# YOCO++: Enhancing YOCO with KV Residual Connections for Efficient LLM Inference

> **一句话总结：** YOCO++在YOCO的底层中引入KV残差连接和缩放因子，将底层K V与第一层K V加权融合后缓存，在保持50% KV缓存压缩率和相同推理效率的同时，超越标准Transformer的性能。

> **TL;DR:** YOCO++ enhances the YOCO cross-layer KV compression method by introducing weighted residual connections between KVs of each bottom-half layer and the bottom layer, plus a scaling factor to facilitate learning, achieving state-of-the-art performance at 50% KV cache compression while matching YOCO's inference efficiency and outperforming the standard Transformer.

## 核心贡献

- 在YOCO架构的下半部分（self-decoder）中引入KV残差连接机制，将每层的K/V与第一层的K/V进行加权组合后缓存，使顶层共享的KV携带更丰富的信息，从而提升模型容量。
- 设计缩放因子（lambda=35）来解决残差权重初始化为零时难以有效学习的问题，实验表明该缩放策略显著改善了残差权重的训练效果。
- 在训练和推理效率上与YOCO完全一致——不增加额外参数量、不增加解码阶段的cache I/O开销、缓存内存占用相同，同时prefilling延迟减半。
- 在1.1B参数模型的实验中，YOCO++在6个常识推理任务上平均准确率达48.99%，超越标准Transformer（48.37%）、YOCO（47.98%）、FusedKV（48.42%）和FusedKV-Lite（48.43%）。

## Key Contributions

- Introduces a weighted KV residual connection mechanism in the bottom-half (self-decoder) layers of YOCO, combining each layer's K/V with the bottom layer's K/V before caching, so the top-half (cross-decoder) layers reuse enriched KV representations.
- Proposes a scaling factor (lambda=35) to address the challenge that residual weights initialized at zero cannot be learned effectively, significantly improving training of the residual connection parameters.
- Achieves identical training and inference efficiency as YOCO: no additional model parameters, no extra cache I/O during decoding, same memory footprint, and half the prefilling latency of the standard Transformer.
- Demonstrates state-of-the-art performance among cross-layer KV compression methods at 50% compression rate, outperforming the standard Transformer, YOCO, FusedKV, and FusedKV-Lite on both training loss and zero-shot commonsense reasoning benchmarks.

## Methodology

YOCO++ builds upon the YOCO (You Only Cache Once) architecture, which is a decoder-decoder model where the bottom-half layers (self-decoder) generate and cache KVs, while the top-half layers (cross-decoder) reuse the middle layer's cached KVs without computing their own. This design reduces KV cache size by 50% and prefilling latency by 50%, but suffers from performance degradation due to direct KV sharing.

The key innovation of YOCO++ is introducing KV residual connections in the bottom-half layers. For each bottom-half layer i (from 2 to L/2), instead of caching the raw K_i, V_i, the method caches a weighted combination:

K'_i = lambda * (alpha^K_{i,1} * K_1 + alpha^K_{i,2} * K_i)
V'_i = lambda * (alpha^V_{i,1} * V_1 + alpha^V_{i,2} * V_i)

where K_1, V_1 are from the bottom layer and alpha parameters are learnable residual weights. A scaling factor lambda (set to 35) is critical for effective learning. The weights are initialized so that the model starts in the same state as YOCO (alpha_{.,1} = 0, alpha_{.,2} = 1/lambda). The top-half layers then reuse K'_{L/2}, V'_{L/2} from the middle layer, which now carry richer information from all bottom-half layers via the residual connections.

Importantly, since the combined KVs are cached rather than the originals, no additional cache I/O overhead is introduced during the decoding stage. The method was validated using 1.1B parameter models (TinyLlama configuration, 22 layers) trained on a 100B token subset of SlimPajama.

## Results & Highlights

- **Zero-shot commonsense reasoning average accuracy**: YOCO++ achieves 48.99%, surpassing Transformer (48.37%), YOCO (47.98%), FusedKV (48.42%), and FusedKV-Lite (48.43%).
- **Training loss**: YOCO++ achieves the lowest training loss among all compared methods on the 100B SlimPajama subset.
- **Prefilling latency**: YOCO++ reduces TTFT to approximately half that of the standard Transformer across all sequence lengths, matching YOCO.
- **Decoding throughput**: YOCO++ maintains the same high throughput as YOCO, consistently exceeding the standard Transformer. Notably, FusedKV suffers throughput degradation due to additional I/O overhead, while YOCO++ avoids this issue entirely.
- **Ablation insights**: Without the scaling factor, YOCO++ degrades to YOCO-level performance (validation loss 2.4333 vs 2.4194), confirming the scaling factor is essential. Removing key residual connections also hurts performance (2.4264 vs 2.4194), showing both key and value residuals contribute.

## Critical Assessment

YOCO++ represents a thoughtful and practical improvement over YOCO. The core idea -- adding weighted residual connections to the KV cache in the self-decoder -- is simple yet effective, and the efficiency analysis is rigorous in demonstrating that no overhead is introduced during the critical decoding stage. The scaling factor trick is an important practical contribution that addresses a real optimization challenge. However, the evaluation is limited to 1.1B parameter models, and it remains unclear how well the approach scales to larger models (7B, 13B, 70B+). The training data (100B tokens) is relatively small by modern standards. Additionally, the paper only evaluates at the 50% compression rate; it would be valuable to understand the trade-off curve at different compression levels.

## Strengths

- **Zero efficiency overhead**: Clever design of caching combined KVs ensures no additional I/O during decoding, which is the bottleneck in production deployments.
- **Strong empirical results**: Outperforms all compared methods including the standard Transformer while using half the KV cache, which is a notable achievement.
- **Simple and practical**: The method adds only 4 learnable scalar parameters per layer, making implementation straightforward and integration easy.
- **Thorough ablation**: Clearly demonstrates the importance of both the scaling factor and the key/value residual connections.
- **Reproducibility**: Code is publicly available and experiments use standard, well-known configurations.

## Limitations

- **Scale of evaluation**: Only tested on 1.1B parameter models; scalability to larger models (7B+) is not demonstrated.
- **Limited training data**: 100B tokens is relatively small; performance gains may differ at larger training scales.
- **Single compression rate**: Only the 50% compression rate is evaluated; the behavior at other compression levels (e.g., 25%, 75%) is unknown.
- **Shallow task evaluation**: Only zero-shot commonsense reasoning tasks are used; performance on more diverse tasks (code generation, math reasoning, long-context tasks) is not assessed.
- **Scaling factor sensitivity**: The choice of lambda=35 is empirical; the sensitivity of performance to this hyperparameter and how to set it for different model sizes is not analyzed.

## Questions for Further Exploration
