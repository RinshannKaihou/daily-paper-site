Syntax Is Easy, Semantics Is Hard: Evaluating LLMs for LTL Translation

Priscilla Kyei Danso, Mohammad Saqib Hasan, Niranjan Balasubramanian, Omar Chowdhury

arXiv: 2604.07321

Date: 2026-04-10

https://arxiv.org/abs/2604.07321

Syntax Is Easy, Semantics Is Hard: Evaluating LLMs for LTL Translation

一句话总结： 本文系统评估了多个主流LLM将自然语言翻译为线性时序逻辑(LTL)公式的效果，发现LLM在语法层面表现尚可但语义正确性严重不足，而将任务重构为Python代码补全问题可显著提升性能。
TL;DR: A systematic evaluation of 7 LLMs on NL-to-LTL translation reveals that models produce syntactically plausible outputs but rarely achieve semantic equivalence with ground truth (~24% average), though reformulating the task as Python AST code completion boosts equivalence to over 61% for the strongest models.

核心贡献

  • 提出了一个六层评估框架，从语法和语义两个维度系统评估LLM在NL->LTL翻译任务上的表现，使用NuSMV作为语义等价性的自动化判定工具
  • 发现LLM在语法正确性方面表现远优于语义正确性（仅24.37%的输出与真值语义等价），时序算子的错误嵌套和作用域误用是主要失败模式
  • 发现将NL->LTL任务重构为Python AST代码补全任务可大幅提升性能（最强模型达到61-72%语义等价率）
  • 在安全领域需求（VERIFY数据集）上进行了额外评估，发现安全需求带来了额外3-16个百分点的等价率下降

Key Contributions

  • A novel six-tier evaluation framework for NL->LTL translation assessing both syntactic and semantic dimensions, using NuSMV as an automated semantic oracle for equivalence and entailment checking
  • Empirical evidence that LLMs achieve ~24% semantic equivalence on average despite producing syntactically valid formulas, with temporal operator mis-scoping as the dominant failure mode
  • Discovery that reformulating NL->LTL as Python AST code completion substantially improves performance (up to 61-72% equivalence for top models like Gemini-2.5-Flash and Gemini-1.5-Pro)
  • Security-domain evaluation on the VERIFY dataset showing 3-16 point equivalence drops and greater sensitivity to correct atomic proposition grounding

Methodology

The authors evaluate 7 general-purpose LLMs (GPT-3.5-Turbo, GPT-4o, GPT-4o-Mini, Gemini-1.5-Flash, Gemini-1.5-Pro, Gemini-2.5-Flash, Claude-3.5-Sonnet) across 6 research questions covering atomic proposition extraction, well-formedness classification, NL-to-future-LTL translation, trace characterization, trace generation, and NL-to-past-LTL translation. They use three progressively more constrained interfaces (minimal, detailed with BNF grammar, and Python AST-based), three prompting strategies (zero-shot, few-shot, zero-shot self-refinement), and multiple ground-truth datasets from textbooks, prior benchmarks, and security specifications. NuSMV serves as the automated semantic oracle for all equivalence and entailment checks.

Results & Highlights

  • Syntax vs. Semantics gap: Average semantic equivalence is only 24.37% across all settings, despite nearly half of outputs being syntactically well-formed; soundness (34.45%) slightly exceeds completeness (31.51%), indicating a tendency toward over-specification
  • Python AST interface effect: The Python code-completion interface yields the strongest results, with top models achieving 72.13% equivalence for future-LTL and 72.96% for past-LTL, a substantial improvement over text-based interfaces
  • Trace tasks: LLMs classify trace satisfiability with moderate accuracy improving from ~57-60% F1 (minimal) to over 80% F1 (Python/AST); they more reliably detect violating traces than satisfying ones
  • Well-formedness: Even the best model (Claude-Sonnet) only achieves 78.33% accuracy on well-formedness classification under the detailed interface; 49.82% of valid formulas are incorrectly rejected
  • Security domain: On the VERIFY dataset (56 samples), equivalence drops 3-16 points compared to the general benchmark; AP grounding causes a 6-19 point gap in security vs. 1-2 point gap in general domain
  • Dominant error pattern: Temporal operator mis-scoping and incorrect nesting remain the primary failure mode across all conditions, not syntactic invalidity

Critical Assessment

This paper makes a timely and well-executed contribution by rigorously separating syntactic from semantic evaluation of LLMs on a formal logic translation task -- a distinction that prior work has largely glossed over. The use of NuSMV as an automated semantic oracle is a methodological strength that enables principled, reproducible evaluation at scale. The finding that Python AST-based interfaces substantially improve performance is practically useful and suggests that structured output constraints can partially compensate for LLMs' weak temporal reasoning. However, the evaluation is limited to short, self-contained NL requirements and does not address longer or more contextual real-world specifications. The data contamination threat is acknowledged but not mitigated, which may inflate reported performance as an upper bound.

Strengths

  • Rigorous separation of syntactic and semantic evaluation dimensions, with formal entailment-based metrics rather than surface-level string matching
  • Comprehensive evaluation across 7 LLMs, 3 interface designs, 3 prompting strategies, and 6 datasets including a security-specific dataset
  • The Python AST reformulation finding has immediate practical implications for improving LLM-based formal specification tools
  • Transparent discussion of evaluation challenges (ontological problem, principle of maximal logical revelation, data contamination)
  • NuSMV-based oracle enables fully automated, reproducible semantic evaluation

Limitations

  • Only evaluates short, self-contained NL sentences; real-world security requirements are typically longer and more interdependent
  • Data contamination concern is acknowledged but not addressed -- results may represent upper bounds
  • The security evaluation uses only 56 samples from the VERIFY dataset, limiting generalizability
  • Model versions changed during the project (Claude-3.5-Sonnet deprecated, Claude-4.5 used for security experiments), introducing some inconsistency
  • Does not evaluate fine-tuned models on equal footing due to their restricted I/O interfaces
  • The Python AST improvement conflates two factors (Python familiarity and structural scaffolding), making it hard to attribute gains solely to either

Relevance

本文对LLM推理质量评估具有重要参考价值。它揭示了LLM在形式语义推理方面的系统性不足——即使语法输出看似合理，语义正确率也远低于预期。这对于依赖LLM进行形式化推理、代码生成或安全策略制定的应用场景具有直接的警示意义，表明在这些关键领域仍需结合符号化验证工具进行交叉检验。

Questions for Further Exploration

  • How would the Python AST approach generalize to more expressive logics beyond LTL, such as CTL*, mu-calculus, or first-order temporal logics?
  • Would incorporating tool-use (e.g., LLMs calling NuSMV as a tool in an agentic loop) further improve semantic correctness beyond the static prompting approaches evaluated?
  • Can the temporal operator mis-scoping errors be systematically categorized and addressed through targeted fine-tuning or specialized prompting strategies?
  • How does performance scale with requirement length and inter-dependency in real-world industrial security specifications?
