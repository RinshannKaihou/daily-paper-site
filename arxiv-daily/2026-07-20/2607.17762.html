01

How AITE works

Two-tier · Orchestrator + ReAct workers

Two-tier loop

ORCHESTRATOR LLM — proposes N distinct ideas per generation, maintains Pareto leaderboard ↓ context: Pareto entries + softmax-sampled off-front WORKERS (M per idea) — ReAct agents in isolated containers, no internet ↓ tools: evaluate · Sionna-docs (FAISS) · run · edit evaluate() → (SUCCESS, metric, complexity) or FAILURE · immutable ↓ ternary verdict: yes / partial / no POST-PROCESS — filter misattribution, refine worker prompt for next gen ↓ optional OPTUNA — multi-objective Bayesian HP tuning expands Pareto front

Input: a task = (natural-language problem, immutable evaluation tool). Output: a set of algorithms spanning a metric–complexity Pareto front — AITE does not pick a single winner; the user chooses the operating point post-run.

Because workers can call the evaluation tool but cannot edit it, the framework prevents "metric gaming."

Anti-gaming by design. Immutable evaluation tool, centralized idea generation (so parallel workers don't collapse onto similar approaches), and ternary verdicts stop a bad implementation from killing a good idea in off-front sampling.

AlphaEvolve / FunSearch lineage, purpose-built for physical-layer problems. Tasks use Sionna Monte Carlo BLER; the authors wrote a new OTFS module (not previously in Sionna).

Metric: NVE = ratio of candidate's coded BLER to a reference equalizer's, averaged over an SNR grid. Complexity proxy: measured latency of torch.compile-compatible code.

03

Strengths & Limitations

side-by-side · no ablations

Strengths 6

  • First convincing demo of agentic LLM evolutionary search discovering novel, SOTA-competitive wireless PHY algorithms — including pilotless OFDM, where essentially no explicit prior art existed.
  • Clean architecture: explicit anti-gaming (immutable eval), anti-collapse (centralized ideas), anti-misattribution (ternary verdict).
  • Open-source release with new Sionna OTFS module and full algorithm derivations in appendices B & C — unusual transparency.
  • Fair baselines: every baseline got identical Optuna tuning and agentic latency optimization.
  • Honest about LLM-quality dependence and the open run-to-run variability question.
  • The pilotless receiver itself (multiscale-moment blind init + EM + ratio-prior BP + affine correction) is publishable signal processing in its own right.

Limitations 8

  • Run-to-run variability unmeasured — authors flag as open; replication not guaranteed.
  • Proprietary LLM dependency: all headline results use GPT-5.5; open-weight models fail to implement good ideas reliably.
  • Narrow NVE SNR grids (2 points each: {13,16} / {15,20} dB) — Pareto comparisons only locally meaningful.
  • Cost: ~USD 2500 / 4 days / 16 GPUs per task limits iteration and independent replication.
  • No controlled ablations isolating verdict filter, prompt refinement, or exploration split.
  • Latency tax: pilotless receiver is ~1.8× slower than the neural receiver it matches in goodput.
  • Eval-tool bias: BLER Monte Carlo + latency dominates runtime, may favor algorithms fast to simulate, not just fast to run.
  • Generalization beyond two tasks and beyond Sionna / PyTorch is not demonstrated.

Open questions

for further exploration
What is the empirical run-to-run variance — how often does a fresh run fail to reach NVE ≤ 1.0 on the pilotless receiver?
How much of the gain comes from each framework component (verdict filter, prompt refinement, off-front sampling T, HP tuning)? Can a controlled ablation attribute it?
Can the framework run on open-weight 70B-class models with stronger tool-use scaffolding, or is GPT-5.5-class capability truly required?
Does the discovered pilotless receiver generalize beyond M=6 — to higher orders, multi-stream, or MIMO?
Could a separate validation tool detect and penalize overfitting to the TDL-C training channel or specific hardware?
How sensitive are results to the complexity proxy (latency on RTX PRO 6000)? Would a different hardware target yield a different winning family?
Can the budget be reduced by replacing some Monte Carlo BLER evaluations with cheaper surrogates without corrupting the search?
