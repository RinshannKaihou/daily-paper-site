Benchmarking Resource-Efficient LLMs for Research Topic Ontology Generation in the Biomedical Field

Tanay Aggarwal · Angelo Salatino · Francesco Osborne · Enrico Motta — Knowledge Media Institute, The Open University
llm-fine-tuning ontology-generation biomedical-nlp small-language-models knowledge-organization-systems
Key Finding

LoRA fine-tuning lifts small-model biomedical relation F1 from 0.55 to 0.89 — closing the reasoning gap that chain-of-thought prompting couldn't.

1 Key metric
0.916F1

Fine-tuned gemma-9b on the 800-pair MeSH-Rel-4K test set — best of five ≤9B models.

0.669 standard → 0.716 CoT → 0.916 fine-tuned
2 Why this matters

A reproducible, consumer-GPU LoRA recipe that brings 3B–9B models to ~92% F1 on biomedical ontology relations — directly practical for domain knowledge-graph construction.

3 Critical edge

No large-LLM or classic BERT baseline, no significance testing on an 800-pair test set, and the headline "resource-efficient" is asserted — never measured (no latency, memory, or cost numbers).

01 Methodology

Four-class relation task over topic pairs (tA, tB) → broader · narrower · same-as · other. Built on MeSH-Rel-4K: 4,000 balanced pairs from the Jan 2025 MeSH release, split 2,800 / 400 / 800 with strict node-isolation to prevent leakage.

Models Five ≤9B open-source LLMs, all 4-bit quantised: mistral-7b, llama-3.2-3b, gemma-2-9b, Phi-3.5-mini, zephyr-sft. LoRA (r, α) tuned per model.
Strategy A Standard prompting — single template, model emits relationship: TYPE.
Strategy B CoT two-way — define → reflect → classify, swap topic order, resolve via referee rules from prior work.
Strategy C LoRA fine-tuning — Unsloth + PEFT + BitsAndBytes on 2,800 pairs; KoboldAI/llama.cpp for inference.
Infra Google Colab, NVIDIA A100 / L4. Code, prompts, configs public on GitHub.

02 Headline results

Fine-tuning is decisive. Average F1 across the five models:

  • Standard prompting — 0.494  baseline.
  • CoT two-way — 0.548  only +5.4pp over standard (±3.5). Surprisingly modest.
  • LoRA fine-tuning — 0.889  +34.1pp absolute, +61.6% relative.

Per-relation (fine-tuned, avg): other 0.943 (easiest) · broader 0.882 · narrower 0.866 · same-as 0.863 (hardest, P 0.873). The dominant error is conflating same-as with hierarchical relations — driven by MeSH's dense lexical overlap (all five models mislabel "alitretinoin" → "tretinoin" as same-as).

04 Strengths

  • Self-contained, balanced, publicly released dataset (MeSH-Rel-4K) with strict node-isolation against leakage.
  • Clean three-way comparison (standard / CoT / fine-tune) under identical prompts and splits.
  • The +60.5pp llama-3b result — a 3B model matching a 9B after LoRA on 2,800 pairs — is striking and practical.
  • Per-class confusion analysis correctly pins the same-as vs. hierarchical boundary as the real open problem.
  • Fully reproducible on consumer Colab GPUs (A100/L4); low barrier for ontology maintenance.

05 Limitations

  • No external baselines — no GPT-4o / Llama-3-70B upper bound, no classic BERT subsumption classifier.
  • 800-pair test set, no confidence intervals, no significance tests, no seed-variance analysis.
  • Single domain (MeSH) and only four coarse relations; SNOMED CT / GO / part-of untested.
  • "Resource-efficient" never quantified — no latency, memory, or cost-per-relation numbers.
  • CoT referee rules borrowed from the authors' engineering study, not re-validated for biomedicine.
  • 4-bit quantisation effect on fine-tune quality not discussed as a limitation.

06 Open questions for follow-up

  1. Does a single large frontier LLM (GPT-4o / Claude / Llama-3-70B) close the gap under few-shot prompting — with zero training data?
  2. Does stratifying the test set by lexical-overlap score re-rank the models, and can targeted augmentation push same-as precision past 0.92?
  3. Does the recipe transfer to SNOMED CT, Gene Ontology, or CSO / ACM CCS without re-tuning LoRA hyperparameters?
  4. Are the symmetric same-as ⇄ hierarchical confusions in mid-sized models reducible via calibration or lexical-overlap-aware prompts?
  5. What are the real latency / memory / cost-per-relation trade-offs — does "resource-efficient" hold up quantitatively?
  6. Can same-as boundary errors be auto-flagged (embedding-cosine thresholds) before insertion, to prevent hallucinated ontology loops?
