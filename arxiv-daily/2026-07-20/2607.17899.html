SEAM-V: A Hybrid-Decoupled RISC-V Vector Processor with Backend-Visible EP Context for Sustained Vector Throughput

arXiv:2607.17899  ·  extends ISCAS 2026 (Ref. [11])  ·  RVV manual v20260120
Key finding

A hybrid-decoupled RVV front end carries execute-packet context across the request interface so a dynamic backend can suppress same-EP hazards and prefetch — reviving vector supply without touching the ISA.

Methodology § how it's built

SEAM-V lifts the software–hardware interaction granularity from single vector instructions to whole kernels via a local execution path, then propagates packet identity through serialization so the dynamic backend can act on it.

Local front-end path

  • TSU — 4-entry task FIFO, one active; submission-order scheduling.
  • IPU — 128-bit fetch into 2×512 B ping-pong buffers; replays loop bodies on resolved backward branches, closing the redirect gap.
  • VLIWPU — scans 16-bit slots in 128/256-bit logical packets; forms EPs up to 8 slots; branch/system/dependence boundaries terminate an EP; cross field spans physical fetch breaks.
  • HEU — splits each EP into a scalar slice (3-wide local backend) and a vector slice (→ VDU).

Backend-visible EP context

  • Each in-flight request carries a 1-bit ep_id toggle + request-bound prefetch context (pf_disable + pf_mode 1X/2X/4X/8X); loop-active supplied independently to the memory path.
  • Software EP contract — compiler/programmer must clear p-bits for any RAW/WAR/WAW or memory-ordering relation needing dynamic enforcement.
  • HINTs ride the standard lui x0, imm20 form — no new architectural state, no RVV semantic change.

Interface contract & workloads

  • Operand-safe handoff ≠ vector completion: VDU acks once scalar operands + vset are captured; lets the front end advance past long-latency vector ops.
  • Baseline: Ara2-based TC, identical backend scale & memory config; 17 kernels across 1-D variable-AVL, BLAS/matrix, and fixed-size applications.

Strengths § what lands

  • Architecturally clean & ISA-compatible — standard RVV + standard HINT form, no new architectural state; preserves portability.
  • Honest ablation — explicitly reports Full is only ~1% above PF+Haz, with per-kernel regressions on STENCIL3/FIR5.
  • Right experiment for the claim — AVL sweep (32→4096) disentangles the short-vector overhead win (3× @32) from sustained per-element efficiency (~1.45–1.50× @1024–4096).
  • Microarchitectural counters tie headline speedups to specific hardware activity (hazard-elimination, PF coverage, residual pressure).
  • Operand-safe handoff is a sound decoupling; stream-break recovery for prefetch is a thoughtful correctness detail.

Limitations § what's thin

  • Physical-design results promised but absent — no mm², gates, mW, or GOPS/W anywhere in the evaluation.
  • Single baseline (Ara2-TC) — no head-to-head vs Saturn, Titan-I, TROOP, Vitruvius+, or Spatz, the most relevant dynamic RVV backends.
  • Microkernel-heavy suite — 12 of 17 are 1-D/BLAS; no end-to-end DNN layer or full scientific kernel.
  • Software EP contract has no described automation or verification — hand-annotated only? compiler pass? no safety-net discussion.
  • Cross-EP overlap empirically near-inert (≤0.18) — one of five advertised mechanisms may not pay off.
  • Significant residual pressure on SYRK/SPMV/JACOBI2D (0.92–0.99) correlates with the smallest gains.

Questions for further exploration § open threads

  1. What are the actual synthesis area / power / energy-efficiency numbers promised in the contributions and §V intro? Without them the abstract's efficiency claims are unsupported.
  2. How does SEAM-V compare head-to-head against Saturn, Titan-I, TROOP and Vitruvius+ on the same kernels and AVL sweep — not only against Ara2-TC?
  3. Can HINT annotations be compiler-generated (LLVM loop-aware pass), and what is the verification story for the software EP contract — static analyzer or runtime checker?
  4. Why does buffered vector early issue contribute so little (≤0.18)? Is it gated too conservatively, or do the kernels lack cross-EP independence?
  5. Behavior on end-to-end workloads (attention, conv-block, sparse CG iteration) where control intensity, irregular memory and AVL heterogeneity coexist — do the SYRK/SPMV/JACOBI2D pathologies generalize?
  6. Does the design extend cleanly to multi-core / multi-task throughput under a shared AXI interconnect, or does the 4-entry task FIFO become a new serialization point?
