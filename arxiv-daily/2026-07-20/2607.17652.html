Methodology

01 / HOW IT WORKS

Block-wise diffusion LLMs decode fixed-length blocks left-to-right — positions inside a block denoise in parallel, but blocks are strictly serial: Bg+1 cannot start until Bg is committed. FlowBlock's insight: LLaDA-2.1's token-to-token (T2T) editing turns that hard data dependency into a schedulable resource — a downstream block needs only an informative upstream draft, not a finalized one.

M1Gated Wavefront Decoding (GWD)

A sliding window [L, R) of up to W blocks runs concurrently; each step does one forward over the W·bl window positions and applies the joint M2T∪T2T update. A new block is admitted when frontier readiness ρ(B) ≥ θ_spawn; the leftmost finished block retires and freezes KV into the prefix. A W-shaped block-causal mask makes committed-prefix KV bit-identical to serial decoding.

M2Heterogeneous Wavefront Packing (HWP)

Each request keeps its own wavefront state (L_b, R_b), but all windows are gathered into a padding-free dense [B, q] batch (q = W·bl) using per-row gathers, absolute-position RoPE indices, and a per-row block-diagonal packed mask. Shape-stable → no recompilation, no allocator churn, no straggler lockstep.

M3Setup

Implemented in dInfer (SGLang backend), no post-training, no kernel changes. Defaults: W=2, θ_spawn≈0.6, block length 32, max len 2048. Baselines run in the same engine to isolate the scheduling effect: LLaDA-2.0-mini, LLaDA-2.1-mini, and D2F (training-based, distilled 2.0-mini).

Strengths & Limitations

03 / BALANCE

▲ Strengths

  • Training-free, applicable to unmodified checkpoints — strong for serving teams.
  • Bit-identical frozen-prefix KV claim rigorously argued via the W-shaped mask + absolute-position layout.
  • HWP is genuinely novel — converts asynchronous per-request progress into dense, shape-stable batches; solves a problem many dLLM serving systems silently suffer.
  • Comprehensive ablations: θ_spawn, W, block length, generation length, plus a dedicated HWP-vs-batch-synchronous study.
  • Open-source code & open-weight model; same-engine baselines remove system confounds.
  • Improves accuracy rather than trading it off — rare for an acceleration paper.

▼ Limitations

  • Single model only — LLaDA-2.1-mini; no larger checkpoints, no other self-correcting families, no non-self-correcting block baselines.
  • D2F baseline is weak (70.52 avg, under-scaling), so the 16× headline is less informative about the field.
  • No direct comparison to Fast-dLLM or dKV-Cache under the same harness — the most direct apples-to-apples gap.
  • θ_spawn is hand-tuned per task; paper notes task-specific selection but gives no auto-tuning recipe.
  • 8×GPU data-parallel setup obscures per-device throughput; GPU type unspecified beyond "80GB".
  • Long-form behavior (>2048 tokens) — where upstream draft drift could compound — is not studied.

Open Questions

04 / WHAT'S NEXT
Does FlowBlock hold on larger self-correcting dLLMs (8B+) where per-block compute and KV pressure differ substantially?
Can the readiness gate ρ(B) be replaced by a learned / adaptive schedule that auto-tunes θ_spawn per task?
What happens on long-form generation (>2048 tokens) where upstream draft drift could outrun T2T repair?
How does it compare head-to-head with Fast-dLLM and dKV-Cache under the same engine and checkpoint?
Is the W-shaped mask + absolute-position packing transferable to non-self-correcting block dLLMs, or does correctness depend on editability?
Can HWP's per-row gather/scatter be fused with FlashAttention kernels to remove the q-token window's data movement overhead?
Does the readiness gate misfire on low-confidence / ambiguous generations (creative writing), and how should θ_spawn be set outside math/code?
Does θ_spawn interact with τ_mask / τ_edit thresholds in ways that change the speed–accuracy frontier across domains?
