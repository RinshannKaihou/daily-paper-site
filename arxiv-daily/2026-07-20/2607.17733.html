MXSens: Sensitivity-Aware Mixed-Precision Quantization for Efficient LLM Inference

Simla Burcu Harma · Danila Mishin · Zhengyuan Su · Ayan Chakraborty · Elizaveta Kostenok · Dongho Ha · Babak Falsafi · Martin Jaggi · Yunho Oh · Amir Yazdanbakhsh
EcoCloud/EPFL · Google DeepMind · Korea University · MangoBoost · Tsinghua  ·  Preprint (ICLR-style), not yet peer-reviewed  ·  code ↗
Key Finding

A training-free, Hessian-guided 4/6/8-bit quantizer on the MXINT format beats QuaRot, RRS, Atom and QUIK.

Why This Matters

Directly relevant to efficient LLM inference, post-training quantization, and hardware–algorithm co-design — especially for anyone tracking the microscaling ecosystem (OCP MX standard, Blackwell / AI100 native support).

Critical Edge

No actual hardware latency was measured — the central ~3% overhead claim is an analytical estimate only, and MXINT is incompatible with rotation-based SOTA, bounding gains to microscaling hardware.

01 Methodology

Three ingredients layered on the MXINT microscaling format — entirely post-training, no fine-tuning, no rotation, no architecture changes.

  1. MXINT format. 32 contiguous values share one power-of-two scale (8-bit exponent); processed by standard INT MAC units. Native on NVIDIA Blackwell & Qualcomm AI 100. Incompatible with rotation (QuaRot/SpinQuant/RRS) — rotations break the per-block shared-scale symmetry.
  2. Column-wise sensitivity (Alg. 1). Empirical Hessian H = XᵀX over 128 C4 calibration samples; diagonal H_jj = Σₖ X²_kj is the per-column sensitivity. Top-32 columns per layer get 8-bit — exactly one MXINT block, zero intra-block waste.
  3. Layer-wise sensitivity (Alg. 3). Quantize one layer, measure L2 error at the final output vs FP32, normalize to [0,1]. Early layers dominate for LLaMA/Mistral; middle layers for Qwen1.5.
  4. Triplet allocation (Alg. 2 + Thm. A.1). Top-32 fixed at 8-bit; 6-bit count per layer ∝ layer sensitivity × global control R; rest default 4-bit. R has a closed form that hits target bitwidth B exactly.
  5. Ablation-justified. 6→8-bit upgrade cuts PPL error 0.5→0.3 at +0.005 bits (A.4); 32 8-bit cols is Pareto-optimal vs 32–160 sweep (A.7); C4 ≈ WikiText calibration (A.9).
  6. Joint scope. Weights, activations, and KV-cache quantized together; KV uses a separate 4–8 bit allocation (avg 4.03 bits). Softmax, LayerNorm, embeddings, and lm_head stay FP16.

03 Strengths & Limitations

Strengths

  • Clean training-free recipe that fits the microscaling block structure instead of fighting it.
  • Closed-form bitwidth allocation (Thm. A.1) — exact average-bitwidth constraint, no heuristic search.
  • Two-axis sensitivity (column-wise Hessian diagonal + layer-wise L2 error) with cross-architecture observations.
  • Extensive ablations: 3-6-8 vs 4-6-8 across four models, 32-vs-160 8-bit cols, 6→8 dynamics, C4-vs-WikiText calibration.
  • Hardware section enumerates three concrete deployment paths on Blackwell / AI100.
  • Public code release and clear pseudocode for all three algorithms.

Limitations

  • No measured latency/throughput on real Blackwell/AI100 — the 3% overhead is an analytical estimate only.
  • Baseline bitwidths not strictly matched (4.00/4.25/4.27/4.50); Atom appears at a disadvantageous config.
  • Loses on Qwen1.5-7B PPL and several Mistral-7B tasks — heuristic not uniformly robust.
  • Narrow evaluation: only WikiText-2 PPL + 0-shot commonsense QA; no chat, code, or long-context tasks.
  • Missing comparisons with QoQ, COMET, MicroScopiQ, AMXFP — all directly relevant.
  • 0.25-bit MXINT exponent overhead added to MXSens but not consistently to baselines.
  • Small calibration set (128 C4 samples); 70B sensitivity estimates may be noisy.

04 Open Questions

  • How does MXSens perform on actual Blackwell/AI100 hardware — is the 3% latency estimate realistic end-to-end?
  • Can the H_jj column-sensitivity signal combine with rotation on non-MX hardware to recover rotation's benefits?
  • Why does MXSens underperform on Qwen1.5-7B PPL — does the middle-layer pattern break the early-layer amplification assumption?
  • Behavior on long-context, chat, code benchmarks (MT-Bench, AlpacaEval, HumanEval) where 0-shot commonsense is a weak proxy?
  • Are gains orthogonal and additive when paired with QoQ-style activation calibration, SmoothQuant, or GPTQ weight updates?
  • Is fixed top-32 8-bit allocation optimal for very wide/deep models (LLaMA-3-70B, Mixtral) where outlier distributions differ?
  • Can sensitivity analysis be made online/adaptive to distribution shift during deployment, rather than fixed at calibration?
  • Does the 0.25-bit exponent accounting flip any headline comparison if applied uniformly to all baselines?
