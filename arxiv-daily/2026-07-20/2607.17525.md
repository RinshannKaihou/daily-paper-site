> **一句话总结：** 作者提出 FailureAtlas (FA)，一个针对 LiteLLM/Portkey/OpenRouter 这类多供应商 LLM 网关的"层 × 可检测性"双层失败模式分类法 (5×2 矩阵)，用 5 条经过证据验证的失败案例 (含 3 个可复现脚本) 说明：在 LLM 服务栈中，运维危害最大的失败是返回 HTTP 200 但悄悄污染语义载荷 (丢历史、工具调用 JSON 错乱) 的"沉默失败"，传统指标 (延迟/错误率/SLA) 对其完全失效。

> **TL;DR:** FailureAtlas (FA) introduces a two-axis (Layer × Detectability) taxonomy of failures in multi-provider LLM gateways (LiteLLM/Portkey/OpenRouter) and populates it with 5 evidence-verified entries (3 with standalone reproduction scripts). Its central finding is that the most operationally severe failures are "Silent": they return HTTP 200, pass every health check, and corrupt the semantic payload (e.g., in the authors' ContinuityBench harness, CPR collapsed from near-perfect at C=5 to ~28% at C=100 concurrent sessions; a thundering-herd retry storm against a mock LLM provider permanently failed 25% of 100 parallel agents). The paper argues that existing Prometheus/Datadog-style monitoring, defined at the transport layer, is structurally blind to this failure class and requires semantic-level observability.

## 核心贡献

- **双轴失败分类法 (5×2 矩阵)：** 第一轴 Layer 把多供应商 LLM 服务栈分成五层 (L1 Network/Transport、L2 Streaming/Protocol、L3 State/Session、L4 Model Behavior、L5 Governance/Cost)；第二轴 Detectability 把失败分成 Loud (能被标准监控抓到) 与 Silent (返回 HTTP 200、需要语义级观测才能发现)。两轴交叉得到一张可操作的故障定位网格。
- **证据驱动的失败目录：** 5 条通过严格准入标准 (具体、可独立验证、机制可解释) 筛出的失败模式，其中 2 条来自作者在 ContinuityBench (CB) 压测中的第一手发现，3 条来自 LiteLLM 公开 issue tracker 的已验证报告 (litellm#33678, litellm#20256, litellm#21033)。
- **两个完整可复现的深度案例：** (1) State/Session 层的并发 read-modify-write 竞态，导致对话历史跨会话"漂移"且只返回 HTTP 200；(2) Governance 层的 thundering-herd 重试风暴，把瞬时 502 放大成自我施加的 DoS。每个案例都附最小复现代码、机制解析、已验证输出与修复方案。
- **三个独立复现脚本：** 在开源仓库 `github.com/Vishal-sys-code/failure-atlas` 提供 YAML schema + 自动化渲染 + 贡献指南，定位为社区可扩展的 CVE-like 故障数据库。
- **沉默失败的主导性论证：** 通过三重架构错配 (无状态架构 vs 有状态负载、传输层成功定义、缺乏语义监控惯例) 解释为什么 LLM 网关特别容易产生"健康但失真"的失败，并把 L4 (Model Behavior) 整行留空作为一项独立发现——揭示该层目前最缺乏可证据化的观测工具。

## Key Contributions

- A **two-axis taxonomy (Layer × Detectability)** producing a 5×2 matrix that classifies LLM-gateway failures both by where in the stack they originate (L1 Network/Transport, L2 Streaming/Protocol, L3 State/Session, L4 Model Behavior, L5 Governance/Cost) and by whether they trigger standard monitoring (Loud vs. Silent). The taxonomy is explicitly designed to be minimal and triage-actionable, rejecting symptom-only or three-axis "severity" alternatives.
- An **evidence-grounded catalog of 5 verified failure modes**, each passing three inclusion criteria (concrete & specific, independently verifiable provenance, mechanistically explainable). 2 entries were discovered first-hand during ContinuityBench stress testing; 3 are sourced from verified LiteLLM GitHub issues (#33678, #20256, #21033).
- **Two fully characterised case studies** — (1) a State/Session concurrency race condition that silently corrupts conversation history across sessions, and (2) a Governance/Cost retry-storm thundering herd that converts transient 502s into a self-inflicted DoS — each with mechanism, minimal reproduction, verified output, and mitigation.
- **Three standalone reproduction scripts** and an open, extensible catalog repository (`github.com/Vishal-sys-code/failure-atlas`) with a structured YAML schema, automated rendering pipeline, and contribution guidelines, explicitly modelled on CVE databases and community-curated "awesome lists."
- A **structural argument about why silent failures dominate** this infrastructure class, grounded in three architectural mismatches (stateless-by-design architecture vs. stateful workloads; transport-layer success metrics; lack of semantic-monitoring conventions). The intentionally empty Model Behavior (L4) row is presented as a legitimate finding that exposes the field's weakest observability tooling.

## Methodology

The paper is empirical and taxonomy-driven rather than benchmark-driven. The authors design FA as a Layer × Detectability matrix and populate it from two evidence streams.

**Taxonomy design (Section 3).** The Layer axis is drawn at boundaries where the failure mechanism, responsible code owner, and mitigation strategy all change simultaneously (a pragmatic anti-over/under-splitting criterion). The Detectability axis distinguishes Loud failures (HTTP 5xx, unhandled exceptions, container crashes, hard 429s, health-check timeouts) from Silent ones (HTTP 200 with corrupted semantic payload). The authors argue severity is almost entirely determined by detectability, which is why they rejected an explicit "severity" third axis.

**Evidence sourcing (Section 4.1).** Two streams feed the catalog:
1. *First-hand stress testing* during ContinuityBench (CB) Phase 2 campaigns, where hundreds of parallel agent instances were run against a mock LLM-provider gateway simulating real-world latency, rate limits, and transient 502s. The race condition and retry storm emerged organically under sustained load.
2. *Structured issue-tracker surveys* over LiteLLM, Portkey, and OpenRouter, using scoped queries like `SSE stream "cut off" OR "truncated"` (Streaming/Protocol layer) and `"connection timeout" gateway failover` (Network/Transport layer).

**Inclusion criteria and validation (Section 4.2).** Each candidate had to satisfy three criteria — concrete and specific, independently verifiable provenance (persistent URL to a GitHub issue, post-mortem, or merged fix), and mechanistically explainable. Validation was manual: the authors read each issue thread, examined logs, traced execution paths through the open-source codebase, and verified that the merged fix matched the described mechanism.

**Standalone reproductions (Section 4.3).** Where possible, failures are reduced to a minimal script: an `asyncio` script for the race condition (shared vs. isolated state under concurrent writes), a mock rate-limited provider with 100 concurrent agents for the retry storm, and the exact reproduction from LiteLLM issue #33678 adapted to statically captured SSE chunks for the index collision. The remaining two entries (event-loop block, Redis semaphore leak) are too tied to external infrastructure (Kubernetes probes, live Redis) for standalone scripts and ship as README instructions instead.

**Case-study depth (Section 6).** The two first-hand failures are presented at mechanism-level detail — Vulnerability → Mechanism → Fix & Implications — using the Continuity Preservation Rate (CPR) from CB as the semantic signal that surfaced the otherwise invisible race condition.

## Results & Highlights

This is a taxonomy/systems-observability paper rather than a benchmark-comparison paper; the quantitative claims describe the catalog composition and the in-house stress tests rather than accuracy gains over baselines. The concrete numbers reported are:

- **Catalog composition:** The FA catalog contains exactly **5 verified failure modes**, with **3 of 5 entries (60%) sourced from a single project's public tracker (LiteLLM)** and **2 of 5 (40%) first-hand discoveries from the authors' own ContinuityBench (CB) campaigns**. **3 of the 5 entries include standalone reproduction scripts**.
- **Silent vs. Loud split:** **2 of the 5 entries (40%) are classified Silent** (the State/Session race condition and the Streaming/Protocol SSE index collision) and the authors argue these are disproportionately operationally severe because no standard metric flags them.
- **Race condition magnitude (CB harness, Section 6.1):** At low concurrency (C = 5 concurrent agent sessions) the failover proxy achieved a **near-perfect Continuity Preservation Rate (CPR)**; at production-scale concurrency (C = 100) **CPR abruptly collapsed to ~28%** while the proxy continued returning **HTTP 200 OK on every request** — i.e., an ~72-percentage-point drop invisible to any latency/error-rate/uptime metric.
- **Retry storm magnitude (CB harness, Section 6.2):** With **100 concurrent agent instances** retrying on a fixed interval after a simulated upstream outage, the synchronised thundering herd **permanently failed ~25% of requests** despite the underlying cause being a "momentary hiccup," while also generating billable partial-completion tokens for zero successful task completions.
- **Empty L4 row:** The **Model Behavior (L4) row of the 5×2 matrix is entirely empty** — explicitly framed as a legitimate finding rather than a survey gap, because the authors could not find a single evidence-grade, reproducible infrastructure-level bug report meeting their inclusion criteria in this layer.
- **Adoption context:** LiteLLM alone is cited as having **over 18,000 GitHub stars** and routing millions of API calls per day, motivating the practical relevance of the failure surface studied.

## Authors & Credibility

The paper is authored by **Vishal Pandey** (Metriqual, London, UK) and **Gopal Singh** (Metriqual, Athens, GR). Credibility signals and risks:

- **Transparency:** The catalog is open-sourced at `github.com/Vishal-sys-code/failure-atlas` with YAML schema and reproduction scripts, which is appropriate for an evidence-grounded catalog paper.
- **Companion-paper linkage:** FA is explicitly a companion to the authors' own ContinuityBench (CB) benchmark [ref 1], and the two headline case studies (race condition, retry storm) were discovered via CB. This gives a coherent narrative but introduces self-sourcing and circular-citation incentives.
- **Evidence provenance:** 3 of 5 catalog entries cite specific, verifiable GitHub issue numbers (litellm#33678, #20256, #21033) and trace through merged fixes, which is the right standard for this kind of work.
- **Venue/blind-review status:** The arXiv listing (cs.LG, v1, 20 Jul 2026) carries no peer-review signal; readers should treat the taxonomy as a position/community-resource paper rather than a formally vetted study.

## Critical Assessment

- **Novelty:** Genuine. The paper correctly identifies a real gap — neither classical distributed-systems taxonomies (Byzantine faults, CAP, Gunawi et al.'s cloud-bug studies) nor LLM evaluation suites (HELM, MMLU, SWE-bench, WebArena, RULER) characterise the failure surface that emerges specifically from serving LLMs through multi-provider gateways. The Detectability axis (Loud vs. Silent) is a useful and non-obvious framing that encodes the infrastructure-correctness vs. semantic-correctness asymmetry. However, the paper is honest that several catalogued patterns (retry storm = thundering herd [ref 2]; race condition = read-modify-write anomaly; semaphore leak = liveness failure) are classical distributed-systems bugs "wearing new clothes" — the contribution is contextualisation, not pattern discovery.
- **Rigor:** Mixed. Inclusion criteria (concrete, verifiable, mechanistic) are appropriate and applied manually, but the sample is tiny (n = 5) and the survey is effectively a LiteLLM tracker scan — the authors themselves concede three of five entries originate from LiteLLM because of its transparency and adoption, not because other gateways are immune. There is no systematic methodology for catalogue completeness (no saturation analysis, no inter-rater reliability on classification, no enumeration of how many candidates were rejected and why at scale). The headline quantitative findings (CPR ~28% at C=100; 25% permanent failure of 100 agents) come from the authors' own mock-provider harness, with no external replication and no reported confidence intervals or run counts.
- **Reproducibility:** Strong on the three scripted entries (standalone scripts in a public repo), weaker on the two infrastructure-tied entries (event-loop block, Redis semaphore leak) that ship only as README instructions requiring Kubernetes probes / live Redis. The YAML schema and rendering pipeline are well specified.
- **Significance:** Practically high for the LLM-infrastructure operations community — this is a useful vocabulary and detection-mental-model document — but academically moderate given the small catalog and the reliance on classical patterns. The empty Model Behavior row is the most provocative finding but is under-developed.
- **Red flags / overclaims:**
  - The phrase "the most operationally severe failures are silent" is asserted strongly but supported by only two silent entries out of five. The survivorship-bias argument is plausible but not measured.
  - The CPR ~28% number is presented as a finding about LLM gateways but is measured on the authors' own prototype failover proxy during their own benchmark; it is more accurately a finding about one specific state-management bug in one specific harness, generalised rhetorically to the whole infrastructure class.
  - The mutually-citing FA ↔ CB pair, both by the same authors at the same company (Metriqual), creates an incentive for each to inflate the other's importance; the paper does not address this conflict of interest.
  - "Three entries include standalone reproduction scripts" is presented as a strength, but two of those three reproduce well-known patterns (race condition, thundering herd) — the only LLM-specific reproduction (the SSE index collision) is adapted from a third party's existing LiteLLM script rather than novelly authored.

## Strengths

- The **two-axis (Layer × Detectability) taxonomy is minimal, memorable, and actionable** — it directly answers the two triage questions practitioners actually ask ("where in the stack?" and "why didn't we catch it?"). The rejection of a third "severity" axis is well argued.
- **Honest framing of empty cells:** The empty Model Behavior (L4) row is presented as a legitimate finding about the field's observability gaps rather than papered over — a sign of intellectual integrity.
- **Mechanism-level depth** on the two first-hand case studies (Vulnerability → Mechanism → Fix) makes them genuinely teachable, including concrete mitigations (per-session deep-copy + locking for the race; exponential backoff with jitter + global circuit breaker for the retry storm).
- **Verifiable provenance:** Citing specific GitHub issue numbers and tracing through merged fixes sets a good evidentiary standard for an empirical catalog.
- **Open, extensible community resource** with YAML schema, rendering pipeline, and CVE-style contribution model is the right design choice for a problem where new failure modes will outpace any single team.

## Limitations

- **Non-exhaustive catalog (n = 5):** Too small to support strong generalisation claims; the authors acknowledge this openly.
- **Single-project sourcing bias:** 3 of 5 entries come from LiteLLM's tracker; Portkey and OpenRouter contribute nothing concrete, and proprietary gateways are not studied at all.
- **Self-sourcing bias:** The two most thoroughly characterised entries were discovered by the authors during their own CB evaluations, which were explicitly designed to stress state management and retry policies — guaranteeing those failure modes would surface.
- **Varying evidence quality:** The two first-hand case studies include controlled experimental data; the surveyed entries range from multi-user reports (Redis semaphore leak) to single detailed bug reports.
- **No statistical methodology:** No saturation analysis, no inter-annotator agreement on Loud/Silent or Layer classification, no confidence intervals on the CPR or failure-rate numbers.
- **Mock-provider, not production, evidence:** The headline numbers (CPR ~28%, 25% permanent failure) are measured against a simulated provider; real production traffic may behave differently.
- **Mutual-citation / conflict-of-interest pattern with CB** is not disclosed or discussed.
- **Taxonomy axes are pragmatic, not formally derived:** Layer boundaries could legitimately be drawn differently and a "recovery time" axis might add value.

## Relevance

这篇论文对研究/工程 "LLM agent 基础设施可靠性、连续性与可观测性" 的方向高度相关：它把"沉默语义失败"提升为一类需要语义级监控 (而非仅 HTTP/latency 监控) 的工程问题，与我们关注的 ContinuityBench / 多轮 agent 状态一致性议题直接互补。对从事 LLM 网关、failover、retry/backoff、streaming 工具调用协议实现的工程师，它提供了一份可直接借鉴的故障定位与缓解清单。

## Questions for Further Exploration

- How would the Layer × Detectability distribution change with a larger, community-sourced catalog (e.g., 50–100 entries)? Would Silent failures still appear disproportionately severe, or merely disproportionately under-reported?
- Can the Model Behavior (L4) row be populated with evidence-grade entries using continuous semantic evaluation (e.g., ContinuityBench) running in production — and what statistical methodology would distinguish genuine model drift from stochastic variation?
- What is the inter-annotator agreement on the Loud vs. Silent classification when applied by independent practitioners to the same set of issue reports?
- For the concurrency race condition, how does CPR degrade as a continuous function of concurrency between C=5 and C=100, and is the collapse threshold predictable from the proxy's locking design?
- Are the catalogued failure modes (e.g., SSE index collision, Redis semaphore leak) reproducible across Portkey and OpenRouter, or are they LiteLLM-specific implementation bugs?
- What would a "semantic observability" stack look like operationally — can existing tools (OpenTelemetry, Prometheus) be extended to inspect conversation-history integrity and tool-call payload correctness at line rate, and at what cost?
- Does the FA taxonomy generalise to non-OpenAI-compatible streaming protocols (Anthropic, Google) where the tool-call index semantics differ?
