> **一句话总结：** 提出一种异构的端侧自适应流水线,将商业推理加速器 Hailo-8L 复用于冻结主干的 INT8 特征提取,只在树莓派 5 CPU 上以 FP32 微调分类头,在 ResNet18/CIFAR-100 上训练墙钟时延相对 CPU 基线加速最高 15.4×、每样本能耗降至约 1/14,同时通过 PTQ 修复策略把精度损失控制在 ~0.06 个百分点。

> **TL;DR:** The authors repurpose a commercial inference accelerator (Hailo-8L, ~13 TOPS at 1.5 W) for on-device training: the pre-trained backbone is frozen and run in INT8 on the accelerator, while only an FP32 classification head is fine-tuned on the Raspberry Pi 5 CPU. With proper post-training quantization (PTQ) restoration, on CIFAR-100 the heterogeneous pipeline trains ResNet18 at 6.04 ms/sample vs 92.85 ms/sample for the RPi5 CPU baseline (15.4× speedup) and 13.70 ms/sample for a Jetson Orin Nano GPU baseline, while MobileNetV3 Large and FastViT-SA12 suffer large accuracy drops (-12.7 pp and -21 pp on CIFAR-100) that only advanced restoration (AdaRound, KD fine-tuning) partially mitigates.

## 核心贡献

- 提出首个将商业推理加速器(Hailo-8L)复用于端侧训练的异构 ONNX Runtime 流水线:在计算图层面切分,前向特征提取走 INT8 加速器,反向传播在加速器边界被截断,只在 CPU 上更新 FP32 分类头,从而避免对完整主干做端到端反向传播。
- 系统性地评估并比较了 5 种 PTQ 性能恢复策略(无恢复 / 通道均衡 Eq. / Eq.+迭代偏置校正 IBC / Eq.+知识蒸馏微调 FT / Eq.+AdaRound),揭示出量化脆弱性按架构分组:标准卷积 (ResNet18、EfficientNet-Lite) 用低成本方法即可恢复,而 Hard-Swish 与 self-attention 架构(MobileNetV3、FastViT-SA12)必须使用数据驱动方法。
- 在统一协议下完成系统级基准:与 RPi5 CPU-only (FP32) 与 NVIDIA Jetson Orin Nano edge GPU (FP32, 40 TOPS/15 W) 三方对比,覆盖 4 种主干(ResNet18、EfficientNet-Lite 4、MobileNetV3 Large、FastViT-SA12)、2 个数据集(CIFAR-100、Oxford-IIIT Pet)、3 个 batch size(1/4/16),并同时测量吞吐与单样本能耗。
- 配套开源实现基于 ONNX Runtime v1.22.1 + HailoRT v4.23.0,可直接复现,为后续 PEFT / 持续学习扩展提供平台。

## Key Contributions

- A heterogeneous, ONNX-Runtime-based on-device training pipeline that, for the first time in the open literature, repurposes a commercial INT8 inference accelerator (Hailo-8L) as a frozen-backbone feature extractor during head-only fine-tuning, with backpropagation mathematically truncated at the accelerator boundary so only the FP32 classification head is updated on the host CPU.
- A systematic study of five post-training quantization (PTQ) performance-restoration strategies — None, channel equalization (Eq.), Eq.+iterative bias correction (IBC), Eq.+knowledge-distillation fine-tuning (FT), Eq.+AdaRound (AR) — quantifying how each trades compute cost for accuracy recovery and exposing an architectural fault line (ReLU/SiLU backbones tolerate cheap restoration; Hard-Swish / self-attention backbones need data-driven recovery).
- A reproducible, three-way system-level benchmark against a Raspberry Pi 5 FP32 CPU baseline and a 40 TOPS NVIDIA Jetson Orin Nano FP32 edge-GPU baseline across four architectures, two datasets, three batch sizes, and energy measured at the wall (or via `tegrastats`) without subtracting idle power.
- Open-sourced toolchain (https://github.com/MatPiech/accelerator-training) integrating the modified ONNX Runtime training API with the Hailo execution provider.

## Methodology

The authors formulate on-device adaptation as a partitioned execution problem. Each network is split into a frozen pre-trained backbone `g(·; θ_b)` and a trainable classification head `h(·; θ_h)`, with `f(x; θ_b, θ_h) = h(g(x; θ_b); θ_h)`. Only `θ_h` is updated; gradients are truncated at the tensor boundary between the accelerator-produced features and the host-side head.

**Heterogeneous pipeline (proposed):** the backbone is exported from PyTorch (timm) to ONNX, quantized to INT8 and "performance-restored" by the Hailo Dataflow Compiler, and embedded as a `HailoOp` delegate node inside an ONNX graph. The ONNX Runtime Training API then compiles four artifacts — training graph (forward + truncated backward), evaluation graph, trainable-parameter checkpoint, and optimizer state — that run on the Hailo-8L (~13 TOPS, 1.5 W, M.2 HAT attached to a Raspberry Pi 5 with 4 GB RAM and a quad-core Cortex-A76 @ 2.4 GHz). The head, cross-entropy loss, backward pass, and AdamW optimizer step (lr = 1e-3) all run in FP32 on the RPi5 CPU.

**Baselines:** (i) RPi5 CPU-only with FP32 backbone + FP32 head; (ii) Jetson Orin Nano (Ampere GPU, 1024 CUDA cores, 32 Tensor Cores, 15 W) with FP32 backbone + FP32 head. The frozen backbone characterization (MACs and params) is reported separately from the trainable head to make the partitioning explicit.

**Quantization restoration strategies (Table 1):** #0 None; #1 Eq. (low cost); #2 Eq.+IBC using 64 unlabeled images; #3 Eq.+FT (knowledge distillation, 4 epochs, 1024 images); #4 Eq.+AR (AdaRound, 320 epochs, 256 images, all layers). A calibration set of 1024 images is sampled from the ImageNet `ILSVRC2012_val` split.

**Evaluation protocol:** head-only fine-tuning for 5 epochs with a fixed random seed; test-set classification accuracy is reported as the headline quality metric. Batch sizes 1/4/16 are swept for throughput; final accuracy is reported at batch size 1. Energy is measured per training epoch for the full workload (feature extraction, loss, backprop, optimizer step) without subtracting idle power: `tegrastats` at 1 Hz on Jetson, an external inline power meter at 5 Hz on the RPi5.

## Results & Highlights

- **Headline speedup (CIFAR-100, batch 1, Table 3):** ResNet18 trains at 6.04 ms/sample on the proposed Hailo-8L + RPi5 pipeline vs 92.85 ms/sample on the RPi5 CPU baseline — a **15.4× wall-clock speedup** — while also beating the Jetson Orin Nano GPU baseline (13.70 ms/sample) at 3.3× less energy per sample (38.65 mJ vs 128.25 mJ). Accuracy is essentially unchanged: 64.62% vs 64.68% (CPU) and 64.67% (GPU), a degradation of only 0.06 pp.
- **Architecture-dependent speedups (Fig. 3):** wall-clock speedup over CPU varies sharply — 15.4× for ResNet18 (CIFAR-100) / 14.4× (Pet), 7.5×/8.4× for EfficientNet-Lite 4, 9.8×/13.1× for FastViT-SA12, but only **4.2×/4.3× for MobileNetV3 Large**, because MobileNetV3's large classifier head dominates the non-accelerated FP32 backward pass.
- **Energy-per-sample reductions (Table 3):** for ResNet18 on Pet the proposed pipeline consumes 46.35 mJ/sample vs 561.25 mJ (CPU, ~12× reduction) and 128.13 mJ (Orin Nano). EfficientNet-Lite 4 on Pet reaches 111.99 mJ vs 1064.18 mJ (CPU) and 184.00 mJ (GPU).
- **PTQ restoration is decisive for fragile architectures (Fig. 4, Table 3):** without strong restoration, MobileNetV3 Large accuracy on CIFAR-100 collapses to 55.53% (vs 68.26% FP32, a **−12.73 pp gap**) and FastViT-SA12 to 50.19% (vs 71.19%, a **−21.0 pp gap**) even with the best AdaRound restoration (#4). In contrast, ResNet18 and EfficientNet-Lite 4 fully recover FP32 accuracy with cheap Eq./IBC, and EfficientNet-Lite 4 even *exceeds* the FP32 baseline (60.23% vs 58.48% on CIFAR-100) thanks to INT8 regularization.
- **Throughput vs batch size (Fig. 5, Pet):** the RPi5 CPU baseline barely scales with batch size; the proposed pipeline saturates above batch 4 and matches or exceeds the Orin Nano GPU at batch sizes 1 and 4 for ResNet18, EfficientNet-Lite 4, and FastViT-SA12 — a notable result given Hailo-8L delivers 13 TOPS at 1.5 W versus Orin Nano's 40 TOPS at 15 W.

## Critical Assessment

- **Novelty:** genuinely underexplored niche — prior on-device-training-accelerator work (TinyML [Lin et al.], smartphone GPU training [Das et al.], and most directly Topko et al. ISVLSI 2025 [ref. 24]) either targets microcontrollers, struggles with full training on GPUs due to data movement, or in the Topko case is limited to a single model (VGG-16) on CIFAR-10. The graph-partition + truncated-backpropagation idea applied to a commercial inference accelerator is a real, if engineering-flavored, contribution.
- **Rigor:** strong on system-level methodology — three-way hardware comparison, two datasets, four architectures spanning CNN/Transformer/Mobile designs, three batch sizes, per-epoch energy measured externally without subtracting idle. Random seed is fixed and all configurations share an identical head-only protocol. Software versions (ONNX Runtime 1.22.1, HailoRT 4.23.0, Hailo Dataflow Compiler 3.31.0) are pinned.
- **Reproducibility:** good but not perfect. Code is public and hardware (Hailo-8L M.2 + RPi5 + M.2 HAT) is commercially available; however, reproducing results also requires the proprietary Hailo Dataflow Compiler, the modified ONNX Runtime fork, and the exact ImageNet calibration split — none of which are openly bundled. The Hailo compilation itself was done on a separate RTX 4090 / Threadripper workstation, so "on-device" is only the training step, not model compilation.
- **Significance:** timely and practical for the lifelong-learning / personalized-edge-AI community, but the contribution is incremental rather than foundational — it is essentially an engineering validation that "inference accelerators can also accelerate training if you freeze the backbone," not a new learning algorithm.
- **Red flags and weak spots:**
  - **Severe accuracy degradation is underplayed.** FastViT-SA12 drops to 50.19% on CIFAR-100 (−21 pp vs FP32) and MobileNetV3 Large drops to 55.53% (−12.7 pp), even with the *strongest* restoration strategy (#4 AdaRound). The abstract's claim of "competitive" accuracy is only true for the two quantization-resilient backbones; for the other two the pipeline is currently unusable for that workload.
  - **Head-only is a narrow setting.** The protocol restricts updates to the classification head; the more interesting PEFT / adapter / LoRA / partial-unfreezing cases are deferred to future work. As acknowledged by the authors, models with heavy heads (MobileNetV3) or small backbones see little benefit.
  - **No peak-memory measurement.** Memory is a first-class constraint on edge devices, and its omission leaves a real gap relative to TinyML-style baselines that report SRAM/Flash footprints.
  - **5-epoch head-only fine-tuning** is a shallow adaptation budget; results may not extrapolate to longer schedules where quantization noise compounds or where restoration overfits the calibration set.
  - **Limited datasets.** Two image-classification sets, both with semantics reasonably close to ImageNet pre-training; cross-domain (medical, satellite, industrial defect) adaptation — which the paper itself motivates — is not tested.
  - **Calibration data leakage risk:** ImageNet `ILSVRC2012_val` is used as the calibration set while CIFAR-100 and Pet are downstream tasks, but the relationship between calibration-domain similarity and restoration effectiveness is not analyzed.

## Strengths

- First systematic, open-source demonstration that a low-power (1.5 W) commercial inference accelerator can be productively repurposed to accelerate on-device *training* of a frozen-backbone + trainable-head setup, with concrete wall-clock and energy wins.
- Clean three-way benchmark (CPU vs edge GPU vs proposed) with honestly reported per-architecture variance and an explicit diagnosis of why MobileNetV3 benefits least (heavy CPU-bound head).
- Practitioner-friendly guidance: a cost-tiered recipe (start with Eq./IBC; escalate to KD-FT or AdaRound only if accuracy is still insufficient) that maps directly to deployment trade-offs.
- Honest limitations section that names the head-only restriction, the missing memory analysis, and the cross-domain-transfer gaps.

## Limitations

- Head-only fine-tuning only; no PEFT, adapter, or partial-backbone updates — the more impactful regime for continual learning is left to future work.
- Two of four architectures (MobileNetV3 Large, FastViT-SA12) suffer double-digit accuracy gaps that the proposed pipeline cannot currently close, limiting applicability to ReLU/SiLU CNNs without further quantization research.
- Peak memory and SRAM/Flash footprints not reported — a significant omission for a paper framed around resource-constrained devices.
- Only two image-classification datasets and a 5-epoch budget; no cross-domain, regression, streaming, or continual-learning-with-forgetting evaluation.
- Reproduction requires proprietary Hailo Dataflow Compiler and a high-end workstation for compilation, partially undercutting the "on-device" framing.
- No statistical significance / variance reporting across seeds — only a single fixed seed is used.

## Relevance

对关注端侧模型自适应、轻量化微调与持续学习的读者高度相关:它给出了一种把推理加速器复用于训练的工程范式,可作为 PEFT/LoRA/adapter 在嵌入式设备落地的硬件基线,也直接对接 test-time adaptation 与 catastrophic forgetting 的后续研究方向。

## Questions for Further Exploration

- Can the heterogeneous pipeline be extended to LoRA / adapter / last-block-unfreezing without breaking the INT8 accelerator's throughput advantage, given that even MobileNetV3's heavy head already becomes the bottleneck?
- How does restoration effectiveness degrade when the calibration domain (ImageNet val) is far from the downstream task (e.g., medical imaging, industrial defect, satellite)?
- What is the peak SRAM / Flash footprint of the proposed pipeline, and how does it compare to the 256 KB SRAM / 1 MB Flash budget of the TinyML baseline?
- For quantization-fragile architectures (MobileNetV3, FastViT), would quantization-aware training of the backbone or mixed-precision (per-layer INT8/INT16) close the >10 pp gap that PTQ restoration leaves?
- Does the 15.4× speedup and energy reduction persist under continual-learning schedules (many sequential tasks with rehearsal), where the head is repeatedly retrained and the calibration distribution drifts?
- How sensitive are the results to the choice of edge GPU baseline power mode (Orin Nano has multiple modes); would the comparison change at 7 W instead of 15 W?
