> **一句话总结：** 该论文提出一种"harness 工程化"系统，将评估 harness 与基于 profiler 的优化控制器解耦，让 Codex/Claude Code 智能体在人工定义的约束/参考/晋级规则下生成 GPU kernel；在 MLSys 2026 FlashInfer 竞赛的 5 个算子上，相对官方 FlashInfer 基线取得 1.12×~29.68× 的平均延迟加速，且显著优于纯自动 LoongFlow Full-Agent。

> **TL;DR:** A Baidu team presents a harness-centered, human-guided workflow (Agent-Assisted) for LLM-driven CUDA/Triton/CuTe kernel optimization in the MLSys 2026 FlashInfer contest on NVIDIA B200 GPUs. By splitting an evaluation harness from a profile-backed controller and encoding expert skill playbooks (reference-first recon, paired gates, full-sweep promotion, artifact memory) used by Codex GPT-5.3 / Claude Opus 4.6 agents, the retained kernels beat the supplied FlashInfer baselines by 1.12×–29.68× mean-latency across the five operators (MoE FP8, DSA top-k, DSA attention, GDN decode, GDN prefill), and outperform matched LoongFlow PES Full-Agent artifacts that run 1.35×–13.25× slower with two operators falling below baseline (MoE FP8 0.27×, GDN Decode 0.83×).

## 核心贡献

- **首次将"harness engineering"（OpenAI/Anthropic 提出的概念）系统化应用到 GPU kernel 优化场景**：把人类智慧固化为可复用的 skill（CUDA 通用 skill + FlashInfer B200 专用 skill），让 LLM agent 在受限环境中做"有界的实现搜索"，而不是无约束的 prompt 试错。
- **解耦"评估 harness"与"优化控制器"**：harness 负责编译、正确性、官方对齐的 B200 计时和产物归档；控制器把 NCU/Torch Profiler 证据转换为可复用的瓶颈状态、选择唯一的优化方向、记录接受/拒绝轨迹。
- **跨 5 个算子、3 个赛道的实测验证**：在 NVIDIA B200（CUDA 13.2 / PyTorch 2.12 / Triton 3.6）官方对齐环境下，FP8 Fused MoE、DSA top-k、DSA attention、GDN decode、GDN prefill 五个算子分别达到 1.62×、18.05×、29.68×、1.12×、13.70× 的平均延迟加速。
- **Agent-Assisted vs Full-Agent 的对照实验**：使用 LoongFlow PES 作为同协议自主搜索基线，证明人工策展的参考、profiler 解读、保守晋级门控对可靠 LLM kernel 生成仍然关键——Full-Agent 在相同归一化下慢 1.35×–13.25×。
- **公开可复现的产物**：开源 harness 与各算子解（`mlsys26-flashinfer-contest` 等 4 个仓库），并附上 Torch Profiler/NCU 大形状证据、ablation、DSA top-k 高 trial 重放证据，便于他人复核。
- **诚实地暴露并建模"trial-budget 正确性盲区"**：用检测概率 `P_detect = 1-(1-p_w)^n` 形式化解释为什么低 trial 验证（默认 n=3）会漏掉稀有 top-k 边界错误，并提出"高 trial 重放作为最终门控"的策略。

## Key Contributions

- First systematic application of "harness engineering" (in the OpenAI Codex / Anthropic Claude Code sense) to GPU kernel optimization, encoding expert practice as reusable skills rather than one-off prompts.
- Architectural separation between an evaluation harness (compile / correctness / official-aligned B200 timing / archival) and a profile-backed optimization controller (NCU/Torch Profiler → bottleneck state → one bounded hypothesis → trajectory memory).
- Empirical validation across all three FlashInfer contest tracks and five operators (FP8 Fused MoE, DSA top-k indexer, DSA sparse attention, GDN decode, GDN prefill), with mean-latency speedups of 1.62×, 18.05×, 29.68×, 1.12×, and 13.70× over supplied FlashInfer baselines on NVIDIA B200.
- A matched-protocol Agent-Assisted vs Full-Agent comparison using LoongFlow PES as the autonomous baseline, showing Full-Agent artifacts are 1.35×–13.25× slower and two operators (MoE FP8, GDN Decode) fall below the FlashInfer baseline.
- Open reproducibility artifacts: four public GitHub repositories with the harness and per-operator solutions, plus retained ablation evidence, NCU/Torch Profiler summaries, and DSA top-k high-trial replays.
- Formal modeling of the trial-budget correctness blind spot (P_detect = 1 − (1 − p_w)^n) for FlashInfer-Bench's default n=3 evaluation, and a harness-policy fix (targeted high-trial replay as a final validation gate).

## Methodology

The system instantiates an AlphaEvolve-style closed loop in which humans define the objective, constraints, evaluation surface, and promotion policy, while coding agents (Codex / Claude Code) perform bounded implementation search.

**Harness/Controller split.** The evaluation harness is the measurement layer: package candidate, compile in target container, run correctness checks, measure official-aligned B200 latency via cupti-python, archive benchmark JSON + profiler artifacts. The optimization controller is the decision layer: extract a state signature from profiler evidence, match it to prior rounds, select exactly one optimization direction, supervise plateau/regression, and update trajectory memory. A "unified CUDA optimization skill" codifies the controller contract; a "FlashInfer B200 contest skill" specializes it for reference-first reconnaissance, real workload-shape discovery, same-round paired gates, B200/CUDA 13.2 tactics, and official-parity evaluation.

**Agent-Assisted loop (Algorithm 1).** For each round: (1) controller builds a state from retained runs and NCU data; (2) agent generates a bounded candidate family around one hypothesis using Triton / CUDA C++ / CuTe-CUTLASS / Python dispatch as permitted surfaces; (3) harness preflight-compiles and runs a paired representative gate against the same-round baseline on 2–3 shape regimes; (4) promising candidates are profiled via Torch Profiler and NCU; (5) a full sweep over the entire workload distribution runs with retry of transient infra failures; (6) promotion happens only if the full distribution improves with no correctness regression; (7) supervisor detects stalls/cycles/plateaus and triggers human steering (model family switch, sub-agent review, reference injection, language switch).

**Hardware- and profiler-driven choices.** Workload axes (sequence length, batch size, page count, outlier regimes) are derived from contest workload UUIDs and JSON files rather than a single hand-picked shape; shape-specialized routes are introduced only when NCU evidence shows distinct limiters (e.g., GEMM+packing cost in MoE, low-wave sparse launches in DSA, register/spill pressure in the GDN prefill chunk kernel). Conservative promotion rejects single-shape local wins and archives negative evidence (failed launch-bound, fusion, tile-size probes) as trajectory memory.

**Full-Agent baseline.** LoongFlow PES is run as an autonomous-search control under the same workload sets, evaluation protocol, and supplied FlashInfer baselines, then re-evaluated under the matched final-evaluation protocol for the comparison in Table 3.

**Environment.** CUDA 13.2, PyTorch 2.12, Triton 3.6, FlashInfer-Bench evaluator, isolated runners, B200 GPUs provided by Modal. Models: GPT-5.3-Codex as primary, Claude Opus 4.6 for independent reasoning and code review; sub-agents for parallel exploration and debate.

## Results & Highlights

All numbers are mean-latency over the contest workload distribution on official-aligned B200, with FlashInfer-relative speedup defined as `mean FlashInfer baseline latency / mean retained solution latency`.

- **DSA Attention — 29.68× speedup vs FlashInfer.** Agent-Assisted retained latency 0.011175 ms vs FlashInfer baseline 0.331650 ms (also 217.17× vs PyTorch reference). The matched Full-Agent artifact reached only 14.54× (0.022811 ms), i.e., roughly half the Agent-Assisted speedup.
- **DSA top-k Indexer — 18.05× speedup vs FlashInfer.** Agent-Assisted 0.006893 ms vs baseline 0.124420 ms (494.13× vs PyTorch). Full-Agent reached only 3.81× (0.032659 ms).
- **GDN Prefill — 13.70× speedup vs FlashInfer.** Agent-Assisted 0.051992 ms vs baseline 0.712166 ms (21,078× vs PyTorch). Full-Agent was essentially at baseline at 1.03× (0.688875 ms).
- **MoE FP8 — 1.62× speedup vs FlashInfer.** Agent-Assisted 0.286342 ms vs baseline 0.463874 ms (63.78× vs PyTorch). Full-Agent fell below baseline at 0.27× (1.74263 ms).
- **GDN Decode — 1.12× speedup vs FlashInfer.** Agent-Assisted 0.006201 ms vs baseline 0.006940 ms (7,970× vs PyTorch). Full-Agent fell below baseline at 0.83× (0.008366 ms).
- **Full-Agent gap is uniform.** Under the matched final-evaluation setting, Full-Agent artifacts are 1.35×–13.25× slower than Agent-Assisted across all five definitions, and two (MoE FP8 0.27×, GDN Decode 0.83×) end up slower than the supplied FlashInfer baseline — direct evidence that human-curated references, profiler interpretation, and conservative promotion gates remain critical.
- **Largest gains came from structural/hardware-aware changes**, not micro-tuning: sparse-attention route specialization (DSA attention 29.68×), top-k score/selection split with CuTe tensor-core scorer (18.05×), and the Blackwell chunked CuTe-DSL path replacing a 40-pair dispatch table for GDN prefill (13.70×, improving 90/100 workloads, mean latency 0.185183 ms → 0.051992 ms).
- **Trial-budget correctness blind spot formally modeled.** For DSA top-k workload `70d53807` (B=12, Pmax=82), a default n=3 trial run reported 0/3 mismatches (PASSED) but 100-trial replay exposed 3/200 INCORRECT with max error 6.033 abs / 0.1587 rel; an older git tag v48 showed 6/100. The conservative fallback tag v37 stayed exact across 1206 trials. Using `P_detect = 1 − (1 − p_w)^n` with the observed p̂ = 3/200, n=3 gives only 4.4% detection while n=100 gives 77.9%.
- **Per-shape ablations drove promotion.** MoE FP8 mean latency improved 0.299662 ms → 0.286342 ms (19/19 passing) after keeping L=1 epilogue + L=901 route specialization. CuTe scorer tile N=16 reduced DSA top-k scorer latency 6.783 µs → 4.896 µs; tile N=32 was rejected because per-CTA work grew faster than CTA-count reduction. GDN prefill Blackwell chunk path reduced mean 0.185183 ms → 0.051992 ms.

## Critical Assessment

**Novelty.** The conceptual move — applying OpenAI's "harness engineering" framing and Anthropic's "skills" pattern to a CUDA kernel optimization loop — is incremental rather than fundamentally new; the underlying closed-loop program search (AlphaEvolve, OpenEvolve, LoongFlow, KernelBench, KernelEvolve) is well established and the paper is honest about that lineage. The genuine novelty is the engineering instantiation: the harness/controller separation, the workload-UUID-derived shape dispatch, and the formal modeling of the trial-budget correctness blind spot.

**Rigor.** Strong on several axes: official-aligned B200 environment, mandatory correctness + latency, paired same-round gates plus full-distribution sweeps, multi-shape ablation evidence, and an honest Full-Agent control using LoongFlow PES under the matched protocol. Weaknesses: (i) the Full-Agent baseline is the authors' own LoongFlow PES run, not an independent replication of a peer autonomous agent — it is the strongest available control but is still a single autonomous baseline; (ii) the headline speedups use a self-defined "ratio-of-means" normalization that differs from the official per-kernel/per-track contest score `S_d`, which the authors flag but a casual reader could miss; (iii) **the post-submission DSA top-k correctness failure is a real red flag** — the retained low-trial artifact has a documented rare-boundary failure mode (3/200 mismatch on workload `70d53807`) that the default n=3 harness could not catch, and only the conservative fallback tag v37 stayed exact. The paper handles this transparently (Appendix E) but it does mean one of the five headline results rests on a tag that has a known correctness gap under high-trial replay.

**Reproducibility.** Above average for the subfield: four public repos with the harness and per-operator solutions, retained benchmark JSON, profiler summaries, and explicit hardware/software stack (CUDA 13.2 / PyTorch 2.12 / Triton 3.6 / cupti-python on B200). However, B200 access is a hard external barrier, the exact agent prompts/skill files are only excerpted (the full skill is "longer than a paper appendix"), and the human steering decisions (which model switch, which reference injection) are not fully logged as a machine-readable trace.

**Significance.** Practically significant for anyone building agent-driven systems-compilation tooling: it provides a concrete, falsifiable template for harness design and a clear negative result that pure autonomous LLM search is not yet competitive with human-guided search on real B200 kernels. Scientifically, the trial-budget detection-probability model is a small but transferable contribution to any kernel-search harness.

**Red flags.** (a) Two of five Full-Agent results fall below the supplied FlashInfer baseline (MoE FP8 0.27×, GDN Decode 0.83×), which slightly undercuts the "agents can do this" message even though the paper's actual claim is "Agent-Assisted beats Full-Agent." (b) The retained DSA top-k tag has a known correctness gap under high-trial replay that is only resolved by retreating to a conservative fallback tag (v37), not exposed as a primary result. (c) The Agent-Assisted workflow is heavy on human effort (harness design, reference curation, plateau steering, promotion approval); the paper is honest about this but it limits scalability claims.

## Strengths

- Clean architectural framing: evaluation harness vs profile-backed controller is a genuinely useful separation that other kernel-agent papers conflate.
- Strong, honest empirical evidence: official-aligned B200, full-distribution sweeps, paired gates, and a matched Full-Agent control rather than a strawman.
- Open artifacts: harness + per-operator solutions + benchmark JSON + profiler summaries are public and citable.
- Workload-grounded shape dispatch derived from real contest UUIDs/JSON, not hand-picked inputs.
- Transparent handling of the DSA top-k correctness failure, including a formal detection-probability model and a conservative fallback policy — exactly the kind of negative evidence most papers omit.
- Per-operator case studies are concrete and profiler-driven (e.g., MoE GEMM1+GEMM2 = 81.6% of CUDA time; GDN prefill chunk kernel = 98.5% of CUDA time with 976 spills and ~50% excessive global sectors).

## Limitations

- Evaluates an Agent-Assisted engineering workflow, not an isolated LLM; human harness design, budget decisions, reference selection, and final promotion are central. The paper itself lists this as the main limitation.
- Workload distributions are contest-specific and the timings are local official-aligned B200 evidence rather than final leaderboard scoring; generalization to other hardware (H100, MI300) or other operator families is untested.
- The headline speedups use a self-defined ratio-of-means normalization, not the official contest score, and the DSA top-k headline rests on a low-trial retained tag with a known high-trial correctness gap (the conservative fallback tag is reported only in Appendix E).
- Only one autonomous baseline (LoongFlow PES) is compared; no comparison against KernelBench/KernelEvolve-style autonomous agents on the same workload set.
- The full skill and exact prompt template are only excerpted; the human steering log is not released as a machine-readable trace, limiting exact replication of the Agent-Assisted results.
- B200 access via Modal is a hard external prerequisite for reproducibility.

## Relevance

对关注"LLM coding agent / 自动化 GPU kernel 生成 / 推理系统优化"方向的研究者高度相关：它提供了一个具体可复用的 harness 设计模板（评估/控制分离、profiler 驱动状态、保守晋级门控），并用对照实验证明在真实 B200 kernel 优化任务上人工引导仍显著优于纯自动 agent——这对评估当前 agent 自主性与构建可靠 AI-for-systems 工具链都有直接参考价值。

## Questions for Further Exploration

- How much of the Agent-Assisted advantage comes from human-curated references vs. profiler interpretation vs. conservative promotion gates? An ablation removing each component would quantify the marginal contribution.
- Can the trial-budget detection-probability model (P_detect = 1 − (1 − p_w)^n) be turned into an adaptive scheduling policy that allocates high-trial replay only to workloads whose top-k boundary margin γ(s) is small, rather than blanket high-trial validation?
- Would a stronger autonomous baseline (e.g., KernelEvolve or a long-horizon research agent) close the gap to Agent-Assisted on these five operators, or do the structural/hardware-aware wins (sparse-attention route, Blackwell chunked CuTe) require human pattern recognition that current agents lack?
- Does the harness/controller separation transfer to non-Blackwell targets (H100, MI300X, Intel GPU) where tile sizes, register pressure, and memory hierarchy differ materially?
- The retained DSA top-k tag has a documented high-trial correctness gap; would replacing it with the conservative v37 fallback change the headline 18.05× speedup, and should the paper's main table report the conservative tag as the primary result?
- How sensitive are the results to the specific model versions (GPT-5.3-Codex, Claude Opus 4.6)? A model-robustness study would clarify whether the harness or the model is doing the heavy lifting.
