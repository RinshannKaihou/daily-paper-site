Harness Engineering for LLM-Driven GPU Kernel Generation

Yue Shui, Chenyu Ma, Hangfei Xu, Shengzhao Wen, Yanpeng Wang — Baidu, Inc.
LLM coding agents GPU kernel optimization harness engineering FlashInfer NVIDIA B200 agent-assisted coding
Key Finding

Human-guided harness engineering beats autonomous agents on B200 GPU kernels — 1.12× to 29.68× over FlashInfer baselines.

01 Key Metric
29.68×
DSA attention mean-latency speedup vs the supplied FlashInfer baseline on official-aligned NVIDIA B200.
02 Why This Matters
A reusable harness design template — eval/control split, profiler-driven state, conservative promotion gates — proves human-guided search still beats autonomous agents on real B200 kernels.
03 Critical Edge
DSA top-k's headline 18.05× rests on a low-trial tag with a known 3/200 high-trial correctness gap the default n=3 harness cannot catch.
§ 01

Methodology

An AlphaEvolve-style closed loop where humans define the objective, constraints, evaluation surface, and promotion policy, while coding agents (Codex / Claude Code) perform bounded implementation search — not unconstrained prompt trial-and-error.

AEvaluation Harness

  • Packages candidate; compiles in target container.
  • Runs correctness checks against reference.
  • Measures official-aligned B200 latency via cupti-python.
  • Archives benchmark JSON + profiler artifacts.

BOptimization Controller

  • Extracts state signature from NCU / Torch Profiler.
  • Matches to prior rounds; selects exactly one optimization direction.
  • Supervises plateau / regression detection.
  • Updates trajectory memory (accept / reject archive).
Agent-Assisted Loop (Algorithm 1)
  1. Controller builds a state from retained runs + NCU data.
  2. Agent generates a bounded candidate family around one hypothesis (Triton / CUDA C++ / CuTe-CUTLASS / Python dispatch).
  3. Harness preflight-compiles and runs a paired representative gate against the same-round baseline across 2–3 shape regimes.
  4. Promising candidates profiled via Torch Profiler + NCU.
  5. Full sweep over the entire workload distribution, with retry of transient infra failures.
  6. Promotion only if the full distribution improves with zero correctness regression.
  7. Supervisor detects stalls / cycles / plateaus → triggers human steering (model switch, reference injection, language switch, sub-agent review).
Stack CUDA 13.2 · PyTorch 2.12 · Triton 3.6 · FlashInfer-Bench · cupti-python · B200 via Modal
Models GPT-5.3-Codex (primary) · Claude Opus 4.6 (independent review) · sub-agents for parallel exploration
§ 02

Results

Mean-latency over the contest workload distribution on official-aligned B200. Speedup = mean FlashInfer baseline / mean retained solution. All five operators retain kernels that beat the supplied baseline; the matched autonomous control does not.

Structural wins, not micro-tuning. Sparse-attention route specialization (29.68×), top-k score/selection split with a CuTe tensor-core scorer (18.05×), and a Blackwell chunked CuTe-DSL path replacing a 40-pair dispatch table for GDN prefill (13.70×, improving 90/100 workloads; 0.185 ms → 0.052 ms).
Trial-budget correctness blind spot. For DSA top-k workload 70d53807 (B=12, Pmax=82), the default n=3 trial run reported 0/3 mismatches (PASSED) — but a 100-trial replay exposed 3/200 INCORRECT (max err 6.033 abs / 0.1587 rel). The conservative fallback tag v37 stayed exact across 1206 trials.

+ Strengths

  • Clean architectural framing: evaluation harness vs. profile-backed controller — a genuinely useful separation that other kernel-agent papers conflate.
  • Strong, honest empirical evidence: official-aligned B200, full-distribution sweeps, paired same-round gates, and a matched Full-Agent control rather than a strawman.
  • Open artifacts: harness + per-operator solutions + benchmark JSON + profiler summaries are public and citable across four repos.
  • Workload-grounded shape dispatch derived from real contest UUIDs / JSON, not hand-picked inputs.
  • Transparent handling of the DSA top-k correctness failure, with a formal detection-probability model and a conservative fallback policy.
  • Profiler-driven case studies: MoE GEMM1+GEMM2 = 81.6% of CUDA time; GDN prefill chunk kernel = 98.5% of CUDA time with 976 spills.

− Limitations

  • Evaluates an Agent-Assisted engineering workflow, not an isolated LLM; human harness design, reference selection, and final promotion are central.
  • Contest-specific workloads; local official-aligned B200 evidence rather than final leaderboard scoring; generalization to H100 / MI300 untested.
  • Headline speedups use a self-defined ratio-of-means, not the official contest score; DSA top-k headline rests on a low-trial tag with a known high-trial correctness gap.
  • Only one autonomous baseline (LoongFlow PES); no comparison against KernelBench / KernelEvolve on the same workload set.
  • Full skill and exact prompt template only excerpted; human steering log is not released as a machine-readable trace.
  • B200 access via Modal is a hard external prerequisite for reproducibility.
§ 04

Questions for Further Exploration

  1. How much of the Agent-Assisted advantage comes from human-curated references vs profiler interpretation vs conservative promotion gates? A component ablation removing each would quantify the marginal contribution.
  2. Can the Pdetect model become an adaptive scheduling policy that allocates high-trial replay only to workloads whose top-k boundary margin γ(s) is small, rather than blanket high-trial validation?
  3. Would a stronger autonomous baseline (KernelEvolve, a long-horizon research agent) close the gap, or do the structural / hardware-aware wins require human pattern recognition that current agents lack?
  4. Does the harness / controller separation transfer to non-Blackwell targets (H100, MI300X, Intel GPU) where tile sizes, register pressure, and memory hierarchy differ materially?
  5. Would replacing the retained DSA top-k tag with the conservative v37 fallback change the headline 18.05×, and should the main table report the conservative tag as the primary result?
  6. How sensitive are the results to the specific model versions (GPT-5.3-Codex, Claude Opus 4.6)? A model-robustness study would clarify whether the harness or the model is doing the heavy lifting.
