> **一句话总结：** 本文形式化定义了"自状态攻击"（被诱导的 agent 通过合法 syscall 改写自己的 memory/instruction/configuration 文件，使攻击操作集是合法操作集的子集 A(R)⊆L(R)），并以 OpenClaw 为对象在 4 个工作负载 profile 上构建 23 单元 × 43 攻击实例矩阵，证明 OS 三层防御（prevention/detection/recovery）各自存在结构性极限——推荐的 L3 权限锁定 + B2 工作负载条件化异常检测 + 15 事件备份栈可关闭除 4 个 Memory 行小修改单元外的全部攻击单元，剩余单元在 OS 层结构上不可区分。

> **TL;DR:** This paper formally defines "self-state attacks" against self-hosted AI agents (e.g., Claude Code, OpenClaw) where a compromised agent corrupts its own memory/instruction/configuration files via legitimate syscalls with A(R)⊆L(R). It introduces a four-axis attack space (Target, Mechanism, Granularity, Temporal), realizes it as a 23-cell × 43-instance matrix on a real OpenClaw-derived agent under four workload profiles (Aider polyglot, FRAMES, Operations, Generalist), and shows that a layered stack of L3 Unix permissions + workload-conditioned B2 anomaly detection at τ=2.0 + 15-event backups closes all but 4 of 23 attack cells, leaving small/daily-log Memory-row writes (Mem-M1-G3/G4, Mem-M2-G3/G4) structurally indistinguishable from legitimate traffic at the OS layer.

## 核心贡献

- **首次形式化"自状态攻击"为新威胁类**：明确刻画了 `A(R) ⊆ L(R)`（攻击操作集是合法操作集的子集）这一与跨边界攻击本质不同的属性，区别于传统 confused deputy 问题——被破坏的对象是 agent 自己的持久状态。
- **提出四轴攻击空间（Target/Mechanism/Granularity/Temporal）**：将每个攻击事件投影到 VFS 边界上的特征元组 `<target-path, op_type, δs, mode-delta, Δt>`，并映射到 MITRE ATLAS (AML.T0080/T0080.000/T0081)、ATT&CK（T1565.001, T1070.004, T1222, T1562）、OWASP Agentic AI（T1/T2/T3/T6/T9）三大目录，从 OS 可观测性而非攻击者意图视角重新分解目录条目。
- **导出 OS 三层防御的结构性极限**：prevention 因 A(R)⊆L(R) 不能在不阻塞合法写入的情况下阻塞攻击；detection 仅记录"that"（发生了什么）而非"whether"（是否恶意），事件元组无法承载语义；recovery 链接到 detection 且承受与备份间隔成比例的数据损失。
- **提出工作负载条件化检测作为设计原则**：通过 V/C/I（Visible/Conditioned/Indistinguishable）三态分类，将"最坏情况不可分"细化为"特定 profile 上的可分边界"，把检测能力从攻击者意图空间拉回到防御者观测空间。
- **在真实 OpenClaw 衍生 agent 上完成实证评估**：用 inotify 采集 4 个工作负载 profile（W1 Aider polyglot / W2 FRAMES / W3 Operations / W4 Generalist）的 5-chain 轨迹池（29/42/1677/98 个自状态事件），对 6 级权限级联 L0–L5、3 种检测器（B0 Wazuh-FIM、B1 pooled-baseline、B2 per-profile）和 4 种备份间隔进行系统实验，提供可复现的 openclaw-core 工具链（Python 3.10+，无第三方依赖，~30 秒一键复现）。

## Key Contributions

- **First formalization of self-state attacks as a distinct class** with the defining property A(R) ⊆ L(R) on writable self-state resources R, distinguishing it from cross-boundary attacks and showing it is a strictly harder instance of the confused-deputy pattern.
- **Four-axis attack space (Target × Mechanism × Granularity × Temporal)** grounded in MITRE ATLAS/ATT&CK/OWASP Agentic AI, where each axis projects onto one component of the VFS feature tuple any kernel-level monitor (inotify/fanotify/auditd) can observe.
- **Structural limits of three OS defense dimensions** derived analytically: prevention meets A(R)⊆L(R) indistinguishability; detection sees "that" not "whether" (event-only, no content); recovery inherits detection as precondition and pays data-loss cost ∝ backup interval.
- **Workload-conditioned detection principle and V/C/I formalism**, turning "globally indistinguishable" into a per-cell, per-profile TPR classification with quantitative thresholds (TPR_min ≥ 0.80 for V, [0.50, 0.80) for C, < 0.50 for I).
- **Empirical evaluation on an OpenClaw-derived agent (openclaw-core)** driven by Gemini 3 Flash across four workload profiles, releasing the full harness, attack catalog (43 instantiations), trace pool, and one-command reproduction as an anonymous artifact.

## Methodology

The paper pairs a formal framework with an empirical evaluation on a real self-hosted agent.

**Self-state model.** Self-hosted agents (Claude Code, OpenClaw) maintain three functional layers as ordinary markdown/JSON files on the local filesystem: *instruction* (SOUL.md, IDENTITY.md, USER.md, AGENTS.md, TOOLS.md), *memory* (MEMORY.md, memory/*.md, daily logs), and *configuration* (openclaw.json, HEARTBEAT.md, credentials/.env). The LLM core reads these directly as prompt context, so writes to them shape future behavior.

**Four-axis attack space.** Every attack reduces at the VFS boundary to a feature tuple `<target-path, op_type, δs, mode-delta, Δt>`: D1 Target (Instruction/Memory/Config); D2 Mechanism (M1 Modify in place, M2 Add/append, M3 Delete/unlink, M4 Deny/chmod); D3 Granularity (G1 whole-file, G2 large-delta, G3 line-scale, G4 ≤4-byte minimal-delta); D4 Temporal (T1 single-shot, T2 burst-aligned, T3 quiet-gap, T4 slow-drip). The 23-cell matrix (Target × Mechanism × Granularity) expands to 43 concrete attack-script instantiations across target files.

**Three structural limits.** Prevention indistinguishability (Claim 1): a per-event, memoryless OS policy cannot block A(R) without blocking some L(R). Detection gap (Claim 2): the OS monitor records event metadata (op_type, δs, mode-delta) but not written content, so benign and malicious writes are indistinguishable in feature space. Recovery gap: temporal ambiguity + selective-recovery ambiguity + cross-layer entanglement (poisoned memory can later be distilled into instruction).

**Workload bridge.** Real workloads concentrate legitimate writes in narrow feature regions; an OS monitor can fit per-(file, op_type) baselines on `(log Δt, sign(δs), log|δs|)` and score events as a z-score (Eq. 1). This yields three regimes: Visible (V, TPR ≥ 0.80 on every profile), Conditioned (C, profile-dependent), Indistinguishable (I, inside at least one profile's legitimate distribution).

**Experimental setup.** Subject agent: openclaw-core, a minimal harness retaining OpenClaw's agent loop and self-state I/O paths (drops channel/plugin/gateway), driven by Gemini 3 Flash. Four workload profiles, 30 tasks each: W1 Coding (Aider polyglot, 29 self-state events/5 chains), W2 Research (FRAMES, 42), W3 Operations (author-constructed, 1677), W4 Generalist (author-constructed, 98). Each trial starts from a clean Ubuntu 22.04 snapshot with inotify recursive watchers. 70/30 session-level train/test split, 30 trials per cell.

**Three experiments.** Exp 1 (Prevention-Functionality Tradeoff): six Unix-permission levels L0–L5 as a strict cumulative cascade from unrestricted (L0) through persona/Instruction/Config/MEMORY.md/daily-log locks up to L5 (all self-state read-only), reporting protection, functionality, and per-profile workload success. Exp 2 (Detection): three detectors — B0 Wazuh-style hash FIM, B1 workload-blind pooled baseline, B2 workload-conditioned per-profile baseline — at fixed τ=2.0, with ablations for temporal evasion (random/burst/quiet, Exp 2-T) and FIM (Exp 2-FIM). Exp 3 (Recovery): nine-attack subset across V/C/I classes plus a backup-destruction sentinel, sweeping k ∈ {5, 15, 30, 60} event backup intervals over 720 scenarios.

## Results & Highlights

- **V/C/I matrix split (Exp 2, B2 at τ=2.0)**: out of 23 canonical cells, **11 are Visible** (TPR ≥ 0.80 on every profile), **8 are Conditioned** (profile-dependent, per-profile minimum TPR in [0.50, 0.80)), and **4 are Indistinguishable** (TPR < 0.50 on at least one profile): Mem-M1-G3, Mem-M1-G4, Mem-M2-G3, Mem-M2-G4 — all small or daily-log Memory-row writes inside the operations-style W3/W4 baseline.
- **Prevention-functionality tradeoff (Exp 1, Table 5)**: protection rises monotonically from 0.00 (L0) to 1.00 (L5), functionality falls 1.00→0.00. **L3 (lock Instruction + Configuration)** is the recommended balanced point with **protection 0.61 and functionality 0.70**; L4 (additionally locking MEMORY.md) raises protection to 0.70 but collapses W3 functionality to **9%** and W4 to 31%; L5 hits perfect protection at zero functionality.
- **Workload-conditioning matters (B1 vs B2)**: B1 (pooled baseline) yields **13 V / 4 C / 6 I**; B2 yields **11 V / 8 C / 4 I** — same net count but bidirectional migration (Cfg-M1-G3/G4 upgrade I→C, while Mem-M1-G2/Mem-M2-G2 downgrade V→C because W3's per-profile Memory σ_s is wider than the pooled one). ΔTPR(B2−B1) is +0.37 on C cells and +0.46 on I cells.
- **Per-event FPR (B2, τ=2.0, Table 7)**: 0.08±0.04 on W3 (densest profile), 0.18±0.19 on W4, 0.36±0.32 on W1, 0.33±0.16 on W2. Sparse-profile FPR is workload-data-bound rather than threshold-bound (per-(file, op_type) sample sizes too small to fit tight σ_s).
- **FIM baseline failure mode (Exp 2-FIM, Table 8)**: B0 (Wazuh-style hash FIM) achieves uniform TPR=1.00 but at per-event FPR **0.76 (W1), 0.76 (W2), 0.99 (W3), 0.87 (W4)**, mean 0.84 — a structurally different failure that floods the legitimate workload with alerts rather than missing the I-cells.
- **Temporal evasion is narrow (Exp 2-T)**: only **2 of 23 cells change class** across random/burst/quiet injection modes. Mem-M1-G2 (W3) drops from TPR 0.77 (random) to 0.47 (burst) → migrates C→I; under quiet-gap it actually rises to 0.80 (C→V). Slow-drip (T4) is excluded by construction since each sub-threshold write is in-distribution.
- **Recovery (Exp 3)**: across the 8-attack V/C/I subset (640 scenarios), recovery achieves **100% remediation** uniformly across V/C/I classes — class-independent given detection. Backup-destruction sentinel (MEMORY.md unlink + backups/ wipe) reaches **0% remediation** by construction, exposing the only structural failure mode.
- **Data-loss scales with backup interval (Exp 3, W3)**: mean lost-event count grows from 4 at k=5 to 35 at k=60; mean lost-event fraction rises 14% (k=5) → ~15% (k=15/30) → 11% (k=60) as the loss window aligns with the inter-cycle gap. Loss is determined by legitimate workload accumulated between snapshots, not by attack class.
- **Recommended stack (Table 9)**: L3 prevention + B2 detection at τ=2.0 + 15-event backup interval closes **all Instruction and Configuration cells uniformly via prevention**, and 2 of 8 Memory V cells via detection. Per-profile closure at strict TPR ≥ 0.80: W1 20/23, W2 21/23, W3 17/23, W4 18/23.

## Critical Assessment

- **Novelty**: High and well-motivated. The A(R)⊆L(R) formalization crisply distinguishes self-state attacks from cross-boundary attacks and from prior memory-poisoning work (AgentPoison, PoisonedRAG, MINJA) that treat memory corruption as a methodology rather than as one row of a broader structural class. The four-axis taxonomy with VFS projection is a genuine reconceptualization of MITRE ATLAS entries along defender-observability axes.
- **Rigor**: Strong on the analytical side (Claims 1–2 are direct from the feature-tuple argument; V/C/I criteria are quantitative with explicit thresholds). The experimental design is unusually transparent about its scope: the paper openly states that the 4 I-cells stay I because W3's training pool carries 162 Memory-row write events populating every relevant bucket with non-degenerate σ_s, and that V cells are detectability claims relative to the profiles measured (not absolute).
- **Reproducibility**: Excellent by current standards. Linux-only inotify harness is a limitation but disclosed (camera-ready 30-session runs on Linux; macOS pilots skip trace collection cleanly). RNG seeded, all seed material checked into the repo, W1/W2 pinned to specific commits/SHAs. The harness is purpose-built rather than a wrapper around the real OpenClaw — both a strength (cleaner VFS events) and a limitation (ecological validity).
- **Significance**: Practical and timely given the deployment trajectory of Claude Code-class agents. The V/C/I formalism and the per-cell defense cascade table (Table 9) are directly actionable for builders of self-hosted agents.
- **Red flags / weaknesses**:
  - **Baselines are limited in scope**: only B0 (hash FIM) and B1 (pooled statistical) are evaluated as detection baselines; semantic-layer defenses (GuardAgent, AgentSpec, VeriGuard) and OS observability systems (AgentSight eBPF, IsolateGPT, CaMeL) appear only in a positional Table 10 with no empirical comparison. The paper explicitly defers semantic-layer detectors to future work, but readers cannot judge the marginal value of B2 over an eBPF-correlation baseline like AgentSight.
  - **Single LLM, single harness**: only Gemini 3 Flash is tested, and the subject is a stripped-down openclaw-core rather than full OpenClaw or Claude Code. Generalization to agents with different memory formats (e.g., vector DBs) is asserted but not demonstrated.
  - **Author-constructed workloads**: W3 (Operations) and W4 (Generalist) task corpora are hand-authored — they produce the densest trace (1677 events for W3) that drives most I-cell conclusions; a different operations workload could shift the boundary.
  - **Fixed τ=2.0**: although justified as profile-independent by design, the operating-point stability claim only covers the V/I gap; the 8 C-cells materially depend on the threshold, so practitioners need per-profile retuning.
  - **High FPR on sparse profiles**: 0.36 on W1 and 0.33 on W2 at τ=2.0 is uncomfortable for autonomous blocking; the paper itself notes sparse-profile FPR is data-bound, not threshold-bound, so more training data is required in practice.
  - **Slow-drip (T4) excluded by construction**: the paper acknowledges each sub-threshold T4 write is in-distribution by design, leaving the cumulative-effect case as an open structural hole at the OS per-event layer.
  - **Recovery semantic ambiguity is described but not solved**: selective rollback across memory/instruction/config layers is left as a hard open problem; only coarse backup-and-restore is evaluated.

## Strengths

- Crisp formalization that makes the A(R)⊆L(R) property mathematically explicit and uses it to derive the three structural limits directly.
- The V/C/I three-regime classification is a genuinely useful conceptual tool that reframes "indistinguishable" as a per-profile, per-cell boundary rather than a global defeat.
- Honest reporting: the paper explicitly states that net cell count alone understates B1-vs-B2 differences (bidirectional migration), that V cells may migrate to C under richer profiles, and that sparse-profile FPR is data-bound.
- Clean artifact design: deterministic (op_type, δs, mode-delta) attack specs decoupled from the legitimate-session driver, with one-command reproduction and zero third-party dependencies.
- The per-cell defense cascade (Table 9) directly tells practitioners which cells are closed by which layer under which profile.

## Limitations

- Only one LLM (Gemini 3 Flash) and one harness (openclaw-core) tested; no production OpenClaw or Claude Code.
- W3/W4 workloads are author-constructed and dominate the trace volume, so I-cell conclusions are sensitive to the operations-workload design.
- No empirical comparison against semantic-layer or eBPF-correlation defenses (AgentSight, GuardAgent, AgentSpec); only positional comparison in Table 10.
- Slow-drip (T4) cumulative-effect attacks are excluded by construction; the paper acknowledges they are structurally out of reach for per-event OS detection.
- Recovery across cross-layer entanglement (memory → instruction → config) is left open; only coarse snapshot rollback is evaluated.
- High FPR (0.33–0.36) on sparse profiles requires either more training data or human-in-the-loop gating for production use.
- Linux-only inotify instrumentation; portability to macOS/Windows kernel monitors is asserted but not tested.

## Relevance

与自托管 AI agent（如 Claude Code、OpenClaw 这类直接读改本地文件 system）的安全防御高度相关，提出的 V/C/I 工作负载条件化检测框架和 L3+B2+备份的分层防御栈可直接指导 agent 文件状态保护、异常监控与回滚策略的设计。

## Questions for Further Exploration

- How would a sliding-window trajectory scorer (the future-work direction in Appendix G.2) recover some of the 4 I-cells without crossing into the semantic layer, and what FPR does it pay?
- Does the V/C/I boundary shift materially when the agent uses a vector database or SQLite-backed memory instead of plain markdown files (different δs distributions)?
- How does B2 compare empirically against AgentSight-style eBPF + TLS-LLM-traffic correlation, which can observe both the syscall and the inference call that triggered it?
- Can a memory-diff audit at read-time (re-score content at the LLM-context-construction boundary) close the daily-log append I-cells (Mem-M2-G3/G4) at acceptable cost?
- What is the production FPR ceiling for sparse profiles as a function of training-trace volume, and is there a transfer-learning approach that lets a sparse profile borrow σ_s from a dense one?
- Does multi-agent/self-evolving memory (where multiple agents cross-write to a shared self-state) change the A(R)⊆L(R) property, or merely widen L(R)?
- How would the backup-destruction sentinel failure mode be closed — by storing snapshots under a separate OS principal as the paper suggests, or by append-only/WORM storage?
