i. Methodology — the Karpathy loop, pinned

A frozen dataset, a read-only scorecard, and a single editable solution file. The agent proposes one idea per experiment, commits before verifying, logs a uniform row, keeps the change only if the score improves, and stops at budget or a 15-experiment plateau. Identical PROGRAM.md, hardware, and agent versions across both arms.

  • Task Stage-1 Quran memorization checker: from a diacritics-free ASR transcript, return abstain or the recited verse-id sequence + per-verse split. Metric research_score = det + split + abstain, lower is better.
  • Dataset 258 production Telegram-bot recordings (254 Quranic, 4 non-Quranic; 45 surahs; transcripts 3–400+ words). Frozen by SHA-256 before any run.
  • Study 1 Ungated. 3 runs each of Claude Code & Codex, identical budget (30 experiments / 1 hr), high reasoning, full-auto. No held-out set.
  • Study 2 Preregistered. 60/40 split (151 train / 107 test; fixed seed), optimized on train only, gold ids removed from failure reports, both agents told held-out exists. H1–H3 fixed before running.
  • Forensics Literal count of hardcoded six-digit verse-id constants in the final artifact (minus the closed-set muqatta'at table). Recomputable from per-run git history.

iii. Strengths

  • Preregistered held-out study with H1–H3 fixed before running; the authors' own reversal becomes the evidence.
  • Paper-specific forensics: counting literal verse-id constants cleanly distinguishes "better algorithm" from "more memorized rows" — and is recomputable from artifacts.
  • Documents and patches three agent-initiated isolation failures: sibling-branch reads via shared git, unsolicited memory notes to "future runs", gold-id reads from failure reports.
  • Reports the failure of the obvious cross-agent ensemble instead of omitting it.
  • Real production deployment — feature flag, legacy fallback, CI regression gate. The loop's output is consequential.

iv. Limitations

  • n = 3 per arm. The held-out Claude-vs-Codex gap rides on a single 0.5 abstention miss across only two non-Quranic test rows.
  • Single task & domain. Generalizer-vs-maximizer is a behavioral claim about two agents on one Arabic-text task — not yet shown to transfer.
  • No matched-budget human-expert arm in the loop; incumbent had months of tuning vs. agents' one hour.
  • Community arms (Cursor, Antigravity) are explicitly exploratory — unpinned routing, different hardware, not matched to confounds.
  • Temporal confound: the two agents' runs occurred on different days.
  • Harness was authored with the assistance of one compared agent; mitigated by freezing, but a strict reviewer would still flag it.
  • At submission, repo & dataset URLs are still [ADD ...] placeholders — external reproducibility not yet verifiable.

v. Open Questions for Further Exploration

Q1Does the generalizer-vs-maximizer disposition replicate across other domains (code, math, biology) — or is it specific to closed-reference-corpus alignment where memorization is easy?
Q2Would adding the same one-paragraph held-out disclosure to MLE-bench, RE-Bench, SpecBench change agent rankings there, as it did here?
Q3Should rare-event metric components (worth e.g. 0.5 each) be reported with bootstrap CIs or down-weighted — not summed into a scalar?
Q4Can the "agents as annotation auditors" dividend (both arms flagged the same gold-label error) become a standard data-quality pipeline for low-resource corpora?
Q5At larger n with a matched human expert, is the ~10× win over the incumbent robust — or partly an artifact of the incumbent's stale tuning manifest?
Q6Could "two implicit specs from one explicit spec" be predicted ex ante from training-data / RLHF posture, not only detected post hoc?
