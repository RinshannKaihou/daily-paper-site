SST is not a new hash table. It is a structural pre-processing layer you stack on any open-addressing scheme to slash collisions by 85%.

At load factor 0.95 with K=8 shaping, maximum probe clusters collapse from 872 to 121 entries and lookup tail latency (p99) drops from 172 probes to as few as 3.75. The cost: 1–3 metadata bits per record and higher build-time CPU. Net gains kick in once reads exceed ~20 queries per record.

How SST Works

  • 1 Insert: Generate K reversible transformations of each key. Map each candidate through the baseline placement rule, pick the least congested slot. Store a 1–3 bit shaping tag.
  • 2 Lookup: Read the tag, apply the inverse transform, probe as usual. Deterministic — no change to the base probing rule.
  • 3 Effect: Fewer collisions at source → shorter probe sequences → fewer RAM accesses per query.
ⓘ SST is composable: “SST off → SST on” can be tested for any open-addressing scheme with the same baseline code.

Strengths

  • ✓Clean composable-layer framing applicable to any open-addressing scheme

  • ✓Tail latency reduction is genuinely valuable for real-time DBs

Limitations

  • ✗Double hashing: structural gain but zero total-time benefit
  • ✗Only tested up to M=500K — industrial scale unknown
  • ✗No comparison vs Swiss tables, F14, or learned indexes
  • ✗No public C++ benchmark source code
  • ✗No energy measurements despite claiming RAM-to-CPU shift
  • ✗Only open-addressing; no chaining-based evaluation

Open Questions

  • Q1 How does SST interact with cache-friendly designs (Swiss tables, F14) that already use metadata bits?
  • Q2 Does reducing RAM accesses actually save energy on modern hardware? No data yet.
  • Q3 Can SST extend to concurrent/lock-free tables where insertions face race conditions?
  • Q4 What is the breakeven point at millions–billions of entries considering cache hierarchy?
  • Q5 Can learned index components be combined with SST for smarter placement?
