Minimal Prompt Perturbations Lead to Code Vulnerabilities

IEM, HES-SO Sierre · Cyber-Defence Campus, armasuisse S+T · 3 models · 5 languages · 18 CWE types
arxiv.org/abs/2605.29737
Headline Claim
Single-character prompt mutations can flip coding LLM outputs from secure to vulnerable code.
Anchor Fact
Input-handling vulnerability AUC 0.753 vs secure-defaults AUC 0.674 (Mann-Whitney p=0.009).
So What
Ordinary developer typos — not just malicious prompts — can cause coding assistants to generate exploitable code.
Critical Edge
Mean probe AUC ~0.70 is modest. No frontier models tested. No mitigation strategies proposed.

01 — Contributions

What This Paper Does First

  • First systematic study of how prompt perturbations affect LLM-generated code security (not just correctness). Prior work (ReCode, NLPerturbator, CREME) only measured functional degradation.
  • Shows mutations as small as a single character suffice to flip secure code to vulnerable across CodeLlama-70B, DeepSeek-Coder-33B, and Qwen3-Coder-30B in C, C++, Go, JS, and Python.
  • Identifies a positional effect: mutations in the middle of prompts (docstring/CWE description region) are most destructive.
  • Uses hidden-state probing to predict vulnerability outcomes before generation begins, revealing two distinct internal decision mechanisms.
  • Extends the threat model beyond prompt injection to include ordinary developer typos and rewording.
02 — Methodology

Experimental Design

Mutation granularities

Benchmark: CWEval (Peng et al., 2025) — jointly evaluates functional correctness and security with paired test oracles. Covers 31 CWE types, 20–25 per language.

Mutation categories:

  • Single-character substitution — smallest perturbation (e.g., typos).
  • Three-character substitution within a single token.
  • Whole-token replacement by top-10 cosine-similar tokens in embedding space.

Probing: Logistic regression and 2-layer MLP classifiers on last-token hidden states. Layer and hyperparameters selected via 2-phase stratified 5-fold CV; evaluated on held-out 20% test split.

Concrete Attack Example

A single-character change (“otherwise,” → “otherwiseV” at token 26/102) in a tar extraction task caused DeepSeek-Coder-33B to drop the path-traversal guard, allowing ../../etc/passwd writes outside dest_path.

Position sensitivity: Destructive effects peak in the middle third of prompts and taper toward the beginning and end.

Asymmetry: Mutations do more harm than good. Degradation consistently exceeds improvement, especially for CodeLlama-70B and DeepSeek-Coder-33B. Qwen3-Coder-30B is the most robust.

04 — Taxonomy

Two Vulnerability Families

18 CWEs manually partitioned into two groups with a statistically significant predictability gap (Mann-Whitney p=0.009). The distinction reflects whether security decisions are made structurally (before generation) or point-wise (during token-level sampling).

Mechanistic Hypothesis

Structural security decisions (adding input validation/guards) are committed to before generation begins — visible in prompt-end hidden states. Point-wise choices (which algorithm to pick, what flag value) are deferred to next-token sampling — not predictable from prompt representations alone.

First-mover on a practically important question. Joint functionality+security evaluation avoids counting non-functional "secure" code as successes. The I/D framework and mechanistic interpretation are compelling. Excellent empirical coverage: thousands of experimental conditions across models, languages, and mutation types.

The manual I/D classification on 18 CWEs is a small, potentially biased partition. Mutations are random character substitutions, not realistic developer edits. Single-turn generation only. Code and data released under Apache 2.0.

06 — Tradeoffs

Strengths & Limitations

Strengths

  • First-mover on security consequences of prompt perturbations in coding LLMs
  • Joint functionality+security evaluation prevents false "secure" positives
  • I/D classification framework with clear mechanistic interpretation
  • Thousands of experimental conditions: 3 models x 5 languages x 3 mutation types x ~20 CWEs
  • Concrete attack examples make the threat tangible
  • Honest about limitations; does not overclaim probe performance

Limitations

  • Probe AUC 0.584–0.857 is moderate; practical deployment needs higher
  • Manual I/D classification on only 18 CWEs; needs independent validation
  • No frontier models (GPT-4, Claude, Gemini) — hidden-state extraction infeasible via APIs
  • Random mutations, not realistic developer edits; ecological validity gap
  • Single-turn generation only; multi-turn interactions untested
  • No mitigation strategies proposed; identifies problem, leaves fixes to future work
