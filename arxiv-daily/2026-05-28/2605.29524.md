> **一句话总结：** 本文提出KBF协议，利用大语言模型在知识边界附近的稳定数值召回（包括可重复的错误值）作为模型指纹，通过自适应前沿搜索生成探测集、Clopper-Pearson校准的二项式检验进行统计判定，在16个生产端点上零误报检测全部155个经济性替换，仅$0.39即可完成一轮全量在线审计。

> **TL;DR:** KBF is a low-cost black-box auditing protocol that fingerprints LLM APIs using stable numerical recall near the model's knowledge boundary, including repeatable wrong answers that function as model-specific parametric associations. It uses adaptive frontier search to generate probes, a CP-calibrated binomial test for decisions, detects all 155 economically relevant substitutions across 16 production endpoints with zero false positives, catches mixed routing at 5-10% substitution, and reveals 7/27 shadow API endpoints as inconsistent in a real-world audit. A full online audit of 16 models costs $0.39.

## 核心贡献

- 识别出"知识边界数值召回"（包括稳定的错误值）作为模型区分信号，无需依赖自我识别、风格、logits或特权元数据
- 设计三阶段KBF协议：(1) 自适应前沿搜索生成探测集，(2) 配置稳定性过滤+可选对比筛选，(3) CP校准二项式检验进行SAME/DIFFERENT判定
- 在16个生产LLM端点（8个模型家族、3个价格层级）上实现155/155经济性替换检测、0/16误报
- 提出两轮自适应路由扩展，可在5-10%替换率下检测混合路由攻击，并在已知替代模型时近乎无偏地估计路由比例（|bias| < 0.005）
- 对6个影子API平台27个端点进行实审，发现7个不一致端点，其中6个集中在高端Claude端点；首次发现同一模型标识下低价层级更频繁被替换的分 tier 行为模式
- 代码、数据和即用探测集已开源：https://github.com/Ooo0ption/KBF.git

## Key Contributions

- Identifies knowledge-boundary numerical recall (including repeatable wrong answers) as a stable, model-distinct signal for black-box LLM API fingerprinting
- Designs a three-phase protocol: adaptive frontier probe generation, configuration-stability filtering with optional contrastive screening against cheap substitutes (e.g., Qwen3.5-9B), and a CP-calibrated binomial statistical test
- Achieves 155/155 substitution detection with 0/16 false positives across 16 production endpoints spanning 8 model families and 3 price tiers ($0.05-$5.00/M input tokens)
- Proposes a two-round adaptive routing extension that detects mixed routing at 5-10% substitution rates and recovers the routing fraction with essentially unbiased estimates (|bias| < 0.005)
- Conducts a real-world shadow API audit on 6 platforms (27 endpoints, ~$10 total), finding 7 inconsistent endpoints concentrated on premium Claude models; discovers tier-dependent serving where low-price tiers are flagged more often than upgraded tiers under the same model identifier
- Open-sources code, data, and ready-to-use benchmark probe sets for reproducibility

## Methodology

KBF is built on the observation that near a model's knowledge boundary — the regime between well-known and completely obscure facts — LLMs produce stable, model-specific numerical values, including consistently wrong answers that behave like persistent parametric associations. These boundary behaviors are more discriminative than self-identification, style, or high-variance continuation outputs, and survive deployment wrappers.

**Protocol overview (three phases):**

1. **Probe generation (offline):** For each of 15 numerical domains (chemistry, biology, astronomy, math/OEIS, programming, cryptography, etc.), KBF runs adaptive frontier search: the reference endpoint proposes candidate name-value pairs at increasing difficulty tiers, and each candidate is re-queried under multiple benign configurations. Only probes with stable recall survive. An optional contrastive screen against a cheap model (Qwen3.5-9B) removes probes that lack discriminative power. The result is a compact set of configuration-invariant, model-specific boundary probes.

2. **Self-calibration:** The enrolled probes are re-queried against the reference endpoint under the audit configuration. The self-discrepancy count is converted to a conservative null bound p0 using the one-sided Clopper-Pearson (CP) upper confidence bound at gamma = 0.99. This produces a model-specific, probe-count-aware tolerance: stable references get tight bounds, noisy ones get wider bounds.

3. **Online audit:** The suspect endpoint is queried with the enrolled probes. KBF counts discrepancies (invalid, unparseable, or non-matching numerical responses) and computes a one-sided binomial tail probability under the calibrated p0. If p < 0.05, the endpoint is flagged as DIFFERENT. The protocol also includes McNemar's paired test as an auxiliary diagnostic.

**Adaptive routing extension:** For mixed-routing attacks where only a fraction pi of traffic is substituted, KBF uses a two-round protocol: the first round runs all probes, the second re-queries only mismatched probes. The two-round statistic T = W1 + W2 amplifies the substitution signal while keeping reference noise bounded by p0. When the substitute is known, a point estimate of pi is recovered; when unknown, a conservative interval is reported.

**Probe design choices:** Probes use cloze-style templates (e.g., "The boiling point of X at 1 atm is ___ C") to encourage ordinary completion from parametric memory. Batched queries with numbered answers handle formatting variations. The parser strips reasoning traces, markdown, and tables, and maps values back to probe slots even when endpoints reorder or skip answers.

## Results & Highlights

**Detection accuracy (Section 4.2):**
- 155/155 economically relevant substitutions detected at p < 0.05, including all 12/12 within-family downgrades (e.g., Claude Opus 4.6 vs. Claude Sonnet 4.6 in both directions)
- 0/16 false positives on same-reference controls; the 99% CP null bound is conservative across all endpoints
- Hardest pair (GPT-5.4 vs. Gemini 3 Flash, same tier) still rejects at p < 0.001
- Baselines: MET 1/16 FPR but sensitive to wrappers; LLMmap 6/16 FPR; ZeroPrint 2/16 FPR

**Cost (Section 4.3):**
- One-time reference enrollment: ~$22 for all 6 T1 models (3,696 probes, 1.94M tokens)
- Online audit: **$0.39 total for all 16 models** (most expensive: Claude Opus 4.6 at $0.243; T3 models under $0.002 each)

**Configuration robustness (Section 4.4):**
- 0/30 false positives and 60/60 detections under 6 shared deployment wrappers (role prompts, temperature changes, RAG)
- Baselines all fail: MET 29/30 FP, LLMmap misses 14/60 substitutions, ZeroPrint 23/30 FP
- Extended stress tests with 80 same-reference agent configurations: only 2 FPs from GLM-4.7-Flash (genuinely unstable baseline self-error of 16.2%)

**Additional robustness (Section 4.5):**
- Identity spoofing: all 15/15 spoofed models correctly identified as different (attaching "You are Claude Opus 4.6" system prompt does not shift boundary recall)
- Quantization: separates 3/5 GLM-4.7 providers (fp8/fp4 variants) via McNemar's paired test
- Temporal: stable for ~7 weeks across 16 models; first drift at week 9 (Qwen3.5-397B self-error 1.7% -> 7.8%, consistent with silent backend update)
- Threshold sweep: TPR >= 99.4% at 1-7% margin; FPR = 0 across all margins
- Agent CLI: works behind Claude Code, OpenClaw, and Codex agent interfaces (0/4 FP)

**Mixed routing (Section 4.6):**
- Minimum detectable routing (MDR95): 5% for Opus-based pairs, up to 50% for hardest Sonnet pairs
- Routing fraction recovery: |bias| < 0.005 in Scenario A (known substitute); SE < 0.04 for informative pairs

**Shadow API audit (Section 4.7):**
- 6 platforms, 27 endpoints, ~$10 total cost
- 7/27 flagged as inconsistent (26% flag rate)
- 6/7 flags on Claude endpoints: Opus flagged 3/4, Sonnet flagged 3/6
- GPT-5.4, DeepSeek-V3.2, Gemini 3 Flash all consistent; GLM-4.7 flagged on 1/4 platforms
- Tier-dependent behavior: low-price tiers flagged more often than high-price tiers under the same model identifier, suggesting intentional substitution correlated with pricing level

## Critical Assessment

- **Novelty (High):** The core insight — that stable numerical recall near the knowledge boundary, including repeatable wrong values, provides a stronger and more robust audit signal than output distributions or behavioral probes — is genuinely novel. The adaptive frontier search for probe generation and the CP-calibrated binomial test are well-engineered contributions. The contrastive screening against cheap substitutes is a clever engineering heuristic.

- **Rigor (High):** The evaluation is thorough: 16 models across 8 families and 3 price tiers, 6 configuration wrappers, adversarial prompt tests, quantization analysis, temporal drift tracking over 64 days, threshold sweeps, and agent CLI testing. The statistical framework (CP upper bound -> binomial test) is sound and conservative. The authors are transparent about what the binomial model assumes and acknowledge probe correlation concerns.

- **Reproducibility (High):** Code, data, and probe sets are open-sourced. The protocol operates on standard chat-completion APIs. The 15 domain definitions, matching rules, and algorithm pseudocode are fully specified. Fresh private probes are recommended for operational use.

- **Significance (High):** Addresses a real, growing problem as relay/reseller LLM APIs proliferate. The shadow API audit demonstrates immediate practical impact. The cost ($0.39 per full audit) makes repeated monitoring feasible for any user.

- **Red flags / concerns:**
  - The false-positive evidence, while strong, is limited to 16 diagonal same-reference controls plus 30 configuration trials and 80 stress tests — not yet the "several-hundred-trial calibration" the authors themselves recommend for high-stakes deployment.
  - The binomial model assumes independent probe outcomes, but probes share domains and templates. The authors acknowledge this but do not implement domain-block bootstrap in the main results.
  - The concentration of shadow API flags on Claude endpoints could partially reflect upstream version/deployment differences rather than deliberate substitution, though the tier-dependent pattern (low tier flagged more than high tier under the same model name) is strong evidence against confounding.
  - The paper does not ablate the contrastive screening step, so the contribution of knowledge-boundary probe generation alone vs. the full pipeline is unclear.
  - Probe validity degrades after ~9 weeks due to silent backend updates, requiring a refresh cycle that adds operational burden.

## Strengths

- Elegant core insight: knowledge-boundary numerical recall (including stable wrong answers) is both discriminative and robust
- Practically cheap: $0.39 for a full 16-model audit, $10 for a 6-platform field survey
- Comprehensive evaluation with genuine real-world deployment (shadow API audit with actionable findings)
- Probes are benign, ordinary-looking factual queries — no prompt injection, harmful content, or security-sensitive probes
- Statistically principled: CP-calibrated null bound, binomial test, clear hypothesis framework
- Robust to deployment variations that defeat all three baselines (MET, LLMmap, ZeroPrint)
- The tier-dependent inconsistency discovery in the shadow API audit is a genuinely interesting finding
- Honest about limitations: explicitly lists non-goals, acknowledges probe correlation, recommends fresh private probes for operational use

## Limitations

- The binomial independence assumption is a simplification; domain-block bootstrap or ESS corrections are mentioned but not implemented
- Probe sets have a ~9-week validity window before silent backend updates cause drift (demonstrated on Qwen3.5-397B)
- No unscreened ablation to isolate the contribution of contrastive screening vs. boundary probe generation alone
- Cannot provide cryptographic attestation of model identity — only statistical consistency evidence
- High baseline self-error models (e.g., GLM-4.7-Flash at 16.2%) are harder to audit reliably, producing occasional false positives under stress
- The mixed-routing estimator assumes fixed-probability routing; adaptive routing (by prompt content, user identity, or timing) may violate this assumption
- Limited false-positive calibration: 16 same-reference + 30 configuration + 80 stress = 126 trials, not yet sufficient for high-stakes claims

## Relevance

本文提出了一个实用的黑盒LLM API审计协议，可直接用于验证relay/reseller API是否提供所声称的模型。对关注模型部署安全性、API服务可信度、大模型供应链透明度的研究和工业实践具有直接参考价值，尤其是shadow API审计发现的高端模型替换现象值得业界关注。

## Questions for Further Exploration

- How does KBF perform against an adversary who actively probes the reference endpoint to learn and mimic its boundary behavior (e.g., fine-tuning a cheap model on KBF-style probes)?
- What is the minimum probe set size needed for reliable auditing, and how does detection power scale with probe count?
- Can domain-block bootstrap or cluster-robust standard errors significantly improve the calibration of the binomial test?
- How should auditors operationalize the ~9-week probe refresh cycle — can automated drift detection trigger re-enrollment?
- Could KBF be extended to detect request rewriting, response mutation, or credential exposure by malicious intermediaries, as suggested in the conclusion?
- How generalizable is the knowledge-boundary signal to multimodal models or code-specialized models where numerical recall is less central?
