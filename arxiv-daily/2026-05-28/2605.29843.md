> **一句话总结：** HARP提出了一种可学习的结构化双面正交旋转处理器，用butterfly-like块正交阶段替代固定Hadamard变换，在QuIP#后端下2-4bit量化中持续降低困惑度（如Llama 2 7B 2bit: 8.22→7.24），并保持128 tok/s推理速度，代码已开源。

> **TL;DR:** HARP replaces fixed randomized Hadamard transforms (RHTs) in LLM post-training quantization with a learnable two-sided orthogonal processor built from sparse butterfly-like block-orthogonal stages. It initializes exactly to the RHT processor and learns a structured orthogonal refinement per-layer on calibration data, using a diagonal Hessian-weighted reconstruction loss plus a block-diagonalization regularizer. Across 2-4 bit settings on Llama 3.2 (1B, 3B) and Llama 2 (7B, 13B, 70B), HARP consistently improves perplexity and zero-shot accuracy over fixed RHT under the same QuIP# backend, with the largest gains at 2 bits (e.g., Llama 2 7B W2 PPL: 8.22→7.24). Inference throughput remains 128 tok/s on an RTX 5080 vs. 61 tok/s for FP16. Code is open-sourced.

## Key Contributions

- Introduces HARP, a learnable two-sided structured orthogonal incoherence processor that replaces fixed RHT while preserving exact full-precision equivalence via orthogonal change of basis
- Parameterizes rotations as products of sparse butterfly-like block-orthogonal stages with Mixed-Radix schedules for arbitrary (non-power-of-two) dimensions, e.g., 5120 = (8,8,8,5,2)
- Proves exact RHT recovery at initialization (Theorem: under stated conditions, HARP at Theta=0 implements QuIP's randomized Hadamard processor exactly)
- Designs a practical fitting objective: diagonal Hessian-weighted reconstruction proxy + block-diagonalization regularizer, optimized with Adam, avoiding nested LDLQ inside the optimizer
- Achieves consistent perplexity and zero-shot accuracy improvements over fixed RHT across 2-4 bit settings on Llama 3.2 (1B, 3B) and Llama 2 (7B, 13B, 70B) under the identical QuIP# backend
- Maintains deployment efficiency: 128 tok/s inference throughput on RTX 5080 vs. 61 tok/s for FP16; supports int8 parameter storage with negligible quality loss

## Methodology

**Core idea.** HARP replaces the fixed randomized Hadamard transform (RHT) used in incoherence-based PTQ (e.g., QuIP#) with a learnable structured orthogonal processor. The key insight is that RHT spreads energy generically across coordinates, but an adaptive rotation can better align the quantization basis with the layer's weight/curvature structure and the quantizer's code family.

**Architecture.** Each HARP processor is a product of m sparse block-orthogonal stages. Stage t divides the d-dimensional vector into d/b_t contiguous blocks, applies a learnable orthogonal block B_{t,c} to each block, and strides the grouping by the product of subsequent radices. Each block B_{t,c} = Q_{t,c}(theta) * G_{b_t}, where G_{b_t} is a fixed base mixer (Hadamard for power-of-two radices) and Q_{t,c} is a learnable orthogonal matrix parameterized via Givens rotations (for b_t=2) or the Cayley map (for b_t>2).

**Initialization.** All learnable parameters are initialized to zero, so Q_{t,c}(0) = I and B_{t,c}(0) = G_{b_t}. The paper proves this recovers the exact RHT processor up to a fixed permutation convention (Theorem, Appendix A).

**Mixed-Radix support.** Non-power-of-two dimensions (e.g., 5120) are handled via factorization into mixed radices (8,8,8,5,2), using Hadamard mixers for power-of-two stages and QR fallback for odd radices. An optional Kronecker fallback aligns with QuIP's convention for dimensions of the form K * 2^L.

**Fitting procedure.** Per-layer fitting uses a surrogate loss: (1) a diagonal Hessian-weighted reconstruction proxy L_diag that measures squared quantization error weighted by rotated curvature, with stopped-gradient through the codebook assignment, and (2) a block-diagonalization regularizer R_bd that penalizes off-block Hessian energy to align the learned curvature with the backend's block structure. Optimized with Adam from RHT initialization, with optional periodic target refresh to reduce calibration cost.

**Computational cost.** O(d log d) application cost (Hadamard-like), O(d * b_t) parameters per stage, controlled by bounded radices. For Llama 2 7B at 2 bits: ~9.7 GPU-hours for full fitting (H100), reducible to ~4.0 hours with target refresh every 4 steps with small quality change.

## Results & Highlights

**Perplexity (main results, QuIP# backend, context 4096 for Llama 2, 8192 for Llama 3.2):**
- Llama 2 7B, 2-bit: RHT 8.22 → HARP 7.24 (Mixed-Radix), 7.85 (Kronecker) on WikiText2
- Llama 2 13B, 2-bit: RHT 6.52 → HARP 6.13 on WikiText2
- Llama 2 70B, 2-bit: RHT 4.55 → HARP 4.39 on WikiText2
- Llama 3.2 1B, 2-bit: RHT 10.64 → HARP 8.93 on WikiText2
- Llama 3.2 3B, 2-bit: RHT 7.88 → HARP 6.91 on WikiText2
- 3-bit and 4-bit: consistent but smaller improvements, as RHT baseline is already closer to FP16
- C4 perplexity shows the same pattern across all settings

**Context-matched comparison (Llama 2, context 2048):**
- 2-bit HARP (7.85/9.68 W2/C4) outperforms AWQ (2.2e5/1.7e5), GPTQ (36.77/33.70), OmniQuant (11.06/15.02), and QuIP# RHT (8.95/11.22)
- 3-bit and 4-bit: HARP matches or slightly exceeds the best published scalar baselines

**Zero-shot accuracy (Llama 2, 2-bit):**
- ARC-Challenge: RHT 37.5 → HARP 39.8 (7B), RHT 39.6 → HARP 41.4 (13B)
- WinoGrande: RHT 61.3 → HARP 62.4 (13B)
- PIQA: improvements on 7B, close on 13B

**Inference latency (RTX 5080, batch=1, seq=1):**
- FP16: 61 tok/s
- HARP 2-bit: 128 tok/s (vs. RHT 2-bit: 141 tok/s) -- retains ~91% of RHT throughput
- HARP 4-bit: 178 tok/s (vs. RHT 4-bit: 196 tok/s)

**Storage overhead:** HARP with int8 parameter storage adds only ~0.05 GB to nominal model size at 2-bit for 7B (2.20 GB vs. RHT 2.15 GB)

**Backend transferability:** HARP successfully transfers to QTIP backend with no backend changes, demonstrating the processor is backend-agnostic.

**Fine-tuning compatibility:** HARP + FT-quant (without E2E FT) achieves PPL 6.16 on Llama 2 7B 2-bit, better than RHT + FT-quant + E2E FT (6.19).

## Critical Assessment

- **Novelty (High):** The idea of making the Hadamard transform learnable for PTQ is a natural but technically non-trivial extension. The butterfly-like factorization with Mixed-Radix support and the exact RHT initialization theorem are genuine contributions. The block-diagonalization regularizer that aligns curvature with the quantizer's block structure is a clean design choice. However, the overall concept of learned rotations for quantization has been explored by SpinQuant and Wush; HARP differentiates itself through strict orthogonality, structured parameterization, and backend-agnostic drop-in compatibility.

- **Rigor (High):** The experimental methodology is commendable. The authors isolate the processor change by keeping the entire QuIP# pipeline fixed (codebooks, solver, random signs, evaluation harness) and only swapping RHT for HARP. This is a clean ablation. The context-matched comparisons against AWQ/GPTQ/OmniQuant are appropriately separated as system-level. The calibration-time ablation (Table 7) is informative. The paper provides theoretical guarantees (Theorem on exact RHT recovery). Missing: no direct comparison against Wush (Chen et al., 2025), which is the most closely related adaptive transform method; it is only cited but not experimentally compared.

- **Reproducibility (High):** Code is open-sourced. Hardware specs, hyperparameters, and evaluation protocols are documented in appendices. The paper includes a reproducibility checklist.

- **Significance (Moderate-High):** HARP demonstrates meaningful perplexity improvements at extreme quantization (2-bit), where most methods struggle. The 2-bit Llama 3.2 1B improvement (10.64 → 8.93) is substantial. The drop-in compatibility with existing pipelines is practically valuable. However, the calibration cost (4-10 GPU-hours for 7B) is non-trivial and may limit adoption compared to simpler methods like AWQ or GPTQ for less extreme settings. The improvements at 3-4 bits are modest.

- **Red flags:** (1) The comparison against Wush is conspicuously absent from experiments, despite being the most directly competing method. (2) The paper evaluates only weight-only PTQ -- no weight-activation quantization results, which the authors acknowledge as future work. (3) Only Llama model families are tested; generalization to other architectures (Mistral, Gemma, etc.) is not demonstrated. (4) The zero-shot evaluation covers only 4 tasks (ARC-C, ARC-E, PIQA, WinoGrande), which is a limited benchmark suite.

## Strengths

- Clean experimental design that isolates the processor change from all other pipeline components
- Exact RHT initialization with formal proof, ensuring HARP is a strict upgrade path
- Supports non-power-of-two dimensions via Mixed-Radix, important for practical model architectures
- Maintains most of the RHT throughput advantage (91% at 2-bit) while improving quality
- Compact int8 storage of parameters adds negligible overhead (~2% at 2-bit 7B)
- Backend-agnostic: demonstrated with both QuIP# and QTIP backends
- Compatible with fine-tuning stages, yielding further improvements
- Open-source code release

## Limitations

- Calibration cost: 4-10 GPU-hours per model (H100) for 7B at 2-bit, significantly more than fixed RHT (~0.7h). Target refresh partially mitigates this but does not eliminate it.
- Weight-only PTQ: no evaluation under weight-activation quantization, which is increasingly important for edge deployment
- Limited model diversity: only Llama 2 and Llama 3.2 tested; no results on other popular architectures
- Zero-shot evaluation limited to 4 tasks; a broader suite (MMLU, HellaSwag, etc.) would strengthen the claim
- No direct experimental comparison against Wush (Chen et al., 2025), the closest competing method
- Modest improvements at 3-4 bits, where the fixed RHT baseline is already strong
- The fitting procedure introduces new hyperparameters (lambda_bd, target refresh interval, number of optimizer steps) that may require tuning

## Relevance

## Questions for Further Exploration

- How does HARP compare experimentally against Wush (Chen et al., 2025), which also proposes adaptive transforms for quantization? The absence of this comparison is notable.
- Can HARP be extended to weight-activation quantization scenarios, and how would it interact with KV-cache quantization?
- What is the sensitivity of HARP's quality gains to calibration data size and composition? The paper does not ablate this.
- How well does HARP generalize to non-Llama architectures (e.g., Mistral, Gemma, Phi)?
- Could the fitting procedure be accelerated via meta-learning or initialization transfer across layers, reducing the 4-10 GPU-hour calibration cost?
- Is the block-diagonalization regularizer necessary, or does the diagonal proxy alone suffice? The paper does not ablate lambda_bd = 0.
- How does HARP interact with mixed-precision quantization strategies where different layers use different bit widths?
