Why This Matters
A test-time paradigm that learns selection from confident samples and transfers to uncertain ones — keeps improving with compute where memoryless methods plateau.
Critical Edge
Error bars are explicitly omitted, and AIME sets have only 30 problems each — a single swing is 3.33 pts, so several headline wins sit inside plausible noise.
Methodology

MILES augments a frozen LLM with an external step-wise memory built from the model's own reasoning. Two modes handle confident vs. uncertain samples differently.

Memory unit — an asymmetric pair (g_i, u_i): g_i is a sub-goal embedding (retrieval key); u_i is a natural-language sub-instruction. Units come from clustering (step, step-goal) tuples in confident trajectories, then LLM-summarizing a generalized instruction per cluster.

score(a,x) = (1−β_c2f)·cos(ĝ,g_a) + β_c2f·π_θa(a|x)

Two-layer coarse-to-fine selection. Layer 1 (coarse): the LLM proposes candidate step-goals, matched to stored g_i by cosine similarity with a per-unit threshold. Layer 2 (fine): a learned per-item selection head π_θa(a|x) scores applicability via a max-similarity-pooled state fingerprint.

Learning the heads. Selection maximizes Σ_a π(a|x)·V(x,a) (expected final-answer correctness) under a KL penalty to the similarity distribution. The closed-form optimum π*(a|x) ∝ π_sim·exp(V/β) is approximated as tractable one-vs-rest BCE classifiers — positives = states where unit a led to a correct answer. Per-item design supports incremental memory growth without retraining existing heads.

Pipeline. For each test question, K responses are generated. Confident samples (majority agreement) drive MCTS to collect (state, outcome) tuples and train heads. Uncertain samples are routed to memory-guided tree search; the final answer is majority-voted. Heads = ensembles of 10 × 3-layer MLPs (16 hidden); embeddings from text-embedding-3-small.

Matches or beats the best baseline on every row across 6 benchmarks × 4 backbones.

  • Accuracy/token frontier (Fig. 2): dominates SC, ToT, rStar, DORA across 2M–6M token budgets and keeps climbing where baselines flatten.
  • Two-layer ablation (AIME 24): none 60.0 → L1 only 63.3 → L2 only 63.3 → both 66.7. LLM-prompt reranking fails (60.0).
  • Cross-model transfer: memory built from Qwen3-30B aux. trajectories lifts GPT-4.1-mini AIME 24 by +6.7.
  • Offline memory on hardest samples: AIME 25 +15.4 pts, GPQA +9.7 pts over no-memory.

Strengths

  • Principled: selection tied to final-answer correctness via a derived KL-regularized optimum, not heuristic similarity.
  • Genuinely test-time-friendly: no param updates, no external labels, no fixed action space, incremental growth.
  • Comprehensive ablations & sensitivity studies (α_sum, cluster size, α_eff, codebook components) across Appendix F.
  • Cross-model and offline-memory transfer show the textual memory is portable, including weaker→stronger backbone.
  • The per-problem efficiency frontier is the most compelling single result.

Limitations

  • Small AIME sets (30 problems); error bars omitted — point differences hard to distinguish from noise.
  • Large open-weight deltas partly inflated by broken baselines (BoT/DC underperform zero-shot CoT).
  • Many interacting components + per-dataset hyperparameters (β_c2f = 0.0 math vs. 0.4–0.6 academic) complicate reproduction.
  • No comparison vs RL-from-feedback memory methods (ICPI, Memento, DynaAct); no confirmed code release.
  • Memory may inherit and amplify biases from collected trajectories; only high-level mitigation proposed.
Open Questions
  1. Do the small-benchmark wins (e.g., +10 on AIME 24) survive on larger eval sets with computed error bars?
  2. What is the wall-clock / total-token cost of MILES's MCTS data collection vs. spending the same compute on Self-Consistency?
  3. Can the asymmetric (embedding, text) memory contribution be isolated from the two-layer selection via a symmetric text-only ablation?
  4. How sensitive is the method to the embedding model — does an open-weight embedder change cross-model transfer?
  5. Does memory degrade or accumulate noise over long test streams (hundreds/thousands of questions)?
  6. Could the learned selection heads combine with process reward models (which the authors avoid) to push the frontier further?
