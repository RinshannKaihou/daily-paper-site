> **一句话总结：** 提出 EVOSOP 框架，让 LLM 智能体在不修改模型参数的前提下，通过"构造-合并-评估-剪枝"的迭代生命周期，把原子动作合成为可复用的高阶 SOP 工具，在 ACEBench 和 Tau2Bench 上显著提升任务成功率并减少交互轮次。

> **TL;DR:** EVOSOP is a model-agnostic, parameter-free framework that lets LLM agents self-evolve by repeatedly extracting recurring atomic-action patterns from execution trajectories into reusable, callable Standard Operating Procedures (SOPs) and then merging/evaluating/pruning them. On ACEBench (Multi-Step/Multi-Turn) and Tau2Bench (Telecom solo), EvoSOP beats the ReAct base by up to +13.4pp and outperforms the one-shot ASI baseline by up to +24pp (e.g., ReAct+GPT-4o Multi-Step: 84.2% vs ASI 60.0%), while cutting reasoning rounds and keeping the toolset compact (typically <10 SOPs).

## 核心贡献

- **提出"迭代式工具优化"的自演化新范式**：把智能体自我改进建模成与机器学习流程对齐的循环——数据采集（执行训练任务）、前向传播（用当前工具集执行任务）、反向传播（从轨迹中抽取可复用 SOP）、正则化（合并与剪枝）、衰减学习率（工具集收敛）。区别于把工具创建当作一次性事件的做法。
- **设计 EVOSOP 四模块生命周期**：CONSTRUCTOR（从轨迹抽取共现的原子动作段并重写为可执行代码的 SOP）、MERGER（合并功能重叠的 SOP，且采用非破坏性策略保留旧工具直到验证阶段）、EVALUATOR（在真实环境重跑训练任务收集实证轨迹）、REVIEWER（作为 critic 把每次 SOP 调用归入最优执行/部分有效/中性/负向干扰/实现缺陷五类，并据此剪枝）。
- **无参数、模型无关**：不改动底层 LLM 权重，仅迭代优化外部工具集，可直接增强 GPT-4o 等黑盒模型。
- **实证有效且工具集保持精简**：在两个基准上同时提升成功率和压缩推理轮次，工具集在动态平衡中维持在个位数规模（典型 <10 个 SOP）。
- **消融揭示剪枝最关键**：去掉 REVIEWER 性能下降最明显，说明"主动管理工具集"与"创建工具"同样重要，解释了相对一次性 ASI 基线的性能差距。

## Key Contributions

- **A new self-evolution paradigm via iterative tool optimization.** The authors formalize agent self-improvement as a loop that mirrors the standard ML pipeline (data acquisition, forward execution, "symbolic" backward propagation of extracted SOPs, regularization via merging/pruning, and a learning-rate-like convergence of the toolset), rather than treating tool creation as a one-off event.
- **The EVOSOP four-module lifecycle (CONSTRUCTOR, MERGER, EVALUATOR, REVIEWER).** Constructor distills recurring atomic-action subsequences into callable code with lightweight conditional/error-handling logic; Merger consolidates functionally overlapping SOPs using a deliberately non-destructive strategy (constituents are kept until empirically validated); Evaluator re-executes training tasks with the expanded toolset F' = F_atomic ∪ schemas(S); Reviewer classifies each SOP invocation into five nuanced states (Optimal/Partial/Neutral/Negative/Defect) and prunes accordingly.
- **Parameter-free, model-agnostic design.** No gradients or fine-tuning of the backbone LLM; the entire "training" is iterative toolset optimization, so it applies to black-box models like GPT-4o.
- **Empirical validation across benchmarks and backbones.** Evaluated on ACEBench (Multi-Step and Multi-Turn) and Tau2Bench (Telecom solo), with construction run on three LLMs (GPT-4o, Gemini-3-Flash-Preview, Qwen-Max) for cross-model robustness while evaluation always uses GPT-4o.
- **Mechanistic findings via ablation and case study.** Removing the REVIEWER hurts most, confirming that lifecycle management — not just creation — drives the gains; a per-SOP lifetime analysis (Figure 4) shows how fundamental/transient/merged SOPs are created, merged, and pruned.

## Methodology

EVOSOP operates on top of a standard ReAct/DFSDT agent whose toolset F initially contains only granular atomic actions. The framework's core is an iterative tool-optimization lifecycle composed of four LLM-based modules:

1. **CONSTRUCTOR** takes raw execution trajectories ξ and runs a two-stage transform: `f_extract(ξ_i)` identifies logically coupled, frequently co-occurring subsequences of tool calls (e.g., search→write_file), then `f_rewrite(ξ'_i, F_atomic)` maps them into SOP source code interleaved with conditional checks and error handling so they generalize across states. SOPs are constrained to contain 2–5 atomic actions. SOPs are only ever *introduced* in this stage, keeping toolset growth grounded in observed experience.

2. **MERGER** inspects newly constructed SOPs within a training batch for functional overlap (shared objective or near-identical logic) and produces consolidated, higher-order SOPs via `f_merge`. Critically, merging is *non-destructive*: constituent SOPs are not immediately removed, because nominally identical SOPs can differ in robustness, coding style, and docstring clarity — all of which affect whether the LLM invokes them correctly. Removal is deferred to the empirical verification phase.

3. **EVALUATOR** loads all candidate SOPs (both newly constructed S_b and merged S') into the real execution environment, exposes their functional schemas (name/description/params) to the agent so F' = F_atomic ∪ schemas(S), and re-executes all training tasks to collect verification trajectories ξ̂ that capture real-world utility and failure modes.

4. **REVIEWER** acts as an LLM critic, classifying each SOP invocation into one of five non-mutually-exclusive states — Optimal Execution, Partial Utility, Neutrality, Negative Interference, or Implementation Defect — each with a natural-language justification for interpretability. Performance statistics are aggregated across the training batch, and a global filtering `f_check` removes SOPs with high error rates, semantic misalignment (misleading name/docstring), low invocation frequency, or latent defects.

**Training workflow and design parallels.** The framework draws an explicit analogy to ML training: training-task execution = data acquisition + forward propagation; trajectory analysis = symbolic "backpropagation" that produces SOPs instead of gradient updates; merging/pruning = regularization that prevents overfitting to narrow, task-specific contexts; the stabilization of the toolset = a decaying learning rate. Mini-batching feeds 5 trajectories per batch for the first 5 iterations (25 total), then the last 5 iterations perform only merging/evaluation/review with no new SOP construction. A **checkpointing** mechanism archives logs and toolset each iteration, and the epoch with the highest *training* success rate is selected as final (no separate validation set is needed because feedback is self-supervised from the environment). The full procedure is given as Algorithm 1.

## Results & Highlights

All numbers below are task success rates (%) with GPT-4o as the evaluation backbone; construction backbone varies by row. Standard errors in parentheses.

- **ACEBench Multi-Step (ReAct base).** EvoSOP reaches **84.2±1.9%** with GPT-4o construction and **83.3±2.4%** with Gemini-3-Flash-Preview, versus the one-shot ASI baseline at **60.0±12.9%** and **45.8±14.8%** respectively — a delta of **+24.2pp / +37.5pp** over ASI, and **+3.4pp / +2.5pp** over the plain ReAct base (80.8%). Across both ACEBench subsets and backbones, gains over base range from **2.5% to 13.4%**.
- **ACEBench Multi-Turn (ReAct base).** EvoSOP hits **79.4±3.0%** (GPT-4o) and **84.4±3.1%** (Qwen-Max), versus ASI at **57.2±8.0%** and **65.6±8.8%** — deltas of **+22.2pp** and **+18.8pp** over ASI; the ReAct base here is 72.2%.
- **ACEBench Multi-Step (DFSDT base).** EvoSOP achieves **85.0±0.0%** (GPT-4o) and **85.8±1.9%** (Qwen-Max), beating the DRAFT baseline at **75.0±5.8%** and **74.2±5.4%** by **+10.0pp / +11.6pp**.
- **Tau2Bench Telecom Solo (ReAct base).** EvoSOP attains **43.3±0.8%** (Gemini) and **40.9±0.4%** (GPT-4o), versus the ReAct base at **37.1%** and ASI at **35.4±1.1%** — gains of **+6.2pp / +5.5pp over ASI** and **+3.8pp** over base on GPT-4o. Margins are deliberately characterized as "more concentrated" due to Tau2Bench's large tool space and intricate stateful transitions.
- **Reasoning compression (Figure 2).** The average number of reasoning rounds per task drops substantially as epochs proceed, stabilizing at a "significantly lower level than baselines" in the final epochs — directly cutting API latency/cost and reducing long-horizon focus loss.
- **Toolset stays compact (Appendix C, Figure 5).** After an initial expansion in epochs 1–5, the maintained SOP count enters a dynamic equilibrium, "typically fewer than 10 SOPs," while still achieving ~80% task success.
- **Ablation (Figure 3).** Removing the REVIEWER causes the largest performance drop on both ACEBench subsets; removing the MERGER also degrades results, confirming gains come from component *synergy* rather than any single module.
- **Failure-mode case study (Figure 4).** In text-message management, ASI-induced SOPs would "delete the latest message instead of the earliest" due to brittle one-shot logic; EVOSOP's Reviewer detects such subtle errors and prunes them, while merging task-specific `order_food_online` + `order_food_from_merchant` into a generalized `enhanced_manage_food_order`.

## Critical Assessment

- **Novelty.** The core idea — synthesizing multi-step workflows into higher-order callable tools — is not new (Voyager, CRAFT, ASI all do this). The genuine novelty is the *closed iterative lifecycle* with explicit merging and LLM-critic-based pruning, and especially the conceptual reframing as a non-parametric analogue of the ML training pipeline (forward/backward/regularization/learning-rate decay). That framing is useful and reasonably well-argued, though it is largely metaphorical rather than formal — no gradient-like convergence guarantee is provided.
- **Rigor.** A key honesty signal: the authors concede the gains on Tau2Bench-Telecom are "more concentrated," and some Tau2 margins are small enough that standard errors are worth scrutinizing (e.g., EvoSOP Qwen-Max 40.2±2.2 vs base ~37.1 on Telecom). The 3–6 complete workflows per benchmark give standard errors but sample sizes are modest.
- **Baseline quality (a real red flag).** The ASI baseline behaves erratically and often *worse than the no-SOP ReAct base* — e.g., ACEBench Multi-Step drops from 80.8% (ReAct, Gemini) to 45.8% (ReAct+ASI, Gemini), a ~35pp collapse. This suggests either ASI is being applied sub-optimally or that one-shot skill induction is genuinely fragile. Either way, it makes ASI a conveniently weak comparator, and the headline "up to +24pp over ASI" deltas are partly inflated by ASI underperforming the plain base. The more meaningful comparison is EvoSOP vs the *plain ReAct/DFSDT base*, where gains are a modest +2.5 to +13.4pp.
- **Missing ablations / analyses.** There is no dedicated ablation isolating the CONSTRUCTOR's contribution (only Merger/Reviewer are removed), no analysis of sensitivity to the "2–5 atomic actions" SOP-size constraint, and no breakdown of how often EvoSOP's gains come from *new* SOPs vs *better-organized* tools. The cost of the optimization loop itself (LLM API calls for Constructor/Merger/Reviewer across 10 epochs × 3 construction backbones) is not reported — a notable gap for a method whose selling point is efficiency.
- **Selection criterion concern.** The final toolset is chosen as "the iteration that yields the highest *training* success rate." Although the authors justify this by saying environment feedback is self-supervised and no validation set is needed, with only 25 training trajectories and checkpoint-based selection this is a textbook overfitting setup. Reported test results include "unseen tasks," but the paper does not cleanly separate in-distribution vs out-of-distribution performance, so generalization claims are under-supported.
- **Reproducibility.** Prompts are shared (Appendix D), settings are detailed, and the method is model-agnostic and CPU-only — good. However, no code/model release is mentioned, and the non-determinism of LLM-as-critic (the REVIEWER) is not addressed; its own failure modes could silently disable useful skills — exactly the phenomenon analyzed in the sibling paper "The Blind Curator" appearing in the same daily batch.

## Strengths

- Genuinely iterative, closed-loop design with a principled four-stage lifecycle; the non-destructive merging decision (keep constituents until empirically validated) is a thoughtful, well-justified engineering choice.
- Parameter-free and model-agnostic — directly applicable to black-box LLMs, which is practically valuable.
- The REVIEWER's five-state failure taxonomy (Optimal/Partial/Neutral/Negative/Defect) with natural-language justifications makes pruning interpretable and traceable.
- Strong results on ACEBench across three construction backbones, plus consistent ablation evidence that pruning (not just creation) drives performance.
- Compact toolset (typically <10 SOPs) avoids the context-bloat failure mode that motivates the whole paper, and the convergence behavior is nicely visualized (Figures 4 and 5).

## Limitations

- Only two benchmark families evaluated (ACEBench, Tau2Bench); no web-agent, code-agent, or long-horizon embodied benchmarks (e.g., GAIA, SWE-bench-style) to test the generality of "SOP" abstraction.
- The headline deltas over ASI are inflated by ASI frequently underperforming the plain base; the more honest gains over base are modest (+2.5 to +13.4pp on ACEBench, smaller on Tau2Bench).
- Final toolset selected by peak *training* success rate with only 25 training trajectories and no separate validation set — an overfitting-prone protocol.
- LLM-as-critic REVIEWER introduces its own bias and stochasticity, which is neither measured nor mitigated (and is the subject of concurrent critical work).
- Computational cost of the 10-epoch, three-backbone optimization loop (and the additional re-execution of all training tasks each epoch by the EVALUATOR) is not reported, undercutting the efficiency narrative.
- No human evaluation of synthesized SOP quality (readability, correctness of docstrings, edge-case handling) beyond the automatic REVIEWER.

## Relevance

This paper is a direct hit on the self-evolving agents and recursive agent optimization interests: it formalizes iterative, non-parametric toolset evolution as a self-improvement loop and provides concrete evidence that *lifecycle management* (merge/prune), not just tool creation, is what makes an agent reliably self-evolve. It also pairs naturally with the reliability/failure-mode theme, since its REVIEWER-critic and checkpointing design are themselves subject to the biased-judge failure modes analyzed in the concurrent "Blind Curator" paper.

## Questions for Further Exploration

- How does EvoSOP perform on truly long-horizon, open-ended agent benchmarks (GAIA, SWE-bench, WebArena) where SOP abstraction should matter most — and does the <10-SOP equilibrium still hold?
- What is the marginal contribution of *new* SOPs versus *reorganization* of the existing toolset? An ablation that disables CONSTRUCTOR after epoch 5 from the start would isolate this.
- How sensitive are results to the REVIEWER's bias? Could a biased or miscalibrated critic silently disable useful skills (the "Blind Curator" phenomenon), and would an ensemble or consensus-based REVIEWER mitigate it?
- Is selecting the final checkpoint by peak *training* success rate a source of overfitting? What does a held-out validation curve look like across epochs?
- What is the total API cost (tokens, dollars, latency) of the 10-epoch, three-backbone optimization loop, and how does amortized cost-per-task compare to one-shot baselines like ASI after many test queries?
- Can the "symbolic backpropagation" analogy be made formal (e.g., a convergence or regret bound), or does it remain purely metaphorical?
