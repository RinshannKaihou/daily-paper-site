Key Finding/LLM-Generated Skills Don't Help

Across 7,560 runs, single-shot LLM-generated skill files give no reliable lift over task-only prompting — and actively harm data preparation.

Do LLM-Generated Skills Make Better AI Data Scientists? A Component Ablation Across Data-Science Workflows
Wei-Jung Huang (independent researcher, USA) — single author
56 tasks · 9 model configs · 3 providers · 10,584 runs  ·  AIDataSci '26, Jeju  ·  2026-07-08
Key Metric
−0.8pp
Full skill < No-Skill on aggregate: 67.5% vs 68.3% pass rate — statistically indistinguishable from a token-matched office-supply control.
total span across 5 conditions: 1.2 pp  ·  all p ≥ 0.396  ·  costs ~4.5× input tokens
Why This Matters

Before adopting family-level LLM-generated SKILL.md files as a default, check baseline difficulty — and prefer task-specific, on-demand, system-priority mechanisms.

Critical Edge

No expert-written positive control — so the null can't distinguish "LLM skills are useless" from "family-level breadth is useless" from "this benchmark is insensitive."

Missing positive control

Methodology — a component ablation designed to isolate mechanism

Four skill sections, ablated mechanically; two controls that rule out the obvious confounds

One skill per lifecycle stage (4 total) was generated by a Gemini 2.5 Pro coding agent in a single autonomous session — no human editing, no candidate selection, no iterative refinement. Each skill has four Anthropic-schema sections: Routing (~50 tok), Core Procedure (~190), Worked Examples (~225), Reference Notes (~225). Worked Examples and Reference Notes are treated as two independent binary factors; variants are made by mechanically deleting sections from the Full skill, so they differ only in which sections are present.

Two mechanism-isolating controls do the heavy lifting: Length-Control (a token-matched office-supply skill, irrelevant domain) rules out "longer prompt helped"; Full+Priority (a user-message directive that task overrides skill) tests the conflicting-heuristics hypothesis. All 10,584 outcomes are binary pass/fail via deterministic verifiers — no LLM-as-judge — analyzed by mixed-effects models, group-stratified bootstrap, McNemar, and permutation tests, all agreeing.

Harm locus
8 of 14 condition-flip cases (Core-Only pass → Core+Refs fail) are data-prep tasks — generic "forward-fill if <30% missing" heuristics override explicit task instructions.
Informative band
Only 17 of 56 tasks fall in the 30–80% range where skills could plausibly help; 39 are near ceiling/floor and uninformative.
Cost
Skill injection costs ~4.5× input tokens (mean 1,293 vs 287) with no measured accuracy gain.
Model groups
No group benefits: compact (0.0 to +0.6 pp), frontier (−1.8 to +0.6), explicit-reasoning (largest −1.8 for Core+Refs).

Strengths

  • Exceptional hygiene — pre-finalized tasks/golds, deterministic verifiers, no LLM-as-judge, condition-flip case review
  • Four converging analyses (mixed-effects, logistic GEE, bootstrap, permutation) that agree, plus honest wide CIs
  • Length-Control design cleanly separates "skill content helps" from "long prefix has generic effect"
  • Three-regime taxonomy (ceiling / conflicting heuristics / floor) is an immediately actionable diagnostic
  • Transparent about authorship overlap and the missing positive control

Limitations

  • No expert-written positive control — biggest threat to the strong title claim
  • Narrow scope — 56 single-turn tasks; only 17 in the informative band; CIs ±4–6 pp can't detect small effects
  • Not a full agent setting — no multi-turn planning, tool use, code execution, or memory
  • Flat injection only — doesn't test Anthropic's recommended selective-loading architecture
  • One skill per stage (n=4) — no inter-generation variance; effects tied to four specific files
  • Authorship overlap — Gemini 2.5 Pro is skill author, task drafter, and evaluated model

Open Questions — for further exploration

Would an expert-written or task-specific positive control show gains on the same 56 tasks — is the null a property of LLM-generated skills, family-level breadth, or benchmark sensitivity?
Does selective / on-demand loading (Anthropic's multi-file architecture with retrieval gating of Reference Notes) recover the data-preparation harm, given Core+Refs is the worst component?
Would multiple independently generated skills per stage (estimating inter-generation variance) change the conclusion, or are these four files unusually conflict-prone?
Can system-level priority rules reliably resolve skill-task conflicts — versus the weak user-message Full+Priority that added only +0.4 pp over No-Skill?
Does adding execution feedback / iterative revision help the floor-regime Statistical Analysis tasks (29.4% baseline), where prompt-level injection adds only +0.8 pp?
