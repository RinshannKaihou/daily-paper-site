> **一句话总结：** Deep4ge 发布了一个包含 14,227 次 DNN 训练运行（来自 59 个 Stack Overflow 的 TensorFlow/Keras 程序，经 27 个变异算子注入 7 类故障）的公开基准数据集，并通过概念验证实验证明：基于完整训练轨迹的特征（MCC 0.227）显著优于仅用末轮 epoch 特征（MCC 0.150）进行故障检测与诊断。

> **TL;DR:** Deep4ge is a public, mutation-derived benchmark of 14,227 per-epoch DNN training runs (9,845 faulty / 4,382 correct) built from 59 TensorFlow/Keras programs harvested from Stack Overflow and 27 source-code mutation operators across 7 fault categories; on held-out-program fault detection, full-trajectory features raise Random Forest MCC from 0.150 (final-epoch) to 0.227 and AUROC from 0.649 to 0.716, beating the trivial always-faulty baseline (F1 0.818) on imbalance-robust metrics.

## 核心贡献

- **首个公开的按 epoch 记录的 DNN 故障训练轨迹数据集**：包含 14,227 次训练运行、719,560 条 epoch 记录，覆盖 7 类故障与 3 种架构（FNN、CNN、RNN），每次运行记录 4 个评估指标和 26 个动态特征（权重、梯度、激活、准确率/损失趋势、学习率、硬件使用）。
- **开源的 AST 故障注入框架**：基于 27 个源码级变异算子（融合 DeepCrime 子集 + 新增 layer 级算子），通过重写抽象语法树注入故障并自动插入日志回调，支持带验证脚本的数据集再生。
- **严格的变异验证流程**：采用 DeepCrime 的统计杀死准则（Cohen's d ≥ 0.5 且 p < 0.05，n=15 次重训），仅保留被杀死的变异体；最终算子集的变异分数为 FNN 0.85、CNN 0.87、RNN 0.83。
- **概念验证的检测/诊断基线**：在分组感知（按 SO 程序分组防泄漏）的 5 折交叉验证下，比较末轮 epoch 与轨迹特征表示，并报告 F1、Macro-F1、平衡准确率、MCC 及 1000 次 bootstrap 置信区间。
- **跨架构迁移与早期检测研究**：训练于两种架构、测试于第三种架构的迁移实验，以及单 epoch 早期检测实验，揭示哪些故障类别可迁移、何时可被检测。

## Key Contributions

- **First public dataset linking adapted DNN programs, mutation-derived fault categories, and per-epoch training trajectories** in a single reusable benchmark — 14,227 runs / 719,560 epoch records, 9,845 faulty and 4,382 correct baselines, spanning 7 fault categories and 3 architecture families.
- **Open-source AST-based fault-injection framework** of 27 source-level mutation operators (a DeepCrime subset plus new layer-level operators not supported in prior work), applying operators by rewriting the program's abstract syntax tree and inserting a per-epoch logging callback; released with validation scripts for full regeneration.
- **Rigorous statistical mutation killing** following DeepCrime (Cohen's d ≥ 0.5, GLM p < 0.05 over 15 retrainings); operators retained only if killable on at least one program of the same architecture, achieving mutation scores of 0.85 (FNN), 0.87 (CNN), 0.83 (RNN).
- **Proof-of-concept baselines** for binary fault detection, 7-class fault diagnosis, cross-architecture transfer, and single-epoch early detection, evaluated with group-aware 5-fold CV (grouped by Stack Overflow program ID) to prevent leakage.
- **Public, citable release** under CC BY 4.0 (data) and MIT (framework), archived on Zenodo (DOI 10.5281/zenodo.20337241) with code at github.com/SigmaJahan/deep4ge.

## Methodology

Deep4ge is constructed via a controlled, four-stage pipeline (Algorithm 1):

1. **Program selection.** 59 complete TensorFlow/Keras DNN programs were collected from Stack Overflow posts containing a full model definition (data loading, compilation, training). Each was manually repaired to run deterministically, with fixed seeds at framework and operator levels, consistent dataset splits, and consistent initialization. Coverage spans feedforward (FNN), convolutional (CNN), and recurrent (RNN) architectures.

2. **Fault injection.** Starting from each corrected program, faulty variants are generated by applying one of 27 AST-based mutation operators grouped into seven categories: training configuration (hyperparameters), activation, loss, optimizer, weight initialization & bias, regularization, and layer structure. The operator set combines a subset of DeepCrime operators with new layer-level operators (e.g., kernel size, LSTM layer count) introduced by Deep4ge. Each released run carries one category label, making diagnosis a seven-class single-label task.

3. **Training and feature extraction.** Each mutant and its correct baseline are trained for up to n=15 retrainings, logging per epoch: an epoch index, 4 evaluation metrics (train/val loss and accuracy), and 26 dynamic features — 4 weight measures (large-weight count, unchanged-mean/std flags, NaN count), 4 accuracy/loss trend measures (gap, oscillation, decrease, increase), 4 activation measures (dying ReLU, saturation, mean, std), 10 gradient measures (vanishing/exploding flags, NaN count, mean/std/min/max/median layer norms, per-weight mean/std), and 4 resource measures (learning rate, CPU%, peak GPU memory MB, system memory%).

4. **Mutation validation and cleaning.** A mutant is killed when the accuracy distribution shift vs. the original reaches Cohen's d ≥ 0.5 and p < 0.05 (GLM) over 15 retrainings. Non-killed mutants (likely equivalent) are dropped. Filenames are normalized, booleans converted, and incomplete runs (no epoch completed) discarded; a validation script enforces consistency.

**Benchmark setup.** Two feature representations are compared: *final-epoch* (30 values at the last epoch) vs. *trajectory* (each of the 30 features summarized by 6 statistics — mean, std, min, max, last, linear slope — yielding 180 features). Logistic Regression and Random Forest are evaluated with 5-fold group-aware cross-validation (grouped by SO program ID so a program's runs never span train and test), reporting F1, Macro-F1, balanced accuracy, MCC, and 95% bootstrap CIs over 1,000 resamples. Trivial always-faulty and stratified-random classifiers serve as baselines.

## Results & Highlights

- **Trajectory features substantially beat final-epoch features on detection.** Using the full per-epoch trajectory instead of only the last epoch raises Random Forest detection **MCC from 0.150 to 0.227**, with non-overlapping 95% bootstrap CIs ([0.132, 0.169] vs. [0.209, 0.245]); AUROC rises from **0.649 to 0.716** and AUPRC from **0.804 to 0.842** on the Deep4ge benchmark.
- **Final-epoch detection collapses to the trivial baseline under imbalance.** Because 9,845 of 14,227 runs are faulty, the always-faulty classifier already scores **F1 = 0.818**, while final-epoch Random Forest reaches only **F1 = 0.816, MCC = 0.150, balanced accuracy = 0.540** — essentially no better than trivial on imbalance-robust metrics (MCC = 0, balanced accuracy = 0.5 for the trivial classifier).
- **Diagnosis improves with trajectory features.** Random Forest 7-class diagnosis **Macro-F1 rises from 0.355 to 0.381** and **MCC from 0.346 to 0.401** when switching from final-epoch to trajectory features.
- **Cross-architecture transfer is mostly preserved.** Models trained on two architecture families and tested on the held-out third achieve MCC of **0.210 (FNN), 0.218 (CNN), 0.198 (RNN)** vs. 0.227 in-distribution — a small but nonzero gap, largest for RNN.
- **Single-epoch early detection underperforms full trajectories.** MCC ranges only **0.14–0.19** across epochs k ∈ {1, 2, 5, 10, 15, 20, 25, 30, 40, 50}, all below the 0.227 full-trajectory model, and no single epoch dominates.
- **Per-category diagnosis varies widely.** Weight-related faults are easiest to diagnose; Optimization and Activation are hardest, with Activation frequently confused with Optimization and Optimization confused with Layer.
- **Dataset scale and validation.** 14,227 runs, 719,560 epoch records (median 50 epochs/run), 9,845 faulty vs. 4,382 correct; mutation scores 0.85 (FNN), 0.87 (CNN), 0.83 (RNN).

## Authors & Credibility

The paper builds directly on the authors' prior ICSE 2025 work DEFault (Reference [9]: S. Jahan, M. B. Shah, P. Mahbub, M. M. Rahman, "Improved detection and diagnosis of faults in deep neural networks using hierarchical and explainable classification," ICSE 2025, pp. 2944–2956), where the dataset was first used. This paper documents the dataset's construction and characterization for the first time. The methodology is grounded in the well-established DeepCrime mutation-testing framework (ISSTA 2021) and the statistical mutation-killing criterion of Jahangirova & Tonella (ICST 2020). The artifact is publicly archived on Zenodo with a citable DOI and on GitHub, with clear licensing (CC BY 4.0 / MIT). The choice of MCC over F1 for imbalanced evaluation explicitly follows Chicco & Jurman (BMC Genomics 2020), indicating methodological awareness.

## Critical Assessment

- **Novelty.** Genuine gap-filling: the authors correctly note that no prior public dataset links adapted DNN programs, mutation-derived categories, and per-epoch trajectories in one reusable artifact. DeepCrime provides operators but not reusable labeled trajectories; AutoTrainer/UMLAUT/DeepLocalize/DeepDiagnosis/DeepFD evaluate on study-specific, less reusable data. Novelty is real but incremental — it is a resource paper, not a new method.
- **Rigor.** Strong on data hygiene: group-aware CV (grouped by SO program ID) prevents leakage, statistical mutation killing with Cohen's d and GLM p-values filters equivalent mutants, and 1,000-resample bootstrap CIs are reported. The honesty about the final-epoch F1 (0.816) being no better than the trivial always-faulty baseline (0.818) is commendable and rare.
- **Reproducibility.** High: fixed seeds at framework and operator level, deterministic dataset splits, public Zenodo DOI (10.5281/zenodo.20337241), public GitHub repo, CC BY 4.0 / MIT licensing, and included construction/regeneration scripts. Each run traces to its originating SO post and operator.
- **Significance.** Moderate-to-high for the SE/testing community as infrastructure; the proof-of-concept baselines themselves are modest (RF detection MCC ~0.23, diagnosis Macro-F1 ~0.38), so the value is primarily as a benchmark substrate rather than as a detection breakthrough.
- **Red flags / specific concerns.**
  - **Ecological validity:** All 59 programs come from Stack Overflow and faults are mutation-derived; the authors explicitly disclaim generalization to industrial code or real-fault distributions. Correct baselines were validated for execution/training, not semantic correctness.
  - **Scope:** Only TensorFlow/Keras FNN/CNN/RNN programs; Transformers (the dominant 2024–2026 architecture family) are explicitly out of scope, limiting current relevance.
  - **Weak baselines / missing method work:** Only Logistic Regression and Random Forest with no hyperparameter tuning are evaluated; no neural/diagnosis-specific models (e.g., the DEFault hierarchical classifier from the authors' own ICSE 2025 paper) are run here, so the numbers are floor estimates rather than strong baselines.
  - **Missing ablations:** No per-feature-group ablation (weights vs. gradients vs. activations vs. resource features) to show which of the 26 dynamic features actually drive detection; the cross-architecture RNN gap is attributed to gradient flow over time steps but the mechanism is not isolated.
  - **Hardware features are environment-dependent** (GPU memory is 0 when no GPU is available), which could silently distort cross-platform comparisons.
  - **Single-label diagnosis only:** multilabel diagnosis is not evaluated even though multiple faults could co-occur.

## Strengths

- First reusable public dataset coupling source programs, mutation categories, and per-epoch trajectories for DNN training faults.
- Transparent, statistically grounded mutation validation (Cohen's d + GLM, n=15) with reported per-architecture mutation scores.
- Methodologically sound evaluation: group-aware CV to prevent program-level leakage, plus MCC and bootstrap CIs appropriate for the heavily imbalanced label distribution.
- Honest reporting that final-epoch F1 is no better than the trivial always-faulty baseline, motivating trajectory features.
- Fully open artifact (Zenodo DOI + GitHub) with regeneration scripts, dual licensing, and provenance tracing to SO posts.
- Rich behavioral characterization (Fig. 3 loss-trajectory groupings, confusion matrix in Fig. 4) that points to concrete future work on hard-to-distinguish categories.

## Limitations

- Programs sourced from Stack Overflow may not reflect industrial development practices or real fault distributions.
- Restricted to TensorFlow/Keras and to FNN, CNN, RNN; Transformer architectures are out of scope.
- Correct baselines validated only for execution/training success, not semantic correctness.
- Hardware-related features (CPU%, peak GPU memory, system memory) are environment-dependent and vary across platforms; GPU memory is zero when no GPU is present.
- Proof-of-concept baselines use only LR and RF with no hyperparameter tuning, leaving headroom unmeasured.
- Diagnosis is single-label (seven-class); multilabel diagnosis is not supported.
- Mutation-derived faults are synthetic by construction and may not capture naturally occurring developer faults.

## Relevance

对从事深度学习软件测试、训练期故障检测与诊断、以及变异测试研究的工程团队与研究者而言，Deep4ge 提供了一个可直接复用、可复现的公开基准数据集与注入框架，可作为评估新检测/诊断算法、特征重要性与跨架构迁移能力的标准化实验平台。

## Questions for Further Exploration

- How much of the detection signal comes from each of the four feature groups (weights, gradients, activations, accuracy/loss trends) vs. the resource features — and do the resource features artificially inflate performance on a fixed hardware config?
- Would Transformer-based programs (encoder-only, decoder-only) exhibit the same seven-category separability, or do new fault categories and operators need to be added?
- Can stronger, diagnosis-specific models (e.g., the authors' own hierarchical DEFault classifier, or sequence models over the trajectory) substantially beat the untuned RF/MCC ≈ 0.23 floor?
- Is the RNN cross-architecture gap (MCC 0.198 vs. 0.227 in-distribution) truly caused by gradient flow over time steps, and can time-step-aware features close it?
- Can the Activation↔Optimization and Optimization↔Layer confusions be resolved with more discriminative features, or do these categories share fundamentally similar training dynamics?
- How do mutation-derived fault detection rates compare against a curated set of real, naturally occurring DNN training faults from industrial codebases?
