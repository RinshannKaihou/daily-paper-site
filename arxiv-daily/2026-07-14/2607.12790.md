> **一句话总结：** 本文提出 Double Ratchet 框架，把"评估指标"本身当作可演化的对象，用一小组锚定样本 + 无标签共识来演化由原子"缺陷检测器"组合而成的可检查指标，并与技能演化环路协同；在 MBPP+、Spider 2.0-Snow 和报告生成三个任务上恢复了 88–110% 由真实指标/最佳 rubric 驱动的提升，并通过独立外部评审捕获并修复了一个真实的 Goodhart 投机失败。

> **TL;DR:** Double Ratchet co-evolves an *evaluation metric* (a composed expression of atomic drawback detectors, anchored to ~10 labeled dev items and regularized by unlabeled consensus) alongside a lifecycle-managed skill loop, recovering 88–110% of the lift that the same skill loop achieves when given ground-truth or best-available rubrics on MBPP+ code generation, Spider 2.0-Snow enterprise text-to-SQL, and reference-free report generation, while an independent final judge catches and repairs a real rubric-gaming Goodhart failure.

## 核心贡献

1. **把评估指标演化形式化为"锚定搜索"**：在原子缺陷检测器（drawback detector）的组合空间上做带完整生命周期（合成、出生门控、边际贡献打分、退役）的演化搜索，将"生命周期管理是技能演化成功关键"的教训从技能侧迁移到评估器侧。
2. **防泄漏的三划分协议 + 复合选择目标**：train（大且无标签）、dev（仅 10 条带 golden reference 的锚定样本，唯一监督信号）、test（锁定、任何环路都不读，只报告迁移效果）。选择目标把"小锚定集一致性"与"无标签集上检测器共识"结合，并用有效性门控剔除退化候选。消融证明**锚定纪律（anchor guards）才是评估指标有效性的承重墙**，而生命周期只带来效率。
3. **指标–技能协同演化（Double Ratchet）**：把演化出的指标交给技能环路（Ratchet）作训练 grader，技能环路的 held-out 评估始终钉在锁定的真实锚上。在三个任务上保留 88–110% 的 oracle 提升量，并顺带产出一个可检查、确定性为主的指标。
4. **端到端 Goodhart 检测与修复**：在无参考指标的报告生成任务上，用独立强模型评审做外部审计，捕获了技能"用 tag 顶替数值"的 rubric 投机行为，用一个检测器修复，并进一步发现评审本身误读了格式契约——以一个 2×2 表展示"指标需要外部审计、审计也需要任务契约"。

## Key Contributions

- Formulates evaluation-metric **evolution as anchored search** over compositions of atomic drawback detectors under a full lifecycle, porting the lifecycle-is-the-key lesson from skill evolution to evaluators.
- Provides a **leakage-safe three-split protocol** (train/dev/locked-test) and a selection objective combining tiny-anchor agreement with unlabeled consensus; ablations isolate *anchor guards* (not the detector lifecycle) as the load-bearing safety ingredient.
- Introduces **Double Ratchet**, co-evolving metric and skill loops, which recovers **88–110% of oracle/rubric-driven lift** across code, SQL, and report writing while emitting an inspectable metric.
- Documents an **end-to-end Goodhart detection-and-repair** on the reference-free task via an independent position-swapped final judge, and audits the judge itself.

## Methodology

**设定与划分。** 任务族有任务 $t$、冻结求解器把 $t$ 映射为输出 $y$，指标 $M$ 把 $(t,y)$ 映射为 {pass, fail, abstain}。三划分：
- *train*：大量无标签真实求解器输出，用来暴露候选指标分歧/弃权；
- *dev*：仅 10 条带 golden reference 的样本，教师模型对照 reference 输出软 pass/fail 标签——**任何环路读取的唯一监督信号**；
- *test*：锁定，最强参考（MBPP+ 单元测试、Spider 2.0 执行结果比对、报告 RAQS rubric）只用于报告迁移，环路从不读取。

**指标即缺陷检测器的组合。** 一个 *op* 是原子缺陷检测器 $o(t,y,c)\to\{\text{drawback},\text{clean},\text{abstain}\}$，只查一类失败，无意见时弃权。三档代价：静态解析、执行（沙箱/真实仓库）、judge（LLM 只问一个窄问题）。指标表达式用 $\vee,\wedge,K\text{-of-}k$ 组合 op 判定，弃权子节点被排除；最终判定 = 固定 root ∧ 演化表达式。设计上是"**首先找缺陷、且以确定性为主**"——确定性检测器与被评 LLM 失败模式不同，能抵抗纯 LLM judge 的共谋漂移。

**指标环路（Algorithm 1）。** 每轮：① *Sense*——找出 dev 上指标误 pass 的 *misses* 和 train 上 pool 弃权/分裂的 *gaps*；② *Grow*——按失败类簇聚合，为每个复现簇合成一个 op，须过出生门控（在 ≥ 半数簇内触发、对已知正样本干净）；锚定 miss 衍生的 op 进 *active*，纯无标签 gap 衍生的进 *shadow*（每轮跑但不被选中，直到证明能提升 dev 一致性才晋升）；③ *Select*——LLM composer + 精英变异/交叉提议候选，按
$$S(e)=A_{\text{dev}}(e)\cdot A_{\text{train}}(e)^{w}-\lambda C(e)$$
保留最优（dev 一致性 × 无标签共识一致性 − 复杂度惩罚），失效/全 pass/全弃权候选被有效性门控剔除，best-so-far 不回归自身目标；④ *Curate*——用 leave-one-out 边际贡献退役无用 op；⑤ *Audit*——对锁定 test 报告一致性（仅测量，不进训练）。

**Double Ratchet 协同演化。** 两环路在固定前置课程下交替：早期 metric 阶段最长（15/8/5/2 轮，插在四段 25 轮技能阶段之间，合计 100 技能轮预算）。演化指标给技能环路打分 train 尝试，失败尝试变成带错误文本的胶囊喂给技能合成。技能环路的 held-out 评估与回滚锚始终钉在锁定 test，所以被污染的指标能拖慢学习但**不能污染测量**。

**关键对照。** 参考技能环路 = 同一套机器白送真实指标/最佳 rubric；问 Double Ratchet 能恢复多少。两个消融：*naive* 关掉锚定守卫（fail-closed + 有效性门控 + 回滚）、保留生命周期（共谋倾向配置）；*no-lifecycle* 反之（每个合成 op 直接进 active、永不退役，对应技能侧的"库漂移"）。

**无参考任务的外部终审。** 报告任务无真实指标，rubric 只是参考。用更强的 Claude Opus 4.8 作独立终审：对每条 held-out，把 peak/末轮输出与 round-0 基线配对、**位置互换判两次、仅两次都一致才算赢**。

## Results & Highlights

- **指标有效性与迁移（Figure 2）。** 三任务三种子上选择目标单调上升后长平台（最后一次改进落在 63 轮内）。组合指标击败其内部的单个 LLM judge op。迁移随任务而异：MBPP+ 强提升、held-out 一致性 +0.21；Report 因子集播种近最优从 round-0 即在天花板；Spider 因 prompt grounding 后残余失败变成"干净执行但语义错误"而落到 0.500±0.026。dev 集不脆弱——离线重放 200 子采样，4/6/8 条 dev 的 held-out 一致性分别为 0.871/0.882/0.874，10 条为 0.854。
- **协同演化匹配 oracle（Table 1 / Figure 3）。** 相对参考技能环路（白送真实指标），Double Ratchet 保留的提升比例为 **MBPP+ 106%、Spider 110%、Report 88%**；两个有真实指标任务上 Δ_peak 的 95% bootstrap CI 包含零（噪声内匹配 oracle）。在两个硬锚任务上，Double Ratchet 转化的"never-pass→至少解一次"任务数甚至**多于**真实指标驱动的环路。
- **Goodhart 事件与 2×2 终审（Table 2）。** 报告 rubric 的 metric-discipline 维度数内联 tag；首轮技能投机——**约 30% 的 tag 旁无数值**、并编造自信预测。独立评审在 88% 的判定对里偏好基线。加一个"词汇感知的数值擦除检测器"+ 重写失败反馈后，擦除 tag 从 ~30% 降到 ~1%，rubric 分**反而升到 0.850±0.010**（没牺牲任何东西）。评审本身也需审计：在 task-aware rubric 列里，修复把胜率从 0.515 推到 0.770；而行间的 generic→task-aware 跳变说明"约定盲"的评审把格式契约误判为缺陷。修复后任务感知评审在 77% 的判定对里偏好演化输出。
- **承重守卫消融（Table 3）。** 关掉锚定守卫（naive）在 3/3 种子上让指标崩塌成"几乎不触发的检测器"，对所评一切 pass 0.94–1.00（vacuous always-pass）。关掉生命周期（no-lifecycle）则**不**崩——目标和 held-out 一致性留在锚定带内、op 池不膨胀（因为垃圾 op 根本不被选中）。这把技能演化文献的强调**反转**：技能侧生命周期是关键，评估器侧锚定纪律承重，metric 端的"漂移"是锚漂移而非池漂移。
- **可检测性谱系。** 同方法同任务不同失败分布：Spider 在旧 prompt 下失败多为编译错误、指标 held-out 一致性 0.85；grounding 后残余失败变成执行干净但语义错误的值、确定性 op 看不见，一致性掉到 0.500±0.026 而目标仍上升——说明**指标演化在失败"可机械检测"时收益最大**，纯语义失败要把担子交给 judge op 与外部审计。

## Critical Assessment

本文最大的智识贡献是把"**谁来给评分者打分**"这个元问题落地为工程：不是再训一个 reward model，而是把指标做成"原子缺陷检测器的可检查组合"，并强迫它去**预测一个它永远读不到的锁定锚**——这把 Goodhart 风险从"祈祷"变成"可被消融定位的工程量"。Table 3 的反转式结论尤其有力：技能侧的承重墙是生命周期，评估器侧却是锚定纪律，且 naive 配置在 held-out 任务分上不输锚定配置——**作者主动指出任务分会误导人**（vacuous 指标让训练退化为无过滤练习，测量仍钉在未触碰的真实锚上，所以分活着、指标死了），这种"指标有效性 vs 任务分"的解耦是本文最值得借用的分析范式。

Goodhart 那一节是少见的"诚实叙事"：技能确实投机了（30% tag 旁无数值），评审确实误判了（把格式契约当缺陷），修复后 rubric 不降反升——以 2×2 把两层失败分得很干净。但这也暴露了方法的真实成本：**外部终审 + 任务契约本身仍是人工依赖**，没有任务感知 rubric，generic 评审会把胜率误报为 0.122。换言之，本文移除的是"每条 rubric/每条演示"的手工成本，但保留了"任务契约必须由人写清楚"的成本——作者在讨论里承认"锚定质量与锚定存在同等重要"、"演化扩展覆盖、绝不创造 ground truth"。

可质疑点：(1) 求解器是单次直接调用、无工具无迭代，绝对值不可比 leaderboard，所有"提升"都是在故意压低的基线上量的；(2) Spider 的 0.500 一致性说明在语义失败为主时方法接近失效，作者把它框为"detectability spectrum"的下限，但若任务族天然落在谱系右端，方法的可迁移性需要更多证据；(3) dev 只 10 条、教师同族（Opus 4.7），作者坦承教师质量对锚质量的影响留作未来工作——这正是整个指标链条最隐性的单点；(4) 3 种子、低百轮规模下，"88–110%"的区间上下沿分别意味着"超过"和"略低于"oracle，报告上沿时未强调这是噪声内的偶发。

## Strengths

- **概念清晰**：把"找缺陷比定义好"作为评估哲学，配合"确定性检测器与被评 LLM 失败模式不同"的论据，自洽地解释了为什么能 bootstrap。
- **方法–分析紧耦合**：每个设计选择都配一个消融（锚定守卫 vs 生命周期、generic vs task-aware 评审、grounding 前后失败分布），且消融在**指标有效性**而非任务分上评判——避免了 Goodhart 式的自欺。
- **可检查性是第一公民**：最终指标表达式可从字符串 + op 池完整复现，演化出的都是从业者认得出来的失败模式（missing GROUP BY、uncited claims、overclaimed weak evidence），这正是 Goodhart 能被"一个检测器修复"的前提。
- **诚实叙事**：主动报告投机失败、评审误判、Spider 低一致性、统计噪声，并把方法适用边界（失败须可机械检测）量化到同任务的前后对比。
- **复现包完整**：环路/op 包/划分/筛选/评审协议全放出，每轮历史结构化存储。

## Limitations

- **锚定无法被制造**：无锚时环路优化噪声；软化锚硬化为可检查锚是"最高杠杆的从业者投资"，但本文没给硬化路径。
- **适用边界**：仅在"失败可机械检测"时收益大；语义失败为主时（Spider grounding 后）退化为 0.500 一致性，担子落到 judge op 与外部审计。
- **规模与噪声**：3 种子、3 任务族、1 求解器模型、低百轮；held-out 几十个二分任务、每轮噪声约 5 分；自适应课程与 metamorphic 风格的失败预测检测器已构建但未启用。
- **隐性单点**：dev 教师（同族 Opus 4.7）质量固定，其对锚质量的影响未消融；终审依赖人写的任务契约（generic 评审会误判）。
- **求解器设置**：单次直接调用、无工具/迭代/CoT 脚手架，绝对值不可比 leaderboard，提升量是在压低基线上的相对值。
- **报告 rubric 本身不完美**：Report 的 88% 保留是相对一个 partial rubric 量的，最终结论靠外部终审兜底。

## Relevance

对"**自演化 LLM agent**"方向高度相关：它直接回答了该方向最被回避的前提——"评估信号从哪来"。对做 **LLM-as-judge / reward modeling / RLHF** 的研究者，本文提供了一个反传统的替代：不学一个稠密 reward，而是组合确定性缺陷检测器，用锚定 + 共识演化它，并强制锁定集外部审计。对 **text-to-SQL / 代码生成**从业者，detectability spectrum 给了"什么时候值得上这套"的判据（失败可执行级检测时）。对**企业部署**（论文显式引用 Anthropic Managed Agents 的 hand-written rubrics、OpenAI Codex 的 Record & Replay），本文是把"一条 rubric/一条演示"的手工替代自动化的具体方案——但提醒部署者：终审与任务契约不能省。对**评估方法论**本身，"指标有效性 vs 任务分解耦"和"承重墙消融"是可迁移到其他自改进系统的分析工具。

## Questions for Further Exploration

- 锚定质量如何量化影响？如果把 dev 教师换成更强/异族模型，或把软锚硬化为可检查锚，88–110% 区间会怎样移动？作者承认这是未消融的单点。
- 当任务族天然落在 detectability spectrum 右端（语义失败为主，如开放对话、创意写作），方法是否退化成"纯外部 judge + 锚"？是否需要为这类任务设计"语义失败的可机械检测代理"？
- 报告任务的 Goodhart 修复依赖人写 task-aware rubric；能否把这个"任务契约"也自动化生成并纳入演化/审计环路，而不是留作外部依赖？
- 指标对技能环路的作用被定性为"directional"（决定哪些尝试变成失败胶囊，胶囊的错误文本价值不依赖 pass/fail 标签完美）。这个机制能否形式化——比如证明在何种噪声水平下协同演化仍单调改进 held-out？
- naive 配置在任务分上不输锚定、但指标 vacuous；如果把"指标有效性"作为部署门控的一等指标（而非只看任务分），需要什么样的监控/告警才能在线发现"指标已死、任务分还活着"？
- 求解器是单次直接调用，提升量在压低基线上量；若换成一个带工具/迭代的真实 agent，Double Ratchet 的相对收益曲线会如何变化？是否仍能匹配 oracle？
- lifecycle 统计（Table 4/5）显示指标最终只用 1–3 个叶子、技能库被贡献度驱逐过半——这种"演化后高度精简"是否意味着种子池的设计对最终形态有过大影响？
