Code-MUE: Measuring Code LLMs' Uncertainty through Execution-based Semantic Interaction Graphs

2Why This Matters

A practical black-box uncertainty signal for closed-source code LLMs — enabling selective prediction and oracle-free Best-of-N reranking in real pipelines.

3Critical Edge

Cannot tell confident-correct from confident-wrong: systematic biases yield low-entropy incorrect solutions indistinguishable from genuine consensus.

01 / METHOD

Three black-box stages — no logits, no oracles

01

Sample & synthesize probes

Query the LLM N=100 times (τ=1.0). Build a hybrid K=20 input-only probe set: type-boundary corner values + LLM-synthesized nominal inputs.

No expected outputs ever generated → bypasses the Oracle Problem.

›
02

Build the Semantic Interaction Graph

Sandbox-execute each program on all probes → Execution Signature Vector (return value / exception type / timeout). Pairwise consensus matrix W encodes functional agreement.

W_ij = (1/K)·Σ 𝟙[o_i,k ≡ o_j,k] ›
03

Quantify global uncertainty

Trace-normalize into density matrix ρ=W/N, eigen-decompose, take normalized Von Neumann entropy. Captures consensus vs. competing clusters vs. fragmentation — invisible to mean agreement.

U(P) = H_spec / ln(N) ∈ [0,1]

03 / ASSESSMENT

Strengths & limitations, side by side

+Strengths

  • Principled black-box. Needs no logits or internal states — works with GPT-4.1, Claude, Gemini APIs, the realistic setting.
  • Oracle-free probing. Generates inputs only, never expected outputs — sidesteps the test-oracle problem.
  • Topology-aware. Von Neumann entropy reads global structure (consensus / competing clusters / fragmentation) that scalar mean agreement cannot.
  • Dual utility. Eigenvector centrality turns a measurement metric into a decision tool for unsupervised response selection.
  • Honest failures. RQ3 transparently characterizes "Overconfident Error" patterns rather than hiding outliers.

!Limitations

  • Confident-but-wrong blind spot. Corpus/prompt biases produce low-entropy incorrect solutions indistinguishable from correct consensus.
  • Timeout asymmetry. Treating distinct timeouts as zero agreement inflates uncertainty for slow-but-correct code (e.g. HumanEval/83 brute force).
  • Input coverage bias. If synthesized inputs miss a semantic region, distinct modes wrongly merge.
  • Cost. ~500K inferences + O(N·K) sandbox runs; dominant N cost is hard to compress without quality loss.
  • Python, function-level only. Repo-level shown only via manual AST decomposition; no multi-language execution semantics.

05 / OPEN

Questions for further exploration

Can confident-but-wrong errors be detected without sacrificing the black-box property — via LLM-as-Judge, metamorphic relations, or temperature-sensitivity analysis?
Would aggressive input generation (fuzzing / symbolic execution / differential testing) further compress coverage bias?
Can the density-matrix framework support weighted / soft semantic modes instead of hard equivalence clustering?
How does Von Neumann entropy compare to other graph-spectral measures — Laplacian gap, modularity, Cheeger constant — on specific failure modes?
Does the framework scale to non-Python languages and repository-level generation beyond translation tasks?
Does the "semantic consensus" hypothesis hold when many legitimately distinct correct solutions exist (e.g. open-ended API design)?
Can the uncertainty signal guide test-time scaling or self-consistency sampling, rather than only filtering a posteriori?
