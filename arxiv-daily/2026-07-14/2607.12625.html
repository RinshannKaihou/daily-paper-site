GUI Agent Memory & Skill Reuse Mobile Agent Cross-Platform Personal Assistant

Key Finding

Decoupling host orchestration from GUI execution — with self-evolving memory and skills — sets open-source SOTA on mobile agents.

Know Deeply, Act Perfectly: Personal GUI Assistant with Self-Evolving Memory and Skill

Li, Li, Hu, Zhang, Su, Xu, Zhao, Bian, Liang, Zhang · HIT-Shenzhen / Lychee Team · arxiv.org/abs/2607.12625
01 Key Metric
64.1% SR

MobileWorld GUI-Only, pass@1, 117 tasks — beating Seed-2.0-Pro 63.2% & GPT-5.5 62.4% with open-source Kimi-K2.6.

02 Why This Matters

Engineering-fuses config-driven personal assistants with visual GUI agents — fixing the two gaps in OpenClaw-style frameworks: no GUI action, and no self-evolution.

03 Critical Edge

Cross-platform claims rest on small, non-standardized sets; failures in wheel pickers, toast clicks, and emoji mapping undercut the "Act Perfectly" promise.

M / 01

Know → Route → Act → Reflect

a four-stage closed loop, two persistent stores
01 · Know

Active Retrieval

Semantic recall of prior GUI memories + candidate skills, kept strictly advisory; policy memory injected; fuzzy instructions surfaced as explicit overridable defaults.

02 · Route

Typed Blackboard

Emits single tasks or multi-app workflows of tuples (g, h, I, O). Missing declared input or output ⇒ fail closed — never fabricate.

03 · Act

Hybrid Action Space

A = GUI ∪ skill ∪ deeplink/intent ∪ ask. Top-5 skills retrieved; per-step state contract validated; bounded recovery on mismatch.

04 · Reflect

Self-Evolution

Distills trajectories into advisory memory vs. state-validated skills; repair before re-extract; per-app de-duplication.

  • Two-tier host–executor split. Host owns conversation, workspace, profile, tools; a lightweight GUI executor owns screenshots, action normalization, device backends & skill validation.
  • "Fail closed" is the headline contribution. Converts the monolithic-agent failure mode (free-text summaries losing values across apps) into a mechanically checkable typed contract.
  • Adaptive host intervention. Host answers eligible info-query subtasks directly via context or non-GUI tools; only genuine device/visual needs get dispatched as GUI tasks.
  • Memory ≠ skill. Textual experience rationales advise; skills are parameterized, state-contracted, executable procedures. Cross-model transfer shown Kimi→Qwen.
POMDP M = (S, A, O, T, Ω, R, γ) Hidden device state — nav stacks, login/permission, async loads, off-screen fields — is never directly observed.

DEFAULT PAIR Qwen3.5-397B-A17B host + Qwen3.5-35B-A3B executor; Kimi-K2.6 serves as both. MobileWorld GUI-Only: 117 tasks, 50-step cap, pass@1.

A / 03

Strengths & Limitations

side by side

✓ Strengths

  • Clean conceptual split of cognitive orchestration vs. visual execution, with efficiency as a first-class goal.
  • Typed blackboard + "fail closed" — a concrete, mechanically checkable fix for cross-app value loss.
  • Unified skill abstraction spanning click-type patterns, multi-step procedures, and validated deeplink/intent shortcuts under one pipeline.
  • Reproducible main result on a standard benchmark, with both pass@1 and pass@3 plus independently logged cost columns.
  • Honest failure analysis (wheel picker, premature termination, Windows grounding) rather than cherry-picked wins.

! Limitations

  • Non-standardized cross-platform eval — manually designed/translated sets, small denominators, qwen3.5-flash judge.
  • Skill brittleness to app updates — package names, coordinates, UI drift; decay over versions is unmeasured.
  • Transfer shown one-way only (Kimi→Qwen); reverse and universal transfer unverified.
  • Sequential pipeline overhead — host–subagent handoff cost (visible 65k host tokens); a unified joint planner is future work.
  • Weak on continuous geometric control, deep app controls, off-viewport & temporal targets (toast clicking).
Q / 04

Questions for Further Exploration

open threads

Version robustness of skills

When app updates drift package names, coordinates, or UI structure, how often do state contracts fail? Is an auto-invalidation + re-validation pipeline needed?

Reverse transfer

Can memory/skills distilled from the weak model (Qwen-35B) backwards improve the strong model (Kimi-K2.6)? Only Kimi→Qwen is verified.

Unified joint planner

Folding knowledge retrieval, non-visual tools, and GUI actions into one decision cycle — how much SR/token gain over the sequential pipeline, and new scheduling ambiguity?

Geometric & temporal grounding

Can wheel pickers, sliders, and toast clicks be eased by modeling geometric/temporal actions separately instead of the unified GUI space?

Cross-platform action sharing

Mobile/desktop share core actions, but how are deeplink/intent vs. Windows equivalents aligned — can one skill description deploy cross-platform?

Memory trust & decay

Advisory experience memory risks "stale lessons" (post-update failures misrouting to fixed paths). Is a confidence/forgetting mechanism needed?
