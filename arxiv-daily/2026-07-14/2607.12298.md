> **一句话总结：** 本文提出 EIR，一种面向 FPGA SoC 的层次化数字孪生自愈框架，利用 PS 端空闲周期运行"兔(Rabbit,粗粒度快速检测)"与"龟(Tortoise,门级精确恢复)"两个孪生模型，在不复制 PL 逻辑的前提下实现瞬态故障的自主检测与就地恢复，在 Zynq-7000 上对 11 个加速器实现 ≥95% 故障检出率，同时保持原始 PL 面积，而 DMR 平均把 LUT/FF 占用提升约 2.0–2.25×、功耗从 2.98 W 升至 5.79 W（+89.1%）。

> **TL;DR:** EIR is a hierarchical digital-twin self-healing framework for FPGA SoCs that runs a coarse-grained "Rabbit" behavioral twin (fast fault detection) and a fine-grained "Tortoise" gate-level twin (precise recovery) on the otherwise-idle Processing System (PS), avoiding PL replication; on an AMD/Xilinx Zynq-7000 across 11 accelerators it reaches ≥95% fault detection with only the top-30% neurons audited while preserving baseline PL area, versus DMR which inflates LUT/FF ~2.0–2.25× and raises power from 2.98 W to 5.79 W (+89.1%).

## 核心贡献

1. **层次化数字孪生架构（首创）**：针对 FPGA SoC 加速器提出首个"双粒度"数字孪生框架——快但粗的 Rabbit（C/C++ 行为级模型，连续监测、快速检出输出偏离）与慢但精的 Tortoise（基于 Yosys CXXRTL 的门级 C++ 模型，仅在故障时被唤醒，从校验点做周期精确重放以重建正确状态）。二者协作：Rabbit 负责"快"，Tortoise 负责"准"，借此兼顾检测延迟与恢复精度。
2. **软硬件协同的校验点机制**：复用 EPOCH 的 PCAP 状态读/写原语，配合 `*.ll`（logic-location）映射文件，只回读 PL 实际使用的逻辑单元到片外 DDR，零 PL 面积开销地完成状态快照；并提出"执行速度感知"的校验点选择策略，在检测延迟与存储开销间权衡。
3. **边端 AI 的自愈方法论**：端到端系统在不修改加速器 RTL、不在可重构 fabric 里复制逻辑的前提下，实现就地状态恢复；定位错误后用 PCAP 把 Tortoise 计算出的"无故障态"写回 PL，从中断点继续执行而非整体重跑。
4. **实验验证**：在 LeNet-5/CIFAR-10 案例与 11 个代表性加速器上证明，仅审计 dense/logit 层中贡献幅值最高的 top-30% MAC 即可捕获所有观测到的误分类（折合仅约 5% 全网 MAC），对应 ≥95% 故障检出率，并量化了对 DMR 基线的面积/能耗优势。

## Key Contributions

- **Hierarchical digital-twin architecture** — the first dual-granularity twin framework for FPGA SoC accelerators, pairing a fast coarse-grained functional twin (Rabbit) for detection with a slow fine-grained gate-level twin (Tortoise) for recovery.
- **HW/SW checkpoint integration** — a practical on-chip checkpoint mechanism built on PCAP readback and logic-location mapping (no PL logic reserved), with execution-speed-informed checkpoint placement.
- **Self-healing methodology for edge AI** — autonomous in-place fault recovery without RTL changes or fabric-level duplication, positioned as an alternative to DMR/TMR.
- **Experimental validation** on AMD/Xilinx Zynq-7000 (XC7Z020) across 11 accelerators (up-counter, SHA-256, AES-128, dense layer, Conv2D, attention, multi-head attention, GEMM, FFT-64, FIR/Conv1D, IIR), demonstrating ≥95% detection at top-30% neuron coverage and quantified area/power savings over DMR.

## Methodology

**系统结构（Fig. 2）**：加速器驻留 PL；AXI-GPIO 监测其输出并在异常时置位；PS 承载校验点策略与两个孪生。状态经 PCAP 读/写，借助 SD 卡上的 `*.ll` 映射只回读被使用的逻辑区到 DDR。

**两孪生分工**：
- *Rabbit (Twin #1)*：高抽象级行为模型（C/C++），不建模单时钟周期，只在校验点处重算同一批输入并与 PL 输出比较 `|O_Rabbit − O_PL| < ε`；连续运行会霸占 PS、能耗过高，因此 EIR 用"延迟感知"策略只在关键阶段唤起。
- *Tortoise (Twin #2)*：用 Yosys CXXRTL 把与 PL bitstream 同源的 RTL 转成周期精确的 C++ 门级模型，平时休眠；Rabbit 报错后，加载最近校验点（含全部寄存器、控制信号、从 t_capture 到 t_fault 的输入序列），逐周期前推，产生无故障输出 O_Tortoise，再经 PCAP 写回 PL。

**校验点选择（Sec. III-B）**：两阶段分析。(i) 文献驱动的层敏感度：早期卷积层多为掩蔽，深层 dense/logit 层最易改变决策；高幅值 MAC、定点高位（MSB）主导故障影响。(ii) 模型专属案例：LeNet-5 on CIFAR-10，按 `|x·w|` 对 dense 层 MAC 排序分位，单/多比特注入。结论：dense6 顶 30%、dense7 顶 ~17%、dense8 顶 ~18% 即覆盖全部观测误分类；聚合后约 25% dense MAC（≈5% 全网 MAC）即可。

**实时预算推导（dense 层）**：PL 单层 ≈43.54 μs（≈10.62 ns/MAC），PS 每 MAC ≈0.14 μs，故 PS 在一层执行窗口内最多可在线校验 `C_MAC ≈ 311` MAC（约 7.6% 的 4096）。若全量校验会 stall PL ≈0.53 ms；改成 top-30%（1228 个）只需 ≈0.17 ms，PL 隔离窗口从 530 μs 降到 170 μs（≈3× 改善）。

**故障注入（Sec. IV-A）**：采用 readback-level 注入——在 PCAP 回读快照后、与 Rabbit 比较前，对选定比特位 XOR 1 以模拟瞬态翻转；这是"方法学便利"而非 EIR 的限制，真实在线故障会被同一套 checkpoint-and-compare 流水线捕获。

**安全恢复（Sec. V-D, V-E）**：仅在 PS–PL 握手进入静默态、最慢时钟驱动的安全同步点处暂停时钟，避免"撕裂态"；注入更新仅落在由离线 `*.ll` 映射确定的目标配置帧/比特位，不动无关 LUT/布线。映射与最终布局布线绑定，RTL/工具版本变化需重新生成。

## Results & Highlights

- **故障覆盖率**：对 dense 层做百分位扫描，仅审计 top-30% 神经元即提供"充分"覆盖，对应 **≥95% 故障检出率**；dense6 顶 30% / dense7 顶 ~17% / dense8 顶 ~18% 已捕获所有观测到的误分类；聚合后**仅约 25% dense 层 MAC（≈5% 全网 MAC）** 即覆盖全部观测误分类。
- **面积/能耗对比 DMR**：跨 11 个加速器，EIR **保持基线 PL 面积**；DMR 把 LUT/FF 利用率提高约 **2.0–2.25×**，并把功耗从 **2.98 W 提到 5.79 W（≈+89.1%，均值）**。
- **延迟剖切（Table II + 正文）**：捕获 t_capture = 31.46 ms，状态校验 ε = 39.3 ms，定位 t_fault = 31.91 ms；**实际状态写回 t_restore 仅 5.7 ms**（经 PCAP）。这些 30–40 ms 级阶段仅在罕见故障时触发，不影响稳态吞吐。
- **PL 隔离窗口优化**：top-30% 选择把在线校验窗口从全量的 ~530 μs 压到 ~170 μs（≈3× 改善），且恰好覆盖最易影响推理精度的计算。
- **孪生速度比 S = Twin/PL**：Rabbit 对多数设计慢约 0.5×–25×（Conv2D 处 Rabbit 反而比 PL 快，S≈0.5×）；Tortoise 则慢 154×–9882×，印证其只能做异常路径用途。
- **基准多样性**：覆盖控制密集加密核（SHA-256、AES-128）、信号处理/ML 核（dense、Conv2D、attention、multi-head attention、GEMM、FFT-64、FIR/Conv1D、IIR）及 8-bit up-counter sanity check。

## Critical Assessment

EIR 的核心赌注是 **PS 在加速器运行期间有可观的空闲周期**，且 **PS 与 PL 间的速度差可被"选择性校验"吸收**。论文用 dense 层的实时预算推导（C_MAC ≈ 311，仅 7.6%）诚实地量化了这一边界——这意味着一旦 PS 被其他任务占用、或加速器变得更"瘦"（每层 PL 时间更短），可在线校验的比例会进一步收缩，检出延迟随之上升。作者承认这一点（Sec. V-A 假设 i，Sec. III-A 末段），但未给出 PS 负载扫描曲线，读者无法判断滑出舒适区有多陡。

第二个真正紧的约束是 **Tortoise 的慢**：dense 层 Tortoise 比 PL 慢 154×，IIR 慢 ~9882×，SHA-256 慢 ~6266×。恢复虽只在异常路径触发，但 worst-case 恢复延迟 = 校验点间距 Δ × S_Tortoise；论文给了 t_restore = 5.7 ms（状态写回）和 t_fault = 31.91 ms（定位），却未把"大 Δ + 复杂加速器"下的 Tortoise 重放耗时明确列入端到端恢复预算，仅以"bounded by checkpoint spacing"概括。

第三，**覆盖率声明是经验性、工作负载特定的**。top-30% 阈值来自 LeNet-5/CIFAR-10 的 dense 层多比特瞬态注入，作者反复强调"不应视为普适值"——对动态控制流、Transformer 全层、或非 ML 加速器（SHA/AES 的故障覆盖率分析被以"space constraints"省略）尚需重做 profiling。论文未提供端到端形式化覆盖保证，Tortoise 的正确性也依赖确定性重放假设（无异步软件干预、无外部不可控输入）。

第四，**`*.ll` 映射与最终 bitstream 强绑定**：任何 RTL/综合/布线/工具版本变更都需重新生成映射并重新验证，这在 CI/CD 式频繁改版的环境中是隐性运维成本。

## Strengths

- 概念优雅：用寓言式"兔与龟"双孪生把"快速检测"与"精确恢复"解耦，避免 DMR/TMR 的连续硬件冗余，思路新颖且叙事清晰。
- 零 PL 面积开销的状态捕获：复用 PCAP + `*.ll` 只回读活跃区，不预留 fabric 逻辑（区别于 StateMover/Zoomie），对资源受限的边端 SoC 尤其友好。
- 诚实的实时性算术：给出 PL/PS 每 MAC 延迟、可在线校验 MAC 数、隔离窗口的逐项推导，而非仅甩一个"低开销"口号。
- 广覆盖基准：11 个加速器横跨加密、DSP、ML，配合 LeNet-5 敏感度案例，论证了方法的可移植性。
- 安全恢复细节扎实：握手式安全同步点、只改目标配置帧、避免撕裂态与协议违例，工程上可落地。

## Limitations

- **PS 空闲假设**：当 PS 被高优先级软件任务占满时，可在线校验比例下降，检出延迟上升（作者承认但未量化）。
- **Tortoise 速度惩罚**：门级孪生对 PL 慢 2–3 个数量级，恢复路径只能作为"罕见异常通道"；对复杂/深流水加速器，重放成本随 Δ 增长，论文未给出 Δ 的系统级调优实验。
- **覆盖率是经验性的**：top-30% 来自单一 LeNet-5/dense 层/多比特瞬态模型；动态工作负载、Transformer、加密核的等价覆盖率分析缺失或被省略。
- **确定性重放前提**：恢复正确性依赖无未追踪的非确定性（异步软件、不可控外部输入流、窗口外时序副作用）。
- **映射绑定 bitstream**：`*.ll` 映射随布局布线/工具版本失效，需重新生成，增加维护成本。
- **故障模型局限**：仅针对瞬态（电压波动、EMI、辐射单事件翻转）；永久故障、BRAM 故障（已由 ECC 覆盖，被显式排除在范围外）不在处理之列。
- **无端到端形式化保证**：作者明确将"更强形式化覆盖保证"列为开放问题。

## Relevance

本文对接多条与本仓库研究方向相关的脉络：(1) **边端 AI 可靠性**——CNN/Transformer 在 FPGA SoC 上的容错是自动驾驶、工业 IoT、航宇电子的真实痛点，EIR 给出了一条区别于 DMR/TMR 的轻量路径；(2) **硬件安全与故障注入**——与该组此前的 CRAFT（pre-silicon 故障注入威胁表征）、THEMIS（多租户 FPGA 调度）形成"攻击-防御-调度"闭环；(3) **数字孪生下沉到芯片级**——把制造/能源域的数字孪生理念压到纳秒级、资源受限的 FPGA，是 twin 概念的工程化延伸；(4) **与校验点/抢占机制（EPOCH）的复用**——展示了"为抢占设计的基础设施"如何被改造成"为容错服务"，对研究基础设施复用有方法论参考价值。对做 FPGA 加速器、硬件容错、或可信边端 AI 的研究者，EIR 的"选择性校验 + 异常路径精确恢复"范式值得纳入对比基线。

## Questions for Further Exploration

1. 当 PS 持续高负载（如同时跑 RTOS + 多任务）时，Rabbit 的可校验比例与检出延迟如何衰减？能否给出一条 PS 利用率 vs. 检出延迟的曲线来界定 EIR 的"舒适区"？
2. Tortoise 的恢复延迟在最坏 Δ 下（例如把 Δ 拉大以减少稳态校验点开销）是否仍能满足边端 AI 的控制环 deadline？Δ 的最优选择是否存在解析或自适应策略？
3. top-30% 阈值在 Transformer（attention/全层）、更深的 CNN、或对抗性权重扰动下是否仍然成立？是否需要一个与模型无关的自适应选择算法来替代离线 profiling？
4. EIR 能否扩展到 **永久故障**（如老化、电迁移）？这需要 Tortoise 不仅重放状态，还要参与缺陷重映射，与 scrubbing/部分重配置如何融合？
5. 面对非确定性源（异步中断、外部流式输入），能否引入确定性重放契约或硬件辅助的执行日志，来拓宽 EIR 的正确恢复前提？
6. EIR 的孪生思路能否反向用于 **安全监测**（如检测恶意 bitstream 篡改或木马激活）而非仅容错？Rabbit 作为"参考实现"本身就是一个可信锚点。
7. 与 DMR 之外的中间方案（HMR-NEureka、HyCA、ApproxABFT 等 selective/partial 冗余）在统一基准下的面积/能耗/覆盖率三维权衡如何？论文以"选择性冗余不可跨设计泛化"为由排除它们，但一个 head-to-head 对比仍具说服力。
