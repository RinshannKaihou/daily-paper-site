Why this matters

Directly answers the most-dodged premise in self-evolving agents — where does the evaluation signal come from? — with inspectable, anchor-locked metrics instead of a learned reward.

Critical edge Caveat

On Spider, held-out metric consistency collapses to 0.500 once residual failures turn semantic — the method nearly fails off the mechanically-detectable end of its own detectability spectrum.

Methodology, distilled

metric = composed drawback detectors

Core move — find defects, don't define "good"

A metric is an inspectable expression of atomic drawback detectors (ops) combined with ∨ ∧ K-of-k. Each op checks exactly one failure class and abstains when it has no opinion — so deterministic detectors (which fail differently from the judged LLM) resist collusion drift. Final verdict = fixed root ∧ evolved expression.

Ablation: anchor guards (not the detector lifecycle) are the load-bearing safety wall.

04 /

Strengths & limitations

side by side

Strengths

  • Concept clarity. "Find defects" beats "define goodness"; deterministic detectors fail differently from the judged LLM, so bootstrap is self-consistent.
  • Method-analysis coupling. Every design choice carries an ablation, judged on metric validity — not the gameable task score.
  • Inspectability first. Final metric reproduces from a string + op pool; evolved detectors are recognizable (missing GROUP BY, uncited claims, overclaimed weak evidence).
  • Honest narrative. Reports the gaming, the judge misreads, the Spider collapse, and ~5-pt per-round noise.
  • Full repro. Loop, op-pack, splits, screening, judge protocol released; every number read from structured JSON/SQLite.

Limitations

  • Anchors can't be manufactured. No anchor → optimizing noise; hardening soft anchors is called "highest-leverage investment" but no path is given.
  • Applicability ceiling. Big gains only when failures are mechanically detectable; semantic-failure tasks degrade to 0.500 consistency.
  • Scale & noise. 3 seeds, 3 task families, 1 solver model, low-hundred rounds; held-out is tens of binary items.
  • Hidden single point. Dev teacher is fixed same-family Opus 4.7; its effect on anchor quality is not ablated. Final audit still needs a human-written task contract.
  • Depressed baseline. Solver is a single direct call, no tools / iteration / CoT; absolute values are not leaderboard-comparable, lift is relative.

Open questions

for further exploration
If the dev teacher swaps to a stronger / out-of-family model, or soft anchors harden into inspectable ones, how does the 88–110% band move? The authors flag this as an un-ablated single point.
When a task family sits at the right end of the detectability spectrum (open dialogue, creative writing), does this reduce to "pure external judge + anchor"? Is a mechanically-detectable proxy for semantic failure needed?
The report-task Goodhart fix relied on a human-written task-aware rubric. Can the task contract itself be auto-generated and pulled into the evolution / audit loop?
The metric's effect on skills is "directional" (decides which attempts become error capsules). Can this be formalized — e.g. prove monotonic held-out improvement under bounded noise?
Naive config loses on metric validity but not on task score. What monitoring would flag, online, that "the metric is dead while the task score lives"?
With a real tool-using / iterative agent (not a single direct call), how does Double Ratchet's relative lift curve change — still oracle-matching?
Post-evolution the metric uses only 1–3 leaves and the skill library is half-pruned. Does seed-pool design then over-determine the final form?
