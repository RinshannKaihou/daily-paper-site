> **一句话总结：** 本文提出 KnowAct-GUIClaw —— 一个基于 "Know-Route-Act-Reflect" 四阶段闭环的个人 GUI 助手框架，通过归因式经验记忆与自演化技能库让 OpenClaw 式 Agent 同时获得跨平台 GUI 操作能力与从执行轨迹中持续学习的能力；在 MobileWorld GUI-Only 上以开源 Kimi-K2.6 取得 64.1% 的 SOTA，超过 Seed-2.0-Pro (63.2%) 和 GPT-5.5 (62.4%)。
>
> **TL;DR:** KnowAct-GUIClaw augments OpenClaw-style personal-assistant agents with a Know-Route-Act-Reflect loop, a typed blackboard for cross-app data flow, and two persistent stores (attribution-aware experience memory + state-validated self-evolving skill library); on the MobileWorld GUI-Only benchmark (117 tasks), the open-source Kimi-K2.6 variant reaches 64.1% SR, beating Seed-2.0-Pro (63.2%) and GPT-5.5 (62.4%), while memory and skills transfer across base models (+8.5 for Kimi-K2.6, +16.2 for Qwen3.5-35B-A3B).

## 核心贡献

1. **"知深行精" (Know Deeply, Act Perfectly) 范式**：把 GUI 自动化重新建模为"宿主 Agent 负责认知编排 + 轻量 GUI 子 Agent 负责视觉执行"的协作过程，而非单一巨型 GUI Agent。系统主张：行动前先检索记忆、技能与策略；行动中以偏可观决策过程记录结构化证据；行动后将轨迹蒸馏为可复用技能或记忆。这统一了认知理解与操作执行。

2. **两级宿主-执行器协作 (Two-tier host–executor collaboration)**：定义了结构化、可恢复的 GUI 任务接口，宿主可下发有限的 GUI 子任务并取回标准化输出（完全完成 / 部分完成 / 阻塞），附带进度与续跑线索。引入"自适应宿主介入规则"——宿主在能用对话上下文或非 GUI 工具解决的子任务上直接作答，只在需要实时 App 会话、视觉定位或设备端验证时才下发 GUI 任务。

3. **记忆驱动的路由与信息契约 (Memory-grounded routing + blackboard)**：路由把请求分为单 App 或多 App 工作流；每个子任务以目标级元组 (g_i, h_i, I_i, O_i) 描述，声明所需输入与产出。临时 blackboard 在子任务间传递类型化的值；若声明输入缺失则"失败闭合 (fail closed)"，杜绝后续子任务凭空捏造。

4. **知识/技能增强的混合执行 (Hybrid GUI–fast-path execution)**：动作空间 A = A_gui ∪ A_skill ∪ A_shortcut ∪ A_ask，把 GUI 原语、蒸馏技能、Android deeplink/intent 快捷方式与用户介入动作统一在同一执行循环里；技能和快捷方式使用前必须通过当前屏幕状态的校验。

5. **轨迹衍生的记忆与技能自演化**：反思阶段把成功/失败轨迹压缩成可检索的经验记忆（受 ReasoningBank 启发）和参数化、带状态契约的可执行技能；技能修复优先于新技能提取，避免"用一个无关工作流替换原技能"。

## Key Contributions

- A **paradigm** ("Know Deeply, Act Perfectly") that decouples high-level orchestration (host agent owning conversation, workspace memory, user profile, tools) from low-level visual device manipulation (lightweight GUI executor owning screenshot perception, action normalization, device backends, skill validation, trajectory recording).
- A **four-stage Know-Route-Act-Reflect closed loop** with two persistent stores (memory+history store, skill+shortcut store) feeding every stage.
- A **typed blackboard protocol** that makes cross-app data flow explicit and "fails closed" rather than letting later subtasks infer or fabricate values — directly attacking the free-text-summary memory-decay failure mode of monolithic GUI agents.
- A **hybrid action space** unifying GUI primitives, distilled skills, validated Android deeplink/intent shortcuts, and intervention actions, with pre-execution state validation.
- A **self-evolution mechanism** that distinguishes textual experience rationales (advisory, never override the current instruction) from executable state-validated skills, and prioritizes in-place skill repair over new extraction.
- Empirical evidence that **memory and skills transfer across base models** (Kimi→Qwen direction), not just within a single model.

## Methodology

**Architecture (Know-Route-Act-Reflect).** GUI automation is formulated as a POMDP M = (S, A, O, T, Ω, R, γ) where the hidden device state (navigation stacks, login/permission status, async loading, off-screen form values) is never directly observed. The framework runs four stages per long-horizon task:

- **Know** — active retrieval: prior GUI memories and candidate skills retrieved by semantic similarity (kept *advisory*, never override the current instruction); policy memory injected directly. Host recalls session history, agent memory, and user profile only when judged relevant; for fuzzy instructions it proposes defaults as explicit, overridable assumptions.
- **Route** — emits either a single GUI task or an ordered multi-app workflow of tuples (g_i, h_i, I_i, O_i). Each subtask sees only declared inputs from B_{i-1} and writes only declared outputs back: G(g_i, h_i, B_{i-1}[I_i]) → τ_i, E(τ_i, O_i) → B_i[O_i]. Missing input/output ⇒ fail closed.
- **Act** — observe–reason–act loop p_t → o_t → a_t → e_t → o_{t+1} over the hybrid action space. Skills: each stored skill has identifier, app/platform scope, parameters, reliability counters, ordered steps with placeholders; stable fields (package names, coordinates, intent payloads) fixed in advance, task-dependent values grounded at run time. Top-5 skills retrieved via lexical+embedding, a lightweight applicability prompt picks one prefix or rejects all; per-step state contract checked (deterministic when available, visual valid-state check otherwise), with bounded recovery subgoal / skip-optional / fall-back-to-ordinary-GUI on mismatch.
- **Reflect** — summarizes trajectory, skips no-signal runs (empty/cancelled/timed-out/skill-replayed), distills reusable procedures via a vision LLM under a declarative-only prompt, separates skill repair (in-place, narrowing description / adding guarded obstacle handling / refreshing stale targets) from new extraction, and induces ≤ a few textual memory items per trace with per-app de-duplication.

**Setup.** Default pair: Qwen3.5-397B-A17B host + Qwen3.5-35B-A3B GUI executor; Kimi-K2.6 configs use Kimi-K2.6 as both. MobileWorld GUI-Only subset (117/201 tasks) scored by native deterministic evaluators, 50-step cap, primary metric = single-run SR (pass@1). AndroidDaily scored on iOS with qwen3.5-flash LLM judge (GUI-only) and two human experts (1.0/0.5/0 rule for answer-returning tasks).

## Results & Highlights

- **MobileWorld SOTA.** KnowAct-GUIClaw with open-source Kimi-K2.6 reaches **64.1% SR** (pass@1) on the 117-task GUI-Only subset, beating closed-source Seed-2.0-Pro (63.2%), GPT-5.5 (62.4%), Gemini-3.1-Pro-Preview (58.1%), Claude-Opus-4.7 (56.4%), and the Kimi-K2.6 public baseline (55.6%) — Table 1.
- **Cross-model transfer of memory+skills.** Equipping the Qwen3.5-35B-A3B executor with experience memory and skills distilled from Kimi-K2.6 trajectories lifts SR from **24.8 → 41.0** (+16.2 pts), 3.1 points above the standard 35B host–memory–skills config (37.9); for Kimi-K2.6 itself, host+memory+skills yields **55.6 → 61.5 → 64.1** (+8.5 pts total) — Table 1.
- **Skills cut cost while raising accuracy.** On skill-using tasks (Table 3), reuse removes ~3.3 GUI steps/task, cuts total tokens ~6% (35B: 303k→284k, −6.2%; 397B: 258k→243k, −5.9%), and improves single-run SR (+4.9 pts for 35B, +1.9 pts for 397B).
- **A capable host should not route everything through GUI.** Letting the 397B host resolve eligible info-query subtasks directly (config F) yields the best SR (46.2) at the lowest GUI-trace cost among Qwen ablations: total tokens 260,516 (down from 278,289 in config C), host total only 10,982 tokens/task (vs 63,792 in C) — Table 2.
- **AndroidDaily (iOS).** KnowAct-GUIClaw scores **78.61% (resolved, 194 tasks) / 64.89% (all, 235 tasks)**, with the largest margins over GUI-only baselines on Analyze (77.08 vs UI-TARS-1.5's 36.71) and complex tasks (62.50 vs 13.64) — Table 4.
- **Cross-platform usability.** Same interface solves **48/63 (76.2%) HarmonyOS** and **21/30 (70.0%) Windows** tasks under a manual-translation protocol — Table 6.
- **Stability.** pass@3 ("all 3") on MobileWorld reaches 32.5% for config F (vs 24.8% single-run for base), showing accuracy gains are not random volatility — Table 5.

## Critical Assessment

The paper's central design move — separating *what* to do (host) from *how* to do it on screen (GUI executor) and making efficiency a first-class objective — is well-motivated and the ablations (Table 2, esp. config F: best SR at lowest cost when the 397B host answers eligible subtasks itself) genuinely support it. The typed-blackboard "fail closed" rule is the strongest engineering contribution: it converts the common failure mode of monolithic GUI agents (free-text intermediate summaries that lose values across apps) into a structurally checkable contract, and the Email→Messages→Maps and cart→SMS case studies (Figs. 5/7/8) concretely show observed prices, addresses, order numbers and phone numbers flowing as declared typed outputs.

However, several aspects warrant scrutiny:

- **Cross-platform numbers are weakly matched.** The HarmonyOS check (48/63, 76.2%) excludes Android-mirrored tasks or manually initializes weakly-tied ones, and the Windows check is a hand-designed 30-task set judged by the authors — neither is a standardized benchmark. The disclosed failure modes are illuminating but the denominators are small.
- **Failure modes undercut the "Act Perfectly" claim.** The HarmonyOS weekend-alarm case fails because GUIClaw cannot settle a wheel picker on "25 minutes"; the TaoBao tattoo case terminates prematurely because the model conflates a "Pay Now" button with the payment page; Windows fails on natural-language affect→icon mapping ("laughing" emoji) and on clicking a short-lived Slack toast. These expose real weaknesses in continuous geometric control, temporal grounding, and language→symbol grounding that the hybrid action space does not yet address.
- **Transfer directionality.** Only Kimi→Qwen transfer is demonstrated; the paper is admirably explicit that this does not assume "universal model-independent transfer," but readers should not over-generalize.
- **Scoring subjectivity on AndroidDaily.** GUI-only tasks scored by qwen3.5-flash and answer-returning tasks by two human experts under a 1.0/0.5/0 rule is reasonable but not deterministic; the resolved-vs-all gap (78.61 vs 64.89) is attributed partly to app availability rather than policy quality.
- **Pipeline rigidity acknowledged.** The conclusion concedes the current Know→Route→Act→Reflect sequence is rigid, and future work targets a "unified joint planner" weighing knowledge retrieval, non-visual tool calls, and GUI actions in one decision cycle — implicitly admitting the sequential handoff overhead the framework currently pays (visible as the 65,224 token "Host Total" when coordinating the weaker 35B executor across 2.3 GUI task invocations).

## Strengths

- Clean conceptual separation of cognitive orchestration vs. visual execution, with efficiency treated as a first-class design goal rather than an afterthought.
- The typed blackboard + "fail closed" rule is a concrete, mechanically checkable fix for cross-app value loss.
- Unified skill abstraction covering click-then-type patterns, multi-step procedures with per-step state contracts, and validated deeplink/intent shortcuts under one retrieval/validation pipeline.
- Strong, reproducible main result on a standard benchmark with both single-run and repeated-attempt metrics, plus independently logged cost columns.
- Honest, detailed failure-mode analysis (HarmonyOS wheel picker, premature termination, Windows temporal/semantic grounding) rather than cherry-picked successes.

## Limitations

- Cross-platform evaluation is non-standardized (manually designed/translated sets, small denominators, qwen3.5-flash judge).
- Skill/shortcut validation depends on on-device page-state checking; brittle to app updates that change package names, coordinates, or UI structure (the paper fixes these in advance and grounds placeholders at run time, but does not measure skill decay over app versions).
- Memory+skill transfer shown only in the Kimi→Qwen direction.
- Sequential pipeline imposes host–subagent communication overhead; a unified joint planner is left to future work.
- Continuous geometric control, deep app-specific controls, off-viewport targets, and temporal grounding (toast clicking) remain weak.
- AndroidDaily depends on mixed LLM-judge + human-expert scoring and on iOS device availability.

## 相关性 (Relevance)

对个人 LLM 助手 / GUI Agent / 计算机使用 (computer-use) 方向高度相关：

- 直指当前主流 Agent 框架（OpenClaw 类）"缺 GUI 操作 + 缺自演化机制"两大短板，是把"配置驱动型个人助手"与"视觉 GUI Agent"两条线工程化缝合的代表工作。
- 其"宿主编排 + 轻量执行器"分工、类型化 blackboard、"失败闭合"契约、技能/记忆双存储的设计，对任何做长程跨 App 自动化、移动端 RPA、或 Agent 记忆系统的研究者都有直接借鉴价值。
- 把经验记忆与可执行技能区分开（前者是 advisory 提示，后者是带状态契约的可验证过程），并给出跨模型迁移的初步证据，呼应了 Reflexion / Voyager / ReasoningBank 这条"无权重更新的 Agent 自演化"主线。
- 对做 MobileWorld / AndroidWorld / OSWorld 评测、或关心 deeplink/intent 快捷方式安全复用的人，是一份可复现的基线与设计模板。

## Questions for Further Exploration

1. **技能/快捷方式的版本鲁棒性**：当 App 更新导致包名、坐标、UI 结构漂移时，状态契约失效频率多高？是否需要一个自动失效检测 + 重验证 pipeline？
2. **反向迁移**：从弱模型 (Qwen-35B) 蒸馏的记忆/技能能否反向提升强模型 (Kimi-K2.6)？论文只验证了 Kimi→Qwen 方向。
3. **"统一联合规划器"的边界**：把知识检索、非视觉工具调用与 GUI 动作放进同一决策周期，相比当前顺序流水线，在 SR 与 token 成本上能带来多少边际收益？是否会引入新的调度歧义？
4. **连续几何控制 / 时序定位**：wheel picker、滑动条、toast 点击这类失败能否通过把几何/时序动作单独建模（而非塞进统一 GUI 动作空间）来缓解？
5. **跨平台动作空间共享度**：Table 7 显示 mobile/desktop 共享核心动作，但快捷方式 (deeplink/intent vs Windows 等价物) 的对齐策略是什么？能否做到一份技能描述多平台落地？
6. **记忆与技能的可信度**：经验记忆作为 advisory 注入，是否存在"过时教训"误导路由的风险（例如 App 改版后旧失败教训反而把任务引向已修复的路径）？是否有遗忘/置信度衰减机制？
7. **与纯端到端 GUI 大模型 (如 UI-TARS / GUI-Owl 系列) 的能力互补**：在哪些任务子集上，KnowAct-GUIClaw 的"宿主+技能"结构相比纯视觉 GUI 模型优势最大、又在哪里被反超？
