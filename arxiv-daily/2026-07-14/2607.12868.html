Why this matters

A reusable, reproducible public benchmark + injection framework for evaluating DNN training-fault detection and diagnosis algorithms.

Relevance

Critical edge

Only untuned Logistic Regression / Random Forest are evaluated — floor estimates, not strong baselines, leaving real headroom unmeasured.

Critical Assessment · Limitations

How it's built

Methodology · 4-stage pipeline
  1. Program selection. 59 complete TensorFlow/Keras programs harvested from Stack Overflow, repaired to run deterministically (fixed seeds, consistent splits, consistent init) across FNN, CNN, RNN.
  2. Fault injection. Each variant generated by one of 27 AST-based operators across 7 categories: training config, activation, loss, optimizer, weight init & bias, regularization, layer structure. DeepCrime subset + new layer-level operators.
  3. Training & feature extraction. Up to 15 retrainings per mutant, logging per epoch 4 eval metrics + 26 dynamic features (weights, gradients, activations, accuracy/loss trends, learning rate, hardware).
  4. Mutation validation. A mutant is killed at Cohen's d ≥ 0.5 and p < 0.05 (GLM, n=15); non-killed (equivalent) mutants dropped. Mutation scores: FNN 0.85, CNN 0.87, RNN 0.83.

 95% bootstrap CIs (final vs traj MCC) do not overlap.
  • Final-epoch collapses to the trivial baseline under imbalance: always-faulty classifier scores F1 0.818; final-epoch RF reaches only F1 0.816, MCC 0.150.
  • 7-class diagnosis also improves: Macro-F1 0.355→0.381, MCC 0.346→0.401.
  • Cross-architecture transfer mostly holds: held-out third arch MCC 0.210 / 0.218 / 0.198 (FNN/CNN/RNN) vs 0.227 in-distribution; gap largest for RNN.
  • Single-epoch early detection underperforms: MCC 0.14–0.19 at every k — no single epoch dominates.

Strengths

  • First reusable public dataset coupling source programs, mutation categories, and per-epoch trajectories for DNN training faults.
  • Statistically grounded mutation validation (Cohen's d + GLM, n=15) with per-architecture mutation scores reported.
  • Group-aware CV (grouped by SO program ID) prevents program-level leakage; MCC + bootstrap CIs fit the imbalanced labels.
  • Brutally honest reporting that final-epoch F1 (0.816) is no better than the trivial always-faulty baseline (0.818).
  • Fully open artifact: Zenodo DOI + GitHub, regeneration scripts, dual licensing, provenance to SO posts.

Limitations

  • Stack Overflow provenance may not reflect industrial practices or real fault distributions; correct baselines checked for execution, not semantic correctness.
  • No Transformers — restricted to TF/Keras FNN/CNN/RNN; the dominant 2024–2026 architecture family is out of scope.
  • Hardware features are environment-dependent (GPU memory = 0 with no GPU), which can silently distort cross-platform comparison.
  • Only LR / RF, untuned — no neural or diagnosis-specific baselines (e.g., the authors' own DEFault hierarchical classifier).
  • Single-label diagnosis only; multilabel (co-occurring faults) not supported. No per-feature-group ablation.

Open questions

For further exploration
  1. How much signal comes from each feature group (weights / gradients / activations / trends) vs resource features — and do resource features inflate performance on fixed hardware?
  2. Would Transformer programs show the same 7-category separability, or need new fault categories and operators?
  3. Can stronger, diagnosis-specific models beat the untuned RF MCC ~0.23 floor substantially?
  4. Is the RNN cross-architecture gap (0.198 vs 0.227) truly gradient-flow over time steps — can time-step-aware features close it?
  5. Can the Activation↔Optimization and Optimization↔Layer confusions be resolved with more discriminative features?
  6. How do mutation-derived detection rates compare against real, naturally occurring DNN training faults from industrial codebases?
