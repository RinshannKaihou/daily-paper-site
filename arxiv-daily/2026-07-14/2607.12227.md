> **一句话总结：** 本文重新审视 LLM 智能体"自动 harness 演化"的评估方法，在统一的反馈与推理预算下与并行采样、序贯精炼等 test-time scaling 基线对照后发现：harness 演化既未稳定超越简单基线，演化出的 harness 也几乎无法泛化到训练集之外的任务，提示现有收益更多来自额外搜索而非真正的 harness 设计改进。

> **TL;DR:** The paper re-evaluates automatic harness evolution for LLM agents by comparing it against parallel sampling and sequential refinement test-time scaling baselines under matched feedback and inference budgets on Terminal-Bench 2.1 with GPT-5.4 / Claude Opus 4.6; harness evolution fails to consistently beat simple baselines (e.g., parallel sampling reaches pass@1 86.0 vs. harness evolution's 75.8 with unit tests) and the evolved harnesses barely generalize to held-out tasks (avg +0.6 points), so reported gains largely stem from extra test-time search rather than genuine harness design.

## 核心贡献

1. **提出统一的预算化评估框架。** 论文将"自动 harness 演化"重新表述为四种在统一计算预算 `K` 下的方法：(a) 并行采样（Parallel Sampling，预算花在独立 trajectory 上）、(b) 序贯精炼（Sequential Refinement，预算花在 trajectory 的深度修订）、(c) harness 演化（Harness Evolution，预算花在跨任务的共享 harness 更新）、(d) harness 缩放（Harness Scaling，预算花在单实例的 harness 修订）。框架明确每种方法"更新什么、能观察到什么反馈"，从而把"是否改动 harness"和"是否只是在 test-time 多搜"区分开来。
2. **指出现有评估协议的两个根本缺陷。** 第一，harness 演化本质是一种反复用任务反馈评估并修订 harness 的迭代搜索，理应在"匹配的反馈与推理预算"下与 test-time scaling 基线比较，以判断增益是来自更优的 harness 设计还是仅仅来自额外搜索；第二，搜索与最终评估共用同一公开基准（如 Terminal-Bench），使增益有过拟合到该任务集的风险。
3. **系统性地实证证伪了 harness 演化的优势。** 在无/有单元测试反馈、以及"训练-评估任务不相交"三种设定下，自动 harness 演化既未稳定超越简单基线，也几乎不泛化到 held-out 任务，揭示了"搜索与评估共用基准"会高估真实 harness 改进。
4. **引入并启用泛化评估协议。** 将 Terminal-Bench 2.1 划分为 45 个训练任务、10 个验证任务、34 个 held-out 测试任务，对 harness 演化做 disjoint split 评估，用以衡量"学到的 harness 是否是可迁移的设计原则"而非任务捷径。

## Key Contributions

- A **unified budget framework** that formalizes four methods (parallel sampling, sequential refinement, harness evolution, harness scaling) by what they update and what feedback they observe, making the confound between "improving the harness" and "spending more test-time search" explicit.
- A **critique of the standard harness-evolution evaluation protocol**, which searches over harnesses using benchmark-derived verifier feedback and then re-evaluates on the same benchmark — conflating harness design with test-time discovery and risking overfitting.
- An **extensive empirical study** on Terminal-Bench 2.1 across three frontier models (Claude Opus 4.6, GPT-5.4, GPT-5.4 mini) showing harness evolution does **not** consistently outperform simple test-time scaling and that its gains **do not transfer** to held-out tasks.
- A **held-out generalization protocol** (45 train / 10 val / 34 test split) demonstrating that when search and evaluation tasks are disjoint, harness evolution's advantage essentially vanishes.

## Methodology

**统一设定。** 给定固定策略 `π_θ`、任务分布 `X` 与计算预算 `K`，所有方法都使用一个由同一底座模型实现的"总结映射" `Φ`，把经验（trajectory、分数、失败）压缩成可被下游智能体消费的形式（如反复出现的失败模式、冗余尝试、开销）。`Φ` 使序贯精炼与 harness 编辑能在前一轮证据基础上诊断与修正。

- **并行采样**：用固定 harness 独立采样 `K` 条 trajectory，无单元测试时由 self-judge `J` 选最优候选；有单元测试时选任一通过 outcome `R(y,g)=1` 的 trajectory。
- **序贯精炼**：把预算分配给深度，每条 trajectory 在前一条基础上修订（self-refine）；无测试信号时返回最后一条，有信号时返回任一被接受的 trajectory。
- **Harness 演化**：保留序贯循环但把预算投向 harness（而非 trajectory），并跨任务分布优化。每轮采样 `m` 条 rollout/task，将 `(harness, rollouts[, outcomes])` 写入经验库；元智能体 `M` 读 `Φ(C_{k-1})` 产生新 harness。论文用 **AHE**（Lin et al., 2026）实例化，但**禁用其 explore agent**（因为它会从外部检索针对评测基准调好的 harness，会把"可复用演化"与"检索已对评测集拟合的解"混淆），从而让所有增益都归因于基于反馈的演化。
- **Harness 缩放**：harness 级的 test-time scaling 类比——为单一评测实例把预算花在 harness 修订上（实例引导的 harness 适配），与"数据集引导的 harness 演化"相对。

**实验设置。** 基准为 Terminal-Bench 2.1（对 Terminal-Bench 2.0 的修正版，修复 89 个任务中的 28 个外部依赖漂移/资源预算错配/指令与测试不一致问题，仍保留 89 个终端任务）。所有四种方法用**同一最小初始 harness**（仅一个 bash 工具，无 skills / middleware / 持久记忆）。AHE 每任务每 harness 采 1 条 rollout（`m=1`），所有方法统一 `K=5`。所有模型用 128k token 上限 + 高 reasoning effort，结果对两次独立运行求平均以降低随机性。所有 rollout 在独立、新鲜供给的 E2B 远程沙箱中执行。

## Results & Highlights

- **无单元测试反馈时（Table 1，pass@1）。** Harness 演化不仅没赢，反而在平均上**低于直接采样基线**：平均 67.4 vs. 直接采样 68.2；最严重的是在 GPT-5.4 上从直接采样 75.3 掉到 69.7，说明"只靠智能体自身判断引导的迭代 harness 修订"会主动伤害强模型。对比之下，并行采样把平均分从 68.2 提到 72.3（三种模型全升），harness 缩放平均 71.8 并在 Claude Opus 4.6 上拿到单点最佳 76.0。
- **有单元测试反馈时（Table 2）。** 即便有了可靠外部信号，harness 演化仍输给简单基线：pass@1 平均仅 75.8（几乎没超过直接采样），而并行采样达 **86.0**（最佳）；pass@5 上序贯精炼 **91.8**（最佳）超过 harness 演化的 86.2 与 harness 缩放的 89.3。关键观察是：如果 harness 修订真的产生了更好的 harness，增益应体现在 pass@1；但收益只在"能从多条 trajectory 中选"时才出现 → 现有 harness 演化的增益主要来自"多尝试几次"，而非"改进了 harness 设计"。
- **泛化设定（Table 3，disjoint train/val/test）。** 在 45 训练 / 10 验证 / 34 测试的划分上，演化 harness 在测试集上仅带来 **+1.2（Claude Opus 4.6）、+0.0（GPT-5.4）、平均 +0.6** 的提升——与"在同一任务集上演化和评测"时的提升形成鲜明反差，强烈表明演化过程学到的是**任务特定捷径**而非可迁移的 harness 设计原则，存在严重过拟合。
- **编辑行为分析。** 元智能体确实做出合理且有动机的编辑（提示层加行为规则→中间件做运行时强制如 turn 预算追踪/工具输出截断/完成门控→工具层修误导性指引并注入恢复提示；harness 缩放则把已知 bug、实现模板、验证检查、文件路径等固化进 prompt/长记忆），但绝大多数编辑是**"记忆修复"而非"提炼策略"**——这些信息一个称职智能体在单次 rollout 内本就能重新发现，于是只节省了它本就能解决的任务的时间，却很少把失败转为成功；而由深度领域推理或 harness 之外的约束导致的稳定难任务核心依旧未被触及，且持续膨胀的 prompt 文本带来的 context 膨胀会部分抵消收益。

## Authors & Credibility

- **机构**：Allen Institute for AI（AI2）+ University of Washington，作者包括 Hannaneh Hajishirzi、Yulia Tsvetkov、Pradeep Dasigi 等资深 NLP/智能体研究者。论文风格严谨、自含完整配置与附录，并公开代码（https://github.com/rethinking-harness-evolution）。
- **基准选择得当**：Terminal-Bench 2.1 是对 2.0 的已验证修正版，并明确说明为何采用三种前沿模型、为何 `m=1`、为何 `K=5`，所有方法初始化自同一最小 harness，对照设计较公平。
- **方法学诚实**：作者主动禁用 AHE 的 explore agent 以避免"检索已对评测基准拟合的 harness"这一混淆，且在讨论中坦诚给出"harness 演化可能只是在当前模型/当前基准上无效"的替代解释（见下）。

## Critical Assessment

论文最大的价值是**把一个被社区默认接受的评估协议（在搜索所用基准上再评估）放到了显微镜下**，并通过"统一预算 + disjoint split"两个干净设计证伪了 harness 演化的核心卖点。证据扎实：Table 1 里 harness 演化在 GPT-5.4 上从 75.3 → 69.7 的退化、Table 2 中 pass@1 几乎原地踏步（75.8 vs 直接采样 72.9）却只在 pass@5 受益、Table 3 中 held-out 上 +0.6 的近乎为零的泛化——三者相互印证"增益来自多采样而非 harness 设计"。讨论部分（5.2）也诚实地承认了反方解释：Terminal-Bench 上智能体已拿高分、剩余失败可能源自模型而非 harness，且该基准对 harness 设计**本身不敏感**（shell 工具 + 基础提示即可解大多数可解任务），意味着结论的**外部效度受限于基准选择**——作者据此明确呼吁未来工作应在"足够难且高度依赖 harness"的基准上重测。这是一篇定位清晰的"评估/复现"型工作，而非新方法，其影响力将取决于社区是否据此修订 harness 演化的评测规范。

## Strengths

- 统一预算框架把四种方法放在同口径下比较，消除了"多花钱"与"真改进"的混淆，方法论上具有示范意义。
- 三种设定（无/有单元测试、disjoint 泛化）层层递进，分别回答"能不能赢""赢了是不是因为 harness""学到的能不能迁移"三个递进问题。
- 对 AHE explore agent 的禁用与对"记忆 vs 策略"编辑的定性分析，显示了作者对混淆变量的清醒意识。
- 公开代码与完整配置（Appendix A），可复现性强。

## Limitations

- **基准单一且可能不 harness-sensitive。** 全部结论基于 Terminal-Bench 2.1；作者自己也指出该基准用"shell + 基础 prompt"即可解大多数任务，因此"harness 演化无效"可能部分是基准特性，而非普适规律。
- **模型覆盖有限。** 仅三种前沿模型，未涉及开源模型或更小模型（后者或许更能从 harness 修订中受益）。
- **harness 演化实例仅 AHE 一种。** 未对 Meta-Harness、AEVO 等同期方法做同等深度的统一预算对照，结论对"harness 演化这一范式"的普适性略有折扣。
- **disjoint split 较小。** 45 训练 / 34 测试的规模有限，泛化结论的统计置信度受任务数制约（虽对两次 run 求平均）。
- **未探索"更强的 prior/更结构化的搜索目标"能否挽救 harness 演化**——作者把这留给未来工作，但读者无法从本文判断该方向的上限。

## Relevance

本文直击当下 LLM 智能体研究的一个热点与潜在泡沫：自动 harness/agent 框架演化（AHE、Meta-Harness、AEVO 等）。对做智能体评测、agent 框架工程、test-time scaling 的研究者与工程师而言，它提供了：(1) 一个可复用的"统一预算 + disjoint 评估"协议模板；(2) 一个清醒的负面结果，提醒不要把"在搜索所用基准上的提升"当成 harness 设计的真实进步；(3) 对"何时 harness 演化才可能有用"的明确判据（任务足够难 + 高度依赖 harness）。与 test-time scaling、self-refine、DSPy/GEPA 等 prompt/程序优化文献构成直接对话。

## Questions for Further Exploration

- 在**高度依赖 harness** 的基准（如需要专有工具/技能/复杂工作流的 SWE 或科研任务）上，harness 演化是否能真正超越 test-time scaling？作者呼吁但未实测。
- 若给 harness 演化提供**更强的先验或结构化搜索目标**（例如约束编辑只能发生在工具/中间件层、或用课程学习按难度分组演化），是否能产生可泛化的设计原则？
- "记忆修复 vs 策略提炼"的定性结论能否**量化**？例如设计一个指标度量"演化编辑的可迁移性"。
- 对于**较弱或开源模型**，harness 演化是否反而比 test-time scaling 更划算（因为弱模型单次 rollout 天花板更低）？
- explore agent 被禁用是合理的混淆控制，但现实部署中"检索外部 harness"与"基于反馈演化"常耦合——如何在评测中**分离但又公平地评估**两者的各自贡献？
- pass@1 几无提升、pass@5 才受益的现象，是否暗示 harness 演化的真正价值在于**扩大可解任务集**而非提高单次成功率？如何设计指标直接衡量这一点？
