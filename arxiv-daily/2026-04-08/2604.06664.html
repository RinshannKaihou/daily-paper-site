Foundry: Template-Based CUDA Graph Context Materialization for Fast LLM Serving Cold Start

一句话总结： 提出Foundry系统，通过模板化CUDA图上下文物化技术将LLM服务冷启动时间减少99%，将Qwen3-235B-A22B的初始化从10分钟缩短至3.9秒，同时保留CUDA图的吞吐量增益。
TL;DR: Foundry materializes CUDA graph execution context during offline processing and reconstructs executable graphs online with negligible overhead, reducing Qwen3-235B-A22B cold-start from 10 minutes to 3.9 seconds (99% reduction) while preserving CUDA graph throughput gains.

核心贡献

  • 提出模板化CUDA图上下文物化系统，在离线阶段持久化图拓扑和执行上下文，在线阶段以极低开销重建
  • 通过确定性内存布局、内核二进制自动提取和拓扑模板化减少在线重建成本
  • 支持单GPU离线捕获生成多GPU部署模板，仅需修补秩相关通信状态
  • 在密集和MoE模型（最大235B参数）上实现99%冷启动延迟减少

Key Contributions

  • Template-based CUDA graph context materialization that persists both graph topology and execution context offline, enabling near-instant online reconstruction
  • Enforces deterministic memory layouts and automatically extracts/reloads kernel binaries required by captured graphs
  • Single-GPU offline capture generates templates for multi-GPU deployments by patching only rank-dependent communication state
  • Reduces Qwen3-235B-A22B initialization from 10 minutes to 3.9 seconds (99% reduction)

Methodology

Foundry addresses the fundamental challenge that CUDA graphs cannot be naively serialized because they are tightly coupled to execution context (device addresses in kernel arguments, lazily loaded kernel code). The system works in three phases: (1) offline capture with deterministic memory layout enforcement, (2) extraction of kernel binaries and topology-based template creation, and (3) online reconstruction by materializing the execution context (memory allocation, kernel binary loading, address patching). For distributed serving, Foundry captures on a single GPU and generates multi-GPU templates by patching only rank-dependent communication state (e.g., NCCL unique IDs, rank assignments).

Results & Highlights

  • Qwen3-235B-A22B: initialization from 10 minutes to 3.9 seconds (99% reduction)
  • Tested across dense and MoE models up to 235B parameters
  • Preserves throughput gains of CUDA graphs (no performance regression)
  • Single-GPU capture generates templates for multi-GPU deployments
  • Open-sourced at github.com/foundry-org/foundry

Critical Assessment

This is a strong systems paper that solves a real and increasingly important problem. As LLM serving infrastructure relies more on autoscaling and parallelism reconfiguration, cold-start latency from CUDA graph capture becomes a critical bottleneck. The 99% reduction (10 min to 3.9 sec) is a dramatic improvement. The single-GPU-to-multi-GPU template generation is an elegant solution for distributed deployments. However, the approach is tightly coupled to NVIDIA's CUDA runtime internals, which may limit portability to AMD (HIP) or other GPU platforms. The paper could also discuss failure modes -- what happens when kernel binaries change across driver versions or when memory layouts become non-deterministic due to external factors.

Strengths

  • Solves a critical real-world bottleneck with dramatic improvement (99% cold-start reduction)
  • Single-GPU capture generating multi-GPU templates is elegant and practical
  • Open-sourced implementation available
  • Preserves CUDA graph throughput gains -- no performance regression
  • Strong systems research team with distributed systems expertise

Limitations

  • Tightly coupled to NVIDIA CUDA runtime internals; portability to AMD/Intel GPUs unclear
  • May break across driver versions or CUDA toolkit updates
  • Does not address the model weight loading component (complementary to recent RDMA-based solutions)
  • Requires deterministic memory layouts, which may conflict with some memory allocators

Relevance

对LLM服务基础设施的弹性扩展和冷启动优化有直接实用价值。模板化CUDA图物化技术为分布式推理系统的快速部署和动态重配置提供了系统级解决方案。对理解CUDA图与执行上下文的耦合关系有技术参考意义。

Questions for Further Exploration

  • Can the template approach be extended to cover model weight loading as well?
  • What happens when kernel implementations change across CUDA toolkit versions?
