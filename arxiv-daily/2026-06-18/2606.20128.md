> **一句话总结：** 本文构造了一个包含 26 个算子（含作者植入的 LLM 风格转录 bug）的受控语料，证明 KernelBench / TritonBench / GEAK 这类 LLM 内核基准所用的"固定形状 + 小样本 + allclose"判分 oracle 会把 GELU 漏写 0.5、softmax 尾部掩码写错、RMSNorm 漏 sqrt、matmul 用 `acc=` 覆盖累加器等真实转录错误误判为"正确"，而 op-schema 感知的种子化模糊测试（配合 fp64 CPU 参考实现与按算子/数据类型设定的绝对容差）能在 RTX 3060 / A10 / A100 SXM4 / L40S / H100 NVL 五类 GPU 上一致地抓出 10/10 个"正确性幻觉"且 15–16 个对照组零误报，每次失败可从存档种子逐字节复现。

> **TL;DR:** LLM-GPU-kernel benchmarks (KernelBench, TritonBench, GEAK) judge correctness with a fixed-shape, small-sample `torch.allclose` check against one shape, one dtype, and a hand-picked tolerance. This paper builds a 26-op controlled corpus seeded with documented LLM transcription bugs (GELU missing 0.5, softmax tail-mask leak, RMSNorm/L2Norm missing sqrt, matmul `acc=` overwrite, attention missing 1/√D, LeakyReLU wrong α, SiLU sigmoid(2x)), and shows the benchmark oracle certifies all of them as "correct" while an op-schema-aware seeded fuzzer with an fp64 reference and per-(op,dtype) absolute tolerance catches 10/10 buggy variants and passes 16/16 controls on five GPU classes (RTX 3060, A10, A100 SXM4, L40S, H100 NVL), with bit-for-bit replay from stored seeds. Worth reading if you build or trust LLM-kernel benchmarks; the bugs are author-seeded (not pulled from real LLM outputs), so this is a constructive demonstration of oracle weakness, not a measured LLM bug rate.

## 核心贡献

- **揭示三类系统性乐观偏差**：固定形状（一个算子只测一个形状，遗漏尾掩码 / 越界累加 / block-size 假设）、固定数据类型（fp32 当默认，fp16/bf16 的溢出/下溢/累积误差基本不测）、手挑容差（atol/rtol 比实际误差包络松 1–3 个数量级，把"真错误"吸收掉）。
- **op-schema 感知的种子化形状生成方法**：每个算子自带 schema（声明共享符号维度与候选值，如 attention 的 B/H/S/D），fuzzer 经 Blake2b 子密钥派生确定性采样，并刻意包含边界值（如 H ∈ {1, 3, 7, 256, 1025}），覆盖算子真实定义域而非任意 rank-3 立方体。
- **高保真数值 oracle**：以 fp64 CPU 参考实现折回目标数据类型作真值，采用**仅绝对容差**（按 op × dtype 设定，区别于 PyTorch `allclose` 的 atol+rtol 组合），并记录完整逐元素误差分布（max/mean/p50/p90/p99 abs、max/mean rel、ULP 分布）以及对 fp16/bf16 的 NaN/Inf 检测。
- **真实多 GPU 上的测量**：单卡 RTX 3060 上 9/9 bug 命中、15/15 对照通过、零误报；扩到 26 算子后，在 5 类 GPU（sm80/86/89/90）上判决完全一致：10/10 幻觉抓出、16/16 对照干净。
- **可复现流水线**：每次失败保存完整输入快照（ReproductionInfo），replay 脚本可逐字节复现判决；公共仓库 gpuemu + gpuemu-corpus，vast.ai 临时实例由 label-strict reaper 与上下文管理器保证销毁，结果上传 Backblaze B2 并带 run id（如 `run-20260611-095210-889b18`）。

## Key Contributions

- **Identifies three systematic optimism modes** in the standard correctness oracle: a single fixed shape per op (misses tail-mask / off-by-one / block-size bugs), a single dtype per op (fp16/bf16 overflow/underflow/accumulation errors essentially untested), and hand-picked tolerances 1–3 orders of magnitude looser than the measured error envelope.
- **Op-schema-aware seeded shape generation**: each kernel ships a schema of shared symbolic dims with concrete candidate values; the fuzzer derives per-dim samples deterministically from a master seed via Blake2b sub-key derivation and deliberately includes boundary values (e.g. H ∈ {1,3,7,256,1025}), covering the operator's real domain instead of an arbitrary cube.
- **A faithful numerical oracle**: fp64 CPU reference rounded to the target dtype, compared with **absolute-only** per-(op, dtype) tolerances (in contrast to PyTorch's combined atol+rtol), recording the full element-wise error distribution (max/mean/p50/p90/p99 abs, max/mean rel, ULP percentiles) plus NaN/Inf detection for fp16 and bf16.
- **A real multi-GPU measurement**: on the 24-op single-GPU corpus (RTX 3060) the seeded oracle flags 9/9 buggy variants and passes 15/15 controls with zero false positives; extending to 26 ops across five GPU classes (RTX 3060, A10, A100 SXM4, L40S, H100 NVL) yields identical verdicts — 10/10 illusions caught, 16/16 controls clean.
- **A reproducible pipeline**: each failure stores a full input snapshot replayed bit-for-bit; the `gpuemu` validator daemon and the 26-op corpus are public; ephemeral vast.ai instances are torn down by a label-strict reaper with context-manager exit, and results (with run IDs) are uploaded to Backblaze B2.

## Methodology

- **Corpus construction (Section 3.3, Table 1):** 24 kernels for the single-GPU run — 15 correct controls (numpy stand-ins for softmax/layernorm/matmul, plus Triton implementations of softmax, GELU, SiLU, RMSNorm, L2Norm, ReLU, LeakyReLU, sigmoid, tanh, ELU, matmul, attention) and 9 LLM-style buggy variants seeded with *documented* transcription errors: GELU drops the leading 0.5, softmax uses `other=0.0` instead of `-inf` (tail-mask leak), RMSNorm and L2Norm forget `sqrt`, matmul uses `acc=` instead of `acc+=` (accumulator overwrite), attention drops the `1/√D` score scale, LeakyReLU uses α=0.1 instead of 0.01, SiLU uses `sigmoid(2x)` (β confusion). The cross-GPU extension adds a flash-attention control + its buggy variant (dropped `acc·α` rescale after the max update), for 26 ops total.
- **Op-schema-aware fuzzing (3.1):** per kernel, a schema declares symbolic dims (e.g. matmul {M,K,N}, attention {B,H,S,D}) with candidate values that explicitly include boundary values; the fuzzer deterministically samples one value per dim per case via Blake2b, then materialises every input.
- **Reference oracle (3.2):** the daemon spawns a Python subprocess that reads base64-JSON inputs on stdin, computes the op in fp64, and casts back to the test dtype; the validator compares with a per-(op,dtype) absolute tolerance, runs NaN/Inf detection, and records full ErrorStats.
- **Pipeline (3.4):** vast.ai ephemeral GPU instance (image `pytorch/pytorch:2.4.0-cuda12.4-cudnn9-devel`), daemon built from source, Python client + Triton installed, P1 driver run against the corpus, results uploaded to Backblaze B2; teardown guaranteed by a context-manager exit with a label-strict reaper.
- **Stated assumptions (3.5):** the 10 buggy variants are author-seeded (not from real LLM outputs); fp64-rounded reference defines "correct" but is not guaranteed byte-identical to any library contract; tolerances fixed ahead of time; Triton kernels compiled fresh per run (no control of ptxas randomness, which the cross-GPU sweep shows doesn't change the verdict); client decodes tensors as contiguous so non-contiguous layout fuzzing is nominal at the client boundary.

## Results & Highlights

- **Single-GPU headline (RTX 3060):** 9/9 LLM-style buggy variants flagged, 15/15 controls passed, zero false positives, over ~1,440 cases (30 iterations × 24 ops × 2 dtypes).
- **Cross-GPU consistency (§4.1):** identical verdicts on five GPU classes (RTX 3060 sm86, A10 sm86, A100 SXM4 sm80, L40S sm89, H100 NVL sm90) — 16/16 controls pass, 10/10 illusions caught, including the real Triton matmul, attention, and flash-attention buggy variants.
- **Shape-dependent bug surface:** `softmax_triton_buggy` fails 13/30 under random sampling, 0/10 under regular-shapes-only, and 6/10 under boundary-only — the bug surfaces at H=3 (tail < BLOCK) and vanishes at H=256 (power of two, no tail), directly demonstrating that shape coverage drives recall.
- **Minimal failing cases:** the daemon's Minimize endpoint reduces `softmax_triton_buggy` to `[1,1,3]` fp16 and `gelu_triton_buggy` to `[2,1,1]` fp16 — shapes the fixed-shape benchmark oracle never exercises.
- **Bug taxonomy (§5):** magnitude-uniform bugs (GELU 0.5, SiLU 2x, LeakyReLU α, RMSNorm/L2Norm sqrt, attention score scale) are caught on essentially every shape — the benchmark missed them only because it tested one shape with an absolute tolerance above the constant bias; shape-dependent bugs (softmax tail mask, matmul `acc=` overwrite) are visible only with boundary-aware fuzzing.
- **Two distinct countermeasures follow:** magnitude-uniform bugs want tighter tolerances (companion paper [8]); shape-dependent bugs want better input generation (companion paper [9]).
- **Bit-for-bit reproducibility:** every failure stores a binary input snapshot that replays through the daemon with an identical verdict.

## Critical Assessment

- **Novelty — Moderate.** The individual techniques (schema-aware fuzzing, fp64 reference, per-dtype absolute tolerances, differential/seeded testing) are well established; FreeFuzz/DocTer/DeepREL/GPU-Fuzz precede this work. The novelty is in (a) the specific application to the *kernel-source* layer where LLM code generation operates, and (b) the framing of existing benchmark oracles as systematically optimistic. This is a valuable re-targeting rather than a methodological breakthrough.
- **Rigor — Mixed.** Reproducibility engineering is excellent (stored seeds, bit-for-bit replay, public repos, run IDs, deterministic Blake2b derivation, multi-GPU cross-check). However, the central empirical headline — "9/9 (then 10/10) bugs caught, 15–16 (then 16/16) controls clean" — is close to *guaranteed by construction*: the author both seeded the bugs and chose the fuzzing/tolerance strategy. This is a constructive proof that the oracle *can* be fooled and a stronger oracle *can* catch these patterns, not an independent measurement. The 5-GPU consistency result is the most genuinely informative experiment.
- **Reproducibility — High.** Public repositories, deterministic seeding, archived run IDs on Backblaze B2, CC-BY licence, explicit assumption list (Section 3.5) that invites auditing. The main unverifiable pieces are the two "in preparation" companion papers that the ablation claims depend on.
- **Significance — Moderate, conditional.** The cautionary message (fixed-shape allclose oracles certify transcription bugs as correct) is real and actionable for anyone building or trusting KernelBench/TritonBench/GEAK, and the countermeasures (boundary-aware shapes, tighter per-(op,dtype) tolerances) are low-cost. Significance will rise substantially once (a) the corpus is run against kernels *actually generated* by current LLMs rather than author-seeded, and (b) the companion calibration paper lands.
- **Red flags / things to scrutinise:**
  - Author-seeded bug corpus of only 26 ops; the bug-finding rate is therefore not a property of any deployed LLM (the paper states this explicitly and repeatedly, which is good, but the headline "9/9 caught" can still be misread as an empirical bug-detection success).
  - Two of three companion papers [8], [9] are unavailable, so the load-bearing ablation claims (boundary-value removal → zero recall; tolerance calibration methodology) are currently uncheckable.
  - The fp64-rounded reference is "not necessarily byte-identical to the kernel's chosen library contract," and tolerances are hand-set per (op,dtype) — the oracle's own correctness is asserted but calibrated elsewhere.
  - bf16 is proxied to fp16 at the Python client (NumPy limitation), and non-contiguous layout fuzzing is nominal at the client boundary, so two advertised axes of variation are effectively not exercised.
  - Single author; small corpus; no comparison against an alternative strong oracle on the same inputs.

## Strengths

- Crisp, falsifiable threat model: the paper names exactly *which three properties* of the standard oracle (one shape, one dtype, loose tolerance) it attacks, and constructs one bug per failure mode.
- Exceptional reproducibility apparatus — stored seeds, run IDs, public repos, deterministic derivation, multi-GPU cross-check, bit-for-bit replay.
- Honest, repeated disclosure that the corpus is author-seeded and the result is about oracle weakness, not LLM bug rate.
- The bug taxonomy (magnitude-uniform vs shape-dependent) maps cleanly onto two distinct, cheap countermeasures, giving benchmark maintainers an actionable checklist.
- Boundary-value insight (H=3 vs H=256) is a memorable, concrete illustration of why single-shape testing is brittle.

## Limitations

- Bug variants are hand-seeded, not extracted from real LLM-generated kernels; no measurement of how often deployed LLMs (e.g. those behind KernelBench/GEAK) actually emit these patterns at scale.
- Corpus is small (26 ops) and self-constructed by a single author.
- Two companion papers that substantiate the tolerance-calibration and input-generation ablations are "in preparation" and not yet verifiable.
- The oracle's own notion of correctness (fp64-rounded reference + hand-set absolute tolerances) is asserted, with calibration deferred to [8]; cross-dtype comparison (kernel-fp16 vs reference-fp64) is future work.
- bf16 effectively not tested (proxied to fp16); non-contiguous layout fuzzing nominal at the client boundary.
- No head-to-head against an alternative strong oracle (e.g. a cross-backend differential oracle) on the same corpus.

## Relevance

对关注"LLM 生成代码的可靠性评估 / 基准本身是否可信"这一方向高度相关：它直接质疑了 KernelBench、TritonBench、GEAK 等主流 LLM 内核基准的判分 oracle，并提供了低成本、可复现的加固方案（边界形状采样 + 按算子/数据类型校准容差），对任何做 LLM 代码生成评测或 GPU 内核验证的人都值得一看。

## Questions for Further Exploration

- When the same protocol is run on kernels *actually generated* by current frontier LLMs (GPT-class, Claude-class, open code models) on KernelBench/GEAK prompts, what fraction of "benchmark-certified correct" kernels fail the seeded oracle, and which bug families dominate?
- Does the fp64-rounded reference ever disagree with the kernel's intended PyTorch/Triton library contract in a way that produces false positives on real (correct) kernels — i.e. how tight can the per-(op,dtype) absolute tolerance be before correct-kernel recall degrades?
- How much does recall improve when cross-dtype comparison (kernel-fp16 vs reference-fp64) and true bf16 support replace the current same-dtype + fp16-proxy setup?
- Can the boundary-value strategy be generalised beyond symbolic dims to stride/layout/contiguity fuzzing without the client-side decoding limitation, and does that surface a new class of bugs?
- Is the "10/10 identical verdict across five GPU classes" result robust to ptxas/compilation non-determinism at much larger sample sizes, or does it only hold because the corpus is small and the bugs are gross?
