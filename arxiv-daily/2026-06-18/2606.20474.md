> **一句话总结：** UltraQuant 在 AMD CDNA4 (MI355X) 上用 FP4 E2M1 + UE8M0 分组缩放(配合 Walsh–Hadamard 旋转、丢弃 QJL 与 per-token 归一化)把 KV cache 压到 4-bit 并将反量化折叠进原生 scaled-MFMA 指令，在多轮 agent 负载的缓存压力后期轮次将 P50 TTFT 降低 3.47×(全轮 2.3×)、输出吞吐提升 1.63×(逼近 FP8 的吞吐上限)，但在 AIME25 上存在 −10 至 −13 pp 的明显精度回退。

> **TL;DR:** A deployment-oriented study of 4-bit KV-cache compression for long-context, multi-turn agent workloads on AMD Instinct CDNA4 GPUs. The authors present two endpoints: Ultra-TQ (an optimized TurboQuant codebook kernel with calibrated Lloyd–Max centroids and a three-tier optimization ladder) and UltraQuant (a hardware-native FP4 E2M1 + UE8M0 group-32 layout whose dequantization is folded directly into the `v_mfma_scale_f32_16x16x128_f8f6f4` instruction via a single model-agnostic scaling constant c=0.156). On a 32-concurrent-session ShareGPT multi-turn workload (MiniMax-M2.5, TP=2, MI355X), UltraQuant cuts P50 TTFT by 3.47× in cache-pressured late rounds (2.3× across all rounds) and lifts output throughput 1.63× over FP8 KV, tracking FP8's decode throughput ceiling (~1.37–1.38× BF16) at half the KV bytes. Accuracy is benchmark-dependent: stable on MATH500 and competitive on GPQA/LCB-128K, but materially regresses on AIME25 (−13.3 pp Qwen3.5-A3B, −10.0 pp MiniMax-M2.5).

## 核心贡献

- **将 4-bit KV cache 显式定位到多轮 agent 负载**：提出应把任务质量、cache 驻留性与服务吞吐联合度量，而不是只看离线压缩率或单 token 延迟。
- **Ultra-TQ**：保留 TurboQuant codebook 表示，通过 GQA 感知 tiling、structure-of-arrays KV 布局、AMD GCN intrinsics / FlyDSL JIT 三层优化梯级，以及每模型校准的 Lloyd–Max 质心(仅对 MSE 偏高的 10% 层做校准)，弥合 codebook 量化与生产服务之间的 kernel 差距。
- **UltraQuant**：用 FP4 E2M1 微张量 + UE8M0(group=32)分组幂指数缩放替代 codebook，反量化仅是一次被吸进 scaled-MFMA 指令的指数移位；FP8(E4M3) query 与 FP4 KV 在同一条 `v_mfma_scale_f32_16x16x128_f8f6f4` 路径上计算，KV 永不物化成 BF16。每元素 4.25 bit(距理想 4-bit 仅 6%)。
- **单一全局常数 c=0.156 的 constOpt 缩放**：通过一次性离线最小化 `E[(z − c·m·q(z/(c·m)))²]` 得到，模型无关、无需 per-token 归一化；消融显示它是相对 FP8 +4.4 pp 的关键(c=0.195 落平 FP8，c=1.0 反而 −4.3 pp)。
- **端到端服务实证**：在缓存压力后期(GMU=0.60)展示 TTFT/TPOT 与解码吞吐的联合收益，并诚实报告 AIME25 的精度回退而非用平均值掩盖。

## Key Contributions

- **Reframes 4-bit KV caching around multi-round agent workloads** where task quality, cache residency, and serving throughput must be measured jointly rather than as separate offline metrics.
- **Ultra-TQ**: an optimized TurboQuant (TQ4/4) codebook kernel with calibrated per-model Lloyd–Max centroids (applied to only the ~10% of layers with higher per-element MSE), GQA-aware tiling, SoA KV layout, and a three-tier implementation ladder (improved Triton → native GCN-ISA intrinsics → FlyDSL JIT).
- **UltraQuant**: replaces the codebook with a hardware-native FP4 E2M1 micro-tensor layout plus UE8M0 group-32 power-of-two scales, so dequantization (`value = code · 2^scale`) folds directly into CDNA4 `MFMA_SCALE_F32_*_F8F6F4` instructions; queries are Hadamard-rotated and cast to FP8 E4M3 so the QK and PV products run on the same FP8×FP4 scaled-MFMA path. Cost: 4.25 bits/element (within 6% of ideal 4-bit).
- **Constant-optimized scaling**: a single model-agnostic global constant c=0.156 (minimizing reconstruction MSE on rotated-unit-vector distribution), which subsumes TurboQuant's per-token ℓ2 K-norm with no extra runtime multiply or per-token state. Ablation shows it is the only constant beating FP8 (+4.4 pp).
- **Practical design decisions made explicit**: asymmetric K/V treatment (KIVI/KVQuant lineage), Walsh–Hadamard rotation (essential, not substitutable by the codebook — un-rotated centroids leave ~2.3× the rotated MSE), QJL removal, block-scale variants, and boundary-layer protection (first + last 2 attention layers kept in BF16).
- **End-to-end serving evidence** on AMD MI355X with honest reporting of accuracy regressions (AIME25) rather than averaging them away.

## Methodology

**Problem framing.** Context-heavy agents maintain long reusable prefixes (system prompts, tool defs, retrieved docs, codebase context) across many short turns; concurrency determines whether GPUs stay utilized. Compression matters because it increases the number of useful prefixes that remain resident, but dequant overhead can negate latency gains — so end-to-end TTFT/TPOT must be measured.

**Ultra-TQ (codebook path).** Starts from TurboQuant's observation that raw KV is a poor scalar-codebook target, but a fixed Walsh–Hadamard rotation whitens outlier energy toward a near-Gaussian / Beta((d−1)/2,(d−1)/2) regime where a 16-entry 4-bit Lloyd–Max codebook works. Calibrated centroids (refit on ~20 captured rotated vectors per layer) lower K-quantization MSE by 10.3% and improve GPQA by +1.20 pp vs the theoretical centroids. Implementation follows FLUTE-style offline restructuring + GQA-aware tiling + AMD MFMA dispatch across three tiers (Triton → GCN intrinsics → FlyDSL).

**UltraQuant (FP4 path).** Stores K and V in an FP4–UE8M0 group-32 layout: head dim partitioned into contiguous 32-channel groups, each = 32 FP4 E2M1 codes (packed 2/byte) + 1 UE8M0 scale byte (17 B per 32 channels). Dequant `value = code · 2^(scale_byte − 127)` is a pure exponent shift consumed by the matrix core. Group scale s = c·m where m = per-group absmax and c = 0.156 is one global constant fit offline by minimizing reconstruction error of the E2M1 grid on captured rotated keys; the per-group absmax absorbs all per-token/per-head/per-model magnitude variation, so no per-token K-norm is needed (and a per-token factor cannot pass through softmax anyway). Boundary layers (first + last 2) kept BF16.

**Evaluation harness.** Accuracy: 3 models (Qwen3.5-A3B, MiniMax-M2.5, Qwen2.5-72B) × 4 benchmarks (GPQA-Diamond, LCB-128K, AIME25, MATH500). Serving: vLLM native multi-turn ShareGPT benchmark, 32 concurrent sessions, MiniMax-M2.5, TP=2, 2×MI355X, 32K/1K, plus a decode sweep at concurrency 64 and inter-token latency sweep 8K–64K. Two GMU operating points (0.60 main, 0.65 low-pressure) are reported.

## Results & Highlights

**Serving (vs FP8 KV baseline, MiniMax-M2.5, TP=2, MI355X):**
- P50 TTFT, late rounds (r4–6): **3.47×** improvement
- P50 TTFT, all rounds: **2.3×**
- Output throughput (agentic): **1.63×**
- Decode throughput at C=64: **1.38×** BF16, matching hardware FP8 KV (1.37×) within ~1%, at half the KV bytes/element
- Median TPOT: 1.40× BF16 (vs FP8 1.37×, Ultra-TQ 1.58×, vLLM OSS TurboQuant 5.56×)
- Inter-token latency vs context length improves monotonically relative to BF16, reaching ~0.5× at 64K (FP8 stays ~1.3–1.5×)

**Accuracy (UltraQuant vs BF16):**
- MATH500: stable (+0.80 Qwen3.5-A3B, +0.00 MiniMax-M2.5)
- GPQA-Diamond: +1.52 (Qwen2.5-72B), −2.02 (MiniMax-M2.5), +0.00 (Qwen3.5-A3B)
- LCB-128K: −2.47 (Qwen3.5-A3B), −4.39 (MiniMax-M2.5)
- **AIME25: −13.33 (Qwen3.5-A3B), −10.00 (MiniMax-M2.5), −3.33 (Qwen2.5-72B)** — flagged as a real limitation

**Ablations (single seed, 1σ≈1.7 pp on GPQA):**
- Per-block absmax scale (group-32) is the load-bearing change; with it in place, Lloyd–Max vs uniform codebook is within noise (0.6559 vs 0.6528). Justifies dropping the calibrated codebook for the fixed FP4 grid.
- Global constant sweep: c=0.156 → +4.4 pp over FP8; c=0.195 → 0.0 pp; c=1.0 (raw absmax) → −4.3 pp. Accuracy monotonic in distance from MSE-optimal c.
- Cache-pressure regime: UltraQuant's residency advantage is largest at GMU=0.60; at GMU=0.65 FP8 no longer degrades and all three schemes track closely.

## Authors & Credibility

- All seven authors are from **Advanced Micro Devices (AMD)**, with three equal-contribution first authors affiliated with UCLA (Chakrabarti) and Purdue (Limpus). This is a vendor paper advocating a format native to AMD's own CDNA4 ISA — a credibility strength (deep hardware/ISA access, real MI355X measurements, vLLM integration) but also an inherent commercial alignment.
- Builds directly on TurboQuant (Zandieh et al., ICLR 2026), KIVI, KVQuant, QJL, and FLUTE — well-grounded in the established KV-quant literature rather than a from-scratch method.
- Acknowledgements cite AMD colleagues and the AMD Quark Team; related prior AMD blog work (Chakrabarti et al., 2026 "Productionizing TurboQuant") is self-cited, indicating continuity of a real engineering effort.

## Critical Assessment

- **Novelty: moderate.** The core algorithmic moves (Hadamard rotation + 4-bit + asymmetric K/V + per-group scales) are well established by TurboQuant/KIVI/KVQuant. The genuine novelty is the **engineering insight** — that a slightly suboptimal but hardware-native FP4 grid with a single global constant can beat an analytically optimal codebook end-to-end because dequant is folded into the MFMA. This is a legitimate and useful contribution, but it is a systems/hardware-co-design result, not a new quantization algorithm.
- **Rigor: mixed.** Serving numbers are thorough (per-round TTFT/TPOT, two GMU regimes, context-length sweep, throughput at multiple concurrencies). But accuracy ablations rely on **single-seed** runs with stated σ≈1.7 pp, so several claimed deltas (e.g., +1.52 GPQA, the Lloyd-vs-uniform 0.31 pp gap) are within noise. The AIME25 numbers are on very small item counts (Qwen2.5-72B AIME25 = 16.67% suggests only 6 items), making point estimates noisy.
- **Baseline concerns.** The headline 3.47× TTFT gain is **conditional on cache pressure (GMU=0.60)**; the paper itself shows the advantage largely disappears at GMU=0.65. This is disclosed (Limitation #2) but the abstract/headline framing emphasizes the large number. The FP8 baseline is vLLM's, which is fair, but no comparison to other 4-bit serving paths (e.g., INT4 with optimized kernels, or NVIDIA-side equivalents) is possible since everything is AMD-specific.
- **"Context-heavy agents" framing vs evidence.** The paper motivates itself around agents (SWE-agent, WebArena, ReAct, Toolformer citations) and claims task quality + residency + throughput must be measured jointly — yet **no actual agent task-quality metric is reported**. The serving proxy is ShareGPT multi-turn chat, and accuracy is measured on static benchmarks (GPQA/AIME/MATH/LCB). The "joint task-quality" claim is not demonstrated; this is an overclaim relative to the framing.
- **Hardware portability is a real red flag for generality.** UltraQuant is intrinsically tied to CDNA4's `MFMA_SCALE_F32_*_F8F6F4` and UE8M0 scale support; it is not portable to NVIDIA or older AMD parts. The result is valuable but narrow.
- **Honest disclosure is a positive signal.** The authors explicitly table the AIME25 regression and the GMU-dependence rather than hiding them behind averages — a credibility plus.

## Strengths

- Clean, deployment-first framing that measures TTFT/TPOT/throughput together rather than only MSE or only compression ratio.
- The constOpt insight (one global c=0.156 subsumes per-token K-norm and transfers across models) is elegant and well-ablated.
- Folding dequant into the MFMA scale operand — eliminating the irregular codebook LUT gather that plagues TurboQuant serving — is the key practical win.
- Honest, granular reporting of accuracy regressions and cache-pressure sensitivity.
- Strong systems engineering: three-tier kernel ladder (Triton → GCN intrinsics → FlyDSL) and integration into vLLM.

## Limitations

- **AIME25 regression of −10 to −13 pp** on the two main models is not minor for a "near-lossless" 4-bit story; the authors concede the accuracy profile is "benchmark-dependent rather than uniformly near-lossless."
- **Single-seed accuracy ablations** leave several deltas within noise.
- **Speed benefit only materializes when context exceeds FP8's resident-cache capacity** — at lower pressure (GMU=0.65) or shorter contexts, FP8 matches or beats it.
- **AMD/CDNA4-only**; no portability to NVIDIA or pre-CDNA4 hardware.
- **No real agent task-quality evaluation** despite the agent framing (ShareGPT chat ≠ SWE-bench/WebArena agent success).
- Single global constant with no layer-wise calibration left as future work; centroid calibration only helps ~10% of layers.

## Relevance

与 LLM 推理/服务优化高度相关：4-bit KV cache 是长上下文 agent 与高并发部署的关键瓶颈，本文给出了在 AMD 硬件上把反量化折进矩阵核、实现 FP8 级吞吐但半数显存占用的可落地路径；对关心 agent 长时记忆驻留、推理吞吐与显存压力的工程读者有直接参考价值，但其结论(尤其 AIME25 精度回退与对缓存压力的强依赖)提示在精度敏感任务上仍需谨慎。

## Questions for Further Exploration

- Why does AIME25 regress so sharply (−10 to −13 pp) while MATH500 stays flat? Is it the small item count, a specific reasoning-chain sensitivity to FP4 value error, or interaction with boundary-layer protection?
- Would layer-wise or head-wise calibration of c recover the AIME25 gap without breaking the "single global constant" simplicity?
- How does UltraQuant behave on real agent benchmarks (SWE-bench, WebArena, τ-bench) where task success — not just TTFT — is the metric the paper claims to care about?
- Can the FP4+UE8M0+constOpt recipe transfer to NVIDIA Blackwell FP4 paths, and does the c=0.156 optimum hold across ISAs?
- The paper drops QJL and per-token K-norm; is there a learned (non-Hadamard) rotation that does better under the same FP4 grid and agent benchmark?
- At what context length / concurrency does UltraQuant's advantage over FP8 cross over in production (the GMU=0.60 vs 0.65 result suggests this is a sharp threshold)?
- Is the single-seed accuracy noise (σ≈1.7 pp) hiding real gains on GPQA/LCB, or is the +1.52 pp Qwen2.5-72B GPQA number essentially luck?
