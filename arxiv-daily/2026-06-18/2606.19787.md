> **一句话总结：** 作者提出 ORAgentBench——首个面向真实端到端运筹优化（OR）任务的执行型评测基准（107 个人工审核任务，沙箱环境+隐藏校验器，三阶段评分），并评测 14 个前沿智能体配置；最强配置（GPT-5.4）仅通过 35.51% 全部任务、20.59% 困难任务，揭示当前智能体"建模策略"是主要瓶颈而非求解器算力，OR 领域专用技能可改善困难任务可行性但难以稳定提升质量与通过率。

> **TL;DR:** ORAgentBench is the first execution-grounded benchmark for end-to-end operations-research agents: 107 human-reviewed, sandbox-isolated tasks where agents must read multi-file operational artifacts, write and run solution code, and produce decision artifacts judged by hidden validators on schema validity, hard-constraint feasibility, and normalized objective quality. Across 14 frontier model-agent stacks, the best (GPT-5.4) passes only 35.51% of tasks (20.59% of hard tasks); 54.8% of failures are modeling-strategy errors, and OR-specific skills improve hard-task feasibility but not reliably quality or pass rate.

## 核心贡献

- **首个端到端 OR 智能体基准**：提出 ORAgentBench，107 个可执行任务，将"运筹研究（OR）"完整工作流（业务文档 → 数据/配置 → 建模 → 求解 → 校验后的决策产物）纳入评测，弥补现有基准"只测求解器"或"只测公式生成"的割裂。
- **任务形式化**：定义任务 τ = (D, C, q, E, V)，即（数据、配置、自然语言需求、沙箱环境、隐藏校验器），并显式刻画"建模策略 ϕ"与"求解策略 ψ"双重隐式决策，区分"能写出公式"与"能交付高质量运营决策"。
- **三阶段隐藏校验协议**：输出 schema 合法性 → 硬约束可行性（可行性为硬门，F=0 即零分）→ 相对参考解 R 与全局界 B 归一化的目标质量 q；通过定义为 F>0 且 q>0.4。
- **六维难度构造准则**：求解策略、公式结构、约束耦合、动态结构、数据规模、问题理解，避免"规模"误代"难度"。
- **大规模诊断实验**：14 个前沿配置（Claude Sonnet/Opus 4.6、GPT-5.3-Codex / GPT-5.4 / GPT-5.4-mini、DeepSeek V4、GLM 5/5.1、Qwen3.5/3.6、Kimi K2.6、MiMo V2.5、MiniMax-M2.7），分析可行性与质量差距、效率/成本权衡、失败模式与技能增强。
- **关键发现**：失败主要由"建模侧策略错误"（54.8%）而非求解器算力或基础设施导致；OR 专用技能会重塑失败分布（提升困难可行性）但非整体提升，甚至会带来新的 pass→fail。

## Key Contributions

- First end-to-end OR agent benchmark: 107 executable, human-reviewed, sandbox-isolated tasks that test the full artifact-to-decision workflow absent from solver-centric (MIPLIB/CVRPLIB) and modeling-only (NL4Opt, IndustryOR) benchmarks.
- Formal task/agent formulation τ=(D,C,q,E,V) with explicit dual latent strategies (modeling ϕθ,τ, solving ψθ,τ), distinguishing plausible-formulation agents from dependable OR agents.
- Three-tier hidden validator (schema validity → hard-constraint feasibility → normalized objective quality), with a feasibility hard gate and a 0.4 quality pass threshold.
- Six construction-time difficulty dimensions to prevent instance scale from masquerading as difficulty.
- Empirical evaluation of 14 frontier stacks plus a skill-enhancement ablation, with failure-mode and scenario-family diagnostics.

## Methodology

**Task definition & dual-strategy framing.** A task τ = (D, C, q, E, V). Given public packet Iτ=(D,C,q), agent Aθ produces program pθ=Exec outcome s=Exec_E(pθ;D,C), scored Score(θ,τ)=V(s). The program is framed as driven by two latent coupled choices: modeling strategy ϕθ,τ ∈ Φ and solving strategy ψθ,τ ∈ Ψ. This dual framing is the conceptual core: realistic OR requires joint design of an effective problem representation and an efficient computational procedure.

**Design principles.**
- P1: Difficulty should come from modeling+solving burden, not instance size. Controlled prospectively along six dimensions (solving strategy, formulation structure, constraint coupling, dynamic structure, data scale, problem understanding).
- P2: Scenarios operationally grounded and automatically verifiable—explicit objective, real resource constraints, all info self-contained in artifacts, deterministic validation, no live APIs/unstated domain knowledge. Prediction/simulation/open-ended consulting excluded.
- P3: Data = realistic multi-file business artifacts (not normalized model instances); numerical values and tightness calibrated to avoid trivial solutions.

**Task construction pipeline (4 stages):** (S1) scenario selection + difficulty framing from OR papers/benchmark seeds/real scenarios; (S2) human-in-the-loop task & data enrichment (instruction, task.toml, Dockerfile, data/, calibrated realistic rules); (S3) agent-assisted validation loop—agent explores env, builds deterministic validator, solves using only public info, checks submission, diagnoses and revises until valid reference solution + reliable validator exist; (S4) human review of data validity, validator correctness, reference quality, instruction clarity, leakage/shortcut/anti-cheating risks. Only passing tasks enter the benchmark.

**Scoring protocol.** V(τ,s) = (F(τ,s), q(τ,s)). F ∈ [0,1] feasibility (hard gate; F=0 ⇒ zero credit). For feasible submissions, quality normalized against verified reference objective R and (if available) global bound B: q = clip(1/2 + gτ(s)/(2·Wτ), 0, 2), with reference-to-bound width Wτ (e.g. 0.05%); tolerance-scaled fallback when no meaningful bound. Pass(τ,s) = 𝟙[F>0 ∧ q>0.4].

**Experimental setup.** 14 model-agent rows. All evaluation in Harbor-based isolated environments, no internet; common optimization stack includes SCIP. Per-agent budget: 45 min interaction + 5 min solve. Reasoning-effort = high. All agents initialized with the same four default OR-base skills.

## Results & Highlights

- **Best stack is far from reliable.** GPT-5.4 (Codex harness) = 35.51% overall pass; GPT-5.3-Codex = 32.71%; Claude Opus 4.6 (Claude Code) = 34.58%. Best hard-task pass only 20.59% (GPT-5.4 and GPT-5.3-Codex). All other completed stacks < 35%.
- **Feasibility–quality gap.** Feasibility consistently exceeds pass rate: agents often produce schema-valid, constraint-feasible submissions that still fall short of the quality threshold — the central diagnostic finding.
- **Failure is strategy, not infrastructure.** 54.8% of non-passing trajectories fail on modeling-side errors (missing/inconsistent/incorrectly encoded operational rules, brittle/oversized formulations). Weak-solver, timeout, and formulation-efficiency errors trail behind. Per-model breakdown varies: some agents miss rules, some burn budget without safeguarding feasibility, stronger agents shift the bottleneck to solution improvement.
- **Quality-threshold sensitivity (Fig 4).** At q=0.4 several agents cluster, but curves diverge at stricter thresholds — separating agents that barely clear the bar from those with real quality margins.
- **Efficiency–quality trade-off (Fig 5).** Pass rate is non-monotonic in runtime and API cost; the best stack is neither slowest nor most expensive. More tokens/time help only when spent on data checking, feasible-model construction, targeted improvement.
- **Scenario-family variation (Fig 7).** Transportation is among the hardest families; software/telecom much easier. Difficulty tracks induced modeling structure (continuity, recourse, sparse feasibility, multi-period coupling) more than domain label.
- **Skills reshape, not uniformly improve (Table 3).** Base Skills (generic) mainly raise feasibility; easy-task pass actually drops 15.62 pts while hard gains only +2.94 pts. Expert Skills (domain-written) raise hard-task feasibility (+0.11) and pass (+3.85 pts) but reduce easy/medium. Paired analysis: Expert Skills recover some infeasible runs but introduce more pass→fail than fail→pass conversions — descriptive, not variance-controlled causal evidence.

## Critical Assessment

- **Novelty — high.** Genuinely fills a gap: prior OR benchmarks either pre-formalize instances (MIPLIB, CVRPLIB) or stop at formulation/code generation (NL4Opt, IndustryOR, OptMATH, MIPLIB-NL, CO-Bench). The artifact-to-validated-decision scope with multi-file inputs, sandbox execution, and OR-style feasibility+quality validation is, as claimed, the first of its kind in this combination (Table 1).
- **Rigor — solid but with caveats.** Strengths: hidden validators, hard feasibility gate, normalized quality against verified references/bounds, six-dimensional difficulty rubric, leakage/anti-cheating human review. Caveats: only 107 tasks (modest N for cross-family claims); 8 scenario families means some families have very few tasks (Fig 7 right shows uneven distribution), so per-family pass rates carry large uncertainty; skill ablation is single-run per condition — explicitly flagged as descriptive, not variance-controlled.
- **Reproducibility — good on paper, needs verification.** Code/data promised; Docker images, task.toml, validators described in detail. But the "verified reference objective R and global bound B" for each task is the crux — quality scores are only as meaningful as these references, and the paper does not fully expose how bounds were obtained or how sensitive q is to Wτ choice (e.g. the 0.05% example). A sensitivity analysis on Wτ would strengthen claims.
- **Significance — high for the agent-evaluation subfield.** The headline finding (modeling-strategy errors dominate; skills help feasibility not quality) is actionable and likely to steer OR-agent research toward formulation/solving co-design rather than solver brute force.
- **Red flags / concerns.**
  1. **Frontier-model naming is fictional/future-dated** (GPT-5.4, Claude Opus 4.6, GLM 5.1, DeepSeek V4, Kimi K2.6, Qwen3.6 Plus, MiMo V2.5, MiniMax-M2.7) — readers should treat absolute numbers as illustrative of a June-2026 snapshot, not stable baselines.
  2. **Validator-as-ground-truth risk**: if a hidden validator encodes an intended formulation, agents that find an equally-valid but differently-structured solution could be mis-penalized; the paper asserts "deterministic" validation but doesn't show robustness to formulation equivalence.
  3. **Coverage bias**: 107 tasks drawn partly from OR papers and benchmark seeds risks overlap with training data of frontier models; the human-review "leakage" check is described but not quantified (no contamination statistics).
  4. **Skill ablation interpretation**: Expert Skills reducing easy/medium pass while raising hard is interesting, but with single runs and no paired variance, "pass→fail > fail→pass" could partly be noise.
  5. **No learning/feedback**: pure zero-shot-style single-pass evaluation; real OR practice is iterative with human feedback, so "end-to-end" here still omits the human-in-the-loop correction loop that the paper itself motivates with in Section 1.

## Strengths

- Conceptually clean dual-strategy framing (ϕ modeling + ψ solving) that explains *why* formulation-only benchmarks are insufficient.
- Three-tier scoring with feasibility hard-gate cleanly separates "submitted something valid" from "submitted a high-quality decision" — the feasibility-quality gap is the most useful diagnostic.
- Six-dimensional difficulty rubric prevents the scale-as-difficulty confound that plagues solver benchmarks.
- Genuine execution grounding: isolated sandbox, multi-file artifacts, hidden validators, anti-cheating review — closer to real OR engineering than any prior OR-LLM benchmark.
- Honest negative results and skill-ablation nuance (skills help feasibility, not quality) rather than overselling.

## Limitations

- Small N (107 tasks) and uneven scenario-family distribution weaken per-family generalization claims.
- Quality score depends on reference objective R and bound B whose derivation and sensitivity to Wτ are under-documented.
- Single-run skill ablation; no variance/seed control for the central RQ4 claim.
- Frontier model names/versions are a moving target; absolute numbers will not age well.
- No iterative/human-feedback setting, despite motivation emphasizing real OR's revision loop.
- Possible train/test contamination from OR-paper-derived tasks is checked qualitatively but not quantified.
- Validator may penalize valid-but-differently-formulated solutions; formulation-equivalence robustness not shown.

## Relevance

对 LLM 智能体研究的直接相关度很高：
- **智能体可靠性**：核心发现"可行但低质量"与"建模侧策略错误占主导"直击当前 coding agent 的可靠性短板——agent 能写出看似合理的优化代码，却无法稳定交付高质量决策。这对所有需要"可执行+可校验"输出的智能体（含 SWE、科学计算、数据分析 agent）都有方法论借鉴。
- **工具使用与执行环境**：沙箱执行 + 隐藏校验器 + schema/约束/质量三段式评分，是一种可迁移到其他"产物型"智能体评测（如数据科学、自动化科研）的评测范式。
- **推理与规划**：双策略（建模 ϕ + 求解 ψ）框架本质上刻画了"问题表征选择"与"计算过程选择"的耦合规划，与 LLM 推理、test-time search、self-refine 的研究方向呼应。
- **可解释性/诊断**：按"首个失败阶段"做失败模式归因（建模错误 / 弱求解器 / 超时 / 公式效率）是一种细粒度可解释诊断，优于单一通过率。
- **技能/工具注入**：RQ4 的"技能重塑而非整体提升"结果，对 prompt/技能工程、agent 工具增强、RAG 式领域知识注入的研究有警示意义——领域知识不是万能，甚至可能引入新的失败模式。

## Questions for Further Exploration

- 质量阈值 q=0.4 与归一化窗口 Wτ（如 0.05%）的选择如何影响模型排名？是否存在对某一类公式化结构有利的系统性偏置？
- 隐藏校验器如何处理"形式不同但等价/更优"的解？是否需要 formulation-equivalence 鲁棒性测试？
- 107 任务中来自 OR 论文/已有 benchmark seed 的比例多大？能否量化前沿模型对这些来源的训练数据污染？
- 若引入迭代式 human-in-the-loop 修正（贴合 Section 1 动机），feasibility-quality 差距会如何变化？多轮交互能否把建模侧错误比例从 54.8% 显著压低？
- OR 技能为何在简单任务上反而降分？是技能本身误导、还是 agent 在简单场景下过度遵从流程而牺牲了直接求解？能否设计难度自适应的技能触发机制？
- 失败模式中"弱求解器"与"建模错误"的边界：当一个 agent 建模正确但求解器选择差，应如何区分以指导后续 agent 设计（更强的求解器调度 vs. 更强的建模验证）？
- 能否把双策略 (ϕ, ψ) 显式化为可观测中间产物（如要求 agent 输出 Model.md 再输出 code），从而分别评分并定位改进点？论文 Figure 1 已暗示这种分离但未在评分中落地。
- 跨场景家族的难度差异（运输最难、软件/电信最易）是否主要源于约束耦合与可行域稀疏性？能否构造受控消融来分离"领域知识"与"结构复杂度"的贡献？
