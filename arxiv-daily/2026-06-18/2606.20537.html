Execution-State Capsules: Graph-Bound Execution-State Checkpoint and Restore for Low-Latency, Small-Batch, On-Device Physical-AI Serving

arXiv: 2606.20537

Date: 2026-06-18

https://arxiv.org/abs/2606.20537

一句话总结： FlashRT 提出第三种受管服务对象——"执行态胶囊"（execution-state capsule），通过把整条前向计算捕获为基于连续静态缓冲区、无 block-table 间接寻址的 CUDA Graph，把任意已提交 token 边界的完整执行状态（含线性注意力递归态/卷积态/MTP 尾）冻结成一个可一次性拷贝恢复的闭包，使会话的恢复/分叉/回滚从"按 token 重算前缀"变成"带宽受限的状态拷贝"，在 RTX 5090 上把冷启动 TTFT 压到 vLLM 的 1/2.6–1/2.8，冷→胶囊加速比随前缀长度从 3.9× (2k) 增长到 27× (16k)，在 Jetson Thor 上更达 9–76×。
TL;DR: This paper introduces FlashRT, a latency-first white-box kernel runtime plus a "third managed object for serving systems" called the execution-state capsule. By capturing the whole forward pass as a graph plan over contiguous static buffers with no block-table indirection, the complete continuation state at any committed token boundary becomes a fixed, named buffer set that can be frozen, restored, forked, and rolled back as a single bandwidth-bound byte copy — turning prefix reuse from a compute-bound recompute into a copy-and-rebuild. On an RTX 5090 the cold TTFT is 2.6–2.8× lower than vLLM's, the capsule TTFT (1.4–2.8× below vLLM's automatic prefix cache) stays flat at ~51–57 ms while cold prefill grows 200→1541 ms over 2k–16k, and KV-only restore is shown to diverge (97.9% mismatch at token 1), proving the recurrent fold is load-bearing.

核心贡献

  • 第三种受管服务对象。 与 vLLM 的 PagedAttention（按 position 索引的 KV 页）和 SGLang 的 RadixAttention（按前缀索引的 KV 子树）并列，FlashRT 提出按"提交边界"索引的"图绑定执行态"作为第一类对象，把服务的复用单位从"token 寻址的 KV 片段"移到"图绑定执行态边界"。
  • 单设计选择，双后果。 关键论点：使用连续静态缓冲区 + 整前向图回放、取消 block-table 间接寻址，这一个选择同时（a）让 kernel 快（无 gather、字节级一致回放、无 per-step launch/Python 开销）和（b）让前向状态可整体冻结。延迟优先执行与胶囊状态管理是同一选择的两个推论，不是两个独立特性。
  • 四动词原语。 snapshot / restore / fork / rollback 都是对边界缓冲闭包的一次字节拷贝 + 已捕获图的回放，永不重算前缀、永不重捕获图。
  • 正确性包络（三层）。 字节恢复、状态完备性、token 等价。识别出分块线性注意力递归扫描的 chunk-alignment 条件：边界必须对齐到 prefill chunk size C 的倍数才能 token 级精确复用。
  • 一个契约跨三域。 同一个最小 C ABI（三种 handle 类型 + 一个不透明 shape key + 严格的 mechanism-not-policy 边界）表达：LLM 编码 agent 的热启动、机器人 RL rollout 的 episode reset、planner-actor 零拷贝交接。
  • 三架构复现。 在 sm_120 (RTX 5090)、sm_110 (Jetson AGX Thor)、sm_121 (DGX Spark/GB10) 上复现 LLM 正确性与结构性质；Thor 上冷→胶囊加速比因冷 prefill 更贵而扩大到 9–76×。

Key Contributions

  1. A latency-first single-stream runtime substrate (FlashRT's NVIDIA CUDA backend) whose hot path is a single captured graph replay over contiguous static buffers with no block-table indirection — a measured low-latency execution floor: cold TTFT 2.6–2.8× below vLLM on the identical hybrid model/GPU, with a tight tail (p99 within ~0.5 ms of p50).
  2. The execution-state capsule: the complete restorable state at a committed boundary frozen as a fixed set of named device buffers, with four verbs (snapshot/restore/fork/rollback) and a cost model that turns prefix reuse from compute-bound recompute into bandwidth-bound copy-and-rebuild.
  3. A three-layer correctness envelope (byte restore, state completeness, token equivalence) plus the identification of the chunk-alignment condition under which exact reuse of a chunked linear-attention recurrent scan holds.
  4. Regime evidence: the runtime floor, the capsule's gain over FlashRT's own cold path, and the retention-control gain over an automatic prefix cache (APC) under an embodied working set, on three architectures.

Methodology

  • 受管对象视角。 论文用一个清晰的概念轴组织全文：服务系统由它"当作受管状态对象"的东西定义。vLLM 管 KV 页、SGLang 管 KV 前缀树、FlashRT 管整个执行态边界。复用单位是核心区分。
  • 静态缓冲区图捕获。 FlashRT 是白盒、面向后端的 kernel 运行时（本文评估 NVIDIA CUDA 后端），不是编译器或图重写器。内存 bound 的算子（norm、activation、fused residual/norm/quant、带 RoPE 的 QKV split）是手写 kernel；GEMM 是 cuBLASLt/CUTLASS FP8/NVFP4 的薄封装。整前向被捕获为图 plan，热路径上单次 cudaGraphLaunch（契约同样允许具名子图 DAG）。
  • 胶囊定义。 胶囊冻结位置 P 的提交边界为一个固定的命名缓冲集：小型定长部分（线性注意力递归态、卷积态、MTP 尾及其压缩 cache、last hidden 作为 MTP seed、边界元数据如 cur_pos 与 token-prefix digest）+ KV 区（[0,P) 的 full-attention KV 及长上下文 FP8 反量化 stage 的 valid end）。只在有意义边界（pinned 共享前缀、episode 起点、turn 边界）取样，绝不在密集 token 网格上。
  • 代价模型。 令 L 为共享前缀长度：冷 turn 付 T_prefill(L+s)（计算 bound，∝ 前缀长度）；胶囊 turn 付 T_restore(L)+T_append(s)（带宽 bound，Θ(L) 字节拷贝）。收益源于计算 vs 带宽不对称 T_prefill(L) ≫ T_restore(L)，且随 L 增大差距扩大。不声称 restore 是 O(1)。
  • 执行契约。 最小 C ABI（188 行头文件的节选）：frt_buffer（唯一状态原语）、frt_graph（ShapeKey→captured graph-exec 表，可捕获或 adopt 外部图）、frt_graph_bind/replay。机制非策略硬边界：契约永不感知 session、KV append/fork/evict 或调度器。胶囊只往契约加一个机制——host-backed buffer + 跨空间异步拷贝，使其可停泊到 GPU 之外。
  • 正确性三层。 (1) 字节恢复；(2) 状态完备性——对抗式测试：snapshot 后把每个 live buffer 用无关 prompt 完全覆盖并 decode，再 restore 并 decode，输出须与从未跑过另一 prompt 时 token 完全一致；(3) 端到端等价——restore 后贪心 decode 与被替换路径 token 完全一致。报告 token 级等价，不单独声称字节级 logits 一致。
  • chunk-alignment 条件。 长分块线性注意力 prefill 按块折叠递归态，位置 N 的状态依赖块边界落点。冷全长 prefill F 把边界放在 prefill chunk size C 的倍数。若胶囊/append 边界 P 不是 C 的倍数，append 引入冷 prefill 没有的块分裂，二者在 FP8 舍入下分叉。修复：在 P'=⌊P/C⌋·C 对齐边界取样。

Results & Highlights

Runtime floor（RTX 5090，4096-token 前缀，paper-TTFT 约定=首 base-logit token，MTP tail 排除）：

  • 冷 TTFT 366.8 ms，p99 367.2 ms（p90−p50 < 0.5 ms，紧尾）；peak GPU 22.8 GB。
  • 胶囊 TTFT 53.0 ms（p99 53.1 ms）；胶囊占 224 MB。

胶囊机制与随前缀长度缩放（Table 7）：

| prefix | capsule | snapshot | restore | append | cold TTFT | capsule TTFT | speedup | token-exact |

|--------|---------|----------|---------|--------|-----------|--------------|---------|-------------|

| 2048 | 160 MB | 0.3 ms | 0.3 ms | 25 ms | 200 ms | 51 ms | 3.92× | yes |

| 4096 | 224 MB | 0.4 ms | 0.4 ms | 26 ms | 365 ms | 53 ms | 6.91× | yes |

| 8192 | 352 MB | 0.7 ms | 0.7 ms | 28 ms | 723 ms | 54 ms | 13.33× | yes |

| 16384 | 608 MB | 1.2 ms | 1.2 ms | 28 ms | 1541 ms | 57 ms | 26.94× | yes |

snapshot/restore 亚毫秒，append 平在 ~25–28 ms，胶囊 TTFT 平在 51–57 ms，冷 TTFT 随长度单调上升。

KV-only 消融（Table 8，4096 边界，贪心 K=0）： 全态恢复 token-exact（0/48 不匹配）；仅恢复 KV、丢递归 fold → token 1 即分叉（47/48 = 97.9% 不匹配）；仅恢复 KV、递归 fold 过期 → token 3 分叉（45/48 = 93.8%）。证明递归态是承重的，位置 KV 单独复用在此架构下是错的，不是慢。

vs vLLM 0.22.0（同模型/GPU，绝对 TTFT ms，Table 9）：

| prefix | vLLM cold | FlashRT cold | vLLM APC | FlashRT capsule |

|--------|-----------|--------------|----------|-----------------|

| 2048 | 519 | 200 | 143 | 51 |

| 4096 | 1026 | 365 | 76 | 53 |

| 8192 | 2057 | 723 | 120 | 54 |

无复用时 FlashRT 冷比 vLLM 冷低 2.6–2.8×；有复用时胶囊比 vLLM-APC 低 1.4–2.8×。vLLM-APC 在工作集 > ~16k token 时缓存崩塌回冷（519–2057 ms miss 路径）。

跨设备（Jetson AGX Thor sm_110 与 DGX Spark GB10 sm_121，Table 12/13）： 冷→胶囊加速比 Thor 为 9–76×、Spark 为 5–33×（冷 prefill 在边缘设备以秒计：Thor 2.2–17.9 s，Spark 0.9–6.6 s），而胶囊 TTFT 始终 ~0.2 s。vLLM-APC 在 Thor/Spark 的大统一内存上不崩塌（无驱逐压力），故复用收益是绝对更低的胶囊 TTFT。

机器人侧： FP8 advantage-conditioned π0 风格扩散策略，rollout 边界快照成 640 字节契约 Buffer，restore 后回放捕获图逐字节复现动作（cosine 1.000000），包括在 live 边界 buffer 被无关 episode 蓄意覆盖之后。planner→actor 交接（两个共驻 Pi05 策略，1:4 多速率）中运行中中断注入新 subgoal 覆盖共享 buffer，下一 actor tick 无重捕获地消费。

Higgs TTS 运行时健全性检查（非胶囊实验）： RTX 5090 单流，FlashRT bf16 TTFA 139 ms vs sglang-omni ~358 ms（2.6×），fp8 达 94 ms（3.8×）；流式输出 vs 一次性 cosine 1.0；匹配精度下 per-frame RTF 持平。

Critical Assessment

  • 新颖性（中-高）。 "第三种受管服务对象"的提法在概念上有清晰增量：把复用单位从 token-寻址 KV 片段移到图绑定执行态边界，并把递归/卷积态作为一等可复用对象暴露——这是 block/radix cache 结构上没有的（它们把 mamba/gated-delta-net 状态放在独立、标记 experimental 的旁路 cache 里）。但底层技术（CUDA Graphs、连续 KV、状态快照/恢复）各有先例（vAttention 连续 KV、CRIU 进程检查点、Hydragen/Prompt Cache 共享前缀）。真正的增量是"把整前向捕获成自包含静态缓冲图，从而把状态变成可冻结一等对象"这一视角翻转及其在混合线性/全注意力模型上的 chunk-alignment 精确性证明。
  • 严谨性（中-高，但有显著自参照）。 正确性三层定义清晰、KV-only 消融设计精巧（用分叉而非断言证明递归态承重）、跨三架构复现、对抗式 dirtied-state 测试、fork/rollback 在 Thor 上 token 级验证（branch A=cold prefix+A 0/40 mismatch，A≠B 真独立分叉）。但所有"加速比"的基线几乎都是 FlashRT 自身的冷路径（cold-vs-capsule 27× 是自参照的），而 vs vLLM 的比较被严格限定在 vLLM 按设计就不擅长的单流并发 1 场景；27× 这类数字若脱离"vs 自身冷"的上下文极易被误读为"比 vLLM 快 27×"（实际 vs vLLM-APC 是 1.4–2.8×）。
  • 可复现性（中）。 代码声称随 GitHub 发布，pytest 九测明确，repro 命令与 raw artifacts 声称释放。但旗舰"Qwen3.6 NVFP4"checkpoint 的获取方式未明说；Higgs TTS 用 sglang-omni、Pi05 策略来源也未完全说明；论文是 arXiv v1 单作者预印本，尚无独立复现印证。
  • 重要性（中）。 若物理-AI/边缘单流服务确实成为主流负载，胶囊化的状态一等对象 + 显式 pin/fork/rollback 是一个真实、互补的设计点。但当下多数生产 LLM 服务仍以高并发吞吐为先，论文自己也反复声明"不替代、不高并发吞吐、不分布式、不跨节点 KV"。对主流服务栈的直接影响有限。
  • 红旗。 (1) 单作者 + 预印本无同行评审；(2) "Qwen3.6"等模型名与版本号风格在未来时间戳下显得可疑，需核验是否真实可获取；(3) vs vLLM 的 APC 崩塌结论绑定于 vLLM 自标 experimental 的混合/递归 prefix-cache 路径，论文也承认"非通用 APC 失败"，但行文与图 5 标题仍可能被误读为通用结论；(4) 命名"FlashRT"与同期他作撞名，存在引用歧义风险；(5) on-robot 评估、生产多轮 server、SGLang 同模型延迟全部 deferred，核心机器人侧仅为离线图机制测试（cos=1.0），非任务成功率/安全性结果。

Strengths

  • 概念清晰：用"受管对象"与"复用单位"两轴把与 vLLM/SGLang 的差异讲得非常锐利（Table 1/2/11），非泛泛"我们也用 CUDA Graph"。
  • "一个设计选择、两个推论"的论证有说服力：静态缓冲图既是速度来源也是可冻结性来源，非两个 bolt-on 特性。
  • 正确性工程扎实：三层包络、对抗式 dirtied-state 测试、chunk-alignment 条件的识别与证明、KV-only 消融用分叉量化证明递归态承重。
  • 诚实到罕见程度：§7.5 主动报告 Thor 上 vLLM-APC 不崩塌（削弱自身对比）、Appendix A 完整披露 SGLang 失败而非凑数、明确"不声称字节级 logits 一致"。
  • 跨三架构（5090/Thor/Spark）复现，且诚实指出加速倍数随设备冷成本变化而胶囊 TTFT 本身稳定 ~0.2 s。
  • 机制跨域统一有美感：LLM 热启动 = 机器人 episode reset = 同一 snapshot/restore 动词；planner-actor 交接用契约已有的零拷贝 buffer pass，无需胶囊。

Limitations

  • 范围"deliberately thin"（自述）。 无生产多轮 agent server、无 on-robot 任务成功率/安全研究、无分布式/集群、无高并发吞吐、decode 吞吐与 speculative decoding 显式排除。
  • SGLang 同模型比较缺失。 因 SGLang 0.5.13 不支持 NVFP4-W4A16 compressed-tensors，同模型延迟赛被推迟到后续，SGLang 仅作结构性（从源码）对比 + Higgs TTS 原生健全性检查。
  • 自参照加速比。 头条 27× 是 FlashRT 冷 vs FlashRT 胶囊；vs vLLM 的真实数字是冷 2.6–2.8×、vs APC 1.4–2.8×，magnitude 远小。
  • APC 崩塌结论的场景依赖。 仅在 5090 离散 VRAM 上观察到 APC 在 ~16k 崩塌（< 其 34.6k KV 容量的一半），且绑定于 vLLM 标 experimental 的混合/递归 prefix-cache 路径；Thor/Spark 大统一内存上不崩塌。崩塌不具普适性。
  • 静态缓冲图牺牲灵活性。 需有界 shape 变体集（ShapeKey 表 + LRU）与固定最大序列长度，适合低并发少 shape，不适合高并发变 shape（paged/radix 的主场）。
  • 胶囊是二进制 blob，绑定精确权重/量化/kernel 版本/graph bucket。 跨版本/跨部署不可移植，仅用于同部署热启动或团队内共享。
  • 机器人侧仅为离线机制测试。 byte-identical 动作回放（cos=1.0）证明正确性，但非 on-robot rollout 延迟、任务成功率或安全研究——后者全部 future work。
  • 单作者预印本，缺乏独立复现与同行评审印证。

Relevance

  • LLM 训练。 弱相关。胶囊是推理期/服务期机制，不涉及训练。但"递归态是承重的、位置 KV 单独复用是错的"这一发现对训练混合线性/全注意力模型后的部署评估有间接指导意义。
  • LLM 推理。 高度相关，正是本文主战场。对单流、低延迟、on-device/边缘的 LLM 服务（编码 agent、语音助手、VLA/机器人策略）提出互补设计点：把会话当可 checkpoint/fork 的对象而非paged cache 上的流。对多轮 agent 的热启动（10–50k 共享前缀每轮重发）、tree-of-thought/best-of-N 分叉、turn 回滚、中断再入有直接机制价值。
  • 可解释性。 低。胶囊是状态管理而非可解释性工具，但其"显式 pin/fork/rollback"策略层给可观测的会话状态控制提供了钩子。
  • Agents。 高度相关。编码 agent 的 warm multi-turn、parallel tool-calls 的 fork、retry 的 rollback 都被显式建模为胶囊动词，是物理-AI 交互环（中断/再入/分叉）的基础设施层。
  • 可靠性。 中-高。bounded re-entry（按场景不变性选恢复深度 + 重新观测 + 验证前置条件 + 回退/转人工）把"恢复陈旧计算对抗已变世界"的不安全性显式建模为策略层，且 capsule 恢复的是不变计算基底（planner context/skill/warm runtime）而非不可逆物理态——这是服务层安全的重要区分。但安全研究与 on-robot 验证尚属 future work。

Questions for Further Exploration

  • 把胶囊集成进真实多轮 agent server（自动前缀匹配、OpenAI 式全历史客户端、graph-cache 健壮性）后，端到端用户可感知延迟与吞吐如何？论文显式把此列为 future work。
  • 在高并发（concurrency > 1）场景下，胶囊的显式 pin 策略与 vLLM/SGLang 的自动 LRU 相比，是否存在一个两者混合的帕累托最优设计点（而非纯粹互补）？
  • chunk-alignment 条件对其他分块递归架构（如不同 chunk size、跨设备数值精度）是否需要重新校准？FP8 舍入导致的非对齐分叉在不同 kernel 实现下有多严重？
  • on-robot rollout 的真实延迟、任务成功率与安全研究所结论如何？离线 cos=1.0 动作回放能否转化为可测的任务性能增益？
