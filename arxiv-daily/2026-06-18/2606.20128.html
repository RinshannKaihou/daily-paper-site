Safety · Benchmark Validity GPU-Kernel Testing Fuzzing

Key Finding Standard LLM-kernel-benchmark oracles certify real transcription bugs as "correct" — while an op-schema-aware seeded fuzzer with an fp64 reference catches 10/10 illusions with zero false positives.

The Correctness Illusion in LLM-Generated GPU Kernels
Dipankar Sarkar (Arizona State University) · 2026-06-18 · arxiv.org/abs/2606.20128

Key Metric

10/10illusions caught
Across 5 GPU classes (RTX 3060, A10, A100 SXM4, L40S, H100 NVL): 10/10 bugs caught, 16/16 controls clean, zero false positives, bit-for-bit replay.

Why This Matters

If you build or trust KernelBench / TritonBench / GEAK, the fixed-shape allclose oracle is systematically optimistic — cheap countermeasures exist.

Critical Edge

The bug corpus is author-seeded (not from real LLM outputs), and the headline 10/10 is near-guaranteed by construction — this demonstrates oracle weakness, not a measured LLM bug rate.

Methodology — three optimism modes, one fuzzer

Fixed shape — one shape/op misses tail-mask & off-by-one
Fixed dtype — fp32 default; fp16/bf16 overflow untested
Hand-picked tolerance — 1–3 orders too loose vs error envelope
Boundary values — e.g. H ∈ {1,3,7,256,1025}
  • Op-schema-aware seeded generation: each kernel ships symbolic dims + candidate values; per-dim sampling via Blake2b sub-key derivation, deliberately including boundaries (H=3 surfaces tail bugs; H=256 hides them).
  • Faithful oracle: fp64 CPU reference rounded to target dtype, compared with absolute-only per-(op,dtype) tolerances (vs PyTorch atol+rtol); records full error distribution + ULP + NaN/Inf.
  • Reproducible pipeline: each failure stores a full input snapshot for bit-for-bit replay; public repos gpuemu + 26-op corpus; vast.ai instances torn down by a label-strict reaper; results archived to Backblaze B2 with run IDs.
  • Corpus: 15 correct controls + 9 LLM-style buggy variants (flash-attention pair added for 26 in the cross-GPU run).

Results — beyond the headline

Single-GPU (RTX 3060)9/9 bugs flagged, 15/15 controls pass, 0 FP, ~1,440 cases

Shape-dependent bugssoftmax tail, matmul acc= overwrite — visible only with boundary fuzzing

Two countermeasures map to two bug families: magnitude-uniform bugs want tighter per-(op,dtype) tolerances (companion paper [8]); shape-dependent bugs want boundary-aware input generation (companion paper [9]) — both currently "in preparation."

Strengths

  • Crisp, falsifiable threat model naming exactly the three oracle weaknesses it attacks.
  • Exceptional reproducibility: stored seeds, run IDs, public repos, multi-GPU cross-check.
  • Honest, repeated disclosure that the corpus is author-seeded.
  • Bug taxonomy maps cleanly onto two cheap countermeasures.
  • H=3 vs H=256 is a memorable illustration of single-shape brittleness.

Limitations

  • Hand-seeded bugs — not extracted from real LLM kernels; no measured bug rate.
  • Small corpus (26 ops), single author.
  • Two load-bearing companion papers are "in preparation" — ablation claims uncheckable.
  • Oracle correctness (fp64 + hand-set tolerance) is asserted; calibration deferred.
  • bf16 proxied to fp16; non-contiguous layout fuzzing nominal.

Questions for Further Exploration

  • Run the protocol on kernels actually generated by frontier LLMs on KernelBench/GEAK — what fraction of "certified correct" kernels fail?
  • How tight can the per-(op,dtype) tolerance be before correct-kernel recall degrades?
  • Does cross-dtype comparison (kernel-fp16 vs reference-fp64) plus true bf16 raise recall?
  • Is the "10/10 identical across 5 GPUs" robust at larger sample sizes, or only because the corpus is small and bugs gross?
