01

How it works

Two things evolve on two timescales — what to do (task skill s) and how to improve (meta-skill m).

A task skill s is a Markdown agent program; its utility U(s) is held-out accuracy. The meta-skill m parameterises the improvement pipeline itself, optimised for meta-productivity P(m|s) — how much utility its descendants produce.

Because m is just another set of Markdown files, the same five-agent pipeline rewrites both s and m. That is the recursion: one level deep, no new model, no new objective. Only strictly-improving children (ΔU>0) are committed.

The full search is a persistent DAG in SQLite; regressing nodes stay eligible as cross-branch inspiration sources.

# fast loop — every iteration optimize U(s) on validation batch
commit child iff ΔU > 0

# slow loop — every H fast iters P̂(m|s) = (1/|H|) Σ ΔU
→ re-run same pipeline on meta-skill files

# frontier pick next parent score = η₁·U + η₂·P̂ + η₃·N
# utility + quality + novelty(cooldown)

Component ablation
Allocator α is the single most critical component on OfficeQA — removing it drops accuracy −19.7 pts (55.32 → 35.58); Proposer π is close behind (−17.7).
Domain-dependent driver
On SealQA the Proposer π dominates instead — ablating it costs −8.4 pts. Gains hinge on edit content, not search fan-out.
Horizon sweep
H=2 (most reactive) wins everywhere. OfficeQA is by far the most sensitive, shedding −9.1 pts from H=2 → H=8 as the meta-skill refreshes less often.
03

Verdict

Strengths

  • Principled separation of U(s) from P(m|s); each frontier term provably targets a distinct greedy-search failure mode.
  • Elegantly bounded recursion — the meta-skill shares the task-skill's Markdown format, so one pipeline serves both, no new architecture.
  • Honest ablations isolating ψ, σ, α, π one at a time, openly reporting the dominant component differs by domain.
  • Transparent about the near-ceiling ALFWorld regime — flags the static-skill regression instead of hiding it.

Limitations

  • Only three curated benchmarks; open-ended, long-horizon, noisy-feedback tasks are explicitly untested.
  • Pipeline topology is fixed — the framework evolves skills, not the agent roles, so recursion is exactly one level deep.
  • H is fixed (default 2); there is no adaptive schedule for the meta-update horizon.
  • Single backbone — cross-family generalisation (Llama/Qwen) is unverified.
  • Slow-loop compute cost (5 extra agent invocations per H iters) is not reported; wall-clock/token comparisons to Single-Level are incomplete.
04

Open questions

Does the slow loop still help when the backbone is not near ceiling — would a stronger/different model shrink the +6.38/+8.05 delta or shift α vs π dominance?
Can recursion extend beyond one level (a meta-meta-skill), and where does it collapse into instability or diminishing returns?
Are the frontier weights η₁, η₂, η₃ benchmark-sensitive, and is there a principled way to set them rather than treat them as hyperparameters?
What is the wall-clock and token overhead of the slow loop, and does per-point-gain cost favour meta-skill evolution or simply more fast iterations?
Does P̂(m|s) correlate with any interpretable skill-file property (length, edit locality, tag-vocabulary size) — making it a useful observability metric?
Would a backbone exposing hidden-state signals let the Analyzer/Allocator condition on internal uncertainty rather than only surface reward?
