§ 01

Methodology

Hard-barriered compute tolerates elastic membership iff no barrier ever waits on a named core.

The Protocol

Implemented inside Anima OS — a bare-metal x86_64 no_std Rust kernel (~232k lines). Four pieces of lock-free SeqCst atomics compose the mechanism:

  • GANG_EPOCHglobal generation-ownership epoch (init 1, not 0); bumped at gang_begin & gang_end → any stale ack dies by construction.
  • GANG_ACK[i]per-core, cache-line-padded; core i joins a token iff ACK[i]==EPOCH at latch time.
  • GANG_ACTIVEthe lend-invitation flag.
  • GANG_OWNERsingle-owner CAS (0 → owner).
resync-before-ack — snapshot the token-generation counter before storing the ack. This one load-bearing ordering rule prevents a latched-but-never-walked hang.

Single Tagged Word & Baselines

Membership is snapshotted once per token and packed as (token_gen<<32)|mask, published at the token-start Release bump. All five barrier wait sites and every worker self-exclusion read this one tagged word — so wait-side & worker-side see the same snapshot, and changes take effect only at the next token.

  • ablationThe static-K baseline is a mode of the same binary (gang_begin returns false) — implementation drift ruled out by construction.
  • metricgeneral_served_s — a monotone served-quanta counter, never re-based on migration. Replaced a contaminated virtual-time metric (118–180% spread) the author proactively disclosed.
  • harnesssched_bench (2,567 lines): in-kernel autorun matrix emitting grep-able [sched:…] lines to NVMe-persisted boot logs.
setup — AMD Ryzen 9 9800X3D (Zen 5, 8C/16T), DDR5-6000 (~75 GB/s), bare metal. SmolLM2-135M (Q4_0, ~0.75 ms/tok) + Qwen2.5-7B-Instruct (Q4_0, ~79 ms/tok). 24 preemptive tenants + 1 pinned latency tenant + 32-token bursts at duty d∈{0,25,50,75,100}%.
§ 02

Results & Highlights

No operating point at which static-8 wins anything.
574.0/s

Work conservation at 25% duty: 574.0 / 496.8 / 417.1 general served-quanta/s vs static-8's flat ≈327 — the 1.75×/1.52×/1.28× headline, at equal-or-better inference.

8 / 8

Full core recovery at idle: 653.9 vs 327 served-quanta/s. At duty 100% the axes converge (326.9 vs 327.0) — elastic never loses.

−1

Bit-exactness: first-divergence index −1, identical final-logits hash under verified mask churn (135M: 8 changes / 6 sets; 7B: 6 changes / 4 sets). Negative control again detected.

width 8

Saturation knee in both memory regimes: 7B 12.72 tok/s (78.6 ms/tok); 135M 1352.5 tok/s. Widths 9–16 buy <1% → ceded cores nearly free.

0.22µs p50

Return latency (epoch bump → ack retraction): p50 0.22 µs, p99 0.38 µs, max 0.42 µs (n=224). Migration: p50 0.22 µs, max 0.7 µs.

~800×

Linux/EEVDF anchor (same silicon): unpinned 8-thread llama.cpp decode collapses ~800× on 135M (1230→1.5 tok/s), ~62× on 7B. taskset restores ~975 but strands half the budget when idle.

+ Governance dial: a trust-weighted lend quota raises inference +66% (343.8→572.4 fwd/s) while general falls 42% across the trust range.

§ 03

Tradeoffs

Exceptional methodological honesty vs. one machine, one vendor.

Strengths

  • Failure-mode-justified mechanism. Each failure mode (barrier deadlock on departure, silent corruption on late/double arrival, stale-ack resurrection) maps to a protocol element that prevents it.
  • Clean ablation. Static-K is a mode of the same binary — the duty sweep isolates scheduling policy with zero implementation drift.
  • Methodological honesty. Author replaced a contaminated metric, disclosed the pinned-latency worst case (2.8 s p99 at duty 100%), and refused to headline SMT-confounded inference gains.
  • Dual bit-exactness proof. Demonstrated under both forced width-churn and genuine ACK-latch lending, with a negative control proving the test can fail.

Limitations

  • One machine, one vendor (single Zen 5 box). Constants — knee at 8, 16 ms quantum — are machine-specific.
  • SMT topology confound. Lending an SMT sibling of a busy core ≠ lending a physical core; likely source of static-8's inference degradation under oversubscription.
  • Busy-core acquisition is quantum-bound (~16 ms), not µs-scale — isolated short generations on fully busy machines benefit little.
  • Linux anchor is directional only (different engine + scheduler); a controlled same-engine port is future work.
  • No GPU evaluation — target class is explicitly GPU-less edge/appliance decode.
  • Mid-token departure argued unrepresentable, not empirically tested; governance sweep is narrow (one model, bench override).
§ 04

Open Questions

Where the elastic frontier bends next.
  1. Would a preempt-for-lend policy (trading tenant latency for sub-quantum gang ramp) materially change the duty-25 acquisition cost — and is the tail-latency hit ever worth it?
  2. Does the bit-exactness invariant survive non-greedy sampling (top-p, temperature), FP16/BF16 kernels, or attention kernels with non-row-partitionable reductions?
  3. Is width-8 saturation a Zen 5 artifact or a deeper property of single-socket decode — i.e. how does the knee (and thus the profitability of elasticity) shift on ARM / Apple Silicon?
  4. Can the single gen-tagged latch word scale beyond 32 cores (its current range), or does the protocol need a different publication structure for high-core-count CPUs?
  5. Would a controlled same-engine port to a Linux userspace harness preserve the collapse-vs-strand tradeoff, or do Linux's NUMA/cache-partitioning features blunt it?
  6. Is the "lost or extra gang core is a correctness event" framing extensible to detecting silent data corruption (SDC) in the matvec itself — could the per-wave exactly-once invariant be instrumented as an SDC detector?
