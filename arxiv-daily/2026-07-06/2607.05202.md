> **一句话总结：** 本文提出 EvoAgentBench——首个通过"轨迹→能力(Ability)"迁移来评测 LLM 智能体自进化的多领域基准，其基于轨迹抽取的 Ability Graph 保证每个测试任务都有训练侧能力支撑；实验显示经策展的能力内容可在跨模型家族间稳定迁移（Anchor 参考条件在 Qwen3.5-397B 上平均 +10.5），但现有所有自动自进化方法（Memento/ReasoningBank/GEPA）在至少一个设置下出现负迁移。

> **TL;DR:** EvoAgentBench reframes agent self-evolution as a controlled experience-to-procedure transfer problem: it builds trace-grounded Ability Graphs (528 train / 267 test tasks across BrowseComp-Plus, SWE-Bench Verified, LiveCodeBench, GDPVal) that guarantee train-side Ability support for every test task. Under matched conditions, a curator-side Ability reference (Anchor) yields uniformly positive transfer on Qwen3.5-397B (+10.5 avg vs Vanilla 49.2→59.7) and all 24 cells, yet no automatic method (Memento, ReasoningBank, GEPA) sustains positive gain — each shows negative transfer in at least one setting, exposing the bottleneck as method-side extraction and routing rather than content availability.

## 核心贡献

- **评测框架重构**：首次将智能体自进化形式化为"受控的经验→过程迁移"评测问题，提出"能力支撑但实例不相交(ability-supported yet instance-disjoint)"的切分原则——测试任务必须与训练任务共享可迁移 Ability，但不共享具体实例，从而杜绝靠记忆答案或近似重复刷分的可能。
- **EvoAgentBench 基准**：跨四个长周期智能体领域（网页研究 BrowseComp-Plus、算法推理 LiveCodeBench、软件工程 SWE-Bench Verified、知识工作 GDPVal），通过三阶段流水线（多骨干轨迹收集→轨迹锚定的 Ability 抽取→规范化与图构建）生成 170 个规范化 Ability 单元与 Ability Graph，保证 267 个测试任务零无支撑。
- **诊断式参考(Anchor Skill)**：提供仅用训练侧证据构建、但用策展侧 Ability 标签路由的能力库，作为自动方法的"理论上限"诊断锚点，揭示能力内容本身是否可迁移。
- **关键实证发现**：跨 2 个 scaffold(OpenClaw/Nanobot)×3 个骨干(Qwen3.5-27B/397B, Gemma-4-31B) 的六设置评测，证明：(1) 能力内容跨模型家族可迁移（Anchor 在与构造骨干不相交的评测骨干上仍全面提升）；(2) 但所有自动方法都脆弱，瓶颈在于方法侧的抽取与路由机制而非内容可用性；(3) 负迁移与"交互成本"均不可被聚合准确率反映。

## Key Contributions

- **Evaluation-framework reframing.** Casts agent self-evolution as a controlled experience-to-transfer evaluation where every test task is provably backed by training-side Ability support, isolating trajectory-to-procedure transfer from mere task solving or information retention.
- **The EvoAgentBench benchmark.** A multi-domain benchmark (4 long-horizon agentic domains, 528/267 train–test split) whose trace-grounded Ability Graphs — not dataset labels or topic metadata — define the transfer structure, with zero unsupported test tasks.
- **Ability abstraction + construction pipeline.** Introduces Abilities as reusable operations (Method/Guard/Workflow roles) with explicit trigger/boundary, extracted jointly across multiple construction backbones (Kimi-K2.5, GLM-5.1, DeepSeek-V3.2), canonicalized via embedding blocking + three-LLM adjudication + domain-expert review, and wired into a graph partitioned by Louvain communities for support-aware splitting.
- **Diagnostic Anchor reference.** A curator-side Ability-grounded skill condition that uses only train-side evidence for content but test Ability labels for routing, serving as an upper-bound diagnostic that decouples "is the content transferable?" from "can an automatic method deliver it?".
- **Multi-setting empirical diagnosis.** Across 6 scaffold–backbone settings, shows ability content transfers across model families (Anchor uniformly positive) but every automatic method is brittle, with paradigm-level failure modes traceable to each method's routing mechanism rather than domain difficulty.

## Methodology

EvoAgentBench is organized around an **Ability Graph** in which nodes are tasks and edges denote shared reusable Abilities — operations extracted from agent executions rather than dataset labels, topics, or repository names. An Ability is a trace-grounded abstraction specifying *when* a procedure, correction, or validation step should apply (trigger + procedure + boundary + role).

**Three-stage construction pipeline:**

1. **Multi-Backbone Trace Collection.** For each of 2,605 source tasks (BrowseComp-Plus, SWE-Bench Verified, LiveCodeBench V6, GDPVal), no-skill executions are collected from three construction backbones — Kimi-K2.5, GLM-5.1, DeepSeek-V3.2 — each run under two scaffolds (OpenClaw, Nanobot) twice, with a 1,800s timeout. Cross-backbone aggregation reduces sensitivity to any single model's shortcuts.

2. **Trace-Grounded Ability Extraction.** An extractor (Claude Sonnet 4.6) jointly reads the query, ground-truth target, and the multi-backbone trace pool per task, comparing successful vs. failed executions to surface recurring procedures, failure modes, and discriminative verification steps. Each raw Ability card is a tuple (trigger γ, procedure π, evidence E, boundary ∂, role ρ). Cards are retained only if grounded in a trajectory span, actionable, and narrow enough to support meaningful transfer. This yields **7,326 raw cards from 2,516 tasks**.

3. **Ability Canonicalization + Graph + Split.** Raw cards are canonicalized by operational-equivalence adjudication: embedding cosine similarity is used only as a recall-oriented blocking step (θ=0.82–0.85), never as a merge criterion. Candidate pairs are judged by three independent LLM adjudicators under a strict rubric (same role, compatible triggers, equivalent procedures, same mechanism, compatible boundaries; shared topic/lexical overlap is explicitly insufficient). Unanimous pairs auto-merge; all others get domain-expert review. Accepted links form a compatibility graph; group-consistency checks promote components to **170 canonical Ability units**. These are mapped back to tasks, producing an Ability Graph (1,108 retained tasks, 77,197 edges) partitioned by Louvain communities. Constrained splitting guarantees every test task shares ≥1 edge-eligible Ability with a training task in its own community, and test tasks are preferentially sampled where construction backbones left accuracy headroom.

**Evaluation protocol.** The primary observable is transfer gain Δ = mean(r_m(x) − r_0(x)) over the test split. Methods build evolution states from D_train only (prompts, verifier outcomes, self-generated rollouts); test answers/traces/shortcuts are excluded. All conditions share identical tasks, tools, scoring contracts, timeouts, and base agent configuration within each scaffold–backbone setting — methods differ *only* in how they encode and reuse training experience. Results are averaged over 3 independent runs per instance with reported standard error.

**Conditions compared:** Vanilla (no evolution), Memento (case retrieval by embedding similarity), ReasoningBank (distilled reasoning-memory retrieval), GEPA (single evolved prompt broadcast to all), and Anchor (diagnostic, curator-side Ability-routed skills).

## Results & Highlights

- **Ability content transfers across model families.** Anchor's construction backbones (Kimi-K2.5, GLM-5.1, DeepSeek-V3.2) are disjoint from the evaluation backbones (Qwen3.5-27B/397B, Gemma-4-31B), yet Anchor produces positive average Δ on **every** backbone — +7.5 (Qwen3.5-27B: Vanilla 37.5 → Anchor 45.0), +10.5 (Qwen3.5-397B: Vanilla 49.2 → Anchor 59.7), and +5.8 (Gemma-4-31B: Vanilla 36.7 → Anchor 42.5) — with positive per-domain Δ in **all 24 method–domain–setting cells**.
- **No automatic method sustains positive gain.** Memento is negative on Qwen3.5-27B (−2.4) and Gemma-4-31B (−0.7); GEPA is positive on average (+1.2/+3.5/+5.7) but still contains **four** negative per-domain cells (e.g., −12.3 on OpenClaw/Qwen3.5-27B/Knowledge-Work); ReasoningBank is positive on average (+0.4 to +3.6) but shows **six** negative per-domain cells.
- **Catastrophic negative transfer is paradigm-driven.** Memento drops by **−36.3 points** on Nanobot/Qwen3.5-27B/SWE due to surface-similarity mismatch, while Anchor remains positive on all 12 SWE and Algo cells — confirming the regressions stem from each paradigm's routing mechanism, not the Ability content itself.
- **Anchor exceeds best automatic method by +3.9 (Qwen3.5-27B) and +7.0 (Qwen3.5-397B) average points**, and on Gemma-4-31B GEPA nearly matches Anchor's average gain but Anchor is the only condition with uniformly positive cell-level transfer.
- **Cost–accuracy decoupling is method-specific.** Anchor reduces overall turn cost in 4 of 6 settings (up to −8.8%); ReasoningBank in 3; GEPA increases cost in 4 of 6. Memento even adds +39.4% turns on solved tasks (OpenClaw/Qwen3.5-27B), showing generic artifacts add overhead without shortcutting exploration.

## Critical Assessment

- **Novelty:** Genuinely novel evaluation framing. The "ability-supported yet instance-disjoint" split invariant is a clean, defensible solution to the tension between relatedness (needed for transfer) and distinctness (needed to prevent memorization), and the Ability Graph construction (trace-grounded, not metadata-based) is a meaningful advance over SkillsBench/SkillLearnBench/SkillFlow, which probe skill quality within or across tasks but never control for training-side ability support (Table 1).
- **Rigor:** Strong experimental controls — identical tasks/tools/scoring/timeouts/agent config across methods within each setting, 3 runs with standard error, native per-domain metrics (programmatic for SWE/LiveCodeBench, LLM-judge for web/knowledge work), and an explicit information-access matrix (Table 8) showing automatic methods receive only train-side evidence.
- **Reproducibility concerns:** Several critical components depend on proprietary/non-public artifacts — the Ability Graph itself, the curator-side Ability labels used for Anchor routing, and the human-reviewed canonicalization decisions. The dataset is public, but re-deriving the Ability vocabulary requires re-running Claude Sonnet 4.6 extraction plus expert review. Anchor is explicitly **not** a deployable method (it uses curator-side test labels for routing), so its numbers are an upper bound, not a target automatic methods can hit without the labels.
- **Significance caveats:** The benchmark is limited to 4 text-based agentic domains and 3 open-source construction backbones; multimodal/embodied agents and proprietary model families are out of scope. Test sets per domain are modest (56–86 tasks), so the authors concede finer-grained per-domain analysis would benefit from larger scale. Absolute Δ magnitudes are specific to the headroom-sampled split, not estimates of expected gain on a random task sample.
- **Red flags / data quirks:**
  - **Test tasks are preferentially sampled where construction backbones left accuracy headroom** (§3.2.5, A.6), which inflates the apparent room-for-improvement and means absolute gains are not representative of random sampling — a choice that favors method comparison but could mislead readers generalizing to deployment.
  - **Single-extractor dependency:** Ability extraction uses only Claude Sonnet 4.6; the resulting vocabulary reflects this choice, though three-judge canonicalization + expert review mitigates (not eliminates) extractor-specific idiosyncrasies. The paper honestly flags characterizing alternative extractors as future work.
  - **Missing ablation:** there is no controlled ablation isolating the contribution of the multi-backbone trace pooling (vs. single-backbone extraction) or of the expert-review step (vs. LLM-only canonicalization) on final transfer quality — so the cost/benefit of the pipeline's most expensive human steps is asserted rather than measured.
  - **Cost metric is turn-count only** (no latency, token, or dollar cost), and "All" turn cost need not lie between solved/unsolved subsets due to equal-domain weighting, which can obscure task-level behavior.

## Strengths

- Clean, well-motivated evaluation principle (ability-supported + instance-disjoint) that resolves a real gap left by all surveyed benchmarks (Table 1 is a crisp positioning).
- Diagnostic Anchor cleanly decomposes *content transferability* (a property of the Ability Graph) from *delivery* (a property of the automatic method), making the central finding — "content transfers, delivery fails" — interpretable rather than a black-box ranking.
- Multi-setting design (2 scaffolds × 3 backbones × 4 domains = 24 cells per method) is rare in agent-self-evolution work and exposes brittleness that single-setting evaluations hide; paradigm-level failure-mode diagnosis (Memento's surface-similarity collapse, GEPA's no-retrieval broadcasting) is insightful.
- Joint reporting of accuracy *and* turn cost, with solved/unsolved breakdowns, surfaces cost–accuracy decoupling that accuracy-only tables miss.
- Honest limitations section that explicitly flags the headroom-sampling quirk, single-extractor dependency, and contamination caveats, rather than overclaiming.

## Limitations

- **Scope:** 4 text-based agentic domains only; multimodal/embodied agents, larger/proprietary model families, and non-English settings are untested.
- **Scale of per-domain test sets** (56–86 tasks) limits statistical power for fine-grained per-domain claims; the authors recommend larger-scale evaluation for sub-cell analysis.
- **Single extractor** (Claude Sonnet 4.6) for Ability extraction; the vocabulary reflects this model's abstractions, and sensitivity to extractor choice is uncharacterized.
- **Headroom-biased test sampling** makes absolute Δ magnitudes split-specific, not population estimates (though method comparisons remain valid since all conditions share the split).
- **No cost model beyond turn count** — no latency, token, or monetary overhead is reported, so "cost" is a coarse proxy.
- **Missing pipeline ablations** on the contribution of multi-backbone pooling and human expert review, the two most expensive construction steps.
- **Anchor is non-deployable** (uses curator-side test Ability labels for routing), so it is an oracle upper bound; the gap to automatic methods is partly an artifact of routing information that automatic methods structurally cannot access.

## Relevance

与"自进化智能体与自动化研究(Self-Evolving Agents & Automated Research)"方向高度相关——本文正是该方向首个保证训练侧能力支撑的迁移评测基准，并定量揭示了自动抽取/路由机制的脆弱性与负迁移问题。同时，其"交互成本与准确率解耦""负迁移诊断"的发现对 LLM 推理评测与训练可观测性亦有方法学借鉴意义，但与计算精度、隐藏态可解释性、SDC 等主题关联较弱。

## Questions for Further Exploration

- How much of the Anchor–automatic gap closes if an automatic method is given the *same* train-side Ability labels but must learn its *own* routing policy (i.e., is the bottleneck routing inference, or routing signal availability)?
- Does the headroom-biased test split systematically favor retrieval-based methods (Memento/ReasoningBank) over broadcast methods (GEPA)? A stratified re-evaluation on a non-headroom sample would clarify generalizability.
- How sensitive is the Ability vocabulary to the extractor model? A swap-in study (e.g., GPT-class or open-weight extractor) would quantify the single-extractor risk the authors flag.
- Can the paradigm-level failure diagnoses (Memento's surface-similarity collapse, GEPA's broadcast brittleness) be used to construct a *hybrid* automatic method that retrieves by operational similarity rather than embedding similarity — and does it eliminate the −36.3 catastrophic-failure cell?
- What is the relationship between Ability Graph density (e.g., BrowseComp-Plus 0.404 vs. SWE 0.289) and the magnitude of achievable transfer gain, and does high density mask retrieval-quality differences?
- Would extending to multimodal/embodied domains preserve the "content transfers, delivery fails" conclusion, or do those domains introduce ability types (perceptual, motor) that change the transferability structure?
