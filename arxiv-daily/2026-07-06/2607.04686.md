> **一句话总结：** 本文提出 ToolFailBench，一个包含 1,000 道跨金融/医学/法律/网络安全/房地产任务的诊断式基准，通过"参数化陷阱 + 对照任务"分离出 Tool-Skip / Result-Ignore / Output-Fabrication / Unnecessary-Tool-Use 四类工具使用失败，揭示 19 个主流模型在相似聚合分数下失败模式差异巨大（最强模型 Grok-4.3 仅达 86.33% Clean Tool-Use Rate，Llama-3.1 系列出现"永远调用工具"的病态行为）。

> **TL;DR:** ToolFailBench is a 1,000-task diagnostic benchmark (5 professional domains) that, via parametric-trap tool-required tasks and no-tool control tasks, decomposes tool-use into Tool-Skip / Result-Ignore / Output-Fabrication / Unnecessary-Tool-Use and labels each trace with a rule classifier + two LLM-judge majority vote; across 19 models the best (Grok-4.3) reaches only 86.33% Clean Tool-Use Rate, and same-scale Llama-3.1-70B vs Qwen2.5-72B diverge by 89 percentage points on control-task accuracy, proving aggregate scores hide distinct failure mechanisms.

## 核心贡献

- **首个聚焦"调用后忠实度"（post-call faithfulness）的诊断式基准**：ToolFailBench 区别于 BFCL / τ-bench / ToolLLM 等只衡量"是否调用"或"最终任务成功率"的工作，专门用受控的 mock tool 返回值（参数化陷阱，刻意与模型记忆中的合理先验相矛盾）来检验模型是否真的把工具返回值写进最终答案，而非退回记忆值。
- **一套可分解的失败模式分类法**：在工具必需任务上区分 Tool-Skip（TS）、Result-Ignore（RI）、Output-Fabrication（OF）；在无需工具的对照任务上区分 Unnecessary-Tool-Use（UTU）、错误直答与正确直答，使"相同聚合分数、不同失败原因"首次变得可见。
- **稳健的复合标注流水线**：每个 trace 由确定性规则分类器 + 两个独立 LLM 评判（Qwen3.5-397B-A17B-FP8 决策树提示 + GLM-4.7-FP8 证据归因提示）三方多数表决，三向 Fleiss' κ = 0.693，并附 same-family 偏差校验（Qwen 评判对 Qwen 家族并未更宽松）。
- **揭示"规模不等于工具纪律"的反直觉发现**：在 70B/72B 同参数量下，Llama-3.1-70B 与 Qwen2.5-72B 在对照任务准确率上相差 89 个百分点（8.91% vs 98.00%），证明"避免无谓工具调用"主要由模型家族与训练行为决定，而非参数量。
- **覆盖 5 个专业领域、19 个头部模型**：金融、医学、法律、网络安全、房地产，每域 200 任务（150 工具必需 + 50 对照），跨 OpenAI / Anthropic / xAI / DeepSeek / Qwen / Llama / Gemma / GLM / Mistral / QwQ。

## Key Contributions

- **A diagnostic benchmark targeting post-call faithfulness, not just call correctness.** Unlike BFCL (call validity), τ-bench (end-state success), ToolLLM (real-world API success), or Toolathlon (long-horizon execution), ToolFailBench uses parametric traps where the mock tool return deliberately contradicts a plausible memorized prior, isolating whether the model writes the returned value into the final answer.
- **A decomposable four-way failure-mode taxonomy** — Tool-Skip, Result-Ignore, Output-Fabrication (tool-required) and Unnecessary-Tool-Use (control) — that surfaces why two models with identical aggregate scores can fail for completely different reasons.
- **A reproducible, auditable labeling pipeline**: a deterministic rule classifier provides an exact audit trail, while two independently-prompted LLM judges (decision-tree vs evidence-attribution framings) reduce surface-form brittleness; the ensemble revises rule labels in 8.4% of judgments and falls back to the rule classifier only on the 1.5% of three-way ties.
- **A 1,000-task, 5-domain, 19-model study** (finance / medicine / law / cybersecurity / real estate) showing faithful tool use is not saturated (best model 86.33% CTUR) and that scale alone does not produce tool discipline (same-scale Llama-3.1-70B vs Qwen2.5-72B differ by 89 points on control accuracy).

## Methodology

**Benchmark design.** 1,000 single-turn tasks across five professional domains (200 each, split 150 tool-required + 50 control). Each domain ships a role-specific system prompt (no model-specific tuning, no in-context answer examples) and nine domain tools (e.g. `get_quote`, `get_filings` for finance; CVE/exploit tools for cybersecurity).

**Parametric traps.** Tool-required tasks are constructed so the mock tool return intentionally diverges from a value the model may have memorized in pretraining/instruction-tuning (e.g. a stock price returns $247.83 where memory suggests ~$180). A faithful model reports the returned value; an unfaithful answer substitutes the prior. The authors are careful to call this a task-design term, not a claim of directly observing internal memory sources.

**Two-step inference protocol.** For every task the model first receives the query + tools with `tool_choice=auto`; if it emits a valid call the harness executes the mock tool and injects the controlled return, then the model produces a final answer with `tool_choice=none`. This deliberately separates the tool-decision step from the answer-writing step so skips, ignores, and fabrications can be cleanly distinguished. All runs use temperature=0, max_tokens=1024, seed=42 where supported; closed-API via providers, open-weight/reasoning via vLLM on H100/H200, judging on B200 via Modal.

**Labeling ensemble.** Each trace gets three labels: (1) deterministic rule classifier (no LLM calls; TS/UTU from presence of executed call; RI from absence of expected value; OF from invented structured fields); (2) Qwen3.5-397B-A17B-FP8 judge with a decision-tree prompt; (3) GLM-4.7-FP8 judge with an evidence-attribution prompt. Final label = unweighted majority vote, three-way ties broken by the rule classifier.

**Six metrics.** On Required (750): Tool-Skip Rate (TSR), Clean Tool-Use Rate (CTUR), Result-Ignore Rate (RIR), Output-Fabrication Rate (OFR) — summing to 100%, so leaderboard rows are verifiable by addition. On Control (250): Unnecessary-Tool-Use Rate (UTR), Control Accuracy (CTRL-Acc).

## Results & Highlights

- **Best model is not saturated.** Grok-4.3 reaches the highest ensemble CTUR of **86.33%** (95% Wilson CI [83.67, 88.61]) on ToolFailBench's 750 tool-required tasks, with Grok-4-1-Fast-Reasoning at 84.11% and Qwen2.5-32B-Instruct at 82.68%; the top two are *not* statistically distinguishable (p=0.23), and Grok-4.3 is only marginally above Qwen2.5-32B (p=0.05).
- **Same scale, opposite discipline.** At matched ~70B scale, Llama-3.1-70B and Qwen2.5-72B differ by **89 percentage points on control-task accuracy** (8.91% vs 98.00%, two-proportion z = −19.9, p < 10⁻⁸⁰), with Llama-3.1-70B showing a 77.73% Unnecessary-Tool-Use Rate vs Qwen2.5-72B's 0.00% — an "Always-Call" pathology that scaling from 8B to 70B does *not* cure.
- **Finance is the hardest domain for result faithfulness.** Among the 17 disciplined models, Finance has a median rule Result-Ignore Rate of **12.24%** vs Cybersecurity's **0.68%** — interpreted as plausible prior associations (stock prices, P/E, market cap) competing more strongly with tool returns.
- **Ensemble labeling is materially less brittle than rules alone.** Mean ensemble CTUR is **75.02%** vs mean rule-only CTUR of **71.05%**; the two LLM judges jointly overrule the rule label in **8.4%** of judgments while genuine three-way ties hit the rule tie-breaker in only **1.5%** (285 of 18,900).
- **Decoding-temperature check holds.** Rerunning QwQ-32B at non-zero temperature moved rule CTUR from 75.47% to **75.60%**, with degenerate repetition traces dropping from 34/1000 to 22/1000 — confirming conclusions are not greedy-decoding artifacts.
- **No same-family judge inflation.** The Qwen judge scores both Qwen-family and non-Qwen models ~6 pp *lower* than the GLM judge (−5.99 vs −6.16), the opposite of a leniency bias.

## Critical Assessment

- **Novelty — real but incremental.** The core insight ("aggregate accuracy hides *why* a tool-using agent failed") is well-trodden; T-Eval, StableToolBench, SpecTool, ToolBeHonest, and SMART each already decompose tool-use failures. ToolFailBench's distinct contribution is narrow but genuine: **controlled post-call faithfulness via parametric traps + paired no-tool control tasks**, isolating a failure mode (does the model trust the return over memory?) that prior benchmarks conflate into a single pass-rate. The "memorized prior" framing is honestly hedged — the authors explicitly disclaim that they observe behavior, not the internal memory source.
- **Rigor — strong for a workshop paper.** Deterministic temperature=0 runs, 95% Wilson CIs, pairwise and Fleiss κ, a temperature ablation, a same-family judge-bias check, and a same-scale Llama-vs-Qwen comparison with a reported z-statistic (z = −19.9, p < 10⁻⁸⁰) are above the bar for an ICML workshop. The four Required metrics summing to 100% (so any leaderboard row is arithmetically verifiable) is a nice integrity safeguard.
- **Reproducibility — good with caveats.** Code, prompts, task files, and the full judge rubrics are released. However, several judged/evaluated models (Grok-4.3, GPT-5.4-Mini, Claude-Sonnet-4.5, Qwen3.5-397B-A17B) are closed-API or otherwise not independently runnable, so the *leaderboard numbers* are only reproducible against those exact API snapshots; the *method and harness* are reproducible. The benchmark is single-turn by design, which the authors flag as a limitation but which also means it cannot catch multi-turn recovery/chaining failures.
- **Missing ablations.** There is **no sensitivity analysis to the trap-value gap** — how does Result-Ignore Rate change as the contradiction between returned value and plausible prior grows from trivial to large? This is the single most important omitted study for a benchmark whose entire mechanism is the trap. There is also **no analysis of how the four failure modes correlate with each other or with model size within a family** (only a 2-point Llama/Qwen same-scale contrast), and no breakdown of *which* tool-return fields (numbers vs dates vs identifiers) are ignored most often.
- **Data quirks worth flagging.** (1) The 250 control tasks share the same system prompts that hammer "the tool return is ground truth … never substitute a memorized value" — yet the headline finding is that Llama-3.1 ignores this and *always* calls tools; this is an interesting result but it means the control-task behavior is measured under an unusually tool-pushing prompt, which could inflate UTR across all models, not just Llama. (2) Two of 22 runs (glm-4-9b, mistral-7b) produced *zero* executed tool calls but tool-style answers ("Per <tool>"), classified as Tool-Skip — a legitimate edge case but one that suggests the harness/parser may be unfairly penalizing models with non-standard tool-call formatting rather than measuring genuine failure. (3) Finance's high RIR is offered as evidence the benchmark discriminates by domain, but the authors themselves note it "should not be read as pure domain difficulty" — leaving open whether it reflects domain hardness or salient-trap design.
- **Significance — moderate.** As a *diagnostic* tool for model developers it is immediately useful; the Llama "Always-Call" finding is a concrete, actionable red flag. As a *scientific* claim it is narrower: it shows failure modes differ across families, but does not yet propose or test a remediation (no fine-tuning, no prompt-intervention ablation, no correlation with training-data composition). The benchmark will likely become a standard evaluation component but is unlikely on its own to redirect the field.
- **Red flags.** Workshop-only (not main-track peer review); model-version snapshot that will date quickly; the "memorized prior" mechanism is behavioral, not mechanistic; no held-out task variants despite the authors acknowledging future train-test overlap risk.

## Strengths

- Genuinely novel failure-mode decomposition with a taxonomy that is mutually exclusive and exhaustive on each task type (Required metrics sum to 100%).
- The parametric-trap design is a clean methodological trick: by forcing the returned value to contradict a plausible prior, the benchmark makes "did the model trust the tool?" directly observable in the answer text without needing internal-state access.
- Paired control tasks with identical tools cleanly separate "always-calls" pathology from "can't use the return" pathology — a distinction no prior benchmark isolates.
- Labeling ensemble is principled (two differently-prompted judges + deterministic rule tie-breaker) and comes with a same-family bias check and stratified κ, addressing the usual "LLM-judge validity" critique head-on.
- Honest about limitations: single-turn, 5-domain, judge blind-spots, and post-release contamination risk are all explicitly enumerated in §6.

## Limitations

- **Single-turn only.** No multi-turn chaining, no recovery from earlier mistakes, no state updates — yet deployed agents fail most often *across* turns, not within one. The authors flag this as future work but it materially bounds the benchmark's external validity.
- **Five professional domains.** Finance/medicine/law/cyber/real-estate omit code, science, e-commerce, travel, and enterprise workflows where failure profiles may differ substantially; absolute failure rates should not be extrapolated.
- **No trap-gap sensitivity study.** The benchmark's entire mechanism is the contradiction between return and prior, but there is no experiment varying the *size* of that contradiction — so we don't know whether Result-Ignore is driven by small-gap confusion or large-gap refusal.
- **Judge dependency.** Although the rule classifier provides an audit trail, the headline ensemble labels depend on two specific LLM judges (Qwen3.5-397B, GLM-4.7) that may share blind spots; κ on rare labels (Unnecessary-Tool-Use Fleiss κ = 0.23) is low even though raw agreement is high.
- **Model-version snapshot.** Several headline models are 2026 closed-API checkpoints; the specific rankings will not transfer to successor or successor-of-successor models, and cannot be independently re-run.
- **Prompt-push confound on control tasks.** The shared system prompt strongly biases toward tool use, which may inflate UTR globally and particularly for models that follow instructions literally (plausibly part of Llama-3.1's Always-Call behavior).
- **Harness/parser edge cases.** Two models (glm-4-9b, mistral-7b) emitted tool-style text without executed calls and one (deepseek-r1-distill-llama-8b) had invalid raw-token output — suggesting the benchmark may conflate formatting/harness failures with genuine tool-use failures for some models.

## Relevance

与你的"LLM 推理评估 (LLM Inference Evaluation)"与"ML 可靠性 (ML Reliability)"方向高度契合：ToolFailBench 本质上是一套面向 LLM agent 的**可分解可靠性评测**，其"参数化陷阱"设计把"模型是否信任外部计算结果而非内部记忆"这一可靠性问题转译为可观测的答案文本信号，对研究"调用后忠实度"这一静默失败模式很有借鉴价值。同时，它对"相同聚合分数、不同失败模式"的诊断式分解，也与你的"评估不应被聚合分数掩盖"立场一致；不过本文不涉及底层精度错误/数值可靠性，也未触及隐藏状态层面的可解释性。

## Questions for Further Exploration

- How does Result-Ignore Rate change as a function of the **magnitude of the contradiction** between the returned value and the plausible prior (trap-gap sweep)? Is there a threshold above which models reliably switch to the tool value?
- Does the Llama-3.1 "Always-Call" pathology survive a **neutral system prompt** that does not push tool use so aggressively, or is it partly an artifact of the current prompt's "tool is ground truth" emphasis?
- Can the four failure modes be predicted from **hidden-state probes** — e.g., is Tool-Skip detectable from pre-call activations, and is Result-Ignore detectable from post-return activations before the answer is written?
- How do the failure-mode rates **correlate within a family across scales** (Qwen 7B/32B/72B/27B/35B), and is there a scaling-law signature for "tool discipline" distinct from "tool capability"?
- Do the parametric-trap failure modes **transfer to multi-turn settings** — i.e., does a model that Result-Ignores on turn 1 also fail to use the return when it re-appears in turn 3?
- Can the rule classifier's 8.4% revision rate by the LLM ensemble be **localized** to specific trace features (paraphrase, formatting, missing fields), and would a learned classifier close that gap deterministically?
- For your SDC/computation-reliability interest: if the mock tool return contained a **numerically corrupted but plausible value** (off-by-one-ULP, off-by-rounding), would models detect it or faithfully propagate the corruption into the final answer?
