> **一句话总结：** 本文提出 Anima OS 内核中的"弹性帮派"（Elastic Gang）机制——通过一个"绝不等待命名核心"的 ACK-latch 历元协议，让硬屏障 LLM 推理帮派能在任意两个 token 之间安全改变核心成员，在 AMD Zen 5 实机上保持输出比特级精确，并在中间推理占空比下取得 1.75×/1.52×/1.28× 的通用进程吞吐，同时等优或更优的推理吞吐。

> **TL;DR:** A bare-metal Rust kernel (Anima OS) treats a hard-barriered LLM inference gang as a first-class schedulable entity whose core membership can change between any two tokens via an ACK-latched epoch protocol that "never waits on a named core"; on an AMD Zen 5 box (8C/16T), inference output stays byte-exact under verified per-token membership churn on SmolLM2-135M and Qwen2.5-7B-Instruct, and elastic membership Pareto-dominates a fair static-8 split by 1.75×/1.52×/1.28× general throughput at 25/50/75% inference duty, recovering all eight stranded cores at idle and converging to the split at saturation.

## 核心贡献

- **"绝不等待命名核心"的 ACK-latch 历元协议**：将 seqlock 风格的 generation-tagged 参与者锁存与 RCU/epoch 风格的成员同意机制组合，使每个 token 的参与者集合 = 请求集合 ∩ 当前 epoch 已 ACK 集合，从构造上保证没有屏障会因某个核心的到达/迟到/离开而死锁。
- **生成标记的单一锁存字**：参与者掩码每个 token 只计算一次，与 token 生成号的低 32 位打包成单个字，在 token 开始的 Release bump 处发布；全部 5 个屏障等待点与工作线程自排除检查都读这一个字，确保等待侧与工作侧视图为同一快照。
- **硅上验证的比特级精确性**：在 135M 与 7B 两模型上，成员掩码真实变化（135M：8 次变化经 6 个不同集合）下，贪婪 token 序列与最终 logits 与定宽参考运行字节一致（first-divergence index −1，final-logits hash `0x9a2917eca1a62bd1`），且负控制被正确检出。
- **工作保守的 Pareto 主导**：在同一二进制内对静态 8 核分区的策略消融中，中间占空比下弹性帮派在等优或更优推理吞吐的同时交付 1.75×/1.52×/1.28× 通用吞吐，空闲时回收全部 8 个搁置核心，饱和时收敛于分区——无固定 K 能跟踪此前沿。
- **两个存储体制下的饱和拐点**：7B（DRAM 带宽受限）与 135M（缓存驻留）均在帮派宽度 8 饱和（7B 12.72 tok/s；135M 1352.5 tok/s），超过拐点核心近乎免费，使弹性出让有利可图。
- **治理作为一等调度杠杆**：每代理宪法信任分数接入为借用准入配额，在信任权重 1→4 范围内单调拨动机器于弹性扫描两端（推理吞吐 343.82→572.35 fwd/s，+66%）。

## Key Contributions

- **The never-wait-on-a-named-core ACK-latch protocol**: A seqlock-style generation-tagged participant latch composed with RCU/epoch-style membership consent, so each token's participant set is `requested ∩ acked-current-epoch`, with a formal invariant argument for why no barrier can hang on a departed or late core and why no stale ack can ever be latched.
- **The generation-tagged participant latch as a single packed word**: Membership is snapshotted once per token (`(token_gen as u32) << 32 | mask`) and published by the token-start Release bump; all five barrier wait sites plus every worker's self-exclusion read this one tagged word, guaranteeing wait-side and worker-side observe the same snapshot and that membership changes take effect only at the next token.
- **Bit-exactness under membership churn on silicon**: For both SmolLM2-135M and Qwen2.5-7B-Instruct, the greedy token stream and final logits are byte-identical to a fixed-width reference under verified per-token membership change, with a deliberately corrupted negative control proving the equality test can fail — a correctness/safety event in a kernel whose safety gate reads logits.
- **Work-conservation Pareto dominance via policy ablation**: Because the static-K baseline is a restriction of the same shipped binary, the duty-cycle sweep is an ablation of scheduling policy alone; elastic membership delivers 1.75×/1.52×/1.28× static-8's general throughput at 25/50/75% duty at equal-or-better inference, recovers all eight stranded cores at idle, and dominates static-12's general throughput 2.3–4.7×.
- **Auto-sizing to the saturation knee in two memory regimes**: Decode saturates at gang width 8 in both the DRAM-bandwidth-bound 7B regime and the cache-resident 135M regime, so ceding cores past the knee is nearly free for inference — the precondition that makes elasticity profitable rather than merely possible.
- **Governance as a first-class scheduler lever**: A per-agent constitutional trust score wired in as a lend-admission quota monotonically dials the gang's core share between the elastic sweep's two endpoints, reported as a measured property of the substrate.

## Methodology

The paper reports the design, implementation, and silicon measurement of the "elastic gang" inside Anima OS, a bare-metal x86_64 `no_std` Rust kernel (~232,000 lines; ~290,000 across the project). The central design observation is that a hard-barriered computation can tolerate elastic membership if and only if no barrier ever waits on a named core.

**The protocol (§3).** Four pieces of lock-free SeqCst atomics compose the mechanism:
1. `GANG_EPOCH`: a global generation-ownership epoch initialized to 1 (not 0), bumped at both `gang_begin` and `gang_end`, so any stale ack dies by construction.
2. `GANG_ACK[i]`: per-core, cache-line-padded; core *i* participates in a token iff `GANG_ACK[i] == GANG_EPOCH` at latch time.
3. `GANG_ACTIVE`: the lend-invitation flag.
4. `GANG_OWNER`: a single-owner CAS (`0 → owner`).

A lending core's service loop obeys one load-bearing ordering rule — **resync-before-ack** (snapshot the token-generation counter before storing the ack), which prevents a latched-but-never-walked hang. The participant mask is computed once per token and packed with the low 32 bits of the token's generation number; this single tagged word is the *only* object read at any barrier wait site or worker self-exclusion check. The generation tag closes failure mode 2 (silent corruption on late/double arrival): a worker whose read straddles a token-start bump sees a tag that is not the token it was admitted to and bails instead of double-walking.

**Membership lifecycle.** `gang_begin` (owner CAS → epoch bump → active-flag set) never aborts a generation; a loser runs un-ganged at base width. `gang_end` is owner-keyed and idempotent, called at all four exit paths (done, faulted, abandoned, lease-fault-cleared), and rings a dispatch doorbell so lent cores wake off MWAIT and retract. Only cores inside the engine's worker range are lend-eligible; a designated high-priority scheduler core never lends; lending is always the lent core's own self-claiming act, never a remote grab. Parked tenants are migrated by the scheduler's service pass (CAS `RUNNABLE→MIGRATING→re-enqueue`, paced ≤2 per pass), and a fault-handler bracket-open flag guarantees that a page fault mid-syscall still releases the gang.

**Implementation & baselines (§4).** The static-K baseline is not separate machinery but a mode of the same binary: in static mode `gang_begin` returns false before touching state, which is bit-identical to the no-lendable-core dead path (goal G5). The measurement harness (`sched_bench`, 2,567 lines) is an in-kernel self-driving autorun matrix emitting grep-able `[sched:...]` lines to NVMe-persisted boot logs. The headline metric is `general_served_s`, a monotone per-slot served-quanta counter never re-based on migration — a deliberate correction after the author found the naive virtual-time metric contaminated by migration (showing physically impossible inflation, 118–180% spread).

**Setup (§5.1).** AMD Ryzen 9 9800X3D (Zen 5, 8 physical cores / 16 hardware threads), DDR5-6000 dual channel (~75 GB/s), bare metal, no hypervisor. Models: SmolLM2-135M (Q4_0, 87.6 MB) at ~0.75 ms/token (stresses per-token elasticity ~100× harder) and Qwen2.5-7B-Instruct (Q4_0, 4,238 MB) at ~79 ms/token (bandwidth-bound). Workload: 24 synthetic preemptive AOT-spin tenants, one timer-paced pinned latency tenant (a disclosed worst case), and a generation driver issuing 32-token bursts at duty d ∈ {0, 25, 50, 75, 100}%.

## Results & Highlights

- **Work conservation vs. static-8 (Table 2, 135M, 24 tenants)**: At inference duty 25/50/75%, the elastic gang delivers **574.0 / 496.8 / 417.1 general served-quanta/s** versus static-8's flat **≈327** — i.e. **1.75× / 1.52× / 1.28×** general throughput at equal-or-better inference throughput (140.0 / 275.5 / 418.8 fwd/s vs. static-8's 59.3 / 120.8 / 177.9 fwd/s). Headline-metric run-to-run spread is **0.1–0.9%** in every cell.
- **Full core recovery at idle**: At duty 0% elastic recovers all eight cores a static split strands (**653.9 vs. 327** served-quanta/s); at duty 100% the general axes converge (**326.9 vs. 327.0**). There is no operating point at which static-8 wins anything.
- **Bit-exactness under membership churn (§5.5)**: Under verified mask changes (135M: 8 changes through 6 distinct participant sets; 7B: 6 changes through 4 sets), output is byte-identical to a fixed-width reference — first-divergence index **−1** and identical final-logits hash **`0x9a2917eca1a62bd1`** — with a deliberately corrupted negative control again detected. A companion experiment (`bitexact-lend`) pins width entirely and lets real lending (14 lend events, 3 retractions, 5 ack-driven mask changes) drive membership; output is again byte-identical.
- **Saturation knee (Table 4, solo)**: Both models saturate at gang width 8 — the 7B at **12.72 tok/s** (78.6 ms/token, vs. a ~55 ms/token theoretical DRAM floor for a 4.2 GB weight stream at 75 GB/s), the 135M at **1352.5 tok/s** (declining ~6% past width 12). Widths 9–16 buy <1% for the 7B, confirming ceded cores are nearly free.
- **Borrow/return latency (§5.6)**: Returning a lent core (epoch bump → ack retraction) costs **p50 = 0.22 µs, p99 = 0.38 µs, max = 0.42 µs** (n=224); the migration primitive costs **p50 = 0.22 µs, max = 0.7 µs** per migration. By contrast, acquiring a *busy* tenant-occupied core costs one WRR quantum — per-core medians **15.2–16.4 ms**, worst **20.1 ms** — by deliberate policy (no mid-slice preemption for lending).
- **External Linux/EEVDF anchor (Table 3, same silicon)**: Under 24-process load, unpinned 8-thread `llama.cpp` decode collapses **~800×** on the 135M (1230 → 1.5 tok/s) and **~62×** on the 7B (14.63 → 0.235 tok/s); `taskset` pinning restores ~975 tok/s but strands ~half the core budget when inference is idle.
- **Governance dial (Table 5)**: A trust-weighted lend-admission quota monotonically raises inference throughput **66%** across the trust range (343.82 → 572.35 fwd/s) while general throughput falls 42% (559.89 → 327.36 served-quanta/s); trust-4 reproduces the shipped full-lend behavior.

## Authors & Credibility

The paper is **single-authored by Daeyeon Son, an independent researcher in the Republic of Korea**, building on two prior companion papers of his own: ProbeLogits (arXiv:2604.11943, the in-kernel inference engine) and Governed MCP (arXiv:2604.16870, the logit-based safety gate). This places the work in a coherent research program rather than an isolated result. The author self-identifies the mechanism as residing in a self-built ~232k-line bare-metal Rust kernel (Anima OS), a Theseus-style ambitious single-contributor OS effort. Credibility is bolstered by an unusually high level of methodological honesty: the author proactively discovered and disclosed a migration-contaminated metric, replaced it, and explicitly warns that the corrected (lower) numbers are "the honest ones." Every claim is gated by a silicon-measured negative control or a fairness sanity check.

## Critical Assessment

- **Novelty**: High and precisely scoped. The per-token membership change of a *running hard-barriered gang* is exactly the case that classic gang scheduling (Ousterhout; Feitelson & Rudolph), demand-based/flexible co-scheduling, and even µs-scale reallocators (Shenango, Caladan) explicitly bracket out — those either co-schedule fixed-membership time slices or reallocate fungible threads. The "never wait on a named core" invariant plus the single gen-tagged latch word is a genuinely novel composition of seqlock + RCU/epoch primitives.
- **Rigor**: Strong. Invariants I1–I8 are stated as a formal table and each is tied to a specific design decision; bit-exactness claims are backed by both a width-churn experiment and a width-pinned lending experiment, with a deliberately corrupted negative control. However, the safety experiments exercise only per-token joins and generation-boundary departures — the author acknowledges mid-token departure is "not because the test declines to force it, but because the protocol makes it unrepresentable," which is a design argument, not an empirical one.
- **Reproducibility**: Mixed. The harness emits every number as grep-able `[sched:...]` lines persisted to NVMe, and the static baseline is a mode of the same binary (a clean ablation). But the entire evaluation runs on **one AMD Zen 5 box, single vendor, single machine**, and the kernel itself (Anima OS) is not stated to be open-source — only the harness logs and parsing scripts accompany the artifact. The mechanism is ISA-generic (atomics, MWAIT, TSC) but the constants (knee at 8, 16 ms quantum) are machine-specific.
- **Significance**: Meaningful for the GPU-less edge/appliance CPU-decode target class, where a barriered CPU-SIMD gang genuinely competes with the OS for cores. Less impactful for datacenter GPU serving, where the "CPU gang this paper elasticizes does not exist" (author's own caveat).
- **Red flags / specific weaknesses**:
  - The **duty-100 inference-axis gap (571.7 vs. 232.4 fwd/s)** between elastic and static-8 is a placement/SMT effect, not a lending effect — both gangs run at native width 8 with lending quiescent — and the author correctly excludes it from headline claims, but it means the headline Table 2's inference column is partly an artifact of SMT-sibling contention in the static baseline rather than a clean policy comparison.
  - The **external Linux baseline is only a "directional anchor"**: it uses a different engine (`llama.cpp`) and a different scheduler (EEVDF), so it supports shape/ratio claims only, never unit-for-unit comparison; a controlled port of the same engine to a Linux userspace harness remains future work.
  - **Acquiring a busy core is quantum-bound (~16 ms), not µs-scale** — the µs-scale numbers quoted for return/re-membership do not characterize the worst-case ramp cost on a fully busy machine; an isolated generation shorter than a quantum runs mostly at its already-acked width.
  - The **governance-share sweep (§5.7)** is limited to one model, one workload matrix, via a bench override, and its granularity is intrinsically admission-time; no headline claim depends on it.

## Strengths

- Mechanism is justified by a precise failure-mode analysis (barrier deadlock on departure, silent corruption on late/double arrival, stale-ack resurrection) with each failure mode mapped to a protocol element that prevents it.
- The static-K baseline being a *mode* of the same binary turns the duty sweep into a clean policy ablation — implementation drift is ruled out by construction.
- Exceptional methodological honesty: the author replaced a contaminated metric, disclosed worst cases (pinned latency tenant: 2.8 s p99 at duty 100%), and refrained from headlining the SMT-confounded inference-axis gains.
- Bit-exactness is demonstrated under *both* membership drivers the protocol has (forced width churn and genuine ACK-latch lending), with a negative control proving the equality test can fail.

## Limitations

- Single machine, single vendor (one Zen 5 box); absolute constants (knee at 8, 16 ms quantum) are machine-specific.
- SMT topology is a confound: lending an SMT sibling of a busy core is not lending a physical core, and is the most likely explanation for static-8's inference degradation under oversubscription.
- The Linux anchor uses a different engine and scheduler; a controlled same-engine port to Linux is future work.
- Busy-core acquisition is quantum-bound (~16 ms), not µs-scale; isolated short generations on fully busy machines benefit little.
- No GPU evaluation; the target class is explicitly GPU-less edge/appliance decode.
- Governance-share sweep is narrow (one model, one matrix, bench override) and admission-time granular.
- The safety experiments exercise per-token joins and generation-boundary departures only — mid-token departure is argued unrepresentable rather than empirically tested.

## Relevance

This paper is highly relevant to **Computation Reliability & Precision Errors** and **ML Reliability & Mathematical Foundations**: its central correctness result — that logits remain bit-exact (final-logits hash `0x9a2917eca1a62bd1`) under verified per-token core-membership churn, where a single extra or missing gang core is treated as a *safety event* because logits feed the kernel's safety gate — directly addresses silent corruption in the LLM forward pass. It also connects to **LLM Inference Evaluation** by characterizing the CPU-decode saturation knee and the collapse-vs-strand tradeoff that any on-device inference substrate must navigate.

## Questions for Further Exploration

- Could a `preempt-for-lend` policy (trading tenant latency for sub-quantum gang ramp) materially change the duty-25 acquisition cost, and would it ever be worth the tail-latency hit?
- Does the bit-exactness invariant survive non-greedy sampling (top-p, temperature), FP16/BF16 (not just Q4_0) kernels, or attention kernels with non-row-partitionable reductions?
- How does the knee (and thus the profitability of elasticity) shift on ARM/Apple Silicon or with a different L3/DRAM ratio — i.e., is width-8 saturation a Zen 5 artifact or a deeper property of single-socket decode?
- Can the single gen-tagged latch word scale beyond 32 cores (the current latch range), or does the protocol need a different publication structure for high-core-count CPUs?
- Would a controlled same-engine port to a Linux userspace harness (closing the §5.3 directional-anchor gap) preserve the collapse-vs-strand tradeoff, or do Linux's NUMA/cache-partitioning features blunt it?
- Is the "lost or extra gang core is a correctness event" framing extensible to detecting silent data corruption (SDC) in the matvec itself — e.g., could the per-wave exactly-once invariant be instrumented as an SDC detector?
