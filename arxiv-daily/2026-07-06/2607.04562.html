Key finding

A weight-free wrapper gates every LLM commit behind a Heaviside step — advancing only when non-LLM checkers confirm the claimed real repository state.

HCRC demotes any LLM to a pure proposer. A commit advances iff H(C·V − τ) = 1 — at τ=1, only when confidence and a parallel pool of filesystem / AST / test-runner / citation checkers all agree. No model weights touched.

一句话总结：HCRC 是一个不改权重的执行层验证包装器，用 Heaviside 门函数（仅当置信度×外部验证分 ≥ τ 时放行）和非 LLM 的 ARIAL 工人池（文件系统/AST/测试检查）把 LLM 降格为"提议者"，在合成软件合成任务上将"虚假完成率"从 4–7% 压到 0%（强模型），并反而更快。

Why this matters

An execution-layer defense against silent corruption.

An execution-layer defense against silent data corruption in autonomous LLM loops — external, decidable predicates block hallucinated state from propagating downstream, decoupling correctness from raw model capability.

Critical edge

The headline FCR→0 result is partly circular: at τ=1 with a full pool, gate-certified FCR is 0% by construction (Thm 6.3) — and the synthetic task's predicate set is generated alongside the task, so the verifier is handed the answer key.

  • Rolling Coefficients R=(R₁…Rₙ): each step carries a predicate set Pᵢ that must hold post-completion.
  • ARIAL worker pool — non-LLM checkers (Validator, Syntax Guard, Test Runner, Citation Guard, Gap Miner) mine disjoint evidence channels, aggregate Vᵢ=Σ wₖvₖ.
  • Key design rule: verification never reads the LLM's internal state / KV-cache — it reads the repository directly (sidesteps model-as-judge bias).
  • Hidden test: a command_succeeds predicate the proposer never sees.
  • Hyperparams: λ=0.15, β=2.0, τ=1, r̄ₘₐₓ=3.

Strengths & limitations

side by side

▲ Strengths

  • Clean reframing: hallucination as a claim/reality divergence (epistemic entropy) a control layer can monotonically reduce — not a calibration problem to train out.
  • Sound design rule: the verification channel reads only the filesystem, never internal state — sidesteps model-as-judge bias and KV-cache contamination.
  • Honest metrics: distinguishes false-completion from honest-halt; residual wrapped FCR attributed to retry-budget exhaustion.
  • Counter-intuitive cost result: HCRC is faster on capable proposers — a non-obvious, well-explained finding.
  • Production extensions (verification-slaved UI, gate-licensed memory compaction, "exhumation rate" vital) are thoughtful reliability engineering.

▼ Limitations

  • Synthetic-only headline: FCR→0 rests on tasks "strictly easier than SWE-bench"; the one real-benchmark run suppresses the hidden test.
  • Load-bearing assumptions: soundness (6.1) and channel independence (6.2) are assumed, not verified — a buggy predicate collapses the guarantee.
  • Predicate completeness: shipped set covers existence/parse/containment/command-success — not semantics, security, or performance.
  • Small / uneven samples: Claude Opus n=3 (excluded "to remove bias"); several partial free-tier runs; non-standard model IDs (gpt-5.4, claude-opus-4-8).
  • Production claims anecdotal: no measured in-the-wild FCR or exhumation-rate telemetry.

Questions for further exploration

open threads
How does HCRC perform when the predicate set P must be discovered (not generated with the task) — e.g. full SWE-bench Lite with test_passes enabled and repos cloned?
What is the measured in-the-wild FCR and exhumation rate from the "Coefficient of Chalk" production deployment? The paper asserts benefits but reports no telemetry.
Can the sound-predicate assumption (6.1) be weakened or verified? A buggy/adversarial predicate collapses Thm 6.3 — a concrete SDC-in-the-verifier problem.
The non-monotonic τ result (5% leak at 0.85, clean at 0.7) — is the τ<1 FCR surface fundamentally workload-dependent, and can it be characterized a priori?
Does the gate's benefit transfer to non-software domains (math proving, DB migration) where decidable predicates are harder to construct than filesystem checks?
Could the honest-halt / retry-exhaustion regime become a streak-conditioned budget policy with provable termination bounds?
