> **一句话总结：** 基于中国计算机等级考试（NCRE）构建了OfficeEval基准，包含200个Word/Excel/PPT实操任务和7118个机器可评分标准，测试7个前沿LLM后发现最强单轮模型仅得36.6%，最强智能体系统得68.8%，远低于95.5%的人类参考分，揭示了当前LLM在细粒度Office文档自动化上的严重不足。

> **TL;DR:** Constructs OfficeEval from China's NCRE exam (200 tasks, 7,118 criteria) and benchmarks 7 frontier LLMs: best single-turn scores 36.6%, best agentic system reaches 68.8%, both far below the 95.5% community reference, exposing a critical gap in fine-grained Office automation capability.

## 核心贡献

- 基于中国计算机等级考试（NCRE）构建OfficeEval基准，涵盖200个实操任务、7118个确定性评分标准，横跨Word、Excel、PowerPoint三个应用
- 提出8类Office技能分类法（文本格式、布局设计、表格、图表、数据公式、图形媒体、动画、文档属性），支持细粒度能力分析
- 揭示关键发现：即使智能体系统代码执行成功率达98-99%，操作准确率仍然很低——模型能写出可运行的代码，但无法实现正确的Office语义操作
- 提供了基于标准等级考试的量化评分锚点（60分及格线、95.5%社区参考分），使基准具有外部可解释性

## Key Contributions

- **OfficeEval benchmark:** 200 NCRE practical-operation tasks across Word (78 tasks), Excel (71 tasks), and PowerPoint (51 tasks), with Level 1 (foundational) and Level 2 (advanced) difficulty, each scored by 20-70 deterministic criteria on a 100-point rubric
- **Comprehensive skill taxonomy:** 8 skill categories across 7,118 criteria — Text & Format (38.6%), Layout & Design (17.6%), Tables (10.8%), Charts (5.1%), Data & Formulas (2.4%), Graphics & Media (9.1%), Animation (7.8%), Doc. Properties (8.6%)
- **Critical performance gap:** No single-turn model reaches the 60-point reference threshold (best: Claude Opus 4.7 at 36.6%); best agentic system (Codex) reaches 68.8% but remains far below the 95.5% community reference
- **Execution vs. accuracy decoupling:** In the coding-agent setting, code execution success rises from <50% to 98-99%, but operation accuracy remains low — agents can write runnable code but struggle with correct Office-specific semantic operations

## Methodology

The benchmark construction and evaluation pipeline:

1. **Data source:** Tasks extracted from NCRE Level 1 and Level 2 MS Office modules, covering basic formatting to mail merge, pivot tables, chart customization, and complex animations. Task instructions are in Chinese with visual reference images.

2. **Task format:** Each task has (a) an input document (.docx/.xlsx/.pptx) with supporting materials, (b) natural language instructions (5-15 sub-tasks), and (c) a machine-readable XML scoring configuration.

3. **Evaluation engine:** Parses Office Open XML (ISO/IEC 29500) using the Open XML SDK, with some checks invoking Microsoft Office COM automation. The engine is deterministic for a fixed environment.

4. **Two evaluation paradigms:**
   - **Single-turn:** Models receive task instructions + document screenshots + file listing, output Python code using python-docx/openpyxl/python-pptx
   - **Coding agents:** Claude Code and Codex operate in an isolated directory with the raw exam paper (RTF) and input files, with execution feedback and iterative repair

5. **Metrics:** Score Rate (SR) = mean percentage of rubric points earned across tasks; 3 independent runs per model at temperature=1.0

## Results & Highlights

- **Single-turn models (7 tested):** Claude Opus 4.7 leads at 36.6% SR, GPT-5.5 at 36.1%, then a steep drop to Gemini 3.1 Pro (16.3%), Kimi-K2.6 (9.4%), and Grok-4.1-fast (2.8%)
- **Coding agents:** Codex reaches 68.8% SR, Claude Code reaches 53.0% — substantial improvement over single-turn but still far from the 95.5% community reference
- **Per-skill analysis:** GPT-5.5 leads in Charts (51.4%), Data & Formulas (48.5%), and Animation (26.7%); Claude Opus 4.7 leads in Text & Format (46.6%), Tables (59.1%), and Doc. Properties (51.1%). Animation and Graphics & Media are consistently weakest across all models
- **Task difficulty:** Only 1 of 200 tasks defeats all models; 33.5% are partially solved by all; 34 tasks discriminate sharply between top and bottom tiers (up to 100 percentage point spread)
- **Key insight:** The gap between execution success (98-99%) and operation accuracy reveals that current agents write syntactically valid code but fail at the semantic level of Office operations

## Critical Assessment

- **Novelty:** High for the benchmark construction. Using a nationally administered professional certification exam as the basis for LLM evaluation is genuinely novel and provides strong external validity. The fine-grained, criterion-level grading is a significant improvement over binary pass/fail benchmarks.
- **Rigor:** Very high. 7,118 deterministic scoring criteria eliminate LLM-as-judge variance. The 60-point reference threshold and 95.5% community reference provide meaningful anchors. Three-run averaging accounts for stochasticity.
- **Reproducibility:** Moderate. The evaluation engine is well-described but requires Windows + Microsoft Office + specific Open XML SDK. The raw NCRE tasks are copyrighted and cannot be redistributed, though the authors provide full prompts and experimental details.
- **Significance:** This is an important contribution to LLM evaluation. It exposes a concrete, practically relevant capability gap that existing benchmarks (SWE-Bench, web navigation) do not capture. The finding that execution ≠ accuracy is particularly significant.
- **Red flags:** The benchmark is Chinese-language-centric, which may conflate language understanding with Office automation capability (though the authors provide English translations for cross-language analysis). The coding agents have access to real execution environments (Windows + Office COM), which creates a dependency on specific infrastructure. The community reference solutions are informal (created by exam candidates, not professionals), so the 95.5% ceiling should be interpreted cautiously.

## Strengths

- Externally validated benchmark grounded in a nationally administered certification exam
- 7,118 deterministic criteria provide dense, fine-grained evaluation signals
- Both single-turn and agentic paradigms evaluated on the same tasks
- Clear, practically meaningful scoring anchor (NCRE pass/fail threshold)
- Per-skill taxonomy enables diagnostic capability analysis
- Honest assessment of limitations and data availability constraints

## Limitations

- Chinese-language task instructions may conflate language and Office automation capability
- Raw NCRE materials are copyrighted and cannot be publicly redistributed
- Evaluation requires Windows environment with Microsoft Office installed
- Community reference solutions (95.5%) are informal and not a formal human baseline
- Only two coding agents tested (Claude Code, Codex); other agent frameworks not evaluated
- The python-docx/openpyxl/python-pptx libraries have limited API coverage for some Office features (especially animations, SmartArt, transitions), which may partially explain low scores in those categories

## Relevance

与用户对LLM推理评估的兴趣直接相关——本文构建了一个基于真实专业考试的LLM能力基准，揭示了当前最强模型在细粒度文档自动化任务上的显著不足，且通过确定性评分标准实现了可复现的评估，属于LLM评估领域的高质量工作。

## Questions for Further Exploration

- How much of the performance gap is due to library limitations (python-pptx lacking animation/SmartArt APIs) versus genuine model capability limitations?
- Would multimodal models with direct screen interaction (e.g., clicking UI elements) perform better than code-generation approaches?
- How does the cross-language analysis (Chinese vs. English instructions) affect model performance? Is the gap language-agnostic or language-specific?
- Can the fine-grained criterion-level scoring be used to create targeted training data for improving Office automation capabilities?
- What is the performance ceiling for agents that can execute code iteratively with unlimited turns?
