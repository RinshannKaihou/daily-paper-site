Decouple parallelism strategy from runtime via a unified training DAG and 5 scheduling directives — PP+EP+ZeRO in tens of lines.

Piper: A Programmable Distributed Training System

Megan Frisella · Shubham Tiwari · Andy Ruan · Yi Pan · Parker Gustafson · Mat Jacob · Gilbert Bernstein · Stephanie Wang

distributed-training pipeline-parallelism system-design compiler-IR LLM-training
Key Metric

6–30%

Throughput gain on composed strategies (DualPipeV) over Megatron-LM, TorchTitan, and DeepSpeed baselines.

Why This Matters

Provides a programmable runtime for auto-parallelism search and complex MoE training — where composing PP+EP+ZeRO cleanly is a real bottleneck.

Critical Edge

No tensor or context parallelism support — essential for training models beyond ~10B parameters on typical clusters.

Methodology

Compiler + runtime, two cleanly separated pieces

Piper splits the problem into a compiler that turns a traced PyTorch model and user directives into a global training DAG, and a distributed runtime that executes it.

Compiler — two phases.

  1. Chunk extraction. TorchDynamo traces the model into an fx.Graph, split at user annotation boundaries into coarse Chunks with no interleaved communication.
  2. DAG transformations. Scheduling directives are applied as mechanical graph rewrites — Place/Replicate/Shard insert comms nodes, Split duplicates the DAG for microbatching, Order adds temporal dependencies with nested filters for overlap.

Runtime. A centralized scheduler decomposes the global DAG into per-PP-rank sub-DAGs and orders tasks greedily by downstream-dependency count. Workers execute tasks asynchronously on assigned GPU streams, synchronized by CUDA events. Separate P2P stream communicators per direction avoid deadlock without requiring exact runtime prediction. Flat buffers and reference-counted activation freeing keep memory predictable.

Scheduling Directives

Five primitives do the work

DirectiveEffect
Place(filter, devices, stream)Assigns device placement; inserts P2P send/recv at cross-device boundaries.
Replicate(filter, devices, …)Replicates with gradient sync (allreduce / reduce-scatter); supports ZeRO-style parameter sharding.
Shard(filter, devices, stream)Shards weights along dim 0; inserts all-to-all before and after.
Split(filter, dim, n_mb)Duplicates a sub-DAG for microbatching.
Order(filter_list)Adds temporal ordering; nested lists enable overlapped execution.

Strengths

What's genuinely new

  • Clean separation of strategy specification from runtime execution
  • Minimal API (5 directives) that is surprisingly expressive — DualPipeV in 63 LoC
  • First system to correctly implement all PP × ZeRO combinations — a real practical contribution
  • Concrete evidence that independent per-dimension scheduling causes resource contention in existing frameworks
  • Honest comparison — acknowledges Megatron's fused-kernel advantage rather than hiding it
Limitations

What to be careful about

  • No TP or CP support — essential for > ~10B-parameter models on typical clusters
  • Requires fully traceable models (TorchDynamo limitation) — dynamic control flow unsupported
  • Evaluated only up to 32 GPUs; behavior at 100+ GPU scale unproven
  • Centralized scheduler is a single point of failure; no fault-tolerance discussion
  • Greedy scheduling heuristic is simplistic; no auto-tuning of directives
Questions for Further Exploration

What this paper leaves open

  1. How would Piper's scheduling extend to tensor parallelism, where communication patterns are more tightly coupled with compute?
  2. Can the greedy heuristic be replaced with a learned or profile-guided policy that adapts to workload characteristics at runtime?
  3. What is the compile-time overhead for large models with many annotations — does the centralized scheduler become a bottleneck at 100+ GPU scale?
  4. How does Piper handle fault tolerance and dynamic cluster changes (preemption, varying node counts)?
  5. Could Piper's IR serve as a compilation target for auto-parallelism search systems like Alpa or Galvatron?
  6. Is the 63-LoC DualPipeV schedule robust to different architectures — e.g. models with multiple MoE layers at different depths?
