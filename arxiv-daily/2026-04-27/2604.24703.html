Defective Task Descriptions in LLM-Based Code Generation: Detection and Analysis

arXiv: 2604.24703v1

Date: 2026-04-27

https://arxiv.org/abs/2604.24703v1

一句话总结： 本文提出 SpecValidator，一个基于 Qwen2.5-Coder-1.5B 的 LoRA 微调轻量分类器，用于自动检测 LLM 代码生成中的任务描述缺陷（词汇模糊、规格不足、语法格式错误），检测性能 F1=0.804 显著优于 GPT-5-mini 和 Claude Sonnet 4，并能发现真实基准测试中潜在的规格缺陷。
TL;DR: This paper introduces SpecValidator, a lightweight LoRA-fine-tuned 1.5B classifier that automatically detects defective task descriptions (Lexical Vagueness, Under-Specification, Syntax-Formatting) for LLM-based code generation, achieving F1=0.804 and significantly outperforming GPT-5-mini (F1=0.469) and Claude Sonnet 4 (F1=0.518). It also reveals that Under-Specification is the most damaging defect type, and benchmarks with richer context (LiveCodeBench) are substantially more resilient.

核心贡献

  • 系统性地定义了三类任务描述缺陷（词汇模糊 LV、规格不足 US、语法格式 SF），并构建了包含 6,573 个缺陷实例的公开数据集。
  • 提出了 SpecValidator，基于 Qwen2.5-Coder-1.5B 的 LoRA 微调分类器，仅需更新 0.14% 参数即可实现 F1=0.804 的缺陷检测，大幅超越 GPT-5-mini 和 Claude Sonnet 4。
  • 对 10 个不同规模的代码生成 LLM 进行了大规模鲁棒性分析，揭示了缺陷类型与基准描述结构对代码正确性的联合影响：规格不足是最严重的缺陷类型，导致 Pass@1 下降高达 15.3%。
  • 发现 SpecValidator 能泛化检测到真实基准测试中未被标注的潜在规格缺陷，经人工验证 72.7% 的标记实例确实存在 Under-Specification 问题。

Key Contributions

  • Defined three categories of task description defects (Lexical Vagueness, Under-Specification, Syntax-Formatting) and created a publicly available dataset of 6,573 defective instances across HumanEval, MBPP, and LiveCodeBench.
  • Proposed SpecValidator, a LoRA-fine-tuned 1.5B parameter classifier that updates only 0.14% of parameters and achieves F1=0.804, outperforming zero-shot and few-shot GPT-5-mini (F1=0.469) and Claude Sonnet 4 (F1=0.518) by wide margins.
  • Conducted a large-scale robustness analysis across 10 code LLMs, demonstrating that Under-Specification is the most damaging defect type (up to 15.3% Pass@1 drop), and that specification structure is a first-order moderator of robustness.
  • Demonstrated that SpecValidator generalizes beyond synthetic training data to detect real latent defects in benchmark ground truth, with 72.7% manual confirmation of flagged instances.

Methodology

Defect Taxonomy and Dataset Construction: The authors define three orthogonal defect types injected into task descriptions: (1) Lexical Vagueness (LV) -- replacing precise terms with vaguer synonyms; (2) Under-Specification (US) -- removing one explicit constraint; (3) Syntax and Formatting (SF) -- introducing typographical and layout noise. Defective descriptions are generated using GPT-5-mini, validated by an independent LLM judge (Qwen2.5-Coder-32B), and manually verified by three annotators.

SpecValidator Architecture: A sequence classifier built on Qwen2.5-Coder-1.5B, fine-tuned with LoRA (rank=16, alpha=32) on 8,766 labeled examples (clean + mutated), split 80/10/10. Three training regimes are compared: linear probing, LoRA, and full fine-tuning. The four-class classification task distinguishes Clean, LV, US, and SF.

Robustness Evaluation: Ten code LLMs spanning small (6-7B), large open-source (15-34B), and closed-source reasoning models (GPT-5-mini, Claude Sonnet 4) are evaluated with Pass@1 under greedy decoding across three benchmarks with original and defective descriptions.

Generalization Test: SpecValidator is applied to 456 original (unmutated) benchmark descriptions to assess whether it can detect real, unseen specification defects.

Results & Highlights

  • Under-Specification is the most damaging defect: US causes Pass@1 drops of 7.9-15.3% on HumanEval across all models, including GPT-5-mini and Claude Sonnet 4 which each decline by 9.7% despite >95% baselines.
  • Lexical Vagueness produces moderate, benchmark-dependent effects: Drops range from 1.3% to 12.2% on HumanEval but only 1.5-3.2% on MBPP, suggesting description style moderates impact.
  • Surface formatting noise is negligible: SF mutations cause drops below 3.7% across all models, indicating code-specialized LLMs are robust to typographical corruption.
  • LiveCodeBench is nearly immune: All Pass@1 deltas remain within +/-2.0% for all defect types, explained by the benchmark's rich contextual grounding (explicit I/O examples that encode constraints implicitly).
  • Model size does not determine robustness: Sensitivity is description-dependent rather than capacity-dependent; even SOTA reasoning models suffer similar degradation.
  • SpecValidator (LoRA) outperforms frontier models: F1=0.804, MCC=0.745 vs. GPT-5-mini few-shot F1=0.462 and Claude Sonnet 4 few-shot F1=0.557. LoRA also outperforms full fine-tuning (F1=0.788), likely due to reduced overfitting.
  • Real-world generalization: Of 456 clean benchmark descriptions, 83 are flagged as under-specified; these achieve near-zero Pass@1 across all 10 models, and 72.7% are confirmed defective upon manual inspection.

Authors & Credibility

The authors are from the University of Luxembourg (SnT / FSTC). Amal Akli is the lead author; Mike Papadakis, Maxime Cordy, and Yves Le Traon are established researchers in software engineering and software testing. The work is closely related to prior work by Larbi et al. (2025) from the same group on prompt robustness in code generation. The paper provides a publicly available replication package at https://github.com/Amal-AK/detecting_prompt_defects. The evaluation is thorough with 10 models, 3 benchmarks, multiple training regimes, and manual validation.

Critical Assessment

Novelty: The core idea of detecting prompt defects with a small fine-tuned model is moderately novel. The defect taxonomy is well-motivated but builds on prior work (Larbi et al., ReCode, NLPerturbator). The finding that a 1.5B LoRA model outperforms frontier models on this classification task is interesting but not entirely surprising given the specialized nature of the task and the limited meta-reasoning capabilities of even large models.

Rigor: The experimental design is strong -- three benchmarks, ten models, multiple training regimes, two random seeds, LLM judge + manual validation. However, the defect injection is artificial (GPT-5-mini generated), which raises questions about how representative these mutations are of real-world user prompts. The generalization experiment (RQ3) partially addresses this concern.

Reproducibility: High. Code and data are publicly available. All models except GPT-5-mini and Claude Sonnet 4 are open-source and evaluated locally. Training hyperparameters are specified.

Significance: The finding that benchmark ground truth contains latent specification defects is impactful for the code generation evaluation community. The practical utility of SpecValidator as a pre-generation quality checker is clear. However, the paper does not address the repair of detected defects, which limits immediate applicability.

Red flags: (1) The four-class classification setup may not reflect real usage where descriptions can have multiple simultaneous defects. (2) The generalization test relies on Pass@1 as indirect evidence for LiveCodeBench since manual inspection was not feasible due to problem complexity. (3) LoRA outperforming full fine-tuning may indicate the dataset is too small rather than a fundamental advantage of LoRA. (4) The paper uses greedy decoding only, which may underestimate variance.

Strengths

  • Comprehensive evaluation across 10 models, 3 benchmarks, and 3 defect types with clear research questions.
  • Strong practical contribution: a lightweight, locally deployable detector that outperforms much larger models.
  • The generalization experiment (RQ3) demonstrating real defects in benchmark ground truth is a compelling finding.
  • Well-structured defect taxonomy with clear mutation operators and quality validation pipeline.
  • Public release of dataset and code for reproducibility.

Limitations

  • Defects are synthetically generated and may not fully represent the distribution of real-world user prompt issues.
  • The three defect categories may not capture compound or mixed defects that occur in practice.
  • No mechanism for defect repair or mitigation is proposed -- detection alone leaves the user without clear next steps.
  • Only Python benchmarks are evaluated; generalization to other programming languages is untested.
  • Greedy decoding only; sampling-based evaluation could reveal additional sensitivity patterns.
  • The clean/US classification boundary remains the dominant error mode, suggesting fundamental ambiguity in detecting information absence.

Relevance

这项工作对理解 LLM 代码生成的鲁棒性和提示工程质量具有重要意义。它揭示了任务描述缺陷对代码正确性的系统性影响，并提供了一个实用的轻量级检测工具，对代码生成基准测试的设计和评估也有直接的启示作用。

Questions for Further Exploration

  • How does SpecValidator perform on multi-language code generation tasks beyond Python?
  • Can SpecValidator be extended to provide defect repair suggestions in addition to detection?
  • What is the interaction between multiple simultaneous defect types (e.g., LV + US) on code generation performance?
  • How does temperature sampling (non-greedy decoding) affect the observed sensitivity to different defect types?
  • Could SpecValidator be integrated into an agentic code generation pipeline to iteratively refine defective specifications before code generation?
  • How do the findings transfer to more complex, library-intensive tasks such as those in BigCodeBench?
