ESPO: Early-Stopping Proximal Policy Optimization

Detect reasoning failures on-the-fly during PPO rollouts using surrogate regret from logits already computed during sampling. Truncate early, concentrate negative TD errors at the failure point, and save 22%+ rollout tokens — with no extra model, no annotation, no additional forward passes.

Zihang Li, Rui Zhou et al. Tongyi Lab · Peking Univ. 2026-05-28
Key Metric
73.17%
avg accuracy · 7B scale · 3 benchmarks

+1.97pp over PPO, +1.41pp over DAPO — while using 839M vs 1,072M cumulative tokens (22% fewer).

Why This Matters

Zero-cost failure detection from logits already computed during sampling. No extra reward model, no human annotation. Drop-in to any existing PPO pipeline — only changes trajectory collection, not the update rule.

Critical Edge

No error bars or statistical tests reported. Only tested on DeepSeek-R1-Distill-Qwen at 1.5B/7B. No direct GRPO comparison. The 1.5B AIME24 score actually trails DAPO (23.87 vs 24.37).

Methodology

Step 01

Per-Step Surrogate Regret

For each sampled action a_t, compute g_t = max(log π(a_greedy|s_t) - log π(a_t|s_t), 0) — measuring deviation from the greedy action. Normalized using frozen batch-level statistics and clipped.

Step 02

EMA Accumulation

Smoothed via exponential moving average: z_t = α · z_{t-1} + (1-α) · g_tilde. Batch statistics are frozen during generation to ensure causal correctness.

Step 03

Value-Gated Termination

When z_t > β · max(V_φ(s_t), ε), the rollout is terminated. Higher predicted return grants more tolerance. A proportional controller adjusts β to hit a target stop rate.

Step 04

Absorbing Failure State

Truncation triggers a terminal failure reward r_fail. GAE produces concentrated negative TD error at the failure point, propagating backward through PPO updates. Critic warmup prevents premature truncation.

Strengths & Limitations

Strengths

  • Zero additional compute — regret computed from logits already available during sampling
  • No extra reward model, human annotation, or process supervision needed
  • Consistent improvements over both PPO and DAPO across nearly all benchmarks at both scales
  • 22%+ token savings that compound over the full training run
  • Thorough ablation (6 variants) isolating each component's contribution
  • Principled absorbing failure state formulation avoids non-stationary reward bias
  • Clean integration — only changes trajectory collection, not the PPO update rule

Limitations

  • Only tested on math reasoning; generalizability to code, logic, or other domains is untested
  • Single model family (DeepSeek-R1-Distill-Qwen) at two scales only
  • No error bars or statistical significance testing reported
  • Hyperparameter sensitivity not systematically analyzed
  • False positives could truncate recoverable trajectories in domains where backtracking is common
  • Critic warmup heuristic adds a tuning dimension; poor warmup causes 2.1pp drop
  • Compatibility with dense/process reward signals is unexplored

Open Questions

Scaling
Does ESPO hold at 32B/70B+ and across other model families (Llama, Mistral)?
Dense Rewards
How does ESPO interact with process reward models? Could surrogate regret combine with step-level reward signals?
False Positives
What is the false positive rate dynamics over training — does it decrease as the policy improves?
Inference
Could the method extend to inference-time early stopping for test-time compute scaling?
Generality
How does ESPO perform on code generation where recovery from early mistakes may be more common?
Auxiliary Signal
Can the surrogate regret be used as an auxiliary training signal — e.g., for learning when to backtrack?
