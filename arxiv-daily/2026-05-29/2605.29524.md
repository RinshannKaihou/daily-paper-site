> **一句话总结：** KBF提出一种低成本黑盒审计协议，利用大模型知识边界附近稳定且独特的数值召回（包括可重复的错误答案）作为指纹，通过自适应前沿搜索生成探测集，在16个生产LLM端点上零误报检测出全部155个经济相关的模型替换，并在6个影子API平台审计中发现7/27端点不一致，问题集中在高端Claude端点。
>
> **TL;DR:** KBF is a black-box relay-auditing protocol that fingerprints LLM APIs using stable numerical recall near the knowledge boundary -- including repeatable wrong answers. Through adaptive frontier search, it generates compact probe sets that are configuration-invariant. Across 16 production endpoints (8 model families, 3 price tiers), it detects all 155 economically relevant substitutions with zero false positives, handles mixed routing at 5-10% substitution rates, and reveals 7/27 flagged endpoints in a 6-platform shadow API audit concentrated on premium Claude endpoints.

## 核心贡献

- 提出核心观察：LLM在知识边界附近的数值召回产生稳定、模型独特的值（包括可重复的错误答案），这比自识别、风格或高方差续延行为是更强的审计信号
- 设计KBF三阶段协议：自适应前沿搜索生成探测集 -> 配置稳定性过滤与可选对比筛选 -> Clopper-Pearson二项检验进行端点一致性判定
- 在16个生产端点（8个模型家族、3个价格层级、100倍价格差距）上验证，零误报检测出全部155个经济相关替换，含12/12族内替换
- 检测仅5%流量替换的混合路由攻击（Opus相关配对达MDR95=5%），两轮协议可估计路由比例（|bias|<0.005）
- 对6个影子API平台进行实地审计，以约$10成本发现7/27端点不一致，6/7集中在Claude端点，且低价格层级被标记频率显著高于高价格层级
- 公开代码、数据和即用探测集（https://github.com/Ooo0ption/KBF.git）

## Key Contributions

- Identifies knowledge-boundary numerical recall -- including stable-but-wrong answers -- as a model-distinct, configuration-invariant audit signal superior to self-identification, style, or high-variance continuation behavior
- Designs the KBF three-stage protocol: adaptive frontier search for probe generation, configuration-stability filtering with optional contrastive screening (against Qwen3.5-9B), and Clopper-Pearson calibrated binomial test for endpoint consistency decisions
- Achieves 0/16 false positives and 155/155 true positives across 16 production endpoints spanning 8 model families and a 100x price spread, including all 12 within-family economically relevant substitutions
- Detects mixed routing with MDR95 as low as 5% for Opus-based pairs; two-round extension provides essentially unbiased routing-fraction estimates (|bias|<0.005) when the substitute is known
- Conducts a real-world shadow API audit across 6 platforms for ~$10, flagging 7/27 endpoints as inconsistent, with 6/7 flags on Claude endpoints and clear tier-dependent serving behavior
- Releases code, data, and ready-to-use benchmark probe sets for reproducibility

## Methodology

KBF operates in three stages:

**Stage 1 -- Probe Generation (Adaptive Frontier Search):** For each of 15 numerical knowledge domains (chemistry, biology, astronomy, math/OEIS, cryptography, etc.), KBF asks the reference endpoint to propose candidate numerical facts at increasing difficulty tiers. Each candidate is re-queried under multiple deployment configurations (prompt variants, decoding settings) to test stability. Only candidates producing valid, stable consensus values survive. An optional contrastive screening step queries a cheap model (Qwen3.5-9B) and discards probes where the contrast model agrees with the reference, biasing the probe set toward higher separation from likely substitutes. A domain stops after two consecutive zero-yield rounds once at least five stable probes are enrolled.

**Stage 2 -- Self-Calibration:** The enrolled probe set is re-queried on the reference endpoint under the audit configuration to measure ordinary self-noise. The self-discrepancy count k_self is converted to a conservative null bound p_0 using the one-sided Clopper-Pearson upper confidence bound at gamma=0.99. This bound is model-specific and probe-count-aware: stable endpoints get tight bounds, noisy ones get wider bounds.

**Stage 3 -- Suspect Endpoint Audit:** Enrolled prompts are sent to the suspect endpoint. Each response is parsed with domain-specific rules; invalid, unparseable, or non-matching answers count as discrepancies. If the observed discrepancy count exceeds what the calibrated null bound p_0 can explain (one-sided binomial test at alpha=0.05), KBF reports "Different"; otherwise "Same."

**Adaptive Routing Extension:** For mixed routing (partial substitution), a two-round protocol concentrates extra queries on probes that mismatched in round 1. The two-round statistic T=W1+W2 is compared against a null distribution derived under fixed-probability routing. When the substitute is known, a method-of-moments estimator recovers the routing fraction with |bias|<0.005.

The entire protocol uses only black-box API access -- no weights, logits, or server-side metadata. Audit probes are ordinary-looking factual queries rendered in cloze-style templates.

## Results & Highlights

**Detection Accuracy (Section 4.2):** 155/155 economically relevant substitutions detected at p<0.05, including all 12 within-family downgrades (e.g., Claude Opus 4.6 vs. Claude Sonnet 4.6 in both directions). Zero false positives on 16 same-reference controls. Hardest cases still reject at p<0.001.

**Cost (Section 4.3):** One-time reference enrollment for 6 T1 models: $7.07 total (~1.94M tokens). Online audit of all 16 models: $0.39 total. Cheapest audit under $0.002 (T3 models); most expensive $0.243 (Claude Opus 4.6 with 681 probes).

**Configuration Robustness (Section 4.4):** Under 6 deployment wrappers (role prompts, temperature 0.1-0.7, RAG augmentation) across 5 models: KBF achieves 0/30 false positives and 60/60 true positives. Baselines fail: MET produces 29/30 FP (distribution shifts under wrappers), LLMmap misses 14/60 substitutions, ZeroPrint has 23/30 FP.

**Extended Robustness (Section 4.5):** Identity spoofing (15 models told "You are Claude Opus 4.6"): 0/15 false positives. Temporal stability: all 16 models stable for ~7 weeks; Qwen3.5-397B-A17B drifts at week 9 (self-error 1.7% to 7.8%), suggesting a ~9-week probe refresh cycle. Quantization: McNemar's paired test separates fp4/fp8 variants for GLM-4.7 and DeepSeek-V3.2. Agent CLI probing: 4/4 agent sessions (Claude Code, OpenClaw, Codex) match API baselines.

**Mixed Routing (Section 4.6):** MDR95 ranges from 5% (Opus->Kimi-K2) to 50% (Sonnet-based pairs). Detection driven by n_01 count (probes where substitute mismatches but reference stays correct). Routing-fraction estimator: |bias|<0.005 when substitute known; interval width 0.15-0.30 in unknown-substitute scenario over 4-model candidate pool.

**Shadow API Audit (Section 4.7):** 6 platforms, 27 platform-model endpoints audited for ~$10. 7/27 flagged as inconsistent (26% flag rate). Claude Opus 4.6 flagged on 3/4 platforms serving it; Claude Sonnet 4.6 on 3/6. Other models (GPT-5.4, DeepSeek-V3.2, Gemini 3 Flash) consistent everywhere. Tier-dependent behavior: low-tier endpoints flagged 5/7 for Opus vs. 1/7 for upgraded tier, confirming economic targeting.

## Authors & Credibility

Authors: Yijia Fang, Yiqing Feng, Bingyu Li, and Mingxun Zhou (corresponding author). The work is submitted to cs.CR (Cryptography and Security) and cs.AI. The evaluation is unusually thorough: 20 pages, 13 figures, controlled experiments across 16 production endpoints, comparison with 3 baselines (MET, LLMmap, ZeroPrint), and a real-world field audit. The paper openly acknowledges limitations, including the binomial independence assumption and the need for larger false-positive calibration studies. Code and probe sets are publicly released.

## Critical Assessment

- **Novelty:** High. The key insight -- that knowledge-boundary numerical recall, including stable wrong answers, provides a strong audit signal -- is both creative and well-grounded. The adaptive frontier search for probe generation is a practical engineering contribution that goes beyond simply observing the phenomenon.
- **Rigor:** Very strong. The paper covers 155 economically relevant substitution pairs with statistical testing (Clopper-Pearson calibrated binomial test at alpha=0.05), extensive robustness tests (configuration wrappers, adversarial prompts, identity spoofing, quantization, temporal drift, threshold sweeps, agent interfaces), and a real-world field audit. Baseline comparisons are fair and well-documented.
- **Reproducibility:** High. Code and benchmark probe sets are publicly available. The protocol is fully specified in pseudocode (Algorithms 1 and 2). Appendix provides implementation details for all baselines. Requires only API access.
- **Significance:** Very high practical impact. The shadow API audit findings -- 26% flag rate with concentration on premium Claude endpoints and tier-dependent behavior -- are directly actionable. The protocol costs ~$0.39 per full audit after one-time enrollment of ~$22.
- **Red flags / Concerns:** (1) The 0/16 diagonal false-positive rate, while encouraging, has a one-sided 95% CP upper bound of 17.1% -- too few trials for high-stakes confidence. The authors acknowledge this. (2) The binomial model assumes independent probe outcomes, but probes share domains and templates; the authors note this but do not provide domain-block bootstrap corrections in the main results. (3) The 15-domain probe library is an engineering choice not systematically justified -- it works, but the sensitivity to domain selection is not analyzed. (4) Adaptive adversaries who recognize and special-case audit probes are discussed but not empirically evaluated. (5) The self-error rates for T3 models are high (up to 16.2% for GLM-4.7-Flash and 13.8% for Gemini 2.5 Flash Lite), suggesting KBF may be less reliable for budget models.

## Strengths

- Addresses a real, growing, and economically motivated problem in the LLM API ecosystem with rigorous methodology
- The insight that stable wrong answers near the knowledge boundary are useful audit signals is elegant and well-demonstrated
- Extremely low false-positive rate under deployment variation -- a critical requirement for practical auditing
- Comprehensive evaluation: controlled experiments + robustness stress tests + real-world field audit
- Low cost makes it practical for repeated monitoring by ordinary users
- The tier-dependent behavior finding in the shadow audit is a compelling real-world validation
- Public code and probe sets support reproducibility and community adoption

## Limitations

- False-positive calibration is based on limited same-reference trials (16 diagonal + 30 configuration + 80 stress tests); high-stakes deployment requires hundreds of trials across accounts, days, regions, and fresh probes
- The binomial independence assumption is a simplification; probes share domains and templates, so discrepancy correlation is possible
- Temporal stability is limited to ~7-9 weeks; probe refresh cycles are needed for ongoing audits, adding operational cost
- Budget models with high self-error rates (11-16%) receive wider null bounds, reducing statistical power against similar substitutes
- No empirical evaluation against adaptive adversaries who specifically try to evade KBF by recognizing and special-casing probe queries
- The contrastive screening heuristic (filtering against Qwen3.5-9B) may over-fit to the current substitute pool and reduce discriminative power against future models
- Cannot provide cryptographic attestation of model identity -- only statistical consistency evidence
- The protocol is designed for endpoint consistency, not for identifying which specific substitute model is being served

## Relevance

这篇论文与LLM安全性和可信AI部署直接相关。对于使用第三方API访问LLM的研究者和开发者，KBF提供了一种实用的验证工具，确保所付费的模型服务与实际交付一致。特别是其发现的高端Claude端点不一致性和层级化服务行为，对任何使用转售API访问前沿模型的用户都具有直接警示意义。

## Questions for Further Exploration

- How robust is KBF against adaptive adversaries who fine-tune proxy models specifically to mimic the target model's knowledge-boundary recall patterns, or who maintain a database of reference answers for known probe queries?
- What is the minimum probe set size needed for reliable auditing, and how does it scale with the number of candidate substitute models and the desired detection sensitivity?
- Can domain-block bootstrap or effective-sample-size corrections replace the binomial independence assumption while maintaining practical audit cost?
- How should auditors handle model version updates (silent backend changes) that shift knowledge-boundary behavior -- can probe set versioning and automated drift detection be integrated?
- Could KBF be extended to verify model configuration properties beyond identity, such as safety alignment level, quantization level, or system prompt integrity?
- What are the implications of the tier-dependent serving behavior discovered in the shadow audit -- should LLM API consumers demand verifiable inference or cryptographic attestation from providers?
