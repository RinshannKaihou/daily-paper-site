> **一句话总结：** 本文提出 CROP 方法，利用保形风险控制（conformal risk control）对大语言模型的推理链前缀进行统计认证——在校准集上选取阈值，返回最长且每步风险代理均低于阈值的连续前缀，保证返回前缀包含标注错误的边际概率不超过 alpha，在六个推理数据集上证明了 AUROC 不能预测前缀效用，并在 Arithmetic 和 GSM8K 上将下游修复准确率提升了最多 +5.5 个百分点。

> **TL;DR:** CROP (Conformal Reasoning Output Prefixes) is a verifier-agnostic, post-hoc calibration layer that certifies the longest contiguous error-free prefix of an LLM reasoning trace. Given any step-level risk proxy (PRM, likelihood, heuristic), it uses conformal risk control to select a threshold guaranteeing marginal prefix-contamination probability at most alpha. Across six datasets (Arithmetic, GSM8K, ProcessBench, Math-Shepherd, PRMBench, PRM800K), CROP demonstrates that AUROC fails to predict prefix utility, achieves the lowest boundary deviation among deployable methods, and improves downstream repair accuracy by +1.0 to +5.5 percentage points over non-CROP baselines (with negative results on same-family Llama repair due to answer repetition). Code available at https://github.com/matthewyccheung/crop.

## 核心贡献

- 提出了 CROP（Conformal Reasoning Output Prefixes），将推理链前缀认证形式化为一个单调二元损失的保形风险控制问题，提供有限样本边际风险保证。
- 证明了前缀污染损失关于阈值单调递增，从而可以直接应用保形风险控制（CRC）框架，选取最大可行阈值以最大化保留前缀长度。
- 在六个过程标注数据集上系统性证明了 step-level AUROC 不能预测前缀效用，提出应以认证前缀长度（fixed-risk prefix utility）作为推理验证器的评价指标。
- 在 Arithmetic 和 GSM8K 上展示了 CROP 在八组修复设置中五组优于最佳非 CROP 基线，最大提升达 +5.53%（DeepSeek on Arithmetic）。
- 将前缀认证定位为过程监督、弃权（abstention）和修复之间的严格桥梁，区别于全量弃权方法，CROP 能保留部分有效推理。

## Key Contributions

- Formalizes clean-prefix certification for sequential reasoning traces as a monotone binary-loss conformal risk control problem, yielding finite-sample marginal guarantees under exchangeability (Proposition 1).
- Proves that prefix contamination loss is nondecreasing in the threshold, enabling direct application of conformal risk control (CRC) to select the largest feasible threshold maximizing retained prefix length.
- Demonstrates across six process-labeled datasets that step-level AUROC is an unreliable predictor of fixed-risk prefix utility, advocating certified prefix length as the primary evaluation metric for reasoning verifiers.
- Shows CROP improves downstream repair accuracy in 5 of 8 settings (Arithmetic + GSM8K, four repair models), with the largest gain of +5.53 percentage points (DeepSeek-R1-0528-Qwen3-8B on Arithmetic), while identifying model-specific failure modes (Llama same-family answer repetition).
- Distinguishes CROP from whole-trace abstention: at similar empirical risk, CROP retains more total step mass by preserving partial reasoning from traces that whole-trace abstention cannot safely accept (Table 3).

## Methodology

CROP is a post-hoc, verifier-agnostic calibration procedure operating on completed reasoning traces. The key technical insight is that prefix contamination (the event that a retained prefix contains at least one erroneous step) can be represented as a binary loss monotone in the threshold, enabling conformal risk control.

**Setup:**
- A reasoning instance X = (q, tau) consists of question q and trace tau = (tau_1, ..., tau_T).
- Each step receives a binary error label Y_t from a fixed annotation protocol (human, programmatic, or LLM-as-judge).
- A step-level risk proxy s_t(X) produces a scalar risk score R_t available at test time (e.g., PRM score, likelihood, hidden-state feature, or heuristic).

**Key definitions:**
- Retained prefix length: M_lambda = max{m : R_t <= lambda for every t <= m} -- the longest prefix where all step risks are below threshold lambda.
- CROP loss (prefix contamination): L_lambda(Z) = 1{exists t <= M_lambda : Y_t = 1} -- binary, nondecreasing in lambda.

**Calibration rule (Proposition 1):**
- Given calibration instances Z_1, ..., Z_n and target risk alpha, define the feasible set F = {lambda : (1 + sum L_lambda(Z_i)) / (n+1) <= alpha}.
- Set lambda_hat = max F (or empty-prefix fallback if F is empty).
- Guarantee: P(exists t <= M_lambda_hat : Y_{n+1,t} = 1) <= alpha under exchangeability.

**Risk proxies evaluated:**
1. Token/format features (surface-level baselines)
2. Trace features (logistic regression on step-level features)
3. Direct PRM (Qwen2.5-Math-PRM-7B, one minus positive-class probability)
4. Trace features + PRM (joint model)

**Datasets:** Six process-supervision datasets: Arithmetic, GSM8K (both from attribution graph work), ProcessBench, Math-Shepherd, PRMBench, PRM800K. Main benchmark: 2,819 traces, 19,311 steps, 1,006 annotated errors across Arithmetic and GSM8K.

**Repair evaluation:** Four repair models (Gemma 4 E4B IT, Qwen2.5-7B-Instruct, DeepSeek-R1-0528-Qwen3-8B, Llama3.1-8B-Instruct) over 20 stratified 60/20/20 splits.

## Results & Highlights

- **Prefix utility vs. AUROC:** Step-level AUROC does not predict fixed-risk prefix utility. Downward red segments in Figure 3 show that higher AUROC verifiers can yield worse certified prefix lengths. This is because AUROC measures global ranking while CROP depends on the first threshold crossing in trace order.
- **Boundary deviation:** CROP achieves the lowest boundary deviation among deployable methods on Arithmetic and GSM8K (Table 1), closely approximating oracle clean prefixes. It avoids both over-withholding (discarding valid steps) and unsafe overshoot (retaining errors).
- **Repair accuracy (Table 2):** CROP prefix input improves over the best deployable non-CROP baseline in 5 of 8 settings:
  - Arithmetic + DeepSeek: 85.0% vs 78.8% (whole-trace abstention), delta +5.53%
  - Arithmetic + Gemma: 88.7% vs 83.4%, delta +5.32%
  - Arithmetic + Qwen: 92.4% vs 90.8%, delta +1.02%
  - GSM8K + Qwen: 87.6% vs 86.3%, delta +0.95%
  - GSM8K + DeepSeek: 87.4% vs 84.6%, delta +1.83%
  - Negative deltas for Llama on both datasets (-1.18%), diagnosed as same-family answer repetition when trace context is provided.
- **CROP vs. whole-trace abstention (Table 3):** At similar empirical risk (~4.3-4.9%), CROP retains comparable or more total step mass (up to 95.1%) while accepting fewer full traces (87.1-91.0% vs 90.5-93.4%). CROP changes the certified object from all-or-nothing to partial-prefix.
- **PRM robustness:** PRM-backed scores remain substantially stronger than surface-feature baselines even after reweighting by token/format cues and running label/order controls (Appendix Table 10), ruling out superficial artifacts.
- **Code available:** https://github.com/matthewyccheung/crop

## Critical Assessment

- **Novelty: High.** The application of conformal risk control to certify contiguous reasoning prefixes (not final answers, not full traces) is genuinely novel. The key insight -- that prefix contamination is a monotone binary loss in the threshold -- is clean and non-obvious. This opens a new operational interface between process supervision and downstream repair.
- **Rigor: Strong.** The theoretical contribution is a well-stated Lemma 1 (monotone binary-loss risk control) and Proposition 1 (clean-prefix risk control), with proper proofs. The exchangeability assumption is clearly stated (Assumption 1) and its scope is discussed. The paper is honest about what CROP does not guarantee (individual prefix correctness, final-answer correctness, semantic truth).
- **Reproducibility: High.** Code is publicly available. The method is straightforward to implement. Experimental setup is well-detailed (20 stratified splits, four repair models, six datasets, four risk proxy families).
- **Significance: High.** As LLM reasoning chains grow longer, the ability to certify how much of a trace is trustworthy is operationally critical. The finding that AUROC is inadequate for verifier evaluation should influence how the community evaluates process reward models.
- **Red flags:**
  - Repair improvements are model-dependent: Llama consistently shows negative deltas due to same-family answer repetition, highlighting a non-trivial failure mode.
  - Only Arithmetic and GSM8K have the dense first-error labels needed for boundary and repair analysis; the other four datasets are used only for prefix utility evaluation.
  - The paper does not address how to handle distribution shift in deployment, only noting that recalibration is needed.
  - The method is post-hoc on completed traces -- it does not intervene during generation.

## Strengths

- Clean theoretical formulation: the monotone binary loss insight elegantly reduces prefix certification to conformal risk control.
- Verifier-agnostic design: CROP wraps around any step-level risk proxy without modification.
- Compelling empirical demonstration that AUROC does not predict prefix utility, with clear visual evidence (Figure 3).
- Honest reporting of negative results (Llama repair degradation) with root-cause analysis.
- The prefix-then-repair paradigm is practically useful and complementary to existing factuality and abstention methods.
- Code availability ensures reproducibility.

## Limitations

- Exchangeability assumption may be violated under distribution shift (new tasks, different generators, changed annotation protocols), requiring recalibration.
- CROP is post-hoc on completed traces -- it does not provide real-time intervention during generation.
- Repair gains are model-specific; same-family repair models (Llama repairing Llama traces) can degrade performance.
- Only Arithmetic and GSM8K support full boundary and repair analysis; the remaining four datasets lack dense first-error labels.
- CROP certifies process-level label compliance (annotated error absence), not semantic truth or causal faithfulness of the reasoning.
- The method does not address non-contiguous valid segments; once a high-risk step is encountered, all subsequent steps are discarded regardless of their individual quality.
- Noisy, incomplete, or misaligned annotations weaken the practical meaning of the statistical guarantee.

## Relevance

该论文将保形风险控制应用于 LLM 推理链的前缀认证，对于关注推理可靠性、不确定性量化、过程监督（process supervision）以及推理链自动修复的研究者具有直接参考价值。CROP 的保形保证框架和前缀-修复范式可应用于任何需要评估和修复长推理链的场景。

## Questions for Further Exploration

- How can CROP be extended to provide stronger conditional (per-instance) guarantees rather than marginal ones, perhaps via adaptive conformity scores or covariate-shift corrections?
- Can the prefix certification idea be applied during generation (online/streaming) rather than post-hoc, enabling real-time early stopping or backtracking?
- How does CROP interact with non-greedy generation strategies (e.g., best-of-N sampling, tree search) where multiple trace candidates exist?
- What is the theoretical and empirical behavior under distribution shift -- how much does calibration quality degrade when test traces come from a different generator or task distribution?
- Can the framework be extended to handle non-contiguous valid segments, allowing recovery of useful reasoning that appears after an isolated error?
- How does CROP compare to learned abstention policies that jointly optimize for prefix length and correctness rather than using a fixed conformal threshold?
