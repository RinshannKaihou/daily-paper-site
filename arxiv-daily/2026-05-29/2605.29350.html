ConMoE: Expert-Pool Consolidation via Prototype Reassignment for MoE Compression

A completely train-free method that reformulates MoE compression as selecting reusable expert prototypes and deterministically remapping original expert calls -- no weight updates, no fine-tuning, no gradients.

Yao, Pan, Dai, Cong, Li, Yang — Peking University 2026-05-29 MoE Compression Train-Free Efficient Inference
Key Finding
Deterministic nearest-prototype remapping alone -- without any weight fusion or fine-tuning -- achieves the best average score on DeepSeek-MoE-16B at both 25% and 50% routed-expert reduction, outperforming all pruning and merging baselines.

Why This Matters
Simple remapping beats complex fusion -- a train-free path to compressing MoE models for deployment where fine-tuning is costly, using only 128 unlabeled calibration examples.
Critical Edge
On Qwen3-30B-A3B at 50% reduction, ConMoE (0.717) falls behind frequency pruning (0.733). The “logical” reduction ratio does not directly translate to physical memory savings without a specialized shared-prototype runtime. Scope size requires per-model tuning (scope 1 vs. 4), and calibration data is task-matched, potentially inflating scores.
Methodology

Prototype Scoring

For each expert in a local scope, compute ae (routing-conditioned contribution: avg gated output norm when routed to) and be (replaceability: nearest-neighbor parameter distance). Final score: se = Norm(ae) × Norm(be), favoring experts that are both useful and hard to replace.

Prototype Selection & Reassignment

Top-K experts by score become the prototype pool. Each original expert is deterministically assigned to its nearest prototype by parameter distance -- inducing a partition with zero weight modification.

Consolidated MoE Operator

The router is preserved unchanged. When multiple routed experts for a token map to the same prototype, their routing weights are aggregated into a single coefficient. Drop-in replacement with standard HuggingFace pipelines.

Strengths & Limitations

+ Strengths

  • Completely train-free: no weight updates, no fine-tuning, no gradients
  • Clean formulation separating prototype selection from reassignment
  • Best average results on DeepSeek-MoE-16B across both reduction ratios
  • Preserves original router interface for drop-in evaluation
  • Post-hoc fusion degrading performance is a valuable negative result
  • Systematic ablations isolate each design choice clearly

– Limitations

  • Not uniformly superior: pruning wins on Qwen3-30B-A3B at 50% reduction
  • Only compresses routed experts; shared experts and attention untouched
  • Logical reduction does not directly equal physical memory savings
  • Scope size is a per-model hyperparameter requiring tuning
  • Task-matched calibration may inflate scores vs. real deployment
  • Evaluated only on multiple-choice benchmarks, not generation tasks
Open Questions
  • Can ConMoE's remapping combine with quantization (INT4/INT8) for compounded gains?
  • What determines whether cross-layer sharing helps -- is it expert specialization heterogeneity?
  • How would task-agnostic calibration (random web text) affect prototype selection?
  • Would soft reassignment (weighted top-2 prototypes) beat hard deterministic remapping?
  • How does the compressed model perform on generation benchmarks (MT-Bench, AlpacaEval)?
  • What is the actual memory/latency reduction with a shared-prototype runtime?
  • Can consolidation extend to shared experts or attention layers?
