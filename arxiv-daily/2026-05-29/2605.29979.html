Fingerprinting Inference Systems of Large Language Models

Floating-point non-associativity in LLM inference stacks creates exploitable numerical fingerprints in text outputs, enabling black-box identification of the engine, attention backend, and GPU from prompt-response pairs alone.

Key Metric
100%
Accuracy at T=0 across all components.
~80% engine ID at T>0
Why This Matters
Opens a new side-channel attack surface for LLM deployments, directly linkable to real CVEs in major inference engines.
Critical Edge
Requires enumerating candidate components in advance — inapplicable to proprietary systems (OpenAI, Anthropic, Google) where the stack is unknown.
Methodology
Stage 1
Rare Tokens
1700 prompts · autoregressive decoding
Stage 2
Binary Decisions
5000 prompts · prefill isolation
Stage 3
Long Context
150 prompts · chunked prefill
Stage 4
Repetition
100 prompts · KV-cache reuse

Threat Model

Attacker knows the deployed model and system prompt (recoverable via existing attacks), has chat-only access (no logits), and can collect training data on reference systems offline.

Classification Pipeline

Responses encoded via set-specific scoring functions into feature vectors. Separate random forest classifiers for inference engine (4 classes), attention backend (6 classes), and GPU type (3 classes). 30 distinct system configurations tested.

Results & Accuracy
Component T=0 T=0.3 T=0.6 T=0.9

Robustness Highlights

Application-specific system prompts reduce accuracy by only ~0.3%. Batch size generalization (train on {32,64,128}, test on 256): 77–99%. Withheld proprietary components: engine ID remains 86–95% for Qwen3-4B.

Key Caveat

Temperature mismatch between training and deployment causes 30–49% accuracy drops. The attacker must match or infer deployment temperature. Llama-3.2 degrades fastest among tested models.

Strengths & Limitations

Strengths

  • Novel attack surface linking floating-point deviations to security-relevant information leakage
  • Elegant prompt design systematically probing distinct computational stages
  • Comprehensive robustness evaluation: system prompts, batch sizes, proprietary components, temperature transfer
  • Security motivation grounded in real CVEs (vLLM CVE-2026-22778, SGLang CVE-2026-5760)
  • Full reproducibility with exact software versions and hardware specs

Limitations

  • Requires reference system access and known component catalog — limited to open-source stacks
  • Stochastic decoding drops attention backend accuracy to 33% (Llama-3.2 at T=0.9)
  • Hundreds of queries needed, potentially detectable via rate limiting
  • Only tested on Nvidia GPUs and 3B–7B parameter models
  • No quantitative defense evaluation (noise addition, rate limiting trade-offs unmeasured)
Open Questions
Q1 Can fingerprinting extend to proprietary APIs (OpenAI, Anthropic) via zero-shot or few-shot transfer without known component catalogs?
Q2 What is the quantitative accuracy-utility tradeoff curve for proposed mitigations like parameter noise or output perturbation?
Q3 Does the fingerprint signal strengthen or weaken with 70B+ models, INT8/INT4 quantization, or non-Nvidia hardware (AMD MI300, Intel Gaudi)?
Q4 Could adversarial training produce numerically indistinguishable outputs, or is the floating-point fingerprint truly irreducible?
