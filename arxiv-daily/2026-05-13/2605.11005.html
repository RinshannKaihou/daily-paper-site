DisagMoE: Computation-Communication overlapped MoE Training via Disaggregated AF-Pipe Parallelism

Zhichen Zeng, Chi-Chih Chang, Jiayi Wang, Zezhou Wang, Ningxin Zheng, Zheng Zhong, Cesar A. Stuardo, Dongyang Wang, Mohamed S. Abdelfattah, Haibin Lin, Banghua Zhu, Ang Li, Ziheng Jiang

arXiv: 2605.11005

Date: 2026-05-13

Tags: [moe-training, distributed-systems, pipeline-parallelism, communication-overlap]

一句话总结： DisagMoE 通过将 Attention 和 FFN 解耦到不同的 GPU 组，配合多级流水线调度（AF-Pipe）和自适应资源分配，在大规模 MoE 训练中实现了高达 1.81× 的加速，有效解决了跨节点 all-to-all 通信瓶颈。
TL;DR: DisagMoE is a disaggregated MoE training system that partitions attention and FFN layers into separate GPU groups, introduces a multi-stage pipeline (AF-Pipe) with many-to-many communication overlap, and uses a roofline-model-guided adaptive allocator to balance compute and network resources, achieving up to 1.81× speedup on 128×H800 clusters.

核心贡献

  • 解耦式组件放置： 将 Transformer 中的 Attention 层和 FFN（MoE）层分离到不同的 GPU 工作组，Attention 组使用数据并行复制，FFN 组使用专家并行分片，从架构层面消除 all-to-all 通信的关键路径瓶颈。
  • AF-Pipe 多级流水线调度： 提出将 M2N/N2M 通信作为一级流水线阶段，与 Attention 和 FFN 计算阶段对齐，实现系统性的计算-通信重叠，流水线气泡降低至传统 1F1B 的约 1/4。
  • 自适应资源分配器： 基于计算-通信 Roofline 模型，将 GPU 和 NIC 带宽在 Attention 与 FFN 组之间的分配建模为混合整数线性规划（MILP）问题，通过 profile-guided 局部搜索优化配置。
  • 大规模实验验证： 在 DeepSeek-MoE、GPT-OSS-120B、Qwen3-235B 三个模型上，使用 16 节点（128×H800）集群进行评测，最高实现 1.81× 加速。

Key Contributions

  • Disaggregated component placement: Separates attention and FFN layers into disjoint GPU worker groups — attention groups use data parallelism (replication), FFN groups use expert parallelism (sharding) — architecturally removing all-to-all from the critical path.
  • AF-Pipe multi-stage pipeline schedule: Treats M2N/N2M token exchanges as first-class pipeline stages alongside attention and FFN compute, systematically overlapping communication with both computations and reducing pipeline bubbles to ~1/4 of standard 1F1B.
  • Roofline-model-guided adaptive allocator: Formulates GPU and NIC bandwidth partitioning between attention and FFN groups as a MILP problem, refined by profile-guided local search to balance stage latencies and maximize MFU.
  • Large-scale empirical validation: Evaluates on DeepSeek-MoE, GPT-OSS-120B, and Qwen3-235B across 16-node (128×H800) clusters with sequence lengths 4K–32K, achieving up to 1.81× speedup over Megatron-1F1B and 1.34× over SOTA overlap methods.

Methodology

DisagMoE addresses the dominant all-to-all communication bottleneck in large-scale MoE training through a three-part design:

  1. Disaggregated Architecture (Section 4.1): The model is partitioned by component type. For a model with L layers and p groups per component, each group g receives every p-th occurrence of its component (attention or FFN) across depth. Attention groups replicate dense parameters via DP; FFN groups shard experts via EP. This homogeneity within each component type enables uniform scaling.
  1. AF-Pipe (Section 4.2): A tailored pipeline parallelism strategy built on the disaggregated placement. Key innovations include:
  • M2N stage boundaries: Traditional all-to-all exchanges become many-to-many (M2N/N2M) communication between disjoint worker groups, fusing P2P and combine into a single unified stage, reducing communication cost by ~1/k.
  • Asynchronous multi-stream execution: Each worker maintains forward, backward, and communication streams, enabling non-blocking overlap of compute and communication.
  • Bubble analysis: AF-Pipe warmup bubble is B_AF = (PP-1)·[max(T_a, T_f) + T_M2N] / (2v), approximately 1/4 of baseline 1F1B bubble with per-layer staging.
  1. Adaptive Worker Allocation (Section 4.3): Extends the classical roofline model to the AFD architecture. The key insight is that attention and FFN have structurally different compute-communication ratios (attention scales quadratically with sequence length; FFN scales linearly). The allocator:
  • Solves a two-phase optimization: first minimize the bottleneck stage latency T = min max(T_a, T_f), then maximize aggregate arithmetic intensity I_attn + I_ffn among configurations attaining T.
  • Uses MILP (Gurobi) for a roofline-guided seed, refined by profile-guided local search.

The system is implemented on Megatron-LM (6K lines Python + 2K C++), using GPUDirect and GPUCopy for efficient M2N/N2M communication.

Results & Highlights

  • Up to 1.81× speedup over Megatron-1F1B interleaved overlap across DeepSeek-MoE, GPT-OSS, and Qwen3 models on 128×H800 GPUs.
  • 1.2–1.5× speedup over Tutel and Comet (operation-level overlap methods), which suffer from unoverlapped communication tails because FFN computation only covers 40–60% of all-to-all time.
  • 1.05–1.13× speedup over DualPipe (microbatch-level bidirectional pipeline), which still has residual communication bubbles from mismatched attention-FFN compute-to-communication ratios.
  • 88% reduction in non-overlapped communication time vs. Tutel, 75% vs. Comet, 45% vs. DualPipe (measured on DeepSeek-MoE 8K).
  • Adaptive resource allocation matters: Optimal A:F GPU ratio varies with sequence length — balanced (16:16) at 4K, attention-heavy (16:10) at 16K, yielding 1.29× improvement over homogeneous allocation at 16K.
  • Robust across top-k and EP size: Consistent 1.08–1.92× speedup across top-k ∈ {2,4,8,16} and EP sizes.

Strengths

  • Principled co-design of architecture placement, pipeline scheduling, and resource allocation — not just a single trick but a holistic system.
  • Strong empirical results across three different MoE architectures and diverse configurations (sequence length, top-k, EP size).
  • The compute-communication roofline model provides actionable hardware-software co-design insights: uniform bandwidth scaling is insufficient; future hardware should target favorable per-component compute-communication ratios.
  • Clean integration into Megatron-LM with minimal user-facing API changes (only module-level definition changes).

Limitations

  • Fixed workload shapes: Assumes pretraining-style stable sequence lengths and batch sizes. Dynamic-shape workloads (e.g., RL training) would require online reallocation, which is not addressed.
  • Symmetric pipeline depth: Uses a single depth p shared by both attention and FFN groups; asymmetric depths (p_A, p_F) could further optimize but introduce scheduling complexity.
  • Homogeneous hardware assumption: All GPUs are H800; the allocator does not account for heterogeneous hardware within a cluster (unlike HeterMoE).
  • Scalability beyond 16 nodes: Experiments top out at 16 nodes (128 GPUs); behavior at hundreds of nodes remains unexplored.
  • Memory overhead of virtual stages: Beyond v=16, OOM occurs, limiting pipeline utilization gains.

Relevance

本文对大规模 MoE 模型训练中的通信瓶颈问题提出了系统性的解决方案，其解耦式架构设计和 Roofline 引导的资源分配思路对分布式训练系统研究具有重要参考价值。对于关注 LLM 训练效率、MoE 系统优化、以及软硬件协同设计的研究者而言，这项工作提供了从架构到调度的完整优化路径。

Questions for Further Exploration

  1. How would DisagMoE's adaptive allocator perform with heterogeneous GPU clusters (e.g., mixing H100 and H800), and can the MILP formulation be extended to handle such asymmetry?
  2. What is the memory overhead of maintaining separate activation storage for attention and FFN groups compared to the baseline, and does this limit the maximum trainable model size?
  3. Can the AF-Pipe scheduling strategy be combined with sequence parallelism or context parallelism to handle ultra-long sequences (>128K tokens)?
  4. How does DisagMoE interact with fault tolerance mechanisms (e.g., checkpoint/restart) in the disaggregated architecture — do separate groups complicate consistent checkpointing?
  5. Would an online version of the adaptive allocator (adjusting ratios dynamically during training as batch composition changes) yield further gains, and what is the overhead of such online profiling?
