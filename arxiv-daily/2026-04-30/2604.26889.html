Revealing NVIDIA Closed-Source Driver Command Streams for CPU-GPU Runtime Behavior Insight

arXiv: 2604.26889

Date: 2026-04-29

https://arxiv.org/abs/2604.26889v1

一句话总结： 通过修改NVIDIA开源内核驱动并在GPU doorbell寄存器上安装硬件watchpoint，本文成功捕获并解析了闭源用户态驱动发出的完整GPU硬件命令流，揭示了CUDA数据搬运的DMA模式选择机制以及CUDA Graph从11.8到13.0版本降低启动开销的底层原因。
TL;DR: This paper intercepts and reconstructs the full hardware command streams emitted by NVIDIA's closed-source userspace driver by installing a CPU hardware watchpoint on the GPU doorbell register mapping. Two case studies demonstrate the methodology: (1) characterizing the two DMA submission modes (inline vs. direct) used by cudaMemcpy and measuring raw DMA engine performance free of driver overhead, and (2) showing that CUDA 13.0's near-constant graph launch cost stems from a dramatically smaller command footprint and a single-submission pattern compared to CUDA 11.8's linearly-scaling, multi-submission approach.

核心贡献

  • 命令流捕获与重建： 通过在内核驱动的内存映射路径中插入instrumentation，并在用户态doorbell映射地址上设置CPU硬件watchpoint，实现了对闭源驱动提交的完整GPU命令流的完整捕获与解析。
  • CUDA数据搬运机制分析： 揭示了NVIDIA驱动根据传输大小在两种DMA模式间切换的机制——小数据（<24 KiB）使用compute engine的inline DMA，大数据使用copy engine的direct DMA——并通过CUDA-bypassing方式直接编程DMA引擎，测量了剥离驱动开销后的原始硬件性能。
  • CUDA Graph优化解析： 对比CUDA 11.8与13.0的命令流，证明13.0版本通过更小的命令足迹（从~45 KiB降至~2.2 KiB）和单次doorbell提交模式（而非多次交替写入），实现了接近常数的graph启动开销，有效写入带宽提升约2倍。

Key Contributions

  • Command-stream capture and reconstruction: Instruments the open-source kernel driver's nv_mmap path and installs a hardware watchpoint on the userspace mapping of the GPU doorbell register, trapping into the kernel at the exact moment the closed-source driver commits a command submission. This yields a complete, integrity-preserving snapshot of GPFIFO entries and pushbuffer commands.
  • Data-movement analysis with CUDA-bypassing DMA control: Identifies the driver's DMA mode selection threshold (24 KiB on A40) between inline DMA (compute engine, ~24 ns startup) and direct DMA (copy engine, ~500 ns startup but higher saturated bandwidth at ~22 GiB/s). Introduces a mechanism to directly program DMA engines, decoupling raw hardware performance from driver overhead. Demonstrates that Nsight-reported "CUDA HW" latency includes substantial non-engine time (up to ~95% for small transfers).
  • CUDA Graph evolution from 11.8 to 13.0: Shows that CUDA 13.0 achieves near-constant launch time (1.9–5.9 µs for 1–2000 kernels) versus CUDA 11.8's linear growth (1.8–209 µs). Attributes this to a ~20× smaller command footprint and a single-submission (one doorbell write) pattern versus 11.8's multi-submission approach that alternates between local RAM writes and PCIe TLP writes, sustaining ~2× higher effective write bandwidth.

Methodology

  1. Doorbell interception: Modifies the NVIDIA OpenGPU kernel driver to intercept the mmap syscall that maps the GPU doorbell register into userspace. Allocates a shadow doorbell page and installs a CPU debug register watchpoint on the userspace virtual address. When the closed-source driver writes the doorbell, the watchpoint fires, pausing execution and providing a static observation window.
  2. Command reconstruction: From the intercepted channel ID, walks the KernelChannel structure to locate USERD and RAMFC, reads GP_PUT to find the new GPFIFO entry, walks GPU MMU page tables to resolve physical addresses, maps and reads the pushbuffer, and decodes commands using open-source driver headers.
  3. Custom command injection: Leverages UVM address unification (GPU VAs = CPU VAs) to identify pushbuffer, GPFIFO, and semaphore buffer allocations. Writes custom commands directly to the pushbuffer and rings the doorbell to measure raw hardware performance without driver intervention. Uses GPU semaphore timestamps for device-side timing.
  4. Experimental setup: NVIDIA A40 (Ampere), PCIe Gen4 x16, Intel Xeon Gold 6338. Tests two software stacks: CUDA 11.8 and CUDA 13.0, each with original (perf) and modified (log) kernel drivers.

Results & Highlights

  • Inline DMA (compute engine): ~24 ns startup latency, saturates at ~17.5 GiB/s around 8 KiB. Direct DMA (copy engine): ~500 ns startup latency, saturates at ~22 GiB/s around 1 MiB.
  • Nsight "CUDA HW" duration overstates raw DMA engine time by 49–95% depending on transfer size and mode, conflating driver/submission overhead with hardware execution.
  • CUDA 11.8 graph launch time scales linearly with kernel count (command size grows from 328 B to ~45 KiB for 2000 kernels, multiple doorbell writes). CUDA 13.0 stays near-constant (340 B to ~2.2 KiB, single doorbell write).
  • CUDA 13.0's effective command-emission bandwidth (~432–450 MiB/s) is roughly 2× that of CUDA 11.8 (~205–244 MiB/s), attributed to the single-submission pattern avoiding costly RAM↔PCIe write alternation.
  • Discovered that GPFIFO resides in GPU VRAM while pushbuffer is in host RAM (Finding 2), creating an asymmetric submission path.

Authors & Credibility

All three authors (Yuang Yan, Ian Karlin, Ryan Grant) are affiliated with Queen's University, Kingston, Ontario, Canada. The work leverages NVIDIA's officially open-sourced kernel driver (OpenGPU Kernel Modules v580.105.08) and public hardware documentation (open-gpu-doc), grounding the methodology in legitimate, vendor-provided infrastructure. The experimental methodology is rigorous, using separate "perf" (unmodified) and "log" (modified) driver configurations to ensure instrumentation does not confound performance measurements. The paper includes detailed command-stream listings and comprehensive benchmarking across two major CUDA versions.

Critical Assessment

  • Novelty: High. This is the first work to reconstruct end-to-end command streams from NVIDIA's closed-source userspace driver with full integrity. The watchpoint-on-doorbell technique is elegant and non-invasive to the userspace driver itself.
  • Rigor: High. The methodology is carefully designed: shadow doorbell page ensures correctness, separate perf/log configurations isolate instrumentation effects, and GPU-side semaphore timestamps provide accurate hardware-level measurements. The GPU MMU page-table walk for address resolution is a sound approach.
  • Reproducibility: Moderate. Requires patching the Linux kernel (exporting sched_task_fork), building a modified OpenGPU kernel module, and matching specific driver versions (520.61.07 / 580.105.08). The open-source driver headers enable command decoding, but many field semantics remain undocumented NVIDIA internals.
  • Significance: High. The methodology opens a practical window into the previously opaque NVIDIA driver stack. The findings on DMA mode selection thresholds, Nsight measurement inaccuracy, and CUDA Graph optimization mechanics are actionable for GPU system designers and performance engineers.
  • Limitations: Evaluated on a single GPU (A40). Command semantics of many fields remain opaque (acknowledged by authors). The watchpoint approach inherently adds latency to each submission, making it unsuitable for production performance monitoring. Long-term viability depends on NVIDIA maintaining the open-source kernel driver's mmap architecture.

Strengths

  • Clean, well-motivated methodology that works entirely from the kernel side without modifying the closed-source userspace driver.
  • Two well-chosen case studies that demonstrate both observation (DMA mode analysis) and intervention (custom command injection for raw measurements).
  • Quantitative demonstration that profiler-reported "hardware" time significantly conflates software overhead with true engine execution.
  • Clear explanation of the two-level GPFIFO/pushbuffer submission hierarchy and channel context structure.

Limitations

  • Single GPU platform (A40); generalization to other architectures (Hopper, Blackwell) requires re-examination of command headers and potential submission path changes.
  • Many decoded command fields use NVIDIA-internal terms without public documentation, limiting depth of semantic analysis.
  • The instrumentation necessarily perturbs timing (watchpoint trap + shadow doorbell indirection), so logged timings cannot be used directly as performance measurements—necessitating the separate perf/log setup.
  • No exploration of multi-stream, multi-GPU, or compute kernel launch command streams beyond the CUDA Graph case study.

Relevance

本文对GPU系统软件栈的逆向分析与性能归因方法具有直接参考价值，尤其是将性能开销分解为硬件执行与驱动软件各阶段的方法论。对于关注GPU中间件优化、异构系统性能建模、以及未来软硬件协同设计的研究者而言，提供了一种实用的命令级可观测性工具。

Questions for Further Exploration

  1. How do the DMA mode selection thresholds and submission patterns differ on newer architectures (Hopper H100, Blackwell B200) with NVLink versus PCIe?
  2. Can the watchpoint-based capture methodology be extended to observe inter-GPU communication commands (NVLink, GPUDirect RDMA)?
  3. What is the full semantic mapping of the undocumented command fields observed during CUDA Graph launches, and do they reveal additional optimization opportunities?
  4. Could the custom command injection technique be used to implement user-scheduled data transfers that outperform the driver's default threshold-based mode selection for specific application patterns?
  5. How does command buffer placement (host RAM vs. GPU VRAM vs. device-attached memory like HBM) affect submission latency on systems with coherent interconnects (e.g., Grace Hopper)?
