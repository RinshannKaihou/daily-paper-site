> **一句话总结：** 本文提出一套受治理的"目标-范围-工具-基准"驱动的多智能体架构，将递归自我改进建模为对外部化、可审计、可回滚的版本化智能体状态的有约束搜索过程，并提出"个人奇点"作为以用户为中心、有界的人机协同成长目标——但全文为立场与系统设计论文，无任何实验或量化结果。

> **TL;DR:** This is a position and systems-design paper (no implementation, no experiments) that proposes a governed multi-agent architecture for recursively self-improving (RSI) LLM agents and introduces "personal singularity" as a bounded, user-directed human-AI co-development objective. It formalizes each specialist agent via signed goal/scope/tool/benchmark/autonomy/routing/memory/improvement contracts, separates a fast planner-executor-verifier task loop from a slow evidence-gated improvement loop under an external governance plane, and defines decentralized owner-controlled agent lineages plus a Personal Singularity OS with four initial agent classes (Daily Working, Computational Imaging, Work-Process Learning, Personal Learning). It references GAIA, OSWorld, SWE-bench, AgentBench, MLAgentBench, ToolBench, and MathTutorBench only as the benchmarks its evaluation framework *should* use; it reports no quantitative deltas of its own.

## 核心贡献

- **递归自我改进的有界形式化**：把改进算子定义为 `A' = I(A, D, B)`，并明确只有当接受的改动改变了改进机制 `I` 本身或其搜索空间时才构成"递归"自我改进，从而把"重复固定反思提示"排除在外；同时给出六级自我改进层次表（输出精炼→记忆→技能→策略→权重→元改进），风险递增。
- **版本化专家智能体规范**：每个智能体由一个签名、版本化的操作画像 `Σᵢ = (G, Q, T, B, A, R, M, I, V)` 定义——目标合约、范围合约、已验证工具注册表、基准套件、自治策略、路由与交接策略、记忆、改进策略、版本与谱系元数据。
- **范围路由器与单一负责主智能体**：每个被接受的任务由且仅由一个 accountable primary agent 负责；范围外子任务通过结构化交接路由给合格专家或新生成的 niche agent，并给出可配置的 `ScopeScore`（领域 0.30 + 任务类型 0.25 + 输入输出 0.15 + 工具就绪 0.15 + 历史 0.10 + 用户偏好 0.05）。
- **工具级评测与端到端评测分离**：明确区分"工具在隔离条件下测试"与"完整轨迹测试"，并要求每个 agent benchmark 至少有四个分区（公开 dev、私有 validation、hidden final test、rotating regression），引入 `ToolFirstRate` 指标。
- **外部治理平面与两速架构**：治理平面（签名策略、deny-by-default、凭证代理、独立评估器、防篡改审计、回滚、带外关停）运行在智能体写权限之外；快速任务环与慢速证据门控改进环分离，给出改进接受谓词 `Accept(A,A')` 的形式化条件（性能 ≥ τp、安全 ≥ τs、风险增量 ≤ τr、权限增量 ⊆ 已批准、谱系完整且可回滚）。
- **Owner 主权的去中心化谱系与 Agent Foundry**：将 "main" 重新定义为可发现性而非控制权；支持克隆、fork、模板派生、多亲重组（child 权限取 `Pₐ ∩ P_B ∩ P_owner-approved` 的交集而非并集）；同一 owner 的 agent 间通过受控消息总线通信，可被置入隔离态。
- **有界目标演化**：五层签名目标栈 `G=(G₀..G₄)`（不可变安全宪法→创始使命→owner 目标→经验派生改进目标→当前任务），低层可收窄但不可覆盖高层；自动子目标仅在不扩权、可测、可逆、有界时允许。
- **个人奇点 OS 与人类能力迁移**：将用户置于中心，定义用户能力向量 `hᵤ(t)=[K,S,J,P,C,H,W,R,F]` 与可行能力前沿 `Fᵤ(t)`（Pareto 前沿而非标量最大值）；提出双目标效用 `U = αQ_task + βΔH_user − γR_safety − δD_dependency − ηC_resource`，并提供 delegate/copilot/tutor/coach/examiner 五种模式与 Algorithm 4 的能力迁移流程。

## Key Contributions

- **Bounded formalization of recursive self-improvement**: defines the improvement operator `A' = I(A, D, B)` and requires that accepted changes alter `I` or its search space to qualify as "recursive" (excluding rote repetition of a fixed reflection prompt), plus a six-level self-improvement hierarchy table mapping changed component to risk.
- **Versioned specialist-agent specification**: each agent is a signed, versioned profile `Σᵢ` over goal, scope, validated tool registry, benchmark suites, autonomy policy, routing/handoff policy, memory, improvement policy, and lineage metadata — any field change is an auditable release delta, not an implicit prompt edit.
- **Scope router and single-accountable primary agent**: every accepted task is owned by exactly one primary agent; out-of-scope subtasks are handed off via structured transfers, with a configurable `ScopeScore` ranking formula.
- **Separation of tool-level from end-to-end agent evaluation**: tool benchmarks (isolated correctness/side-effects/security) are reported separately from agent benchmarks (full trajectory), with a mandatory four-partition benchmark design (public dev, private validation, hidden final, rotating regression) and a `ToolFirstRate` metric.
- **External governance plane and two-speed architecture**: governance (signed deny-by-default policy, credential broker, independent evaluator, tamper-evident audit, rollback, out-of-band shutdown) lives outside the agent's write authority; a fast task loop is separated from a slow evidence-gated improvement loop with a formal `Accept(A, A')` promotion predicate.
- **Decentralized, owner-sovereign lineages and agent foundry**: "main" is reframed as discoverability rather than authority; supports clone/fork/template/multi-parent recombination with child permissions as an intersection (not union); same-owner agents communicate via a controlled broker and can be fully isolated.
- **Bounded goal evolution**: a five-layer signed goal stack `(G₀..G₄)` where lower layers may narrow but never override higher layers; auto-derived subgoals are permitted only when they add no permissions and are measurable, reversible, and bounded — directly motivated by goal misgeneralization.
- **Personal Singularity OS and human-capability transfer**: places the user (not a central agent) at the center, defines a user capability vector and a Pareto feasibility frontier rather than a scalar maximum, and introduces a dual objective `U = αQ_task + βΔH_user − γR_safety − δD_dependency − ηC_resource` with delegate/copilot/tutor/coach/examiner modes plus a formal capability-transfer algorithm.

## Methodology

This is a conceptual systems-design paper, not an empirical study. The method is architectural formalization combined with a literature-grounded design rationale:

1. **Definitions and state formalization** — An agent instance is a tuple `Aᵢᵗ = (θ, π, M, S, G, P, E, V)` (parameters/adapters, runtime policy, memory, skills/tools, goal stack, externally enforced permissions, evaluation evidence, version/lineage). The paper distinguishes six levels of self-improvement (output → memory → skill → policy → weight → meta) and defines the improvement operator `A' = I(A, D, B)`, marking recursiveness as the condition that accepted changes alter `I` itself.
2. **Reference architecture** — A fast planner-executor-verifier task loop (Algorithm 1: bounded end-to-end execution with task-contract compilation, sandboxed execution, external authorization, verification, repair, and budget exhaustion) is separated from a slow consolidation/improvement loop (Algorithm 2: risk-gated recursive improvement with quarantined candidates, static + held-out tests, and human review for authority-expanding changes). An external governance plane enforces signed policy, short-lived credentials, independent evaluation, audit, rollback, and shutdown.
3. **Specialist-agent operating profile** — Each agent is specified by `Σᵢ = (G, Q, T, B, A, R, M, I, V)`. Goal contracts separate stable mission from task/improvement objectives; scope contracts are semantic/operational (domain, task kinds, input/output classes, required tools, permissions, risk ceiling) — explicitly *not* context-window length. A scope router (Algorithm 3) assigns one accountable primary agent and routes out-of-scope work via structured handoffs.
4. **Decentralized lineages and agent foundry** — Owner-directed operations (clone, fork, template-derive, multi-parent recombine) through an external foundry; child permissions are intersected; identities, credentials, private memories, and signing authority are never inherited.
5. **Personal Singularity OS** — A user constitution and personal capability graph place the user at the center; every demand maps to one accountable primary agent; a dual objective jointly optimizes task outcome and durable human-capability gain (Algorithm 4); health and other high-stakes domains are compartmentalized.
6. **Safety invariants and evaluation framework** — Ten architectural invariants (no self-granted authority, no ambient credentials, creation-is-not-activation, risk-based promotion, independent evidence, incremental capability ceilings, reversible deployment, goal integrity, no uncontrolled reproduction, user welfare over engagement). An evaluation framework specifies four measurement axes (task performance, self-improvement quality, governance/red-team, human outcomes) and eight falsifiable hypotheses (H1–H8).
7. **Seven-stage implementation roadmap** — From registry/router/Daily Working Agent (Stage 1, no persistent self-modification) through tool/benchmark harness, Process-Learning and Computational-Imaging agents, memory and Personal-Learning agent, improvement manager, decentralized lineages/Personal Singularity OS, and finally (Stage 7) weight/meta-level improvement only after earlier controls are reliable.

The design rationale is anchored in cited prior work: ReAct, Self-Refine, Reflexion, Voyager, CoALA, Gödel Agent, SICA, Darwin Gödel Machine, MetaSkill-Evolve, MemGPT, Generative Agents, plus benchmarks (GAIA, OSWorld, SWE-bench, AgentBench, MLAgentBench, ToolBench, MathTutorBench) and safety/interaction literature (goal misgeneralization, model collapse, cognitive forcing functions, NIST AI RMF, WHO guidance).

## Results & Highlights

No quantitative results reported.

This paper is explicitly a position and systems-design paper with no implementation and no experiments. The author states directly: *"This is a position and systems-design paper rather than evidence that unrestricted recursive self-improvement or personal singularity has been achieved"* and *"This paper introduces a conceptual architecture and reports no original experimental dataset."* Instead of empirical results, the paper offers the following as its substantive content:

- **Four fully specified algorithms** (bounded end-to-end task execution; risk-gated recursive improvement with promotion predicate `Accept(A,A')`; scope-aware assignment and handoff with `ScopeScore`; human capability transfer after task completion), plus formal definitions of agent state, improvement operator, scope, capability frontier, and the dual utility `U`.
- **A six-level self-improvement hierarchy table** mapping changed component to risk band (Output refinement=Low → Meta-improvement=High–very high).
- **An Auto-Index table** (Levels 0–5: Explain-only → Step approval → Plan-first hybrid → Workspace autonomous → Delegated external → Scheduled operational), clarifying that the index changes *interaction policy, not permissions or intelligence*.
- **A four-class agent table** (Daily Working, Computational Imaging, Work-Process Learning, Personal Learning) with primary missions, representative tools, and primary benchmark targets.
- **A seven-dimension core evaluation table** (task autonomy, self-improvement, safety, goal integrity, human growth, privacy, ecosystem health), each with metrics and an explicit failure signal.
- **Eight testable hypotheses (H1–H8)** that the architecture is designed to make empirically falsifiable — e.g., H2 (external verification + protected benchmarks reduce false promotion vs. self-judgment alone), H4 (explicit capability-transfer modes improve delayed independent performance vs. delegate-only), H7 (optimizing unnecessary-question rate under a non-decreasing quality constraint beats optimizing raw question count).
- **A seven-stage implementation roadmap** that deliberately defers weight/meta-level self-modification (Stage 7) until governance, routing, benchmarks, memory, and the improvement manager are reliable.

The only quantitative constants that appear are design hyperparameters presented as configurable rather than validated — e.g., the `ScopeScore` weights (0.30 domain + 0.25 task-kind + 0.15 I/O + 0.15 tool + 0.10 history + 0.05 preference), which the paper explicitly says "should be calibrated on routing data rather than treated as universal constants." No baseline comparisons, no accuracy numbers, and no ablation results exist in the paper.

## Critical Assessment

- **Novelty — moderate, mostly synthesis**: The individual building blocks (planner-executor-verifier, evidence-gated promotion, scope routing, owner-controlled autonomy, decentralized lineages, capability transfer) each have clear precedents (Gödel Agent, Darwin Gödel Machine, MetaSkill-Evolve, CoALA, cognitive forcing functions). The paper's genuine novelty is in *integration and formalization* — especially the signed `Σᵢ` profile, the four-partition hidden/rotating benchmark design, the intersection-not-union child-permission rule `P_C ⊆ P_A ∩ P_B ∩ P_owner-approved`, and the Pareto-frontier framing of "personal singularity" as distinct from a scalar human-maximum score. The reframing of "main" as discoverability rather than authority is a clean conceptual contribution.
- **Rigor — strong on formalization, zero on validation**: The formal notation (agent-state tuple, improvement operator, `Accept` predicate, capability vector, dual utility) is internally consistent and the safety invariants are concrete and architectural (not prompt-level). However, there is literally no experimental validation: the `ScopeScore` weights (0.30/0.25/0.15/0.15/0.10/0.05), the promotion thresholds (τp, τs, τr), and the utility weights (α, β, γ, δ, η) are all unestimated free parameters. The eight hypotheses H1–H8 are well-formed but none are tested.
- **Reproducibility — not applicable in the empirical sense**: There is no code, no dataset, no model, and no experiment to reproduce. The author transparently states a reference implementation is future work. The *architecture* is described precisely enough to guide implementation, but the `ScopeScore` calibration and benchmark partitions are left to future work, so two independent implementations could diverge substantially.
- **Significance — uncertain, agenda-setting**: If the field converges on governed, owner-controlled RSI architectures, several of this paper's distinctions (tool-test vs. agent-test separation; creation-is-not-activation; intersection child permissions) could become standard vocabulary. But absent any demonstration that the architecture is buildable or that its constraints preserve performance, its influence will depend on whether subsequent work adopts and tests it.
- **Red flags**:
  - **Category mismatch**: filed under `cs.CV` while being an agent-architecture/governance paper — suggests possible miscategorization and limits discoverability by the intended audience.
  - **Single author + non-standard affiliation + personal email**: lower-confidence provenance compared to multi-author lab papers; no institutional validation.
  - **Untested central claims**: the headline "recursively self-improving agents" and "personal singularity" are concepts the paper explicitly says have *not* been achieved, yet the title and abstract can read as more accomplished than the content supports. The author does mitigate this with repeated disclaimers.
  - **Subjective core construct**: the "personal singularity" capability vector `[K,S,J,P,C,H,W,R,F]` includes inherently subjective dimensions (wellbeing, relationship capability, judgment) that the paper itself admits "should be estimated from evidence and owner feedback rather than treated as objective personality scores" — operationalizing these longitudinally is an open and hard problem the paper does not solve.
  - **No comparison to existing agent frameworks**: the paper does not contrast its governance plane against, e.g., existing sandboxed-agent systems, OpenAI/Anthropic safety scaffolding, or prior agent-OS proposals in any structured way — relying on the reader to map the differences.

## Strengths

- Intellectual honesty about scope: the paper states repeatedly and prominently that it is conceptual, that RSI/personal singularity have not been achieved, and that no dataset or code exists.
- Clean, reusable formal vocabulary: the `Σᵢ` agent profile, the six-level self-improvement hierarchy, the `ToolFirstRate` metric, and the four-partition benchmark design are immediately portable to other agent-systems work.
- Safety treated as architectural, not prompt-level: the ten invariants (no self-granted authority, creation-is-not-activation, no ambient credentials, etc.) are enforced outside the agent's write authority, which is the correct threat model for self-modifying systems.
- Non-trivial treatment of human-side harm: the explicit distinction between assisted and unassisted performance, the `D_dependency` penalty term, and the requirement that critical capabilities be periodically assessed without assistance directly engage the cognitive-offloading/de-skilling literature.
- The multi-parent "intersection, not union" permission rule and the "main as status not authority" reframing are concrete, non-obvious design choices that meaningfully reduce blast radius.
- The staged roadmap sensibly defers the highest-risk capability (weight/meta-level self-modification, Stage 7) until governance and evaluation are proven.

## Limitations

- No implementation, no experiments, no benchmarks run, no human study — every empirical claim is deferred. The paper is a research agenda, not evidence.
- All key quantitative knobs (ScopeScore weights, τ thresholds, α/β/γ/δ/η utility weights) are unestimated; the architecture's actual behavior is undefined without calibration the paper does not provide.
- The `personal singularity` construct relies on subjective, multi-objective, longitudinal human-capability measurement that the paper acknowledges is hard and does not operationalize beyond a Pareto-frontier sketch.
- The Computational Imaging Agent and the Personal Learning Agent are described at a much coarser grain than the Daily Working Agent; the four-class taxonomy is unevenly developed.
- No discussion of computational cost, latency, or feasibility of the external governance plane (independent evaluator, hidden rotating benchmarks, signed releases, canary deployment) in realistic settings.
- Federated/secure-aggregation treatment is acknowledged to remain vulnerable to poisoning, inference attacks, and governance failures, but no concrete defense beyond "clipping, anomaly detection, robust aggregation" is offered.
- Single author, non-peer-reviewed, `cs.CV`-categorized preprint — provenance and discoverability concerns.

## Relevance

This paper is highly relevant to agent-architecture, self-improvement, and AI-governance research lines: it provides a concrete, formal vocabulary (signed agent profiles, scope routing, evidence-gated promotion, decentralized lineages) that directly informs the design of safe self-improving LLM agent systems. It is most useful as a conceptual scaffold and a source of falsifiable hypotheses (H1–H8) for future empirical work on governed RSI agents, rather than as a source of deployable results.

## Questions for Further Exploration

- Can the `ScopeScore` weights (0.30/0.25/0.15/0.15/0.10/0.05) be empirically calibrated on real routing data, and does scope-aware routing (H6) actually beat a single generalist agent on completion and unsafe-tool-use rate at fixed base model?
- Does external verification with protected benchmarks measurably reduce false-promotion rate relative to self-judgment (H2), and at what cost in compute and human review?
- In a longitudinal study, do explicit capability-transfer modes (tutor/coach/examiner) improve delayed *independent* performance over delegate-only assistance (H4), or do they merely slow task throughput without durable gain?
- Is the "intersection, not union" child-permission rule (`P_C ⊆ P_A ∩ P_B ∩ P_owner-approved`) sufficient to prevent privilege accumulation across multi-parent recombination, or can adversarial skill/prompt inheritance still enlarge effective capability?
- How would the external governance plane (independent evaluator, hidden rotating partitions, signed releases, canary rollout) be implemented efficiently for personal-scale single-user deployments without prohibitive overhead?
- Can the subjective dimensions of the capability vector (wellbeing, relationship, judgment) be measured with enough rigor to support the Pareto-frontier framing, or does the construct collapse into a few objective proxies?
- Does treating "main" as discoverability rather than authority actually prevent registry/reputation concentration, or does network effect still produce a dominant lineage the architecture assumes away?
- What is the minimal viable Stage-1 implementation (Daily Working Agent + registry + router + Auto-Index) that could be built today to begin testing H1, H3, H7, H8?
