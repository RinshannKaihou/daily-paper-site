01

How Deep4ge is built

STAGE 01

Program selection

59 complete TensorFlow/Keras programs pulled from Stack Overflow, each tied to a unique SO question ID, manually repaired for deterministic runs and instrumented with a logging callback. Seeds fixed; covers FNN / CNN / RNN.

STAGE 02

Fault injection

27 source-level mutation operators applied via Python AST rewriting — DeepCrime's set plus full layer-level operators. Data- & process-level faults excluded. Each run carries exactly one category label.

STAGE 03

Train & extract

n=15 retrainings per mutant. Each epoch logs index, 4 eval metrics + 26 dynamic features: weights, gradients, activations, trends, and hardware (LR, CPU%, peak GPU mem, sys mem).

STAGE 04

Mutation validation

A mutant is "killed" only if statistically significant (p<0.05) and medium-effect (Cohen's d ≥ 0.5). Equivalent mutants dropped; yields mutation scores 0.85 / 0.87 / 0.83 (FNN/CNN/RNN).

Benchmark eval: group-aware 5-fold CV (grouped by SO program ID, no same-program leakage) · 1,000-resample bootstrap 95% CIs · Logistic Regression & Random Forest · features compared as final-epoch (30 vals) vs trajectory (180 = 30×6 stats).

03

Strengths & limitations

Strengths

  • Clean, reproducible artifact: explicit schema, data dictionary, validation scripts, pinned version v1.0.0 on Zenodo + GitHub (MIT framework / CC-BY-4.0 data).
  • Methodologically honest: group-aware CV, bootstrap CIs, trivial baselines included, and an explicit F1→MCC switch once imbalance is acknowledged.
  • Rich per-epoch feature set (26 dynamic features across weights, gradients, activations, trends, hardware) — well beyond loss/accuracy curves.
  • Actionable negative findings: early single-epoch detection is weak, Activation/Optimization are confusable, final-epoch features insufficient.
  • Extensible: new operators, programs, or features can be added and metadata regenerated.

Limitations

  • Restricted to TensorFlow/Keras FNN/CNN/RNN — Transformers are out of scope, leaving the dominant 2026 architecture family uncovered.
  • Stack Overflow programs ≠ industrial code; mutation-derived faults may not represent naturally occurring faults — no generalization claim.
  • Correct baselines validated only for execution/training, not semantic correctness — some may harbor silent bugs.
  • Severe category imbalance (Activation: 312 runs, 0 in CNN; Regularization: 254) makes rare-class diagnosis unreliable.
  • Diagnosis is single-label only; co-occurring multi-fault scenarios are not modeled.
  • Hardware features (GPU mem, CPU%) are environment-dependent; baselines are deliberately minimal (untuned LogReg/RF) ⇒ absolute MCC ≈ 0.23 is modest.
04

Open questions for further work

Q1

Can trajectory-aware sequence models (RNN/Transformer over per-epoch series, contrastive learning) push detection MCC substantially above 0.227?

Q2

Why are Activation & Optimization so confusable — are the 26 logged features simply insufficient, and would per-layer gradient cosine similarity or activation sparsity recover discriminability?

Q3

Can the benchmark extend to Transformers (attention-head mutations, LR-schedule / positional-encoding faults) while preserving the statistical killing criterion — and does the RNN-weaker transfer finding hold?

Q4

Does a multi-label diagnosis formulation (several operators per mutant) change the relative difficulty ranking of fault categories?

Q5

How sensitive are the hardware features (CPU%, peak GPU mem, sys mem) to platform changes, and do they contribute once environment is controlled?

Q6

Are the "correct" baselines semantically correct — would a stricter semantic-validation pass change the detection MCC?
