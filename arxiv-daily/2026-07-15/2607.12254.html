Recursive Self-Improving Agents · Governed Architecture

A governed multi-agent architecture frames recursive self-improvement as bounded, audited, rollback-able search over versioned, owner-controlled agent states.

Each specialist agent is a signed, versioned operating profile; a fast task loop is separated from a slow, evidence-gated improvement loop under an external governance plane — and "personal singularity" is reframed as a bounded, user-directed Pareto frontier, not a scalar maximum.

02Why this matters

Supplies a concrete, portable formal vocabulary — signed agent profiles, scope routing, evidence-gated promotion, decentralized owner-controlled lineages — that researchers can adopt today when designing safe self-improving LLM agent systems.

03Critical edge

No implementation, no experiments, no benchmarks run, no human study — every empirical claim is deferred. All key knobs (ScopeScore weights, τ thresholds, α–η utility weights) are unestimated free parameters. It is a research agenda, not evidence.

§ 01Methodology — a seven-stage, risk-ordered roadmap

Conceptual systems-design, not empirical study. The roadmap deliberately defers the highest-risk capability (Stage 7) until governance, routing, benchmarks & memory are proven.

STAGE 1Registry · Router · Daily Working Agent (no persistent self-mod)
STAGE 2Tool & benchmark harness · four partitions
STAGE 3Process-Learning & Computational-Imaging agents
STAGE 4Memory & Personal-Learning agent
STAGE 5Improvement manager · slow loop · Accept(A,A′)
STAGE 6Decentralized lineages · Personal Singularity OS
STAGE 7 ⚠Weight / meta-level self-modification (gated)

Core formal objects

Agent state Aᵢᵗ = (θ, π, M, S, G, P, E, V); improvement operator A′ = I(A, D, B) — recursiveness holds only when accepted changes alter I itself; promotion predicate Accept(A,A′) requires perf ≥ τp, safety ≥ τs, risk-Δ ≤ τr, permission-Δ ⊆ approved, lineage intact & rollback-able.

Six-level self-improvement hierarchy

Output refinement (Low) → Memory → Skill → Policy → Weight → Meta-improvement (High–very high). A six-level Auto-Index separately governs interaction policy (Explain-only → Scheduled operational) — explicitly not permissions or intelligence.

§ 03Strengths & Limitations — side by side

Read across: the same formalization that is the paper's main virtue is also the source of its validation gap.

Strengths

  • Intellectual honesty — repeatedly flags itself as conceptual; states RSI & personal singularity have not been achieved.
  • Clean, reusable formal vocabulary: Σᵢ profile, six-level hierarchy, ToolFirstRate, four-partition benchmarks — portable to other agent-systems work.
  • Safety treated as architectural, not prompt-level — the ten invariants live outside the agent's write authority (the correct threat model).
  • Non-trivial human-side treatment: assisted vs. unassisted performance, the Ddependency penalty, periodic unassisted re-assessment.
  • "Intersection, not union" child permissions & "main as discoverability, not authority" meaningfully reduce blast radius.

Limitations

  • No implementation, experiments, benchmarks, or human study — every empirical claim deferred.
  • All quantitative knobs unestimated: ScopeScore weights (0.30/0.25/0.15/0.15/0.10/0.05), τ thresholds, α–η utility weights.
  • "Personal singularity" relies on subjective, multi-objective, longitudinal measurement (wellbeing, relationship, judgment) — not operationalized beyond a Pareto sketch.
  • Four-class taxonomy unevenly developed — Imaging & Personal-Learning agents far coarser than Daily Working.
  • No cost / latency / feasibility analysis of the external governance plane at personal scale.
  • Single author, non-peer-reviewed, cs.CV-categorized — provenance & discoverability concerns.

§ 04Open questions for further exploration

Each maps to a falsifiable hypothesis the architecture is designed to make testable.

Q1 → H6Can ScopeScore weights be calibrated on real routing data, and does scope-aware routing beat a single generalist on completion & unsafe-tool-use at fixed base model?
Q2 → H2Does external verification with protected benchmarks reduce false-promotion vs. self-judgment — and at what compute / human-review cost?
Q3 → H4Do explicit capability-transfer modes (tutor/coach/examiner) improve delayed independent performance, or just slow throughput without durable gain?
Q4Is the intersection child-permission rule sufficient against adversarial skill/prompt inheritance, or can effective capability still enlarge?
Q5How is the external governance plane (hidden rotating partitions, signed releases, canary rollout) implemented cheaply for single-user personal deployments?
Q6Can the capability vector's subjective dimensions be measured rigorously enough to support the Pareto framing, or does it collapse into a few proxies?
Q7Does "main as discoverability" actually prevent registry/reputation concentration, or do network effects still produce a dominant lineage?
Q8What is the minimal viable Stage-1 build (Daily Working Agent + registry + router + Auto-Index) to start testing H1, H3, H7, H8 today?
