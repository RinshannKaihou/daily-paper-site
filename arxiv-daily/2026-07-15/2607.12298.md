> **一句话总结：** 提出EIR（Emulated Integrity Replica）——一种用于FPGA SoC的层级化数字孪生自愈框架，在处理系统（PS）空闲周期中运行"Rabbit"（粗粒度行为模型）快速检测故障、运行"Tortoise"（细粒度门级模型）精确恢复，从而在不复制可编程逻辑（PL）硬件的情况下，覆盖≥95%的故障，相比DMR平均降低约2.0–2.25倍的LUT/FF面积和约89.1%的功耗。

> **TL;DR:** EIR is a hierarchical digital-twin self-healing framework for FPGA SoCs that runs two complementary software twins (Rabbit for coarse-grained detection, Tortoise for gate-level recovery) on the otherwise-idle Processing System (PS), avoiding DMR/TMR fabric duplication; on an AMD/Xilinx Zynq-7000 across 11 benchmarks (LeNet-5/CIFAR-10 dense layers, Conv2D, attention, AES-128, SHA-256, FFT-64, GEMM, FIR, IIR), auditing the top 30% of neurons yields ≥95% fault-detection rate while DMR raises LUT/FF utilization ~2.0–2.25× and power from 2.98 W to 5.79 W (~89.1% higher), versus zero PL area increase for EIR.

## 核心贡献

- **首个面向FPGA SoC加速器的双粒度数字孪生框架**：提出"Rabbit"（粗粒度、快速、C/C++行为模型，运行于PS）与"Tortoise"（细粒度、门级、由Yosys CXXRTL生成的C++模型）组合，实现"快速检测 + 精确恢复"的层级分工，避免传统DMR/TMR对PL逻辑的连续复制。
- **无需修改加速器RTL的自愈机制**：EIR通过PCAP接口周期性读取PL状态到片外DDR，在检测到Rabbit与PL输出不一致时，由Tortoise从最近可信检查点逐周期重放出正确的无故障状态，再经PCAP写回PL，实现"原地恢复"（in-place recovery），而无需对加速器RTL做任何改动或重综合。
- **基于执行速度感知的延迟敏感型检查点策略**：根据PL执行时间的性能分析（profiling），只在决策关键的稠密层/logit层等高敏感阶段放置检查点，并按MAC贡献幅值排序，选择top-30%的高贡献MAC进行校验，把PL隔离窗口从全量校验约530 µs压缩到约170 µs（约3倍改善）。
- **面向边缘AI的软硬协同端到端系统**：在资源受限的FPGA SoC（Zynq-7000, XC7Z020）上完成完整实现与11项基准评测，覆盖密码学、信号处理与机器学习内核，证明EIR可作为连续硬件冗余的按需软件替代路径。
- **量化的资源/能耗收益**：在所评估的故障模型与工作负载下，EIR保持基线PL面积不变，而DMR使LUT/FF利用率平均增加约2.0–2.25×、功耗从2.98 W升至5.79 W（约+89.1%），实际状态写回`trestore`仅需5.7 ms。

## Key Contributions

- **Hierarchical digital-twin architecture**: First dual-granularity digital-twin framework purpose-built for FPGA SoC accelerators, separating fast functional detection (Rabbit, a coarse-grained C/C++ behavioral twin running on the PS) from precise gate-level recovery (Tortoise, a cycle-accurate CXXRTL-generated twin), instead of always-on fabric-level duplication.
- **Hardware-software checkpoint integration**: A practical checkpoint-based fault-tolerance implementation that reuses the PCAP readback/writeback primitives and logic-location (`*.ll`) maps of prior work (EPOCH), with a speed-informed checkpoint-selection policy that balances detection latency against off-chip storage traffic.
- **Self-healing without RTL redesign or PL duplication**: On a detected Rabbit-vs-PL mismatch, Tortoise replays from the last trusted checkpoint to reconstruct a fault-free state, which EIR writes back into PL registers via PCAP, enabling in-place recovery with zero added PL area and no accelerator RTL changes.
- **Magnitude-aware selective validation**: Ranks dense-layer MAC operations by absolute contribution `|x·w|` and audits only the top-k (default k=30%) highest-impact neurons, shrinking the PL isolation window from ~530 µs (full recompute) to ~170 µs (~3× improvement) while retaining observed coverage.
- **Empirical validation on 11 heterogeneous benchmarks**: Demonstrates generality across up-counter, SHA-256, AES-128, dense/Conv2D/attention layers, GEMM, FFT-64, FIR/Conv1D, IIR, and multi-head attention on an AMD/Xilinx Zynq-7000 (XC7Z020), with all accelerators written in Verilog and synthesized with Vivado 2024.2.

## Methodology

**System architecture.** EIR partitions responsibility by SoC island: the target accelerator runs in the Programmable Logic (PL), while an AXI-GPIO IP monitors its outputs and flags faults. Detection and recovery run entirely on the Processing System (PS), so no fabric-level replica is ever built. PL state is captured through the PCAP interface, using a logic-location (`*.ll`) file (stored on an SD card) to selectively read back only the active regions into off-chip DDR memory. The PS hosts both digital twins and the checkpoint policy.

**The two twins.**
- *Rabbit (Twin #1, detection)*: a coarse-grained behavioral model written in C/C++ that re-executes the accelerator's functional logic at a higher abstraction level (no cycle-by-cycle fidelity). At selected checkpoints it runs the same input batch as the PL and compares outputs; a divergence `|O_Rabbit − O_PL| ≥ ε` flags a potential fault. Rabbit is intentionally not always-on because (i) PS is usually slower than PL, (ii) continuous running would starve other PS tasks, and (iii) it would raise energy.
- *Tortoise (Twin #2, recovery)*: a fine-grained, gate-level twin produced by the Yosys CXXRTL backend (v0.50) from the *same* Verilog used for the PL bitstream, preserving registers, combinational logic, and signal-level detail. It stays dormant until Rabbit flags a fault, then emulates forward cycle-by-cycle from the latest checkpoint to produce the correct fault-free state.

**Three-phase operation.** (1) Normal execution with periodic validation against Rabbit; (2) fault localization via Tortoise, loading the checkpoint at `t_capture` plus the intervening input stream and replaying to `t_fault`; (3) state correction/recovery — the corrected state `O_Tortoise` is written back through PCAP as a partial-reconfiguration update confined to target flip-flops/pipeline registers, after a PS↔PL handshake quiesces clocks at safe synchronization points to avoid "torn" state.

**Checkpoint-selection derivation (two stages).** Stage 1 (layer-level sensitivity) injects per-layer bit flips into MAC results on a LeNet-5 / CIFAR-10 model and finds convolutional layers tend to mask faults whereas dense/logit layers corrupt the decision. Stage 2 (magnitude-aware) computes per-MAC contribution `m_{l,i}(x) = |x_{l,i}·w_{l,i}|`, ranks MACs into percentiles, and injects faults to show outcome-altering faults cluster in the top slices: ~30% of MACs in layer 6, ~17% in layer 7, ~18% in layer 8 — aggregated to ~25% of dense-layer / ~5% of all network MACs capturing all observed misclassifications. EIR therefore defaults to k=30%.

**Fault model and injection.** EIR targets single- and multi-bit transient faults (voltage fluctuation, EMI, radiation-induced bit flips) in PL configuration memory and on-chip data memories holding weights/activations; BRAMs, the PS, and DDR are assumed ECC-protected and trusted. Permanent faults are out of scope. To systematize campaigns without radiation/glitch hardware, EIR uses a *readback-level* injection technique (per prior work): it XORs selected bit positions in the checkpoint readback data to emulate transient bit flips. The authors stress this is a methodological convenience — EIR's detect/compare pipeline is unchanged when real live PL faults occur.

## Results & Highlights

- **Fault coverage vs. neuron budget (dense layer, LeNet-5/CIFAR-10 fault campaign):** Auditing the **top 30% of neurons** delivers **≥95% fault-detection rate**; aggregated sensitivity analysis shows that auditing **~25% of dense-layer MACs (only ~5% of all network MACs)** is sufficient to capture **all observed misclassifications** under the evaluated multi-bit transient model — the smallest top slice capturing all misclassifications is **30% of MACs in layer 6, ~17% in layer 7, ~18% in layer 8**.
- **PL area overhead — EIR vs. DMR:** Across the 11 accelerators, EIR preserves the baseline PL area, whereas DMR raises LUT/FF utilization by **~2.0–2.25× on average** (e.g., SHA-256 729/2160 → 1592/4364; AES-128 1796/406 → 3595/815; Conv2D 12250/21711 → 24500/43430; FFT-64 8656/2129 → 17415/4260).
- **Power overhead — EIR vs. DMR:** DMR raises PL power consumption from **2.98 W (EIR/baseline avg) to 5.79 W (DMR avg), ≈89.1% higher**; per-design, e.g., FFT-64 8.13 W → 16.7 W, Attention 4.25 W → 8.71 W, AES-128 2.13 W → 4.40 W.
- **Selective validation timing win:** Full PS-side dense-layer verification would stall PL for ~530 µs (4096 − 311 ≈ 3785 overlapping MACs at 0.14 µs/MAC); auditing only the top-30% subset (N_sel = 0.3 × 4096 = 1228 MACs) reduces the isolation window to **~170 µs — a ~3× improvement** while focusing on the most outcome-impactful computations.
- **Recovery latency budget:** The `t_capture` checkpoint phase ≈ 31.46 ms and `t_fault` localization ≈ 31.91 ms (both dominated by conservative software + PCAP bandwidth and triggered only on rare faults), state validation ε ≈ 39.3 ms, while the actual corrected-state write-back to PL (`t_restore`) completes in **only 5.7 ms** — leaving steady-state accelerator throughput unaffected.
- **Speed-ratio characterization (twin vs. PL, Table II):** Rabbit ranges from ~0.5× (Conv2D) to ~25× (IIR) of PL speed; Tortoise is 2–4 orders of magnitude slower (e.g., IIR ~9882×, SHA-256 ~6266×, AES-128 ~6063×), justifying its use only as an exception-path mechanism rather than a continuous companion.

## Authors & Credibility

- **Arsalan Ali Malik** (NC State) and **Aydin Aysu** (NC State) — the authors of the underlying checkpoint-preemption work EPOCH (arXiv:2501.16205), the CRAFT pre-silicon fault-injection characterization, the "Precision Strike" CNN clock-glitch attack, and prior Zynq SoC isolation-flow methodology. Their track record in FPGA security/fault tolerance directly underpins EIR's PCAP-based state capture and fault-injection methodology.
- **Ali Suvizi** and **Guru Venkataramani** (George Washington University) — authors of *Auto-Healer: Self-Healing Hardware for Perception Stage Faults in Autonomous Driving Systems* (ICS 2025), the most directly related prior self-healing work that EIR positions itself against (EIR removes Auto-Healer's reliance on DMR-style replication).
- The team is therefore well-placed to make the architectural comparison to DMR/TMR and to the Auto-Healer software-level approach, and the toolchain (Vivado 2024.2, Yosys CXXRTL 0.50, Vitis SDK) is mainstream and reproducible.

## Critical Assessment

- **Novelty — genuine but incremental, and the framing is candid about it.** The core idea (run a coarse twin on the PS for detection and a cycle-accurate twin for occasional recovery, instead of duplicating the PL) is a sensible systems contribution and, to the authors' credit, they explicitly state this is the *first* such combination for FPGA SoCs. However, the checkpoint/restore primitive itself is not new — EIR's Sec. III-C and Sec. V-E openly acknowledge they reuse the PCAP + `*.ll` readback/writeback machinery of EPOCH [17] (by the same authors), repurposing it from preemption to fault tolerance. The honest framing is a plus; the incremental delta over EPOCH is the twin hierarchy and the fault-triggered recovery path.
- **Rigor — the headline "≥95% coverage" is for a narrow, carefully conditioned setup.** That number is measured specifically on the LeNet-5 dense-layer case study under a multi-bit transient fault model injected at the MAC-result register via *readback-level* corruption. The paper itself concedes (Sec. IV-C) that extending the per-fault coverage analysis to the other 10 benchmarks (SHA-256, AES-128, GEMM, attention, etc.) "is omitted due to space constraints" and that "it does not alter the main conclusion" — a claim that is asserted rather than demonstrated. A reviewer should treat "≥95% coverage" as workload-specific, not as a system-wide guarantee.
- **Methodological convenience that weakens external validity.** Injecting faults by XOR-ing readback bits is, as the authors note, only an emulation accelerator; but it means the experiments do *not* exercise faults that originate inside live PL combinational logic or that propagate through paths the readback never captures. The claimed equivalence ("EIR detects this by comparing the checkpoint-derived state/output against Rabbit") is argued, not measured against a live-fault oracle.
- **Missing ablations.** There is no sensitivity study over the checkpoint spacing Δ, no study of how coverage degrades as PS utilization rises (the paper only argues qualitatively that detection latency increases but correctness is preserved), and no comparison against a partial/selective redundancy baseline — the authors explicitly *exclude* partial redundancy as a baseline on the grounds that it "does not generalize," which is convenient because it removes the most direct competitor (e.g., HMR-NEureka, HyCA) from the comparison table.
- **Reproducibility — moderately good.** Benchmarks are standard (MachSuite-inspired, secworks SHA-256, an 8-bit quantized Edge-TPU-style systolic array), toolchain versions are pinned (Vivado 2024.2, Yosys CXXRTL 0.50), and the `*.ll`-driven PCAP flow is described in enough detail to reimplement. But no code/artifact release is mentioned, the `*.ll`→frame mapping is bitstream-specific, and the magnitude-aware top-30% policy is derived from per-model profiling, so results will not transfer verbatim to other networks without re-profiling.
- **Significance — real for the narrow niche, oversold at the margins.** The area/power advantage over DMR is essentially definitional (EIR adds nothing to the PL, so any always-on PL redundancy will look worse), so the "89.1% power reduction" headline is less a research result than a consequence of the design point. The genuine contribution is the demonstration that PS-side twins can *detect and recover* within a few-ms budget on a Zynq-7000 — useful for cost-sensitive edge AI where brief stalls are tolerable.
- **Red flags.** (i) A Wikipedia citation for "Tesla Autopilot" in the motivating examples is a weak sourcing choice for a safety-critical-claims introduction. (ii) The phrase "comparable empirical fault coverage at reduced energy and area cost" is doing a lot of work — "comparable" is measured only against DMR on the dense-layer case, not against ABFT or selective hardening. (iii) No end-to-end accuracy report on CIFAR-10 *after* recovery (i.e., does the recovered classification actually match the fault-free label?), only the detection rate.

## Strengths

- Clean and well-motivated decoupling of *detection* (cheap, frequent, coarse) from *recovery* (expensive, rare, precise) — a sound engineering division of labor that maps naturally onto the PS/PL resource asymmetry of FPGA SoCs.
- Zero PL area overhead and no accelerator RTL modification is a genuinely attractive property for deployment, since it preserves the customer's synthesized netlist and timing closure.
- The magnitude-aware MAC ranking is a principled way to spend the limited PS verification budget, and the 530 µs → 170 µs isolation-window reduction is a concrete, defensible timing win.
- Honest scoping: the assumptions (PS slack available, faults infrequent, deterministic replay, ECC-protected BRAM/DDR/PS, transient-only) are stated explicitly in Sec. II-E and Sec. V-A rather than buried.
- Strong author pedigree in exactly this subfield (EPOCH, Auto-Healer, CRAFT, Precision Strike), giving the related-work positioning and the DMR/TMR comparison real authority.

## Limitations

- **No formal coverage guarantee.** EIR only guarantees detection of faults that propagate to monitored outputs within Δ cycles of a checkpoint; faults outside the protected subset or that do not manifest at monitored outputs may go undetected. The authors acknowledge stronger formal guarantees are "an open research problem."
- **Single prototype platform.** All results are on Zynq-7000 (XC7Z020); scalability to UltraScale+/larger SoCs (deeper pipelines, larger state) is explicitly left as future work, and Tortoise's replay cost will grow with accelerator complexity.
- **Workload-specific thresholds.** The default k=30% is derived from LeNet-5/CIFAR-10 dense-layer profiling; the paper warns it is "not a universal value" and that networks with irregular control flow or dynamic execution graphs will need adaptive checkpoint placement — which EIR does not yet provide.
- **Recovery assumes deterministic replay.** Any untracked nondeterminism (asynchronous software intervention, uncontrolled external input streams, timing-dependent side effects outside the replay window) breaks correctness — a real constraint for systems with DMA or async I/O.
- **Detection latency is PS-load-sensitive.** As PS utilization climbs, the fraction of MACs Rabbit can validate online shrinks, increasing detection latency (though not recovery correctness); the mitigation (coarser checkpointing) is only sketched.
- **Fault-injection methodology is readback-level emulation.** Results are not validated against a live physical fault source (radiation, voltage glitch), so the equivalence to real in-field transient faults is argued rather than measured.

## Relevance

与用户的边缘AI/硬件安全研究方向高度相关：EIR直接处理FPGA SoC上CNN加速器的瞬态故障检测与恢复，与CNN加速器安全（bit-flip攻击、时钟毛刺攻击）、边缘部署的容错设计主题强相关，可作为"软件侧数字孪生替代硬件冗余"这一思路的具体案例参考。

## Questions for Further Exploration

- How does EIR's coverage and recovery latency scale beyond the Zynq-7000 to UltraScale+ devices with deeper pipelines and 10–100× larger accelerator state — does Tortoise's replay cost stay bounded and usable within real-time control deadlines?
- What is the end-to-end post-recovery *classification accuracy* on CIFAR-10 (i.e., does the Tortoise-reconstructed state actually yield the correct fault-free label), as opposed to the reported detection rate?
- How does EIR compare against partial/selective redundancy baselines (HMR-NEureka, HyCA, ePredictNet) on the *same* benchmarks, rather than only against full DMR — would the area/power advantage survive that fairer comparison?
- Can the top-30% magnitude-aware threshold be derived *adaptively* at runtime for networks with dynamic execution graphs or shifting input distributions, and how much coverage is lost if the threshold is fixed offline and the workload drifts?
- Does the readback-level fault-injection methodology (XOR-ing checkpoint bits) produce the same detect/recovery behavior as live physical faults (voltage droop, SEU) — can the equivalence claim be validated against a radiation or FPGAHammer-style voltage-glitch oracle?
- What happens to detection latency and effective coverage when the PS is heavily loaded by other real-time software tasks — is there a graceful degradation curve or a cliff?
