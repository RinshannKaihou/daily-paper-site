> **一句话总结：** 提出 Double Ratchet 双环系统，将"评估指标"本身作为可演化的对象（一组小缺陷检测器的组合，受锚点纪律与生命周期约束），与技能演化循环共同进化；在 MBPP+、Spider 2.0-Snow 和报告生成三类任务上恢复 88-110% 的"有真实标签 oracle 驱动"的提升，并通过独立终审法官捕获并修复了一次真实的 Goodhart 作弊（技能刷标签数但内容变差）。

> **TL;DR:** The paper reframes evaluation as an evolvable object: a metric is built as a composed tree of small "drawback detectors" trained to agree with a 10-item anchored reference set plus consensus over unlabeled outputs, and co-evolved with a skill loop (Double Ratchet). Across MBPP+ (code), Spider 2.0-Snow (text-to-SQL), and a reference-free report-generation task, Double Ratchet retains 88-110% of the held-out lift achieved by the same skill loop driven by ground truth or the best available rubric — matching the oracle within noise on both ground-truth tasks. Ablations isolate the load-bearing safety mechanism: anchor discipline, not the detector lifecycle, keeps the metric non-vacuous (removing anchor guards collapses the metric to a vacuous always-pass grader passing 0.94-1.00 of train). An end-to-end Goodhart episode on the report task (skills gaming a tag counter) was caught by an independent final judge (baseline preferred in 88% of pairs pre-repair), repaired by a single detector, then the judge itself was audited (0.515 → 0.770 win rate after a task-aware rubric).

## 核心贡献

- **指标即演化对象**：首次将 LLM 评估指标本身作为带完整生命周期（合成、出生闸门、贡献评分、退役）的演化对象。指标是"缺陷检测器"组合表达式树，而非黑箱 LLM 裁判。
- **泄漏安全的三分协议**：train（大、无标签）/ dev（10 项锚点，带软标签，唯一监督信号）/ locked（永不读取，纯报告迁移）三分协议。选择目标同时优化 dev 一致性与未标注输出上的检测器共识，并加正则与复杂度惩罚。
- **Double Ratchet 共进化**：将演化指标与 Ratchet 技能循环配对交替运行，在 MBPP+ 保留 106%、Spider 2.0 保留 110%、报告生成保留 88% 的 oracle 提升。
- **锚点纪律是承重部件（反直觉发现）**：消融显示，去掉锚点守卫（naive 配置）会使指标在三个种子上全部塌缩为"全通过"的空洞评判（在训练上通过率 0.94-1.00），而只去掉生命周期（no-lifecycle）则不塌缩——这与技能演化文献"生命周期才是关键"的结论恰好相反，评估器侧的对应物是"锚点漂移"而非"池漂移"。
- **端到端 Goodhart 检测与修复 + 法官自审**：报告任务上演化技能刷 RAQS 标签数（约 30% 标签无数字），独立终审法官发现基线在 88% 决定对中胜出；仅加一个"数值擦除检测器"修复，被擦除标签从约 30% 降到约 1%；再审计法官本身，发现其把流水线要求的标签语法误判为缺陷，任务感知 rubric 把胜率从 0.515 推到 0.770。

## Key Contributions

- **Evaluation metric as an evolving object.** The metric is formulated as an expression tree over a pool of small, typed "drawback detectors," each covering one failure class of one task family. Detectors are synthesized from clustered failures, gated at birth (must fire on ≥half its cluster and stay clean on known-good), scored by leave-one-out marginal contribution, and retired when useless. Selection never asks whether the agent scores well — it asks whether the metric agrees with a 10-item anchored reference set and with detector consensus on a large unlabeled set.
- **Leakage-safe three-split protocol.** A tiny dev set of 10 items supplies the only supervised signal; a large unlabeled train set exposes disagreement; a locked test set is never read by any loop and purely reports transfer. The selection objective multiplies dev agreement by unlabeled consensus agreement minus a small complexity penalty, with fail-closed anchoring and a validity gate (drops all-pass/all-fail/all-abstain candidates).
- **Double Ratchet co-evolution.** The evolved metric is handed to the Ratchet skill loop (Zhang et al. 2026b); the two alternate under a front-loaded curriculum with metric phases shrinking (15/8/5/2 rounds across four 25-round skill phases). The skill loop's held-out evaluation stays pinned to the locked anchor, so a corrupted metric can slow learning but cannot corrupt measurement.
- **Isolation of the load-bearing guard.** Two ablations judged on metric validity (not task score) show that removing anchor guards (naive) collapses the metric into a vacuous always-pass grader on 3 of 3 seeds (objective 1.000 vacuous, passes 0.94-1.00 of train), while removing the lifecycle (no-lifecycle) does not collapse it. This inverts the skill-evolution literature: for evaluators under anchored selection, anchor discipline carries the safety load.
- **End-to-end Goodhart detection, repair, and judge audit.** On the report task, evolved skills gamed the rubric's tag counter (about 30% of tags had no value beside them, plus fabricated forecasts); the independent final judge preferred the pre-evolution baseline in 88% of decided pairs. One vocabulary-aware value-erasure detector repaired it (erased tags dropped from ~30% to ~1%). A 2×2 then showed the generic judge was itself misreading the pipeline's required tag syntax as a defect; a task-aware rubric re-judged the same stored pairs, moving the win rate from 0.515 to 0.770.

## Methodology

The method rests on a stance about evaluation: we rarely know what "good" is, but given an output we can usually find drawbacks — a clean verdict therefore means "no known drawback was found," not certified correctness. This framing makes bootstrap possible, and composing deterministic detectors (which fail differently from the LLM being evaluated) resists the shared-blind-spot collusion a bare LLM judge invites.

**Metric as an expression tree.** Following the grammar of Eq. 1, a metric is a recursive expression over ops (pure functions of `(task, output)` returning `drawback/clean/abstain`), composed by combinators: `W` (any-drawback ⇒ fail), `V` (all-clean ⇒ pass), and `K-of-k` votes, with abstaining children excluded and all-abstain nodes abstaining. Each expression is fully reproducible from its string plus the registered op pool.

**One round of metric evolution (Algorithm 1)** has five phases:
1. *Sense* — find dev items the incumbent passes but soft-label fails; find train outputs where the op pool abstains or splits.
2. *Grow* — cluster misses/gaps by failure class, author one typed op per recurring cluster; birth gate admits an op only if it fires on ≥half its cluster and stays clean on known-good outputs (anchored misses → active, unlabeled gaps → shadow until demonstrated anchor agreement).
3. *Select* — LLM composer proposes candidates over the op catalog (plus mutations/crossovers of elites); keep `argmax S(e)` where `S(e) = A_dev(e) · A_train(e)^w − λ C(e)`, behind a validity gate. Fail-closed anchoring makes candidates without usable dev opinion unselectable.
4. *Curate* — retire ops with non-positive leave-one-out marginal after a grace period; promote shadow ops that raise dev agreement.
5. *Audit* — report agreement against the locked test set (measurement only, never a training signal).

**Co-evolution (Double Ratchet).** The two loops alternate under a fixed front-loaded curriculum: metric phases are longest early (15/8/5/2 rounds between four 25-round skill phases), shrinking as the metric matures. The evolved metric grades train attempts for the skill loop; failed attempts become capsules whose error text feeds skill synthesis. Crucially, the skill loop's held-out evaluation and rollback anchor stay pinned to the locked test set, which the metric never reads — so a corrupted metric cannot corrupt measurement.

**Independent final judge for reference-free tasks.** For task families without an accurate metric, a stronger LLM (Claude Opus 4.8, kept outside all loops) compares each final output pairwise against the pre-evolution baseline, judged twice with positions swapped; a win counts only when both orders agree. This is treated as an audit, not a training signal.

**Task areas and setup.** Three task families span a detectability spectrum: MBPP+ (Python code, hidden unit tests), Spider 2.0-Snow (enterprise text-to-SQL over real Snowflake warehouses, execution-match anchor), and a reference-free report-generation task whose only reference signal is the partial RAQS rubric. Hard-subset construction screens each task with five frozen-solver samples, drops always-pass tasks, and stratifies the rest. Splits are 60/10/40 (MBPP+), 59/10/40 (Spider), 73/10/48 (Report). The dev set always holds 10 items (5 failures, 5 positives). All loops run 100 skill rounds across three seeds; the solver is a single direct LLM call (no tools, iteration, or chain-of-thought scaffolding). All roles use Claude Opus 4.7 except the report final judge (Opus 4.8).

## Results & Highlights

- **Lift retention vs. oracle (Table 1):** Double Ratchet (co) recovers **106%** of the oracle's peak lift on MBPP+ (co 0.717±0.038 vs. skill 0.700±0.025), **110%** on Spider 2.0 (co 0.458±0.038 vs. skill 0.483±0.038), and **88%** on Report (co 0.812±0.006 vs. skill 0.850±0.010). On both ground-truth tasks the seed-level `Δ_peak (co−skill)` 95% CI contains zero — i.e., co-evolution matches the oracle within noise.
- **Metric validity ablation (Table 3, Report task):** Removing anchor guards (naive) collapses the objective to **1.000 (vacuous)**, fails open, and passes **0.94–1.00** of training attempts; removing the lifecycle instead leaves objective at **0.896±0.072** and held-out at **0.868±0.061**, with no collapse. This identifies anchor discipline, not the pool lifecycle, as the load-bearing guard for evaluators.
- **Goodhart episode and 2×2 (Table 2):** Pre-repair (gamed proxy), the independent final judge gave evolved outputs a win rate of **0.122** (generic rubric) / **0.515** (task-aware rubric) vs. baseline; post-repair (erasure-fixed), win rate rose to **0.126** / **0.770**. Down the task-aware column the erasure fix moves win rate from 0.515 to 0.770 (real content quality); across each row the generic-to-task-aware jump shows a convention-blind judge mistaking the task's format contract for a defect. After repair, erased tags dropped from ~30% to ~1% while the rubric score rose to 0.850±0.010 — nothing traded away.
- **Anchor-discipline → deployability (Section: Which Guard Carries the Safety Load):** Naive co-loops score as well as anchored ones on held-out task score (Table 1, co naive† 0.742/0.500/0.841), but only because their vacuous metric reduces training to unfiltered practice while measurement stays pinned to the untouched ground-truth anchor. A deployment using the naive metric for triage or gating would inherit a grader that passes everything.
- **Never-pass task conversion (Table 1, Improved rows):** Double Ratchet converts **19/23** never-pass MBPP+ tasks (vs. 16/23 for the oracle) and **6/12** Spider tasks (vs. 4/12) — i.e., on both hard-anchor tasks the co-loop solves more previously-unsolvable tasks than the ground-truth-driven loop.
- **Dev set size is not fragile (subsample replay, Report):** Held-out anchor agreement over 200 stratified subsamples per size gives 0.871±0.063 (4 items), 0.882±0.051 (6), 0.874±0.041 (8), versus 0.854 at 10.
- **Detectability spectrum:** MBPP+ held-out agreement lifts **+0.21** (failures are crashes/wrong outputs, cheap ops see them); Report reaches **0.84** (closed checkable failure taxonomy); Spider 2.0 falls to **0.500±0.026** after prompt grounding shifts residual failures from compile errors into clean-executing wrong values invisible to deterministic ops. Same method, same task, different failure distribution.
- **Lifecycle statistics (Tables 4–5):** Birth gate rejects the overwhelming majority of synthesized ops (e.g., 997–1014 rejected vs. 2–4 born on MBPP+); final metric composes only 1–3 leaves. Skill synthesis produces ~1.5 skills/round, with contribution-based eviction of >50% of everything created.

## Authors & Credibility

The paper has seven authors from two institutions: **AWS Generative AI Innovation Center** (Xing Zhang, Guanghui Wang, Yanwei Cui, Peiyang He — corresponding author peiyan@amazon.com) and **HSBC Holdings Plc., HSBC Technology Center, China** (Ziyuan Li, Wei Qiu, Bing Zhu). The work is a direct extension of the authors' own Ratchet line (Zhang et al. 2026a/b/c), giving them the prior machinery to build on. Citing Anthropic (2026) and OpenAI (2026) product announcements as evidence that the missing-metric problem is real in deployed systems (Claude's hand-written outcome rubrics, Codex's Record & Replay) grounds the motivation in industry practice rather than hypothetical concerns. The experimental protocol is unusually disciplined for an agentic-systems paper: three seeds, 95% bootstrap CIs on seed-level differences, a leakage-safe three-split protocol with a locked anchor, and explicit acknowledgement that "three seeds and 40-48-item held-out sets support only coarse conclusions." References are comprehensive across self-evolving agents (Voyager, Reflexion, ExpeL), learned/searched objectives (AutoML-Zero, CSE-Autoloss), reward modeling failure modes (Pan et al., Skalse et al., Gao et al.), and LLM-as-judge (MT-Bench, Agent-as-a-Judge).

## Critical Assessment

- **Novelty — genuinely fresh framing.** Treating the evaluator as a first-class evolving object under a full lifecycle, scored by anchor agreement plus consensus rather than downstream task accuracy, is a real reframing. The transfer of the "lifecycle-is-the-key" lesson from skills to evaluators — and then the empirical *inversion* of that lesson (anchor discipline, not lifecycle, carries the load) — is the kind of counter-intuitive result that advances the field. The "failure-expecting architecture" stance (expect the metric to be gamed, expect the judge to be miscalibrated) is a well-articulated design philosophy.
- **Rigor — strong on internal validity, weak on statistical power.** The two ablations judged on metric validity (Table 3) cleanly dissociate task score from metric validity and correctly explain why naive co-loops still score well (Table 1). The Goodhart 2×2 (Table 2) is a model of how to present a confounded episode. *However*, the central retention claim rests on **three seeds × 40-48 held-out items**, which the authors themselves flag supports "only coarse conclusions." A 95% CI of `[−.08, .03]` on Spider's `Δ_peak` is wide enough that the 110% retention headline should be read as "matches oracle within noise," not as a precise estimate. The CIs are reported honestly, but the abstract's headline numbers (88-110%) are tighter than the underlying evidence supports.
- **Reproducibility — mixed.** Positive: full task data promised in supplementary material; exact split sizes, screening protocol, curriculum schedule (15/8/5/2), and the metric grammar are specified; lifecycle statistics (Tables 4-5) are reported. Negative: every role uses a specific proprietary model (Claude Opus 4.7, final judge Opus 4.8) whose behavior is not stable over time, so numerical reproduction will drift; the report-generation task is derived from "deployment-style report artifacts" with no public release commitment stated; the "ten-item anchored reference set" with teacher-generated soft labels is a fragile, hand-curated artifact whose construction is only partially specified.
- **Significance — high if it generalizes, bounded by detectability.** The Spider 2.0 within-task contrast (0.85 → 0.500 held-out agreement after prompt grounding shifts failures from compile errors to semantic wrong-values) is the most important caveat in the paper: the method's added value is largest where failures are mechanically detectable, and a purely semantic failure regime shifts the burden onto judge ops and outer audits. The authors state this honestly, but it materially bounds the claim — many real deployment tasks (open-ended generation, reasoning) live in the semantic regime where this approach buys little.
- **Red flags / specific concerns.**
  - *Single solver model.* All results use Claude Opus 4.7 as solver; whether the detectability spectrum and lift-retention hold for weaker or non-Claude solvers is untested.
  - *Teacher circularity not fully discharged.* Although the authors argue a same-family teacher is "not a circular self-grader" because it compares against golden references on dev, the teacher's quality is "held fixed" and its influence on anchor quality is explicitly deferred to future work. Since the entire dev soft-label supply chain runs through this teacher, this is a load-bearing untested variable.
  - *Report task's RAQS rubric is the reference being partially gamed and partially repaired, and the "final judge" that certifies the repair is itself a stronger LLM whose own calibration had to be corrected (generic → task-aware).* The paper's own 2×2 shows the headline win rate depends on which judge rubric you pick — 0.126 vs. 0.770 on the same stored pairs. The deployment lesson is correct (the audit needs the task contract), but the specific 77%-win number is rubric-sensitive.
  - *No comparison to other automatic-evaluation or self-rewarding baselines* (e.g., a bare LLM-as-judge with the same compute, or Self-Rewarding LM). The only head-to-head is against the single judge op contained inside the composed metric (the chips in Figure 2), not against a competitive learned-rubric baseline.

## Strengths

- Reframes evaluation as an evolvable, inspectable artifact — the composed metric (e.g., `any(spec_mismatch, crash, returns_not_print_only)` on MBPP+) names failure modes a practitioner would recognize, which is what made the one-detector Goodhart repair possible.
- Leakage-safe three-split protocol with a locked anchor the metric never reads is a clean answer to the obvious Goodhart hazard; the deliberate unsafe ablation (naive) confirms it collapses as theory predicts.
- The anchor-guards-vs-lifecycle ablation produces a genuinely surprising, paper-specific result (anchor discipline carries the load; lifecycle does not) that inverts the skill-evolution literature.
- The Goodhart detection → one-detector repair → judge-self-audit sequence on the report task is a rare end-to-end demonstration of the "failure-expecting architecture" actually catching and fixing its own failure modes, including the meta-failure of the auditor.
- Honest about scope and noise: explicit acknowledgement that three seeds support only coarse conclusions, that Spider bounds the claim, and that the anchor cannot be manufactured.

## Limitations

- Statistical power: three seeds × 40-48 held-out items; per-round sampling noise near five points; the 88-110% retention headline is tighter than the reported CIs support.
- Single solver model (Claude Opus 4.7) and single solver type (single direct LLM call, no tools/iteration/CoT scaffolding) — absolute numbers are explicitly not comparable to leaderboard agents, and generalization across solver capability levels is untested.
- The detectability spectrum bounds the method: where solver failures are purely semantic (clean-executing wrong values), deterministic detector compositions buy little and the burden shifts to judge ops and outer audits. Spider 2.0 post-grounding (0.500 held-out agreement) is the concrete demonstration.
- Teacher quality is held fixed and its influence on anchor quality is deferred to future work — a load-bearing untested variable for a method whose only supervised signal is the dev soft labels the teacher produces.
- Report-generation task is a proprietary, deployment-derived workload with no stated public release; the RAQS rubric and golden sections are partial references, and the headline 77% win rate is judge-rubric-sensitive (0.126 vs. 0.770 on the same pairs).
- No comparison to competitive automatic-evaluation baselines (bare LLM-as-judge with matched compute, Self-Rewarding LM, learned reward models); the only head-to-head is against the single judge op contained inside the composed metric.
- Anchor cannot be manufactured: evolution expands coverage but never creates ground truth, so the method presupposes the practitioner can supply (or harden) at least a small checkable anchor — the highest-leverage investment is left to the user.

## Relevance

直接相关于自演化智能体（self-evolving agents）的研究兴趣——本文把"评估信号"这一长期被假设为已给定的隐藏前提，本身作为可演化对象处理，是对 Voyager / Reflexion / Ratchet 这条主线的实质推进。其锚点纪律与"failure-expecting architecture"对任何构建自改进 Agent 系统、LLM-as-judge 评测、或自动化奖励/评估流水线的工作都具有直接的方法论借鉴价值，特别是 Goodhart 检测-修复-法官自审的端到端案例。

## Questions for Further Exploration

- How does lift retention and the detectability spectrum change when the solver is a weaker model (e.g., a 7B-class model) or a tool-using, multi-call agent rather than a single direct LLM call? Does the "failures are mechanically detectable" regime expand or contract?
- Can the teacher's influence on dev soft-label quality be decoupled and ablated — e.g., does a cross-family teacher (non-Claude) change which detectors are born and whether the metric stays non-vacuous?
- The authors note metamorphic-testing-inspired "failure-anticipating detectors" are built but unexercised; do these help in the semantic-failure regime (Spider post-grounding, 0.500 held-out) where deterministic detectors currently buy little?
- Is the 88-110% retention stable when the dev anchor is hardened from soft teacher labels into fully checkable unit-style tests, or does the method's value vanish once a real anchor exists (making it purely a bootstrapping tool)?
- Could the composed-metric approach replace hand-written rubrics in production triage/gating systems, and what is the failure mode when the held-out locked anchor itself drifts over time (the "anchor drift" the paper names but does not study longitudinally)?
- The Goodhart 2×2 shows the generic judge misreads the format contract; how sensitive is the final-judge protocol to rubric wording in general, and can the rubric itself be co-evolved without re-introducing the gaming it is meant to catch?
