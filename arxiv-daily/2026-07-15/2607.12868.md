> **一句话总结：** 作者构建了 Deep4ge——一个公开的、基于变异注入的 DNN 训练基准，包含来自 59 个 Stack Overflow TensorFlow/Keras 程序的 14,227 条逐 epoch 训练轨迹（9,845 条故障 + 4,382 条正常），覆盖 7 类故障与 FNN/CNN/RNN 三种架构；基准实验证明使用完整轨迹特征比仅用末轮特征能显著提升故障检测与诊断的 MCC（检测 0.227 vs 0.150，诊断 0.401 vs 0.346）。

> **TL;DR:** Deep4ge is a controlled, public benchmark of 14,227 per-epoch DNN training runs (9,845 faulty, 4,382 correct) generated by injecting 27 mutation operators across 7 fault categories into 59 adapted TensorFlow/Keras programs from Stack Overflow, covering FNN/CNN/RNN architectures with 719,560 epoch records. Proof-of-concept baselines (Logistic Regression, Random Forest) under group-aware cross-validation show that trajectory features beat final-epoch features on the Deep4ge benchmark: detection MCC rises 0.150→0.227 (RF) and diagnosis MCC rises 0.346→0.401, while detection AUROC improves 0.649→0.716.

## 核心贡献

- 首个公开的、将真实 DNN 源码程序、变异派生故障类别与逐 epoch 训练轨迹三者关联的可复用基准数据集，填补了 SE 社区缺乏可复用 DNN 训练时故障数据的空白。
- 数据规模可观：14,227 条训练轨迹、719,560 条 epoch 记录（中位数 50 epoch/运行）、27 个变异算子、7 个故障类别（Hyperparameter、Loss、Weight、Layer、Optimization、Activation、Regularization）、覆盖 FNN/CNN/RNN 三种架构。
- 提出可复现的故障注入框架：通过 AST 改写实现 27 个变异算子，采用 DeepCrime 的统计杀死准则（Cohen's d ≥ 0.5、GLM p < 0.05、n=15 次重训练），算子集在 FNN/CNN/RNN 上分别达到 0.85/0.87/0.83 的变异分数。
- 提供 proof-of-concept 的故障检测（二分类）与故障诊断（7 类单标签）基线，并设计 group-aware 5 折交叉验证防止同程序数据泄漏，给出 1,000 次 bootstrap 的 95% 置信区间。
- 实证证据：完整训练轨迹比末轮特征更富信息量；并给出跨架构迁移、单 epoch 早期检测、逐类别诊断与混淆矩阵等多维度分析，明确指出 Activation/Optimization 类故障是难点。

## Key Contributions

- The first public dataset linking adapted DNN source programs, mutation-derived fault categories, and per-epoch training trajectories in a single reusable benchmark, addressing the SE community's lack of reusable DNN training-time fault data.
- Substantial scale: 14,227 training runs (9,845 faulty, 4,382 correct), 719,560 epoch records (median 50 epochs/run), 27 mutation operators, 7 fault categories (Hyperparameter, Loss, Weight, Layer, Optimization, Activation, Regularization), spanning FNN/CNN/RNN architectures.
- A reproducible fault-injection framework: 27 operators implemented via AST rewriting, using DeepCrime's statistical killing criterion (Cohen's d ≥ 0.5, GLM p < 0.05, n=15 retrainings); the operator set attains mutation scores of 0.85/0.87/0.83 on FNN/CNN/RNN.
- Proof-of-concept baselines for binary fault detection and 7-class single-label fault diagnosis, evaluated with group-aware 5-fold cross-validation to prevent same-program leakage, and reported with 1,000-resample bootstrap 95% CIs.
- Empirical evidence that full training trajectories carry more signal than final-epoch features, plus cross-architecture transfer, single-epoch early detection, per-category diagnosis, and a normalized confusion matrix analysis that identifies Activation/Optimization as the hardest categories to separate.

## Methodology

Deep4ge is constructed in four stages. (1) **Program selection**: 59 complete TensorFlow/Keras DNN programs (training code included) were collected from Stack Overflow, each tied to a unique SO question ID, manually repaired to run deterministically, and instrumented with a logging callback. Seeds were fixed at framework and operator level and dataset splits/inits were kept consistent to make regeneration reproducible; architectures cover feedforward, convolutional, and recurrent networks. (2) **Fault injection**: starting from each correct baseline, the framework applies one of 27 source-level mutation operators implemented via Python AST rewriting. The operator set reuses a subset of DeepCrime's operators (training configuration, activation, loss, optimizer, weight init/bias, regularization) and adds the full set of layer-level operators not supported in prior work; data- and process-level operators were deliberately excluded. Each released run carries exactly one category label, so diagnosis is a 7-class single-label task (multilabel is not evaluated). (3) **Training & feature extraction**: each mutant and its correct baseline are trained for n=15 retrainings; per epoch the system records an epoch index, 4 evaluation metrics (train/val loss and accuracy), and 26 dynamic features — 4 weight measures, 4 accuracy/loss trend measures, 4 activation measures, 10 gradient measures, and 4 environment measures (learning rate, CPU %, peak GPU memory, system memory %; GPU memory is 0 when no GPU is available). (4) **Mutation validation**: a mutant is "killed" only if the accuracy distribution difference between original and mutant over the test set is both statistically significant (GLM p < 0.05) and of at least medium effect (Cohen's d ≥ 0.5). Equivalent mutants are dropped; operators are retained only if killable on at least one program of the same architecture, yielding mutation scores 0.85 (FNN), 0.87 (CNN), 0.83 (RNN).

For the proof-of-concept benchmarks the authors compare two feature representations: a **final-epoch** representation (30 values at the last epoch) and a **trajectory** representation (each of 30 features summarized by 6 statistics — mean, std, min, max, last, linear slope — yielding 180 features). They evaluate Logistic Regression and Random Forest using 5-fold group-aware cross-validation grouped by SO program ID (so all runs of one program stay on the same side of the train/test split), reporting F1, Macro-F1, balanced accuracy, MCC, and AUROC/AUPRC, with 95% bootstrap CIs over 1,000 resamples, against trivial always-faulty and stratified-random baselines. Additional analyses probe per-category diagnosis, cross-architecture transfer (train on two families, test on the third), and single-epoch early detection across k ∈ {1,2,5,10,15,20,25,30,40,50}.

## Results & Highlights

- **Dataset scale**: 14,227 training runs containing 719,560 epoch records (median 50 epochs/run), split as 9,845 faulty + 4,382 correct baselines, from 59 unique Stack Overflow programs. On Deep4ge, RNN dominates run count (7,752 runs, 5,253 faulty) versus FNN (3,792/2,555) and CNN (2,683/2,037).
- **Operator validation**: the 27-operator set achieves mutation scores of 0.85 (FNN), 0.87 (CNN), and 0.83 (RNN) under the statistical killing criterion (Cohen's d ≥ 0.5, p < 0.05, n=15).
- **Trajectory beats final-epoch on detection**: on the Deep4ge detection task, Random Forest MCC rises from 0.150 (final-epoch) to 0.227 (trajectory) with non-overlapping 95% bootstrap CIs ([0.132, 0.169] vs [0.209, 0.245]); AUROC rises 0.649→0.716 and AUPRC 0.804→0.842.
- **Trajectory beats final-epoch on diagnosis**: for 7-class diagnosis, Random Forest Macro-F1 rises 0.355→0.381 and MCC rises 0.346→0.401 when moving from final-epoch to trajectory features.
- **F1 is misleading under class imbalance**: with 9,845/14,227 runs faulty, the trivial always-faulty classifier already scores F1 = 0.818 on detection, and RF's final-epoch F1 of 0.816 is essentially tied with it — the authors correctly rely on MCC and balanced accuracy, where RF only modestly exceeds the trivial baseline.
- **Per-category diagnosis F1**: Weight 0.700, Hyperparameter 0.523, Loss 0.520, Regularization 0.462, Layer 0.384, Optimization 0.075, and Activation 0.000 — the confusion matrix shows Activation faults are mostly misclassified as Optimization, and Optimization faults frequently confused with Layer.
- **Cross-architecture transfer**: in-distribution detection MCC = 0.227; cross-architecture models trained on two families and tested on the held-out third achieve MCC 0.210 (FNN), 0.218 (CNN), 0.198 (RNN), showing transfer is largely preserved but slightly weaker for RNN.
- **Early detection is hard**: single-epoch RF detection MCC ranges only 0.14–0.19 across all tested epochs k, with no single epoch dominating and all underperforming the full-trajectory model (0.227).

## Authors & Credibility

Sole author Sigma Jahan (Dalhousie University, Canada), with the dataset linked to prior DEFault work (ICSE 2025, co-authored with Shah, Mahbub, Rahman) that already used this dataset to evaluate detection/diagnosis. The work builds directly on the well-established DeepCrime mutation framework (ISSTA 2021) and cites the canonical DNN debugging tool line (AutoTrainer, UMLAUT, DeepLocalize, DeepDiagnosis, DeepFD). The artifact is publicly archived on Zenodo with a citable DOI (10.5281/zenodo.20337241), CC BY 4.0 for data and MIT for the framework, plus a GitHub repo and pinned version v1.0.0 (commit 3f30d85e4f81), giving strong reproducibility provenance. The single-author, short-paper (5-page) format limits the depth of independent validation.

## Critical Assessment

- **Novelty**: genuinely useful gap-fill — it is, by the author's own claim, the first public dataset linking adapted SO DNN programs, mutation-derived categories, and per-epoch trajectories in one reusable benchmark. The novelty is in the curation + release rather than a new algorithm; the AST-based mutation approach and statistical killing criterion are inherited from DeepCrime.
- **Rigor**: evaluation methodology is sound and honestly reported — group-aware CV prevents same-program leakage, bootstrap CIs are given, and the authors openly expose that headline F1 (~0.82) is inflated by the 69% faulty base rate and that the trivial always-faulty classifier essentially matches RF on final-epoch F1. The choice to emphasize MCC over F1 is correct and refreshingly transparent.
- **Reproducibility**: high. Public Zenodo + GitHub, pinned commit, documented schema with a data dictionary, validation scripts, and a construction loop (Algorithm 1) that lets others regenerate the dataset. Seeds and dataset splits are fixed. Hardware-dependent features (GPU memory, CPU%) are flagged as environment-sensitive.
- **Significance**: primarily infrastructure. The benchmark baselines are deliberately weak (untuned LogReg/RF), and even the best trajectory MCC of 0.227 is low in absolute terms — the contribution is the substrate that lets future methods be compared, not a new detection algorithm.
- **Red flags / overclaiming**: (1) Only 5 Activation-category faulty logs (312 total, with 0 in CNN) and only 254 Regularization logs — the per-class F1 of 0.000 for Activation is partly an artifact of tiny, architecture-skewed sample sizes rather than purely intrinsic difficulty. (2) Baselines were validated only for *execution* and *training*, not *semantic correctness*, so some "correct" baselines may themselves be subtly buggy. (3) Mutations are derived from real-fault taxonomies but are still synthetic; the paper explicitly disclaims generalization to real industrial faults, which limits external validity. (4) Diagnosis is restricted to single-label, so co-occurring multi-fault scenarios are not evaluated. (5) Scope excludes Transformers, which are the dominant architecture family in practice — a significant coverage gap for 2026.

## Strengths

- Clean, reproducible, well-documented artifact with explicit schema, data dictionary, validation scripts, and pinned versioning on Zenodo + GitHub.
- Methodologically honest evaluation: group-aware CV, bootstrap CIs, trivial baselines included, and an explicit switch from F1 to MCC/balanced accuracy once imbalance is acknowledged.
- Rich per-epoch feature set (26 dynamic features covering weights, gradients, activations, trends, and hardware) that goes well beyond loss/accuracy curves.
- Useful negative-style findings: early single-epoch detection is weak, Activation/Optimization faults are confusable, and final-epoch features are insufficient — these are actionable pointers for future work.
- Extensible design: new operators, programs, or features can be added and metadata regenerated.

## Limitations

- Restricted to TensorFlow/Keras FNN/CNN/RNN; **Transformers are out of scope**, leaving the most relevant modern architecture family uncovered.
- Programs come from Stack Overflow and may not reflect industrial code; mutation-derived faults may not represent naturally occurring faults, so results do not establish generalization to real-fault distributions.
- Correct baselines were validated for execution/training, not semantic correctness.
- Severe class imbalance across fault categories (Activation: 312 runs, 0 in CNN; Regularization: 254 runs) makes per-category diagnosis numbers unreliable for the rare categories.
- Diagnosis is single-label only; multi-fault scenarios are not modeled.
- Hardware features (GPU memory, CPU%) are environment-dependent and may not reproduce identically on other machines.
- Baselines are deliberately minimal (untuned LogReg/RF); absolute performance (MCC ≈ 0.23 detection, ≈ 0.40 diagnosis) is modest, so the paper is best read as a benchmarking substrate rather than a strong empirical result.

## Relevance

对关注深度学习系统测试与软件工程、训练时故障诊断以及训练动力学分析的研究者，Deep4ge 提供了一个高质量、可复用的公开基准与故障注入框架，可直接用于评估和对比新的故障检测/诊断方法、训练轨迹特征工程及跨架构迁移研究；但其仅覆盖 FNN/CNN/RNN 而排除 Transformer，对以大模型为主的研究方向参考价值有限。

## Questions for Further Exploration

- How would modern trajectory-aware models (RNN/Transformer sequence classifiers over per-epoch feature series, or contrastive representation learning) compare against the untuned RF baseline, and can they push detection MCC substantially above 0.227?
- Why are Activation and Optimization faults so confusable in the training dynamics — are the logged 26 features simply insufficient to separate them, and would additional features (e.g., per-layer gradient cosine similarity, activation sparsity over time) recover discriminability?
- Can the benchmark be extended to Transformers (attention-head-level mutations, learning-rate-schedule faults, positional-encoding faults) while preserving the statistical killing criterion, and would the cross-architecture transfer finding (RNN slightly weaker) still hold?
- Does adding a multi-label diagnosis formulation (multiple operators applied per mutant) change the relative difficulty ranking of fault categories?
- How sensitive are the hardware features (CPU%, peak GPU memory, system memory) to platform changes, and do they meaningfully contribute to detection once environment is controlled?
- Are the "correct" baselines semantically correct, and would a stricter semantic-validation pass change the detection MCC (since some labeled-correct runs may harbor silent bugs)?
