The policy LLM rewrites its own training environment after every GRPO round using failure-aware context — beating larger proprietary models on multi-agent path-finding.

From Trainee to Trainer: LLM-Designed Training Environment for RL with Multi-Agent Reasoning

Chao Chen, Chengzu Li, Zhiwei Li, Yinhong Liu, Zhijiang Guo 2026-06-16
reinforcement-learning large-language-models curriculum-design multi-agent-path-finding environment-generation GRPO
Key Metric
51.67 / 33.14 / 18.67
Valid rate of Qwen3-4B + GRPO + V6 on 3/4/5-agent MAPF generalization sets — surpasses Kimi-K2.5 (46.17/26.95/18.95), GPT-5.4, Grok-4.2, Gemini-3.1-Pro as environment engineers.
Why This Matters
The RL-trained checkpoint is a better environment engineer than its own base model.
Policy learning sharpens self-diagnosis, not just capability — reframes self-improvement loops as meta-cognitive refinement, not mere skill acquisition.
Critical Edge
All headline numbers come from a single custom testbed with undisclosed proprietary prompts and single-seed reporting.
MAPF-FrozenLake is purpose-built; transfer to ALFWorld / WebShop / WebArena is unverified. Only Qwen3-4B tested across the whole study.

Methodology 方法

Testbed — MAPF-FrozenLake. A controllable multi-agent extension of FrozenLake built on Conflict-Based Search. Agents must reach goals without collisions or holes; the wait action resolves conflicts. The generator exposes a configuration C = {(rs, hs, ws)} over eight grid sizes (3×3 → 10×10), with data ratio (training share), hole ratio (obstacle density), and wait ratio (instances requiring a wait).

Training. Backbone Qwen3-4B, algorithm GRPO. Per update: 128 prompts × 8 trajectories = 1024 samples on 4×H100 80GB; AdamW, lr=2e-6, KL β=1e-3 in the actor, no critic, reference policy = start-of-round policy.

Reward. R = wacc·Racc + wlen·Rlen. Accuracy is gated by 8 strict validity checks, then shaped by cost gap to ground truth. (wacc, wlen) shifts (0.5,0.5) → (0.8,0.2) as the short-response EMA rises.

Loop. Dn ← G(Cn) → Mn+1 ← GRPO(Mn, Dn) → validate → compose context ctxn from selected modules → Cn+1 ← Mn+1(ctxn).

Context modules. F (failure breakdown) · G (guideline) · H (history) · S (model summary) · T (training details). Variants V1–V6 add modules in sequence; V6 drops the round-0 default and adds self-summary + bookkeeping-only training info.

Results & Highlights 结果

Valid rate (%) on 3/4/5-agent held-out sets. Higher is better.

Method (Engineer) 3-agent 4-agent 5-agent
Base Qwen3-4B (no RL) 14.83 3.43 ≈0
Fixed curriculum (random-curr.) 40.42 26.67 14.00
GPT-5.4 as engineer 32.50 17.05 9.11
Grok-4.2 as engineer 33.42 23.43 12.89
Gemini-3.1-Pro as engineer 24.50 12.95 4.78
Kimi-K2.5 as engineer 46.17 26.95 18.95
RL checkpoint + V6 (ours) 51.67 33.14 18.67

Counter-intuitive ablations. Bookkeeping-only training context (51.67/33.14/18.67) beats full RL details (38.83/24.76/14.22) — extra optimization info distracts the engineer from the failure evidence.

Behavioral axes. V6 wins via signal salience, edit granularity, feature-based templates, cross-round self-correction, and task-grounded modeling — replacing monotone "bigger map = more holes" with non-monotone, evidence-driven schedules.

Strengths & Limitations 优缺点

Strengths

  • Clean reframing: the same LLM is both trainee and trainer, but the engineer role is restricted to a low-dimensional generator interface — no reward hacking.
  • Full release: code, model weights (LARK-Lab/Trainee2Trainer), and project website — well above the arXiv-only bar.
  • Mechanism-level analysis: the 5-axis behavioral decomposition explains why V6 wins, with falsifiable hypotheses for the community.
  • Two counter-intuitive findings reported honestly: bookkeeping > full RL context; checkpoint > base as engineer.
  • "Edit granularity" insight (full rewrites degrade healthy sizes) is broadly useful for any closed-loop prompt design.

Limitations

  • Single backbone family (Qwen3-4B); the headline claim is unverified on Llama / GPT / Claude-class models.
  • Single custom testbed; transfer to ALFWorld, WebShop, WebArena, ScienceWorld is not shown.
  • No standard deviation or seed-variance reporting in the main tables — single-run numbers.
  • Proprietary-LLM baselines lack version, prompt, and sample-count disclosure — a weaker prompt would inflate the Qwen3-4B win.
  • Framework requires a clean validity oracle (the 8 hand-coded checks) — not available for all RL tasks.
  • Short redesign horizon (3 rounds); long-horizon stability of the engineer is unknown.
  • Training is 2-agent only; 5-agent evaluation collapses to 18.67 — performance is dominated by simple agent counts.
  • Engineering restricted to discrete generator parameters — no continuous relaxation, noise injection, or lookahead.

Questions for Further Exploration 延伸问题

  • Does the framework transfer to standard agentic-RL benchmarks (ALFWorld, WebShop, WebArena, ScienceWorld) where the engineer must operate over high-dimensional, partially-observable environments?
  • Does the "checkpoint > base as engineer" phenomenon persist across model families (Llama, GPT, Claude) and scales (1B / 8B / 70B), or is it specific to Qwen3-4B?
  • What is the long-horizon behavior of the engineer (10+ rounds)? Stable curriculum, oscillation, or collapse to a degenerate schedule?
  • Can the engineer be made robust to noisy / hallucinated structured context — e.g., when the failure summary is itself LLM-generated?
  • Does the same framework apply to reward / loss design or to data-mixing ratios in pre-training curricula, where the engineer interface is also parameterized?
  • How does the engineer behave under a budget constraint (max K configuration edits per round)? Does it learn to amortize exploration?
  • Is "bookkeeping > full RL context" a general prompt-engineering principle (less is more, when the LLM has good priors) or a quirk of this task?
  • Could the engineer benefit from retrieval over past rounds (long-context summary tree) instead of the current fixed-length summary?
  • How sensitive is final performance to the order in which the 5 modules are added during prompt construction? Is V6 (bookkeeping-last) optimal?
