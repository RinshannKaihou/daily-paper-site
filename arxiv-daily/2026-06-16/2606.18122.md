> **一句话总结：** 这篇单作者 IEEE 综述以"系统工程"视角贯穿微控制器级嵌入式机器学习的完整流水线,以惯性动作识别(2 s、62.5 Hz 三轴加速度 → 33 维 RMS/PSD 特征)与音频关键词唤醒(抗混叠 → MFCC → 1D CNN)两条链路为统一示例,系统总结数据策展、特征压缩、类别不平衡评估、模型/运行时协同设计、量化、阈值化、调度与现场监测等工程要点,并提出 8 条实用设计规则与若干开放研究方向。

> **TL;DR:** A single-author systems-oriented synthesis of the end-to-end embedded-ML workflow for microcontrollers, anchored on two reference pipelines — inertial gesture recognition (3-axis accelerometer at 62.5 Hz, 2-s window → 33 RMS/PSD features) and small-footprint keyword spotting (anti-aliased audio → MFCC → 1-D CNN). The paper consolidates engineering rules across data curation, feature compression, imbalance-aware evaluation, model/runtime co-design, quantization, thresholding, scheduling, and field monitoring, and distills 8 practical design rules plus open research directions (continual learning, OOD/uncertainty, verification, application-specific acceleration, better scientific reporting).

## 核心贡献

- 提出面向 MCU 平台的端到端嵌入式 ML 流水线框架,显式覆盖数据采集/窗口化 → 特征提取 → 训练 → 评估 → 运行时 → 现场监测的闭环
- 用两条代表性信号链路统一示例:(1) 惯性动作识别 2 s、62.5 Hz 三轴加速度 → RMS + PSD 特征 → 小型分类器;(2) 关键词唤醒 音频 → 抗混叠 → MFCC → 1D CNN + Softmax
- 明确提出"特征提取即嵌入式压缩"思想:把 375 个原始样本压缩到 33 维特征,降低 MAC 与内存需求
- 强调模型输出是 *概率* 而非 *事实*,主张在模型外层用阈值、去抖、迟滞、状态机、看门狗等确定性保护机制封装
- 总结 8 条从实验室模型到可靠现场推理的实用设计规则
- 提炼若干开放研究方向:持续学习、OoD/不确定性、形式化验证、应用特定加速、规范化实验报告

## Key Contributions

- An end-to-end embedded-ML lifecycle framing (Fig. 1) that closes the loop from sensing, preprocessing, feature extraction, training, deployment, and field monitoring back into data collection
- Two fully worked example pipelines used throughout the paper:
  - **Inertial motion recognition**: 2-s, 62.5 Hz, 3-axis accelerometer → 125 samples/axis (375 raw values) → 33-dimensional feature vector (per-axis RMS + 3 peak amplitudes + 3 peak frequencies + 4 spectral-band powers) → small dense classifier with Softmax over left-right, up-down, circle, idle
  - **Audio keyword spotting**: microphone → anti-aliased ADC at kHz rate → sliding MFCC matrix → 1-D convolution + max-pooling blocks → Softmax over noise / unknown / target keywords
- A "feature extraction as embedded compression" argument: explicitly computes the per-window raw buffer (N_raw = f_s · T · C) and the resulting feature dimensionality to motivate handcrafted features over end-to-end models on MCU budgets
- An explicit model/runtime co-design stance: TFLite Micro / CMSIS-NN as implementation vehicles of one architectural pattern; deterministic acquisition (timer interrupts, RTOS, DMA) paired with bounded/static buffers and bounded inference latency
- Eight practical design rules for moving from an offline trained model to a robust on-device inference system (see Methodology)
- A consolidated set of evaluation metrics (Table II) and runtime considerations (Table III) relevant to embedded deployment
- Open research directions: continual learning under memory constraints, uncertainty calibration and OOD detection, runtime verification of probabilistic models, application-specific acceleration that goes beyond MAC throughput, and stricter scientific reporting standards for embedded ML papers

## Methodology

This is a **synthesis / tutorial / position paper** rather than a methods paper. The author structures the argument as a vertical walk through the embedded-ML lifecycle, with each section tied back to one of the two case studies. Concretely:

1. **Target modeling.** Distinguishes single-board computers (OS, CLI/GUI, more memory) from microcontrollers (bare-metal/RTOS, tens-to-hundreds of KB RAM, deterministic control loops, always-on sensing). Embeds a useful sizing formula: N_raw = f_s · T · C. Worked example: 62.5 Hz × 2 s × 3 axes = 375 raw values, occupying 750 bytes at 16-bit precision — *before* double buffers, features, weights, activations, or arena.
2. **Data curation & windowing.** Argues that data collection is itself a design decision, not a prerequisite. Sample matching must hold across sample rate, bit depth, axis orientation, mounting, enclosure, etc. Stresses *subject/session/environment-aware* splits over random splits for edge generalization. Calls out the need for *negative and unknown* classes (background noise, idle, OOD) since naïve accuracy can mask a majority-class predictor.
3. **Feature extraction as embedded compression.** RMS + PSD descriptors for the inertial case (375 → 33 dims, >10× reduction preserving amplitude and frequency information). Anti-aliasing + STFT/spectrogram + mel filterbank + log compression + DCT for MFCC, then 1-D CNN for keyword spotting (weight sharing exploits local temporal structure; ref. Sainath & Parada 2015).
4. **Model architecture and training.** Softmax outputs treated as *scores*, not probabilities. Class-specific thresholds (e.g., P(left-right) > 0.5 or P(circle) > 0.8) are *product parameters*. Under/overfitting diagnostics tied to loss + accuracy curves; learning-rate diagnostics. The accompanying Table I summarizes the model configurations of the two example pipelines.
5. **Evaluation metrics for deployment.** Confusion matrix is treated as a deployment artifact. Per-class precision, recall, F1, FPR, FNR reviewed before threshold selection. Macro-averaged F1 is preferred over total accuracy under imbalanced priors. Temporal behavior (sliding-window flicker, partial-word triggers, transition windows) is part of evaluation, not an afterthought. Table II catalogs evaluation metrics.
6. **Embedded runtime and deployment.** TFLite Micro (FlatBuffer + C/C++ runtime) and CMSIS-NN as the implementation spine. Static buffers preferred over dynamic allocation. Sampling is paced by timer interrupt / RTOS task / DMA; inference runs on a completed buffer while acquisition fills the next. Profiling must be done on the actual hardware. Decision policy lives *outside* the model (thresholds, debouncing, hysteresis, anomaly state). Quantization (int8) is essentially required for MCU; calibration data and post-quantization validation are required. Table III catalogs runtime/deployment considerations.
7. **Case study patterns.** Two worked case studies reinforce the pipeline: (a) inertial gesture recognition emphasizes orientation mismatch, sensor saturation, gesture-speed variability, and transition windows; (b) keyword spotting emphasizes negative/unknown classes, background augmentation, sample matching, and streaming MFCC updates. Table IV summarizes the deployment patterns that the author derives from them.
8. **Eight practical design rules.** The closing checklist: (1) start with a resource/timing budget, (2) treat the dataset as a product test artifact, (3) validate across realistic sources of variation, (4) design the runtime as a real-time embedded subsystem, (5) version the *complete* deployed pipeline (model + preprocessing + features + thresholds + post-processing), (6) evaluate predictive and deployment metrics together, (7) plan field monitoring before deployment (privacy-preserving aggregate logging), (8) match on-device inference with privacy/reliability safeguards.
9. **Open research directions.** Continual learning under catastrophic-forgetting and field-behavior constraints; uncertainty calibration + OOD detection; verification of threshold logic, rejection behavior, and confidence margins at the system level; application-specific acceleration that supports the *whole* pipeline (filtering, windowing, MFCC, normalization) rather than just MAC throughput; better scientific reporting (device, clock, precision, model size, peak memory, operator set, end-to-end latency, class-wise metrics, sampling conditions).

## Results & Highlights

This paper does **not** report new empirical benchmarks. Its "results" are qualitative:

- A *worked sizing example* showing 62.5 Hz × 2 s × 3 axes = 375 raw values (750 bytes at int16) before double-buffering/features/weights/activations/arena
- A *worked feature-compression example* showing 375 → 33 dimensions for the inertial case (>10× reduction)
- A *consolidated lifecycle* (Fig. 1) and two pipeline diagrams (Fig. 2 inertial, Fig. 3 audio) as pedagogical artifacts
- Four reference tables (I: representative model config; II: evaluation metrics for deployment; III: runtime/deployment considerations; IV: deployment patterns)
- A list of 8 practical design rules and 5+ open research directions
- The author anchors the discussion with a 9-entry reference list that maps the synthesis onto established TinyML literature (Edge Impulse, Warden & Situnayake's *TinyML* book, TFLM (David et al. 2021), CMSIS-NN (Lai et al. 2018), Speech Commands (Warden 2018), Sainath & Parada 2015, MCUNet (Lin et al. 2020), Jacob et al. 2018 quantization, Horowitz 2014 energy)

## Critical Assessment

**Novelty:** Low-to-moderate. The two reference pipelines (inertial → RMS/PSD + dense net; audio → MFCC + 1D CNN) are textbook TinyML. The novelty is in the *lifecycle framing* (closed-loop workflow with explicit field-monitoring feedback) and in pulling deployment-time concerns (sample matching, thresholding, deterministic acquisition, versioning the whole pipeline) to the foreground. This is useful packaging, not a new method.

**Rigor:** Reasonable for a synthesis paper. The N_raw = f_s · T · C sizing example and the 375→33 dimensionality reduction are concrete and reproducible. The 8 design rules and the 4 tables are well-organized and internally consistent. However, the author does not provide ablations showing that, e.g., a feature-engineered model beats an end-to-end 1D CNN at matched MCU budget on a specific dataset, nor does he measure latency/energy/flash on a specific target. The synthesis is therefore *qualitative* in its claims about which choices are "robust."

**Reproducibility:** The paper is fully text-described and the two pipelines are concrete enough that an experienced embedded-ML engineer could re-implement them from the prose. No code, datasets, or trained models are released, which is acceptable for a synthesis paper but means the reader must take the author's word that the recommended pattern is in fact best practice.

**Significance:** High *for the target audience* (engineers new to TinyML, or experienced ML researchers trying to land a model on a Cortex-M). Low for a research audience looking for new algorithms or SOTA numbers. The paper is a useful "north star" reference that an engineering team could adopt as a deployment checklist, but it does not move the empirical frontier.

**Red flags:**
- No new experiments, no new model, no new benchmark — readers expecting a research contribution will be disappointed. The contribution is a *position paper* in disguise and should be marketed as such.
- The 9-entry reference list is short for a synthesis covering this much ground; major works on quantization-aware training (e.g., TQT, LSQ), on-device NAS (MCUNet series), or robust TinyML evaluation (MLPerf-Tiny) are only partially cited.
- The "8 practical design rules" are presented as actionable, but several (e.g., "validate across realistic sources of variation", "treat the dataset as a product test artifact") are aspirational engineering advice rather than falsifiable claims.
- The author repeatedly conflates "softmax outputs" with "probabilities" in a way that the *uncertainty calibration* open-research bullet itself flags as an open problem — readers should be careful not to take softmax scores at face value without calibration.
- The paper is silent on specific MCU targets (Cortex-M0 vs M7, RISC-V, ESP32), so the "MCU-class" claim is broad; deployment numbers will vary widely across this class.

## Strengths

- End-to-end lifecycle framing that ties data, features, evaluation, runtime, and field monitoring into a single closed loop (Fig. 1)
- Two concrete, contrasting pipelines (inertial vs audio) make the abstractions tangible
- Explicit emphasis on *probabilistic model + deterministic safeguards*, which is often missing from ML-for-embedded material
- Useful quantitative anchors: the 375-raw / 33-feature / 750-byte sizing example, the 60/20/20 split discussion, the N_raw = f_s · T · C formula
- Practical design rules that are phrased as actionable engineering heuristics
- Honest articulation of open research problems rather than overclaiming
- Clear, structured tables that an engineering team could adopt as a deployment checklist

## Limitations

- No new empirical contribution; the paper is best read as a tutorial/position paper, not as a research contribution
- No code, datasets, or model releases; reproducibility is by re-implementation
- The 9-entry reference list is sparse; the synthesis could be strengthened by deeper engagement with MCUNet, MLPerf-Tiny, recent QAT methods, and robustness/OOD literature
- "MCU-class" is treated monolithically; per-class numerical guidance (e.g., flash/RAM budgets for a Cortex-M4F vs M33) is absent
- The 8 design rules are largely restatements of engineering best practice; they are useful but not falsifiable
- The 1D CNN for keyword spotting is described qualitatively; the paper does not compare it to recent MCU-deployable alternatives (DS-CNN, MicroNet, TC-ResNet)
- Continual learning and on-device personalization are mentioned as open problems but not addressed
- Single-author work — internally consistent but lacks the challenge that a multi-author synthesis would bring

## Relevance

与"边缘/端侧 AI + 模型/系统协同设计"方向高度相关。对需要在 MCU 上落地分类模型(可穿戴、工业传感、HMI、音频/惯性监测)的工程读者,这是一份结构化、可作为部署 checklist 使用的参考;对研究新算法或 SOTA 表现的读者,贡献有限。

## Questions for Further Exploration

- On which specific MCU target (e.g., Cortex-M4F, Cortex-M33, ESP32-S3) is the inertial pipeline profiled, and what is the resulting latency, flash, RAM, and energy per inference?
- How does the recommended 33-dimensional RMS/PSD feature vector compare head-to-head against an end-to-end 1D CNN at matched accuracy on, e.g., the UCI HAR or WISDM datasets, when both are quantized to int8 and deployed on the same MCU?
- Does the keyword-spotting 1D CNN architecture match the structure of modern MCU-friendly KWS backbones (DS-CNN, MicroNet, TC-ResNet), or is it a more generic baseline?
- What is the recommended approach to *uncertainty calibration* of the softmax outputs that the author flags as an open problem, beyond simply setting a global threshold?
- How would the 8 design rules change for *streaming* applications with non-stationary class priors (e.g., keyword spotting in a noisy factory floor over months of deployment)?
- What is the relationship between the "subject/session-aware" validation strategy advocated here and the more formal *cross-domain* evaluation protocols used in MLPerf-Tiny?
- For continual learning on-device, what is the empirical memory/energy/accuracy cost of the approaches currently being explored (e.g., replay buffers, parameter-efficient adapters) on the same class of MCUs?
- Does the paper's "version the complete pipeline" recommendation align with MLOps practice for embedded devices (e.g., TFLite Micro + Edge Impulse + ONNX + TFLM model metadata), and what tooling actually supports that?
