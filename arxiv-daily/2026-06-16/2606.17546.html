Why This Matters

Aligns directly with the long-term reliability of LLM agents — single-score or single-curve evaluation hides regression, overfitting, and instability that the multi-view protocol now surfaces.

Critical Edge

One n=1 seed per (method, batch, source, backend) cell — no error bars, no statistical test; the dramatic HLE-only collapse to 0.0 at the final snapshot is a single-run observation that could be seed-sensitive.

Methodology

MDP over a mutable harness Ht

A snapshot decomposes as At = (M, Ht) — fixed base model M plus mutable harness Ht (prompts, memory, tools, middleware, runtime). The environment samples a task batch Bt ⊂ Dtrain; the method applies its native update Ht+1 = U(Ht, Bt, Tt, Ft). SEAGym only fixes schedule, splits, and views — the policy is each method's own. Validation and test tasks never enter update evidence, so a higher score cannot be explained by leakage.

Six Views

update-evidence, frozen update-validation, ID transfer, OOD transfer, replay/retention, cost — snapshot frozen at every evaluation point.

Schedule Parameters

state persistence, task reuse, batch size, update repeats, assessment timing — exposed as variables to prevent confounding method with exposure pattern.

Metrics

UVG (prev / base), IDG, OODG, FR (forgetting) — domain-level macro averages so larger task groups do not dominate.

Strengths
  • Reframes the unit of evaluation as the harness update, not model weights — a more honest description of how production agents actually change.
  • Six views directly target online-adaptation failure modes: overfitting to recent tasks, forgetting older ones, paying hidden rollout cost for illusory gains.
  • Clean separation of Harbor (task runner) from SEAGym (schedule + views); new baselines need only a thin wrapper.
  • The replay diagnostic (Figures 3–4) turns a single retention number into a per-task fix / churn / forget narrative.
  • Three counter-intuitive results in one paper: non-monotonic batch size, single-source collapse to 0, asymmetric cross-model transfer.
Limitations
  • Only 3 harness-update methods compared; batch-size non-monotonicity shown only for AHE.
  • 2 benchmarks × 3 backends — narrow evidence base for the "complementary signals" claim.
  • No seed variance, no error bars, no statistical test; HLE-only → 0 and epoch-4 replay collapse are single-run.
  • Restricts to model-external harness evolution; parameter-level self-evolution and hybrid systems are future work.
  • "Complementary signals" claim is qualitative — no redundancy analysis across the six views.
  • Cross-model transfer does not control for prompt format, tool-calling convention, or sampling temperature.
Questions for Further Exploration
Q01
Are the six views information-theoretically redundant, or does each contribute unique signal for predicting final harness reliability?
Q02
Does the batch-size non-monotonicity for AHE generalize to ACE and TF-GRPO, or is it specific to saturating update rules?
Q03
Can update-validation serve as an online early-stopping signal for harness updates — vs. checkpoint averaging, ensembling, EWC-style regularization?
Q04
Is the HLE-only → 0 collapse reproducible across seeds, or explained by distribution shift or an unlucky update?
Q05
How much of the cross-model asymmetry is model identity vs. prompt format / tool-calling convention / temperature?
Q06
Can process diagnostics (harness diffs, middleware/runtime regressions) be formalized as a stability/plasticity test, e.g., bounding per-step edit magnitude?
Q07
Does the same multi-view protocol predict downstream reliability under parameter-level self-evolution and hybrid harness + weight updates?
Q08
What is the minimum cost-controlled evaluation schedule that still surfaces regression and OOD failure modes at scale?
