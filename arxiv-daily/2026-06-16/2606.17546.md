> **一句话总结：** SEAGym 把"agent harness"（prompt + memory + tools + middleware + runtime + 模型-工具 loop）作为自演化的对象，提供一个 RL 风格的多视角评测环境（train batch、frozen update-validation、ID/OOD transfer、replay、cost），并在 Terminal-Bench 2.0 + HLE 上对比 ACE、TF-GRPO、AHE 三种更新方法，揭示了"频繁更新可能不提升 held-out 性能"、"有用的中间 snapshot 可能在后续崩溃"以及"源多样性和 rollout 后端对 harness 可靠性有显著影响"三个非平凡结论。

> **TL;DR:** SEAGym reframes evaluation of self-evolving LLM agents away from a single task score or a single sequential improvement curve, and instead exposes the harness update process via six complementary views (training, update-validation, ID transfer, OOD transfer, replay, cost) on Harbor-compatible benchmarks. Instantiated on Terminal-Bench 2.0 and HLE under a shared epoch/batch protocol, it compares three harness-update methods — AHE (harness-editing), ACE (skill/strategy-memory), TF-GRPO (grouped-rollout RL) — and shows that AHE is the only method that lifts validation, ID, and OOD together, that batch size is non-monotonic (batch 20 is the sweet spot), that a single-benchmark source can collapse to 0 at the final snapshot despite a useful intermediate, and that evolved harnesses transfer asymmetrically across rollout backends (DeepSeek / GLM / GPT-5.4).

## 核心贡献

- 提出 **agent harness** 概念：把 prompt、context、memory、tools、orchestration、middleware、runtime、verifier 等围绕 base model 的"结构化执行层"作为一个整体化、可持久化、可演化的对象，并把评测焦点从单任务分数转向 harness 更新本身。
- 形式化一个 **MDP-style 演化环境** $M=(S,A,P,R,\rho)$，其中 state $s_t$ 包含当前 agent snapshot、schedule 位置与可用任务；snapshot 拆分为固定的 $M$（base model + 不可变 runtime）和可变的 $H_t$（harness state）；更新规则 $H_{t+1}=U(H_t, B_t, T_t, F_t)$ 由各方法自行提供，SEAGym 只规定 task distribution、feedback、schedule 与 evaluation views。
- 设计**六视图评测协议**：update evidence（$B_t \subset D_{\text{train}}$）、process assessment（frozen $V_{\text{update-val}} \subset D_{\text{val}}$）、ID transfer（$V_{\text{ID}} \subset D_{\text{test}}$）、OOD transfer（$V_{\text{OOD}} \subset D_{\text{test}}$）、replay/retention（$V_{\text{replay}}$）以及 token/tool/wall-clock cost 记录；快照在 evaluation point 上被冻结保存，验证任务从不进入 update evidence，以避免"把 validation 当训练样本"造成的伪增益。
- 在统一 epoch/batch 协议下对比 **AHE、ACE、TF-GRPO** 三种 harness 更新方法，并就 batch size、源多样性、cross-model transfer 做消融，给出多视角下可解释的失败模式（snapshot 崩溃、source 单一时最终全 0、跨后端不对称退化等）。
- 工程上把 **Harbor 框架**作为 task runner，把 SEAGym 作为 schedule/evaluation 层，二者通过 rollout/update wrapper 解耦：基准作者按 Harbor 模板接入，自演化方法只需实现一个把 trajectory batch 翻译成原生 update 输入的薄壳。

## Key Contributions

- **Agent-harness abstraction.** Treats prompts, memory, tools, orchestration, middleware, runtime, and the model–tool interaction loop as one persistent, evolvable substrate $H_t$, alongside an immutable base model $M$. This is a more honest description of how production agent systems change over time than "the model got better."
- **Multi-view evaluation environment.** Converts Harbor-compatible static benchmarks into dynamic self-evolution task sources with explicit schedule parameters (state persistence, task reuse, batch size, update timing), and assesses every saved snapshot on update-validation, ID transfer, OOD transfer, replay, and cost — never just the final score.
- **Process diagnostics.** Beyond verifier scores, SEAGym saves trajectory references, public feedback, update summaries, harness diffs (when available), and per-snapshot metric records, so one can decompose regression vs. recovery vs. forgetting at the task level (e.g., "after epoch 4 the replay rate drops from ~50% to 6/80 due to a middleware/runtime contract regression, then the final epoch restores it").
- **Empirical comparison of three update mechanisms.** AHE (harness editing) — the only method that improves validation, ID, and OOD together, but with the largest reliability surface; ACE (skill-/strategy-memory) — small but robust gains, less leverage over runtime failures; TF-GRPO (grouped-rollout RL) — strong source-validation gain, OOD regression, and highest rollout cost.
- **Non-trivial ablations.** Batch size is non-monotonic for AHE (10 → −14.3 val, 20 → +17.1 val, 40 → +2.9, 80 → −17.1); HLE-only training reaches a useful intermediate snapshot but **collapses to 0 on validation/ID/OOD at the final snapshot** (−22.0 / −21.2), while mixed-source training both passes through a bad intermediate and recovers; evolved harnesses transfer asymmetrically across rollout backends (DeepSeek / GLM / GPT-5.4).

## Methodology

**Problem formulation.** A self-evolution run is modeled as an MDP $M=(S,A,P,R,\rho)$. State $s_t$ contains the current agent snapshot, the schedule position, and the available task context. A snapshot decomposes as $A_t = (M, H_t)$, where $M$ is the fixed base model + immutable runtime and $H_t$ is the mutable harness state (prompts, memories, skills, experience libraries, tools, middleware, project files, runtime config, etc.). At step $t$, the environment samples a task batch $B_t \subset D_{\text{train}}$; the agent executes tasks, producing trajectories $T_t$ and feedback $F_t$ according to the benchmark's visibility policy; the method then applies its native update rule $H_{t+1} = U(H_t, B_t, T_t, F_t)$. SEAGym specifies the observed environment; the policy and update rule are left to each method.

**Schedule parameters.** State persistence, task reuse, batch size, update repeats, and assessment timing are exposed as experimental variables rather than hard-coded — this prevents confounding "the method" with "the exposure pattern" and lets one ablate whether a gain came from the update rule, from more evidence per update, from more frequent updates, or from re-visiting the same distribution.

**Splits vs. views.** Base splits $(D_{\text{train}}, D_{\text{val}}, D_{\text{test}})$ control task visibility and update evidence; evaluation views are materialized from these splits to assess different aspects of self-evolution. Validation and test tasks are reserved for frozen assessment and never mixed into update evidence, so a higher score cannot be explained by "the agent saw the test task during updating."

**Metrics.** Verifier score $r(A, x) \in [0, 1]$ drives $\mathrm{Perf}(A, D) = \frac{1}{|D|}\sum_{x\in D} r(A, x)$ and $\mathrm{SR}(A, D) = \frac{1}{|D|}\sum_{x\in D} \mathbb{1}[r(A, x) = 1]$. For each evaluation point $E_i$, update-validation gain is reported as $\mathrm{UVG}^{\mathrm{prev}}_i = \mathrm{Perf}(E_i, V) - \mathrm{Perf}(E_{i-1}, V)$ and $\mathrm{UVG}^{\mathrm{base}}_i = \mathrm{Perf}(E_i, V) - \mathrm{Perf}(E_0, V)$. Final ID gain $\mathrm{IDG} = \mathrm{Perf}(A_T, D_I) - \mathrm{Perf}(A_0, D_I)$ and OOD gain $\mathrm{OODG} = \mathrm{Perf}(A_T, D_O) - \mathrm{Perf}(A_0, D_O)$ use the initial and final snapshots; forgetting is $\mathrm{FR} = \max(0, \mathrm{Perf}(A_0, D_R) - \mathrm{Perf}(A_T, D_R))$. Token usage, tool calls, wall-clock time, and (when available) cost reduction are recorded alongside. Main tables use domain-level macro averages so larger task groups do not dominate.

**Integration.** SEAGym reuses **Harbor** (Harbor Framework Team, 2026) as the task runner for task definitions, environments, verifiers, and parallel jobs; SEAGym adds the self-evolution schedule on top. On the method side, rollout and update are decoupled: a rollout component runs tasks under the current harness and returns a trajectory batch; an update component consumes that batch and applies the method's native self-evolution rule. A new baseline is connected by a thin wrapper that converts SEAGym trajectory batches to the method's native update input and saves the resulting harness state.

**Experimental setup (default).** 80 source train tasks (Terminal-Bench 2.0 + HLE text-only Math/Physics), 35 source validation tasks, 55 source test tasks, 80 HLE CS/AI and Engineering OOD transfer tasks. 5 epochs, train batch size 20. Main comparison: ACE, TF-GRPO, AHE under **DeepSeek-V4-Flash**; ablations use AHE for batch size, source diversity, and cross-model transfer (DeepSeek-V4-Flash / GLM / GPT-5.4).

## Results & Highlights

**Main results (Table 2, DeepSeek-V4-Flash, best-validation snapshot, success rates %, gains in percentage points):**

| Agent | V0 | V* | UVG | ID0 | ID* | IDG | OOD0 | OOD* | OODG | Rollout tok/task | Update tok/update |
|------|----|----|-----|-----|-----|-----|------|------|------|------------------|-------------------|
| AHE | 40.0 | 57.1 | **+17.1** | 40.0 | 49.1 | **+9.1** | 22.5 | 28.8 | **+6.3** | 1.46M | 3.91M |
| ACE | 37.1 | 40.0 | +2.9 | 30.9 | 34.5 | +3.6 | 22.5 | 25.0 | +2.5 | 1.93M | – |
| TF-GRPO | 31.4 | 48.6 | +17.1 | 30.9 | 34.5 | +3.6 | 26.3 | 23.8 | **−2.5** | 2.33M | 1.60M |

AHE is the only method that simultaneously improves validation (+17.1), ID (+9.1), and OOD (+6.3) — and it does so at the lowest rollout cost (1.46M tok/task). TF-GRPO matches AHE on validation (+17.1) but its OOD gain is negative (−2.5) and its rollout cost is the highest (2.33M tok/task). ACE is the most conservative: small positive gains across the board and a meaningful saving on update tokens (none reported, vs. 3.91M for AHE and 1.60M for TF-GRPO).

**Training fix and forgetting (AHE, Figures 3–4).** Replaying the 80 train tasks at each snapshot, the final AHE agent solves 43/80 vs. 34/80 initially (net +9). But the trajectory is non-monotonic: after epoch 4, replay performance collapses to 6/80 with many rollout errors, driven by an evolved message-construction regression in the middleware/runtime contract; the final epoch restores that path and most recovered tasks come back in one step. Relative to $A_0$, the final agent fixes 13 initially-failed tasks and forgets 4 initially-solved tasks. This is the headline example of why "final-vs-initial" tables hide damaging intermediate regression.

**Batch size (AHE, Table 3).** Non-monotonic: batch 10 → −14.3 val / −14.5 ID, batch 20 → **+17.1 val / +9.1 ID** (the only positive), batch 40 → +2.9 / +1.8, batch 80 → −17.1 / −16.4. Update token cost is roughly 3–4M regardless of batch size, so larger batches do not give the evolving agent proportionally more analysis capacity per update; at batch 80, one update must inspect too many trajectories and the per-task attention dilutes, while at batch 10 each update sees too little evidence and the high update frequency (39/40 budget consumed) accumulates runtime regressions.

**Source diversity (Tables 4–5).** Mixed Terminal-Bench + HLE: $V_T=57.1$, $ID_T=49.1$ (+9.1), $OOD_T=28.8$ (+6.3). HLE-only at the **final** snapshot collapses to 0.0 on validation, ID, and OOD (−22.0 / −21.2), but reaches a useful **intermediate** snapshot at epoch 3 with $ID^*=47.3$ (+7.3) and $OOD^*=25.0$ (+3.8). Interpretation: a single benchmark can drive the harness toward benchmark-specific local optima; mixed sources still pass through a bad intermediate state, but their advantage is recovery — Terminal-Bench exposes tool/environment failures while HLE exposes reasoning failures, giving later updates more varied evidence to restore a broken harness.

**Cross-model transfer (Figure 5, AHE best-validation snapshot, gains in pp).** Same-backend ID gains are all positive (DeepSeek +9.1, GLM +7.3, GPT-5.4 +5.5) — AHE can adapt the harness to the rollout backend that produced the updates. Cross-backend transfers are asymmetric: DeepSeek-evolved → GLM +7.3 ID / 0.0 OOD, but → GPT-5.4 −3.6 ID / −8.8 OOD; GLM-evolved → GPT-5.4 −7.3 ID; GPT-5.4-evolved → GLM +5.5 ID / +3.8 OOD. **ID gain does not imply OOD gain** — GPT-5.4-evolved AHE improves GPT-5.4 ID by +5.5 but drops GPT-5.4 OOD by −7.5. Update-artifact analysis: DeepSeek trajectories drive AHE to edit verification, tool-recovery, artifact-cleanup, and message-contract paths; GLM trajectories emphasize text-only reasoning and research; GPT-5.4 trajectories emphasize artifact constraints and validation sufficiency — so transfer succeeds when the evaluation trajectory exposes similar failures and weakens when the edited subsystem no longer matches the dominant failure surface.

## Critical Assessment

**Novelty.** The conceptual move — evaluating self-evolving agents on the *update process* rather than the *final score*, and decoupling "what is being optimized" (harness vs. memory vs. RL policy) from "what task the agent ends up solving" — is timely and well-motivated, but is not entirely new in spirit: continual-learning evaluation has long separated adaptation, transfer, retention, and replay (Parisi et al., 2019; Lopez-Paz & Ranzato, 2017; Chaudhry et al., 2019; van de Ven & Tolias, 2019). The novelty here is the specific instantiation on LLM agents with a multi-view snapshot protocol and the explicit framing of the **harness** as the unit of evolution, which is genuinely useful and under-supplied in the current literature. The cross-model and source-diversity ablations that go beyond "final ID number" are the strongest contribution.

**Rigor.**
- *Strengths.* The protocol is well-specified: frozen update-validation, no leakage of val/test into update evidence, snapshot saves, replay diagnostics, cost recorded. The same-backend/cross-backend contrast (Figure 5) is the cleanest demonstration in the paper that "ID gain" alone is misleading.
- *Weaknesses.* The numbers come from a **single primary configuration per cell** (one seed per (method, batch size, source, backend)); no error bars, no seed-variance discussion, and no statistical test. The "5 epochs, batch 20" default is a single point in a schedule space, and the batch-size sweep uses AHE only — so the non-monotonicity claim is not shown to hold for ACE or TF-GRPO. The OOD transfer set is restricted to HLE CS/AI + Engineering, all of which is text-only; OOD across modalities (e.g., web, desktop) is not tested.

**Reproducibility.** The paper releases methodology and references Harbor; the rollout/update wrapper is described as a "thin wrapper," but the paper does not (in the main text) release full code, prompts, or per-task trajectories. The cited "Appendix Tables 6 and 7 / Appendix Figures 6–14" are not in the excerpt read here, so claim-level reproducibility depends on those appendices and the eventual code release. Update-token costs are reported as "3–4M regardless of batch size" — this is an important detail to verify, because it underpins the "more evidence per update is not the same as a better update" argument.

**Significance.** The headline message — that single-score or single-curve evaluation of self-evolving agents hides regression, overfitting, and instability, and that a multi-view protocol reveals them — is correct, important, and likely to be cited as a baseline. SEAGym is positioned as the first such environment for harness-level self-evolution, and the cross-model asymmetry in Figure 5 is a concrete warning against over-claiming cross-backend transfer in this line of work.

**Red flags / things to be skeptical of.**
- The TF-GRPO OOD regression (−2.5) is reported but the paper does not show whether AHE's OOD gain of +6.3 holds under the same source diversity conditions that made HLE-only collapse to 0; "AHE is the only method that wins on all three views" is conditional on the mixed-source / 5-epoch / batch-20 setting used here.
- The HLE-only collapse to 0 is striking and is presented as evidence that diversity helps — but with $n=1$ run and no error bars, the alternative hypothesis ("HLE-only just hit a bad update that mixed-source happened to avoid") cannot be ruled out. The fact that the mixed-source run also passes through a "bad intermediate state" is acknowledged, which is honest, but the recovery mechanism is described qualitatively (Terminal-Bench exposes tool failures, HLE exposes reasoning failures) rather than quantitatively.
- The cross-model results conflate "rollout backend" with "model identity" (DeepSeek-V4-Flash, GLM, GPT-5.4) and do not control for prompt format, temperature, or tool-calling convention — all of which are known to dominate cross-model transfer in practice. The asymmetry is consistent with the paper's claim, but not uniquely so.
- "Complementary signals" is asserted rather than measured: no information-theoretic or correlational analysis showing that, e.g., replay score is not redundant with OOD score for predicting final harness reliability.

## Strengths

- Reframes the unit of evaluation as the **harness update** rather than the model weights, which is a more honest description of how production agent systems actually change over time.
- Six evaluation views (training, update-validation, ID, OOD, replay, cost) directly address known failure modes of online adaptation: overfitting to recent tasks, forgetting older ones, paying hidden rollout cost for illusory gains.
- Clean architectural separation between Harbor (task execution) and SEAGym (schedule + views), with rollout/update decoupled on the method side — a new baseline only needs a thin wrapper.
- The "training fix and forgetting" replay diagnostic (Figures 3–4) is a particularly good idea: it turns a single retention number into a per-task churn / fix / forget narrative that explains *why* the final number looks the way it does.
- Empirically exposes three counter-intuitive results in one paper: non-monotonic batch size for AHE, single-source collapse to 0 at the final snapshot, and cross-model asymmetric transfer where ID gain does not imply OOD gain.

## Limitations

- Only three harness-update methods compared (ACE, TF-GRPO, AHE); the conclusions about "self-evolving agents in general" are bounded by this small set, and the batch-size non-monotonicity is shown only for AHE.
- Two benchmark suites (Terminal-Bench 2.0, HLE) and three rollout backends (DeepSeek-V4-Flash, GLM, GPT-5.4) — a narrow evidence base for the "complementary signals" claim and the cross-model transfer claim.
- No seed variance, no error bars, no statistical test; the dramatic "HLE-only → 0" and "epoch-4 replay collapse" results are single-run observations and could in principle be seed-sensitive.
- Restricts to model-external harness evolution (prompts, memory, skills, middleware); parameter-level self-evolution, online RL fine-tuning, and hybrid systems are explicitly left to future work, so the environment does not yet arbitrate the harness-vs-weights debate it implicitly raises.
- The "complementary signals" claim is qualitative; no analysis of how redundant the six views actually are, or which view is most predictive of final harness reliability.
- Cost/coverage tradeoff is acknowledged but not addressed: snapshotting and re-evaluating across six views for many epochs is expensive, and adaptive / budget-aware evaluation is left to future work.
- Cross-model transfer analysis does not control for prompt format, tool-calling convention, or sampling temperature — all known confounders of cross-backend agent transfer.

## Relevance

与"自演化智能体 / 持续学习 / agent 自我改进"方向高度相关——把 harness 而非模型权重作为演化与评测对象，与本项目对 LLM agent 长期可靠性、continual adaptation 与 evaluation rigor 的关注完全对齐，对如何严谨地报告"会自我改进的 agent"实验具有直接借鉴价值。

## Questions for Further Exploration

- Are the six views (training, update-validation, ID, OOD, replay, cost) information-theoretically redundant, or does each contribute unique signal for predicting final harness reliability?
- Does the batch-size non-monotonicity for AHE generalize to ACE and TF-GRPO, or is it specific to methods whose updates are saturating (broad harness rewrites) rather than filtering (memory or RL)?
- Can the update-validation view be used as an **online early-stopping signal** for harness updates, and how does that compare to checkpoint averaging, ensembling, or EWC-style regularization on the harness state?
- The HLE-only → 0 collapse is a striking single-run result — is it reproducible across seeds, and is it explained by the absence of tool/environment signal, by distribution shift relative to validation, or by an unlucky update?
- How much of the cross-model asymmetry (Figure 5) is driven by the model identity vs. by **prompt format, tool-calling convention, and sampling temperature** — i.e., would the same model on a different harness contract transfer cleanly?
- Can SEAGym's process diagnostics (harness diffs, middleware/runtime regressions) be formalized as a stability/plasticity test, e.g., bounding per-step edit magnitude on the harness?
- How does the protocol extend to **parameter-level** self-evolution (online RL fine-tuning) and to **hybrid** harness + weight updates — does the same multi-view protocol predict downstream reliability there as well?
- What is the minimum cost-controlled evaluation schedule that still surfaces regression and OOD failure modes, and how does it scale to long-horizon online task streams?
