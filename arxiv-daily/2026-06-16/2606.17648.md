> **一句话总结：** 提出"酿造—消解(Brewing-to-Resolution)"双阶段代码推理生命周期,在线性探测(信息可用性)与上下文剥离解码 CSD(信息就绪度)组成的双诊断框架下,在 6 类任务、16 个模型(Qwen/Llama/DeepSeek 系列,0.5B-14B)上系统发现酿造支架跨架构稳定(归一化时长 24-42%)、而消解成功率随能力变化,且四类失败模式(Resolved/Overprocessed/Misresolved/Unresolved)对应可因果干预的内部计算状态。

> **TL;DR:** The paper models LLM code reasoning as a two-phase "brewing-to-resolution" lifecycle. Using layer-wise linear probing (availability) and Context-Stripped Decoding/CSD (readiness) on 16 decoder-only LLMs (Qwen2.5-Coder/2.5/3, DeepSeek-Coder, CodeLlama, Llama-2; 0.5B-14B) across 6 synthetic code-reasoning families with single-digit outputs (24,300 samples/model), the authors find that the answer becomes linearly recoverable at ~14% depth (FPCL) and self-decodable only at ~50% (FJC) — a 38% "brewing" interval — and then samples diverge into four causally validated outcomes (Resolved 41.5% / Overprocessed 26.4% / Misresolved 8.5% / Unresolved 23.7%). Function Call depth-1→3 collapses Resolved from 61.1% to 2.5%, while the normalized brewing duration remains a near-universal 24-42% across all 16 models. A ground-truth-free Resolution Functional reaches AUC 0.850 for Resolved-vs-Rest.

## 核心贡献

- 提出"酿造—消解"双阶段代码推理生命周期:答案在层间先可被外部线性探针恢复(FPCL,~14% 深度),经约 38% 的酿造区间才被模型自身解码管线可读(FJC,~50%),最后收敛为四类消解结果。
- 设计双诊断框架:层级线性探测(Linear Probing,信息可用性) + 上下文剥离解码(CSD,灵感来自 Patchscopes,信息就绪度),二者构成"答案是否在 / 模型是否用得上"的不对称测度。
- 构造 CUE-Bench:6 类代码推理任务(Value Tracking、Computing、Conditional、Function Call、Loop、Loop-unrolled),答案为单数字 0-9(11 类诊断空间含 d̄ 残差类),每模型 24,300 样本,覆盖 150 个受控配置。
- 实证四类结果 Resolved 41.5% / Overprocessed 26.4% / Misresolved 8.5% / Unresolved 23.7% 普遍存在,准确率指标严重高估可用代码能力;三任务 Resolved 低于 30%。
- 通过 3 类因果干预(FJC 处激活补丁、Overprocessed 层跳过 + α 混合注入、Unresolved 重注入)验证四分类对应真实可干预的计算状态,Overprocessed 伴随 L22-24 Hoyer 稀疏度上升。
- 跨 16 个模型(0.5B-14B,Qwen/Llama/DeepSeek)复现酿造支架,归一化酿造时长稳定在 24-42%,消解成功率随能力/规模/训练变化;并以 CRUXEval-O 外基准印证模型排序。

## Key Contributions

- **Dual diagnostic framework**: layer-wise linear probing (information availability) paired with Context-Stripped Decoding/CSD (information readiness, inspired by Patchscopes), distinguishing "the answer is present in the residual stream" from "the model can decode it from its own representations."
- **Brewing-to-resolution lifecycle**: a measurable ~38% of network depth between FPCL (First Probe-Correct Layer) and FJC (First Joint-Correct Layer) where the answer is linearly readable but not yet self-decodable.
- **Four-way outcome taxonomy**: Resolved / Overprocessed / Misresolved / Unresolved, each carrying substantial mass on every task (overall 41.5% / 26.4% / 8.5% / 23.7%).
- **Causal validation**: activation patching at FJC (20-44% answer-flip vs. 3-18% at pre-brewing layers), layer-skipping with α-blend injection that rescues 47.8% of Overprocessed samples (vs. 10.4% with raw replacement), and FPCL re-injection that recovers 22-38% of Unresolved samples.
- **Empirical regularity vs. capability**: normalized brewing duration falls in a 24-42% band across all 16 decoder-only models, while resolution success varies with capability, scale, and training — the scaffold is stable; resolution is not.
- **Resolution Functional ρ** (closed-form, endpoint statistics only) reaches AUC 0.850 for Resolved-vs-Rest, showing the lifecycle is observable without ground truth.
- **External validation** on CRUXEval-O reproduces the same model ordering (DeepSeek-Coder > CodeLlama > Llama-2, code-specialized ≥ general).

## Methodology

- **Models**: 16 decoder-only Transformers (0.5B-14B) from six families — Qwen2.5-Coder, Qwen2.5, Qwen3, DeepSeek-Coder, CodeLlama, Llama-2 — spanning code-specialized vs. general and three architecture lineages.
- **Benchmark (CUE-Bench)**: 6 task families along the classical axes of program analysis — data flow (Value Tracking, Computing), control flow (Conditional), data+control combined (Function Call, Loop, Loop-unrolled). Each sample has a short code snippet + question suffix and a single-digit answer (0-9), so the target is always one token and the diagnostic space is 11 classes (digits + a near-OOD residual class d̄ to debias the probe). 24,300 samples/model, 4,050/task, 150/config.
- **Linear Probing Φ_P^ℓ(h^ℓ) = σ(W^ℓ h^ℓ + b^ℓ)**: a per-layer logistic probe trained on h^ℓ (last-token residual stream) to test whether the answer is linearly recoverable at depth ℓ. A necessary-condition lens.
- **Context-Stripped Decoding Φ_C^ℓ**: extract h^ℓ from the source run, inject it into a target prompt T = Q (question suffix only, code context stripped) and continue from layer ℓ+1 to L-1. Subtract the clean T-only logit baseline z_b to cancel the language prior; the resulting distribution tests whether the model itself can decode the answer from h^ℓ alone.
- **FPCL / FJC definitions**: FPCL = min layer where probing argmax = gold; FJC = min layer where probing and CSD argmaxes both equal gold. Brewing interval ∆_brew = FJC − FPCL (mean 10.7 layers, 38% of depth, anchor setting).
- **Outcome taxonomy**: Resolved = ∃FJC ∧ output correct; Overprocessed = ∃FJC ∧ output wrong (the "overthinking" of Halawi et al.); Misresolved = ∄FJC ∧ tail-window CSD max ≥ 0.5 (stable wrong); Unresolved = ∄FJC ∧ tail-window CSD max < 0.5 (not finished).
- **Causal interventions**:
  1. *Activation patching at FJC* (Meng et al. 2022 style): patch h^ℓ from S_x into a neutral target prompt at FJC vs. earlier/later layers, measure answer-flip rate.
  2. *Layer skipping + α-blend injection* for Overprocessed: h_target = 0.7·h_orig + 0.3·h_FJC; rescue rate measures how often an Overprocessed sample recovers when the late-layer corruption is "diluted."
  3. *Re-injection from FPCL* for Unresolved: same α-blend formula at the penultimate 2-4 layers.
- **Cross-task controls**: per-task sweeps over call depth (1→3), distractor count (0→2), loop iterations, operator mix, condition type (boolean/numeric/membership), and body type (single/dual variable).
- **Ground-truth-free signals**: layer-wise entropy H(ℓ) and MaxConf C(ℓ) of the CSD 10-class softmax; combined Resolution Functional ρ built from endpoint statistics only.

## Results & Highlights

- **Aggregate outcome (anchor Qwen2.5-Coder-7B, 24,300 samples)**: 41.5% Resolved, 26.4% Overprocessed, 8.5% Misresolved, 23.7% Unresolved (2.7% NO_BREWING excluded). Three of six tasks fall below 30% Resolved (Computing, Function Call, Loop-unrolled).
- **Brewing timing is stable**: FPCL at mean 14% depth, FJC at 50%, ∆_brew ≈ 38%. Across all 16 models normalized ∆_brew/L ∈ [24%, 42%]; small models at the low end (0.5B ≈ 29%) because both probes and CSD are weaker, medium-large models plateau near 33-39%.
- **Depth-induced collapse**: Function Call Resolved plunges 61.1% (depth 1) → 2.5% (depth 3); Unresolved rises 19.1% → 57.4%; FJC existence falls 78% → 40%. Function-call indirection is the single largest bottleneck observed.
- **Coder vs. Base (7B, matched architecture)**: Computing Resolved 26.2% (Coder) vs. 21.5% (Base); code pretraining shifts the outcome mix but leaves the FJC position nearly invariant (mean shift 0.67 layers, 2.4% of depth; Pearson r = 0.901 across 6 tasks).
- **Scaling**: Unresolved drops sharply with size (most "not-finished" samples become resolvable with extra depth). Coder-0.5B: NO_BREWING ≈ 10%, Resolved 18%; Coder-14B: NO_BREWING ≈ 1%, Resolved 50.3%. Brewing duration stays inside the 24-42% band — scaling sharpens an existing mechanism rather than creating a new one.
- **Cross-architecture**: brewing scaffold is robust in CodeLlama-7B, Llama-2-7B, DeepSeek-Coder-6.7B; Computing Resolved for CodeLlama/Llama-2 is only 7.6% / 8.3% (CSD max 17-20% — the bottleneck is readiness, not availability). DeepSeek-Coder-6.7B reaches 21.3% on Computing with the same LlamaForCausalLM backbone, isolating pretraining data quality as the differentiator.
- **Causal validation, quantified**:
  - Patching at FJC flips the answer at 20-44% rate (vs. 3-18% at FJC-8 to FJC-2); 27-51% at FJC+2 to FJC+4 (post-FJC nonlinear transforms can amplify).
  - Layer-skipping with α-blend injection (0.7·h_orig + 0.3·h_FJC) rescues 47.8% of Overprocessed samples on average (range 36.7-67.3% across tasks) vs. 10.4% with raw replacement — confirming that Overprocessed = "correct info present but geometrically mismatched."
  - Re-injection from FPCL recovers 22-38% of Unresolved (conditional 38%, loop 26-30%) while Resolved controls stay at 84-100% — Unresolved is "not finished," not "incapable."
  - Overprocessing is geometric: Hoyer sparsity spikes at the final layer (L27); gap between Overprocessed and non-Overprocessed peaks at L22-24 for most tasks.
- **Loop vs. Loop-unrolled**: identical arithmetic; explicit `for` loops reach 35.5% Resolved vs. 28.0% for unrolled code, and dual-variable tracking UR is 23.5% (loop) vs. 53.6% (unrolled). Across 107 trajectory-matched pairs, trajectories are near-identical through layer 21, then loop's late MLP (layers 25-26) writes the correct digit more forcefully. Transplanting that MLP write into the unrolled twin closes 82.6% of the readiness gap; attention-direction and random-donor controls do nothing.
- **Family-specific late-stage component**: the late-segment rewrite is stable across 16 models, but its responsible component differs by family — attention in Qwen, MLP in Llama/DeepSeek. Reported as an empirical localization, not a universal circuit.
- **Ground-truth-free observability**: a closed-form Resolution Functional ρ built from endpoint statistics reaches AUC 0.850 (binary Resolved-vs-Rest). Leave-one-lens-out ablations drop 4-class balanced accuracy from 0.62 to ~0.40, confirming both lenses are load-bearing.
- **External corroboration (CRUXEval-O)**: the same model ordering emerges — DeepSeek-Coder > CodeLlama > Llama-2, accuracy rises with scale, code-specialized ≥ general. A Coder-0.5B failure mode (copying the answer placeholder) maps cleanly onto the paper's readiness-vs-format distinction.

## Critical Assessment

- **Novelty**: The brewing/resolution framing is a useful and pedagogically clean relabeling of long-standing probe-vs-decode disagreement diagnostics. The empirical claim that the brewing *scaffold* is stable (24-42% across architectures and scales) while *resolution* scales with capability is the most original and provocative contribution. The four-outcome taxonomy is sensible but largely reorganizes known phenomena (overthinking, FJC-null cases); the value-add is the causal validation, not the labels themselves.
- **Rigor**: Strong on breadth (16 models × 6 tasks × controlled sweeps × 3 intervention types). Three specific concerns:
  1. The α-blend (0.7·h_orig + 0.3·h_FJC) rescue rate of 47.8% is presented as evidence of geometric mismatch, but it could partly reflect generic partial-recovery from any reasonable signal injection; a stronger control (inject from an unrelated sample) would tighten the causal claim.
  2. The "stable brewing scaffold" claim is at *layer granularity*; whether the same 24-42% band holds for sub-circuit-level dynamics (attention heads, MLP sublayers) is explicitly left as future work and the paper is honest about this.
  3. The 41.5% aggregate Resolved is striking but the paper does not appear to report confidence intervals or inter-seed variance for the diagnostic probes; for a quantitative interpretability claim this matters.
- **Reproducibility**: Public code is a plus. The 24,300-sample-per-model evaluation is heavy but feasible. Probe training details (Section D.1) are documented in the appendix. Less clear: the cost of layer-wise probing across 16 models and the exact CSD baseline-subtraction implementation.
- **Significance**: For interpretability, the scaffold-vs-capability dissociation is a genuinely interesting empirical regularity worth pressure-testing on more architectures (MoE, encoder-decoder, very small / very large). For practitioners, the outcome taxonomy is actionable: early-exit is correct for Overprocessed but harmful for Unresolved, motivating outcome-aware inference (the paper explicitly frames this as the takeaway for Mixture-of-Depths and looped Transformers).
- **Red flags to watch**:
  1. The benchmark is short, single-primitive code — generalization to real repository-scale code is unproven.
  2. The "causal intervention" for Overprocessed relies on a hand-chosen α = 0.3; the 47.8% rescue rate is not robust to α in a way the abstract suggests.
  3. CSD assumes the target prompt T = Q carries a near-trivial language prior; for longer or more structured code, the baseline-subtraction may not fully cancel.
  4. "Brewing scaffold is stable" is a within-Transformer empirical claim, explicitly not a circuit-level universality result; readers should resist overgeneralizing.
  5. Single-digit output space (0-9) is convenient for probe design but constrained — these are toy primitives, not real code.

## Strengths

- Clean conceptual separation between "information availability" (linear probe) and "information readiness" (CSD), which most prior interpretability work conflates.
- Strong empirical breadth (16 models, 6 architecture lineages, 0.5B-14B) for a mechanistic-interpretability study.
- Controlled per-axis sweeps (depth, distractors, iterations, operators, condition type) localize failure axes rather than reporting aggregate accuracy.
- Three independent causal interventions all converge on the four-outcome taxonomy — Resolved/Overprocessed/Misresolved/Unresolved are intervention-sensitive computational states, not post-hoc labels.
- Actionable diagnostic for inference: an Overprocessed-aware early-exit policy and an Unresolved-aware re-injection policy are diametrically opposed, motivating outcome-aware depth allocation.
- Code release, external CRUXEval-O corroboration, and explicit GT-free signal (Resolution Functional ρ) make the framework practically usable at inference time.

## Limitations

- Restricted to decoder-only Transformers; encoder-decoder (CodeT5, T5) and MoE (Mixtral, DeepSeek-MoE) architectures are untested.
- Single-digit answer space (0-9) constrains the benchmark; the framework's behavior on multi-token outputs (variable names, multi-digit numbers) is not validated.
- Synthetic/structured code primitives (CUE-Bench) are short single-primitive programs; compositional, multi-statement, real repository-scale code is not covered.
- Linear probe assumes a linear decision boundary; richer probes (MLPs, sparse autoencoders, attention-head ablations) could reveal finer structure within the brewing interval.
- The α-blend injection for Overprocessed is hyperparameter-dependent (α = 0.3 chosen by hand); systematic α-sweep robustness is not in the main text.
- CSD baseline subtraction may be confounded by how models behave with stripped context (refusal, hallucination, format issues — the Coder-0.5B placeholder-copying failure is a concrete instance).
- Layer-granularity diagnostics cannot pinpoint attention heads or MLP sublayers driving the brewing-to-resolution transition; path patching is acknowledged future work.
- Aggregate Resolved = 41.5% is a single-model (Coder-7B) anchor result; cross-model confidence intervals are not foregrounded.

## Relevance

- 与你的研究兴趣高度相关:该工作属于 LLM 内部机制可解释性(mechanistic interpretability)与代码推理评测方向,与你关注的 LLM 评估/解释高度契合。
- "酿造支架稳定 vs. 消解随能力变化"的双轴解耦,以及基于信息可用性/就绪度的失败模式分类,为设计更有诊断性、可因果干预的代码能力评测与自适应推理策略(早退 / 重注入)提供了可直接复用的范式与基线。

## Questions for Further Exploration

- Does the 24-42% brewing-scaffold band generalize to **encoder-decoder** code models (CodeT5, T5-coder) and **MoE** architectures (Mixtral, DeepSeek-MoE) — or is it specific to decoder-only Transformers with residual streams?
- Can the **Resolution Functional ρ** be deployed at inference time as a runtime monitor for outcome-aware decoding, and does it generalize out-of-distribution to multi-token outputs (variable names, multi-digit numbers)?
- Is the **Overprocessed "geometric corruption"** (Hoyer sparsity peak at L22-24, late-layer norm growth) also present in natural-language reasoning (math, multi-hop QA), or is it specific to code's discrete-symbol structure?
- The **Loop vs. Loop-unrolled** late-MLP consolidation finding (loop's MLP writes the correct digit more forcefully) is a striking mechanistic insight — does it survive in larger models and in models trained on more code data, where unrolled code is rarer in pretraining?
- Can targeted **fine-tuning of the resolution phase** (post-FJC layers only) shift the outcome distribution more efficiently than full-model fine-tuning, given the scaffold's apparent invariance?
- The α-blend injection rescues 47.8% of Overprocessed samples — does this generalize to a **practical inference-time correction** (early-exit + re-decode) that improves Resolved rate without retraining?
- Why do **late layers corrupt correct answers** (Overprocessed)? Is this a necessary consequence of residual-stream geometry (norm growth, capacity saturation) or a contingent property of current training regimes — and would alternative training objectives (e.g., late-layer self-distillation) reduce Overprocessed mass?
- The **availability-before-readiness** ordering (FPCL < FJC universally) — is this a structural property of residual-stream Transformers, or an artifact of how pretrained models are optimized? Would a model trained with explicit intermediate-decoding objectives (e.g., InfoNCE on h^ℓ) invert or eliminate the brewing interval?
- How do the four outcome distributions shift under **chain-of-thought, tool use, or test-time compute scaling** — does CoT primarily fix Misresolved (wrong-branch confidence) or Unresolved (insufficient depth)?
- Are the four outcomes **predictable from the prompt alone** (e.g., from code length, nesting depth, identifier familiarity), enabling pre-hoc routing to a model depth or compute budget?
