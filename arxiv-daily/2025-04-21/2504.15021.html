Is Intelligence the Right Direction in New OS Scheduling for Multiple Resources in Cloud Environments?

arXiv: 2504.15021

Date: 2025-04-21

https://arxiv.org/abs/2504.15021

OSML+ 通过三个协作的机器学习模型（离线预测OAA/RCliff的MLP、离线QoS预测MLP、在线DDPG强化学习Agent）来智能调度云环境中混合部署服务的缓存、内存带宽和CPU等多维资源，收敛速度比启发式基线快2.6倍、比贝叶斯优化基线快5.5倍，并通过迁移学习实现跨平台泛化。

TL;DR

This paper proposes OSML+, a multi-model collaborative ML-based resource scheduling system for co-located cloud services. It addresses the Resource Cliff (RCliff) problem -- where slight resource reductions cause catastrophic QoS degradation (e.g., latency jumping from 34ms to 4644ms) -- by identifying the Optimal Allocation Area (OAA). Three models work together: Model-A (offline MLP, 32-64-32 architecture) predicts OAA and RCliff boundaries; Model-B (offline MLP) predicts QoS for new allocations enabling QoS-resource tradeoffs; Model-C (online DDPG reinforcement learning agent) dynamically adjusts resource allocations. Evaluated across three real server platforms (Intel Xeon with PMU, Intel with RDT/CAT, AMD with L3 cache partitioning), OSML+ converges 2.6x faster than heuristic baselines (5.8s vs 15.0s) and 5.5x faster than Bayesian optimization baselines (5.8s vs 32.1s). Transfer learning enables generalization across different hardware platforms. This is a practical, systems-oriented paper with real hardware evaluation, directly extending the authors' prior OSML work (FAST '23).

  2. Resource Cliff与OAA的形式化定义：首次系统性地定义了资源悬崖现象和最优分配区域的概念，为理解混合部署中的资源敏感性提供了理论基础。
  3. 跨平台迁移学习：通过迁移学习技术使训练好的模型能够适配不同硬件平台（Intel PMU、Intel RDT、AMD QoS），解决了ML模型在系统领域泛化性差的问题。
  4. QoS-资源权衡机制：Model-B能够预测不同资源分配下的QoS，允许在可接受范围内用QoS换取资源效率，而非简单地满足或违反QoS约束。

Key Contributions

  1. Multi-model collaborative scheduling architecture: A three-model pipeline (Model-A: OAA/RCliff prediction via MLP, Model-B: QoS prediction via MLP, Model-C: online DDPG RL agent) that combines offline profiling with online adaptation for co-located service resource scheduling.
  2. Formalization of Resource Cliff (RCliff) and Optimal Allocation Area (OAA): Systematic characterization of the nonlinear resource-performance relationship where minor resource reductions trigger catastrophic QoS degradation, and the identification of the ideal allocation zone.
  3. Cross-platform transfer learning: Demonstrates that models trained on one server platform can be transferred to different hardware architectures (Intel PMU, Intel RDT/CAT, AMD with L3 partitioning) with minimal retraining overhead.
  4. QoS-resource tradeoff mechanism: Model-B enables principled QoS-resource trading by predicting QoS outcomes for arbitrary allocations, moving beyond binary QoS satisfaction/violation to nuanced resource efficiency optimization.

Methodology

System Architecture

OSML+ operates in three phases:

  • Phase 1 -- Offline Profiling (Model-A): A feedforward MLP (32-64-32 neurons, ReLU, 30% dropout) is trained on profiling data collected by systematically varying resource allocations (cache ways, memory bandwidth, CPU cores). Model-A predicts the OAA boundaries and RCliff locations for each co-located service pair. Training data is collected by sampling allocations across a grid and measuring P99 tail latency.
  • Phase 2 -- Offline QoS Prediction (Model-B): Another MLP trained to predict QoS metrics (P99 latency) given a resource allocation configuration. This enables "what-if" analysis: given a candidate allocation, predict whether QoS will be met, and by what margin. This allows trading QoS headroom for resources allocated to Best-Effort (BE) services.
  • Phase 3 -- Online Dynamic Scheduling (Model-C): A Deep Deterministic Policy Gradient (DDPG) reinforcement learning agent that takes as input the OAA/RCliff predictions from Model-A and QoS predictions from Model-B, and outputs continuous resource allocation actions. The reward function balances LC service QoS satisfaction against BE service throughput maximization.

Resource Types Managed

  • Last-level cache (LLC): Partitioned via Intel CAT or AMD QoS technologies
  • Memory bandwidth: Allocated via Intel RDT Memory Bandwidth Allocation (MBA)
  • CPU cores: Allocated via cgroups/cpuset

Experimental Setup

Three server platforms:

  1. Intel Xeon Gold 5218 with hardware performance counters (PMU-based monitoring)
  2. Intel Xeon Platinum 8360Y with RDT/CAT hardware partitioning
  3. AMD EPYC 7742 with AMD QoS cache partitioning

Workloads: Latency-Critical (LC) services (e.g., memcached, web serving) co-located with Best-Effort (BE) batch jobs (e.g., graph analytics, data processing).

Critical Assessment

Strengths

  1. Real hardware evaluation on 3 platforms: Most ML-for-systems papers evaluate on a single platform or simulators. OSML+ tests on Intel PMU, Intel RDT, and AMD -- providing stronger generalizability evidence.
  2. Practical multi-model design: The three-model decomposition is sensible -- separating OAA identification, QoS prediction, and online scheduling into specialized models rather than one monolithic approach.
  3. Addresses a real operational problem: Resource cliffs are a genuine pain point in production cloud environments where co-locating LC and BE services. The problem formulation is well-motivated.
  4. Fast convergence: 2.6x-5.5x faster convergence than baselines is operationally meaningful in cloud environments where scheduling decisions must be made quickly.
  5. Transfer learning for generalization: Explicitly addressing the cross-platform generalization problem that plagues many ML-for-systems approaches.

Limitations

  1. Limited workload diversity: The evaluation primarily uses memcached and a few BE benchmarks. Production cloud environments have much more diverse service mixes (databases, ML inference, streaming, etc.). The generalizability to arbitrary workload combinations is not established.
  2. Offline profiling overhead: Model-A and Model-B require extensive offline profiling data collection. The paper does not quantify how much profiling time is needed for a new workload, which is a critical deployment cost.
  3. DDPG instability: DDPG is known for training instability and sensitivity to hyperparameters. The paper does not discuss failure modes, hyperparameter sensitivity analysis, or how robust Model-C is to distribution shift.
  4. Comparison baseline completeness: The paper compares against heuristic and BO baselines but does not compare against other recent ML-based schedulers (e.g., CLITE, Parties, or other DRL-based approaches), making it hard to assess relative state-of-the-art standing.
  5. Scalability unclear: Experiments appear to focus on 2-service co-location scenarios. Production servers often run 5-10+ co-located services. How OSML+ scales with the number of co-located services and resource dimensions is not addressed.
  6. Safety guarantees: The RL agent (Model-C) explores resource allocations online. The paper lacks discussion of safety mechanisms to prevent QoS violations during exploration, which is critical for production deployment.

Questions for Further Exploration

  1. How much offline profiling time is required for a completely new workload, and can this be reduced via few-shot learning or meta-learning approaches?
  2. How does OSML+ scale to scenarios with 5+ co-located services competing for shared resources?
  3. What safety mechanisms can be added to Model-C (DDPG) to provide formal QoS guarantees during online exploration?
  4. Can the transfer learning approach generalize to ARM-based servers or GPU resource partitioning, which are increasingly relevant in modern data centers?
  5. How does OSML+ compare with more recent DRL-based schedulers (e.g., those using SAC or PPO instead of DDPG) in terms of both performance and training stability?
