LithOS: An Operating System for Efficient Machine Learning on GPUs

arXiv: 2504.15465

Date: 2025-04-21

https://arxiv.org/abs/2504.15465

一句话总结

LithOS 将 GPU 视为共享计算资源，通过 TPC 粒度空间调度、内核原子化、硬件右尺寸化和透明 DVFS 等 OS 级机制，在不修改应用的前提下实现了推理堆叠 3 倍尾延迟降低、混合负载 1.18 倍吞吐提升，以及显著的能效与资源节省。

TL;DR

LithOS is a GPU operating system that transparently intercepts CUDA driver calls to provide four key mechanisms: (1) TPC-granularity spatial scheduling via kernel atomization, (2) hardware right-sizing using an Amdahl's-law-based scaling model, (3) TPC stealing across applications, and (4) transparent DVFS for power management. Evaluated on NVIDIA A100 GPUs against time slicing, MPS, MIG, REEF, Orion, and TGS, LithOS achieves up to 13x tail latency reduction versus MPS and 3x versus state-of-the-art for inference stacking, while right-sizing saves a quarter of GPU capacity for less than 4% performance overhead and DVFS saves a quarter of GPU energy for approximately 7% overhead. Published at SOSP 2025.

核心贡献

  1. TPC 粒度空间调度与内核原子化：将 GPU 内核透明拆分为更小的"原子"（线程块组），以 TPC（线程处理集群）为粒度进行空间调度，消除传统时间切片中的队头阻塞问题，使得多个 ML 工作负载能够高效共享同一 GPU。
  1. 硬件右尺寸化：基于 Amdahl 定律的扩展模型（l = m/t + b），在线预测每个内核所需的最少 TPC 数量，在性能损失不到 4% 的情况下释放约四分之一的 GPU 容量供其他工作负载使用。
  1. 透明 DVFS 与 TPC 窃取：根据在飞工作负载动态调整 GPU 频率以节省约四分之一能耗（约 7% 性能开销），并通过 TPC 窃取机制将空闲 TPC 重新分配给需要更多资源的应用。
  1. 端到端透明部署：通过 CUDA 驱动 API 拦截实现，无需修改任何应用程序代码，支持 PyTorch、TensorFlow、JAX、TensorRT 和 cuDNN 等主流框架，以约 5000 行 Rust 代码实现。

Key Contributions

  1. TPC-Granularity Spatial Scheduling with Kernel Atomization: Transparently splits GPU kernels into smaller "atoms" (thread block chunks) and schedules them at TPC (Thread Processing Cluster) granularity, eliminating head-of-line blocking inherent in traditional time slicing. This enables efficient spatial multiplexing of multiple ML workloads on a single GPU.
  1. Hardware Right-Sizing: Uses an Amdahl's-law-based scaling model (l = m/t + b) to online-predict the minimal number of TPCs each kernel requires, freeing up approximately a quarter of GPU capacity for other workloads with less than 4% performance overhead.
  1. Transparent DVFS and TPC Stealing: Dynamically adjusts GPU frequency based on in-flight work to save approximately a quarter of GPU energy for roughly 7% performance overhead, and reassigns idle TPCs across applications via a stealing mechanism to improve overall utilization.
  1. Transparent Deployment via CUDA Driver Interposition: Requires zero application modifications by intercepting CUDA driver API calls. Supports PyTorch, TensorFlow, JAX, TensorRT, and cuDNN. Implemented in approximately 5000 lines of Rust.

Methodology

LithOS operates as a transparent layer between ML applications and the GPU hardware, intercepting all CUDA driver API calls. The design rests on four pillars:

Kernel Atomization: When a kernel launch is intercepted, LithOS splits it into multiple "atoms," each containing a subset of the original thread blocks. These atoms are then scheduled onto TPCs independently. This prevents large kernels from monopolizing the entire GPU and creating head-of-line blocking for other applications' kernels.

Latency Prediction Module: An online module predicts kernel execution time based on historical observations. This prediction drives all scheduling decisions: how many TPCs to allocate (right-sizing), whether to steal TPCs from other applications, and when to adjust GPU frequency.

Hardware Right-Sizing via Amdahl's Law: LithOS models kernel execution time as l = m/t + b, where m is the parallelizable portion, t is the number of TPCs, and b is the serial portion. By fitting this model online, LithOS determines the minimal TPC count that keeps performance within acceptable bounds, freeing excess TPCs for concurrent workloads.

TPC Stealing and DVFS: Idle TPCs detected through the latency prediction module are transparently reassigned to applications that can benefit from additional compute. DVFS adjusts GPU clock frequency based on current in-flight work intensity, reducing power consumption during low-utilization periods without requiring application awareness.

The system is implemented as a shared library loaded via LD_PRELOAD, intercepting CUDA driver calls and managing all scheduling internally. It runs entirely in userspace on unmodified NVIDIA drivers.

Results & Highlights

  • Inference Stacking: Running multiple latency-critical inference workloads on a single A100 GPU, LithOS achieves up to 13x tail latency reduction compared to NVIDIA MPS and 3x improvement over the best prior art (REEF, Orion, TGS).
  • Hybrid Inference-Training Stacking: Mixing latency-critical inference with best-effort training, LithOS achieves 4.7x improvement over MPS and 1.18x over state-of-the-art for tail latency, while maintaining training throughput.
  • Right-Sizing: Frees approximately 25% of GPU capacity for less than 4% performance degradation, validated across diverse ML workloads including CNNs, Transformers, and recommendation models.
  • DVFS: Saves approximately 25% of GPU energy for roughly 7% performance overhead by transparently adjusting clock frequencies.
  • Production Motivation: Analysis of Meta's Ads inference service reveals average GPU utilization of only 27%, demonstrating the real-world need for better GPU multiplexing.
  • Broad Framework Support: Works transparently with PyTorch, TensorFlow, JAX, TensorRT, and cuDNN without any code changes.

Critical Assessment

LithOS presents a compelling and well-engineered solution to the real problem of GPU underutilization in ML workloads. The production analysis from Meta (27% average utilization) provides strong motivation, and the transparent deployment approach via CUDA driver interposition is pragmatic and immediately deployable.

Methodological Strengths: The evaluation is thorough, comparing against five baselines (time slicing, MPS, MIG, REEF, Orion, TGS) on A100 GPUs across diverse workloads. The ablation study isolates the contribution of each mechanism. The use of an Amdahl's-law-based scaling model is elegant and computationally lightweight.

Concerns: The evaluation is limited to NVIDIA A100 GPUs. The TPC abstraction is NVIDIA-specific (TPCs are a hardware concept in NVIDIA architectures), raising questions about portability to AMD or Intel GPUs. The latency prediction module relies on historical observations, which may be less effective for kernels with highly variable execution times or cold-start scenarios. The paper does not deeply explore interactions with NVIDIA's newer hardware features (e.g., the thread block cluster mechanism in Hopper architecture). Additionally, while the 27% utilization figure from Meta is striking, it may not generalize across all ML deployment scenarios --- some workloads (e.g., large-scale training runs) may already achieve high utilization.

Intellectual Honesty: The authors provide ablation studies showing that no single mechanism dominates; the gains come from the combination. They also acknowledge the overhead of kernel atomization for very small kernels and discuss limitations of the Amdahl model for kernels with poor scaling behavior.

Strengths

  • Transparent deployment requiring zero application modifications, supporting all major ML frameworks.
  • Strong quantitative results: 13x tail latency improvement over MPS, 3x over state-of-the-art.
  • Production-motivated research grounded in real Meta deployment data.
  • Clean systems design decomposing the problem into four composable mechanisms.
  • Implemented in Rust (~5000 lines), suggesting robustness and maintainability.
  • Thorough evaluation with six baselines and ablation studies.

Limitations

  • Evaluated only on NVIDIA A100; portability to other GPU vendors (AMD, Intel) is unaddressed.
  • TPC is an NVIDIA-specific hardware concept; the design may not map cleanly to other architectures.
  • Latency prediction module may struggle with cold-start kernels or highly variable workloads.
  • Does not explore interactions with newer GPU features (e.g., Hopper thread block clusters, H100 TMA).
  • The 7% performance overhead for DVFS may be unacceptable for latency-critical serving scenarios.
  • Right-sizing assumes kernels exhibit Amdahl-like scaling, which may not hold for all operations (e.g., memory-bound kernels with poor scaling).

训练稳定性（Training Stability）：虽然本文主要关注推理和混合负载场景，但其混合推理-训练堆叠实验表明，在训练工作负载与推理共存时，LithOS 能够有效隔离延迟敏感型推理的影响，同时维持训练吞吐。这对大规模训练集群的资源利用率有直接意义。

Questions for Further Exploration

  1. How would LithOS perform on NVIDIA H100/H200 GPUs with Hopper architecture features like Thread Block Clusters and the Tensor Memory Accelerator (TMA)? Would the atomization mechanism need fundamental redesign?
  2. Can the latency prediction module be extended to handle multi-GPU or distributed training scenarios where kernel execution times depend on inter-node communication patterns?
  3. What is the interaction between LithOS's TPC stealing mechanism and NVIDIA's upcoming Confidential Computing features, which may restrict cross-application resource sharing?
  4. How does kernel atomization affect the behavior of CUDA graphs, which are increasingly used for optimized inference pipelines in LLM serving frameworks like TensorRT-LLM and vLLM?
  5. Could the right-sizing model be extended to incorporate memory bandwidth as a constraint, enabling more precise resource allocation for memory-bound kernels common in LLM inference (e.g., attention layers)?
