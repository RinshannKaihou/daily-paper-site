> **一句话总结：** 本文提出 Skillware —— 一种面向持久化"行为制品"（Agent Skills 的自然语言行为源码）的软件本体与工程生命周期，通过三个必要条件（行为主导性、独立软件身份、Agent Host 执行关系）+ Lifecycle Continuity 属性界定其范畴，并基于 138,133 条 SKILL.md 语料、15 个边界案例与 13 个固定版本工程实现，论证了经典软件工程模块/组件/生命周期/设计模式可平滑迁移到这一新型 AI 原生软件载体。

> **TL;DR:** The paper defines **Skillware** — a software ontology and engineering lifecycle for persistent natural-language "Behavioral Artifacts" (Agent Skills, e.g. a directory with SKILL.md). It operationalizes membership via three necessary conditions (behavioral primacy C1, independent software identity C2, Agent-Host execution C3) plus a separate Lifecycle-Continuity property, then grounds the claim on the Agent Skills spec, a frozen **SkillMD-138K** corpus of **138,133 deduplicated SKILL.md files** (98.73% with frontmatter) tied to **20,556 repositories**, 15 boundary cases, and 13 fixed-revision implementations, showing that classical GoF patterns (Facade/Adapter/Composite/Observer/State/Strategy) transfer to natural-language skill carriers. No head-to-head benchmark deltas are reported — the evidence is existence/recurrence/construct-validity rather than accuracy gains.

## 核心贡献

- **Skillware 软件本体**：把 Agent Skill 提升为可独立寻址、拥有版本与生命周期的软件对象，明确区分 *Behavioral Source → Skill Artifact → Skillware Unit → Agent Host → Agent Runtime → Execution Trace → Task Outcome* 的责任链；其中 Skill Artifact 携带可复用行为，Skillware Unit 是围绕它建立的独立软件身份（原子实现二者重合，复合实现如 Superpowers 多技能共享一个包级身份）。
- **可审计的范畴判定准则**：三个必要条件 C1（Skill 行为主导性）、C2（独立软件身份：地址/包/版本/兼容性/来源）、C3（至少一个兼容 Agent Host 的发现-激活-加载-使用路径，如 Codex/Claude Code/Copilot/Hermes/OpenClaw），另以 *Lifecycle Continuity* 单独衡量"同一身份穿越更新/维护/回滚/卸载"的软件级管理强度。Membership 不评分质量/安全/流行度。
- **Skillware 工程生命周期**：一个最小基线（Atomic）+ 六个可正交组合的实现维度（Resource-Backed、Tool-Backed、Event-Driven、Plugin-Packaged、Companion-Coupled、Adaptation-Enabled）刻画结构剖面；并用六步工作流（定义单元/选维度/选模式/实现机制/评估打包/发布维护）将经典 SE 范式接入。
- **设计模式可迁移性证据**：给出 Superpowers using-superpowers=Facade、gstack 生成绑定=Adapter、OpenMontage 分阶段工作流=Composite、ECC/OpenMontage 钩子=Observer、OpenMontage/SkillOpt 检查点=State、Superpowers 选择策略=Strategy 的具体映射，并附"7 要素"准入协议（源意图/设计力/参与者对应/后果/实现证据/聚焦验证/滥用判别），补充材料覆盖全部 23 个 GoF 模式。
- **Identity-Preserving Evolution 与 Collaborative Evolution**：提出"执行证据→变更提案→评估→受治理发布"的演化框架；将 SkillOpt 的证据驱动改进定位为 Self-Evolution 的部分先例，把跨人多方的 review→release→rollback 归约为 Collaborative Evolution（明确为未来工作）。
- **可复现证据包**：发布冻结快照（SkillMD-138K，SHA-256 与 commit 固定）、检测器定义、聚合输出、15 个边界案例（12 含/3 刻意排除）与 13 个固定版本技术评审单元的 canonical repo + 40-char commit + 精确路径 + 正反证据。

## Key Contributions

- **A software ontology for Agent Skills**: elevates Agent Skills from "prompt files" to independently addressable, versioned, lifecycle-managed software objects, with a clear responsibility chain (Behavioral Source → Skill Artifact → Skillware Unit → Agent Host → Agent Runtime → Execution Trace → Task Outcome) and an explicit separation of *artifact specification* from *runtime interpretation*, *situated performance*, and *task outcome* (so a description cannot be taken as evidence of effectiveness).
- **Auditable membership conditions**: three necessary membership tests — C1 behavioral primacy (the Skill source organizes the reusable task contract), C2 independent software identity (address/package/version/compatibility/provenance), C3 an identified Agent Host discovery-activation-loading-use path (Codex, Claude Code, Copilot, Hermes, OpenClaw) — plus a separately coded *Lifecycle Continuity* property tracking identity persistence across update/maintenance/rollback/removal.
- **Skillware Engineering Lifecycle**: one minimal profile (Atomic) and six orthogonal implementation dimensions (Resource-Backed, Tool-Backed, Event-Driven, Plugin-Packaged, Companion-Coupled, Adaptation-Enabled) that compose into multi-label profiles, plus a six-step authoring workflow (define unit → pick dimensions → pick patterns → implement mechanisms → evaluate/package → release/observe/adapt).
- **Evidence that classical SE transfers to natural-language carriers**: six grounded GoF mappings (Superpowers using-superpowers Facade; gstack generated bindings Adapter; OpenMontage staged workflows Composite; ECC/OpenMontage hooks Observer; OpenMontage/SkillOpt checkpoints State; Superpowers selection Strategy) governed by a 7-element admission protocol; supplement screens all 23 GoF patterns plus Pipes&Filters and Specification.
- **Evolutionary framing**: *Engineering Consolidation* (evidence-driven accretion of resources/scripts/bindings/tests around a stable unit identity), *Identity-Preserving Evolution* (execution-evidence → proposal → evaluation → governed release under one identity), and *Collaborative Evolution* as a multi-party governance realization (explicitly future work).
- **Reproducible evidence package**: frozen SkillMD-138K snapshot (fixed commit + SHA-256), detector definitions, aggregate outputs, 15 boundary cases (12 inclusions, 3 deliberate negatives), and 13 fixed-revision technical reviews (Superpowers, gstack, ECC, OpenMontage, SkillOpt, etc.) each with canonical repo, 40-char commit, exact paths, positive/counterevidence/limitations.

## Methodology

This is a **conceptual / ontology-building empirical paper**, not a benchmark ML study. The methodology is organized as four evidence layers with deliberately separated inference boundaries (Figure 3, Table 3):

1. **Specification evidence** — The Agent Skills specification and official docs (agentskills.io; OpenAI/Anthropic/GitHub/Nous implementation docs) establish the *artifact envelope*: a required `SKILL.md`, YAML metadata, Markdown instructions, optional `scripts/`, `references/`, `assets/`, plus progressive-disclosure loading. Named "skills-compatible agent" hosts (Codex, Claude Code, GitHub Copilot, Hermes, OpenClaw) supply C3 execution paths.

2. **Collection-scale corpus signals** — A frozen, content-hash-deduplicated snapshot **SkillMD-138K** (HuggingFace dataset, fixed revision `0d73048a…`, raw Parquet SHA-256 `c3c07af0…`) is analyzed with *narrow lexical detectors* for (a) frontmatter delimiters and (b) explicit path tokens into `references/`/`scripts/`/`assets/`. The authors are explicit that path tokens do **not** prove a resource exists or is used — they only measure authoring convention prevalence.

3. **Fixed-revision case review** — A *category-boundary review* of 15 declared units (12 inclusions, 3 deliberate negatives: external Skill index, MCP filesystem reference server, agent-system-managed GPT config) and a *technical review* of 13 fixed-revision implementations (Superpowers, gstack, ECC, last30days, scientific-schematics, financial-analysis, dot-skill, ui-ux-pro-max, Caveman, design-taste-frontend, darwin-skill, SkillOpt-Sleep, OpenMontage), each pinned to a canonical repository + 40-character commit + exact paths with positive observations, counterevidence, and limitations recorded.

4. **Independent empirical studies** — Chen et al. (Skill OS, 97,755 Skills), Gao et al. (18,463 registry + 23,199 personal Skills from 5,876 repos, 3,709 reuse links), Hong et al. (238-Skill smell study), Metere (verifiable artifacts), SkillOS (curation), FederatedSkill, SkillFab — cited under *their own samples and denominators* as converging evidence of execution, reuse, maintenance, and quality pressure.

The **formal core** is Definition 1 (Skillware) plus the C1–C3 + LC test battery (Table 2), and a runtime-semantics equation `π = R_H(S, τ, M, C, T, P, σ)` (Eq. 1) that deliberately assigns *no deterministic semantics to unrestricted natural language* — it only enumerates the variables any compatibility/robustness claim must control. Design-pattern transfer uses a 7-element admission protocol (source intent, design forces, participant correspondence, consequences, implementation evidence, focused verification, misuse discriminator) to reject name-only mappings. An initial six-step engineering workflow and 8 named mechanisms (Skill Router, Progressive Disclosure, Deterministic Kernel, Canonical Skill + Generated Adapters, Lifecycle Bootstrap, Verification Gate, Human-Gated Companion, Evidence-Gated Self-Evolution) operationalize the abstraction.

## Results & Highlights

The paper reports **no benchmark accuracy/F1/task-success deltas** — its quantitative claims are corpus-scale prevalence and case-count evidence for the *existence and recurrence* of the Skillware form. Key quantitative findings:

- **Corpus scale & frontmatter prevalence**: After content-hash deduplication the SkillMD-138K snapshot contains **138,133 SKILL.md records** associated with **20,556 repository identifiers**; **136,380 files (98.73% of the corpus)** carry YAML frontmatter delimiters — taken as narrow evidence that a structured metadata envelope is a *dominant visible convention* (not proof of field validity or identical activation semantics across hosts).
- **Support-artifact references**: A conservative lexical detector finds **32,069 files (~23.2%)** with at least one explicit path token into `references/`, `scripts/`, or `assets/`, totaling **236,421 such tokens** across the corpus — evidence that explicit references to adjacent packaged artifacts are a *measurable authoring convention* (the authors flag that it misses implicit dependencies, hooks, generated adapters, and can count illustrative/stale paths).
- **Document length (engineering-pressure signal)**: median file = **169 lines / 687 words**; 90th percentile = **517 lines / 1,971 words** — read as the amount of behavioral source that may enter discovery, context assembly, review, and maintenance (explicitly "carries no direct quality or maturity meaning").
- **Evidence inventory**: **15 category-boundary cases** (12 inclusions, 3 deliberate negatives) and **13 fixed-revision engineering implementations**, each pinned to a canonical repository + 40-character commit. **No complete Adaptation-Enabled Skillware case** was found among the 13 reviewed units (SkillOpt and darwin-skill are only *partial* Self-Evolution precedents) — an explicit negative result.
- **Pattern transfer coverage**: 6 main-text GoF mappings (Facade, Adapter, Composite, Observer, State, Strategy) with verified open-source correspondences; supplement screens **all 23 GoF patterns** plus Pipes&Filters and Specification, with **10 GoF patterns** having detailed constructive implementations (Facade, Adapter, Composite, Decorator, Strategy, Template Method, Observer, State, Memento, Mediator).
- **Independent-study corroboration (cited under their own denominators)**: Chen et al. characterize **97,755 Skills**; Gao et al. analyze **18,463 registry + 23,199 personal-use Skills from 5,876 GitHub repositories with 3,709 reuse links**; Hong et al. study **238 Skills** for smells — all converging on execution/reuse/maintenance/quality pressure without population estimates of *high-quality* Skillware.

If a strict benchmark table is required: **No quantitative results reported** in the ML-evaluation sense; the above are the paper's quantitative existence/recurrence evidence.

- **Provenance discipline is unusually strong for a non-benchmark paper**: every corpus snapshot is pinned to a HuggingFace revision + raw-Parquet SHA-256; every technical case is pinned to a canonical repository + 40-character git commit + exact paths; negative cases and counterevidence fields are explicit. A released study package (corpus manifests, detector definitions, aggregate outputs, repository aliases, verification records, reconstruction scripts) is provided — though it excludes third-party Skill bodies.

## Critical Assessment

- **Novelty — moderate, well-scoped**: The contribution is a *taxonomy/ontology + evidence protocol*, not a new algorithm. Its genuine novelty is (a) the C1–C3+LC test battery that makes membership auditable and (b) the design-pattern transfer protocol with its 7-element admission gate. However, the conceptual distance from Chen et al.'s "Skill OS" (the named closest precursor, [22]) and from component/software-evolution theory is real but incremental; the paper is honest that it "absorbs" rather than supersedes these.
- **Rigor — strong on construct discipline, weak on inter-rater reliability**: The inference boundaries (Table 3) are exemplary — the authors refuse to let a path token or frontmatter count prove execution semantics. But the category-boundary review of 15 units is *purposive* with **no inter-rater reliability estimate** (the paper admits this: "It supplies no estimate of inter-rater reliability. Independent coding and agreement analysis remain required"). The whole ontology's practical value hinges on whether third parties reproduce the same boundary decisions, and that test has not been run.
- **Reproducibility — high for corpus, partial for cases**: The frozen SkillMD-138K snapshot with SHA-256 and the 40-character commits make the corpus and case evidence reconstructible; content-level replication requires separately acquiring the corpus, and fixed commits "age as projects change." Direct multi-system execution for cross-Host parity claims is *not* performed — Adapter/Composite mappings assert "strong correspondence" but defer runtime parity to future tests.
- **Significance — potentially high if adopted, currently speculative**: If the Agent-Skills ecosystem continues to consolidate, a shared software ontology is genuinely useful (versioning, compatibility, supply-chain security). Today the impact is mostly framing: there is **no complete Adaptation-Enabled case** among 13 reviewed units, and Collaborative Evolution + Forward-Deployed Engineering are explicitly *prospective*.
- **Red flags / specific weaknesses**:
  - The **lexical detector for support paths (~23.2%)** is acknowledged to miss implicit dependencies, hooks, generated adapters, and to count illustrative/stale paths — so even the prevalence number is a floor/ceiling of unknown bias, yet it is one of the headline quantitative claims.
  - **Several closest precursors are 2026 arXiv preprints** (SkillOS, SkillFab, the Gao/Hong smell studies) whose methods "may change through review" — the synthesis rests on a fast-moving, not-yet-vetted literature.
  - **Construct validity of C1 "behavioral primacy"** is openly hard to judge in packages with substantial executable services (e.g., gstack, ECC) — the very cases meant to demonstrate the compound profile are where the central test is most contestable.
  - The runtime equation `π = R_H(S, τ, M, C, T, P, σ)` is essentially a *variable list*, not a model; the paper is transparent about this ("assigns no deterministic semantics"), but readers expecting executable formalism will find none.
  - No quantitative evidence that adopting the Skillware ontology *improves* any engineering outcome (defect rate, reuse, time-to-deploy) — all such benefits are framed as future field studies.

## Strengths

- Exceptionally disciplined **inference-boundary hygiene**: each evidence layer (spec / corpus / case / independent study) is mapped to the exact condition it can and cannot support (Table 3), preventing the common over-claim that "a file exists therefore a behavior is realized."
- **Operational, falsifiable membership test** (C1–C3 + LC) instead of a vague definition — a refreshing change for an ontology paper, and one that makes disagreement *inspectable* via the counterevidence fields.
- **Reproducible-by-construction evidence**: frozen corpus hashes, 40-character commits, released detector definitions and reconstruction scripts.
- **Concrete, source-grounded design-pattern mappings** with a 7-element admission protocol and explicit *misuse discriminators* (e.g., "`adapter.yaml` is only navigational evidence until it translates a canonical interface") — resists the trap of pattern-name rubber-stamping.
- Honest, non-promotional tone: explicit negative results (no complete Adaptation-Enabled case), explicit disclaimers on word coinage and population estimates.

## Limitations

- **No ML benchmarks**: zero task-success/accuracy/F1 deltas; the quantitative evidence is prevalence and case-count, which limits comparability with empirical SE or agent-capability benchmarks.
- **Purposive case selection, no inter-rater reliability** — the central discriminability claim rests on the authors' own coding of 15 boundary cases.
- **Lexical-only corpus signals**: frontmatter and path-token detectors cannot establish semantic correctness, runtime use, or cross-Host parity.
- **Fast-moving, partly unreviewed precursors**: several load-bearing cited works are 2026 preprints (SkillOS, SkillFab, Gao et al., Hong et al.).
- **Construct ambiguity at the compound end**: C1 primacy is hardest to judge exactly where the paper needs it most (heavy plugin/execution units like gstack, ECC).
- **Prospective, unvalidated extensions**: Engineering Consolidation trajectory, Collaborative Evolution, and Forward-Deployed Engineering are framings without longitudinal or field-trial evidence.
- **Single-group authorship** (MetaInFLow) with no independent replication of the classification protocol yet.

## Relevance

对关注 Agent Skills、AI 原生软件工程与"以自然语言为编程介质"的研究者，本文提供了一套可直接用于为 Skill 体系建立版本/兼容性/供应链治理的范畴框架与可审计判据，与你对 skill 设计模式、自演化（Self-Evolution）与生产化治理的兴趣高度契合；若你在构建或维护 skill 仓库，C1–C3 + Lifecycle Continuity 可作为现成的工程评估清单。

## Questions for Further Exploration

- Would blind third-party coders, given only the C1–C3 + LC rubric, reproduce the authors' 12-inclusion / 3-exclusion boundary decisions on the same 15 units? What is the inter-rater agreement (e.g., Cohen's κ)?
- How does Skillware membership correlate with downstream engineering outcomes — defect rate, reuse frequency, time-to-first-useful-deployment, regression frequency — in a controlled field trial vs. code-first delivery?
- For the compound units where C1 primacy is contestable (gstack, ECC), can a behavioral-primacy *metric* (e.g., share of runtime decisions routed through Skill source vs. executable services) operationalize the test instead of analyst judgment?
- The runtime equation `π = R_H(S, τ, M, C, T, P, σ)` lists variables but quantifies nothing: can paired trials fix (task, Host, model, runtime version, tools, permissions) and measure between-run variance, latency, cost, and recovery, as the authors' own research agenda proposes?
- The ~23.2% support-path-token prevalence is biased by an unknown amount (implicit deps, generated adapters, stale paths): how does the estimate change under repository-level resolution and dynamic trace collection?
- Since no complete Adaptation-Enabled Skillware case exists among the 13 reviewed units, what is the minimal evidence-gating + rollback apparatus needed to make Evidence-Gated Self-Evolution satisfy identity preservation end-to-end?
