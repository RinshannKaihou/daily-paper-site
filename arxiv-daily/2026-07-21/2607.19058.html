2 Why this matters

A plug-and-play “allocate state by role” principle that moves a 6.78B MoE off an 80 GB-class device onto a 40 GB one — code & logs open-sourced.

Targets anyone doing memory-constrained MoE training, optimizer engineering, or Adafactor/ZeRO/GaLore-style state compression.

3 Critical edge

The tiers buy memory, not accuracy. The perplexity lead rests on single seeds, and SkewAdam was never tuned while the baselines were.

Methodology

Tiered allocation

SkewAdam is a standard Adam-style update with one idea: an MoE is not homogeneous. Its three parameter populations — dense backbone (5%), experts (95%), router (<0.01%) — differ so sharply in size and gradient statistics that they should not share one state recipe. State is allocated by role. 

MoE parameters · 6.78B
   │
   ├── Backbone  5%     fp32 momentum + factored 2nd moment   1.27 GB
   │              (embed · attn · dense FFN · norms)
   │
   ├── Experts  95%     factored 2nd moment ONLY             ~0 GB momentum
   │              (128 SwiGLU · top-2-of-128 routing)
   │
   └── Router  <0.01%   EXACT unfactored 2nd moment            2 MB
                  (128×4096 gate · no weight decay)

Total optimizer state: 1.29 GB   ·   AdamW baseline: 50.55 GB

Reused, not novel

  • Factored 2nd moment (Adafactor): row+col EMA → rank-1 V̂; a 4096×4096 matrix drops 64 MB → 32 KB.
  • Update-RMS clipping (also Adafactor's).

Matched protocol

  • bf16 master weights; fp32 step written back through dithered stochastic rounding — identical path for every optimizer.
  • 2-block decoder, width 4096, GQA 32/8, ~440M active/token; OpenWebText hash-split 95/5, 10k steps, batch 64×128, 81.9M tokens, cosine + 3% warmup.

Results

Beyond the headline

The structural memory win is unambiguous; the perplexity win is real but underpowered. SkewAdam overtakes AdamW and Muon by step 4,000, and its lead survives tuning the baselines (AdamW → 118.5, Adafactor → 139.7).

Strengths

  • Role-aware, not uniform — allocation targets the dominant cost (expert momentum) instead of shrinking all state equally.
  • Large structural savings — 97.4% state / 61% peak-memory reduction, independent of quantization and compounding with it.
  • Exceptionally controlled comparison — shared initialization, identical data order, identical bf16 / dithered-rounding write-back path; no optimizer is precision-privileged.
  • Intellectual honesty — the paper itself frames the result as memory-at-parity, and pre-empts over-reading the near-chance downstream numbers.
  • Strong artifact — open code, per-step logs, exact CLI commands, closed-form memory model, and cross-vendor replication.
!

Limitations

  • Shallow by design — only a 2-block model; how tiered allocation behaves when routing composes across many MoE layers is untested.
  • Statistically underpowered — most configs are single-seed; only LR sweeps repeat seeds, and SkewAdam itself was never tuned while baselines were.
  • Tiny horizon — 82M tokens / 128-token context, three to four orders of magnitude below production; weight decay rounded to a no-op, so the recipe is effectively unregularized.
  • Near-chance downstream evals (PIQA / WinoGrande / HellaSwag / ARC-C) — a completeness check only, no optimizer separation.
  • Weak baselines caveats — the GaLore run (1,839.9 PPL) is a single untuned own-implementation config; Lion's 393.7 is one LR / one seed.
  • Single-author, self-funded preprint — the compute budget, not the design, set the scale.

Questions for further exploration

Open threads
  1. Does tiered allocation keep its perplexity / memory edge on deeper MoEs (16–32 blocks), where routing decisions compose across many expert layers?
  2. Would a cheap, low-rank expert momentum (vs. full fp32) close any gap at longer horizons where expert momentum might “begin to earn its 24 GB”?
  3. How does SkewAdam interact with quantized state (8-bit, FP8) — does factoring's lack of a scale ceiling compound with quantization, as speculated?
  4. Does Muon's late load-balance drift (22% above the 0.05 floor in the final thousand steps) foreshadow a real instability at longer horizons or larger expert counts?
  5. Can the tiered principle extend to other heterogeneous-parameter architectures — retrieval-augmented models, multi-modal towers — where gradient density also varies sharply by component?
