ReLibra: Routing-Replay-Guided Load Balancing for MoE Training in Reinforcement Learning

Chao Jin, Xinming Wei, Yinmin Zhong, Chengxu Yang, Bingyang Wu, Ruidong Zhu, Zili Zhang, Yuliang Liu, Xin Jin — Peking University
ReLibra exploits RL's rollout-training loop to know exact token-to-expert routing before training begins, then decomposes load balancing into inter-batch expert reordering (swap-based simulated annealing) and intra-batch dynamic expert replication (MILP with layer-shared replica buffer), achieving near-optimal throughput on 20-node Hopper clusters.
Key Finding

In RL's rollout-training pipeline, both stages process identical tokens with identical MoE parameters — so routing is fully determined during rollout. This eliminates the load prediction problem entirely, and the optimization overlaps with post-rollout processing to hide solver latency.

Key Metric
1.58×
throughput over Megatron-LM · 90–94% of ideal balanced baseline · tested on Qwen3-30B/106B/235B + GLM4.5-106B
Why This Matters

MoE+RL training (GRPO, RLHF) is now the standard post-training pipeline for frontier models. 1.21–1.58× throughput gains translate directly to millions in production training cost savings at scale.

Critical Edge

Fundamentally limited to RL training where rollout and training share identical routing — not applicable to MoE pretraining or inference. Off-policy RL with parameter drift breaks the routing replay assumption. Also requires a separate 128-core CPU node for the solver.

Routing Replay
  • During RL rollout, tokens are routed to MoE experts — this routing is recorded
  • Training processes the same tokens with the same MoE parameters, so routing is identical
  • Solver runs concurrently with reward computation and KL divergence evaluation — zero overhead on the training critical path
Timescale Decomposition
  • Inter-batch (reordering): Swap-based simulated annealing across GPUs, with LSE-smoothed objective (beta=20) to gradient-signal near-bottleneck nodes. Includes data locality optimization via sample-swap annealing
  • Intra-batch (replication): MILP-formulated expert replication with incremental greedy heuristic. Places replicas on least-loaded feasible GPU, re-solves token-splitting LP
Layer-Shared Replica Buffer
  • Each GPU allocates only r=2 replica slots, reused across all MoE layers
  • Memory overhead: 0.13%–1.62% of total parameter memory
  • Synchronization via persistent async TMA kernel on NVIDIA Hopper, limited to few SMs to avoid contention with attention computation
  • Replica traffic confined to intra-node NVLink (high bandwidth)
Communication Model
  • Jointly models computation and all-to-all communication under rail-optimized topology
  • Three link types: intra-node NVLink, same-rail RDMA, cross-rail RDMA with NVLink relay
  • Captures realistic bandwidth asymmetry between link tiers
  • Objective: minimize T_MoE = max_g(T_g^comp) + max_g(T_g^comm)

Overhead Breakdown
  • Solver time: fully hidden behind post-rollout processing
  • Expert reordering: 1.4–2.1% of training batch time
  • Replica buffer memory: 0.13–1.62% of parameter memory
  • Replica synchronization: fully overlapped with attention via async TMA kernel
  • Rank-level skewness: 1.00–1.08 across EP sizes 8–64 (vs. much higher for Megatron-LM)
Strengths
Clean, elegant core insight (routing replay) with solid systems engineering to exploit it
Comprehensive evaluation: 3 models (30B–235B), 4 RL domains, multiple baselines including oracle-strengthened ones
Thorough ablation quantifying contributions of each component independently
Layer-shared replica buffer is a simple but effective design making micro-batch-level replication practical
Joint optimization of computation and communication goes beyond prior work that only considers computation load
All overheads quantified and shown to be negligible or hidden
Limitations
Fundamentally limited to RL training — not applicable to MoE pretraining or inference
Only 2 replica slots per GPU tested; scalability to more aggressive replication is unexplored
Requires a separate 128-core CPU node for the solver — practical barrier for smaller teams
Off-policy RL with stale parameters may break routing replay assumptions; degradation not quantified
No evaluation of end-to-end RL training convergence/quality — only throughput is measured
Mixed workload follows DeepSeek-V3.2 recipe; may not generalize to other RL configurations

Models & Datasets
  • Models: Qwen3-30B-A10B, Qwen3-106B-A22B, Qwen3-235B-A22B, GLM4.5-106B
  • RL domains: Reasoning, coding, instruction following, mixed
  • EP sizes: 8, 16, 32, 64
  • Mixed dataset: Follows DeepSeek-V3.2's recipe
Baselines
  • Megatron-LM: With DeepEP + GroupedGEMM (production baseline)
  • PopFetcher: Recent MoE load balancing approach
  • EPLB+: Expert-parallel load balancer strengthened with oracle loads from routing replay
  • LPLB+: Layer-parallel load balancer strengthened with oracle loads
  • Balanced: Ideal upper bound (randomly reroutes tokens — does not preserve model semantics)
Open Questions
  • How does replica slot count affect gains? Diminishing returns?
  • Can a lightweight routing correction compensate for off-policy parameter drift?
  • On-GPU CUDA-based LP solver to eliminate separate CPU node?
  • Interaction with emerging MoE communication optimizations (DeepEP hierarchical, Comet)?
  • Temporal stability of expert reordering across RL iterations for amortized planning?
