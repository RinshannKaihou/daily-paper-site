> **一句话总结：** ReLibra利用RL训练中rollout阶段的token-to-expert路由信息（routing replay），通过跨batch的swap-based模拟退火专家重排序和batch内的MILP优化动态专家复制（layer-shared replica buffer实现零额外显存），在20节点Hopper集群上将训练吞吐量提升至Megatron-LM的1.58x、EPLB（oracle loads）的1.21x，达到理想均衡基线的90%-94%。

> **TL;DR:** ReLibra exploits routing replay in RL's rollout-training loop to know exact expert loads before training starts. It decomposes load balancing into inter-batch expert reordering (swap-based simulated annealing) and intra-batch expert replication (MILP with layer-shared replica buffer at 0.13%-1.62% memory overhead), achieving 1.21-1.58x over Megatron-LM and 90-94% of ideal balanced throughput on Qwen3-30B/106B/235B and GLM4.5-106B models.

## 核心贡献

- 提出routing replay机制：利用RL中rollout和训练处理相同token且使用相同MoE参数的特点，在训练开始前精确获知每个micro-batch的token-to-expert路由信息，无需依赖历史负载预测。求解过程与post-rollout处理（reward计算、KL散度评估）重叠，不占训练关键路径。
- 设计两层负载均衡策略：跨batch粒度使用swap-based模拟退火算法进行专家重排序（联合优化计算与通信负载，含数据局部性优化）；batch内粒度使用MILP建模+增量贪心启发式算法进行动态专家复制与token分配。
- 提出layer-shared replica buffer设计：每GPU仅分配2个replica slot，跨所有MoE层复用，额外显存仅占总参数量的0.13%-1.62%，同步操作通过异步TMA kernel与attention计算重叠。
- 在三个不同厂商/规模的MoE模型（Qwen3-30B/106B/235B, GLM4.5-106B）和四个RL领域（reasoning, coding, instruction following, mixed）上验证，训练吞吐量达到理想均衡基线的90%-94%。

## Key Contributions

- **Routing replay insight:** In RL's rollout-training pipeline, both stages process identical tokens with identical MoE parameters, so token-to-expert routing is fully determined during rollout. This eliminates the prediction problem entirely. The optimization is overlapped with post-rollout processing (reward computation, KL divergence), hiding solver latency.
- **Timescale decomposition:** Expert reordering operates at inter-batch timescale (per training batch) via swap-based simulated annealing with LSE-smoothed objective for cross-node balance; expert replication operates at intra-batch timescale (per micro-batch) via MILP-formulated incremental greedy heuristic for within-node fine-grained adaptation.
- **Layer-shared replica buffer:** Each GPU allocates only r replica slots (r=2 in experiments), reused across all MoE layers. Memory overhead is 0.13%-1.62% of total parameter memory. Replica synchronization uses a persistent async TMA kernel on NVIDIA Hopper, limited to few SMs to avoid contention with attention.
- **Communication-aware formulation:** Jointly models computation and all-to-all communication loads under rail-optimized topology (NVLink, same-rail RDMA, cross-rail RDMA), capturing realistic bandwidth asymmetry. Also optimizes data locality via a second round of sample-placement simulated annealing.
- **Strong baselines:** Compares against Megatron-LM (with DeepEP and GroupedGEMM), PopFetcher, EPLB+ (with oracle loads from routing replay), and LPLB+ (with oracle loads). All methods use the same layer-shared buffer and synchronization for fair comparison.

## Methodology

ReLibra is implemented as ~14K lines of Python/C++/CUDA, built on Slime (RL orchestration), Megatron-LM (MoE training), and SGLang (rollout backend).

**Problem formulation (Section 4.1):** For each MoE layer, the system models both computation load (token count per GPU) and communication load across three patterns in rail-optimized topology: intra-node NVLink, inter-node same-rail RDMA, and inter-node cross-rail RDMA with NVLink relay. The objective is to minimize total MoE execution time T_MoE = max_g(T_g^comp) + max_g(T_g^comm), considering both send and receive directions on NVLink and RDMA links.

**Inter-batch expert reordering (Section 4.2):**
1. Initializes with greedy LPT (Longest Processing Time) placement.
2. Applies swap-based simulated annealing with multiple parallel seeds. Key techniques: (a) incremental state update (O(|G|) per swap by rescanning only swapped experts' routing), (b) LSE smoothing of max operators (beta=20) to provide gradient signal on near-bottleneck GPUs, (c) second round of sample-swap annealing for data locality optimization under data-parallel attention.

**Intra-batch expert replication (Section 5):**
1. Formulated as MILP with binary placement variables z_{e,g} and continuous token-splitting variables y_{j,e,g}, minimizing T_MoE subject to token completeness, placement-token coupling, home GPU fixing, and per-GPU replica buffer constraints.
2. Solved via incremental greedy heuristic: iteratively identifies bottleneck GPU and expert, places replica on least-loaded feasible GPU, re-solves token-splitting LP. Time complexity is bounded by replica slot count.

**Layer-shared replica buffer:** Forward pass pushes expert parameters to destination GPUs before expert computation; backward pass aggregates replica gradients back. All confined to intra-node, using high-bandwidth NVLink. Synchronization overlapped with attention via limited-SM TMA kernel.

## Results & Highlights

- **Training throughput (mixed dataset):** 1.21-1.58x over Megatron-LM, 1.17-1.27x over PopFetcher, 1.06-1.21x over EPLB+ and LPLB+ across three model scales (30B to 235B).
- **Near-optimal:** ReLibra achieves 90%-94% of Balanced baseline throughput across all configurations.
- **Ablation (Qwen3-235B-A22B):** Inter-batch reordering alone gives 12-15% speedup; adding intra-batch replication gives additional 24-34% speedup. The two mechanisms are synergistic.
- **Inter-batch reordering:** Swap-based SA achieves 1.05-1.16x over greedy LPT baselines, demonstrating value of global optimization.
- **Intra-batch replication:** Holistic MILP approach outperforms separate placement-only (1.12-1.24x) and splitting-only (1.09-1.12x) optimizations.
- **Skewness:** ReLibra maintains rank-level skewness of 1.00-1.08 across EP sizes 8-64 (vs. much higher for Megatron-LM), and 1.02-1.07 across all datasets.
- **Overhead:** Solver time fully hidden behind post-rollout processing. Expert reordering overhead: 1.4-2.1% of training batch time. Replica buffer memory: 0.13-1.62% of parameter memory. Replica synchronization fully overlapped with attention.

## Critical Assessment

- **Novelty (High):** The routing replay insight is genuinely novel and elegantly exploits a structural property of RL training. The timescale decomposition (inter-batch reordering + intra-batch replication) is well-motivated by network topology asymmetry. The LSE-smoothed SA objective and the MILP formulation with incremental greedy heuristic are technically sound contributions.
- **Rigor (High):** Experimental design is thorough. EPLB+ and LPLB+ are strengthened with oracle loads from routing replay, which isolates ReLibra's advantage to the *fine-grained temporal adaptation* rather than just information access. All baselines share the same system infrastructure (layer-shared buffer, synchronization) for fair comparison. Three models from different vendors, four RL domains, and multiple EP sizes provide broad coverage.
- **Reproducibility (Moderate-High):** Implementation is detailed (14K lines, specific frameworks named, algorithms pseudocoded). Hardware configuration is fully specified. However, the system requires a 20-node Hopper cluster with rail-optimized InfiniBand and a separate 128-core CPU node for the solver, which limits who can reproduce.
- **Significance (High):** Addresses a real and growing bottleneck. As MoE+RL training (GRPO, RLHF) becomes the standard post-training pipeline, this work has immediate practical impact. The 1.21-1.58x throughput gains are substantial for production training costs.
- **Potential concerns:** (1) Applicable only to RL training; the routing replay property is fundamentally specific to the rollout-training paradigm. The paper acknowledges this but does not explore generalization to pretraining. (2) Off-policy RL algorithms that reuse data across multiple updates or operate asynchronously break the routing replay assumption — the paper discusses this briefly but relies on industrial practice adopting routing replay anyway. (3) The solver runs on a separate CPU node with 128 cores; this is additional infrastructure that may not always be available. (4) Only 2 replica slots per GPU were tested; the sensitivity to this parameter is not explored. (5) The "Balanced" baseline randomly reroutes tokens and does not preserve model semantics, making it a loose upper bound.

## Strengths

- Clean, elegant core insight (routing replay) with solid systems engineering to exploit it.
- Comprehensive evaluation: 3 models (30B-235B), 4 RL domains, multiple baselines including oracle-strengthened ones.
- Thorough ablation quantifying contributions of each component.
- Layer-shared replica buffer is a simple but effective design that makes micro-batch-level replication practical.
- Joint optimization of computation and communication (including rail-optimized topology modeling) goes beyond prior work that only considers computation load.
- All overheads quantified and shown to be negligible or hidden.

## Limitations

- Fundamentally limited to RL training scenarios where rollout and training share the same routing. Not applicable to MoE pretraining or inference.
- Only 2 replica slots per GPU tested; scalability to more aggressive replication is unexplored.
- Requires a separate CPU node (128 cores) for the solver. For smaller teams, this may be a practical barrier.
- Off-policy RL with stale parameters may break routing replay assumptions. The paper notes industrial adoption of routing replay but does not quantify degradation when routing diverges.
- The mixed workload follows DeepSeek-V3.2's recipe; results may differ for other RL training configurations (different batch sizes, micro-batch counts, or reward functions).
- No evaluation of end-to-end RL training convergence/quality — only training throughput is measured.

## Relevance

该工作直接针对MoE模型在RL后训练（GRPO/RLHF）中的负载均衡瓶颈，对于大规模LLM后训练的系统优化具有直接的工程参考价值。如果你关注分布式训练系统优化、MoE在RL场景中的部署效率，或正在做RLHF/GRPO的大规模训练基础设施，这篇论文的系统设计和优化思路值得深入研读。

## Questions for Further Exploration

- How does the number of replica slots per GPU affect throughput gains? Is there a diminishing-returns point or an optimal configuration for different model scales?
- What happens to routing replay accuracy when off-policy RL causes parameter drift between rollout and training? Can a lightweight routing correction mechanism compensate?
- Can the solver be made fast enough to run on-GPU (e.g., using CUDA-based LP solvers) to eliminate the need for a separate CPU node?
- How does ReLibra interact with emerging MoE communication optimizations like DeepEP's hierarchical all-to-all or Comet's computation-communication overlapping?
- Does the expert reordering plan exhibit temporal stability across RL iterations, potentially allowing amortized planning across multiple training steps?
