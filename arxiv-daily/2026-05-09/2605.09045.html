Containment Verification: AI Safety Guarantees Independent of Alignment

Royce Moon & Lav R. Varshney · 2026-05-09 · cs.AI, cs.CR, cs.SE

Key Finding
First deductive formal verification of an agentic framework. Safety guarantees are invariant to AI model capability under havoc oracle semantics.
Key Metric
~100lines of PocketFlow verified · 1.2–2.5s per-path Dafny proof
Why This Matters
Safety becomes a property of deterministic code, not learned behavior. The guarantee holds no matter how capable or adversarial the AI becomes.
Critical Edge
Demonstrated only on a ~100-line minimalist framework. The gap to production systems like LangChain is enormous and unaddressed.

Approach

Containment verification places safety guarantees at the agentic framework layer rather than in the AI model. The AI is modeled as a havoc oracle — a Dafny method with an empty body that may return any value of the typed Action enum. The verified containment layer must enforce the boundary policy for every possible output.

The soundness theorem (Theorem 3.2) proves that if abstract havoc safety and boundary-event refinement hold, every trace of the concrete implementation satisfies the policy at every step — regardless of oracle behavior. The proof is mechanized in Dafny using the IronFleet refinement idiom.

Agentic Synthesis Pipeline

A seven-phase PocketFlow graph takes a Flow file as input and emits a machine-checked Dafny artifact. An information barrier prevents the LLM from seeing implementation context during property derivation, ruling out tautological specifications by construction.

2InterfaceContract
3GenerateSpec
4ValidateSpec
5Verify

7ProofRepair

Uses Claude Sonnet 4.6 for codebase exploration, Claude Opus 4.7 for synthesis and reasoning. Total token usage: 1.6M–3.4M per run.

Three-Gate Validation

G1
Resolution
Synthesized Dafny must parse and type-check within a 30-second timeout.
G2
Vacuity
A permissive-stub mutation that guts the inductive invariant must fail verification. If it passes, the specification cannot demand what refinement requires.
G3
Discrimination
An LLM-generated mutation introducing a plausible modeling error must fail. If it passes, the spec cannot distinguish faithful from faulty refinements.

All three gates are necessary but not sufficient. A property-completeness gap exists when the template schema lacks a field for a Flow's privileged effect.

Strengths

  • Guarantee is a property of deterministic code, not of learned behavior — structurally distinct from alignment
  • Havoc oracle semantics cleanly separates verification from model behavior
  • Mechanized proofs in Dafny following IronFleet methodology — gold standard rigor
  • Unusually transparent about limitations, trusted base, and failure modes
  • Guarantee-strength / locus taxonomy (Table 1) is a clean conceptual contribution
  • Correctly positioned as complementary to alignment and sandboxing, not alternative

Limitations

  • PocketFlow is ~100 lines — scalability to LangChain-scale frameworks is unaddressed
  • Effect exclusivity assumption is fragile in Python (dynamic imports, FFI, ambient I/O)
  • Three-gate validation has a known property-completeness gap for schema mismatches
  • LLM-generated specifications introduce bootstrapping concerns: who verifies the specs?
  • Verifies the dispatch loop, not end-to-end system behavior or tool implementations

Open Questions

  • How does verification complexity scale with action-space richness in production frameworks?
  • Can the approach handle dynamic action spaces with runtime tool registration?
  • How to systematically address the property-completeness gap beyond three gates?
  • What is the trusted computing base size for a production deployment?
  • How does the guarantee interact with multi-agent shared-state communication?
  • What are the practical latency overheads of the verified containment layer?
