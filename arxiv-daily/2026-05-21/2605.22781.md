> **一句话总结：** DeltaBox 通过 DeltaFS（动态叠加层文件系统）和 DeltaCR（增量进程快照+模板 fork 恢复）两个 OS 级协同机制，将 AI 智能体沙箱的检查点/回滚延迟降至毫秒级（14ms 检查点、5ms 恢复），使 MCTS 搜索的状态管理开销从 47–77% 降至 3–6%。

> **TL;DR:** DeltaBox introduces DeltaFS (runtime hot layer switching over OverlayFS) and DeltaCR (incremental CRIU dumps + frozen-template fork restores) to achieve millisecond-level coupled filesystem+memory checkpoint/rollback for AI agent sandboxes, reducing state-management overhead from 47–77% to 3–6% in MCTS workloads and enabling near-saturating GPU utilization in RL training fan-outs.

## 核心贡献

- 提出核心观察：AI 智能体连续检查点之间高度相似，只需复制变化量而非全量状态。
- 设计 **DeltaFS**：基于 OverlayFS 扩展，支持运行时热层切换（无需卸载），通过检查点代数计数器实现跨检查点的惰性文件描述符重定向，配合 XFS reflink 实现 4KB 粒度的写放大控制。
- 设计 **DeltaCR**：双路径进程状态管理——检查点时异步增量 CRIU 转储 + 同步模板 fork，恢复时优先从冻结模板 fork（~5ms），模板被回收时回退到 CRIU lazy-pages 恢复（~8ms）。
- 设计 **Network Proxy Daemon (NPD)**：将 LLM SDK 网络连接从智能体进程地址空间解耦，确保模板可安全 fork。
- 在 SWE-bench 和 RL 微基准上实现毫秒级 C/R，状态管理开销占比从基线的 47–77% 降至 3–6%。

## Key Contributions

- Identifies the key insight that consecutive checkpoints in AI agent workloads are highly similar, motivating change-based (delta-based) state management instead of full duplication.
- Designs **DeltaFS**, a runtime-reconfigurable OverlayFS extension with hot layer switching (no unmount required), per-inode generation counters for lazy file descriptor redirection across checkpoints, and XFS reflink integration for 4 KB granularity write amplification.
- Designs **DeltaCR**, a dual-path process state manager: asynchronous incremental CRIU dumps (hidden in LLM I/O window) + synchronous frozen-template fork at checkpoint; fast-path restore via fork() from template (~5 ms), slow-path via CRIU lazy-pages (~8 ms).
- Introduces the **Network Proxy Daemon (NPD)** to decouple LLM SDK network threads from the agent's address space, making templates safely forkable.
- Introduces a **bounded template pool** with LRU eviction and reachability-aware MCTS garbage collection, ensuring memory overhead stays capped while correctness is preserved.
- Achieves 14 ms mean checkpoint and 5 ms mean restore on SWE-bench workloads, reducing state-management overhead from 47–77% to 3–6% compared to coupled-FS baselines.

## Methodology

DeltaBox is built on a four-layer architecture atop Firecracker microVMs:

1. **Base Storage (Layer 1):** XFS with reflink enabled, providing block-level CoW so that copy-up operations create shared extent references rather than duplicating data blocks.
2. **DeltaFS (Layer 2):** A modified Linux 6.8 OverlayFS kernel module with a custom ioctl for atomic layer-stack reconfiguration at runtime. On checkpoint, the writable upper layer is frozen (renamed to a read-only lower) and a fresh upper layer is inserted. A per-filesystem `checkpoint_gen` counter enables lazy re-resolution of open file descriptors: on write, the kernel checks the cached generation against the current generation and re-resolves the inode if stale. XFS reflink ensures that CoW copy-up operates at 4 KB block granularity.
3. **DeltaCR (Layer 3):** Builds on CRIU for incremental process dumps using soft-dirty page tracking. At each checkpoint, two operations run concurrently: (a) an asynchronous CRIU incremental dump to tmpfs, and (b) a template-creating fork() where the parent self-suspends (SIGSTOP) as a frozen template and the child becomes the new active agent. On restore, if the target template is alive, fork() it (~3.75 ms); if evicted, fall back to CRIU lazy-pages restore (~7.25 ms). An async-warm thread pre-absorbs CoW page faults off the critical path.
4. **Search Strategy (Layer 4):** The StateManager coordinates Layers 2–3 to maintain consistent (filesystem, memory) state pairs. It supports Linear, Best-of-N, and MCTS search patterns.

The **Network Proxy Daemon (NPD)** runs LLM SDK clients in a separate process, communicating with the agent via FIFOs and shared memory, keeping the agent's address space free of persistent socket state. A **semantic classifier** skips checkpoints for read-only commands (62% of events in production MCTS runs). **Inference-masked checkpointing** hides the ~14 ms checkpoint cost inside the LLM round-trip window.

## Results & Highlights

- **Checkpoint latency:** 14.57 ms mean (DeltaFS ioctl ~1.12 ms + CRIU dump + fork ~13.45 ms, hidden in LLM I/O window).
- **Restore latency:** 5.14 ms mean fast path (DeltaFS switch 1.66 ms + template fork 3.75 ms); 8.04 ms slow path (CRIU restore).
- **vs. baselines:** 1–3 orders of magnitude faster than copytree+replay, Docker commit+restart, Firecracker VM snapshot, and CubeSandbox across all percentiles.
- **End-to-end MCTS:** On 100-iteration MCTS trajectories (Qwen3-Coder-30B), DeltaBox runs at 1.03–1.06x the LLM-only floor; Firecracker+dm-snapshot runs at 1.87–3.84x; cloud-hypervisor+dm at 2.62–4.29x.
- **SWE-bench MCTS lift:** MCTS search improves pass rates by +2.2pp to +5.9pp over linear ReAct across Claude Sonnet 4.6, MiMo V2.5-Pro, and Qwen3-Coder-30B.
- **RL training fan-out:** DeltaBox achieves near-saturating GPU utilization (93–96% at N=64) vs. 51–63% for Docker/copytree. Staleness stays below the verl threshold at production scales.
- **Write amplification:** Bounded to O(4KB) per edit via XFS reflink, compared to O(directory) for copytree and O(VM image) for Firecracker.
- **Lightweight skip:** 62% of checkpoint events are classified as read-only and elided entirely, reducing tmpfs pressure.
- **GC effectiveness:** Reachability-aware GC reduces end-of-trajectory dump storage by 46–63%.

## Critical Assessment

- **Novelty (High):** The key insight of change-based (delta-based) coupled C/R for agent sandboxes is well-motivated. DeltaFS's runtime hot layer switching without unmount is a genuine OS-level contribution. The dual-path DeltaCR design (template fork + CRIU fallback) with NPD decoupling is novel and well-engineered. The reachability-aware GC for MCTS is a clean solution to a real problem.
- **Rigor (High):** The evaluation covers 9+ SWE-bench repositories, 5 baselines (copytree, Docker, Firecracker, cloud-hypervisor, CubeSandbox), and both MCTS and RL workloads. Component-level breakdowns (Table 4) enable understanding of where time is spent. The inference-masking argument is validated with real LLM round-trip measurements.
- **Reproducibility (Medium):** Built on open-source components (Linux kernel, CRIU, Firecracker, OverlayFS), but requires a custom kernel module (DeltaFS) and non-trivial integration. No open-source release is mentioned in the paper. The microbenchmark setup is well-specified (hardware, VM config, model versions).
- **Significance (High):** This work addresses a real and growing infrastructure bottleneck as agent systems increasingly adopt MCTS and RL training. The 10–100x latency reduction directly enables deeper search and higher GPU utilization. The OS-level abstractions (DeltaState, DeltaFS hot-switch) have potential applicability beyond agent sandboxes.
- **Red flags:** (1) The paper does not mention an open-source release plan, which would significantly limit adoption. (2) The NPD introduces a single point of failure for LLM connectivity within the sandbox. (3) Network I/O rollback is explicitly not supported, which may limit applicability to agents that make external API calls beyond LLM inference. (4) The evaluation uses relatively small agent processes (~15 MB RSS); scaling to agents with large in-memory state (e.g., code indexing, retrieval caches) is not explored.

## Strengths

- Clean OS-level design that exploits a fundamental property of agent workloads (high checkpoint similarity).
- The inference-masked checkpointing insight effectively hides CRIU dump cost behind LLM round-trips.
- Dual-path restore (template fork vs. CRIU fallback) provides consistent low latency while gracefully handling memory pressure via bounded template pool.
- Comprehensive evaluation with strong baselines and real-world workloads (SWE-bench MCTS, RL training fan-out).
- The semantic classifier for lightweight skip (62% of events) is a practical optimization that reduces unnecessary work.
- The reachability-aware GC design is provably safe for MCTS selection semantics.

## Limitations

- No open-source release mentioned; the custom kernel module (DeltaFS) is a significant barrier to adoption.
- Network I/O rollback is not supported; external side effects from non-LLM API calls cannot be undone.
- The NPD does not handle LLM SDK features that require streaming responses or complex authentication flows beyond basic HTTP/2.
- Template pool sizing is a new tuning parameter that users must configure; the paper provides limited guidance on optimal pool size.
- Evaluation focuses on coding agents (SWE-bench); applicability to other agent domains (web navigation, robotics) with different state profiles is not demonstrated.
- The Firecracker microVM substrate adds overhead compared to container-based alternatives for simple workloads.

## Relevance

这篇论文与分布式训练鲁棒性和系统可靠性高度相关。DeltaBox 解决的核心问题——AI 智能体在高频状态探索中的检查点/回滚瓶颈——直接关系到 MCTS 搜索和 RL 训练的基础设施可靠性，其 OS 级抽象设计对理解 LLM 训练和推理系统的底层行为有参考价值。

## Questions for Further Exploration

- How does DeltaBox scale with agent processes that maintain large in-memory indices (e.g., RAG retrieval caches of hundreds of MB)?
- What is the interaction between DeltaFS's hot layer switching and concurrent multi-agent sandbox sharing the same base filesystem image?
- Can the DeltaState abstraction be extended to support network I/O rollback (e.g., via request replay or idempotency tokens)?
- What is the optimal template pool size as a function of MCTS branching factor and search depth?
- How does DeltaBox perform under memory pressure when multiple concurrent MCTS trajectories compete for the same physical RAM?
- Could the DeltaFS layer-switching mechanism benefit container-based sandboxes (e.g., Docker) without the Firecracker microVM overhead?
