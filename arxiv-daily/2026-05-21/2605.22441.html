A Constant-Time Implementation Methodology for Activation Functions on Microcontrollers
Andrii Tyvodar, Andreas Rechberger, Dirmanto Jap, Shivam Bhasin et al. · 2026-05-21
side-channel-attacks embedded-inference constant-time activation-functions hardware-security
Key Metric
88 cycles (3-function) / 108 cycles (5-function) — identical execution time for all activations on ARM Cortex-M4, regardless of input.
Why This Matters
Template attacks break desynchronization defenses with only ~2,000 measurements, proving that random delays are insufficient for embedded AI security.
Critical Edge
Protected sigmoid and tanh are 60–78% faster than standard math library versions, with sigmoid MSE < 3×10¹² — security and speed gains are simultaneous.

Methodology

The methodology combines four tightly integrated components to ensure input-independent execution time on ARM Cortex-M4 (STM32F411, 84 MHz). First, branchless masking-based selection uses bit-level masks on IEEE-754 floats instead of conditional branches, eliminating data-dependent control flow. Second, fixed-cost Padé rational approximation shares a single R_tanh(x) = x·P(x²)/Q(x²) across all nonlinear activations, requiring a fixed sequence of arithmetic operations. Third, masking-based saturation handles large-magnitude inputs via the same branchless technique, with mathematically derived thresholds. Fourth, cycle alignment with dummy arithmetic pads simpler functions (ReLU) with discarded R_tanh computation plus minimal NOP padding to match the common budget. The implementation is pure C++, verified under -O1, -O2, and -O3, with timing measured via the DWT cycle counter.

Strengths

  • Clean decomposition into independently verifiable components
  • Protected nonlinear activations are faster than standard library versions
  • Timing regularity verified over 500,001 input values with 5 repetitions
  • Template attack on desynchronization provides compelling motivation
  • Open-source implementation at github.com/andrewtyv/activation_timing

Limitations

  • Evaluated on only one hardware platform (ARM Cortex-M4)
  • Constant-time guarantee is empirical, not formally proven
  • Power and EM side channels remain unaddressed
  • 12× ReLU overhead is nontrivial for activation-heavy networks
  • Requires careful compiler management that may be fragile in large projects

Questions for Further Exploration

  1. How does the methodology scale to quantized (int8) activation functions for edge deployment?
  2. Can the approach protect entire inference pipelines under a unified constant-time framework?
  3. What is the end-to-end performance impact on realistic models (MobileNet, TinyML workloads)?
  4. Would combining with power/EM countermeasures provide multi-channel protection?
  5. Can formal verification tools (ct-verif, dudect) provide machine-checked guarantees?
