> **一句话总结：** 本文提出 Life-Harness，一种生命周期感知的运行时接口适配框架，通过环境契约、过程技能、动作实现和轨迹调控四层机制，在不修改模型权重的前提下，让冻结的 LLM 智能体在确定性任务环境中平均提升 88.5%，且从一个小型模型演化出的 harness 能跨模型迁移到 17 个不同骨干网络。

> **TL;DR:** Life-Harness improves frozen LLM agents by adapting the runtime interface (not model weights) across four lifecycle layers, achieving 88.5% average relative improvement across 7 benchmarks and 18 model backbones, with strong cross-model transfer from a single 4B source model.

## 核心贡献

- 提出了 **运行时接口适配** 的新范式：不修改模型权重或评测环境，仅通过适配 LLM 与环境之间的运行时 harness 来提升智能体表现
- 设计了 **四层生命周期架构**：环境契约层（Environment Contract）、过程技能层（Procedural Skill）、动作实现层（Action Realization）和轨迹调控层（Trajectory Regulation），分别解决不同阶段的接口失败
- 证明了 **跨模型迁移性**：仅用 Qwen3-4B-Instruct 的训练轨迹演化出的 harness，可以迁移到 17 个不同模型（包括 Llama、xLAM 系列），116/126 个设置获得提升
- 展示了 **训练与 harness 互补**：harness 可以超越专门的工具使用训练（xLAM-2-32B 在 τ-bench 上被 base 模型+harness 反超 7.5 个百分点），且训练后模型仍能从 harness 中获益

## Key Contributions

- Introduces **runtime interface adaptation** as a complementary alternative to model-centric agent training — adapting the model-environment boundary rather than model parameters
- Designs a **four-layer lifecycle harness**: Environment Contract (pre-interaction constraints), Procedural Skill (task-conditioned skill retrieval), Action Realization (pre-execution validation/canonicalization), and Trajectory Regulation (post-execution monitoring/recovery)
- Demonstrates **strong cross-model generalization**: a harness evolved solely from Qwen3-4B-Instruct trajectories transfers to 17 other backbones (instruct, reasoning, and agent-trained models), improving 116 out of 126 model-environment settings
- Shows that **harnessing outperforms specialized tool-use training** on in-domain tasks and remains beneficial even after training, while specialized training can hurt OOD generalization

## Methodology

Life-Harness follows a lifecycle-aware design with four layers, each targeting a distinct failure category identified through trajectory analysis:

1. **Environment Contract Layer** — Injects stable tool constraints, policy rules, and answer-format requirements into the prompt *before* interaction begins. For example, informing the agent that cancellation only applies under four specific conditions, or that SQL results must follow a specific format.

2. **Procedural Skill Layer** — Maintains a skill library distilled from training trajectories, retrieved via BM25 ranking against the current task. Skills encode reusable procedural patterns (e.g., "check overdue bills → send payment request → resume line → reboot" for telecom service recovery). Only the top-1 skill is injected to avoid context contamination.

3. **Action Realization Layer** — Validates and canonicalizes model-generated actions *before* environment execution. Catches invalid tool calls, malformed arguments, wrong tool choices, and semantically incorrect formats. Also rescues actions that the model writes as plain text instead of structured tool calls.

4. **Trajectory Regulation Layer** — Monitors post-execution trajectories for degeneration patterns: repeated actions, oscillation, budget exhaustion, no-progress feedback. Appends diagnostic information to observations and proposes recovery actions.

**Evolution process**: A coding agent (Codex) iteratively inspects failed training trajectories, annotates failure categories (action realization > contract mismatch > trajectory degeneration > residual reasoning), and implements targeted harness updates. The process converges in ~5 iterations. The harness is frozen after evolution and applied unchanged during evaluation.

**Key constraint**: The harness must not use test labels, modify benchmark tasks, alter environment transitions, or change evaluation criteria.

## Results & Highlights

- **88.5% average relative improvement** across 7 benchmarks (Airline, Retail, Telecom from τ-bench/τ²-bench; ALFWorld, WebShop, OS, DBBench from AgentBench) and 18 model backbones
- **116/126 model-environment settings** improved; no single model or environment is left worse off on average
- **Relative gains range 10–84%** depending on benchmark; smaller models become competitive with much larger ones
- **Cross-model transfer**: Harness evolved from Qwen3-4B generalizes to Qwen, Llama, and xLAM families (including reasoning models like DeepSeek-R1-distilled and agent-trained xLAM models)
- **Harness vs. training**: Qwen2.5-32B + Life-Harness beats xLAM-2-32B (specialized tool-use training) by 7.5pp on τ-bench in-domain; applying harness to xLAM adds another 6.8–28.9pp across OOD benchmarks
- **Harness vs. prompt evolution**: Life-Harness achieves 120% more relative improvement than prompt-only evolving methods, confirming that interface adaptation goes beyond prompt optimization
- **Ablation**: All four layers are necessary — removing any layer causes substantial drops on at least some benchmarks (e.g., removing Trajectory Regulation causes 86.5% relative drop on DBBench)

## Critical Assessment

**Novelty: High.** The core insight — that many agent failures stem from model-environment interface mismatches rather than model capability deficits — is well-argued and distinct from existing model-centric or prompt-centric approaches. The four-layer lifecycle design is a principled decomposition of the interface problem.

**Rigor: Strong.** The paper provides extensive ablations (leave-one-layer-out), cross-model transfer experiments, comparisons with both prompt evolution and specialized training, and per-benchmark per-model results. The failure annotation protocol (priority-based classification) is clearly defined.

**Reproducibility: Good.** Code is released. The methodology is deterministic (temperature 0.0). However, the harness evolution relies on Codex as a coding agent, which introduces some variability. The full harness inventory for each environment is provided in appendices.

**Significance: High.** This work challenges the assumption that agent improvement requires model training. The cross-model transfer result is particularly impactful — it suggests that environment-side structure can be decoupled from model-specific behavior.

**Red flags:**
- The harness evolution process depends on a proprietary coding agent (Codex) and is not fully automated — it involves manual inspection of iteration results
- The scope is limited to deterministic, rule-governed environments; extending to open-ended tasks is acknowledged as future work
- Some harness components (e.g., the extensive action validation rules in Airline/Retail) are highly environment-specific and may require substantial engineering effort per new environment
- The 88.5% relative improvement metric can be misleading for low baselines (e.g., going from 2% to 4% is a 100% relative gain)

## Strengths

- Clean problem formulation: separating interface adaptation from model training
- Strong empirical results with broad coverage (18 models × 7 environments)
- The four-layer decomposition is both principled and practically effective
- Cross-model transfer from a single small model is a compelling result
- Harness can outperform and complement specialized agent training
- Detailed appendix with full harness inventory enables replication

## Limitations

- **Environment-specific engineering**: Each new environment requires building a tailored harness, though the evolution process automates much of this
- **Deterministic-only scope**: Does not address open-ended or non-deterministic agent environments
- **Evolution depends on training trajectories**: Quality of the harness is bounded by the coverage of failure modes in the training set
- **Codex dependency**: The evolution pipeline uses a proprietary coding agent; replacing it may affect harness quality
- **Not a replacement for model improvement**: Harnessing addresses interface failures but cannot fix fundamental reasoning deficits

## Relevance

本文与 LLM 智能体的可靠性高度相关：Life-Harness 通过运行时接口适配提升确定性环境中智能体的成功率，本质上是一种面向模型-环境交互的鲁棒性工程方法，涉及轨迹监控、异常检测和失败恢复，与训练稳定性/可靠性研究有方法论上的共通之处。

## Questions for Further Exploration

- Can the harness evolution process be made fully autonomous without relying on a separate coding agent?
- How does harness quality scale with the number of training trajectories — is there a diminishing returns point?
- Could Life-Harness be combined with model-level improvements (e.g., RL fine-tuning) for additive gains in non-deterministic settings?
- What is the cost of adapting a harness to a new environment compared to training a specialized agent model?
- Do the four failure categories generalize to multi-modal or embodied agent environments?
