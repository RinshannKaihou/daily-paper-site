> **一句话总结：** 提出分层KV缓存架构，在GPU存储INT8键/INT4值的同时将FP16原始数据保留于系统RAM，通过在线逐头逐步计算两项量化误差界（注意力分布失真+值重建误差），驱动自适应精度选择与四级回退阶梯，在PG-19/NIAH/RULER基准上恢复朴素量化导致的灾难性检索失败，代价是2.7–4.8倍的延迟开销。

> **TL;DR:** A single-author work presenting a tiered KV cache that stores INT8 keys (per-channel) and INT4 values (group=16) in GPU VRAM while retaining FP16 originals in pinned system RAM. A formal two-term error decomposition provides per-head, per-step bounds on attention distribution distortion (from key quantization) and value reconstruction error. An adaptive top-K selector promotes high-mass blocks to FP16 keys at each decode step, and a four-rung fallback ladder guarantees exact FP16 recovery when bounds are violated. On LLaMA 3.1-8B (8K–128K context), the system matches dense FP16 quality on PG-19 perplexity (Δppl ≤ 0.001) and NIAH retrieval (McNemar p=1.0 at 8K/64K). A naive INT8/INT4 baseline catastrophically fails on retrieval (NIAH drops to 5–10%), confirming the certification machinery is essential, not just overhead. The cost: 2.7–4.8× decode latency vs. dense Flash Attention, with meaningful VRAM savings only at 64K+ context. The certification is local (per-head, per-step) and does not guarantee end-to-end model correctness.

## 核心贡献

- **分层KV缓存架构**：Tier-1在GPU存储INT8键（逐通道量化）和INT4值（分组g=16量化），占FP16的56% VRAM；Tier-2在系统RAM保留完整FP16 KV用于确定性回退。
- **两项误差分解**：独立界定键量化导致的注意力分布失真（基于Total Variation距离的E_key界）和值量化导致的重建误差（E_val界），可在线逐头逐步计算，互不耦合。
- **自适应精度选择**：每步解码根据实际注意力模式选择top-K高权重块升级至FP16键（覆盖率阈值τ_cov=0.995），并按局部值误差贡献(ρ̂_b·η_b)选择性升级FP16值。
- **四级回退阶梯**：Rung 1（K*扩展至2·K_max）、Rung 2（FP16值升级）、Rung 3（逐头FP16全计算）、Rung 4（全头FP16全计算），最终调用与Dense基线相同的`torch.scaled_dot_product_attention`代码路径。
- **排序一致性检查**：检测INT8评分噪声导致的注意力分布失真和注意力稀释效应，在64K NIAH上边界检查触发率为0.7%，逐头回退仅占步时间的1.7%。
- **将KV缓存量化重构为运行时可验证计算**：每个注意力计算要么相对于FP16参考有界，要么通过回退精确恢复——不再是固定近似。

## Key Contributions

- **Tiered KV cache**: INT8 keys (per-channel) + INT4 values (g=16) in GPU VRAM (56% of dense); full FP16 originals in pinned system RAM for deterministic fallback.
- **Two-term online error decomposition**: Independent per-head, per-step bounds on (i) attention distribution distortion from key quantization (E_key, via Total Variation distance) and (ii) value reconstruction error (E_val). The two bounds have separate monitors and escalation paths.
- **Adaptive top-K precision selector**: Promotes high-softmax-mass blocks to FP16 keys based on actual attention patterns at each decode step (coverage threshold τ_cov=0.995, K_max=128).
- **Four-rung fallback ladder**: Rung 1 (K* expansion to 2·K_max) → Rung 2 (FP16 value escalation) → Rung 3 (per-head dense FP16 recomputation) → Rung 4 (all-head dense recomputation via `torch.scaled_dot_product_attention`).
- **Ranking-consistency check**: Detects when INT8 scoring noise swaps block rankings at the FP16/INT8 boundary. At 64K, boundary corrections fire on 0.7–8.0% of head-steps; per-head Rung 3 fallback adds only 1.7% of step time.
- **Runtime-verified computation framing**: Every attention output is either certified within bound relative to FP16 reference or exactly recovered via fallback—reframing KV cache quantization from fixed approximation to verifiable computation.

## Methodology

**Tiered storage architecture.** Tier-1 stores INT8 per-channel keys and INT4 group-quantized (g=16) values in VRAM with associated metadata (scales, offsets, per-block η_b annotations). Tier-2 retains full FP16 originals in pinned system RAM over PCIe 5.0. An FP16 scratch buffer (2048 blocks, ~2.3–4.3 GB) in VRAM caches recently promoted blocks to avoid repeated host-to-device transfers.

**Two-term error decomposition (Section 4).** The output perturbation ‖O_quant − O_ref‖ is bounded by two independent terms:
- *Key error (E_key)*: Bounded via Total Variation distance on the softmax distribution. Theorem 4.4 gives E_key ≤ 2·V_max·e^{2Δ}·α̂_T·(e^{2Δ} − 1), where α̂_T is the estimated INT8 tail mass and Δ is the per-channel quantization bound. The implementation uses conservative e^{3Δ} inflation to account for INT8×INT8 tensor-core scoring.
- *Value error (E_val)*: Bounded by Σ_b ρ_b·η_b (Corollary 4.2), where ρ_b is the block's attention mass and η_b is its maximum per-token INT4 reconstruction error, computed at quantization time.

**Adaptive top-K selector.** At each decode step, Phase 1 scores all blocks using INT8 keys to estimate block softmax masses. The selector promotes blocks in descending mass order until estimated coverage reaches τ_cov=0.995, capped at K_max=128. Rung 1 expands to 2·K_max as a safety margin.

**Multi-stage fallback ladder.** If E_key exceeds the certified threshold for a head, Rung 3 triggers per-head dense recomputation using FP16 keys/values paged from system RAM. Score-consistency canaries compare FP16 vs. INT8 scores on promoted blocks; ranking-consistency checks detect ordering inversions at the FP16/INT8 boundary. An exploration budget (1–5% of blocks) randomly spot-checks INT8 scores as defense-in-depth.

**Implementation details.** Evaluated on LLaMA 3.1-8B (INT8 model weights via bitsandbytes) on a single NVIDIA RTX PRO 6000 WS (96 GB VRAM, Blackwell). Native Blackwell attention backend with FP32 online-softmax accumulators. FP16 scratch cache sized at 2048 blocks (asymmetric: 2048K/256V at 64K, 2048K/512V at 128K).

## Results & Highlights

**PG-19 perplexity (language modelling):** Certified matches dense within noise at all context lengths:
- 8K: Δppl = −0.002, 95% CI [−0.010, +0.007]
- 64K: Δppl = +0.001, 95% CI [−0.001, +0.004]
- 128K: Δppl = +0.001, 95% CI [−0.001, +0.002]
- All 20-chunk CIs include zero. Per-chunk deltas symmetrically distributed around zero.

**NIAH (multi-needle retrieval, 10 needles/trial, 100 paired trials):**
- 8K/64K: Identical accuracy to dense (39%/66%), McNemar p=1.0 at both.
- 32K: −2% gap (51% vs. 49%), McNemar p=0.727 (not significant).
- 128K: Omitted (base model scores 0% on dense).

**RULER (7 subtasks):**
- 64K: +0.015 (certified slightly exceeds dense). All 4 NIAH retrieval variants at 100%/near-100%.
- 128K: +0.004, matching dense.
- 32K: −0.019 gap driven by CWE (−0.100), consistent with INT4 value error.
- 8K: −0.069 gap localized to value-sensitive subtasks (VT −0.250, CWE −0.140, FWE −0.117). All retrieval subtasks perfect.

**Naive INT8/INT4 baseline (no certification):** Catastrophic degradation confirming certification is essential:
- NIAH: 5–10% accuracy vs. 39–66% dense (McNemar p < 10⁻⁹ at all lengths).
- RULER: −44 to −56pp at all lengths, VT scores 0% everywhere.

**Key ablations:**
- FP16 values close 8K RULER gap from −6.9pp to −0.4pp with zero critical failures.
- Tightening vtol from 0.05 to 0.02 removes the 32K gap entirely (Δ=0.000, 5 slices).
- K_max sweep (64–512) does NOT close the 8K gap—value quantization, not key quantization, is the bottleneck.
- Value group-size cliff: g=16 near-lossless on perplexity; g=32 causes severe +26.4 Δppl.

**Runtime performance (2048-block scratch cache):**
- Latency: 2.73× (8K) to 4.76× (128K) dense. Dominated by ranking-consistency check (28.3% of step time at 64K) and H2D page-in (11.1%).
- VRAM savings: 0.83× dense at 64K (asymmetric 2048K/256V), 0.72× at 128K (asymmetric 2048K/512V). Below 64K, scratch exceeds dense KV.
- System RAM cost: Tier-2 equals dense KV size (~17 GB at 128K).

**Certificate telemetry (RULER):**
- E_key increases with context: median 0.055 (8K) to 0.392 (128K), max 3.731 (8K) to 18.141 (128K). But max values are *never returned*—fallback intercepts them.
- E_val stable: median 0.023–0.038, capped at vtol=0.050 by promotion policy.

## Critical Assessment

- **Novelty: High.** Runtime certification of KV cache quantization with per-head, per-step formal bounds is genuinely novel. The two-term decomposition, ranking-consistency check, and attention-dilution insight (promoting more blocks can *worsen* retrieval) are original contributions. Most prior work validates quantization empirically on benchmarks without any per-step guarantees.

- **Rigor: High.** Formal TV bounds with proofs (Appendix A). Statistical testing done properly: exact McNemar tests for paired NIAH trials, bootstrap CIs for PG-19/RULER, systematic ablations isolating key vs. value vs. kernel-path error sources. Honest about the local-only nature of certification (does not compose across layers/steps).

- **Reproducibility: Good.** Code released under MIT license. Exact seeds, model configs, and evaluation scripts in the artifact repository. However, the system requires a specific GPU (RTX PRO 6000 WS, Blackwell architecture) and 96 GB VRAM + 221 GB system RAM, limiting who can reproduce results.

- **Significance: Moderate-to-high.** Practical impact is currently narrow: the 2.7–4.8× latency overhead and requirement for system RAM equal to dense KV size mean this is only viable for long-context (64K+) deployments where VRAM is scarce and quality regressions are unacceptable. The certification framing is conceptually important but the current implementation is a prototype, not production-ready.

- **Red flags:**
  1. **Latency overhead is severe.** 4.76× at 128K. The ranking-consistency check alone consumes 28.3% of step time and is Python-orchestrated. The paper acknowledges this is the primary optimization target.
  2. **VRAM savings only at 64K+.** Below 64K, the fixed scratch buffer makes the system use *more* VRAM than dense. The practical operating regime is narrow.
  3. **System RAM cost equals dense KV.** You need as much system RAM as you'd need VRAM for dense. At 128K, that's ~17 GB. This is not "free" memory.
  4. **Per-step certification does not compose.** Small per-step errors accumulate through 32 heads × 32 layers × N decode steps. The 8K RULER gap (−6.9pp on VT/CWE/FWE) demonstrates this limitation clearly.
  5. **Single-author, single-GPU, single-model.** Only tested on LLaMA 3.1-8B with INT8 weights. No results on larger models (70B+), multi-GPU serving, or batching (all experiments use batch size 1).
  6. **Non-monotonic precision vs. accuracy.** More FP16 keys can *worsen* retrieval via attention dilution. This is honestly reported but complicates deployment tuning.

## Strengths

- Principled formal error bounds with proofs, not just empirical validation.
- Exceptionally transparent about limitations and negative findings.
- The naive-baseline comparison powerfully demonstrates that certification machinery is essential, not ornamental.
- Two-term decomposition enables independent key/value precision tuning—a practical deployment knob.
- Ranking-consistency check converts silent failure mode (attention distribution corruption) into a detectable, recoverable event.
- Comprehensive ablations (K_max sweep, FP16 values, vtol tuning, precision decomposition) leave no ambiguity about error mechanisms.

## Limitations

- **Per-step certification does not guarantee end-to-end correctness.** Accumulated value errors across layers/steps cause measurable degradation on value-sensitive tasks at short contexts (8K: −6.9pp RULER).
- **Severe latency overhead (2.7–4.8×).** Dominated by ranking-consistency check (Python-orchestrated) and H2D page-in, not fundamental to the certification framework but limiting practical adoption.
- **No VRAM savings below 64K.** The fixed FP16 scratch buffer exceeds dense KV cache size at short contexts.
- **System RAM requirement equals dense KV size.** Tier-2 FP16 originals consume ~17 GB system RAM at 128K.
- **Single model and hardware configuration.** Only LLaMA 3.1-8B on one GPU type, batch size 1, greedy decoding.
- **Kernel numerical path gap (~4pp on NIAH).** Different rounding between the certified kernel and Flash Attention is an implementation artifact that adds noise to the certification story.
- **Value group-size cliff.** g=16 works, g=32 catastrophically fails (+26.4 Δppl)—no intermediate operating point.

## Relevance

与LLM高效推理直接相关，特别是长上下文场景下的KV缓存压缩与安全部署。运行时误差认证机制为量化部署提供了形式化保障的新范式，两项误差分解和自适应精度选择的思路对实际系统设计有重要参考价值。虽然当前实现开销较大，但认证框架本身是架构无关的，可适配未来硬件和量化方案。

## Questions for Further Exploration

- Can the ranking-consistency check be fused into the Phase 2 attend kernel to eliminate the 28.3% step-time overhead? What is the theoretical minimum overhead for certification?
- How does the system behave on models with different attention patterns (e.g., mixture-of-experts, grouped-query attention with different GQA ratios)?
- Can the certification framework extend to other compression schemes (FP8 KV, 2-bit quantization, vector quantization) while maintaining tight enough bounds to keep fallback rates low?
- What is the latency overhead on multi-GPU serving with tensor/pipeline parallelism, where H2D transfers may overlap with computation on other GPUs?
- Could predictive prefetching of FP16 blocks based on attention-pattern forecasting reduce the H2D page-in cost?
- Is there a path to end-to-end certification that composes per-step bounds across layers and autoregressive steps, even as a loose upper bound?
- How does the attention-dilution effect (more FP16 keys worsening retrieval) interact with different model architectures and attention head specializations?
