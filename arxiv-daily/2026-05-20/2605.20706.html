Llamas on the Web: Memory-Efficient, Performance-Portable, and Multi-Precision LLM Inference with WebGPU

A WebGPU backend for llama.cpp that runs LLM inference in the browser with static memory planning, a tunable kernel library, and templated quantization kernels supporting 23 formats.

Reese Levine, Rithik Sharma, Nikhil Jain, Abhijit Ramesh, Zheyuan Chen, Neha Abbas, James Contini, Tyler Sorensen

Key Finding

Static memory planning and templated quantization kernels enable WebGPU-based LLM inference that uses 29–33% less memory and achieves 54–69% higher decode throughput than existing browser frameworks — while supporting 177K+ GGUF models and running competitively with native backends on select hardware.

Key Metric
54-69 % faster decode

vs. WebLLM / Transformers.js (geometric mean, 4 GPUs). Memory use down 29–33%. Evaluated across 481 benchmark runs on 16 devices from 8 GPU vendors.

Why This Matters

Zero-install, privacy-preserving LLM inference in any browser tab. Access to 177K+ GGUF-compatible models vs. ~400 for WebLLM. Already merged into llama.cpp via 100+ PRs — real engineering impact, not a prototype.

Critical Edge
  • Prefill 51% slower than WebLLM — missing kernel fusion + subgroup matrix ops
  • Browser safety checks cost up to 42% prefill overhead on NVIDIA RTX 5080
  • Firefox WebGPU essentially broken (~1 vs ~52 tok/s same hardware)
  • Quantization doesn't linearly improve throughput — primarily a memory play

Architecture
Static Memory Arena
Pre-allocates all GPU memory at startup: parameter arena, FlashAttention intermediates, KV-cache. Eliminates dynamic allocation and prevents browser tab crashes.
Tunable Kernel Library
Separates runtime scheduling from kernel compilation. Device-specific parameter sweeps (thousands of configs × 4 GPUs) yield cross-device portable defaults.
Templated Dequant
21 quantization formats compiled as templates into matmul/attention kernels. Adding a new format requires only a dequant function — no kernel rewrite.
Assessment

Strengths

  • Largest cross-device browser LLM evaluation to date
  • Clean kernel library / runtime separation
  • Extensible quantization via templates
  • Honest reporting of all weaknesses
  • 100+ merged PRs into llama.cpp
  • Static memory prevents competitor memory leaks

Limitations

  • Prefill 51% behind WebLLM (no kernel fusion)
  • Quantization is memory-only, not throughput gain
  • Safety checks up to 42% overhead
  • No MoE or multi-modal support
  • Firefox WebGPU unusable
  • No batching — limits server use
