> **一句话总结：** 作者将现代多模态助手 MiniCPM-V-4.6（SigLIP2 视觉塔 + 门控增量网语言骨干）完全手工移植到一块 2011 年的 6GB NVIDIA Tesla C2075（Fermi, sm_20）上端到端运行，通过"先测量再下结论"的工程方法，把预填充投影交给厂商 cuBLAS sgemm（64% FP32 peak，约为手写 GEMM 的 1.8 倍）、把循环层改写成分块 delta-rule 扫描（修正一个坏核后比顺序快 2.8 倍）、并两次用 cuBLAS 注意力重写移除了 10k 上下文下的 O(N²) 墙（10k 预填充 479s→27.7s，17.3 倍），最终单图问答端到端 1.7s。

> **TL;DR:** An engineering/measurement report that runs the MiniCPM-V-4.6 multimodal assistant (SigLIP2 vision tower + 0.8B hybrid gated-delta-net LLM) entirely on a 2011 6 GB NVIDIA Tesla C2075 (Fermi, sm_20). Three measured results: (1) an all-GPU language engine built on dequantize-then-cuBLAS-sgemm prefill (659 GFLOP/s = 64% FP32 peak, vs 378 GFLOP/s for the best hand-written GEMM) and a chunked delta-rule scan (2.8× faster than sequential recurrence); (2) a stage-validated hand port of the SigLIP2 tower matching a locally generated reference forward to 1.4×10⁻⁵ per element; (3) a tiered 2k–10k prefill benchmark where a vendor-GEMM attention rewrite removes an O(N²) wall, taking 10k-token prefill from 479.3s (naive attention kernel, 20.9 tok/s) to 27.7s (361.2 tok/s), a 17.3× delta, with a needle planted at 60% depth retrieved exactly. The system answers a single-image question end-to-end in 1.7s on hardware older than the transformer literature it runs.

## 核心贡献

- **在废弃硬件上跑通现代多模态助手：** 首次将 MiniCPM-V-4.6（SigLIP2 视觉编码器 + 窗口注意力合并器 + 0.8B 混合 gated-delta-net 语言骨干）完全驻留在单块 2011 年 Tesla C2075（Fermi, 6GB, sm_20）上，无需任何 CPU 协同或主机↔设备流式传输——这是其伴随工作（35B MoE 模型）的结构性对立面。
- **三个"经测量而非凭直觉"的工程基础：** (a) 预填充投影先反量化 8-bit 权重再调用厂商 cuBLAS sgemm，达到 659 GFLOP/s（64% FP32 peak），纠正了之前"手写 GEMM 37% 即为天花板"的错误结论；(b) 把 token 级顺序递推改写成分块 delta-rule 形式，第一版慢 2.4×、经组件归因发现是单个前代换核占用 92% 时间，修正后反而比顺序快 2.8×；(c) 用测量否定了"4-bit 解码更快"的直觉——Fermi 移位指令半速发射使 4-bit 比 8-bit 慢 12%。
- **阶段验证式视觉塔移植：** 通过 CPU 能力覆盖让无法原生运行的参考实现强制走遗留代码路径以在本地生成 ground truth，对 31 个阶段逐一移植并校验（每阶段 ~10⁻³ 门控），全塔匹配至 1.4×10⁻⁵/元素，多切片至 1.2×10⁻⁴。
- **可推广的浮点 tie-breaking 规则：** 发现 NaViT 风格位置嵌入的桶化在精确有理平局（10/28 = 25/70）处是"实现定义"的，任何重实现都会在某些网格尺寸翻转，由此提出普适规则——"含可达精确平局的索引算术必须调用参考算子而非重写它"。
- **零额外显存的 cuBLAS 注意力重写：** 把双向（视觉塔）和因果（语言预填充）softmax 注意力都改写成每头两次 cuBLAS GEMM，并把分数直接写入引擎已有的得分缓冲区（零新增显存）；图像编码快 6×，10k 预填充快 17.3×。

## Key Contributions

- **Runs a current-generation multimodal assistant fully on a 2011 GPU:** Deploys MiniCPM-V-4.6 (SigLIP2 vision encoder + mid-stack window-attention merger giving 16× visual token compression + 0.8B hybrid gated-delta-net LLM from the Qwen3 family) entirely resident on a single NVIDIA Tesla C2075 (Fermi, 6GB, sm_20) — the structural opposite of the companion 35B MoE study that required streaming GPU-prefill / CPU-decode.
- **Three measurement-driven engineering foundations for the language engine:** (a) prefill projections dequantize 8-bit weights once into an FP32 scratch and call the vendor cuBLAS sgemm still shipped in the last Fermi toolchain — 659 GFLOP/s = 64% FP32 peak, overturning a prior false claim that the 37% hand-written GEMM was the ceiling; (b) a hand-written chunked intra-/inter-chunk delta-rule scan for the 18 recurrent layers, whose first version was 2.4× slower until per-component attribution localized 92% of the time in one bad forward-substitution kernel, after which it is 2.8× faster than sequential recurrence; (c) a measured negative — 4-bit decode is 12% slower than 8-bit on this GPU because Fermi issues the nibble-unpacking shifts at half rate.
- **A stage-validated methodology for hand-porting a vision stack** when the reference implementation cannot run natively (no-AVX host): a CPU-capability override forces the tensor library onto a legacy code path to generate a local 31-stage reference trace, and each ported stage is gated at ~10⁻³ agreement before proceeding; the full tower matches to 1.4×10⁻⁵ per element.
- **A generalizable tie-breaking rule for index arithmetic:** the discovery that NaViT-style position bucketization at the exact rational tie 10/28 = 25/70 is implementation-defined and that every reimplementation (float64, float32, exact integer) disagrees with the reference on some grid in {1,…,70}; the rule "call the reference operator, do not reimplement it" extends to any index arithmetic with reachable exact ties.
- **Two cuBLAS attention rewrites with a zero-allocation layout trick:** the per-head score matrix is written directly into the engine's existing attention buffer (zero new device memory), cutting vision-tower image encoding 6× (5.7s→0.93s) and removing the O(N²) long-context prefill wall (10k tokens: 479.3s→27.7s, 17.3×).

## Methodology

This is explicitly an **engineering and measurement report on one machine, not an algorithmic novelty claim.** The recurring methodology is that "every ceiling and every bug yielded to measurement against a ground truth" and resisted intuition.

**Target hardware:** NVIDIA Tesla C2075 (Fermi/GF110, 2011): 448 CUDA cores, 6 GB GDDR5 (~5.3 GB usable with ECC) at ~144 GB/s, ~1.03 TFLOP FP32 peak, compute capability sm_20, no tensor cores / fast FP16 / DP4A, 48 KB shared memory/block, logical shifts issued at half FMA rate. Last supported toolchain is CUDA 8.0 (run in a container, used only to build) — critically, CUDA 8.0 still ships cuBLAS, which the engine calls three separate times. The host is a pre-AVX Westmere-class Xeon (SSE4.2/SSSE3 only).

**Model:** MiniCPM-V-4.6 has three parts: (a) a SigLIP2 vision encoder — 27-layer, 1152-wide ViT, 16 heads of dim 72, 14×14 patches, learned position embeddings on a 70×70 reference grid resampled by fractional-coordinate bucketization (NaViT style), no rotary embeddings; (b) a two-stage spatial compressor — after layer 6 a window-attention merger (self-attention within each 2×2 patch window + wide MLP 4608→17216→1152) and after the final layer a second 2×2 merge-and-project (4608→4608→1024), net 16× compression so a 448² image costs 64 language tokens; (c) a 0.8B Qwen3-family hybrid backbone — 24 blocks (18 gated-delta-net + 6 softmax-attention), hidden width 1024, tied output head, 1-D rotary.

**Quantization & residency:** Language weights 8-bit group-wise symmetric (one FP32 scale per 64 weights) ≈ under 1 GB; vision tower FP32 ≈ 2.1 GB. Everything resident: 1.6 GB engine + 2.7 GB tower = 4.4 of 5.3 usable GB at rest, ~0.8 GB headroom, no host→device transfer inside any forward pass.

**Three measured foundations of the language engine:**
1. *Prefill projections* are genuine matrix–matrix products, so the achievable GEMM ceiling was measured against the vendor: hand-written 6×6 register-tiled GEMM = 378 GFLOP/s (37% peak), cuBLAS sgemm = 675 GFLOP/s (66% peak); the engine dequantizes once and calls cuBLAS, reaching 659 GFLOP/s (64% peak, 7.2× the per-token GEMV it replaces).
2. *Chunked delta-rule scan* for the 18 recurrent layers: within a 512-token chunk the gated delta rule becomes a lower-triangular system solved by two small GEMMs + a forward substitution + batched GEMMs; across chunks a short recurrence of small state products remains. Verified at three levels (chunked algebra vs sequential on host ~3×10⁻⁵ rel. error; GPU vs host ~2×10⁻⁵; full engine produces identical greedy tokens).
3. *Decode* stays at 8-bit by measurement: the 8-bit kernel reads four signed bytes per vector load (no shift) at ~48 GB/s effective bandwidth (48.6 tok/s greedy); the best 4-bit register-blocked kernel is 12% slower (0.31 vs 0.28 ms) because nibble-unpacking shifts at half rate move the kernel off its bandwidth bound.

**Stage-validated vision port:** A single script hooks all 31 reference stages (patch embedding, 27 encoder layers, window-attention merger, final layer norm, downsampling projector) and dumps each output for a test image; each ported stage must match its reference dump to ~10⁻³ (in practice 10⁻⁴ or better). The one stage that failed — position-embedding bucketization, off by 1.8 max per element on a 28×36 grid — led to the tie-breaking rule. The merger is the only genuinely new kernel (one block per (window, head), 4×4 score matrix in shared memory).

**Two cuBLAS attention rewrites:** The score matrix S = QK⊤ per head is a real matrix product that belongs in sgemm; the key layout observation is that the engine's existing per-(position, head) row-major score buffer is byte-for-byte a valid column-major score matrix if the GEMM is issued with leading dimension heads×seq, so the vendor GEMM writes into the buffer the old kernel already owned (zero new memory). Causal attention adds a masked column softmax that normalizes only the valid prefix and zeroes the tail.

**Correctness verification throughout:** greedy decoding compared token-by-token against the reference implementation on held-out prompts (identical); tower output compared to a locally generated reference trace; needle-in-haystack planted at 60% depth of a 10k-token document and retrieved exactly before and after the attention rewrite.

## Results & Highlights

- **GEMM ceiling (C2075 prefill projection, ~1030 GFLOP/s FP32 peak, Table 1):** the vendor cuBLAS sgemm reaches **675 GFLOP/s (66% of FP32 peak)**, the hand-written 6×6 tiled GEMM reaches only **378 GFLOP/s (37%)**, and the engine's dequantize-then-cuBLAS pipeline reaches **659 GFLOP/s (64%)** — i.e., 7.2× the 91 GFLOP/s per-token GEMV it replaced, and the hand-written "37% wall" claim was wrong by nearly 2×.
- **Chunked delta-rule scan (one recurrent layer, 16 heads, 512-token chunk, Table 2):** after per-component attribution localized 92% of time in one bad forward-substitution kernel (54.5 ms → 4.2 ms after fix), the total scan dropped from **59.4 ms (first version, 0.42× sequential) to 9.0 ms (2.8× faster than the 25.1 ms sequential recurrence)** — a complete reversal of the initial "2.4× slower" conclusion.
- **Decode weight-format decision (Table 3):** 8-bit four-wide signed load takes **0.28 ms at 48.4 GB/s** effective bandwidth vs 4-bit register-blocked **0.31 ms at 22.9 GB/s (1.12× slower)** — the measured negative that keeps decode at 8-bit; greedy decode runs at **48.6 tok/s**.
- **Tiered long-context prefill (production model, Table 4 & Figure 1):** naive attention sustains **114.2 tok/s at 2k but collapses to 20.9 tok/s at 10k** (prefill wall 479.3s); the cuBLAS attention rewrite is nearly flat at **408.5 tok/s at 2k and 361.2 tok/s at 10k** (prefill wall 27.7s) — a **17.3× speedup at 10k** — with the needle planted at 60% depth of the 10k-token document retrieved exactly both before and after.
- **Vision tower accuracy & speed:** the full SigLIP2 tower (patch embedding through projector) matches the reference forward to **1.4×10⁻⁵ max per element** (multi-slice 1.2×10⁻⁴ over all 5×63 output vectors of an 896×672 image); the cuBLAS attention rewrite cuts single-image encoding from **5.7s to 0.93s (6.1×)** and a five-slice image from **27.5s to 4.6s**, with tower agreement unchanged at 1.2×10⁻⁴.
- **Position-embedding tie failure:** the bucketization stage initially mismatched the reference by **1.8 max per element** on a 28×36 grid at the exact rational tie 10/28 = 25/70; every reimplementation (float64, float32, exact integer) flipped on some grid in {25, 28, 49, 50, 55}; calling the reference operator reduced mismatch to **2×10⁻⁶ on every grid**.
- **End-to-end production (all on the serving path):** a single-image shape/color question answers in **1.7s**; an image-plus-question prompt is ≈92 tokens (image cost near-invisible after 16× compression); a five-slice 896×672 image with rendered text answers in **≈7s (tower 4.6s of it)**, reading the rendered text exactly; device memory at rest is **4.4 of 5.3 usable GB** with ~0.8 GB headroom. An eight-item smoke battery (plain text, color/shape, rendered-text OCR, two images in one prompt, multi-slice image, multi-turn follow-up, plain text after vision, Spanish) passes in full.

## Critical Assessment

- **Novelty:** The paper claims no algorithmic novelty and is correct that there is none — every technique (dequantize-then-sgemm, chunked delta-rule scan, cuBLAS attention) is a port of known ideas to an unsupported architecture. The genuine contributions are the *composition* (an all-GPU current-gen multimodal assistant on Fermi), the **stage-validated porting methodology** with the locally generated reference trace, and the **tie-breaking rule for index arithmetic** at exact rational ties (the 10/28 = 25/70 case is a real, transferable debugging lesson). Reviewers looking for a new algorithm will find nothing.
- **Rigor:** Measurement discipline is the paper's strongest trait and is visibly real — every number is paired with the ground truth it was measured against (vendor BLAS, per-component timer, reference dump, tiered benchmark, planted needle). The five explicitly enumerated reversals ("37% is the GEMM wall," "the chunked scan is slower," "4-bit decode must be faster," "2k benchmarks represent prefill," "index arithmetic can always be reimplemented") are a credibility-enhancing pattern.
- **Reproducibility — strengths:** single-source build, public checkpoint, regenerable reference dumps, all experiments on one workstation. **Weaknesses:** the entire setup hinges on a no-AVX Westmere host and an NVIDIA Tesla C2075 with a CUDA 8.0 container — hardware most readers cannot acquire, so "regenerable" is true in principle but rarely in practice; no code repository URL is given (the artifact is described but not linked), and the one-line bucketization "call the reference operator" depends on having the exact framework version installed.
- **Significance:** Primarily of interest to the low-resource / legacy-hardware / extreme-edge-inference community and as a methodology tutorial; it is unlikely to change mainstream multimodal deployment. The transferable takeaways (measure the GEMM ceiling against the vendor before asserting a wall; benchmark at the longest tier not the shortest; index arithmetic with reachable exact ties cannot be reimplemented) are the most broadly valuable artifacts.
- **Red flags / weak points:**
  - **No standard accuracy benchmark.** Correctness is validated by token-identical greedy decoding to a reference, by a single planted needle, and by an eight-item hand-authored smoke battery — there is no MMVet / MMMU / MMBench / standard VLM benchmark, so the assistant's *quality* is essentially taken on trust from the upstream MiniCPM-V-4.6 model card rather than re-measured here.
  - **Single-hardware, single-image-suite evaluation.** Every throughput number comes from one C2075 on one workstation; there is no variance reporting, no second unit, and the eight-item smoke battery is small and self-authored (selection-bias risk).
  - **The "needle at 60% depth" is a single retrieval probe**, not a systematic needle-in-haystack sweep across depths/positions/quantities; it establishes that the rewrite did not break retrieval, not a quantitative recall curve.
  - **4-bit "slower" claim is architecture-specific** to Fermi's half-rate shifts and is correctly framed as such, but readers could over-generalize it to modern GPUs where INT4 matmul kernels dominate.
  - **The author name "A. C. Opus"** (and the companion paper's framing) suggests possible pseudonymous / AI-assisted authorship; the corresponding author and institution are real, but the authorship provenance is worth noting.

## Strengths

- Exceptional measurement discipline: every ceiling and bug is resolved against a ground truth, with five named reversals of prior intuition.
- The stage-validated porting methodology — using a CPU-capability override to generate a local reference trace on a no-AVX host — is a genuinely transferable recipe for porting to unsupported hardware.
- The tie-breaking rule ("call the reference operator for index arithmetic with reachable exact ties") is a crisp, generalizable debugging lesson backed by a concrete failure (10/28 = 25/70) and the observation that grids 25/28/49/50/55 each flip differently because the tie set depends on gcd(g, 70).
- The zero-allocation cuBLAS attention layout trick (writing scores into the existing per-(position, head) buffer with leading dimension heads×seq) elegantly fits a 6 GB memory budget.
- Honest negative results (hand-GEMM ceiling was wrong; 4-bit is slower) that prevent the community from repeating the same mistakes on legacy hardware.
- Token-identical greedy decoding vs the reference and tower agreement to 1.4×10⁻⁵ give strong correctness assurance for the port itself.

## Limitations

- No standard multimodal accuracy benchmarks (MMMU, MMBench, MMVet, etc.); task quality is inherited from the upstream model card rather than re-measured, so the paper proves the port is *faithful* but not that the assistant is *useful* on this hardware.
- Single-hardware, single-workstation evaluation with no variance bars and no second unit; throughput numbers are point measurements on one Tesla C2075.
- Correctness probes are light: one planted needle at one depth (60%), one eight-item self-authored smoke battery, token-identical greedy decoding on a few held-out prompts — no systematic needle-in-haystack recall curve or adversarial cases.
- Hardware and toolchain (CUDA 8.0 container, pre-AVX host, Tesla C2075) are largely unobtainable, so practical reproducibility by others is low despite the regenerable-in-principle reference dumps.
- No public code/repository URL; the artifact is described but not linked.
- Architecture-specific findings (e.g., 4-bit decode being slower) are correctly scoped to Fermi but easy to misread as general claims.
- The paper does not address energy/cost-efficiency comparisons vs a modern GPU or vs the companion 35B MoE system on the same machine — only throughput.

## Relevance

与"在受限/老旧硬件上部署现代大模型"及"多模态 LLM 高效推理（量化、CUDA 内核、注意力/线性注意力优化）"方向高度相关，其"先测量厂商库再断言天花板、长上下文分层基准、阶段验证移植"的方法论对任何做边缘推理与内核移植的工作都有直接借鉴价值。

## Questions for Further Exploration

- How does the system perform on standard multimodal benchmarks (MMMU, MMBench, MMVet, OCRBench) rather than a self-authored smoke battery — is the 1.4×10⁻⁵ tower error and token-identical decoding sufficient to preserve reported accuracy, or do small numerical differences compound on harder tasks?
- Would a systematic needle-in-haystack sweep across depths (0%, 25%, 50%, 75%, 100%), multiple needles, and varying document types reveal retrieval degradation that the single 60%-depth probe hides?
- Can the zero-allocation cuBLAS attention layout trick and the chunked delta-rule scan be ported back to *modern* low-end GPUs (e.g., mobile/embedded) where memory budget, not compute, is the binding constraint — and does the tie-breaking rule surface in other NaViT/SigLIP deployments?
- The paper disables speculative decoding at ~70% draft-acceptance because "the extra verification passes outweigh the tokens saved" — is that break-even point sensitive to draft-model size, and could a smaller draft tip the balance on Fermi?
- Given that the 4-bit-is-slower finding is specific to Fermi's half-rate shifts, what is the corresponding crossover analysis on architectures with native INT4 matmul (e.g., Turing/Ampere), and does the methodology (measure the vendor ceiling first) generalize as a pre-deployment audit tool?
- The "call the reference operator for index arithmetic with reachable exact ties" rule is demonstrated for NaViT bucketization — are there other index computations in modern vision towers (e.g., rotary position indices, RoPE frequency tables) where the same implementation-defined-tie hazard exists?
