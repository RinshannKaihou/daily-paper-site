> **一句话总结：** 作者提出 LQCDMaster，一个面向格点 QCD 的领域专用智能体，通过专家标注的 LQCD 技能、确定性的 Wick 收缩工具（generate_einsum）与 solve-critique-rewrite 规划循环，将自然语言物理任务转换为可执行的 PyQUDA 计算工作流，在 70 任务基准上以 GPT-5.4 为骨干达到 90.0%（63/70）的机器精度复现率，并把单可观测量实现时间从数小时压缩至数分钟（17×–54× 加速）。

> **TL;DR:** LQCDMaster is a tool-augmented, skill-guided, domain-specialized agent that turns natural-language lattice QCD research tasks into reviewable PyQUDA workflows. On a 70-task benchmark (C24P29 CLQCD ensemble) spanning five observable classes, it reproduces expert-written reference implementations on 63/70 tasks at machine precision (|Δ|≲10⁻¹², 90.0% accuracy) with GPT-5.4, versus a Claude Code baseline that fails on all 20 of the simplest local two-point tasks, while cutting per-observable implementation time by 17×–54× (e.g., baryon 3pt: 8 h → 10.9 min).

## 核心贡献

- **降低 LQCD 研究门槛：** 将自然语言任务（如"计算算符 ūγ₅d 的 π 介子两点函数，使用 [0,0,0,0] 点源"）自动转换为完整的 PyQUDA 测量脚本、SLURM 提交脚本、执行日志与数值输出，使物理学家无需手写实现代码即可开展格点 QCD 研究。
- **构建首个全面的 LQCD 科学计算基准：** 70 个任务覆盖五大可观测量类别（局域/非局域两点函数、Wilson 圈、介子三点函数、重子三点函数），全部取自前沿科研场景，并配以与专家手写实现的逐位数值校验。
- **确定性符号工具约束脆弱代数：** 引入 `generate_einsum` 工具，将 Wick 收缩串（味配对、费米子符号、γ 插入、ε 张量、自旋-色指标序）从语言模型的自由输出空间中剥离，交由确定性引擎生成，改变了系统失败模式——模型只负责高层物理判断。
- **开创智能体科学计算范式：** 以两个真实探索案例验证其价值——首次使用对角 Wilson 线几何在格点上计算介子光锥分布振幅（LCDA），以及质子、氘核、氚核、超子 Λ、超氘核 pΛ、超氚核 pnΛ 的多强子谱计算，证明降低实现成本能扩展"值得尝试"的科学想法集合。
- **跨骨干模型鲁棒性：** 在仅替换 chat-completion 端点的情况下，DeepSeek-V4-Pro 骨干在相同 70 任务上达到 56/70（80.0%）机器精度复现，表明 LQCDMaster 架构对骨干 LLM 有实质性鲁棒性。

## Key Contributions

- **Lowering the barrier to LQCD research** by converting natural-language research requests into executable PyQUDA computing workflows (measurement scripts, job-submission artifacts, execution logs, numerical outputs), removing the need to hand-write implementation code.
- **A comprehensive LQCD scientific-computing benchmark** of 70 tasks spanning five observable classes (local/nonlocal two-point functions, Wilson loops, meson and baryon three-point functions), all drawn from frontier research and validated by direct numerical comparison against hand-written expert references at machine precision.
- **A deterministic Wick-contraction tool (`generate_einsum`)** that removes the most algebraically brittle part of correlator code generation (flavor pairings, fermion signs, gamma insertions, epsilon tensors, spin-color index ordering) from the LLM's free-form output, establishing an explicit reliability boundary between physical planning and algebraic realization.
- **Expert-annotated, stage-routed domain skills** (`lqcd-physics-correlator` for the planner, `pyquda-tool` for the executor, `pyquda-gauge` for gauge-only tasks) that prevent invented PyQUDA API usage and align domain knowledge with the tools valid for each physics workflow.
- **Pioneering agentic scientific computing in lattice QCD** via two real explorations made practical by low marginal cost: the first lattice computation of light-cone distribution amplitudes with a diagonal Wilson-line geometry, and multi-hadron spectroscopy (proton, deuteron, triton, Λ, pΛ, pnΛ) where Wick-contraction topologies balloon to 2880 and 576 terms.

## Methodology

LQCDMaster decomposes LQCD code generation into two stages, separating planning from execution:

**Planning stage.** A planner agent operates in a *solve–critique–rewrite* loop: it performs physical analysis and proposes a detailed technical plan (ensemble, observables, hadrons, propagators, correlator measurements, expected output format); an independent critique checks whether the observables, propagators, and settings are reasonable and consistent, and the plan is revised if flaws are found. Before the plan reaches the executor, a **human checkpoint** lets physicists review and optionally refine the plan — an explicit acknowledgment that agents cannot yet guarantee correctness for all LQCD observables.

**Execution stage.** The executor generates PyQUDA measurement scripts plus SLURM submission scripts (a test script and a full-run script) guided by PyQUDA skills, with automated verification via static analysis and LLM critique.

**The `generate_einsum` tool** is the central reliability mechanism. Rather than emit contraction strings directly, the executor requests a contraction *in physics terms* (e.g., "meson 2pt", "baryon 3pt sequential-source sink block"); the tool enumerates valid Wick pairings, applies sign/topology conventions, constructs spin-color labels, and returns a ready-to-insert contraction specification. The final PyQUDA script contains the resulting einsum strings and does *not* depend on the tool at runtime. Five contraction families are exposed: meson 2pt, baryon 2pt, multi-hadron 2pt, meson 3pt, baryon 3pt.

**Skills are stage-routed**: the planner receives correlator physics skills (observables, interpolating operators, Wick-contraction structure, momentum projection, sequential-source requirements, γ₅-hermiticity), the executor receives PyQUDA skills (source construction, solver parameters, MPI data layout, contraction implementation, output formatting), and a dedicated `pyquda-gauge` skill handles gauge-only measurements (Wilson loops, Polyakov loops).

**Validation protocol.** Every generated workflow is validated by direct numerical comparison with a hand-written reference implementation for the same observable, ensemble, and kinematic setup, not merely by successful execution — because in LQCD a syntactically valid program can execute cleanly while measuring the wrong observable (wrong spin–color contraction, flavor assignment, or fermionic sign).

## Results & Highlights

- **Overall accuracy (GPT-5.4 backbone):** 63 of 70 tasks reproduced expert-written reference implementations at machine precision (|Δ| ≲ 10⁻¹²) on the CLQCD C24P29 ensemble (a = 0.105 fm, 24³ × 72, m_π ≈ 290 MeV) — an overall accuracy of **90.0%**; 3 additional cases were convention mismatches (fixable by a single global sign transform), 4 were unresolved failures, all in baryon 3pt.
- **Per-class accuracy:** Local 2pt 20/20 (100%), Nonlocal 2pt 8/10 (80.0%), Wilson loop 12/12 (100%), Meson 3pt 13/13 (100%), Baryon 3pt 10/15 (66.7%).
- **Implementation-time speedups vs human experts:** Wilson loop 1 h → 3.5 min (**17×**), Local 2pt 2 h → 4.0 min (**30×**), Nonlocal 2pt 4 h → 4.5 min (**54×**), Meson 3pt 4 h → 6.2 min (**38×**), Baryon 3pt 8 h → 10.9 min (**44×**); speedup grows with observable complexity.
- **Cross-backbone robustness:** Swapping GPT-5.4 for **DeepSeek-V4-Pro** (architecture, skills, tools, pipeline unchanged) yields 56/70 exact matches = **80.0%** accuracy (Wilson loop 100%, Local 2pt 80.0%, Baryon 3pt 86.7%, Meson 3pt 53.8%, Nonlocal 2pt 80.0%), showing the architecture, not just the model, drives reliability.
- **Baseline comparison:** **Claude Code**, given the same tasks plus PyQUDA documentation but no specialized skills/tools, *failed to produce executable scripts for any of the 20 local two-point-function tasks* — the simplest observable class — demonstrating that a guidebook alone is insufficient.
- **New science enabled:** First lattice computation of a meson light-cone distribution amplitude with a **diagonal Wilson-line** geometry (cross-checked against an independent PyQUDA implementation, agreement within statistical uncertainties); and multi-hadron spectroscopy for proton, deuteron, triton, hyperon Λ, hyperdeuteron pΛ, hypertriton pnΛ — cases where manual Wick-contraction topologies reach 36 and 2880 (2pt) / 12 and 576 terms.

## Critical Assessment

- **Novelty:** Genuinely novel framing — this is the first agentic system targeting *executable workflow generation* in lattice QCD (prior ML work in the field uses AI only as a numerical ingredient inside otherwise human-centered pipelines, e.g., flow-based sampling). The `generate_einsum` reliability boundary (delegating symbolic contraction construction to deterministic code while leaving judgment to the LLM) is a transferable design pattern for any domain where a single wrong index yields a plausible-but-wrong number.
- **Rigor:** The validation protocol is the paper's strongest feature. Correctness is established by *direct numerical regression against hand-written references at machine precision* (|Δ| ≲ 10⁻¹²), not by "it ran." The benchmark's failure taxonomy (matched / convention mismatch / failure) and Table 3's analysis of first-pass issues (einsum-label mismatches, gauge-context lifetime errors, sequential-propagator sign conventions) are concrete and falsifiable. A subtle rigor question: the "convention mismatch" bucket is somewhat lenient — a single global sign flip is counted as a near-success, which flatters the headline 90.0%; arguably 3 of those are still wrong-by-default outputs that needed expert intervention to diagnose.
- **Reproducibility:** High. Code, benchmark, generated scripts, and outputs are publicly released; the architecture is backbone-agnostic (demonstrated across two frontier LLMs); the benchmark runs on a named, available CLQCD ensemble with stated lattice spacing, volume, and pion mass.
- **Significance:** The Claude Code baseline is a damning and important data point — *0/20 on the simplest tasks* — quantifying exactly what the skills+tool layer contributes. The two exploratory case studies (diagonal Wilson-line LCDA; hypernuclear spectroscopy) are credible demonstrations that lower implementation cost can change which science gets done.
- **Red flags / caveats:** (1) The benchmark uses **unreleased/proprietary backbone models named "GPT-5.4" and "DeepSeek-V4-Pro"**, which readers cannot independently verify at the stated capability level — exact reproduction of the headline numbers is therefore not guaranteed to hold for any particular reader's model snapshot. (2) The **4 baryon-3pt failures** (p→p vector, Λ→Λ vector/axial, Λ_b→Λ_c vector currents) are precisely the hardest, most physically interesting case, and the paper admits the critique–rewrite pipeline "did not fully catch" them, so the headline accuracy is uneven across observable complexity. (3) The baryon 3pt category also contains a "convention mismatch" — with only 15 tasks, a single reclassification materially changes that row's accuracy. (4) The human checkpoint in the planning loop is essential but its effort/iteration count is not quantified, so "hours → minutes" reflects wall-clock agent time, not total human-in-the-loop cost. (5) The benchmark omits disconnected diagrams, ≥4-point functions, and finite-temperature observables — i.e., the easy-to-medium regime; generalization to the hardest LQCD computations is untested.

## Strengths

- Numerical, not behavioral, evaluation: machine-precision agreement with expert references is the right gold standard for scientific code, and cleanly separates "plausible correlator" from "correct correlator."
- The `generate_einsum` design is elegant and generalizable: it converts the most failure-prone sub-task (symbolic contraction strings) from an LLM-output problem into a deterministic-tool problem, fundamentally changing the failure mode.
- The Claude Code ablation (0/20 on simplest tasks) is an unusually honest and informative baseline that quantifies the value of the domain skills + deterministic tool layer.
- Cross-backbone validation (GPT-5.4 vs DeepSeek-V4-Pro, 90.0% vs 80.0%) tests architecture-dependence rather than treating a single model result as the finding.
- Real new physics produced as proof of value (first diagonal Wilson-line LCDA; hypernuclear multi-hadron spectrum), not just benchmark numbers.
- Full release of code, benchmark, and outputs.

## Limitations

- Benchmark coverage is representative, not exhaustive: **disconnected diagrams, ≥4-point correlations, finite-temperature observables, and large multi-particle operators are not tested** — these are among the most important and error-prone LQCD computations.
- The hardest category (baryon 3pt) has the worst accuracy (66.7%) with 4 genuine failures on physically central transition currents (p→p, Λ→Λ, Λ_b→Λ_c); static checks and test execution did not catch them, so **direct numerical regression against trusted references remains mandatory** for new observable classes.
- The system performs **code generation only** — no automated fitting, no physical interpretation/analysis pipeline; it accelerates implementation, not the downstream scientific inference.
- Residual failures are sensitive to the backbone LLM's in-domain knowledge (DeepSeek-V4-Pro's meson-3pt failures were traced to incorrect quark-flavor assignments), so semantic validation of observable-specific task parameters is not fully solved by the tool layer.
- The human-in-the-loop planning checkpoint is acknowledged as necessary but its cost is not quantified; the headline "hours → minutes" is agent wall-clock time.
- Headline models (GPT-5.4, DeepSeek-V4-Pro) are not independently reproducible by readers at the stated capability.

## Relevance

这篇文章对"LLM 智能体 + AI for Science"研究方向高度相关：它示范了如何用"确定性工具 + 专家技能 + 规划-执行双阶段"的架构，把脆弱的领域代数（Wick 收缩、γ 矩阵、自旋-色指标）从语言模型自由生成中剥离，从而在硬核科学计算（格点 QCD）上达到机器精度级别的可验证正确性，对设计面向其他领域的可信赖科学代码生成智能体具有直接的范式参考价值。

## Questions for Further Exploration

- How does accuracy degrade as observable complexity increases beyond the benchmark — specifically on disconnected diagrams, 4-point functions, and finite-temperature observables that are currently untested?
- Can the `generate_einsum` reliability-boundary pattern be formalized and ported to other domains where a single index/sign error yields plausible-but-wrong numeric output (e.g., quantum chemistry contractions, tensor-network simulations)?
- What is the quantified human-in-the-loop cost at the planning checkpoint (review time, number of plan revisions), and how does total human time compare to the headline agent wall-clock speedup?
- Could the 4 baryon-3pt transition-current failures (p→p, Λ→Λ, Λ_b→Λ_c) be closed by extending `generate_einsum` to cover transition-specific contraction topologies, or do they require a fundamentally different verification step?
- Does the architecture's robustness across backbones (90.0% → 80.0%) hold for substantially weaker open-weight models, or is there a capability floor below which the skill/tool layer cannot compensate?
- Can the system be extended downstream into automated fitting and physical interpretation (e.g., mass extraction, excited-state fits), turning it from a code generator into an end-to-end analysis infrastructure?
