> **一句话总结：** 本文提出 Atrex-Bench,首个直接从生产推理 trace 采样的 GPU kernel 生成基准(30 个算子、440 个 shape,按生产 GPU 时间占比加权并配以 per-problem roofline 上限),评估 6 个前沿编码 agent 后发现最好的模型也只达到硬件 roofline 的约 10.7%,且部分"正确"结果来自 PyTorch fallback 而非模型手写的 DSL kernel;为此协同发布 Atrex-Kernel-Agent (AKA),通过 profile 驱动的 measure-revise 循环、optimization dropout 与分层 GPU 知识库,把零 FlyDSL 的 fallback 转换为接近 100% FlyDSL 的真实 kernel,并在 attention_forward 等算子上反超手调生产基线。

> **TL;DR:** The authors release Atrex-Bench, the first GPU-kernel-generation benchmark whose 30 operators and 440 shapes are sampled from full-cluster production inference traces on >10k XPU-A/H20 accelerators and scored by an importance-weighted, per-problem roofline metric. Six frontier coding agents (Claude Opus 4.7, GPT-5.5, Qwen3.7-Max, Kimi-K2.6, GLM-5.1, DeepSeek-V4-Pro) reach at best Sagg = 0.107 (10.7% of roofline) on Atrex-Bench, and none beats the deployed production kernel (best 0.99x); the paper exposes a "correctness illusion" where Qwen3.7-Max scores 84.8% correctness with only 43.8% FlyDSL adoption. The co-released Atrex-Kernel-Agent (AKA) converts 0%-FlyDSL fallbacks into near-100%-FlyDSL kernels that beat hand-tuned production baselines by 1.06x-1.31x on attention_forward, chunk_gated_delta_rule_state, and mla_decode_attention.

## 核心贡献

- **Atrex-Bench:首个生产 trace 驱动的 kernel 生成基准。** 从 XPU-A 与 H20 推理集群(>10k 加速卡)的在线服务 trace 中采样,经 1,303 个 profile、~20 个部署模型(vLLM/SGLang/AITER/RTP-LLM)归并出 30 个算子、440 个 hot shape,每个算子携带按应用卡时与 prefill/decode 阶段加权得到的重要性权重 wi,以及 per-shape roofline 上限。
- **重要性加权 roofline 评分契约。** 引入 Sagg = Σ wi·Si(算子级别取正确 shape 的中位数,否则 0),让占据真实服务时间的算子主导得分;top 5 算子(unified_attention 36.1% 等)占约 64% 权重。同时强制隐藏 metadata.json(上游符号、生产时序)与 roofline.json,从生成端隔离答案,防作弊。
- **对 6 个前沿编码 agent 的系统性评估与"正确性幻觉"诊断。** 揭示最好的 GPT-5.5 仅达 Sagg = 0.107,无 agent 击败生产 kernel;并提出 FlyDSL adoption 指标暴露 correctness 靠 PyTorch/vender kernel fallback 的现象(Qwen3.7-Max 84.8% correctness vs 43.8% FlyDSL)。
- **Atrex-Kernel-Agent (AKA) 优化 agent。** 由 profile-driven measure-revise 循环、optimization dropout(部分重启以跳出局部最优)与分层 GPU 知识库(298 个参考 kernel、244 篇优化文档、外部上游项目)组成;在受控 case study 中把 fallback 转为近 100% FlyDSL 的真实 kernel,并超越手调生产 kernel。
- **两项成果均开源**(github.com/alibaba/atrex-bench 与 atrex-kernel-agent)。

## Key Contributions

- **Atrex-Bench, the first benchmark sourced from full-cluster production inference traces.** 30 operators and 440 hot shapes are mined from 1,303 profiles over ~20 deployed models served by vLLM, SGLang, AITER, and RTP-LLM on >10k XPU-A/H20 accelerators, then collapsed from framework-specific kernels into computation-level operator families so the same problem is shared across engines.
- **An importance-weighted, per-problem roofline scoring contract.** Each (op, shape) carries a serving-phase-aware weight wi derived from its share of app-card-hour-weighted device time, and a speed-of-light ceiling Troofline from reference semantics; the headline Sagg = Σ wi·Si credits operators in proportion to production wall-time, while hidden metadata.json/roofline.json prevent the model from retrieving the reference by name or fitting the scoring formula.
- **A six-frontier-agent evaluation that quantifies the deployability gap and a "correctness illusion."** GPT-5.5 leads at Sagg = 0.107 but still falls to 0.85x of the deployed production kernel; a FlyDSL-adoption metric exposes that Qwen3.7-Max's 84.8% correctness rests on only 43.8% target-DSL device time, with the rest delegated to scaled_dot_product_attention or precompiled AITER entry points.
- **Atrex-Kernel-Agent (AKA).** A routed profile-driven optimization agent combining iterative measure-revise search, optimization dropout (a partial restart that masks stale iteration memories while preserving the accepted kernel and audit trail), and a layered GPU Wiki (298 reference kernels across NVIDIA/AMD/agnostic + 244 optimization-knowledge documents, with mandatory source citations for every hardware spec).
- **A controlled case study** on three compute-bound operators showing AKA converts 0%-FlyDSL fallbacks into near-100%-FlyDSL kernels that beat hand-tuned production baselines by 1.06x-1.31x, and lifts already-FlyDSL Opus kernels toward the roof (attention_forward S 0.28 → 0.42).

## Methodology

**Benchmark construction (Atrex-Bench).** A tracer attaches to live serving workloads on XPU-A and H20 clusters without relaunch, combining Python-frame context (phase, batch/token shape, framework state) with vendor runtime traces (kernel launches, timing, streams, memory ops, graph anchors) and correlating host/device events. Raw kernel records are mapped to computation-level operator families so semantically identical kernels from vLLM/SGLang/AITER/RTP-LLM share one problem. Each operator ships as a fixed five-file contract: visible to the candidate are reference.py, input.py, shapes.json, and the prompt; hidden from generation (used only by the evaluator) are metadata.json (provenance, production baseline time) and roofline.json (per-shape speed-of-light denominators). Tool surface during generation is restricted to local workspace inspection, search, edit, and shell build/test; web retrieval, notebook editing, and auxiliary agents are excluded in the comparable benchmark runs.

**Per-problem roofline bound.** With dtype-specific compute peak Pτ and memory bandwidth β calibrated once per accelerator, the reference-defined semantic work Fj and traffic Mj yield Troofline,j = max(Fj/Pτ, Mj/β).

**Importance weighting (Equation 3).** For each application a with card-hour share pa, operator i's device-time share per phase ϕ ∈ {prefill, decode} is wi = Σa pa · (1/|Φi|) Σ D_{i,a}/D_a, then normalized so Σ wi = 1. The top five operators hold ≈64% of weight (unified_attention 36.1%, fused_moe 10.4%, block_scaled_mm 8.5%, fp8_blockscale_fused_moe 4.7%, paged_attention_decode 4.0%).

**Scoring.** Per-shape achievement Sj = Troofline,j / Tcand,j ∈ (0,1]; per-operator Si is the median over correct shapes, or 0 if none pass; headline Sagg = Σ wi Si (arithmetic, not geometric, so a single zeroed operator does not erase all signal). Evaluation is a three-stage gate: compile → correctness (vs reference.py over K=5 seeds, atol=1e-2, rtol=5e-2) → performance (median end-to-end forward time, frequency-lock check, per-iteration L2 flush, per-shape subprocess isolation). The target DSL is deliberately FlyDSL, absent from pre-training corpora, to probe learning rather than recall.

**AKA agent.** A seven-stage outer workflow (user input → router/init → hardware roofline grounding → correctness-first baseline → profile-driven optimization loop → optimization-dropout partial restart → evaluator-ready output contract) and an eight-stage inner loop (profile with official profiler → extract bottleneck evidence → query GPU Wiki → write evidence-driven plan → apply one optimization category → validate → update structured memory → stop check). Every hardware value must be cited from GPU Wiki; every kernel change must be justified by profiler evidence rather than intuition.

## Results & Highlights

- **Production-weighted roofline ceiling is far from reached.** On Atrex-Bench (30 ops, 440 shapes, XPU-A), the best vanilla agent GPT-5.5 reaches Sagg = 0.107 (i.e., 10.7% of the reference-derived hardware roofline), while Claude Opus 4.7 reaches only Sagg = 0.059; the remaining four models fall to 0.012-0.047. GPT-5.5's lead over Opus 4.7 widens from 0.129 vs 0.104 (unweighted operator median) to 0.107 vs 0.059 (≈1.8x) under importance weighting, because the heaviest production operators are compute-bound.
- **No candidate beats the deployed production kernel.** Opus 4.7 comes closest at 0.99x, GPT-5.5 at 0.85x, Qwen3.7-Max at 0.19x, and DeepSeek-V4-Pro trails at 0.12x. Versus torch.compile, only Opus 4.7 (2.29x) and GPT-5.5 (3.06x) beat it on the median operator; Qwen3.7-Max is marginal at 1.10x.
- **Correctness illusion is measurable.** Qwen3.7-Max scores 84.8% correctness but only 43.8% FlyDSL adoption (i.e., device time in target DSL); Kimi-K2.6 has 23 correct vs 10 FlyDSL-dominant operators, and DeepSeek-V4-Pro has 18 vs 6. Opus 4.7 is the exception (24 correct vs 26 FlyDSL-dominant, negative gap).
- **Hardness is operator-specific, not model-specific.** Nine operators are solved by every model, while the hardest — fp8_blockscale_fused_moe (22.2% mean pass), fused_rmsnorm_quant (34.8%), per_token_group_quant_fp8 (44.7%), attention_forward (45.2%) — are uniformly low-precision-fused or matrix-engine kernels, marking a shared capability frontier.
- **Memory-bound is solved, compute-bound is not.** Opus 4.7's memory-bound median S = 0.207 is over 20x its compute-bound median 0.009; GPT-5.5 is the only model with non-trivial compute-bound achievement (median 0.074, e.g., 0.22 on fused MoE GEMM), which alone lifts it above Opus on Sagg.
- **Failure taxonomy over 683 failing units.** 53.4% are compile failures (365 units, GLM-5.1 alone contributing 232), 43.0% runtime (257 numeric mismatches = 37.6% of all failures), and 3.5% correct-but-slow. GPT-5.5's 30 failures are almost entirely numeric mismatches; Opus 4.7's 46 are uniquely half correct-but-slow — the soft performance-side signature of the strongest backend.
- **Generation volume does not predict quality.** DeepSeek-V4-Pro emits 6.56M tokens for only 18 correct operators (365K tokens/op), while GPT-5.5 emits 1.19M tokens for 26 correct (46K/op).
- **AKA closes both gaps in the controlled case study.** On Qwen3.7-Max it converts 0%-FlyDSL fallbacks to ~100% FlyDSL kernels: chunk_gated_delta_rule_state 0.001→0.03 S and 0.03x→1.20x vs production (the vanilla fallback was ~29x slower), attention_forward 0.06→0.40 S (6.7x over fallback) and 0.17x→1.11x vs production, mla_decode 0.0003→0.0035 S and 0.10x→1.06x. On Opus 4.7 it lifts attention_forward from S=0.28 to 0.42 (0.78x→1.17x production) and mla_decode from 0.71x to 1.27x.

## Authors & Credibility

The paper is authored by the ATREX Team (Alibaba Tensor Research & Engineering for XPUs) at Alibaba Group, with Lingyun Yang and Yuxiao Wang as equal contributors. The work is grounded in real fleet telemetry (>10k deployed accelerators, 1,303 production profiles, ~20 production-model deployments across four serving frameworks), and both artifacts are released as open source (github.com/alibaba/atrex-bench and github.com/alibaba/atrex-kernel-agent). The benchmark construction details (tracer design, five-file contract, hidden-provenance policy, anti-hacking isolation) are described in operational depth, and the failure taxonomy over all 683 failing units is disclosed rather than cherry-picked. The target DSL (FlyDSL) and accelerator naming (XPU-A as a desensitized non-NVIDIA accelerator) are openly stated as deliberate design choices.

## Critical Assessment

- **Novelty.** The combination is genuinely new: prior benchmarks (KernelBench, BackendBench, TritonBench, MultiKernelBench, FlashInfer-Bench, CUDABench, SOL-ExecBench) are uniformly synthetic/curated and none ships both a per-problem roofline ceiling and an importance-weighted aggregate; the cross-benchmark matrix in Table 1 is the paper's own positioning evidence. The correctness-illusion framing via FlyDSL adoption is a sharper diagnostic than pass-rate alone. AKA's optimization dropout is a recognizable but reasonable variation on partial-restart ideas from AKO/KDA, and is honestly framed as a "controlled case study, not a benchmark-wide ranking."
- **Rigor.** The hidden-information policy (metadata.json, roofline.json excluded from generation), per-shape subprocess isolation, frequency-lock, and L2 flush are non-trivial anti-gaming measures. Multi-seed correctness (K=5), operator-balanced aggregation, and a separate optimizer-augmented setting (web tools enabled) are methodologically clean. The arithmetic-vs-geometric-mean justification for Sagg is correct.
- **Reproducibility.** Strong on benchmark construction (released five-file contract, refreshable weights, versioned snapshots) but weaker on the AKA case study: Table 8 reports "a single timed run per (op, shape)" with no variance bars and only one production-representative shape per operator, so the 1.06x-1.31x production-beating margins are point estimates on a narrow slice.
- **Significance.** The headline finding that the best frontier agent reaches ~10.7% of roofline and beats no production kernel is a credible, deployability-aligned signal that rebukes over-optimistic claims from synthetic benchmarks. The compute-bound vs memory-bound split is the most actionable diagnostic for the field.
- **Red flags / missing pieces.** (1) Empirical results are reported only on XPU-A (AMD-class, desensitized); the H20 fleet is described as sampled but not benchmarked, so the production-readiness claim is single-vendor. (2) The target DSL is FlyDSL, explicitly "absent from the models' pre-training corpora" — this is a fair test of learning but means the absolute Sagg numbers are not directly comparable to CUDA/Triton benchmarks; the paper acknowledges Triton/Gluon/CuteDSL are deferred. (3) AKA's wins are restricted to three compute-bound operators hand-picked to "stress the agent in different ways"; there is no full-benchmark AKA run, no ablation isolating the contribution of optimization dropout vs the knowledge base vs the profile loop, and no comparison to AKO/KDA under the same contract. (4) The "overtake hand-tuned production" headline on attention_forward is partly explained away by the paper itself — the general AITER varlen kernel only reaches ~36% of the roof on that awkward non-power-of-two shape (head dim 72), so a shape-specialized kernel passing it is less surprising than it sounds. (5) Importance weights are derived from one cluster's serving mix and are skewed by unified_attention at 36.1%; a fleet with a different model mix could re-rank agents.

## Strengths

- First benchmark whose problem distribution, weights, and denominator all come from production serving, making the score a deployability proxy rather than a synthetic average.
- The FlyDSL-adoption metric turns an invisible "specification shortcutting" pass into a recorded number, exposing the correctness illusion that simpler pass-rate benchmarks miss.
- Honest, fine-grained failure taxonomy (683 units broken down by stage and model) and a model-independent operator-difficulty ranking.
- Clean separation of generation vs evaluation surfaces with concrete anti-leakage engineering (hidden metadata/roofline, restricted tool surface, isolated subprocesses).
- AKA's design principles — every hardware spec cited, every edit justified by profiler evidence — are good practice that other kernel agents should adopt.

## Limitations

- **Single-hardware empirics.** All headline numbers are on XPU-A; H20 is sampled but not evaluated, and broader accelerator validation is explicitly future work.
- **Inference-only.** Training kernels (backward, optimizer-step) are out of scope for this release.
- **Narrow AKA evaluation.** Three operators, one shape each, one timed run per cell, no variance, no full-benchmark run, no ablation of the three AKA mechanisms against each other or against AKO/KDA baselines.
- **FlyDSL-specific.** Because the target DSL is deliberately out-of-distribution, absolute Sagg values are not comparable to CUDA/Triton benchmarks; Triton, Gluon, CuteDSL are deferred.
- **Weight skew.** unified_attention alone holds 36.1% of weight; a different fleet mix could change both Sagg and the agent ranking.
- **No peer model comparison for AKA.** AKA is paired only with Qwen3.7-Max and Opus 4.7; behavior on the four other panel models is untested.

## Relevance

本论文直接命中"LLM 代码 agent 能否写出可上线的 GPU kernel"这一核心问题,提供了首个生产 trace 驱动、按服务时间加权、带 per-problem roofline 的基准,以及一个可复用的 profile 驱动优化 agent;对于关注 LLM for code、推理系统优化、kernel auto-tuning 与 code agent benchmark 设计的研究者,既是评估方法论参考,也是可接入的开源工具。

## Questions for Further Exploration

- How does the agent ranking change when the target DSL is Triton or CUDA (in-distribution for pre-training) instead of FlyDSL — does the correctness illusion shrink, stay, or morph?
- Does AKA's production-beating margin hold on the full 440-shape benchmark with variance bars, or does it collapse outside the three hand-picked compute-bound operators?
- What is the individual contribution of optimization dropout vs the GPU Wiki knowledge base vs the profile loop in an ablation under the same contract?
- How transferable are the production-importance weights and the operator ranking to an NVIDIA-only or H20-heavy fleet with a different model mix?
- Can the memory-bound vs compute-bound diagnostic be turned into an automatic routing signal that selects which AKA optimization category to apply per operator?
- Since AKA relies on retrieving from 298 reference kernels + 244 docs, how sensitive is it to knowledge-base coverage on operators outside its current scope (e.g., training backward kernels)?
