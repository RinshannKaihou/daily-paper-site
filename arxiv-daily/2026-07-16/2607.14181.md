> **一句话总结：** 本文系统对比了六种 4-bit 权重量化方法（GPTQ、AWQ、QuIP#、AQLM、BitsAndBytes、GGUF）在 Qwen2.5-Coder-7B 和 CodeLlama-7B 上的表现，发现在 pass@1、SonarCloud 代码质量指标及 prompt 复杂度鲁棒性三方面方法差异显著：AQLM 一致地追平或超过 FP 基线，QuIP# 退化最严重，安全性则全面不受影响。

> **TL;DR:** A 4-bit, weight-only quantization benchmark of six PTQ techniques (GPTQ, AWQ, QuIP#, AQLM, BitsAndBytes, GGUF) on Qwen2.5-Coder-7B and CodeLlama-7B across McEval, CoderEval, and BigCodeBench (Python/Java). AQLM consistently matches or beats the FP16 baseline on pass@1 (e.g., BCB-Python CodeLlama 0.27 vs FP 0.23), while QuIP# is the worst offender (BCB-Python CodeLlama 0.12 vs FP 0.23, p<0.05, OR=5.10); Security stays at 0 across all configs, and prompt-complexity sensitivity (Shannon entropy / token length) is strongly model-dependent.

## 核心贡献

- 首个覆盖六种主流 weight-only PTQ 方法（横跨四类算法策略加两类生态路径）的统一对比基准，弥补了以往"一项研究只测一种量化方法"的空白。
- 超越 pass@1，使用 SonarCloud 静态分析从可维护性、可靠性、安全性、圈复杂度（CyC）、认知复杂度（CoC）五个维度评估量化对代码质量的影响。
- 首次引入 prompt 复杂度鲁棒性分析，用 token 长度和 Shannon 熵刻画输入复杂度，通过点二列相关分析量化方法在不同复杂度任务上的退化差异。
- 在 2 个模型族 × 3 个基准 × 2 种语言上做严格统计检验（McNemar、Wilcoxon、Holm-Bonferroni 校正、Cliff's delta 效应量），并提供完整复现包。
- 给出可操作的实践指南：部署资源受限代码模型时如何选择量化策略，强调仅看功能正确性不足以刻画量化模型的真实代价。

## Key Contributions

- First unified comparison of six widely-used weight-only PTQ techniques (spanning four algorithmic families plus two ecosystem paths), closing the gap left by prior single-technique studies.
- Goes beyond pass@1 to assess code quality via SonarCloud across maintainability, reliability, security, cyclomatic complexity (CyC), and cognitive complexity (CoC).
- Introduces a novel prompt-complexity robustness analysis that characterizes input difficulty through token length and Shannon entropy, and quantifies technique-specific degradation via point-biserial correlation.
- Applies rigorous statistics (McNemar, Wilcoxon signed-rank, Holm-Bonferroni correction, Cliff's delta) across 2 model families, 3 benchmarks, and 2 languages, with a public replication package.
- Delivers actionable deployment guidance: how to pick a quantization strategy for resource-constrained code models, stressing that functional correctness alone is insufficient.

## Methodology

The study evaluates six weight-only post-training quantization (PTQ) approaches, all at 4-bit precision, chosen one-per-category from a taxonomy of PTQ methods:

- **GPTQ** (second-order/Hessian-informed, group size 128)
- **AWQ** (activation-aware saliency, default config)
- **QuIP#** (rotation/incoherence-based, randomized Hadamard transforms + E8 lattice codebooks)
- **AQLM** (vector quantization, two codebooks)
- **BitsAndBytes** (library/kernel-based, NF4 with double quantization)
- **GGUF** (format/runtime-based, Q4_K_M mixed-precision)

**Models:** Qwen2.5-Coder-7B-Instruct and CodeLlama-7B-Instruct (instruction-tuned, 7B only — justified as representative of resource-constrained deployment and to keep the combinatorial space manageable).

**Benchmarks:** McEval (simple, concise function-level), CoderEval (mid-length, context-dependent, using the Crupi et al. filtered subset), and BigCodeBench (long, multi-library orchestration, Python only). McEval and CoderEval are used in both Python and Java.

**Metrics:**
- Functional correctness: pass@1 (single-attempt, whether top output passes all unit tests) in a sandboxed Docker environment.
- Code quality via SonarCloud: Reliability (bug density), Maintainability (code smells), LoC, CyC (control-flow structural complexity), CoC (human-perceived complexity).

**Quantization procedure:** Pre-quantized Hugging Face checkpoints where available (AWQ, GPTQ, GGUF, BitsAndBytes); AQLM and QuIP# quantized locally with official pipelines calibrated on WikiText-2 (QuIP# via QuIP-for-all to support Qwen). All models loaded via Hugging Face transformers; temperature 0; input cap 1024 tokens. Hardware: 4× NVIDIA L40S (48GB each).

**RQ3 complexity analysis:** Python tasks pooled across McEval/CoderEval/BigCodeBench, split at median Shannon entropy into High/Low buckets; complexity–degradation relationships tested with McNemar's test on discordant pairs, stratified pass@1/degradation analysis, point-biserial correlations (entropy, length), and Mann–Whitney U tests.

**Statistical framework:** McNemar's test (binary correctness) and Wilcoxon signed-rank (continuous quality metrics), Holm–Bonferroni correction for six pairwise comparisons, Cliff's delta effect sizes (negligible/small/medium/large). Output variability checked via 10 predictions per instance on Qwen/McEval-Python with Friedman's test (non-significant).

## Results & Highlights

- **RQ1 — Functional correctness largely preserved, but technique matters:** On McEval-Java, CodeLlama-7B (FP pass@1 = 0.25) spans from QuIP# at 0.08 to GPTQ/AQLM at 0.34; BitsAndBytes preserves the baseline exactly (0.25). Qwen2.5-Coder-7B (FP 0.45) jumps to 0.60 with BitsAndBytes (+15pp) and 0.58 with AQLM, while GPTQ and QuIP# drop to 0.32. On BCB-Python, AQLM yields the best CodeLlama pass@1 (0.27 vs FP 0.23) whereas QuIP# drops to 0.12 — roughly halving the baseline. McNemar's test reaches significance only on BCB-Python and one McEval-Java case; the lone significant Java finding is CodeLlama FP vs QuIP# on McEval (p<0.05, OR = 19.00). On BCB-Python, CodeLlama QuIP# (p<0.05, OR = 5.10) and BitsAndBytes (p<0.05, OR = 2.05) significantly degrade, while AQLM significantly *improves* (p<0.05, OR = 0.62).
- **RQ2 — Security universally unaffected; quality effects are localized:** Security stays at 0 hotspots across every configuration, model, benchmark, and language. AWQ systematically increases Maintainability issues in Java (CodeLlama d = −0.346, Qwen d = −0.394, both medium effect, on McEval-Java). BitsAndBytes produces the largest single effect in the whole study on Qwen McEval-Python: CyC d = −0.452 (medium) and CoC d = −0.487 (large). On CoderEval-Python Qwen, all six techniques significantly increase LoC (1,367–1,618 vs FP 988), CyC, and CoC — the most quality-sensitive configuration. Java quality differences are mostly non-significant; Python reveals the statistically robust shifts.
- **RQ3 — Prompt-complexity sensitivity is strongly model-dependent:** For CodeLlama, five of six techniques show significant positive correlation between input complexity and degradation, with QuIP# strongest (entropy r = +0.325, p<0.001; length r = +0.321, p<0.001), followed by BitsAndBytes (r = +0.203) and AWQ (r = +0.192). BigCodeBench concentrates the worst losses — QuIP# reaches 12.3–15.3% degradation on CodeLlama. For Qwen, only two of twelve correlations reach significance and one is *negative* (GPTQ r = −0.156, p<0.001), i.e., simpler prompts degrade more; the within-BigCodeBench replication confirms this model-level divergence is not a benchmark artifact.
- **Cross-cutting:** Effects are both technique- and model-dependent — a technique that helps one model may hurt another (e.g., GPTQ improves CodeLlama McEval-Java pass@1 by 9pp but reduces Qwen's by 13). Statistical significance concentrates on the hardest benchmark (BCB-Python), suggesting quantization-induced correctness shifts become detectable as task complexity rises. On complex benchmarks, correctness and quality degradation do not always co-occur (e.g., GGUF and BitsAndBytes reduce CodeLlama LoC and Maintainability on BCB despite correctness losses).

## Authors & Credibility

The three authors (Saima Afrin, Md. Zahidul Haque, Antonio Mastropaolo) are from the AURA Lab, Department of Computer Science, William & Mary. Mastropaolo is an established software-engineering researcher with a strong empirical track record (e.g., the well-cited ICSE 2023 Copilot robustness study). The work is a direct extension of Afrin et al.'s ICSME 2025 paper (the first to examine quantization's impact on code quality, limited to AWQ) and Giagnorio et al.'s 2025 differentiated replication. Funding from NSF grant CCF-2451058 is disclosed. A full replication package (generated code, evaluation scripts, detailed efficiency measurements) is published at the cited GitHub repository, and statistical rigor (McNemar, Wilcoxon, Holm-Bonferroni, Cliff's delta, Friedman run-stability check) is markedly above the norm for this subfield.

## Critical Assessment

- **Novelty:** Genuine incremental contribution. The headline six-technique sweep across algorithmic families is the broadest in the code-quantization literature, and the prompt-complexity robustness analysis (Shannon entropy + token length) is a fresh, previously unexplored dimension. However, the constituent techniques and benchmarks are all standard; novelty is in the comparative synthesis, not in any new method.
- **Rigor:** Strong. Appropriate paired statistical tests, Holm-Bonferroni multiple-comparison correction, effect-size reporting, a 10-run Friedman stability check, and a within-BigCodeBench replication to rule out the entropy-bucket confound (High bucket dominated by BigCodeBench). The paper is honest that many directional "improvements" from quantization reflect stochastic variation rather than systematic gains.
- **Reproducibility:** High. Public replication package, named model variants/checkpoints, explicit calibration corpus (WikiText-2) and quantization frameworks (QuIP-for-all), fixed temperature 0 and 1024-token cap, and named hardware (4× L40S). The biggest reproducibility caveat is BitsAndBytes NF4 / GGUF Q4_K_M / AWQ / GPTQ using *pre-quantized* Hugging Face checkpoints whose exact provenance and revision are not pinned per-config.
- **Significance:** Practical rather than theoretical. Findings (AQLM is the safe default; QuIP# is risky on complex prompts; AWQ has a maintainability tax in Java; security is robust) directly inform deployment decisions for the large and growing population of local/Ollama users.
- **Red flags / specific weaknesses:** (1) Scope is narrow — only 7B models, only 4-bit, only Python/Java, and RQ3 is Python-only; the authors themselves flag that 2–3-bit and larger scales may invert the trade-offs (e.g., QuIP# is designed for extreme sub-4-bit regimes, so judging it only at 4-bit arguably undersells it). (2) The pass@1 baselines are modest in places (CodeLlama 0.17–0.31), and with only 42–53 McEval tasks and ~184–230 CoderEval tasks, per-bucket degradation percentages rest on small denominators — McEval-Python High-bucket percentages like "AQLM 16.7%" come from very few tasks. (3) McEval-Python shows pass@1 *gains* under quantization that are implausible as real improvements (e.g., Qwen GGUF/AQLM 0.43 vs FP 0.31) and are correctly but somewhat uncomfortably attributed to noise without deeper investigation. (4) SonarCloud is the sole quality oracle despite code quality being multi-faceted; the paper defends this by citing agreement with Pylint/Flake8/PMD from prior work, but security in particular is likely under-tested given all-zero hotspots. (5) The paper studies only weight-only quantization — it explicitly excludes weight-and-activation methods, so "quantization for code generation" is narrower than the title implies.

## Strengths

- Broadest head-to-head of weight-only PTQ families in the code-LLM literature, with a clean one-per-category taxonomy.
- Quality evaluation beyond pass@1 using a single language-agnostic tool, enabling direct Python/Java comparison.
- Methodologically careful: paired tests, multiple-comparison correction, effect sizes, run-stability check, confound replication.
- The prompt-complexity robustness analysis is genuinely new and yields a crisp, actionable model-dependent finding (CodeLlama is complexity-sensitive; Qwen is not).
- Honest framing of noise-driven "improvements" and of the technique-vs-model interaction, with a public replication package.

## Limitations

- Single model size (7B), single bit-width (4-bit), two languages; generalization to other scales, precisions, or languages is unproven and explicitly out of scope.
- RQ3 is Python-only due to BigCodeBench's language coverage, and the High-entropy bucket is BigCodeBench-dominated (mitigated but not eliminated by the within-BCB replication).
- Only weight-only PTQ is studied; weight-and-activation quantization (ZeroQuant, SmoothQuant, LLM.int8(), RPTQ, PB-LLM) is out of scope.
- Small per-benchmark task counts make some degradation percentages noisy; QuIP# being evaluated only at 4-bit may understate its intended sub-4-bit niche.
- SonarCloud as the sole quality tool, and uniformly-zero Security hotspots, suggest security may be under-tested.
- No efficiency/cost numbers are reported in the main tables despite memory savings (≈70% at 4-bit per prior work) being a primary motivator; efficiency is deferred to the replication package.

## Relevance

对关注大模型高效推理、模型压缩与代码生成的研究者，本文提供了在 Qwen2.5-Coder/CodeLlama 等主流代码模型上选择 4-bit 量化策略的直接实证依据，并指出仅靠 pass@1 评估量化模型存在系统性盲区——这与"代码大模型部署效率"及"代码生成质量评估"两个方向都高度相关。

## Questions for Further Exploration

- Do the AQLM-best / QuIP#-worst rankings and the CodeLlama-vs-Qwen robustness gap persist at 2–3 bit precision (QuIP#'s intended regime) and at larger scales (e.g., 32B Qwen2.5-Coder)?
- Would code-specific calibration data (instead of WikiText-2) close the AWQ-Java maintainability gap and the CoderEval-Python Qwen quality regression?
- Why do some quantized variants *exceed* FP pass@1 (e.g., Qwen McEval-Python GGUF/AQLM 0.43 vs FP 0.31) — is this genuine, a decoding-temperature artifact, or benchmark test-suite weakness?
- How do these six weight-only techniques interact with inference-time optimizations (speculative decoding, KV-cache quantization, structured pruning)?
- Is the uniformly-zero Security finding real, or an artifact of SonarCloud's security rule coverage on short generated snippets? Would a security-focused tool (e.g., SALLM-as-judge, dedicated vulnerability scanners) surface latent issues?
