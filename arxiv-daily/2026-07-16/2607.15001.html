LQCDMaster: Agentic Scientific Computing for Lattice Quantum Chromodynamics Research

Haofei Gao, Tingjia Miao, Wenkai Jin & 11 others  —  SJT U · Beihang · Tsung-Dao Lee Inst. · SCNU · Indiana U.
Co-corresponding: Qi-An Zhang, Siheng Chen, Wei Wang
lattice-qcd llm-agents scientific-computing code-generation wick-contraction pyquda
Key finding

A skill-guided agent turns natural-language lattice QCD tasks into reviewable PyQUDA workflows, matching expert code at machine precision on 63 of 70 tasks.

Key metric
90.0%
63 / 70 tasks reproduced expert references at machine precision (|Δ| ≲ 10⁻¹²) on the CLQCD C24P29 ensemble, GPT-5.4 backbone.
Why this matters
Proves a deterministic-tool + expert-skill architecture can peel brittle domain algebra (Wick contractions, γ matrices, spin–color indices) off the LLM and reach verifiable correctness in hardcore scientific computing.
Critical edge
Read critically
The hardest, most physically interesting category (baryon 3pt drops to 66.7%), and the headline models GPT-5.4 / DeepSeek-V4-Pro are unreleased — readers cannot independently reproduce the 90.0% number at the stated capability.

01 Methodology

LQCDMaster splits code generation into a planning stage and an execution stage, with a deterministic symbolic tool constraining the most algebraically fragile part of the work.

solve → critique → rewrite planner · PyQUDA-skilled executor · human checkpoint before run
  • Planner (solve–critique–rewrite): emits a technical plan (ensemble, observables, hadrons, propagators, correlator measurements); an independent critique checks consistency; a human physicist reviews before execution.
  • generate_einsum tool: the central reliability mechanism. The executor asks for a contraction in physics terms ("meson 2pt", "baryon 3pt sequential-source sink block") and a deterministic engine enumerates Wick pairings, applies sign/topology conventions, builds spin–color labels. The emitted PyQUDA script needs no runtime tool dependency.
  • Stage-routed skills: planner gets correlator physics; executor gets PyQUDA skills; a dedicated pyquda-gauge skill handles Wilson/Polyakov loops.
  • Validation: direct numerical regression against hand-written references — because in LQCD a clean-running program can still measure the wrong observable.

03 Strengths & Limitations

Strengths
  • Numerical, not behavioral evaluation — machine-precision agreement is the right gold standard for scientific code.
  • generate_einsum converts the most failure-prone sub-task from an LLM-output problem into a deterministic-tool problem — a transferable design pattern.
  • Claude Code ablation (0/20) is an unusually honest baseline that quantifies the value of the skill+tool layer.
  • Cross-backbone validation (90.0% vs 80.0%) tests architecture-dependence, not a single model snapshot.
  • Real new physics (LCDA, hypernuclear spectrum), not just benchmark numbers; full public release.
Limitations
  • Coverage is representative, not exhaustive: disconnected diagrams, ≥4-point functions, finite-T observables are untested — among the most error-prone LQCD computations.
  • Baryon 3pt: 66.7% with 4 genuine failures on central transition currents (p→p, Λ→Λ, Λ_b→Λ_c); static checks did not catch them.
  • Code generation only — no automated fitting or physical interpretation; it accelerates implementation, not downstream inference.
  • Human-in-the-loop cost at the planning checkpoint is acknowledged as necessary but not quantified; "hours → minutes" is agent wall-clock.
  • Headline models GPT-5.4 / DeepSeek-V4-Pro are not independently reproducible by readers.

04 Questions for Further Exploration

  1. How does accuracy degrade beyond the benchmark — on disconnected diagrams, 4-point functions, finite-temperature observables?
  2. Can the generate_einsum reliability-boundary pattern be ported to other domains where a single index/sign error yields plausible-but-wrong output (quantum chemistry, tensor networks)?
  3. What is the quantified human-in-the-loop cost at the planning checkpoint, vs the headline agent speedup?
  4. Could the 4 baryon-3pt transition-current failures be closed by extending generate_einsum to transition topologies, or do they need a different verification step?
  5. Does cross-backbone robustness hold for substantially weaker open-weight models — is there a capability floor?
  6. Can it extend downstream into automated fitting and physical interpretation, becoming an end-to-end analysis infrastructure?
