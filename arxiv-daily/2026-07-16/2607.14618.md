> **一句话总结：** PolyQ 通过 CPU 友好的 {2,3,4,8,16}-bit 调色板配合激活感知的逐通道水 filling 比特分配和编译期布局正则化（DAG 级排列传播），在 Falcon-H1-3B/Llama2-13B/Qwen3-32B 上把 3-bit 目标的 WikiText-2 困惑度相对 AWQ/GPTQ/Slim-LLM/AMQ 降低 2.4–32.1%，并将激活重排流量减少最高 70.8%，使分数比特预算在纯 CPU 边缘部署上成为可预测的实用设计变量。

> **TL;DR:** PolyQ is a CPU-oriented quantizer+compiler co-design that assigns per-channel bit-widths from a CPU-aligned {2,3,4,8,16}-bit palette under a fractional average-bit budget and uses a compile-time model compiler to permute/cluster channels into bit-homogeneous blocks; on WikiText-2 across Falcon-H1-3B, Llama2-13B, and Qwen3-32B at a 3-bit target it improves perplexity by 2.4–32.1% over AWQ/GPTQ/Slim-LLM/AMQ (e.g., Llama2-13B 4.88 vs AWQ 5.29 vs AMQ 5.68), reduces activation reorder traffic by up to 70.8%, and stays within 2% energy overhead of an optimized LUT (T-MAC) backend on Ryzen 9950X/7840U and Intel N250.

## 核心贡献

- **CPU 对齐的多精度调色板**：首次将逐通道量化调色板扩展为 {2,3,4,8,16}-bit，覆盖 LUT 友好的低比特（2/3/4）与 SIMD 友好的高比特（8/16），既能匹配激活显著性的重尾分布，又能避免 Slim-LLM 的 5–7-bit 条目破坏 2^b LUT 扩展规律的问题。
- **分数比特预算的水填充分配**：把每层视为独立的离散资源分配问题，在用户指定的分数平均比特预算 B（如 3.7 b）下，用基于激活加权重建误差的贪心水填充（max-heap 增益）选每通道比特宽度，再用 ISA 感知的 SIMD/LUT 量子匹配将通道数对齐到内核 tile 大小。
- **编译期布局正则化与 DAG 排列传播**：模型编译器把不规则的通道级比特图置换成同质比特块，为 W{2,3,4}A16 生成 T-MAC LUT 内核、为 W{8,16}A16 派发 OpenBLAS SIMD；并通过算子 DAG 上的基向量传播把可合并排列合并进权重矩阵，使运行期几乎不需要显式激活 shuffle。
- **端到端可预测的部署曲线**：在三类 CPU（工作站 Ryzen 9950X、笔记本 Ryzen 7840U、移动 Intel N250）上证明同一个 B 同时控制质量恢复、峰值内存、prefill 延迟、decode 吞吐与能耗，把分数比特预算变成可用设计变量而非 W2/W3/W4 三选一。

## Key Contributions

- **CPU-aligned poly-precision palette**: A per-channel palette of {2,3,4,8,16}-bit that respects both LUT (2^b table scaling for 2/3/4-bit) and SIMD (8/16-bit OpenBLAS) CPU kernel constraints, unlike Slim-LLM's b±1 palette whose 5–7-bit entries force INT8 dequantization and forfeit LUT efficiency.
- **Waterfilling under a fractional bit budget**: Each linear layer is solved as a discrete resource-allocation problem under a user-specified fractional average budget B; a max-heap greedy waterfilling picks the largest gain G_{b→b'}[j] = ΔE/Δb promotions until B is exhausted, then an ISA-aware snap stage rounds channel counts to backend tile quanta with error-aware reassignment.
- **Compile-time layout regularization**: A model compiler factors each linear as O = (A×P_W^T)×(P_W×W), clusters same-bit channels into contiguous blocks, and dispatches W{2,3,4}A16 blocks to T-MAC LUT kernels and W{8,16}A16 blocks to OpenBLAS SIMD GEMM; the permutation matrix P_W is never stored at runtime.
- **DAG-level permutation propagation**: The compiler treats the model as a static operator DAG, assigns channel-basis labels to activation edges, and forms maximal regions where downstream activation permutations are absorbed into upstream weight permutations (Eq. 10), eliminating explicit shuffles beyond single transformer blocks.
- **End-to-end validation of a single budget knob**: Demonstrates that one fractional B parameter monotonically controls perplexity recovery, peak memory footprint, prefill latency, decode throughput, and energy across workstation/laptop/mobile CPUs, turning the bit budget into a usable deployment design curve.

## Methodology

PolyQ couples a quantizer (Section 3) with a model compiler (Section 4).

**Quantizer (Section 3).** For each linear layer ℓ with weight W^(ℓ) and per-channel scale s^(ℓ), and for each candidate bit-width b ∈ {2,3,4,8,16}:
1. **Activation-aware per-channel error proxy (3.1).** Computes E_b^[ℓ][j] ≈ ||w̃_j^(ℓ,b) − w_j^(ℓ)||² · E[x_j²] in a single pass over weights plus precomputed activation second moments; channels with large E[x_j²] are salient and their quantization error is penalized more heavily.
2. **Waterfilling under fractional budget (3.2).** Initializes all channels to b_min=2, then greedily applies the highest-gain feasible promotion (max-heap keyed by G_{b→b'}[j] = (E_b[j]−E_{b'}[j])/(b'−b)) until the per-layer budget B·d_in is exhausted.
3. **ISA-aware SIMD-quanta matching (3.3).** Rounds per-bit channel counts ĉ_b ≈ round(c_b / q_b^(t))·q_b^(t) to backend tile sizes (e.g., AVX2 int8 multiples of 32, LUT per-bit quanta), with a second error-aware reassignment step that promotes/demotes channels by activation-aware gain to keep B_real close to B (Eq. 7).
4. **Bit-specific activation-aware scaling (3.4).** Shares a single exponent α_b ∈ [0,1] per bit-width group so scales follow one power law per bit (s_j = m_j^{α_b}), chosen by grid search to minimize proxy error; cheap to store and folds into packed weights.

**Model compiler (Section 4).**
- **Channel permutation/clustering (4.1).** Factorization O = (A×P_W^T)×(P_W×W) clusters same-bit input rows without changing output; only P_W×W is materialized.
- **Bit blocks and kernel assignment (4.2).** Each clustered weight decomposes into 2/3/4/8/16-bit blocks; low-bit blocks use W{2,3,4}A16 LUT GEMM/GEMV, high-precision blocks use W{8,16}A16 OpenBLAS SIMD.
- **Merging permutations across layers (4.3).** A downstream activation permutation is absorbed into compile-time weight permutations via O' = A×(W×P^T)×(P×W'), so the intermediate activation is produced directly in the required basis; non-linear activations (ReLU/GELU) commute with channel permutations.
- **Graph-based permutation planning (4.4).** A finite graph-labeling problem assigns a channel-basis label π_e to each DAG edge, forming maximal mergeable regions; topological propagation with local conflict resolution inserts shuffles only at incompatible fan-out/fan-in/path-boundary edges, discovering mergeable paths beyond a single transformer block (e.g., FFN output basis matching next attention block's first projection).

**Experimental setup.** Baselines: AWQ, GPTQ, Slim-LLM, AMQ (group size 128, FP16 activations, 128 WikiText-2 calibration sequences). Models: Falcon-H1-3B, Llama2-13B, Qwen3-32B on WikiText-2 perplexity and capped MMLU/WinoGrande/ARC-Easy/ARC-Challenge/HellaSwag. CPUs: Ryzen 9 9950X, Ryzen 7 7840U, Intel N250 (16/8/4 cores) on Ubuntu 24.04; energy via RAPL. Runtime reference is uniformly quantized AWQ on the same T-MAC path at 2/3/4-bit.

## Results & Highlights

- **3-bit WikiText-2 perplexity (Table 1):** PolyQ reaches 5.82 on Falcon-H1-3B vs AWQ 5.96 / GPTQ 6.28 / Slim-LLM 6.15 (AMQ unsupported †); 4.88 on Llama2-13B vs AWQ 5.29 / GPTQ 5.75 / Slim-LLM 5.03 / AMQ 5.68; and 7.66 on Qwen3-32B vs AWQ 8.23 / GPTQ 10.75 / Slim-LLM 10.00 / AMQ 11.29. Stated as 2.4–7.8% over AWQ, 7.4–28.7% over GPTQ, 3.0–23.4% over Slim-LLM, 14.1–32.1% over AMQ at the 3-bit target.
- **Fractional budget realization (Figure 7):** After ISA-aware quanta matching across the 3.0–6.0 b sweep, the maximum average-bit mismatch is only 0.045 b on Falcon-H1-3B, 0.031 b on Llama2-13B, and 0.016 b on Qwen3-32B (within 0.016 b at B=3 and 0.005 b at B=4); raw-map padding drops from 1.30–5.77% to 0.46–1.68%.
- **Activation reorder traffic reduction (Figure 10):** Versus the best prior fixed/template policy (RPTQ- or Atom-style), PolyQ reduces per-token A16 read+write reorder traffic by 46.6% on Falcon-H1-3B (3.28→1.00 MiB/token after full propagation vs 1.88 MiB for RPTQ), 70.8% on Llama2-13B (6.80→1.14 MiB/token vs 3.91 MiB for RPTQ), and 52.8% on Qwen3-32B (14.50→2.95 MiB/token vs 6.25 MiB for RPTQ).
- **CPU latency vs AWQ(T-MAC) (Figure 11, Llama2-13B):** Prefill latency stays within 4.7% on Ryzen 9950X, within 5.8–7.3% on Ryzen 7840U, and within 0.1–0.4% on Intel N250; decode throughput shortfall is ~0.2% on Ryzen 9950X, 8.1–8.2% on Ryzen 7840U, 1.2–6.1% on N250. Latency scales near-ideally: B=2→3 raises prefill by 1.56–1.58× (ideal 1.50×); B=3→4 raises by 1.38–1.46× (ideal 4/3).
- **Energy/token overhead (Table 3, Llama2-13B at B=3):** 1.13% on Ryzen 9950X (6.10 vs 6.03 J/token AWQ+T-MAC), 0.80% on Ryzen 7840U (2.80 vs 2.77), 1.89% on Intel N250 (2.41 vs 2.37) — consistently <2% across all three CPU classes.
- **Peak memory footprint (Figure 8):** At matched integer budgets PolyQ stays within 1.1% of same-backend AWQ(T-MAC) footprint across all three models; for Llama2-13B at B=2.6, storage is 4.75 GiB of the 4.98 GiB peak while layout metadata + reorder buffer contribute only 0.006 GiB, confirming fractional footprint is controlled by weight budget, not mixed-precision bookkeeping.
- **Low-bit recovery threshold (Table 2):** PolyQ identifies model-specific recovery points B_min = 2.1/2.6/2.2 b for Falcon-H1-3B/Llama2-13B/Qwen3-32B, saving 17.1%/11.6%/21.8% peak memory versus a uniform W3 fallback at the first recovered operating point; the W3 overhead versus B_min is 42.9%/15.4%/36.4%.

## Authors & Credibility

Authors are from the University of California, Irvine (Imani Lab). The group recently published T-SAR (arXiv:2511.13676, ref [16]), a full-stack CPU-only ternary LLM inference co-design, which PolyQ builds on as its W{2,3,4}A16 LUT backend. The work is positioned squarely in the same T-MAC/T-SAR lineage of CPU-oriented LLM systems research. Evaluation is unusually thorough on the systems side: three distinct CPU classes (workstation/laptop/mobile), RAPL-measured energy, peak-memory modeling that includes KV cache, FP16 residue, live activations, backend scratch, layout metadata, and reorder buffer rather than weight-only theoretical savings. No public code release is mentioned in the paper.

## Critical Assessment

- **Novelty:** The genuinely novel contribution is the DAG-level permutation propagation (Section 4.4) — turning cross-operator layout regularization into a finite graph-labeling problem that discovers mergeable paths beyond a single transformer block (e.g., FFN output basis matching the next attention block). The per-channel waterfilling itself is a straightforward greedy over an activation-aware proxy similar in spirit to AWQ/Slim-LLM saliency; the {2,3,4,8,16} palette is an engineering choice more than a scientific insight. The claimed "fractional-bit deployment" is essentially the combination of these two ideas plus quanta matching, not a fundamentally new quantization algorithm.
- **Rigor:** The systems-side rigor is strong (three CPU classes, RAPL energy, full working-set memory model, 0.1-bit sweep granularity). However, the accuracy-side comparison is uneven: AMQ "currently does not support Falcon-H1-3B's hybrid attention architecture" (Table 1, †), which conveniently removes the strongest recent AutoML-based mixed-precision baseline from one of three models. Slim-LLM, RPTQ, and Atom are noted as "implement GPU-only execution backends and have no CPU inference path," so direct runtime comparison is only possible against uniformly-quantized AWQ on T-MAC — runtime claims rest on a single baseline.
- **Reproducibility:** No public code or artifact release is mentioned. The quanta values q_b^(t) (e.g., "AVX2 int8 kernels consume channels in multiples of 32") are described only qualitatively for one ISA; ARM NEON/SVE retargeting is hand-waved as "substituting their backend-specific {q_b^(t)}" without validation. The "capped downstream multiple-choice accuracy" protocol (Figure 6) is non-standard and makes cross-paper comparison of MMLU/ARC/HellaSwag numbers difficult.
- **Significance:** Practical value is real for CPU-only edge deployment: the demonstrated 0.1-bit granularity over the 2–4 b range, sub-2% energy overhead, and up to 70.8% reorder-traffic reduction address a genuine gap between coarse inter-layer methods and execution-unfriendly intra-layer mixed precision. But significance is bounded by the CPU-only scope — at batch=1 only, with no GPU/accelerator comparison, and the gains concentrate at the tight 3-bit budget where the field is already crowded.
- **Red flags:** (1) The 2.4–32.1% perplexity range is very wide; the lower bound (2.4% over AWQ on Falcon-H1-3B) is within the kind of noise one might expect from 128-sequence calibration, yet is presented alongside the 32.1% headline figure without discussing calibration-set sensitivity. (2) No ablation isolates the contribution of the activation-aware proxy E[x_j²] versus, e.g., mean-absolute-activation or a GPTQ-style second-order metric; the proxy choice is asserted, not justified empirically. (3) Only three models are tested and none exceed 32B — the claims about scaling to "diverse edge targets" would be stronger with a 70B-class data point. (4) Figure 6 shows "small non-monotonic variations" in downstream accuracy attributed to "sensitivity of discrete benchmark metrics to evaluation scale [19]" (the Schaeffer emergent-abilities critique) — a convenient way to hand-wave away non-monotonic accuracy behavior that could equally indicate quantization instability.

## Strengths

- Genuinely end-to-end: co-designs quantizer, kernel assignment, and graph-level layout, with one budget parameter B predicting quality, memory, latency, throughput, and energy.
- Strong systems evaluation: three CPU tiers (workstation/laptop/mobile), RAPL-measured energy, full working-set peak-memory accounting rather than weight-only theoretical savings.
- Compelling reorder-traffic reduction (up to 70.8% vs best prior fixed/template policy) with a clean per-token A16-traffic model that isolates layout policy from quantizer/packing/scheduler differences.
- The compile-time factorization (Eq. 9, Eq. 10) ensures P_W is never materialized at runtime and the merge rule propagates across transformer-block boundaries.
- Honest about Slim-LLM's marginal advantage at 5–6 b ("within 0.01–0.03 perplexity") and about the GPU-only baselines precluding direct runtime comparison.

## Limitations

- No public code release mentioned; quanta values and ARM NEON/SVE retargeting are under-specified.
- Only three models (3B/13B/32B); no 70B+ validation; no batch>1 study despite LLM edge inference often involving small batching.
- AMQ missing on Falcon-H1-3B (†) weakens the comparison on one of three models; runtime comparison rests on a single baseline (AWQ+T-MAC).
- Downstream accuracy uses a non-standard "capped multiple-choice accuracy" protocol, hindering cross-paper comparison.
- No ablation on calibration set size, no comparison of activation-aware proxy choices, no sensitivity analysis of the α_b grid search.
- Non-monotonic downstream accuracy variations are dismissed via the Schaeffer et al. emergent-abilities critique rather than investigated.
- The {2,3,4,8,16} palette lacks 5/6/7-bit entries by design — this is presented as a feature (LUT-friendly) but also caps the achievable quality ceiling at a given average budget versus finer-grained palettes.

## Relevance

PolyQ 直接命中「LLM 量化 + CPU/边缘推理」这条主线，其「编译器协同的分数比特预算控制」视角对端侧/PC 端 LLM 部署（量化、推理加速、内存受限场景）有强参考价值，可作为 mixed-precision PTQ 与 model compiler 协同设计的代表性工作跟进。

## Questions for Further Exploration

- How does PolyQ's fractional-bit advantage change with calibration set size (e.g., 32 vs 128 vs 512 sequences) — is the 2.4% lower-bound gain robust or an artifact of the 128-sequence calibration?
- Can the DAG-level permutation propagation generalize to architectures with non-trivial channel mixing across attention/FFN boundaries (e.g., Mamba, hybrid SSM-transformer like Falcon-H1 itself, MoE expert routing)?
- What is the quality/latency trade-off at 70B+ scale, where the absolute reorder traffic and the cost of OpenBLAS SIMD dispatches for 8/16-bit blocks may scale differently?
- How sensitive is the activation-aware proxy E[x_j²] to activation distribution shift between calibration and deployment (e.g., code vs natural language, long-context vs short)?
- Could the quanta-matching snap stage be extended to GPU group-quantization tile sizes to make the poly-precision palette portable across CPU+GPU heterogeneous edge platforms?
