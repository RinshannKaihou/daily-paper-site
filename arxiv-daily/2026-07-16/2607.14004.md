> **一句话总结：** 作者提出一个两阶段持续学习评测协议（Terminal-Bench 2.0 上 12+10 个 hard 任务），发现 GEPA、Meta Harness、RELAI-VCL 在静态单阶段上都能超过基线，但只有把"无回归约束"写进搜索循环内的 RELAI-VCL 同时实现正向迁移和二次优化增益，lifelong 平均通过率 76.4%，显著优于 GEPA(66.0%)、Meta Harness(64.6%) 和基线(58.7%)。

> **TL;DR:** This RELAI.ai technical report introduces a two-phase continual-learning evaluation on 22 hard Terminal-Bench 2.0 tasks (12 in T1, 10 in T2) and shows that while GEPA (70.8%), Meta Harness (66.6%), and RELAI-VCL (79.2%) all beat the baseline (62.5%) in the static Phase-1 setting, only RELAI-VCL both transfers positively to unseen T2 tasks (72.7% vs GEPA's below-baseline 54.5%) and keeps improving under Phase-2 re-optimization (77.3% vs Meta Harness's stalling at 59.1%), reaching the highest lifelong average of 76.4% — a +17.7-point delta over the baseline — with the authors attributing compounding to in-loop regression control.

## 核心贡献

- 提出两阶段持续学习评测协议：Phase 1 在初始任务集 T1 上优化，然后不进一步优化直接评测在 T1∪T2 上的迁移能力；Phase 2 从 Phase 1 优化后的 agent 出发，在 T1∪T2 上做第二次等预算优化。该协议把单个 benchmark 分数所混淆的三种属性（静态优化强度、对未见任务的迁移、把新任务纳入目标后能否继续改进）显式拆开。
- 在 Terminal-Bench 2.0 的 22 个 hard 任务（T1=12 个 900s 超时任务，T2=10 个 1800s 超时任务）上，对 GEPA、Meta Harness、RELAI-VCL 三种 agent-harness 优化器做了等预算（每阶段 200 rollouts）、共享相同基线 agent（GPT-5.5 / TerminusKira）的头对头比较。
- 给出经验证据：要让优化增益"复利累积"，回归控制必须内嵌进搜索循环里，而非事后分析。RELAI-VCL 因拒绝任何以牺牲已解任务为代价的候选编辑，是唯一同时获得正向迁移和持续改进的方法。
- 量化刻画了三种典型失败模式：GEPA 对 Phase 1 过拟合（迁移落到基线以下），Meta Harness 迁移好但二次优化停滞（Phase 2 每个候选都比现有 agent 差），而单看 lifelong 平均分（GEPA 66.0% vs Meta Harness 64.6%）几乎相同，掩盖了截然相反的失败。
- 讨论了更贴近真实部署的持续学习 benchmark 应满足的条件（任务间相关、单条轨迹有限观测、不完美验证器、多轮长程优化）。
- 公开了全部基线、Phase 1、Phase 2 的 agent artifacts（https://github.com/relai-ai/Continual-Learning-Terminal-Bench）。

## Key Contributions

- A two-phase continual-learning protocol that decomposes a single benchmark score into three measurable properties — static optimization strength on T1, transfer to an expanded task union T1∪T2, and the ability to keep improving once T2 is folded into the search objective — applied to Terminal-Bench 2.0 hard tasks.
- A head-to-head, equal-budget (200 rollouts/phase) comparison of three agent-harness optimizers (GEPA, Meta Harness, RELAI-VCL) starting from an identical GPT-5.5 baseline agent across 22 hard Terminal-Bench 2.0 tasks (T1 = 12 tasks at 900s timeout, T2 = 10 tasks at 1800s timeout).
- Empirical evidence that regression control must live inside the search loop (rejecting candidate edits that trade previously solved behavior for new-task gains), not as post-hoc analysis, for optimization gains to compound across phases.
- A characterization of three distinct failure modes that a static score hides: GEPA overfits Phase 1 (negative transfer below the unoptimized baseline), Meta Harness transfers well but stalls under re-optimization (every Phase-2 candidate worse than its Phase-1 agent), and these opposite failures produce nearly identical lifelong averages (66.0% vs 64.6%).
- A roadmap for more realistic continual-learning agent benchmarks requiring correlated task streams, limited observability (single trajectory), imperfect/partial verifiers, and many repeated optimization rounds.
- Public release of all baseline, Phase-1, and Phase-2 agent artifacts for the three optimizers.

## Methodology

The paper builds a minimal continual-learning evaluation around the "compounding question": after an agent is optimized once on T1, can the same method be applied again on an expanded task set T1∪T2 without unwinding the first round's gains? The protocol instantiates three evaluation stages from a common baseline agent A0:

1. **Phase 1 (static optimization).** Each optimizer receives a 200-rollout budget to optimize A0 against the 12 tasks in T1, producing A1, then is evaluated on T1 (the conventional static score).
2. **Transfer.** The Phase-1-optimized agent A1 is evaluated on the full 22-task union T1∪T2 with no further optimization, measuring generalization to tasks that never drove search.
3. **Phase 2 (re-optimization).** Starting from its own A1, each optimizer receives another identical 200-rollout budget on T1∪T2, producing A2, which is then evaluated on the union. This tests whether the method can keep improving on top of its own earlier edits.

The single-number summary is **LifelongAvg = (Phase-1 + Transfer + Final)/3**.

Three optimizers are compared under matched conditions, all sharing the same GPT-5.5 baseline agent ("TerminusKira," a Harbor/Terminus2 wrapper with native tool calling):

- **GEPA** — reflective evolutionary prompt mutation; its code-mutation variant (GEPA-code) failed to produce a valid candidate in Phase 1 and was excluded, so only the prompt-only variant carries forward. Regression handling: post-hoc only.
- **Meta Harness** — an outer loop in which Codex (GPT-5.5) directly edits harness code, with candidates accepted/rejected on current-task-set feedback. Regression handling: post-hoc only.
- **RELAI-VCL** — searches the broadest space (prompt, tool, workflow, memory, skill, code) but enforces a no-regression constraint inside the candidate-selection loop, rejecting any edit that improves new tasks by regressing on previously solved tasks.

Every task is run twice per agent and scored as the fraction of trials solved (0, ½, or 1). All methods use GPT-5.5 as the proposer model and an identical per-phase rollout budget to keep the comparison controlled.

## Results & Highlights

- **Phase 1 (static, T1, 12 tasks):** RELAI-VCL reaches 79.2% versus the baseline's 62.5%, a +16.7-point gain, and beats the next-best optimizer GEPA (70.8%) by +8.4 points; Meta Harness sits at 66.6%.
- **Transfer (T1∪T2, 22 tasks, no Phase-2 optimization):** RELAI-VCL transfers best at 72.7%; Meta Harness at 68.2%; the unoptimized baseline is 56.8%; and GEPA transfers negatively at 54.5%, falling below the baseline it beat in Phase 1 — evidence of Phase-1 overfitting. RELAI-VCL's pass rate on the unseen T2 tasks alone is 65.0%, about +15 points above baseline despite never optimizing against those tasks.
- **Phase 2 final (T1∪T2 after re-optimization):** RELAI-VCL reaches 77.3%; GEPA recovers to 72.7% once Phase-2 tasks enter its objective; Meta Harness regresses to 59.1% because every candidate it generated in the second round scored worse than its Phase-1 agent.
- **Lifelong average:** RELAI-VCL 76.4% vs GEPA 66.0% vs Meta Harness 64.6% vs baseline 58.7% — RELAI-VCL leads by +10.4 points over the best alternative and +17.7 points over the baseline, and is the only method that leads at all four evaluated stages.
- **Mechanistic evidence:** GEPA's prompt grew from 5 lines to 103 (Phase 1) then 195 (Phase 2) lines, with named per-task lessons hardcoding Terminal-Bench task IDs (e.g., `configure-git-webserver`), exact file paths, and literal verifier strings — a concrete overfitting signature. Meta Harness's accepted candidates were generic robustness fixes (command-completion regex, output-noise compaction) with no task literals, which explains its good transfer but conservative Phase-1 gain. RELAI-VCL added a generic task-type classifier and a verifier-shaped completion gate with no task IDs or literal outputs.

## Authors & Credibility

- Authors are **Wenxiao Wang, Priyatham Kattakinda, and Soheil Feizi**, all affiliated with **RELAI.ai** — the same organization that produces RELAI-VCL, the method that wins the comparison. Soheil Feizi is a well-known ML researcher (UMD/Reliable AI) with a substantial publication record, which lends technical credibility to the experimental design, but the paper is explicitly labeled a "Technical Report" and is a vendor self-evaluation.
- Baseline, Phase-1, and Phase-2 artifacts for all three optimizers are publicly released on GitHub (https://github.com/relai-ai/Continual-Learning-Terminal-Bench), including the full prompts and code diffs, which materially improves reproducibility and auditability of the per-method claims (e.g., GEPA's hardcoded task IDs).
- The framing is unusually careful for a vendor report: the compounding interpretation is repeatedly hedged as tied to "this specific protocol and task population" rather than a general law, and limitations get a dedicated section.

## Critical Assessment

- **Novelty:** The protocol (decomposing a static benchmark score into static/transfer/re-optimization) is a genuinely useful contribution and, per the authors, no existing agent-harness optimizer evaluation directly measures both transfer and re-optimization. However, the conceptual ingredients (Goodhart's-law benchmark overfitting, IFBench-style overfitting to verifiable constraints, EWC-style regression regularization) are drawn from prior work; the novelty is in applying and isolating them for harness optimization.
- **Rigor:** The matched-budget, shared-baseline, shared-LLM design is well controlled, and the authors publish exact numbers and per-stage tables. But three issues weaken the comparison. (1) **Search-space asymmetry**: RELAI-VCL searches over "prompt, tool, workflow, memory, skill, code," whereas GEPA is restricted to prompts and Meta Harness to harness code — so RELAI-VCL is given strictly more expressive search space under the same rollout budget, confounding the regression-control claim with a larger hypothesis class. (2) **Disabled competitor config**: GEPA-code (the stronger code-mutation variant) "failed to produce a valid candidate" in Phase 1 and is excluded from all Phase-2 comparisons, so GEPA is handicapped to its prompt-only variant. (3) **Tiny N**: 22 tasks, 2 trials each, so each pass rate is in units of ~1/44; differences of a few points (e.g., Meta Harness Transfer 68.2% vs RELAI-VCL 72.7%) are within plausible run-to-run noise, and no variance/significance is reported.
- **Reproducibility:** Strong on artifacts (released code/prompts/diffs, named LLM GPT-5.5, fixed 200-rollout budget, explicit task IDs in Appendix A). Weak on the RELAI-VCL method itself: the paper describes it at a conceptual level ("constrained candidate search") but the actual regression-detection logic and search procedure are not specified in enough detail to reimplement independently — the reader depends on the artifact repo.
- **Significance:** The core message — static benchmark gains do not predict continual-learning behavior, and in-loop regression control matters — is valuable and actionable regardless of which method wins. But because the winning method is the authors' own product evaluated under a search-space handicap favoring it, the headline "RELAI-VCL is the only method that compounds" should be read as a hypothesis supported by this specific setup, not a settled result.
- **Red flags:** Vendor self-evaluation with the vendor's product winning; the competing method (GEPA) is quoted largely from a 2025 arXiv preprint and its stronger code variant was dropped; no statistical significance, confidence intervals, or seed-variance analysis on a 22-task benchmark; "LifelongAvg" is the authors' own aggregate defined to make a three-way tradeoff collapse to one number.

## Strengths

- The phased protocol cleanly separates three properties (static strength, transfer, re-optimization) that a single score conflates, and the data shows these properties can diverge sharply — a useful methodological contribution independent of the winner.
- Honest, detailed per-method artifact analysis: the paper inspects *what* each optimizer actually changed (GEPA's hardcoded task IDs, Meta Harness's generic IO fixes, RELAI-VCL's verifier gate) and ties those qualitative differences to the quantitative transfer/stall patterns, rather than reporting only aggregate scores.
- Transparent about limitations — a whole section on loosely-related tasks, verifier availability, and the gap to realistic benchmarks, plus explicit hedging that the compounding interpretation is protocol-bound.
- Open release of all agent artifacts and full prompt/code diffs, enabling independent audit of the overfitting and shortcut claims.
- Connects the agent-harness setting to the established continual-learning literature (EWC, catastrophic forgetting) and to the Goodhart's-law/IFBench overfitting literature, giving the work a clear conceptual scaffolding.

## Limitations

- Only 22 hard tasks total (12 + 10), with just 2 trials per task; no variance, confidence intervals, or significance testing is reported, so small point differences are not demonstrably meaningful.
- Asymmetric search spaces favor RELAI-VCL (full harness package) over GEPA (prompts only) and Meta Harness (harness code only) under identical rollout budgets; an equal-expressiveness comparison would be more convincing.
- GEPA's code-mutation variant was excluded after failing to produce a valid Phase-1 candidate, so the strongest GEPA configuration did not actually participate in Phase 2.
- Vendor self-evaluation: the authors are from RELAI.ai and RELAI-VCL is their product, creating an inherent conflict of interest that the paper does not fully mitigate (e.g., no neutral third method with in-loop regression control is tested).
- The tasks in Terminal-Bench 2.0 are largely independent across distinct domains, unlike production failures that share tools/data/policies; the authors flag this but it limits external validity.
- The RELAI-VCL method is described conceptually but not specified precisely enough to reimplement without the artifact repo; the "generalization filter" interpretation is acknowledged as speculative and not separable from alternative explanations under this protocol.
- Only two phases are studied; whether these compounding patterns hold over many rounds, longer horizons, or distribution shift between rounds is untested.

## Relevance

这项工作对"agent 自我改进/持续优化"方向高度相关：它直接回答了"反复用优化器改 agent harness 会不会把之前的收益改没"这一部署场景的核心问题，并指出把回归检测写进搜索循环是关键，这对构建可长期演进、不自毁的 agent 系统有直接指导价值。如果你关注 agent benchmark 过拟合、prompt/工具自动优化（GEPA/Meta Harness/Maestro 一类）的可靠性评估，本文提出的两阶段协议是一个值得借鉴的评测范式。

## Questions for Further Exploration

- Would RELAI-VCL still win if GEPA and Meta Harness were given the same broad search space (prompt + tool + workflow + memory + code) rather than being restricted to prompts or harness code only?
- How do the rankings change if the disabled GEPA-code variant is repaired and included, or if a neutral third-party optimizer with in-loop regression control is added?
- Do the compounding patterns persist over 5+ phases or under distribution shift between phases, rather than just the two phases studied here?
- On a benchmark with correlated, same-domain tasks (closer to production failures), does regression control help as much, or do cross-task shortcuts change the picture?
- What is the run-to-run variance of these pass rates (e.g., over random seeds and proposer stochasticity), and which of the reported point differences are statistically significant given only 22 tasks and 2 trials each?
- Can the "regression control = generalization filter" interpretation be separated from confounds (larger search space, different accepted edits) by an ablation that adds no-regression rejection to GEPA and Meta Harness directly?
