# Dual-Pool Token-Budget Routing for Cost-Efficient and Reliable LLM Serving

## Paper Summary

This paper addresses a fundamental inefficiency in production LLM serving: homogeneous fleets provision every GPU instance for the worst-case context length (e.g., 64K tokens), even though 80-95% of real requests are short. The authors call this **configuration-traffic mismatch** and propose a lightweight dual-pool routing mechanism that splits a fleet into a high-throughput short-context pool and a high-capacity long-context pool, achieving 31-42% GPU reduction, 5.4x fewer preemptions, and multi-million-dollar annual savings.

## Key Contributions

1. **Token-budget pool routing**: A fleet-level dispatch algorithm that splits a homogeneous vLLM fleet into right-sized short and long pools with O(1) routing overhead. It operates across instances (unlike per-GPU optimizations) and composes with PagedAttention, chunked prefill, and prefill-decode disaggregation.

2. **Self-calibrating token estimation**: A per-category exponential moving average (EMA) that learns the bytes-per-token ratio from `usage.prompt_tokens` feedback, with asymmetric-error-aware conservative bias. This eliminates the need for a model-specific tokenizer at the routing layer -- a practical necessity in multi-model deployments.

3. **Closed-form cost model**: The formula `savings = alpha * (1 - 1/rho)` predicts fleet-level GPU savings from two observable quantities: the short-traffic fraction (alpha) and the throughput gain ratio (rho). This allows teams to audit savings potential before deployment without simulation.

4. **Comprehensive evaluation**: On Azure LLM Inference Dataset and LMSYS-Chat-1M traces, serving Llama-3-70B on A100 GPUs, plus a Qwen3-235B-A22B case study on AMD MI300X.

## Technical Details

### Problem Analysis

- On an 80 GB A100 serving Llama-3-70B, `max_model_len=64K` yields only ~16 concurrent sequences; reducing to `max_model_len=8K` yields ~128 sequences -- an 8x concurrency gain.
- Chunked prefill solves compute scheduling but not the memory provisioning problem: KV cache is still allocated for the full sequence, and `Cmax` still dictates concurrency limits.
- The mismatch causes OOM crashes, preemption storms, request rejections, and head-of-line blocking.

### Routing Mechanism

Each request's total token budget is estimated as: `Ltotal = ceil(|r| / c_hat_k) + r.max_output_tokens`, where `c_hat_k` is the per-category bytes-per-token ratio. The router uses a conservative estimate (`c_route = c_hat - gamma * sigma_hat`) with `gamma=1.0` to bias borderline requests toward the safer long pool.

**Three-step dispatch**: (1) Hard capacity check -- if `Ltotal > Cmax(short pool)`, route to long pool; (2) Budget-based routing -- compare against threshold `Bshort`; (3) Load-aware spillover -- redirect to alternate pool if preferred pool is overloaded.

**Calibration**: The EMA with `beta=0.95` converges within ~50 observations per category to within 3.5% of true values, reducing mis-routing from 4.1% (global static ratio) to under 1%.

### Cost Model

The formula `savings = alpha * (1 - 1/rho)` where:
- `alpha = F(Bshort)` is the fraction of short traffic
- `rho = mu(Cshort) / mu(Chigh)` is the throughput gain ratio

For typical production values (alpha=0.80, rho=4), predicted savings = 60%. The formula is a conservative upper bound because it ignores PagedAttention occupancy headroom, activation memory asymmetry, and KV-read bandwidth benefits.

**Qwen3-235B-A22B on MI300X at 10,000 req/s**: 197 -> 137 nodes (30.5% reduction), saving $15.4M/year at cloud rates.

The threshold `Bshort` is forgiving: any value in 4K-16K tokens achieves >80% of peak savings. Savings are scale-invariant across request rates (100-2000 req/s).

## Critical Assessment

**Strengths**:
- The problem identification is sharp and well-motivated with real production data. The "configuration-traffic mismatch" framing is compelling and likely resonates with anyone operating vLLM fleets.
- The solution is elegantly simple: O(1) routing, no tokenizer dependency, and composability with existing optimizations. This is a practical engineering contribution rather than a theoretical one.
- The closed-form cost model is a useful operational tool -- teams can estimate savings before committing to deployment.
- The self-calibrating EMA design is well-thought-out, particularly the conservative bias for asymmetric routing errors and per-category tracking for multilingual traffic.

**Weaknesses**:
- The evaluation relies entirely on simulation (calibrated against Vidur), not actual deployment. The gap between the cost model prediction (60%) and simulated result (41.9%) is 18 percentage points, which is substantial. Real-world results could differ further due to network overhead, load balancing dynamics, and traffic pattern shifts.
- The paper assumes Poisson arrivals, which is a simplification. Production LLM traffic often exhibits burstiness and diurnal patterns that could affect spillover dynamics.
- The dual-pool approach introduces operational complexity: capacity planning now requires reasoning about two pool sizes, threshold tuning, and monitoring two separate sets of SLOs. The paper acknowledges this briefly but does not deeply analyze the operational overhead.
- The claim of "no tokenizer needed" is somewhat misleading -- the router still needs content category labels for per-category calibration. How these labels are obtained (user-specified? classifier?) is not discussed in detail.
- The Qwen3-235B/MI300X case study is a projection, not an evaluation. It applies the cost model formula rather than running a simulation, which weakens the frontier hardware validation claim.
- The two-pool design is presented as optimal, but no systematic comparison with three-pool or adaptive multi-pool designs is provided beyond a brief mention of "marginal ~2% improvement."

**Questions for the authors**:
- How sensitive are results to the EMA decay rate (beta=0.95) under concept drift (e.g., a sudden shift from English to CJK traffic)?
- What happens when the short-traffic fraction changes significantly during deployment (e.g., a new feature generates longer prompts)?
- How does the spillover mechanism perform under correlated burst arrivals?

## 中文摘要

本文针对生产环境中 LLM 服务的一个核心低效问题：同构 GPU 集群为最坏情况上下文长度（如 64K tokens）配置每个实例，但实际 80-95% 的请求是短文本。作者将这一现象定义为 **"配置-流量失配"**（configuration-traffic mismatch），并提出了双池 token 预算路由机制：将同构集群拆分为高吞吐短上下文池和高容量长上下文池，根据请求的估计总 token 预算进行路由。

核心技术贡献包括：(1) O(1) 复杂度的路由算法，无需 tokenizer，通过 EMA 在线学习每类别的字节-token 比；(2) 闭式成本模型 `savings = alpha * (1 - 1/rho)`，可在部署前预估节省幅度；(3) 在 Azure 和 LMSYS 真实轨迹上的全面评估，GPU 使用量降低 31-42%（年节省 $2.86M），抢占率降低 5.4 倍，P99 TTFT 改善 6%。

该方法与 PagedAttention、连续批处理、预填充-解码分离等优化正交互补，工程实用性强。主要不足在于：评估完全基于仿真（未实际部署），成本模型预测与仿真结果差距达 18 个百分点；路由阈值 Bshort 在 4K-16K 范围内鲁棒，但缺乏对多池设计的系统比较；Qwen3-235B/MI300X 的案例研究仅为公式投影而非实际测试。
