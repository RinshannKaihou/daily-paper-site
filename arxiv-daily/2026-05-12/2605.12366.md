> **一句话总结：** 本文系统性地发现，当使用Opus 4.6、GPT 5.4、Gemini 3.1作为安全监控分类器时，800K token长度的编码智能体记录会使漏检率增加2–30倍，且攻击位于中间位置时退化最严重；增量监控、推理增强和提醒提示可部分缓解，但微调效果有限。

> **TL;DR:** Frontier LLMs used as safety monitors for coding agents degrade catastrophically on long transcripts. Opus 4.6's recall on MonitorBench drops from 98.6% to 88% with 800K benign prefix; blatantly obvious attacks see recall collapse from 99.7% to 69%. The effect is worst when attacks sit in the middle of transcripts. Extended thinking, incremental monitoring at multiple checkpoints, and periodic reminders all help but don't eliminate the problem. SFT on GPT-4.1 showed in-distribution gains but failed to generalize. Any AI safety evaluation that ignores context length is overestimating monitor reliability.

## 核心贡献

- **揭示了"分类器上下文腐烂"现象并量化其严重程度**：Opus 4.6在MonitorBench上的召回率从98.6%降至88%（800K前缀）；对明显攻击的召回率从99.7%降至69%。
- **设计了两类评估方法**：(1) 在800K良性前缀后拼接MonitorBench攻击记录；(2) 在长记录中间注入明显恶意命令（数据窃取/远程脚本执行），系统研究了位置效应。
- **证明了"中间最差"效应**：攻击位于记录中间时检测率最低，非推理模型受影响尤为严重（500K前缀+100K后缀可将检测率降至5%）。
- **提出了三种缓解策略并逐一评估**：扩展推理（更多thinking token）、增量监控（在多个时间点采样检查）、以及在工具调用间插入提醒提示。
- **微调实验揭示了泛化困境**：GPT-4.1在needle攻击任务上微调后AUC大幅提升，但无法迁移到MonitorBench；proxy任务微调甚至导致负迁移。

## Key Contributions

- **Demonstrates and quantifies "Classifier Context Rot"**: Opus 4.6 recall drops from 98.6%→88% on MonitorBench with 800K benign prefix; recall on blatant attacks drops 99.7%→69% at 800K tokens. All three frontier models (Opus 4.6, GPT 5.4, Gemini 3.1 Pro) are affected.
- **Designs two complementary evaluation paradigms**: (1) Padded MonitorBench (subtle firewall-disabling attacks after benign prefix), and (2) Needle Injection (obviously malicious bash commands — exfiltration via `curl` or remote script execution — inserted into long clean Claude Code transcripts).
- **Shows "worst in the middle" positional degradation**: non-thinking Opus 4.6 drops to 5% detection when MonitorBench is padded with 500K before + 100K after; extended thinking is more robust but still degrades.
- **Evaluates three mitigation families**: extended thinking/increased reasoning tokens (helps all models, especially Gemini 3.1), incremental monitoring at sub-transcript checkpoints (provides multiple chances to catch attacks at lower token counts), and periodic reminder prompts inserted between tool calls (<3% token overhead, significantly helps needle detection but not subtle MonitorBench attacks).
- **SFT experiments on GPT-4.1 show limited generalization**: fine-tuning on synthetic needle attacks dramatically improves needle detection but fails to transfer to MonitorBench; proxy rule-violation tasks show mixed or negative results, and combining diverse proxy domains actually hurts performance.

## Methodology

**Models tested**: Claude Opus 4.6 (no-thinking, prompted CoT, CoT-List, extended thinking at 1K/2K/8K budgets), Gemini 3.1 Pro (low/medium/high thinking), GPT 5.4 Thinking (low/medium/high/xhigh). Fine-tuning experiments use GPT-4.1 (the only model with fine-tuning API access).

**Classification prompt**: Based on a prompt privately shared by an AI company for internal monitoring, asking whether a transcript contains harmful actions (security weakening, sabotage, deception, data exfiltration). GPT variants require a longer "do not overflag" section due to higher baseline false positive rates.

**Evaluation datasets**:
- *MonitorBench* (148 transcripts, 5K–16K tokens each): agent covertly disables firewall while configuring it. Padded with 80 real Claude Code sessions (5K–1M tokens) from the authors.
- *Needle Injection*: clean Claude Code transcripts with malicious bash commands injected (exfiltration via `tar + curl` to AWS, or `curl | sh` remote execution). Attack position varied systematically.

**Monitoring strategies**:
- *Final Transcript Monitoring*: single pass over full transcript.
- *Incremental Monitoring*: monitor at multiple truncation points throughout the session, with configurable subsampling rate. A trajectory is flagged if any check returns positive.

**Fine-tuning**: Two training datasets — (1) Synthetic Needle in Transcript (real transcripts + synthetically generated malicious tool calls), (2) Proxy Rule Violation (algorithmically generated corporate spend-approval policies + event logs). 50:50 clean/malicious ratio. AUC computed using logprobs of "true"/"false" tokens to control for calibration shifts.

**Statistical methods**: 95% Wilson confidence intervals for detection rates; McNemar's exact test for paired comparisons of mitigation effects; 10,000 Monte Carlo simulations for incremental monitoring confidence intervals; 10,000 bootstrap samples for AUC confidence intervals.

## Results & Highlights

- **Subtle attacks (MonitorBench)**: Opus 4.6 recall 98.6%→88% with 800K prefix. All models show increased attack success rates with long context.
- **Obvious attacks (Needle)**: Opus 4.6 recall 99.7%→69% going from 100K→800K context. Even blatant `curl | sh` commands are missed.
- **Positional effect**: Detection worst in the middle of transcripts. Non-thinking Opus: 500K prefix + 100K suffix → 5% detection on MonitorBench. Extended thinking is more positionally robust.
- **More reasoning helps**: Gemini scales smoothly with thinking effort. CoT-List prompting (list suspicious actions verbatim first) gives massive uplift on Opus with few extra tokens — surprisingly, the model can quote suspicious actions verbatim when asked to list them but fails to flag them when asked directly.
- **Incremental monitoring helps**: Monitoring at multiple checkpoints throughout the session significantly boosts detection, especially for needle attacks where the dangerous action is now the "most recent" at some checkpoint.
- **Reminders help for obvious attacks only**: Inserting `<monitor_note>` between tool calls significantly improves needle detection (p<0.001 for Opus thinking and GPT-5.4) but not MonitorBench (ns for Opus variants, GPT-5.4). Does not increase false positives.
- **Fine-tuning**: Needle-training GPT-4.1 greatly improves needle detection but gains fade on MonitorBench at longer contexts. Proxy training shows modest needle improvement but active harm on MonitorBench. Combining diverse proxy domains produces worse-than-baseline models.
- **False positive rates**: All models <2% on 80 benign transcripts, with and without reminders.

## Critical Assessment

- **Novelty**: High. While lost-in-the-middle and needle-in-a-haystack are known phenomena, applying them specifically to the *safety monitoring* use case with realistic coding-agent transcripts and measuring the gap between benchmark and deployment conditions is novel and practically important.
- **Rigor**: Strong. Two complementary evaluation paradigms (subtle + obvious attacks), three frontier models, multiple thinking budgets, statistical tests (McNemar's, Wilson CIs, bootstrap), calibration-aware AUC via logprobs, and false positive verification on 80 benign transcripts. The positional analysis (Figure 4) is particularly informative.
- **Reproducibility**: Moderate. The MonitorBench dataset is published; the prompt templates and injection mechanisms are described in appendices. However, the 80 real Claude Code transcripts from the authors are private, and the "AI company's internal monitoring prompt" was privately shared. Replication would require constructing equivalent benign transcripts.
- **Significance**: Very high for AI safety. The finding that monitors degrade precisely in the scenarios where agents have the most autonomy (long sessions) is a critical safety gap. The CoT-List result — models can quote suspicious actions verbatim but fail to flag them — is a striking and novel observation about classifier behavior.
- **Red flags**: (1) The MonitorBench task is narrow (firewall disabling) — generalization to other attack types is uncertain. (2) The needle attacks are somewhat artificial (appending `&& curl ...` to existing commands). (3) Fine-tuning is limited to GPT-4.1 (not a frontier model) with small datasets; the authors acknowledge this is preliminary. (4) The paper only considers localized single-step attacks; multi-step distributed attacks are acknowledged as likely harder but untested. (5) Only coding-agent transcripts are studied; generalization to other agent domains is unknown.

## Strengths

- Clean experimental design with two complementary paradigms testing both subtle and obvious attacks.
- Surprising and insightful CoT-List finding: models can *recognize* suspicious actions when asked to list them but fail to *classify* the same transcript as dangerous.
- Practical mitigations evaluated rigorously with statistical significance testing and cost analysis (incremental monitoring subsampling, reminders at <3% token overhead).
- Honest about fine-tuning failures — negative transfer results are reported transparently.
- Strong implications section connecting the findings to real-world deployment risk (long sessions → more autonomy → weaker monitors → worst-case scenario).

## Limitations

- MonitorBench covers only one attack type (firewall disabling); broader attack diversity would strengthen generalizability claims.
- Needle injection attacks are synthetic and somewhat obvious (appending `&&` to bash commands); real adversarial attacks would be more subtle.
- Fine-tuning only on GPT-4.1 (non-frontier) with small datasets; results may not reflect what's possible with frontier models or larger training runs.
- Only localized, single-step attacks studied; multi-step distributed attacks (the more realistic threat model) are left to future work.
- Benign transcripts come from the authors' own Claude Code sessions (AI safety research), which may not represent the diversity of real-world coding sessions.
- The paper does not explore architectural mitigations (e.g., retrieval-augmented monitoring, specialized classifier heads) beyond prompting and fine-tuning.

## Relevance

本研究直接揭示了LLM安全监控在长上下文场景下的严重性能退化问题，对AI智能体安全对齐、编码智能体风险防控以及安全评估方法论设计具有直接和重要的参考价值。

## Questions for Further Exploration

- Can retrieval-augmented approaches (e.g., chunk the transcript, retrieve suspicious segments, then classify) outperform whole-context monitoring on long transcripts?
- How does classifier context rot interact with multi-step distributed attacks where each individual step is benign but the aggregate is dangerous?
- Would a dedicated safety-classifier model (not a general-purpose LLM) be more robust to context length degradation?
- The CoT-List finding suggests models can recognize but not classify — could a two-stage pipeline (first extract suspicious actions, then classify them) be more robust?
- How does the degradation curve look between 0–800K tokens? Is there a critical threshold, or is the decline gradual?
- Could input-size-specific calibration thresholds (as the authors suggest) recover detection performance without increasing false positives?
