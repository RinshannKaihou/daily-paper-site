It's Not the Size: Harness Design Determines Operational Stability in Small Language Models

arXiv: 2605.12129

Date: 2026-05-12

https://arxiv.org/abs/2605.12129

一句话总结： 本文通过实验证明，对2-3B参数的小语言模型，精心设计的4阶段执行管线（规划→执行→验证→恢复）可将任务成功率从0.762提升至0.952，且发现最小化的外壳包裹反而会降低性能（非单调效应），揭示了管线设计比模型规模更能决定小模型的操作稳定性。
TL;DR: A systematic study showing that a 4-stage execution harness (plan→execute→verify→recover) boosts SLM task success rate to 0.952 on Gemma4 E2B, reveals a non-monotonic effect where minimal wrappers hurt performance, and introduces the concept of "scaffold collapse" where LLaMA 3.2 abandons JSON format without harness support.

核心贡献

  • 系统测量了三种管线复杂度（无管线、最小外壳、4阶段管线）对2-3B小语言模型操作性能的影响，覆盖3个模型和24个任务
  • 发现并表征了非单调现象：最小外壳的TSR反而低于无管线基线（Gemma和Qwen均观察到此现象），表明"半吊子"的管线设计可能适得其反
  • 提出"脚手架坍塌"（scaffold collapse）概念：LLaMA 3.2在无管线支持时放弃JSON格式结构（TSR=0.429），但VTSR=0.952说明内容生成能力完好
  • 消融实验显示规划阶段和恢复阶段各贡献约24.7%的总增益，验证捕获率VCR=0.625
  • 将故障模式按管线可修复性分类（可修复、部分可修复、不可修复），为实际部署提供故障预期框架

Key Contributions

  • Systematic measurement of three harness levels (model-only, minimal-shell, 4-stage pipeline) across 3 SLMs and 24 tasks, using TSR and VTSR metrics
  • Discovery of a non-monotonic harness effect: minimal wrapper tags reduce TSR below raw model-only in both Gemma4 E2B and Qwen3.5:2B
  • Introduction of "scaffold collapse" — LLaMA 3.2(3B) abandons JSON formatting under complex requirements without harness support (TSR=0.429 vs. VTSR=0.952)
  • Ablation study quantifying planning and recovery each contributing ~24.7% of total pipeline gain; verification catch rate (VCR) = 0.625
  • Failure mode taxonomy classifying errors as harness-fixable, partially fixable, or unfixable, providing a practical remediation framework

Methodology

The authors design a controlled experiment with three harness conditions applied to three SLMs (Gemma4 E2B 2B, Qwen3.5:2B, LLaMA 3.2 3B) across 24 tasks in 6 categories (structured knowledge, search-and-ground, comparison, workflow completion, constraint-sensitive, web search):

Harness conditions:

  • Model-only: Raw prompt with no structural wrapper
  • Minimal-shell: Task wrapped in [TASK START] / [OUTPUT] tags, no execution flow
  • 4-stage pipeline: Plan → Execute → Verify → Recover (up to 2 retries with error feedback)

Ablation conditions: Remove one stage at a time (no-plan, no-verify, no-recover) to isolate component contributions.

Evaluation: Manual scoring on a 0/1/2 rubric. TSR (score-2 rate) and VTSR (score≥1 rate) with thresholds TSR≥0.65 and VTSR≥0.80. All models run via Ollama v0.21.2 on a single RTX 4060 GPU.

Key design choices: Each task is run once (retries only within pipeline recovery); tasks are fixed JSON files; results include full metadata and pipeline traces for reproducibility.

Results & Highlights

  • Gemma4 E2B baseline (T1–T5): Pipeline achieves TSR=0.952 and VTSR=1.000, the only condition reaching perfect VTSR. Model-only TSR=0.762, minimal-shell TSR=0.714 (reversal).
  • T4 Workflow Completion shows largest pipeline gain: TSR jumps from 0.50 (model-only) / 0.25 (minimal-shell) to 1.00 (pipeline), confirming pipeline value for multi-step tasks.
  • Non-monotonic effect confirmed in two models: Minimal-shell TSR < model-only TSR in both Gemma (0.714 < 0.762) and Qwen (0.857 < 0.952). Hypothesized cause: wrapper tags increase cognitive load.
  • LLaMA 3.2 scaffold collapse: Model-only TSR=0.429 (7 format violations out of 21 tasks) but VTSR=0.952. Content generation is intact; format compliance collapses without harness support. Pipeline rescues TSR to 0.762.
  • Qwen pipeline reversal: Qwen model-only already achieves TSR=0.952; pipeline actually drops TSR to 0.857 because verify-recover loop generates repeated score=1 outputs in T2 (structurally correct but incomplete citations).
  • Ablation: Planning stage acts as a "format anchor" for quantitative constraints (e.g., 200-char limit: 248 chars without plan → 193 chars with plan). Recovery stage handles timeout retries. Both contribute ~24.7% of total gain.
  • T6 design flaw acknowledged: Web search tasks structurally disadvantage model-only/minimal-shell (no tool access), making T6 a test of tool availability, not orchestration quality. Authors propose an equal-tool redesign.

Critical Assessment

Novelty (Moderate): The finding that harness design matters more than model size for operational stability is intuitive rather than surprising. The non-monotonic effect and scaffold collapse concept are more novel contributions. The 4-stage pipeline itself is a straightforward composition of known patterns (plan-execute-verify-recover).

Rigor (Moderate):

  • Strength: Well-structured experimental design with clear research questions, ablation study, and cross-model comparison.
  • Weakness: Single run per task with no variance estimation or confidence intervals — this is a significant gap. The non-monotonic effect could be noise. Single scorer with no inter-rater reliability. Non-uniform timeouts (300s vs. 600s) confound cross-model comparisons.
  • The authors are commendably transparent about these limitations.

Reproducibility (Good): All tasks are fixed JSON files, models are publicly available via Ollama, and the authors commit to version-controlling all artifacts. Consumer hardware makes replication accessible.

Significance (Moderate): Relevant for practitioners deploying SLMs in edge/on-premises settings. The failure mode taxonomy and harness fixability classification have practical value. However, the 24-task scope limits generalizability, and the results may not extend to more complex real-world deployments.

Red flags:

  • LLaMA minimal-shell/pipeline conditions were run in a separate session with different GPU load — cross-condition comparison for LLaMA is "indicative only" per the authors themselves.
  • Qwen pipeline actually hurts performance, suggesting the pipeline design is model-specific rather than universally beneficial.
  • The paper has a single author, which is unusual for empirical ML work of this scope.

Strengths

  • Honest and transparent about limitations — the authors openly acknowledge the T6 design flaw, single-run evaluation, and cross-session inconsistency
  • Well-structured experimental methodology with clear research questions and ablation design
  • Introduction of useful concepts: "scaffold collapse" and "non-monotonic harness effect" provide vocabulary for discussing SLM deployment challenges
  • Practical relevance: demonstrates SLM viability on consumer hardware with proper harnessing
  • Failure mode taxonomy with harness-fixability classification offers actionable guidance for practitioners

Limitations

  • 24-task scope is too small for strong generalization claims; task diversity is limited
  • No variance estimation — single run per task means observed effects could be noise (especially the non-monotonic finding)
  • Single scorer with no inter-rater reliability assessment
  • Non-uniform timeouts (300s for 2B models, 600s for 3B) confound timeout-based comparisons
  • LLaMA conditions were not run in the same session, making cross-condition comparison unreliable
  • Qwen results show the pipeline can hurt performance, suggesting the approach is not universally beneficial
  • Pipeline incurs 2-3× inference overhead, limiting applicability in latency-sensitive scenarios
  • External service dependency (DuckDuckGo rate limits) compromised T6 results entirely

Relevance

这篇论文对部署小语言模型的工程实践有直接参考价值——特别是4阶段管线设计和故障模式分类框架，对于在边缘设备上构建可靠的LLM应用具有实用意义。非单调效应的发现也提醒我们，在模型部署中不能简单地叠加封装层次，而需要系统性验证每层封装的实际效果。

Questions for Further Exploration

  1. Would the non-monotonic effect persist with multiple runs per task and statistical significance testing?
  2. How does the pipeline scale to more complex, multi-turn conversational tasks beyond single-shot structured outputs?
  3. Can rule-based verification (regex, code execution) improve VCR beyond the current 0.625 without increasing pipeline overhead?
  4. Is the scaffold collapse phenomenon specific to JSON output formats, or does it generalize to other structured outputs (XML, YAML, code)?
  5. How would the pipeline perform on larger models (7B–13B) — does the harness effect diminish as model capability increases?
