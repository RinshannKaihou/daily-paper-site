> **一句话总结：** 本文提出 RisCoSet 方法，利用多重假设检验（Learn Then Test 框架）为 LLM 代码生成构建风险可控的预测集（部分程序），在保证高置信度包含正确程序的同时，相比现有 PAC 方法减少了高达 24.5% 的代码节点移除。

> **TL;DR:** RisCoSet reframes LLM code uncertainty quantification as a multi-label problem using the Learn Then Test framework, constructing partial programs (AST subtrees) as risk-controlling prediction sets that guarantee a correct solution with high confidence while removing up to 24.5% fewer AST nodes than prior PAC-based methods.

## 核心贡献

- **将 LTT 框架首次引入结构化代码生成任务**：利用多重假设检验替代 PAC 预测集中的单调性约束，将代码不确定性量化重新定义为多标签分类问题，天然适配代码生成中多正确输出的特性。
- **设计面向代码结构化风险函数**：基于 AST 节点的负对数概率构建优化问题，在满足树结构约束的前提下保留尽可能多的节点，生成信息量更大的部分程序。
- **提出选择性执行策略**：通过不确定性分数（如困惑度）学习阈值，减少校准阶段需要运行测试用例的程序数量，在可控标签错误率下节省高达 84.4% 的测试执行开销。

## Key Contributions

- First application of the Learn Then Test (LTT) framework to structured code generation, eliminating the monotonicity constraint required by PAC prediction sets and naturally accommodating multiple correct programs via multi-label formulation.
- A structured risk function and corresponding optimization problem that produces partial programs (AST subtrees) with theoretical risk-controlling guarantees (Theorem 4.2), using binomial trial p-values for hypothesis testing.
- A selective execution strategy (Algorithm 2) that leverages program uncertainty scores to skip test-case execution for high-confidence programs, trading a bounded label-error rate $\epsilon$ for significant computational savings during calibration, with formal guarantees (Proposition 4.4).

## Methodology

RisCoSet operates in three main stages:

1. **Partial program construction via optimization (Section 4.1):** Given an LLM-generated program represented as an AST $T = (V, E)$, the method introduces binary variables $\beta_{i,v}$ for each node to indicate removal. The optimization minimizes node removals subject to an uncertainty budget $\lambda_k$ (Eq. 5) and two structural constraints: (SC1) tree connectivity (children can only be removed if their parent is removed) and (SC2) a cap $t_{\max}$ on subtree removals. The optimization is solved via the Z3 SMT solver.

2. **Multi-label formulation and LTT (Section 4.2):** Unlike PAC prediction sets that treat code generation as single-label classification, RisCoSet augments the calibration set by sampling $m$ additional programs per task and identifying correct ones via test-case execution. The structured risk (Eq. 4) measures the fraction of calibration instances where the partial program contains no correct program. Valid parameters $\hat{\lambda}$ are selected through multiple hypothesis testing with binomial trial p-values and FWER-controlling algorithms (Bonferroni, Holm-Bonferroni, or fixed sequence testing).

3. **Selective execution strategy (Section 4.3):** To reduce the cost of verifying $M$ sampled programs, the method learns a threshold $\hat{u}$ on uncertainty scores such that programs with scores below $\hat{u}$ are directly accepted as correct without test execution. The error bound $L(u)$ is estimated via importance sampling with Hoeffding's inequality (Lemma 4.3), yielding a formal trade-off: the resulting prediction sets satisfy $P(R_s(\hat{\lambda}) \leq \alpha + \epsilon(1-\alpha)) \geq (1-\gamma)(1-\delta)$.

## Results & Highlights

- Evaluated on three LLMs (Deepseek-Coder-33B, Qwen2.5-Coder-32B, Llama3.1-70B) across three benchmarks (HumanEval, MBPP, APPS) with 100 random calibration/test splits.
- **Node removal reduction:** RisCoSet removes significantly fewer AST nodes than both the PAC baseline and greedy baseline across all settings. At $\alpha = 0.15$, reductions of 24.5% on MBPP and 20.3% on APPS compared to the best competitor.
- **Risk control validity:** The satisfying code set coverage consistently exceeds the target bound $1 - \alpha$ across all experiments, empirically validating Theorem 4.2.
- **Sampling sensitivity:** Performance improves with more additional samples $m$ (up to 80 tested), but gains diminish beyond $m = 5$, suggesting practical efficiency with small sample sizes.
- **Selective execution savings:** At $\epsilon = 0.3$ label error rate, the strategy saves 84.4% of test-case executions while maintaining coverage above the target bound. Even at $\epsilon = 0.05$, it saves 12% of executions.
- **FWER algorithm comparison:** Fixed sequence testing (FST) outperforms Bonferroni and Holm-Bonferroni by leveraging domain knowledge that smaller $\lambda$ values are more likely to satisfy risk targets.

## Critical Assessment

- **Novelty (High):** The reframing from single-label PAC to multi-label LTT for code generation is a meaningful conceptual advance. The selective execution strategy with formal error bounds is a practical contribution. This is the first work applying LTT to structured code generation.
- **Rigor (High):** Theoretical guarantees are formally stated and proved (Theorem 4.2, Proposition 4.4). The empirical evaluation is thorough with 100 random trials, multiple LLMs, and multiple benchmarks.
- **Reproducibility (Moderate):** Key implementation details are provided (Z3 solver, 24-second timeout, temperature 0.8, top-p 0.95, grid search with 0.02 increment). However, the Z3-based optimization with structural constraints may require non-trivial engineering effort to reproduce. Code availability is not mentioned.
- **Significance (Moderate-High):** The practical impact depends on whether partial programs are useful to developers. The paper argues they help mitigate code hallucinations and reduce manual review, but no user study validates this claim. The method also adds calibration overhead (sampling + test execution), though this is offline-only.
- **Red flags:** The $t_{\max} = 1$ default setting limits the method to removing at most one subtree, which is restrictive. The Z3 solver introduces a 24-second timeout that may fail on complex programs. The evaluation uses only introductory-level APPS problems. The gap between partial programs and practical developer utility remains unexplored.

## Strengths

- Strong theoretical foundation with formal risk-controlling guarantees, eliminating the restrictive monotonicity assumption of prior work.
- Multi-label formulation correctly captures the reality that multiple semantically equivalent programs can solve the same task.
- Selective execution strategy provides a principled accuracy-cost trade-off with theoretical backing.
- Consistent improvements across three diverse LLMs and three benchmarks demonstrate generalizability.
- Empirical risk control is validated: coverage always exceeds the target bound in practice.

## Limitations

- The method requires test-case execution during calibration, adding computational overhead that is not needed by sample-free baselines.
- The partial program representation (AST subtree) may not be immediately actionable for developers — the paper does not evaluate how developers would interact with or benefit from partial programs.
- The default $t_{\max} = 1$ constraint limits the expressiveness of partial programs; larger values lead to worse results due to miscalibration, suggesting sensitivity to model calibration quality.
- Evaluation is limited to Python benchmarks with available test cases; applicability to other languages or domains without test oracles is unclear.
- The Z3 solver may struggle with very large programs or complex ASTs, as indicated by the timeout mechanism.

## Relevance

本文将 conformal prediction 风险控制框架引入 LLM 代码生成的不确定性量化，与 LLM 可靠性、不确定性估计和代码智能等研究方向高度相关。其多重假设检验方法也可推广至其他结构化预测任务。

## Questions for Further Exploration

- How would RisCoSet perform on more complex code generation tasks (e.g., repository-level generation, multi-file projects) where test-case availability is limited?
- Can the partial program representation be integrated into an interactive code completion or repair tool, and would developers find it useful?
- How does the method scale with program length and AST complexity beyond the introductory-level APPS problems tested?
- Could alternative uncertainty scores (beyond perplexity) improve the selective execution strategy's efficiency?
- Is the approach applicable to non-code structured generation tasks (e.g., SQL, formal specifications, molecular generation)?
