Formalize, Don't Optimize: The Heuristic Trap in LLM-Generated Combinatorial Solvers

arXiv: 2605.12421

Date: 2026-05-12

https://arxiv.org/abs/2605.12421

一句话总结： 本文通过 100 个组合优化问题（4,577 个实例）的系统性实验揭示，LLM 生成求解器时应专注于形式化建模而非搜索优化——使用 Python + OR-Tools 约束建模 API 可获得最高正确率，而提示 LLM 进行启发式优化会触发六种范式特异的"启发式陷阱"，在弱模型上导致正确率显著下降。
TL;DR: Across 100 combinatorial problems and three LLMs, Python + OR-Tools constraint modeling achieves the highest correctness. Prompting for heuristic optimization yields negligible speed-ups (1.03-1.12x) but triggers six paradigm-specific failure modes that sharply degrade correctness, especially on weaker models—a "heuristic trap" that supports using LLMs for formalization, not search optimization.

核心贡献

  • 提出 CP-SynC-XL 基准，包含 100 个异构组合优化问题（58 个 CSP + 42 个 COP）和 4,577 个参数化实例，支持实例级求解器评估
  • 系统比较三种求解器构造范式：原生 Python 算法搜索、Python + OR-Tools 约束建模、MiniZinc + OR-Tools 声明式建模，发现 Python + OR-Tools 在所有 LLM 上正确率最高
  • 揭示"表示表面分歧"（representational divergence）：MiniZinc + OR-Tools 和 Python + OR-Tools 共享相同 CP-SAT 后端，但 LLM 对 MiniZinc 的建模流畅度显著更低
  • 发现"启发式陷阱"（heuristic trap）：提示 LLM 进行效率优化只带来微小中位加速（1.03-1.12x），但引发六种范式特异的失败模式，在弱模型上导致高达 -7.4 pp 的正确率下降
  • 通过配对代码审计定位了四类基线静默失败机制（M1-M4）和六类启发式退化模式（A-F），均源自自然语言到形式语言的非等价转换

Key Contributions

  • Introduce CP-SynC-XL, an XL-scale benchmark with 100 heterogeneous combinatorial problems and 4,577 parameterized instances for evaluating LLM-synthesized solvers at the instance level
  • Demonstrate a consistent representational divergence: Python + OR-Tools achieves the highest correctness across all three LLMs (GPT-5.3-Codex, Gemini 3.1 Pro, DeepSeek-V3.2), while MiniZinc + OR-Tools lags despite sharing the same CP-SAT backend
  • Identify the heuristic trap: efficiency-oriented prompting produces only marginal median speed-ups (1.03-1.12x) with strongly bimodal outcomes—many instances slow down and correctness drops sharply on a long tail of problems
  • Catalog four baseline mistranslation mechanisms (M1: associative recall, M2: long-spec attention drift, M3: output-schema ambiguity, M4: sentinel-value misreading) and six heuristic failure modes (A: unverified bounds, B: propagation-weakening rewrites, C: hand-computed seeding, D: silent completeness loss, E: COP-to-SAT collapse, F: redundant-machinery blow-ups)
  • Propose a conservative "formalize, don't optimize" design principle: use LLMs to formalize variables, constraints, and objectives for verified solvers, and separately verify any LLM-authored search optimization

Methodology

The paper uses a controlled factorial design with two axes: solver-generation paradigm (Python, Python + OR-Tools, MiniZinc + OR-Tools) and prompt type (baseline vs. heuristic). Three frontier LLMs serve as code generators: GPT-5.3-Codex, Gemini 3.1 Pro, and DeepSeek-V3.2. Each LLM produces an instance-agnostic executable solver for each natural-language combinatorial problem.

Pipeline: Each generated solver passes through a two-stage validation (smoke check on a small instance, scaling check on a harder instance) with up to four refinement rounds. Neither stage verifies solution correctness—this is deferred to evaluation against a reference solver. Both the first-working and final artefacts are evaluated on the full instance pool.

Evaluation protocol: For every (artefact, instance) pair, the framework records whether a schema-conformant solution was provided within a 256-second budget and whether it passes the reference verifier (CSP feasibility + COP optimality). Fidelity is defined as correct/provided—the share of returned solutions that pass verification. Outcomes are decomposed into five bins: correct, suboptimal, UNSAT, invalid, and no solution.

Analytical methods: Paired code-level audits compare baseline and heuristic solvers problem-by-problem. Conservative source-level detectors identify six failure modes (A-F). A per-problem audit of the top twelve UNSAT-heavy problems (covering 89.6% of wrong mass) identifies four baseline mistranslation mechanisms (M1-M4).

Results & Highlights

  • Python + OR-Tools dominates correctness: 70.2% (GPT-5.3-Codex baseline) vs. 63.4% (Python) and 52.2% (MiniZinc + OR-Tools) at the 256s budget
  • MiniZinc + OR-Tools has highest fidelity (≥83%) but lowest coverage: it returns fewer answers overall, but those it returns are most likely correct
  • Native Python has the highest silent-failure rate: fidelity as low as 51.9% (DeepSeek-V3.2 heuristic), meaning nearly half of returned solutions are wrong
  • Heuristic prompting is net-negative for DeepSeek-V3.2: -7.4 pp correctness and -16.1 pp fidelity on Python + OR-Tools
  • Heuristic prompting is mildly positive for GPT-5.3-Codex: +1.2 to +2.2 pp across paradigms, because GPT-5.3-Codex prefers native solver hooks over hand-rolled substitutions
  • Refinement is an error-recovery loop, not a performance loop: median speed-up is essentially 1.0x; correctness gains come from fixing crashes, not speeding correct solvers
  • M2 (attention drift) and M3 (schema ambiguity) account for ~74% of baseline wrong mass, not solver reasoning failures
  • Mode F (redundant-machinery blow-ups) is the most common heuristic failure in MiniZinc (29.1% incidence), and can drop accuracy from 1.00 to 0.00 on individual problems

Critical Assessment

  • Novelty: High. The "heuristic trap" concept and the systematic paradigm comparison with instance-level decomposition are novel contributions. The six failure modes are identified through careful engineering analysis rather than assumed a priori.
  • Rigor: Very strong. The factorial design controls for LLM, decoding parameters, instances, timeouts, and refinement budget. The paired analysis (baseline vs. heuristic on the same problem) is methodologically sound. The 256s instance filtering (excluding instances whose reference validation times out) is disclosed transparently.
  • Reproducibility: Strong. The benchmark, generated solvers, and evaluation scripts are promised. The mode detectors have explicit regex-level definitions in the appendix.
  • Significance: High practical impact for neuro-symbolic system design. The "formalize, don't optimize" principle is actionable and counter to the prevailing trend of asking LLMs to also supply heuristics.
  • Red flags: (1) Only greedy decoding (T=0) is tested; sampling-based approaches might change the picture. (2) The 256s reference-validation filter removes hard instances, so the findings may not extend to the hardest instances. (3) The paper studies single-shot generation with bounded refinement—results may differ under evolutionary or population-based approaches (FunSearch-style). (4) The representational divergence finding may be partly an artifact of current training data distributions shifting toward Python-heavy code corpora.

Strengths

  • Instance-level evaluation on a large and diverse benchmark (4,577 instances across 100 problems)
  • Clean factorial design with strong methodological controls (external time control, paradigm enforcement)
  • Mechanistic code-level audits that go beyond aggregate metrics to identify specific failure patterns
  • The fidelity metric captures an important practical concern: systems that confidently return wrong answers
  • The paper honestly reports negative results (heuristic prompting hurts on DeepSeek-V3.2) rather than only highlighting positive findings

Limitations

  • Only three LLMs are evaluated; the findings may not generalize to open-source models or smaller models
  • The 256s per-instance budget is arbitrary and may not reflect all deployment scenarios
  • MiniZinc's lower performance could partly reflect insufficient MiniZinc training data in the LLMs rather than an intrinsic limitation of declarative modeling
  • The heuristic prompt is a single fixed prompt; different formulations of the efficiency instruction might yield different outcomes
  • The study focuses on single-problem, single-shot generation; multi-problem transfer or few-shot prompting is not explored

Relevance

本文对 LLM 代码合成与神经符号系统的研究具有直接参考价值。其"先形式化、后优化"的设计原则为使用 LLM 辅助组合优化问题建模提供了实用的工程指导，同时提出的启发式陷阱现象对理解 LLM 在复杂推理任务中的局限性有重要意义。
