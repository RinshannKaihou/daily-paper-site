arXiv 2605.17613

Turning Lossy KV Cache into Lossless LLM Inference

First framework to draft from compressed KV cache and verify against off-GPU full KV — identical outputs, up to 4.26× throughput.

kv-cache lossless-inference speculative-decoding llm-serving memory-optimization
↗ arxiv.org/abs/2605.17613
Key Finding
Compressed KV caches can serve as high-fidelity self-drafters (acceptance rate > 0.8 at draft 30), enabling lossless verification against offloaded full KV via cross-resource staggering.
Key Metric
4.26× throughput on Qwen-32B (w/ EAGLE). Standalone 1.92×–2.73×. KL < 0.01 nats.
Why This Matters
Resolves the accuracy-throughput dichotomy in KV cache compression — a critical production blocker for code generation and tool-calling where output fidelity is non-negotiable.
Critical Edge
Total memory is not reduced (full KV still lives in CPU/storage). Peak 4.26× requires EAGLE composition; benefits shrink on slow interconnects or with aggressive >4× compression.

Methodology

VeriCache runs a draft-then-verify loop: (1) draft x tokens autoregressively using the compressed KV cache on GPU; (2) verify via one forward pass over all x positions conditioned on the full KV loaded from secondary storage; (3) accept all matching tokens + a bonus token, reject and correct the first mismatch.

  • P1 — Cross-resource staggering: only ~B/x requests verify at any iteration, overlapping HBM-bound drafting with PCIe/network-bound transfers. Avoids the burst congestion of lock-step speculative decoding.
  • P2 — Extended verification period: same-model compressed KV sustains ~0.8 acceptance at draft 30 (vs. 2–3 tokens for small-model drafters), amortizing each full-KV swap over 25–40 accepted tokens.
  • Uniform Compressor interface: exposes only dropped_indices + bit_scheme; supports 7 methods (KVzip, KVZap, ExpectedAttention, SnapKV, KIVI, KVQuant, RotateKV) without changing scheduling logic.
  • Dual deployment: long-context (full KV in CPU DRAM, PCIe reload) and remote prefix caching (full KV on storage node, fast-link local verification).

Results & Highlights

2.73× Llama-70B 4.26× Qwen-32B 2.11× Remote prefix KL < 0.01 nats
  • Lossless across 3 models, 7 compressors, 5 datasets — KL < 0.01 nats (hardware nondeterminism only).
  • Lossy baselines diverge exponentially: KVzip 0.5× on Llama-70B accumulates ~14.4 nats.
  • Matches Full KV on ComplexFuncBench accuracy; lossy KVzip loses up to 30 points.
  • Speedup grows as KV budget shrinks (1.61×→2.71×) and as interconnect improves (H100 → GH200).

Authors & Credibility

U. Chicago — Jiayi Yao, Samuel Shen, Kuntai Du, Shaoting Feng, Dongjoo Seo, Rui Zhang, Yuyang Huang, Yuhan Liu, Shan Lu, Junchen Jiang.

  • Built on production-grade vLLM + LMCache (~8K LoC).
  • Evaluated on Mistral-24B, Qwen-32B, Llama-70B; hardware: RTX PRO 6000, H100 NVL, GH200.
  • 16-page paper with formal throughput model (Appendix B) and KL divergence proof (Appendix A).
  • No explicit open-source release mentioned for VeriCache itself.

Strengths

  • Elegant insight: same-model compressed-KV drafting yields far higher acceptance than small-model drafting.
  • Cross-resource staggering avoids burst overhead of lock-step speculative decoding.
  • Zero-friction integration of 7 compression methods via uniform interface.
  • Dual-setting coverage (long-context + remote prefix caching).
  • Composable with EAGLE for additive speedups on orthogonal bottlenecks.

Limitations

  • Full KV must be retained in CPU/storage — not a total memory reduction.
  • Static draft length; adaptive policy could handle heterogeneous workloads better.
  • Benefits sensitive to HBM-to-interconnect bandwidth ratio.
  • Aggressive >4× compression risks dropping acceptance rate below the sweet spot.
  • No evaluation on sampling-based decoding or models > 70B.

Questions for Further Exploration

  • Could a learned compressor optimizing directly for acceptance length push throughput beyond the current Pareto frontier?
  • How does VeriCache interact with non-prefix chunk caching (e.g., CacheBlend) where draft-then-verify might also apply?
  • What is the break-even compression ratio below which verification overhead exceeds drafting speedup — and does it differ by architecture?
  • Can an online adaptive draft-length policy (based on running acceptance rate) improve gains without scheduling complexity?
  • How would VeriCache perform on 405B+ models or MoE architectures where per-expert KV differs?
