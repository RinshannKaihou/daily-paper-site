Key Finding

Why This Matters

A “compile once, deploy anywhere” agent skill library: strong-model skills lift weaker executors and cut deployment cost while accumulating reusable procedural knowledge.

Critical Edge

— The “structural” PFSM framing is aspirational: every stage still leans on an LLM for normalization, clustering, and codegen, so SkillDisCo is in practice a well-engineered LLM-orchestration stack — and its FSM assumption excludes non-deterministic, open-ended tasks.

M Methodology

SkillDisCo takes a set of successful traces T™ and returns a skill library K. It grounds an FSM-defined scenario M = (S, A, δ, S₀, Sgoal) with deterministic transitions, then abstracts concrete states/actions into a parameterized finite-state machine (PFSM) so traces of different objects, locations, or lengths instantiate the same control-flow pattern. A skill is a PFSM subgraph shared across many traces; since neither the true PFSM nor the lifting function is observable, SkillDisCo approximates both through two phases.

Distillation (3 stages): (1) an LLM normalizes each raw trace into a Python-like program with explicit primitive calls and code-level control flow; (2) programs decompose into multi-step subgoal operators, with single-step operators discarded; (3) operators sharing parameterized control-flow structure are clustered, and each cluster receives a coverage-based reusability score — only high-coverage clusters survive.

Compilation (2 stages): each candidate is given a formal specification (signature, pre/post-conditions, metadata), then a Python skill is synthesized and run through a synthesize → verify (on held-out tasks) → re-synthesize loop with feedback, discarding skills that still fail. Evaluation uses strict induction/eval splits: ALFWorld (200 / 134) and WebArena (406 / 406), against ReAct, CodeAct, AWMoffline, and ASIoffline.

± Strengths & Limitations

✓ Strengths

  • Clean formal grounding — PFSM + parameter binding distinguishes it from NL-only or per-trajectory skill methods.
  • Compact libraries (5 / 20 skills) with strong coverage and high per-call compression.
  • Striking transfer — GPT-4o skills let Qwen3.5-9B exceed the inducer on ALFWorld.
  • Honest ablations attributing gains to distillation, not compilation cosmetics.
  • Verified execution (0% skill-call error on ALFWorld) fixes a real weakness of text-based libraries.

✗ Limitations

  • Only procedural tasks with reusable structure benefit; no gain for pure NLP tasks.
  • Depends on the compiler LLM's codegen quality; degrades with weaker models.
  • Needs a corpus of successful traces — sparse in low-success domains.
  • FSM-defined assumption is restrictive and hard to verify for new environments.
  • No code/data release; clustering & verification details underspecified.
  • WebArena absolute SR still modest (~29%); transfer gains noisier.

? Questions for Further Exploration

Q1How to extend beyond the strict FSM assumption to non-deterministic or partially observable environments?
Q2Is there a principled, model-agnostic way to detect whether a new domain is FSM-defined before committing?
Q3How does performance scale with induction-corpus size and diversity — is there a saturation point?
Q4Can currently-discarded failed traces be used as negative evidence to sharpen pre/postconditions?
Q5How robust are compiled skills to environment drift (e.g., WebArena UI changes), and can verification re-trigger online?
Q6Could a learning-based matcher replace LLM clustering / reusability scoring?
Q7Does transfer hold for much weaker executors (1–3B), or does skill selection itself become the bottleneck?
