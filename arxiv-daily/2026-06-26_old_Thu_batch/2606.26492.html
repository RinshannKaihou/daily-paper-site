Evaluation-Strategy Gap in Fault Diagnosis of Deep Learning Programs

Sigma Jahan · Preprint (cs.SE, cs.AI, cs.LG)

Key Finding

DL fault-diagnosis techniques overstate accuracy by ~0.19 under within-program evaluation — the gain is program identity, not transferable fault signal.

deep-learning fault-diagnosis software-engineering generalization empirical-study dynamic-analysis evaluation-methodology
Key Metric
0.190
Balanced-accuracy gap: within-program 0.469 vs program-held-out 0.279 (4-class fault-type).
Widens to 0.34 under random-forest classifiers.
Why This Matters
The dominant within-program CV convention hides deployment failures; future diagnosis papers must report program-held-out results and program-identity predictability.
Critical Edge
The headline gap rests on mutation-injected faults across only 38 programs (Activation = 2, Regularization = 1), so the near-chance 6-class cross-program number is partly structural, not evidence of no-transfer.

Methodology

DynFault corpus. 38 real-world DL programs (FFNN / RNN / CNN) curated from Stack Overflow, each keyed by so_id. Faults are injected via extended mutation operators spanning 7 categories (hyperparameter, loss, activation, layer config, optimizer, weight init, regularization) grounded in empirical DL-bug taxonomies (Islam; Humbatova; DEFault). Mutants are classified killed / survived by a GLM criterion; killed mutants receive a fault label. Weight-init mutants produced no killed runs, leaving 6 categories and a 4-class primary task (Layer, Hyperparameter, Loss, Optimization).

Two runtime feature sets. Curvature (CF): 4 performance metrics + Hessian-vector-product mean/std and per-layer HVP norms (Pearlmutter R-operator, O(|θ|)). Optimizer (OF): 4 metrics + gradient norm/stats, learning rate, activation mean/std, memory usage (15 channels). For each scalar channel over the first k epochs the pipeline extracts window mean and OLS slope; windows k ∈ {1,3,5,10,20,50}; standardization uses training-split stats, missing channels imputed by training medians.

Classifier & tasks. Logistic regression with class-balanced one-vs-rest weights — a deliberate linear probe to fix capacity and isolate feature/evaluation effects (RF widens the gap to 0.34, so LR is conservative). Three tasks: fault-type classification (4/6/3-class), catastrophic-instability triage (loss > 10⁶), train/validation mismatch detection (|acc gap| ≥ 0.1). Two strategies: within-program stratified 5-fold CV vs program-held-out grouped 5-fold by so_id; plus leave-one-program-out (LOPO) for the unsupervised triage rule. Metrics: balanced accuracy, ROC-AUC, PR-AUC; 95% CIs via cluster bootstrap over programs.

Four gap-attribution controls. Program-identity prediction, within-program label permutation, per-program normalization, and distribution-level pairwise MMD² (RBF, median-heuristic bandwidth) — converging evidence that the gap is program-level shortcut, not transferable signal.

Strengths

  • Cleanly isolates evaluation strategy from feature design and shows they interact, via controlled experiments.
  • The four-control convergence argument (identity, permutation, normalization, MMD) is a textbook attribution of a gap to a specific cause.
  • Practitioner-actionable cost/benefit guidance for instrumentation (when HVP logging is worth it).
  • Honest reporting: full CIs, chance baselines, conservative LR probes, transparent CF/OF subset confound.
  • Releases both DynFault corpus and evaluation framework, lowering adoption barriers.

Limitations

  • Mutation-injected faults only — external validity to real-world failures unconfirmed (authors' own caveat).
  • Small, imbalanced category coverage (Activation = 2, Regularization = 1 programs) undermines some cross-program numbers.
  • Cross-program 4-class CI includes chance — no deployable fault-type classifier is demonstrated.
  • Flagship zero-FP triage rule generalizes poorly in recall (LOPO recall 0.056).
  • Curvature collection has real per-step overhead; cost analysis is qualitative, not benchmarked.

Questions for Further Exploration

  1. Do the 0.19–0.34 gaps and the curvature-transfers / optimizer-doesn't pattern hold on real-world fault corpora rather than mutation-injected mutants?
  2. Can domain-adaptation / program-invariant feature learning (IRM, adversarial alignment, training-time normalization) recover the lost cross-program fault-type discrimination?
  3. Is there a transferable fault signal beyond instability — features with provably low program-identity predictability but high fault-type predictability?
  4. How does the gap scale with program diversity (architecture, dataset size, optimizer) — is 38 programs enough to saturate the distribution?
  5. Can the epoch-0 curvature heuristic be made higher-recall cross-program by combining clusters or learning a program-invariant threshold?
  6. Is the fixed 0.1 T/V-mismatch threshold architecture-aware (RNN vs CNN vs FFNN)?
