> **一句话总结：** 本文提出 ClaMPAPP 混合系统，将 LLM 限定为"接口"（用工具调用从叙述式临床文本中抽取 schema 受限特征并做硬规则合理性校验），由 XGBoost 在两个德国儿科阑尾炎队列上完成最终预测；系统在内/外部验证上都取得最高的整体 F1，且在句序重排鲁棒性测试中退化幅度仅 5.2%，而端到端 LLM 基线（如 MedGemma、Llama-3.1-8B）退化达 28–30%，证明"LLM 当接口、ML 当预测器"是高风险临床决策更安全的工程范式。
> **TL;DR:** ClaMPAPP decouples language understanding from clinical prediction: an LLM (Llama-3.1-8B as a rewriter; tool-calling LLM as the extractor) parses free-text pediatric appendicitis narratives into a fixed schema (demographics, exam, labs, ultrasound, Alvarado/PAS), a deterministic Feature Validator rejects implausible values (mapping them to NaN rather than clipping), and an XGBoost classifier emits a continuous appendicitis risk score. Trained on n=580 Regensburg records and tested on n=202 internal + n=301 external (Düsseldorf) records, ClaMPAPP beats a panel of 6 LLM baselines (MedGemma-4B, Llama-3.1-8B, GPT-5.5, o3-pro, Claude Opus 4.7, OASST-Pythia-12B) on accuracy/F1, with 0.851/0.848 internal and 0.807/0.881 external; critically, it misses only 2 cases internally and 15 externally (vs 40–89 for the strongest LLM baselines). Under sentence-order permutation ClaMPAPP's accuracy drops only 5.2% (vs 28–30% for LLM-only baselines) and it still hits 100% sensitivity on permuted Regensburg. The honest caveats: external specificity collapses to 39.4%, raw probabilities are miscalibrated (Brier 0.118/0.150, ECE 0.167/0.096), and the evaluation narratives are synthetic (templated + LLM-rewritten), not real clinician free-text — so the real-world extraction drop is unknown.

## 核心贡献
- 提出并实现 "LLM-as-interface, ML-as-predictor" 架构 ClaMPAPP：LLM 严格限定为基于工具调用的 schema 受限特征抽取 + 临床报告生成，XGBoost 负责最终风险预测。
- 设计了"硬规则 Feature Validator"作为安全门：超出临床/生理合理区间的提取值被映射为 NaN（而非裁剪到边界），交给 XGBoost 的原生缺失值处理。
- 构建四阶段数据-到-叙述管线（结构化 EHR → 固定模板 → Llama-3.1-8B 受约束改写 → 二段式句序置换），既保留 ground truth，又生成可注入位置鲁棒性测试的自然化自由文本。
- 手工 fidelity 审计显示改写阶段特征级错误率仅 3.4%（Regensburg, 620 检查项）与 2.9%（Düsseldorf, 170 检查项），绝大多数为单条遗漏/矛盾。
- 在两个独立德国儿科队列上系统对比 6 个 LLM 基线（开源 + 闭源 + 医学专用），证明混合架构在整体 F1、漏诊率、句序重排鲁棒性三项指标上同时占优。
- 提供了完整的概率校准分析（discrimination vs. calibration 分离讨论），并诚实指出 raw 概率需 post-hoc 重新校准。

## Key Contributions
- **Architectural reframing**: the LLM is repositioned from "decision-maker" to a constrained semantic parser with a deterministic safety gate, leaving the predictive risk to a validated tabular ML model.
- **Feature Validator as a clinical safety net**: out-of-range or implausible LLM extractions (e.g., age 0–18, temp 35–42°C, WBC 3–50×10⁹/L) are converted to NaN rather than clipped, avoiding silent input bias.
- **Schema-constrained extraction schema** (Table 2 in paper): demographics, BMI, clinical scores (Alvarado/PAS — computed deterministically only when all components present), symptoms/exam findings, lab variables, ultrasound findings, and encounter metadata.
- **Synthetic-narrative evaluation harness**: a four-stage data-to-text pipeline that produces realistic first-person free-text from structured EHR, with a sentence-order permutation (bipartite structural inversion) stress test.
- **Empirical demonstration** that the same frozen Llama-3.1-8B goes from 21/57 false negatives end-to-end to 2/15 false negatives in the hybrid pipeline — i.e., the LLM's clinical utility is preserved *without fine-tuning*.
- **Calibration-aware evaluation**: AUC/AP, Brier, log loss, ECE, calibration intercept/slope, and Hosmer–Lemeshow all reported; the paper candidly flags raw-probability miscalibration.

## Methodology

**Datasets (structured EHR, not free text):**
- Regensburg (internal, n=782): pediatric abdominal-pain admissions at St. Hedwig Hospital. Split 580 train / 202 internal eval.
- Düsseldorf (external, n=301): pediatric suspected appendicitis at Florence Nightingale Hospital; excluded prior appendectomy, chronic intestinal disease, or active antibiotics. Used *only* for external validation — not for training, prompt refinement, or threshold selection.

**Narrative generation (4 stages):**
1. Raw structured patient data.
2. **Template application**: a fixed first-person English template with placeholders for every variable.
3. **LLM rewriting** with Llama-3.1-8B using a strict preservation prompt ("only rewrite, do not add/remove clinical facts, no diagnoses, no commentary").
4. **Bipartite structural inversion**: divide narrative into two halves, swap order, re-rewrite — preserves coherence while varying information position.
- Manual fidelity audit on 31 Regensburg (≈15% of internal eval) + 10 Düsseldorf reports: 3.4% and 2.9% feature-level error rates, mostly omissions and a few contradictions.

**ClaMPAPP pipeline (5 stages):**
1. Clinician inputs free-text note.
2. LLM tool-call `extract_features` → schema-constrained JSON.
3. Deterministic Feature Validator: range/type checks, NaN-on-fail.
4. XGBoost classifier → continuous appendicitis risk score.
5. LLM as clinical summarizer → structured clinician-facing report with non-prescriptive triage language (e.g., "High risk: consider surgical consultation").

**Why XGBoost?** A 5-model benchmark (LightGBM, XGBoost, Decision Tree, Random Forest, Logistic Regression) on the same Regensburg train/internal-eval/Düsseldorf-external setup showed XGBoost gave the best balance of internal performance and external stability. XGBoost also natively handles NaN, which pairs naturally with the validator's NaN-on-fail policy.

**Baselines (6 LLMs):** medgemma-4b-it, llama-3.1-8b-instant, OpenAI GPT-5.5, OpenAI o3-pro, Anthropic Claude Opus 4.7, oasst-sft-1-pythia-12b — prompted to return a strict JSON binary decision.

**Metrics:** Acc, Sensitivity, Specificity, Precision, F1, TP/TN/FP/FN, AUC-ROC, AP, Brier, log loss, ECE (10 bins), calibration intercept/slope, Hosmer–Lemeshow.

## Results & Highlights

**Internal Regensburg (n=202):**
- ClaMPAPP: Acc 0.851, F1 0.848, Sen 0.977, Spec 0.759, FN=2, AUC=0.982, AP=0.981.
- OASST-Pythia-12B "wins" sensitivity (0.988) by predicting positive for *every* patient — Spec=0.026, FP=113, Acc=0.436. (Tells you that bare sensitivity numbers for LLM baselines can be vacuous.)
- MedGemma-4B: Sen 0.535 (FN=40), Spec 0.888 — trades away too many true cases.
- Claude Opus 4.7: Sen 0.581 (FN=36), Spec 0.776.
- OpenAI GPT-5.5: Acc 0.748, Sen 0.802, FN=17.

**External Düsseldorf (n=301) — ClaMPAPP still leads F1:**
- ClaMPAPP: Acc 0.807, F1 0.881, Sen 0.935, Spec 0.394, FN=15, AUC=0.804, AP=0.929.
- Note the specificity collapse (0.759 → 0.394) — ClaMPAPP trades specificity for sensitivity, i.e., the "high-sensitivity triage" posture.
- OpenAI GPT-5.5: Spec 0.859, but FN=78 (misses more than 5× as many).
- OpenAI o3-pro: Spec 0.775, FN=67.
- MedGemma-4B: FN=89 (worst miss rate).

**Permutation robustness (bipartite structural inversion, Regensburg):**
- ClaMPAPP: Acc 0.807, Sen 1.000, FN=0 — *improves* FN count vs. unpermuted.
- OpenAI o3-pro: Acc 0.515 (essentially random); GPT-5.5: 0.624; MedGemma-4B: 0.515 with Sen 0.128 (FN=75).
- Relative accuracy drops: MedGemma -30.2%, Llama-3.1-8B -28.4%, ClaMPAPP -5.2%.

**Permutation robustness (Düsseldorf):**
- ClaMPAPP: Acc 0.764, Sen 0.926, FN=17 — most stable and safety-oriented.
- GPT-5.5: Acc 0.797, but FN=41; o3-pro: FN=36.

**Calibration (raw, not recalibrated):**
- Regensburg: Brier 0.118, log loss 0.375, ECE 0.167, calibration intercept −2.936, slope 1.113. Discriminates well, but predicted probabilities are not on a calibrated absolute scale.
- Düsseldorf: Brier 0.150, log loss 0.508, ECE 0.096, intercept −0.060, slope 0.513 (probabilities spread too wide).
- Exploratory post-hoc logistic recalibration (Platt-style) on the same cohort improves apparent calibration (Regensburg ECE 0.167 → 0.039; Düsseldorf ECE 0.096 → 0.026) but the paper is honest that this is "apparent" not transportable.

**Architecture-contrast table (paper's Table 1):**
- Pure LLM = high hallucination risk; RAG = moderate; Traditional ML = low but rigid; ClaMPAPP = low hallucination risk + free-text input + auditable LLM explanation.

- Not a large multi-center clinical trial team — the work is a focused methodology paper that re-uses two publicly available pediatric appendicitis datasets (Regensburg / Düsseldorf; both from the Marcinkevičs et al. group).

## Critical Assessment

**Novelty:** Moderate-high. The "LLM as interface, ML as predictor" split is a well-trodden idea in clinical AI, but the specific combination of (i) tool-call-based schema-constrained extraction, (ii) NaN-on-fail validation that *deliberately* punishes implausible LLM output, and (iii) demonstrating that the *same* frozen Llama-3.1-8B goes from 21/57 FN to 2/15 FN in this hybrid form is a concrete, defensible contribution. The fidelity-audited synthetic-narrative harness is also a useful methodological template for other clinical-LLM work.

**Rigor:** Above-average for this paper class. Two independent cohorts, true external validation (Düsseldorf not used for any training/tuning), 6 baseline LLMs spanning open + proprietary + medical-specialized, discrimination + calibration + permutation all reported. The OASST-Pythia-12B "near-perfect sensitivity" being revealed as a degenerate always-positive predictor is honest reporting. **But** the paper leans on a synthetic-narrative evaluation; the authors are explicit about this but the gap to real free-text clinician documentation is non-trivial and the relative ordering of methods on *real* notes is not established.

**Reproducibility:** Moderate. The complete template and generation code are promised to be released "upon publication" — not yet in the paper, which means the most important prompts and the schema are not currently in the artifact. XGBoost hyperparameters, prompt engineering details, and the validator's exact range table are visible in the paper, but full reproduction requires the promised code release.

**Significance:** Real. This is one of the cleanest demonstrations I've seen of why "end-to-end LLM as diagnostic engine" is the wrong default in safety-critical clinical workflows, and the architectural pattern (NaN-on-fail + XGBoost + LLM summarizer) is highly transferable to other tabular-ML-meets-free-text use cases (sepsis triage, radiology report summarization, ED chest-pain risk scoring, etc.).

**Red flags / honest concerns:**
1. **External specificity collapse (0.759 → 0.394) is glossed as "acceptable for safety-oriented triage."** In an ED with finite imaging slots, halving specificity to gain a small number of additional true cases has real cost (unnecessary imaging, parental anxiety, surgical consults). The paper does not perform a decision-curve / net-benefit analysis to justify the operating point.
2. **Raw probability calibration is poor and the "fix" is post-hoc on the same cohort.** Without an external recalibration cohort, transportable calibration is unproven. For a system whose clinician-facing report is *generated* by the LLM in natural language, the gap between "the model says 0.83" and "the actual risk is 0.83" matters.
3. **Synthetic narratives ≠ real free-text.** The fidelity audit shows 3% feature-level error *in the rewriting*, but real clinical notes have copy-forward artifacts, abbreviations, typos, half-documented negatives, and inter-clinician variability that the synthetic pipeline does not capture. The headline robustness numbers likely overestimate real-world extraction reliability.
4. **No prospective or shadow-mode evaluation.** All numbers are retrospective on already-curated datasets.
5. **No subgroup analysis.** Pediatric appendicitis has important subgroups (younger children with atypical presentation — explicitly called out as a motivation in §1 — pre-/post-pubertal, complicated vs. uncomplicated). Not reported.
6. **Threshold choice is implicit.** ClaMPAPP is presented as a single operating point; no PR-curve operating-point sweep or threshold sensitivity analysis is shown.
7. **Baseline prompting matters.** Comparing to "end-to-end LLM with a strict JSON prompt" underestimates the ceiling you could get from chain-of-thought or tool-using LLM agents. The MedGemma 0.535 sensitivity on the unpermuted set, in particular, looks suspiciously low for a medical model and could reflect a brittle prompt setup, not the model's true capability.
8. **OASST-Pythia-12B is a strange baseline choice** for a clinical paper — it's a general-purpose conversational assistant. Including it mostly serves to dramatize LLM unreliability.
9. **No clinician-in-the-loop or prospective usability evaluation.** A triage tool that is "safety-oriented" still has to fit into an ED workflow.
10. **Authors' claim that the design is "disease-agnostic" is forward-looking, not demonstrated.** Only pediatric appendicitis is shown.

## Strengths
- Clean, well-motivated architectural split with explicit safety framing (FN minimization as primary endpoint, not headline accuracy).
- True external validation on a second hospital's data, untouched during development.
- The Feature Validator (NaN-on-fail) is an underused and elegant pattern — it converts the LLM's hallucination mode into *missing data* rather than corrupt data, which XGBoost handles well.
- Demonstrates the value of the architecture with the *same* underlying LLM (Llama-3.1-8B), removing fine-tuning/confounding as an explanation.
- Sentence-order permutation is a concrete, fact-preserving robustness test that surfaces a real LLM weakness.
- Calibration analysis is reported with multiple metrics and the limitations are candidly discussed.
- Comparison panel spans open, proprietary, and medical-specialized models, with honest reporting of where each one fails.

## Limitations
- **Synthetic-narrative evaluation**: the most consequential limitation. The pipeline is evaluated on templated + LLM-rewritten text, not real clinician free-text. The full distribution of abbreviations, typos, copy-forward artifacts, and inter-site documentation conventions is not represented.
- **Two German pediatric cohorts only**: geographic, linguistic, and practice-pattern generalization is unaddressed; the XGBoost would likely need retraining/recalibration for any new site.
- **Calibration is poor on raw scores and the "fix" is post-hoc on the same cohort.** Apparent calibration improvement ≠ transportable calibration improvement.
- **External specificity of 0.394** is operationally expensive (many false-positive workups) and the paper does not quantify the cost.
- **No prospective or shadow-mode evaluation**; the paper is explicit that prospective validation is future work.
- **No subgroup analysis** (age strata, sex, complicated vs. uncomplicated appendicitis).
- **No decision-curve / net-benefit analysis** to justify the chosen operating point.
- **Validator catches implausible values but not plausible-but-incorrect ones** — e.g., a confidently extracted wrong WBC will sail through.
- **Code/prompts not yet released**, despite the paper's reproducibility-relevant claims.
- **OASST-Pythia-12B as a "baseline"** is somewhat strawman-ish; it doesn't represent current frontier capability and mostly serves the narrative.
- **Threshold sensitivity / operating-point robustness** is not analyzed; the reported numbers are at a single (apparently fixed) threshold.

## Relevance
- 直接对应"LLM 在高风险决策中应退化为受约束接口、而非自由生成器"这一核心工程主张，是少数用真实儿科阑尾炎外部队列 + 多种 LLM 基线 + 校准/鲁棒性多维评估把这个范式做扎实的论文。
- 对任何要把"自由文本→结构化预测"管线部署到高风险场景（ED 分诊、影像报告、检验报告解读、ICU 早期预警）的团队都给出了可立即借鉴的架构模板：tool-call schema 抽取 + NaN-on-fail 校验 + XGBoost 决策 + LLM 仅做报告生成。

## Questions for Further Exploration
- How does ClaMPAPP perform on *actual* clinician-written free-text notes (with abbreviations, typos, copy-forward artifacts), not synthetic narratives? The fidelity audit only checks the *rewriter*, not the *real-world* extractor.
- Can the same architecture transfer to other acute triage tasks (pediatric sepsis, intussusception, DVT, bowel obstruction) where validated tabular ML models exist?
- How do you do external **calibration** properly — i.e., fit Platt or isotonic recalibration on a held-out calibration cohort, not the same cohort used to train XGBoost?
- What is the marginal value of running ClaMPAPP on top of a structured-data-only XGBoost in a prospective A/B or cluster-randomized workflow study? Is the LLM interface worth the latency/cost?
- How brittle is the architecture to LLM upgrades (Llama-4, future MedGemma, GPT-6)? Does the frozen-extractor approach require prompt re-tuning on every base-model upgrade?
- What is the cost/latency profile of the tool-calling extraction step in production, and is it compatible with ED throughput constraints?
- How do you extend the Feature Validator to catch *plausible-but-wrong* extractions (the gap the authors candidly flag)? LLM self-consistency? A second-pass LLM verifier? Cross-field clinical rules?
- Is there a meaningful subgroup (e.g., age <5 with atypical presentation — the very group §1 cites as the motivation) where the model underperforms?
- Could a tool-using LLM agent with retrieval + chain-of-thought close most of the gap to ClaMPAPP, or is the hybrid architecture's advantage structural rather than prompt-engineering-driven?
- How should the clinician-facing LLM report be audited / constrained to prevent it from drifting into recommendation territory?
