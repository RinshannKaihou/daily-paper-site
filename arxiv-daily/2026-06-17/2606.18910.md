> **一句话总结：** REVES 把 test-time scaling 形式化为 meta-RL 问题，证明了 sequential-revision 的单步恢复价值可以迁移到所有"使用 revision"的 TTS 算法（Theorem 3.1），并据此提出两阶段"在线数据增强 + 单轮 RL"框架：把成功恢复轨迹里的"near-miss"中间步拆成 revision / verification 两种 prompt，在 Qwen3-4B 上用 4B 模型在 circle packing 上追平 Gemini-2.0 Pro/Flash 等大模型进化搜索 SOTA。

> **TL;DR:** REVES reframes test-time scaling as a meta-RL problem, proves that improving one-step revision recovery transfers to every revision-using TTS algorithm, and operationalizes this with a two-stage loop (off-policy near-miss prompt augmentation + single-turn RL). On LiveCodeBench it beats the RL baseline by +6.5 and standard multi-turn by +4.0; on circle packing a 4B Qwen3 matches AlphaEvolve V2/ThetaEvolve/TTT-Discover SOTA; gains also transfer to MCTS, AB-MCTS variants, and Mind Evolution, and generalize to OOD constraint puzzles (n-queens, mini-sudoku).

## 核心贡献
- **概念层面：** 将 TTS 形式化为 meta-RL 问题，定义由 TTS 算法 ϕ 诱导的决策过程，提出"修订调用覆盖"概念，并证明提升 sequential-revision (SR) 的单步恢复概率可迁移到所有 revision-using 的 TTS 算法（**Theorem 3.1**）。同时给出反例说明优化单次 pass@1 不等价于优化 J^SR_ϕ（**Theorem 3.2**）。
- **方法层面：** 给出 J^SR_ϕ 的精确分解（**Lemma 4.1**），把它写成对访问状态上"一步恢复概率"按访问频次 ρ_θ(z) 的加权和，从而把"长链路信用分配"问题转化为单状态单轮的监督信号。提出 REVES 两阶段迭代框架：Stage I 用当前策略跑 SR rollout，保留成功恢复的轨迹，把其中"差一点对"的中间步转换为 (i) **revision prompt**（要求基于前序错误答案做修订）和 (ii) **verification prompt**（要求判断中间步是否正确）；Stage II 用普通单轮 RL（RLVR/GRPO 风格）在原始 prompt ∪ 增强 prompt 上更新策略，下一轮再刷新增强数据。
- **工程层面：** 由于修订/验证是单步前向，增强数据可**离线/异步**生成，与主训练并行（不需要 on-policy 的长链路采样），单步梯度即可推进训练，避免多轮 RL 的"每步内层串行采样"主要成本。
- **经验层面：** 在 math (AIME24/25, MATH500) / code (LiveCodeBench, CodeContest) / OOD 约束谜题 (n_queens, mini_sudoku) / circle packing 上都取得一致提升；且 REVES 训练出的策略同时提升 MCTS、AB-MCTS (β/Guassian/M) 和 Mind Evolution 等所有依赖 revision 的 TTS 算法，与 Theorem 3.1 一致。

## Key Contributions
- **Conceptual:** Casts TTS as a meta-RL problem with a TTS-induced decision process; defines Φ_R (revision-using TTS algorithms) and proves a **transfer guarantee** (Theorem 3.1): raising the one-step recovery probability on SR's revision inputs raises J_ϕ for any ϕ ∈ Φ_R that covers them, up to a coverage drift term.
- **Negative result:** **Theorem 3.2** shows two policies can be indistinguishable under single-shot pass@1 yet differ substantially under J^SR_ϕ — justifying a dedicated training objective.
- **Theoretical decomposition:** **Lemma 4.1** exactly rewrites J^SR_ϕ as Σ_z ρ_θ(z) · V_π(z), where V_π(z) is the one-step recovery probability at revision state z. This converts trajectory-level credit into per-state, single-turn supervision with no horizon credit assignment.
- **Algorithmic contribution (REVES):** Two-stage iterative framework — Stage I performs on-policy SR rollouts, retains only trajectories that succeed within budget K, and converts near-miss intermediates into decoupled **revision prompts** and **verification prompts**. Stage II trains the policy with standard single-turn RL on the union of original and augmented prompts; the loop alternates each epoch.
- **Off-policy efficiency:** Revision/verification targets are produced from cached successful trajectories, so Stage II is single-turn; augmented data can be generated in parallel with training, sidestepping the on-policy long-horizon sampling of multi-turn RL.
- **Empirical breadth:** LiveCodeBench +6.5 over RL, +4.0 over multi-turn; matches SOTA on circle packing with a 4B model; OOD generalization to constraint puzzles; gains transfer to MCTS / AB-MCTS variants / Mind Evolution.

## Methodology

### Setup
- **X**: distribution over problems; **Y**: answer space; **r⋆(x, y) ∈ {0, 1}**: ground-truth verifiable reward.
- **TTS algorithm ϕ**: orchestrates up to K policy calls, returns ŷ = y_τ at the random stopping time τ = K ∧ min{t : r⋆(x, y_t)=1}.
- **Sequential Revision (SR):** y_1 ~ π_θ(·|x); for t≥2, y_t ~ π_θ(·| x, y_{t−1}, f_{t−1}) where f is feedback (compiler error, verifier signal, etc.).
- **Best-of-N:** N parallel calls, external scorer picks ŷ.
- **TTS-induced MDP:** state s_t = (x, y_{i_t}, f_{i_t}), kernel P_ϕ chooses which prior attempt to branch from; same π_θ is Markov in s_t.

### Theorem 3.1 — Why train on SR?
Define:
- V^π(z) := P_{y~π(·|z)}[r⋆(x, y) = 1]  (one-step recovery probability)
- ρ^π_ϕ(z) := E_{(ϕ,π)}[Σ_t 1{z_t = z}]  (expected visit count to z under ϕ)

Under (C1) coverage and (C2) monotone recovery improvement assumptions, the gap J_ϕ(π_1) − J_ϕ(π_0) is bounded below by (1/C_ϕ)·[J^SR_ϕ(π_1) − J^SR_ϕ(π_0)] minus a small visit-count drift term. So training π on SR's inputs transfers to every ϕ ∈ Φ_R that covers them.

### Theorem 3.2 — Why single-shot RL is not enough
For any revision length K ≥ 2, there exist policies π_1, π_2, Δ(K) > 0, and a problem distribution X such that J_OneShot(π_1) = J_OneShot(π_2) but J^SR_ϕ(π_2) − J^SR_ϕ(π_1) ≥ Δ(K). So J_OneShot and J^SR_ϕ are different objectives, and aligning post-training with TTS deployment requires its own objective.

### Lemma 4.1 — Hazard decomposition
For a stopped SR trajectory ζ = (z_1, y_1, …, z_τ, y_τ) under π_θ:
J^SR_ϕ(θ) = Σ_t E_{x,ζ}[ 1{τ ≥ t} · V_π(z_t) ] = Σ_z ρ_θ(z) · E_{y'~π_θ(·|z)}[r⋆(x, y')]
This is a nonneg-weighted sum of per-state one-step recovery probabilities. Optimizing V_π at any visited z directly raises J^SR_ϕ with no horizon credit assignment.

### REVES algorithm
- **Stage I (data augmentation):** Run SR under current π_θ for each x ∈ batch. Discard trajectories that fail to recover within K or are already correct at y_1. From each retained (x, y_1, …, y_ℓ) with r⋆(x, y_ℓ)=1, build
  - **Revision data:** prompts asking the model to revise y_i → y_{i+1}.
  - **Verification data:** prompts asking the model to judge whether y_i is correct (so the model can self-stop at deployment where r⋆ is unavailable).
- **Stage II (single-turn RL):** Train π_θ with standard single-turn RL (RLVR/GRPO style) on the union of original RL prompts and Stage-I prompts. The updated policy seeds the next epoch's Stage I.

### Three design choices
1. **Verification prompts alongside revision data** — needed because oracle stopping is unavailable at test time; improves confidence calibration (AUROC 0.8641 vs 0.8313 for revision-only on AIME25).
2. **Offline generation** — augmented samples produced once per epoch; Stage II is single-turn; pipeline can be parallelized with training.
3. **Successful-trajectory filtering** — restrict Stage I to rollouts that succeed within K; this filter loosens naturally as π improves.

### Test-time stopping rules
- **Math:** TailConfidence surrogate (Fu et al., 2025); stop at first t ≥ 3 with c_t / Σ_{j≤t} c_j > c; on budget exhaustion pick argmax_t c_t (breaks ties uniformly).
- **Code:** Public test cases as surrogate r̃; stop at first t with r̃(x, y_t)=1, else output y_K.

### Models and data
- Base models: Qwen2.5-3B, Qwen2.5-7B, Qwen3-4B (non-thinking), DeepSeek-R1-Distill-Qwen-7B.
- Training data: Skywork/Skywork-OR1-RL-Data (7298 math + 1015 coding = 8313).
- Eval: MATH500, AIME 24/25 (math); LiveCodeBench Aug'24–Jan'25, Jan'25–May'25, CodeContest (code); n_queens, mini_sudoku from ReasoningGym (OOD puzzles).
- Baselines: **RL** (single-shot), **MultiTurn** (multi-turn rollouts without explicit self-verification), **PAG** (Jiang et al., 2025) (multi-turn + self-verification reward).

## Results & Highlights

### Coding (Table 1)
- Qwen-2.5-7B-REVES: LCB(Aug'24–Jan'25) TC-32 29.5 vs Multi-turn 25.5, PAG 25.7; TC-4 23.8 vs Multi-turn 21.3, PAG 23.2.
- Qwen3-4B-REVES: LCB(Aug'24–Jan'25) TC-32 **50.9** vs Multi-turn 49.5, PAG 43.6; CodeContest TC-32 **37.9** vs Multi-turn 34.9, PAG 32.4.
- Gains are consistent across budget levels B=4 and B=32, and across both LCB time windows.

### Math (Table 3, Oracle-4 / SelfConf-4)
- Qwen-2.5-7B-REVES: AIME24 O-4 25.9 vs RL 21.4, Multi-turn 20.0, PAG 19.6; MATH500 O-4 85.6 vs RL 79.4, Multi-turn 80.1, PAG 80.8.
- Qwen-2.5-3B-REVES: AIME24 O-32 33.7 vs RL 16.9, Multi-turn 20.5; roughly **doubles** the multi-turn baseline on AIME-style benchmarks at budget 32.
- Gains persist under SelfConf-4 (the practical, oracle-free stopping rule), showing the learned capability is not contingent on a perfect verifier.

### Circle packing (Table 2) — headline efficiency claim
At n=26, sum of radii: REVES on Qwen3-4B reaches **2.635983** (the optimum), matching AlphaEvolve V2 (Gemini-2.0 Pro+Flash), ShinkaEvolve (ensemble), ThetaEvolve (R1-Qwen3-8B), and TTT-Discover (Qwen3-8B) — all far larger systems. Notably, REVES uses the **smallest base model** in the comparison and reports **far fewer rollouts** than evolutionary baselines.

### OOD generalization (Figure 5)
- Trained on math+code only, evaluated on n_queens / mini_sudoku: Qwen-2.5-7B-REVES hits **73.3%** on mini_sudoku vs 45.7% for Qwen-2.5-7B-RL, 42.5% for Multi-turn, 35.0% for PAG; n_queens **59.7%** vs ~15% for vanilla base.
- RL-Augmentation (vanilla RL + revision-prompt aug) reaches 60.0% / 47.0% — REVES's full decoupling still wins.

### Transfer to other TTS algorithms (Table 4)
- Qwen-2.5-7B-REVES is best or tied-best on 1-shot, SeqRev, BoN, MCTS, MindEvo, ABM-A(β), ABM-A(G), ABM-M. Empirically confirms Theorem 3.1's prediction that revision gains transfer.

### Ablations (Table 5, Figure 6, Figure 7)
- **Revision vs. verification:** Revision-only → AIME25 O-4 16.5, SC-4 8.3; Verification-only → O-4 10.3, SC-4 7.1; **Both (REVES) → O-4 21.2, SC-4 15.8**. Verification alone is weaker, but combining with revision yields the largest gains and improves confidence calibration (AUROC 0.8641 vs 0.8313).
- **Continual augmentation matters:** Augmenting only at the initial epoch underperforms re-augmenting every epoch (Figures 6 right, 7 left) because stale augmented data no longer matches the model's current failure modes.
- **Budget scaling:** Increasing the augmentation budget K (1, 2, 4, 8, 16, 32) consistently improves accuracy on AIME24 and MATH500, with diminishing returns above ~8.

## Critical Assessment
- **Novelty:** the "split near-miss into revision + verification prompts" trick and the J^SR_ϕ hazard decomposition are clean and pedagogical. However, individual ingredients — self-revision (Self-Refine / STaR), verifier training, process reward models, and off-policy reuse of successful trajectories — are well-established. The real novelty is the **meta-RL framing** (Theorem 3.1) and the resulting **transfer guarantee** that explains *why* improving SR also improves MCTS/AB-MCTS/MindEvo. The decomposition (Lemma 4.1) is essentially a hazard-form identity for SR rollouts, which is straightforward once stated.
- **Rigor:** the empirical side is comparatively strong — multiple model sizes (3B, 4B, 7B), two baseline paradigms (single-shot RL, multi-turn RL with and without self-verification via PAG), both oracle and self-confidence stopping, and an OOD probe. However:
  - Confidence intervals / seed variance are not visible in the extracted tables; the +6.5 / +4.0 headline numbers are point estimates.
  - **LiveCodeBench reward hacking:** using public test cases as the surrogate r̃ is a known footgun for code post-training; the paper does not analyze whether REVES overfits to test-case-passing heuristics rather than genuine code synthesis. The LCB (Jan 25 – May 25) split also overlaps with the post-training data window, raising contamination concerns.
  - **Circle-packing SOTA claim** is impressive but should be qualified: SOTA on circle packing at n=26 has plateaued at 2.635983 (the optimum), so "matching SOTA" is really "matching the optimum" — this is a quality result, but it does not establish a new frontier.
  - **Theorem 3.1's coverage assumption (C1)** is a worst-case constant that could be loose; the bound in the paper is qualitative, and the paper does not measure C_ϕ empirically.
- **Reproducibility:** code is released; the SR/verification prompt templates are promised in Appendix A (not extracted in the main text but presumably included). The paper uses standard training data (Skywork-OR1-RL-Data) and well-known base models, so re-running is feasible. The self-confidence stopping threshold and the surrogate r̃ design are the trickiest pieces to port.
- **Significance:** the **transferability result** (Theorem 3.1 + Table 4) is the most important contribution in the paper — it gives a principled reason to invest in revision-specific post-training and says the gains are not tied to a specific inference algorithm. The efficiency claim (off-policy augmentation, single-turn training) is also practically important for production post-training pipelines.
- **Red flags to watch:**
  1. **Successful-trajectory-only filtering** (design choice 3) is a non-trivial inductive bias: it concentrates training on problems the base model can already recover from. For far-from-optimal base models, this could be vacuous; the paper does not show the breakdown by base-model capability.
  2. **The "verification prompt" is a soft, learned alternative to a real verifier.** Mixing a reward model and a learned verifier into the same data stream can mask reward-model failure modes; the paper's TailConfidence stopping rule is essentially the model's own confidence, which can be miscalibrated.
  3. **Table 1 REVES sometimes loses to Multi-turn on OneShot** (e.g., Qwen-2.5-7B OneShot 18.9 vs Multi-turn 19.4; Qwen3-4B OneShot 32.7 vs RL 32.2 only marginally). The headline is about TTS, not single-shot — fair, but worth noting.
  4. **PAG is a strong baseline** (Jiang et al., 2025) that already adds self-verification; REVES beating it on math/code while being methodologically simpler is a real win, but the gap is sometimes small (e.g., Qwen-2.5-3B MATH500 O-4 77.8 vs PAG 69.9, but 1-shot 64.5 vs PAG 64.0 is essentially tied).
  5. **The "matching SOTA with 4B" framing is technically correct but rhetorically loaded**: the comparison class is evolutionary search systems on a single well-studied optimization instance, not a broad reasoning benchmark. Strong evidence for the method, narrow evidence for general capability.

## Strengths
- Clean, theoretically grounded framing (meta-RL, hazard decomposition, transfer theorem) that goes beyond an empirical recipe.
- Decoupling revision vs. verification as a single design move is intuitive and ablated cleanly (Table 5).
- Engineering-friendly: off-policy augmentation can be parallelized with training; no per-step inner loop during Stage II.
- Strong evidence on **OOD** (n-queens, mini-sudoku) and on **transfer to other TTS algorithms** (MCTS, AB-MCTS, Mind Evolution), not just on a single decoding strategy.
- The circle-packing result with a 4B model matching 8B/ensemble/proprietary systems is a compelling efficiency story.

## Limitations
- Requires a **verifiable reward** (public test cases, ground-truth checkers, constraint checkers) at training time. Behavior under noisy or learned verifiers is not analyzed.
- **Successful-trajectory filter** biases the augmented set toward problems the current policy can already solve. Far-from-optimal base models may produce too few successful rollouts for the augmentation to be informative; the paper does not show this scaling behavior.
- The verification prompt trains a self-judge that is **not robustly evaluated** as a stand-alone reward model — only its effect on TailConfidence calibration (AUROC) is reported.
- All experiments use Qwen-family base models and the Skywork-OR1-RL-Data training set. Generalization across model families (e.g., Llama, DeepSeek) is not shown.
- The SelfConfidence stopping rule has a threshold c that is presumably hand-tuned; robustness of the reported numbers to c is not reported in the main text.
- The "off-policy data generation" is described at a high level; wall-clock comparisons to multi-turn RL are in Appendix F.2 (not extracted here), but the main paper does not give a full compute-matched comparison to MCTS/evolutionary baselines on circle packing (only rollouts, not FLOPs).

## Relevance
- 与"LLM 推理 / 后训练 / 高效 RL / test-time scaling"方向高度相关：REVES 直接把"如何让多轮 RL 训练目标与 test-time 推理策略对齐"这个长期痛点变成一个 meta-RL 框架，并给出可迁移的理论保证，对关注 test-time compute、revision-style 推理和 post-training 流水线的研究者/工程师都值得细读。
- 4B 模型在 circle packing 上追平 8B / 闭源进化搜索 SOTA 的结果，也给"小模型 + 强 revision 后训练"的路径提供了正面证据。

## Questions for Further Exploration
- How sensitive is the augmentation signal to the **successful-trajectory filter** as the base model becomes weaker? Is there a minimum recovery rate below which Stage I produces too few useful revisions?
- Can the **verification prompt** be replaced or augmented by a learned process reward model (PRM) to improve verification AUROC, and does this break the theoretical guarantee in Theorem 3.1?
- The transfer bound in Theorem 3.1 is qualitative; can the coverage constant C_ϕ be **estimated empirically** for MCTS / AB-MCTS / Mind Evolution, and how tight is the bound on real rollouts?
- Does REVES compose with **inference-time parallel scaling** (BoN, majority voting) on top of sequential revision, or do the gains saturate?
- The "off-policy augmentation" argument assumes the offline generated data is still informative for the updated policy. How stale can the augmentation be before performance degrades (and is there a principled refresh schedule)?
- How does the method behave on tasks where **public test cases / verifiers are unreliable** (e.g., competitive programming with weak test suites, or math where the verifier is a numeric parser that accepts near-misses)?
- The "SelfConfidence" stopping rule uses the model's own confidence as a surrogate. How robust is REVES's reported math gains to **miscalibration** of this signal, and can the verification prompt explicitly correct it?
- Can the near-miss / verification pipeline be extended to **multi-modal** or **tool-use** reasoning tasks, where the "verifier" is the result of an external tool call?
- How does REVES interact with **DPO / preference-based post-training** when preference data is collected from revision rollouts — is the resulting policy better than either approach alone?
