Up to 40% of a transformer's attention can be replaced by short, auditable Python — with QA accuracy preserved.

Explaining Attention with Program Synthesis · Amiri Hayes, Belinda Z. Li, Jacob Andreas · arXiv:2606.19317
interpretability program-synthesis attention mechanistic-interpretability transformer
用 Claude Sonnet 4 逐头合成 Python 程序作为注意力头的可执行代理，在四个模型上共生成 1,664 个程序；按 IoU 降序贪心替换 25–40% 的头时，困惑度仅小幅上升，六个下游 QA 基准不掉点。
— Source one-liner, original Chinese preserved alongside the distilled English claim above.
Key metric
30–40%of attention heads in BERT-base, GPT-2-small, TinyLlama-1.1B, and Llama-3.2-3B can be swapped for synthesized programs while HellaSwag, PIQA, SciQ, ARC-Easy, Social IQA, and COPA stay flat.
Why this matters

A third of an LLM's attention is now replaceable by auditable code — opening a path to mechanistic editing, safety audits, and neuro-symbolic hybrids without ever touching the weights.

Caution · critical edge

The 30–40% number is IoU-greedy, the perplexity test is in-distribution on TinyStories, and a substantial fraction of heads sit below 40% IoU. Cross-head reuse is real — but any single program may not be the causally correct one for that head.

01

Methodology — one pipeline, run per head

An LLM (Claude Sonnet 4) is asked to write Python that imitates one attention head at a time. The pipeline runs in nine small steps, then a global program library is assembled and tested by causal interchange intervention:

01 · EXTRACTRun the model on TinyStories; record each head's n×n attention matrix.
02 · FILTERKeep only the top 2.5% of weights by magnitude — drop the noise floor.
03 · SUMMARIZEFormat surviving token-pair weights as a ~4k-token text prompt.
04 · SYNTHESIZEClaude (with NumPy / spaCy / NLTK) emits a candidate program π : X → Â.
05 · RANKValidate syntax; score survivors by Jensen-Shannon distance to A.
06 · REFINEOne round of contrastive feedback (best vs. worst examples) → π′.
07 · SELECTPick π* by IoU on a held-out set; append to the global library.
08 · INTERVENEReplace A with π(X) during a forward pass; measure PPL + downstream QA.
09 · CATEGORIZECluster programs by docstring-prompt into 6 functional types.

Models audited: BERT-base (144 heads, bidirectional) · GPT-2-small (144) · TinyLlama-1.1B (704) · Llama-3.2-3B (672, 28 layers). Library size: 1,664 programs. Compute cost: ~$150 in Claude API spend, ~35M input + 3.5M output tokens. Code & data: github.com/AmiriHayes/explaining_attention_heads

02

Results — beyond the headline

The headline metric is supported by a sharper story across architectures and scales. Approximate values from the paper's Figures 4–7 and §Results:

Model Heads Architecture Best-program IoU PPL Δ at 25% replace
BERT-base144encoder · bidi~ 60%+18%
GPT-2-small144decoder · causal≈ 69%+14%
TinyLlama-1.1B704decoder · causal≈ 74%+15%
Llama-3.2-3B672decoder · causal≈ 79%+17%
  • Spearman r < −0.91 between per-head IoU and substitution cost — the same metric that measures fit also predicts downstream harm.The only result of its kind for program surrogates; strong and falsifiable.
  • Structural baselines explode. A lower-diagonal or uniform program pushes perplexity up >1000% after just 5% of TinyLlama heads — the headline number isn't "any program works."
  • Depth-dependent categories in GPT-2. First-token programs cluster in early layers; syntactic / coreference programs dominate middle and later layers — a re-discovery of the classical NLP pipeline, in code.
  • Best beats intended. The globally-best program (picked from any head's library) usually fits a head better than the program synthesized for that head — cross-head reuse is real, supporting a MAP-Elites-style view of the head space.
  • Decoder > encoder. Causal heads are systematically easier to approximate than bidirectional ones, attributed to MLM producing more diffuse attention distributions.
03

Strengths ↔ Limitations

Strengths +

  • Clean four-stage pipeline (extract → summarize → synthesize → re-rank) that extends to any model with extractable attention.
  • Three independent evaluation lenses — IoU, perplexity, downstream QA — none of which is sufficient alone.
  • Genuine causal evidence via interchange interventions, not correlational interpretability claims.
  • Cross-architecture and cross-scale replication: encoder vs. decoder, 144 → 704 heads, 0.1B → 3B parameters.
  • Total compute ~$150; code, programs, and prompts released — the result is genuinely reproducible.
  • Artifacts are auditable Python, not natural-language neuron descriptions — they can be read, modified, and re-run.
  • Honest reporting: the paper flags the heads it cannot capture and attributes high-replacement degradation to them.

Limitations −

  • No head is fully captured — a substantial fraction sits below 40% IoU.
  • Programs are short and not very diverse; many high-scorers collapse onto a few motifs (first-token, prior-token, sentence-initial).
  • Single training corpus (TinyStories); transfer to Wikipedia, code, multilingual text is untested.
  • Perplexity evaluation is in-distribution; out-of-distribution PPL under replacement is not measured.
  • Replacement is IoU-greedy — a random-order baseline would be a more conservative stress test.
  • Only one code-LM (Claude Sonnet 4) is used; transfer to weaker or stronger code-LMs is not characterized.
  • Only attention heads are addressed — MLPs, embeddings, layer norms, and cross-attention are out of scope.
  • Programs approximate attention outputs, not the full head contribution — the output projection after attention is still neural.
  • No adversarial / safety robustness analysis; the work is interpretability research, not a deployable surrogate.
04

Questions for further exploration

What this paper invites next open directions

  1. Does random-order replacement still rise gracefully, or does the library only work for the easy heads?
  2. How does this scale to ≥7B models where heads may be more polysemantic and the 30–40% ratio may collapse?
  3. Can the 1,664 programs be factored into a few dozen reusable sub-routines (circuit-style motifs)?
  4. Do the programs transfer across distributions — Wikipedia, code, multilingual, long-context inputs?
  5. Could multi-round synthesis with test-case generation, symbolic execution, or counterexample search close the <40% IoU gap?
  6. Can the programs be used as edits in a causal-scrubbing style — remove one program, observe the behavior delta, localize function?
  7. Do the surrogates hold up under adversarial / OOD prompts (jailbreaks, code-switching, prompt injection)?
  8. How sensitive is the result to the code-LM choice — would an open Code-Llama reproduce the IoU curve at higher compute but lower API cost?
  9. Could the same pipeline be applied to MLP layers — substantially harder synthesis, but a much larger payoff, since MLPs hold most of the parameters?
