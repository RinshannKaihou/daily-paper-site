> **一句话总结：** SWE-chat 是首个大规模"野外"编码智能体交互数据集（约6000个真实会话、63K用户提示、355K工具调用），实证发现：41%的会话属于"vibe coding"、仅44%的智能体产出代码被提交、"vibe-coded"代码的安全漏洞率约为人类代码的9倍，且用户在44%的轮次中主动打断或反驳智能体。

> **TL;DR:** SWE-chat is the first large-scale in-the-wild dataset of real coding agent sessions (~6K sessions, 63K user prompts, 355K tool calls) from opt-in open-source developers. Empirically, 41% of sessions are "vibe coding," only 44% of agent-produced code survives into commits, vibe-coded code introduces ~9x more security vulnerabilities per line than human-written code, and users push back or interrupt in 44% of turns—suggesting agent autonomy is outpacing oversight.

## 核心贡献

1. **首个野外编码智能体数据集**：通过 Entire.io CLI 的 checkpoint logging，自动从 200+ 公开 GitHub 仓库收集了约 6000 个真实会话、13K checkpoints、63K 用户提示、355K 工具调用、共 2.7M 日志事件。覆盖 Claude Code、OpenCode、Gemini CLI、Cursor、Factory AI Droid 等多种智能体（85% 来自 Claude Code）。
2. **首次实现"人类 vs 智能体"逐行代码作者归属**：区分 human-only（22.7%）、collaborative（36.5%）、vibe coding（40.8%）三种模式。
3. **使用模式实证分析 (RQ1)**：揭示用户意图远比现有基准多样——理解代码（19%）是最常见请求，而不是生成代码（13.4%）；1/3 工具调用是 bash 命令；用户行为极度"nitpicky"。
4. **失败模式与安全性量化 (RQ2)**：智能体代码只有 44.3% 被提交；vibe coding 带来约 9× 于人类、5× 于协同的安全漏洞注入率（Semgrep 静态分析）；vibe-coding 花费约 3× tokens、0.13$/100 行（人类为 0.05$）。
5. **"自主性跑赢监督"的系统性观察**：智能体仅在 1.4% 的轮次主动寻求澄清，而用户在 44% 的轮次打断或反驳。

## Key Contributions

1. **First in-the-wild coding-agent interaction dataset** with complete traces (prompts, tool calls, code diffs, line-level authorship) rather than curated patch-generation tasks.
2. **Line-level human-vs-agent code attribution**, enabling quantitative study of three coding modes: human-only, collaborative, vibe coding.
3. **Empirical RQ1 analysis** of how users actually use agents: understanding code is the #1 intent (19%), task distribution is much broader than GitHub-issue-style benchmarks assume, and interactions are multi-turn and nitpicky.
4. **Empirical RQ2 analysis** of failures: code survival rates, token/dollar/time efficiency per committed line, and security-vulnerability introduction rates via Semgrep static analysis.
5. **Living dataset + data pipeline** that is continuously updated; published with ethics-reviewed PII/credential scrubbing (Presidio + TruffleHog).

## Methodology

- **Collection pipeline**: Users opt-in by installing Entire.io CLI, which hooks into git to automatically capture agent session transcripts and link each commit to the corresponding session with line-level attribution. Only public repos with redistribution-friendly licenses are included.
- **Annotation scheme** (Table 2): LLM-as-judge labels at two granularities—session-level (success score 0–100, user persona: expert nitpicker / vague requester / mind changer / other) and prompt-level (intent among create/refactor/debug/understand/connect/git/test/other; pushback type: correction/rejection/failure-report/none). Inter-annotator agreement is reported as moderate-to-high; zero-shot LLM judges were validated against human gold labels with multiple prompt paraphrases.
- **Efficiency metrics**: coding efficiency (fraction of agent output reaching commit), code survival rate (excluding agent self-overwrites), plus tokens / dollars / wall-clock minutes per 100 committed lines.
- **Security metrics**: Run Semgrep on pre- and post-commit snapshots, restricted to modified files, counting Introduced vs Fixed findings per 1K added lines.
- **PII handling**: Microsoft Presidio + SpaCy NER and TruffleHog for credential redaction, following WildChat.

## Results & Highlights

- **Dataset scale (as of April 2026)**: ~6K sessions, 205 public GitHub repos, 13K checkpoints, 63K user prompts, 355K tool calls, 2.7M logged events. About 85% from Claude Code.
- **Coding-mode distribution**: Human-only 22.7% / Collaborative 36.5% / Vibe coding 40.8%. Vibe-coding share doubled from 20% to >40% over a three-month window.
- **Task diversity**: Understanding code 19.0%, create new code 13.4%, git ops 13.4%, debugging 13.0%; "other" 26.6%. Only ~33% of tool calls are file edits; another ~33% are bash.
- **Session success**: Mean 75.7, median 82.0 (LLM-rated 0–100); 90% of sessions score ≥50. Low-scoring tail dominated by user interrupts and off-target agent work.
- **Agent code survival**: Overall 50.3% survival, 44.3% coding efficiency; vibe coding 64.6% survival / 59.0% efficiency; collaborative 44.1% / 38.2%.
- **Cost per 100 committed lines**: Vibe $0.13 (~3× collaborative, ~2× human-only tokens); collaborative $0.05 (most efficient); human-only $0.07.
- **Security** (Semgrep, per 1K added lines): Vibe introduces 0.76 vulns (vs 0.08 human-only, 0.14 collaborative) — ~9× human-only and ~5× collaborative. Vulns include path traversal, command injection, unsafe format strings, SQL injection.
- **Oversight asymmetry**: Claude Code asks for clarification in only 1.1–2.6% of turns; users interrupt in 3.3–6.0% and push back (correction / rejection / failure report) in ~39% of turns. Combined pushback+interrupt ≈ 44%.
- **Autonomy trend**: 99.9th-percentile turn duration now exceeds 100 minutes and is trending upward.

## Critical Assessment

The paper is best viewed as a **data + descriptive-statistics** contribution rather than a causal study, and most of its findings should be read with that framing in mind.

- **Selection bias is acknowledged but severe.** The sample is restricted to (a) developers who installed the Entire.io CLI, (b) opted into public checkpoint logging, and (c) pushed to public GitHub. The authors note a "large fraction of data comes from Entire.io's own repository," which could dominate behavioral patterns. There is essentially no coverage of enterprise / proprietary workflows.
- **Claude Code monoculture**: ~85% of sessions come from Claude Code. Claims about "coding agents in general" are actually claims about Claude Code with some noise. Cross-agent comparisons in the paper are weak.
- **LLM-as-judge labels are load-bearing but admittedly unreliable.** The authors themselves cite their own prior work on "LLM hacking" and caution against taking these labels at face value—yet most RQ1/RQ2 narratives (success scores, personas, intent) ride on these annotations. Inter-annotator agreement is reported as "moderate to high" without concrete kappa/ICC numbers in the main text.
- **Security analysis leans entirely on Semgrep** and raw counts per 1K added lines. Semgrep has known false-positive rates, and per-line normalization may unfairly penalize vibe coding (which simply produces more code). No severity weighting, no manual triage reported in the main text.
- **Code-survival is a biased metric.** Sessions where users abandon agent output entirely (never commit) are not captured at all, which the authors note would over-estimate success rates. Semantic rewrites of agent suggestions are counted as human overwrites, under-counting agent usefulness.
- **The "9× more vulnerabilities" headline is likely to be over-cited.** It is a relative rate on a small absolute base (0.76 vs 0.08 per 1K lines), conditioned on Semgrep findings only, for a self-selected user population. It should not be read as a causal claim that vibe coding is 9× less safe.
- **No ablation on the LLM judge** (e.g., swapping judges, sensitivity to prompt wording beyond paraphrase robustness) is shown in the main text, only in appendix.
- **Three-month observation window** is short for the "doubling of vibe coding" trend; it may reflect tool adoption curves rather than a durable behavioral shift.

The paper is refreshingly honest about these limitations in Appendix A, which partially offsets the concerns—but readers citing the headline numbers should preserve that caution.

## Strengths

- **Fills a genuine gap**: no public dataset previously captured real multi-turn agent sessions with human-authorship attribution (Table 1 comparison is convincing).
- **End-to-end pipeline**: instrumentation, PII scrubbing, annotation, and continual updates are all thought through; data + code + website are released.
- **Line-level attribution is a real technical artifact** and enables coding-mode analysis that prior datasets (SWE-smith-trajectories, CoderForge, AgentPack, etc.) cannot support.
- **Framing is well-calibrated**: explicitly positions findings as a starting point for a research agenda, not definitive claims, and points to concrete follow-up directions (realistic benchmarks, user simulators, human-agent interaction design).
- **Security findings are actionable** even if imperfect: they establish that the autonomy/oversight gap has measurable downstream consequences, consistent with Perry et al. 2023 and Pearce et al. 2025.
- **Connects well to "invisible failures" literature** (Potts & Sudhof 2026) — the observation that agents rarely self-signal uncertainty while silently producing problematic code is a clean empirical hook for a growing subfield.

## Limitations

- Early-adopter + open-source + public-repo selection bias; unclear generalization to enterprise.
- Heavy Claude Code skew (~85%).
- LLM-judge-derived metrics (success scores, personas, intents) are not independently validated at scale in the main text.
- Abandoned/uncommitted sessions are invisible to the pipeline, biasing efficiency upward.
- Line-level attribution under-counts semantic agent contributions that humans reformulate.
- Security evaluation is Semgrep-only, with no severity weighting or manual confirmation.
- Observation window (3 months) is short for behavioral-trend claims.
- No controlled comparison across agent families—differences between Claude Code, Cursor, Gemini CLI, etc. cannot be isolated given the imbalance.

## Relevance

与用户研究兴趣的契合度较高：

- **ML 可靠性 / silent data corruption**：论文与 Potts & Sudhof (2026) 的 "Invisible failures in human-AI interactions" 直接对接——智能体在 98%+ 的轮次不主动标注不确定性，但在近一半的轮次被用户打断/反驳，是"沉默失败"的典型场景。
- **推理评估 (inference evaluation)**：为"真实工作流"基准提供原始语料，能直接用于构建非 one-shot、多轮、含用户反馈的评估集，是对 SWE-bench 系列的重要补充。
- **系统卡 / system cards**：LLM-judge 打分与 Semgrep 漏洞率可以作为部署时的风险度量原型，和 Anthropic/Cursor 的系统卡披露直接相关。
- **可解释性 / 训练稳定性 / 量化**：相关性较弱，论文没有涉及 hidden states、precision 或训练稳定性层面的分析。
- 对构建"用户模拟器"、"自主性 vs 监督"框架的研究者（Naous 2025, Buening 2026, Zhou 2026 sim2real）非常有用。

总体：**高相关性**，特别是对 ML 可靠性与推理阶段评估方向。

## Questions for Further Exploration

1. How much do the headline numbers (41% vibe coding, 9× vulnerability rate, 44% pushback) change if Entire.io's own repo is removed from the sample?
2. Can the security finding be replicated with a severity-weighted analyzer (e.g., CodeQL with CVE-mapped rules) and with manual triage of a Semgrep subset, so we can distinguish true vulnerabilities from lint-style findings?
3. What is the kappa / ICC for LLM-judge vs human labels on session success and user persona, and how does it vary across the three coding modes?
4. Do the pushback and interruption rates predict downstream commit-revert or bug-fix rates—i.e., are users' "nitpicks" correct signal?
5. If we restrict analysis to non-Claude-Code agents (the remaining ~15%), do the coding-mode distributions and failure modes look materially different?
6. Is the rising 99.9th-percentile turn duration causally linked to improved agent capability, or is it an artifact of users delegating longer tasks over time as trust grows?
7. Can SWE-chat sessions be used to train a user simulator whose behavior on held-out sessions matches the observed nitpicker/vague/mind-changer distribution, enabling cheap offline eval of new agents?
8. For commits classified as "vibe coded," what fraction of Semgrep-flagged vulnerabilities were already present in surrounding human-authored code vs genuinely introduced by the agent edit?
