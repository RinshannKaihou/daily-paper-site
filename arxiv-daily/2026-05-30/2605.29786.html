Croissant Tasks decouples ML benchmark problem definitions from solutions via a declarative JSON-LD metadata format.
Croissant Tasks: A Metadata Format for Reproducible Machine Learning Evaluations Benjelloun, Bianco, Guyon, Vanschoren et al. · 2026-05-28
Reproducibility Metadata LLM Agents Benchmarking

Key Metric

97.1%
evaluation metrics correctly implemented from Croissant Tasks alone, outperforming 90% from raw PDF.

Why This Matters

Eliminates brittle code replication by letting autonomous agents implement benchmarks from compact, machine-readable specifications alone.

Critical Edge

Reproduced scores can diverge sharply from reported values (e.g., 40.0 → 23.36 F1), so conceptual reproducibility does not yet guarantee numerical agreement.

Methodology

  • Declarative schema. JSON-LD grounded in schema.org / RDF splits every benchmark into a cr:TaskProblem (givens + placeholder specs) and a cr:TaskSolution (concrete code, data, results) linked by schema:isBasedOn. Six properties—input, output, implementation, execution, evaluation, subTask—cover the full pipeline.
  • Automated extraction. An LLM agent reads five diverse NeurIPS 2025 D&B papers and emits Croissant Task descriptions; SHACL validation plus expert review checks quality.
  • Agentic reproduction. A coding agent receives either the raw PDF (baseline) or only the Croissant Tasks file, then builds a complete evaluation pipeline—data loading, model invocation, metric computation, and TaskSolution generation—with web access blocked and up to three human guidance prompts allowed.

All five benchmarks—spanning text, code, medical imaging, brain MRI, and safety—were fully expressible. The compact specification also prevented the context-overload failures that caused the PDF-only agent to terminate prematurely on long documents.

Strengths & Limitations

Strengths

  • Elegant problem-solution duality maps naturally onto real-world challenges.
  • Agentic workflow (pdf2ct → ct2code) is well-documented and reproducible.
  • Authors openly report failure modes and coverage gaps.
  • Built on web standards (schema.org, JSON-LD, RDF) for interoperability.
  • LLM extraction removes the need for manual schema authoring.

Limitations

  • Only five benchmarks with one baseline each—feasibility demo, not large-scale proof.
  • Heavy dependence on LLM quality; no ablation with weaker models.
  • Missing or incorrect metadata propagates directly into broken reproductions.
  • No evidence of community adoption in major platforms yet.
  • Up to 3 human prompts per benchmark limits true autonomy.

Questions for Further Exploration

  1. How does the pipeline scale to hundreds of benchmarks—does the 97% success rate hold?
  2. What is the minimal LLM capability required for reliable extraction and implementation?
  3. Can the format cover training pipelines, data augmentation, or multi-stage workflows?
  4. What institutional mechanisms (conference mandates, platform integrations) would drive adoption?
  5. Can problem-solution duality be leveraged for automated benchmark generation, not just reproduction?
