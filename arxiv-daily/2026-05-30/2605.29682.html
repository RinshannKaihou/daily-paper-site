Why This Matters
EFC replaces empirical guesswork in agent framework design with a quantitative scaling law rooted in feedback quality rather than token count.
Critical Edge
EFC's four factors are hand-designed heuristics relying on hidden state; real-world prospective holdout drops to R² = 0.85.

Methodology

The paper treats an agent harness as a closed-loop computation producing a trajectory of states, actions, observations, and memory updates. It defines Effective Feedback Compute (EFC) as a trace-level scalar that credits each feedback event only when four conditions hold simultaneously:

  • Informativeness (It) — reveals task-relevant information
  • Validity (Vt) — supported by reliable evidence (checkers, execution)
  • Non-redundant Relevance (Rt) — addresses active subgoal with new information
  • Memory Update (Mt) — changes plan/state affecting later actions

Run-level EFC sums the product of these four bounded factors. A task-demand normalization term Dtask (reasoning depth × tool entropy × state pressure × observation noise × verifier visibility) enables cross-task comparison. Validation spans five layers: synthetic controllable tasks, executable code, real benchmark traces, held-out splits, and prospective holdout.

In matched-budget interventions, improving feedback quality raised success from 0.27 to 0.90 while raw cost and tool calls remained fixed. Harness efficiency η alone explained success variation at R² = 0.97, while raw cost explained almost none (R² = 0.01).

Strengths

  • Addresses a genuine gap: raw expenditure is a poor scaling coordinate for agent harnesses
  • Five-layer validation (synthetic, executable, real, held-out, prospective) is exceptionally thorough
  • Matched-budget experiments cleanly isolate feedback quality from raw spending
  • Decomposition into harness efficiency η and task demand provides actionable design guidance

Limitations

  • Oracle-EFC requires hidden state; gap between Oracle and Estimated is not fully characterized
  • EFC factor definitions are heuristic and may not generalize across all agent architectures
  • Task suite is author-curated; independent third-party evaluation would be more convincing
  • No discussion of computational cost of EFC estimation itself
  • Limited to single-agent settings; multi-agent harnesses are not addressed

Questions for Further Exploration

1Is the product form the optimal aggregation, or would attention-weighted or additive forms perform better?
2How stable is Estimated-EFC calibration across different base models (open vs. closed)?
3Can EFC extend to multi-agent collaboration, and how should inter-agent feedback be measured?
4Can harness efficiency η serve as an automatic optimization objective for agent framework design?
5Are Dtask's five components sufficient for open-domain dialogue or creative generation tasks?
