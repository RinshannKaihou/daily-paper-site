## 一句话总结

本文提出"有效反馈计算量"(Effective Feedback Compute, EFC)作为智能体框架(agent harness)的缩放坐标,证明决定智能体性能的不是花多少原始计算,而是原始预算被多高效地转化为有用的、持久的反馈。

## TL;DR

The paper introduces Effective Feedback Compute (EFC) as a trace-level scaling coordinate for agent harnesses. EFC credits feedback only when it is informative, valid, non-redundant, and retained for later decisions. Across synthetic tasks, executable code tasks, real benchmark traces, held-out splits, and prospective validation, EFC-based coordinates consistently outperform raw-compute baselines (R^2=0.33--0.42) and a strong multivariate SAS baseline (R^2=0.88), with Oracle-EFC/D_task reaching R^2=0.99. The key insight: harness scaling is governed not by how much computation is spent, but by how efficiently raw budget is converted into durable, task-sufficient feedback.

## 核心贡献

1. **形式化EFC指标**: 定义了一个事件级的反馈质量度量,每个反馈事件通过四个因子(信息量 $I_t$、有效性 $V_t$、非冗余相关性 $R_t$、记忆更新 $M_t$)的乘积来评估,只有同时满足"有用、可靠、不重复、被保留"的反馈才被计入。

2. **提出任务需求归一化**: 引入 $D_{\text{task}}$ 作为任务需求尺度,将EFC除以任务需求以实现跨任务比较,解决了不同难度任务间反馈需求量差异的问题。

3. **分解缩放机制**: 将智能体框架缩放分解为两个机制——框架效率 $\eta = \text{EFC}/C_{\text{raw}}$ (原始预算到有效反馈的转化率)和任务归一化反馈 $\text{EFC}/D_{\text{task}}$ (反馈量是否足够满足任务需求)。

## Key Contributions

1. **EFC Formalization**: Defines Effective Feedback Compute as a trace-level scalar that credits feedback events only when they are simultaneously informative ($I_t$), valid ($V_t$), non-redundant ($R_t$), and retained for later decisions ($M_t$), using a product form $\text{EFC}_t = \kappa I_t V_t R_t M_t$.

2. **Task-Demand Normalization**: Introduces $D_{\text{task}} = L \cdot H_{\text{tool}} \cdot S_{\text{state}} \cdot (1 + N_{\text{obs}}) \cdot (1 - V_{\text{oracle}})$ to normalize EFC across tasks with different feedback requirements, enabling cross-task scaling comparisons.

3. **Decomposition into Efficiency and Demand**: Shows that harness scaling decomposes into harness efficiency ($\eta = \text{EFC}/C_{\text{raw}}$, controlled by routing, verification, memory, and observation quality) and task-demand sufficiency ($\text{EFC}/D_{\text{task}}$).

4. **Multi-layer Validation**: Validates EFC across five experimental layers: synthetic controllable tasks, semi-realistic executable code tasks, real benchmark traces (HumanEval, Terminal, SWE), held-out splits, and prospective validation, demonstrating consistent superiority over raw-compute baselines.

5. **Oracle-Free Estimation**: Develops Estimated-EFC that recovers the oracle signal from trace-observable features alone (checker use, plan updates, memory retention, error avoidance), achieving $R^2 = 0.93$ vs oracle at $R^2 = 0.95$.

## Methodology

### Problem Formulation

The paper models an agent harness as a closed-loop computation. Given task distribution $\mathcal{T}$, a harness $h$ paired with base model $m$ produces a trajectory $\tau = \{(s_t, a_t, o_t, u_t)\}_{t=1}^T$, where each step includes state, action, observation, and memory update. The goal is to find a scalar scaling coordinate $z$ that predicts failure rate $E(z) = E_\infty + Az^{-\alpha}$.

### EFC Definition

Each feedback event $e_t$ receives four bounded factors in $[0,1]$:
- **Informativeness** ($I_t$): reveals task-relevant information
- **Validity** ($V_t$): supported by reliable evidence (deterministic checkers, execution results)
- **Non-redundant Relevance** ($R_t$): addresses active subgoal, adds new information
- **Memory Update** ($M_t$): changes plan/state/memory affecting later actions

Run-level EFC: $\text{EFC}(\tau) = \kappa \sum_{t=1}^T I_t V_t R_t M_t$ with $\kappa=10$.

### Variants for Different Settings

- **Oracle-EFC**: Uses hidden task state and ground-truth progress (synthetic tasks only)
- **Estimated-EFC**: Uses trace-observable features (checker firing, tool-result references, plan changes, memory retention, repeated-error avoidance, observation consistency)
- **NRS-EFC** (Non-redundant Stable): Filters out repeated/unstable observations for real traces
- **Task-demand-normalized**: $\text{EFC}/D_{\text{task}}$ for cross-task comparison

### Experimental Layers

1. **Synthetic controllable tasks**: Procedurally generated with known hidden state; provides Oracle-EFC calibration
2. **Semi-realistic executable code tasks**: HumanEval-like format with executable test validation
3. **Real benchmark traces**: HumanEval-style code generation, terminal interaction, SWE tasks
4. **Held-out validation**: Across task families, harness variants, models, and combined configurations
5. **Prospective validation**: New real traces evaluated under pre-specified metric and calibration protocol

## Results & Highlights

### Controlled Scaling (Synthetic Tasks)
- Raw tokens: $R^2 = 0.33$, tool calls: $R^2 = 0.42$
- SAS baseline: $R^2 = 0.88$
- Oracle-EFC: $R^2 = 0.94$, Estimated-EFC: $R^2 = 0.94$
- **Oracle-EFC/$D_{\text{task}}$: $R^2 = 0.99$, MAE = 0.02**

### Matched-Budget Interventions
- Improving feedback quality raises success from **0.27 to 0.90** while raw cost and tool calls remain fixed

### Trace-Time Estimation
- Estimated-EFC/$D_{\text{task}}$ reaches $R^2 = 0.93$ (vs Oracle at 0.95), demonstrating oracle-free estimation is viable

### Executable Code Tasks
- Estimated-EFC/$D_{\text{task}}$: $R^2 = 0.97$
- Tool calls become a strong baseline ($R^2 = 0.94$) due to executable feedback alignment, but EFC still outperforms

### Module Ablations
- Harness efficiency $\eta$ explains success variation with $R^2 = 0.97$; raw cost explains almost none ($R^2 = 0.01$)
- Router quality (+0.28), verifier strength (+0.22), and memory fidelity (+0.20) are the largest efficiency gains

### Real Benchmark Traces
- NRS-EFC/$D_{\text{task}}$ reaches $R^2 = 0.92$, while raw compute has near-zero or negative fit
- Slice-specific $\eta$ analysis: efficiency is harness-task-dependent, not globally fixed

### Prospective Holdout
- NRS-EFC/$D_{\text{task}}$: $R^2 = 0.85$ (pre-specified protocol)
- Raw tokens, tool calls, wall time, operations, raw cost all obtain **negative** held-out $R^2$

## Critical Assessment

The paper makes a compelling case that raw expenditure (tokens, tool calls, wall time) is a poor predictor of agent performance, and that a feedback-quality-aware metric like EFC is a much better scaling coordinate. The multi-layer validation and prospective holdout are genuine methodological strengths.

However, several concerns merit attention:

1. **EFC's four factors ($I_t, V_t, R_t, M_t$) are themselves hand-designed heuristics**. While the product form is principled, the operationalization of each factor in synthetic tasks relies on access to hidden state that is unavailable in practice. The Estimated-EFC bridges this gap, but its accuracy depends on the quality of trace-observable proxies.

2. **The product form assumes independence of the four factors**. If informativeness and validity are correlated (e.g., valid observations tend to be more informative), the product may double-count or undercount. The paper does not analyze this.

3. **Task demand $D_{\text{task}}$ introduces additional hand-specified components** (reasoning depth, tool entropy, state pressure, observation noise, verifier visibility) that require estimation per task. The fitted variant helps, but portability to entirely new task domains remains uncertain.

4. **The synthetic task suite is designed by the authors** to validate EFC. While the paper also tests on real benchmarks, the strongest results ($R^2 = 0.99$) come from the synthetic layer where Oracle-EFC has privileged access. The real-world gap is evident: prospective $R^2$ drops to 0.85.

5. **Comparison to baselines is somewhat unfair**. SAS uses only system-level quantities, while EFC uses fine-grained event-level information including checker results, plan changes, and memory states. A more informative comparison would be against a similarly granular non-EFC baseline.

6. **Scalability of EFC estimation** is not discussed. Computing trace-observable features for every event in a long trajectory may be costly, and the paper does not address computational overhead.

## Strengths

- Addresses a genuinely important gap: raw expenditure is a poor scaling coordinate for agent harnesses
- The EFC formulation is intuitive and well-motivated
- Five-layer validation (synthetic, executable, real, held-out, prospective) is exceptionally thorough
- The decomposition into harness efficiency and task demand provides actionable design guidance
- Matched-budget experiments cleanly isolate feedback quality from raw spending
- Prospective holdout with pre-specified protocol is a gold-standard validation practice

## Limitations

- Oracle-EFC requires hidden state; the gap between Oracle and Estimated variants is not fully characterized
- EFC factor definitions are heuristic and may not generalize across all agent architectures
- The task suite, while multi-layered, is curated by the authors; independent third-party evaluation would be more convincing
- No discussion of computational cost of EFC estimation itself
- The power-law failure model $E(z) = E_\infty + Az^{-\alpha}$ is assumed rather than derived
- Limited to single-agent settings; multi-agent or collaborative harnesses are not addressed

## Relevance

本文对智能体系统研究具有重要意义。当前LLM智能体框架(如ReAct、Reflexion、Tree of Thoughts等)的设计很大程度上依赖经验性调参,缺乏统一的理论指导。EFC提供了一个量化框架,可以帮助理解和优化以下问题:

- 为什么增加推理时间计算不一定提升性能
- 如何设计更高效的智能体框架(重点在于反馈质量而非数量)
- 如何在不同任务间公平比较框架效率
- 框架中哪些模块(路由器、验证器、记忆)对性能贡献最大

对于从事LLM智能体、test-time compute、推理时缩放方向的研究者,本文提供了一个有用的分析工具和新的视角。

## Questions for Further Exploration

1. EFC的四因子乘积形式是否是最优聚合方式?加法形式或注意力加权形式是否会更好?

## Abstract

Agent harnesses increasingly determine the performance of language-model systems by deciding how models call tools, receive feedback, verify intermediate states, store memory, and revise solutions. Yet current test-time scaling analyses often parameterize this process by raw expenditure -- tokens, tool calls, operations, wall time, or cost -- which does not distinguish useful feedback from redundant or unstable interaction. We introduce Effective Feedback Compute (EFC), a trace-level scaling coordinate that credits feedback only when it is informative, valid, non-redundant, and retained for subsequent decisions, and we normalize it by task demand when comparing tasks with different feedback requirements. Across synthetic controllable tasks, executable code tasks, real benchmark traces, held-out splits, and a prospective validation batch, EFC-based coordinates consistently predict failure rates better than raw-compute baselines and a strong multivariate SAS baseline. In controlled scaling, raw tokens and tool calls explain limited variation ($R^2=0.33$ and $0.42$), SAS reaches $0.88$, while Oracle-EFC and Estimated-EFC reach $0.94$ and Oracle-EFC/$D_{\mathrm{task}}$ reaches $0.99$. Matched-budget interventions show that improving feedback quality raises success from $0.27$ to $0.90$ while raw cost and tool calls are fixed. On mixed real traces, NRS-EFC/$D_{\mathrm{task}}$ reaches $R^2=0.92$ while raw compute has near-zero or negative fit, and it remains the best predictor in a prospective holdout ($R^2=0.85$). These results suggest that harness scaling is governed less by how much computation is spent than by how efficiently raw budget is converted into durable, task-sufficient feedback.
