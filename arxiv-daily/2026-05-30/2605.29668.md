> **一句话总结：** 本文提出GRASP框架，通过回归感知的验证门控机制对LLM Agent的技能库进行有边界的编辑，每次候选技能只有在保留下过去正确行为的前提下才能被接受，在FHIR临床基准上将gpt-oss-120b从40.6%提升至88.8%，超过最强基线21.0个百分点，且技能库可跨模型和跨基准迁移。

> **TL;DR:** GRASP treats LLM agent self-improvement as a sequence of validated edits to a bounded skill library. Each candidate skill is accepted only if it produces a net improvement on a balanced held-out probe under a hard regression budget (no new failures allowed). Evaluated across five base models on two FHIR clinical benchmarks, GRASP lifts gpt-oss-120b from 40.6% to 88.8% on MedAgentBench (+21.0 over the strongest baseline), improves every base model by 17.2 to 40.3 points, and generalizes to non-clinical environments. Frozen libraries from stronger models transfer to weaker executors, producing a distillation-like effect without parameter updates.

## 核心贡献

- 提出将Agent自我改进建模为对有界技能库的序列化编辑问题，而非单调积累自然语言指导，从根本上解决了"修复一条轨迹却 silently 破坏另一条"的问题
- 设计回归感知的验证门控（acceptance gate）：每个候选技能必须在平衡探测集（balanced probe）上产生净改进且不引入新的回归错误才能被接受，包含硬回归预算约束 R(c) <= R_0
- 揭示关键发现：技能写作本身不如不用技能，真正的增益来自验证门控——没有门控的技能写作（K=1）准确率仅40.1%，与无技能基线持平
- 证明冻结的技能库具有跨模型迁移能力：强模型写的技能库可以提升弱执行器，且这种不对称性是GRASP独有的，其他五个基线均无法复现
- 技能库平均仅5条技能、5.6k tokens，远小于基线方法的34-53k tokens，实现了更高的推理效率

## Key Contributions

- Reframe agent self-improvement as bounded library editing rather than monotonic accumulation, solving the silent regression problem inherent in prior verbal-feedback methods
- Introduce a regression-aware acceptance gate that validates each candidate skill on a balanced held-out probe (previously-failing + previously-passing examples) under a hard regression budget
- Demonstrate that skill writing alone is no better than no skills (40.1% vs 40.6%); the gain comes from the validation mechanism, not from better skill generation
- Show that frozen skill libraries transfer across models with an asymmetric pattern: stronger writers improve weaker executors beyond self-training, but not vice versa
- Achieve compact libraries (5 skills, 5.6k tokens on average) that outperform unbounded memory baselines (34-53k tokens) at a fraction of the inference cost

## Methodology

GRASP operates in three stages per training batch:

**1. Failure-Driven Skill Proposal:**
- An LLM classifier assigns each failing trace a mechanism-specific open-vocabulary label (e.g., "date_filter_omitted" not "wrong_answer")
- Failures are grouped by label; the skill-writing model cycles from the most frequent failure mode to the least
- Each call to the skill writer provides failing traces from one group, a sample of passing traces, active skill summaries with provenance and effectiveness stats, and labels of other active failure modes to discourage regressions
- The writer returns an ADD, MODIFY, or REMOVE edit; ADD is blocked at capacity unless paired with a REMOVE

**2. Regression-Aware Selection:**
- A balanced probe of up to N/2 previously-failing and N/2 previously-passing examples is drawn from earlier batches in the epoch (out-of-sample relative to the proposing batch)
- The current library S is re-run on the probe for a fresh causal baseline (F0 fixes, R0 regressions)
- Each candidate S_c is evaluated on the same probe; acceptance requires:
  - Net improvement: F(c) - F0 - (R(c) - R0) > 0
  - Hard regression budget: R(c) <= R0
- If the winning candidate introduces any regression, a contrastive revision step narrows the trigger or adds a guard clause

**3. Skill Representation:**
- Skills are structured Markdown documents with YAML frontmatter, specifying trigger conditions, behavioral rules, optional verification steps, and contrastive examples
- The library starts empty and stays capacity-bounded, versioned, and reversible

**Training Protocol:** 5 epochs over the dev split, batch size 48, 36-example probe, 4 candidate edits per batch. Best checkpoint by validation accuracy is restored.

**Models:** gpt-oss-120b (open-source, single H200 GPU), DeepSeek V4 Flash, Gemini 3.1 Flash Lite, GPT-4.1, GPT-5.4 (low reasoning effort).

**Baselines:** No skills, Sequential Memory, Batch Memory, ExpeL, Evo-MedAgent, SkillX -- all using same agent, decoding settings, and training episodes.

## Results & Highlights

**MedAgentBench (Primary Benchmark):**
- gpt-oss-120b: 40.6% -> 88.8% (+48.2 over no skills, +21.0 over best baseline Evo-MedAgent at 67.8%)
- GPT-5.4: 45.1% -> 85.4% (+40.3)
- GPT-4.1: 45.5% -> 84.9% (+39.4)
- DeepSeek V4 Flash: 47.7% -> 70.0% (+22.3)
- Gemini 3.1 Flash Lite: 54.2% -> 71.4% (+17.2)
- OOD split: even larger margins, e.g., gpt-oss-120b OOD reaches 56.3% vs 31.3% for best baseline (+25.0)

**MedAgentBench-v2 (Secondary Benchmark):**
- More mixed results; GRASP leads in-domain on 3 of 5 models
- gpt-oss-120b: 61.1% -> 76.9% (+15.8, +9.1 over best baseline ExpeL)
- On GPT-4.1 and GPT-5.4, within noise of strongest baseline (dominant failure: exhausting 8-action budget)

**Learning Stability:**
- GRASP improves nearly monotonically and stabilizes above 85% from epoch 3
- Sequential and Batch memory regress below no-skills baseline as context accumulates
- Evo-MedAgent oscillates >20 points between adjacent checkpoints (std dev 14.1 vs GRASP's 5.8)

**Ablation Results (gpt-oss-120b, MedAgentBench):**
- Full GRASP: 88.8%
- Without failure grouping: 84.4% (-4.4)
- Without regression budget: 81.8% (-7.0)
- Fixes-only selection: 80.2% (-8.6, high variance +-13.7)
- Append-only: 80.2% (-8.6)
- Without acceptance gate (K=4): 63.5% (-25.3)
- Without acceptance gate (K=1): 40.1% (-48.7, matches no-skills baseline)
- Matched-compute control (proposer): 70.8%
- Matched-compute control (random): 67.2%
- Key takeaway: the acceptance gate is the dominant factor; compute alone does not explain the gain

**Cross-Model Transfer:**
- GPT-5.4 skills applied to gpt-oss-120b: OOD 77.8% vs self-trained 56.3% (+21.5), trading 12.8 points in-domain
- GPT-5.4 skills applied to Gemini: OOD 41.7% -> 71.1% (+29.4), in-domain 71.4% -> 76.6% (+5.2)
- Reverse direction (weaker writer to stronger executor) consistently worse than self-training
- This asymmetry is unique to GRASP; all five ungated baselines fail to reproduce it

**Non-Clinical Environments (gpt-oss-120b):**
- ALFWorld: +28.4 (structured, verifiable)
- WebShop: +20.6 (structured, verifiable)
- DBBench: +5.0 (partially structured)
- OS Interaction: +0.9 (open-ended, not significant)

## Critical Assessment

This is a well-executed paper that identifies a real and underappreciated problem (silent regression in self-improving agents) and proposes a clean, principled solution. The core insight -- that skill writing without validation is useless and the value comes entirely from the acceptance gate -- is both surprising and well-supported by the ablation evidence. The experimental design is thorough: five models, two primary benchmarks, five baselines, eight ablation variants, cross-model transfer, cross-benchmark transfer, and non-clinical generalization.

**Strengths:**

- The regression-aware gate is a conceptually simple but powerful idea, well-motivated by the failure mode of monotonic accumulation
- Exceptional ablation study that cleanly isolates each component's contribution, including matched-compute controls that rule out the "more compute" explanation
- Honest about mixed results on MedAgentBench-v2 and OS Interaction, without overclaiming
- Cross-model transfer analysis revealing the asymmetric writer-executor relationship is novel and practically important
- Compact libraries (5 skills, 5.6k tokens) are practically appealing for deployment
- Statistical reporting is unusually rigorous for the field
- The paper clearly delineates where GRASP works (structured, verifiable tasks) and where it does not (open-ended action spaces)

**Limitations:**

- The validation probe is the dominant cost: ~440 agent calls per batch, roughly 3x more than simple baselines. The "no parameter updates" framing understates this inference cost. The authors acknowledge this but the trade-off could be prohibitive in slow or expensive tool-interface settings
- Probe compute is not budget-matched across methods. While the matched-compute ablation addresses this within GRASP, it does not settle whether giving the same extra compute to baselines would narrow the gap
- Cross-model transfer is asymmetric and not fully model-agnostic. Weaker writers produce libraries that degrade stronger executors. The paper offers no principled method for detecting writer-executor mismatches beyond "use the strongest available writer"
- All clinical evaluation is on English-language FHIR data with curated patient records and exact-match scoring. Generalization to real clinical environments (incomplete records, free-text notes, non-standard codes, non-English) is entirely open
- Skills are policies tuned to benchmark performance, not medical expertise. Clinical deployment would require clinician review of each skill, prospective comparison, and live monitoring
- The mechanism helps most where tasks recur with predictable structure and verifiable signals. This is a meaningful but narrow slice of real-world agent deployment scenarios
- On MedAgentBench-v2, GPT-4.1 and GPT-5.4 show no benefit from GRASP (or any method), because the dominant failure mode (exhausting the 8-action budget on paginated-search tasks) is not addressed by any skill-based approach

## Relevance

本文对LLM Agent的自我改进机制研究有重要贡献。核心思路——将技能积累改为有界库编辑并通过回归测试门控——具有普遍适用性，不仅限于临床场景。对于关注Agent可靠性、持续学习和知识管理的研究者而言，这是一项值得关注的工作。特别是跨模型迁移中发现的不对称性（强模型指导弱模型）暗示了一种无需参数更新的知识蒸馏效应，这对实际部署中如何管理和复用技能库具有直接指导意义。论文明确界定了方法的适用边界（结构化、可验证的任务），这种诚实态度值得肯定。

## Questions for Further Exploration

- Would adding the acceptance gate to the baseline methods (ExpeL, Evo-MedAgent, SkillX) narrow the performance gap, or is the gate's effectiveness tied to GRASP's specific skill representation and failure-grouping mechanism?
- How does probe size scale with task complexity? Is there a theoretical or empirical minimum probe size below which the gate becomes unreliable?
- Can the asymmetric cross-model transfer be predicted a priori? For instance, can one measure "procedural capability gap" between writer and executor to forecast transfer quality?
- How do skills degrade over time in non-stationary environments where the underlying task distribution shifts?
- What is the overhead of human-in-the-loop skill review? Could clinician vetting of proposed skills further improve reliability, and at what cost in iteration speed?
- The paper shows skills are environment-interface-specific (FHIR convention matters). Could a two-layer skill architecture -- one layer for abstract procedural patterns, another for interface-specific implementations -- improve cross-environment transfer?
