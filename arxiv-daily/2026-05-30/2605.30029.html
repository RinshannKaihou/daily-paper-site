RAISE: RAG Design as an Architecture Search Problem
Zhen Chen et al. · 2026-05-30
RAG Architecture Search Benchmark
Key Metric
13 × 7 × 3
13 optimizers across 7 datasets with 3 seeds each. Every dataset had a different winner.
Why This Matters
No universal RAG config exists — task structure decides which pipeline modules matter, and simple methods often beat complex ones.
Critical Edge
Small-scale proxy evaluation (100 QA pairs, 30 trials) may not reflect production RAG optimization.

Methodology

  • Three-layer framework: parameterized RAG pipeline → evaluation layer → controller API. New algorithms plug in via a unified black-box interface.
  • Search space spans query rewriter (3 templates), chunker (256–2048 tokens, 0–192 overlap), hybrid retriever (BM25 + dense, α∈[0,1], top-k∈{1,3,5,10,20,50}), reranker, pruner, and generator.
  • Objective: maximize equal-weight average of ROUGE-L, METEOR, token-F1, and BLEU.
  • Setup: Qwen3-4B-Instruct, 30 evaluations × 3 random seeds, 100 QA proxy pairs per dataset.

Strengths

  • Clean NAS/HPO formulation for RAG tuning
  • Rigorous controls: matched budgets, multiple seeds
  • Comprehensive: 13 methods across 7 families
  • Honest mixed-result reporting
  • Extensible open-source framework

Limitations

  • 100 QA pairs, 30 trials — small proxy scale
  • Single model family (Qwen3-4B)
  • Lexical metrics miss semantic quality
  • Fixed discrete search space
  • No computational cost analysis

Questions for Further Exploration

  1. Does scaling evaluation budget to hundreds or thousands change optimizer rankings?
  2. Would larger LLMs (70B+) shift module importance and optimizer rankings?
  3. Would continuous search spaces change optimizer-environment interactions?
  4. Are rankings consistent on full production-scale datasets?
  5. Would LLM-as-judge or BERTScore rankings differ from lexical metrics?
  6. Can task features auto-predict the best optimizer (meta-learning)?
