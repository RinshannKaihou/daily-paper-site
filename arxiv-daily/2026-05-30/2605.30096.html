How Reliable Are AI Attackers Against a Fixed Vulnerable Target? A 400-Run Empirical Study of LLM Penetration Testing Consistency
Galip Tolga Erdem · 2026-05-28 · LLM Security Penetration Testing Empirical Study
Key Metric
85%
Gemini full exploitation rate vs qwen's 25% — a 60-point spread on the identical honeypot across 400 runs.
Why This Matters
Refusal-based safety mechanisms fail in practice: LLMs attack with zero pushback, giving defenders a ~30-second detection window and exposing signature-based defenses.
Critical Edge
Progression-forcing interventions bias trajectories — Markov chains describe orchestrator-shaped behavior, not raw model capabilities.

Methodology

Four models (Claude Sonnet 4, Gemini 2.5 Flash-Lite, GPT-4o-mini, qwen2.5-coder:14b) each ran 100 autonomous penetration tests against an identical Azure VM honeypot hosting three vulnerable services: OWASP Juice Shop (SQLi), OpenSSH with weak credentials, and vsftpd anonymous access.

A command-execute-observe loop drove each run, with three progression-forcing safeguards: refusal re-prompting, loop detection (identical 200-char prefix), and phase-stall forcing. Cloud models received ~5 exchanges of trimmed history; the local model retained full history.

Analysis included Wilson-score confidence intervals, first-order Markov chains over attack phases, strategy-diversity ratios, and Holm-Bonferroni-corrected pairwise tests (m=37) with Cliff's delta and Cohen's h effect sizes.

Strengths & Limitations

Strengths

  • First study at N=100 per model — genuine methodological advance
  • Transparent documentation of every decision and confound
  • Proper multiple-testing correction (Holm-Bonferroni, m=37) with effect sizes
  • Novel, model-specific failure mode taxonomy
  • Complete dataset and code released on Zenodo
  • Largest multi-provider zero-refusal observation to date

Limitations

  • Single target (OWASP Juice Shop + 2 services) limits generalizability
  • Training data contamination risk with well-known vulnerabilities
  • Claude confounded by temperature asymmetry (T=1.0 vs T=0.3) and API capacity event
  • Conversation history trimming asymmetry: cloud ~5 exchanges vs local full
  • 25-iteration cap artificially truncates GPT-4o-mini runs
  • Only "authorized pentest" framing tested; no control or jailbreak variants

Questions for Further Exploration

  2. How sensitive are results to prompt engineering variations?

  6. Would re-running Claude at corrected temperature significantly alter its exploitation rate and diversity?
