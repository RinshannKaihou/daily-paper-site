Skills Are Not Islands — measuring dependency & risk in agent skill supply chains

A first-of-its-kind empirical study that rebuilds the heterogeneous skill ↔ package ↔ service dependency graph across 1.43M agent skills and surfaces live malicious artifacts hiding behind transitive edges.

Changguo Jia · Tianqi Zhao · Runzhi He · Minghui Zhou · arxiv.org/abs/2607.01136 ↗
Key finding · 一句话总结

22.42% of agent skills carry a hidden package inventory, and 98.01% of axios exposure reaches root skills purely through transitive dependencies — the supply chain is real, it is deep, and it is already carrying known malware.

Why this matters · Relevance

It ports the SBOM playbook — evidence-calibrated graphs, BOM-IR, responsible disclosure — to natural-language-described agent skills, giving registries, platform maintainers and downstream consumers a first measurable handle on agent-skill supply-chain risk.

Critical edge · sharpest caveat

SDA is both artifact and measurement instrument. A ~5% F1 slack propagates into every 1.43M-scale count, and the static 2026-06-06 snapshot cannot see supply-chain churn — generalization across Claude / OpenAI / Vercel skills is asserted, not tested.

Methodology · 四阶段流水线

  1. Structure-aware skill parsing — split SKILL.md into YAML front matter + markdown body; recover root metadata, mine dependency clues via keywords + patterns.
  2. Evidence-calibrated dependency analysis — score every candidate against context: strong evidence → confirmed edge; weak evidence → annotation (preserved, not graphed). A typed scanner routes edges into package / skill / service channels.
  3. Incremental BOM construction — seed SkillBOM, recursively expand the package and skill channels (cycle-aware via visited-skill list), keep service records non-transitive, canonicalize duplicates.
  4. Schema-validated SkillBOM output — serialize, validate against the SkillBOM schema, preserve evidence + annotations; compatible with SPDX / CycloneDX via SBOM-IR.

Strengths

  • First mover at ecosystem scale — defines a new object of study (ASSC) before the agent ecosystem ossifies.
  • Concept + tool + benchmark + measurement in one paper, each layer supporting the next.
  • Cross-type graph — captures skill + package + service together, more realistic than single-ecosystem studies.
  • Quantified governance recommendations — typed manifests, dependency-cluster management, risk-warning, lockfile-style records.
  • Responsible disclosure — concrete cases reported (clawhub1, postmark-mcp).
  • Built on existing standards — SkillBOM extends SBOM-IR; SPDX / CycloneDX interop preserved.

Limitations

  • Static snapshot (single date) — no growth dynamics, no churn.
  • SDA is its own measurement instrument for RQ2 / RQ3; F1 = 0.95 leaves ~5% slack that propagates into large-scale counts.
  • Only one LLM baseline (DeepSeek-v4-pro) — generalization of the "LLM underperforms" claim is untested.
  • No detailed false-positive analysis of malicious-skill detection beyond the manually confirmed cases.
  • Strict package / skill / service taxonomy — skills that conflate channels are not modeled.
  • Threat model limited to known-malicious labels + regex patterns; novel obfuscation is out of scope.
  • Public registries only — enterprise / private skill ecosystems are excluded.

Questions for further exploration

  1. How does SDA degrade on non-SkillsMP formats — Claude marketplace, OpenAI Codex, Vercel skills.sh?
  2. What is the time-to-detection of newly published malicious skills? Could SDA run incrementally in CI?
  3. Does Gini 0.925 / 0.944 reflect genuine workflow hubs, or template proliferation?
  4. How stable is the 22.42% hidden-package inventory when upstream skills in a cluster are deprecated?
  5. Could the typed-evidence calibration generalize to agent prompts (tool deps hidden in prose) and agent traces?
  6. Would an automatic cluster-resolution with conflict blocking (à la npm v7) suffice, or do skills need a "cluster manifest" spec?
  7. What is the false-positive rate of the regex security families on benign skills that legitimately shell out?
  8. How fast is the supply chain evolving — re-running on a later SkillsMP snapshot, how much churn?
  9. Could a signed SkillBOM (cryptographic provenance) close the governance gap without breaking copy-paste sharing?
  10. Would a graph-aware LLM (LLM + SDA's typed scanner as a tool) close the gap on transitive skill dependencies?
