LogbQuant: Quantizing Language Models in Logarithmic Space

Jeremias Bohn, Tizian Dippold, Mahdi Koubaa, Elias R. Wahl, Georg Groh TUM · School of CIT
log-domain quantization 4-bit PTQ GPTQ-compatible FLUTE kernel edge / consumer GPU Llama · Qwen3
Key finding
A log-domain 4-bit codebook with closed-form base & shift keeps LLM accuracy — where tensor-wise linear quantization collapses to random guessing.
Key metric
10.10GB
Qwen3-14B at 4-bit, batch 1 — down from 28.88 GB BF16, a 65% memory cut. Fits inside a 12 GB consumer GPU (e.g. RTX 5070). Linear 4-bit, by contrast, drops Qwen3-14B MMLU from 77.15 → 24.68 (≈ random).
Why this matters

A non-uniform codebook that retains 4-bit tensor-wise PTQ accuracy without per-channel or per-group scaling — directly valuable for private / on-device inference on consumer GPUs, where calibration cost and deploy complexity are the bottlenecks.

Critical edge

The headline win is over the weakest baseline in modern PTQ.

  • No comparison to GPTQ-g128, AWQ, OmniQuant, QuaRot, FlatQuant.
  • Batch-128 throughput regresses 0.56–0.84× (LUT bank contention).
  • Per-tensor real-valued b blocks int8 weight×activation multiply.

Methodology

Redefine the quantization codebook in logb space. For weight x, index q ∈ ℤ, bitwidth n, qmax = 2n-1−1: 

Quant(b,s,x)  =  sign(x)·(⌊log_b|x|⌉ + s − q_max)
Dequant(b,s,q)=  sign(q)·b|q|−s      (q≠0)

Per-tensor b, s (closed form)

From the effective positive range [wmin, wmax]: b = (wmax/wmin)1/qmax, s = qmax − ⌊logb wmax⌋. No non-convex search.

Energy-based pruning

Set wmin as the smallest threshold where discarded weights contribute ≤ ε of total signal energy. ε = 4×10⁻³ at 4-bit, 1×10⁻⁶ at 8-bit.

Pipeline

  • Calibration: GPTQ on WikiText-2 (128 × 2048 tokens, seed 0) — only the Quant/Dequant functions are swapped.
  • Inference kernel: FLUTE LUT decoder, group size 64 ≈ tensor-wise granularity.
  • Scope: weight-only, symmetric log codebook, embedding + LM head stay BF16.

Throughput & memory at 4-bit

Speedup 1.01×–1.51× at batch 1–32; 0.56×–0.84× regression at batch 128. Memory cut 50–65% at batch 1, 14–59% at batch 128. At 8-bit, linear saturates the base models and is the recommended choice.

Strengths

  • Self-calibrating closed-form b, s — no per-tensor optimization.
  • Energy-based pruning adapts the effective range to each layer's distribution.
  • Drop-in with the GPTQ PTQ pipeline; only Quant/Dequant swap.
  • Shows that coarse-grained PTQ need not fail at 4-bit if the codebook shape is right.
  • Full per-model tables — every claim is verifiable.

Limitations

  • Weakest baseline chosen — no GPTQ-g128, AWQ, OmniQuant, QuaRot, FlatQuant.
  • Batch-128 throughput regressions from FLUTE LUT bank contention.
  • No weight×activation quantization path; per-tensor real b blocks int8 mul.
  • Global ε; per-layer search or learned thresholds could squeeze more accuracy.
  • No code/model release mentioned.
  • Only dense decoder LMs; MoE and encoder-decoder out of scope.

Questions for Further Exploration

  1. How does LogbQuant stack up against GPTQ with group size 128 on the same 8 models?
  2. Does the method compose with QuaRot rotations or SmoothQuant/KIVI activation quantization?
  3. Can per-layer ε or a learned wmin close the remaining 4–6 pp gap to BF16?
  4. Why does LUT decode regress at batch 128 — and would a dedicated CUDA logb kernel fix it?
  5. Could the exponent b be snapped to a small candidate set (e.g. 8 values) to enable int8 weight×activation without losing the log advantage?
  6. How sensitive is the 4-bit win to calibration data and seed — does it survive RedPajama / C4 with ≥ 3 seeds?
  7. Does LogbQuant transfer to MoE and encoder-decoder models with different weight distributions?
