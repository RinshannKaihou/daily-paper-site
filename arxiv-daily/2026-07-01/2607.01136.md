> **一句话总结：** 该工作提出 **Agent Skill Supply Chains (ASSC)** 概念与自动化分析工具 **SkillDepAnalyzer (SDA)**，从 143 万条 SkillsMP 真实智能体技能中以自然语言证据驱动地重建混合 skill–package–service 依赖图，揭示技能元数据"激活就绪但治理薄弱"、依赖复用集中度（Gini 0.925/0.944，高于 npm）、递归复用导致 22.42% 技能携带隐藏包库存（npm/PyPI 71–73% 包仅经传递获得），以及 30.41% 根技能参与依赖簇，并通过 SkillBOM 与 `risk-warning`/lockfile 等治理建议披露仍在供应链中存活的已知恶意技能。

> **TL;DR:** *Skills Are Not Islands* formalizes **Agent Skill Supply Chains (ASSCs)** — heterogeneous dependency graphs mixing LLM agent skills, traditional packages, and external services — and ships **SkillDepAnalyzer (SDA)** to mine natural-language dependency evidence from skill files. On the human-annotated SKILL-DEP benchmark (500 single-layer + 100 multi-layer skills), SDA achieves **F1 = 0.95** on dependency extraction and **1.00 accuracy** on metadata, beating all of Syft, ScanCode, ORT, Microsoft sbom-tool, Cdxgen, and a DeepSeek-v4-pro LLM baseline. Applied to **1.43M** SkillsMP skills, the study surfaces four patterns — governance-poor metadata, multi-channel reuse with Gini 0.925 (skills) / 0.944 (packages), recursive reuse that hides 22.42% of package inventories, and 30.41% of roots in dependency clusters — and uses the recovered graphs to uncover still-live malicious skills (`clawhub1`), vulnerable packages (`axios` 98.01% transitive, `Nx` 65.43%), and malicious MCP services (`postmark-mcp`, 93.10% transitive). Recommendations: typed dependency manifests, first-class dependency-cluster management, an `npm audit`-style risk command, and lockfile-like skill records.

## 核心贡献

- 提出 **Agent Skill Supply Chains (ASSC)**，将 LLM 智能体技能的"技能-包-服务"三层异构依赖形式化为有向图，补足传统 SBOM 在智能体生态中的结构性缺口。
- 设计 **SkillDepAnalyzer (SDA)**：结构化解析 YAML front matter、按类型校准依赖证据（高置信入图，低置信作 annotation）、递归构建依赖图、并以 **SkillBOM**（基于 SBOM-IR 的技能定向表示，与 SPDX/CycloneDX 兼容）序列化输出。
- 构建 **SKILL-DEP** 基准（500 单层 + 100 多层）并由两名软件工程研究者按冻结指南人工标注，可重复且分层（包/技能/服务三类分别计分）。
- 在 **1,434,046** 条 SkillsMP GitHub 来源技能（2026-06-06 快照）上进行首次 ASSC 大规模测量，归纳四项结构性发现。
- 通过依赖图审计识别仍在 ASSC 中可触达的已知恶意技能（如 `clawhub1`/`clawbhub` 副本于 `Demerzels-lab/elsamultiskillagent`）、未固定版本的脆弱包（`axios`, `Nx`）与恶意 MCP 服务（`postmark-mcp`），并向开发者负责任披露。
- 给出面向技能基础设施维护者与技能作者的可操作治理建议：类型化多通道依赖清单、一级"技能依赖簇"管理、`risk-warning` 审计命令与 lockfile 风格的固定记录。

## Key Contributions

- **Conceptual framing:** Defines *Agent Skill Supply Chains (ASSCs)* as directed mixed skill–package–service dependency graphs, generalizing SBOM to LLM agent ecosystems.
- **Tool:** *SkillDepAnalyzer (SDA)* — structure-aware parsing → evidence-calibrated dependency analysis → incremental BOM construction → schema-validated SkillBOM output. Cycle-aware, annotation-preserving, evidence-traceable.
- **SkillBOM:** A skill-oriented BOM built on SBOM-IR; assigns UUIDs to skills, introduces `USES_SERVICE`-style typed relations, and remains compatible with SPDX/CycloneDX so existing SBOM toolchains can be reused.
- **Benchmark:** *SKILL-DEP* — 500 stratified single-layer skills (436 package + 708 skill + 442 service dependencies; 99.55% front-matter, 1.40% dependency-like fields) and 100 multi-layer graphs (80 depth-3, 18 depth-4, 2 depth-5), adjudicated from a 50-skill pilot.
- **Empirical study:** First large-scale measurement on 1.43M skills — four structural patterns + security signal propagation across three channels.
- **Responsible disclosure:** Reports `clawhub1`/`clawbhub` copies and unpinned `axios`/`Nx` issues to developers.
- **Actionable governance:** Typed manifests, first-class dependency-cluster management, `risk-warning` audit CLI, and lockfile-style skill records.

## Methodology

SDA is a four-stage pipeline that converts a skill document into a typed SkillBOM:

1. **Structure-aware skill parsing** — Split `SKILL.md` into YAML front matter and markdown body. Recover root-skill metadata (name, version, description, license) from front matter; if absent, fall back to body. Mine *dependency clues* via metadata keyword matching + dependency pattern matching across front matter, commands, and natural-language mentions.
2. **Evidence-calibrated dependency analysis** — Each candidate is scored against its surrounding context. Strong evidence (installation commands, manifest entries) → confirmed edge. Weak evidence → retained as *annotation* (preserves traceability without polluting the graph). A *typed scanner* routes confirmed edges into three channels: **package** (installation commands, manifests, Docker snippets), **skill** (multi-field identity: name + repo + path + owner + stars), and **service** (MCP/API/webhook cues).
3. **Incremental BOM construction** — Seed SkillBOM from root skill + first-layer dependencies. Recursively expand along the package channel (prefer local running environment, fall back to registries like npm/PyPI) and skill channel (import each matched skill's SkillBOM; cycle-aware via visited-skill list). Service records describe external services but are *not* transitively expanded. A merge step canonicalizes duplicates and preserves relationship evidence.
4. **Schema-validated SkillBOM output** — Serialize the graph into SkillBOM, validate against the SkillBOM schema, and preserve evidence + annotations. SkillBOM extends SBOM-IR with skill-specific component types and typed relations (e.g., `USES_SERVICE`), so SPDX/CycloneDX consumers can ingest it.

**Evaluation (RQ1):** SDA is evaluated against SKILL-DEP. Baselines include five package-centric SBOM generators (Syft, Cdxgen, ScanCode, ORT, Microsoft sbom-tool) and a DeepSeek-v4-pro LLM baseline (T=0, fixed prompt, JSON-only output). Metrics: precision/recall/F1 for dependency extraction (keyed by source skill × target kind × canonicalized name × relationship type) and field-level accuracy for metadata.

**Measurement (RQ2):** Apply SDA to a 2026-06-06 snapshot of SkillsMP (1,640,440 listed skills; 1,434,046 GitHub-backed, ~87.4% retrieved). Compute dependency channel counts, normalized Gini (concentration), amplification factors (transitive/direct), hidden-package shares (npm/PyPI), and cluster/cycle statistics.

**Security (RQ3):** Collect security-relevant indicators from two complementary sources: (i) public threat reports (Koi, Snyk, Antiy, GitHub Advisory Database, Huntress, Model Context Protocol, OWASP) for known-malicious skills, packages, services; and (ii) regex-based families covering remote payload execution, dangerous code patterns, prompt injection, credential exfiltration, persistence/backdoor commands, secret exposure (skills), and code-repository/issue-tracker authority, local-resource/browser authority, collaboration/messaging authority, data-backend authority, mechanism-only MCP attack patterns (services). Scan skills + all generated SkillBOMs and observe propagation via dependency-only paths. Manually confirm key cases.

## Results & Highlights

**RQ1 — SDA accuracy (single-layer):**

| Method | Overall F1 | Package F1 | Skill F1 | Service F1 |
|---|---|---|---|---|
| Cdxgen | 0.19 | 0.40 | — | — |
| Syft | 0.20 | 0.43 | — | — |
| ScanCode | 0.25 | 0.49 | — | — |
| ORT | 0.18 | 0.45 | — | — |
| Microsoft sbom-tool | 0.24 | 0.58 | — | — |
| DeepSeek (LLM) | 0.52 | 0.52 | 0.53 | 0.49 |
| **SDA** | **0.95** | **0.93** | **0.96** | **0.94** |

- Metadata: SDA = 1.00 / 1.00 / 1.00 / 1.00 (name/repo/path/license) vs DeepSeek 1.00 / 0.99 / 0.99 / 1.00.
- Multi-layer comprehensiveness: **Precision 0.98, Recall 0.93, F1 0.95** (no baseline supports this).

**RQ2 — ASSC characteristics (1.43M skills):**

- **Activation-ready but governance-poor:** 99.55% of skills have front matter (name 99.49%, description 99.52%); but only **11.25%** declare license, **20.12%** declare version, and only **1.40%** carry dependency-like fields (yet >30% actually have dependencies). **58.73%** of effective skill names collide with another record; 88.76% of collisions span multiple repos and 99.31% have different SHA-256 hashes. **25.88%** share normalized descriptions.
- **Multi-channel, concentrated reuse:** Dependency channels among roots — direct skill **8.92%** (127,891), direct package **15.48%** (221,925), direct service **22.25%** (319,013); any of the three **36.60%**; all three **0.77%**. Normalized Gini — skills **0.925**, packages **0.944** (npm reference 0.87); top-20 skill targets capture 14.12% of all skill edges, top-20 packages capture ~8.8% of package edges.
- **Dependency expansion + hidden package inventory:** Amplification p50/p90/p99/max — Total 0.5 / 23.0 / 130.5 / **979.0×**; Skill 0 / 4.1 / 34.0 / 347.0; Package 0 / 47.0 / 350.0 / **1,754.0**; Service 0 / 5.0 / 63.5 / 194.0. **22.42%** of dependency-bearing skills gain packages only through reused skills; **71.87%** of npm exposures and **73.33%** of PyPI exposures are inherited through skill reuse. Extreme case: `windows-95-web-designer` declares zero packages but pulls in **1,754 packages / 1,938 components** (645× amplification) as a workflow orchestrator.
- **Dependency clusters:** **30.41%** of dependency-bearing roots include a cycle; 30.03% have convergent downstream nodes. Clusters are workflow-coupled — e.g., `OpenTideHQ/AgentTide`'s `microsoft-defender-endpoint` → `detection-engineering` ↔ `microsoft-sentinel` ↔ `threat-hunting`; `claude-dev-suite`'s `bitcoin-psbt` suite organizes 140 mutually referencing skills around Bitcoin custody knowledge.

**RQ3 — Security signal propagation:**

- *Skill-level:* Publicly reported malicious labels reach 194 roots (13.40% dependency-only); regex families — remote payload execution **6,097** roots (63.77% dep-only), dangerous code patterns **3,342** (78.43%), prompt injection **1,413** (64.61%), credential exfiltration **1,189** (68.29%), persistence/backdoor **454** (60.13%), secret exposure **577** (65.34%). Concrete case: 44/47 roots reaching `github-repo-management` (with remote-execution patterns) do so purely through dependency.
- *Package-level:* `axios` reaches 3,413 roots (98.01% dep-only); `Nx` 81 roots (65.43%); `Clinejection` payload 38 roots (44.74%), entry 2 (0%). Caused by unpinned install instructions inheriting whatever version the local environment serves.
- *Service-level:* Reported vulnerable MCP services reach 29 roots (93.10% dep-only). Broader authority surfaces — code-repository/issue-tracker (3,566 roots, 72.91% dep-only), local-resource/browser (882, 73.47%), data-backend (218, 72.02%), collaboration/messaging (303, 53.14%), mechanism-only MCP attack patterns (568, 66.55%). Example: `activator-authoring-cli` inherits GitHub MCP authority through a two-hop dependency via `check-updates` without ever naming the GitHub MCP server.

## Critical Assessment

- **Novelty:** High. SBOM supply-chain transparency is mature, but applying it to *natural-language-described agent skills* and treating skills as first-class artifacts with cross-type (skill + package + service) dependencies is a fresh framing. No comparable 1M-scale empirical study of skill supply chains exists in the public record.
- **Rigor:** Strong. (i) Held-out human-annotated SKILL-DEP with explicit guidelines and inter-annotator agreement (1,581/1,583 raw records matched, 11 disputed adjudicated). (ii) Comprehensive baselines — five SBOM generators + an LLM baseline (DeepSeek-v4-pro) at T=0. (iii) Stratified sampling for benchmark construction. (iv) Cycle-aware, annotation-preserving pipeline with evidence tracing. (v) Threats-to-validity section explicitly addresses construct/internal/external threats.
- **Reproducibility:** Authors publish SDA, the SKILL-DEP benchmark, and the SkillBOM schema (open dataset is implied by the SkillsMP + GitHub-snapshot retrieval workflow, though an artifact/badge is not explicitly mentioned in the body). Concerns: (a) exact reproduction depends on the 2026-06-06 SkillsMP snapshot, which will drift; (b) the natural-language scoring rules and confidence thresholds are critical to rerun and should be released in code; (c) the LLM baseline depends on a specific model version (DeepSeek-v4-pro) whose behavior may shift.
- **Significance:** High and timely. The agent-skill ecosystem is growing ninefold in three months (1.43M skills). The findings — 22.42% of skills with hidden package inventory, 71–73% of npm/PyPI exposures transitive through skills, 98% of `axios` exposure via transitive paths, and `clawhub1`/`postmark-mcp` still reachable through registries — are operationally consequential for skill registry operators, agent-platform maintainers, and downstream consumers.
- **Red flags / concerns:**
  - **Internal validity:** SDA is *both* the artifact *and* the measurement instrument for RQ2/RQ3. The authors mitigate this with the independently annotated benchmark and manual case confirmation, but residual extraction errors could propagate into the large-scale numbers (e.g., the 98.01% `axios` transitive figure could over- or under-state reachability if SDA's skill-matching has systematic FP/FN).
  - **Recall on rare dependencies:** The paper does not stratify performance on long-tail or adversarial skill formats; the F1=0.95 headline may not hold for obfuscated or unconventional skills.
  - **Static snapshot:** 2026-06-06 only — generalization across registries (Claude skills, OpenAI Codex skills, Vercel Labs `skills.sh`) and over time is asserted but not tested.
  - **No negative results on remediation:** Did registries act on the responsible-disclosure reports? No follow-up evidence presented.
  - **"All three channels" = 0.77%:** Strong typing assumption; some skills may bundle dependencies across channels in ways that break the strict package/skill/service taxonomy.
  - **DeepSeek baseline scope:** Only one LLM is tested. Generalizing the "LLM-based baselines underperform" claim would require multiple models (GPT-4o, Claude, etc.).
- **Overall:** A well-scoped, well-benchmarked, well-disclosed empirical study whose claims are largely backed by the numbers in the paper. The methodology (typed evidence calibration + cycle-aware incremental construction) is the most reusable artifact beyond the findings themselves.

## Strengths

- **First mover at ecosystem scale** — defines an important new object of study (ASSC) before the agent ecosystem ossifies.
- **Concept + tool + benchmark + measurement** — all four layers in one paper, with each supporting the next.
- **Cross-type graph analysis** — captures skill + package + service together, more realistic than single-ecosystem dependency studies.
- **Concrete, quantified governance recommendations** — typed manifests, dependency-cluster management, `risk-warning` audit, lockfile-style records — all directly actionable for Vercel Labs Skills CLI, SkillsMP, `claude-dev-suite`, etc.
- **Responsible disclosure** — appropriate for a security paper; concrete cases reported (`clawhub1`, `clawbhub`, `postmark-mcp`).
- **Built on existing standards** — SkillBOM extends SBOM-IR rather than inventing a new format, preserving SPDX/CycloneDX interop.

## Limitations

- Static snapshot (single date) of SkillsMP — does not capture growth dynamics or churn in supply-chain composition.
- Internal-validity concern: SDA is its own measurement instrument for RQ2/RQ3; F1=0.95 leaves ~5% slack that propagates into large-scale counts.
- DeepSeek-v4-pro is the only LLM baseline; claims about LLM-based extractors should be replicated across models.
- No detailed false-positive analysis of malicious-skill detection beyond the manually confirmed cases (`clawhub1`, etc.).
- The "package/skill/service" taxonomy is strict — skills that conflate channels (e.g., a skill file that itself acts as a service description) are not modeled.
- Threat model is limited to known-malicious labels and regex patterns; novel obfuscation or semantic-only attacks are not addressed.
- No user-facing performance numbers (latency, cost of recursive SkillBOM construction at 1.43M scale).
- Public-registry only — enterprise/private skill ecosystems are excluded from external validity.

## Relevance

高度匹配用户关注点。该工作直面 LLM 智能体技能供应链中的"隐式依赖—隐式风险"问题，把 SBOM、依赖图谱与可靠性审计方法迁移到智能体生态，是 *Self-Evolving Agents / ML Reliability foundations* 主题下的代表性实证工作。其方法论（证据校准、增量 BOM、责任披露）也可借鉴用于评估**智能体训练数据/技能注册中心**的可靠性与可复现性。

## Questions for Further Exploration

- How does SDA's precision/recall degrade on *non-SkillsMP* skill formats (Claude marketplace entries, OpenAI Codex skills, Vercel Labs `skills.sh` skills)? Cross-registry generalization is asserted but untested.
- What is the *time-to-detection* of newly published malicious skills? Could SDA be run incrementally in CI to flag skills at submission time?
- Does the high concentration (Gini 0.925 / 0.944) reflect *genuine* workflow hubs, or are some "popular" skills simply copy-paste templates? Distinguishing functional reuse from template proliferation would sharpen governance priorities.
- How stable is the hidden-package inventory (22.42% of skills gaining packages only transitively) when the *upstream* skills in a cluster are removed or deprecated?
- Could SDA's typed-evidence calibration be applied to agent *prompts* (which often hide tool dependencies in prose) and to *agent traces* (which record what was actually called)?
- The "dependency cluster" governance proposal echoes npm's peer-dependency lessons (npm 4–6) — would *automatic* cluster-resolution with conflict blocking (à la npm v7) be sufficient, or do skills need a stronger "cluster manifest" specification?
- What is the false-positive rate of the regex-based security families (e.g., remote payload execution, prompt injection) on benign skills that legitimately use shell commands or external services?
- How do the SkillBOM graphs change when re-running on a *later* SkillsMP snapshot — i.e., how fast is the supply chain evolving?
- Could a *signed* SkillBOM (cryptographic provenance) close the "governance-poor metadata" gap without breaking copy-paste skill sharing?
- The DeepSeek baseline underperforms on transitive skill dependencies via path/workflow cues — would a *graph-aware* LLM (LLM + SDA's typed scanner as a tool) close the gap without losing semantic flexibility?
