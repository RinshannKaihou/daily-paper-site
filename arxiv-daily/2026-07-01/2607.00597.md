> **一句话总结：** PaperPilot 把多轮科学文献检索重新定义为"工作流归纳"——以 17 个强类型算子构建可执行 DAG,通过对受控损坏工作流的偏好优化训练,使 9B 模型在多轮场景下将 Hit@5 从 58.0 提升到 77.0、MRR 从 47.5 提升到 59.4,并将工作流执行错误率从 9.5% 降至 0%。

> **TL;DR:** PaperPilot replaces opaque chain-of-thought tool-use with **explicit, editable DAG-structured search workflows** composed from a 17-operator typed toolkit (sourcing / filtering / scoring / reranking / keyword generation / evidence extraction / graph construction). It is trained with two-phase supervision — SFT on 5,540 teacher-generated workflows over 2,723 anchor–query cases across five search directions, then IPO-style DPO over 1,733 hard chosen-vs-corrupted pairs — yielding a 9B model that lifts Hit@5 from 58.0 → 77.0 and MRR from 47.5 → 59.4 versus a Qwen3.5-9B toolset baseline under multi-turn interaction, while dropping workflow execution errors from 9.5% to **0%** and matching or beating GPT-5.4 + Toolset on cost-effectiveness (665 vs 7 successful cases per dollar on the workflow-generation task). A 6-participant human study reinforces the metrics: 74.7% success rate vs 32.0% for GPT-5.4 and 8.0% for OpenAI DeepResearch.

## 核心贡献

- 将多轮科学文献检索重新形式化为**工作流归纳(workflow induction)**:每次检索显式生成一个由强类型算子组成的有向无环图(DAG),而非调用黑盒的 tool-use LLM。
- 设计 **PaperPilot-Toolset**:17 个论文检索算子(关键词检索、引用扩展、合并/去重/过滤、打分/截断/重排、LLM 关键词、NLI 过滤、细粒度阅读、证据抽取、关系图构建),每个算子具有类型化输入/输出,DAG 需满足类型一致。
- 提出**两阶段训练**:SFT 在 5,540 个高质量工作流样本上模仿教师模型,然后通过**受控工作流损坏**(invalid references / missing inputs / wrong operator / dropped nodes / shifted filters / vague NLI axes)构造偏好对,用 IPO 风格 DPO 进行偏好优化。
- 在多轮交互中将用户澄清同时映射到 *查询改写* 与 *DAG 结构编辑*;支持 ask-clarification / modify-workflow / finalize 三类动作,可按轮次逐步对齐用户意图。
- 在 hold-out 检索基准、workflow-level 评估(生成 + 编辑)和 6 人人因实验上全面验证,获得显著的检索质量、workflow 稳定性与成本-效率优势。

## Key Contributions

- **Workflow-induction formulation** of multi-turn scientific literature search: cast retrieval as building a typed-DAG workflow conditioned on an evolving dialogue, instead of a fixed retrieval pipeline or implicit language-only reasoning.
- **17-operator PaperPilot-Toolset** with typed input/output signatures covering sourcing (keyword_search, citation_expand, web_resolve), combine/filter (union, dedupe, filter), scoring (score), cutting (top_k, above), rerank (llm_rerank, nli_filter, fine_read), LLM (llm_keywords, llm_keywords_from), and output (extract_evidence, pairwise_nli, build_graph).
- **Two-phase supervision pipeline**:
  - SFT on 5,540 teacher-generated workflows from 2,723 anchor-query training cases (5 directions: predecessor / successor / sibling / benchmark / survey), 3 epochs, lr 2e-4, seq 14,336, LoRA on attention + MLP.
  - IPO-style DPO on 1,733 hard chosen-vs-corrupted pairs, 3 epochs, lr 3e-5, β=0.2, seq 16,384.
- **Multi-turn interactive refinement**: G_{t+1} = Refine(G_t, f_t, H_{t+1}) — user feedback drives both query updates *and* DAG edits (add / modify / remove node, adjust filters / keywords / NLI axes).
- **Hold-out benchmark** with 6–15 hidden gold papers per case, fixed Qwen3.5-397B-A17B user simulator with leakage control (deterministic string-match + Qwen3.5-9B leak checker + regeneration / replacement).
- **Empirical wins** on retrieval quality, workflow structural stability, and cost-effectiveness (see Results).

## Methodology

### Problem formulation
- **Inputs**: anchor paper p₀ (with cached Semantic Scholar metadata), user query q, evolving interaction history H_t.
- **Action space**: ask clarification, induce / edit DAG, execute workflow, finalize ranked paper list P̂.
- **Transition**: s_{t+1} = T(s_t, a_t, f_t) — updates candidate paper set and history.
- **Objective**: maximize expected cumulative utility — paper relevance, alignment with feedback, intent coverage, and interaction efficiency.

### Workflow representation
- Workflow = DAG G = (V, E); each node v_i = (o_i, θ_i) is an instantiated operator with parameters; each edge encodes typed data flow (PaperSet, Scored, Evidenced, Graph, Keywords).
- **Type consistency** is required: the output type of every predecessor must match the input type expected by every successor.
- Terminal node produces a ranked list, evidenced list, or a relation graph depending on the query.
- A **fixed-workflow baseline** (PaperPilot-Workflow) using the same toolset but a static pipeline (keyword → citation expand → union / dedupe → generic score → llm_rerank → extract_evidence) isolates the value of *adaptive* workflow induction.

### Rollout
- Per turn: G_t = Induce(q, p₀, P_t, H_t); (P_{t+1}, y_t) = Execute(G_t, P_t).
- Intermediate outputs are cached across turns, so user edits only touch the nodes / parameters that matter.
- A user turn can produce a clarification question, a ranked list, an evidence snippet, or a workflow edit.

### Training data
- 2,723 anchor-query training cases × 5 search directions → teacher-model trajectories.
- SFT supervision: retain turns where the gold paper appears in top-5 *and* direction-specific success conditions hold → **5,540 workflow supervision examples**.
- DPO supervision: pair each successful workflow with deliberately corrupted versions (structural validity errors, search-quality errors) → filter out easy pairs → **1,733 hard chosen–rejected pairs**.

### Two-phase training
- Phase 1 — SFT with next-token cross-entropy on the assistant response in chat-completion format.
- Phase 2 — IPO-style DPO (preferred for stable preference signal on long-context workflow generation) over the hard pairs.
- Both phases use LoRA; sequence length grows from 14,336 (SFT) to 16,384 (DPO) to fit long DAGs.

### Evaluation protocol
- **Single-turn**: agent searches directly from the original query.
- **Multi-turn**: clarifier LLM asks questions with 3–5 options, simulator (Qwen3.5-397B-A17B) responds, leakage-control enforces no gold-paper title / author leakage, then retrieval runs once with enriched query.
- Max clarification turns: 4–5 depending on experiment.
- Top-50 paper list scored against hidden gold set; failed executions get zero.

### Headline gains (PaperPilot-9B vs base Qwen3.5-9B Toolset, multi-turn)
- Hit@5: 58.0 → **77.0** (+19.0; +33% relative)
- Recall@50: 34.8 → **40.0** (+5.2)
- MRR: 47.5 → **59.4** (+11.9)
- nDCG@10: 26.8 → **32.5** (+5.7)
- Workflow execution error: 9.5% → **0%**
- PaperPilot-9B also surpasses the single-turn GPT-5.4 Web Search baseline on Hit@5/10/15 and MRR despite being a much smaller model at ~8× lower cost.

PaperPilot-9B has the strongest cost-effectiveness frontier (Rel. Eff. 3.26 at $0.00108/case) on workflow generation.

### Workflow-level refinement
- PaperPilot-9B achieves the highest final full-workflow cosine (**0.9469**) and query-level cosine (**0.8777**).
- Strongest local-edit consistency under add-node, modify-node, and remove-node edits; largest gain over base Qwen3.5-9B on remove-node.

### Action distribution across turns
- Turn 1–3: dominated by *Ask user* (clarification gathering).
- Turn 4–7: shifts to *Modify workflow* (refinement).
- Turn 8+: dominated by *Finalize*.

### Sensitivity on candidate-pool size K
- Sweet spot K = 8–10; larger pools add distractors. R@50 drops 0.411 (K=8) → 0.374 (K=20); Hit@5 0.705 (K=10) → 0.660 (K=20).

## Critical Assessment

- **Novelty.** Reframing literature search as *explicit, editable DAG workflow induction* is a genuine step beyond opaque CoT-style tool-use agents; treating user feedback as a *workflow edit instruction* (not just query text) is a clean conceptual move. The "controlled workflow corruptions" preference data construction is also a clever way to inject robustness supervision without expensive human relabeling.
- **Rigor.**
  - Standard IR metrics with a hold-out set, fixed user simulator, and explicit leakage-control pipeline (deterministic string match + LLM leak checker + regeneration / replacement) — above-average rigor for this sub-field.
  - Multi-axis evaluation (retrieval, workflow generation, workflow refinement, human study, cost) makes the claims harder to dismiss.
  - The 0% execution error is striking but plausible: the operator sandbox rejects invalid plans, and training explicitly corrupts workflows of the same failure modes the base model exhibits under multi-turn.
- **Reproducibility.**
  - The benchmark, simulator, leakage-control, and baselines are described in appendices C.1–C.4; hyperparameters are given in C.6.
  - No public code/dataset URL is mentioned in the abstract, but a project website is referenced.
  - Reliance on a fixed teacher (Qwen3.5-397B-A17B) and on Semantic Scholar metadata ties results to a moving external service.
- **Significance.** Useful blueprint for downstream agent-for-science systems; the explicit-DAG pattern is transferable to other tool-use agents (web agents, code agents) where user feedback should drive structural edits rather than query rewording.
- **Red flags.**
  - The benchmark is CS-only and uses curated 6–15 gold papers per case; real-world literature search has noisier relevance signals.
  - LLM user simulator is convenient but cannot fully substitute for humans — even with leakage control, simulator bias can inflate apparent gains.
  - The user simulator has access to hidden gold metadata to provide preference-aware feedback; this is a strong-but-leaky assumption about real users (who often know *what* they want but not *exactly which paper*).
  - PaperPilot-9B still trails the GPT-5.4 + Toolset upper bound on raw retrieval quality (84.0 vs 77.0 Hit@5) — the 9B story is *cost-effectiveness*, not absolute SOTA.
  - Sensitivity analysis shows diminishing returns beyond K=10, hinting that the operator sandbox's filtering is a major contributor and may not generalize to noisier / larger corpora.

## Strengths

- Explicit, inspectable, *editable* workflow representation — rare in agentic-IR literature and a transferable design idea.
- Strong empirical gains on standard IR metrics (Hit@5 +19 absolute) and a near-total elimination of workflow execution errors.
- Two-phase training pipeline (imitation + corruption-based DPO) is principled and reusable for other tool-use agents.
- Comprehensive evaluation: retrieval, workflow generation, workflow editing, sensitivity, cost, and human study.
- Excellent cost profile: ~80× cheaper than GPT-5.4 + Toolset, ~340× cheaper than OpenAI DeepResearch, while remaining competitive on quality.
- Well-designed operator toolkit that covers the realistic retrieval loop end-to-end (sourcing → filtering → scoring → reranking → evidence → graph).

## Limitations

- Operator library is fixed and symbolic; may not cover domain-specific search behaviors (e.g., clinical-trial retrieval, patent search, cross-corpus linkage).
- Teacher trajectories inherit the teacher model's biases and blind spots.
- Benchmark is CS-only with curated 6–15 gold papers per case; generalization to open-ended real-world search is not yet demonstrated.
- Multi-turn evaluation relies on an LLM user simulator (Qwen3.5-397B-A17B) with leakage control — even with that, it cannot fully replace real human feedback.
- Operator-sandbox filtering likely drives part of the workflow-execution-error reduction; results may not transfer to noisier tool environments where invalid plans are not rejected cleanly.
- Smaller models (e.g., 3B or below) are not evaluated; scaling behavior at the low end is unknown.

## Relevance

该工作与"自演化智能体与自动化研究"兴趣高度相关:PaperPilot 把检索策略从隐式语言推理转为**显式可编辑 DAG**,并通过"工作流损坏 → 偏好优化"的闭环让 agent 学会从用户反馈中恢复与重构工作流——这正是自动化文献调研流水线急需的可控、可检视范式,也可作为后续"agent 自我演化"研究(工作流库自我扩展、损坏检测等)的天然基础设施。

## Questions for Further Exploration

- How does the explicit-DAG pattern transfer to other tool-use domains (code agents, web agents, database agents) where the operator toolkit is less standardized than paper search?
- Can the "controlled corruptions" preference data construction be scaled beyond paper search — e.g., generating corrupted web-browsing trajectories for general web agents?
- How sensitive is performance to the choice of user simulator? A study swapping Qwen3.5-397B-A17B for GPT-5.4 / Kimi K2.6 would bound simulator-induced gains.
- Can the operator library be learned or grown automatically (auto-extending PaperPilot-Toolset) instead of being hand-engineered?
- How does PaperPilot behave when the user is *wrong* about what they want, or when gold papers are adversarial / out-of-corpus?
- Workflow execution error going to exactly 0% — what fraction comes from the operator-sandbox rejection layer vs. the model itself? An ablation isolating the sandbox would clarify the contribution.
- Is the 0% error rate robust to a noisier tool environment where invalid plans are silently executed rather than rejected?
