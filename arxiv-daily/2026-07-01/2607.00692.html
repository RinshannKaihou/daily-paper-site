02 Why This Matters↗ relevance
Reframes agent context as runtime objects with lifecycle guarantees — fold/mask/prune over indexed IDs with sidecar recoverability. A vocabulary that maps cleanly onto real engineering concerns: which spans are actually needed downstream.
Deployable today · harness-portable · not a provider rewrite
03 Critical Edge⚠ caveat
Judge-based no-impact is a proxy, not a recovery test. Sidecar byte-exact restore under adversarial edits is never demonstrated — and planners still attempt to cut the latest user turn 12–25 / 330 times.
"The prompt usually works, but residual risk justifies mandatory last-turn protection."

Methodologyharness interface

  1. Indexed IDs — conversation:user:kfunction:<tool>:n assigned by a session-local monotonic allocator; assistant turns are not first-class GC targets.
  2. Object-action contract — planner emits XML with grouped fold / mask / prune over existing identifiers only. No "summarize old context" prompt.
  3. Rehearse — harness forks the prefix, replays candidate edits into a local projection, drops invalid/cut-turn edits, normalizes overlapping actions.
  4. Cache-aware commit check — applies CommitBenefit > 0; rejects the plan otherwise. Rejected plans never reach the main loop.
  5. Safe-boundary commit — repairs parent lineage to nearest surviving ancestor, persists folded sidecars, normalizes provider messages. Mandatory last-turn retention.
  6. Operating policy — 30% compression threshold; 0.3 active-view-pruning commit floor (described as policy, not law).

Failure Modesdependency-centric

  • Exact evidence — URL, row value, table cell dropped.
  • Locators & handles — callback URLs, job IDs lost.
  • Behavioral contracts — e.g. "do not replace the table header" overwritten.
  • Verbatim source text — paraphrased into wrong citation.
  • Live execution state — "explain which rerun finally succeeded" gone.
Chronological / type-based policies have no representation for these distinctions — they delete without telling whether the span was trace or the only bridge to the next action.

Strengths+

  • Object-action contract targets the real failure mode: lost URLs/values/handles, not lost narrative.
  • Sidecar recoverability beats self-summaries on byte-exact restoration.
  • Harness invariants let a mid-tier planner be safely deployed.
  • Plan → Rehearse → Commit: model mistakes never directly edit live conversation.
  • Cache-aware commit formula replaces magic thresholds.
  • Multi-axis evaluation: offline + hard-set + scaled prod + A/B calibration + online.

Limitations−

  • No public artifacts — production data confidential; sanitized repro package promised but not shipped.
  • Small samples (33 / 332) → wide CIs on Hard Set.
  • No end-to-end sidecar byte-exact restore test; judge is a proxy.
  • Online signal is operational, not experimental; planner overhead not billed.
  • No direct run against AgentDiet / AgentFold / LongCodeZip / ACON / ContextBudget.
  • Hard Set skewed to browser/shell/web-fetch — not generalizable without rebalancing.

Questions for Further Exploration10

  1. Can the GC planner be replaced by a learned task-conditioned policy at materially lower inference cost without dropping below 90% no-impact on the Production Suite?
  2. Does Self-GC compose with external memory (Mem0, A-MEM, TiMem, HippoRAG) — migrate durable knowledge out of the active prompt while fold/mask/prune governs transient trace?
  3. What happens under multi-modal / vision agent traces where some "objects" are screenshots or generated artifacts? Is the function:<tool>:n scheme general enough?
  4. How sensitive is the 0.3 active-view-pruning commit floor to provider cache pricing and TTL? Does the formula generalize across OpenAI / Anthropic / Google / Kimi / Aliyun cache designs?
  5. The sidecar recoverability claim is judged via no-impact proxy. What does a direct byte-exact restore test (recover folded artifact from sidecar by ID, score 0/1) look like — and what fails?
  6. On Production Suite, the gap over the strongest baseline is only ~7 pp (92.77 vs 87.46). With more sessions and a fully randomized deployment, does that gap survive or compress?
  7. How does Self-GC interact with trained context compressors (Composer-style, ACON, ContextBudget, Chroma Context-1)? Can the planner contract absorb their outputs as additional actions?
  8. The paper gates commit on CommitBenefit > 0 from the active agent's view. Does a multi-tenant / shared-prefix setting need a coordinated commit scheduler?
  9. Can online bandit / RL over the planner prompt's few-shot calibration improve no-impact under shifting production workload mix (DOC-heavy vs CODE-heavy), without provoking cut-turn edits?
  10. Evaluation only checks future-dependency preservation. Does Self-GC change downstream answer quality or user-perceived helpfulness on a randomized holdout, or only token economics?
