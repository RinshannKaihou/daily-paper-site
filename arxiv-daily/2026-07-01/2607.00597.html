Multi-Turn Agentic Scientific Literature Search via Workflow Induction

A 9B model learns to induce explicit, editable DAG search workflows instead of opaque chain-of-thought tool calls — turning scientific retrieval into inspectable code.

Jisen Li, Bingxuan Li, Nanyi Jiang, Xuying Ning, Xiyao Wang, Yifan Shen, Heng Wang, Yuqing Jian, Xiaoxia Wu, Ben Athiwaratkun, Pan Lu, Jiaxuan You, Bingxin Zhao

PaperPilot-9B eliminates runtime workflow errors under multi-turn interaction — every induced DAG executes cleanly against the typed operator sandbox.
Why This Matters
Explicit, editable DAGs turn opaque agentic search into inspectable, debuggable code — a natural substrate for self-evolving research pipelines and operator-library growth.
Critical Edge
Not absolute SOTA. PaperPilot-9B still trails GPT-5.4 + Toolset on raw Hit@5 (77.0 vs 84.0). The win is cost-effectiveness, and the operator-sandbox rejection layer likely inflates the 0% error.

Methodology

Problem Formulation

Inputs: anchor paper p₀ (with cached Semantic Scholar metadata), user query q, evolving interaction history Ht. Action space: ask clarification, induce / edit DAG, execute workflow, finalize. Transition: st+1 = T(st, at, ft). Objective maximizes expected cumulative utility — relevance, feedback alignment, intent coverage, efficiency.

Workflow Representation

A workflow is a DAG G = (V, E); each node vi = (oi, θi) is an instantiated operator with parameters, edges encode typed data flow (PaperSet / Scored / Evidenced / Graph / Keywords). Type consistency is required — every predecessor output type must match the successor input. A fixed-workflow baseline (PaperPilot-Workflow) isolates the value of adaptive induction.

17-Operator PaperPilot-Toolset

Sourcing (keyword_search, citation_expand, web_resolve) → Combine/Filter (union, dedupe, filter) → Scoring (score) → Cutting (top_k, above) → Rerank (llm_rerank, nli_filter, fine_read) → LLM (llm_keywords, llm_keywords_from) → Output (extract_evidence, pairwise_nli, build_graph).

Two-Phase Training

  • SFT on 5,540 teacher-generated workflows from 2,723 anchor–query cases × 5 directions (predecessor / successor / sibling / benchmark / survey), 3 epochs, lr 2e-4, seq 14,336, LoRA on attention + MLP.
  • IPO-style DPO on 1,733 hard chosen-vs-corrupted pairs (invalid references, missing inputs, wrong operator, dropped nodes, shifted filters, vague NLI axes) — 3 epochs, lr 3e-5, β=0.2, seq 16,384.
  • Multi-turn refinement: Gt+1 = Refine(Gt, ft, Ht+1) — user feedback drives both query updates and DAG edits.

Strengths vs Limitations

Strengths

  • Explicit, editable DAG workflow — rare in agentic-IR; transferable to code/web agents.
  • Hit@5 +19 absolute; workflow execution errors 9.5% → 0%.
  • Two-phase SFT + corruption-DPO is principled and reusable.
  • Comprehensive eval: retrieval, workflow generation, refinement, sensitivity, cost, human study.
  • Excellent cost profile: ~80× cheaper than GPT-5.4 + Toolset, ~340× cheaper than OpenAI DeepResearch.
  • 17-operator toolkit covers the full retrieval loop (sourcing → filtering → scoring → reranking → evidence → graph).

Limitations

  • Operator library is fixed & symbolic — may miss clinical-trial, patent, or cross-corpus search behaviors.
  • Teacher trajectories inherit the teacher's biases and blind spots.
  • Benchmark is CS-only with curated 6–15 gold papers per case; open-domain real-world search untested.
  • Multi-turn eval relies on a LLM user simulator with leakage control — cannot fully replace humans.
  • Operator-sandbox rejection likely drives part of the 0% error; may not transfer to noisier tool environments.
  • No evaluation for ≤3B models — scaling behavior at the low end is unknown.

Questions for Further Exploration

  1. How does the explicit-DAG pattern transfer to code agents, web agents, or database agents where the operator toolkit is less standardized than paper search?
  2. Can "controlled corruptions" preference-data construction be scaled beyond paper search — e.g., generating corrupted web-browsing trajectories for general web agents?
  3. How sensitive is performance to the user simulator? Swapping Qwen3.5-397B-A17B for GPT-5.4 / Kimi K2.6 would bound simulator-induced gains.
  4. Can the operator library be learned or grown automatically (auto-extending PaperPilot-Toolset) instead of being hand-engineered?
  5. How does PaperPilot behave when the user is wrong about what they want, or when gold papers are adversarial or out-of-corpus?
  6. The 0% execution error — what fraction comes from the operator-sandbox rejection layer vs. the model itself? An ablation isolating the sandbox would clarify the contribution.
  7. Is the 0% error rate robust to a noisier tool environment where invalid plans are silently executed rather than rejected?
