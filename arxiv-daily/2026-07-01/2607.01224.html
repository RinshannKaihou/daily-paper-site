AutoMem: Automated Learning of Memory as a Cognitive Skill

Shengguang Wu · Hao Zhu · Yuhui Zhang · Xiaohan Wang · Serena Yeung-Levy Stanford University  ·  arXiv:2607.01224  ·  1 Jul 2026

TL;DR  memory as a separable skill Base model  Qwen2.5-32B-Instruct Games  Crafter · MiniHack · NetHack
Key finding

Treating memory as a learnable skill, Qwen2.5-32B gains 2–4× on long-horizon games — with task-action weights frozen.

Two meta-LLM outer loops automate scaffold + proficiency. No new parameters on the gameplay policy — only on a dedicated "memory specialist" LoRA.

02 · WHY THIS MATTERS
Implication

A 32B open-weights model becomes competitive with Claude Opus 4.5 and Gemini 3.1 Pro Thinking on long-horizon procedural games — not by scaling, but by learning how to use memory. A concrete, reproducible instance of the AutoResearch paradigm.

03 · CRITICAL EDGE
Caveat
Read this first

The "memory specialist" is per-game, and its training signal is filtered by a closed meta-LLM (Claude Opus) — so Claude's biases are baked into both the gate and the LoRA data. No real long-horizon tool-use (web, code, OS) was tested.

Methodology § Inner-loop + two outer loops

An inner-loop agent (Qwen2.5-32B-Instruct served via vLLM) plays Crafter / MiniHack / NetHack with a directory of files as external memory. Every step runs two routines — LOG (what to record) and PLAN (what to recall before acting). Memory file-ops live in the same action vocabulary as gameplay actions, so every memory decision is a traceable token in the trajectory.

Loop 1 · Scaffold

Claude Opus 4.6 reads full trajectories (up to 10⁵ steps), diagnoses failures, edits code / prompts / file schema / action vocab. Revisions gated on same-seed eval improvement.

Loop 2 · Proficiency

Claude Opus 4.7 picks training data + LoRA config; LoRA-finetunes a dedicated "memory specialist" on the model's own good memory decisions. Gameplay model stays frozen.

Deployment split

At inference, the memory-specialist LoRA handles LOG and the memory-consult portion of PLAN; a handoff passes context to the frozen gameplay model for the final world action. Keeps training signal focused and preserves base-model action formatting.

Eval protocol

10 fixed seeds [42–51]; same updated BALROG harness for baselines and AutoMem; training seeds disjoint from eval seeds → no contamination possible.

Strengths § What holds up
  • A clean, falsifiable claim — the architecture explicitly freezes task weights, isolating memory as the only variable changed.
  • Behavioral diagnostics beyond headline progression rate: stuckness −32 to −65%, repeat-WRITE −68 to −83%, empty-SEARCH −13 to −50%, input tokens/step −3 to −30%.
  • Iteration is gated on identical eval seeds — a proper hill-climbing gate, not vibes; auditable meta-loop.
  • Open code + open-weights base + standard harness — strong reproducibility surface area given closed meta-LLM dependencies.
  • Loop 1 ≈ Loop 2 in magnitude — both axes earn their place in the ablation, neither is buried.
Limitations § What to push back on
  • Domain scope — three procedurally generated games. Web / OS / coding-agent transfer is asserted, not shown.
  • Per-game specialists — each environment gets its own scaffold + its own LoRA. No evidence of a single transferable memory skill.
  • Episodic only — memory resets every episode; persistent / cross-episode memory is future work.
  • Offline SFT, not online RL — Loop 2 inherits base-model biases via the meta-LLM's "good memory" filter.
  • Compute footprint — repeated Claude Opus --effort max calls in both loops; favours API-rich teams over small labs.
Questions for further exploration § Open ground
  1. Does the separability claim (memory ≠ task actions) hold on real tool-use — web agents, code agents, OS agents — where the two are tightly interleaved?
  2. Can Loop 2 be replaced by online RL (RLOO / GRPO / PPO) against a memory-quality reward, removing dependence on a closed meta-LLM judge?
  3. What is the transfer curve for a memory specialist trained on Crafter → applied to MiniHack / NetHack / web tasks?
  4. Could a single scaffold + single memory specialist be jointly trained across all three games via curriculum, and match per-game specialists?
  5. How robust is the meta-LLM reviewer to adversarial or noisy trajectory summaries — what fails when the reviewer is wrong?
  6. How does per-step input-context reduction (−3 to −30%) interact with hidden-state interpretability — what does the leaner memory change in the residual stream?
