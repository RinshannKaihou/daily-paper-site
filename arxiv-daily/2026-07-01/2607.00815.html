LRAT-Catcher: Importing SAT Solver Certificates into Lean 4 by Reflection

Standalone Lean 4 tool that takes any DIMACS formula plus an LRAT proof and turns it into a Lean theorem — using Lean's already-verified core checker, invoked by reflection, instead of building an explicit proof term.

Key Metric
8.9GB VS 95+GB

Peak memory on php(10,9) via native reflection, vs. Mathlib's lrat_proof which the OS OOM-kills at the same size. Native reflection scales at ~14–35× the certificate; the explicit import grows at ~1500×.

Why This Matters

A reusable template for any minimize-the-trusted-base effort: keep the trust surface minimal, then use a reflection-compiled native path to make it usable. The verification-vs-usability trade-off the paper navigates is the same one that recurs wherever strict guarantees meet real workloads.

Critical Edge

Certified, but at what cost? R(4,4)=18 still needs 188 GB peak and emits a 51.6 GB compiled Lean object; native_decide adds the Lean compiler to the trusted base, while CaDiCaL 3 has to be run with --no-factor to play with the verified checker.

Methodology

Reflection over the verified core. Lean's already-formally-checked LRAT checker (Std.Tactic.BVDecide.LRAT, Böving et al.) is invoked by reflection — native_decide compiles it to native code to scale, and a +kernel mode restricts trust to the Lean kernel + three axioms (but is much slower).

CNF → combinatorial lifting. Schur and Ramsey are encoded as CNF; an encoding-soundness lemma (proved once by hand, separate from the certificate) makes UNSAT on the CNF imply UNSAT on the original problem.

Cube-and-conquer composition. Each leaf is independently UNSAT-certified and a second LRAT certificate proves the cubes cover the formula. Composition (Theorem 5.1) is a single Boolean function checked end-to-end inside Lean — no trusted cover combinator.

Parallel leaf build. Each leaf compiles in its own Lean module, so peak memory is bounded by the largest single leaf, not the total certificate volume.

Solver integration. CaDiCaL 3 with --lrat --no-binary --no-factor; verified checker is full-RAT.

Strengths vs. Limitations

Strengths

  • Reuses Lean's already-verified LRAT checker with full RAT support — no reimplementation risk.
  • Peak memory scales with the certificate (~14–35×), not with the embedded formula (~1500×).
  • Cube-and-conquer composition is end-to-end inside Lean — cover completeness is itself an LRAT proof, not a trusted combinator.
  • Parallel leaf build caps per-process memory at the largest single leaf, decoupling memory from leaf count.
  • Verified CNF encodings plus soundness lemmas keep the CNF-to-problem bridge auditable.
  • Open-source artifact, plus a companion PBLean announcement, signals ecosystem continuity.

Limitations

  • Trusts the SAT solver as a black box; the verified checker catches certificate bugs, not solver misbehaviour.
  • native_decide adds the Lean compiler to the trusted base; +kernel mode is too slow for non-trivial certificates, so most users sit on the larger trusted surface.
  • Still 188 GB peak + a 51.6 GB compiled object for R(4,4)=18 — scalable, not cheap.
  • CaDiCaL 3 factoring must be disabled (--no-factor) — solver-side compatibility friction that may recur with future releases.
  • Single-author scope; no industrial SMT/ILP or SAT-modulo-theories workload characterization.
  • Kernel-reducible replacements for opaque core constructs add maintenance surface tied to Lean core updates.

Questions for Further Exploration

  1. How does the per-module native_decide axiom count compare with proof-assistant community standards, and could cake_lpr-style post-hoc checking reduce it further?
  2. Will CaDiCaL 4 (or any successor) produce LRAT certificates the verified Lean-core checker accepts without --no-factor workarounds — or should the checker be extended?
  3. Can the reflection + verified-checker pattern lift to VeriPB (the PBLean direction) and to SMT/ILP certificates, yielding a uniform Lean-import framework?
  4. For very large cube-and-conquer runs (10⁴–10⁵ leaves), does the per-module native_decide axiom count become an audit or usability problem?
  5. Is there a path to certifying the hand-proved encoding-soundness lemma itself by reflection or external checker, removing the last non-mechanical component?
  6. Could a similar architecture import certificates from proof-producing neural or LLM-based provers, where the certificate format and the "trusted base" question look quite different?
