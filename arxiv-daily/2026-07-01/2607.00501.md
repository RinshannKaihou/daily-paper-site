> **一句话总结：** BaseRT 是一个直接构建在 Apple Metal GPU API 上的 C++ LLM 推理运行时（无 MLX/PyTorch/CoreML 中间层），通过数据驱动的架构描述符、零分配 decode loop、手写 Metal kernel 融合（含内联反量化）与 FlashAttention，在 M4 Pro / M3 上 Q4/Q8 量化下相比 llama.cpp decode 提升 1.04–1.56×、相比 MLX 最高 1.35×，并在 MoE 模型 prefill 上相对 llama.cpp 取得 1.59–1.81× 的领先。

> **TL;DR:** BaseRT is a C++ LLM runtime built directly on Apple's Metal GPU API (no MLX/PyTorch/CoreML layer). It reports the highest open-source decode throughput on Apple Silicon to date: 1.04–1.56× over llama.cpp and up to 1.35× over MLX on M4 Pro at Q4/Q8 across Qwen3, Llama 3.2, and Gemma 4 families, with an even sharper 1.59–1.81× lead over llama.cpp on MoE prefill (Qwen3-30B-A3B, Gemma-4-26B-A4B). The win comes from four engineering bets — data-driven architecture descriptors, a zero-allocation decode loop, hardware-adaptive hand-fused Metal kernels with inlined dequantisation, and a Metal-simdgroup FlashAttention path — rather than a new algorithm. Numbers are self-reported, with prefill on dense models essentially tied to MLX/llama.cpp and one notable regression (0.90× vs MLX on Gemma-4-26B-A4B decode).

## 核心贡献

- 提出 BaseRT：一个不依赖 MLX / PyTorch / CoreML 等中间框架、直接面向 Apple Metal GPU API 的 C++ 推理运行时，对外暴露 C API 便于跨语言互操作。
- **数据驱动的架构描述符**：将激活/归一化变体、MoE 细节、注意力与位置编码约定等架构差异表达为紧凑的数据描述符而非控制流，使热路径不按架构分支；新增架构（LLaMA / Qwen3 / Gemma / Whisper / BERT）只需声明式修改。
- **零分配 decode 循环**：模型加载后 decode 阶段分配 0 字节——残差、注意力、FFN 草稿、logits、token 缓冲区以及最大上下文长度的 KV cache 均在加载时预分配；错误处理也使用静态线程局部缓冲区，热路径只剩 GPU 命令派发与 CPU 端轻量修补。
- **芯片自适应、手写 Metal kernel 库**：matmul、attention、归一化、RoPE、embedding、激活、采样七大类；matmul 针对每种量化格式（2–16 bit）分别提供 GEMV（decode, M=1）与 GEMM（prefill, M>1）变体，并把反量化内联到内层循环，避免在 global memory 上物化反量化权重。
- **逐算子融合**：把通常分开发射的注意力路径与 FFN 内部算子序列融合，每消除一次 kernel launch 与一次 global memory 往返，在 decode 阶段收益尤其大。
- **Apple Silicon 硬件自适应配置**：根据 GPU 核心数与家族（M1–M5）选择 threadgroup 大小、kernel 选择阈值与融合决策。
- **Metal simdgroup 矩阵 intrinsic 的 tiled GEMM + chunked prefill + FlashAttention**（online softmax、O(n) 内存）prefill 路径。
- **低开销命令派发 + decode 调度**：在 decode 热路径上去除每算子调度/分配/图评估开销，并将 CPU–GPU 同步摊销到多个生成 token。
- 在 M3 base（8 GPU 核 / 16 GB / 100 GB/s）与 M4 Pro（16 GPU 核 / 24 GB / 273 GB/s）上系统评测 Qwen3-0.6B、Llama-3.2-1B/3B、Gemma-4-E2B（dense）以及 Gemma-4-26B-A4B、Qwen3-30B-A3B（MoE）的 Q4 / Q8 decode 与 128–2048 token 的 prefill。

## Key Contributions

- A native-Metal C++ LLM inference runtime for Apple Silicon that bypasses MLX, PyTorch, CoreML, and other intermediate array frameworks; exposes a C API; currently supports LLaMA, Qwen3, Gemma, Whisper, and BERT families.
- **Data-driven architecture descriptors** that capture per-architecture variation (activations, normalisation, MoE, attention variants, positional encoding) as compact data rather than control flow, keeping the hot path uniform across models.
- **Zero-allocation decode loop**: after model load, every per-token intermediate buffer (residual, attention, FFN scratch, logits, token buffer) and the full KV cache are pre-allocated; even error handling uses a static thread-local buffer.
- **Hand-written, hardware-adaptive Metal kernel library**: dedicated GEMV (M=1) and GEMM (M>1) variants per quantization format (2–16 bit), with dequantisation inlined into the inner loop; per-chip configuration selects threadgroup sizes, kernel thresholds, and fusion choices for M1–M5.
- **Operator fusion** along the attention path and inside FFN blocks, removing kernel launches and global-memory round-trips that disproportionately hurt decode.
- **Metal simdgroup-based tiled GEMM with chunked prefill + FlashAttention (online softmax, O(n) memory)** for compute-bound prompt processing.
- **Low-overhead command dispatch and decode scheduling** that amortises CPU–GPU synchronisation across multiple generated tokens — explicitly targeting the decode hot path.
- **Comprehensive Apple Silicon benchmark** against llama.cpp (commit b9630, June 2026, Metal backend) and MLX (v0.31.2, mlx-lm), plus the closest architectural competitor uzu (Mirai) on M3 base and M4 Pro.
- Open-source release at `github.com/basecompute/baseRT`; supports Q2, Q3, Q4, Q5, Q6, Q8, BF16, FP16 on every Apple M-series generation.

## Methodology

- **Architecture**. C++ runtime targeting Metal directly; no MLX, PyTorch, CoreML, or other array-framework dependency. C API for cross-language interop; models loaded in BaseRT's native base weight format. Current architectures: LLaMA, Qwen3, Gemma, Whisper, BERT.
- **Data-driven architecture descriptors**. Per-architecture knobs (activations, normalisation, MoE specifics, attention/positional-encoding conventions) are expressed as a compact descriptor; the core engine never branches on architecture identity, so adding a new architecture is a declarative change that leaves the inference loop untouched.
- **Zero-allocation decode loop**. All forward-pass scratch, logits, token buffer, and the maximum-context KV cache are pre-allocated at load time, with KV layout chosen for coalesced GPU memory access during attention. The decode hot path is reduced to GPU command dispatch plus lightweight CPU-side patching — no allocator contention, no per-step memory management.
- **Kernel library and fusion**. Hand-written Metal shaders organised across matmul, attention, normalisation, RoPE, embedding, activation, and sampling. Matmul is split into GEMV (M=1, decode) and GEMM (M>1, prefill) variants per quantization format, each integrating dequantisation directly into the inner loop. Operator sequences that other runtimes dispatch as separate kernels are fused, eliminating one kernel launch and one global-memory round-trip per fusion — particularly impactful during decode where individual kernel runtimes are short.
- **Hardware-adaptive kernel selection**. A per-chip configuration (derived from GPU core count and family, M1–M5) selects threadgroup sizes, kernel thresholds, and fusion decisions; each quantization format is mapped to its specialised kernel and launch geometry.
- **Prefill**. Uses GEMM rather than GEMV on activation tensors of shape `[seq_len × dim]`. Tiled GEMM kernels built on Metal's simdgroup matrix intrinsics with tile geometries tuned to sequence-length regime; chunked prefill caps scratch buffer memory. A FlashAttention kernel computes attention in tiles with online softmax and running statistics, achieving O(n) memory instead of materialising O(n²).
- **Decode-path CPU overhead reductions**. A low-overhead command dispatch removes per-operator scheduling, allocation, and graph evaluation; a decode scheduling scheme amortises CPU–GPU synchronisation across multiple generated tokens.

### Experimental setup

- **Hardware**: primary M4 Pro (16 GPU cores, 24 GB unified memory, 273 GB/s); secondary M3 base (8 GPU cores, 16 GB, 100 GB/s).
- **Models**: six spanning Qwen3, Llama 3.2, Gemma 4 — dense Qwen3-0.6B, Llama-3.2-1B, Llama-3.2-3B, Gemma-4-E2B; MoE Gemma-4-26B-A4B and Qwen3-30B-A3B.
- **Quantisation**: Q4 and Q8.
- **Baselines**: llama.cpp (commit b9630, June 2026, Metal backend, default settings, GGUF); MLX (v0.31.2, mlx-lm, default generation settings, MLX SafeTensors); uzu (Mirai, native Metal + MPSGraph hybrid).
- **Metrics**: prompt processing throughput `pp` (tokens/s) at prompt lengths 128, 256, 512, 1024, 2048; decode throughput `tg128` over 128 generated tokens. Standard deviations omitted for readability (reported as <3% for decode, <10% for prefill).

## Results & Highlights

### Decode (tg128) on M4 Pro

| Model | Quant | BaseRT | llama.cpp | vs llama.cpp | MLX | vs MLX |
|---|---|---:|---:|---:|---:|---:|
| Qwen3-0.6B | Q4 | 464.5 | 297.4 | **1.56×** | 343.6 | **1.35×** |
| Qwen3-0.6B | Q8 | 321.2 | 219.8 | 1.46× | 255.3 | 1.26× |
| Llama-3.2-1B | Q4 | 295.4 | 230.4 | 1.28× | 257.8 | 1.15× |
| Llama-3.2-1B | Q8 | 183.8 | 160.7 | 1.14× | 159.2 | 1.15× |
| Llama-3.2-3B | Q4 | 117.3 | 102.4 | 1.15× | 112.1 | 1.05× |
| Llama-3.2-3B | Q8 | 70.9 | 65.1 | 1.09× | 65.5 | 1.08× |
| Gemma-4-E2B | Q4 | 127.7 | 107.0 | 1.19× | — | — |
| Gemma-4-E2B | Q8 | 84.5 | 59.5 | 1.42× | — | — |
| Gemma-4-26B-A4B | Q4 | 62.2 | 58.0 | 1.07× | 69.3 | **0.90×** |
| Qwen3-30B-A3B | Q4 | 84.1 | 80.7 | 1.04× | 83.1 | 1.01× |

- Largest gains are on **small dense models at Q4** (Qwen3-0.6B: +56% over llama.cpp, +35% over MLX) where fixed per-token overhead dominates latency.
- The advantage **narrows on MoE models** (1.04–1.07× vs llama.cpp) because large-model decode is memory-bandwidth-bound.
- **One regression**: Gemma-4-26B-A4B Q4 decode runs at **0.90× of MLX** (62.2 vs 69.3 tok/s) — the only MLX loss in the table.
- The relative advantage is **larger at Q4 than Q8** (smaller weights make per-token overheads a larger share of the budget).

### Prefill (pp128–pp2048) on M4 Pro

- **vs llama.cpp (Table 2)**: on dense models, BaseRT trails by 5–10% on Qwen3-0.6B, 1–6% on Llama-3.2-1B, ±6% on Llama-3.2-3B (no systematic trend across prompt length); Gemma-4-E2B trails 3–7% at Q4 but **leads by 19–23% at Q8**. On MoE, the ordering reverses: **up to 1.59× on Gemma-4-26B-A4B** and **up to 1.81× on Qwen3-30B-A3B at pp128** (738 vs 407 tok/s); margins narrow at longer prompts as GEMM saturates the GPU.
- **vs MLX (Table 3)**: dense models run close — BaseRT tends to lead at pp128, MLX pulls ahead from pp512 onward. On MoE, BaseRT leads **up to 1.42× on Gemma-4-26B-A4B** and **up to 1.78× on Qwen3-30B-A3B at pp128** (738 vs 415 tok/s), shrinking toward parity by pp2048.
- Headline MoE prefill gap (vs MLX, Qwen3-30B-A3B pp128): **1.78× = +78%** — the source of the "+78%" callout in Figure 1.
- Authors' framing: on dense models, GEMM-based prefill is compute-bound and saturates GPU matmul units across runtimes, so throughputs converge; on MoE models, BaseRT's GPU-only path holds a clear, consistent lead.

### Cross-generation (M3 base, tg128)

Against llama.cpp: BaseRT leads all eight dense configurations by **1.13–1.34×** (best Gemma-4-E2B Q8 at 1.34×; Qwen3-0.6B Q8 1.25×). Against MLX: leads all six supported configurations by **1.01–1.22×**. Confirms the decode advantage carries across chip generations.

### Comparison with uzu (native Metal + MPSGraph)

- **Decode**: BaseRT faster on 5/6 configurations (1.17–1.19× on Qwen3-0.6B; 0.99–1.06× on Llama models).
- **Prefill**: uzu leads on most configurations by 4–13% on Qwen3-0.6B and 1–3% on Llama models, widening at longer prompts. The authors attribute this to **uzu's MPSGraph dispatch into the Apple Neural Engine (ANE)**, which provides additional compute bandwidth for GEMM-dominated prefill — a path BaseRT's GPU-only stack does not currently exploit.

## Critical Assessment

- **Novelty**: incremental in the lineage of llama.cpp → MLX → vllm-mlx → MetalRT/uzu. The substantive contribution is a stack-engineering decision (commit to Metal only, no portability layer) plus the four implementation principles (data-driven descriptors, zero-alloc decode loop, inlined-dequantisation kernel library, FlashAttention on Metal simdgroup). No new algorithm is proposed.
- **Rigor**: above-average for an industry benchmark paper. Baselines pinned to specific commits and versions; standard deviations explicitly bounded (<3% decode, <10% prefill); both decode and prefill reported across five prompt lengths; cross-generation check on M3 in addition to M4 Pro; head-to-head with the closest architectural competitor (uzu). Weakness: prefill numbers for dense models show BaseRT *trailing* llama.cpp and MLX in places, which the authors acknowledge but do not deeply analyse beyond "GEMM is compute-bound, throughputs converge."
- **Reproducibility**: strong. Open-source code, pinned dependencies, standard models, fixed quantisation recipes. Anyone with the hardware can reproduce in a day. Hardware availability is the only real constraint (M3 base and M4 Pro are required for the headline numbers).
- **Significance**: meaningful for the on-device LLM community. If the speedups hold on independent re-runs, BaseRT raises the ceiling of Apple Silicon LLM inference — relevant for privacy-sensitive (medical, legal, financial), latency-sensitive (real-time agents, robotics), and air-gapped deployments. The MoE prefill result (1.81× over llama.cpp on Qwen3-30B-A3B at pp128) is particularly notable given the rapid rise of small-active MoE architectures for local deployment (Qwen3, Gemma 4, Phi-MoE, etc.).
- **Red flags**:
  1. **Self-reported, vendor-affiliated benchmarks** in inference are notorious for cherry-picked baselines. The paper pins llama.cpp to a specific commit but does not report MLX with `mlx-lm`'s faster `--max-kv-size` or speculative decoding settings, nor llama.cpp with its experimental Metal-Split backend.
  2. **MoE prefill is the headline, but decode on MoE is essentially a tie** (1.04× / 1.07× vs llama.cpp; one outright loss to MLX at 0.90×). The framing "best-in-class throughput" is selectively drawn from the strong cells.
  3. **No quality analysis** — no perplexity, MMLU, GSM8K, or any downstream accuracy check at Q4/Q8. Given that fused dequantisation paths can introduce numerical drift, this is a real gap.
  4. **No thermals / sustained-throughput / energy analysis**. Apple Silicon throttles aggressively; burst numbers can be 30–50% above sustained, which matters for any "edge inference" deployment story.
  5. **Single-device only**. The paper concedes no continuous batching, no parallel decoding, no tensor parallelism — all of which limit practical serving claims.
  6. **uzu comparison is qualitatively framed** ("uzu's use of MPSGraph … likely accounts for this") without a quantitative ANE vs. GPU utilisation breakdown.
  7. **Metal-only** means the contribution does not generalise to the broader LLM-systems research community.

## Strengths

- Targets a real, growing pain point — on-device LLM inference performance on Apple Silicon — with a credible open-source artifact.
- Headline speedups (1.56× decode vs llama.cpp on Qwen3-0.6B; 1.81× MoE prefill vs llama.cpp) are non-trivial and reproducible in principle.
- Four design principles are well-explained and tied to specific sources of overhead in existing runtimes; the architecture-descriptor idea is genuinely clean.
- Honest about where it does *not* win: dense-model prefill is tied to MLX/llama.cpp; MoE decode is tied to llama.cpp; one Gemma MoE decode is a loss vs MLX.
- Broad quantisation (Q2–FP16), broad architecture coverage (LLaMA / Qwen3 / Gemma / Whisper / BERT), and M1–M5 compatibility lower adoption friction.
- Open-source release + pinned baselines + standard models make independent verification cheap.

## Limitations

- **Apple-only** — no CUDA or Vulkan backend today, so the contribution is locked to a single hardware vendor.
- **Single-device only** — no continuous batching, no parallel decoding across requests, no tensor parallelism; unsuitable for server-class workloads as admitted.
- **No quality / accuracy analysis** at Q4 or Q8 — perplexity, downstream benchmarks, and numerical drift from fused kernels are not characterised.
- **No sustained-performance or thermal analysis** — burst numbers are likely much higher than what a real device delivers over a 30-minute agentic session.
- **No energy per token** despite framing the work around the "intelligence per watt" edge-inference argument.
- **Dense-model prefill is essentially tied** with llama.cpp and MLX, so the "best-in-class" claim is selective to decode and MoE prefill.
- **Engineering artifact, not research contribution** — no analytical model predicting *why* Metal-specific fusion should outperform by this margin; no ablation isolating the four design principles' individual contributions.
- **No discussion of speculative decoding** (a known lever for Apple Silicon LLM inference), which is left to future work.

## Relevance

与本关注的「Inference Evaluation（推理评测）」和「Computation Reliability & Precision Errors（计算可靠性与精度误差）」方向高度相关：BaseRT 提供了一个在 Apple Silicon 上系统对比 llama.cpp、MLX、uzu 的吞吐量基准，覆盖 Q4/Q8 多种量化，并报告 MoE prefill 的显著加速——这是评估本地 LLM 推理栈时常用的参考数据。同时由于其重度依赖 fused kernel 中内联反量化，却未报告量化后模型质量，是「计算精度/可靠性」方向值得追问的盲点。

## Questions for Further Exploration

- 1.04–1.56× 的速度差距在 llama.cpp、MLX 的实验性优化（如 Metal-Split、speculative decoding、`--no-mmap`、batch size 调优）下是否仍然成立？即**上限是 1.56× 还是已经达到 wall**？
- 数据驱动的架构描述符相比 MLX 的延迟执行与 vllm-mlx 的批处理框架，能否在不破坏 zero-alloc decode loop 的前提下支持 continuous batching？
- 量化路径内联反量化（特别是 2-bit/3-bit 的极端路径）是否引入了可测量的精度漂移？建议用 perplexity 与下游 MMLU/GSM8K/HumanEval 在 Q2/Q3/Q4/Q8 上系统对比 BaseRT 与 llama.cpp 同一 GGUF 量化版本。
- 在持续推理（30 分钟以上 agentic 工作负载）下，热降频会使吞吐量下降多少？论文完全没有触及这一点。
- 是否计划提供 CUDA 后端？这将直接关联到 GPU-class 推理服务器的对比，并把 BaseRT 从"苹果专属"提升到"通用 Metal-style kernel 库"层级。
- uzu 的 MPSGraph → ANE prefill 加速（4–13%）是否意味着在 Apple Silicon 上最优 prefill 策略是 GPU + ANE 混合？BaseRT 是否考虑集成 ANE？
- 在统一内存 24 GB 限制下，Qwen3-30B-A3B / Gemma-4-26B-A4B 的 Q8 量化无法加载（论文脚注），那么对这些 MoE 模型的最佳实际部署精度组合是什么（Q4 + KV cache 量化 + speculative draft？）？
