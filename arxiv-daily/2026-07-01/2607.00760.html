MosaicKV: Serving Long-Context LLM with Dynamic Two-D KV Cache Compression

Sheng Qiang · Ruiwei Chen · Yinpeng Wu · Jinyu Gu · Zhichao Hua · Yubin Xia · Binyu Zang · Haibo Chen  ·  IPADS, Shanghai Jiao Tong University

Dynamic two-D compression hits 16× attention speedup at 1M context with only 1.76% average accuracy loss.

Key metric · attention speedup vs. uncompressed
16×attention · up to · batch 8 / 256K · 12.5% retention

Averages drop to ~1.5–2.3× across regimes. 3.0× smaller KV-cache memory vs. Full; 4.8× lower TBT and 7.3× higher decode throughput vs. vLLM at 1M.

Why this matters

KV cache dominates long-context LLM serving cost — 1 TB at 8× LLaMA-3.1-8B / 1M tokens, ~64× the model weights. MosaicKV proves channel + sequence compression can be combined without the accuracy collapse that killed prior single-axis schemes.

Critical edge

The "16×" is a ceiling. Averages are 1.5–2.3×; some LongBench sub-tasks drop 5+ points (Qwen3 LCC 5.25→5.10 ≈ total miss); RULER is only reported at 64K, never at 128K–1M where compression usually breaks.

Methodology · how it works

Two-stage two-D compression. Stage 1 — for each segment, compute an SVD rotation per K/V from the segment's covariance matrix, rotate to concentrate magnitude in leading channels, then keep the top-r absolute-value elements per vector individually (no global mask). Per-block key vectors (mean of keys within a token block) are channel-compressed too. Stage 2 (decode only) — before each attention layer, score every block with the current query × per-block key and attend only to the top-k blocks. Both stages run on the same compressed cache.

Dynamic compression strategy. Prefill is partitioned into fixed 64K-token segments. Decode segments finalize when accumulated compression loss crosses a threshold (capped at 64K). Each segment's strategy has four parameters — channel rate, element group size 1/2/4, sequence rate, block size. SVD rotation is incremental (rank-1 outer-product updates per token). Channel params pick the most aggressive combo below the loss threshold; block size picks the largest candidate whose intra-block key variance stays below its threshold.

Packed KV + PackedAttention. Retained elements are stored contiguously with a D-bit position bitmap; bottom-25% globally unimportant channels are truncated (no bitmap). During decode, dot-products run directly on packed vectors via CUDA cores (not tensor cores) — exploiting measured HBM saturation (~90.5%) with idle CUDA cores (~10.35% busy on A800). The SVD inverse is folded into the rotated query vector to avoid rotating the huge cache matrix. Built on FlashAttention's chunked pipeline.

Heterogeneous double compression buffer. GPU buffer holds the unfinished current segment under the previous strategy so decode never blocks. CPU buffer asynchronously partitions segments, updates SVD covariance, selects strategy, and recompresses — then a switch swaps in the new compressed segment. A 32-token update buffer drains every ~40.6 μs.

Strengths

  • Attacks the real bottleneck — KV cache memory at long context — with a working prototype, not a model.
  • Combines two orthogonal compression axes and rescues accuracy via per-vector selection + dynamic per-segment strategy.
  • PackedAttention exploits measured resource imbalance (HBM saturated, CUDA cores idle) on the same device — no offloading needed.
  • Heterogeneous double buffering keeps decode latency-tight while CPU does the heavy async segment work.
  • Drops cleanly into PagedAttention / RadixAttention / FlashAttention stacks; orthogonal to KIVI / KVQuant / DiffKV quantization.
  • Strong multi-model, multi-benchmark evaluation with clean ablations crediting each design choice.

Limitations

  • RULER only at 64K; performance at 128K–1M RULER / Needle-in-a-Haystack is unreported.
  • Hard LongBench sub-tasks miss (Qwen3 QA-1 70→58; LCC 5.25→5.10; LLaMA QASper 28→24) — matters for multi-doc QA users.
  • Decode-stage only; prefill acceleration needs chunked prefill (future work).
  • PackedAttention relies on CUDA-core slack that may vanish on future bandwidth-rich hardware.
  • Per-vector bitmap + CPU coordination overhead not reflected in attention latency; tail-latency jitter not profiled.
  • No code release mentioned; only evaluated on NVIDIA H800/A800 — portability to AMD / non-CUDA unclear.

Questions for further exploration

  1. How does MosaicKV hold up on RULER at 128K–1M (its claimed regime) and on multi-needle Needle-in-a-Haystack, where per-vector selection may miss globally important tokens?
  2. Do the worst-case sub-task drops (QA-1 70→58; LCC 5.25→5.10) propagate into multi-doc QA or repo-level code generation workloads?
  3. Can the dynamic compression policy be learned end-to-end — a small predictor for (rate, group size) given a segment's energy distribution — replacing hand-tuned thresholds?
  4. How does MosaicKV interact with INT4 / FP8 KV quantization layered on top — does the packed-with-bitmap representation still pay off, or does quantization obviate the channel step?
  5. Does PackedAttention remain CUDA-core-bound when PagedAttention page tables, speculative decoding, or chunked-prefill kernels compete for the same cores?
  6. How portable is it to non-NVIDIA hardware (AMD MI300, TPUs) and to MLA-structured KV caches that share KV across heads at a per-block level?
  7. Does the incremental per-segment SVD + 25%-retained BF16 dot product introduce numerical drift that breaks reproducibility in long agent trajectories?
  8. For multi-turn agent workloads, does segment recompression during a long decode introduce between-turn latency jitter that breaks interactive SLOs?
