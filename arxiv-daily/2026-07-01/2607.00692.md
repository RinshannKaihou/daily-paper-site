> **一句话总结：** Self-GC（Xiaohongshu）将长任务 LLM 智能体的对话历史视作带生命周期需求的"索引化运行时对象"集合，由一个旁路规划器提议 fold/mask/prune 操作，并由执行框架（harness）在安全回合边界做沙盒排练+sidecar 恢复+缓存感知提交；离线与线上分别将"未来依赖保留率"从 54–88% 提升到 84–95%，生产环境日间主模型输入 token 减少 10–15%（峰值约 20%）。

> **TL;DR:** Self-GC reframes context management for long-horizon LLM agents as **runtime lifecycle control over indexed objects**, not cheap text cleanup. The harness assigns stable `conversation:user:k` and `function:<tool>:n` IDs to transcript spans; a side-channel planner proposes structured fold/mask/prune actions over those IDs; the harness then rehearses the plan, drops invalid edits, normalizes provider messages, persists folded payloads in recoverable sidecars, and only commits at safe turn boundaries — and only when expected savings outweigh prefix-cache disruption (`CommitBenefit ≈ N_future·(C−C′) − L_cache − L_GC`). On a 33-session Hard Set and a 332-session Production Suite derived from real Xiaohongshu traces, Self-GC reaches **84.85% / 91.27–94.58% no-impact** (GPT-5.5 judge with Wilson CIs, diff-grounded at the granularity of URLs/paths/row values/editable bodies/source-backed evidence) while pruning 43.95% / 31.04–33.98%, beating four heuristic baselines (oldest-turn fold, greedy tool-prune, two-stage tool-mask+prune, hybrid) by 15–17 percentage points of no-impact. An account-level deployment split with email-first-character ≥ `o` for treatment shows **10–15% average (peaks ~20%) daytime input-token reduction** during May 25–30, 2026 covered traffic.

## 核心贡献

- **对象级语义重塑长任务上下文管理**：把"agent 历史 = 线性 token 缓冲区"的隐含假设推翻，主张 agent 历史是生命周期各异的 **索引化运行时对象** 集合——失败命令日志可删除，重复浏览器快照可遮蔽，但单一来源的 URL / 行值 / 可编辑正文 / 句柄 / 回调地址必须精确可恢复。
- **Self-GC 框架：Plan → Rehearse → Commit 分工**：由轻量级 `context-engine` 钩子给用户轮次和工具跨度发稳定 ID（如 `conversation:user:k`、`function:tool:n`），旁路 LLM 规划器只输出**结构化对象操作合同**（不是"压缩成摘要"），主 agent 主循环之外的执行框架（harness）负责目标校验、最近用户轮保护、覆盖归一化、sidecar 持久化、谱系修复、提供商消息规范化，最后只在安全回合边界增量提交。
- **fold / mask / prune 三动作语义差异**：fold 把完整 payload 移入可恢复 sidecar 并留指针（适合需要幂等恢复的大体内容）；mask 保留对象边界并只省略低信号中段（适合重复日志）；prune 仅在依赖已闭合的废弃 trace 上使用（无恢复保证）。
- **缓存感知提交策略**：用 `CommitBenefit ≈ N_future·(C−C′) − L_cache − L_GC` 决定是否落地；实验给出经验阈值——active-view 剪裁 > ~30% 立即正收益，否则挂起到缓存过期或下个任务边界。
- **生产级评估协议**：把"未来依赖是否仍可访问"做成 GPT-5.5 diff-grounded judge（Wilson 95% CI），对比单判与 A/B 校准；端到端在 332 生产会话上，三种规划器（Qwen3.6-Plus / Qwen3.7-Max / GLM-5.1）均跨过 90% no-impact，证伪"必须最强模型当规划器"的常见假设。

## Key Contributions

- **Reframing agent context as runtime objects with lifecycle requirements, not a passive token buffer** — the central insight. Whether an edit is safe depends on whether the object will be a future dependency (URL, table value, editable body, live handle, callback), not on chronological position or message type.
- **Harness-portable `Self-GC` framework**: a small `context-engine` hook (turn/tool-span boundaries + stable IDs + side-channel planner + local rehearsal + safe-boundary commit) rather than a provider-specific agent rewrite. Provider message normalization stays inside the harness; the planner only emits object actions over existing identifiers.
- **Three explicit actions with different recoverability guarantees**:
  - **Fold** — exact payload to recoverable sidecar, compact recovery pointer left (for large bodies that may need idempotent recovery).
  - **Mask** — keep object boundaries, elide repetitive/low-signal middle content (for log-like repeating spans).
  - **Prune** — remove obsolete trace, no recovery guarantee (only when dependency is cleared).
- **Cache-aware commit policy** with explicit cost equation `CommitBenefit ≈ N_future·(C−C′) − L_cache − L_GC`, **mandatory last-turn retention**, and a 0.3 active-view-pruning operating threshold below which the plan stays pending until cache expiry or the next task boundary.
- **Production-grade evaluation methodology**: GPT-5.5 judge as binary no-impact over diff-grounded evidence (URLs, paths, row values, task IDs, editable bodies, source-backed evidence) with Wilson 95% CIs; A/B calibration on disagreement cases; planner-backbone robustness sweep showing framework is not dependent on a single model. Hard Set 43.95% prune → 84.85% no-impact (+15.15 pp over the strongest completed heuristic); Production Suite 31.04–33.98% prune → 91.27–94.58% no-impact across Qwen3.6-Plus / Qwen3.7-Max / GLM-5.1.
- **Online deployment evidence**: account-level split (`o` ≤ email first char → treatment) yields **10–15% average daytime input-token reduction (peaks ~20%)** on covered traffic May 25–30, 2026 — not a randomized quality experiment but a deployment signal consistent with offline gains.

## Methodology

**Harness interface.** Porting Self-GC requires only a small context-engine hook, not a provider rewrite. The harness exposes (1) turn and tool-span boundaries, (2) stable object identifiers assigned by a session-local monotonic allocator, (3) ability for a side-channel planner to read a forked prefix, (4) replay of candidate edits into a local projection, (5) persistence of folded payloads to sidecar storage, and (6) commit of an accepted projection only at a safe turn boundary. Provider-specific message normalization stays inside the harness.

**Indexed context objects.** Two ID schemes:
- `conversation:user:k` for a user request and its following execution span.
- `function:<tool>:n` for tool-level spans editable independently.
Assistant turns are not first-class GC targets (they carry connective text and tool-call envelopes); they are preserved/normalized by the harness when their adjacent objects change. Object IDs are control-plane metadata (`turn_id`, `turn id="function:read:1"`) rather than assistant prose, so the planner targets exact objects while the harness can replay/validate/recover edits without fuzzy text matching.

**Planner contract.** The planner prompt is an **object-action contract**, not a "summarize old context" prompt. Rules (verbatim from the appendix):

1. Exclude the latest visible turn, active instruction files, live handles, editable artifact bodies, and exact evidence that future turns may need verbatim.
2. Prefer tool-level actions when they preserve the same future support more precisely than whole-turn folding.
3. Use `prune` only for obsolete low-signal trace, `mask` for repetitive log-like middle content, and `fold` for large bodies that may require idempotent recovery.
4. Emit XML with a short conversation summary and grouped fold/mask/prune targets over existing identifiers only (see Listing 2).

**Reflective Plan → Rehearse → Commit.** When token pressure, a turn boundary, or policy triggers a governance round, the harness forks the current context prefix and sends it (with planner-only instructions) to the side-channel planner. The planner returns an XML plan; the harness then:
1. Resolves targets, drops invalid or cut-turn edits.
2. Normalizes overlapping actions.
3. Materializes the projected active view locally.
4. Estimates token savings and runs the cache-aware commit check.
5. On safe turn boundary: merges plan with current view, repairs parent lineage to nearest surviving ancestors, persists folded sidecars, normalizes provider messages.
6. Rejected plans never reach the main agent loop.

**Default operating policy.** Offline: 30% compression threshold, mandatory last-turn retention. All edits go through deterministic validation first.

## Results & Highlights

### Hard Set (33 sessions; GPT-5.5 judge; Wilson 95% CI)

| Method | Prune (%) | No-impact (%) | 95% CI |
|--------|-----------|---------------|--------|
| Oldest-turn | 63.45 | 66.67 | [49.61, 80.25] |
| Tool-prune | 67.93 | 69.70 | [52.66, 82.62] |
| Tool-mask+prune | 61.90 | 54.55 | [37.99, 70.16] |
| Hybrid | 69.87 | 57.58 | [40.81, 72.76] |
| **Self-GC (Qwen3.6-Plus)** | **43.95** | **84.85** | [69.08, 93.35] |

**Δ vs strongest completed heuristic: +15.15 percentage points of no-impact at lower pruning.**

### Production Suite (332 sessions; A/B-calibrated judge)

| Method | Prune (%) | No-impact (%) | 95% CI |
|--------|-----------|---------------|--------|
| Oldest-turn | 40.19 | 87.46 | [83.46, 90.60] |
| Tool-prune | 47.76 | 77.71 | [72.93, 81.86] |
| Tool-mask+prune | 47.75 | 80.12 | [75.49, 84.06] |
| Hybrid | 46.28 | 83.73 | [79.38, 87.31] |
| **Self-GC Qwen3.6-Plus** | 31.51 | 92.77 | [89.47, 95.09] |
| **Self-GC Qwen3.7-Max** | 33.98 | 94.58 | [91.59, 96.54] |
| **Self-GC GLM-5.1** | 31.04 | 91.27 | [87.74, 93.85] |

All three planner backbones independently exceed 90% no-impact, supporting the claim that **a mid-tier planner suffices provided the harness enforces deterministic safety checks**.

**A/B calibration on the Hard Set**: single-prompt judge gives 89.76% (oldest-turn) vs 92.47% (Self-GC); on 20 disagreement cases the A/B judge prefers Self-GC 11×, ties 7×, prefers baseline 2× → calibrated 92.77% vs 87.46%.

**Cut-turn safety.** Auditing how often planners try to edit the latest visible user turn (which the safety filter then removes): Qwen3.6-Plus 25/330, Qwen3.7-Max 15/330, GLM-5.1 12/328. "The prompt usually works, but the residual risk justifies mandatory last-turn protection."

### Online deployment

Account-level split (email first character `≥ o` → Self-GC, `< o` → control). Coverage: `context-gc` for interactive chat cleanup, `skill-gc` for long-lived skill-state pruning. Across covered daytime traffic May 25–30, 2026 (08:00–22:00):
- **Average main-agent input-token reduction: 10–15%**.
- **Peak reductions ~20%** (month-end batch API window is volatile and excluded from the headline).
- The paper explicitly notes this is **operational monitoring evidence, not a fully randomized quality experiment**; offline replay provides the controlled future-dependency comparison.

### Failure-mode taxonomy (dependency-centric, not message-type-centric)

The paper argues the loss modes of fixed heuristics are not generic "bad summaries" but **specific missing future dependencies**: exact evidence (URL, value, table row), locators and handles (callback URLs, job IDs), behavioral contracts ("do not replace the table header"), verbatim source text, and live execution state ("explain which rerun finally succeeded"). A chronological or type-based policy has no representation for these distinctions and so deletes many tokens without telling whether the removed span was low-signal trace or the only bridge to the next action.

## Authors & Credibility

- **Affiliation**: All five authors (Xubin Hao, Hongjin Meng, Xin Yin, Jiawei Zhu, Chenpeng Cao) are at **Xiaohongshu (小红书)**, the major Chinese content/community platform — i.e. the system is being tested against real long-horizon agent traces from a production office/browser/shell/web-fetch environment, not a synthetic benchmark.
- **Industrial signal**: The headline numbers come from production traffic and a production account-level split, which is unusual credibility for a context-management paper. The paper is candid about the limits of the online signal (operational, not a randomized quality experiment, planner overhead not billed).
- **Planner-model provenance**: `Qwen3.6-Plus`, `Qwen3.7-Max`, and `GLM-5.1` as planner backbones; `GPT-5.5` is **only** the offline judge, never the planner under test — a clean separation that supports the no-impact numbers.
- **Reproducibility**: Raw production traces contain private user data and cannot be released directly. The paper commits to a future public reproduction package (sanitized fixtures, planner outputs, candidate patches, token estimates, prompt templates, judge outputs, recompute scripts).

## Critical Assessment

- **Novelty**: Genuine reframing of agent context management from "text trimming" to "object-lifecycle control with sidecar recoverability" — the kind of paradigm shift the authors are also careful about not overclaiming. The fold/mask/prune action vocabulary is borrowed from GC and from KV-cache eviction, but the **integration with planned rehearsal, mandatory last-turn retention, and an explicit cache-aware commit formula** is a substantive architectural step beyond vanilla summarization or message pruning.
- **Rigor**: Strong. Same cut points, same token accounting, mandatory last-turn retention, GPT-5.5 no-impact judge with Wilson 95% CIs, **A/B-judge calibration on disagreement cases** to correct single-judge bias, three independent planner backbones, and an explicit dependency-centric failure taxonomy (not just "summaries sometimes lose things"). The Hard Set is intentionally skewed toward browser/shell/web-fetch to stress the system; that's disclosed.
- **Reproducibility**: Partially limited by production data confidentiality (acknowledged). The Hard Set and Production Suite sizes are small (33 and 332 sessions), and 95% CIs are wide on the Hard Set (e.g. Self-GC CI [69.08, 93.35]). The online account-level split is non-randomized and has only n≈1 day-window of evidence. Full binary-payload recoverability tests (can the sidecar actually restore the original?) are missing — judge-based no-impact is a proxy.
- **Significance**: High for practitioners shipping long-horizon agents. The paper offers (1) a vocabulary (objects / fold / mask / prune / sidecar / safe-boundary commit) that maps cleanly to engineering concerns (which spans are needed downstream), (2) a working empirical recipe (30% threshold, last-turn retention, 0.3 active-view pruning commit floor), and (3) a deployable — not just paper — system. Most relevant to engineering teams building office/IDE/code/web agents where exact evidence dominates over narrative coherence.
- **Red flags**:
  - The 0.3 active-view-pruning commit floor is described as an **"operating policy rather than a universal constant"** — guidance, not law; expect to retune per workload.
  - "Code/IDE traces" baselines (AgentDiet, AgentFold, LongCodeZip) are cited as **motivating future comparisons** but not directly run; the strongest claim is against heuristic replay, not against the most recent learning-based compressors.
  - The online signal covers daytime main-agent input tokens only; side-channel planner overhead is **not** part of a matched billed-cost audit. Net-cost and quality claims should be treated as separate from the input-token reduction.
  - The reviewer-bait number is **+15.15 pp on the Hard Set**; on the Production Suite baselines are already doing 77–87% no-impact and the gap is smaller. Headline framing should be read with that nuance.
  - The system **does not eliminate** the need for external memory — the paper explicitly leaves durable knowledge migration into hierarchical memory / retrieval stores to future work.

## Strengths

- **Object-action contract** (not "summarize") directly targets the actual failure mode in agent traces: lost URLs/values/handles, not lost narrative.
- **Recoverability via sidecar** beats final self-summaries on auditability, byte-exact restoration, and "no longer loses editable bodies."
- **Harness-side invariants** (protocol validity, safe-boundary commit, lineage repair, sidecar persistence) let a mid-tier planner be safely deployed behind strong safety checks — relieving the team from "must use the best model for the GC call."
- **Plan → Rehearse → Commit** separation means model mistakes never directly edit the live conversation; only accepted, validated plans do.
- **Cache-aware commit formula** gives a deployable heuristic instead of a magic threshold.
- **Multi-axis evaluation**: offline future-dependency preservation + hard-set stress + scaled production suite + A/B calibration + online deployment signal = unusually comprehensive for a context-management paper.
- **Disclosed failure taxonomy** gives engineering teams something to design tests against (exact evidence loss / locator loss / contract loss / verbatim text loss / live state loss).

## Limitations

- **No public artifacts** — production data confidentiality blocks re-running the exact evaluation; only a future sanitized reproduction package is promised.
- **Small absolute sample sizes** — 33 (Hard Set) and 332 (Production Suite) sessions; both reported metrics include this in their CIs.
- **No end-to-end recoverability test** — judge-based no-impact is a proxy; whether sidecars can actually restore byte-exact folded payloads in adversarial edits is not demonstrated.
- **Online evidence is operational, not experimental** — account split is by email-first-character (clearly disclosed), daytime average only, planner overhead excluded, batch-API window volatile. Net billed-cost and user-quality impacts are not measured.
- **No direct comparison against the most recent learning-based agent compression work** — AgentDiet, AgentFold, LongCodeZip, ACON, ContextBudget, Chroma Context-1, SWE-Pruner are referenced but not run on the same fixtures.
- **Judge sensitivity** is partially addressed via A/B calibration but not systematically stress-tested (model swap, prompt swap).
- **Vision / multimodal** objects are out of scope; only text spans.
- **A 0.3 active-view pruning commit threshold is workload-specific** — not a universal constant.
- **Hard Set is intentionally biased** toward browser/shell/web-fetch workloads, so headline no-impact numbers should not be generalized without re-running on a balanced workload mix.

(Self-GC ↔ 用户兴趣画像)

直击 **Self-Evolving Agents & Automated Research** 与 **ML Reliability foundations**：把"上下文管理"从被动字符裁剪提升为带可恢复性、谱系修复、缓存感知提交的运行时对象生命周期治理，本质上是把一种新型可靠性约束（未来依赖保留 / byte-exact 恢复）引入 agent 推理栈。对正打造长任务 office / IDE / web agent 的工程团队，这套 fold/mask/prune + sidecar + safe-boundary commit 范式比"末尾 self-summary"显著更稳健，也能直接挂到 prompt-cache 计费模型与 KV-cache eviction 的上下游。该工作与**计算可靠性 & 隐藏状态**角度也有共鸣——harness 显式做了对象身份、提供商消息规范化、谱系修复，这正是把"软提示文本"重构成"具名对象"的一次系统级工程改造。建议优先看 Hard Set 的失败模式分类与 Listing 5 排练伪代码。

## Questions for Further Exploration

- Can the GC planner be replaced by a learned task-conditioned policy (or a small in-house classifier) at materially lower inference cost without dropping below 90% no-impact on the Production Suite?
- Does Self-GC compose with **external memory** (Mem0, A-MEM, TiMem, HippoRAG) — i.e. migrate durable knowledge out of the active prompt while fold/mask/prune governs transient trace — without violating the dependency-preservation guarantees?
- What happens under **multi-modal / vision** agent traces where some "objects" are screenshots or generated artifacts? Is the `function:<tool>:n` scheme general enough?
- How sensitive is the cache-aware commit threshold (0.3 active-view pruning) to **provider cache pricing and TTL**? Does the formula generalize across OpenAI/Anthropic/Google/Kimi/Aliyun cache designs?
- The **sidecar recoverability** claim is currently judged via no-impact proxy. What does a direct byte-exact restore test (ask the agent to recover a folded artifact from sidecar by ID, score 0/1) look like — and what's the failure mode?
- On the Production Suite, the **gap over the strongest baseline is only ~7 pp** of no-impact (92.77 vs 87.46). With more sessions and a fully randomized deployment, does that gap survive or compress? Worth knowing before betting production load on it.
- How does Self-GC interact with **trained context compressors** (Composer-style, ACON, ContextBudget, Chroma Context-1) — can the planner contract absorb their outputs as additional actions, or do they need different lifecycle semantics?
- The paper gates commit on `CommitBenefit > 0` from the active agent's perspective. Does a **multi-tenant / shared-prefix** setting (many sessions share prefix-cache) need a coordinated commit scheduler, or is local per-session discipline enough?
- Can **Online Bandit / RL** over the planner prompt's few-shot calibration improve no-impact under shifting production workload mix (e.g. when DOC-heavy vs CODE-heavy traffic swings), without provoking the cut-turn edits that the safety filter currently removes?
- The evaluation only checks **future-dependency preservation**. Does Self-GC change downstream answer quality or user-perceived helpfulness on a randomized holdout, or only token economics?
