Persistent LLM threads pass <spawn><send><recv><stop> directives point-to-point — cutting max context by Θ(N/k) versus fork-join.

Message Passing Enables Efficient Reasoning  →
Xuecheng Liu · Daman Arora · Gokul Swamy · Andrea Zanette
2026-07-01 · Carnegie Mellon University · arXiv:2607.01077
Key Metric
72%

on 25×25 Sudoku — a fine-tuned Qwen3-0.6B-Base beats GPT-5 Pro (20%) and DeepSeek-R1 (0%), at 1017s. Serial and Fork-Join cannot even train at this size.

Why It Matters

First learned protocol for persistent inter-thread communication at inference time — both theory-grounded (O(T·k·M) bound, Theorem 2) and immediately deployable as a zero-shot prompting scaffold.

Critical Edge

The Θ(N/k) advantage collapses to a constant under dense dependencies; zero-shot gains only verified on MoE Qwen3-A3B — replication on dense frontier LLMs (GPT-5 Pro, Claude) is missing.

Methodology

Architecture. N persistent threads, each holding a distinct context and KV cache, decoded concurrently via a standard batched engine (vLLM / SGLang). Threads emit four learnable directives — <spawn>, <send>, <recv>, <stop>. A thin controller runs only at directive boundaries; both <send> and <recv> are stop-tokens at the engine level.

Sudoku. One worker per cell. Parent (ID 0) spawns the grid; each cell queries its 3N² − 2N − 1 row/column/box neighbours, sends its naked-single value, then <stop>s. SFT data is generated by simulating the naked-singles solver in Python.

3-SAT. Distributed DPLL: any child emitting <send> with SAT pre-empts the entire sibling sub-tree — no FJ-style sync barrier.

LongBench-v2. Parent splits context into ≤25K-token chunks, spawns one worker per chunk, then iteratively re-queries specific workers for finer evidence. Workers stay alive — re-querying is free.

Strengths

  • Falsifiable efficiency claim via Theorem 2 + bounded-local-processing assumption.
  • Engine-agnostic controller over vLLM / SGLang — no KV-cache or sampler changes.
  • Three independent task families (Sudoku, 3-SAT, LongBench-v2) all show consistent wins.
  • 0.6B params beats frontier models on 25×25 Sudoku — a ~100× size inversion.
  • Latency cuts (1.7×–2.2×) come with accuracy gains, not at their expense.
  • SFT runs are cheap: <48 H100-hours per task.

Limitations

  • Θ(N/k) gap shrinks to a constant under dense dependencies (e.g. theorem proving).
  • Sudoku is restricted to naked-singles-solvable instances — no real combinatorial search.
  • 3-SAT is evaluated only on random instances, not industrial hardness.
  • Zero-shot MPLM-following verified only on MoE Qwen3-A3B — dense frontier models untested.
  • Single SFT base model (Qwen3-0.6B-Base); cross-scale / cross-family transfer unclear.
  • Static, hand-coded communication topology in Sudoku — topology not learned.
  • Preemption is a hard-coded rule (“first SAT child wins”) — no learned policy.
  • No head-to-head with recent MPI-style / async-decoding schemes (ThreadWeaver, ASPD).

Questions for Further Exploration

  1. Does the O(T·k·M) gap hold empirically on dense-dependency tasks like multi-step theorem proving? What is the smallest k for which the bound is tight?
  2. Can the directive grammar be combined with reinforcement learning so the model learns which neighbours to talk to — rather than relying on a hand-coded topology?
  3. Does preemption transfer to non-Boolean search (shortest-path, MCTS, tool-use loops with early termination)?
  4. How does MPLM interact with KV-cache compression / eviction? Persistent workers grow monotonically — what is the memory-throughput tradeoff vs. transient FJ?
  5. Would a dense frontier model (GPT-5 Pro-class, not MoE) follow MPLM zero-shot? The current evidence is MoE-only — a notable confound.
  6. Can MPLM threads carry tool-use (web, code, calculator) instead of just text messages?
  7. How robust is the prompted-only LongBench-v2 gain to directive-template ablations — is it the protocol, or prompt-format familiarity?
  8. Could broadcast / all-reduce primitives be added as learnable directives to handle dense-dependency tasks where point-to-point is too narrow?
