> **一句话总结：** AutoMem 将 LLM 智能体的"外部文件记忆管理"提升为一种可独立学习的认知技能（metamemory），通过两层自动化外循环——(1) Claude Opus 回溯千步轨迹并迭代改写 agent 代码/提示/文件 schema；(2) 在自身轨迹上以 LoRA 微调一个独立"记忆专员"模型而冻结任务模型——在 Crafter、MiniHack、NetHack 三个长视野程序生成游戏上仅优化"记忆"就让 Qwen2.5-32B 取得 ~2–4× 提升，达到与 Claude Opus 4.5、Gemini 3.1 Pro Thinking 同档的水平。

> **TL;DR:** AutoMem promotes file-system operations to first-class actions in an LLM agent and treats memory management as a separable, learnable skill along two axes — *scaffold* (prompts/schema/vocabulary) and *proficiency* (the parametric ability to use memory well). Two meta-LLM-driven outer loops automate both: Loop 1 revises the agent scaffold from full trajectory review, Loop 2 LoRA-finetunes a dedicated "memory specialist" on the model's own good memory decisions while freezing the task model. On Crafter, MiniHack, and NetHack, a Qwen2.5-32B base agent improves 2–4× in progression rate without ever touching task-action weights, becoming competitive with Claude Opus 4.5 and Gemini 3.1 Pro Thinking.

## 核心贡献
- **概念重构**：将"记忆管理"从工程机制提升为可独立学习、可优化、可评测的认知技能（metamemory 的 LLM 实例化）。
- **统一动作空间**：把文件级操作（read / write / append / upsert / search）与任务动作放在同一动作空间，记忆决策在轨迹中完全可观察——这既隔离了"结构性"瓶颈，也隔离了"使用熟练度"瓶颈。
- **双自动化外循环**：
  - **Loop 1（Scaffold Optimizer）**：Claude Opus 4.6 读取长至 10⁵ 步的完整回合轨迹，像阅读完整 execution log 的代码审查者一样识别失败模式，迭代修改代码、提示、文件 schema、动作词表，每次修改通过"同种子评测提升"门控才被采纳。
  - **Loop 2（Memory Ability Optimizer）**：Claude Opus 4.7 充当"训练引擎"，从 100+ 回合轨迹池中挑选模型自身优质的记忆操作作为监督数据，匹配挑选 LoRA 超参（rank/alpha/epochs），仅微调一个独立的"记忆专员"LoRA 适配器；游戏任务模型权重保持冻结。
- **实证收益**：Qwen2.5-32B-Instruct 上纯记忆优化带来 Crafter 25.0%→47.27%→51.36%（×2.05），MiniHack 7.5%→27.5%→30.0%（×4.0），NetHack 0.42%→1.57%→1.85%（×4.4），与 Claude Opus 4.5、Gemini 3.1 Pro Thinking 同档、显著优于 72B 同族基线。
- **可观测行为变化**：无用动作（卡墙/抖动）率 −32%~−65%、重复 WRITE 率 −68%~−83%、空 SEARCH 率 −13%~−50%、每步输入 token 下降 −3%~−30%；NetHack 地图文件从 138 字符/step 压缩到 6 字符/step（−95%）。

## Key Contributions
- **Conceptual reframing** of agent memory management as an *independently learnable* skill rooted in cognitive-science metamemory, realized through a unified file-system–plus-task-action interface.
- **Two automated meta-LLM outer loops:**
  - *Loop 1 (Scaffold):* frontier meta-LLM acts like a code reviewer over complete long-horizon trajectories (up to 10⁵ steps), iteratively editing prompts, file schemas, and action vocabulary; revisions gated on same-seed evaluation improvements.
  - *Loop 2 (Proficiency):* meta-LLM acts as a *training engine* — it sets selection criteria over a large trace pool, decides data composition, *and* chooses a LoRA configuration matched to that data; only a dedicated "memory specialist" LoRA adapter is trained, while the gameplay model weights are frozen.
- **Empirical wins without touching task behavior:** Qwen2.5-32B-Instruct + AutoMem reaches Claude Opus 4.5 / Gemini 3.1 Pro Thinking tier on three procedurally generated long-horizon games — out-performing Qwen2.5-72B by a wide margin despite being <½ the size.
- **Behavioral evidence that memory skill is real:** downstream game actions (not just memory ops) become substantially more productive (less stuckness, less oscillation) as the memory scaffold improves — even though the optimizer never directly edits gameplay policy.

## Methodology
- **Inner-loop agent.** A single Qwen2.5-32B-Instruct served via vLLM executes one episode of Crafter / MiniHack / NetHack with a directory of files as its external memory. Each step runs two routines:
  - **LOG:** "What is worth recording about what just happened?" → append / create / rewrite a memory file.
  - **PLAN:** "What must I recall to act now?" → search across files, read entries, then commit a world action.
- **Unified action space.** Memory file ops (e.g. `<|APPEND|>`, `<|SEARCH|>`, `<|UPSERT_MAP|>`) live in the same action vocabulary as gameplay actions, so every memory decision is a *traceable* action that the outer loops can observe and optimize.
- **Loop 1 — Scaffold optimization.** Claude Opus 4.6 (`--effort max`) receives the agent code, full per-step traces, and the resulting memory directory; it diagnoses where the scaffold caused memory failures and proposes code edits. A revision is kept only if its mean progression on the same fixed eval seeds (42–51) strictly beats the previous version (≤ 1 same-session retry; then restart from clean session). Concrete edits include: pre-loading reference files (e.g. Crafter's crafting tree, NetHack strategy rules), instrumenting low-HP / repeated-action warnings, switching from append-only to coordinate-keyed deduplication for maps, and blocking impossible actions at the scaffold level (e.g. blocking "sleep next to a monster").
- **Loop 2 — Memory-proficiency training.** Three stages on top of the converged Loop-1 scaffold:
  - **(a) Trace pool.** Run the base agent for 100 (Crafter) / 50×8 (MiniHack) / 50 (NetHack) episodes under disjoint random seeds.
  - **(b) Training engine.** Claude Opus 4.7 sets (i) selection criteria, (ii) data composition, (iii) the LoRA config to match; iterates against eval-trajectory feedback. Post-processing strips non-memory examples and trims action-commit portions of mixed examples.
  - **(c) LoRA finetune (DeepSpeed ZeRO-3, bf16, cosine LR, two GPUs).** Per-environment hyperparameters: Crafter r=256/α=512/epochs=4; MiniHack r=128/α=256/epochs=3; NetHack r=256/α=512/epochs=1. Final training sets: 1,597 / 444 / 800 examples.
- **Two-model deployment.** At inference the memory specialist (LoRA-fine-tuned) handles LOG and the memory-consult portion of PLAN; a handoff passes context to the *frozen* gameplay model for the final world action. This split keeps training signal focused (no dilution by action-format tokens) and preserves the base model's well-formatted action competence.
- **Evaluation.** 10 fixed seeds [42–51]; 10 episodes for Crafter, 5×8=40 for MiniHack (5 per task), 5 for NetHack — matching BALROG defaults. Progression-rate metric (BALROG definition): Crafter = fraction of 22 achievements; MiniHack = fraction of 8 tasks solved; NetHack = dungeon-depth/EXP progression.

## Results & Highlights
- **Memory alone delivers the headline gains.** All task-action weights stay frozen through Loop 1; Loop 2 touches only the memory-specialist LoRA. Net 2×–4× progression on every game (Table 1):

  | Agent | Crafter (%) | MiniHack (%) | NetHack (%) |
  |---|---|---|---|
  | Qwen2.5-32B v0 (base file-memory) | 25.00 ± 5.50 | 7.50 ± 4.16 | 0.42 ± 0.37 |
  | + Loop 1 (scaffold) | **47.27 ± 2.05** | **27.50 ± 7.06** | **1.57 ± 0.35** |
  | + Loop 2 (memory training) | **51.36 ± 3.81** | **30.00 ± 7.25** | **1.85 ± 0.44** |
  | Claude Opus 4.5 (frontier ref.) | 49.5 ± 3.1 | 27.5 ± 7.1 | 2.0 ± 0.5 |
  | Gemini 3.1 Pro Thinking | 55.0 ± 6.4 | 27.5 ± 7.1 | 2.6 ± 0.3 |

  The 32B AutoMem agent beats Qwen2.5-72B-Instruct by a wide margin on all three, demonstrating that well-managed memory is higher-leverage than model scale on long-horizon tasks.
- **Loop 1 ≈ Loop 2 in magnitude.** The proficiency-training loop stacks a clean additional ~9–18 % relative gain (e.g. Crafter 47.27→51.36, MiniHack 27.5→30.0) without disrupting the scaffold's gains — strong evidence that the two axes are nearly orthogonal.
- **Behavior gets visibly cleaner (Fig. 4).** Across Crafter/MiniHack/NetHack after Loop 1 only:
  - Unproductive-action rate (stuck + oscillation): **−32% to −65%**.
  - Repeat-WRITE rate: **−68% to −83%**.
  - Empty-SEARCH rate (searches returning nothing): **−13% to −50%**.
  - Input tokens per step: **−3% to −30%**.
  - NetHack map file size: **138 → 6 chars/step (–95%)**.
- **Loop 2 internalizes a "consult-before-write" discipline (Table 2).** LOG-phase writes-per-SEARCH drops sharply: Crafter 0.84→0.39 (−54 %), MiniHack 2.89→0.82 (−72 %), NetHack 4.66→1.31 (−72 %).
- **Qualitative traces (Fig. 6) make the story concrete.** In MiniHack Corridor-R3 (requires navigating branching corridors), neither v0 nor the loop-1 scaffold solves it; the trained specialist does (prog 0 % → 0 % → 100 %). In NetHack, the base agent dies at XP level 1; the loop-1 scaffold survives thousands of steps reaching XP level 2; the trained specialist reaches XP level 4 with far longer survival.

## Critical Assessment
- **Novelty:** High and well-positioned. Reframing memory as a separable, learnable skill with a *unified* action space is a clean conceptual move; the dual-loop decomposition (structure + proficiency) is more disciplined than most "self-improving agent" papers. Distinguishes itself from sibling work (MemSkill, MeMo, MemEvolve, Memory-R1, Agentic Memory) by jointly optimizing both axes.
- **Rigor:** Strong overall.
  - Evaluation uses 10 fixed seeds with mean ± SE; baseline and AutoMem use *the same updated BALROG harness* (autopickup, fallback actions) — a careful choice that removes confounds.
  - Training seeds are explicitly *disjoint* from eval seeds → no train/eval contamination possible.
  - The "iteration" in both loops is **gated by measured progression on the same seeds** — a proper hill-climbing gate, not vibes.
  - What's missing/weak: no formal statistical-significance tests across scaffolds vs. baselines; standard errors are reported but CIs or paired tests are not. Only the single base model (Qwen2.5-32B-Instruct) is used — claims of "general" separability need cross-base-model evidence.
- **Reproducibility:** Mostly good but partly closed.
  - Code, prompts, schemas: **released** on GitHub.
  - Game environments (BALROG): public.
  - Meta-LLMs (Claude Opus 4.6 / 4.7): closed, paid API — full reproduction requires non-trivial spend.
  - The **judgement of "good memory decisions"** in Loop 2 is opaque: it's the meta-LLM applying criteria it engineered, which inherently embeds Claude's biases.
- **Significance:** High for the self-evolving-agents and long-horizon-agent communities. Demonstrates an alternative to raw RL/agentic-scale-up: *re-targeting an objectively meaningful, isolated sub-skill (memory) can close the gap to frontier closed models for 32B-class open weights.* Methodologically a useful primitive for AutoResearch-style pipelines.
- **Red flags / caveats:**
  - All three environments are *game-like and procedurally generated* — strong, but web/office/coding-tool generalization is asserted, not demonstrated.
  - Per-environment *separate* scaffolds and *separate* LoRA specialists were optimized (not one shared specialist across all games) — limits claims of a single "memory skill."
  - Comparisons vs. frontier closed models rely on the public BALROG leaderboard and identical harness settings — fair, but those settings may slightly favor procedural-game-savvy systems.
  - "LLM-as-judge" criteria in Loop 2 are themselves a black box; if the meta-LLM has consistent blind spots for, say, *reading-then-acting* patterns, the trained specialist will reproduce them.
  - The "MetaMem" / "EvolveMem" / "MeMo" / "Memory-R1" / "Agentic Memory" / "MemSkill" cluster is moving quickly; differentiation here is real but the field will re-evaluate quickly.

## Strengths
- A **clean, falsifiable claim**: optimizing memory *alone* yields 2–4× gains, with task weights frozen. The architecture explicitly enforces this isolation.
- **Behavioral diagnostics** that go far beyond the headline progression rate (stuckness, redundant writes, empty searches, input tokens) — these are exactly the metrics a memory-focused paper should report, and they corroborate the headline.
- The **gating mechanism on identical eval seeds** prevents scaffold drift and makes the meta-loop auditable.
- **Open code + open weights base model + standard harness** — strong reproducibility surface area given closed meta-LLM dependencies.
- Concrete ablation showing **Loop 1 ≈ Loop 2** in magnitude, cleanly motivating both axes rather than burying the smaller contributor.

## Limitations
- **Domain scope:** Three procedurally-generated games. No experiments on real long-horizon tool-use (browsing, software engineering, research assistants), so transfer claims are speculative.
- **Per-environment optimization:** Each game gets its *own* scaffold and its *own* memory-specialist LoRA. There is no evidence yet for a single transferable memory skill across domains.
- **Episodic memory only:** Memory resets every episode. Persistent / cross-episode memory is explicitly flagged as future work.
- **Loop 2 is offline SFT on self-traces**; it does not use online RL, and inherits the base model's biases via "good" memory decisions filtered by another LLM.
- **Compute / dependency footprint:** Loop 1 and Loop 2 both require repeated frontier-LLM trajectory review (Claude Opus `--effort max`), which is expensive and time-consuming — practical adoption will favor API-based teams over small labs.

## Relevance
直接命中"自演化智能体与自动化研究"主轴：将"记忆"拆为 *结构* 与 *熟练度* 两个可独立优化的子轴，并通过"元 LLM 回溯完整轨迹 + 自动修订代码 / 自动筛选自我训练样本"完成闭环，是当前 AutoResearch 范式的一个具体、可复现的实例。同时其"长视野场景下输入 token 冗余下降、可观察动作抖动率下降"也对 *Inference Evaluation* 与 *Hidden States / 外部记忆可靠性* 有间接价值——它表明一个 32B 开源模型通过把"如何用记忆"训出来，可以在不增加权重参数的情况下逼近前沿系统。

## Questions for Further Exploration
- Does the **separability claim (memory ≠ task actions) hold on real tool-use** (web agents, code agents, OS agents), or does it break down when memory and task actions are tightly interleaved?
- Can Loop 2 be replaced or augmented by **online RL** (RLOO/GRPO/PPO) against a memory-quality reward, to remove the dependence on a closed meta-LLM filter?
- What is the **transfer curve** for a memory specialist trained on Crafter → applied to MiniHack / NetHack / web tasks? Does "consult-before-write" generalize?
- Could a **single scaffold + single memory specialist** be trained jointly across all three games via curriculum, and would it match per-game specialists?
- How robust is the meta-LLM reviewer to **adversarial or noisy trajectory summaries**, and what is the failure mode when the reviewer is wrong?
- How does **per-step input context reduction (–3% to –30%)** interact with hidden-state interpretability analyses — does the leaner memory change what the residual stream "needs" to represent?
- Could Loop 1's iteration count (typically 2–5) be **predicted from task statistics** (episode length, action-space size) so the method is auto-budgeted for new environments?
