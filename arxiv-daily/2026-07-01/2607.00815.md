> **一句话总结：** LRAT-Catcher 是一个独立、通用的 Lean 4 工具，通过 reflection 机制运行 Lean core 中已形式化验证的 LRAT 检查器（默认 native_decide 编译为本地代码，可选 +kernel 模式仅信赖内核），将 SAT 求解器的 LRAT 证书作为定理导入，从而绕开 Mathlib 显式 proof-term 导入的内存爆炸问题，并支持 cube-and-conquer 拆分运行（256/1024 个 leaf）在 Lean 内部组合为单一 UNSAT 定理，最终在 Lean 4 中建立了 Schur 数 S(4)=44 与 Ramsey 数 R(4,4)=18。

> **TL;DR:** LRAT-Catcher is a standalone Lean 4 tool that turns any DIMACS formula plus LRAT certificate into a Lean theorem by running Lean's verified core LRAT checker via reflection (native by default, kernel-only as an option). It scales where Mathlib's explicit proof-term import runs out of memory (peak ~8.9 GB vs. 95+ GB on the same benchmarks), composes cube-and-conquer runs end-to-end inside Lean (cover completeness checked as another LRAT proof, not a trusted combinator), and uses verified encodings to lift CNF-level UNSAT to the original combinatorial claim — delivering Lean theorems for Schur S(4)=44 and Ramsey R(4,4)=18.

## 核心贡献
- 提供一个独立、通用的 LRAT 证书导入工具（`lrat_reflect`、`lrat_reflect_cnf`、`lrat_decide`、`lrat_cover_reflect`、`lrat_cover_reflect_cnf`），将 DIMACS 公式与 LRAT 证明作为 Lean 定理导入。
- 通过 Lean 4 的 reflection 机制运行 Lean core 中已形式化验证的 LRAT 检查器 `Std.Tactic.BVDecide.LRAT`（Böving et al.），并以 native_decide 编译为本地代码运行，避免显式 proof-term 构造带来的内存爆炸；同时提供 `+kernel` 模式以最小化可信基（仅依赖 Lean 内核及三个标准公理）。
- 在 Lean 内部完整支持 cube-and-conquer 范式：通过 Theorem 5.1（每叶子 UNSAT + 立方覆盖完备 ⇒ 整体 UNSAT），将所有 per-cube 反驳与一份覆盖完备证书（本身也是 LRAT 证明，无可信组合算子）合并为一条单一 UNSAT 定理。
- 利用并行 leaf 编译（`lratcatch-cover-parallel`）将单次大运行按 cube 拆分为独立 Lean 模块，使峰值内存受限于最大单个 leaf 而非总体积。
- 通过已验证的 CNF 编码（Schur：`encodeK k n`；Ramsey：`encode n s t`）及其健全性引理，将 CNF 层 UNSAT 提升回原始组合问题，编码错误无法通过健全性证明被遮蔽。
- 在 Lean 4 中建立了 Schur 数 S(3)=13、S(4)=44 与 Ramsey 数 R(3,3)=6、R(4,4)=18 四个定理，其中 S(4) 与 R(4,4) 依赖 256 / 1024 个 leaf 的 cube-and-conquer 运行。

## Key Contributions
- A standalone, general-purpose Lean 4 import tool for arbitrary DIMACS formulas paired with LRAT certificates, usable as a reusable certificate-import bridge rather than a one-off proof step.
- Reflection-based invocation of Lean's formally verified core LRAT checker (full RAT rule, textual + binary parsing) compiled to native code, with an opt-in `+kernel` mode that restricts the trusted base to the Lean kernel plus three standard axioms.
- Full in-Lean cube-and-conquer composition: a single composition theorem (Thm 5.1) plus a cover-completeness LRAT certificate replaces any trusted cover combinator.
- Parallel leaf compilation that bounds peak memory to the largest single leaf rather than the aggregate certificate volume.
- Verified CNF encodings for Schur and Ramsey problems with soundness lemmas that lift UNSAT to the original combinatorial claim.
- Empirical evaluation against Mathlib's `lrat_proof` and the external HOL4-verified checker `cake_lpr` on pigeonhole, Schur, and Ramsey instances — culminating in verified Lean theorems for S(4)=44 (256 leaves, 316 s elaboration, 4.6 GB peak) and R(4,4)=18 (1024 leaves, 83 min import, 188 GB peak, 51.6 GB compiled object).

## Methodology
- **Trusted base hierarchy.** Four import workflows (Mathlib `lrat_proof`, external `cake_lpr` + axiom, LRAT-Catcher `+kernel`, LRAT-Catcher native) are characterized by their trusted bases — kernel + 3 axioms, kernel + 3 axioms + `cake_lpr` + asserted axiom, kernel + 3 axioms, kernel + 3 axioms + Lean compiler (`native_decide`).
- **Reflection-based checking.** The formally verified checker `checkLrat` is evaluated either by `decide +kernel` (slow, minimal trusted base) or `native_decide` (compiled, scales). Kernel mode requires kernel-reducible replacements for constructs opaque to the kernel (well-founded recursion, hash-based clause storage); equality to the originals is proved, so the soundness theorem transfers unchanged.
- **DIMACS parsing.** A lenient, auditable parser `parseDimacs` is paired with an elaboration-time validator that rejects malformed lines / missing terminator, guaranteeing that the theorem statement commits to exactly the formula in the file.
- **CNF-to-problem lifting.** An encoding-soundness lemma proved once by hand shows every model of the original combinatorial structure satisfies the CNF, so UNSAT on CNF implies UNSAT on the original problem. A faulty encoding cannot be masked by the certificate.
- **Cube-and-conquer composition.** A Boolean function checks every leaf certificate against its leaf and the cover certificate against the negated-cubes formula; its soundness theorem yields `F.Unsat`. Per-leaf checking happens in separate Lean modules so peak memory is bounded by the largest leaf.
- **Solver integration.** Defaults to CaDiCaL 3 with `--lrat --no-binary --no-factor` (factoring in CaDiCaL 3 introduces extension variables whose certificates the verified checker rejects). Solver overridable via `LRATCATCHER_SOLVER` env var.
- **Evaluation harness.** Lean v4.30.0, CaDiCaL 3.0.0, `cake_lpr`, LRAT-Catcher v0.1.0 on one machine for comparable timing.

## Results & Highlights
- **Schur S(3) ≤ 13:** Native reflection imports a 22 KB certificate in 13.8 s; kernel reflection takes 245 s at 28.5 GB peak (already extreme for the size).
- **Pigeonhole ladder (`php(m+1, m)`):** Native reflection completes all sizes up to `php(10, 9)` (63 MB) in 8–30 s, peaking near 2 GB. `lrat_proof` reaches 36.6 GB on `php(9, 8)`, peaks at 95.7 GB on `php(10, 9)`, and is OOM-killed by the OS on the two largest certificates — about 1500× the certificate size.
- **Schur S(4) = 44:** Split into 256 leaves totaling 777 MB of LRAT text; assembled via `lrat_cover_reflect_cnf` in 316 s of elaboration at 4.6 GB peak. The monolithic S(4) certificate (628 MB) is 77 s under native reflection but is OOM-killed under `lrat_proof` (peaks at 96.6 GB).
- **Ramsey R(4, 4) = 18:** Monolithic certificate generation did not finish — solver was still writing a 47 GB partial file after 16 hours. Cube-and-conquer pipeline solved + trimmed + imported 1024 leaves in 2.8 hours total; final import alone: 83 min at 188 GB peak, producing a 51.6 GB compiled object. `cake_lpr` independently verified every leaf.
- **Memory profile (Fig. 1):** Native reflection stays roughly 14–35× the certificate size in a normal Lean process; `lrat_proof` grows with the embedded formula and is OOM-killed near 100 GB on the largest certificates.
- **Trusted-base accounting (via `#print axioms`):** Each theorem depends only on Lean kernel + 3 standard axioms + two `native_decide` axioms (one for the cover check, one for the lower-bound witness), or one `native_decide` per module under parallel build.
- **Comparison vs. `cake_lpr`:** `cake_lpr` is fastest on every instance (e.g., 27.4 s vs. 77 s native on S(4)) but returns only a verdict, not a kernel-checked Lean theorem.

## Critical Assessment
- **Novelty:** Reflection-based verified-checker invocation is not new (Gallicchio et al.'s Keller proof already did this), but LRAT-Catcher is the first to expose it as a reusable, general-purpose import bridge over arbitrary DIMACS + LRAT pairs, with full RAT support and a parallel cube-and-conquer pipeline.
- **Rigor:** The verified checker, the composition theorem, and the verified encodings together push most of the argument onto formal ground. Remaining trust sits in CaDiCaL's certificate production (and the Lean compiler under native mode) — standard but real boundaries. The encoding-soundness lemma is proved by hand rather than generated, so a faulty encoding could slip past; the author mitigates this by separating the encoding proof from the certificate check.
- **Reproducibility:** Tool is open-source at `github.com/leansolving/lrat-catcher`; pinned versions (Lean v4.30.0, CaDiCaL 3.0.0, v0.1.0) are given; benchmark scripts and machine specs are implicit but the artifact is sufficient for independent replication.
- **Significance:** Materially narrows the gap between industrial-strength SAT solving and interactive theorem proving in Lean, enabling formal certification of combinatorial results (Ramsey, Schur, Keller, Empty Hexagon) whose proofs exceed human or ITP-native reasoning capacity. The R(4,4)=18 result in Lean is, to the author's claim, the largest such formally certified theorem to date.
- **Red flags:** Single-author paper; `native_decide` adds the Lean compiler to the trusted base, which some formalists may resist for "axiom-clean" developments; the R(4,4) compiled object (51.6 GB) and 188 GB peak memory show the approach still hits real resource limits; RAT-rule certificates still require `--no-factor` on CaDiCaL 3, hinting at solver-side compatibility friction; the four-way benchmark in Table 1 uses RUP-only certificates, leaving RAT-only behaviour only empirically characterized via the S(4) and R(4,4) cases.

## Strengths
- Reuses Lean's already-verified LRAT checker with full RAT support, avoiding re-implementation risk.
- Native reflection keeps peak memory roughly an order of magnitude above the certificate, in contrast to `lrat_proof` which grows with the embedded formula.
- Cube-and-conquer composition is end-to-end inside Lean: cover completeness is a checked LRAT proof rather than a trusted combinator.
- Parallel leaf build caps per-process memory at the largest single leaf, decoupling memory from the number of cubes.
- Verified CNF encodings + soundness lemmas keep the CNF-to-problem bridge trustworthy and auditable.
- Open-source artifact with a companion PBLean announcement signals ecosystem continuity.

## Limitations
- Trusts the SAT solver's certificate as a black box; a buggy or compromised solver could produce a wrong theorem (the verified checker would still catch certificate-level errors, but not solver misbehaviour).
- `native_decide` mode adds the Lean compiler to the trusted base; kernel mode is too slow for non-trivial certificates, leaving most users on the larger trusted base.
- The Schur monolithic certificate (628 MB) and R(4,4) (1024 leaves) still need 8.9 GB and 188 GB peak memory respectively; the approach is scalable but not cheap.
- CaDiCaL 3's factoring must be disabled (`--no-factor`) because its extension-variable certificates are rejected by the verified checker — a solver-side compatibility caveat that may recur with future CaDiCaL releases.
- Single-author scope; broader workload characterization (e.g., SAT modulo theories, industrial benchmarks beyond pigeonhole) is not provided.
- Kernel-reducible replacements for opaque constructs in the core checker add non-trivial engineering surface that must be kept in sync with Lean core updates.

## Relevance
与用户研究兴趣的直接相关性较低：本工作属于形式化验证 / SAT 求解桥接领域，不涉及 LLM 训练稳定性、推理评估、隐藏状态可解释性、计算可靠性 / 精度错误、模型系统卡、静默数据损坏、ML 可靠性基础或自进化智能体等方向。可作为"通过已验证外部检查器建立高可信计算结果"这一可靠性方法学的范例参考——其"可信基最小化 + reflection 编译折中"的取舍思路与计算可靠性研究中"在严格验证与可用性之间权衡"的关切有方法学上的类比价值。

## Questions for Further Exploration
- How does the trusted-base axiom count (one `native_decide` per module in parallel build vs. one for the monolithic check) compare with proof-assistant community standards, and could `cake_lpr`-style post-hoc checking reduce it further?
- Will CaDiCaL 4 (or any future solver) produce LRAT certificates that the verified Lean-core checker accepts without `--no-factor`-style workarounds, or should the checker be extended?
- Can the reflection + verified-checker pattern be lifted to VeriPB (the companion PBLean direction) and to SMT/ILP certificates, producing a uniform Lean-import framework?
- For very large cube-and-conquer runs (10⁴–10⁵ leaves), does the per-module `native_decide` axiom count become a usability or audit problem?
- Is there a path to certifying the encoding soundness lemma itself (currently proved by hand) by reflection or by an external checker, eliminating the last hand-written component?
- Could a similar architecture be applied to import certificates from proof-producing neural or LLM-based provers, where the certificate format and the "trusted base" question look quite different?
