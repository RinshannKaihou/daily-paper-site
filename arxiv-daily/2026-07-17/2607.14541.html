Key Finding

The best frontier agent hits just 10.7% of the GPU roofline — and none matches the deployed production kernel.

Across 30 operators / 440 shapes sampled live from Alibaba inference clusters (>10k XPU-A & H20 cards, ~20 deployed models), six frontier coding agents targeting the unfamiliar FlyDSL fall far short of deployable — exposing a measurable "correctness illusion" where high pass rates ride on PyTorch fallbacks, not real kernel authorship.

Methodology

how the benchmark is built

Benchmark construction Atrex-Bench

  • A non-intrusive tracer attaches to live serving on XPU-A & H20 clusters — no relaunch, no app instrumentation — fusing Python-frame context with vendor runtime traces.
  • Raw kernel records collapse to 30 operator families across vLLM / SGLang / AITER / RTP-LLM; 440 hot shapes shipped per operator.
  • Each problem = a five-file contract: reference.py, input.py, shapes.json, plus hidden metadata.json + roofline.json — a structural anti-hack boundary.
  • Target DSL = FlyDSL, deliberately absent from training corpora, to probe in-context learning vs. recall.

Scoring & gating

  • Per-shape roofline bound derived from reference semantics: T = max(F/P, M/β) — fixed across candidates, never the candidate's own profile.
  • Importance weighting: each operator weighted by fleet card-hour share (top-5 ≈ 64%; unified_attention alone = 36.1%) → aggregate Sagg = Σ wᵢ·Sᵢ.
  • Three-stage gate: compile (AOT MLIR artifact) → correctness (K=5 seeds, atol=1e-2 / rtol=5e-2) → perf (sandbox, freq-lock, L2 flush, per-shape subprocess isolation).
  • Panel: 6 frontier agents (Opus 4.7, GPT-5.5, Qwen3.7-Max, Kimi-K2.6, GLM-5.1, DeepSeek-V4-Pro) + 2 baselines (torch.compile, deployed production kernel).

  • Opus 4.7 is the exception: 24 correct vs. 26 FlyDSL-dominant — it actually writes the DSL.
  • Failure taxonomy over 683 units: 53.4% compile, 37.6% silent numeric mismatch, 3.5% correct-but-slow.

AKA optimization agent · controlled case study vanilla → augmented

  • AKA = profile-driven measure-revise loop (8 stages) + optimization dropout (partial restart) + layered GPU Wiki (298 ref kernels, 244 docs).
  • On attention_forward: Qwen3.7-Max S 0.06 → 0.40 (6.7×), 0.17× → 1.11× production; Opus 4.7 S 0.28 → 0.42, 0.78× → 1.17×.
  • On chunk_gated_delta_rule_state: 0% → ~100% FlyDSL; Qwen 0.03× → 1.20× prod, Opus 0.04× → 1.31× prod.
  • All six augmented kernels overtake the deployed hand-tuned production kernel — but only at one shape each (see Critical Edge).

Strengths & Limitations

two-sided read

Strengths what holds up

  • Directly attacks the three production-relevance axes others miss: which shapes (skew vs. grid), which kernels matter (weighted vs. averaged), how good is enough (roofline vs. mutable baseline).
  • Structural anti-hacking: hidden provenance/roofline + the FlyDSL-adoption metric turns invisible fallback passes into a recorded number.
  • FlyDSL target cleanly removes the "model just remembers Triton/CUDA" confound.
  • Failure taxonomy over all 683 failing units is unusually thorough and reusable.
  • Memory- vs. compute-bound split & "generation volume ≠ quality" are genuinely diagnostic beyond the headline.
  • Both artifacts open-sourced with versioned snapshots & a planned refresh cadence.

Limitations where it's thin

  • AKA evidence is 3 ops × 2 models × 1 shape — not a benchmark-wide ranking; no per-component ablations, so attribution is unclear.
  • Single timed run per (op, shape) — no variance reported, despite the 1.06–1.31× wins being narrow.
  • Production comparisons use shapes AITER wasn't tuned for (e.g. the 65,556-token attention where AITER itself only hits ~36% of roof).
  • Evaluated on XPU-A only; H20 is traced for sampling but never used as an eval target.
  • Inference-only: backward / optimizer-step kernels out of scope.
  • Only FlyDSL exercised this release; Triton / Gluon / CuteDSL deferred. Forward-dated model names & arxiv ID suggest a draft.
