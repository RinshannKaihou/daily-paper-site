> **一句话总结：** LQCDMaster 是一个面向格点 QCD 的工具增强、技能引导型智能体，结合规划—执行两阶段架构、专家标注技能与确定性的 `generate_einsum` Wick 收缩工具，在 70 项前沿任务基准上将 63 项精确复现专家实现（机器精度），并将每个观测量的实现时间从小时级压缩到分钟级（最高 54× 加速），还首次用对角 Wilson 线几何计算了介子光锥分布振幅。

> **TL;DR:** LQCDMaster is a tool-augmented, skill-guided LLM agent that converts natural-language lattice QCD tasks into executable PyQUDA workflows; on a 70-task benchmark (C24P29 ensemble, 5 observable classes) using GPT-5.4 it exactly reproduces expert-written references in 63/70 tasks at machine precision (90.0% accuracy, with baryon 3pt being the weakest class at 66.7%), cuts per-observable implementation time by 17×–54× (e.g. baryon 3pt from 8h to 10.9 min), and—against the only general-agent baseline—Claude Code fails to produce a single executable script for any of the 20 simplest local 2pt tasks.

## 核心贡献

- **降低格点 QCD 门槛：** 把自然语言研究请求（如"用算子 ūγ5d 计算π介子两点函数"）直接转换为可审查、可执行的 PyQUDA 计算工作流（测量脚本、SLURM 作业脚本、执行日志、数值输出），让物理学家无需手写实现代码。
- **构建首个全面的 LQCD 编码基准：** 70 项前沿任务覆盖五类可观测量——20 个局部两点函数、10 个非定域 Wilson 线两点函数、12 个 Wilson 圈、13 个介子三点函数、15 个重子三点函数，所有任务均要求 Wick 收缩、γ 矩阵代数、规范链传输、源—汇运动学等正确性。
- **机器精度级专家复现：** GPT-5.4 主干下 63/70 任务以 |∆|≲10⁻¹² 精度复现专家实现（90.0%），其余 3 项为可修正的符号约定不一致，仅 4 项（全部为重子三点函数）未通过。
- **确定性 `generate_einsum` 工具：** 将代数最脆弱的"收缩字符串生成"从 LLM 自由生成空间移交给枚举 Wick 配对的确定性引擎，从根上消除单索引/符号错误导致的"可运行但错误"的可观测量。
- **首次实现若干非标准可观测量：** 首次用对角 Wilson 线几何计算介子光锥分布振幅（LCDA），并自动化多强子谱学（质子、氘核、氚核、超子 Λ、超氘核 pΛ、超氚 pnΛ）中随强子数指数增长的 Wick 收缩（2→36→2880 项）。
- **跨主干模型稳健性：** 用 DeepSeek-V4-Pro 替换 GPT-5.4 后整体仍达 80.0%（56/70）机器精度复现，证明架构本身而非单一模型是性能来源。

## Key Contributions

- **Lowering the barrier to LQCD research:** Converts natural-language LQCD research requests into reviewable, executable PyQUDA workflows (measurement script, SLURM submission script, test/full-run scripts, execution logs, and numerical outputs), letting physicists start research without hand-writing implementation code.
- **First comprehensive LQCD coding benchmark:** 70 frontier tasks across five observable classes—local 2pt, nonlocal Wilson-line 2pt, Wilson loops, meson 3pt, baryon 3pt—deliberately probing spin–color contractions, γ-matrix algebra, gauge-link transport, source–sink kinematics, sequential-source conventions, MPI data layout, and normalization.
- **Machine-precision expert reproduction:** With GPT-5.4, 63/70 tasks reproduce expert references at |∆|≲10⁻¹² (90.0%); the 7 remaining cases are 3 single-global-sign convention mismatches (recoverable after a fixed sign transform) plus only 4 true algebraic failures, all in baryon 3pt.
- **Deterministic `generate_einsum` tool:** Removes the most algebraically brittle component—Wick-contraction einsum strings—from the LLM's free-form output by delegating enumeration of valid Wick pairings, sign/topology conventions, and spin–color label construction to a deterministic engine; the generated PyQUDA script no longer depends on the tool at runtime.
- **New-observable exploration enabled by the agent:** First lattice computation of meson light-cone distribution amplitudes (LCDAs) using a diagonal Wilson-line geometry, plus automated multi-hadron spectroscopy (p, deuteron, triton, Λ, pΛ, pnΛ) where the number of Wick contractions grows as 2→36→2880 with hadron number.
- **Architecture-level cross-backbone robustness:** Re-running the full benchmark with DeepSeek-V4-Pro instead of GPT-5.4 still achieves 56/70 (80.0%) exact matches with the same skills and tools, showing the architecture—not a single model—is the source of reliability.

## Methodology

LQCDMaster is built as a tool-augmented, skill-guided, domain-specialized scientific computing agent that converts a natural-language LQCD request into reviewable PyQUDA code. The design deliberately separates **planning from execution** (Fig. 4):

**Planning stage.** The planner agent runs a *solve–critique–rewrite* loop: it performs physical analysis and proposes a detailed technical plan (ensemble, observables, hadrons, propagators, correlator measurements, expected output format); an independent critique checks whether observables, propagators, and settings are reasonable and consistent, and the plan is revised if any flaw is found. A human checkpoint then lets physicists review and optionally interact with the plan before it is passed downstream—acknowledging that the agent cannot yet guarantee correctness for every observable.

**Execution stage.** The executor realizes the approved plan, generating PyQUDA measurement code and SLURM submission scripts. It is guided by PyQUDA-related skills (to prevent invented API usage) and is verified via static analysis and LLM critique before submission. Artifacts per stage are summarized in their Table 4 (planner: computation plan + concise summary; executor: reliable PyQUDA script, test script, full-run script).

**Deterministic `generate_einsum` tool (Table 5).** The central specialized tool: instead of letting the model emit contraction strings directly, the executor requests a contraction in physics terms (meson 2pt, baryon 2pt, multi-hadron 2pt, meson 3pt sequential-source sink block, baryon 3pt sequential-source sink block). The tool enumerates valid Wick pairings, applies sign and topology conventions, constructs spin–color labels, and returns a ready-to-insert contraction specification. This converts the failure mode from "silently wrong observable" to "high-level judgment is on the model, exact symbolic construction is on deterministic code."

**Expert-annotated, stage-routed skills (Table 6).** Three skills: `lqcd-physics-correlator` (planner; hadronic operators, Wick conventions, flavor, γ5-hermiticity), `pyquda-tool` (executor; solver setup, MPI gather, contraction usage, common code-gen mistakes), and `pyquda-gauge` (planner/executor; gauge loading, Wilson/Polyakov loops). Skills and tools are jointly stage-routed so the agent's knowledge is aligned with the tools valid for the selected physics workflow.

**Evaluation protocol.** Generated scripts must (a) be syntactically valid and complete test execution, and (b) be validated by direct numerical comparison with a hand-written reference implementation on the same observable, ensemble, and kinematic setup, since in LQCD a syntactically valid, runnable program can still measure a wrong observable due to incorrect spin–color contraction, flavor assignment, or fermionic sign. Match = numerical agreement at machine precision |∆|≲10⁻¹²; Convention mismatch = single global sign/phase transform recovers full correlator; Failure = anything else.

## Results & Highlights

- **Overall benchmark accuracy:** On the 70-task C24P29 benchmark (CLQCD ensemble, a=0.105 fm, volume 24³×72, m_π≈290 MeV), LQCDMaster with GPT-5.4 reproduces expert references in **63/70 = 90.0%** of tasks at machine precision |∆|≲10⁻¹²; the remaining 7 are 3 convention mismatches plus 4 true failures (Table 2).
- **Per-class accuracy (Table 2):** Local 2pt **100% (20/20)**, Nonlocal 2pt **80.0% (8/10)**, Wilson loop **100% (12/12)**, Meson 3pt **100% (13/13)**, Baryon 3pt **66.7% (10/15)** — baryon 3pt is the dominant failure class.
- **The 4 unresolved failures are all in baryon three-point functions** involving specific transition-current combinations: p→p vector current, Λ→Λ vector and axial currents, and Λ_b→Λ_c vector current. These passed preliminary static checks and test execution but failed numerical comparison.
- **Implementation-efficiency speedups (Fig. 1):** Per-observable human-expert time vs. LQCDMaster time — Wilson loop 1h→3.5 min (**17×**), Local 2pt 2h→4.0 min (**30×**), Nonlocal 2pt 4h→4.5 min (**54×**), Meson 3pt 4h→6.2 min (**38×**), Baryon 3pt 8h→10.9 min (**44×**). Speedup grows with observable complexity.
- **Cross-backbone robustness (Table A1):** Replacing GPT-5.4 with DeepSeek-V4-Pro (same skills/tools/pipeline) yields **56/70 = 80.0%** exact matches; Wilson loop stays at 100%, baryon 3pt improves to 86.7% (13/15), but meson 3pt drops sharply to **53.8% (7/13)** — failures traced to incorrect quark-flavor assignments in model-generated task parameters, indicating semantic validation still depends on backbone in-domain knowledge.
- **General-agent baseline (Appendix B):** Claude Code, given the same tasks plus PyQUDA documentation but no specialized skills/tools, **failed to yield executable scripts for any of the 20 local 2pt tasks** (the simplest class) — establishing that a guidebook alone is insufficient and the deterministic tools + domain skills are the load-bearing components.
- **Failure-mode taxonomy (Table 3):** Routine errors (2 einsum-label mismatches, 1 array-shape issue in nonlocal displaced-field workflows, 2 output-format/naming deviations, 1 gauge-context lifetime error) were all detected and auto-corrected by static analysis/critique/test execution; only the 2 sequential-propagator sign/phase mismatches and the 4 baryon 3pt failures required direct numerical reference comparison.
- **First-of-its-kind physics output:** The first lattice QCD evaluation of a meson distribution-amplitude observable using a diagonal Wilson-line geometry, cross-checked against an independently written PyQUDA implementation and agreeing within statistical uncertainties; plus automated multi-hadron spectroscopy whose Wick-contraction counts scale 2→36→2880 across p / deuteron / triton systems.

## Critical Assessment

- **Novelty:** High and genuinely first-of-its-kind. Prior AI-in-LQCD work (flow-based sampling, learned effective actions, refs [9]–[12]) used ML as a numerical ingredient inside a human-centered pipeline; LQCDMaster is the first to cast LQCD *code/workflow generation* itself as an agentic task. The `generate_einsum` tool's framing as a "reliability boundary between physical planning and algebraic realization" is a clean, transferable idea.
- **Rigor:** Strong on validation philosophy. The authors correctly insist that LQCD correctness is *not* implied by execution success and validate every workflow by direct numerical comparison to hand-written references at machine precision (|∆|≲10⁻¹²). The three-level outcome taxonomy (Matched / Convention mismatch / Failure) is honest and lets the reader see exactly what failed.
- **Specific weaknesses worth naming:**
  - *Baryon 3pt is the soft underbelly.* The 4 failures (p→p vector, Λ→Λ vector+axial, Λ_b→Λ_c vector) all involve transition currents and sequential-source routing — exactly the class where spin–color routing is hardest and the critique–rewrite loop did not catch them. The headline 90% number is buoyed by perfect scores on the simpler classes.
  - *Asymmetric flavor sensitivity across backbones.* DeepSeek-V4-Pro flips the failure profile — meson 3pt collapses to 53.8% (vs. 100% with GPT-5.4) due to incorrect quark-flavor assignments, while baryon 3pt actually improves to 86.7%. This shows semantic correctness of observable-specific task parameters is still backbone-dependent, undermining the "architecture-not-model" robustness claim to some degree.
  - *Weak general-agent baseline.* Appendix B compares only to Claude Code given *documentation but no skills/tools* — a deliberately handicapped baseline that failed all 20 local 2pt tasks. The paper does not test a more competitive middle ground (e.g., Claude Code + skills but without `generate_einsum`, or vice versa), so the individual contributions of skills vs. the deterministic tool are not ablated.
  - *Missing ablations.* There is no ablation isolating the value of the solve–critique–rewrite loop, of individual skills, or of the human checkpoint. The relative contribution of each component is asserted, not measured.
  - *Narrow ensemble.* Everything is on a single CLQCD C24P29 ensemble at one lattice spacing/volume/pion mass; generalization across ensembles is not demonstrated.
- **Reproducibility:** Good in principle — public GitHub with benchmark, skills, tool, and generated outputs. However, results depend on proprietary backbone models (GPT-5.4, DeepSeek-V4-Pro) whose exact versions will drift; the deterministic `generate_einsum` portion is the truly reproducible core.
- **Significance:** Conceptually important for AI-for-science: it demonstrates that automating the *engineering* bottleneck (not the physics reasoning) can unlock computations that were physically well-motivated but practically never attempted (diagonal Wilson-line LCDAs, multi-hadron spectroscopy). This "exploration bias" framing is the paper's most broadly exportable insight.
- **Red flags:** None severe. No cherry-picking of metrics (failures are reported openly); claims are appropriately scoped to "code-generation assistance" rather than "autonomous physicist." The authors explicitly state that the agent's contribution is reducing engineering cost, not independent discovery of operators — a refreshingly honest framing.

## Strengths

- Clear separation of planning and execution with a *solve–critique–rewrite* loop plus a human checkpoint — architecturally sound and pragmatic.
- The `generate_einsum` deterministic tool directly addresses the highest-risk failure mode (wrong Wick contractions producing plausible-but-wrong correlators) and is a transferable design pattern for other symbolic-heavy scientific domains.
- Honest, granular reporting: per-class accuracy, named failing observables, and an explicit failure-mode taxonomy rather than a single headline number.
- Cross-backbone replication (GPT-5.4 vs. DeepSeek-V4-Pro) is rare and valuable for assessing architecture-level vs. model-level credit.
- Delivers genuinely new physics results (first diagonal Wilson-line LCDA; multi-hadron spectrum up to hypertriton) rather than only a benchmark score.
- Strong baseline reality-check: a general coding agent with documentation alone produces zero executable scripts on the simplest class, motivating the entire design.

## Limitations

- Benchmark is representative but not exhaustive: **disconnected diagrams, four-point and higher correlations, finite-temperature observables, and large multi-particle operators are not tested** — all explicitly flagged by the authors as future work.
- The 4 baryon 3pt failures (transition currents) show the critique–rewrite pipeline does not yet fully catch the hardest spin–color routing; direct numerical reference comparison is still required for novel observable classes.
- Semantic correctness of observable-specific task parameters (e.g., quark-flavor assignments) remains partially dependent on backbone LLM in-domain knowledge — visible in DeepSeek-V4-Pro's meson 3pt regression.
- All experiments use a single ensemble (C24P29); cross-ensemble and multi-collaboration validation is absent.
- No automated downstream stages (fitting, physical interpretation, error analysis) — the agent stops at correlator generation.
- Reliance on proprietary frontier LLMs (GPT-5.4, DeepSeek-V4-Pro) limits bit-exact reproducibility of the LLM-driven portions.
- The general-agent baseline (Claude Code + docs only) is deliberately weak; intermediate ablations are missing, so the marginal value of skills vs. the deterministic tool vs. the planning loop is not isolated.

## Relevance

对关注 "AI for Science / LLM agents 自动化科研工作流" 的研究者，本文是 agentic scientific computing 范式在格点 QCD 这一高知识门槛前沿领域的标杆性落地，其"确定性符号工具 + 专家技能 + 规划—执行分离"的可靠性设计模式可迁移到其他代数脆弱、运行无误却可能错误的科学代码生成场景（如量子化学、张量网络）。对 LLM-agent 工程方向，它提供了"把脆弱符号构造移出 LLM 自由输出空间"这一可复用的工程原则。

## Questions for Further Exploration

- What is the marginal contribution of each component — `generate_einsum` alone, skills alone, the solve–critique–rewrite loop alone, and the human checkpoint? A component-wise ablation would clarify where the reliability actually comes from.
- Can the baryon 3pt transition-current failures (p→p vector, Λ→Λ vector+axial, Λ_b→Λ_c vector) be closed by extending `generate_einsum` to cover sequential-source sink blocks for transition currents, or do they require a different validation mechanism?
- How does the system behave on **disconnected diagrams and ≥4-point functions**, where Wick-contraction combinatorics and gauge-noise statistics are qualitatively harder than anything in the current benchmark?
- Does cross-ensemble generalization hold — i.e., are skills/contracts portable to ensembles with different lattice spacings, volumes, boundary conditions, or actions (e.g., Wilson vs. staggered vs. domain-wall)?
- Why does the failure profile invert across backbones (meson 3pt worse with DeepSeek-V4-Pro but baryon 3pt better)? Is the flavor-assignment sensitivity a prompt/skill artifact or a pretraining-data artifact?
- Could the pipeline be extended past correlator generation to **automated fitting, excited-state decomposition, and physical interpretation**, and at what point does the human-in-the-loop checkpoint become unnecessary?
- Is the "exploration bias" claim (non-standard observables go uncomputed for engineering reasons) quantifiable across the LQCD literature, and can LQCDMaster's adoption be shown to broaden the observable distribution of subsequent publications?
