> **一句话总结：** PolyQ 提出面向 CPU 的"量化-编译器协同设计"框架，用激活感知的逐通道 {2,3,4,8,16} 比特分配配合编译时布局正则化，在 Falcon-H1-3B/Llama2-13B/Qwen3-32B 上以 3 比特目标相比 AWQ/GPTQ/Slim-LLM/AMQ 将 WikiText-2 困惑度降低 2.4–32.1%，并把激活重排流量减少最高 70.8%，将"分数比特预算"变成 CPU 上可落地、可预测的部署方式。

> **TL;DR:** PolyQ is a CPU-only LLM quantization + compiler co-design that assigns each weight channel one of {2,3,4,8,16}-bit via activation-aware waterfilling under a fractional average-bit budget, then at compile time permutes/clusters channels into bit-homogeneous blocks, emits SIMD (W{8,16}A16 OpenBLAS) and LUT (W{2,3,4}A16 T-MAC) kernels, and propagates compatible permutations across the operator DAG. On WikiText-2 it improves 3-bit perplexity by 2.4–32.1% over AWQ/GPTQ/Slim-LLM/AMQ on Falcon-H1-3B/Llama2-13B/Qwen3-32B (e.g., Qwen3-32B: 7.66 vs AMQ 11.29), realizes fractional budgets within 0.045 b of target, cuts activation reorder traffic by up to 70.8% over the best RPTQ/Atom-style policy, and stays within 5.8% prefill latency and <2% energy/token of the optimized AWQ+T-MAC backend across Ryzen 9950X, Ryzen 7840U, and Intel N250.

## 核心贡献

- **可部署的逐通道混合精度方案**：首次将"激活感知显著性"与覆盖 2/3/4/8/16 比特的"CPU 友好调色板"结合，使得既能匹配通道显著性，又能做细粒度的分数比特预算控制（如 3.7 b）。
- **编译时协同设计**：将不规则的逐通道比特映射，经置换、聚类转换为"同比特块"，按块生成 LUT/SIMD 内核，并通过算子图传播让运行期布局重排几乎为零。
- **层内分数比特"注水"算法**：以"误差/比特增益"为最大堆，从全 2 比特初始态贪心升级显著通道，直至填满平均比特预算。
- **端到端可预测的部署曲线**：同一比特参数 B 同时控制质量、峰值显存、吞吐和能耗，把原本 W2/W3/W4 的离散三选一变成稠密设计曲线。
- **跨三档 CPU 的实测验证**：在工作站（Ryzen 9950X）、笔记本（Ryzen 7840U）、移动（Intel N250）三档硬件上验证分数比特部署的真实可行性与能效。

## Key Contributions

- A **deployable channel-wise mixed-precision formulation** over a CPU-aligned {2,3,4,8,16}-bit palette, guided by activation-aware saliency, that simultaneously enables saliency-matched allocation and fractional-bit budget control (e.g., 3.7 b) — something neither inter-layer methods (HAQ/AMQ, limited to 2/3/4) nor intra-layer methods (Slim-LLM, with its CPU-hostile 5–7-bit entries) provide.
- A **compile-time quantizer/compiler co-design** that clusters irregular channel bit assignments into bit-homogeneous blocks, dispatches W{2,3,4}A16 to LUT (T-MAC) and W{8,16}A16 to SIMD (OpenBLAS) kernels, and propagates compatible channel permutations across the operator DAG so layout regularization is moved off the runtime path.
- A **layer-local waterfilling algorithm** that starts every layer at 2-bit and greedily promotes the highest-gain channels (by activation-weighted error per extra bit) until the fractional average-bit budget B is exhausted, followed by an ISA-aware quanta-matching snap that re-optimizes channel membership to fit real SIMD/LUT tile sizes.
- A **graph-level permutation planner** (finite graph-labeling on the operator DAG) that merges adjacent activations' permutations into compile-time weight rewrites, so only incompatible fan-out / fan-in / boundary edges pay a runtime shuffle.
- **End-to-end evidence** on three CPU tiers (workstation / laptop / mobile) that fractional budgets are practical, predictable, and energy-efficient, turning the budget sweep into a usable deployment design curve.

## Methodology

PolyQ is a quantizer/compiler co-design with two halves.

**Accuracy side (Section 3, "Poly-Precision Quantization"):**
- **Activation-aware per-channel error proxy (Eq. 2):** For each linear layer, each candidate bit-width b ∈ {2,3,4,8,16}, and each input channel j, compute an error proxy E_b[j] ≈ ||w̃_j^(b) − w_j||² · E[x_j²], combining weight reconstruction error with the empirical second moment of the j-th input activation from a small calibration set (128 WikiText-2 sequences). E_16[j] = 0 (treated as unquantized). All five error values per channel are computed in one pass.
- **Layer-wise waterfilling under fractional budget (Eq. 3–5):** Each linear layer is treated as an independent discrete resource-allocation problem under a per-layer average budget B. Initialize all channels to 2-bit; compute residual budget Δ = round(B·d_in − 2·d_in); repeatedly pop the channel with the largest gain G_{b→b'} = (E_b[j] − E_{b'}[j]) / (b' − b) from a max-heap and promote it, until no discrete promotion fits the residual budget. Different layers can end with different 2/3/4/8/16-bit mixes while targeting the same B.
- **ISA-aware SIMD-quanta matching (Eq. 6–7):** Round per-bit channel counts c_b to multiples of backend quanta q_b^(t) (e.g., AVX2 int8 → multiples of 32, LUT kernels impose per-bit quanta). A second error-aware reassignment step promotes/demotes channels by the same proxy to keep realized average B_real within ~0.016–0.045 b of target.
- **Bit-specific activation-aware scaling (Eq. 8):** A single exponent α_b per bit-width per layer (power-law scale s_j = m_j^{α_b}), chosen by a small grid search to minimize average proxy error, kept cheap to store and foldable into packed weights.

**Execution side (Section 4, "Model Compiler Architecture"):**
- **Channel permutation & clustering (Eq. 9):** Factor O = A·W = (A·P_W^T)·(P_W·W). Only the clustered weight P_W·W is materialized at compile time; the permutation matrix is never stored at runtime.
- **Bit-block kernel assignment:** Low-bit blocks (W{2,3,4}A16) use LUT-based GEMM/GEMV (T-MAC, vqtbl1q/vpshufb); high-precision blocks (W{8,16}A16) use OpenBLAS SIMD GEMM because their larger group sizes make table lookup less efficient than direct MAC.
- **Cross-layer permutation merging (Eq. 10):** For consecutive linears O = A·W and O' = O·W', the downstream activation permutation P^T is rewritten into compile-time permutations of upstream output columns and downstream input rows: O' = A·(W·P^T)·(P·W'). Non-linear activations (ReLU/GELU/SiLU) commute with channel permutations, so the basis propagates unchanged.
- **Graph-based permutation planning:** The model is treated as a static operator DAG; each linear carries a preferred input-channel basis, each edge a current basis. A topological pass assigns basis labels to edges and inserts shuffles only at incompatible fan-out/fan-in/path-boundary edges, discovering mergeable paths that span more than one Transformer block (e.g., last FFN linear → next attention block's first projection).

Baselines: AWQ, GPTQ, Slim-LLM, AMQ (group size 128, FP16 activations, 128 WikiText-2 calibration sequences). Models: Falcon-H1-3B, Llama2-13B, Qwen3-32B. Quality: WikiText-2 perplexity plus capped accuracy on MMLU, WinoGrande, ARC-Easy/Challenge, HellaSwag. Hardware: Ryzen 9 9950X (16 cores), Ryzen 7 7840U (8 cores), Intel N250 (4 cores) under Ubuntu 24.04; energy via RAPL.

## Results & Highlights

- **3-bit perplexity on WikiText-2 (Table 1):** PolyQ beats all baselines on every model — Llama2-13B: 4.88 vs AWQ 5.29, GPTQ 5.75, Slim-LLM 5.03, AMQ 5.68; Qwen3-32B: 7.66 vs AWQ 8.23, GPTQ 10.75, Slim-LLM 10.00, AMQ 11.29 (a 32.1% reduction over AMQ); Falcon-H1-3B: 5.82 vs AWQ 5.96 and Slim-LLM 6.15 (GPTQ 6.28). Aggregate improvement range: 2.4–32.1% over the best prior method at the 3-bit target.
- **Fractional-budget realization (Fig. 7):** After ISA-aware quanta matching across B = 3.0–6.0 b, the maximum realized-bit mismatch is 0.045 b (Falcon-H1-3B), 0.031 b (Llama2-13B), and 0.016 b (Qwen3-32B); at B=3 the mismatch stays ≤ 0.016 b and at B=4 within 0.005 b. Raw-map padding is reduced from 1.30–5.77% (pre-snap) to 0.46–1.68%.
- **Activation reorder traffic (Fig. 10):** PolyQ's modeled per-token reorder traffic is 1.00 / 1.14 / 2.95 MiB on Falcon-H1-3B / Llama2-13B / Qwen3-32B respectively, versus the best prior-style policy (RPTQ or Atom) at 1.88 / 3.91 / 6.25 MiB — reductions of **46.6%, 70.8%, and 52.8%** (peak reduction 70.8% on Llama2-13B). With no merging at all the cost is 3.28 / 6.80 / 14.50 MiB per token.
- **CPU latency & throughput (Fig. 11):** Against AWQ on the same T-MAC LUT path for Llama2-13B, PolyQ prefill latency stays within 4.7% on Ryzen 9950X, 5.8–7.3% on Ryzen 7840U, and 0.1–0.4% on Intel N250; decode-throughput shortfall is ~0.2% (9950X), 8.1–8.2% (7840U), and 1.2–6.1% (N250). Latency scaling with B closely tracks the ideal weight-traffic ratio (B:2→3 gives 1.56–1.58× latency vs ideal 1.50×; B:3→4 gives 1.38–1.46× vs ideal 4/3).
- **Energy per token (Table 3, Llama2-13B at B=3):** PolyQ overhead vs AWQ+T-MAC is **1.13%** on Ryzen 9950X (6.10 vs 6.03 J/token), **0.80%** on Ryzen 7840U (2.80 vs 2.77 J/token), and **1.89%** on Intel N250 (2.41 vs 2.37 J/token) — consistently below 2%.
- **Peak memory footprint (Fig. 8):** At matched integer budgets, PolyQ stays within 1.1% of AWQ(T-MAC) peak footprint; for Llama2-13B at B=2.6, storage is 4.75 GiB of the 4.98 GiB peak, with layout metadata + reorder buffer contributing only 0.006 GiB.
- **Low-bit recovery (Table 2):** PolyQ identifies model-specific recovery points B_min just past the W2 collapse — Llama2-13B recovers from PPL 44.78 at B=2.5 to PPL 5.96 at B=2.6, saving 15.4% over a uniform-W3 fallback; equivalent savings 42.9% (Falcon) and 36.4% (Qwen3) with peak-footprint reductions of 17.1% / 11.6% / 21.8% respectively.

## Authors & Credibility

The work comes from the **University of California, Irvine** group of **Mohsen Imani** (senior/corresponding author, m.imani@uci.edu). The first author Hyunwoo Oh also appears on reference [16] T-SAR, a prior full-stack CPU-only ternary LLM co-design — i.e., the team has a direct, recent track record on exactly this problem (CPU low-bit LLM inference), giving the systems-engineering claims (T-MAC dispatch, OpenBLAS dispatch, RAPL-based energy, multi-CPU end-to-end timing) substantial credibility. The choice of T-MAC as the LUT backend and OpenBLAS as the SIMD backend are mainstream, not strawman, comparisons. The paper is an arXiv preprint (cs.LG, v1, 16 Jul 2026) and not yet peer-reviewed, so claims should be read as author-reported until independently reproduced.

## Critical Assessment

- **Novelty — genuine but incremental.** The core insight — a CPU-aligned {2,3,4,8,16} palette that avoids Slim-LLM's 5–7-bit LUT-incompatible entries while still enabling fractional budgets — is the strongest novel claim. However, the constituent pieces are well-precedented: activation-aware per-channel saliency is AWQ/SmoothQuant, channel-wise mixed precision is Slim-LLM, reordering is RPTQ, operator-boundary fusion is Atom, and the LUT backend is T-MAC. The contribution is the integration via a compiler that propagates permutations across the DAG, not any single new algorithm.
- **Rigor — solid on quality/efficiency, weaker on accuracy ablations.** Perplexity numbers are reported at matched B with a sensible stable-window framing (3.0–6.0 b). But several gaps stand out: (1) the 3-bit perplexity win on Llama2-13B (4.88) is strikingly low compared to typical AWQ-3 reports in the literature, and the paper does not provide a reconstruction-error / Hessian-based error metric to convince the reader the proxy (Eq. 2, a first-order activation-weighted Frobenius error, not GPTQ-style second-order) is sufficient — a perplexity-vs-proxy correlation ablation is missing. (2) AMQ is reported as unsupported on Falcon-H1-3B (Table 1 footnote †), so the "PolyQ beats AMQ" claim is partially on unrun configurations. (3) Downstream task results are "capped" multiple-choice accuracy whose non-monotonic variations are hand-waved away with a citation to the emergent-abilities-mirage paper rather than reported with confidence intervals or seeds.
- **Reproducibility — mixed.** Positives: real models (Falcon-H1-3B, Llama2-13B, Qwen3-32B), real CPUs (three tiers), real energy numbers via RAPL, calibration = 128 WikiText-2 sequences (standard), group size 128 (standard), baselines at standard settings. Negatives: the compiler and graph-planning implementation are not stated to be open-sourced; the AWQ comparison uses "AWQ weights on the same T-MAC path" rather than AWQ's own kernels, which is fair for isolating layout but means AWQ is not run in its native optimized form; the reorder-traffic metric (Fig. 10) is a *modeled* MiB/token traffic count, not measured cache/DRAM traffic, so the "70.8% reduction" is a model number, not a hardware measurement.
- **Significance — high for the CPU edge-deployment niche.** The thesis that "fractional-bit deployment is a real systems need because usable memory is fragmented and W2/W3/W4 are too coarse" is well-motivated and the demonstrated ≤0.045 b budget realization and <2% energy overhead make a credible case that this is deployable today. If the compiler is released it could become a reference CPU mixed-precision backend.
- **Red flags / things to probe.** (a) The Llama2-13B PolyQ-3 perplexity of 4.88 is suspiciously strong vs published AWQ-3 (~5.3 is more typical) and warrants a sanity check — is the baseline AWQ here under-tuned, or is the proxy genuinely better? (b) The reorder-traffic and prefill-latency gains are reported against "prior-style policies" reimplemented by the authors (RPTQ-style / Atom-style), not the original RPTQ/Atom codebases — direct comparison would be more convincing. (c) No INT8/INT4 activation quantization is attempted (activations stay FP16/A16); the energy/latency wins are about weight traffic only, so the title's "End-to-End" framing slightly oversells scope. (d) Slim-LLM, RPTQ, and Atom are explicitly excluded from end-to-end CPU timing because they "implement GPU-only execution backends and have no CPU inference path," which is fair but means the runtime comparison is essentially PolyQ vs AWQ-on-T-MAC only.

## Strengths

- Cleanly diagnoses the real gap: inter-layer methods are too coarse, intra-layer methods use CPU-hostile bit-widths, and execution-side work doesn't propagate across the operator DAG.
- The {2,3,4,8,16} palette choice is a genuinely well-motivated CPU fit, and the waterfilling + quanta-matching + error-aware reassignment pipeline is principled and modular.
- Compiler DAG-level permutation planning (including cross-block merges like FFN-out → next-attention-in) is the most original systems contribution and goes beyond RPTQ/Atom's local fusion.
- Honest about where the gains come from: at 5–6 b all methods converge (acknowledged), and the differentiation is explicitly scoped to the tight-budget regime.
- Real three-tier CPU evaluation with RAPL energy numbers, not just speedup tables on a single chip.

## Limitations

- Activation quantization is out of scope (A16/A8/A32 always); the method is weight-only, so memory savings on activations/KV-cache are not addressed.
- The reorder-traffic reduction (Fig. 10) is a *modeled* byte count, and the CPU timing baseline is essentially AWQ-on-T-MAC (the original Slim-LLM/RPTQ/Atom backends are GPU-only and excluded).
- No reconstruction-error or Hessian-based justification for the first-order activation-weighted Frobenius proxy; no perplexity-vs-proxy-quality ablation.
- AMQ is unsupported on Falcon-H1-3B, weakening one head-to-head.
- Downstream task results are "capped" evaluations reported without seeds, confidence intervals, or variance, with non-monotonic kinks attributed to emergent-abilities literature rather than measured.
- The bit-specific activation-aware scaling exponent α_b and group size are chosen by grid search; sensitivity to calibration set size/composition is not reported.
- No release statement for the compiler; reproducibility hinges on artifacts that are not confirmed public.

## Relevance

与用户关注的高效 LLM 推理 / 边缘部署 / 模型量化方向高度契合：PolyQ 提供了一套面向 CPU 的"量化-编译"协同设计范式，把分数比特预算从理论概念落到三档真实 CPU 上（含移动端 Intel N250），对在做端侧 LLM 部署、低比特量化或推理后端优化的人有直接参考价值，尤其是其跨算子置换合并与 LUT/SIMD 混合分派思路。

## Questions for Further Exploration

- How does PolyQ behave under INT8/INT4 activation quantization (currently A16-only), and does the {2,3,4,8,16} palette argument still hold when activations are also low-bit?
- Is the suspiciously strong Llama2-13B PolyQ-3 perplexity (4.88 vs typical AWQ-3 ~5.3) reproducible with the public AWQ recipe, or is the AWQ baseline here under-tuned?
- How sensitive is the realized budget B_real and final perplexity to calibration set size (128 sequences) and domain (WikiText-2 vs code/multilingual)?
- Can the graph-level permutation planner handle non-Transformer architectures (Mamba/RWKV, MoE experts with shared fan-in) where basis compatibility is less clean?
- What is the breakdown of the <2% energy overhead — is it dominated by the residual shuffles at incompatible DAG edges, or by the sparse OpenBLAS dispatch for the few 8/16-bit blocks?
- Does the waterfilling proxy (first-order, activation-weighted Frobenius) leave accuracy on the table vs GPTQ-style second-order Hessian updates when budgets are very tight (B ≈ 2.1–2.6)?
