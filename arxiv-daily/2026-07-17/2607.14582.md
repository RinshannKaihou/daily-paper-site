> **一句话总结：** 提出 MathCoPilot，一个以"活态证明蓝图"为核心的人机协同定理证明系统，整合交互式工作台、技能编排与知识库构建；在 FormalMATH 子集和两个真实 PDE 定理上系统对比 Gemini 3.1 Pro / GPT-5.4 / Claude Opus 4.7 等模型，发现先写自然语言证明再形式化可将严格通过率从 15.87% 翻倍至 31.75%，但研究级定理仍高度受制于自动形式化质量。

> **TL;DR:** MathCoPilot is a human-in-the-loop theorem-proving system whose editable "living proof blueprint" lets the mathematician steer high-level strategy while AI agents (Brainstorm, Blueprint, Prover, Formalizer, Aggregation) execute step-level work in Lean 4, supported by adaptive skill orchestration (NL-first, formalization-first, knowledge-grounded) and a personal verified knowledge base built from retrieved arXiv/Mathlib papers. On a 21-problem FormalMATH subset it shows NL-first raising the strict pass rate from 10/63 (15.87%) to 20/63 (31.75%, +15.87 pp) over formalization-first, and on the full interactive workflow it recovers 10/12 of GPT-5.4's failures; on two real PDE theorems (upwind DG L2 estimate, Gauss-Radau projection) Claude Opus 4.7 gets 6/6 Lean assembly certificates vs. Gemini 3.1 Pro's 2/6, with GPT-formalized statements breaking due to Lean-version incompatibility. The headline message: autoformalization quality and abstraction-boundary choice dominate over raw model size.

## 核心贡献

- 提出基于"活态证明蓝图"(living proof blueprint) 的人机协同证明范式：把证明分解为可点击、可编辑、可递归的节点图，由人类掌管高层方向、AI 执行节点级细化，从而把"端到端黑盒证明"改成可控、可解释、可中途干预的协作流程。
- 设计了五个专职智能体（Brainstorm / Blueprint / Prover / Formalizer / Aggregation）与一套反馈驱动的"证明技能编排器"（NL-first、formalization-first、knowledge-grounded），能根据 Lean 反馈在技能间动态切换、递归暴露子目标并自适应检索知识库。
- 实现自动知识库构建：从 Semantic Scholar / arXiv / Mathlib 按主题检索论文，抽取定义、定理、证明骨架与记号，按依赖序自动形式化为 Lean 4 代码，只有通过 Lean 4 类型检查的条目才进入个人知识库；知识库同时保留形式化代码和自然语言推理过程的双表征。
- 提供严格的、可复现的基准评估：在 21 题 FormalMATH 子集（3 题 × 7 领域）上对比 Gemini 3.1 Pro / GPT-5.4 / Claude Opus 4.7，并明确把"含 sorry / 空证明"判为失败；同时在两个真实 PDE 定理（半离散迎风 DG 的 L2 误差估计、Gauss-Radau 参考单元投影估计）上评测三种 autoformalization 路径（GPT-Formalized、Opus-Formalized、Direct），并额外测试 DeepSeek-Prover-V2、Goedel-Prover-V2、Claude Sonnet 4。
- 通过"wrong-proof detection"任务证明：即便 Lean 形式化需要把 PDE 理论抽象为 opaque 声明/axiom，模型仍能正确拒绝带错证明并指出错误所在（如 DG 证明中 φ=e=u−u_h 非容许测试函数；Gauss-Radau 中 F(ŵ)=‖Π̂ŵ−ŵ‖L2 非线性）。

## Key Contributions

- A human-in-the-loop proving paradigm built on an editable **living proof blueprint**: proofs are decomposed into a navigable node graph (lemmas/cases/claims) where the mathematician clicks a node to inspect/direct/refine, while AI agents discharge the details. This replaces the black-box single-pass Q&A prover with a controllable, explainable workflow.
- A five-agent pipeline (Brainstorm → Blueprint → Prover → Formalizer → Aggregation) plus a **feedback-driven skill orchestrator** that dynamically switches among NL-first, formalization-first, and knowledge-grounded proving based on Lean's error feedback, supports arbitrary-depth recursion on stuck nodes, and learns skill→problem preferences over time.
- **Automated personal knowledge-base construction**: topic-driven retrieval (Semantic Scholar / arXiv / Mathlib) followed by dependency-ordered autoformalization into Lean 4, with only Lean-verified entries retained and both Lean code and the natural-language reasoning chain stored in each entry for downstream retrieval and explanation.
- A **strict, reproducible evaluation harness**: a 21-problem FormalMATH subset (3 per domain × 7 domains, totaling 126 attempts) verified under Lean 4.30.0-rc2 / Mathlib v4.30.0-rc2 with 1200s timeout, where a pass requires no `sorry` and a non-empty proof; plus a PDE study on two real theorems with three formalization routes (GPT-Formalized, Opus-Formalized, Direct) verified under Lean 4.30.0 / Mathlib v4.30.0 with 600s timeout.
- A **wrong-proof diagnostic** showing that critique is easier than construction: both Gemini 3.1 Pro and GPT-5.4 correctly reject all four deliberately flawed PDE proofs and pinpoint the precise illegal step (illegal test function `φ=e` in DG; non-linear `F(ŵ)=‖Π̂ŵ−ŵ‖L2` in Gauss-Radau).

## Methodology

The system rests on three integrated capabilities (Figure 1):

1. **Interactive Human-AI Proof Co-Construction.** Given a target proposition the mathematician enters a five-agent loop. The *Brainstorm Agent* proposes several NL proof strategies; the human edits/selects one. The *Blueprint Agent* decomposes the chosen strategy into a flowchart of self-contained nodes (lemma / case / claim) annotated with NL intent and status (open / in-progress / verified); interaction is *pointing-based* — the user clicks a node rather than steering through dialogue. The *Prover Agent* is the user-facing interface to the skill orchestrator (Sec. 3.2); for hard nodes the user can *recurse*, re-applying the whole pipeline to a single node to arbitrary depth. The *Formalizer Agent* maintains the correspondence with Lean 4, emitting an initial `sorry`-skeleton of `have` statements and translating each node into Lean as it is completed. The *Aggregation Agent* linearizes the dependency graph and stitches node-level arguments into a single paper-ready NL proof plus assembled Lean proof. On-demand explanation is provided at multiple granularities.

2. **Proving Skill Orchestration.** Rather than a fixed pipeline, an orchestrator maintains a pool of skills — *NL-first* (NL sketch then incremental formalization), *formalization-first* (formalize statement, then search tactics from Lean's proof state), *knowledge-grounded* (retrieve analogous theorems from the user's KB). It parses Lean feedback (type mismatch, missing hypothesis, unsolved goal), diagnoses the failure mode, and chooses the next action: retry with another skill, refine based on the error, retrieve additional lemmas, or request human intervention. The orchestrator accumulates per-skill success statistics.

3. **Knowledge-Base Construction and Maintenance.** Given a topic, the system retrieves papers from Semantic Scholar, arXiv, and Mathlib, ranks by relevance / citation / formalization potential, then an extraction agent pulls formal definitions and dependencies, theorem statements with hypotheses, proof sketches, and notation. Content is autoformalized to Lean 4 with Mathlib as context, in dependency order (definitions → auxiliary lemmas → main theorems), and only files that compile under the Lean 4 type checker enter the KB. Each entry stores both Lean code and NL reasoning. Entries are re-verified automatically when Mathlib versions update.

**Evaluation methodology.** Two studies: (a) FormalMATH subset — 21 problems × 3 models × 2 skills = 126 attempts, with strict no-`sorry` verification; plus a 12-case "GPT-unsolved" subset re-attacked with the full interactive blueprint workflow (first pass + recursive local-lemma refinement). (b) PDE study — two real theorems (semi-discrete upwind DG L2 estimate; Gauss-Radau reference-cell estimate), three formalization routes (GPT-Formalized, Opus-Formalized, Direct), four models (Gemini 3.1 Pro, GPT-5.4, Claude Opus 4.7, Claude Sonnet 4 on GPT-Formalized route), 20 attempts total, with the explicit caveat that success = Lean *assembly* certificate under explicit opaque/axiom assumptions for deep analytic facts (not a from-first-principles formalization of PDE theory). An additional wrong-proof diagnostic gives Gemini 3.1 Pro and GPT-5.4 deliberately flawed NL proofs of both PDE theorems.

## Results & Highlights

- **NL-first doubles strict pass rate on FormalMATH.** Overall strict pass rate is **20/63 (31.75%) for NL-first vs. 10/63 (15.87%) for formalization-first**, a +15.87 pp gain. Per-model gains: Gemini 3.1 Pro **4/21 → 7/21 (+14.29 pp)**; GPT-5.4 **3/21 → 6/21 (+14.29 pp)**; Claude Opus 4.7 **3/21 → 7/21 (+19.05 pp)** — the Opus number corrects for 3 empty formal-proof outputs under FF that would otherwise have counted as 6/21 raw verifier passes.
- **Domain-level zeros.** Integral, Other, and Sequences/Series stay at **0/18 across all models and both skills**, while Differentiation jumps from **0/9 FF → 6/9 NL** and Precalculus from **5/9 FF → 7/9 NL**. The dominant failure category is residual `sorry` (19 times under FF, 23 under NL).
- **Interactive blueprint recovers GPT-5.4 failures.** On 12 cases GPT-5.4 could not solve, MATHCOPILOT verifies **9/12 (75.0%) on the first full-system pass and 10/12 (83.3%) after recursive local-lemma refinement**, including hardmath_162, whose endpoint Laplace lemma no baseline could close.
- **PDE results are autoformalization-bound, not model-bound.** Per-route totals on the two real PDE theorems: **GPT-Formalized 2/8 (25%), Opus-Formalized 6/6 (100%), Direct 4/6 (66.67%)**. By model on assembly certificates: **Claude Opus 4.7 6/6, GPT-5.4 4/6, Gemini 3.1 Pro 2/6, Claude Sonnet 4 0/2**. The GPT-Formalized route collapses not because of math but because of Lean-version/declaration-boundary incompatibilities that only Opus repairs.
- **Critique is easier than construction.** Both Gemini 3.1 Pro and GPT-5.4 correctly reject all four planted wrong PDE proofs (2/2 each), identifying the exact illegal step (illegal test function `φ=e=u−u_h` in the DG estimate; non-linear `F(ŵ)=‖Π̂ŵ−ŵ‖L2` treated as a functional in Gauss-Radau).
- **Specialization does not transfer.** Out-of-distribution test of two open-source provers, DeepSeek-Prover-V2 and Goedel-Prover-V2, on the DG theorem yields **0/2 valid Lean 4 code** for both — outputs stay in NL meta-analysis and truncate.
- **Mathematically sharper refinement.** The hardmath_162 refined Lean certificate avoids treating the rounded `-0.10 e^{3.98x}/x` as a ratio-limit and instead proves the exact endpoint formula `I(x) ∼ -0.9·(-0.7)^2 / (-φ'(-0.7)) · e^{xφ(-0.7)}/x` plus the separate interval bounds `|φ(-0.7) − 3.98| ≤ 0.001` and `|leading coeff + 0.10| ≤ 0.002` — a semantic upgrade the blueprint workflow exposes.

## Authors & Credibility

The paper comes from the **State Key Laboratory of Cognitive Intelligence at USTC** (University of Science and Technology of China), jointly with the **School of Mathematical Sciences** at USTC. The author list mixes well-known ML/AI researchers (Enhong Chen, Qi Liu, Guoping Hu, Zhenya Huang, Doudou Wang — all associated with the USTC cognitive-intelligence group) with research mathematicians from the same institution (Mengping Zhang and Xiangdong Ye are the corresponding senior mathematicians). This combination is exactly what a serious "mathematical research with AI" paper needs: the PDE case studies (DG methods, Gauss-Radau projection, Bramble-Hilbert) are non-trivial and reflect genuine domain expertise, not toy algebra. The bibliography cites both the canonical PDE literature (Johnson-Pitkäranta 1986, Cockburn-Shu 2001, Zhang-Shu 2004, Bramble-Hilbert 1970, Ciarlet 1978) and the relevant recent LLM-proving work (DeepSeek-Prover, AlphaProof/Hubert et al. Nature 2025, LeanDojo, Lean Copilot, Numina-Lean-Agent, Rethlas/Archon, Draft-Sketch-Prove, Hilbert, Goedel-Prover-V2). Models named are concrete and current (Gemini 3.1 Pro, GPT-5.4, Claude Opus 4.7, Claude Sonnet 4), Lean versions and Mathlib versions are pinned, and an accompanying webpage (mathcopilot.cn) is referenced. The honesty about what counts as a "certificate" (assembly under opaque/axiom assumptions rather than from-first-principles PDE formalization) is a credibility plus.

## Critical Assessment

- **Novelty.** The system-level framing — five named agents around a *living, node-graph, pointing-based* blueprint combined with skill orchestration and an auto-built personal KB — is a genuine integration rather than a single new model. The closest comparables, Numina-Lean-Agent and Rethlas/Archon, are explicitly *autonomous*; MATHCOPILOT's explicit positioning as "serve the mathematician, not the benchmark" is the novel thesis. That said, the individual mechanisms (NL-first vs. FF from Draft-Sketch-Prove; recursive decomposition from Hilbert; retrieval from LeanDojo) are individually known, so novelty is in the composition.
- **Rigor.** Genuinely strong on the FormalMATH side: pre-registered strict no-`sorry` rule, explicit correction of Opus's 6/21→3/21 empty-proof discrepancy, full per-domain breakdown, failure-category counts. The PDE side is appropriately hedged — the authors explicitly state that PDE "success" means *assembly certificates* under opaque/axiom assumptions and are **"not directly comparable"** to the end-to-end FormalMATH strict pass rates. This is the right call and should be respected by anyone citing the 60% / 100% numbers.
- **Reproducibility.** Mixed. Lean and Mathlib versions are pinned (4.30.0-rc2 for FormalMATH, 4.30.0 for PDE), timeouts specified (1200s / 600s), the 21 FormalMATH problem IDs are listed in Appendix A.1, and the 12 GPT-unsolved problem IDs are enumerated. But: (i) the FormalMATH subset is only 21 problems, hand-sampled 3-per-domain, so the headline "NL doubles FF" rests on a 63-cell matrix with several all-zero domains — wide confidence intervals; (ii) the system prompts and orchestrator control logic are not released in the paper; (iii) the appendix references "tex/lean/Main.lean" but the Lean listing is unavailable ("Lean listing unavailable"); (iv) the comparison to DeepSeek-Prover-V2 / Goedel-Prover-V2 is just 0/2 with no prompt details.
- **Significance.** Practitioner-relevant for the formal-math community: the empirical claim that *autoformalization quality dominates model size* is actionable and confirms a growing trend in the literature. The wrong-proof-detection finding (models can reject but not construct) is a useful data point for "AI as reviewer" workflows.
- **Red flags.** (a) **Tiny FormalMATH sample (n=21)** with three domains at 0/18 — the reported deltas (e.g., "0/9 → 6/9 on Differentiation") are 3-cell per-model counts and statistically fragile; "doubles the pass rate" is a real but noisy headline. (b) **The PDE "100% on Opus-Formalized" route is partly circular**: Opus formalizes the statement into opaque/axiom blocks and then *Opus* (and others) verify the assembly — the deep analytic content is offloaded to unproved assumptions, so 6/6 should not be read as "Opus proved the PDE theorem." The authors are honest about this in Sec. 5.1 but the table headers (Table 5, Table 11) still risk being misread. (c) **The GPT-Formalized route's "Syntax" failure mode is presented as a Lean-version issue** but it is plausibly a GPT-5.4 formalization-style issue (declaration boundaries that should be opaque axioms); attributing it primarily to "Lean-version incompatibility" may understate the formalization-design lesson. (d) **No user study** despite the system's central claim being about mathematician workflow — the authors acknowledge this in Future Work, but the core "human-in-the-loop improves productivity" thesis is not empirically tested on actual mathematicians. (e) **The skill orchestrator's "learns from historical proof attempts" claim is asserted but never quantified** — there is no experiment showing orchestration improves over a fixed policy.

## Strengths

- Honest, strict verification protocol (no `sorry`, no empty proofs) that catches an otherwise-inflated Claude Opus number (6/21 → 3/21 on FF).
- The *recovery experiment* on 12 GPT-5.4-missed cases is a clean demonstration that the interactive blueprint workflow is additive beyond single-pass prompting — 9/12 immediately, 10/12 after refinement, including a mathematically sharper theorem than the original benchmark.
- Real, domain-rich case studies (DG methods, Gauss-Radau projection, Bramble-Hilbert) authored by people who clearly know the underlying numerics — a refreshing contrast to algebra-only autoformalization benchmarks.
- The wrong-proof diagnostic is a small but well-chosen experiment: it isolates *review* as a distinct, easier skill than *construction*, with concrete planted errors that exercise genuine math understanding (admissibility of test functions; linearity of operators).
- Clean per-domain heatmap that exposes which domains are bottlenecked by formal infrastructure (Integral/Other/Sequences at 0/18) vs. by reasoning (Differentiation).

## Limitations

- The FormalMATH sample is only 21 problems / 126 attempts; several domains sit at 0/18, so per-domain numbers are individually weak and the headline "NL doubles FF" should be read with wide error bars.
- PDE "assembly certificates" assume deep analytic facts as opaque declarations or axioms; they are explicitly **not** end-to-end formalizations of PDE analysis, and the authors flag this — but headline PDE percentages (60%, 100%) still overstate mathematical depth if quoted out of context.
- No human-subject study: the central "human-AI symbiotic paradigm improves mathematician productivity" claim is asserted by design, not measured. Plans for a "rigorous user study with larger participant pools" are deferred to future work.
- The skill orchestrator's *learning* behavior is described qualitatively ("improving its selection over time") but never quantified; no ablation isolates the contribution of orchestration vs. a fixed NL-first policy.
- The system prompts, agent control logic, and orchestrator decision rules are not released; the webpage (mathcopilot.cn) is referenced but a complete artifact release is not described.
- Appendix A.3 explicitly notes "Lean listing unavailable" for the showcased hardmath_162 endpoint block (tex/lean/Main.lean), weakening reproducibility of the headline recovery case.
- Out-of-distribution comparison to DeepSeek-Prover-V2 / Goedel-Prover-V2 is only 0/2 with no prompt or setup details — interesting signal but too thin to support strong claims about transfer.

## Relevance

与"用 LLM 做形式化定理证明"和"人机协同科研助手"两条研究主线高度相关：MathCoPilot 把 Lean 4 形式化、autoformalization、Agent 编排和可解释证明工作流整合到一个面向真实数学家的系统中，是观察前沿大模型（Gemini 3.1 Pro / GPT-5.4 / Claude Opus 4.7）在研究级数学（包括数值 PDE）上能力边界的一个高质量实证样本，对构建科研 Agent、形式数学评估基准、以及把"AI 作为审稿人"的工作都有参考价值。

## Questions for Further Exploration

- Does the **+15.87 pp NL-first advantage** survive on the full FormalMATH (not a 21-problem subset), and does it hold once formalization-first is given a Lean-aware repair loop comparable to the blueprint recursion?
- What fraction of the **PDE "assembly" certificates' correctness depends on the chosen opaque/axiom set**? A sensitivity study varying the abstraction boundary would clarify how much math is actually being verified.
- Can the **wrong-proof-detection capability** (currently 4/4 correct rejections on PDE) be benchmarked at scale, e.g., on a corpus of subtly-flawed Mathlib-style proofs, to calibrate model reviewers?
- How does the **skill orchestrator's learned policy** compare to (i) always-NL-first and (ii) a random skill selection, in terms of Lean success rate and token cost? The current paper asserts learning but does not isolate it.
- What is the **realized productivity gain for working mathematicians**? A controlled user study (e.g., formalizing a section of a recent analysis paper with vs. without MATHCOPILOT) would directly test the central thesis.
- Do the **GPT-Formalized route failures** disappear under a newer Lean/Mathlib version, or do they reflect a genuine formalization-style gap that GPT-class models need to learn (choosing axiom/opaque boundaries)?
- Can the **personal knowledge base** bootstrap from a larger Mathlib-connected seed to reduce cold-start, and does cross-group KB sharing actually compose without Lean-version conflicts?
