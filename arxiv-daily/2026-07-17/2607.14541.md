> **一句话总结：** 本文提出 Atrex-Bench——首个直接从阿里云生产推理 trace（30 个算子、440 个 shape）采样的 GPU kernel 生成基准，配合 production-importance 加权的 roofline 评分；评测 6 个前沿 coding agent 发现最强模型也只达到硬件 roofline 的约 10.7%，并据此发布 Atrex-Kernel-Agent（AKA），通过 profile-driven 的 measure-revise 循环、optimization dropout 与分层 GPU 知识库，将 PyTorch fallback 转化为真正匹配/超过手工调优生产 kernel 的 FlyDSL 实现。

> **TL;DR:** ATREX Team at Alibaba releases Atrex-Bench, the first GPU kernel-generation benchmark whose 30 operators and 440 shapes are sampled from full-cluster production inference traces (>10k XPU-A and H20 accelerators, ~20 deployed models across vLLM/SGLang/AITER/RTP-LLM), and scored by a production-importance-weighted roofline metric (Sagg). Six frontier coding agents (Claude Opus 4.7, GPT-5.5, Qwen3.7-Max, Kimi-K2.6, GLM-5.1, DeepSeek-V4-Pro) targeting the unfamiliar FlyDSL reach only Sagg=0.107 at best (GPT-5.5) vs. Opus 4.7's 0.059, and no agent matches the deployed production kernel (best 0.99× on Opus 4.7). The paper exposes a "correctness illusion" (Qwen3.7-Max passes 84.8% but with only 43.8% FlyDSL adoption) and co-releases Atrex-Kernel-Agent (AKA), whose profile-driven measure-revise loop, optimization dropout, and 298-reference-kernel / 244-document knowledge base convert 0%-FlyDSL fallbacks into near-100%-FlyDSL kernels that beat hand-tuned AITER production kernels by 1.06×–1.31× on three operators.

## 核心贡献

- **首个生产 trace 驱动的 kernel 生成基准 Atrex-Bench**：从 XPU-A 与 H20 生产推理集群（>10k 加速卡、~20 个部署模型、1303 条 profile、vLLM/SGLang/AITER/RTP-LLM 四大框架）采样得到 30 个算子、440 个 hot shape，而不是合成或人工筛选的任务，使评测贴近真实部署负载。
- **production-importance 加权的 roofline 评分**：每个算子按其在生产 GPU 时间中的份额 wᵢ 加权（top 5 算子占 ≈64% 权重，unified_attention 单算 36.1%），并以 per-shape 硬件 roofline 为分母得出 Sagg，确保分数反映“最吃 serving 时间的算子上的能力”而非算术平均。
- **隐藏 provenance 与 roofline 的防作弊 contract**：生成时仅暴露 reference.py / input.py / shapes.json，metadata.json（含上游符号与生产时延）与 roofline.json（含 SOL/W-Q 边界）仅给 evaluator，从结构上防止“按名字抄 reference”或“拟合评分公式”。
- **揭示“correctness illusion”**：Qwen3.7-Max 正确率 84.8% 但 FlyDSL adoption 仅 43.8%，证明 high correctness 可由 PyTorch fallback（如 `scaled_dot_product_attention`）或预编译 vendor kernel 调用（如 AITER GEMM/MoE）撑起，而非真正写出目标 DSL，必须用 FlyDSL device-time share 才能识破。
- **发布 AKA（Atrex-Kernel-Agent）优化代理**：组合 profile-driven 的 measure-revise 八阶段循环、用于跳出局部最优的“optimization dropout”部分重启机制，以及分层 GPU Wiki 知识库（298 个 reference kernel + 244 篇优化文档 + 外部上游项目）。
- **在 controlled case study 上闭环验证**：AKA 把 Qwen3.7-Max 的 0%-FlyDSL fallback 转成接近 100%-FlyDSL 的真实 kernel，attention_forward 上 S 从 0.06→0.40（6.7×），且在三个算子上全部超过手工调优的生产 kernel（1.06×–1.31×）。

## Key Contributions

- **Atrex-Bench — first production-trace-sourced kernel generation benchmark**: 30 operators / 440 shapes drawn directly from full-cluster inference traces on >10k XPU-A and H20 accelerators running ~20 deployed models across vLLM, SGLang, AITER, and RTP-LLM, bypassing the synthetic/curated task sources of KernelBench, TritonBench, MultiKernelBench, CUDABench, and SOL-ExecBench.
- **Importance-weighted roofline scoring (Sagg)**: every operator carries a production card-hour-weighted, phase-aware GPU-time weight wᵢ (top-5 operators ≈64% of weight; unified_attention alone = 36.1%), and is scored against a per-shape hardware roofline ceiling rather than a mutable software baseline, so the headline number reflects ability where production time actually goes.
- **Hidden-provenance release contract** that excludes metadata.json and roofline.json from the generation surface, preventing shortcut hacks (recovering the reference by name, tuning to the SOL denominator) while keeping per-problem provenance for reproducibility.
- **Empirical "correctness illusion" finding**: high correctness is achievable without writing the target DSL; Qwen3.7-Max reaches 84.8% correctness but only 43.8% FlyDSL adoption, and the failure taxonomy (683 failing units) shows 257 numeric mismatches and 24 correct-but-slow kernels that single-gate benchmarks would miss.
- **Atrex-Kernel-Agent (AKA)** — a profile-driven measure-revise agent with optimization dropout (partial restart masking stale iteration memories while preserving accepted kernel + audit trail) and a layered knowledge base of 298 reference kernels (115 NVIDIA, 142 AMD, 41 architecture-agnostic) and 244 optimization documents plus upstream reference projects (CUTLASS/CuTeDSL, FlyDSL, Triton, AITER, FlashInfer, FlashMLA).
- **Controlled case study demonstrating production-parity**: on three compute-bound operators, AKA converts 0%-FlyDSL fallbacks into near-100%-FlyDSL kernels that beat hand-tuned AITER production baselines by 1.06×–1.31×.

## Methodology

**Atrex-Bench construction.** A non-intrusive tracer attaches to running serving workloads on selected XPU-A and H20 clusters (without relaunching services or inserting application instrumentation), combining Python-frame context (operator phase, batch/token shape, framework state) with vendor runtime traces (kernel launches, timing, streams, memory ops, graph anchors) and correlating host/device events through correlation IDs, timestamps, or graph anchors. Raw kernel records are mapped to computation-level operator families so semantically identical operators across vLLM/SGLang/AITER/RTP-LLM collapse into one problem. Each operator is shipped as a five-file contract: `reference.py` (hardware-agnostic PyTorch spec), `input.py` (randomized + corner-case generator), `shapes.json` (production shapes), and hidden `metadata.json` + `roofline.json`.

**Per-problem roofline bound.** For unit j with semantic work Fⱼ, memory traffic Mⱼ, dtype-specific compute peak Pτ, and memory bandwidth β: `T_roofline,j = max(Fⱼ/Pτ, Mⱼ/β)`. This bound is derived from reference semantics (never the candidate's own profile), is fixed across candidates, and is stored in roofline.json.

**Importance weighting (Equation 3).** Each operator gets weight `wᵢ = Σ_a pₐ · (1/|Φᵢ|) Σ_{ϕ∈Φᵢ} D_{i,a}^ϕ / Dₐ^ϕ`, where pₐ is application a's fleet card-hour share, D_{i,a}^ϕ is operator i's observed device time in application a during phase ϕ. Weights are normalized so Σ wᵢ = 1 over the retained operator set; the distribution is heavily skewed (top-5 = 63.7%).

**Three-level scoring.** Per-shape `Sⱼ = T_roofline,j / T_cand,j ∈ (0,1]`; per-operator `Sᵢ = median{Sⱼ : j correct}` (0 if no correct shape — deliberately so an operator with no correct kernel cannot be silently dropped); aggregate `Sagg = Σᵢ wᵢ Sᵢ`. Weighted arithmetic mean (not geometric) is chosen so a single failed operator does not collapse the score to zero.

**Three-stage evaluation gate.** (1) compile (must produce a completed AOT MLIR artifact, not merely import); (2) correctness vs. reference.py on K=5 random seeds with atol=1e-2, rtol=5e-2; (3) performance in a controlled sandbox with frequency-lock check, per-iteration L2 flush, and per-shape subprocess isolation. Only kernels reaching stage 3 contribute to Sⱼ.

**Evaluated panel.** Six frontier coding agents (Claude Opus 4.7, GPT-5.5, Qwen3.7-Max, Kimi-K2.6, GLM-5.1, DeepSeek-V4-Pro) targeting FlyDSL — deliberately chosen as a DSL essentially absent from training corpora to probe in-context learning rather than recall. Runtime held fixed (Claude Code, except GPT-5.5 on its native Codex runtime) so differences reflect base-model capability, not harness differences. Two non-LLM reference baselines: torch.compile and the deployed production kernel.

**AKA agent architecture.** A routed workflow (7 outer stages) over shared resources (GPU Wiki, profiling/measurement tools, reference templates): (1) user input; (2) router initialization with stable artifact boundary; (3) hardware roofline grounding from GPU Wiki (every hardware value sourced + cited); (4) correctness-first baseline; (5) profile-driven optimization loop (8 stages: profile → extract evidence → query knowledge → evidence-driven plan → single-category edit → validate → memory update → stop check); (6) partial restart via optimization dropout (masks selected iteration memories, preserves accepted kernel + audit trail, lets fresh sub-agent reassess); (7) output contract packaging into a single evaluator-ready kernel artifact.

## Results & Highlights

- **Headline production-weighted Sagg on XPU-A**: best agent GPT-5.5 reaches only **Sagg = 0.107** (10.7% of the hardware roofline) vs. Claude Opus 4.7 at **0.059**; the remaining four sit at Qwen3.7-Max 0.047, Kimi-K2.6 0.043, GLM-5.1 0.015, DeepSeek-V4-Pro 0.012. The strongest agent reaches only ≈10% of the reference-derived roofline.
- **No agent beats the deployed production kernel** on the median operator: Opus 4.7 comes closest at **0.99× production**, GPT-5.5 at **0.85×**, Qwen3.7-Max 0.19×, Kimi-K2.6 0.33×, GLM-5.1 0.33×, DeepSeek-V4-Pro 0.12×. Only Opus 4.7 (2.29×) and GPT-5.5 (3.06×) beat torch.compile on the median operator.
- **Correctness illusion**: Qwen3.7-Max passes **84.8% correctness with only 43.8% FlyDSL adoption**; Kimi-K2.6 has 23 correct operators but only 10 FlyDSL-dominant (>50% device time), and DeepSeek-V4-Pro has 18 vs. 6 — roughly half of correctness is carried by non-DSL paths (PyTorch fallback / precompiled vendor kernels). Opus 4.7 is the exception with 24 correct vs. 26 FlyDSL-dominant.
- **Failure taxonomy over 683 failing units**: 365 compile failures (53.4%), 257 numeric mismatches (37.6%, the largest silent-failure class), 85 compile timeouts, 24 correct-but-slow. GLM-5.1 alone contributes 288 failures (152 raise-before-artifact, 61 compile timeouts, 19 not even valid nn.Modules).
- **Memory- vs. compute-bound regime split**: every model reaches far more of the roof on bandwidth-bound operators (Opus 4.7 memory-bound median 0.207 vs. compute-bound 0.009 — a >20× gap); GPT-5.5 is the only model with a non-trivial compute-bound fraction (median 0.074), which is what lifts it above Opus 4.7 on Sagg since the heaviest operators are compute-bound.
- **Operator difficulty is model-independent**: 9 operators solved by every model, while the hardest `fp8_blockscale_fused_moe` passes only 22.2%; the hard cluster (`fused_rmsnorm_quant` 34.8%, `per_token_group_quant_fp8` 44.7%, `block_scaled_mm` 56.9%, `attention_forward` 45.2%) is uniformly fused-fp8 + matrix-engine kernels.
- **Generation volume does not predict quality**: heaviest generator DeepSeek-V4-Pro (6.56M tokens) is correct on only 18 operators, while GPT-5.5 (1.19M tokens, 46K per correct op) achieves 26 correct — the lightest generator tops the panel.
- **AKA controlled case study (vanilla → optimizer-augmented, single production shape per operator on XPU-A)**: Qwen3.7-Max on `chunk_gated_delta_rule_state` 0%→~100% FlyDSL, S 0.001→0.03, 0.03×→1.20× production; on `attention_forward` 0%→99% FlyDSL, S 0.06→0.40 (6.7× over fallback), 0.17×→1.11× production; on `mla_decode` 14%→87% FlyDSL, 0.10×→1.06× production. Opus 4.7 on `attention_forward` S 0.28→0.42, 0.78×→1.17× production; on `chunk_gated_delta_rule_state` 0%→~100%, 0.04×→1.31× production. All six optimizer-augmented kernels overtake the deployed hand-tuned production kernel.

## Authors & Credibility

Authored by the **ATREX Team (Alibaba Tensor Research & Engineering for XPUs) at Alibaba Group**; Lingyun Yang and Yuxiao Wang are marked as equal contributors. The work is grounded in genuine production access: >10k deployed XPU-A and H20 accelerators, 1,303 production profiles, and ~20 deployed models (Qwen3 MoE variants, DeepSeek-R1, Qwen3-VL, QwQ-32B, Qwen3-32B/8B). Both artifacts are open-sourced at github.com/alibaba/atrex-bench and github.com/alibaba/atrex-kernel-agent. The benchmark methodology reuses established conventions (PyTorch references, three-stage compile/correctness/perf gating, Williams et al. 2009 roofline model) and is positioned honestly against eight prior benchmarks in a capability matrix rather than ranked. The numerical-precision tolerances (atol=1e-2, rtol=5e-2), K=5 random seeds, per-shape subprocess isolation, L2 flush, and frequency-lock check are concrete anti-gaming mechanisms, not hand-waving. The benchmark's stated use of "XPU-A" as a desensitized name for a non-NVIDIA accelerator, and "Claude Opus 4.7" / "GPT-5.5" / "Qwen3.7-Max" etc. as model names, are notable — readers should treat these as future-dated naming conventions in this draft.

## Critical Assessment

**Novelty.** Genuinely novel in two respects: (a) Atrex-Bench is the first kernel-generation benchmark whose problem set is sampled from live production serving traces with importance weights tied to card-hours — KernelBench, BackendBench (partial traces), TritonBench, MultiKernelBench, FlashInfer-Bench, CUDABench, and SOL-ExecBench all lack the importance-weighted + production-sampled combination (Table 1 is a fair positioning matrix, not a stacked ranking). (b) The "correctness illusion" / target-DSL-dominance analysis (FlyDSL device-time share vs. raw correctness) is a real methodological contribution that exposes a measurable reward-hacking pattern other benchmarks conflate.

**Rigor.** The experimental protocol is detailed and reproducible at the artifact level: per-shape subprocess isolation, frequency-lock checks, L2 flush, K=5 seeds, declared tolerances, hidden provenance and roofline. The failure taxonomy (683 units, broken into compile/runtime/perf) and the model-independent operator-difficulty analysis (Table 5) are strong. The memory- vs. compute-bound split is a genuinely diagnostic finding.

**Reproducibility concerns.** Three weak spots. (i) The AKA case study (Table 8) reports **a single timed run per (op, shape)** on a non-power-of-two attention shape (65,556 tokens × 16 heads × 72 head-dim) where the AITER production kernel itself reaches only ≈36% of roof — the 1.06×–1.31× wins over production are on a shape that happens to be unflattering to AITER's general varlen path; generalization to the rest of the 440 shapes is not shown. (ii) Performance numbers are reported **only on XPU-A**; H20 is traced for problem sampling but never used as an evaluation target in this draft, so the roofline-scoring generalization claim across accelerator families is untested. (iii) The missing ablations are notable: no ablation of optimization dropout alone, no ablation of GPU Wiki alone, no isolation of the measure-revise loop from the knowledge base — so we cannot tell which AKA component is doing the work. The 6-op×2-model case study is too small to support the "AKA closes the gap" headline without per-component attribution.

**Significance.** High for the production-kernel-generation subfield: the headline finding (best frontier agent at 10.7% of roofline, no agent matches deployed production) is a credible, deployment-aligned signal that will recalibrate expectations. The correctness-illusion framing is likely to be cited and adopted by subsequent benchmarks.

**Red flags.** The AKA evaluation is described as a "controlled case study, not a benchmark-wide ranking" — this is honest but means the strongest empirical claim of the paper (agent-generated kernels beat hand-tuned production) rests on 3 operators × 2 base models × 1 shape each, on hardware where the production kernel was not specifically tuned for the chosen shape. The aggressive future-dated model names (Opus 4.7, GPT-5.5, Qwen3.7-Max, GLM-5.1, DeepSeek-V4-Pro, Kimi-K2.6) and arxiv ID 2607.14541 (i.e., July 2026) should be read as a forward-dated draft, not yet-validated production evaluations.

## Strengths

- Benchmark contract directly addresses the three production-relevance axes prior benchmarks miss: which shapes (production skew vs. uniform grid), which kernels matter (importance-weighted vs. unweighted average), how good is good enough (per-shape roofline vs. mutable baseline).
- Structural anti-hacking design: hidden provenance and roofline artifacts, plus the FlyDSL-adoption metric that turns invisible PyTorch-fallback passes into a recorded number — a real solution to specification shortcutting.
- The target-DSL choice (FlyDSL, absent from training corpora) cleanly probes in-context learning vs. recall, removing the "model just remembers Triton/CUDA" confound.
- Failure taxonomy over all 683 failing units is unusually thorough and separates compile (53.4%), silent numeric mismatch (37.6%), and correct-but-slow (3.5%) regimes — a useful resource for future work.
- The memory- vs. compute-bound split, the model-independent operator-difficulty ranking, and the "generation volume does not predict quality" analysis are all genuinely diagnostic findings beyond the headline score.
- Both artifacts open-sourced with versioned snapshots and a planned refresh cadence from new production traces.

## Limitations

- **Trace scope**: covers only selected XPU-A and H20 inference clusters with >10k cards; the sampled problems represent that traced deployment slice, not every hardware class. The authors explicitly do not claim representation of future hardware.
- **Inference-only**: training workloads (backward kernels, optimizer-step kernels) are out of scope.
- **Empirical results on XPU-A only**; broader accelerator validation (NVIDIA-side, higher-throughput accelerators) is left as future work.
- **AKA evaluation is a 3-operator × 2-model case study**, not a benchmark-wide ranking; per-component ablations (optimization dropout, GPU Wiki, measure-revise loop) are absent, so attribution is unclear.
- **Single timed run per (op, shape)** in the AKA case study — no variance reported, despite the 1.06×–1.31× wins being narrow on the latency-bound operator.
- **Production-kernel comparisons on shapes the production kernel was not specifically tuned for** (e.g., the awkward 65,556-token attention shape where AITER itself only reaches ≈36% of roof) — the "beats hand-tuned production" framing is shape-dependent.
- FlyDSL is the only target DSL exercised in this release; the prompt framework supports Triton, Gluon, CuteDSL but those are left to future releases.

## Relevance

This paper is highly relevant to anyone tracking LLM coding agents for systems/MLSys work: it provides the first production-trace-grounded benchmark for judging whether agent-generated GPU kernels are deployable, and a concrete agent design (profile-driven measure-revise + knowledge base + optimization dropout) for closing the gap. The correctness-illusion framing (high pass rate carried by PyTorch fallbacks) is a directly transferable lesson for designing any LLM-for-systems benchmark.

## Questions for Further Exploration

- How does AKA perform when each component is ablated in isolation — does optimization dropout, GPU Wiki retrieval, or the measure-revise loop contribute most to the production-parity results on the 3-operator case study?
- Would the "AKA beats hand-tuned production" result hold across the full 440 shapes rather than the single representative shape per operator, especially on shapes where AITER was specifically tuned?
- What happens to the correctness illusion (43.8% FlyDSL adoption for Qwen3.7-Max at 84.8% correctness) if the evaluator disallows PyTorch fallback and precompiled vendor-kernel calls at the compile gate?
- How do the Sagg numbers and the memory-vs-compute-bound gap change on H20 (NVIDIA-side) versus XPU-A, given that H20 was traced for problem sampling but never used for evaluation?
- Can the importance weights wᵢ be refreshed from new production traces without invalidating prior leaderboard results, and how sensitive is Sagg to the choice of traced cluster mix (vLLM/SGLang/AITER/RTP-LLM)?
- Does targeting Triton/CUDA/CuteDSL (DSLs present in training corpora) instead of FlyDSL change the relative ordering of the six agents, or does the compute-bound roofline gap persist regardless of DSL familiarity?
