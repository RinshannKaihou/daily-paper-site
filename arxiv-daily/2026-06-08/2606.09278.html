Internalizing Geometric Law: Learning from Solver Residuals for Precision-Critical Generation

["Rafael Cabral", "Pang Zixi", "Ziyi Shou", "Shen Xin"]

arXiv: 2606.09278

Date: 2026-06-08

一句话总结： 本文把几何约束求解（GCS）做成 LLM 对齐任务，发布可微分几何 DSL 引擎 PyGeoX 与 300 题基准 PyGeoX-Bench；提出"饱和累加奖励"（SAR），通过逐约束有界核求和解决"离群梯度掩蔽"问题，使 Qwen3-8B 在 Hard 层求解率比 MSE 基线提升 2.3×（RL 0.41 vs 0.18），并与更大的前沿系统竞争力相当。
TL;DR: The authors formulate end-to-end Geometric Constraint Solving (natural-language description → exact metric coordinates) as an RLVR alignment task and release PyGeoX (a differentiable geometric DSL/engine, 35 object types × 38 relationships) plus PyGeoX-Bench (300 problems, Easy/Medium/Hard tiers) and PyGeoX-Wild (86 OOD problems). They identify a failure mode "Outlier Gradient Masking" where a single severely-violated constraint drives global-norm rewards (e.g., exp(−MSE)) to zero, nullifying the policy gradient from all satisfied constraints, and propose Saturating Additive Rewards (SAR) = Σ_i exp(−r_i/T). On Qwen3-8B via GRPO, the composite SAR+S+D reward raises Hard-tier solving rate (SR) to 0.41 vs 0.18 for MSE+S+D (2.3×) and 0.35 for the field-standard sparse I[‖r‖²₂<10⁻³] baseline; on PyGeoX-Wild (OOD) SAR+S+D reaches 0.66 RL vs 0.55 MSE+S+D and 0.59 sparse. The resulting 8B model (Hard SR 0.41) beats DeepSeek-V3.2 (0.25) and two of three proprietary frontier systems on the Hard tier, where the strongest baseline reaches 0.51.

核心贡献

  • 把端到端几何约束求解（自然语言描述 → 精确度量坐标）定义为一个全新的 LLM 对齐任务：模型直接输出 Python 代码给出满足所有约束的坐标，而非把几何推理委托给外部求解器；现有基准均不覆盖此输入/输出签名。
  • 发布 PyGeoX：可编程几何 DSL 引擎，支持 35 种对象、38 种关系，将声明式约束经 SymPy 后端编译为可微损失（Numba JIT 加速 10–50×），并支持不等式、多边形关系等先前求解器缺失的表达力；既用于数据生成又用作 RL Gym。
  • 提供数据管线（~100k 已验证训练题）和两个基准：PyGeoX-Bench（300 题，Easy/Medium/Hard 各 100，按 8/10/16 个约束分层）与 PyGeoX-Wild（86 题人工撰写散文的 OOD 诊断集）。
  • 识别并形式化"离群梯度掩蔽"失败模式：在全局范数奖励（exp(−‖r‖_p)、二值稀疏）下，单个严重违约约束会把聚合奖励压到 0，抹掉所有已满足约束的学习信号；给出高维信号体积定理（Theorem A.1：全局范数有效奖励体积随约束数 C→∞ 趋 0，SAR 趋 1）与梯度定理（Theorem A.3：全局范数梯度 ∥∇_ri R^G∥ = |ϕ′(L)|·(r_i/L)^(p−1) 与全局误差耦合，SAR 梯度 ∥∇_ri R^SAR∥ = |ϕ′(r_i)| 严格局部）。
  • 提出 SAR（饱和累加奖励）R = Σ_i exp(−r_i/T) 及其组合形式（加稀疏成功奖励 + 退化惩罚），证明其在 Hard 层显著优于领域标准的稀疏与全局范数奖励。

Key Contributions

  • End-to-end GCS as a new LLM alignment task. The model emits Python that produces a structured dictionary of exact point coordinates / circle radii satisfying every constraint, with a single-turn code-agent interface (numpy/scipy/sympy sandbox, 90s timeout); PyGeoX is used off-policy only (data generation + per-constraint residual reward), never as an in-context tool, so the agent cannot learn DSL syntax instead of spatial reasoning.
  • The PyGeoX engine. A declarative→symbolic (SymPy)→differentiable (Numba-JIT) pipeline covering 35 object types and 38 relationships, with a scalar error function E(u) aggregating equality / non-strict-inequality / strict-inequality / not-equal constraint classes; supports arbitrary algebraic constraints on any derived scalar property (e.g., pentagon.perimeter = circle.area), which AlphaGeometry/FormalGeo's fixed predicate sets cannot express.
  • Data + benchmarks. A procedural pipeline (seeded by Qwen3-30B-A3B) yielding ~100k validated training problems stratified into Easy (single-polygon, ~13 objects, 8 constraints), Medium (two-polygon, ~15 objects, 10 constraints), Hard (three-polygon, ~23 objects, 16 constraints); plus PyGeoX-Bench (300 held-out problems) and PyGeoX-Wild (86 human-authored OOD problems).
  • Outlier Gradient Masking, formalized. In RL the reward is only a scalar multiplier on ∇θ log πθ, so a global-norm reward collapses to ~0 when any single residual r_i≫0; proven in two regimes — high-dimensional signal volume (Theorem A.1: Vol(V^G_ϵ)/Vol(Ω)→0 vs Vol(V^SAR_ϵ)/Vol(Ω)→1 as C→∞) and gradient sensitivity (Theorem A.3: global gradient coupled to total error L, SAR gradient strictly local).
  • Saturating Additive Rewards (SAR). R_SAR = Σ_i ϕ(r_i) for monotonically-decreasing bounded kernel ϕ (Boltzmann exp(−r/T), T=0.1; also Cauchy/sigmoidal variants); composite reward adds a sparse success bonus I_suc (‖r‖²₂<10⁻³) and a degeneracy penalty min(4, C_deg), bounding reward in [−4, 10] with w=6.0, R_bonus=4.0.

Methodology

Task formulation (§1.1). Given an NL description x of geometric objects and relational constraints, produce a Python program whose output is a structured dict of exact point coordinates and circle radii satisfying all constraints simultaneously (‖r‖²₂<10⁻³ to allow floating-point propagation). The agent operates single-turn with numpy/scipy/sympy; it may use constructive ruler-and-compass, numerical optimization, or hybrids. The LLM never calls PyGeoX in-context.

PyGeoX engine (§4.1, App. B). Declarative DSL (scene.add / scene.relate / scene.constraint) → lazy SymPy symbolic buffer (handles equalities, non-strict G_j≥0, strict S_k>0, not-equal N_ℓ≠0) → aggregated differentiable loss E(u)= αΣE_i² + βΣmax(0,−G_j)² + βΣmax(0,ε−S_k)² + γΣ1_{|N_ℓ|<ε}, default weights α=β=γ=1.0, ε=10⁻⁴, compiled via Numba JIT (10–50× over Python). Data generation uses Basin-hopping (1000 iters, step 0.5, T=2, L-BFGS-B local refinement) with an optional separation penalty L_sep=δΣmax(0, d²_min−‖p_i−p_j‖²) where d_min=1/200 of domain span, δ=1.0, to prevent coordinate collapse; configurations that fail to converge or degenerate are auto-discarded.

Data generation (§4.2). Seeds (n objects, m relationships, k extra constraints sampled from the 35/38 vocabulary) are expanded by a Qwen3-30B-A3B LLM assembler into full PyGeoX specs, validated by the numerical solver; ~100k validated problems. Each sample is a 4-tuple (NL description, PyGeoX DSL code, compiled per-constraint reward function, rendered image). Difficulty tiers: Easy ~13 objects/8 constraints, Medium ~15 objects/10 constraints, Hard ~23 objects/16 constraints.

SAR reward design (§3.2–3.3). R_SAR = Σ_{i=1..C} exp(−r_i/T), T=0.1. Composite reward (Eq. 2): R = w·Σ exp(−r_i/T) + I_suc·R_bonus − min(4, C_deg) with w=6.0, T=0.1, R_bonus=4.0, I_suc active when ‖r‖²₂<10⁻³. The sparse bonus overcomes the SAR reward plateau (a config satisfying 14/16 constraints gets high dense reward but is still "wrong"); the degeneracy penalty prunes trivial solutions (all points collapsed to one coordinate). Compared baselines: MSE reward exp(−SSE/T_mse) with T_mse=10 (chosen to maximize reward spread, i.e., favorably tuned for MSE); Sparse I[‖r‖²₂<10⁻³].

Evaluation. One-shot solving rate (SR) = fraction of problems with ‖r‖²₂<10⁻³ on PyGeoX-Bench (300) and PyGeoX-Wild (86).

Results & Highlights

  • SAR+S+D beats the field-standard sparse RLVR baseline (Table 1). On Hard tier, RL SAR+S+D reaches SR 0.41 vs sparse 0.35; SFT 0.32 vs sparse 0.23. On PyGeoX-Wild, RL 0.66 vs 0.59; SFT 0.65 vs 0.57.
  • SAR crushes MSE on the hard tier (Table 1). RL SAR+S+D 0.41 vs MSE+S+D 0.18 (a 2.3× gap); SFT SAR+S+D 0.32 vs MSE+S+D 0.04 (8×). Strikingly, MSE+S+D (0.18 RL) underperforms sparse alone (0.35 RL) — i.e., replacing SAR's per-constraint kernel with the global-norm MSE objective actively harms training, exactly as the gradient-masking analysis predicts.
  • The composite (dense + sparse) design is essential. Pure dense rewards in isolation underperform: SAR-alone reaches only 0.09–0.10 Hard SR under RL and MSE-alone 0.10, well below sparse alone (0.35); the sparse outcome bonus acts as a hard correctness gate while SAR supplies the partial-progress signal that lets learning proceed on problems the sparse indicator alone reports as failures.
  • Reward discrimination empirically confirms gradient masking (Table 8, 3,893 partially-correct solutions). 97.1% of SAR rewards fall in the useful [0.1, 0.9] range vs only 19.0% for MSE (and 60.4% of MSE rewards collapse to near-zero <0.1, making most partial solutions indistinguishable to GRPO); sparse puts 100% in the collapsed-~0 bucket.
  • The 8B model is competitive with frontier systems on the Hard tier (Table 9, zero-shot). Qwen3-8B-RL SAR+S+D reaches Hard SR 0.41, beating DeepSeek-V3.2 (0.25), Proprietary-A (0.37), and Proprietary-B reasoning (0.38); the strongest baseline Proprietary-C reasoning reaches 0.51. On PyGeoX-Wild, the 8B model (0.66) is below Proprietary-C (0.81) and Proprietary-B (0.77) but above DeepSeek-V3.2 (0.64).
  • Token efficiency (Fig. 2). SAR+S+D-trained model averages ~4,060 generated tokens/task vs ~5,260 for the sparse baseline (a 22.8% reduction), finding geometric solutions more directly without lengthy numerical-search traces.
  • Evidence against memorization (§6). ~90% of successful Hard-tier traces follow constructive ruler-and-compass derivations; SAR+S+D obtains the highest PyGeoX-Wild SR under three-way distribution shift (human prose, unseen constraint types, novel multi-step chains); a conservative lower bound on distinct Hard-tier configurations exceeds 10¹⁷, dwarfing the ~10k RL training problems by 13 orders of magnitude.

Critical Assessment

  • All RL and SFT results use a single base model (Qwen3-8B) and a single seed (42). The authors themselves flag this: SFT on Qwen3-1.7B and Llama-3.1-8B-Instruct failed (Hard SR ≤ 0.02, unstable training), and a multi-scale study across Qwen3-4B/14B, Mistral-7B, Gemma-4 "exceeded our compute budget." So the central claim "SAR beats MSE/sparse" is established for one model family at one scale with no variance bars across seeds — the relative ranking could be seed-sensitive, and generalization to other base models is asserted (PyGeoX-Wild OOD) but not directly demonstrated across architectures.
  • The MSE baseline, while favorably tuned (T_mse=10 "chosen to maximize reward spread"), is still arguably a weak comparator for the headline 2.3×. The authors note MSE still collapses 60% of partially-correct solutions to near-zero and call the comparison "conservative rather than a strawman," but a practitioner's first dense reward would more likely use a per-constraint mean or a clipped/normalized variant rather than raw exp(−SSE/10) over 16 squared residuals — the paper does not benchmark such intermediate aggregators, so the MSE-vs-SAR gap may overstate the gain relative to a better-engineered global reward.
  • The frontier-model comparison (Table 9) is uncontrolled and uses vendor-default settings. The 8B RL model is compared zero-shot against DeepSeek-V3.2 / Proprietary-A/B/C with their default or "reasoning" inference modes, no prompt engineering, and no tool sandbox parity (frontier systems may not have been given the same numpy/scipy execution environment the trained 8B model relies on); the authors acknowledge this is "to contextualize task difficulty, not as a controlled baseline," so the "competitive with frontier" framing should not be over-read.
  • PyGeoX-Wild is small (86 problems) and its source citation is withheld "for anonymity," which makes the OOD generalization claim hard to independently verify; with n=86 the SR differences (e.g., 0.66 vs 0.60) are within ~2-problem granularity, and the paper does not report confidence intervals on any SR.
  • PyGeoX targets only 2D static geometry. The engine cannot express kinematic synthesis or 3D CAD primitives, so the claim of relevance to "engineering domains like kinematic synthesis" (motivation in §1) is aspirational rather than demonstrated by any engineering experiment.

Strengths

  • Clean problem formulation: end-to-end GCS as RLVR with the LLM emitting coordinates (not delegating to a DSL) is genuinely novel and well-motivated by the precision-hallucination gap.
  • The Outlier Gradient Masking analysis is both formal (Theorems A.1/A.3 with proofs in Appendix A) and empirically validated (Table 8: 97% SAR rewards in useful range vs 60% MSE collapsed) — the failure mode is demonstrated, not just asserted.
  • SAR is a simple, general, well-motivated fix (sum of bounded per-constraint kernels) with a clear composite design rationale (dense shaping + sparse correctness gate + degeneracy penalty), and the authors honestly show pure-SAR alone underperforms, so the composite is essential rather than a free lunch.
  • Strong release: engine, DSL, 100k training problems, two benchmarks, checkpoints, and configs to reproduce every Table 1 row, at a public GitHub URL.

Limitations

  • Single base model (Qwen3-8B) and single seed (42); no variance reported; smaller/different base models failed.
  • MSE comparator is favorably tuned but still arguably weak; no intermediate global aggregators (mean, clipped, normalized) benchmarked.
  • Frontier comparison (Table 9) is uncontrolled, vendor-default, likely without tool-sandbox parity.
  • PyGeoX-Wild is only 86 problems with withheld citation and no confidence intervals on any SR.
  • Engine is 2D static only; no kinematic/3D/CAD engineering experiment despite the motivation.
  • Evaluation is one-shot (no pass@k), so SR reflects a single sample per problem.

Questions for Further Exploration

  • The headline 2.3× (RL Hard: SAR+S+D 0.41 vs MSE+S+D 0.18) rests on a single seed (42) on Qwen3-8B — does the SAR-beats-MSE ranking survive across multiple seeds and across the other base models the authors tried but found too weak (Qwen3-1.7B, Llama-3.1-8B), or does it interact with base-model math/code capacity?
  • The paper cites Tao et al. 2025 ("Hybrid reinforcement: When reward is sparse, it's better to be dense") as the inspiration for the composite dense+sparse design — would the SAR kernel specifically (vs Tao et al.'s dense formulation) change the PyGeoX-Wild OOD ranking, given the authors claim SAR is the differentiator but never ablate against a non-saturating dense+sparse baseline?
  • The MSE baseline uses exp(−SSE/T_mse) with T_mse=10 tuned to maximize spread — how does SAR compare against a per-constraint mean reward (Σ r_i²/C transformed) or a clipped/normalized global reward, which a practitioner might try before SAR, since the paper only benchmarks the two extremes (global norm vs per-constraint saturating sum)?
  • PyGeoX-Wild is 86 problems adapted from a middle-school geometry benchmark whose citation is withheld — once released, do the OOD SR differences (e.g., 8B model 0.66 vs DeepSeek-V3.2 0.64) remain significant given n=86 implies ~1-problem granularity?
  • The frontier-model comparison (Table 9) uses vendor-default settings without tool-sandbox parity — if DeepSeek-V3.2 / Proprietary-C were given the same numpy/scipy/sympy execution environment the 8B model relies on, would the Hard-tier gap (0.41 vs 0.25/0.51) close?
