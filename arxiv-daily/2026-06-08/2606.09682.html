AutoMegaKernel: A Statically-Checked Agent Harness for Self-Retargeting Megakernel Synthesis

["Jaber Jaber", "Osama Jaber"]

arXiv: 2606.09682

Date: 2026-06-08

https://arxiv.org/abs/2606.09682

一句话总结： AutoMegaKernel（AMK）将 HuggingFace Llama 系列模型整个前向过程编译为单一持久协作 CUDA megakernel（无手工 CUDA），其冻结的 schedule-IR 验证器在 7,160 个对抗调度（6,091 不安全）中零误判地拒绝所有不安全调度，并在 SmolLM2-135M 上逐 token 复现 HuggingFace greedy 解码、perplexity 误差 2.5×10⁻⁷；自动生成的 int8 权重量化 megakernel 在推理级 GPU（L4 最高 1.33×、L40S 1.25–1.27×、A10G 1.08×、RTX 5090 1.19–1.23×）上以 W8A16 对比 cuBLAS bf16 取得真实优势。
TL;DR: AMK compiles a HuggingFace Llama-family model into a single persistent cooperative CUDA kernel with no per-model hand-written CUDA, and makes correctness a static property: a frozen schedule-IR validator had zero false-accepts across 7,160 adversarial schedules (6,091 unsafe) while accepting all 360 real lowerings (24 bit-exact vs eager). On the real SmolLM2-135M checkpoint it reproduces HuggingFace greedy decode token-for-token and matches teacher-forced perplexity to 2.5×10⁻⁷. A search-found int8 weight-only (W8A16) megakernel beats CUDA-graphed cuBLAS bf16 at batch-1 on the NVIDIA L4 by up to 1.33×, L40S by 1.25–1.27×, A10G by up to 1.08×, and RTX 5090 by 1.19–1.23× (a precision-asymmetric win that holds on inference-class but not training-class A100/H100 silicon, disclosed honestly).

核心贡献

  • 静态安全门控的 megakernel 编译器：冻结的 validate()（schedule/ir.py, 1150 行）通过一组静态图检查（DAG 无环、wait 满足性 1≤t≤#producers、每 SM 队列序、shared-counter all-join、传递 happens-before 等）证明无死锁与无竞态；在 7,160 个对抗调度中零误判（接受全部 360 个真实 lowering）。
  • 零手工 CUDA 的自动生成：对全部 10 个支持模型（含 SmolLM2-135M/360M、TinyLlama-1.1B 真实 checkpoint，IR 任务数最高 3,410）自动生成正确 megakernel，并在 SmolLM2-135M 上逐 token 匹配 HuggingFace greedy 解码、perplexity 误差 2.5×10⁻⁷。
  • 自重定向多架构：同一份源码在 sm_80（A100）、sm_90（H100）、sm_120（RTX 5090）上生成正确 megakernel，gencode 由设备自动派生，fp32 误差 ≤4.2×10⁻⁷。
  • int8/int4 权重量化自动生成：int8（W8A16）greedy 无损且每 token 快 1.12×；int4 将权重流量下限降 2.42×（混合）但有精度代价。
  • 无人值守 agent 自改进循环：propose→validate→correctness-gate→measure→keep/revert，带漂移鲁棒的 per-sample 交错测量与物理 roofline-floor 诚实守卫，自身基线上获得 1.25–1.72× 提升。

Key Contributions

  • A statically-checked schedule IR with a frozen validator that certifies deadlock-freedom and race-freedom before any launch via static graph checks (not a mechanized proof), stress-tested to zero false-accepts across 7,160 adversarial schedules (6,091 oracle-unsafe) while accepting all 360 real lowerings (24 re-run bit-for-bit vs eager).
  • Automatic generation coverage: a correct megakernel is auto-generated for 10 of 10 supported models with zero hand-written CUDA, including three real HF checkpoints up to TinyLlama-1.1B (3,410 IR tasks), token-for-token vs eager; 3 of 4 unsupported variants are rejected loudly at import.
  • Self-retargeting: the same source builds and runs correct megakernels on sm_120 (RTX 5090), sm_80 (A100), and sm_90 (H100), matching eager to ≤4.2×10⁻⁷ (fp32) on toy/synthetic and 3.8×10⁻⁵ on the real SmolLM2-135M checkpoint.
  • int8/int4 weight-only quantized megakernel auto-generation: int8 (W8A16) is greedy-lossless and 1.12× faster per token; int4 cuts the weight-traffic floor 2.42× (blended) at a documented accuracy cost.
  • An unattended agent-drivable autoresearch loop that autonomously improves the megakernel over its own baseline (1.25–1.72× measured) with drift-robust per-sample-interleaved measurement and a physical-roofline-floor honesty guard.

Methodology

System architecture (four layers + two loops). Layer 0 is the VM (vm/scheduler.cu, 132 lines; vm/loader.py, 857 lines): the persistent cooperative megakernel — one cudaLaunchCooperativeKernel co-resides one threadblock per SM; each block owns one SM-queue and walks it in global topological order executing wait→dispatch→signal. It is hand-written, exhaustively verified, and frozen per architecture. Layer 1 (instructions/) holds ABI-conformant micro-kernels (GEMV, attention, RMSNorm, RoPE), each correctness-checked against its reference op in isolation. Layer 2 (schedule/ir.py 1150 lines; schedule/graph.py 548 lines) imports an HF model, lowers it to a tiled task-DAG over an SM-level DAG synchronized by monotonic uint32 counters, and validates it. Layer 3 is a roadmap placeholder.

Synchronization model. Cross-task signalling is only through monotonic uint32 counters: each task on completion issues a device-scope release fence then atomicAdd(+1) on exactly one out_counter; before executing, a task waits on a set of (counter, threshold) pairs with statically known thresholds via an acquire-load spin with exponential backoff and a WDDM watchdog escape. No locks, no arbitrary signalling. One kernel launch = one forward pass = one decoded token; counters are host-memset to zero before each launch and the KV cache persists in HBM across launches.

Validator (validate()). Returns a ValidationResult and never raises; a rejected result prevents launch. Checks (Table 2): referential integrity/arity/params, ABI capacity caps (inputs>8, outputs>4, waits>8, rank>4), wait satisfiability (1≤t≤#producers), DAG acyclicity (Kahn's algorithm, safe at 5000+ nodes), per-SM queue order (linear extension of DAG), shared-counter all-join (rejects partial waits 1

The ScheduleConfig edit surface. The Layer-2 agent proposes a structured object, not kernel code: tiling, fusion_grouping, sm_assignment (round_robin/load_balance/explicit), pipelining_depth, page_allocation, threads_per_block, smem_bytes_per_block. The frozen VM lowers any point deterministically and validate() guarantees safety regardless of the point chosen.

Hardware. Seven GPUs: RTX 5090 Laptop (sm_120, 82 SMs, 896 GB/s spec / 731 GB/s measured), A100-SXM4-40GB (sm_80, 1555/1383 GB/s), H100-80GB-HBM3 (sm_90, 3350/3089 GB/s), plus inference-class L4 (sm_89, 300 GB/s), L40S (sm_89, 864 GB/s), A10G (sm_86, 600 GB/s), T4 (sm_75, 320 GB/s). torch 2.11.0+cu128; datacenter runs on Modal.

Measurement. CUDA-event timing, 25 warmup + 100 iterations, median with p10/p90; clocks unpinned in the main study (clock-pinned re-measurement within ±0.8 pp). Every latency is correctness-gated (eval/bench.py refuses to emit without a PASS vs eager oracle). int8-vs-cuBLAS ratios are kernel-only, per-sample paired-interleaved (AMK vm.relaunch whole forward vs cuBLAS g.replay of a captured graph), ratio = cuBLAS/AMK so drift cancels.

Results & Highlights

  • Validator soundness (headline safety claim): across 6,091 oracle-confirmed-unsafe schedules, validate() produced zero false-accepts (rate 0.0000%), rejecting all 6,091 while accepting all 360 real lowerings; 24 of 24 re-lowered accepted schedules ran in the ReferenceVM and matched eager PyTorch bit-for-bit (Table 5). Validation runs at ~5,150 schedules/s on CPU.
  • Numerical correctness on real checkpoint: on SmolLM2-135M, AMK matches HuggingFace teacher-forced perplexity to 6 significant figures (14.948473 vs 14.948473, gap 2.45×10⁻⁷), 64-token greedy decode is byte-identical to model.generate (64/64 tokens); on A100/H100 the megakernel reproduces HF greedy decode token-for-token (divergence index 8 of 8) and matches eager to ≤3.2×10⁻⁷.
  • Inference-fleet int8 win (W8A16 vs CUDA-graphed cuBLAS bf16, batch-1): L4 1.18× (1.3B) → 1.33× (4B, peak, still climbing); L40S 1.25× (4B) → 1.27× (6.7B); A10G parity (2.7B) → 1.08× (4B); consumer RTX 5090 1.19× (4L/623MB) → 1.23× (8L/984MB) — all with p10 > 1 at every size (Table 10).
  • Quantization trade-offs (RTX 5090, 4-layer model): int8 1371.2 µs/token vs bf16 1537.4 (1.12× per-token, 1.18× kernel-only, greedy-lossless 32/32 tokens); int4 1450.6 µs/token (1.06×) but lossy (~22% token agreement) — naive round-to-nearest nibble-unpack is compute-bound (only 0.18× cuBLAS on L4).
  • Autoresearch self-improvement: cold campaign reaches 1.72× drift-free (default 2168→1261 µs at iteration 2); agent-in-the-loop drift-robust median 1.25× (2514→1991 µs, 12 experiments, 6 kept); a 10-minute campaign self-improves 1.47× over AMK's own start and reaches 0.88× of cuBLAS-graphed eager (within ~13%).
  • Honest boundary (loses): on training-class A100/H100 the int8 megakernel trails cuBLAS (A100 ~0.79× at 1.3B down to ~0.55× at 13B; H100 ~0.72× down to ~0.60×); the harness localizes the cause as per-tile cross-SM synchronization (both a cp.async staging ring and split-KV attention regressed). Whole-decode H100 SmolLM2-135M: AMK 14.30 ms/token vs vLLM cudagraph 8.65 ms (1.65× slower).
  • Kernel-quality gap (disclosed): optimized bf16 GEMV reaches ~460 GB/s (~51% spec / ~63% measured peak) vs cuBLAS ceiling ~661 GB/s (~90% measured peak); whole-decode loses to CUDA-graphed cuBLAS by 1.13× (~13%).

Critical Assessment

  • The headline cuBLAS win is precision-asymmetric (W8A16 vs bf16), not a faster kernel. The authors flag this honestly, but the abstract's framing ("beats cuBLAS bf16") could mislead: the equal-precision control (AMK bf16 vs cuBLAS bf16) trails cuBLAS at every depth (0.76–0.88×, Figure 6), so the win comes entirely from streaming ~0.61× the weight bytes via near-lossless int8, not from a better per-byte GEMV. A Marlin-class hand-built quantized GEMV would likely widen the margin further and is exactly the Layer-1 micro-kernel AMK does not yet have.
  • Largest real checkpoint is TinyLlama-1.1B; the fleet speed sweeps use random-initialized weights. The int8-fleet wins in Table 10 are on random-initialized, Llama-shaped models (justified because batch-1 latency is shape/byte-determined), but the near-lossless W8A16 quality is only established on the 135M SmolLM2 checkpoint — there is no real-weight quality validation at 4B/6.7B where the peak speedups are claimed.
  • Validator is empirical soundness over a trusted base, not formal verification. The authors state this plainly, but it means the "zero false-accepts" claim holds only over the 8 injected unsafe classes and the random-DAG generator; a ninth hazard class not in the test population is not covered. The Qwen2 silent-accept bug (hardcoded q/k/v biases pass the config-only check, logit error 2.47) is a concrete instance of the config-inspection blind spot.
  • vLLM comparison uses dtype=float32, not vLLM's bf16 default. The H100 row (AMK 14.30 ms vs vLLM cudagraph 8.65 ms, AMK 1.65× slower) is the only fair default-mode comparison and AMK loses; the A100 row is enforce_eager only (no CUDA-graph baseline collected). Combined with the self-funded compute constraint, the datacenter performance gap partly reflects optimization effort not afforded rather than a fundamental ceiling.

Strengths

  • Genuinely novel static-safety gate: turning "an agent can edit the schedule" from a hazard into a safe operation, with the 7,160-schedule stress test providing empirical evidence (zero false-accepts) that MPK and hand-built megakernels do not provide.
  • Exceptional honesty: the paper discloses every losing direction (A100/H100 deficit, vLLM loss, int4 lossiness, cuBLAS kernel-quality gap, Qwen2 blind spot) rather than cherry-picking, and reports both spec and measured roofline denominators.
  • Correctness-by-construction architecture (typed schedule IR + counter-synced task-DAG + frozen VM) that decouples the agent's edit surface from the trusted base — a clean design for safe agent-driven systems.
  • Real-checkpoint numerical validation (token-for-token HF greedy match, 6-sig-fig perplexity) is stronger than the typical systems-paper "looks close" check.

Limitations

  • Architecture coverage is the bias-free, full-rotary, SiLU-SwiGLU, RMSNorm, GQA Llama family only; MoE routing, sliding-window/fused-QKV attention, partial/scaled RoPE, and biased projections are rejected at import (one documented Qwen2 silent-accept exception).
  • Only batch-1, position-0, empty-KV decode is measured; long-context per-token latency (where KV traffic becomes a second bandwidth term) is not measured.
  • ncu/Nsight hardware counters were unavailable on the Modal account, so all utilization figures are wall-clock + analytic roofline, not measured DRAM-throughput counters.
  • Self-funded compute bounded the experimental scope: no long sustained autotuning campaigns or iterative tensor-core/DP4A GEMV development, so the datacenter gap partly reflects effort, not a fundamental ceiling.

Relevance

本文与"Computation Reliability & Precision Errors"高度相关：它系统比较了 bf16/int8(W8A16)/int4 权重量化在不同 GPU 架构上的精度保真度（int8 greedy 无损、int4 约 22% token 不一致）与速度，并量化了精度非对称比较（W8A16 vs bf16）对结论的影响。同时其冻结的静态验证器（zero false-accepts 保证无死锁/无竞态）是"ML Reliability & Mathematical Foundations"中关于形式化/静态安全保证的典型案例；无人值守的 agent 自改进循环直接映射到"Self-Evolving Agents & Automated Research"。

Questions for Further Exploration

  • The fleet int8 speedups (Table 10) are on random-initialized Llama shapes; would the near-lossless W8A16 quality (argmax-exact, greedy-token-identical) hold at the 4B/6.7B sizes where the peak 1.33× is claimed, or does weight distribution shift with scale?
  • The validator is empirical soundness over 8 injected unsafe classes — does adding a ninth hazard class (e.g., the Qwen2 hardcoded-bias config-inspection blind spot, which silently passed with logit error 2.47) reveal further gaps that a state-dict bias scan alone would not close?
  • The diagnosed A100/H100 boundary is per-tile cross-SM synchronization (both cp.async staging and split-KV attention regressed) — would a coarser-sync scheduler (fewer grid-wide barriers per layer) extend the int8 win onto training-class silicon, as the paper hypothesizes?
  • The paper cites Marlin [11] as the hand-built quantized GEMV that would close the kernel-quality gap; can the agent-driven loop autonomously discover and consume a Marlin-class Layer-1 micro-kernel, or does that step still require human kernel engineering?
