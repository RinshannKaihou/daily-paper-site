AutoMegaKernel: A Statically-Checked Agent Harness for Self-Retargeting Megakernel Synthesis

⚠️ Abstract-only summary

一句话总结： AutoMegaKernel 将 HuggingFace Llama 模型编译为单一 CUDA megakernel，通过静态图检查保证无死锁/无竞态，并在 int8 精度下于推理级 GPU 上超越 cuBLAS bf16。
TL;DR: Compiles entire LLM forward passes into single CUDA kernels with static safety validation; achieves 1.19-1.33x speedup over cuBLAS bf16 on inference-class GPUs (L4, L40S, RTX 5090) using W8A16 quantization.

核心贡献

  • 首个将 Llama 系列模型编译为单一持久协作 CUDA megakernel 的系统，无需手工 CUDA 代码
  • 静态 IR 验证器通过图检查（非机械化证明）保证无死锁和无竞态：7160 个对抗调度中零误报，360 个真实 lowering 全部通过
  • 同一代码库自动适配 sm_80/sm_90/sm_120 三代 GPU 架构，自动生成 10/10 模型的正确 megakernel
  • Agent 驱动的自改进循环可自动优化 megakernel（1.25-1.72x 加速）
  • int8 (W8A16) megakernel 在推理级 GPU 上超越 CUDA-graphed cuBLAS bf16

Key Contributions

  • First system to compile HuggingFace Llama-family models into single persistent cooperative CUDA kernels with no per-model hand-written CUDA
  • Frozen schedule-IR validator certifies deadlock-freedom and race-freedom via static graph checks (zero false-accepts across 7,160 adversarial schedules)
  • Single codebase retargets sm_80/sm_90/sm_120, auto-generates correct megakernels for all 10 supported models
  • Agent-driven autoresearch loop self-improves megakernels (1.25-1.72x over baseline)
  • W8A16 megakernel beats CUDA-graphed cuBLAS bf16 at batch-1 decode on inference-class GPUs

Methodology

AMK takes a HuggingFace Llama-family model and compiles the entire forward pass into a single CUDA kernel launch. A schedule-IR layer enables an agent to propose kernel schedules, which are then validated by a frozen static checker that performs graph-based analysis for deadlock and race conditions. The system supports multi-architecture targeting (sm_80/sm_90/sm_120) from one codebase. An unattended agent-driven autoresearch loop iteratively improves the generated megakernels. The int8 variant uses weight-only quantization (W8A16) for decode workloads.

Results & Highlights

  • Safety validation: 0 false-accepts out of 6,091 unsafe schedules; all 360 real lowerings accepted
  • Numerical correctness: Reproduces HuggingFace greedy decode token-for-token on SmolLM2-135M (perplexity match 2.5e-7)
  • Inference-class GPU speedup (W8A16 vs cuBLAS bf16):
  • L4: up to 1.33x
  • L40S: 1.25-1.27x
  • A10G: up to 1.08x
  • RTX 5090: 1.19-1.23x
  • Training-class GPU gap: Trails cuBLAS on A100/H100 (bandwidth-bound, cross-SM-sync bottleneck reported honestly)

Critical Assessment

  • Novelty: High — the combination of static safety checking + agent-driven kernel synthesis + single-kernel compilation is novel
  • Rigor: Strong — adversarial evaluation with 7,160 schedules, numerical verification via perplexity matching
  • Reproducibility: Good — open-source code, clear hardware specs, honest reporting of where the system trails
  • Significance: High for inference deployment — the inference vs training-class GPU distinction is an important insight
  • Red flags: Largest real checkpoint tested is TinyLlama-1.1B; unclear if the approach scales to 7B/70B models. The comparison is precision-asymmetric (W8A16 vs bf16) which is somewhat unfair

Strengths

  • Zero-trust safety model: unsafe agent proposals are caught before launch
  • Honest about limitations on training-class GPUs (A100/H100)
  • Practical focus on inference-class GPUs where deployment economics matter most
  • Multi-architecture support from single codebase

Limitations

  • Only tested up to TinyLlama-1.1B — scaling to larger models is unproven
  • Precision-asymmetric comparison (W8A16 vs bf16) complicates interpretation
  • Static checking is graph-based, not a formal mechanized proof
  • Trails cuBLAS on high-bandwidth training GPUs
