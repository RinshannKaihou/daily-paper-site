Toward Compiler World Models: Learning Latent Dynamics for Efficient Tensor Program Search

["Haolin Pan", "Lianghong Huang", "Xulin Zhou", "Mingjie Xing", "Yanjun Wu"]

arXiv: 2606.09312

Date: 2026-06-08

Tags: ["ML", "cs.PL", "Tensor Compilers", "Auto-Scheduling", "World Models", "TVM", "Cost Model"]

一句话总结： 本文将张量程序调度评估重新建模为"世界模型"式的动作条件隐状态演化——用 CodeBERT 编码 TensorIR 初始状态、用 TransH 风格转移模型逐步 rollout 调度动作、用 XGBoost LambdaRank 排序候选——在 TVM AutoScheduler 中于 Intel Xeon Gold 6430 CPU 与 NVIDIA RTX 4090 GPU 上实现：相同 64-trial 预算下相对 Ansor 在 GPU/CPU 上代表子图延迟加速 1.37×/1.54×，用 10× 更少测量在算术均值上略超 Ansor-10K、几何均值差距仅 2.2%，整机推理相对 PyTorch/cuDNN 几何均值加速 4.61×/3.67×。
TL;DR: The authors recast tensor-program schedule evaluation as action-conditioned latent dynamics over program states (a "compiler world model"): a CodeBERT-based encoder maps a TensorIR state to a dense vector, a TransH-style transition model F(z_{t−1}, a_t) rolls out the scheduling-action sequence step-by-step in representation space, and an XGBoost LambdaRank ranker scores the predicted terminal-state embedding + structured action features + hardware features. Instantiated in TVM AutoScheduler and evaluated on an Intel Xeon Gold 6430 CPU and NVIDIA RTX 4090 across 7 models (ResNet-50, BERT-base, MobileNetV2, GPT-2, ResNet18-3D, OPT-1.3B-attn, GPT-Neo-2.7B-attn, 103 subgraphs), under the same 64-trial budget it beats Ansor on 21/22 GPU and 22/22 CPU representative subgraphs (geomean speedup 1.37× GPU / 1.54× CPU). Ours-1024 matches Ansor-10240 within 2.2% geomean latency using 10× fewer trials and outperforms it on 11/22 subgraphs; full-model inference reaches 4.61×/3.67× geomean and up to 58.61×/35.50× peak speedup over PyTorch / PyTorch-opt(cuDNN). The ablation shows the learned transition module is the most important component (removing it slows 19/22 subgraphs, median +56.5% latency).

核心贡献

  • 把"世界模型"视角引入张量程序调度评估：把编译器+硬件视为环境、TensorIR 程序视为状态、调度变换视为动作，候选调度不再是"最终静态代码快照"而是从初始程序出发 rollout 一串动作得到的终态——从而捕捉调度决策间的依赖上下文，并抑制表面代码差异（如 N=1024 vs N=1023 触发的 loop peeling 语法噪声）。
  • 设计三阶段学习框架（状态编码器 E、动作条件多步转移模型 F、排序代价模型 G）：CodeBERT-based 编码器用对比学习（文件内/文件间双负样本）训练；TransH 风格转移模型在隐空间逐步模拟动作；XGBoost LambdaRank 在 workload 内排序而非直接回归运行时。
  • 构建 TVM/TenSet 状态预测数据集：把调参日志与对齐的 TensorIR 状态组织成 (s_0, a_1, s_1, …, a_K, s_K) 动作-状态轨迹，用于多步状态预测训练（CPU/GPU 各约 39 万状态-动作样本，排序模型 CPU 50 万/GPU 14 万样本）。
  • 在 TVM AutoScheduler 中实测：相同 64-trial 预算下相对 Ansor 在 21/22 GPU、22/22 CPU 代表子图上更优；用 1K-trial 接近 Ansor-10K（几何均值差 2.2%）；消融证明学习到的状态转移模块是最关键组件。

Key Contributions

  • Compiler-world-model formulation. Schedule evaluation is recast as learning action-conditioned latent state transitions: the static "final code snapshot" view (which misses how a later scheduling decision depends on earlier context, and is misled by superficial syntax variations like loop peeling for N=1023 vs N=1024) is replaced by a rollout from the initial program state s_0 through the action sequence a_{1:T} to a predicted terminal-state representation ẑ_T = F(E(s_0), a_{1:T}).
  • Three-stage learned framework. (1) A CodeBERT-based state encoder trained with contrastive learning (intra-file + inter-file negatives with separable weights) maps long TensorIR text (sliding-window chunked, masked-mean-pooled) to a dense state vector. (2) A TransH-based action-conditioned transition model (each action = a translation in an action-specific hyperplane, combined via a residual MLP) is trained with multi-step rollout using a state-matching loss + a margin ranking loss against hardest negatives. (3) An XGBoost LambdaRank ranker scores (ẑ_T, structured action features φ(a_{1:T}), optional hardware features h) for within-workload candidate prioritization rather than absolute latency regression.
  • A TVM/TenSet-based state-prediction dataset organized as action-state trajectories: contrastive dataset 10k samples; transition model 391,240/395,761 state-action samples (CPU/GPU); ranking model 500,425 (CPU) / 140,262 (GPU) re-measured samples.
  • Strong empirical gains under matched TVM settings. Under the same 64-trial budget, beats Ansor on 21/22 GPU and 22/22 CPU representative subgraphs; Ours-1024 matches Ansor-10240 within 2.2% geomean with 10× fewer measurements and wins on 11/22 subgraphs.

Methodology

Problem formulation (§3.2). For a fixed workload, given initial state s_0 and candidate action sequence a_{1:T}, estimate the relative quality of the resulting candidate s_T before hardware measurement. Decomposed into three functions: state encoder E(·), transition model F(·) computing ẑ_T = F(E(s_0), a_{1:T}), and ranking model G(·) computing scalar score y = G(ẑ_T, φ(a_{1:T}), h). The score y is for relative within-workload ranking, not direct runtime prediction.

State representation (§3.3). TensorIR text tokenized, encoded by a trainable CodeBERT-based backbone with sliding-window chunking (TensorIR inputs exceed standard Transformer length), masked-mean-pooled per chunk, aggregated to a program-level vector, projected to state embedding z(s)∈R^d. Trained with contrastive learning: positives are two stochastic views of the same TensorIR; negatives split into intra-file (different schedules, same workload source) and inter-file (different files), with separable weights; loss L_repr = CE(u, 0) over cosine-similarity logits.

Action-conditioned multi-step transition (§3.4). Each action serialized to text, encoded by a frozen CodeBERT action encoder + trainable projection. Transition ẑ_t = F(z_{t−1}, a_t) implemented by a TransH-based predictor: the current state is transformed in an action-conditioned geometric space (relation-specific hyperplane translation from knowledge-graph embedding), combined with the action embedding through a residual multilayer, then normalized. Trained with multi-step rollout (predicted state at step t becomes input at step t+1); per-step loss combines a state-matching term L_match = ‖ẑ_t − z_t‖²₂ and a margin ranking term L_rank = max(0, d(ẑ_t, z_t) − d(ẑ_t, z_{t−}) + γ) against the hardest negative, accumulated over rollout steps with mask m_t and weight λ.

Ranking model (§3.5). XGBoost LambdaRank (learning-to-rank), workload-level groups; candidates within a workload sorted by measured runtime fast→slow and mapped to discrete relevance labels (not regressed on absolute runtime). Inputs: predicted terminal-state embedding ẑ_T, structured action feature vector φ(a_{1:T}) (action counts, sequence length, action-specific numeric attributes — distinct from the dense transition action embedding), and optional hardware features h.

Online integration (§3.6). Drop-in replacement of TVM AutoScheduler's default candidate-scoring pathway: for each candidate, extract its action sequence, encode the task's initial TensorIR state, rollout through the transition model, append action + hardware features, output a scalar score used to prioritize candidates for hardware measurement.

Experimental setup (§4.1). Hardware: Intel Xeon Gold 6430 CPU + NVIDIA RTX 4090 GPU. Software: TVM 0.8.dev0, CUDA 12.6. Benchmarks: 7 models (ResNet-50, BERT-base, MobileNetV2, GPT-2, ResNet18-3D, OPT-1.3B Self-Attention, GPT-Neo-2.7B Self-Attention) → 103 extracted subgraphs, grouped into 22 representative subgraph types (Table 2). All learned components trained from TenSet tuning logs; ranking labels re-measured on the authors' own platforms.

Results & Highlights

  • Representative-subgraph speedup vs Ansor at matched 64-trial budget (Table 3, §4.4). Geomean speedup on GPU: 2.14× over Ansor and 2.36× over TenSet at 16 trials, 1.37× over Ansor and 1.79× over TenSet at 64 trials, 1.09× over Ansor and 1.32× over TenSet at 1024 trials. On CPU (Fig. 7): 1.54× geomean over Ansor at 64 trials, 1.01× at 1024 trials. Per-subgraph, wins on 21/22 GPU and 22/22 CPU representative subgraphs; up to 2.40× (GPU) and 3.76× (CPU) peak speedup.
  • Sample efficiency vs large-budget Ansor (§4.5, Fig. 8). Ours-1024 matches or outperforms Ansor-10240 on 11/22 representative subgraphs and reaches 0.98× of Ansor-10240 on the geomean (i.e., within 2.2%) using 10× fewer measurements; it exceeds Ansor-1024 by 1.09×, Ansor-2048 by 1.05×, and matches Ansor-4096 at 1.01×. Even Ours-64 reaches 0.95× of Ansor-1024 and 0.88× of Ansor-4096.
  • Model-level weighted-latency speedup vs Ansor (§4.3, Fig. 5). GPU geomean: 1.49× at 16-trial matched budget, 1.22× at 64-trial; under time budget 1.33× at 200s, 1.22× at 500s, ≥1.05× at 3000s. CPU geomean: 1.62× at 64 trials (per-model up to 1.88×), 1.05× at 1024 trials. Cross-budget: Ours-16 already beats Ansor-64 by 1.25–1.44× on average; Ours-64 nearly matches Ansor-1024.
  • End-to-end model latency (§4.2, Fig. 4). Under the same 64-trial setting, reduces geomean latency from 3.91 ms (Ansor-64) to 3.02 ms (1.29× speedup) across the 7 models; Ours-800 reaches 2.54 ms vs 2.79 ms for Ansor-1024 (1.10×), matching/beating Ansor-1024 on 6/7 models (largest gain OPT 1.55×; GPT-Neo the only loss). vs frameworks: Ours-800 gives 4.61× geomean speedup over PyTorch and 3.67× over PyTorch-opt(cuDNN), with peak speedups up to 58.61×/35.50×.
  • Ablation (§4.6, Fig. 9, 22 subgraphs, 64 trials). Removing the learned transition and feeding the initial TensorIR embedding slows 19/22 subgraphs, median latency +56.5%, mean +136.6%. Replacing the learned transition with the compiler-produced final-state embedding also slows 19/22, median +69.0%, mean +120.1%. Removing structured action features: median +12.0%, 14 subgraphs slower. Removing program state: median +0.0% but mean +14.3%. Replacing ranking with regression (reg:squarederror): median +5.1%, mean +12.9% — confirming the transition module is the most important component and group-wise ranking beats global latency regression.

Critical Assessment

  • The headline "matches Ansor-10K within 2.2%" is a geometric-mean claim that hides substantial per-subgraph variance and is not a strict dominance. Figure 8(b) shows that against Ansor-10240, Ours-1024 wins on only 11/22 representative subgraphs and loses on the other 11; on some subgraphs (e.g., gpt2-t0, gptneo-t0) Table 3 shows the Ansor/TenSet ratios swing wildly across trial budgets (gpt2-t0: 13.24×/12.07× at 16 trials collapsing to ~1.02× at 1024), so the near-parity is a geomean over heterogeneous outcomes, not a consistent win, and the authors themselves state "our method does not strictly dominate Ansor-10240 at every subgraph."
  • The learned evaluator only improves candidate evaluation, not candidate generation or search-space design. The authors explicitly flag this as the first limitation: "Its effectiveness is still bounded by the coverage of the underlying scheduler and the quality of the search space it explores. If strong schedules are not proposed during search, a stronger evaluator alone cannot recover them." So the gains are conditional on Ansor's existing sketch/annotation generation already proposing good candidates.
  • Multi-step latent rollout can accumulate prediction error on long/complex schedules. The authors list this as the second limitation: terminal-state representations come from multi-step rollout in latent space, so "prediction errors may accumulate on long or complex scheduling trajectories, reducing the reliability of the predicted terminal state in harder cases" — but the paper provides no analysis of how prediction error grows with trajectory length T, which is exactly the regime where the method would be most valuable.
  • The model is trained for within-workload ranking, not absolute latency prediction, so it "is less directly applicable to settings requiring calibrated performance estimates or direct comparisons across different workloads" (the authors' third limitation) — this bounds its transferability to cross-workload/cross-hardware cost estimation use cases.
  • The end-to-end comparison includes TensorRT as a baseline but frames it as out-of-scope ("a vendor-optimized inference engine rather than a TVM AutoScheduler variant"), so the paper does not claim to beat TensorRT, and on standard Transformer workloads TensorRT "remains a strong deployment baseline" — readers should not interpret the 4.61×/3.67× PyTorch speedups as beating vendor inference engines.

Strengths

  • Genuinely novel framing: modeling schedule evaluation as action-conditioned latent dynamics (a world model) rather than static code encoding directly addresses the documented failure modes (referential ambiguity of action tokens like Tile(k) where k is created by an earlier Fuse; syntax sensitivity to loop peeling), and the ablation confirms the transition module is the single most important component (removing it: median +56.5% latency).
  • Strong, well-controlled empirical story under matched TVM settings: beats Ansor on 21/22 GPU and 22/22 CPU subgraphs at matched 64-trial budget, and reaches within 2.2% geomean of a 10× larger Ansor budget (Ansor-10240) — a real sample-efficiency gain.
  • Clean three-stage decomposition (encoder / transition / ranker) with each component trained separately and reusable, plus a published-style dataset construction (action-state trajectories from TenSet logs) that others could replicate.
  • Honest about scope: the limitations section explicitly enumerates the three bounds (evaluation-only, rollout error accumulation, ranking-not-regression), and the sample-efficiency claim is hedged ("does not strictly dominate").

Limitations

  • Evaluation-only: cannot recover schedules the underlying search never proposes; gains bounded by Ansor's candidate generation.
  • Multi-step latent rollout may accumulate error on long/complex schedules; no error-vs-trajectory-length analysis provided.
  • Trained for within-workload ranking, not calibrated absolute latency or cross-workload comparison.
  • "Matches Ansor-10K within 2.2%" is a geomean over 11/22 wins; not strict per-subgraph dominance and per-subgraph ratios are highly volatile across budgets.
  • No public code/data URL in the PDF; reproducibility depends on reconstructing the TenSet-based trajectory dataset.
  • Only two hardware targets (Intel Xeon Gold 6430 CPU, RTX 4090 GPU); no mobile/accelerator/AMD results.

Relevance

这篇论文直接映射到用户的「Computation Reliability & Precision Errors」与「ML Reliability & Mathematical Foundations」兴趣区——它的核心动机就是现有代价模型把候选程序当作"静态代码快照"评估，会被表面语法差异（如 N=1024 vs N=1023 触发的 loop peeling 边界处理分支）误导，把两个硬件上等价的程序误判为不同，本质上是一类"数值/结构等价性识别"的可靠性问题；其"动作条件隐状态演化"用连续隐空间抑制表面噪声、捕捉调度决策间的依赖，与计算可靠性的目标一致。它对 TVM AutoScheduler 搜索效率（在更少硬件测量下达到同等质量）的改进也关系到大规模 ML 系统的工程可靠性。

Questions for Further Exploration

  • The authors flag rollout error accumulation as a limitation but provide no error-vs-trajectory-length analysis — how does the predicted terminal-state representation ẑ_T drift from the true z_T as the action-sequence length T grows, and does this explain the per-subgraph volatility in Table 3 (e.g., gpt2-t0 going from 13.24× at 16 trials to ~1.02× at 1024)?
  • The ablation shows removing the learned transition and using the compiler-produced final TensorIR state still slows 19/22 subgraphs (median +69.0%) — is this because encoding the materialized final AST re-introduces the syntax sensitivity (loop-peeling noise) the latent rollout was designed to suppress, and would a denoised/normalized final-state encoding close that gap?
  • The method is evaluation-only and the authors concede "if strong schedules are not proposed during search, a stronger evaluator alone cannot recover them" — would coupling the latent-transition evaluator with an LLM/agent-based candidate generator (the paper cites TLM [42], AutoKernel [17], Bayesian code diffusion [18]) compound the gains, or does the evaluator's within-workload ranking objective mis-rank cross-generator candidates?
  • The ranking model is trained per-workload with LambdaRank and cannot compare across workloads — does this prevent cross-hardware transfer (e.g., training the transition model on RTX 4090 logs and evaluating on an AMD MI300), and would a contrastive cross-hardware state encoder (à la the encoder's intra/inter-file negatives) enable it?
  • Table 3 shows TenSet (a learned cost-model baseline) is beaten by 1.79×/1.32× at 64/1024 GPU trials — is this because TenSet also uses a static-snapshot view, and would adding the action-conditioned transition module on top of TenSet's encoder recover most of the gap, isolating the contribution of dynamics from the contribution of the CodeBERT encoder?
