> **一句话总结：** 本文首次对 MCP（Model Context Protocol）服务端**运行时协调故障**做实证分类：从 473 个活跃 MCP 服务器 GitHub 仓库中手工分析 837 条已确认故障线程，自下而上归纳出 **11 个顶层类、27 个子类、73 个叶节点**；55 名 MCP 开发者调研显示人均遇到过约 20 个子类且无一子类“零观测”，为协议感知测试、故障注入与 AI 软件运维提供结构化基础。

> **TL;DR:** This paper presents the **first empirical taxonomy of MCP server runtime coordination faults**, derived from manual analysis of **837 confirmed fault threads** across **473** actively maintained MCP server GitHub repos (bottom-up open coding → **11 top-level categories, 27 subcategories, 73 leaf types**). A practitioner survey (**n=55**) reports encountering ~**20/27** subcategories on average, with **no subcategory unobserved** and no new fault class in open-ended feedback. Dominant failure surface: **Tool** (194/837), **Data & Schema** (147), and **State & Configuration** (143)—together **57.8%** of cases—plus practitioner emphasis on **Tool Execution** (73% recognition), **Schema Compatibility** (69%), and high-severity **State Transition** / **Security** faults despite lower repo frequency.

## 核心贡献

- **首个 MCP 服务端运行时故障分类体系**：聚焦与实现语言/框架无关的**协议级协调义务**（JSON-RPC、会话生命周期、能力协商、工具/资源/提示、取消与错误报告等），而非客户端、部署基础设施或普通编译错误。
- **大规模仓库挖掘 + 五轮开放编码**：从 MCP Market 与 Awesome MCP Servers 汇总的 1,876 个候选仓库（2025 年 9 月采集）筛至 **473** 个活跃维护仓库；挖掘 **76,460** 条英文 issue–PR 合并线程，经故障相关与 MCP 关键词过滤后，**837** 条确认为 MCP 运行时协调故障（双人独立标注，分歧 **52/837，6.21%**，一致率 **93.79%**）。
- **11 / 27 / 73 层次结构**：覆盖 Base Protocol、Transport、Session Lifecycle、State & Configuration、Tool、Resource、Prompt、Model Provider Integration、Data & Schema、Security、Timeout & Cancellation；叶节点含“配置参数被接受但未强制执行”“成功 JSON-RPC 响应掩盖工具失败”等 MCP 特有模式。
- **开发者外部效度验证**：**55** 名 MCP 服务器维护者/贡献者（目的性抽样，未参与分类构建）确认各子类在实践中出现；平均确认率 **44%**，最高 **Tool Execution 73%**、**Schema Compatibility 69%**、**Tool Result Propagation 67%**；开放反馈未提出新的运行时故障类。
- **可复现制品与工程启示**：Figshare 复制包含数据集、编码记录与脚本；论文将分类用于**诊断分诊清单**、**一致性测试用例设计**、**基于观测模式的故障注入**及 issue 标签化研究议程。

## Key Contributions

- **First empirical taxonomy of MCP server runtime faults** scoped to protocol-level coordination (JSON-RPC, session lifecycle, capability negotiation, tools/resources/prompts, cancellation, structured errors)—explicitly excluding MCP clients, host/agent orchestration, build/deploy infrastructure, and generic non-MCP bugs.
- **Large-scale GitHub mining + five-round open coding:** 1,876 unique repo candidates (Sept 2025) → **473** active MCP servers; **76,460** English merged issue–PR threads → **837** confirmed MCP runtime fault threads after filtering; dual-author labeling with **52/837 (6.21%)** disagreements resolved against the MCP spec (**93.79%** agreement).
- **Hierarchical taxonomy (11 / 27 / 73):** Eleven top-level categories from Base Protocol through Timeout & Cancellation, with leaf types capturing MCP-specific patterns (e.g., accepted-but-unenforced config flags, JSON-RPC success masking tool failure, non-protocol stdout corrupting stdio transport).
- **Independent practitioner validation (n=55):** Purposively sampled MCP server maintainers/contributors not involved in taxonomy construction; mean **~20/27** subcategories encountered; **≥25%** reported every subcategory; top recognition: Tool Execution (73%), Schema Compatibility (69%), Tool Result Propagation (67%); no new runtime fault class in open-ended responses.
- **Replication package and actionable implications:** Public Figshare artifact; taxonomy supports diagnostic triage checklists, conformance test derivation from leaf types, fault-injection benchmarks grounded in observed coordination faults, and protocol-aware issue labeling.

## Methodology

**Scope and fault definition.** An **MCP fault** is a defect causing violation of coordination obligations under the protocol contract or schema constraints—manifesting as omission, incorrect content, or timing violations at the service interface (Avižienis-style dependability framing), even when the server remains “up” and returns structurally valid JSON-RPC.

**Repository discovery (§4.1).** Sources: MCP Market crawler + Awesome MCP Servers regex parse → **2,112** entries → **1,876** unique after dedup → **1,774** accessible via GitHub API. Exclusion criteria (logged, non-mutually exclusive): &lt;5 contributors, no open issues, &lt;100 commits, &lt;10 stars and &lt;10 forks, archived/disabled → **473** repos retained (**1,301** excluded).

**Issue/PR mining (§4.2).** Up to **200** most recent issues/PRs per repo; reference linking (closes/fixes/resolves/#id/URLs); **78,267** merged threads → **76,460** English after language detection + manual non-English removal.

**Fault-related filter (§4.3).** Labels bug/defect/error; commit messages with (fix|solve) + (bug|issue|problem|defect|error) → **4,649** fault-related artifacts; **398** repos with ≥1 such artifact.

**Manual analysis selection (§4.4).** Top **200** repos by fault-related artifact count (information-rich, not prevalence-estimation sampling) → **2,823** issues/PRs + **937** fault-fix commits analyzed.

**MCP-specific filter (§4.5).** Vocabulary mining → **31** MCP keywords after manual inspection of high-frequency stems → **2,016** MCP-relevant threads → manual confirmation → **837** runtime fault threads (excluded: non-fault, build/env-only, insufficient MCP runtime evidence).

**Taxonomy construction (§4.6).** Bottom-up open coding on **837** threads; **five** refinement rounds until saturation (final: 11 categories, 27 subcategories, 73 leaves). Mapping aligned to MCP spec constructs (JSON-RPC, transports stdio/Streamable HTTP, tools/resources/prompts, OAuth 2.1 security, cancellation).

**Practitioner survey.** Google Forms; demographics + per-subcategory encounter (Yes/No), **severity** (Minor/Major/Critical), **diagnostic effort** (Low/Medium/High); open-ended missing categories. **55** respondents (software engineers, architects, AI/ML engineers, DevOps, leads, founders); median experience **5–10** years SE, **1–2** years MCP-related (24/55).

|-------------------|---------|-------|------------------------------|
| **Tool** | 194 | 23.2% | Identification (38), **Execution (108)**, Result Propagation (49) |
| **Data & Schema** | 147 | 17.6% | Schema Compatibility (51), Data Type & Structure (79), Output Representation (17) |
| **State & Configuration** | 143 | 17.1% | Configuration Handling (77), Session State (37), Resource State (29) |
| **Session Lifecycle** | 65 | 7.8% | Initialization (28), State Transition (14), Termination (23) |
| **Resource** | 58 | 6.9% | Identification (57), Synchronization (1) |
| **Security** | 54 | 6.5% | Authentication (32), Token Validation (8), Authorization (13) |
| **Base Protocol** | 45 | 5.4% | Message Structure (21), Message Semantics (23) |
| **Transport Layer** | 45 | 5.4% | Stdio (36), Streamable HTTP (9) |

**Top three categories = 484/837 (57.8%)**—failure surface concentrates on **tool execution and schema enforcement**, not transport/base protocol alone.

### Recurrent fault patterns (examples)

- **Gray/silent failures:** Well-formed JSON-RPC **success** despite incorrect coordination (tool failed but success returned; null LLM content accepted by schema but breaks downstream)—analogous to distributed **gray failures**.
- **Configuration accepted but not enforced:** e.g., `include_resp_body=false` ignored when another plugin shares logging utility with conflicting flag.
- **stdio transport corruption:** Non-protocol output on stdout breaks JSON-RPC parsing (36 threads in stdio subcategory).
- **Session/init race:** Requests before initialization complete; missing `initialized` notification; async/sync control-flow errors in active sessions.
- **Tool-result pipeline:** Results lost before model injection (23 leaf threads); schema/runtime mismatch at tool output.

### Practitioner validation (Table 1, n=55)

| Signal | Finding |
|--------|---------|
| Coverage | All **27** subcategories reported by **≥25%**; mean Yes rate **44%** |
| Most recognized | Tool Execution **73%**, Schema Compatibility **69%**, Tool Result Propagation **67%** |
| Least recognized (still &gt;0) | Resource Identification, Termination, Resource Synchronization **25–29%** |
| Highest severity | **State Transition**: **77%** Major/Critical; **Authorization**: **55%** Major, **33%** High diagnostic effort |
| Missing categories | None new in open-ended responses |

Survey mean: developers encountered **~20 of 27** subcategories—aligns with broad practical relevance.

### Developer & maintainer implications (§6)

**Five practices for MCP server authors:**

1. Validate tool inputs against declared schemas **before** execution.
2. Represent failures with **structured JSON-RPC error objects**, not success payloads hiding errors.
3. **Gate** request handling until initialization completes.
4. Preserve **request–response identifiers** through tool invocation and result propagation.
5. Keep **diagnostics off the JSON-RPC stream** (especially stdio).

**SDK/protocol maintainers:** Enforce id propagation, default input validation, initialization gating, and error-object helpers in SDKs rather than per-repo fixes.

**Benchmarking:** Each subcategory is a seedable fault pattern for detection/diagnosis/recovery evaluation (cf. DL mutation coverage gaps).

## Critical Assessment

- **Novelty:** High for MCP/LLM-tooling reliability. Prior MCP work covers ecosystem growth, security, and architecture [Guo, Hasan, Hou, Li, Zhao, Watanabe] but **not** a bottom-up **runtime fault taxonomy** validated with practitioners. Bridges classical dependability (omission/value/timing) with **schema-governed protocol coordination**.
- **Rigor:** Strong qualitative methodology (837 confirmed threads, dual coding, five saturation rounds, spec-referenced resolution). Repository selection is **information-rich** (top fault-artifact repos), not representative prevalence sampling—authors state this explicitly; frequency counts describe **public artifact bias**, not global MCP fault rates.
- **Reproducibility:** Good—Figshare replication package with datasets, coding records, scripts, figures/tables; Sept 2025 snapshot documented with exclusion logs.
- **Significance:** High for teams building or operating **tool-augmented LLM systems** (Cursor-style hosts, inspectors, enterprise MCP gateways). Provides shared vocabulary for issues, tests, and monitoring; directly relevant to “configuration accepted but not enforced” class of production surprises.
- **Red flags / caveats:**
  - **Server-only scope**—client/agent faults deferred.
  - **GitHub-public bias**—security issues may be underrepresented in issues vs. private audits (authors discuss severity–frequency divergence).
  - **Keyword filter risk**—31 MCP terms may miss atypical phrasing (mitigated by manual confirmation pass).
  - **English-only threads**—ecosystem is international.
  - **Evolving protocol**—taxonomy snapshot may need revision as MCP spec/SDK mature.

## Strengths

- **Timely and practitioner-grounded:** Addresses MCP’s rapid adoption and the gap between “compiles and starts” vs. **correct coordination at runtime**.
- **Clear scope discipline:** Excludes generic bugs, build/env errors, and client-side faults—keeps taxonomy actionable for **server implementers**.
- **Rich hierarchy with concrete repo examples:** Each category tied to real GitHub threads (fastmcp, awslabs/mcp, apache/apisix, openinference, etc.).
- **Dual validation:** Repository-derived structure + **55-developer** survey with severity/effort—not just label frequency.
- **Explicit engineering translation:** Triage checklist, conformance tests, fault injection, SDK enforcement recommendations.
- **Honest threats to validity:** Incomplete cross-references, multi-category threads, purposive sampling limits.

## Limitations

- **No prevalence estimates** for the broader MCP ecosystem—top-200-repo selection optimizes qualitative coverage.
- **No automated classifier or tool release** in-paper—taxonomy is descriptive, not an operational detector (future work implied).
- **MCP clients and agent orchestration** out of scope—end-to-end agent failures may differ.
- **Survey non-probability sample** (n=55)—cannot infer population rates; confirms recognition, not incidence.
- **Static snapshot (Sept 2025)**—protocol and SDK churn may obsolete some leaf types.
- **Output Representation** and other rare subcategories have few threads (e.g., Resource Synchronization = 1)—leaf-level statistics are sparse.

## Relevance

该论文与 **LLM Agent / 工具调用可靠性** 高度相关：MCP 已是连接模型与外部 API/数据的事实标准之一，本文把“线上真实报障”系统化为可测试、可注入、可打标签的故障空间，对构建稳健的工具增强 AI 工作流、设计协议一致性测试、以及 daily-paper 关注的 **agent 系统运维与演化** 具有直接参考价值（`filtered_papers.json` relevance_score: 4）。

## Questions for Further Exploration

- Can a **lightweight classifier** (issue text → taxonomy leaf) be trained on the 837 labeled threads to scale triage on new MCP repos?
- How do **MCP client-side faults** (host routing, multi-server orchestration, prompt injection at the tool boundary) complement this server taxonomy in an end-to-end agent fault model?
- What **conformance test suite** covering all 73 leaves would maximize mutant detection vs. real-fault detection (mutation-testing analogy from DL fault work)?
- Do **SDK-default enforcement** (schema validation, init gating, error helpers) reduce fault rates measurably in a longitudinal study of the same 473 repos?
- How does taxonomy coverage change across **transport bindings** (stdio vs. Streamable HTTP) and **deployment modes** (local vs. OAuth-protected remote)?
- Can **runtime monitors** assert behavioral oracles (session-state consistency, required notifications, tool-result content) beyond crash/HTTP-error detection for gray failures?
- For security: can public-issue frequency be **calibrated** with private audit data to prioritize Authorization/Token Validation testing?
