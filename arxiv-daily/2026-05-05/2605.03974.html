LIPPEN: A Lightweight In-Place Pointer Encryption Architecture for Pointer Integrity

arXiv: 2605.03974

Date: 2026-05-05

https://arxiv.org/abs/2605.03974

一句话总结： LIPPEN 通过对 64 位指针进行全字段加密（使用 PRINCEv2 轻量级分组密码），在零元数据开销的前提下实现了 2^64 的抗暴力破解安全强度，性能与 ARM PAC 相当，并在 RISC-V FPGA 原型上完成了验证。
TL;DR: LIPPEN replaces ARM PAC's truncated authentication codes with full 64-bit pointer encryption using PRINCEv2, achieving 2^64 brute-force resilience at zero metadata overhead. Prototyped on RISC-V Rocket/BOOM FPGA cores, it matches or slightly outperforms PAC on SPEC CPU2017 (2.9% geo-mean overhead on Rocket for return-address protection) with negligible area/power increase (<4%), while maintaining binary compatibility with existing PAC compiler infrastructure.

核心贡献

  • 提出 LIPPEN 架构，将整个 64 位指针视为加密块，使用 PRINCEv2 轻量级分组密码进行全字段加密，提供指针完整性和机密性双重保护，且零元数据开销。
  • 通过系统-密码协同设计（modifier 拆分为 m1 混入明文、m2 混入密钥），避免了使用更昂贵的可调密码（tweakable cipher），同时提供足够的上下文绑定位数。
  • 形式化证明了伪造 LIPPEN 加密指针的难度不低于破解底层分组密码，安全性归约到 PRINCEv2 的密码学强度（最佳已知攻击复杂度 > 2^112）。
  • 在 RISC-V Rocket（顺序）和 BOOM（乱序）核心上实现 FPGA 原型，并对比自研 RISC-V PAC 实现和 Apple M1 上的 ARM PAC，证明全指针加密不比认证更贵。
  • 与 PacTight 等现有 PAC 编译器框架的兼容性验证仅需修改不到 50 行代码。

Key Contributions

  • Proposes full 64-bit pointer encryption using PRINCEv2 in ECB mode as a drop-in replacement for ARM PAC, eliminating PAC's 7--16 bit entropy bottleneck that enables PACMAN-style brute-force attacks (~2.94 minutes on Apple M1).
  • Co-designs the system and cipher to avoid tweakable block ciphers: splits the context modifier into m1 (XORed with plaintext, using unused address bits) and m2 (XORed with key), achieving single-cycle encryption/decryption.
  • Provides a formal security reduction (Theorem 1): forging a LIPPEN-encrypted pointer is no easier than distinguishing PRINCEv2 from a random permutation, with best known attacks requiring > 2^47 chosen plaintexts and > 2^112 time.
  • Implements and evaluates on 64-bit RISC-V FPGA (VCU118) with both in-order Rocket and out-of-order BOOM cores, comparing against an in-house QARMA-based PAC baseline and Apple M1's commercial PAC.
  • Demonstrates seamless integration with existing PAC compiler infrastructure (PacTight) requiring < 50 LOC changes.

Methodology

LIPPEN is a hardware-software co-design built around three pillars:

  1. Cipher selection: After surveying 10+ lightweight block ciphers, the authors select PRINCEv2 for its single-cycle unrolled datapath (404 ps latency, 14,181 GE area in NanGate 15nm), outperforming QARMA (553 ps, 22,131 GE) on both metrics. The +mod variant (adding XOR logic for modifier injection) achieves 42 MHz Fmax on the FPGA vs. QARMA's 40 MHz.
  1. Modifier design: The context modifier m = m1 || m2 is split to co-design with the cipher: cipher = Enc_{K⊕m2}(plain ⊕ m1) and plain = Dec_{K⊕m2}(cipher) ⊕ m1. Component m1 occupies only non-address-affecting bits (unused high-order bits + 2 LSB alignment bits, yielding |m1|_max = 64 − A + 2 = 18 bits for 48-bit VA), preventing target bit-flip attacks. Component m2 XORs into the key, with the OS ensuring the unaffected 128 − |m2| key bits are unique per domain to prevent cross-domain collisions. Empirical analysis of SPEC CPU2017's largest benchmark (xalancbmk) shows 16 modifier bits suffice for unique pointer contexts.
  1. ISA extensions: Four new instructions (SET_KEY, SET_M_SIZE, PTR_SEAL, PTR_UNSEAL) mirror PAC's PAC/AUT semantics. Compiler support implemented in LLVM v18.1 targeting RISC-V, instrumenting call/return sites analogously to Arm's RETAA/RETAB.
  1. Implementation: Prototyped on Xilinx VCU118 (Virtex UltraScale+) using Chipyard v1.8, connected via Rocket Custom Coprocessor (RoCC) interface. Configured with M1=16 bits, M2=0.

Results & Highlights

  • SPEC CPU2017 (return-address protection): Geometric mean overhead of 2.9% for LIPPEN vs. 3.6% for PAC on Rocket; near-zero on BOOM. Control-flow-intensive benchmarks (perlbench_r, leela_r, deepsjeng_r) show 6--12% overhead; compute-bound workloads (namd_r, lbm_r) remain at baseline.
  • nbench: 0.2% overhead for both LIPPEN and PAC on Rocket (O2); BOOM effectively 0%.
  • Microbenchmarks: LIPPEN on BOOM is comparable to Apple M1 across all return-address and data-pointer protection scenarios. Data-pointer overhead is dominated by load-use chain serialization, not cipher latency.
  • Hardware cost: LIPPEN adds 1,034 LUTs / 131 FFs to Rocket (vs. PAC's 2,071/132). On BOOM: 862/193 vs. 2,054/131. Power increase < 4% on both cores (e.g., Rocket: 3.935W baseline → 4.035W with LIPPEN).
  • PacTight compatibility: Adapting PacTight from ARM PAC to LIPPEN required < 50 LOC changes to the IR pass, with negligible performance overhead under O0.
  • Security: 64-bit brute-force space (vs. PAC's 7--16 bits), eliminates PACMAN oracle attacks, formally reduces to PRINCEv2's IND-CCA security.

Critical Assessment

Novelty: High. This is the first work to rigorously evaluate full 64-bit pointer encryption for integrity (as opposed to truncated MACs or partial encryption like C3). The co-design insight — that preserving raw pointer bits is unnecessary since PAC's speculative-use benefit is marginal in practice — is a genuinely useful observation.

Rigor: Strong overall. The formal security reduction (Theorem 1) is clean and connects LIPPEN's forgery difficulty to PRINCEv2's IND-CCA security. The cipher comparison is systematic with real FPGA implementations (not just literature numbers). However, some concerns:

  • Only return-address protection is evaluated end-to-end on SPEC. Full data-pointer protection across SPEC is not reported; the nbench data-pointer results show ~20% overhead (consistent with PARTS' earlier findings), but this is only for a pointer-chasing microbenchmark. The paper acknowledges this gap.
  • SPEC benchmarks with C++ exception handling are excluded due to stack unwinding issues with sealed pointers. This is a practical limitation that weakens the deployment story.
  • O2 PacTight overhead spikes to 7.5% on Rocket because LIPPEN instructions are treated as opaque barriers. This compiler limitation is acknowledged but not resolved.
  • The FPGA runs at 100 MHz, far below commercial processors. While the relative comparison to PAC is valid, absolute overhead numbers may differ at GHz frequencies where pipeline depth and cache behavior change.
  • No evaluation on real silicon — all RISC-V results are FPGA-based. The Apple M1 comparison only covers microbenchmarks where PAC instructions can be isolated, not full SPEC runs.

Reproducibility: Excellent. Open-source code, Zenodo artifact, detailed FPGA setup instructions, and the artifact has been evaluated for reproducibility.

Significance: High for the architecture security community. If the results hold at commercial frequencies, LIPPEN presents a compelling case that ARM should have encrypted pointers from the start rather than using truncated MACs.

Red flags: None serious. The paper is transparent about limitations (C++ exception handling, O2 compiler barriers, data-pointer overhead).

Strengths

  • Clean security story: full encryption eliminates the entropy-vs-metadata trade-off that has plagued pointer integrity for a decade.
  • Practical compatibility: reuses PAC compiler infrastructure with minimal changes; PAC-style key management applies directly.
  • Thorough cipher analysis: real FPGA implementations of multiple candidates, not just paper comparisons.
  • Open-source with artifact evaluation badge, enhancing trust in reported results.
  • The modifier analysis (16 bits suffices for xalancbmk) is a useful data point for the community.

Limitations

  • Full data-pointer protection across real workloads (SPEC) is not benchmarked; only return-address protection and microbenchmarks for data pointers are evaluated end-to-end.
  • C++ exception handling (stack unwinding) is incompatible without system library modifications.
  • O2 compiler optimization barriers cause non-trivial overhead (7.5%) when composed with PacTight on Rocket.
  • FPGA-only evaluation at 100 MHz; real silicon behavior at GHz frequencies with deeper pipelines remains unvalidated.
  • PRINCEv2, while well-analyzed, is a relatively new cipher (2024) and has not received the decades of scrutiny that AES has.

Relevance

本文提出了一种硬件-软件协同设计的指针加密架构，使用轻量级分组密码保护指针完整性。虽然主要面向硬件安全和处理器架构领域，但其对 ARM PAC 安全缺陷的分析（特别是 PACMAN 暴力破解攻击）和密码学安全归约方法对系统安全研究具有参考价值。

Questions for Further Exploration

  1. How does LIPPEN's overhead scale when protecting all data pointers (not just return addresses) on full SPEC CPU2017 workloads, as systems like RSTI and AOS require?
  2. Can the compiler barrier issue at O2 be resolved through targeted LLVM middle-end optimizations that understand LIPPEN instruction semantics?
  3. What is the interaction between LIPPEN and memory tagging extensions (MTE) when both are active — does the reduced m1 bit-budget meaningfully weaken the security guarantee?
  4. How would LIPPEN perform on a superscalar out-of-order core at 2+ GHz with deeper pipelines, where the single-cycle cipher latency may become a larger fraction of the clock period?
  5. Could LIPPEN's encryption approach be extended to protect against side-channel attacks (e.g., cache-timing on the cipher) in addition to the current pointer-forgery threat model?
