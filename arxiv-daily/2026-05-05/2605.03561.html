Enhancing Performance Insight at Scale: A Heterogeneous Framework for Exascale Diagnostics

arXiv: 2605.03561

Date: 2026-05-05

https://arxiv.org/abs/2605.03561

一句话总结： 本文为hpcanalysis框架设计了C++/GPU异构加速基础设施，在Aurora十万MPI规模下实现9.69秒数据导入和最高314倍GPU加速分析，并引入三维性能模型从执行轨迹中重建迭代行为，在Frontier上为GAMESS量化出32.28%的潜在加速空间。
TL;DR: This paper accelerates the hpcanalysis HPC performance analysis framework with a C++ Read API (mmap + OpenMP) and a GPU-accelerated HpcDataFrame abstraction supporting NVIDIA (cuDF), AMD (hipDF), and Intel (SYCL) backends. The C++ API ingests 100K MPI rank profiles from Aurora in 9.69s; GPU backends achieve up to 314x speedup over Pandas for grouping/sorting 100K traces. A novel tri-dimensional (Node × Trace × Iteration) performance model "re-materializes" iterative behavior from traces, predicting a 32.28% speedup for GAMESS on Frontier. A topology-aware workflow maps performance outliers to Slingshot interconnect coordinates, localizing network congestion across 22 racks on Aurora.

核心贡献

  • C++加速Read API： 用mmap替代Python文件I/O、OpenMP多线程替代joblib多进程，实现10万MPI profile在9.69秒内导入Aurora。
  • GPU加速HpcDataFrame抽象层： 统一接口支持cuDF（NVIDIA）、hipDF（AMD）、自定义SYCL wrapper（Intel），在10万trace规模上实现最高314倍加速。
  • 三维性能模型（Node × Trace × Iteration）： 从执行轨迹中"重新物化"迭代行为，能在聚合profile丢失的细粒度负载不均衡信息中量化潜在加速收益。
  • 拓扑感知诊断工作流： 将MPI rank级别的性能异常通过DBSCAN聚类映射到Slingshot互连物理坐标，定位了Aurora上22个机柜的网络拥塞。

Key Contributions

  • High-throughput C++ Read API: Replaces Python file I/O with mmap and joblib multiprocessing with OpenMP threads. Ingests 100,000 MPI rank profiles on Aurora in 9.69 seconds, compared to minutes with the original Python API.
  • GPU-accelerated HpcDataFrame abstraction: A unified DataFrame interface that dispatches to cuDF (NVIDIA), hipDF (AMD), or a custom SYCL wrapper (Intel) based on detected hardware. Achieves up to 314x speedup for grouping and 220x for sorting 100K traces over CPU Pandas.
  • Tri-dimensional performance model: Integrates hpcanalysis with Thicket to build a (Node × Trace × Iteration) model that reconstructs per-iteration profiles from execution traces, enabling fine-grained load imbalance analysis across iterations and ranks.
  • Topology-aware interconnect mapping: Maps logical performance outlier nodes to physical Slingshot interconnect coordinates (rack/chassis/slot), localizing network congestion across 22 racks on a 1,000-node Aurora allocation.
  • Iteration-aware diagnostics for GAMESS: Demonstrated the tri-dimensional model on a GAMESS Hartree-Fock workload on Frontier, identifying severe GPU kernel imbalance (balance ratio as low as 0.50) and predicting 32.28% potential speedup.

Methodology

The paper enhances the three-layer hpcanalysis framework (Read API → Query API → Data Analysis):

  1. C++ Read API (Section III): Reimplements the Python Read API as a C++ shared library using mmap for zero-copy file access and OpenMP for native multi-threaded parsing of parallel profiles. Uses pybind11 to bridge C++ objects back to Python. This eliminates Python's GIL bottleneck and double-copy I/O overhead.
  1. GPU-accelerated Query API (Section IV): Introduces HpcDataFrame, a hardware-agnostic abstraction that defaults to Pandas (CPU) but can dispatch to vendor-specific GPU backends: cuDF for NVIDIA, hipDF for AMD, and a custom SYCL-based wrapper using dpnp for Intel GPUs. The SYCL wrapper stores each column as a discrete GPU array and implements core operations (comparison, arithmetic, sorting, filtering, reductions, cumulative sums) as SYCL kernels.
  1. Tri-dimensional model via Thicket integration (Section V): Scans execution traces to detect iterative boundaries, reconstructs separate Hatchet profiles per iteration, and merges them into a Thicket object with a 3D DataFrame index (calling context node × trace × iteration). This enables queries like groupby("iteration")["time"].agg(["mean","min","max","std"]) to quantify per-iteration load imbalance.
  1. Topology-aware diagnostic workflow (Section VI-B): Clusters MPI ranks by communication cost using DBSCAN, maps outlier ranks to compute nodes via metadata DataFrames, then translates POSIX node IDs to Slingshot physical addresses (rack/chassis/slot format) to localize congestion.

Benchmarks were collected on three DOE supercomputers: Polaris (NVIDIA A100), Frontier (AMD MI250X), and Aurora (Intel Max 1550). Each operation was run 10 times and averaged.

Results & Highlights

  • C++ Read API on Aurora: Ingested pruned profiles for 100,000 MPI ranks in 9.69s. The Python API with 128 joblib tasks took 160.7s for just 10,000 ranks — the C++ API processed 10K ranks in 1.13s.
  • GPU DataFrame speedups (100K traces):
  • Polaris (cuDF): 314.5x grouping, 209.4x sorting, 67.1x merging.
  • Frontier (hipDF): 154.2x grouping, 220.1x sorting, 83.7x merging.
  • Aurora (SYCL): 81.7x cumulative sum, 68.5x reduction sum on array operations.
  • Small-data GPU overhead: For 10–100 traces, CPU often wins (0.1x–0.8x speedups) due to GPU memory transfer and kernel launch overhead.
  • GAMESS case study (Frontier): GPU kernel gpu_rhf_j04_psps had a balance ratio of 0.50 and across-rank CV of 46.31%. The tri-dimensional model estimated 28.08s savings across 11 iterations, yielding a 32.28% potential speedup (out of 87s total runtime).
  • AMG network congestion (Aurora): MPI_Allreduce (Instance 2) showed a global variance ratio of 0.39. DBSCAN separated 202 outlier nodes from 798 normal ones. The outlier group averaged 5.19s total runtime vs. 3.12s for the normal group. Congestion spanned 22 racks on the Slingshot interconnect.

  • The work was evaluated on two DOE leadership-class supercomputers (Frontier at ORNL, Aurora at ANL), indicating significant institutional support and access to cutting-edge systems.
  • The paper builds on the well-established HPCToolkit ecosystem and integrates with Thicket (LLNL), demonstrating cross-institutional collaboration.
  • Single-author paper, which is notable for a systems paper of this scope — the breadth of contributions (C++ rewrite, GPU abstraction across 3 vendors, tri-dimensional model, topology mapping) is ambitious for one author.

Critical Assessment

  • Novelty: Moderate-to-high. The individual techniques (mmap, OpenMP, GPU DataFrames, DBSCAN clustering) are well-known. The novelty lies in their integration into a coherent, production-grade exascale performance analysis pipeline and the tri-dimensional model concept. The topology-aware mapping workflow is a genuinely useful systems contribution.
  • Rigor: Good. Benchmarks are run 10x and averaged. Three distinct GPU vendors are tested. The case studies use real applications (GAMESS, AMG) on production supercomputers, not synthetic proxies. The cluster stability analysis (K-Means with 2–5 groups showing minimal intersections) strengthens the congestion findings.
  • Reproducibility: Limited. The work depends on access to Aurora and Frontier, which are restricted systems. The hpcanalysis framework itself is part of HPCToolkit (open source), but the C++ API, GPU backends, and tri-dimensional model integration may not yet be publicly available. No code repository is referenced.
  • Significance: High for the HPC community. Being able to ingest and analyze 100K-rank datasets interactively is a genuine capability gap that this paper addresses convincingly.
  • Red flags:
  • The 314x speedup headline number is for grouping on cuDF — this is largely cuDF's achievement rather than a novel algorithm. The paper should more clearly distinguish what is engineering integration vs. algorithmic contribution.
  • The SYCL/Intel GPU wrapper is limited to single-column operations; multi-column operations fall back to CPU Pandas. This is a significant limitation on Aurora (one of the two primary evaluation platforms).
  • The GAMESS case study uses only a single-node, 16-rank execution — a relatively small scale for a paper focused on exascale diagnostics. The predicted 32.28% speedup is an idealized estimate assuming perfect load balancing, not a demonstrated improvement.
  • The congestion analysis on Aurora identifies the problem but does not propose or evaluate a solution.
  • No comparison against other accelerated analysis frameworks (e.g., Intel VTune, TAU with GPU acceleration).

Strengths

  • Evaluated on three real DOE supercomputers with production workloads — exceptional experimental setup.
  • The HpcDataFrame abstraction elegantly handles vendor heterogeneity (NVIDIA/AMD/Intel) with a single API.
  • The tri-dimensional model is a conceptually clean approach to recovering time-resolved information from collapsed profiles.
  • The topology-aware congestion localization workflow addresses a real operational need for system administrators.
  • Comprehensive benchmark tables with per-operation, per-platform, per-scale breakdowns.

Limitations

  • Intel GPU support is incomplete (single-column SYCL operations only; multi-column falls back to CPU).
  • The GAMESS case study is single-node; the tri-dimensional model is not demonstrated at exascale scale.
  • The "re-materialization" of iterations requires applications with regular, repetitive behavior — the paper explicitly notes it is not optimized for irregular execution phases.
  • GPU acceleration shows negative returns for small datasets (<100 traces), requiring users to be aware of the crossover point.
  • No end-to-end wall-clock comparison of a full analysis workflow (ingestion + query + visualization) on the accelerated vs. original stack.
  • Single author raises questions about long-term maintenance of such a broad engineering effort.

Relevance

本文面向HPC性能分析与系统工具链，与AI/ML研究方向关联较弱。但其中GPU异构加速数据处理、大规模并行trace分析的思路，对分布式训练系统的性能诊断有一定借鉴意义。

Questions for Further Exploration

  1. How does the tri-dimensional model scale to truly exascale executions (e.g., 100K+ ranks with thousands of iterations)? Does the Thicket DataFrame become a memory bottleneck?
  2. Can the congestion localization workflow be generalized beyond Slingshot to other interconnects (InfiniBand, custom networks)?
  3. What is the overhead of the hpcanalysis measurement collection itself on the target application? The paper focuses on post-hoc analysis but does not quantify measurement perturbation.
  4. How would the framework handle streaming/online analysis where performance data is processed concurrently with application execution?
  5. Could the tri-dimensional model be extended to capture hierarchical parallelism (MPI ranks × GPU streams × kernel iterations × thread blocks)?
