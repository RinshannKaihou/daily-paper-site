> **一句话总结：** 本文隔离出 MCP 协议中"人类审批视图渲染"与"模型上下文字节投递"两条独立代码路径之间的保真度缺口，从 Unicode 码点分配出发模型无关地预测并实测验证：在 8 种工具投毒技术中仅 TAG-block 编码 (U+E0000–U+E007F) 能同时对人类审阅和字符串匹配清洗器隐身，8/8 全部到达模型上下文，0/8 触发协议级重新授权；且跨 3 个独立 Python MCP 库的 32 个结果单元完全一致，证明这是协议级属性而非单一代码库缺陷。

> **TL;DR:** This paper isolates the structural gap in MCP between what the human approval dialog renders and the verbatim bytes injected into the model's context, and predicts from Unicode codepoint assignment alone that the TAG-block encoding (U+E0000–U+E007F) is invisible to every mainstream renderer while surviving byte-for-byte into the model's tokenizer. Implementing 8 tool-poisoning/confused-deputy techniques across 5 metadata surfaces against a genuine JSON-RPC/stdio client, the authors find 8/8 reach the model's context, 4/8 evade a representative string-matching sanitizer, only 1/8 (the TAG-block technique T7) evades human review (the only one defeating both defense layers), and 0/8 force re-approval even under a TOCTOU rug-pull — all 32 cross-library cells agree exactly across 3 independently developed Python MCP server libraries (mcp.server.lowlevel.Server, mcp.server.fastmcp.FastMCP, third-party fastmcp 3.4.2), and the baseline sanitizer flags 0/25 benign tool descriptions.

## 核心贡献

- **隔离出单一结构性机制（隐藏编码 concealment encoding）**：从 MCP 协议中精确分离出"人类审批渲染管线"与"模型上下文投递管线"两条独立消费同一段不可信字节的代码路径，并指出二者没有任何协议要求保持字节级一致。
- **模型无关、协议无关的事前预测**：仅依据 Unicode 码点分配（TAG 块 U+E0000–U+E007F 无字体字形，但 UTF-8 解码器与分词器照常接受）即预测出哪些隐藏编码能逃过人类审阅，再由实测验证，将机理分析与经验结果解耦。
- **真协议概念验证 + 8 技术 × 5 表面的完整目录**：T1–T8 覆盖描述字段、inputSchema、工具名、错误通道、TOCTOU 改写五类表面，包含可直接复现的 JSON-RPC/stdio 客户端、确定性测量线束（每技术记录 4 个协议级观测），以及 verbatim 的恶意 tools/list 响应与 sanitizer 源码。
- **跨库验证 32/32 单元完全一致**：在官方 SDK 低层 API、官方 SDK 高层 FastMCP、独立维护的第三方 fastmcp 3.4.2 上分别重实现全部 8 项技术，四比特结果模式（Ctx/Base/Rev/Reap）逐格吻合，把"协议级属性"与"单代码库缺陷"区分开。
- **结构化修复方案**：提出四项互补的协议层修复——字节忠实审批、定义突变强制重新授权、按来源划分的工具命名空间、schema 默认值不等于授权——分别对应 T7、T3、T6、T4/T8 的根因。
- **将 5 个已披露 CVE 关联到同一机理**：CVE-2026-13341、CVE-2025-52573、CVE-2025-6514、CVE-2025-58357、CVE-2025-63603 均可在本文的表面/机理框架下被解释为同一信任边界的不同实例。

## Key Contributions

- **Isolates a single structural mechanism (concealment encoding)**: precisely separates MCP's two independent code paths — the human-facing approval-render pipeline and the model-context delivery pipeline — that both consume the same untrusted bytes, and notes that nothing in the protocol forces them to agree byte for byte.
- **Model-free, protocol-free a-priori prediction**: predicts which concealment encodings evade human review purely from Unicode codepoint assignment (the TAG block U+E0000–U+E007F has no font glyph in any mainstream renderer yet is accepted by UTF-8 decoders and tokenizers), then confirms the prediction empirically, decoupling mechanism analysis from measurement.
- **Real-protocol proof-of-concept plus an 8-technique × 5-surface catalogue**: T1–T8 cover the description field, inputSchema, tool name, error channel, and TOCTOU re-list mutation; ships a reproducible JSON-RPC/stdio client, a deterministic harness recording four protocol-level observations per technique, and verbatim malicious tools/list responses and sanitizer source.
- **Cross-library validation with 32/32 cell agreement**: re-implements all eight techniques against the official SDK's low-level Server, the official SDK's high-level FastMCP, and the independently maintained third-party fastmcp 3.4.2; the four-bit outcome pattern (Ctx/Base/Rev/Reap) agrees in every cell, distinguishing a protocol-level property from a single-codebase artifact.
- **Structural fix proposal**: four complementary protocol-level mitigations — byte-faithful consent, forced re-consent on definition mutation, provenance-scoped tool namespaces, and "schema defaults are not consent" — each targeting the root cause of T7, T3, T6, and T4/T8 respectively.
- **Connects five disclosed CVEs to one mechanism**: CVE-2026-13341, CVE-2025-52573, CVE-2025-6514, CVE-2025-58357, and CVE-2025-63603 are each reinterpreted within the paper's surface/mechanism framework as different instances of the same trust boundary.

## Methodology

The paper combines a model-free mechanism analysis (§4) with a deterministic protocol-level harness (§3.3), keeping the two deliberately separate.

**Mechanism analysis (§4).** Before any experiment, the author traces a single payload character through the rendering pipeline. In T1 (plain ASCII) the byte 0x65 ('e') maps to U+0065, one UTF-8 byte, with an assigned glyph in every font. In T7 (TAG-block) `tag_encode` maps the same source character to `0xE0000 + (0x65 & 0x7F)` = U+E0065, whose four-byte UTF-8 encoding `F3 A0 81 A5` a decoder accepts identically, yet which has no assigned glyph in any mainstream terminal, chat client, or IDE panel. The asymmetry — "renderable-or-not for a human, decodable regardless for a model" — is the entire mechanism behind T7 and is independent of which MCP client, sanitizer, or model is downstream. The same analysis predicts T1 and T2 (Basic Latin) should NOT evade human review, because their payloads use codepoints every renderer displays.

**Technique set (§3.2, Table 3).** Eight techniques grouped by the MCP surface they poison:
- Description field: T1 direct `<SYSTEM>` block injection (plain ASCII), T2 cross-tool shadowing (rewriting how a different trusted `send_email` is used, plain ASCII), T7 TAG-block concealment.
- Input schema: T4 confused-deputy credential relay (coercive instruction inside a parameter description), T8 dangerous-default coercion (`--allow-network --no-sandbox --disable-seatbelt` pre-selected as a `default`/`enum`).
- Tool name: T6 namespace collision (registering `read_file`/`list_dir` to shadow host built-ins).
- Error channel: T5 injection into the text of an `isError` `tools/call` result.
- Post-approval mutation: T3 rug-pull — benign `list_dir` at approval time, poisoned description on the second `tools/list`.

**Threat model (§3.1).** A malicious or compromised MCP server the user has connected to their agent; the attacker controls all tool metadata and tool-call results but has no other access (no code execution on host, no network position, no client modification). The developer's only defense-relevant choices are which sanitizer (if any) filters tool metadata and how faithfully the client renders it to the user. Proof-of-concept performs no real egress; confused-deputy tools relay data to a local file and attacker endpoints use reserved `example.com`.

**Measurement harness (§3.3).** A real MCP client launches the malicious server as a subprocess and speaks genuine JSON-RPC/stdio (`initialize` → `tools/list` → `tools/call`). For each technique it records four deterministic protocol-level observations on the exact bytes received: `reaches_model_context`, `evades_baseline` (a conjunctive keyword sanitizer with NFKC normalization, imperative-indicator ∧ sensitive-action-indicator), `evades_human_review`, and `reapproval_forced`. Because each observation is a deterministic protocol-level fact, no confidence interval or significance test applies.

**Baseline sanitizer sanity check (§3.2).** A corpus of 25 representative non-malicious tool descriptions (file I/O, version control, issue tracking, messaging, calendars, search, build tooling) is run through the same `baseline_sanitizer`; 0/25 are flagged, ruling out the degenerate case where the sanitizer rejects nearly everything.

**Reproducibility (§3.4).** A driver script reproduces the results dataset deterministically; an independent `verify_numbers.py` re-derives every numeric macro from the dataset and asserts it matches the text.

## Results & Highlights

- **All 8/8 techniques reach the model's context**, uniformly across all 5 metadata surfaces (description, inputSchema, tool name, error channel, post-approval mutation) — Table 5.
- **4/8 techniques evade the baseline string-matching sanitizer**: the schema-coercion pair (T4, T8), the namespace collision (T6), and the TAG-block concealment (T7); the sanitizer only catches the overt plain-ASCII injection cases (T1, T2, T5, T3).
- **Only 1/8 (T7) evades the human-review render**, making it the single technique in the set that defeats both the automated and the human defense layer at once, exactly as the mechanism analysis predicted.
- **0/8 techniques force protocol-enforced re-approval**: the TOCTOU rug-pull (T3) mutates `list_dir`'s description from benign to poisoned across two `tools/list` calls (hash `e95a8489` → `7d10aab8`) after the user has already approved, with no re-consent signal, even though the `notifications/tools/list_changed` mechanism exists but was scoped only to cache invalidation.
- **Cross-library agreement: 32/32 cells agree exactly** across the three independently developed Python MCP server libraries (mcp.server.lowlevel.Server, mcp.server.fastmcp.FastMCP, third-party fastmcp 3.4.2); the four-bit outcome pattern is identical for every technique in every library (Table 6), e.g. T7 = 1110 in all three.
- **Baseline sanitizer specificity: 0/25 benign tool descriptions flagged**, ruling out the trivial explanation that the 4/8 evasion count is an artifact of a detector that rejects almost everything.
- **Byte-level trace (Table 4)**: a single source character 'e' (0x65) becomes U+0065 with UTF-8 byte `65` (one byte, glyph 'e') under T1, versus U+E0065 with UTF-8 bytes `F3 A0 81 A5` (four bytes, no assigned glyph) under T7, while both reach the tokenizer verbatim.
- **Five independently confirmed CVEs** mapped onto the studied surfaces: CVE-2026-13341 (Kong Konnect confused-deputy, matching the T4/T8 schema-coercion mechanism), CVE-2025-52573 (ios-simulator-mcp shell injection via tool args), CVE-2025-6514 (mcp-remote OAuth OS command injection matching the malicious-server threat model), CVE-2025-58357 (5ire desktop client content-injection script gadgets via compromised servers — same client-render-safety gap as §4), CVE-2025-63603 (data-science MCP `exec()` sandbox escape).

## Authors & Credibility

Single-author paper by Mohammadreza Rashidi (Department of Computer Science, AI and Media Analysis Lab, Berlin, Germany). The work is unusually rigorous for a security measurement paper: it ships verbatim JSON-RPC payloads (Listing 1), the full baseline sanitizer source (Listing 4), byte-level codepoint arithmetic (Table 4), a cross-library agreement table (Table 6), a complete technique cross-reference (Table 7), an end-to-end worked scenario for T4 (Appendix B), and a position table (Table 2) honestly comparing itself against nine prior works on four explicit dimensions, marking competitors' claims only where their own abstracts support them. The author ties findings to five real CVEs confirmed against the NVD at time of writing, and the `verify_numbers.py` invariant that fails the build if any cross-library cell disagrees is a strong reproducibility signal. The framing is careful to claim delivery and defense-evasion rates, not attack success rates against specific models — a narrower and defensible scope.

## Critical Assessment

**Novelty.** The paper does not claim a new attack family; it explicitly positions itself as isolating the single property (approval-view fidelity) that decides whether an *existing* family of tool-metadata attacks is visible to the human who is supposed to catch it, and as the first to predict that outcome from a model-free Unicode codepoint analysis before measuring. Table 2's honest four-dimension comparison against Wang et al. (MCPTox benchmark), Li et al. (MCP-ITP), Huang et al., Ye et al. (TRUSTDESC), Bhatt et al. (ETDI), He et al., Song et al., Jamshidi et al., and Zuvic shows none of the nine prior works combines all four properties (real protocol attack + defense-layer measurement + mechanism analysis + cross-library replication). This is a genuine and well-scoped contribution.

**Rigor.** Strong. Deterministic protocol-level harness, verbatim payload listings, byte-level codepoint arithmetic, an independent verification script that fails the build on disagreement, and the 0/25 benign-corpus check that preemptively answers the obvious "your sanitizer just rejects everything" objection. The four-bit observation schema (Ctx/Base/Rev/Reap) is crisp and reproducible.

**Reproducibility.** Very high relative to the field. The harness, payload catalogue, sanitizer source, and cross-library dispatch fixes (Listing 5) are reproduced in full; the author names the exact library versions (third-party fastmcp 3.4.2) and the specific dispatch quirk (`to_mcp_tool()` conversion needed for the Pydantic validation error) that had to be solved independently. A reader could re-derive the dataset.

**Limitations the author concedes (and one they underplay).**
- The baseline sanitizer is a deliberately simple conjunctive keyword matcher, and both defenses are described as "lower bound" approximations of today's shipping guardrails; the author is honest that a production client with a semantic classifier or a renderer that normalizes unassigned-glyph codepoints would close some gaps. The 4/8 sanitizer-evasion number is therefore a *lower* bound, but a more sophisticated sanitizer is never tested, so the headline "most evade a representative sanitizer" rests entirely on how representative the keyword baseline is — and the author does not benchmark any second, stronger sanitizer to bracket this.
- **Missing ablation on the renderer side**: the paper predicts (and notes it is "not aware of a mainstream MCP client that currently does this") that a client performing unassigned-glyph normalization would not miss T7, but never actually tests such a normalization against any real client. The single most consequential claim — that *every* mainstream renderer is vulnerable — is asserted from font coverage rather than measured across a panel of real clients (Cursor, Claude Desktop, Cline, Continue, etc.).
- **Construct-validity gap the author flags but does not close**: "reaches the model's context" is operationalized as "present in the JSON-RPC payload the agent serializes," not "the model acted on it." This is a defensible narrowing but means the numbers cannot be read as attack success rates, and no end-to-end agent-behavior measurement is offered — so the practical severity for any specific coding agent is unmeasured.
- The benign corpus of 25 descriptions is hand-written rather than sampled from deployed traffic, so false-positive prevalence in the wild is unmeasured, and the cross-library agreement shares the underlying `mcp` wire-protocol types for two of three libraries, making the result strongest against tool-management-layer artifacts and weaker against a hypothetical shared wire-layer bug (the author concedes this internal-validity threat in §9.1).
- Single-author, and the "three independent engineering teams" framing for the cross-library result slightly overstates independence: the author wrote all three re-implementations against three libraries, two of which are from the same SDK.

**Significance.** High for the MCP and coding-agent security community. The TAG-block mechanism is concrete, cheap, and immediately actionable, and the structural fix proposal (byte-faithful consent, hash-pinned re-consent, provenance-scoped namespaces, no-inherited-dangerous-defaults) gives client and spec authors a concrete checklist. The work is best read as a measurement-and-mechanism paper, not as an attack-succes-rate study.

**Red flags.** None rising to the level of fabrication or overclaiming — the claims are carefully scoped — but the absence of any test against a real production MCP client's actual renderer (relying instead on font-coverage argumentation) is the weakest link in an otherwise rigorous paper, and the headline framing could be misread by casual readers as proving deployed clients are all vulnerable when in fact no deployed client was measured for the render claim.

## Strengths

- A genuinely model-free, protocol-free mechanism analysis that predicts the empirical outcome before any client is run, then is confirmed by the data.
- Real JSON-RPC/stdio protocol implementation rather than a simulated one; verbatim payload and sanitizer source listings make the work trivially inspectable.
- Cross-library replication across three independent Python MCP server libraries (32/32 cell agreement) is the strongest possible defense against the "single-codebase artifact" critique.
- Honest scoping: deliberately separates delivery/defense-evasion rates from attack success rates, and explicitly positions the result as a lower bound against minimally-defended clients.
- Concrete, mechanism-targeted structural fixes (§8.1–8.5) mapped one-to-one onto the techniques they defeat, plus ties to five real CVEs.
- Self-critical Threats to Validity (§9.1) addressing construct, internal, and external validity.

## Limitations

- The two baseline defenses are deliberately simple; no stronger (semantic, LLM-based) sanitizer is benchmarked, so the 4/8 sanitizer-evasion figure is a lower bound without an upper-bound bracket.
- The claim that *every* mainstream renderer is vulnerable to T7 is argued from Unicode font coverage, not measured against a panel of real production MCP clients (no Cursor/Claude Desktop/Cline/Continue render test).
- "Reaches the model's context" is not "the model acted on it"; no end-to-end agent-behavior or attack-success-rate measurement against any specific deployed model is provided.
- The benign corpus is hand-written (25 descriptions), not sampled from deployed traffic, so real-world sanitizer false-positive prevalence is unknown.
- Two of the three cross-library implementations share the official `mcp` package's wire-protocol types, so the cross-library agreement is strong evidence against tool-management-layer artifacts but weaker against a shared wire-layer bug.
- Single author; the "three independent engineering teams" framing modestly overstates the independence of the re-implementations, all of which were written by the same person.
- The technique set covers five identified surfaces but is acknowledged as non-exhaustive; marketplace- or client-level mandatory security review is not evaluated.

## Relevance

对研究 LLM 智能体安全与工具调用（tool-use）的工作高度相关：它精确刻画了 MCP 协议中"审批视图"与"模型上下文"两条路径的保真度缺口，并给出可直接落地的协议层修复（字节忠实审批、定义突变重新授权、来源命名空间、schema 默认值需确认），为编码型智能体（coding agent）可信工具集成与防御研究提供了机理分析与可复现的攻击目录。

## Questions for Further Exploration

- How do real production MCP clients (Claude Desktop, Cursor, Cline, Continue) actually render the TAG-block payload — does any of them already normalize or reject unassigned-glyph codepoints, falsifying the universal-vulnerability claim?
- What is the attack-success rate (model actually acting on the payload) of T7 and T4 against frontier coding-agent models, as opposed to the delivery rate measured here?
- Would a semantic / embedding-based sanitizer trained on benign MCP descriptions substantially change the 4/8 evasion figure, and at what false-positive cost on deployed traffic?
- Is the TOCTOU re-consent gap (T3) closeable purely client-side by pinning the approved metadata hash, and do any current clients already do so?
- Does the concealment mechanism generalize beyond MCP to other agent architectures that show a user one rendering of untrusted data while forwarding different bytes to the model (e.g., browser-tool, OpenAI function-calling, A2A protocols)?
- Can the provenance-scoped namespace fix be enforced without breaking the legitimate case of a server deliberately overriding a host built-in?
- How prevalent are these eight techniques in the wild, given that the benign corpus was hand-written rather than sampled from a population of deployed servers?
