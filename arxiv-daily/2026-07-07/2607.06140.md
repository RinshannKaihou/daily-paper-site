> **一句话总结：** CURATE EVO 把 agent 后训练的数据清洗策略表示为可执行代码，并用留出开发集上的失败轨迹驱动多轮演化（GPT-5.4 + mini-SWE-agent 作为代码演化智能体），在 ACEBench-Agent/BFCL-V4/τ²-Bench 上以 Qwen3-4B 超过使用 8B 模型的最强基线，标注数据与野生数据平均分别提升 3.2 与 2.7 分，同时把数据清洗的 token 开销降低约 48%。

> **TL;DR:** CURATE EVO reframes agentic post-training data curation as evolving executable code: an LLM-based code-evolution agent (GPT-5.4 + mini-SWE-agent) rewrites the curation strategy each epoch using failed dev trajectories, jointly producing SFT data, RL data, and an inference-time memory bank under a cost-aware objective J(ρ)=P(ρ)−λC(ρ). On Qwen3-4B it beats the strongest prior baselines (MUA-RL, EnvScaler, RODS, FunReason-MT) on ACEBench-Agent, BFCL-V4, and τ²-Bench, improving average score by +3.2 (labeled) and +2.7 (wild data), while cutting curation token overhead by ~48%.

## 核心贡献

- 提出 CURATE EVO，一个失败驱动的动态演化框架：把数据清洗策略 ρ 表示为可执行代码，并基于留出开发集上的失败轨迹多轮改写该代码，而非把清洗当作固定的预处理步骤。
- 联合优化三类资源（SFT 数据、RL 数据、推理时记忆库 M_mem），并提出成本感知目标 J(ρ)=P(ρ)−λC(ρ)：先通过增广/过滤/精修覆盖失败模式提升有效性，再通过裁剪冗余与低效用训练轮次降低训练规模。
- 在标注数据（SWE-chat、AgentRewardBench、OpenHands-Feedback）与野生数据（真实用户交互轨迹）两种设置下，分别比最强基线平均提升 3.2 与 2.7 分，且 Qwen3-4B 超越多个使用 Qwen3-8B 的基线方法。
- 验证了 CURATE EVO 是一个数据侧模块，可与 GRPO、AgentGym-RL、ProRL-Agent 等不同后训练配方叠加，平均额外提升 21.3 分；并把每千条保留训练轮次的清洗开销降至 0.51M tokens / 405 秒。
- 提供了细粒度消融（操作级 + 资源级）以及失败模式分析（State Binding Drift、Adaptive Recovery Failure、Grounded Execution Gap），指出 agent 数据清洗的关键瓶颈。

## Key Contributions

- Proposes CURATE EVO, a failure-driven dynamic evolution framework that represents the data-curation strategy as executable code and iteratively rewrites it using failed trajectories from a held-out dev set, instead of treating curation as a static preprocessing step.
- Jointly constructs three resources (SFT data, RL data, and an inference-time memory bank) under a cost-aware objective J(ρ)=P(ρ)−λC(ρ), first covering diagnosed failure modes via augmentation/filtering/refinement, then pruning redundant or low-utility training turns.
- Validates the approach under two raw-corpus settings (labeled human-annotated trajectories vs. wild real user logs), improving average score by 3.2 and 2.7 over the strongest baselines, with Qwen3-4B outperforming several Qwen3-8B baselines.
- Demonstrates that CURATE EVO is a data-side module compatible with different post-training recipes (GRPO, AgentGym-RL, ProRL-Agent), yielding an average +21.3 gain, and reducing curation overhead to 0.51M tokens / 405s per 1K retained turns (~48% token and ~50% wall-clock reduction vs. baselines).
- Provides fine-grained operation- and resource-level ablations plus a failure-mode analysis (State Binding Drift, Adaptive Recovery Failure, Grounded Execution Gap) that pinpoints residual agent bottlenecks for future curation.

## Methodology

CURATE EVO keeps the agentic post-training procedure (base model + SFT+GRPO recipe + environment + inference budget) fixed and instead optimizes how training/inference resources are constructed from a fixed raw corpus D_raw.

**Formulation.** A curation strategy ρ maps the raw corpus into three outputs: ρ : D_raw → (D_sft, D_rl, M_mem). The trained policy is π_ρ = Train_{SFT+GRPO}(π_base, D_sft, D_rl), and at inference the policy additionally conditions on retrieved memory Retrieve(q, M_mem). The goal is ρ* = argmax_ρ Perf_dev(π_ρ, M_mem).

**Evolution loop (per epoch e).**
1. Execute the current code ρe on D_raw to produce D_e^{sft}, D_e^{rl}, M_e^{mem}.
2. Train π_e from the same base model with the fixed SFT+GRPO recipe.
3. Evaluate (π_e, M_e^{mem}) on the held-out dev set Q_dev; collect failed trajectories T_e^-, dev score s_e, and a curation profile Σ_e (data scale, turn distributions, trajectory lengths, redundancy).
4. An LLM-based code-evolution agent (GPT-5.4 with mini-SWE-agent) rewrites ρe to ρ_{e+1}.

**Cost-aware objective.** J(ρ) = P(ρ) − λC(ρ), where P(ρ) is dev performance and C(ρ) = tanh(log(1+Cost_train(ρ))/log(1+N_ref)) is a normalized training-turn cost. The discrete code optimization is approximated by two ordered revisions:
- *Effectiveness revision*: diagnose recurring failure modes F_e = g_diag(T_e^-) (e.g., incorrect tool selection, invalid argument grounding, weak multi-turn planning, poor error recovery, memory mismatch, noisy supervision); then rewrite code to augment/filter/refine data and extract memory entries targeting those gaps.
- *Efficiency revision*: remove duplicates, prune low-utility/noisy samples, compress repeated interaction patterns, and truncate over-long trajectories, conditioned on the diagnosed failure modes so that rare-failure and critical-recovery samples are retained.
- If a revised strategy does not improve J, it is rolled back to prevent error cascading.

**Final evolved curation code** (Appendix C) decomposes trajectories into decision-centric segments, separates transferable policy behaviors (kept for SFT/RL) from long-tail task knowledge (compressed into memory) and noisy fragments (discarded), and concentrates supervision on high-frequency failure families. The test set is never used during evolution.

## Results & Highlights

CURATE EVO uses Qwen3-4B as the base policy and a fixed SFT+GRPO recipe (LoRA r=16; SFT lr 2.5e-5, ≤140 steps; GRPO lr 5e-6, β_KL 0.14, ratio clip 0.05). All numbers below are from Table 1 (mean ± std over 3 runs).

- **Labeled data, ACEBench-Agent:** CURATE EVO scores 56.7 ± 0.6 vs. the strongest baseline MUA-RL at 53.3 (Δ +3.4), with Qwen3-4B beating Qwen3-8B baselines.
- **Labeled data, BFCL-V4:** 52.4 ± 1.0 vs. FunReason-MT at 50.3 (Δ +2.1).
- **Labeled data, τ²-Bench:** 41.9 ± 0.6 vs. EnvScaler at 37.9 (Δ +4.0), the largest gain, indicating the method helps multi-turn state-tracking beyond tool-call accuracy.
- **Wild data (noisy real user logs):** 55.8 ± 1.4 / 50.0 ± 0.6 / 37.2 ± 1.2 on ACEBench/BFCL/τ², beating the strongest wild-data baselines (MUA-RL 50.7/40.9/30.6, EnvScaler 48.8/46.2/35.5). Average improvement: +2.7 on wild, +3.2 on labeled.
- **GRPO without any curation is far weaker:** 32.1 / 13.8 / 26.8 on labeled data, showing raw trajectories give weak training signals.
- **Efficiency (Figure 3):** 0.51M tokens and 405s per 1K retained training turns — roughly 48% token overhead and 50% wall-clock reduction versus the average prior baseline (e.g., MUA-RL 1.18M/980s, EnvScaler 1.21M/1040s).
- **Recipe compatibility (Table 3):** adding CURATE EVO to GRPO, AgentGym-RL, and ProRL-Agent improves all benchmarks and both settings by an average of +21.3; best result is with ProRL-Agent (labeled τ² 43.4).
- **Ablation (Table 2):** removing the effectiveness revision costs ~7 avg points (e.g., labeled ACEBench 56.7 → 49.3); removing the efficiency revision causes a smaller but consistent drop (56.7 → 55.5); removing each resource hurts (SFT data: 56.7 → 51.9; RL data: → 50.2; memory: → 53.9), with larger drops under wild data.
- **Evolution dynamics (Figure 4):** most gains appear in the first 3 epochs (labeled avg 24.2 → 50.3; wild 21.8 → 47.7); after the main-experiment epoch performance saturates while curated scale keeps shrinking, supporting the two-phase design.
- **Cost weight (Figure 5):** λ=0.3 achieves the best average performance for both settings; λ=0 keeps a larger but worse set, while too-large λ over-prunes and removes useful long-horizon trajectories.

## Authors & Credibility

The paper is authored by Dingzirui Wang, Xuanliang Zhang, Keyan Xu, Qingfu Zhu, and corresponding author Wanxing Che from the Harbin Institute of Technology (HIT), a well-established NLP/group. The work is framed as a preprint/work-in-progress (arXiv 2607.06140v1, 7 Jul 2026). The experimental design is reasonably careful: fixed base model and fixed SFT+GRPO recipe across all compared methods, mean±std over 3 runs for the main table, a held-out test set never touched during evolution, and operation-level plus resource-level ablations isolating each component of J(ρ)=P(ρ)−λC(ρ). The benchmark suite (ACEBench-Agent, BFCL-V4, τ²-Bench) and baselines (MUA-RL, EnvScaler, AWM, RODS, FunReason-MT) are contemporary and relevant.

## Critical Assessment

- **Novelty.** Genuine and fairly specific: the key move is to treat the *data-curation code itself* as the optimization target and evolve it with an LLM coding agent using dev-set failures, rather than evolving prompts, policies, or replay buffers as in RODS/EnvScaler/FunReason-MT. The cost-aware objective J(ρ)=P(ρ)−λC(ρ) with the two ordered (effectiveness → efficiency) revisions is a clean formulation, though each individual ingredient (failure-driven augmentation, deduplication, trajectory truncation) is well-known.
- **Rigor.** Strong on controls: fixed base model + fixed recipe across methods, 3-run std reported for the main table, held-out test set walled off from evolution, λ and epoch-count ablations, and a resource-level ablation showing SFT/RL/memory are complementary. However, the baseline comparison has a notable confound: the strongest baselines (MUA-RL, EnvScaler, AWM) are reported with Qwen3-8B while CURATE EVO uses Qwen3-4B. The authors spin this as a positive ("4B beats 8B"), which is impressive, but a same-model-size (Qwen3-4B) run of MUA-RL/EnvScaler/RODS/FunReason-MT is not shown in Table 1, making the headline deltas not strictly apples-to-apples. RODS and FunReason-MT are at 4B, but they are the weaker baselines.
- **Reproducibility.** Mixed. Hyperparameters are fully specified (LoRA r=16, SFT/GRPO lrs, β_KL=0.14, clip settings, λ=0.3, N_ref=10^5, 3 epochs) and prompts are provided in Appendix A. But the entire framework hinges on a powerful proprietary coding agent (GPT-5.4 with mini-SWE-agent) rewriting curation code; no public code/release is mentioned, and the final evolved curation code is only summarized qualitatively in Appendix C rather than dumped, so reproducing the exact curated datasets is effectively impossible without the same evolution runs. The "evolved code" is itself a moving, model-dependent artifact.
- **Significance.** High if the claim holds: it argues that *adaptive curation* can matter as much as scaling the policy, and that curation should be a closed-loop, failure-aware process. The +21.3 average gain when stacked onto GRPO/AgentGym-RL/ProRL-Agent (Table 3) is the most compelling practical result, positioning CURATE EVO as a data-side module orthogonal to policy optimization.
- **Red flags / overclaiming.** (1) The 3.2/2.7 headline deltas mix model sizes for the strongest baselines (8B vs. the method's 4B) — not a head-to-head at matched capacity. (2) Wild-data baselines look conspicuously low relative to labeled-data baselines, and the paper does not explain why the same baseline method degrades so much; the deltas on wild data may partly reflect weak baseline curation on noisy data rather than a uniformly strong method. (3) Only 3 evolution epochs were run; Figure 4 actually shows gains continuing and the main-experiment point chosen before saturation, with no clear justification for stopping at 3 epochs for the main table. (4) Heavy dependence on GPT-5.4 as the evolution brain is under-discussed as a cost/fairness concern — the reported 0.51M-token curation overhead does not appear to include the evolution agent's own reasoning cost. (5) Memory bank retrieval's contribution is entangled with policy training; the memory ablation only removes the resource, not retrieval-vs-memorization effects.

## Strengths

- Clean, well-motivated formulation: curation as evolving executable code optimized against a cost-aware objective, with rollback to prevent cascading errors.
- Jointly produces SFT data, RL data, and an inference-time memory bank from one strategy, and ablations confirm all three are complementary (especially under noisy wild data).
- Strong empirical results on three diverse benchmarks (tool selection, function calling, long-horizon dual-control), under both clean labeled and noisy wild settings.
- Substantial efficiency gains: ~48% token and ~50% wall-clock reduction in curation overhead while improving accuracy.
- Recipe-agnostic: the +21.3 average gain when combined with GRPO/AgentGym-RL/ProRL-Agent is a strong practical selling point.
- Honest failure-mode taxonomy (State Binding Drift, Adaptive Recovery Failure, Grounded Execution Gap) with concrete future directions, rather than only celebrating wins.

## Limitations

- Entire framework depends on a powerful proprietary coding agent (GPT-5.4 + mini-SWE-agent); open-source or smaller evolution models are not tested, and the evolution agent's own compute/token cost is excluded from the overhead comparison.
- Headline baseline comparison mixes model sizes (strongest baselines MUA-RL/EnvScaler/AWM at Qwen3-8B vs. CURATE EVO at Qwen3-4B); no matched-size comparison of the strongest baselines is provided.
- Only 3 evolution epochs were used for the main table, even though Figure 4 shows gains continuing past that point; the choice of stopping epoch is not justified.
- The final evolved curation code is described only qualitatively (Appendix C); no code or data release is mentioned, limiting reproducibility.
- λ and N_ref are tuned; sensitivity is shown for λ but not for N_ref or for the choice of dev/test split ratio (fixed 9:1).
- Evaluation is limited to a 4B base model; scaling behavior to larger policies is untested.

## Relevance

这项工作直接服务于 agent 后训练这一主线课题，把数据清洗与策略优化解耦，与强化学习数据合成、failure-driven 训练以及 RAG/记忆增强推理高度相关，方法论（用代码演化驱动数据分布迭代）对自动化 agent 数据工程有借鉴价值。

## Questions for Further Exploration

- How does CURATE EVO behave when the code-evolution agent is an open-source model (e.g., Qwen-Coder) instead of GPT-5.4, and how much of the gain comes from the evolution agent's capability versus the evolution loop itself?
- What are head-to-head results when the strongest baselines (MUA-RL, EnvScaler) are also run at Qwen3-4B, so the +3.2/+2.7 deltas are measured at matched capacity?
- Does the curated memory bank generalize to unseen task families, or does it mostly memorize benchmark/schema-specific anchors (a potential test-set leakage risk via schema matching)?
- How does performance scale with base model size (7B/8B/14B), and does the relative benefit of curation grow or shrink?
- If the evolution runs for more epochs (Figure 4 suggests gains continue), what is the asymptotic performance, and does over-evolution start to overfit the dev set?
- Can the failure-mode taxonomy (State Binding Drift, Adaptive Recovery Failure, Grounded Execution Gap) be turned into explicit reward or verifier signals rather than only curation guidance?
