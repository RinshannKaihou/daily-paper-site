> **一句话总结：** Akashic 提出基于分块的 MemAttention 内存维护机制与软硬件协同的内存管理器，通过跨块推理恢复跨段语义证据并按亲和度共置物理存储，在 LoCoMo/SWE-bench/BrowseComp/WebArena 四个基准上同时实现最高 10.2 点精度提升、1.21× 吞吐与 1.88× 可持续请求率。

> **TL;DR:** Akashic is a low-overhead LLM inference service whose MemAttention performs bounded chunk-granular memory maintenance with cross-chunk reconciliation, paired with a hardware-software co-designed memory manager that co-locates likely co-retrieved chunks; on LoCoMo, SWE-bench, BrowseComp, and WebArena it improves accuracy by up to 10.2 points and throughput by up to 1.21× over the strongest prior memory baselines (Mem0, MemGAS, MemGPT, RMM), and sustains up to 1.88× higher request rate under concurrent serving.

## 核心贡献

- **识别两类瓶颈**：刻画了智能体记忆中的两个系统性瓶颈——固定摘要策略与异构上下文密度之间的失配，以及语义相关性与物理放置之间的"局部性鸿沟"（locality gap）。
- **MemAttention 算法**：提出分块粒度的记忆维护机制，仅压缩一个有界分块（默认 1024 token），并对每个新块与少量语义相关的历史块做"跨块推理"，从而在不重写全量历史的前提下恢复跨段证据。
- **软硬件协同的内存管理器**：用模型推断块间亲和度，将常被共同检索的分块物理共置，并以 LSM 风格的 out-of-place 写入与 75% 无效比触发的后台 GC 控制空间/写放大。
- **端到端实现与评测**：在 vLLM v0.10.0 控制路径上实现（GPU 执行路径不动），覆盖 Qwen-8B/OPT-30B/Llama-70B 三种规模与四类工作负载，证明同时在精度与效率上处于 Pareto 前沿。

## Key Contributions

- **Bottleneck characterization**: Identifies two systems-level bottlenecks in agent memory — the mismatch between fixed summarization policies and heterogeneous (cross-dataset and within-trajectory) context density, and the "locality gap" between semantic relatedness and physical placement.
- **MemAttention algorithm**: A chunk-granular memory architecture with gate-triggered compaction (bounded per-step cost O(L_chunk) ≤ τc) plus cross-chunk inference that reconciles each new chunk with the top-5 related prior chunks, preserving cross-chunk evidence that independent segment summaries would fragment.
- **Hardware-software co-designed Memory Manager**: Uses model-inferred chunk affinity to co-locate likely co-retrieved chunks out of place, with background block compaction triggered at invalid ratio φ(B) ≥ 0.75, closing the locality gap at moderate space/write amplification.
- **Unified relevance**: The same model-driven semantic matching mechanism is reused for both compaction-time reconciliation and inference-time retrieval, yielding a single notion of memory relevance with bounded maintenance and retrieval cost.

## Methodology

**MemAttention** treats memory construction as incremental chunk-level maintenance rather than global history compression. Context is partitioned into fixed-size chunks (default 1024 tokens). For each active session the system keeps one append-only active chunk plus a set of compacted chunks keyed by (user_id, session_id, metadata). When the active chunk length crosses the gate τc, the system (1) compresses only that bounded chunk, (2) extracts metadata keywords, (3) gathers metadata of previously compacted chunks in the same namespace and asks the LLM to select the top-5 most related prior chunks, and (4) performs joint compaction that refines the new chunk and updates or deletes stale prior memories (cross-chunk inference, inspired by Claude Code). At inference time, the current query keywords and candidate chunk metadata are fed to the model, which selects the top-5 relevant chunks; these are fetched, reordered chronologically, and prepended to the prompt under a memory token budget. The same model-driven matcher is used for both maintenance and retrieval.

**Memory Manager** targets the locality gap (|P(R_t)| dominating retrieval cost even when |R_t| is small). Chunks live in a disk-resident store; logical identity is (user_id, session_id, metadata), while physical placement is a separate block layout. Two stages run: (i) an online relocation path that, during joint inference, uses model-inferred affinity to pack co-accessed chunks into new block(s) out of place, tombstoning old copies; and (ii) a background compaction path that, for blocks whose invalid ratio φ(B) ≥ 0.75, extracts live chunks, partitions by (user_id, session_id) to preserve tenant/session isolation, groups them by model-inferred affinity, and repacks them temporally ordered. Updates follow an LSM-tree-style log of sequential writes + tombstones + background reclamation.

**Implementation** extends the control path of vLLM v0.10.0 while leaving the GPU execution path, attention backend, and KV-cache manager unchanged. Memory retrieval is hooked at the request admission layer, memory write-back at the request completion layer, and physical layout maintenance is split into online relocation + background storage manager — keeping long-term memory outside the hot GPU decoding loop. Core parameters: chunk_gate_tokens (default 1024), top_p=5, memory_budget_tokens, gc_invalid_ratio=0.75.

## Results & Highlights

- **Accuracy/throughput under single-request (batch=1) on LoCoMo & BrowseComp**: Akashic improves throughput over Mem0, MemGAS, and RMM by **1.08×–1.10×, 1.10×–1.21×, and 1.33×–1.54×** respectively, while improving accuracy by **8.4–10.2, 4.2–7.2, and 2.1–5.2 points**; vs. Full-context it is **1.27×–2.05× faster** with **1.2–3.4 points higher accuracy**.
- **On SWE-bench** (high information density, less compressible): Akashic improves throughput over Mem0, MemGAS, and RMM by **1.31×–1.33×, 1.15×–1.19×, and 1.41×–1.56×**, and resolved rate by **4.6–6.6, 2.9–3.7, and 1.8–2.2 points**; it retains **90.5%–91.8%** of Full-context throughput while improving resolved rate by 0.6–1.1 points.
- **Concurrent serving (sustainable request rate)**: On LoCoMo, Akashic sustains **1.22×–1.43×** higher rates than Mem0, **1.28×–1.74×** than MemGAS, and **4.67×–6.64×** than Full-context; on WebArena **1.40×–1.54× / 1.26×–1.34× / 3.40×–4.30×**; on SWE-bench **1.53×–1.88× / 1.32×–1.58× / 1.11×–1.26×** — overall up to **1.88× higher sustainable request rate**.
- **BrowseComp/OPT-30B headline (Figure 1b)**: Akashic beats the strongest prior method by about **2.0 accuracy points** and about **1.35× throughput**.
- **Locality gain (Figure 10)**: blocks/request on LoCoMo drops from 6.8 (Append-only) / 5.1 (Semantic-only) to **3.4** (−50.0% / −33.3%); p95 cold-cache retrieval latency falls from 38 / 30 ms to **22 ms** (−42.1% / −26.7%); storage cost stays moderate at **1.17× space amplification and 1.34× write amplification**.
- **Ablation**: Removing bounded chunk-level maintenance (Whole-context) collapses throughput to 82.8%–88.9% while task quality drops to 93.2%–95.5%, identifying it as the dominant efficiency contributor; cross-chunk reconciliation adds modest overhead (102.2%–103.4% throughput) for 95.9%–96.4% quality.
- **Motivating cost of memory**: Naive memory-augmented inference reduces end-to-end throughput by **48.0%–63.9%** on SWE-Bench/BrowseComp/WebArena, and under concurrency the standard-vs-memory throughput ratio only narrows from 2.08× (2 req) to 1.43×–1.44× (16 req) — quantifying the locality gap Akashic targets.

## Authors & Credibility

The work is a collaboration between **Xiaohongshu Inc.** (industry, corresponding author Zhaokai Luo), **Shanghai Jiao Tong University**, and **Peking University**. The same group authored the related RedKnot long-context serving system (arXiv:2606.06256), and Akashic is pledged to be merged into the RedKnot open-source branch (https://github.com/rednote-machine-learning/RedKnot). Evaluation spans realistic production-style hardware (NVIDIA H800 with PCIe Gen5 and 4 TB NVMe SSD) and four community-standard agent benchmarks (LoCoMo, SWE-bench, BrowseComp, WebArena) at three model scales (8B/30B/70B), against five representative baselines covering whole-context (Mem0), segment-level (MemGAS), and hierarchical (MemGPT, RMM) strategies.

## Critical Assessment

- **Novelty**: Genuine. The framing of agent memory as a joint logical + physical optimization problem is fresh: most prior memory work (Mem0, MemGAS, A-MEM, RMM) treats storage as an opaque backend, and the explicit "locality gap" formalization (|P(R_t)| vs. |R_t|) ties agent memory to the disk-resident ANN/SPANN/Starling locality literature in a non-obvious way.
- **Rigor**: Largely solid. Four benchmarks, three model sizes, five baselines, latency-vs-concurrency sweeps, locality/storage-amplification microbenchmarks, and a parameter sensitivity study over τc, Bmem, τassoc, θgc. The ablation credibly isolates bounded chunk maintenance as the dominant contributor.
- **Weak baselines / caveats**: The baselines are reasonable but not all tuned equally — Mem0's whole-context summarization is a known weaker operating point, and the paper does not report whether Mem0/MemGAS/RMM were given the same memory token budget or retrieval depth (top-5) as Akashic, which matters for fair comparison. The reported accuracy ranges (e.g., "8.4–10.2 points over Mem0") collapse across models/benchmarks and obscure where gains actually land; the headline 10.2 points is the upper bound on LoCoMo/BrowseComp, the more compressible regime, while SWE-bench gains are 4.6–6.6 at best.
- **Missing ablations**: There is no ablation isolating the *model-driven* matcher from a strong dense-embedding retriever at matched budget (the "Dense-only" ablation is presented but the gap to full Akashic is small, 97.2%–97.3% quality, which undercuts the claim that model-driven matching is essential — it may be that a well-tuned retriever suffices). No latency breakdown of the extra LLM calls added by MemAttention (cross-chunk inference + retrieval matching both consume GPU), which is a real serving cost not fully accounted for in throughput figures.
- **Reproducibility**: Promised open source in RedKnot; vLLM/SGLang versions and hardware specs are stated. However, prompts for the model-driven matcher, affinity inference, and joint compaction are not detailed in the paper, and these prompts are load-bearing for reproduction.
- **Significance**: High if the gains hold — a Pareto improvement on *both* axes against established memory systems is uncommon. But the SWE-bench regime (the hardest, highest-density case) shows the smallest margins and the least compression headroom, which is exactly where the "memory maintenance is worth it" thesis is most contested.
- **Red flags**: The motivation data (memory-augmented inference is 48–64% slower than standard inference) somewhat undercuts the case that any of these memory systems, including Akashic, beat simply dropping history — the strongest baseline Akashic beats on LoCoMo concurrency is Full-context by 4.67×–6.64×, but that reflects Full-context's collapse under concurrency, not Akashic's absolute efficiency. The claim to be "the first system to efficiently realize long-context filtering for LLM agents through end-to-end software–hardware co-design" is defensible but narrowly worded.

## Strengths

- Jointly optimizes the semantic (what to remember) and physical (where to place it) axes, which is rare and well-motivated.
- Bounded, threshold-triggered chunk maintenance gives predictable per-step cost independent of total history length — a clean systems insight borrowed from LSM/Bigtable/LFS.
- Reusing one model-driven matcher for both maintenance reconciliation and retrieval is elegant and removes a hand-tuned scoring function.
- Comprehensive evaluation across diverse density regimes (dialogue, code, web) and model scales, with honest reporting that SWE-bench gains are smaller because it is less compressible.
- Storage overhead is explicitly measured (space/write amplification) and compared against Aggressive-Relocation and No-GC alternatives, not hand-waved.

## Limitations

- The paper relies heavily on additional LLM calls for cross-chunk inference, metadata matching, and affinity grouping; the GPU cost of these calls is not isolated in the throughput accounting and could erode gains in compute-bound (vs. latency-bound) deployments.
- Gains are heavily concentrated in low-density, highly compressible workloads (LoCoMo, BrowseComp); on information-dense SWE-bench the throughput advantage over Full-context shrinks to ~90% retention, narrowing the value proposition precisely where agents most need long context.
- The model-driven matcher prompts are not specified, threatening reproducibility; the "Dense-only" ablation landing at 97%+ of full quality suggests the LLM-based matching may be replaceable by a cheaper retriever.
- No evaluation against very recent memory systems (e.g., A-MEM) as baselines, and MemGPT is listed as a baseline category but its specific numbers are not broken out in the results narrative.
- All experiments use the group's own OPT-30B as the memory/relevance model; sensitivity to the matcher model quality is not studied.

## Relevance

This paper is directly relevant to long-horizon LLM agent serving and memory systems — it tackles exactly the cost/quality trade-off that arises when agents accumulate multi-turn, multi-session context, and its chunk-granular maintenance plus locality-aware storage design are practical for production inference stacks. For research on agent memory, KV/context management, and LLM serving efficiency, it offers both a concrete architecture and a benchmarking methodology worth following.

## Questions for Further Exploration

- How do Akashic's gains change when the matcher/relevance model is a small, fast model (e.g., 1–3B) versus the 30B model used here — is the model-driven matcher cost-effective at all scales?
- What is the breakdown of end-to-end latency into (a) the extra LLM calls for cross-chunk inference and affinity grouping, (b) retrieval I/O, and (c) the actual forward pass — and does Akashic still win in a compute-bound regime where GPU is saturated?
- Could a strong dense retriever + lightweight cross-encoder reranker replace the model-driven matcher at near-equivalent quality, given that the "Dense-only" ablation already reaches ~97% of full quality?
- How does the locality benefit scale with the number of tenants/sessions and the write churn rate — does the 75% GC threshold need to be adaptive under heavy multi-tenant load?
- Does the chunk gate τc = 1024 generalize to other model families and tokenizers, or is it overfit to Qwen/OPT/Llama tokenization?
