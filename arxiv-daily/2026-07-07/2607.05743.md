> **一句话总结：** 本文是一篇系统化知识(SoK)综述，将 2023-2026 年间 39 篇 AI 编码代理执行安全论文归入 17 个类别，对照 NIST NVD 确认了 4 个已披露 CVE(其中 2 个在 Claude Code 自身)，并通过跨类别阅读揭示出 5 个未被任何单篇论文察觉的系统性研究空白(如隔离机制与访问控制从未在同一基准上对比、防御机制未在真实策略失效率下复测、TOCTOU 与 MCP 威胁实为同一类 validate-then-act 竞态却被当作两支独立文献研究等)。

> **TL;DR:** This SoK systematizes 39 execution-security papers for AI coding agents (2023-2026) into 17 verified categories, independently confirms 4 patched CVEs (2 in Claude Code itself) against the NIST NVD, and exposes 5 cross-cutting gaps invisible within any single category — most notably that no isolation architecture (e.g., IsolateGPT) has ever been benchmarked head-to-head against any access-control system (e.g., PORTICO, SEAgent), and that defenses reporting strong rates against author-built attackers are never re-tested against ShellSieve's empirically discovered bypasses that defeat 69.0%–98.6% of real command denylists.

## 核心贡献

- **系统化的执行安全语料库**：将 2023-2026 年间 39 篇论文按主执行安全机制归入 17 个类别(隔离架构 5 篇、逃逸/对抗基准 6 篇、访问控制与能力模型 6 篇、策略执行脆弱性 2 篇、TOCTOU 2 篇、MCP 威胁与防御 4 篇等)，每篇均直接对照原始 arXiv 摘要页核实标题、作者与贡献，而非采信二手摘要。
- **真实 CVE 接地**：对照 NIST 国家漏洞数据库独立确认了 4 个已披露并修复的 CVE，直接影响生产级代理工具链——CVE-2024-21626(runc 容器逃逸 "Leaky Vessels")、CVE-2025-53773(GitHub Copilot/VS 命令注入，CVSS 7.8)、CVE-2025-59536(Claude Code 启动信任对话框代码注入，CVSS 8.8)、CVE-2026-21852(Claude Code 项目加载流数据外泄，CVSS 7.5)，将综述建立在真实事件而非论文自述威胁模型之上。
- **五个跨类别研究空白**：通过跨类别阅读而非类别内阅读，发现(1) 隔离与访问控制从未在共享基准上对比；(2) 防御机制未在 ShellSieve 测得的真实策略失效率下复测；(3) TOCTOU 与 MCP 威胁实为同一 validate-then-act 竞态却被当作两支互不引用的文献；(4) 所有执行机制都假设诚实的策略作者，从未研究策略撰写错误本身；(5) 新测量的 scope-creep 失效模式(过度执行率达 17.1%)不被任何访问控制机制覆盖。
- **四类根因抽象**：将 17 个类别坍缩到 4 个反复出现的根本缺陷——RC1(不可信内容与控制输入无结构分离)、RC2(授权一次检查即永久信任)、RC3(机制判定"动作类型是否允许"而非"此调用此刻是否符合意图")、RC4(防御仅对照作者自构造攻击者验证而非真实观测失效)。
- **可审计的机器可读语料文件**：发布 CSV 语料文件与生成器/校验器脚本，使每个计数(39 篇、各类别计数、年份分布)均可由校验器从数据文件机械重导并与正文核对，分类可被读者逐条质疑或扩展。
- **面向五个空白的研究议程**：未来工作直接指向闭合空白(如用 SandboxEscapeBench/RedCode 共享攻击语料对比隔离 vs 访问控制、用 ShellSieve 已发布绕过语料复测各防御)，而非重述各论文自述的未来工作。

## Key Contributions

- **A systematized corpus of 39 execution-security papers (2023-2026) into 17 categories**, each verified directly against its arXiv abstract page rather than a secondary summary; the verification protocol caught and corrected two misattributions during the review (one paper miscategorized from a search summary, one conflating two MCP papers by the same first author).
- **Independent confirmation of 4 disclosed, patched CVEs against the NIST NVD**, grounding the corpus in real resolved incidents: CVE-2024-21626 (runc "Leaky Vessels" container escape), CVE-2025-53773 (GitHub Copilot/VS 2022 command injection, CVSS 7.8), CVE-2025-59536 (Claude Code startup trust-dialog code injection, CVSS 8.8, CWE-94), CVE-2026-21852 (Claude Code project-load data exfiltration including API keys, CVSS 7.5) — two of which affect Claude Code itself and are structurally the same trust-boundary race the TOCTOU category studies.
- **Five cross-cutting gaps identified by reading across categories** rather than within any single one: (1) isolation vs. access-control evaluation gap, (2) enforcement-fragility vs. defense-evaluation gap, (3) TOCTOU-MCP vocabulary gap, (4) honest-policy-author assumption, (5) unaddressed scope-creep failure mode.
- **Four recurring root causes (RC1-RC4) that collapse the 17 categories**, showing that no single existing mechanism addresses both a missing-state failure (RC1/RC2) and a wrong-question failure (RC3/RC4), and that partial defenses compose in the taxonomy but do not compose along the causal chains.
- **A released, machine-readable corpus file plus generator and verifier scripts** so every count asserted in prose is mechanically re-derived from one data file; the verifier reports a single pass/fail summary, catching any drift between corpus and body before the paper builds.
- **A research agenda organized around closing the five gaps** (e.g., running isolation and access-control mechanisms against a shared attacker corpus, re-testing defenses against ShellSieve's released bypass set, a unified formal treatment of validate-then-act races across filesystem/browser/MCP settings), rather than restating individual papers' future-work sections.

## Methodology

This is a Systematization of Knowledge (SoK), not a new attack or defense. The method has four stages:

1. **Search and inclusion.** The authors searched arXiv and web literature with combinations of terms (sandbox, execution isolation, agent harness, coding agent, capability, access control, TOCTOU, Model Context Protocol), iterating the query set as each round surfaced new author groups, venues, and vocabulary. A paper was included if it (a) addresses the execution layer of an LLM-based agent — the boundary between the agent's decisions and the OS/filesystem/network/tool-calling substrate — and (b) was published 2023 to time of writing. Papers whose primary contribution is model-level alignment, text-level prompt-injection defense without an execution-layer component, or agentic security surveyed only as one catalog item were excluded.

2. **Verification protocol.** For every candidate, the authors fetched its arXiv abstract page directly and recorded the exact title, full author list, venue, and one-sentence contribution before admission; a candidate was admitted only when these agreed with the drafted citation. Two candidates failed and were corrected or dropped. The four CVEs were verified separately against NIST NVD listing pages (CVE-ID, affected product, version range, CVSS score, disclosure date).

3. **Categorization.** Each verified paper was assigned to exactly one of 17 categories by primary mechanism (Table 1). The full corpus with verified title, authors, venue, and arXiv ID is released as a plain CSV so the classification is falsifiable rather than asserted.

4. **Reproducibility.** Table 1 and every count in the paper are generated from a single machine-readable corpus file recording each paper's bibliographic identity, category, root-cause attribution, and pipeline stage. An independent checker re-derives every count and compares it to the prose/tables; it also verifies no bibliography entry is duplicated and every citation resolves. The data file and checker are released as supplementary artifacts.

A second synthesis re-reads the same 39 papers along two axes: (i) **root cause** — what design defect a technique actually responds to, collapsing the taxonomy onto four recurring root causes (RC1-RC4) — and (ii) **pipeline stage and role** — where in the action pipeline (pre-action / at-action / post-action / post-hoc) a technique intervenes and whether it proposes a mechanism or measures one. A pairwise category-similarity heatmap (cosine similarity of RC-weight vectors plus a shared-stage bonus) independently surfaces the same gaps Sections 6.1-6.3 identify by close reading.

## Results & Highlights

- **Corpus scale and growth**: 39 papers systematized into 17 categories; publication-year breakdown is 2023: 4, 2024: 4, 2025: 14, 2026: 17 — the majority of the corpus was published in the most recent year alone, indicating execution security emerged as an active concern only after coding agents began operating with broad, largely unsandboxed tool access.
- **Denylist fragility (ShellSieve)**: tested against 1,709 command denylists scraped from GitHub, 69.0% to 98.6% are fragile depending on bypass class — the most consequential finding in the corpus because it measures real shipped artifacts rather than a mechanism's own claimed guarantee; YoloFS independently documents the same failure mode from 290 public misuse reports across 13 agent frameworks.
- **Scope-creep overeager-action rate (OverEager-Bench)**: across a purpose-built benchmark of 500 validated scenarios run over ~7,500 runs across four agent products and six base models, simply removing an explicit statement of authorized scope from the prompt raised the measured overeager rate on Claude Code from 0.0% to 17.1% with no change to the task or the model — a failure mode not addressed by any access-control or capability paper in the corpus.
- **Pwn2Own Berlin 2026 real-world validation**: the ZeroDay Initiative's dedicated Coding Agents category (targeting Claude Code, OpenAI Codex, Cursor) paid out $1,298,250 across 47 unique zero-days; OpenAI Codex fell to a single input-neutralization bug (CWE-150) for a $40,000 award, and the contest's qualifying bar explicitly required a security-boundary crossing reachable through ordinary use while ruling out pure model jailbreaks — direct evidence the attack surface catalogued here is neither hypothetical nor niche.
- **MCP server empirical study**: of 1,899 open-source MCP servers examined, 7.2% carry traditional-style vulnerabilities, 5.5% exhibit MCP-specific tool poisoning, and 66% show code smells — concluding MCP security tooling has not kept pace with MCP adoption; most of seven prominent MCP clients implement no static validation against tool-poisoning (the standout client-side risk).
- **Defense-mechanism headline rates (each from a single paper, author-built attacker)**: IsolateGPT reports under 30% overhead; DeltaBox cuts checkpoint/rollback latency to single-digit milliseconds; a transactional sandbox reports 100% high-risk-command interception and 100% rollback success in its own evaluation; Open Agent Passport drops attacker success from 74.6% to 0% under a strict policy (assuming the enforcement framework itself is uncompromised); TOCTOU mitigation reduces TOCTOU flaws in executed trajectories from 12% to 8%; Agent-Sentry stops 94.3% of successful injections while allowing 95.1% of benign executions.
- **SecureVibeBench (static analysis)**: anchored to 105 C/C++ secure-coding tasks from 41 real OSS-Fuzz projects; even the strongest agent+model combination produced solutions that were correct and secure only 23.8% of the time — agents are, on their own evaluation, more often wrong about security than right.
- **CVE CVSS scores confirmed against NVD**: runc CVE-2024-21626; GitHub Copilot/VS CVE-2025-53773 (CVSS 7.8 High); Claude Code CVE-2025-59536 (CVSS 8.8 High, CWE-94, fixed in 1.0.111); Claude Code CVE-2026-21852 (CVSS 7.5 High, fixed in 2.0.65).

## Authors & Credibility

Single author: Mohammadreza Rashidi, Department of Computer Science, AI and Media Analysis Lab, Berlin, Germany. The paper is a SoK (systematization of knowledge) and is explicit that it runs no new attacks and builds no new defense — its credibility therefore rests entirely on the rigor of its verification protocol rather than on experimental novelty. That rigor is unusually well-evidenced for a survey: every paper was fetched at its own arXiv abstract page, two misattributions were caught and corrected during verification, the four CVEs were confirmed directly against NIST NVD listing pages (not secondary descriptions), and the entire corpus plus a mechanical checker that re-derives every count are released as supplementary artifacts so any classification can be challenged by editing one line and re-running the checker. The author is candid about the limits of verification (it confirms bibliographic identity, not independent replication of empirical claims) and about coverage limits (iterative citation-following search rather than exhaustive). The grounding in Pwn2Own Berlin 2026 results and four real CVEs gives the threat model concrete, externally verifiable backing rather than relying solely on papers' self-reported threat models.

## Critical Assessment

- **Novelty**: Genuine. This is, to the author's knowledge and supported by the related-work section, the first survey to treat execution security for AI coding agents as a subject with its own internal structure rather than as one entry in a wider agentic-security catalog. The four broader surveys it identifies ([1]-[4]) each treat sandboxing as one defense among many alongside prompt injection, retrieval poisoning, and multi-agent risk; none organize execution-security work specifically across the mechanisms the field has actually built. The four-root-cause abstraction (RC1-RC4) and the pipeline-stage/role re-reading are original synthesis, not a recapitulation of prior surveys' taxonomies.

- **Rigor (verification)**: High and transparent. The verification protocol is applied uniformly, the two caught misattributions are reported honestly (one paper miscategorized from a search summary, two candidate titles conflated into one entry), and the released corpus file plus mechanical checker make the counts falsifiable. The author explicitly states the verifier "takes no arguments and reports a single pass/fail summary, so any change to the corpus that would silently drift the body numbers is caught before the paper builds" — a reproducibility commitment stronger than most SoKs.

- **Rigor (synthesis evidence)**: Mixed, and the author admits it. The five gaps are "argued from the absence of a citation or a shared benchmark across categories, which is a real but indirect form of evidence; it is possible relevant cross-category work exists in a venue or preprint outside the corpus we searched." The pairwise-similarity heatmap (Figure 18) is offered as independent confirmation but its off-diagonal scores depend on the author's RC-weight assignments, which are themselves interpretive — the author concedes "several papers, notably SEAgent and FORGE, combine elements of more than one category, and we assign each to the single category that best matches its primary contribution," which understates overlap.

- **Reproducibility**: Strongest aspect. Four artifacts released (data file, two generators, verifier); the paper itself builds warning-free on a standard LaTeX/BibTeX cycle. A reader can re-run the verification pipeline (a script that walks the corpus, fetches each abstract page, parses metadata, reports mismatches) on their own machine.

- **Significance**: High for the subfield. The single most actionable finding is that the two enforcement-fragility papers (ShellSieve's 69.0%-98.6% bypass rate against 1,709 real denylists; YoloFS's 290 real misuse reports) are the natural adversarial test set for every access-control mechanism in Section 4, yet none of those mechanisms has been re-tested under them — this is a concrete, low-cost evaluation that would convert untested claims into measured ones, and the paper names it explicitly as "the most consequential finding in the whole literature."

- **Red flags / overclaiming**: Minimal. The author is disciplined about scope: this is "a systematization, not a measurement study," the corpus file "is intended to make later extension straightforward rather than to claim completeness," and eight of seventeen categories contain exactly one paper, reported honestly as ambiguous between "a genuinely nascent research direction" and "an incomplete search." One mild concern: the headline defense rates cited from individual papers (100% interception, 74.6%→0% attacker success, 94.3% injection stop rate) are reported as each paper's own claim against its own author-constructed attacker — the paper correctly flags this as Gap 2/RC4 but a casual reader could still mistake these cross-paper numbers as comparable, when they are measured against non-comparable attacker sets.

- **Paper-specific weak point (named)**: The honest-policy-author gap (Gap 4) is identified but not itself measured — the author notes "policy-authoring error is plausibly at least as large a source of real-world risk as enforcement failure" yet the paper offers no data on it, only the agenda item that a ShellSieve-style study aimed at access-control policies (rather than command denylists) "would tell the field whether the mechanisms it has built are being undermined more by weak enforcement or by policies that were never correctly specified in the first place." This is the gap most consequential and least addressed.

## Strengths

- First dedicated systematization of execution security for AI coding agents as a subject with its own internal structure, filling a gap four broader agentic-security surveys left open.
- Verification protocol is unusually rigorous for a survey: direct fetch of each paper's arXiv abstract page, two misattributions caught and corrected, four CVEs confirmed against NIST NVD rather than secondary descriptions.
- Released machine-readable corpus file plus generator and verifier scripts make every count mechanically re-derivable and falsifiable — a reproducibility commitment stronger than most SoKs.
- The four-root-cause abstraction (RC1-RC4) and the pipeline-stage/role re-reading are genuine cross-cutting synthesis that surface non-obvious structural similarities (e.g., TOCTOU filesystem races and MCP tool-poisoning are the same validate-then-act defect at different layers).
- Grounding in real-world incidents (Pwn2Own Berlin 2026 coding-agents category, four patched CVEs including two in Claude Code) makes the threat model concrete and externally verifiable rather than speculative.
- Honest about limitations: coverage rather than measurement error, iterative rather than exhaustive search, single-category assignment understates overlap, gaps argued from citation absence (indirect evidence).

## Limitations

- SoK, not a measurement study: no new attacks or defenses; central limitation is coverage rather than measurement error. The search was iterative and citation-following, not exhaustive, and arXiv posting norms mean some relevant work may not surface under the search terms used.
- Eight of seventeen categories contain exactly one paper (and several contain two); the author reports this honestly as ambiguous between a genuinely nascent direction and an incomplete search, and advises weighting claims about those categories accordingly.
- The verification protocol confirms bibliographic identity (title, authors, venue match the abstract page) but "does not constitute independent replication of any paper's empirical claims; we report those claims as stated by their authors." The four CVEs were confirmed against NVD for affected product, version, and CVSS score, but the underlying exploits were not independently reproduced.
- The five gaps are argued from the absence of a citation or shared benchmark across categories — "a real but indirect form of evidence; it is possible relevant cross-category work exists in a venue or preprint outside the corpus we searched."
- Single-category assignment understates overlap: SEAgent and FORGE combine elements of multiple categories but are assigned to one, so the degree of inter-category overlap is understated.
- The pairwise-similarity heatmap (Figure 18) depends on the author's interpretive RC-weight assignments (X=1.0, (x)=0.5), which are themselves a synthesis claim rather than an objective measurement.
- Single author: a 39-paper systematization with per-paper root-cause and pipeline-stage attribution is a large interpretive load for one reviewer, increasing the chance of a classification a second reviewer would dispute.

## Relevance

本文与 AI 编码代理执行安全高度相关：它系统化梳理了沙箱隔离、访问控制、TOCTOU 竞态、MCP 威胁等机制及其相互空白，对构建安全编码代理(如 Claude Code、Codex、Cursor 类工具)的防御层选型与基准评估具有直接指导价值，尤其指出了用真实失效语料(ShellSieve 的 1,709 条 denylist 绕过集)复测防御机制的低成本可执行路径。

## Questions for Further Exploration

- If a representative isolation architecture (e.g., IsolateGPT) and a representative access-control system (e.g., PORTICO or SEAgent) were run against a shared attacker corpus like SandboxEscapeBench or RedCode, are they substitutes or complements — and does layering both produce strictly stronger containment than either alone?
- How do the access-control mechanisms in Section 4 perform when re-tested against ShellSieve's released bypass corpus (1,709 real denylists, 69.0%-98.6% fragility) rather than their own author-constructed attackers?
- Can a unified formal treatment of validate-then-act races cover filesystem TOCTOU, browser-DOM TOCTOU, and MCP tool-metadata poisoning under one model, so a mitigation designed for one setting can be checked against the others without translation?
- What is the empirical rate of policy-authoring error in real agent deployments (overly permissive, contradictory, or time-pressured scopes), and does it dominate enforcement failure as a source of real-world risk — the ShellSieve-style study aimed at access-control policies rather than command denylists that Gap 4 calls for?
- When OverEager-Bench's already-collected overeager trajectories are replayed against a capability system (PORTICO) or an information-flow framework (SEAgent), what fraction of scope-creep actions does an authorization mechanism built for a different threat model actually catch — and does the answer change the design premise that constraining "what an agent is allowed to do" also constrains "whether an authorized capability is invoked when it should not be"?
