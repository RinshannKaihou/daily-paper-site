> **一句话总结：** 提出 Floor First 残差驱动分诊方法，将 LLM 解码步骤建模为五维资源向量并算出 [max,sum] 双侧 floor，在 16×H20 上的 DeepSeek-V3.2 风格 671B MoE/MLA 案例中用 wall ordering 解释了 TP16 与 EP16+DP-attention 在生产中相反的选择，并指出单流场景下 TP 反而领先 2.4×。

> **TL;DR:** Floor First is a residual-driven triage workflow for LLM serving that builds a two-sided analytical floor ([max,sum]) from a five-dimensional decode resource vector (HBM bytes, FLOPs, network bytes, network messages, KV capacity), then escalates to Nsight profiling only when a benchmark residual exceeds 1.3×. On a DeepSeek-V3.2-style 671B MoE/MLA model on 16×H20 GPUs, the floors show TP16 is KV-capacity-limited to B_max≈70 concurrent 8K requests while EP16+DP-attention reaches B_max≈644 (≈9× higher), yet at single-stream B=1 TP16 wins by 2.4× (TPOT floor 4.9 ms vs 11.8 ms) — making the layout choice a computable function of the operating point that explains why production deployments shipped opposite attention layouts on identical hardware.

## 核心贡献

- **残差驱动的分诊工作流**：将目标定义、floor 构造、benchmark 对账、profiler 升级拆成显式阶段，带有停止规则（1.3× 阈值），并把裁定结果分派到 pipeline 优化或 kernel 优化两条执行臂，避免无脑 grid search。
- **双侧 floor 模型 [max, sum]**：同一硬件资源内竞争项相加、独立引擎取 max 得到乐观 floor topt，全相加得悲观 floor tsum；测量值落在区间内的位置本身是零成本的 overlap 诊断，弥补了既有解析模型固定 overlap 假设的信息损失。
- **Wall ordering 作为输出**：用"哪个资源墙先 bind"（KV 容量 / KV 带宽 / 权重带宽 / 计算）来比较部署方案，而非仅看同 batch 延迟；容量作为一等资源维度而非事后 OOM 检查。
- **H20 ridge-point 首次刻画**：给出 H20 ridge point ≈74 FLOP/byte（vs H100 ≈590），内存受限区比 H100 宽 8×，从几何上解释为何这款出口受限芯片是天然 decode 引擎。
- **容量优先推导 DP-attention 判据及其反转**：在未修改 MLA 的前提下，DP-attention 把 KV 容量墙从 70 抬到 644（≈9×），但单流场景因 all-reduce/all-to-all 延迟主导而反转，TP16 领先 2.4× —— 量化了"MLA serving 该用 DP attention"的工程经验，并解释了官方 DeepSeek 与 LMSYS–Ant 在同款 16×H20 上相反选择的根因。
- **零依赖 artifact + agent skill**：纯 Python 的 floor.py / mbu_mfu.py / specs.py，外加 SKILL.md，让 floor-first 纪律在 agentic optimization loop 中可强制执行（agent 必须先出 floor 账本再允许 benchmark）。

## Key Contributions

- **Residual-driven triage workflow** that separates target definition, floor construction, benchmark reconciliation, and profiler escalation with explicit stopping rules (1.3× threshold), dispatching verdicts to either the pipeline-optimization arm or the kernel-writing arm — Profiling is treated as exception handling, not a pipeline stage.
- **Two-sided floor model [max, sum]**: same-resource terms add, independent engines (HBM/SMs/NIC) take the max for an optimistic floor topt; the plain sum gives a pessimistic floor tsum. The measurement's position inside the interval is a zero-cost overlap diagnostic that existing analytical models discard by fixing an overlap assumption up front.
- **Wall ordering as a first-class output**: deployment alternatives are compared by which wall (KV capacity, KV bandwidth, weight bandwidth, compute) binds first as load grows, rather than by same-batch point benchmarks. KV capacity is treated as a fifth resource dimension, not a post-hoc OOM check.
- **First published ridge-point characterization of the export-grade H20 GPU**: ridge ≈74 FLOP/byte vs ≈590 for H100, an 8× wider memory-bound zone, geometrically explaining why H20 is a natural decode engine and weak for prefill.
- **Capacity-first derivation of the DP-attention judgment and its single-stream inversion**: for unmodified MLA, EP16+DP-attention lifts the KV capacity wall from B_max≈70 to ≈644 (≈9×) at 8K context, but at B=1 TP16 wins by 2.4× (TPOT floor 4.9 ms vs 11.8 ms) because per-op latency dominates — quantifying the folklore "MLA serving should go DP" and explaining why official DeepSeek and the LMSYS–Ant H20 deployment shipped opposite attention layouts.
- **Zero-dependency artifact plus an agent-readable SKILL.md** that makes the floor-first discipline enforceable in agentic optimization loops: a triggered agent must produce the floor account before any benchmark, and profiler use must cite a residual above threshold.

## Methodology

Floor First models each autoregressive decode step as a five-dimensional resource vector **r = (HBM bytes, FLOPs, network bytes, network messages, KV capacity)**. The first four convert to time via calibrated hardware rates; the fifth bounds feasible batch size and enters the goodput judgment.

**Two-sided floor.** Terms contending for the same hardware resource add (weight reads + KV reads share HBM; bandwidth cost + per-message latency share the NIC), while independent engines run concurrently under perfect overlap, so only the slowest shows: `t_opt = max(t_HBM, t_compute, t_network)` and `t_sum = t_HBM + t_compute + t_network`. The interval [t_opt, t_sum] is kept rather than collapsed: a measurement near t_opt implies good overlap (next gain must reduce the dominant account), near t_sum implies serialized engines (scheduling/overlap suspects), and above t_sum implies time leaking outside the model (host gaps, preemption, stragglers). A structural caveat: at B=1 the layer-serial dependency chain leaves little to overlap, so the honest single-stream floor is t_sum.

**MoE union correction.** For MoE, FLOPs follow activated parameters but weight bytes follow the union of experts activated across the batch (a weight tile is read once per step regardless of how many tokens use it). Under uniform routing with E routed experts and top-k, the expected distinct-expert fraction is `f_expert(B) = 1 − (1 − k/E)^B`, bounded by `min(1, kB/E)`. At B=64, E=256, k=8 this touches ≈87% of experts, tightening weight terms by ≈13%.

**MLA replication.** For MLA-style latent attention, the latent KV is shared by all heads, so head-parallel TP cannot shard it — every TP rank stores and reads the full cache. This replication cost makes DP attention the structurally favored high-concurrency layout; the paper quantifies by how much.

**Communication.** TP decode does two all-reduces per layer (ring all-reduce traffic `2(n−1)/n · B·H·b_act` bytes plus a latency term). EP+DP attention replaces all-reduces with dispatch/combine all-to-alls per MoE layer. The paper separates two questions the literature usually merges: which route has the better ceiling, and does a near-ceiling implementation of that route exist (ceiling-versus-implementation distinction).

**Capacity wall.** `B_max = (C_HBM − C_weights − C_overhead) · s_KV / (S · bytes_KV/token)`, where the KV sharding factor s_KV is 1 for MLA under head-parallel TP (full replication), min(n, n_kv) for GQA under TP, and n for DP attention. Because decode goodput is B/t_step under a TPOT SLO, a layout with slightly worse same-batch t_step but an order-of-magnitude larger B_max can dominate the goodput frontier.

**Compositionality.** Each layer type (attention, routed FFN, dense FFN, future state-space block) declares five things: weight bytes, state bytes, per-step state reads, FLOPs, communication. A model is a list of modules; a step account is the per-resource sum. New architectures enter by declaring one module, not by rewriting the framework.

**Roofline identity for metric discipline.** `MFU/MBU = I_work / I_ridge` holds identically, so the two utilizations are not independent — report the one whose ceiling is 100% for the binding resource (MBU for bandwidth-bound decode, MFU for compute-bound prefill).

**Artifact.** Pure-Python, zero runtime dependencies: `specs.py` (model/GPU/cluster constants), `floor.py` (resource table, [max,sum] floors, capacity wall, single-stream bound, prefill floor), `mbu_mfu.py` (measured TPOT/TTFT → MBU/MFU + triage verdict), and `SKILL.md` (agent-readable workflow). Calibration chain: HBM via nvbandwidth, GEMM rates via production serving shapes, communication via nccl-tests and the production all-to-all benchmark.

**Workflow loop (Figure 1).** (0) Define SLO (P99 TTFT/TPOT, goodput); (1) compute floors and capacity walls, discard configurations whose walls exclude the target; (2) benchmark survivors under open-loop, steady-state, realistic-length, tail-percentile discipline; (3) reconcile — if measurement is within 1.3× of floor, stop; (4) else open Nsight Systems and answer exactly three questions (gaps between kernels? exposed communication? kernel class over budget?); (5) open Nsight Compute only for the indicted kernel class.

## Results & Highlights

- **TP16 decode floor at B=64, S=8192 on 16×H20**: HBM account 19.70 ms dominates compute (2.99 ms) and network (8.91 ms), giving a floor of **[19.7, 31.6] ms** and a KV-capacity wall of **B_max ≈ 70** concurrent 8K requests (Table 2).
- **TP16 + DSA-style sparse attention**: KV term drops from 9.21 ms to 2.30 ms, lowering the floor to **[12.8, 23.2] ms**, but the capacity wall stays at **B_max ≈ 70** — DSA removes the KV-bandwidth wall but not the capacity wall, because top-k selection reduces reads, not residency (Table 2, Table 3).
- **EP16 + DP attention**: per-GPU weight traffic rises to 14.70 ms (worse than TP16's 10.48 ms at equal batch) but KV reads shrink to 0.58 ms and the capacity wall moves from 70 to **B_max ≈ 644** — an order-of-magnitude higher concurrency (Table 2).
- **Single-stream inversion at B=1**: TP16 TPOT floor ≥ 4.9 ms (≲ 205 tok/s) vs EP16+DP-attention TPOT floor ≥ 11.8 ms (≲ 85 tok/s), so **TP16 wins single-stream by 2.4×** — the mirror image of the high-concurrency judgment (§5.6).
- **H20 ridge point ≈ 74 FLOP/byte vs ≈ 590 for H100** (FP8 dense 296 T vs 1979 T; HBM BW 4.00 TB/s vs 3.35 TB/s), an **8× wider memory-bound zone**; decode at B=64 sits at I_work ≈ 11 FLOP/byte, well below the H20 ridge, so it is governed by memory traffic rather than tensor peak (Table 1, Figure 2).
- **Calibration findings**: effective all-reduce rate at decode message sizes is **43 GB/s against a 100 GB/s per-node line rate**; small-message latency ≈ 33 µs equal within noise across and within nodes (identifying NCCL launch overhead, not wire time); DeepEP op-level dispatch latency **60 µs vs a ~30 µs pure-transport bound**; raw-NCCL all-to-all measured 12.6 GB/s where DeepEP approaches line rate. Calibration moved one communication constant by 8×, yet **no optimistic floor and no deployment judgment changed** — a property of wall ordering, since the HBM account and capacity wall carry the verdicts (§4).
- **Fully calibrated floors are 15–24% looser than datasheet ones**; GPU-side rates landed at 81% of datasheet HBM bandwidth (under a torch-level fallback tool) and 89% of FP8 peak (§4).
- **Reconciliation example (hypothetical)**: a measured 25 ms P50 TPOT for TP16 yields MBU = 78.8%, residual 1.27× the optimistic floor (below the 1.3× escalation threshold), interval position 0.45 — the workflow says stop, no profiler session warranted (§5.7). A 45 ms measurement (MBU 44%, 1.42× the pessimistic floor) would sit outside the interval entirely, escalating to Nsight Systems.
- **Compute wall is unreachable in the feasible region**: the MoE-corrected GEMM-only knee is B* ≈ 670; including 8K MLA attention FLOPs lowers it to ≈225 (≈450 with DSA), but the TP16 capacity wall caps B at 70 first — "chasing decode MFU here is optimizing a wall the deployment cannot reach" (§5.3, Table 3).

## Critical Assessment

- **Novelty**: Genuine. The two-sided [max,sum] floor that keeps overlap observable (rather than fixing an overlap assumption) and wall ordering as the primary comparison axis are real conceptual contributions distinct from prior analytical models (LIMINAL fixes overlap; Pope et al. target dense TPU partitioning; AIC++ trades closed forms for measured kernel databases). The H20 ridge-point portrait and the capacity-first DP-attention derivation for unmodified MLA appear to be new.
- **Rigor**: Mixed and self-acknowledged. The analytical model is internally consistent and the artifact is reproducible (Appendix A gives exact commands). However, the central empirical claim — that the 1.3× escalation threshold and the MBU bands (70%/40%) correctly triage — is **not yet validated**: §8 lists the residual sweep as "planned, not yet performed." The reconciliation example in §5.7 uses a **hypothetical 25 ms measurement**, not a real one. The goodput-frontier comparison of TP16 vs DP-attention on the actual hardware (the payoff of the whole case study) is §8 item 5, also not done.
- **Reproducibility**: Strong on the analytical side (zero-dependency Python, public model dimensions, cluster-calibrated constants documented, reproduction commands given) and weak on the empirical side (calibration data and measurement scripts "will be released with the next revision"; the residual sweep does not exist yet).
- **Significance**: High if the planned validation holds. The explanation of why official DeepSeek (DP attention) and LMSYS–Ant (TP16 attention) shipped opposite layouts on identical 16×H20 hardware — as different operating points on the same frontier — is a concrete, falsifiable, and practically important claim. The wall-ordering framing ("a judgment carried by the first wall survives errors in the accounts of walls never reached") is a useful robustness argument backed by the calibration evidence that an 8× change in one communication constant left all judgments unchanged.
- **Red flags**: (1) The paper is explicitly a draft whose headline empirical validation is not yet performed — readers must not treat the case-study numbers as measured system performance, only as analytical floors. (2) The triage thresholds (1.3×, 70%/40% MBU bands) systematize practitioner experience from Databricks and the author's own deployments but are not derived from first principles; the calibration protocol to validate them is itself part of the unfinished §8 campaign. (3) The MoE union correction assumes **uniform routing** — a known blind spot the paper flags (§7(ii)), since hot experts push per-GPU weight traffic above the account and the sum floor loses meaning when stragglers, not resources, set step time. (4) Single-author; no co-authors or external reviewers are credited. (5) The artifact currently hard-codes the two case-study layouts; the per-module plan interface that would generalize it is "staged as the next revision" (§7), so the compositional generality is partly aspirational.

## Strengths

- Conceptually clean two-sided floor that turns overlap from an assumption into an observable, with a concrete reading of interval position (near max → reduce dominant account; near sum → fix overlap; above sum → look outside the model).
- Wall ordering reframes deployment comparison from point benchmarks to "which resource binds first," yielding the robustness property that first-wall verdicts survive errors in later-wall accounts.
- The H20 ridge-point portrait is a genuinely useful single-number reframing (74 vs 590 FLOP/byte) that explains the export part's decode-friendliness from geometry rather than folklore.
- Honest, detailed calibration section that distinguishes datasheet floors from calibrated floors and explicitly separates the ceiling-versus-implementation question for communication.
- Compositional module-declaration design that makes the framework architecture-agnostic in principle (DSA, future state-space layers, MLA-sharding redesigns all enter by changing one module's declaration).
- Intellectual honesty about draft status, blind spots, and the epistemic status of thresholds — rare and commendable.
- Ships as both a human checklist and an enforceable agent skill, addressing a real failure mode of LLM coding agents that reflexively benchmark and profile.

## Limitations

- **Headline empirical validation is not yet performed**: the residual sweep (§8 item 2), triage traces (item 3), cross-platform replication (item 4), and judgment validation (item 5) are all planned. The case-study numbers are analytical floors, not measured end-to-end results.
- **Triage thresholds are not derived**: 1.3× escalation and 70%/40% MBU bands are systematized practitioner experience, with the calibration protocol to validate them listed as the first required experiment.
- **Uniform-routing assumption** for the MoE union correction breaks under expert imbalance, which is common in production; the paper flags this but does not bound the error.
- **Continuous batching not modeled**: B varies within a window, so floors should bracket the batch distribution rather than assume a stationary point — a known blind spot.
- **Single-stream floors bound feasibility, not prediction**: the paper cites evidence that batch-1 decode on high-bandwidth parts reaches only ~27% of its analytic floor due to kernel-launch overheads, so the 2.4× single-stream TP advantage is a physical bound, not an achievable expectation.
- **Artifact scope**: currently hard-codes pure TP and EP+DP-attention decode plus compute-bound prefill; mixed TP×EP, prefill/decode disaggregation, chunked-prefill interference, speculative decoding, and attention–FFN disaggregation are future extensions on the (not-yet-implemented) per-module plan interface.
- **Single author, draft status**: no external validation cycle; calibration data and measurement scripts promised for the next revision.
- **Host-bound regimes outside the model**: Python and scheduler overheads are outside the resource vector by construction; the < 40% MBU band plus kernel-gap inspection is the only catch, which is indirect.

## Relevance

这篇论文与 LLM 推理系统性能优化、roofline 分析以及 MoE/MLA 部署调优高度相关，其"先建解析 floor、再用残差决定是否升级到 profiler"的分诊思路可直接指导大模型服务系统的容量规划与并行策略选型，尤其适合在稀缺 GPU 资源下避免无效 grid search 的工程场景。

## Questions for Further Exploration

- When the planned residual sweep (§8 item 2) is performed across {TP16, TP16+DSA, EP16+DPA} × B × S, does the 1.3× escalation threshold hold, or does it need to be layout- and batch-dependent? How does the interval-position distribution actually look?
- The paper claims the 33 µs per-op latency is NCCL launch overhead rather than wire time, and that CUDA-graphed production paths reach ~10 µs. How much of the single-stream TP advantage survives once production graph paths close the launch-overhead gap?
- Under realistic non-uniform expert routing (hot experts), how much does the union correction underestimate per-GPU weight traffic, and does EPLB-style balancing fully restore the floor, or do stragglers invalidate the sum floor before balancing helps?
- Can the wall-ordering robustness argument (first-wall verdicts survive errors in later-wall accounts) be quantified as a sensitivity bound — i.e., how wrong can a later-wall constant be before it promotes to the first wall and changes the verdict?
- The case study fixes MLA as unshardable across heads; how do the recently proposed TP-enabling latent attention variants (GQLA, TPLA) shift the capacity wall and the TP-vs-DP crossover point, and do they change the operating-point judgment or merely move it?
- Does the floor-first discipline, encoded as an agent skill, measurably reduce wasted benchmark/profiler invocations in an autonomous LLM-serving optimization loop compared to a baseline agent without the floor gate?
