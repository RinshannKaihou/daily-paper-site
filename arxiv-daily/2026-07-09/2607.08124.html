Critical Assessment · Limitations

Methodology

Adaptation state is arbitrary executable code. The model M is frozen; the harness H is a Python class orchestrating M, tools, and an execution environment. Gold labels and hidden tests are withheld during adaptation and enter only post-selection, for measurement.

From each trace, TTHE derives lightweight label-free proxies — execution health (runs clean, well-formed), round-trip consistency (output inverted and re-scored against the input), and public-test pass rate. These are exposed as raw traces to the proposer and judge, never collapsed into a single scalar, because cross-sample agreement is not correctness and any one proxy is gameable.

The loop processes the unlabeled stream in batches: each batch evolves G parallel lineages over R rounds via three operators. Solver, proposers, and judge are different roles around the same frozen backbone — no stronger teacher, no separately trained adaptation model. The judge commits one harness without ever seeing gold, so reported accuracy is transductive.

OBSERVE
Run every harness on every input; collect traces, cached outputs, proxy signals.
PROPOSE
G parallel proposers edit their parent branch; roles steered for diversity; invalid → fallback.
JUDGE
Agentic selector inspects code/traces/signals, may re-execute, commits one harness — no gold.

Strengths

  • Clean, strict formulation. Test-time adaptation as evolution over executable control programs; gold never enters proposer / judge / traces.
  • Breadth across five execution-grounded domains with large absolute gains on every slice.
  • Exemplary honesty. Quantifies its own failure modes (selection regret, coverage ceiling) against gold-only oracles; states the transductive limit plainly.
  • Inspectable artifacts. Discovered grounding / verify / repair policies are readable source, not opaque weights.
  • Cross-model consistency (DeepSeek / MiMo / Kimi all beat their own baselines).

Limitations

  • Tiny adversarial hard slices (30–60 each); BIRD +38 is inflated by slice construction on baseline failures — not a model ranking.
  • Transductive only. No prequential or shuffled-batch evaluation; forward generalization unproven, gains may be batch-local.
  • Single-run ablations, no multi-seed variance; the "best" G=3 / R=3 could be partly noise.
  • No compute-matched baselines (best-of-N execution selection, dev-time optimizers like DSPy / ADAS / AFlow).
  • Sandboxed execution. Open-world / safety-critical use would need guardrails on what the proposer may write.

Open questions

Generalization

Does the committed harness survive prequential scoring (Ht on batch t+1) and a shuffled-batch rerun?

Judge calibration

Would a learned verifier or process-reward model close the selection-regret gap (3/10 chosen over 4/10)?

Baselines

How does TTHE compare to a compute-matched best-of-N selection and dev-time optimizers on the same frozen backbone?

Coverage ceiling

Candidates plateau at 70% from one shared backbone — would a diverse multi-backbone proposer break through?

Stability

Can a regression guard (never commit a candidate that regresses on cached prior-batch outputs) tame the non-monotonic G=1 degradation?

Slice validity

Do the gains hold on the full standard leaderboard splits, not just adversarially-filtered hard subsets?
