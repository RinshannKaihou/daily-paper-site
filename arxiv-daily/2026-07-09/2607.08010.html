§ 01

Methodology — two modes

Baseline · Eluna + CodeAct

A main agent walks a 44-node decision tree (44 decision / 19 action nodes) and delegates each node to a CodeAct sub-agent that writes and executes MCP metric queries in a loop. Per alarm this loop can exceed 100 LLM turns — repeatedly recompiling the same SOP text against an unchanging interface. That repetition is the dominant per-alarm cost.

Offline tool-making pipeline

Each SOP node compiles to one deterministic tool with a uniform signature (warehouse, timestamp, context) → verdict, grounded by three components:

  1. 01Data-collector sub-agent runs on 3 balanced cases/node, emitting a trace of query code, MCP responses, observed schema & graded verdict.
  2. 02Tool-maker LLM writes a candidate conditioned on SOP text, node position & the traces.
  3. 03Reflector test-repair loop (budget 3) rewrites on disagreement; highest pass-rate candidate is deployed.
Training set 8,011 train / 8,022 eval, 100–200 cases/node Inference: direct call or sub-agent, CodeAct fallback on exception Student robustness: teacher distill w/ 25% tool-dropout → action acc 97.4% even tools-off Tools are model-agnostic, regenerated independently of LoRA fine-tuning

§ 03

Strengths & limitations

✓ Strengths
  • Real production scale — 5,000+ alarms, no rollback; not a benchmark-only study.
  • Decisive ablations proving both the data-collection trace and the test-repair loop are necessary; repair compute cannot substitute for environmental grounding.
  • Honest about confounds — the architecture-vs-model confound in Table 3 is explicitly isolated via a controlled offline ablation.
  • Label-free variant (91.6%, within 3pp of labeled ceiling) lowers the bar to adding new SOP nodes.
  • Operational payoffs beyond latency: auditability, drift detection, one 800K-token tool library shared across both deployed students.
! Limitations
  • Single-domain deployment — outbound-dock alarm triage; free-form runbooks & non-decision-tree workflows not covered.
  • Human oversight still required — drift detection and candidate generation are automated, but not every failure is guaranteed caught.
  • Clarifying underspecified SOPs is inherently human; new tools still benefit from labeled cases.
  • Absolute latencies withheld; all comparisons normalized to an undisclosed baseline.
  • Proprietary data & backends prevent external replication; no code/data release.
§ 04

Open questions

  • 1Does the offline tool-making pipeline generalize to free-form runbooks where node boundaries and labeled cases are less well-defined?
  • 2How does per-node pass@1 degrade as upstream schemas drift between generation and deployment, and what is the monitoring agent's detection latency?
  • 3Could the label-free variant close the remaining ~3pp gap with more candidates or stronger judges, removing SME labels entirely?
  • 4What is the cross-site failure rate when numeric thresholds vary widely (greenfield vs mature FCs)?
  • 5Can the direct-call architecture preserve fallback robustness under tool-availability dropout beyond the 25% used in fine-tuning, or does it collapse past some threshold?
