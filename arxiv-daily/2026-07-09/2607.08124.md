> **一句话总结：** TTHE 提出在评测阶段、仅利用无标签执行轨迹对 LLM 智能体的可执行 harness（控制程序）进行种群式演化优化——不更新模型权重、不使用 gold 标签——在 BIRD、LiveCodeBench、SWE-bench Verified、DS-1000 及 claw-eval 五个执行型任务上显著超越固定的 ReAct 基线 harness（如 BIRD 12.0%→50.0%、SWE-bench 20.0%→35.0%）。

> **TL;DR:** TTHE reframes test-time adaptation of LLM agents as evolution over their executable harness program: during evaluation it maintains a population of candidate harnesses, an agentic proposer rewrites them from unlabeled execution traces, and a judge commits one via execution-derived proxy signals—without weight updates or gold labels. It lifts a fixed ReAct baseline from 12.0% to 50.0% on a hard BIRD text-to-SQL slice, 20.0% to 35.0% on SWE-bench Verified, 30.0% to 38.3% on LiveCodeBench, and 48.9% to 69.8% mean score on the claw-eval agentic tool-use benchmark.

## 核心贡献

- **提出"测试时 harness 演化"新范式：** 把 LLM 智能体的可执行控制程序（harness）作为测试时自适应的状态对象，在评测过程中、仅凭无标签执行轨迹进行优化——模型权重冻结、gold 标签隔离到选择之后才用于度量。
- **种群式 generate-and-judge 循环：** solver、proposer、judge 都是围绕同一个冻结 LLM 的不同角色与 harness，通过 G 条并行谱系、R 轮迭代演化候选 harness，judge 在无标签的执行健康度、往返一致性、公开测试通过率等代理信号上选定一个并持久化。
- **跨五个执行型领域的实证：** 在 text-to-SQL（BIRD）、竞争编程（LiveCodeBench）、软件工程（SWE-bench Verified）、数据科学代码（DS-1000）、智能体工具使用（claw-eval）上全面超越固定 ReAct 基线 harness，并暴露可检视的 grounding/验证/修复策略源码。
- **深入的诊断与失败模式剖析：** 给出 search-budget、batch-size、cross-model、cross-batch accumulation 等消融，并通过 gold 仅用于诊断的 oracle 定位两大瓶颈——候选覆盖率（coverage gap）与选择遗憾（selection regret），指出 judge 在不完美代理信号下的校准问题是核心技术难题。

## Key Contributions

- Formulates **test-time harness evolution**: adapting a persistent executable harness during evaluation, with model weights fixed and gold labels isolated to post-selection measurement, realized as a population-based generate-and-judge loop in which the solver, proposers, and judge are different roles/harnesses around the same frozen LLM.
- Demonstrates gains across **five execution-grounded domains** (BIRD, LiveCodeBench, SWE-bench Verified, DS-1000, claw-eval) while exposing the resulting harness code, traces, and per-batch decisions for behavioral inspection.
- Complements results with **diagnostics** (search-budget, batch-size, cross-model ablations and per-batch trace audits) that characterize the setting's failure modes, including limited candidate coverage and selection regret.
- Recasts test-time adaptation for LLM agents as **evolution over executable control programs** and identifies execution-derived proxy reliability as a central challenge for robust unsupervised agent improvement.

## Methodology

TTHE optimizes the agent's harness during evaluation using only the evidence that execution itself produces. The two organizing questions are: (Q1) with gold labels, reference outputs, and hidden tests withheld during adaptation, what provides the optimization signal? and (Q2) given only that imperfect signal, how is the harness actually improved?

**Setup.** Let M be a model with fixed parameters and H a harness program orchestrating calls to M, tools, and an execution environment E. For input x, H(x) produces an output-trace pair (o, τ). The measurement objective J⋆(H) is expected accuracy against gold y, but gold enters only through measurement and is never available during adaptation.

**Adaptation state = executable code.** The harness is a Python class, so the adaptation state is arbitrary program code (strictly containing prompt-only and fixed-workflow scaffolds). A proposer may express any strategy traces motivate: grounding generation in retrieved context, executing candidate outputs and reading results back, decomposing tasks, sampling/aggregating multiple solutions, adding deterministic contract checks, or repairing outputs from execution feedback. The baseline H0 is a standard ReAct scaffold.

**Optimization signal (no trusted scalar reward).** From each trace TTHE derives lightweight, task-dependent proxies:
- *Execution health* s∈{0,1}: output runs without error and is well-formed/shape-consistent.
- *Round-trip consistency* b∈[0,1]: output is inverted into a description and scored against the input to flag mis-grounding.
- *Public-test pass rate* p∈[0,1]: fraction of public examples/repository tests passed (direct but incomplete).
Crucially these signals are exposed together with raw traces to the proposer and judge rather than collapsed into a single scalar, since cross-sample agreement is not correctness and any single proxy is gameable.

**The loop (Algorithm 1).** TTHE processes the unlabeled stream in batches. For batch Xt it starts from the committed harness Ht, evolves G parallel branches over R rounds using three label-free operators:
- `OBSERVE(H, Xt)`: executes every harness on every input, returns traces, cached outputs, and proxy signals.
- `PROPOSE({Hg}, T)`: G proposers run in parallel; proposer g edits its own parent branch into a child while free to read other branches' code/traces; an invalid child falls back to the parent. Branches are steered toward different roles (conservative-repair, exploratory, adversarial) for diversity.
- `JUDGE(F, T, Xt)`: an agentic selector over the final branch set that inspects code/traces/signals, may re-execute candidates, and commits one harness—never seeing gold.

Two properties make this a population search rather than single-candidate reflection: *Lineage* (each branch carries its parent forward, so useful routines compound across rounds) and *Diversity* (branches explore different directions and may borrow ideas from one another's traces). The committed harness is scored on Xt from cached outputs without re-execution; because it is both selected on and measured on Xt, reported accuracy is *transductive*.

**Adaptation boundary.** Solver, proposers, and judge all use the same frozen backbone (deepseek-v4-flash in main runs), differing only in harness; the proposers/judge run inside a Claude Code agent scaffold used purely as a coding interface routing every call to the same frozen backbone—so no stronger teacher and no separately trained adaptation model.

## Results & Highlights

TTHE improved fixed ReAct-style baselines across all evaluated domains (deepseek-v4-flash as solver/proposer):

- **BIRD (hard text-to-SQL slice):** accuracy rose from **12.0% → 50.0%** (+38 points). The slice is constructed from questions the baseline systematically fails, so the gain measures recovery on those hard queries; under MiMo V2.5 it improved 32.0%→52.0% and under Kimi K2.5 28.0%→48.0%, showing the loop is not tied to one backbone.
- **SWE-bench Verified (40-instance hard slice):** **20.0% → 35.0%** (+15 points); the evolved harness added a reproduce-first, locate-root-cause, fix, re-verify, run-existing-tests workflow plus a post-rollout repair branch that re-runs the agent when the first rollout returns an empty patch.
- **LiveCodeBench (60-problem contamination-controlled hard slice):** **30.0% → 38.3%** (+8.3 points); the discovered policy is a conditional verify-and-repair that uses the low-cost path by default and escalates only after a public-test failure.
- **DS-1000 (50-problem data-science hard slice):** **38.0% → 44.0%** (+6 points).
- **claw-eval (30-task agentic tool-use slice, graded rubric in [0,1] ×100):** mean task score **48.9% → 69.8%** (+20.9 points, improved on 28 of 30 tasks).

**Search-budget ablation (Table 1, BIRD):** more rounds help only at G=3; the best setting is **G=3, R=3 at 50.0%**, whereas G=1/R=3 falls to 40.0%—parallel lineages make a useful behavior likelier to survive to selection.

**Batch-size ablation (Figure 3):** accuracy is non-monotonic and peaks at **B=10 (50.0%)**; both smaller (B=5: 44.0%) and larger (B=25: 38.0%, B=50: 44.0%) batches do worse, though all remain far above the 12.0% baseline.

**Cross-batch accumulation (Table 3):** carrying the committed harness across batches reaches 50.0% versus 44.0% for resetting to baseline each batch—about a six-point warm-start edge concentrated in middle batches.

**Where gains are lost (post-hoc gold-diagnostic oracles on BIRD):** a judge-restricted oracle reaches 64.0% vs the judge's 50.0% (seven tasks solvable by a seen-but-uncommitted candidate = selection regret), and an oracle over all candidates ever generated still reaches only 70.0%, so roughly fifteen tasks were never produced by any candidate (coverage gap). The judge is measurably miscalibrated—committing a 3/10 branch over an available 4/10 in one batch and a 6/10 over a 7/10 in another—while claiming near-perfect correctness.

## Critical Assessment

- **Novelty:** Genuinely novel framing. Unlike pre-deployment prompt/workflow optimizers (DSPy, ADAS, AFlow) or per-query self-refinement (Reflexion, self-debugging), TTHE moves harness optimization *inside* evaluation under hidden gold, and the adaptation state is arbitrary executable code rather than weights, prompts, or a bounded config space. The nearest concurrent system, Meta-Harness (Lee et al., 2026), uses an agentic proposer but optimizes with an explicit development objective; TTHE's contribution is the label-free, proxy-driven, transductive setting.
- **Rigor:** The honesty is a strength—selection regret and coverage are measured against gold-only oracles, and search-budget curves are shown to be non-monotonic. However, **evaluation slices are tiny and adversarially constructed**: 50/60/40/50/30 hard instances, with the BIRD slice *conditioned on baseline failures*, which inflates the +38-point delta and makes the absolute 12.0%→50.0% not a model ranking. A defender could argue the gain reflects recovery on cherry-picked hard cases rather than broad capability.
- **Missing ablations / baselines:** No compute-matched comparison to best-of-N execution selection or development-time optimizers (acknowledged in limitations). No multi-seed runs—each ablation entry is a single completed run, so variance is unreported and the G=3/R=3 "best" setting could be partly noise. The missing **prequential** evaluation (scoring H_t on batch t+1) means forward generalization is not established; because each batch is both adapted to and scored on the same inputs, the transductive gains partly reflect batch-local specialization. A shuffled-batch rerun is also absent.
- **Reproducibility:** Code is public; the harness source for discovered policies is in the appendix. But the loop depends on a Claude Code agent scaffold and frozen deepseek-v4-flash/MiMo/Kimi backbones whose versions may drift, and the slices are custom hard subsets not part of the standard leaderboard—exact reproduction will require care.
- **Significance:** If the prequential generalization holds up in future work, this is a meaningful new axis for unsupervised agent improvement. The central, well-argued takeaway—that proxy reliability and agentic-judge calibration are the binding bottlenecks, not search budget—is the paper's most durable contribution.
- **Red flags:** The judge commits a 3/10 branch over a 4/10 and a 6/10 over a 7/10 while claiming near-perfect correctness—clear evidence the agentic judge over-trusts proxy signals, which the authors surface rather than hide. The claw-eval slice is difficulty-ordered, making cross-batch accumulation results confounded with ordering (acknowledged).

## Strengths

- Novel and clean formulation: test-time adaptation as evolution over executable control programs, with a strict label-free boundary (gold never enters proposer/judge/traces).
- Strong empirical breadth across five diverse execution-grounded domains, with large absolute gains on every slice.
- Exemplary intellectual honesty: quantifies its own failure modes (selection regret 64.0% pool-oracle vs 50.0% committed; coverage ceiling 70.0%) and states the transductive limitation plainly.
- Inspectable artifacts: the discovered grounding/verification/repair policies are readable source code (appendix Figures 4–5), not opaque weights.
- Cross-model consistency (DeepSeek/MiMo/Kimi all improve over their own baselines) argues the effect is not an artifact of one backbone.

## Limitations

- Tiny, adversarially-constructed hard slices (30–60 instances each); the BIRD +38-point delta is inflated by slice construction on baseline failures, so magnitudes are not a controlled cross-task comparison.
- Transductive scoring only—no prequential or shuffled-batch evaluation, so forward generalization is unproven and part of the gain may be batch-local specialization.
- Single-run ablations with no multi-seed variance; the "best" G=3/R=3 setting could be partly noise.
- No compute-matched baselines (best-of-N execution selection, dev-time optimizers).
- Restricted, sandboxed execution environment; open-world or safety-critical deployment would need additional guardrails on the programs the proposer may write.

## Relevance

本文与测试时自适应（test-time adaptation）和无标签监督下的智能体自我改进高度相关，其"把可执行 harness 作为自适应状态、仅凭执行轨迹演化"的范式为自演化智能体研究提供了新视角，并诚实地暴露了代理信号可靠性这一核心瓶颈，对本研究方向的方法选型与后续工作有直接参考价值。

## Questions for Further Exploration

- Does the committed harness generalize under **prequential scoring** (evaluating H_t on batch t+1 before adaptation), and how much of the reported gain survives a shuffled-batch rerun?
- Would adding a **learned verifier or process-reward model** (or a small information ablation exposing only scores vs. full traces) close the selection-regret gap where the judge commits 3/10 over an available 4/10?
- How does TTHE compare against a **compute-matched best-of-N execution selection** baseline, and against development-time optimizers (DSPy/ADAS/AFlow) given the same frozen backbone?
- The coverage oracle plateaus at 70.0% with correlated candidate variants from a single shared backbone—would a **diverse-population / multi-backbone proposer** escape this coverage ceiling?
- Can the loop be stabilized with a **regression guard** (e.g., never commit a candidate that regresses on cached prior-batch outputs) to prevent the non-monotonic search-budget degradation at G=1?
