Why this matters

First detectable, cross-lingual evidence that code models internally encode formal type information — a foundation for type-constrained decoding and causal interpretability of code LLMs.

Critical edge

A narrow synthetic universe — only 4 base types + list, 2 languages, 2 models, ≤2 candidate functions — is the biggest threat to generalization.

No real code · no generics · no causal test

01 Methodology

A controlled parallel Java/Python corpus isolates type signal from lexical shortcuts — then per-layer linear probes test whether that signal is decodable, transferable, and robust.

Three 90K-example partitions (Java, untyped Python pyUnt, typed Python pyTag) implement the same programs. Identifiers and literals are independently randomized per partition so equivalent examples never share names or values across languages; each masked call site b:τ_b = f_⟨FIM⟩(a) forces a choice between two functions distinguishable only by type compatibility.

Post-residual activations at the FIM-token position are probed with a separate linear probe per (model, layer, task, partition), exhaustively across layers (AdamW, lr 1e-4, 20 epochs, 4-fold CV). The headline metric is selectivity S = acc_task − acc_control, against an architecturally-identical random-label control. Robustness = absolute drop Δ under an adversarial partition where every identifier is renamed to a contradictory type.

03 Strengths

  • Clean three-partition design that cleanly isolates typed-vs-untyped and Java-vs-Python signals.
  • Adversarial renaming is a genuinely useful diagnostic — separates lexical from type-level signal.
  • Function/argument/result decomposition shows where representations succeed and fail, not one collapsed number.
  • Honest limitations section; transparent re-run after a disclosed CodeLlama tokenization bug.
  • "Cross-lingual type manifold" is a falsifiable, follow-up-friendly hypothesis for SAE / activation-patching work.

04 Limitations

  • Only 4 base types + list; no generics, subtyping, user-defined types, or low-level systems.
  • Only two decoder-only models (SantaCoder-1.1B, CodeLlama-7B); no scale sweep or newer code LLMs.
  • Purely synthetic data with ≤2 candidate functions; no real-codebase evaluation.
  • Linear probes can't recover superposed features or establish a causal role in generation.
  • Cross-lingual transfer is task-dependent (strong T1–2, weak T0) — broad claims need qualification.
  • Per-token control task departs from the standard per-type control, complicating cross-paper comparison.

05 Open questions

  1. Do the recovered type directions play a causal role in generation? Activation patching on these call sites would test whether steering a type direction changes the emitted function or inferred type.
  2. Do the findings survive on real-world code (GitHub Java/Python) and richer type systems (generics, subtyping, TypeScript structural types)?
  3. How do newer/larger code LLMs (StarCoder2, DeepSeek-Coder, CodeQwen) compare — is the type manifold scale-dependent or an artifact of the two studied models?
  4. Can Sparse Autoencoders disentangle type directions from superposed lexical features and explain the large adversarial drops (Δ up to 41)?
  5. Why is Task 0 transfer near chance while Tasks 1–2 transfer well — is function identity genuinely language-bound, unlike base-type encoding?
  6. Could the directions be used to audit type-constrained decoding — flagging when a model's internal type signal disagrees with an external type checker?
