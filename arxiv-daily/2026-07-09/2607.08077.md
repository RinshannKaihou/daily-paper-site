> **一句话总结：** 提出 GRAM（gradient-routed auxiliary modules）预训练方法，通过为每个"双用途"能力（病毒学、网络安全、核物理、专有代码等）配置独立模块并按数据标签路由梯度，实现单次训练即可在推理时按需"摘除"模块从而精准移除指定能力；在 50M–5B 参数的 Chinchilla-optimal 缩放实验中，GRAM 在保留能力上接近数据过滤基线、在被移除能力上随规模增大反而更彻底，且训练成本不随能力画像数量增长，5 画像场景下相比数据过滤节省 5× 开销。

> **TL;DR:** GRAM augments each MLP block of a Transformer with small, per-capability auxiliary modules and uses data-label-driven gradient routing so that each dual-use capability (virology, cybersecurity, nuclear physics, Lisp code) is learned into its own module; ablating a module at inference removes that capability while preserving the rest. On Simple Stories and an 800M realistic dual-use setup, GRAM matches data filtering and FT-LoRA on retain/forget compute ratios (retain ~0.95–0.98) while being more robust to malicious finetuning than post-hoc unlearning (MaxEnt), and a Chinchilla-optimal 50M→5B scaling sweep shows the data-filtering gap on removed capabilities widens with scale and GRAM tracks it throughout — at 5× lower amortized training cost than data filtering for the 5-profile setting.

## 核心贡献

- **提出 GRAM 预训练方法**：在 Transformer 的每个 MLP 块中引入一个常驻的"核心"模块加若干小型"辅助"模块（每个辅助模块对应一种双用途能力），通过数据标签决定前向激活与梯度更新（即"梯度路由"），无需学习型 router，也不是 MoE 的 per-token 路由。
- **单次训练得到多种能力画像**：推理时按需激活/摘除模块即可生成不同能力组合，支持 2^(N-1) 种画像，模块化成本与画像数量无关——在 5 画像设置下相比数据过滤节省约 5× 训练 FLOPS。
- **在被移除能力上比 post-hoc unlearning 更鲁棒**：GRAM/FT-LoRA 通过"从未让模型获得该能力"的架构隔离，使恶意 finetuning 难以恢复；MaxEnt 等 unlearning 在 finetuning 后基本失效。
- **随规模改善的有利缩放趋势**：在 Chinchilla-optimal 的 50M–5B 参数缩放分析中，被过滤/被摘除能力的计算比（forget compute ratio）随规模下降（即移除更彻底），保留能力差距保持较小；GRAM 与 FT-LoRA 在各尺度都紧贴数据过滤。
- **部分标注场景下的显著优势**：仅 50% 数据有标签时，GRAM 的 forget compute ratio 为 0.60±0.01，远低于 filtering 与 FT-LoRA 的 0.85±0.01，归因于"吸收效应"——相似无标签样本的更新被吸收进对应模块而非核心参数。
- **更好的人员/能力组合性**：同时激活多个 GRAM 模块时保留性能不退化（即使训练时从未见过全部模块同时激活），而 FT-LoRA 在多个 adapter 相加时普遍退化。

## Key Contributions

- Introduces **Gradient-Routed Auxiliary Modules (GRAM)**, a pre-training method that adds a permanent core MLP plus small per-capability auxiliary MLPs to each Transformer block and uses data-label-driven forward activation / gradient updates (no learned router, not per-token MoE routing) to localize each dual-use capability into its own module.
- Demonstrates that **a single training run yields an exponential number (2^(N-1)) of capability profiles**: any subset of auxiliary modules can be ablated at inference to remove the corresponding capability, with training cost independent of the number of profiles.
- Shows GRAM **matches data filtering and FT-LoRA** on core/retain/forget compute ratios while being **more robust to malicious finetuning than post-hoc unlearning (MaxEnt)**, whose removal is largely reversed by finetuning.
- Provides a **Chinchilla-optimal scaling study from 50M to 5B parameters** establishing two favorable trends: the data-filtering gap on removed capabilities widens with scale (removal gets cleaner), while the gap on retained capabilities stays small; GRAM and FT-LoRA track data filtering throughout.
- Reveals a **partial-labeling advantage**: with only 50% labeled data, GRAM's forget compute ratio (0.60) is far below filtering/FT-LoRA (0.85), attributed to an "absorption effect" that routes similar unlabeled updates into the dedicated module.
- Establishes a clean **compute-ratio metric** (baseline-compute-normalized cross-entropy) enabling cross-dataset, cross-scale comparison, plus an elicited-forget protocol finetuning on 512 sequences to probe whether removal is real or merely suppressed.

## Methodology

GRAM modifies the standard decoder-only Transformer by augmenting every MLP ("core") block with N−1 smaller auxiliary MLP modules, one per auxiliary capability dataset. The model parameters are partitioned into one core set (embeddings, attention, norms, unembedding, and the core MLP) and N−1 disjoint auxiliary sets, each optimized by a separate AdamW optimizer (following Shilov et al., 2025). Crucially, there is no learned router and no per-token routing — selection is data-driven.

Module activation is governed by a binary indicator m∈{0,1}^N with m1=1 always; the GRAM-layer output is hout = Σ mi·Ei(hin). Three hyperparameters control batch composition:

- **Auxiliary Factor p_af(i)**: sampling frequency of auxiliary dataset D_i (oversampling with undersampling of D_1 to stay compute-equal to baseline).
- **Auxiliary Spread p_as∈[0,1]**: on an auxiliary batch, the core params are updated with this probability (lower → more modular isolation, less core contamination).
- **Core Robustness p_cr∈[0,1]**: on a core batch, a randomly sampled auxiliary module is co-activated/updated with this probability, training the core to be robust to the presence/absence of auxiliary modules.

On **core batches** the core MLP is always active; with probability p_cr one random auxiliary module is also activated and all active params update. On **auxiliary batches** for D_i both the core MLP and auxiliary module i are active in the forward pass; in the backward pass module i is always updated while core params update only with probability p_as. Modules not selected for a batch receive no forward activation and no gradient. At inference, capability profile S is served by activating the core plus modules i∈S and ablating the rest.

Evaluation uses a **compute ratio** CR(M,D_i) = L_i^{-1}(ℓ(M,D_i)) / L_i^{-1}(ℓ(M_BL,D_i)) — the fraction of baseline compute needed to reach the same loss (ideal: ≥1 on retain sets, 0 on forget sets). Aggregate metrics are CRC (core), CRR (mean retain), CRF (mean forget). **Elicited Forget** finetunes each model on 512 sequences (~0.5M tokens) from each forget category and re-measures CRF, modeling a limited adversary. The method is evaluated on (1) Simple Stories (synthetic 2M children's stories, 4 of 48 categories as auxiliary, 26M model); (2) a realistic 800M model on FineWeb-Edu + arXiv + The Stack (core) plus virology (Europe PMC), cybersecurity (arXiv), nuclear physics (OSTI + arXiv), and Lisp code (auxiliary); and (3) a 50M–5B Chinchilla-optimal scaling sweep retaining only virology. Baselines are Baseline (all data), Filtering (separate data-filtered models), FT-LoRA (branched low-rank adapters with rehearsal), FT-Full (branched full finetune), and MaxEnt (post-hoc unlearning).

## Results & Highlights

- **5× training-cost reduction vs data filtering**: In the 800M realistic dual-use setting with 5 capability profiles, Filtering requires 5× the baseline training FLOPS and 5× total MLP parameters, whereas GRAM uses 1× training FLOPS, 1×–1.1× active-MLP parameters, and 1.4× total MLP parameters (Figure 4, right table).
- **GRAM matches data filtering on retention/removal**: Aggregated over 5 profiles, retain compute ratio is 0.95 for GRAM vs 0.96 for Filtering (near-identical), core ~0.99–1.01, with auxiliary forget compute ratios near 0.50–0.63 (Figure 4). On Simple Stories, GRAM reaches CRR=0.95 vs Filtering's 0.96, with a forgotten category (A Deadline) at 0.61 (GRAM) vs 0.65 (Filtering).
- **Robustness advantage over unlearning (MaxEnt)**: MaxEnt achieves low forget compute ratio (~0.52) but its removal is largely undone by finetuning — Elicit compute ratio rebounds to ~0.89 — whereas GRAM/FT-LoRA keep elicited-forget low, confirming the removal is structural rather than merely suppressed (Figure 4).
- **Partial-labeling superiority**: With only 50% of training data labeled, GRAM's aggregate forget compute ratio is 0.60±0.01 versus 0.85±0.01 for both Filtering and FT-LoRA — a large, statistically separated gap attributed to the absorption effect (Figure 6).
- **Scaling trend widens the removal gap**: Across 50M/100M/200M/400M/800M/2B/5B Chinchilla-optimal models, the data-filtering forget compute ratio decreases with scale (cleaner removal at larger scale) while retain stays near baseline; GRAM and FT-LoRA track filtering at every scale. (The authors flag one anomalous GRAM retain point at 5B, attributed to a scale-dependent hyperparameter interaction and examined in Appendix H.)
- **Composability**: Activating all GRAM modules simultaneously (a configuration never seen in training) yields high retained-capability compute ratios on all auxiliary categories, whereas FT-LoRA's compute ratios degrade when multiple LoRA adapters are summed (Figure 5).
- **Scales to many categories**: Increasing auxiliary categories from 4 to 20 on Simple Stories keeps retain near baseline and core ~0.88, with forget/elicited-forget staying well below retain — clear separation persists across the range (Figure 3).

## Critical Assessment

- **Novelty — moderate, honestly scoped.** GRAM is a thoughtful recombination of known ingredients (DEMix-style domain modules, gradient routing, per-partition optimizers) rather than a fundamentally new mechanism. The authors are candid about this lineage in both the intro and Related Work. The genuine contribution is the *systematic application to access control with favorable scaling evidence* and the partial-labeling finding, not a new primitive.
- **Rigor — strong on metrics, with honest caveats.** The compute-ratio metric is a principled way to normalize across datasets/scales, and the elicited-forget protocol (finetuning on 512 sequences) is a reasonable limited-adversary test. Error bars (90% CIs over 3 seeds) are reported for sub-2B runs, but **only a single run was done at 2B and 5B** — exactly the regime where the headline scaling claim lives. The 5B GRAM retain anomaly is disclosed but leaves the largest-scale point under-resolved.
- **Reproducibility — mixed.** Hyperparameters, datasets, and architecture are described in reasonable detail with appendices, but there is no mention of code or model release; the realistic 800M run uses custom data mixtures (Europe PMC, OSTI, specific arXiv categories) that would take effort to reconstruct. Three seeds for small models help; the single 5B seed hurts.
- **Significance — potentially high if trends hold.** The claim that *data filtering itself scales favorably* (removal gap widens with scale) is arguably more important than GRAM itself, and is underemphasized relative to the method. If the 5B trends extend to frontier scale, this directly informs how providers could ship differentiated-capability models.
- **Red flags / overclaiming to watch.** (1) The evaluation is entirely **cross-entropy loss / compute ratio**, not downstream task benchmarks — the authors flag this but it means "virology capability removed" is operationalized as higher perplexity on virology text, not inability to answer virology questions or synthesize pathogens. (2) MaxEnt is a relatively **weak unlearning baseline**; stronger recent unlearning methods (e.g., representation-engineering or NPO-style) are not compared, so "GRAM beats unlearning" is narrower than it sounds. (3) The "5× reduction" is specific to the 5-profile setting and amortized over profiles; a deployment needing few profiles sees little benefit. (4) Capability entanglement (the hardest case — overlapping target/retained capabilities) is acknowledged as a limitation but never stress-tested empirically. (5) No interaction with instruction tuning / RLHF is tested, yet real deployments would apply post-training that could erode the separation.

## Strengths

- Clean, well-motivated problem framing tying AI safety (dual-use capability control) to a concrete, deployable mechanism, with the access-control / least-privilege framing making the practical use case vivid.
- A genuinely informative **scaling study** (50M→5B, Chinchilla-optimal) — rare in the unlearning/access-control literature, which usually fixes one model size — and the favorable data-filtering scaling trend is a valuable finding in its own right.
- Fair, compute-matched comparison against a comprehensive set of baselines (Filtering, FT-LoRA, FT-Full, MaxEnt, plus DEMix in appendix), holding active parameters and FLOPS constant.
- The **partial-labeling result** is a surprising and practically important win, with a plausible mechanistic explanation (absorption effect) and supporting per-token analysis in the appendix.
- Intellectual honesty: anomalies (5B retain point), limitations (six explicitly enumerated), and the lineage of ideas are openly discussed rather than hidden.

## Limitations

- All results use **cross-entropy / compute ratio** as the capability proxy; no downstream behavioral evaluations (e.g., actual virology QA, code execution, cyber-exploit generation) confirm that higher loss translates to unusable capability.
- **Single run at 2B and 5B parameters** — the scales that carry the central scaling claim — so the largest points lack seed-based error bars.
- The strongest unlearning comparison is **MaxEnt**; more recent / stronger unlearning methods are absent, making the "more robust than unlearning" claim less definitive.
- **Capability entanglement** is acknowledged but not empirically stress-tested; closely overlapping retain/forget capabilities (the realistic hard case for any shaping method) could defeat GRAM.
- **No post-training interaction** tested: instruction tuning and RLHF, standard in real deployments, could reinforce or erode the pre-training-time separation — explicitly listed as unknown.
- Productionization is non-trivial: per-module routing, separate optimizers, and module management add implementation complexity frontier developers may avoid.
- The "5× cost reduction" is amortized over 5 profiles and is modest when fewer profiles are needed; the benefit scales with the number of differentiated deployments an organization actually wants.

## Relevance

对关注"AI 安全 / 双用途能力管控 / 模型治理"方向的研究者高度相关：GRAM 提供了一条区别于 refusal training 与 post-hoc unlearning 的、可在预训练阶段实现"差异化能力部署"的工程路径，并给出了少见的缩放规律证据。对于做大模型预训练、unlearning、或安全对齐的工程实践，其计算比指标、部分标注优势与"数据过滤随规模改善"的结论都具有直接的方法论与策略参考价值。

## Questions for Further Exploration

- Does the favorable scaling trend (widening forget gap, GRAM tracking filtering) persist to **frontier scale (tens of billions+)** and under **multi-epoch / larger auxiliary-data fractions**, where the authors note GRAM/FT-LoRA may diverge from filtering?
- How does GRAM's capability separation survive **instruction tuning and RLHF** — does post-training reinforce the modular structure or re-entangle capabilities into shared parameters?
- Can capability removal be validated with **downstream behavioral benchmarks** (virology QA, exploit generation, Lisp code completion) rather than cross-entropy, to confirm higher loss means actual loss of hazardous capability?
- How does GRAM behave under **capability entanglement**, where the target capability overlaps substantially with a retained one (e.g., general biology vs. virology) — the hardest case the paper flags but does not test?
- Against **stronger unlearning baselines** (NPO, representation engineering, task arithmetic), does GRAM's robustness-to-finetuning advantage still hold, and how does it compare to adversarial-training-based elicitation defenses?
- What is the theoretical basis for **GRAM's superior composability** over FT-LoRA (multi-module activation never seen in training yet working), and does the absorption-effect explanation survive direct mechanistic investigation?
- For **open-weight releases**, can module ablation be made tamper-resistant, since an adversary with weights could in principle re-train or re-add a removed module?
