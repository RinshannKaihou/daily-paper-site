> **一句话总结：** 作者用逐层线性探针在 SantaCoder-1.1B 与 CodeLlama-7B 的残差流上探测类型语义，发现即便从无类型 Python 训练也能恢复跨语言（Java↔Python）可线性解码的类型表示，且对对抗性重命名只表现出部分鲁棒性——揭示了代码模型内部存在一个跨语言的"类型流形"。

> **TL;DR:** TypeProbe trains per-layer linear probes on the FIM-token residual stream of SantaCoder-1.1B and CodeLlama-7B over a synthetic parallel Java/Python (typed and untyped) corpus, and finds linearly decodable type representations that transfer cross-lingually — e.g., Task 1 (Argument Type) selectivity for CodeLlama Java→Java reaches S=87 at L16 and still transfers Java→pyUnt at S=73 and pyUnt→Java at S=47 — while adversarial renaming reveals only partial robustness (drops as large as Δ=41 in some transfer cells), against a random-label control baseline.

## 核心贡献

- **首次系统探测形式类型语义**：据作者所知，此前代码模型可解释性工作未直接针对形式类型语义或跨语言类型表示；本文把探测目标从语法/标识符层面推进到类型层面。
- **构造受控并行语料**：构建三个 9 万样本的并行分区（Java、pyUnt 无类型 Python、pyTag 带类型 Python），实现同一程序的不同语言实现，并对标识符/字面量独立随机化，杜绝跨语言词法捷径。
- **跨语言类型表示涌现于无类型代码**：在无类型 Python 上训练的探针可迁移到 Java（Task 1 中 SantaCoder pyUnt→Java 达 S=73），说明类型相关方向不依赖显式标注即可恢复，提示存在跨语言"类型流形"。
- **对抗性重命名评估**：设计标识符系统性地与真实类型矛盾的对抗分区，用以区分类型级推理与词法启发式，量化鲁棒性为标准→对抗的选择性绝对下降 Δ。
- **解耦三个子任务**：将"函数预测""参数类型""应用结果类型"分离，可定位失败模式（如结果类型可预测但函数选择错误）。
- **公开代码与数据**：https://github.com/anticleiades/TypeProbe

## Key Contributions

- First systematic probing study targeting **formal type semantics** and **cross-lingual type representations** in pretrained code models, extending prior probing work (which focused on syntax/identifiers/namespaces) to the type level.
- A **controlled parallel corpus** of three 90K-example partitions (Java, pyUnt = Python without annotations, pyTag = Python with annotations) implementing the same programs, with identifiers and literals independently randomized per partition to eliminate lexical shortcuts across languages.
- Evidence that **type-relevant directions are recoverable from untyped code**: probes trained on pyUnt transfer to typed Java/pyTag, suggesting an aligned cross-lingual "type manifold" analogous to multilingual word-embedding alignment in NLP.
- An **adversarial renaming protocol** in which identifier names systematically contradict their true types, used to quantify robustness via the absolute selectivity drop Δ between standard and adversarial settings.
- A **three-task decomposition** (Function Prediction, Argument Type, Application Result) that isolates failure modes — e.g., correct result-type recovery despite incorrect function selection.

## Methodology

**Research questions.** (1) *Representation*: do pretrained code models develop linearly decodable type-level semantics, and in which layers? (2) *Invariance/robustness*: is the representation invariant across Java/Python syntax and robust to lexical adversarial perturbation? (3) *Inference*: can internal states resolve data flow / typed function application (predict the result type τ_b from argument type τ_a)?

**Dataset.** A programmatically generated corpus with three 90K-example partitions — Java, pyUnt, pyTag — all implementing the same underlying programs. Identifiers and literals are independently randomized per partition so operationally equivalent examples never share names/values across languages. Each example contains a masked call site `b : τ_b = f_⟨FIM⟩(a)` with a fill-in-the-middle token; the model must pick between two candidate functions distinguishable only by type compatibility. Input-output type pairs are uniformly distributed to avoid trivial shortcuts. Type systems map operationally equivalent base types across languages (str↔String, bool↔boolean, int↔int, float↔float) plus list containers. An adversarial partition renames every variable `v` to `var_{τ'}` where τ' is uniformly sampled from the *incorrect* types, and analogously for function names.

**Models.** SantaCoder-1.1B (24 layers, 16 heads, hidden 2048, MQA) and CodeLlama-7B (32 layers, 32 heads, hidden 4096, MHA), both decoder-only.

**Probing.** Post-residual activations are extracted at the FIM-token position. A separate linear probe is trained for every (model, layer, task, training-partition) combination, exhaustively across layers, using cross-entropy with AdamW (lr 1e-4, weight decay 0.01), 20 epochs, no feature normalization, 4-fold cross-validation with 10% held-out validation. Batch size 512 (CodeLlama) / 256 (SantaCoder). Runs on a single NVIDIA H100 80GB via PyTorch + TransformerLens.

**Metrics.** *Selectivity* S = acc_task − acc_control, where acc_control is an architecturally identical probe trained on a per-token random-label control task (a variant of Hewitt & Liang's control task). *Robustness* = absolute drop Δ in S between standard and adversarial settings. *Cross-lingual transfer* is assessed zero-shot: train on one partition (Java/pyUnt/pyTag), test on the remaining OOD partitions.

## Results & Highlights

- **Task 1 (Argument Type, 8-class) shows strong cross-lingual transfer.** CodeLlama-7B trained on Java reaches peak selectivity **S=87 at L16** on Java, and transfers to pyUnt at **S=73**; pyUnt→Java reaches **S=47**. SantaCoder-1.1B Java→Java is **S=87 at L13**, with Java→pyTag **68**, Java→pyUnt **64**, and pyUnt→Java **73** (Table 2) — all against a random-label control at ≈chance (so selectivity S ∈ [−12.5, 87.5] for the 8-class tasks).
- **Task 2 (Application Result) confirms typed-function-application structure is decodable**, but transfer is non-uniform. CodeLlama Java→Java **S=91 at L14**, pyTag→Java **84**, yet Java→pyUnt collapses to **04**. SantaCoder Java→Java **S=83 at L12** with pyUnt→Java at **53** (Table 3).
- **Task 0 (Function Prediction) cross-lingual transfer is largely weak.** CodeLlama Java→Java is only **S=50 at L16** and Java→pyUnt drops to **04** (Table 1) — the cross-lingual headline rests mainly on Tasks 1–2, not on selecting the function itself.
- **Adversarial renaming reveals only partial robustness.** Several transfer cells suffer large drops: in Task 1, CodeLlama Java→pyTag falls from **77 to 36** (Δ=**41**) and Java→pyUnt from **73 to 39** (Δ=**33**); in Task 2, CodeLlama pyTag→Java falls **83→69** (Δ=**14**) and Java→Java **91→80** (Δ=**11**) (Tables 2–3).
- **Type info peaks in mid-to-late layers** — SantaCoder L11–L16, CodeLlama L14–L18 — and pyUnt-trained probes show a late-layer decodability drop that *amplifies* under adversarial evaluation.
- **Container structure is recovered more reliably than element type** from untyped code (pyUnt confusion matrices), whereas in-domain typed evaluation (Java, pyTag) confusion matrices are nearly diagonal.
- **Motivation stat**: in generated TypeScript, nearly **94% of compilation errors stem from type-check failures** (cited from Mündler et al., type-constrained decoding), framing why internalized type structure matters.

## Critical Assessment

- **Novelty: moderate, genuine.** Combining layer-wise linear probing with a *type-level* target and a *cross-lingual* (Java↔Python) plus *typed-vs-untyped* design is, as claimed, not directly covered by prior code-model probing (Troshin & Chirkova 2022 targeted syntax/identifiers, not formal type semantics), and it complements Lucchetti & Guha's activation-steering work. The framing as a "cross-lingual type manifold" is a fair conceptual contribution.
- **Rigor: solid on protocol, weak on scope.** The control-task selectivity, 4-fold CV, exhaustive layer sweep, and adversarial renaming are methodologically clean. However, the **baseline is only a random-label control** — there is no comparison to alternative probing methods (MDL probes, SAEs, SVCCA), no causal intervention (activation patching is listed as future work), and no comparison against an actual type-inference system or type-prediction baseline. The claims are correlational only, which the authors acknowledge.
- **The "cross-lingual" headline is task-dependent and somewhat oversold.** Task 0 cross-lingual transfer is largely negative (CodeLlama Java→pyUnt = 04, SantaCoder pyUnt→Java = 16); the strong cross-lingual story lives almost entirely in Tasks 1–2. The abstract's blanket claim of cross-lingual type representations should be read as "for argument/result-type prediction, not for function selection."
- **"Partly robust" is doing heavy lifting.** Robustness is highly conditional: some adversarial cells keep most signal (Δ=8–11) while others collapse (Task 1 CodeLlama Java→pyTag Δ=41, Java→pyUnt Δ=33). Saying the structure is "partly robust" is technically accurate but understates how brittle transfer becomes when identifiers mislead.
- **Narrow, synthetic universe — the biggest threat to generalization.** Only **4 base types + list containers**, only **2 languages**, only **2 models**, all programmatically generated with at most two candidate functions in scope. There are no user-defined types, no generics/subtyping, no real-world code distribution effects, and no modern/larger code LLMs (no StarCoder2, DeepSeek-Coder, CodeQwen). Authors flag this honestly in Limitations.
- **Data quirk worth noting**: the per-token (rather than per-type) control task is a departure from Hewitt & Liang; the authors justify it via their randomization scheme but never ablate the choice, so the absolute selectivity values are not directly comparable to numbers in the probing literature that use per-type controls.
- **Reproducibility: good.** Synthetic data generator described, fixed FIM format, exact hyperparameters given, single H100 sufficient, code/data public. The disclosed-and-corrected CodeLlama tokenization bug is itself evidence that re-implementation care matters here.

## Strengths

- Clean, well-motivated experimental design with three parallel partitions that isolate typed vs. untyped and Java vs. Python signals.
- Adversarial renaming is a genuinely useful diagnostic that goes beyond standard probing and exposes how much signal is lexical vs. type-level.
- Task decomposition (function / argument / result) lets the reader see *where* representations succeed and fail rather than collapsing everything into one number.
- Honest limitations section and transparent reporting of a corrected setup bug.
- Conceptually clean takeaway: an aligned cross-lingual type manifold is a falsifiable, follow-up-friendly hypothesis for SAE/activation-patching work.

## Limitations

- Only 4 base types and list containers; no generics, subtyping, user-defined types, or low-level systems (Rust ownership, C++ templates).
- Only two models (SantaCoder-1.1B, CodeLlama-7B), both decoder-only; no scale sweep, no instruction-tuned or newer code LLMs.
- Purely synthetic data with trivially small scope (≤2 candidate functions); no real codebase evaluation.
- Linear probes cannot recover features in superposition and cannot establish a causal role of the recovered directions in generation — explicitly acknowledged.
- Cross-lingual transfer is inconsistent across tasks (strong for Tasks 1–2, weak for Task 0), so broad claims need qualification.
- Per-token control task departs from the standard per-type control, complicating cross-paper comparison of selectivity magnitudes.

## Relevance

与代码大模型可解释性、类型推理及类型受控解码的研究兴趣高度相关——它把"模型内部是否编码形式类型信息"这一基础问题给出了可探测、跨语言的实证证据，可作为类型受控解码（type-constrained decoding）与代码模型内部机制分析的参考，并为后续用 SAE/activation patching 做因果分析提供清晰的实验范式。

## Questions for Further Exploration

- Do the recovered type directions play a *causal* role in generation? Activation patching / causal tracing on these same call sites would test whether steering along a type direction changes the emitted function or inferred type.
- Do the findings survive on **real-world code** (e.g., GitHub Java/Python repos) and on **richer type systems** (generics, subtyping, user-defined classes, TypeScript structural types)?
- How do **newer/larger code LLMs** (StarCoder2, DeepSeek-Coder, CodeQwen) compare — is the cross-lingual type manifold a scale-dependent phenomenon or an artifact of the two studied models?
- Can **Sparse Autoencoders / dictionary learning** disentangle the type directions from superposed lexical features and explain the large adversarial drops (Δ up to 41)?
- Why is **Task 0 cross-lingual transfer near chance** while Tasks 1–2 transfer well — does function identity live in a representation that is genuinely language-bound, unlike base-type encoding?
- Could the recovered directions be used to **improve or audit type-constrained decoding**, e.g., detecting when a model's internal type signal disagrees with an external type checker?
