Auto-deriving hardware input bounds from PLCopen addresses closes PLC verification's deployment gap — 54 false alarms vanish, all 32 proofs hold.

ESBMC-Arduino: Closing the Deployment Gap for Formal Verification of Open-Hardware PLCs ↗ · Pierre Dantas, Lucas Cordeiro, Waldir Junior · 2026-07-09 arXiv:2607.08550
Key Metric

Why This Matters

Repairs the deployment gap between abstract verification and real open-hardware PLC semantics at near-zero cost — a ~40-line hook reusing ESBMC unchanged — making deployable safety arguments feasible for automation engineers.

Critical Edge

The headline 54→0 is a false-alarm result, not a bug-finding result — the paper finds zero genuine defects in real PLC code, and 91/123 verdicts remain unknown.

01 Methodology

Thesis: what you verify must be what runs on the device — model both the target word width and the hardware-realizable input ranges.

A PLC scan cycle is a transition system T=(S, s₀, R) over state partitioned into inputs I, outputs O, internals V. The abstract model M_abs takes a large width and D(i)=ℤ; the deployment model M_dev(P,H) takes both from a declarative HAL descriptor H=⟨w, b_adc, b_pwm, φ⟩ — small, declarative, and independent of P.

Faithful lowering (4 steps): (1) interpret all integers as w-bit bit-vectors; (2) inject assume(i ∈ D(a)) immediately after each input re-sample — this single step restores soundness in practice; (3) model ADC reads as nondeterministic within the representable set, clamp PWM writes; (4) over-approximate non-linear function blocks (PID, thermistor) as fresh nondeterministic outputs.

Soundness (Theorem 1): Reach_dev ⊆ Reach_M_dev(P,H), so every reported counterexample is physically realizable. Implementation: a ~40-line hook in the ESBMC LD frontend, toggled by the HAL_ANNOTATE env var — the SMT backend and k-induction engine are reused unchanged.

Strengths

  • Crisply formalizes a genuine, under-articulated problem — the deployment gap sits exactly where computation meets the physical process (ADC + finite-width arithmetic).
  • Elegant, minimal, adoptable. The repair is a four-field descriptor and a ~40-line hook reusing ESBMC unchanged; one binary toggles baseline vs. faithful.
  • Automatic input-range derivation removes the manual-annotation burden Frama-C / Astrée / CBMC impose — the key to corpus-scale deployment.
  • Intellectually honest throughout — "real but rare," zero genuine defects, 91/123 unknown, value = soundness not bug-count.
  • Strong cross-tool corroboration (identical witness on 3 engines) + soundness proof (Theorem 1, Lemma 1, Appendix A).

Limitations

  • Finds essentially zero real bugs in real PLC code; value is soundness-restoration, not defect discovery.
  • Partial completeness: 91/123 unknown. k-induction fails on loop-bearing function blocks (timers, counters); only 32 proofs.
  • Integer/Boolean fragment only. Non-linear FP blocks (PID, transcendental linearization) are soundly over-approximated — analog REAL corpora out of scope.
  • Uniform 10-bit ADC assumption for every integer input; per-address domains, 14-bit UNO R4 channels, digital & network variables are future work.
  • No hardware-in-the-loop validation; strongest baseline is the authors' own ESBMC-PLC line (PLCverif not run head-to-head).

03 Questions for Further Exploration

  1. How does the method perform on analog REAL-arithmetic ICS-testbed corpora (PID loops), where the authors concede the gap may be more prevalent but which fall outside the frontend?
  2. Can the 91/123 unknown verdicts be converted to safe proofs via loop-invariant inference or compositional k-induction, raising the count above 32?
  3. Would a fixed-point abstraction of non-linear blocks (vs. nondeterministic over-approximation) recover precision and change the prevalence finding?
  4. Hardware-in-the-loop: does running verified binaries on physical Uno/Due boards and confronting witnesses (e.g. adc=898) confirm the theorem against silicon?
  5. Extending (M1) pin-binding/aliasing and (M2) ADC quantization within the framework — do they expose extra defect classes?
  6. Could the HAL discipline be layered onto PLCverif or Arcade.PLC, and would the soundness/automation results transfer?
