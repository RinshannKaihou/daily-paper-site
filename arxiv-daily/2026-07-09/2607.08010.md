> **一句话总结：** 作者提出一种离线"工具制造"流水线，把 CodeAct 风格推理代理中反复生成的 SOP 代码步骤编译成经验证、可版本化的工具库，部署在亚马逊 FC 出货月台告警分诊系统中：相对"无工具子代理"基线，p50 延迟下降 42%，1,500 条历史告警的端到端错误率最高降低 53%，配合直接调用架构进一步使 p50 再降 62%。

> **TL;DR:** This Amazon Fulfillment Technologies paper replaces a CodeAct-style inference-time coding loop with an offline agentic tool-making pipeline that compiles each of the 44 decision nodes in a Fulfillment Center alarm-triage SOP into a validated, versioned tool grounded in live MCP execution traces. Deployed in production, tool calls cut p50 latency by 42% on Qwen3 32B versus the no-tools sub-agent baseline; on 1,500 historical alarms they reduce end-to-end error rate by up to 53% (GLM-4.5-Air: 1.7% → 0.8%), and a direct-call architecture ablation cuts p50 a further 62% (p99 30%). Residual errors concentrate on 4 underspecified SOP nodes, where targeted spec/data fixes raise per-node pass@1 from 94.5% to 99.9%.

## 核心贡献

- 提出离线"工具制造"流水线：对每个 SOP 决策节点，用"数据收集子代理 + 工具生成 LLM + 测试-修复循环（reflector + tool-maker）"把反复出现的推理步骤一次性编译为经验证、可版本化的确定性工具，运行时直接调用，失败时回退到 CodeAct 子代理。
- 通过"数据收集轨迹 + 测试-修复循环"两种 grounding 机制弥合 SOP 文本缺失的执行细节（字段名、数据类型、空值、边界约定），证明二者缺一不可：移除修复循环 pass@1 降 9.6pp，移除轨迹降 5.2pp。
- 真实生产部署：在亚马逊 FC 出货月台告警分诊系统（44 决策节点 / 19 动作节点的决策树）中上线，已跨 5,000+ 生产告警被调用，无回滚。
- 量化收益显著：p50 延迟降 42%，端到端错误率最高降 53%，直接调用架构进一步使 p50 降 62%；同时版本化工具暴露了上游 3 处既有数据不一致（类型不一致、累计 vs 当前事件、百分比 vs 小数格式漂移）。
- 给出可复用的运营模式：LLM 离线构建并自修复自身动作空间，使工业 LLM 系统更快、更一致、更可审计，并指出限制因素从"代理能力"转移到"SOP 规格质量"。

## Key Contributions

- An offline agentic tool-making pipeline that compiles each repeated SOP step into a single deterministic, validated, versioned tool via three components: a data-collector sub-agent run against the production MCP, a tool-maker LLM conditioned on SOP text + node position + the trace, and a reflector-tool-maker test-repair loop (budget 3 rounds) graded against labeled cases.
- Two grounding mechanisms that close the gap left by human-authored SOP text: (a) a data-collection trace (query code, MCP responses, observed schema, graded verdict) and (b) a test-repair loop. Both are shown to be necessary through ablation.
- A real production deployment in an Amazon Fulfillment Center outbound-dock alarm-triage agent operating over a 44 decision-node / 19 action-node SOP across heterogeneous metric backends, with tools invoked across 5,000+ production alarms and no rollback needed since deployment.
- A latency + reliability win, plus an architectural simplification: compact tool outputs let the main agent call tools directly (no sub-agent), and versioned tools surface pre-existing upstream data drift.
- A diagnostic finding that once grounded tools exist, the bottleneck shifts from agent capability to SOP specification quality, with targeted fixes raising residual-node pass@1 from 94.5% to 99.9%.

## Methodology

The system operates in two modes (offline build time, online inference time).

**Base agent (baseline).** Built on the Eluna hierarchical multi-agent architecture (Liu et al., 2026b): a main agent traverses a decision tree and delegates each evaluated node to a CodeAct-style sub-agent (Wang et al., 2024a) via a free-text question. The sub-agent generates and executes code that queries metrics through MCP (Model Context Protocol), iterating until it returns a free-text answer the main agent parses into a verdict (true/false/no data). Per alarm the agent evaluates 3-10 nodes and the sub-agent loop can exceed 100 LLM turns. The dominant per-alarm cost is the sub-agent loop repeatedly translating the same SOP text into the same MCP queries against an unchanging interface.

**Tool-making pipeline (offline).** Each tool is a function with a uniform signature mapping a (warehouse, timestamp, context) input to the node's verdict plus supporting detail. The pipeline has three model components:

1. *Data-collector sub-agent* = the baseline CodeAct sub-agent with production MCP access. For each node it runs on three sampled training cases balanced across verdict labels, producing a trace containing query code, MCP responses, observed schema (field names, datatypes, value ranges), and the graded verdict.
2. *Tool-maker LLM* produces a candidate tool conditioned on SOP node text, node position (parent/child), and the traces.
3. *Test-repair loop*: the candidate is tested against the full labeled training set; disagreements go to a reflector LLM that writes a short diagnosis, after which the tool-maker rewrites. Budget of 3 rounds, early termination on full pass. The highest training-set pass-rate candidate is deployed.

Each node has a labeled training set of (input, verdict) pairs (per-node case counts 100-200, 8,011 train / 8,022 eval total) from an offline labeler whose logic SMEs specified. At inference, tools are used in two configurations: the main agent delegates a node to a sub-agent that calls the tool, or the main agent calls the tool directly. "No data" is reserved for missing metrics and exceptions trigger fallback to the CodeAct sub-agent. Tools are model-agnostic and regenerable independently of agent fine-tuning.

**Agent fine-tuning.** Production latency caps the deployed agent at small model size, so a larger teacher is distilled into the student with LoRA. Trajectories are kept only when extracted observations/root-causes/actions all match ground truth (~92% acceptance). To keep the student robust when tools are missing, the teacher generates trajectories under varied tool availability: 75% with tools available but 25% randomly disabled per trajectory, and the remaining 25% with all node tools removed (full fallback). This preserves fallback: a Qwen3 32B agent fine-tuned with tools but evaluated with tools disabled stays at action accuracy 97.4% / overall 96.9%, statistically indistinguishable from the no-tools baseline.

## Results & Highlights

- **Production p50 latency (Qwen3 32B):** replacing the sub-agent CodeAct loop with tool calls cuts per-alarm p50 by **42%** and p99 by **41%** (normalized to no-tools sub-agent baseline = 100); the direct-call main-agent architecture on GLM-4.5-Air reaches p50 = 26 and p99 = 29 (Table 3). Per-alarm output tokens fall 58% (14,998 → 6,225) and sub-agent turns fall 45% (33.7 → 18.7).
- **Architecture ablation (controlled, GLM-4.5-Air, 3 seeds, 1,500 alarms):** direct calls alone reduce p50 by **62%** and p99 by **30%** relative to the sub-agent architecture, isolating the architecture effect from the model.
- **End-to-end error rate (1,500 historical alarms):** tools reduce error from 2.8% to 1.8% on Qwen3 32B (36% relative reduction) and from 1.7% to 0.8% on GLM-4.5-Air (**53% relative reduction**) (Table 4). Best end-to-end accuracy is GLM-4.5-Air main-agent direct-call at 99.3%.
- **Per-node pass@1 (44 decision nodes, GLM-4.7, 30 seeds):** the full SOP+Data+Repair configuration reaches **94.5%**; SOP-only is 75.7%, SOP+D 84.9%, SOP+R 89.3%. Removing the repair loop costs 9.6pp, removing the trace costs 5.2pp. Only SOP+D+R exceeds 90%. GLM-5 reaches 93.6%.
- **Residual-error fix (Opus, 4 nodes):** augmenting training data raises pass@1 to 96.6% and clarifying the ambiguous SOP raises it to **99.9%** (Table 2), showing the residual ceiling is specification quality, not model capacity.
- **Data efficiency:** with grounding, pass@1 reaches 91.8% at N=5 training cases per node (within 3pp of the full-set ceiling); a fully label-free variant using judge-filtered pseudo-labels + voting reaches 91.6%.
- **Deployment scale:** tools have been invoked across 5,000+ production alarms with no rollback needed since deployment.

## Authors & Credibility

All seven authors (Kalle Kujanpää, Ning Liu, Shahnawaz Alam, Yeshwanth Reddy Sura, Tianyu Yang, Kristina Klinkner, Shervin Malmasi) are at Amazon Fulfillment Technologies & Robotics. The work is built directly on the Eluna system (Liu et al., 2026b, a companion paper by overlapping authors), giving the team authoritative standing on the deployed FC agent. The paper's credibility rests on genuine production evidence (5,000+ alarms, real drift incidents caught) rather than only benchmarks. Absolute latencies are withheld for competitive reasons, so latency is reported normalized — a reasonable industrial convention but one that limits external verification of the magnitude.

## Critical Assessment

- **Novelty:** Moderate and well-scoped. The core insight (amortize a repeated inference-time coding loop into a build-time tool, validated against labeled cases) is a clean industrial framing of the tool-making line (Cai et al. 2024; Wölflein et al. 2025; Yuan et al. 2024). The genuinely novel pieces are the *data-collection trace* grounding mechanism and the production direct-call architecture simplification, not tool synthesis per se. The closest peer (Pei et al., 2025, Flow-of-Action) is SOP-enhanced but benchmark-only; this paper's production deployment is the differentiator.
- **Rigor:** Strong on ablations. The four-way tool-maker × four-configuration matrix (Table 1), the trace-content ablation (Table 5), the repair-strategy comparison (Figure 3, reflection vs branching vs no-reflection), and the trace×repair interaction (Table 6) cleanly establish that *both* grounding mechanisms are necessary and that repair compute cannot substitute for the trace. Pass@1 uses 30 seeds and end-to-end accuracy uses 3 seeds with significance tests reported (Appendix F, Tables 9-14).
- **Reproducibility:** Weak to moderate. No code or data release is mentioned; the SOP, alarm data, and metric backends are proprietary Amazon systems and cannot be replicated externally. Latency is normalized rather than absolute. The method description is detailed enough to reimplement on a new domain, and model versions (GLM-4.7, GLM-5, Qwen3 32B/235B, GLM-4.5-Air, Claude Opus 4.6) and hardware (8×H100 inference, 64-128 H100 fine-tuning, vLLM, LoRA rank/α, MS-SWIFT) are specified.
- **Significance:** High for industrial LLM agents. The shift of the limiting factor from "agent capability" to "SOP specification quality," and the mechanism by which versioned tools surface pre-existing upstream drift (three concrete incidents: type mismatch offline-vs-production MCP, cumulative-vs-current events, percentage-vs-decimal format change) are operationally valuable beyond the latency/accuracy numbers.
- **Red flags / weaknesses:**
  - The headline 94.5% pass@1 is achieved on GLM-4.7 but the *deployed* triage runs on much smaller Qwen3 32B / GLM-4.5-Air students — the pass@1 numbers and the production latency/accuracy numbers come from different models, which the text handles honestly but which a casual reader could conflate.
  - Generalization is unestablished: the paper itself states (Limitations) deployment is on a single application (outbound dock alarm triage) and free-form runbooks are not covered.
  - The 99.9% pass@1 result is on only 4 residual nodes with a manually clarified SOP and augmented data (Opus) — it is a targeted, human-in-the-loop fix, not an automated gain, and is explicitly noted as not affecting the headline 94.5%.
  - Latency figures are normalized (no absolute values), and the 62% direct-call ablation is confounded by also switching model/inference in production rows (the authors acknowledge this and run a controlled offline ablation to isolate the architecture, which is the right move).
  - The three "upstream drift" incidents are reported without quantification of their frequency or blast radius, making the auditability claim somewhat anecdotal.

## Strengths

- Real production deployment with meaningful scale (5,000+ alarms, no rollback), not a benchmark-only study.
- Clean, decisive ablations proving both the data-collection trace and the test-repair loop are necessary, and that repair compute cannot substitute for environmental grounding — a result consistent with the text-to-SQL and self-repair literature.
- Honest about confounds: the architecture-vs-model confound in Table 3 is explicitly isolated via a controlled offline ablation, and residual errors are attributed to specification quality rather than overclaimed as solved.
- The label-free variant (91.6% pass@1, within 3pp of the labeled ceiling) materially lowers the barrier to adding new SOP nodes.
- Operational payoffs beyond latency: auditability, drift detection, and the transfer of one tool library across both deployed student models at a one-time 800K-token generation cost.

## Limitations

- Single-domain deployment (outbound dock alarm triage); generalization to free-form runbooks or non-decision-tree workflows is not established.
- Human oversight still required for production safety; the loop is not fully autonomous (drift detection and candidate generation are automated, but every failure cannot be guaranteed caught without human review).
- Clarifying an underspecified SOP is inherently a human task since the SOP encodes author intent; creating a new tool still benefits from labeled cases.
- Absolute latencies withheld for competitive reasons; all latency comparisons are normalized to an undisclosed baseline.
- Proprietary data and backends prevent external replication; no code/data release is mentioned.
- Pass@1 ceiling numbers (94.5%, 99.9%) come from a different (larger) model than the deployed small students, so tool-library quality in production depends on the offline tool-maker choice.

## Relevance

这项工作高度契合"LLM 代理 / self-evolving agents / 工具使用与生产部署"研究方向：它给出了把推理时编码循环离线编译为可验证、可版本化工具的完整流水线，并在真实工业场景（亚马逊 FC 告警分诊）中证明了对延迟、准确性与可审计性的实质收益，对研究低延迟、可运维的工业 LLM 代理具有直接参考价值。

## Questions for Further Exploration

- Does the offline tool-making pipeline generalize to free-form runbooks (non-decision-tree workflows) where node boundaries and labeled cases are less well-defined?
- How does per-node pass@1 degrade as upstream schemas drift between tool generation and deployment, and what is the quantitative detection latency of the monitoring agent?
- Could the label-free variant (judge-filtered pseudo-labels + voting) close the remaining ~3pp gap to the labeled ceiling with more candidates or stronger judges, removing the need for SME labels entirely?
- What is the cross-site generalization failure rate when numeric thresholds vary widely (e.g., greenfield vs mature fulfillment centers with different baseline robot error rates)?
- Can the direct-call architecture preserve fallback robustness under tool-availability dropout at higher rates than the 25% used in fine-tuning, or does fallback accuracy collapse beyond some dropout threshold?
