> **一句话总结：** Cassandra 是一种无需额外训练的自推测解码框架，通过细粒度数据选择（非结构化剪枝+尾数截断+指数压缩）构建轻量草稿模型，并配套专用编解码器硬件，在低批次边缘推理场景下实现最高 2.41 倍无损加速。

> **TL;DR:** Cassandra is a training-free, hardware-software co-designed self-speculative decoding framework that constructs a high-quality draft model via fine-grained pruning, mantissa truncation, and exponent compression, achieving up to 2.41x lossless speedup over BF16 for low-batch edge LLM inference.

## 核心贡献

- 提出了 Cassandra，一种面向低批次场景的算法-硬件协同设计自推测解码框架，无需额外训练即可构建高性能草稿模型。
- 通过非结构化数值剪枝（Wanda 激活感知剪枝 + KV cache 逐 token 剪枝）、尾数截断和指数压缩（无损失的 Unary coding 或高压缩比的 MX 格式），从原始模型中提取最关键的数据位，形成轻量草稿模型。
- 设计了轻量级 Cassandra 编码器-解码器硬件模块，支持 Unary 和 MX 两种指数压缩格式的高效格式转换，面积开销仅约 2%。
- 提出了 Superblock 内存管理方案，解决了可变长度编码带来的不规则内存访问问题。
- Cassandra 是首个专门针对消费级边缘设备低批次 LLM 推理的算法-硬件协同设计自推测解码框架。

## Key Contributions

- Proposes Cassandra, a training-free algorithm-hardware co-designed self-speculative decoding framework optimized for low-batch LLM inference on consumer-grade devices.
- Constructs the draft model through fine-grained data selection: unstructured value pruning (activation-aware for weights, magnitude-based for KV cache), mantissa truncation, and exponent compression (lossless unary coding or MX format).
- Designs a lightweight encoder-decoder hardware module supporting both unary and MX exponent compression with ~2% area overhead on a 64 TFLOPS NPU (28nm).
- Introduces a superblock-based memory management scheme to handle variable-length encoded data and prevent irregular memory access patterns.
- Achieves up to 2.41x speedup over BF16 baseline without training, and generates 1.81x more tokens than Eagle-3 under the same memory budget on RTX 4090.

## Methodology

Cassandra operates in two phases during inference:

**Draft Model Construction (Offline/Online):** The original model's weights and KV cache are transformed into a specialized format partitioned into "speculation data" (selected salient values) and "verification data" (remaining values). Three techniques are applied sequentially:

1. **Exponent Compression** — Applied first. Cassandra-1 uses unary coding (lossless, ~2.85 bits/element average) for the exponent field; Cassandra-2 uses the MX format (shared exponent, higher compression but minor accuracy loss). The BF16 exponent occupies 8 of 16 bits (50%), so compressing it is critical.

2. **Unstructured Value Pruning** — For weights, uses Wanda (activation-aware importance scoring with minimal calibration, ~128 samples). For KV cache, uses per-token magnitude-based pruning. Unstructured pruning preserves more information than structured alternatives.

3. **Mantissa Truncation** — Simply truncates low-order mantissa bits. Unlike quantization, this preserves a strict subset of original bits, so the draft model parameters are a subset of the target model — eliminating memory overhead for a separate draft model.

**Speculative Decoding (Runtime):** During draft inference, only speculation data is loaded and zero-padded back to standard floating-point, reducing memory bandwidth. During verification, both speculation and verification data are loaded to reconstruct the full-precision target model for parallel token verification via rejection sampling.

**Hardware Integration:** The Cassandra decoder sits between L2 cache and interconnect (GPU) or within the DMA controller (NPU). Page table spare bits distinguish Cassandra-format from standard data pages. The parallel unary decoder uses a parallel zero counter architecture that processes 8-bit chunks with cross-chunk state propagation.

**Hyperparameter Optimization:** A grid search over pruning ratios (30-60%) and truncation bits (0-5) on 8 calibration samples takes ~5 minutes on an A100 (vs. 96-192 GPU hours for Eagle-3). Default config (40% pruning, 4-bit truncation) transfers well across models.

## Results & Highlights

- **Speedup:** 1.78x-2.41x over BF16 baseline across RTX 4090, Jetson AGX Orin, and a custom 64 TFLOPS NPU, evaluated on 4 benchmarks (LiveCodeBench, GPQA-Diamond, LongBench-QMSum, Math-500) and 3 reasoning models (DeepSeek-R1-Distill-Llama3-8B, Qwen3-8B-Thinking, Qwen3-4B-Thinking-2507).
- **Acceptance Rate:** 0.74-0.91 across models/benchmarks for Cassandra-1 (γ=5), and 0.74-0.91 for Cassandra-2 (γ=3).
- **Accuracy Preservation:** Cassandra-1 matches BF16 accuracy exactly on all benchmarks. Cassandra-2's accuracy is comparable to SmoothQuant (W8A8), far superior to aggressive methods like Wanda pruning which drops to 0% on AIME 2025.
- **Memory Efficiency:** Cassandra requires *less* memory than the BF16 baseline due to exponent compression. It generates 11.59x more tokens than Draft&Verify and 1.81x more than Eagle-3 on RTX 4090 under fixed memory budget.
- **Hardware Overhead:** ~2% area overhead (1.76 mm² of 91.6 mm² total) and ~1.7% power overhead on a 64 TFLOPS NPU at 28nm.
- **vs. Other Speculative Decoding:** Outperforms Draft&Verify, MagicDec, and Lookahead Decoding across all benchmarks. Eagle-3 outperforms on some benchmarks (AIME, GPQA) but shows high variance and requires expensive training.
- **vs. Lossy Compression:** Demonstrates that existing lossy methods (Wanda, SqueezeLLM, DuQuant) suffer severe accuracy drops on reasoning benchmarks, while Cassandra-1 is completely lossless.

## Critical Assessment

- **Novelty:** The combination of unstructured pruning + mantissa truncation + exponent compression for self-speculative decoding is novel. The insight that the draft model should be a strict bit-level subset of the target model (eliminating memory overhead) is elegant. The unary coding approach for hardware-friendly lossless exponent compression is a practical contribution.
- **Rigor:** The evaluation methodology is comprehensive with proper baselines, multiple hardware targets, and ablation studies. However, the software emulator cannot measure actual speedup on GPUs — performance numbers come from cycle-accurate simulators (Accel-Sim, Scale-Sim), which may not perfectly match real hardware behavior.
- **Reproducibility:** The method is described in sufficient detail for reproduction. Default hyperparameters (40% pruning, 4-bit truncation) are provided and shown to transfer across models. Calibration requires only 128 samples.
- **Significance:** Addresses a real and growing problem — reasoning LLMs produce 2.99x-5.20x longer outputs, making decode-stage optimization critical, especially on edge devices where batching is infeasible. The work is timely and practical.
- **Red Flags:** (1) Cassandra has not been validated on real hardware with the proposed encoder/decoder — all results are simulated. (2) The encoder/decoder integration assumes modifications to GPU/NPU memory systems (page table bits, L2 cache placement) that require vendor cooperation. (3) The 2.41x figure is the best case (Cassandra-2 on NPU with favorable benchmark); typical speedups on RTX 4090 with Cassandra-1 are more moderate. (4) Compatibility with quantization is claimed but not empirically demonstrated beyond the MX format used in Cassandra-2.

## Strengths

- Completely training-free with minimal calibration overhead (~5 minutes on A100), making it practical for edge users.
- No additional memory capacity overhead — the draft model is a strict subset of the target model's data.
- Cassandra-1 is truly lossless (identical accuracy to BF16), which is critical for reasoning tasks where lossy compression causes severe degradation.
- Strong hardware-software co-design story with concrete synthesis results and integration guidelines for both GPU and NPU architectures.
- Consistent performance across benchmarks, unlike task-specific methods (e.g., Eagle-3 excels on AIME but struggles on LongBench).

## Limitations

- Only validated through simulation (Accel-Sim, Scale-Sim), not on real hardware with the proposed encoder/decoder. Real-world performance may differ.
- Requires hardware modifications (encoder/decoder modules, page table spare bits, L2 cache integration) that depend on GPU/NPU vendor adoption.
- Performance gains are primarily demonstrated on 4B-8B parameter models. Scaling to larger models (70B+) or smaller models (<3B) is not evaluated.
- The acceptance rate (0.74-0.91) means a non-trivial fraction of draft tokens are rejected, limiting speedup ceiling compared to training-based methods like Eagle-3.
- Cassandra-2 trades some accuracy for speed; its advantage over simple INT8 quantization on edge devices is less clear-cut given quantization's broader ecosystem support.

## 相关性

本文对边缘端 LLM 推理加速研究有直接参考价值。Cassandra 的自推测解码思路——通过细粒度数据选择构建无额外内存开销的草稿模型——为资源受限设备上的无损推理加速提供了一条新路径。对于关注推理模型部署效率和计算机架构协同设计的研究者而言，本文的方法论和硬件设计具有重要借鉴意义。

## Questions for Further Exploration

1. How does Cassandra perform when combined with INT8/FP8 quantization on the target model? The paper claims compatibility but does not empirically evaluate a quantized target model with Cassandra's draft model.
2. What is the sensitivity of Cassandra's performance to the pruning ratio and truncation bit-width across model families beyond Llama3/Qwen3 (e.g., Gemma, Mistral)?
3. Can the unary coding exponent compression be extended to FP8 or other reduced-precision formats where exponent entropy is even lower?
4. How does Cassandra's acceptance rate degrade with increasing draft length (γ > 5), and is there an optimal γ for different sequence lengths?
5. What is the end-to-end latency impact when the Cassandra encoder must perform online KV cache format conversion during decode, especially for very long sequences?
