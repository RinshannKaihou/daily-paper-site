Cassandra: Enabling Reasoning LLMs at Edge via Self-Speculative Decoding

2605.26558 2026-05-26 Choi, Kim, Son, Kim · KAIST
Key Finding

Cassandra achieves up to 2.41x lossless speedup for edge LLM inference via training-free self-speculative decoding with fine-grained data selection.

Key Metric
2.41x
speedup over BF16 baseline
Why This Matters

Training-free lossless acceleration makes reasoning LLMs deployable on consumer edge devices without vendor cooperation or retraining.

Critical Edge

Only validated through simulation, not real hardware. Requires vendor-proprietary hardware modifications (encoder/decoder modules, page table bits, L2 cache integration).

Methodology

Cassandra constructs a draft model as a strict bit-level subset of the target model, eliminating separate draft-model memory overhead. Three transforms are applied sequentially to weights and KV cache:

Exponent Compression Cassandra-1: lossless unary coding (~2.85 bits/elem). Cassandra-2: shared-exponent MX format, higher compression, minor accuracy tradeoff.
Unstructured Pruning Wanda activation-aware pruning for weights (128 calibration samples). Magnitude-based per-token pruning for KV cache.
Mantissa Truncation Truncates low-order bits. Draft parameters are a subset of target bits, so zero memory overhead for the draft model.

Runtime Speculative Decoding

During draft inference, only speculation data is loaded (reduced bandwidth). During verification, both speculation + verification data reconstruct full-precision target model for rejection sampling.

Hardware Module

Lightweight encoder-decoder between L2 cache and interconnect (GPU) or in DMA controller (NPU). Parallel unary decoder processes 8-bit chunks with cross-chunk state propagation. Superblock memory management handles variable-length encoded data. ~2% area overhead on a 64 TFLOPS NPU at 28nm.

Calibration

Grid search over pruning ratios (30-60%) and truncation bits (0-5) on 8 samples takes ~5 min on an A100. Default (40% pruning, 4-bit truncation) transfers well across models.

Results & Highlights

  • Speedup across 3 platforms (RTX 4090, Jetson Orin, 64T NPU)1.78x – 2.41x
  • Acceptance rate (Cassandra-1, γ=5)0.74 – 0.91
  • Cassandra-1 accuracy vs BF16Identical (lossless)
  • Token output vs Eagle-3 (same memory budget, RTX 4090)1.81x more
  • Token output vs Draft&Verify (same memory budget)11.59x more
  • Hardware area overhead (64T NPU, 28nm)~2% (1.76 mm²)
  • Hardware power overhead~1.7%
  • Calibration time~5 min (vs 96-192 GPU-hrs for Eagle-3)

Evaluated on LiveCodeBench, GPQA-Diamond, LongBench-QMSum, Math-500 with DeepSeek-R1-Distill-Llama3-8B, Qwen3-8B-Thinking, Qwen3-4B-Thinking-2507.

Strengths

  • Completely training-free; calibration takes ~5 min with 128 samples on a single A100.
  • Zero additional memory — draft model is a strict subset of target model data.
  • Cassandra-1 is truly lossless, critical for reasoning tasks where lossy compression causes severe degradation.
  • Concrete hardware synthesis results at 28nm with integration guidelines for both GPU and NPU.
  • Consistent performance across benchmarks, unlike task-specific methods.

Limitations

  • Only validated through cycle-accurate simulation (Accel-Sim, Scale-Sim), not real hardware.
  • Requires hardware modifications that depend on GPU/NPU vendor adoption.
  • Demonstrated on 4B-8B models only; scaling to 70B+ or <3B not evaluated.
  • Acceptance rate 0.74-0.91 limits speedup ceiling vs training-based methods like Eagle-3.
  • Cassandra-2's advantage over simple INT8 quantization is less clear-cut given quantization's broader ecosystem support.

Questions for Further Exploration

  1. How does Cassandra perform when combined with INT8/FP8 quantization on the target model?
  2. What is the sensitivity to pruning ratio and truncation bit-width beyond Llama3/Qwen3 (e.g., Gemma, Mistral)?
  3. Can unary coding exponent compression extend to FP8 or other reduced-precision formats?
  4. How does acceptance rate degrade with increasing draft length (γ > 5)?
  5. What is the end-to-end latency impact of online KV cache format conversion for very long sequences?
