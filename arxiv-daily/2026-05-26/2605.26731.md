> **一句话总结：** 通过432次受控实验证明，LLM代理的最优harness（任务框架）复杂度与模型能力层级之间并非单调反比关系，而是取决于模型类型（对话型vs推理型）和指令微调质量。

> **TL;DR:** A 432-run controlled experiment refutes the assumption that higher-capability LLM agents need less structured harnesses. The optimal harness depends on model type (chat vs. reasoning) and instruction-tuning quality, not capability tier alone.

## 核心贡献

- 提出了HEAT-24基准：24个合成任务，6个类别，使用git工作区验证，每个任务都有确定性的二元验证器
- 首次通过受控实验（6个模型×4个能力层级×3种harness条件=432次运行）检验了"能力越高，所需harness越简单"的单调反比假说
- 发现了harness复杂度悖论：严格harness使前沿对话模型（Gemini 2.5 Flash）的VTSR下降29-38个百分点
- 发现推理模型（Qwen3.5-122B）在严格harness下表现最好（91.7%）且延迟最低，与预测相反
- 2B参数的Gemma4:e2B在所有harness条件下均达91.7% VTSR，匹配强开源层级的稳定性，证明指令微调质量比参数量更重要
- 提出六标签失败分类法：format_violation主导高能力模型失败，wrong_file主导低能力模型失败

## Key Contributions

- Introduces HEAT-24, a deterministic 24-task synthetic benchmark with git-based workspace verification across six task categories (inspect local, structured edit, format sensitive, verification recovery, repair, multi step ops)
- Conducts the first controlled empirical test of the monotone inverse capability-harness hypothesis across 432 runs (6 models × 4 tiers × 3 harness conditions)
- Documents a harness-complexity paradox: strict harness reduces frontier chat model (Gemini 2.5 Flash) VTSR by 29–38 percentage points, with format violations as the dominant failure mode
- Shows frontier reasoning model (Qwen3.5-122B, extended thinking) benefits most from strict harness (91.7% VTSR, lowest latency), contradicting the hypothesis
- Demonstrates that a 2B parameter model (Gemma4:e2B) matches strong-open-tier stability at 91.7% across all harnesses, challenging parameter count as a capability proxy
- Introduces a six-label failure taxonomy and derives practical tier-aware harness selection guidelines

## Methodology

The study uses a controlled factorial design crossing six models across four capability tiers with three harness conditions:

**Models & Tiers:**
- Frontier-Proprietary: Gemini 2.5 Flash (proprietary API)
- Frontier-Reasoning: Qwen3.5-122B-A10B (MoE, extended thinking enabled)
- Strong-Open: GPT-OSS-120B (Groq Cloud)
- Constrained: Qwen3.5:2B, LLaMA 3.2, Gemma4:e2B (Ollama local, 4096-token context)

**Harness Conditions (increasing complexity):**
- Light: Two-line prompt (role + raw task instruction)
- Balanced: Adds four-step process template + allowed file list
- Strict: Six explicit stages (preflight/plan/execute/verify/recover/report), allowed files, success criteria, verification spec

**Benchmark (HEAT-24):**
- 24 tasks across 6 categories, each with deterministic binary verifiers
- Shared synthetic workspace with 12 files (YAML, JSON, Python, Markdown, CSV)
- Git-based verification: workspace initialized as git repo, verification uses git diff
- File modifications via structured `<<<WRITE:path>>>...<<<END>>>` markers

**Metrics:** VTSR (Verified Task Success Rate) and mean inference latency per task. Failures classified by automated rule-based system into six categories.

## Results & Highlights

- **Gemini 2.5 Flash (Frontier Chat):** Light harness 95.8% → Balanced 58.3% (−37.5pp) → Strict 66.7% (−29.2pp). Format violations account for 18/18 failures in complex conditions.
- **Qwen3.5-122B (Frontier Reasoning):** Strict achieves highest VTSR (91.7%) and lowest latency (23.3s vs 35.4s light), with balanced performing worst (75.0%).
- **GPT-OSS-120B (Strong-Open):** Robust across light/balanced (95.8% each), modest strict penalty (87.5%).
- **Gemma4:e2B (Constrained):** 91.7% VTSR across all harnesses — matches strong-open tier despite ~60× fewer parameters.
- **Qwen3.5:2B:** Inverted-U pattern — 0% (light) → 58.3% (balanced) → 4.2% (strict), showing no harness is universally optimal.
- **LLaMA 3.2:** Uniformly low (16.7–20.8%) across all conditions — insufficient baseline capability for structural guidance to help.
- Category analysis: `inspect_local` and `format_sensitive` tasks are the primary locus of harness sensitivity; `structured_edit` and `repair` are robust across all conditions.

## Critical Assessment

- **Novelty: High.** This is the first controlled empirical test of the widely-held monotone inverse hypothesis for harness complexity. The harness-complexity paradox finding is genuinely surprising and practically impactful.
- **Rigor: Moderate-High.** Clean experimental design with 432 runs. However, k=1 repeat per condition means individual cell estimates are noisy (wide Wilson CIs). The author is commendably transparent about this.
- **Reproducibility: Moderate.** HEAT-24 and benchmark code promised upon acceptance. API-hosted models use provider-default settings that may change. Constrained models use fixed Ollama Modelfiles (reproducible). GPT-OSS-120B strict required re-evaluation with a second API key due to rate limiting — introduces potential confound.
- **Significance: High.** Directly challenges a core assumption in LLM agent deployment. The practical guidelines for tier-aware harness selection are immediately actionable.
- **Red flags:** (1) Each tier represented by a single model — the author acknowledges this limits generalization. (2) The chat/reasoning distinction was post-hoc, not pre-registered. (3) Qwen3.5-122B's "reasoning" classification is based on enabled extended thinking, not architecture; the harness effect may be a thinking-mode interaction rather than a pure harness effect. (4) The constrained tier's 4096-token context limit may artificially penalize longer-output tasks.

## Strengths

- Clean, well-controlled experimental design with transparent methodology
- Genuine hypothesis testing with clear refutation — not just observational correlations
- Practically immediately useful: tier-aware harness selection guidelines
- Comprehensive failure taxonomy provides diagnostic value beyond aggregate metrics
- The Gemma4:e2B finding challenges the pervasive assumption that parameter count predicts capability
- Honest, thorough limitations section that identifies specific threats to validity

## Limitations

- Single repeat per condition (k=1) — statistical reliability requires k≥3
- Each tier represented by one model — cannot distinguish model-specific from tier-level effects
- Synthetic workspace with only 12 files — may not transfer to production software engineering
- Constrained models run with 4096-token context limit, potentially penalizing performance
- Cannot isolate extended thinking contribution from harness effect for Qwen3.5-122B
- The chat/reasoning distinction was identified post-hoc, not pre-registered
- GPT-OSS-120B strict condition had a confound (re-evaluation with different API key)

## 相关性

本文对LLM Agent的部署实践具有直接指导意义。它揭示了"一刀切"的harness策略是次优的，最优框架复杂度取决于模型类型而非单纯的参数规模。对于构建自动化流水线和Agent系统的研究者，这项工作提供了基于实证的harness选择指南，特别是在格式敏感任务（如JSON输出）中需要特别注意对话模型与推理模型的差异。

## Questions for Further Exploration

1. Would the harness-complexity paradox hold across other frontier chat models (GPT-4.1, Claude 3.5/4, Llama 4)?
2. Does separating process instructions from output-format specifications (as suggested by Deng et al., 2025) eliminate format violations in strict harness conditions?
3. How does the chat vs. reasoning model distinction interact with harness complexity across different task domains (e.g., code generation, data analysis, multi-turn dialogue)?
4. Can a short harness probe (4–6 tasks from HEAT-24) reliably predict optimal harness condition for an unseen model?
5. What role does the 4096-token context limit play in constrained-tier performance — would unconstrained context change the inverted-U pattern observed for Qwen3.5:2B?
