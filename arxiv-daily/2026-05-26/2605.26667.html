MemFail: Stress-Testing Failure Modes of LLM Memory Systems

I. Garg, N. Kolhe, D. Song, X. Zhao UC Berkeley 2026-05-26
Key Finding

No single memory system dominates—architectural constraints, not model intelligence, bound LLM memory performance.

Key Metric
98%+

human-judge agreement on error-type classification across 100 examples

Why This Matters

Shatters the “bigger model” assumption—memory architecture is the binding constraint, not LLM capability.

Critical Edge

Only 4 open-source systems tested; LLM-generated datasets may narrow phrasing diversity; latency omitted.

Methodology

MemFail decomposes any LLM memory system into three canonical operations and isolates where each fails.

Storage → Query → Grading

An LLM-as-judge (gpt-5-mini) classifies each failure into storage / summary / retrieval / reasoning error or correct, using get_all_memories() to diagnose which operation failed.

TaskStress-testsMechanism
Conditional-Facts Summary failure Conditional rules hidden among unrelated facts; Hard variant spreads rules across 3 non-adjacent sentences
Coexisting-Facts Storage + Retrieval N ∈ {2,3,4,5} compatible preferences stored—tests whether systems incorrectly treat them as contradictions
Persona-Retrieval Storage failure 50/50 toggle between direct queries and misleading distractor queries—tests abstention behavior
Long-Hop Retrieval failure Transitive causal chains of K ∈ {1,2,3} hops, each stored separately

Results & Highlights

  • StructMem excels at causal reasoning (Long-Hop, Conditional-Facts) but collapses on Coexisting-Facts; Mem0 shows the inverse.
  • SimpleMem's verbatim storage avoids summary errors entirely but is retrieval-inefficient at scale.
  • Scaling retrieved memories (k) helps only retrieval-bottlenecked tasks; marginal returns when summary errors dominate.
  • Stronger internal models degrade accuracy—smarter models produce overly verbose memories that pollute embedding spaces.
  • Token-performance tradeoff is task-dependent: verbose memories help summary-bottlenecked tasks but hurt retrieval-bottlenecked ones.
  • A-MEM's descriptive notes explode token usage without retrieval gains; RAG-style embeddings fail on inter-entity relationships.

Strengths

  • Clean formal framework generalizing across diverse memory architectures via three canonical operations.
  • Adversarial task design that effectively isolates individual failure modes rather than reporting end-to-end accuracy.
  • Surprising, actionable findings: stronger models don't help, more tokens can hurt, no system dominates.
  • Practical future directions (mixture-of-memories, task-adaptive token scaling) grounded in evidence.
  • Excellent reproducibility—MIT-licensed code, datasets, and a three-function API contract.

Limitations

  • LLM-generated datasets may not capture the phrasing/entity diversity of real-world conversations.
  • Only four open-source systems evaluated; proprietary or non-standard architectures are excluded.
  • Latency analysis collected but intentionally omitted—a missed practical dimension.
  • The three-operation formalism may not capture systems with implicit or learned memory (e.g., weight-editing).
  • Binary failure classification (correct vs. one of four error types) may oversimplify interacting failures.

Open Questions

  • Q1 How would a mixture-of-memories architecture (routing causal chains to a graph, persona facts to a flat store) perform across all tasks—and what is the routing overhead?
  • Q2 Can task-adaptive token scaling be learned online, with the system dynamically choosing compression level per information type?
  • Q3 How do MemFail failure modes correlate with real deployment metrics—user satisfaction, task completion, conversation length?
  • Q4 Would temporal decay or forgetting mechanisms shift the failure distribution, especially for Coexisting-Facts and Persona-Retrieval?
  • Q5 How sensitive are results to embedding model choice, and could cross-encoders or learned retrieval mitigate retrieval failures?
