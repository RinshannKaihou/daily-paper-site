How Fast Can Reward Models Score?

A systems study of C++ and PyTorch inference runtimes for RLHF — by Venkata Naga Sai Vishnu Rohit Pulipaka, Anish Katta & Deva Rohit Reddy Peddireddy.

RLHF Reward-Model Inference ONNX Runtime torch.compile Systems Benchmarking DeBERTa-v3-large Electra-large
Key finding

The CPU win comes from ONNX Runtime, not C++; on GPU torch.compile beats the dedicated C++ engine; and naive padding is a hidden throughput killer.

Key metricCPU · p50
335.9ms
C++ ONNX engine scores one row, vs 581.6–628.8 ms for every PyTorch baseline.
≈ 1.7–1.9× faster · zero CI overlap · Welch p < .001
Key finding02

Calling the same ONNX session from Python (≈349 ms) statistically ties the C++ engine (335.9 ms) — the language isn't the lever, the graph runtime is.

Why this matters03

You don't need C++ to win on CPU — just export ONNX and drop eager mode for ~1.8× gains. On GPU, torch.compile is the faster answer.

Critical edge04

A single 6 GiB laptop GPU, a 60-row eval set, and no end-to-end RLHF wall-clock measurement leave the real-world training speedup unproven.

Methodology

DeBERTa-v3-large + Electra-large · Anthropic hh-rlhf
  • ENGINENative C++ wraps ONNX Runtime 1.26.0 with built-in SentencePiece (DeBERTa) / WordPiece (Electra) tokenization, batching & post-processing — no Python at inference time.
  • MODELSTwo transformer-encoder reward models: DeBERTa-v3-large (primary) and Electra-large-discriminator (generalization check).
  • BASELINESHF Transformers eager · torch.compile (inductor) · FastAPI wrapper · plus a Python-calls-ONNX-RT isolation control.
  • CORRECTNESSMax abs reward error 5.7×10⁻⁶ (CPU) / 4.2×10⁻³ (GPU). Found & fixed a real Electra export bug (token_type_ids).
  • SWEEPBatch 1/2/4/8 · naive vs length-bucketed · concurrency 2/4/8 shared vs multi-instance · both models, both devices.
  • HARDWAREAMD Ryzen 7 5800H · NVIDIA RTX 3060 Laptop (6144 MiB) · fp32 · CUDA 12.6 · PyTorch 2.10.0.

Statistical discipline

Every number is a mean with 95% CI across independent process launches — 5 launches (CPU baselines 3, due to torch.compile recompile cost). CIs must not overlap for a result to count; Welch's t-test confirms all headline comparisons.

03

Strengths & Limitations

side by side

Strengths

  • Rigorous, reproducible methodology — independent process launches, CI non-overlap as the reality threshold, Welch's t-tests, pinned library versions, open-sourced code.
  • Genuinely surprising, actionable findings — ONNX-Runtime-not-C++ is the real CPU lever; torch.compile wins on GPU; naive padding is actively harmful; concurrency doesn't scale.
  • Intellectual honesty — publicly retracts a Table 8 result (Electra 4.7×) that didn't reproduce, and reports a real export bug found & fixed.
  • Clear, opinionated engineering guidance tied to specific hardware and backed by non-overlapping CIs.

Limitations

  • Single hardware target — Ryzen 7 5800H + RTX 3060 Laptop (6 GiB). The GPU torch.compile win and 4–5-session OOM ceiling are silicon-specific; untested on server GPUs or many-core CPUs.
  • fp32 only — no half/mixed/INT8 paths; effects of quantization are unmeasured.
  • 60-row primary eval set (mitigated by a 150-row robustness set); torch.compile's recompile cost limited CPU baselines to 3 launches.
  • No end-to-end RLHF loop measurement — scoring speedup is isolated; translation to faster training is unmeasured.
  • Excluded systems — Triton cut entirely; no multi-GPU/distributed serving; only two OpenAssistant backbones; multi-instance tested as threads, not OS processes.
04

Questions for Further Exploration

  1. Do the GPU results (torch.compile beating ONNX Runtime) hold on data-center GPUs (A100/H100, 40–80 GiB), where the OOM ceiling and per-session CUDA-context overhead shift dramatically?
  2. What fraction of an actual RLHF training step does reward scoring occupy, and does the speedup translate into measurable wall-clock improvement inside a live OpenRLHF loop?
  3. How large is the torch.compile recompile spike when a rollout hits a length outside the cached shape distribution — is the shape cache warm enough in practice to avoid it?
  4. Does the picture change under fp16/bf16 or INT8 quantization, which are standard in production reward-model serving but entirely absent here?
  5. Would length-aware bucketing remain the only throughput lever on a many-core server CPU, or does ONNX Runtime's CPU backend gain real batch parallelism with more cores?
  6. How does a properly-run Triton Inference Server — the excluded baseline — compare, given it is a standard production choice for serving ONNX models?
  7. Is the single non-reproducing Table 8 row (Electra vs DeBERTa magnitude) a genuine architecture-dependent effect, or a measurement artifact resolvable by a fresh repeated-launch remeasurement?
