> **一句话总结：** 本文提出 Skillware——围绕持久化 Agent Skill 的 AI 原生软件抽象，用三条必要条件（行为优先性、独立软件身份、Agent Host 执行关系）与 Lifecycle Continuity 软件级属性，把"以自然语言为主要载体的可复用任务行为"纳入软件工程本体与生命周期管理，并以 13.8 万条 SKILL.md 记录语料与多个固定修订案例作为证据。

> **TL;DR:** Skillware is a proposed AI-native software abstraction treating Agent Skills (SKILL.md directories) as independently managed software units rather than ephemeral prompt files; using a frozen corpus of 138,133 content-deduplicated SKILL.md records across 20,556 repos, 13 fixed-revision engineering cases, 15 category-boundary cases, and independent empirical studies, the paper defines a 3-condition membership test (behavioral primacy, independent identity, Agent Host execution) plus a Lifecycle Continuity property, and shows Gang-of-Four design patterns transfer to natural-language behavioral source (e.g., Superpowers as Facade, gstack bindings as Adapter, OpenMontage staged workflows as Composite).

## 核心贡献

1. **Skillware 本体（Skillware Ontology）**：把"自然语言行为源 → Skill Artifact → Skillware Unit → Agent Host → Agent Runtime → Execution Trace → Task Outcome"形式化为一条责任链。核心区分在于：Skill Artifact 是行为载体，而 Skillware Unit 是围绕该载体、拥有可独立寻址的名称/地址/包/版本/兼容性/溯源的"软件身份"。
2. **可审计的成员资格判定（C1–C3）与 Lifecycle Continuity（LC）**：
   - C1 行为优先性：一个 Skill 或一致 Skill 套件承担并组织可复用任务行为；
   - C2 独立软件身份：可被独立获取/安装/跟踪于 Agent Host 之外；
   - C3 Agent Host 执行关系：至少一个已识别的兼容 Host（Codex、Claude Code、GitHub Copilot、Hermes、OpenClaw 等）可发现/激活/加载/使用该 Unit；
   - LC：在 update、maintenance、rollback、removal 中保持同一 Unit 身份（作为软件级属性而非成员资格）。
3. **Skillware 工程生命周期（Engineering Lifecycle）与结构性维度**：1 个最小基线（Atomic）+ 6 个可组合维度（Resource-Backed / Tool-Backed / Event-Driven / Plugin-Packaged / Companion-Coupled / Adaptation-Enabled）；并提出 6 步工程工作流（定义 Unit 与契约 → 选维度 → 选模式 → 实现机制 → 评测打包 → 发布维护演化）。
4. **Identity-Preserving Evolution（身份保持演化）视角**：区分 Self-Evolution（候选变更生成）与 Collaborative Evolution（治理型多贡献者发布），使人类、agent、部署证据都能在保持软件身份的前提下产出新版本。
5. **设计模式迁移作为"软件连续性"证据**：展示 Facade / Adapter / Composite / Observer / State / Strategy 等经典 GoF 模式可由自然语言行为源 + 脚本/hooks/插件/子 Skill 实现（需满足七要素迁移判定：源意图、设计力、参与者对应、结果、实现证据、聚焦验证、误用判别）。
6. **多层级证据协议**：四层证据（Agent Skills 规范、冻结语料、固定修订案例、独立实证研究）各自承担不同的推理边界，避免用词频/路径 token 擅自推断语义正确性或跨系统行为等价。

## Key Contributions

- **A declared software category**, not a tool: "Skillware" is positioned as the AI-native software abstraction parallel to code-centric software (modules/components) and model-centric intelligence (learned parameters). A Skill Artifact specifies reusable behavior; a Skillware Unit manages it as separately identifiable software.
- **Operational, auditable membership tests** (C1–C3) plus a separately-reported Lifecycle Continuity property, each with a "positive observable" and a "frequent exclusion signal" (Table 2) — enabling boundary cases (e.g., a pure Skill index, an MCP filesystem server, a GPT config) to be excluded even when they are installable/persistent.
- **A structural + lifecycle two-axis model**: structural profile (snapshot) is decoupled from lifecycle record (same identity across releases), explicitly rejecting a "maturity ladder" interpretation.
- **Empirical grounding on real artifacts**: the SkillMD-138K snapshot (revision `0d73048a…`), 15 category-boundary cases (12 inclusions, 3 negatives), and 13 fixed-revision technical implementations (Superpowers, gstack, ECC, OpenMontage, SkillOpt, etc.) each pinned to a 40-character commit.
- **Design-pattern transfer evidence** demonstrating software-engineering principles are "medium-independent" when behavior is carried by natural-language source rather than formal code.
- **A research artifact package** (repo `MetaInFLow/skillware`) with manifests, fixed-revision case packets, pattern fixtures, and reconstruction scripts.

## Methodology

The paper is a **conceptual/software-engineering ontology paper with an empirical evidence protocol**, not a benchmark experiment. Its method:

1. **Phenomenon identification**: Agent Skills (SKILL.md directories with YAML metadata, instructions, and optional `scripts/`, `references/`, `assets/`, hooks, plugin manifests, tests, companions) are now acquired, installed, activated, updated, maintained, and removed across multiple independent systems.
2. **Ontology construction**: Define distinct analytical objects (Behavioral Source, Skill Artifact, Skillware Unit, Agent Host, Agent Runtime, situated performance, task outcome) and an execution-semantics equation: `π = R_H(S, τ, M, C, T, P, σ)` — explicitly assigning no deterministic semantics to unrestricted natural language.
3. **Membership formalization**: Three necessary conditions (C1 primacy, C2 identity, C3 Host execution) must converge on the same unit; LC is reported separately and does not score quality/security/popularity.
4. **Evidence design with separate inference boundaries** (Table 3 maps evidence→condition with inference limits):
   - Specification & corpus (narrow: envelope existence, metadata prevalence, support-path references);
   - Fixed-revision cases (declared units pinned to commits, recording positive evidence, counterevidence, limitations);
   - Independent empirical studies (Chen et al. 97,755 Skills; Gao et al. 18,463 registry + 23,199 personal Skills + 3,709 reuse links; Hong et al. 238-Skill smell study; SkillOS, SkillFab, FederatedSkill).
5. **Engineering-continuity test**: Show conventional SE concepts (module boundary, component identity, lifecycle standards, design patterns) remain applicable; transfer is admitted only when 7 elements (source intent, design forces, participant correspondence, consequences, implementation evidence, focused verification, misuse discriminator) are present.

## Results & Highlights

- **Corpus scale & metadata convention**: The frozen SkillMD-138K snapshot contains **138,133 content-deduplicated SKILL.md records across 20,556 repository identifiers**; **136,380 files (98.73%)** contain frontmatter delimiters — evidence that a structured metadata envelope is a dominant visible convention in collected artifacts (the paper is careful that this does not validate frontmatter fields or cross-system activation semantics).
- **Support-path reference prevalence**: A narrow lexical detector finds **32,069 files (~23.2%)** with at least one explicit path token into `references/`, `scripts/`, or `assets/`, totaling **236,421 such tokens** across the corpus — a measurable authoring convention for packaged adjacent artifacts.
- **Document-length engineering signal**: The **median SKILL.md is 169 lines / 687 words**, with the **90th percentile at 517 lines / 1,971 words** — quantifying the amount of behavioral source that may enter discovery/context assembly/review/maintenance (length carries no direct quality meaning).
- **Independent-study convergence** (under their own samples): Chen et al. characterize **97,755 Skills** and motivate a Skill OS abstraction; Gao et al. analyze **18,463 registry + 23,199 personal-use Skills from 5,876 GitHub repos with 3,709 reuse links**; Hong et al. report widespread detected smells across **238 Skills** — together establishing reuse, maintenance, and quality pressure.
- **Case coverage**: 15 category-boundary cases (12 inclusions / 3 deliberate negatives) and 13 fixed-revision technical implementations; **none of the 13 technical units fully satisfied the Adaptation-Enabled dimension** — SkillOpt and darwin-skill provide only partial Self-Evolution precedent.
- **Pattern transfer**: 10 GoF patterns (Facade, Adapter, Composite, Decorator, Strategy, Template Method, Observer, State, Memento, Mediator) have detailed constructive implementations in the supplement; 6 are argued in the main text with verified or candidate status against real repos (Superpowers Facade verified; OpenMontage Composite as candidate + constructive fixture; Observer pending complete registration/delivery evidence).

## Authors & Credibility

- **Haodi Fan** and **Zucong Lan** (anthonyfan@metainflow.cn, neillan@metainflow.cn), affiliated with **MetaInFlow**. They explicitly disclaim word coinage or historical priority — the term "Skillware" already appears in an ARPA Hellenic Logical Systems PyPI modular-skills framework (cited as ref [27]); their contribution is framed as a cross-project operational definition and evidence protocol.
- **Reproducibility commitments**: frozen corpus revision (`0d73048abf2fb6ee91f6f9f5ac598d5be8d6bdd7`), SHA-256 of the raw Parquet (`c3c07af0…403`), 40-character commit pins for every technical case, and a public research-artifact repo (`MetaInFLow/skillware`) with manifests, case packets, pattern fixtures, and reconstruction scripts.
- **Honest scoping**: The paper repeatedly bounds its claims — "emerging" means recurring artifacts are observable while conventions remain unsettled; evidence "supplies no population estimate of high-quality Skillware, no universal adoption claim, and no behavioral-equivalence claim across agent systems." Several close studies cited are recent preprints whose conclusions may change through review.
- **Caveat on venue/independent validation**: As a preprint proposing a category definition, it has not yet passed peer review, and the authors themselves flag that inter-rater reliability has not been measured ("It supplies no estimate of inter-rater reliability").

## Critical Assessment

- **The core unit distinction (Skill Artifact vs. Skillware Unit) is genuinely useful.** Treating "the independently managed identity around one Skill" as the classification unit — rather than a single SKILL.md file or a containing plugin — is a defensible abstraction that matches how the ecosystem actually packages things like Superpowers (a coherent suite with hooks, manifests, tests, and a Visual Companion sharing one package-level identity).
- **The C1–C3 + LC test is operational but partly judgment-laden.** C1 "behavioral primacy" can be genuinely hard to judge in packages with substantial executable services (the authors admit this in §7.2). Whether a given unit's primary behavior lives in its Skill source vs. in application code is sometimes a matter of analyst framing, and no inter-rater reliability is reported — a real construct-validity gap the authors acknowledge.
- **Corpus evidence is deliberately narrow and the paper is unusually honest about it.** The 98.73% frontmatter and 23.2% support-path-token numbers are explicitly framed as metadata/lexical signals that cannot prove semantic correctness, runtime participation, or operational use. Table 3's "Inference limit" column is a strong epistemic discipline rare in this kind of position paper.
- **The design-pattern transfer is the most testable claim and the strongest evidence of SE continuity** — but several mappings carry caveats: Observer is "candidate pending complete registration and delivery evidence," Composite is "candidate correspondence plus constructive repository fixture," and Strategy's "comparative benefit requires runtime study." In other words, some mappings are constructive (built by the authors as fixtures) rather than observed in the wild, which slightly weakens the "patterns already recur in real Skillware" claim.
- **Scope of "emerging" is appropriate but limits impact.** Because conventions, compatibility contracts, evaluation methods, and lifecycle infrastructure are admittedly unsettled, the paper is more of a conceptual scaffolding + research agenda than a validated framework. The FDE (Forward Deployed Engineering) deployment hypothesis and Collaborative Evolution workflow are explicitly prospective.
- **Self-Evolution coverage is a known hole.** The authors forthrightly state that none of the 13 frozen technical units satisfied every Adaptation-Enabled requirement, so the "identity-preserving self-evolution" pillar rests on partial precedents (SkillOpt, darwin-skill) rather than a complete real example.

## Strengths

- Clearly separates what the abstraction **is** (ontology) from how it **changes** (lifecycle), and separates both from **situated performance** (a particular run under a particular model/context/toolset) — preventing the common error of treating an artifact description as evidence of behavior.
- The boundary-case methodology (declared unit → frozen revision → positive/counterevidence/limitations for C1–C3 with LC coded separately) is reproducible and makes disagreements traceable.
- Genuine epistemic humility: inference limits are stated per-condition (Table 3), and the threats-to-validity section (§7.2) names collection bias, purposive sampling, construct validity, and recent-preprint dependence without deflecting.
- Connects an emerging artifact type to a deep SE tradition (Parnas modules, Szyperski components, ISO/IEC/IEEE 12207 lifecycle, Lehman evolution laws, GoF/POSA/Evans patterns), lowering the conceptual barrier for SE researchers entering the agent-skills space.

## Limitations

- **No behavioral/empirical validation of the abstraction's utility**: there are no experiments showing that classifying something as a Skillware Unit, or applying the 6-step workflow, improves task success, maintenance cost, reuse, or defect rate vs. ad-hoc Skill authoring. The FDE comparison study is explicitly future work.
- **Construct validity of C1 is under-tested**: no inter-rater reliability, no independent coding; different analysts could pick different unit boundaries or Host roles.
- **Corpus is metadata/lexical only**: registry-heavy collection, aliases, forks, generated artifacts, and missing source revisions constrain external validity; content-level replication requires access to a separately acquired corpus.
- **Pattern transfer is partly constructive**: some mappings are built as fixtures by the authors rather than discovered in production Skillware, so "patterns recur in the ecosystem" is only partially supported; ecosystem frequency and comparative benefit remain open.
- **Adaptation-Enabled (the lifecycle-evolution capstone) has no complete real example** among the 13 reviewed units; Self-Evolution and Collaborative Evolution rest on partial precedents.
- **Cross-Host parity is explicitly out of scope**: source inspection cannot establish behavioral equivalence across Codex / Claude Code / Copilot / Hermes / OpenClaw; that requires direct multi-system execution.
- **Author count (2) and non-academic affiliation** reduce institutional signaling; the paper leans heavily on the rigor of its evidence protocol and public artifacts to compensate.

## Relevance

This paper is **highly relevant** to anyone building, curating, or researching Agent Skill ecosystems (Codex Skills, Claude Code Agent SDK Skills, GitHub Copilot Skills, Hermes, OpenClaw, plugin/skill marketplaces). It matters for:

- **AI-native software engineering researchers** — it provides a shared vocabulary (Skillware Unit, Behavioral Artifact, Agent Host, Agent Runtime, Lifecycle Continuity) and a testable membership protocol that can structure future empirical work on skill reuse, quality, and evolution.
- **Agent platform & tooling builders** — the 6 implementation dimensions and 8 engineering mechanisms (Skill Router, Progressive Disclosure, Deterministic Kernel, Canonical Skill + Generated Adapters, Lifecycle Bootstrap, Verification Gate, Human-Gated Companion, Evidence-Gated Self-Evolution) are a concrete design checklist.
- **Skill authors / maintainers** — the pattern mappings and the "Engineering Consolidation" trajectory give a pragmatic path from a natural-language behavioral seed to a packaged, tested, versioned unit without forcing premature code.
- **Security & governance folks** — the paper foregrounds provenance, capability gates, prompt-injection-through-references, companion sessions, adaptive supply chains, and rollback as first-class concerns.
- It sits at the intersection of `cs.SE` and `cs.AI` and engages a lively 2026 preprint line (SkillOS, SkillFab, FederatedSkill, the Gao/Hong/Chen empirical studies), making it a useful node for mapping current research on "skills as software."

## Questions for Further Exploration

1. **Inter-rater reliability of C1–C3**: If independent coders classify the same 15 boundary cases and 13 technical units, do they reproduce the authors' inclusions/exclusions? What disagreement patterns emerge around "behavioral primacy"?
2. **Cross-Host behavioral parity**: Holding a Skillware Unit, task, model, and toolset fixed, how much does situated performance vary across Codex vs. Claude Code vs. Copilot vs. Hermes vs. OpenClaw? Can a compatibility/parity benchmark be defined?
3. **Does Engineering Consolidation actually happen longitudinally?** Commit- and release-level studies: over N releases, do instructions migrate into resources/deterministic kernels, and does test coverage / maintenance cost follow a predictable trajectory?
4. **Self-Evolution under identity preservation**: Can a complete Adaptation-Enabled Skillware Unit be built where candidate changes from execution evidence pass evaluation, review, version assignment, provenance, release, and rollback — and does it beat human-only maintenance on held-out tasks?
5. **Pattern frequency in the wild**: Across the 138K corpus, how often do Facade/Adapter/Composite/Observer/State/Strategy participant structures actually occur (vs. only in constructive fixtures)? Does pattern presence correlate with maintenance or quality outcomes?
6. **Security surface validation**: Empirically, how exploitable are the named risks (prompt injection via `references/`, companion-session hijacking, adaptive supply-chain attacks)? Are capability gates + verification gates sufficient mitigations?
7. **FDE field trial**: In a real Forward Deployed Engineering setting, does Skillware-centered delivery beat code-first delivery on time-to-first-useful-deployment, cross-site reuse, defect rate, and review effort?
8. **Relationship to MCP and tool-primary units**: Where exactly should the Skillware / tool (MCP) boundary be drawn when a unit's value is split between natural-language orchestration and callable tools? Is there a stable rule, or is it inherently analyst-dependent?
