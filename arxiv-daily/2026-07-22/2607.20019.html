Key Metric
83.6%
avg DRV reduction across 7 DAC26 blocks (vs. 73.5% baseline); Block3 & Block5 fully cleaned.
Why This Matters

Automates a multi-day manual DRC ECO bottleneck at tapeout, scaling LLM agents to 765-violation blocks for the first time.

Critical Edge

Largest block still leaves 275 DRVs; connectivity check is simplified DFS, not LVS — no sign-off correctness guarantee.

Methodology — three-stage pipeline

EvoDRC runs three stages (Fig. 2). Each repair crop gets an independent agent using a ReAct loop with four internal roles: planner · adversarial reviewer · DRC engineer · patch reviewer.

STAGE 1

Layout Decomposition

Classify routing; build first-order crops touching violations bounded to 2 μm × 2 μm; merge by layer overlap into second-order crops; merge by standard-cell row into final repair crops.

STAGE 2

Agentic Repair

Per-crop agent edits only objects of editability classes A–C (vias, polygons, clipped metal ends); class D fixed. Three preview tools: local DRC, full-connectivity check, neighbor-impact preview.

STAGE 3

Assess & Skill Evolution

Record only accepted ops + DRV-delta per layer into a Knowledge DB. Skill Refiner & DB Summarizer propose candidates; mechanical checker + Skill Judge majority-vote decide iteration t+1 skills.

Implementation: Anthropic Claude Sonnet 4.6 (reasoning effort = medium) · max 5 iterations, early stop at zero DRV · crop cap 2 μm × 2 μm · KLayout 0.30.1 + ASAP7 rule deck.

Assessment

Strengths

  • First at block scale — extends agentic DRC repair from standard-cell to module-level (up to 765 DRVs).
  • Skill evolution mirrors real engineers — reuse across designs, refine on new ones; every skill traceable to a repair record.
  • Decomposition proven necessary — Ablation 3 quantifies the single-agent coverage bottleneck (81 vs. 490 fixed on Block7).
  • Engineered against LLM unreliability — 4-class editability, 3 preview tools, mechanical checks, Skill Judge majority vote.
  • Fully open — code, sample skills, benchmark released.

Limitations

  • Block7 not closed — 275 residual DRVs (−64.1%); weak on cross-layer rules like V2.M3.AUX.2.
  • "73.5% baseline" is ambiguous — source not fully specified in text; read 83.6% as the headline.
  • Cost understated — only LLM tokens reported; KLayout full-chip DRC time/compute excluded, potentially dominant at advanced nodes.
  • Connectivity = DFS, not LVS — no sign-off-level logic guarantee.
  • Single PDK only — ASAP7; no commercial-node rule-complexity evaluation.
  • Hard iteration cap — 5 iterations; no sensitivity analysis for Block7-scale designs.
Questions for further exploration
  1. Why do generated-via rules (V2.M3.AUX.2 / V4.M5.AUX.2) on Block7 resist repair — missing skill knowledge, insufficient action primitives, or crops splitting cross-layer coordination?
  2. Would 10 iterations further converge Block7, or stall once a layer's skill saturates?
  3. Under a sign-off commercial DRC deck replacing ASAP7 + KLayout, how much does repair quality degrade? Does rule complexity exceed current primitive expressiveness?
  4. Skill mechanical checks ensure traceability but not correctness — could formal verification or counterexample-driven skill falsification be added?
  5. Can Knowledge DB transfer scale to multi-reference-design distillation instead of a single CLA? How large is initial-skill quality variance?
  6. Upgrading connectivity to full LVS: how do acceptance rate and repair quality change — and would DFS-missed logic breaks surface?
  7. Does skill evolution "forget" — can early effective skills be lost to later Skill Judge replacement? Should a versioned skill store with rollback exist?
