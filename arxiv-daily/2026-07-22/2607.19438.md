> **一句话总结：** 本文为 BaseRT 推理运行时新增一套手写的 Metal 4 张量核（cooperative-tensor）内核，专门调度到 Apple M5 每核 GPU 神经加速器上，仅用于计算受限的 prefill 阶段，在 M5 Pro 上对 15 个模型配置实现 prefill 吞吐相对 llama.cpp 最高 6.4×、相对 MLX 最高 3.9× 的提升，decode 因受内存带宽限制提升有限（最高 1.75× / 1.33×）。

> **TL;DR:** BaseRT extends its framework-free Metal inference runtime with hand-written Metal 4 cooperative-tensor kernels that drive the Apple M5's per-core GPU Neural Accelerators for compute-bound prefill GEMM/MoE/attention, delivering up to 6.4× prefill throughput over llama.cpp and 3.9× over MLX on an M5 Pro across 15 Qwen3/Qwen3.5/Llama 3.2/Gemma 4 configs, while decode gains stay bandwidth-bound (≤1.75× / 1.33×).

## 核心贡献

1. **一套手写的 Metal 4 cooperative-tensor 内核**，驱动 Apple M5 每核 GPU 神经加速器，覆盖推理中矩阵乘主导的算子：4-bit/8-bit 量化下的 dense GEMM、MoE 专家 GEMM、融合 gate/up 投影（在张量核累加器上一次性完成 gate/up 投影并应用 SiLU 激活，省去每个专家一次全局内存往返）、以及 prefill 的 Flash-Attention（QK^T 与 PV 经由 matmul2d，中间保留与 SIMD 路径数值等价的精确在线 softmax）。

2. **工作负载感知的调度策略**：仅在计算受限阶段（prefill GEMM、MoE 专家 GEMM、prefill attention）选择张量核路径；decode 仍走 BaseRT 现有的特化 GEMV 与 decode-attention 内核（内存带宽受限，张量核无收益）。加载时检测 M5 家族，并做形状约束检查，不满足时回退到 SIMD 内核；pre-M5 芯片上整条张量路径被旁路，行为与既有 SIMD 内核完全一致。

3. **在 Apple M5 Pro 上覆盖 15 个模型配置的系统评测**（Qwen3、Qwen3.5/3.6、Llama 3.2、Gemma 4，从 sub-1B 到 35B 参数，含 4 个 MoE 模型），确立 Apple Silicon 上新的推理吞吐上限，并定量回答"张量核在哪里起作用"：prefill 提升显著（最高 6.4× over llama.cpp），decode 收益受内存带宽封顶。

## Key Contributions

- A **family of hand-written Metal 4 cooperative-tensor kernels** targeting the M5's per-core GPU Neural Accelerators, covering dense GEMM (4-/8-bit, reusing BaseRT's packed-quant weight layout), MoE expert GEMM, **fused MoE gate/up projection with in-accumulator SiLU**, and a prefill flash-attention whose QK^T/PV products run on matmul2d while keeping an exact online softmax (numerically equivalent to the SIMD FlashAttention path).
- A **workload-aware dispatch** that selects the tensor path only for compute-bound phases and keeps decode on the existing specialised GEMV/decode-attention kernels; shape checks fall back to SIMD, and the whole path is bypassed on pre-M5 hardware with unchanged behaviour — a single runtime that transparently exploits the accelerators only where they matter.
- A **comprehensive M5 Pro evaluation across 15 configs** (Qwen3, Qwen3.5/3.6, Llama 3.2, Gemma 4; sub-1B–35B; four MoE models) that sets a new on-device throughput ceiling and isolates where tensor cores help: large prefill gains, decode bounded by memory bandwidth.

## Methodology

**Hardware & runtimes.** All measurements on an Apple M5 Pro (48 GB unified memory, macOS Darwin 25.4). BaseRT v0.1.6 (a C++ runtime that targets Metal directly, with no MLX/PyTorch/CoreML/array-framework dependency; design described in prior work arXiv:2607.00501) is compared against llama.cpp build b9960 (Metal backend, default settings) and mlx-lm 0.31.3 (MLX 0.32.0, default generation settings), all loading the same checkpoints at matched quantisation on the same device.

**Why the split path works.** Prefill multiplies the full prompt activation matrix (one row per token) by layer weights, so arithmetic intensity grows with sequence length → **compute-bound**, dominated by GEMMs and the QK^T/PV attention products — exactly the regime where a dedicated matmul engine pays off. Decode is an M=1 GEMV that streams the whole weight set per step → **memory-bandwidth-bound**, so the tensor cores cannot help (a matmul engine cannot accelerate a phase that is waiting on memory). This asymmetry is the organising principle of the M5 design.

**Cooperative-tensor kernel structure.** Each threadgroup owns an output tile and, along the contraction dimension, feeds operand tiles into an MPP `matmul2d` that accumulates into a register-resident cooperative tensor. Notably the authors note threadgroup-memory staging is not required for peak GEMM on Apple Silicon — kernels are fastest reading operands straight from device memory and letting the cache hierarchy work, which is a materially different kernel structure from the simdgroup-matrix model; code must be compiled with `-std=metal4.0`.

**Metrics.** Prompt-processing throughput pp (tok/s) at lengths 128, 256, 512, 1024, 2048; token-generation throughput tg128 (tok/s) over 128 generated tokens. Each point is the average of five repetitions.

## Results & Highlights

- **Prefill vs llama.cpp: up to 6.4×.** On Gemma-4-E2B (Q8) the ratio rises from 2.8× at pp128 to **6.4× at pp2048** (16,264 vs 2,547 tok/s); on the Q4 variant the gain reaches 6.3× at pp2048 (16,677 vs 2,656 tok/s). BaseRT is faster on **every configuration at every prompt length**.
- **Prefill vs MLX: up to 3.9×.** The largest gap is **Qwen3-30B-A3B at pp128 (2,478 vs 639 tok/s, 3.9×)**; BaseRT leads on all but 3 of 75 measured points (the exceptions being the two smaller Llama models at pp1024–2048, where MLX edges ahead by ≤3%).
- **MoE models see the widest margins.** Across the four MoE models, BaseRT leads llama.cpp by **29–120%** and MLX by **15–288%** over the sweep; e.g. Qwen3.6-35B-A3B (Q4) leads both baselines at every prompt length (Figure 1 right).
- **Decode: bounded by bandwidth as predicted.** BaseRT beats llama.cpp on all 15 configs by **1.02–1.75×** (largest on 35B-A3B MoE: Qwen3.6-35B-A3B 110.7 vs 63.1 tok/s = 1.75×). Against MLX it leads on 13/15 (1.02–1.33×) but trails narrowly on Gemma-4-26B-A4B and Qwen3.6-27B (0.98×) — both run near the bandwidth ceiling where tensor cores confer no decode advantage.
- **Takeaway for latency.** Because time-to-first-token is dominated by prefill, the prefill gains translate directly into lower interactive latency for long-prompt and agentic on-device workloads.

## Authors & Credibility

Authors are from **Base Compute, Melbourne, Australia** (Fabian Waschkowski, Prabod Rathnayaka, Lukas Wesemann). This extends the group's prior BaseRT work (arXiv:2607.00501, "BaseRT: Best-in-Class LLM Inference on Apple Silicon via Native Metal"), so the system under test is a real, maintained runtime — **the code is publicly released at https://github.com/basecompute/baseRT** (BaseRT v0.1.6), which materially strengthens reproducibility. Credibility caveats: this is an **industry/vendor paper** (the authors sell the runtime) rather than an independent academic benchmark; evaluation is on a **single M5 Pro device only**; and the strongest headline numbers come precisely on the newest model families (Qwen3.5/3.6, Gemma 4) that were "newly supported as of BaseRT v0.1.6" — i.e. where the baselines may not yet be as well tuned as for long-supported models.

## Critical Assessment

The paper's central technical claim — that the M5's per-core Neural Accelerators are reachable only through the Metal 4 tensor API (not via a capability flag; the chip still advertises `MTLGPUFamily.apple9`) — is non-trivial and well explained, and the **workload-aware dispatch rationale is correctly grounded in the compute-bound vs memory-bound dichotomy** (prefill arithmetic intensity grows with sequence length; decode is M=1 GEMV). The decode results honestly *confirm* the bandwidth-bound prediction rather than overclaiming (BaseRT trails MLX by 0.98× on two large models), which is a sign of intellectual honesty. However, several specifics warrant scrutiny: (1) **single-device evaluation** — only M5 Pro (48 GB) is measured; the M5 base and Max tiers, which differ in core count and bandwidth, are explicitly deferred, so the headline speedups cannot be assumed to transfer; (2) **baselines at default settings** — llama.cpp (b9960) and MLX (0.32.0) are run with defaults, and no tuning sweep (e.g., batch sizes, KV layout, thread/group configuration) is reported, so part of the gap may reflect baseline under-tuning on brand-new hardware/APIs rather than pure kernel superiority; (3) the **newest model families** (Qwen3.5/3.6, Gemma 4) are exactly where BaseRT shows its largest margins and where baselines are least mature; (4) the paper measures **single-user, single-device inference only** — no continuous batching, no multi-request parallel decode, no tensor parallelism — so the relevance to any serving-style (even local multi-tenant) workload is untested; (5) decode, the metric that dominates steady-state interactive token cost, gains comparatively little from the headline contribution, and the authors themselves flag speculative decoding as the route to间接 decode gains, which is future work.

## Strengths

- Clean, well-motivated architectural insight: correctly identifies that M5 tensor cores help only compute-bound phases and builds a dispatch policy around it, rather than naively routing everything through matmul2d.
- Numerical-equivalence guarantee for prefill attention (exact online softmax between QK^T and PV on the tensor cores) avoids silently degrading quality.
- Transparent fallback story: same runtime, unchanged behaviour on pre-M5 chips via SIMD kernels — practical for a deployed runtime.
- Open-source release (BaseRT v0.1.6) with the actual version benchmarked, plus broad 15-config / 4-family coverage including 4 MoE models.
- Honest framing of decode: explicitly states decode gains come from dispatch/fusion, not the tensor path, and admits the two cases where MLX wins.

## Limitations

- Tensor cores do **not** raise the decode ceiling set by memory bandwidth; the headline contribution is prefill-only.
- Requires **M5-family hardware and a Metal 4 toolchain** to realise the gains; pre-M5 falls back with no speedup.
- Evaluation is on a **single M5 Pro device**; M5 base/Max behaviour (different core counts and bandwidth) is left for future work.
- Targets **single-device, single-user inference only** — no continuous batching, no multi-request parallel decoding, no tensor parallelism; **Metal only**.
- Baselines run at **default settings** with no tuning sweep, on hardware/APIs (Metal 4) and model families (Qwen3.5/3.6, Gemma 4) that are very new, leaving open how much of the gap is kernel superiority vs baseline immaturity.
- Decode-side wins, the metric most relevant to sustained interactive cost, are modest and partially owed to existing dispatch/fusion work rather than the new contribution.

## Relevance

对端侧 LLM 推理与 Apple Silicon 生态有直接且较高的相关性。本文是把 Apple M5 新引入的"每核 GPU 神经加速器 + Metal 4 张量 API"落地到真实推理运行时的早期工作之一，给出的结论（prefill 计算受限、张量核收益巨大；decode 内存带宽受限、张量核无收益）为后续所有想在 M5 上做推理优化的工作提供了清晰的工程指南。对做端侧 Agent / 长上下文 / 多模态的团队尤其有价值，因为 TTFT 由 prefill 主导，而 prefill 正是本文提升最大的环节。局限在于：仅 M5 Pro 单机、单用户、Metal 独占、基线为默认设置，因此若你的场景是 M5 Max、多请求批处理或跨平台，结论不能直接外推。总体属于"工程性强、可复现（开源）、新硬件首发红利"的一类论文，适合作为 M5 推理优化的基准参照。

## Questions for Further Exploration

- 在 M5 Max（更多核、更高带宽）和 M5 base 上，prefill/decode 的领先幅度如何变化？MoE 模型的收益是否会因带宽提升而相对收敛？
- 若对 llama.cpp / MLX 在 Metal 4 后端做针对性调优（batch、KV 布局、线程/分组配置），BaseRT 的领先会缩小多少？当前默认设置的差距中，有多少是"内核更优"而非"基线未调优"？
- 把 speculative decoding 的验证批 GEMM 走张量核路径，能否如作者预期那样间接抬高 decode 吞吐？实测增益有多大？
- 连续批处理 / 多请求并行解码 / 张量并行等 serving 式特性缺席，对本地多租户或高并发 Agent 场景意味着什么？BaseRT 的单用户最优是否在并发下被反超？
- prefill 的精确在线 softmax 与 SIMD 路径"数值等价"——在极长上下文（>>2048）下数值稳定性是否仍然成立？是否做过端到端生成质量评测而非仅吞吐？
- Metal 4 cooperative-tensor 的 threadgroup-staging 反直觉地不需要（直读 device memory 最快）——这一结论在更小 tile / 更低带宽档位芯片上是否依旧成立？
