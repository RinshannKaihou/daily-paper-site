> **一句话总结：** 本文提出"以知识为中心的自改进"范式——让Agent保持通用、一次性使用，把改进的全部负载放到一个由"任务级论坛 + 跨任务论坛 + 蒸馏"三阶段协议持续维护的共享知识库上，在ARC-AGI-1/2、Polyglot、SWE-bench Pro、Terminal-Bench 2五项基准上以更低美元成本取得更高求解率，且蒸馏出的知识可跨任务、跨LLM家族迁移。

> **TL;DR:** The paper inverts agent-centric self-improvement: agents are kept generic, stateless, and disposable, while the only evolving object is a curated shared knowledge base maintained by a three-stage (task-level forum → cross-task forum → distillation) protocol. On ARC-AGI-1/2, Polyglot, SWE-bench Pro, and Terminal-Bench 2, this beats agent-centric baselines (HyperAgents, Darwin Gödel Machine, Meta-Harness) and prompt optimizers (GEPA, OpenEvolve) on solve rate at lower dollar cost (e.g., ARC-AGI-1 86.7% vs HyperAgents 70%, $76 vs $234), and the distilled knowledge transfers zero-shot to held-out tasks and across Haiku↔GPT LLM families.

## 核心贡献

1. **提出并形式化"以知识为中心的自改进"范式**：与主流"以Agent为持续改进对象"（优化prompt/workflow/harness/Agent自身代码）相对，作者把Agent设为通用、无状态、一次性的worker，唯一持久改进的载体是一个共享、可检视、可压缩、可迁移的知识库。

2. **设计了一个刻意简单的三阶段知识策展协议**作为受控实验装置：
   - **Stage 1 任务级论坛**：每个任务一个线程，执行后的Agent发布基于本地证据的声明（什么有效/失败/需复核的假设、约束），并引用其他帖子相互支持或反驳。
   - **Stage 2 跨任务论坛**：把各任务级声明放到共享线程，提出跨任务复现的声明（常见错误类型、验证策略、不变量、环境假设、分解模式），后发帖须对先前声明明确表态（agree/disagree/synthesize），把"分歧"当作证据而非噪声。
   - **Stage 3 蒸馏**：将存活的本地声明与全局声明压缩成 typed bundle（字段：可迁移见解、已确认约束、被拒假设、陷阱、检查项、下一步），下一代 fresh agent 在行动前消费。蒸馏被设计为"为新任务量身的选择"而非通用摘要。

3. **实证支持该范式的三个论断**：
   - 知识策展在求解率与成本上同时优于Agent中心式自改进基线；
   - 同一协议在不同LLM家族下都有效；
   - 蒸馏出的知识库作为独立artifact，能零样本迁移到留出任务与不同LLM。

## Key Contributions

- **A controlled framework isolating curated knowledge as the sole improvement mechanism.** Agents are fixed as generic, stateless, disposable instances; the only changing object is the shared knowledge base. This lets the authors attribute any cross-generation gains to the curated knowledge rather than to agent specialization.
- **A three-stage curation protocol** (task-level forum → cross-task forum → distillation) that treats disagreement as evidence, preserves competing local hypotheses, and retains only actionable, scoped, evidence-grounded claims.
- **Empirical Pareto improvements** over agent-centric self-improvers (HyperAgents, Darwin Gödel Machine, Meta-Harness) and prompt optimizers (GEPA, OpenEvolve) across abstract reasoning, coding, and terminal benchmarks, at lower dollar cost.
- **Demonstration of genuine transfer**: the frozen generation-10 knowledge asset improves zero-shot solve rates on held-out tasks and remains effective when consumed by a different LLM family than the one that produced it.

## Methodology

- **Agent设计**：每个Agent是fresh实例（干净上下文），收到任务描述、标准工具访问（文件读写、shell执行、测试调用）、以及从共享知识库抽取的 seed bundle，单次会话内尝试任务并记录结果。无私有记忆、无身份、无专门化、无定制prompt/编排。
- **知识库三类artifact**：typed attempt table（每次尝试与结果）、forum posts（执行轨迹→证据化声明）、distilled bundles（压缩成可重用指导）。
- **生成式运行**：每项编码/抽象推理基准50任务池，每个未解决任务spawn一个agent；运行10代；已解决任务移出活跃池避免重复花费token；本方法跑3个seed。
- **Agent实现**：任务求解Agent用 off-the-shelf Anthropic Claude Agent SDK (Haiku 4.5) 与 OpenAI Agents SDK (GPT-5.4-mini, medium reasoning)；论坛与蒸馏阶段直接API调用同一LLM，不带agent工具。
- **关键设计意图**：因为Agent每代重新实例化，任何跨代性能提升只能来自其消费的累积与蒸馏知识——这使受控评估成为可能。
- **迁移评估协议**：donor LLM 在自改进split上产出冻结的 generation-10 知识资产；recipient LLM 在20个双方都失败的留出难任务上零样本消费该冻结资产（不跑新论坛/不蒸馏），用 task-conditioned adapter 把共享资产转为针对当前任务的简短备忘。

## Results & Highlights

- **ARC-AGI-1**：本方法 (Haiku 4.5) 86.7%±4.2，成本 $76±16；HyperAgents 70% / $234；OpenEvolve 54%；GEPA 44%。（求解率 +16.7pp，成本约为HyperAgents的1/3。）
- **ARC-AGI-2**：本方法 82.7%±6.1 / $80±1 vs HyperAgents 60% / $188（+22.7pp）。
- **Polyglot**：本方法 68.0%±2.0 / $126±6 vs HyperAgents 52% / $190 vs DGM 58% / $281；prompt优化器 GEPA 36%、OpenEvolve 46%。
- **SWE-bench Pro**：本方法 64.0%±2.0 / $208±19 vs HyperAgents 42% / $431 vs DGM 54% / $713（求解率高于DGM +10pp，成本约DGM的1/3.4）。
- **Terminal-Bench 2**：本方法 43.8%±3.4，优于 OpenHands 13.9%、Terminus 2 28.3%、Mini-SWE-Agent 29.8%、Terminus-KIRA 33.7%、Goose 35.5%、Meta-Harness 37.6%（Meta-Harness 用 Opus 4.6 提议器 + Haiku 4.5 base agent）。
- **跨LLM**：GPT-5.4-mini 在四项基准上均更高求解率更低成本（如 ARC-AGI-1 93.3%±7.0 / $16±3；Polyglot 72.7%±2.3 / $56±1）；Haiku runs 仍以更低成本超过所有Agent中心式基线。
- **零样本迁移（20留出难任务）**：Polyglot 上 GPT recipient 无知识 8.3%→+GPT-authored bundle 20.0%；ARC-AGI-1 上 GPT recipient 23.3%→43.3%；跨家族迁移双向均为正（如 Haiku-donor→GPT-recipient 在 ARC-AGI-1 上 23.3%→38.3%±12.6），证明资产携带donor无关的结构而非donor特定习惯。

## Authors & Credibility

全部作者来自 **Caltech**（Xuefei "Julie" Wang, Lauren Hyoseo Yoon, Chengrui Qu, Amanda Zichang Wang, Atharva Sehgal, Eric Mazumdar, Yisong Yue）。Yisong Yue 与 Eric Mazumdar 是知名资深学者。工作得到 NSF (#2505096, #2240110) 以及 OpenAI、Point72 的资助。代码开源在 https://github.com/recursive-knowledge/KSI 。基线（HyperAgents, DGM, Meta-Harness, GEPA, OpenEvolve）均在作者自己的评估协议下重跑，OURS 条目为3个seed的均值±标准差，DGM 仅在编码基准评估（标记 n/a 于ARC）。

## Critical Assessment

- **实验设计的核心强项**：通过把Agent"冻"成通用一次性实例，确实把改进归因隔离到知识库，这在自改进系统文献中是少见的干净对照。三阶段协议（尤其把分歧当证据、要求跨任务复现、蒸馏丢弃模糊建议）在机制上合理。
- **基线公平性有取舍**：基线为单次运行而 OURS 为3 seed均值；DGM/HyperAgents 的官方设定可能未完全对齐；Meta-Harness 的 Terminal-Bench 分数是 Opus 提议器+Haiku base，并非纯 Haiku——作者对此有标注，但仍使直接比较略显错配。
- **prompt优化基线偏弱**：GEPA/OpenEvolve 仅在 ARC-AGI-1 与 Polyglot 跑，且因其增益有限未扩展到更耗资源的基准；这使"优于prompt优化"的结论覆盖面有限。
- **迁移评估的小样本风险**：迁移用20个留出任务，部分单元格方差大（Haiku-donor→GPT 在 ARC-AGI-1 为 ±12.6），作者自己也说明把跨家族幅度读作"指示性而非精确"。
- **范围限制**：仅5个基准，未触及真正长视野/多步分解任务（作者在结论里承认这是未来方向）；蒸馏协议本身未被证明最优。
- **可复现性**：公开代码库 + 明确的SDK/LLM/超参附录是加分项。

## Strengths

- 范式清晰、可操作：把"什么是持续改进的对象"这一根本问题显式化，并以一个简单协议落地。
- 改进可检视、可压缩、可分享（externalized knowledge），不与某个Agent设计或某次自适应运行耦合。
- 双轴收益：求解率更高且美元成本更低（多基准上成本降到基线的约1/3）。
- 跨任务与跨LLM家族的迁移证据，支持"改进不是LLM/运行特定行为"的论断。
- 把分歧作为有用证据而非不稳定信号，是知识策展协议的亮点。

## Limitations

- 基线为单次运行 vs OURS 三seed均值，且部分基线（DGM、Meta-Harness）在模型/范围上不完全对齐。
- prompt优化比较仅覆盖两个基准，未扩展到更昂贵设定。
- 迁移评估样本量小（20任务），部分单元格方差高，跨家族幅度仅作指示性。
- 仅评估5个基准，未涉及长视野、多子任务分解的真实复杂场景。
- 论文明确声明该协议并非最优，蒸馏阶段的LLM依赖（同一LLM做蒸馏）可能引入偏置。
- 未纳入人类专家贡献（作者列为未来方向）。

## Relevance（相关性）

对"自改进Agent / Agent记忆 / 经验复用 / 知识管理"方向高度相关。其"以知识为中心"的视角为长期困扰该领域的"改进与特定Agent设计耦合、难以维护与迁移"问题提供了新解法。对构建可检视、可审计、可跨模型复用的Agent生态有直接工程价值；蒸馏bundle作为可迁移artifact的思路对RAG、记忆系统、agent workflow memory（Voyager, Agent Workflow Memory, AgentKB等）都是一种从"存储"到"策展"的范式升级。

## Questions for Further Exploration

1. 蒸馏bundle的质量是否会随代数增长而退化或冗余累积？是否需要显式的遗忘/压缩机制来防止知识库膨胀？
2. 跨任务论坛中"分歧→证据"的机制在任务分布高度异质时是否仍有效（例如纯数学证明 vs 终端运维）？
3. 把以知识为中心与以Agent为中心结合（例如用策展知识初始化一个随后自改进的Agent）能否进一步推高Pareto前沿？
4. 蒸馏阶段使用同一LLM是否会引入系统性偏置？引入第三方/更强蒸馏模型能否提升bundle的可迁移性？
5. 在真正长视野、需子任务分解的任务上，子组件间的知识策展是否会成为新的研究单元（论文结论暗示的方向）？
6. 迁移评估中部分高方差单元格（如 Haiku-donor→GPT）背后是否暗示知识资产存在 donor-specific 与 donor-agnostic 两类成分，能否显式分离？
