§ 01

Methodology

Ontology + bounded evidence
  1. Phenomenon identification SKILL.md directories (metadata + instructions + optional scripts/, references/, hooks, manifests, tests) are now acquired, installed, updated, and removed across many independent agent systems.
  2. Ontology construction Distinct analytical objects — Behavioral Source → Skill Artifact → Skillware Unit → Agent Host → Runtime → Execution Trace → Outcome — with an execution-semantics equation that assigns no deterministic meaning to unrestricted natural language.
    π = R_H(S, τ, M, C, T, P, σ)
  3. Membership formalization Three necessary conditions (C1 primacy, C2 identity, C3 Host execution) must converge on the same unit; Lifecycle Continuity is reported separately and never scores quality or popularity.
  4. Evidence with separate inference limits Spec + corpus (narrow envelope signals), fixed-revision cases pinned to commits, and independent studies — each with its own inference boundary, so word frequency never implies semantic correctness.
  5. Engineering-continuity test Classic SE concepts (module boundary, component identity, lifecycle, design patterns) must remain applicable; pattern transfer is admitted only when 7 elements align — from source intent to misuse discriminator.

Membership & Structure

C1 Behavioral primacy — the Skill organizes a reusable task behavior.
positive observable · exclusion if behavior lives in app code
C2 Independent identity — acquirable / installable / trackable outside any one Host.
name · address · package · version · compatibility · provenance
C3 Agent Host execution — ≥1 compatible Host (Codex, Claude Code, Copilot, Hermes, OpenClaw …) can discover / activate / load it.
LC Lifecycle Continuity — same unit identity across update / maintenance / rollback / removal.
software-level property, not a membership condition

§ 03

Strengths & Limitations

side-by-side

Strengths

  • Three cleanly separated concerns. Splits what the abstraction is (ontology) from how it changes (lifecycle) and from situated performance (one run under one model/context).
  • Reproducible boundary method. Declared unit → frozen revision → positive/counterevidence/limitations for C1–C3, with LC coded separately — disagreements become traceable.
  • Genuine epistemic humility. Per-condition inference limits (Table 3) and a threats-to-validity section that names collection bias, purposive sampling, and construct validity.
  • Bridges to deep SE tradition. Connects agent skills to Parnas modules, Szyperski components, ISO 12207, Lehman laws, and GoF/POSA/Evans patterns.

Limitations

  • No utility validation. No experiment shows classification or the 6-step workflow improves task success, reuse, or defect rate; FDE comparison is future work.
  • C1 construct validity under-tested. No inter-rater reliability, no independent coding — different analysts may draw different unit boundaries.
  • Corpus is lexical/metadata only. Registry-heavy collection, aliases, forks, generated artifacts limit external validity.
  • Pattern transfer is partly constructive. Some mappings are author-built fixtures, not discoveries — ecosystem frequency stays open.
  • Adaptation-Enabled has no complete example. 0/13 units satisfy it; evolution pillars rest on partial precedents.
  • Cross-Host parity out of scope. Source inspection cannot prove behavioral equivalence across Codex / Claude Code / Copilot / Hermes / OpenClaw.
§ 04

Questions for Further Exploration

open agenda
1Inter-rater reliability of C1–C3. Do independent coders reproduce the authors' inclusions/exclusions on the same 15 + 13 cases?
2Cross-Host behavioral parity. Holding unit/task/model/toolset fixed, how much does situated performance vary across Codex vs. Claude Code vs. Copilot vs. Hermes?
3Does Engineering Consolidation happen? Over N releases, do instructions migrate into resources / deterministic kernels along a predictable trajectory?
4Self-Evolution under identity preservation. Can a complete Adaptation-Enabled unit close the loop — generate, evaluate, review, version, release, rollback — and beat human-only maintenance?
5Pattern frequency in the wild. Across the 138K corpus, how often do Facade / Adapter / Composite / Observer structures actually occur vs. only in fixtures?
6Security surface validation. How exploitable are prompt injection via references/, companion-session hijacking, and adaptive supply-chain attacks?
7FDE field trial. In a real Forward Deployed Engineering setting, does Skillware-centered delivery beat code-first on time-to-first-useful-deployment and reuse?
8Skillware vs. MCP boundary. Where should the line be drawn when a unit's value splits between natural-language orchestration and callable tools?
