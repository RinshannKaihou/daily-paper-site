> **一句话总结：** EvoDRC 是首个面向模块级 (block-level) DRC 违例修复的智能体框架，通过跨设计技能迁移、布局分区分解与迭代式技能进化，在 DAC26-DRC-Benchmark 七个模块上平均降低违例 83.6%（相比基线的 73.5% 报告值），可完全清洗部分设计。

> **TL;DR:** EvoDRC is the first agentic framework for block-level post-route DRC violation repair; by transferring layer-specific repair skills from an unrelated reference design (CLA), decomposing large layouts into bounded repair crops, and evolving skills via a Knowledge DB, it achieves an 83.6% average DRV reduction across seven DAC26-DRC-Benchmark blocks (vs. the 73.5% reported baseline), fully cleaning Block3 and Block5.

## 核心贡献

- **首个模块级 agentic DRC 修复框架**：自动化传统手工的 post-route DRC ECO 流程，使布局工程师从繁琐的违例修复中解放出来。据作者所知，此前 agentic DRC 修复仅在标准单元级小规模任务上验证过，无法扩展到含数百违例的模块级设计。
- **将 DRC 修复形式化为「技能进化」任务**：结合来自参考设计 (CLA) 的可迁移初始技能与目标设计上经核实的修复结果，持续演化各层修复知识。
- **布局分区分解机制 (Layout Decomposition)**：通过 DRV 分布裁切一阶/二阶 crop 并按行合并，将大模块切分为有界 repair crop，解决单 agent 覆盖能力不足的扩展性瓶颈。
- **可追溯的 Knowledge DB**：仅记录每个 agent 的最终修复操作及其对应的局部 DRV 分布变化，丢弃中间试探痕迹，避免噪声累积并保持技能与结果的可追溯连接。

## Key Contributions

- **First agentic framework for block-level post-route DRC ECO repair**, automating a traditionally manual, multi-day engineering task; prior agentic work only handled standard-cell-scale cases.
- **Skill-evolution formulation of DRC repair**: layer-specific repair skills are seeded by distilling knowledge from transforming an unrelated DRC-violating reference design (CLA) into a DRC-clean design, then continuously refined using verified repair outcomes from the target design.
- **Layout decomposition mechanism** that classifies routing types, builds first-/second-order crops bounded to k μm × k μm, and merges row-aligned crops — essential for scalability on designs with up to 765 DRVs.
- **Traceable Knowledge DB with a dual-path skill-evolution pipeline**: the Skill Refiner (incremental refinement) and DB Summarizer (full-history reconstruction) generate candidate skill files, validated by a mechanical script checker and adjudicated by a Skill Judge agent.

## Methodology

EvoDRC 由三个阶段构成（Fig. 2）：

1. **Layout Decomposition**：解析设计并分类 routing types；按 DRV 分布构建一阶 crop（触及违例的金属线/via 扩展），限制在 2 μm × 2 μm；按 x/y 与金属层重叠合并为二阶 crop；再按标准单元行高合并为最终 repair crop，多行 crop 保持独立。

2. **Agentic Repair**：每个 repair crop 分配一个修复 agent，输入包括 DRM 图像、KLayout Python 脚本、DRV 分布与各层 skill 文件。对象按可编辑性分四类（A: via 结构；B: 区域内完整 polygon；C: 被 crop 裁切的金属线端；D: 标准单元/其他 routing 类型，禁止修改）。提供三种 preview tool：Local DRC analysis（沙箱快速 DRC）、Connectivity check（全设计连通性，禁止破坏网连接）、Impact preview（受影响的邻近对象）。agent 采用 ReAct 范式，内部执行 planner / adversarial reviewer / DRC engineer / patch reviewer 四角色工作流。

3. **Assess & Record + Skill Evolution**：仅当操作通过全设计连通性检查才被接受；对同一生成 via 的冲突操作分别测试取净 DRV 降幅最大者。按金属层分组记录操作-DRV 分布变化到 Knowledge DB。随后两个 agent 独立生成候选 skill 文件：Skill Refiner 增量精修，DB Summarizer 从完整历史重构。候选经三项机械检查（Source check、Claim citation、Ambiguity check）后，由 Skill Judge 按 Reference check / Coverage / Writing quality 三指标多数表决，决定是否替换迭代 t+1 的 skill 文件。

实现细节：使用 Anthropic Claude Sonnet 4.6，reasoning effort 设为 medium；最多 5 次迭代，零违例提前停止；crop 限制 2 μm × 2 μm；DRC 用 KLayout 0.30.1 + ASAP7 rule deck。

## Results & Highlights

- **整体修复质量**：在 DAC26-DRC-Benchmark 七个模块上，EvoDRC 平均 DRV 降低 **83.6%**，而论文标题与摘要中报告的对比基线为 **73.5%**；每个模块至少降低 **55.3%**（Block1），并完全清洗 Block3（89→0，-100%）和 Block5（68→0，-100%）。最大模块 Block7（765 初始 DRV）从 765 降至 275（-64.1%，移除 490 个违例）。
- **消融验证三大机制**：Ablation 1（移除 CLA 技能迁移）平均仅 51.0%，证明初始可迁移技能的价值；Ablation 2（冻结技能）在六个未完全清洗的模块上全部低于 EvoDRC，Block7 残留 358 vs. EvoDRC 的 275；Ablation 3（移除布局分解）在 Block7 仅移除 81 个（vs. EvoDRC 的 490），单 agent 无法覆盖大规模违例——平均 73.0%。
- **LLM 成本可负担**：全部七个模块总成本 **$1152.23**（按 Sonnet 4.6 定价），其中 Repair agent 占 86.4%（$995.33），技能进化相关 agent（Skill Refiner 5.4% + DB Summarizer 6.3% + Skill Judge 1.9%）合计约 13.6%；最大单模块 Block7 为 $458.89。相比人工修复数百违例需数天，成本极低。
- **技能进化动态**：迭代 1 后 DB Summarizer 几乎赢得所有模块（首次获得目标设计证据）；从迭代 3 起 Skill Refiner 占主导（已在含目标知识的 skill 上增量精修）。Block7 的技能 situation 覆盖从 25 增长到 108，对应其最大最 diverse 的违例集（Table 8）。
- **非贪婪探索的鲁棒性**：Block5 在迭代 2 引入 13 个 M5.AUX.3 新违例（38→39），迭代 3 即清除 90% 余量；所有中间 DRV 增加均在两轮内恢复，得益于 Knowledge DB 记录副作用而非走贪婪路径（Table 9）。

## Critical Assessment

论文的核心论点——「跨设计技能迁移 + 迭代技能进化 + 布局分解」三者缺一不可——通过三个精心设计的消融实验得到充分支持，且消融结果与机制解释高度一致（如 Ablation 3 在 Block7 仅修复 81/765 清晰说明单 agent 覆盖瓶颈）。技能进化的可追溯设计（仅存最终操作、机械检查强制引用与去歧义、Skill Judge 多数表决）是对 LLM agent 不可靠输出的工程化约束，体现了对「盲目信任 agent 输出」风险的清醒认识。

但需注意几点：第一，摘要/标题宣称的「73.5% compared to the reported baseline」措辞模糊——文中未明确该基线是 DAC26-Benchmark 自带 agentic workflow 还是其他；而 EvoDRC 自身平均为 83.6%，73.5% 实为 Table 5 中 Ablation 3（无分解）对 Block2 的数值或对报告基线的引用，建议读者直接以 83.6% 为 EvoDRC 的主结果。第二，最大模块 Block7 仍残留 275 个 DRV（-64.1%），说明在最大规模、最 diverse 规则集上仍未接近闭合，且 V2.M3.AUX.2 类违例在 Block7 全程卡在 225/153 直到迭代 4 才协调清除 72，暴露出对特定 cross-layer 规则的修复仍弱。第三，成本仅报告 LLM API 费用，未含 KLayout DRC 全芯片运行的时间/算力开销，而迭代式全设计 DRC 在真实先进节点 sign-off deck 下可能非常昂贵。第四，连通性检查是简化的 DFS 而非 LVS，无法替代 sign-off 逻辑验证。

## Strengths

- 首次将 agentic DRC 修复从标准单元级扩展到模块级（最高 765 DRVs），填补了重要空白。
- 技能进化范式优雅地模拟了真实 DRC 工程师「跨设计复用 + 针对新设计精修」的工作方式，且每条技能可追溯到具体修复记录。
- 布局分解 + 并行 repair crop 是朴素但有效的扩展性方案，消融 3 量化证明了其必要性。
- 工程严谨：可编辑对象四分类、preview tool 三件套、机械检查三道关、Skill Judge 多数表决，层层约束 LLM 不可靠性。
- 完全开源（代码、skill 文件示例、benchmark），可复现性强。

## Limitations

- 最大模块 Block7 仍有 275 个残留 DRV（仅 -64.1%），离 DRC closure 有距离，尤其 V2.M3.AUX.2 等 cross-layer 规则修复力弱。
- 73.5%「报告基线」的来源在正文中未充分界定，存在表述歧义。
- 成本核算未包含 KLayout 全芯片 DRC 运行的实际时间与算力，仅 LLM token 费用；在真实 sign-off deck 下该部分可能成为瓶颈。
- 连通性检查为简化 DFS，非 LVS，无法保证 sign-off 级电路正确性。
- 仅在 DAC26-DRC-Benchmark（ASAP7 PDK）上评估，未覆盖其他工艺节点或商业 PDK 的规则复杂度。
- 最多 5 次迭代为人为上限，对 Block7 等大规模设计是否充分未做敏感性分析。
- skill 文件的机械检查只保证「可追溯」而非「事实正确」，错误但自洽的技能可能被保留。

## Relevance

本研究对 EDA / VLSI 物理设计社区具有直接价值：DRC closure 是先进节点流片的关键瓶颈，业界报告显示迭代式 DRC 收敛可能耗时数天至数周。EvoDRC 证明了 LLM agent 配合可进化技能库可在模块级自动化这一手工流程，为「AI for EDA」在物理验证落地提供了可扩展范式。其技能迁移 + 进化 + 任务分解的思路也可迁移到其他需要跨实例复用专家经验、且单次推理覆盖能力不足的 agentic 任务（如时序修复、IR drop 修复）。对研究 agentic self-improvement、tool-augmented reasoning、知识库治理的读者亦有方法论参考价值。

## Questions for Further Exploration

- Block7 上 V2.M3.AUX.2 / V4.M5.AUX.2 等生成 via 类违例为何长期难以清除？是 skill 知识缺失、操作原语不足，还是 crop 分割割裂了跨层协调？
- 增加迭代次数（如 10 轮）能否进一步收敛 Block7，还是会在某层技能饱和后停滞？
- 若用 sign-off 级商业 DRC deck 替换 ASAP7 + KLayout，技能迁移与修复质量会如何退化？规则复杂度上升是否超出当前操作原语表达力？
- skill 文件机械检查只保证可追溯不保证正确，能否引入形式化验证或反例驱动的技能证伪机制？
- Knowledge DB 的跨设计复用能否扩展到多参考设计蒸馏，而非单一 CLA？不同参考设计的初始技能质量差异有多大？
- 连通性检查升级为完整 LVS 后，接受率与修复质量如何变化？是否会暴露当前 DFS 漏检的逻辑破坏？
- 技能进化过程是否存在「遗忘」（早期有效技能被后续 Skill Judge 替换丢失）？能否引入技能版本库与回滚？
