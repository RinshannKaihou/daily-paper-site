Knowledge-Centric Self-Improvement

Xuefei (Julie) Wang, Lauren Hyoseo Yoon, Chengrui Qu, Amanda Zichang Wang, Atharva Sehgal, Eric Mazumdar & Yisong Yue  ·  Caltech

self-improving-agents knowledge-curation agent-benchmarks llm-agents knowledge-transfer
Key Finding

Keep agents generic and disposable; the only thing that improves is a curated shared knowledge base — winning more solves at lower cost.

Key Metric
86.7%

ARC-AGI-1 solve rate at $76 vs HyperAgents' 70% at $234 — roughly a third of the cost.

Why This Matters

Decouples improvement from any specific agent design — the evolving artifact is inspectable, compressible, and transfers zero-shot across tasks and LLM families.

Critical Edge

Baselines ran once while the method averaged three seeds, and DGM & Meta-Harness used mismatched models — so headline gaps may overstate the true margin.

The Mechanism

A three-stage curation loop treats disagreement as evidence, not noise:

task forum → cross-task forum → distillation

02

Methodology

Each agent is a fresh, stateless, disposable instance — clean context, standard tools (file I/O, shell, test calls), plus a seed bundle drawn from the shared knowledge base. No private memory, no identity, no specialization, no bespoke prompt or orchestration. Because agents are re-instantiated each generation, any cross-generation gain can only come from the accumulated, distilled knowledge they consume.

STAGE 1
Task-level forum

One thread per task. Post-execution agents file evidence-grounded claims (what worked / failed / needs rechecking) and cite each other.

→
STAGE 2
Cross-task forum

Claims meet on a shared thread. Late posts must take a stance (agree / disagree / synthesize) on earlier ones — disagreement is treated as evidence.

→
STAGE 3
Distillation

Surviving claims compress into a typed bundle (insight, confirmed constraints, rejected hypotheses, traps, checks, next step) sized for the new task.

Generative run

50-task pool per coding / reasoning benchmark; an agent is spawned per unsolved task; 10 generations; solved tasks leave the active pool to avoid duplicate spend; 3 seeds for the proposed method.

Transfer protocol

Donor LLM produces a frozen generation-10 asset on the self-improvement split. Recipient LLM consumes it zero-shot on 20 jointly-failed hard tasks (no new forum, no distillation), via a task-conditioned adapter that turns the shared asset into a short memo.

03

Strengths & Limitations

Strengths

  • Paradigm is explicit and operational: forces the question "what is the object of improvement?" and lands it in a simple protocol.
  • Knowledge is externalized — inspectable, compressible, shareable — not coupled to one agent design or run.
  • Dual-axis win: higher solve rate and lower dollar cost (costs fall to ~1/3 of baselines on several benchmarks).
  • Genuine transfer across tasks and LLM families supports "improvement is not LLM- or run-specific."
  • Disagreement as evidence is a thoughtful curation primitive, not a stability nuisance.

Limitations

  • Baseline fairness: single-run baselines vs 3-seed mean; DGM / Meta-Harness mismatched on model and scope.
  • Prompt-optimizer comparison covers only two benchmarks and wasn't extended to costlier settings.
  • Small transfer sample (20 tasks); some cells high-variance (Haiku→GPT ±12.6); cross-family magnitude is indicative only.
  • Only 5 benchmarks — no genuinely long-horizon, multi-subtask decomposition scenarios.
  • Distillation uses the same LLM (possible systematic bias); protocol not proven optimal; no human-expert contributions yet.
04

Questions for Further Exploration

  1. Does bundle quality degrade or accumulate redundancy over generations — is an explicit forgetting / compression mechanism needed to curb knowledge-base bloat?
  2. Does "disagreement-as-evidence" survive highly heterogeneous task distributions (e.g., pure mathematical proof vs. terminal operations)?
  3. Could combining knowledge-centric and agent-centric views — seeding a subsequently self-improving agent with curated knowledge — push the Pareto frontier further?
  4. Does using the same LLM for distillation introduce systematic bias? Would a stronger / third-party distillation model improve bundle transferability?
  5. On truly long-horizon, decomposable tasks, does knowledge curation between subcomponents become the new research unit (as the conclusion hints)?
  6. Do high-variance transfer cells (Haiku-donor → GPT) imply separable donor-specific vs donor-agnostic components in the asset — and can they be split explicitly?
