Two-stage clustering with provable guardrails cuts 38M-customer LLM cost 50× — 10–1000× faster than baselines.

Efficient Clustering with Provable Guardrails for LLM Inference at Scale

Longshaokan Wang, Wai Tsang Keung, Punit Ghodasara, Roman Wang, Ali Dashti, Francesc Moreno-Noguer — Amazon

provable-guardrails clustering set-cover llm-inference recommendation-systems scalability

Four assets

▎ Key finding

A Mini-batch K-Means seed plus a greedy set-cover representative step provably enforces minimum cosine similarity & exact attribute match, cutting inference cost ~50×.

▎ Key metric

50× / cost + latency

38M-customer A/B (2025-06, α=0.77): LLM compute $1.13M → ~$22.5K, relevance drop only 0.7pp.

▎ Why this matters

Clusters embeddings before LLM invocation — collapsing per-sample calls to per-cluster calls, an orthogonal cost lever to quantization and serving.

▎ Critical edge

The guardrail holds by construction, and the headline 10–1000× speedup is measured only at 100K (where baselines are infeasible), not at the 38M deployment scale.

Methodology

  • ProblemDataset D={(Pᵢ, Aᵢ)}: embed text Pᵢ → Eᵢ, compare by cosine. Learn an assignment so every sample maps to a representative j with sim(Eᵢ,Eⱼ) ≥ α and Aᵢ = Aⱼ. Relation ∼α is reflexive, symmetric, but not transitive — the source of difficulty.
  • Stage 1Mini-batch K-Means on {Eᵢ} yields K initial clusters {Cₖ}. The guardrail holds for any initial partition (even K=1); Stage 1 only changes final cluster count and runtime.
  • Stage 2Per cluster: greedy Johnson–Chvátal set-cover over α-balls. Repeatedly pick the representative covering the most uncovered samples. Bound: |Rₖ| ≤ OPTₖ·(1 + ln|Cₖ|).
  • ReassignOptional Alg. 2 re-points each sample to its most-similar feasible rep — strictly raises avg within-cluster similarity while keeping guardrails and |R| fixed.
  • ComplexityTime O(nd + n²d/K), memory O(nd + n²/K²). With K = Θ(n) both go linear in n, enabling n ∼ 10⁷.
  • Scalability knobThe quadratic pairwise work is confined to each ≈ n/K cluster. Larger K linearly cuts time, quadratically cuts memory — at a slight cost in reduction efficiency.
  • Embeddingall-MiniLM-L6-v2 (SentenceTransformer), chosen because Stage 1 needs Euclidean distance and Stage 2 needs cosine similarity from the same space.
  • BaselinesK-Means, Mini-batch K-Means, Agglomerative, BIRCH, Spectral, GMM — scikit-learn 1.7.2 defaults, cluster counts aligned to match data-reduction ratio.
  • HardwareSingle r7i.12xlarge, no parallelism — all wall-clock figures are single-machine.

Strengths & limitations

Strengths

  • Guardrails are exact by construction and auditable — citable for compliance/safety (e.g. child choking-hazard avoidance as a hard attribute constraint).
  • Complete scalability story: O(nd) K-Means → confine O(n²) to clusters → K as an explicit, tunable knob.
  • Revenue-grade deployment: 38M customers, ~50× LLM cost/latency cut, statistically significant A/B win.
  • Right-skewed cluster sizes are exploited as a feature — greedy naturally yields a heavy tail enabling aggressive tail-trim.
  • Robust across 12 batch×init×K combos: cluster count varies only ~50% and Min. Sim. stays exactly 0.75 throughout.

Limitations

  • The (1 + ln|Cₖ|) bound is on data-reduction efficiency, not quality, and only vs the partition-limited optimum.
  • Cosine similarity is a proxy for relevance (r = 0.781 / 0.797); ~39% of variance unexplained.
  • The 100K benchmark scale is dictated by baselines being infeasible beyond it — the 10–1000× figure cannot be directly extrapolated to 38M.
  • Acutely sensitive to α and the attribute set; deployment needs heavy offline calibration.
  • Assumes discrete, low-cardinality categorical attributes; continuous/high-cardinality cases untested.
  • Tail-trim needs upstream oversampling to compensate dropped users — a hidden cost, and a potential fairness risk.

Questions for further exploration

  1. Could a learned output-consistency similarity (a small model predicting representative–member LLM-output divergence) lift the 0.781 correlation and turn the "similarity guardrail" into a true "output guardrail"?
  2. Can representatives be synthesized rather than drawn from real samples — learning a constrained embedding with sim≥α and matching attributes — to compress clusters further?
  3. Is there a computable upper bound on the gap |R*partition| − |R*global| from initial-clustering quality (e.g. silhouette), giving theoretical guidance for K instead of only empirical robustness?
  4. For high-cardinality attributes (fine SKU taxonomy, geo), can LSH / inverted indices take the O(nₖ²·a) attribute match down to near-linear?
  5. If the trimmed tail over-represents protected subgroups, does recommendation quality become unevenly distributed — and can a fairness constraint be folded into the guardrail?
  6. Can Stage-2 greedy set-cover be made incremental / online, so drifting personas (over 38M customers) don't force periodic full re-clustering?
Bottom line — The contribution is not a new clustering idea but the rare combination of per-sample exact similarity + exact attribute guardrails + linear scalability. In production on 38M Amazon customers it turned a $1.13M / 508-day LLM pipeline into ~$22.5K / 10 days with only a 0.7pp relevance cost — a deployable, auditable template for "cluster before you call the LLM."
