# How Fast Can Reward Models Score? A Systems Study of C++ and PyTorch Inference Runtimes for RLHF

> **一句话总结：** 作者用 ONNX Runtime 构建了一个原生 C++ 奖励模型推理引擎，发现在 CPU 上它全面碾压 PyTorch eager/`torch.compile`/FastAPI 三种基线（提速约 1.7–1.9×，置信区间完全不重叠），但在 GPU 上 `torch.compile` 反而更快；进一步消融证明提速真正来自 ONNX Runtime 图执行模式而非 C++ 语言本身，而把请求按长度分桶（而非朴素 padding）在 GPU 上能带来约 35% 的吞吐增益，多实例并发在 GPU 上还会因 OOM 全部失败。

> **TL;DR:** The authors build a native C++ reward-model inference engine on ONNX Runtime and benchmark it (on Anthropic's hh-rlhf, 60-row eval set, DeBERTa-v3-large + Electra-large) against HF eager mode, `torch.compile`, and FastAPI. On CPU the C++ engine beats every baseline (p50 335.9 ms vs 581.6–628.8 ms, ~1.7–1.9×, CIs non-overlapping). On GPU `torch.compile` wins at both p50 (19.0 vs 27.4 ms) and p95 (25.6 vs 116.2 ms). An isolation test shows the CPU win comes from ONNX Runtime, not C++ (Python ONNX ≈ 349 ms vs C++ 335.9 ms), and length-aware batching (not naive padding, not concurrency) is the only real throughput lever on this hardware.

## 核心贡献

1. **首个针对 RLHF 奖励模型打分阶段的系统性推理后端基准**：填补了既有系统文献几乎只关注生成（generation）而忽略打分（scoring）的空白。在 DeBERTa-v3-large 与 Electra-large-discriminator 两个奖励模型上，对比了 HF eager、`torch.compile`、FastAPI、Python ONNX Runtime 与自研 C++ ONNX 引擎共 5 条路径。
2. **纠正"快是因为 C++"的直觉**：通过把同一个 ONNX Runtime 会话分别用 Python 与 C++ 调用，证明 CPU 提速来自图执行（graph execution）而非语言本身（349 ms ≈ 335.9 ms），团队无需写 C++ 即可拿到几乎全部加速，只需停止使用 eager 模式。
3. **揭示 GPU 上 `torch.compile` 反超专用 C++ 引擎**：在 p50 与 p95 两个口径下 `torch.compile` 均以严格不重叠的置信区间胜出，推翻了"静态 ONNX 图能更好地抑制尾延迟尖峰"的假设。
4. **量化"朴素 padding 是负优化"**：朴素分批（按最长样本对齐）在 CPU 上损失 5–8× 吞吐、在 GPU 上损失 3.5–4×；按长度分桶只在 GPU 上能挽回损失，CPU 后端因缺乏批并行性无论怎么分桶都不超过 batch=1 基线。
5. **方法论上的严谨性示范**：所有指标均来自多次独立进程启动（5 次重复，CPU 基线 3 次）的均值与 95% 置信区间，并用 Welch's t 检验（全部 p < .001）佐证；并诚实标注了一处未能复现的 Electra/DeBERTa 比较结果。

## Key Contributions

- A first-of-its-kind systems benchmark isolating **reward-model scoring latency** in RLHF (as distinct from generation), measured on DeBERTa-v3-large and Electra-large-discriminator reward models over Anthropic's hh-rlhf data.
- A native C++ inference engine on ONNX Runtime that takes over tokenization, batching, and post-processing, verified for numerical parity (max abs error 5.7×10⁻⁶ on CPU, 4.2×10⁻³ on GPU).
- An isolation ablation proving the CPU win is attributable to **ONNX Runtime's graph execution, not the C++ language** — Python ONNX Runtime (≈349 ms) statistically ties the C++ engine (335.9 ms), both far below HF eager (602.4 ms).
- A counter-intuitive **GPU result**: `torch.compile` beats the dedicated C++ engine at both p50 (19.0 vs 27.4 ms) and p95 (25.6 vs 116.2 ms), invalidating the "static graph = safer tail latency" hypothesis.
- Quantified evidence that **naive padding is actively harmful** (5–8× CPU, 3.5–4× GPU throughput loss) and that length-aware bucketing only pays off on GPU; and that **concurrency does not scale** throughput (≤11% gain shared; multi-instance is worse and OOMs on GPU).

## Methodology

- **Engine & models.** A custom C++ engine wraps ONNX Runtime 1.26.0 and implements native tokenization (SentencePiece for DeBERTa, WordPiece for Electra), batching, and post-processing — no Python calls at inference time. Two transformer-encoder reward models tested: OpenAssistant's DeBERTa-v3-large (primary) and Electra-large-discriminator (generalization check).
- **Correctness validation.** Identical inputs through both engines; max abs reward error 5.7×10⁻⁶ on CPU, 4.2×10⁻³ on GPU (the GPU gap is expected FP accumulation-order noise, not a bug). A real Electra export bug (missing `token_type_ids`) was found and fixed, dropping its error to 2.9×10⁻⁵.
- **Baselines.** HF Transformers eager mode, `torch.compile` (default inductor backend), and a FastAPI service wrapping the PyTorch model; plus a Python-calls-ONNX-RT isolation control.
- **Statistical method.** Every number is a mean with 95% CI across **independent process launches** (5 launches; 3 for CPU baselines due to `torch.compile` recompile cost). Within a launch, p50/p95 are computed per-row then averaged across launches (a "mean of medians"). A two-sample Welch's t-test (unequal variances) confirms all headline comparisons at p < .001. CIs must not overlap for a result to count as real.
- **Dataset.** Anthropic hh-rlhf; primary eval set = 60 rows (fixed seed, length-stratified). Robustness: two additional 60-row seeds + one 150-row set (~2.5×). No formal power analysis; reasoning checked empirically.
- **Batching & concurrency sweep.** Batch sizes 1/2/4/8, naive vs length-bucketed, across 4 systems on CPU and C++-only on GPU. Concurrency 2/4/8 (plus single-run checks at 5/6/7 for GPU memory ceiling) for shared vs per-request multi-instance, on both models, both devices, plus real-HTTP FastAPI.
- **Hardware/software.** Single dev machine: AMD Ryzen 7 5800H, NVIDIA RTX 3060 Laptop GPU (6144 MiB VRAM). fp32 throughout. ONNX Runtime 1.26.0 (CPUExecutionProvider / CUDAExecutionProvider), PyTorch 2.10.0 (+cu126), CUDA 12.6, driver 592.00.

## Results & Highlights

1. **CPU latency (p50, Table 1).** The C++ engine scores 335.9 ms [297.7, 374.0] vs HF eager 602.4 ms [551.3, 653.4], FastAPI 581.6 ms [570.4, 592.9], `torch.compile` 628.8 ms [613.7, 644.0] — a ~1.7–1.9× point-estimate win, with **zero CI overlap** against all three baselines; even the most conservative CI-bound comparison still shows ≥1.4×.
2. **GPU latency (p50, Table 1).** C++ engine 27.4 ms beats HF eager (57.2 ms) and FastAPI (62.8 ms) with no CI overlap, but **`torch.compile` wins at 19.0 ms [17.6, 20.3]** (also non-overlapping).
3. **GPU p95 tail latency (Table 2).** `torch.compile` 25.6 ms vs C++ engine 116.2 ms — the gap actually *widens* at the tail, refuting the "static ONNX graph is safer against spikes" theory.
4. **Source of the CPU win (§4.2).** Same ONNX session called from Python: ~349 ms ≈ C++ 335.9 ms — essentially tied, both beating HF eager by a wide margin. The native C++ tokenizer is 3.8× faster than Python AutoTokenizer (64.3 µs vs 245.8 µs) but negligible vs a full forward pass. Zero-copy buffers and preallocation showed **no measurable effect** under repetition.
5. **Batching (Tables 4 & 5).** Naive padding costs **5–8× CPU throughput** and **3.5–4× GPU** vs batch=1. Length-bucketing recovers the loss on GPU only (C++ GPU bucketed batch=8 = 53.7 rows/s vs naive 10.0; ~35% over batch=1's 39.8) and plateaus on CPU (≤3.10 rows/s regardless of batch size, since the CPU backend folds the batch into one larger sequential matmul).
6. **Concurrency (Tables 6 & 7).** Shared-instance throughput grows ≤**11%** from concurrency 2→8 across all device/model combos; multi-instance is *worse* (CPU thread-pool oversubscription; GPU ~19× slower for DeBERTa at conc=2) and **OOMs every one of 5 launches at conc=8** on the 6 GiB card — the real ceiling is ~4–5 sessions. Real-HTTP FastAPI throughput stayed flat at 0.53–0.65 rows/s from conc 1→8.
7. **Generalization (Table 8).** ONNX Runtime beats eager for Electra just as for DeBERTa (direction holds); however the magnitude (4.7× vs 1.7×) **did not reproduce** on a fresh run, and the authors flag this one row honestly rather than silently editing it.
8. **Statistical robustness (§4.6).** Two alternative seeds give C++/eager ratios of 1.79× and 1.73×; the 150-row set reproduces all core findings within noise.

## Authors & Credibility

The three authors (Venkata Naga Sai Vishnu Rohit Pulipaka, Anish Katta, Deva Rohit Reddy Peddireddy) appear to be independent researchers (Gmail affiliations, no institution listed). Credibility signals are strong despite the lack of institutional branding: (a) the **full engine, benchmark harness, and analysis scripts are open-sourced** at https://github.com/vishnup22/reward-model-benchmarks, with ONNX Runtime pinned identically across C++ (CMake FetchContent) and Python (pyproject.toml); (b) the methodology explicitly follows the rigorous benchmarking discipline of Mytkowicz et al. (ASPLOS'09) and Georges et al. (OOPSLA'07) — independent process launches, CI non-overlap as the threshold of reality, and Welch's t-tests; (c) the paper is unusually self-critical, publicly retracting a result that did not reproduce (the Electra 4.7× claim) and reporting a genuine export bug they found and fixed. Limitations are stated plainly (single machine, 6 GiB GPU, fp32 only, no end-to-end RLHF loop, Triton excluded). The authors disclose Gemini was used only for copyediting.

## Critical Assessment

The paper's central intellectual move is to **invert the obvious narrative**. The natural assumption — "C++ is faster than Python, so a C++ engine wins" — is shown to be wrong: the win comes from ONNX Runtime's graph execution, not the language, since calling the identical ONNX session from Python (≈349 ms) statistically ties the C++ engine (335.9 ms). The GPU result is even more counter-intuitive: `torch.compile` (19.0 ms p50, 25.6 ms p95) beats the dedicated C++ ONNX engine (27.4 ms p50, 116.2 ms p95) under the same strict CI standard the authors apply everywhere else, and the gap *widens* at the tail — killing the "static graph = safer tail" hypothesis the authors initially held.

Three specific details anchor the credibility:
- The Electra export bug (missing `token_type_ids` → 0.003–0.1 reward discrepancy → fixed to 2.9×10⁻⁵) demonstrates the correctness-checking pipeline actually catches real errors, not just FP noise.
- The zero-copy/preallocation ablation *initially* showed a spurious 10% win that vanished under reruns of the identical baseline — a textbook illustration of why the paper's repeated-launch discipline matters, deployed on the authors' own pet optimization.
- The honest retraction of the Table 8 Electra/DeBERTa ratio (4.7× vs 1.7×) that "didn't survive a fresh check," with the original numbers left in and explicitly distrusted, is exactly the failure mode the cited benchmarking literature warns about.

The weak spots are real but openly disclosed: a single 6 GiB laptop GPU and a single Ryzen CPU mean the torch.compile GPU win and the multi-instance OOM ceiling are silicon-specific; the primary eval set is 60 rows (mitigated by the 150-row robustness set); and crucially, **no end-to-end RLHF wall-clock measurement** is provided — the authors lean on outside DeepSpeed-Chat figures (generation ≈85% of step time) to temper their own claims, explicitly warning that optimizing scoring may not be where a training loop loses time. The FastAPI-over-HTTP concurrency check is a single non-repeated run, which the authors flag as confirmatory only.

## Strengths

- Rigorous, reproducible methodology: independent process launches, CI non-overlap threshold, Welch's t-tests, pinned library versions, open-sourced code.
- Genuinely surprising, actionable findings: ONNX-Runtime-not-C++ is the real lever on CPU; torch.compile wins on GPU; naive padding is actively harmful; concurrency doesn't scale.
- Intellectual honesty in retracting a non-reproduced result and reporting a real export bug.
- Clear, opinionated engineering guidance tied to specific hardware.

## Limitations

- **Single hardware target** (AMD Ryzen 7 5800H + RTX 3060 Laptop, 6 GiB VRAM): the GPU torch.compile win and the 4–5-session OOM ceiling are silicon-specific and untested on server-class GPUs or many-core CPUs.
- **fp32 only**: no half/mixed-precision paths exist in the codebase, so effects of quantization are unmeasured.
- **60-row primary eval set** (mitigated but not eliminated by the 150-row robustness set); torch.compile's recompile cost is the stated reason CPU baselines ran only 3 launches.
- **No end-to-end RLHF loop measurement**: reward-scoring speedup is isolated; its translation to faster training depends on the (likely small) fraction of step time scoring occupies, which this paper does not measure.
- **Excluded systems**: Triton Inference Server cut entirely; multi-GPU and distributed serving not tested; only two OpenAssistant reward-model backbones.
- **Table 8 Electra/DeBERTa magnitude not reproducible** — flagged by authors.
- Multi-instance tested only as threads in one process, not as separate OS processes.

## Questions for Further Exploration

1. Do the GPU results (torch.compile beating ONNX Runtime) hold on data-center GPUs (A100/H100, 40–80 GiB VRAM), where the multi-instance OOM ceiling and per-session CUDA-context overhead would shift dramatically?
2. What fraction of an actual RLHF training step does reward scoring occupy on this hardware, and does the C++/ONNX speedup translate into measurable wall-clock training improvement when wired into a live OpenRLHF loop?
3. How large is the `torch.compile` recompile spike when a rollout hits a length outside the cached shape distribution, and is the shape cache warm enough in practice to avoid it? (The authors flag this as an open question.)
4. Does the picture change under fp16/bf16 or INT8 quantization, which are standard in production reward-model serving but entirely absent here?
5. Would length-aware bucketing still be the only throughput lever on a many-core server CPU, or does ONNX Runtime's CPU backend gain meaningful batch parallelism with more cores?
6. How does a properly-run Triton Inference Server (the excluded baseline) compare, given it is a standard production serving choice for ONNX models?
7. Is the single non-reproducing Table 8 row (Electra vs DeBERTa magnitude) a genuine architecture-dependent effect or purely a measurement artifact resolvable by a fresh repeated-launch remeasurement?
