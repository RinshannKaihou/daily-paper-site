Grounded Inference: Principles for Deterministically Encapsulated Generative Models

arXiv: 2606.19753

Date: 2026-06-19

https://arxiv.org/abs/2606.19753

一句话总结： 本文定义了四个用于把生成式模型确定性封装进传统系统的"原子原语"（概率引擎、模型封装器、状态注册表、确定性编排器），并指出两类常见反模式（把 LLM 当数据工人 / 当有状态机）；引用实证证据证明"temperature=0 即确定性"是错觉（同输入输出可漂移 15%，仅切换推理后端可造成 16.6 个百分点差异），主张用外部确定性边界（沙箱校验 + 有界重试 + 断路器）约束生成模型的"爆炸半径"。
TL;DR: A single-author architectural position paper that defines four atomic primitives (Probabilistic Engine, Model Encapsulator, State Registry, Deterministic Orchestrator) for deterministically wrapping generative models inside traditional software, names two industry anti-patterns, and grounds its case in empirical non-determinism results (up to 15% output variance across identical hosted runs; up to 16.6 percentage-point swings purely from changing the inference backend). The headline rule: treat LLMs as strictly stateless functions invoked only to bridge gaps that deterministic code cannot fill, with all retry/state/validation owned by deterministic layers and a circuit breaker tripped on exhaustion.

核心贡献

  • 定义四个"原子原语"（Grounded Inference Primitives），并断言任何稳健的企业级"AI 混合架构"都只是这四个组件的有意配置：
  1. 概率引擎（Probabilistic Engine）：LLM/多模态基座，严格无状态推理引擎，永不作为数据库，输出未经验证一律不可信。
  2. 模型封装器（Model Encapsulator）：传统计算层，唯一与概率引擎通信的组件；按刚性 schema 校验输出，失败则拦截、生成错误轨迹并确定性决定是否重试。
  3. 状态注册表（State Registry）：确定性 DB / 向量库 / 文档库，持有不可变真相，与概率引擎隔离。
  4. 确定性编排器（Deterministic Orchestrator）：传统计算环境（Python 脚本、K8s、消息队列），管理状态流、API 路由与业务逻辑。
  • 归纳两类普遍反模式及各自系统性故障：
  • 反模式 A：把生成模型当数据工人（审计性/可解释性丧失、模型即服务漏洞、静默退化、并发瓶颈、概率控制流）。
  • 反模式 B：把生成模型当有状态机（静默修改"冻结"资产、上下文崩塌；援引 Liu et al. 的 U 型长上下文曲线）。
  • 提出 Order / Entropy 数据分类法与 Entropy Minimization 原则：在摄入管道最早期把所有可确定性解析的部分剥离给传统计算，只让模型处理真正需要语义推理的片段。
  • 给出参考架构 Adaptive Resolution Agent（用于 schema 漂移修复）：让 LLM 写解析器而非直接处理数据，配合沙箱语法测试 + 语义校验 + 有界重试（如上限 3 次）+ 断路器，把校验负担从模型权重转移到不可妥协的执行环境。
  • 明确拒绝"内在自我纠错"范式，援引 Huang et al.（LLMs cannot self-correct reasoning yet）与 Chen et al.（Self-Debugging）：没有客观外部锚点时，模型常把正确答案翻转为错误。
  • 给出审计启发式：在时序图中，只有确定性原语允许"连续粗激活"lifeline；概率引擎应表现为一系列"不连续的闪烁"——若架构要求概率引擎保持连续激活 lifeline，则该架构是"泄漏状态"且本质上有缺陷。

Key Contributions

  • Four atomic primitives (Probabilistic Engine, Model Encapsulator, State Registry, Deterministic Orchestrator) claimed to be the only building blocks of any robust blended AI architecture; the Model Encapsulator is the sole component permitted to talk to the Probabilistic Engine.
  • Two anti-patterns with enumerated systemic failures: (A) Generative Model as a Data Worker (loss of auditability, model-as-a-service vulnerability, silent degradation, concurrency bottlenecks, probabilistic control flow); (B) Generative Model as a Stateful Machine (silent modification of frozen assets via the "Lost in the Middle" U-shaped degradation, context collapse).
  • Order vs. Entropy data taxonomy plus an Entropy Minimization principle: route everything parseable deterministically to traditional compute; invoke the model only on fragments requiring semantic reasoning.
  • Adaptive Resolution Agent reference architecture for schema-drift remediation: the LLM synthesizes a parser rather than touching data, validated through sandboxed syntax test + documentation-grounded semantic check + bounded retry + circuit breaker.
  • Explicit rejection of intrinsic self-correction, citing Huang et al. (LLMs cannot self-correct reasoning yet) and Chen et al. (Self-Debugging): without an external objective anchor, models frequently flip correct answers into incorrect ones.
  • A diagram-based audit heuristic (Validation Micro-Pattern): continuous "thick" activation is allowed only on deterministic primitive lifelines; if the Probabilistic Engine needs a continuous activation lifeline, the architecture is state-leaking and compromised.

Methodology

This is a conceptual/position paper, not an empirical study. The author builds the framework from first principles:

  • Classifies data into Order (deterministically parseable, including text parseable by regex/AST) vs. Entropy (requires semantic reasoning), and asserts the execution boundary is defined by transformation mechanics, not by data format.
  • Defines the four primitives and shows the Encapsulation Micro-Pattern (Figure 1) and Validation Micro-Pattern (Figure 3) as sequence diagrams.
  • Instantiates the framework in the Adaptive Resolution Agent (Section 5, Figure 2): Agentic Controller pulls an anomaly cluster, queries a Document Store by timestamp, packages docs + baseline logic + data sample + error trace to the Remediation Execution Engine (REE), which forwards to the Arbiter Model. REE runs a bounded retry loop: sandbox syntax test → (on pass) reset session context → semantic validation against docs → binary pass/fail from Arbiter → on exhaustion, hard exception to Orchestrator which trips the circuit breaker, commits telemetry to an Audit Ledger, and halts.
  • Grounds the risk arguments in cited empirical evidence rather than new experiments:
  • Atil et al. (2025): hosted LLMs configured for strict determinism show up to 15% output accuracy variation across identical runs.
  • Yashwanth (2025): floating-point noise in batch-invariant GPU matmul is structured and systemic, not random.
  • Pape et al. (2026): choice of inference backend (vLLM, SGLang, llama.cpp, LMDeploy, Ollama) acts as a hidden hyperparameter, swinging benchmark scores by up to 16.6 percentage points even under greedy decoding on identical hardware.
  • Liu et al. (2024, "Lost in the Middle"): U-shaped retrieval over long contexts.

Results & Highlights

  • No quantitative results of the author's own — the paper is a framework + reference architecture. The headline "results" are architectural invariants:
  • Statelessness invariant: the Probabilistic Engine must be invoked as an ephemeral, stateless function; its session context is explicitly reset before each new submission; only task-relevant context is passed in and only engine output is returned ("Information Pass-Through" model is itself an instance of Anti-Pattern B).
  • Blast-radius containment: stochastic variance is confined to a sandboxed control plane; the primary data stream is physically barred from model mutation.
  • Failure semantics: on retry-budget exhaustion the Encapsulator returns a hard exception; the Orchestrator trips a deterministic circuit breaker, halts the agent thread, and commits error telemetry for human review.
  • Section 7 catalogs three layers of generative-model risk:
  • Service-layer instability: hidden system-prompt injections, the "seed illusion" under managed/PRNG-paged-attention multi-tenant serving, unannounced model/alignment updates shipped under frozen version aliases.
  • Distributed/hardware non-determinism: non-associative floating-point arithmetic aggregated across dynamic parallel threads; backend variability as a hidden hyperparameter.
  • Intrinsic stochastic architecture: statistical boundary failures; sampling-distribution realities (any non-zero temperature guarantees a spectrum of variants).
  • Section 8 argues conversational interfaces are a distinct valid pattern only because the human is a "Sovereign Validator" / out-of-band Model Encapsulator absorbing the risk — a status autonomous enterprise pipelines do not have.

Critical Assessment

  • Novelty: Moderate. The individual ideas (LLMs are stateless, wrap them with validation, don't put them on the data plane, circuit-break on failure, distrust self-correction, floating-point non-determinism) are largely known to experienced practitioners. The contribution is the packaging into four named primitives + two anti-patterns + a reusable reference architecture and an audit heuristic ("flickers vs. thick lifelines"). The vocabulary (Order/Entropy, Grounded Inference, Sovereign Validator) is memorable but not technically novel.
  • Rigor: Mixed. The non-determinism section is the strongest part — it correctly explains why temperature=0 does not guarantee determinism (non-associative FP arithmetic, paged attention, hidden system prompts, backend kernels) and cites real measurements. However, the architecture itself is asserted, not validated: no case study, no measurements of the Adaptive Resolution Agent, no comparison to existing agent frameworks. The "atomic primitives" claim (that every robust pattern is a configuration of exactly these four) is unfalsifiable as stated.
  • Reproducibility: N/A empirically. As guidance, it is reasonably operationalizable — the sequence diagrams (Figures 1–3) and the retry/circuit-breaker flow are concrete enough to implement, though some participants (token compaction, "authoritative knowledge pipeline") are left to the engineer.
  • Significance: Practitioner-facing and timely; likely useful as a teaching/reference doc for engineers first introducing LLMs into production data pipelines. Lower significance as a research contribution.
  • Red flags:
  • Single author with no affiliation and personal-email contact.
  • Heavy, disclosed LLM assistance across all writing phases — appropriate that it is disclosed, but the prose has the polish/generic-quality of LLM-assisted drafting and some claims are asserted rather than argued.
  • The central "atomic primitives" framing is presented as definitive ("Every robust enterprise pattern ... is simply a deliberate configuration of these four components") without proof.
  • Some risk taxonomy is derivative of vendor blog posts and well-known practitioner talks, repackaged as "first principles."
  • The "next-generation generative model interfaces" promised in the abstract is gestured at (Section 8) but never specified.

Strengths

  • Correctly and accessibly explains why LLM inference is non-deterministic even at temperature 0, with citations to actual measurements (15% run-to-run variance; 16.6 pp backend swing). This is genuinely useful and often misunderstood.
  • The audit heuristic ("thick lifelines only on deterministic primitives; probabilistic engine must flicker") is a clever, immediately usable diagram-reading rule.
  • The Adaptive Resolution Agent ("use the LLM to write the parser, not to be the parser") is a concrete, defensible design that directly inverts Anti-Pattern A.
  • Honest disclosure of LLM-writing assistance and a clear, narrow scope statement.
  • The Order/Entropy framing gives engineers a clean decision rule for when to invoke the model at all.

Limitations

  • No empirical evaluation of the proposed reference architecture; no comparison with existing LLM-orchestration frameworks (LangGraph, DSPy, AutoGen, agent SDKs) that already implement validation loops, retry budgets, and circuit breakers.
  • "Four atomic primitives" and "two anti-patterns" counts feel pre-decided; the taxonomy is not justified against alternatives and ignores hybrid cases (e.g., models with explicit memory/tool/state APIs, agentic runtimes with native state).
  • Hardware-determinism discussion is correct but shallow on mitigation (the paper's answer is essentially "don't trust it, validate externally," which sidesteps the actual reproducibility engineering literature).
  • Single author, no peer review indicated, LLM-assisted prose.
  • Future-work promise ("next-generation generative model interfaces") is hand-wavy.

Questions for Further Exploration

  • How does the Adaptive Resolution Agent compare empirically (success rate, cost, latency, hallucination rate) against existing agent frameworks (LangGraph, DSPy, AutoGen) that already wrap LLM calls in validation + retry + circuit breakers?
  • Can the "four atomic primitives" claim be formalized — e.g., is there a class of safe blended architectures that cannot be decomposed into exactly these four, and does the framework's expressive power match real systems with tool-calling, native memory, or streaming?
  • The paper asserts bitwise reproducibility is "practically infeasible" at scale; how does this interact with recent deterministic-inference techniques (deterministic CUDA graphs, fixed reduction trees, reproducible batching)? Is full determinism achievable, and at what throughput cost?
  • The 15% (Atil et al.) and 16.6 pp (Pape et al.) variance figures are for hosted/greedy settings — how do they scale with model size, context length, and reasoning-model (CoT) workloads, and can the Encapsulator's schema-validation catch the semantic portion of that variance?
  • The "Sovereign Validator" framing for conversational AI shifts all risk to the human — but at what cognitive load / error rate does that human-in-the-loop guarantee break down?
  • What would a concrete "next-generation generative model interface" look like under Grounded Inference — is there an API contract spec (e.g., deterministic Pydantic schema + signed prompt manifest + backend pinning) the author would propose to providers?
