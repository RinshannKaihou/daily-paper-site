The Correctness Illusion in LLM-Generated GPU Kernels

arXiv: 2606.20128

Date: 2026-06-19

https://arxiv.org/abs/2606.20128

一句话总结： 本文构造受控的 24 内核语料库（15 正确控制 + 9 植入文档化转录错误的 LLM 式 bug 变体），通过 op-schema 感知的种子化 fuzzing + fp64 CPU 参考 + 每 (op, dtype) 绝对容差，证明 KernelBench/TritonBench/GEAK 所用的"单形状 allclose"判定器会把 buggy 内核错误认证为正确；种子化 oracle 在 5 类 GPU 上 100% 捕获 10/10"正确性幻觉"、控制样本零误报，每次失败可从存储种子逐字节复现。
TL;DR: Existing LLM-kernel benchmarks (KernelBench, TritonBench, GEAK) score correctness with fixed-shape, single-sample allclose checks; this work builds a controlled 24-kernel corpus (15 correct controls + 9 LLM-style transcription-bug variants, extended to 26 ops with a flash-attention pair) and re-evaluates it with op-schema-aware seeded fuzzing using an fp64 CPU reference and per-(op, dtype) absolute tolerances. The seeded oracle catches 9/9 (single-GPU) and 10/10 (cross-GPU) buggy kernels, passes 15/15 and 16/16 controls with zero false positives, gives identical verdicts across RTX 3060, A10, L40S, A100 SXM4, H100 NVL, and every flagged failure replays byte-for-byte from a stored seed.

核心贡献

  • 揭示并实证检验了 LLM 生成 GPU 内核基准（KernelBench/TritonBench-G/GEAK）所用"单形状 allclose"正确性判定器存在系统性"正确性幻觉"——它将通过文档化转录错误引入的 bug 认证为正确实现。论证该 oracle 在三方面系统性乐观：(i) 每个 op 只测一个形状；(ii) 每个 op 只测一个 dtype（fp16/bf16 几乎不被测）；(iii) atol/rtol 通常比内核实测误差包络松 1–3 个数量级。
  • 构造受控、可复现语料库：24 个内核（15 正确控制 + 9 植入真实 LLM 式转录错误的变体），扩展到 26 ops（加入 flash-attention 控制+bug 对）。Bug 模式包括：softmax 尾掩码 other=0.0 而非 -inf、GELU 漏 0.5×、SiLU β 混淆 sigmoid(2x)、RMSNorm/L2Norm 漏 sqrt、LeakyReLU α=0.1 而非 0.01、matmul acc= 而非 acc+=、attention 漏 1/√D 缩放、flash-attention 漏 max 更新后的 acc·α 重缩放。
  • 提出 op-schema 感知的种子化 fuzzing 评估协议：从共享符号维度（如 matmul M,K,N；attention B,H,S,D）按 Blake2b 子密钥派生采样，候选集刻意包含边界值（如 H∈{1,3,7,256,1025}），配合 fp64 CPU 参考与每 (op, dtype) 绝对容差，并记录完整逐元素误差分布（max/mean/p50/p90/p99 abs、ULP 百分位、fp16/bf16 的 NaN/Inf 检测）。明确改用 absolute-only 容差而非 PyTorch 的 atol+rtol·|y| 组合。
  • 跨硬件一致性：在 RTX 3060(sm86)、A10(sm86)、L40S(sm89)、A100 SXM4(sm80)、H100 NVL(sm90) 五类 GPU 上判定完全一致（10/10 幻觉被捕获、16/16 控制干净），且 Triton 每次重新编译、未控制 ptxas 随机性也不影响判定。
  • 提供逐字节可复现性：每个失败存储完整输入快照（ReproductionInfo payload）并提供最小化失败形状（如 softmax_buggy 最小化为 [1,1,3] fp16、gelu_buggy 为 [2,1,1] fp16）；replay 脚本逐位匹配判定。

Key Contributions

  • Demonstrates a systematic "correctness illusion": the single-shape allclose oracle used by KernelBench (5 inputs at reference shape) / TritonBench-G / GEAK certifies LLM-style transcription-buggy kernels as correct. Argues the oracle is systematically optimistic in three ways: one shape per op, one dtype per op (fp16/bf16 rarely exercised), and per-op atol/rtol typically 1–3 orders of magnitude looser than the kernel's measured error envelope.
  • Releases a controlled, ground-truth-labeled corpus: 24 kernels (15 correct controls + 9 LLM-style buggy variants), extended to 26 ops by adding a flash-attention control/buggy pair. Seeded bug patterns include softmax tail-mask (other=0.0 vs -inf), GELU dropped 0.5×, SiLU β-confusion sigmoid(2x), RMSNorm/L2Norm missing sqrt, LeakyReLU α=0.1 vs 0.01, matmul acc= vs acc+=, attention dropped 1/√D scale, flash-attention dropped acc·α rescale after max update.
  • Introduces an op-schema-aware seeded-fuzzing protocol: symbolic-dim schemas (matmul M,K,N; attention B,H,S,D), Blake2b sub-key derivation per dim, candidate sets deliberately including boundary values (e.g., H∈{1,3,7,256,1025}); companion paper [9] shows dropping boundary values collapses shape-bug recall to zero.
  • Builds a faithful validator: fp64 CPU reference rounded to target dtype, absolute-only per-(op,dtype) tolerances (contrast PyTorch |x−y|≤atol+rtol·|y|), full element-wise error distribution (max/mean/p50/p90/p99 abs, max rel, ULP percentiles), plus NaN/Inf detection extended to fp16/bf16.
  • Cross-hardware determinism: identical verdicts (10/10 illusions caught, 16/16 controls clean) on RTX 3060, A10, L40S, A100 SXM4, H100 NVL, despite Triton recompilation per run and uncontrolled ptxas randomness.
  • Byte-for-byte reproducibility: every failure stores a full input snapshot with minimized failing shapes (softmax_buggy→[1,1,3] fp16, gelu_buggy→[2,1,1] fp16); a replay script matches verdicts bit-for-bit.

Methodology

The paper explicitly measures an oracle, not a deployed LLM's bug rate. The author argues the legacy benchmark oracle is systematically optimistic in three ways (single shape, single dtype, hand-picked loose tolerance per op) and then tests that empirically.

Corpus. 15 known-correct controls (numpy stand-ins for softmax/layernorm/matmul plus Triton controls) and 9–10 LLM-style buggy variants seeded with documented transcription patterns. The cross-GPU extension adds a flash-attention control+buggy pair (rows 25–26).

Op-schema-aware fuzzer (crates/gpuemu-daemon/src/fuzzer.rs). Each kernel ships a schema declaring shared symbolic dims (e.g., matmul dims {M,K,N}, inputs {a:[M,K], b:[K,N]}, output [M,N]; attention dims {M,N,D}). The fuzzer deterministically samples one value per dim per case via Blake2b sub-key derivation from a master seed, and candidate sets deliberately include boundary values (e.g., H∈{1,3,7,256,1025}). A companion paper ablates this and reports shape-bug recall collapses to zero without boundary values.

Reference oracle. A Python subprocess reads base64-encoded JSON inputs on stdin, computes the op in fp64, casts back to the test dtype — a high-precision reference rounded to target dtype (not claimed byte-identical to any library contract). The validator uses absolute-only per-(op,dtype) tolerances (vs PyTorch's atol+rtol·|y|), records full element-wise ErrorStats (count, num exceeding, max/mean/p50/p90/p99 abs, max/mean rel, max/mean ULP), and runs NaN/Inf detection extended to fp16/bf16.

Pipeline. Ephemeral GPU instances on vast.ai with label-strict reaper + context-manager teardown for guaranteed cleanup; daemon built from source; results uploaded to Backblaze B2 under a run id (e.g., run-20260611-095210-889b18). Setup: RTX 3060, image pytorch/pytorch:2.4.0-cuda12.4-cudnn9-devel, 30 iterations per (op,dtype), 2 dtypes, ~1,440 cases for the 24-op single-GPU run.

Stated assumptions (auditable). (1) Bug variants are author-seeded from documented patterns, not pulled from real LLM outputs — the claim is about which patterns the oracle certifies-as-correct, not deployed-LLM bug rate. (2) fp64 round-off treated as zero (standard mixed-precision practice, citing PyTorch team and Kalamkar BF16 study). (3) Per-(op,dtype) tolerances fixed ahead of P1 evaluation; calibration addressed in companion paper. (4) Triton recompiled per run; ptxas randomness not controlled (shown not to affect verdicts). (5) Client decodes tensors as contiguous, so non-contiguous layout variation is nominal at the client boundary.

Results & Highlights

  • P1 headline (24-op single-GPU, RTX 3060): seeded oracle flags 9/9 LLM-style buggy variants, passes 15/15 correct controls, zero false positives. ~1,440 cases (30 iters × 24 ops × 2 dtypes).
  • Per-kernel fail counts (Table 2 subset): gelu_buggy 29/30, rmsnorm_buggy 30/30, silu_buggy 29/30, softmax_llm_buggy 10/30, softmax_triton_buggy 13/30, l2norm_buggy 30/30, leaky_relu_buggy 30/30, matmul_buggy (K>1), attention_buggy (D≥8). All correct Triton controls (gelu/softmax/matmul) pass 0/30.
  • Shape-dependent illusion demo: softmax_triton_buggy fails 13/30 under random sampling, but 0/10 under a regular-shapes-only strategy and 6/10 under boundary-only. Bug surfaces at H=3 (tail < BLOCK) and vanishes at H=256 (power of two, no tail). This directly shows why a single-shape oracle misses it.
  • Minimal failing shapes: softmax_triton_buggy→[1,1,3] fp16; gelu_triton_buggy→[2,1,1] fp16 — shapes the allclose oracle would never exercise.
  • Cross-GPU (26 ops, 5 GPU classes): identical verdicts — 16/16 controls pass, 10/10 illusions caught (including real Triton matmul, attention, flash-attention buggy variants) on RTX 3060, A10, L40S, A100 SXM4, H100 NVL. Two providers hit "port already allocated 0.0.0.0:18433" during provisioning but the safety contract prevented orphan instances.
  • Reproducibility: each failure stores a binary input snapshot; replay fetches the run, decodes, re-runs through the daemon, verdict matches bit-for-bit.
  • Scope (explicit): about LLM-style transcription bugs certified-as-correct by the allclose-on-one-shape oracle, NOT the bug rate of any specific deployed LLM.

Critical Assessment

  • Novelty: Moderate-to-high. "Single-shape allclose is unsound" is plausible and arguably known in spirit to practitioners, but the controlled ground-truth corpus + op-schema fuzzing + absolute-only per-(op,dtype) tolerances + five-GPU replication package it rigorously and citably. The framing as a "correctness illusion" certified by existing benchmarks is sharp. The absolute-only tolerance choice (vs PyTorch atol+rtol) and the boundary-value ablation are non-trivial design decisions backed by companion work.
  • Rigor: Strong within its stated scope. Labeled controls vs injected bugs, fp64 reference, per-(op,dtype) tolerances, full ULP error distributions, five-GPU replication, byte-for-byte replay, and unusually honest assumption enumeration (5 assumptions stated plainly for audit). Per-kernel fail counts (e.g., softmax 13/30 random vs 0/10 regular vs 6/10 boundary) transparently show how a single-shape oracle misses shape-dependent bugs.
  • Reproducibility: Claimed excellent — named GPU classes, named run ids on Backblaze B2, stored binary input snapshots, public corpus repository, minimized failing shapes, bit-for-bit replay script. Caveat: independent verification depends on the public repo and seeds actually being available at review time; the abstract/PDF references a "public corpus repository (Section 7)" but a reader must confirm the URL.
  • Significance: High for benchmark consumers. If the corpus + protocol are released, gpuemu could become a standard audit layer for KernelBench/TritonBench/GEAK scorecards, and the bug taxonomy (tail-mask, off-by-one accumulation, block-size assumptions, dropped scale factors) is reusable. Limited external validity to general LLM correctness rates by the author's own explicit admission.
  • Red flags: (1) Perfect 9/9, 15/15, 10/10, 16/16 numbers are clean to the point of scrutiny — the seeded bugs are author-curated from documented transcription patterns, so there is a real risk the bug set was chosen precisely because the legacy oracle misses them (a circularity the author acknowledges in Assumption 1 / Section 6 but does not fully close). (2) The per-(op,dtype) absolute tolerances are fixed ahead of P1 but calibrated in a companion paper [8] — a reader cannot rule out that tolerances were tuned on or near the same control kernels, biasing toward zero false positives. (3) The fp64-rounded reference is "not necessarily byte-identical to any library's correctness contract" by the author's own statement, so "correct" is the oracle's definition, not an industry one. (4) Single-author preprint; no independent co-author cross-check; companion papers [8][9] needed to validate tolerance calibration and boundary-value ablation, so this paper's strongest empirical claims rest on work not fully in scope here. (5) Title/framing risk overstating: this is a controlled critique of an oracle on seeded bugs, not evidence that real deployed-LLM kernels are systematically wrong.

Strengths

  • Sharp, falsifiable thesis with a clean controlled experimental design separating correct controls from injected bugs.
  • Op-schema-aware shape generation with deliberate boundary values beats an arbitrary rank-3 cube; ablation in companion paper shows recall collapses without them.
  • High-precision fp64 reference plus absolute-only per-(op,dtype) tolerances and full ULP error distributions are a meaningful upgrade over single-global-tolerance allclose.
  • Cross-hardware replication on five GPU classes (consumer + datacenter) strengthens external validity and rules out ptxas-randomness confound.
  • Byte-for-byte replayable failures with minimized shapes are best-practice for regression and audit; run ids and Backblaze B2 storage named.
  • Honest scoping: five assumptions stated plainly, explicit acknowledgment that the result is about oracle-certified transcription bugs not deployed-LLM bug rate.

Limitations

  • Deliberately narrow scope: LLM-style transcription bugs certified-as-correct by one oracle; not a deployed-LLM bug-rate study (Assumption 1).
  • Small controlled corpus (24, extended to 26 ops) — generalization to the full KernelBench (250 workloads) / TritonBench-G (183 kernels) / GEAK op coverage is not shown.
  • Perfect detection numbers invite suspicion of a bug set curated to be missed by the legacy oracle; the seed/bug inventory must be public to defuse this, and the author flags it without fully resolving it.
  • No comparison to alternative fuzzers (FreeFuzz, DocTer, DeepREL, NablaFuzz, GPU-Fuzz) on the same kernel-layer corpus; only positioning against API-layer fuzzers is given. A 2023/TOSEM-2025 study is cited showing API-level fuzzers collectively catch only 6.5% of real bugs, but the proposed method's recall on independently-sampled real LLM bugs is not measured.
  • Tolerance calibration deferred to companion paper [8]; boundary-value ablation deferred to [9]; this paper's central "zero false positives" and "perfect recall" claims rest on work not fully self-contained here.
  • Single author; fp64 reference not byte-identical to any library contract, so "correct" is the oracle's definition.

Relevance

高度契合"LLM 推理评测"与"计算可靠性 / 精度错误"两条主线：它直接质疑当前 LLM 生成 GPU 内核基准的正确性判定器是否可信，并提出基于 fp64 参考 + 每 (op,dtype) 绝对容差 + 种子化 fuzzing + 完整 ULP 误差分布的可复现审计方案，正好补足评测可重现性、数值容差校准与跨硬件数值一致性这一研究链条；对"静默错误检测"亦有方法论价值——逐字节可复现的最小化失败用例可用于 CI 回归。

Questions for Further Exploration

  • Are the 10 seeded bugs drawn independently of the legacy oracle's blind spots, or curated precisely because the oracle misses them? What is the seeded oracle's recall on a held-out set of randomly sampled real LLM kernel outputs (not author-seeded)?
  • How does gpuemu scale and what is its wall-clock cost relative to the legacy allclose check across the full KernelBench (250) / TritonBench-G (183) op sets?
  • Were the per-(op,dtype) absolute tolerances calibrated on held-out shapes/controls, or on/near the same controls used for the zero-false-positive claim (companion [8])?
  • Is a cheaper mitigation sufficient — e.g., a small multi-shape / multi-dtype grid without full fuzzing — to close most of the correctness illusion, or is op-schema-aware boundary sampling essential?
  • Can the byte-for-byte replay seeds be integrated into CI for LLM-kernel benchmarks as a standard regression layer, and how do they interact with nondeterministic reductions/atomics on older sm_80/sm_86 hardware?
  • How sensitive are verdicts to non-contiguous layout / stride variation, given the client decodes tensors as contiguous (Assumption 5)?
  • Does the absolute-only tolerance choice generalize beyond the 26-op corpus, or does it need per-op-family recalibration as op coverage grows?
