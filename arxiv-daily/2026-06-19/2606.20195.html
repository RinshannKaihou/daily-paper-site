Randomized Sketching is Robust to Low-Precision Rounding on GPUs

arXiv: 2606.20195

Date: 2026-06-19

https://arxiv.org/abs/2606.20195

一句话总结： 作者在 NVIDIA H100 GPU 上把 Higgins 等人的 CountSketch 内核推广为每列含 ζ 个非零的 SparseStack，并实现三种 FP16 舍入（确定性就近、精确随机、抖动）；实测发现 sketch 的嵌入质量对 FP16 舍入规则几乎不敏感——决定精度的是 sketch 分布本身，因此开销最低的确定性舍入给出最佳性能-精度折中，在相干对抗最小二乘上将残差从 CountSketch 的 2.69e-1 降到 6e-4。
TL;DR: Sparse GPU sketching is memory-traffic- and atomic-bound, so the authors extend the GPU CountSketch kernel into a multi-nonzero-per-column SparseStack and implement three FP16 accumulation variants (deterministic round-to-nearest, exact stochastic, dithered). On an H100, across incoherent/coherent/adversarial tests, the FP16 rounding rule has essentially no effect on distortion or sketch-and-solve accuracy — the sketch distribution dominates — so cheap deterministic rounding wins on the performance-accuracy tradeoff.

核心贡献

  • GPU SparseStack 实现：在 Higgins 等人优化版 GPU CountSketch（ELLPACK 风格的 row_idx/val 元数据，每行一个 atomicAdd）基础上推广为每输入行散射到 ζ 个独立目标行（默认 ζ=4），符号乘以 1/√ζ 保持几何；ζ=1 即退化为 CountSketch。
  • 混合精度稀疏 sketching 内核：用 CUDA half2 与打包 32-bit 原子加实现向量化 FP16 累加（一个 32-bit 指令同时更新两个相邻列），避免相邻线程争用同一 32-bit 地址。评测三种 quantize 策略：确定性就近偶舍入（RN，硬件 __float2half）、精确随机舍入（SR，显式取上下界+插值+随机判别）、非减法抖动舍入（按 ULP 缩放的零均值噪声 + 硬件 RN）。
  • 舍入无关的经验精度结论：在不可信高斯、相干单位阵、以及相干对抗最小二乘三类问题上，FP32 SparseStack 与三种 FP16 SparseStack 的子空间失真 η(S,U)=max{|σ_max(SU)−1|, |1−σ_min(SU)|} 几乎重合；最小二乘相对残差与 κ(SA) 也基本一致。
  • 性能-精度折中：吞吐排序为 CountSketch-MxP > CountSketch > SparseStack-detm ≈ SparseStack-dither > SparseStack-stoch ≈ SparseStack > FlashSketch；因失真曲线不可区分，确定性 RN 在 SparseStack 变体中最快，给出最佳折中。
  • 高质量软件产物：提供一组 FP32/FP16 稀疏 sketch 累加 GPU 内核（CountSketch、SparseStack、多舍入模式），作者称其构成当前 GPU 随机 sketching 的 state of the art，计划集成进 Ginkgo 数值线性代数库。

Key Contributions

  • GPU SparseStack kernel generalizing Higgins et al.'s CountSketch: each input row scatters to ζ independent target rows (default ζ=4) with sign scaled by 1/√ζ; ζ=1 reduces to CountSketch.
  • Mixed-precision sparse sketching kernels using CUDA half2 packed 32-bit atomic adds to vectorize the feature dimension and avoid adjacent-thread address conflicts, with three quantize strategies: deterministic round-to-nearest-even, exact stochastic rounding (boundary interpolation + 24-bit random), and non-subtractive dithered rounding (ULP-scaled zero-mean noise + hardware RN).
  • Central empirical claim of rounding-agnostic accuracy: across incoherent Gaussian, coherent identity, and adversarial coherent least-squares problems, FP32 and all three FP16 SparseStack variants produce nearly indistinguishable subspace distortion η(S,U) and near-identical least-squares residuals and κ(SA).
  • Performance-accuracy tradeoff: consistent throughput ordering CountSketch-MxP > CountSketch > SparseStack-detm ≈ SparseStack-dither > SparseStack-stoch ≈ SparseStack > FlashSketch; since distortion is identical, deterministic RN is fastest and optimal among SparseStack variants.
  • State-of-the-art GPU sketching kernel suite (CountSketch, SparseStack, FP16 rounding modes), with plans to integrate into the Ginkgo numerical linear algebra library for mixed-precision sketch-and-solve / sketch-and-precondition workflows.

Methodology

Problem setting. Sketch A∈R^{m×n} (m≫n) via random S∈R^{d×m}; require S to be a δ-embedding of range(A). Sparse sketches are attractive on GPUs because they apply in O(nnz(A)) but are bottlenecked by global-memory traffic and atomic-update contention rather than FLOPs — motivating mixed precision.

SparseStack construction. Extend CountSketch (one nonzero per column, Se_j=ρ_j e_{s_j}) to ζ stacked independent CountSketch blocks; column j has ζ nonzeros, target index for block j uniform in the j-th subdomain [j·d/ζ, (j+1)·d/ζ−1], value Rademacher ×1/√ζ. This breaks the quadratic dependence on n up to polylog factors. The kernel (Listing 1) keeps A reads coalesced and loops over ζ subdomains per thread; metadata arrays (row_idx, val) are cache-friendly.

Mixed-precision quantization & vectorization. Naive FP16 atomicAdd suffers because two half values share one 32-bit word, serializing adjacent threads. Fix: map one thread to two consecutive columns, quantize each in FP32, pack via __halves2half2, and issue a single packed atomicAdd. The quantize function is one of:

  • RN: native __float2half (minimal overhead).
  • SR: fetch lower/upper FP16 bounds, interpolate, compare against 24-bit uniform random (high arithmetic latency).
  • Dither: add (u−0.5)·(exact_val·ULP_SCALE_FACTOR) then hardware RN (one FMA, decorrelates error).

Key intuition: the sketch's random sign flips and random routing already inject zero-mean noise into the accumulation, so unbiased rounding techniques may be redundant.

Experimental setup. NVIDIA H100 GPU. n=1024; sketch ratio d_sketch/n ∈ {2,4,8,16,32,64,128,256}; m ∈ {2^19,2^20,2^21}; distortion tests at m=2^21. Baselines: FP32 CountSketch (Higgins et al.), mixed-precision CountSketch (CountSketch-MxP), FP32 SparseStack, three FP16 SparseStack (detm/dither/stoch), FlashSketch. Metrics: effective throughput (GB/s), distortion η(S,U), sketch-and-solve relative residual ‖Ax−b‖₂/‖b‖₂, κ(SA), runtime breakdown by phase (sketching / transpose / factorization / precondition / Gram&RHS / solve). Coherence sweep uses M(α)=(1−α)Q_rand+αQ_coord, α∈[0,1], orthogonalized, reporting coherence μ(U)=m·max_i‖U_{i,:}‖²/n. Adversarial least-squares uses A=[I_n; τG/√(m−n)] with τ=10⁻³ and 5×10⁻¹ noise on high-leverage identity block, rows randomly permuted.

Results & Highlights

  • Incoherent Gaussian: all methods track the Gaussian-reference distortion √(n/d); sketch distribution and rounding rule both have little visible effect. Throughput ordering holds across m; CountSketch-MxP fastest, SparseStack-stoch slowest among SparseStack variants.
  • Coherent identity (adversarial for one-sparse): CountSketch and FlashSketch show substantially larger distortion; SparseStack variants remain grouped together — FP32 and all three FP16 rounding rules essentially identical. Distortion gap is driven by distribution, not quantization.
  • Coherence sweep (Fig. 2): as coherence μ(U) increases, CountSketch/FlashSketch degrade (especially at small d/n), while SparseStack stays stable; FP32 and FP16 SparseStack curves nearly superimposed across all rounding rules — strongest evidence of rounding insensitivity.
  • Dense synthetic least-squares (easy/hard): residuals converge for all methods at large sketch ratios; FP16 SparseStack tracks FP32; the three rounding variants visually indistinguishable. Runtime breakdown at d/n=2 shows sketch construction dominates, explaining why FP16 helps end-to-end.
  • Coherent adversarial least-squares (Table 1, d/n=2): NormalEq residual 4.28e-4; CountSketch/CountSketch-MxP residual 2.69e-1 with κ(SA)=4.85e3 (1–3 orders worse); SparseStack and all three FP16 SparseStack variants residual ≈6.1e-4, κ(SA)≈6.70 — matching NormalEq scale with near-identical accuracy. Deterministic SparseStack fastest among FP16 variants (10.14 ms vs 10.19 dither vs 10.79 stoch vs 12.03 FP32).
  • Throughput: deterministic and dithered FP16 SparseStack recover much of the performance lost to FP32 SparseStack's extra ζ atomic updates; exact stochastic is slower due to inner-loop boundary/interpolation work; CountSketch-MxP shows nonmonotone d-dependence (cache residency / atomic contention effects from smaller FP16 footprint).

Critical Assessment

  • Novelty: Moderate. SparseStack itself is borrowed (cited to Chenakkod et al. / Camaño et al. / Chen 2025); the GPU kernel is an extension of Higgins et al. 2025. The genuine contribution is the empirical isolation of the FP16 rounding rule as a non-factor for sparse sketching — a useful but narrow finding.
  • Rigor: Solid empirical methodology: three rounding schemes × multiple baselines × three input regimes × throughput + distortion + end-to-end least-squares + condition-number diagnostics, on current H100 hardware. Appropriate metric (η distortion, relative residual, κ(SA)). Scope clearly bounded (n=1024 fixed; single GPU family).
  • Reproducibility: Kernel-level implementation details (Listings 1–4, half2 packed atomics) are concrete and implementable. No explicit code-release URL in the body; integration into Ginkgo is "planned" rather than delivered, which slightly lowers reproducibility confidence.
  • Significance: High practical value for HPC/RandNLA practitioners — confirms FP16 sparse sketching is safe with the cheapest rounding rule, a directly deployable result. Broader ML impact is limited: this is a numerical-kernels paper, not an LLM/ML-systems result.
  • Red flags:
  • The headline "robust to low-precision rounding" is explicitly regime-scoped ("for the tested regimes"); the authors honestly flag they prove no OSE guarantee for the quantized operator and offer only a hand-wavy decorrelation explanation.
  • Single n (=1024); behavior at very different column dimensions or extreme aspect ratios is untested.
  • FP16 only — no BF16/FP8/INT8, and no engagement with cases where deterministic RN is known to drift.
  • Tail/worst-case behavior across random seeds is not reported (results read as single-run or few-run means); "insensitive" claims would be stronger with seed-variance bands.
  • CountSketch-MxP's nonmonotone throughput is noted but not investigated, a minor loose end.

Strengths

  • Directly answers a practical question (is low-precision accumulation safe for GPU sparse sketching?) with a clean, decisive negative result on rounding-rule sensitivity.
  • Strong baseline set (FlashSketch, CountSketch, CountSketch-MxP, FP32 SparseStack) and well-chosen adversarial construction that separates distributions cleanly.
  • Engineering contribution (half2 packed atomics, three rounding kernels) is concretely described and reusable.
  • Honest framing: conclusions labeled empirical, theory explicitly deferred to future work, precedent (Carson & Daužickaitė Nyström analysis) correctly cited.
  • Comes from a group with the track record and library (Ginkgo) to actually ship these kernels.

Limitations

  • Empirical-only; no formal bound linking quantization error to OSE distortion (acknowledged).
  • Fixed n=1024 and a single GPU architecture (H100); generalization to other shapes, GPUs, or precisions (BF16/FP8) unproven.
  • No seed-variance / tail analysis for the rounding-insensitivity claim.
  • SparseStack's throughput penalty vs CountSketch is real; the paper shows FP16 helps but does not always make SparseStack beat CountSketch-MxP in throughput — the quality advantage only pays off on coherent inputs.
  • Adversarial construction is somewhat synthetic; real-world coherent matrices (e.g., from ML data pipelines) are not tested.

Questions for Further Exploration

  • Does the FP16 rounding-insensitivity result survive BF16 (narrower mantissa) and FP8 / INT8 accumulation, or does it break once precision drops below a threshold?
  • Can a formal bound be proven — e.g., quantization error bounded by the inherent statistical distortion of the OSE — analogous to Carson & Daužickaitė's Nyström analysis?
  • How do the results scale with n (e.g., n=10⁴–10⁶) and with extreme aspect ratios m/n, where accumulation length and atomic contention grow?
  • What is the per-seed tail (worst-case) distortion distribution for deterministic vs stochastic rounding, beyond the reported mean curves?
  • Could newer GPU FP16/BF16 atomic-add instructions further close the throughput gap between SparseStack and CountSketch-MxP?
  • Does the "random sketch decorrelates quantization error" intuition extend to other randomized primitives (randomized SVD, Nyström, randomized range finders) on GPUs?
  • For real-world coherent matrices from ML pipelines (features with leverage-score concentration), does SparseStack-FP16 still match FP32 accuracy end-to-end?
