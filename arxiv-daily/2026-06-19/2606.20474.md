> **一句话总结：** 面向长上下文多轮智能体场景，UltraQuant 通过 FP4 + UE8M0 分组缩放将 KV 缓存压到约 4.25 bit/元素，并把反量化直接折叠进 AMD CDNA4 的原生 scaled-MFMA 指令，从而在缓存吃紧的后段会话中把 P50 首 token 延迟降低 3.47×，吞吐提升 1.63×。

> **TL;DR:** UltraQuant is a deployment-grade 4-bit KV-cache path for context-heavy LLM agents: it stores keys/values as FP4 (E2M1) micro-tensors with a single calibrated UE8M0 power-of-two scale per 32-channel group, letting AMD CDNA4 matrix cores dequantize natively. On a multi-turn agentic workload it cuts P50 TTFT 3.47× in cache-pressured late rounds and raises output throughput 1.63× over an FP8 KV baseline, while roughly halving the KV footprint per element.

## 核心贡献

- **针对智能体工作负载重新定义 4-bit KV 缓存的评价维度**：把任务质量、缓存驻留率与推理吞吐三项指标联合衡量，而不是只看离线压缩比。
- **给出可上线的 4-bit 设计决策清单**：非对称 K/V 处理、Walsh–Hadamard 旋转、去除 QJL、block-scale 变体，以及仅对约 10% 的层做 Lloyd–Max 质心标定。
- **UltraQuant：硬件原生的 FP4 路径**：用 FP4 E2M1 + UE8M0 group-32 缩放（约 4.25 bit/元素）替代 codebook 查找，反量化被折叠进 `MFMA_SCALE_F32_*_F8F6F4` 指令，FP8 query × FP4 KV 直接走矩阵核。
- **提出一个模型无关的全局常数 c = 0.156**：基于旋转后坐标近似服从 Beta((d−1)/2,(d−1)/2) 的事实，用单常数做离线最优缩放，无需逐 token 归一化。
- **端到端服务收益**：长上下文多轮负载下，后段 P50 TTFT 3.47×，全轮 2.3×，输出吞吐 1.63×（相对 FP8 KV 基线，MiniMax-M2.5 / MI355X）。

## Key Contributions

- Reframes 4-bit KV caching around **multi-round agent workloads**, evaluating task quality, cache residency, and serving throughput jointly rather than as an isolated compression metric.
- Documents the **practical design choices** that make a 4-bit path deployable: asymmetric K/V treatment, Walsh–Hadamard rotation, QJL removal, block-scale variants, and boundary-layer BF16 protection (first + last 2 attention layers).
- Introduces **UltraQuant**, an FP4 (E2M1) + UE8M0 group-32 approximation in which dequantization is folded into the native AMD CDNA4 scaled-MFMA instruction, removing software LUT gather from the critical path.
- Derives a **single model-agnostic global constant c = 0.156** from the rotated unit-vector Beta distribution, replacing per-token ℓ2 normalization (K-norm) with a self-scaling per-block absmax.
- Delivers **end-to-end serving wins** under memory pressure: 3.47× P50 TTFT in late rounds, 2.3× across all rounds, and 1.63× output throughput vs FP8 KV on MiniMax-M2.5 (TP=2, MI355X).

## Methodology

The paper studies two endpoints on AMD Instinct (CDNA4 / MI355X) GPUs, both anchored to prior work: TurboQuant (rotation + codebook, the 4-bit quality anchor) and vLLM FP8 KV caching (the deployment anchor).

**Ultra-TurboQuant (Ultra-TQ).** Keeps the TurboQuant TQ4/4 codebook representation but closes the kernel gap via GQA-aware tiling, a structure-of-arrays KV layout, AMD GCN-intrinsic MFMA dispatch, and a FlyDSL (MLIR Python DSL) JIT path. It also adds **calibrated Lloyd–Max centroids**: the production centroids assume post-rotation coordinates follow Beta((d−1)/2,(d−1)/2), but real activations deviate, so refitting the 16-entry table on captured keys lowers K quantization MSE 10.3% and gains +1.20 pp on GPQA. Calibration is applied to only ~10% of layers.

**UltraQuant (the deployed format).** Replaces the arbitrary codebook with the hardware-native FP4 E2M1 grid:
- **Layout**: head dim partitioned into 32-channel groups; each group = 32 FP4 codes (packed 2/byte) + 1 UE8M0 scale byte = 17 B, i.e. 4.25 bits/element (within 6% of ideal 4-bit).
- **Scale**: per-group absmax m, with a single global constant c = 0.156 → exponent E = round(log2(c·m)) stored as UE8M0. No per-token ℓ2 norm (it cannot pass through softmax and would add runtime multiply). The per-block absmax absorbs all per-token/per-head/per-model magnitude variation, so only the normalized shape matters, and that shape depends only on head dim d after rotation — hence one constant transfers across models.
- **Dequantization**: value = code × 2^scale, where scale = byte − 127. Because the scale is a power-of-two exponent this is an exponent shift, folded into the MFMA accumulator. Queries are Hadamard-rotated into the key basis and cast to FP8 E4M3, so attention runs on an FP8 × FP4 scaled-MFMA path; KV is never materialized as BF16. Keys are rotated; values are stored unrotated (asymmetric).

**Evaluation harness.** Agentic memory pressure is simulated with vLLM's native multi-turn benchmark on ShareGPT data, 32 concurrent chat sessions, reporting P50 TTFT and P50 TPOT. Serving runs on MiniMax-M2.5 (TP=2, 2×MI355X), 32K/1K at concurrency 64. Accuracy is measured on GPQA-Diamond, LCB-128K, AIME25, and MATH500 across Qwen3.5-A3B, MiniMax-M2.5, and Qwen2.5-72B.

## Results & Highlights

- **Serving (Table 1):** vs FP8 KV — P50 TTFT 0.86× (FP8 faster) in warm rounds, **3.47× in late rounds**, **2.3× across all rounds**, output throughput **1.63×**. The win comes from cache residency (keeping prefixes resident) rather than re-prefill speed.
- **Throughput (Fig. 4):** at C=64, UltraQuant reaches 1.38× BF16 output throughput, within ~1% of hardware FP8 KV (1.37×) while halving KV bytes/element.
- **Per-token latency (Fig. 5):** median TPOT 1.40× BF16, within ~2% of FP8 KV (1.37×), ahead of Ultra-TQ (1.58×) and vLLM OSS TurboQuant (5.56×).
- **Context scaling (Fig. 6):** inter-token latency improves monotonically relative to BF16, reaching ~0.5× at 64K, while FP8 stays at 1.3–1.5× — the half-precision footprint sustains longer contexts before HBM pressure forces re-prefill.
- **Accuracy (Table 2):** stable on MATH500 (+0.80 pp Qwen3.5-A3B, +0.00 MiniMax), competitive on GPQA (Qwen3.5-A3B +0.00, Qwen2.5-72B +1.52, MiniMax −2.02), mixed on LCB-128K (−2.47, −4.39), but a **material AIME25 regression** (−13.33 pp Qwen3.5-A3B, −10.00 MiniMax, −3.33 Qwen2.5-72B).
- **Ablation — per-block scale (Table 3, GPT-OSS-20B GPQA):** switching the adaptation statistic from per-token ℓ2 to per-block absmax is the load-bearing change (0.6237 → 0.6559); with per-block scaling fixed, swapping calibrated Lloyd for a uniform grid stays within noise (0.6559 vs 0.6528) — justifying the fixed FP4 grid.
- **Ablation — global constant (Table 4):** only c = 0.156 beats FP8 (+4.4 pp); c = 0.195 ties FP8; c = 1.0 (raw absmax) regresses below FP8 (−4.3 pp).

## Critical Assessment

- **Novelty (Moderate):** The core idea — approximate a rotated codebook with the hardware-native FP4 grid and a single calibrated constant — is a clean engineering insight, but the rotation/codebook machinery is inherited from TurboQuant/QJL. The genuinely novel pieces are the UE8M0 single-constant scaling argument and the kernel-level folding of dequant into MFMA, both AMD/CDNA4-specific.
- **Rigor (Mixed):** Serving benchmarks are end-to-end and well-motivated, and the constant ablation is informative. However, accuracy is single-seed in places (explicitly noted ~1.7 pp 1σ for the GPQA ablation), sample counts for AIME25 are small (20–90 items), so the −13.3 pp AIME25 regression could be noisy. The authors deserve credit for not hiding the AIME25 regression behind an average.
- **Reproducibility (Moderate-Low outside AMD):** Implementation is fully tied to AMD CDNA4 hardware and vLLM internals; the load-bearing MFMA instruction (`v_mfma_scale_f32_16x16x128_f8f6f4`) and FlyDSL paths are AMD-only. No code release is referenced in the text, so independent reproduction on NVIDIA hardware is effectively impossible without re-derivation.
- **Significance (High for a niche):** Highly relevant for the AMD serving ecosystem and for the general principle that hardware-native formats can beat analytically-optimal codebooks end-to-end. Narrower impact on non-AMD stacks.
- **Red flags / concerns:**
  - The headline 3.47× TTFT gain is **conditional** — it only appears when the workload exceeds FP8's resident-cache capacity (the authors' own stated limitation #2). In warm rounds FP8 is actually faster (0.86×).
  - AIME25 regression is large and unexplained beyond "benchmark-dependent"; no mitigation is offered.
  - Benchmarking is confined to a single hardware config (MI355X, TP=2) and a small set of models; generalization claims across model families/concurrency are asserted but not demonstrated.
  - Typos ("codeboook", "quantizaiton", "ommit") suggest a rushed draft.

## Strengths

- Genuinely end-to-end: jointly reports task accuracy, cache residency, TTFT, TPOT, and throughput rather than compression ratio alone.
- Honest reporting of accuracy regressions (AIME25) instead of burying them in averages.
- Clean, well-motivated design (Beta-distribution argument for why a single constant suffices) with supporting ablations.
- Strong systems engineering: removes software dequantization from the critical path via native MFMA, delivering throughput within ~1% of FP8 KV at half the footprint.
- Calibrated-centroid trick for Ultra-TQ is a cheap, drop-in improvement (+1.20 pp GPQA, zero inference overhead).

## Limitations

- (Authors' own #2, the most important) Speed benefit over FP8 **only materializes when context length exceeds FP8's resident-cache capacity**; for shorter contexts there is no win (vs Ultra-TQ benefits always hold).
- AIME25 accuracy regression (−10 to −13 pp on two models) is real and unresolved.
- Constant calibration is single-valued and model-agnostic; per-layer or per-distribution calibration is explicitly left to future work.
- Ultra-TQ centroid calibration helps on only ~10% of layers.
- Entirely AMD/CDNA4-bound; not portable to NVIDIA Hopper/Blackwell without re-derivation.
- Limited model/hardware/concurrency coverage; single-seed accuracy in places.

## Questions for Further Exploration

- What causes the AIME25-specific regression? Is it sensitive to particular answer distributions / pass@1 vs majority voting, and can it be recovered with per-layer constants or value-side rotation?
- How does UltraQuant transfer to NVIDIA hardware (e.g., FP4 support on Blackwell)? Does the single-constant Beta argument hold, or does the grid spacing change the optimum?
- How stable is the c = 0.156 constant across much larger head dimensions, MoE models, or models with strong outlier channels (e.g., Llama-style)?
- Does the benefit hold under non-ShareGPT agentic traces (tool-heavy, SWE-style, very long single-turn)? The current workload is conversational, not truly tool-invoking.
- Can the boundary-layer BF16 protection (first + last 2 layers) be made adaptive, and what is its accuracy contribution?
- What is the sensitivity of the headline 3.47× TTFT to concurrency and HBM capacity — is there a sharp regime transition, and how does it scale on larger MI355X node configurations?
- Could the FP4 path interact badly with silent data corruption / bit-flips in HBM, given the very low bit budget per element?
