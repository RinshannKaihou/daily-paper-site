DriftSched: Adaptive QoS-Aware Scheduling under Runtime Token Drift for Multi-Tenant GPU Inference

Kathiravan Palaniappan

arXiv: 2606.02982

一句话总结： 本文提出 DriftSched，一种针对多租户 GPU 上大语言模型推理服务的自适应 QoS 感知调度框架，通过运行时反馈驱动的指数移动平均（EMA）偏差修正机制补偿 token 漂移，将工作负载估计误差平均降低 38.8%（MAE）和 40.5%（RMSE），并在最短作业优先（SJF）策略下将端到端中位延迟降低约 42%、P99 延迟降低约 16%。
TL;DR: This paper presents DriftSched, an adaptive QoS-aware scheduling framework for multi-tenant LLM inference on NVIDIA L4 GPUs. It uses runtime feedback-driven exponential moving average (EMA) bias correction to compensate for token drift, reducing workload estimation error by 38.8% (MAE) and 40.5% (RMSE). Under Shortest-Job-First (SJF) scheduling, it achieves ~42% median end-to-end latency reduction and ~16% P99 reduction versus FIFO under sustained GPU contention.

核心贡献

  • 自适应漂移感知调度架构（DriftSched）：提出了一套端到端的多租户 LLM GPU 推理调度框架，集成了基于 FastAPI 的 API Gateway、Redis 分布式队列、租户感知队列管理与 vLLM 推理执行引擎。
  • 运行时 token 漂移补偿机制：利用指数移动平均（EMA）在线更新各工作负载类别的偏差因子，将静态的准入阶段 token 预算估计转化为自适应的动态估计，显著减少工作负载误分类。
  • 多策略调度实验对比：在 NVIDIA L4 GPU 上系统评估了 FIFO、优先级调度、加权调度、最短作业优先（SJF）与老化优先级调度五种策略，揭示了队列管理与租户 QoS 之间的权衡关系。
  • 可复现的基准测试框架：提供了包含约 1180 条独立 prompt 的异构工作负载语料、完整的实验自动化脚本与运行时指标管道，并在 GitHub 上开源。
  • 量化的性能提升：自适应偏差修正将 MAE 降低约 39%、RMSE 降低约 40%；SJF 在持续 GPU 争用下相对 FIFO 降低中位延迟约 42%、P99 延迟约 16%。

Key Contributions

  • Adaptive drift-aware scheduling architecture (DriftSched): Proposes an end-to-end framework for multi-tenant LLM GPU inference that integrates a FastAPI-based API Gateway, Redis distributed queues, tenant-aware queue management, and vLLM-based inference execution.
  • Runtime token-drift compensation mechanism: Uses exponential moving average (EMA) online updates for per-workload-category bias factors, transforming static admission-time token-budget estimation into adaptive dynamic estimation and significantly reducing workload misclassification.
  • Multi-policy scheduling evaluation: Systematically evaluates FIFO, Priority, Weighted, Shortest-Job-First (SJF), and Aging Priority scheduling on an NVIDIA L4 GPU, revealing trade-offs between queue management and tenant QoS.
  • Reproducible benchmarking framework: Provides a heterogeneous workload corpus of ~1180 unique prompts, complete experimental automation scripts, a runtime metrics pipeline, and open-sources the artifact on GitHub.
  • Quantified performance improvements: Adaptive bias correction reduces MAE by ~39% and RMSE by ~40%; SJF achieves ~42% median latency reduction and ~16% P99 reduction versus FIFO under sustained GPU contention.

Methodology

The DriftSched framework consists of seven core components operating in a closed feedback loop:

  1. Workload Generation: Produces heterogeneous concurrent inference requests using Python ThreadPoolExecutor with configurable burst concurrency. Requests span four semantic categories—short_qa, summary, technical, and report—and are assigned to three tenant tiers (Premium, Standard, Batch) via weighted probabilistic selection. Each experimental run generates 3000 requests with 50 concurrent clients.
  1. API Gateway (FastAPI): Receives incoming requests and routes them to the workload analysis layer. Provides lightweight REST-based integration between workload generation, queue management, and inference execution services.
  1. Workload Analysis Layer: Performs admission-time inference cost estimation via two sub-components:
  • Adaptive Token Estimator: Computes T_budget = T_input + T_estimated_output × S_tenant × F_input, where T_estimated_output = T_base × B_runtime. B_runtime is a per-workload-category adaptive bias factor learned from execution feedback; S_tenant is a tenant safety scaling factor; F_input captures prompt complexity.
  • Workload Classifier: Maps the estimated budget into runtime scheduling classes: short (≤128 tokens), medium (128–512 tokens), and long (>512 tokens).
  1. Tenant Queue Management: Maintains three independent Redis-backed queues (Premium, Standard, Batch). Queue assignment depends on workload classification from the adaptive estimation layer, so improvements in drift compensation directly influence queue composition.
  1. Scheduling Engine: Implements five policies:
  • FIFO: Strict arrival-order dispatch.
  • Priority: Selects from highest-priority non-empty queue; preserves FIFO within each tier.
  • Weighted: Cyclically dispatches according to a 50/30/20 ratio (Premium:Standard:Batch).
  • SJF: Prioritizes requests with smaller estimated token budgets.
  • Aging Priority: Increases priority of long-waiting requests to mitigate starvation.
  1. GPU Inference Execution: Uses vLLM (development build) with PagedAttention and continuous batching on an NVIDIA L4 GPU, loading the Qwen1.5-1.8B-Chat model in FP16 via PyTorch 2.4.0. GPU batch size is 32 with a 0.01 s batch wait interval.
  1. Adaptive Runtime Token Drift Compensation: After each inference completes, observed output length (measured as whitespace-delimited word count as a computationally inexpensive proxy) is compared against the originally estimated token budget. The bias factor is updated via EMA:
  • B_new = (1 − α) × B_old + α × B_measured, where B_measured = T_actual / T_base.

Experimental Phases: Each run consists of a 1000-request calibration phase (bias learning) followed by a 2000-request stress evaluation phase. Online adaptation continues throughout execution.

Results & Highlights

  • Runtime token drift is measurable and systematic: Across all scheduling policies, learned bias factors converge from an initial value of 1.0 to stable operating regions between ~0.79 and ~0.84, indicating that static token-budget estimates consistently overestimate actual token generation behavior.
  • Adaptive bias convergence is robust: Bias values rapidly converge during calibration and remain stable throughout the stress phase despite increased request pressure, demonstrating effective online learning.
  • Semantic categories do not directly map to runtime cost: Report-generation workloads exhibit both medium and long runtime classifications, while technical and summarization workloads are predominantly medium. This validates the need for dynamic cost estimation rather than static semantic labeling.
  • Estimation error reduction is consistent across all schedulers: Adaptive bias correction reduces MAE by 38.80% on average (range 36.82%–39.74%) and RMSE by 40.49% on average (range 37.18%–41.40%) across FIFO, Priority, Weighted, SJF, and Aging Priority.
  • SJF achieves the best overall latency performance:
  • Median end-to-end latency reduced by ~42% versus FIFO.
  • P95 and P99 latency reduced by more than 15% versus FIFO.
  • Short jobs wait only ~2.87 seconds on average, compared to ~166.89 seconds under FIFO.
  • Tenant-aware schedulers enforce QoS differentiation but do not reduce tail latency: Priority and Aging Priority reduce Premium tenant latency to ~76 s while increasing Batch latency to ~433 s. Weighted Scheduling provides a practical compromise (~158 s for Premium, ~256 s for Batch).
  • GPU execution latency is largely independent of scheduling policy: P50 GPU inference latency stays near ~10.5 s and P99 near ~11.3 s across all policies. This confirms that observed end-to-end latency improvements originate from queue management and workload estimation, not from changes in model execution speed.
  • GPU utilization and memory are stable: GPU memory consumption stabilizes near 14.5 GB; GPU utilization remains consistently high (85%–92%) across all schedulers.

Critical Assessment

  • Novelty: Moderate-to-high. While multi-tenant GPU scheduling, continuous batching (Orca, vLLM), and QoS-aware inference serving are well-studied, the explicit framing of runtime token drift compensation as a first-class admission-time scheduling signal is a distinct angle. Prior systems largely assume static max_tokens or rely on user-provided constraints; DriftSched closes the estimation–execution feedback loop.
  • Rigor: Moderate. The experimental design includes calibration/stress phases, three independent runs, and explicit separation of queue-wait time from GPU execution latency. However, the evaluation is limited to a single model (Qwen1.5-1.8B) on a single NVIDIA L4 GPU. The paper does not report confidence intervals, statistical significance tests, or variance across runs for latency percentiles. The use of word-count proxy instead of tokenizer-exact token counts introduces a known measurement simplification.
  • Reproducibility: Strong for an independent work. The GitHub repository, detailed software/hardware configurations, and open workload corpus (1180 prompts) are all positive signals. True reproducibility depends on whether the repository contains complete runnable code and exact prompt datasets; the reference suggests this is the intent.
  • Significance: Moderate. Token drift is a genuine operational pain point in production LLM serving. The ~42% median latency improvement under SJF is practically meaningful. However, the work would be more significant if it demonstrated integration with a mainstream production runtime (e.g., as a vLLM plugin or scheduler backend) and evaluated larger models (7B–70B) where scheduling inefficiencies are more pronounced.
  • Red flags:
  • Single-author independent research without visible peer review or institutional backing.
  • The comparison baseline is static estimation (BIAS=OFF) rather than state-of-the-art adaptive schedulers in existing engines; there is no comparison against vLLM's native scheduler or other recent works.
  • The output-length proxy (whitespace word count) systematically deviates from true tokenizer token counts, which the author acknowledges as future work.
  • The 1.8B model size may understate the severity of token drift and scheduling challenges seen in larger models with longer KV-cache lifetimes.

Strengths

  • Clear and practical problem formulation targeting a real operational issue (runtime token drift) in production LLM serving.
  • Well-structured modular architecture (API Gateway → Analysis → Queues → Scheduler → GPU Worker → Feedback Loop) that is easy to understand and potentially incrementally adopt.
  • Explicit separation of semantic workload category from runtime scheduling class, which is architecturally sound and practically important.
  • Quantitative improvements are reported with well-chosen metrics (MAE, RMSE, P50/P95/P99 latency, queue wait time by workload class).
  • Honest self-assessment of limitations in the Conclusion section (single GPU, single model, word-count proxy).
  • Public artifact (GitHub repository) provided, aligning with reproducible systems research norms.

Limitations

  • Single model and single GPU: All experiments use Qwen1.5-1.8B-Chat on one NVIDIA L4. Results may not generalize to larger models (7B+, MoE) or multi-GPU deployments.
  • Output-length proxy: Uses whitespace-delimited word counts instead of exact tokenizer token counts. The author acknowledges this as a simplification that reduces estimation fidelity.
  • No comparison against production schedulers: The baseline is static token-budget estimation (BIAS=OFF) rather than competing adaptive schedulers in vLLM, TensorRT-LLM, or TGI.
  • Limited fairness metrics: While tenant latency and queue wait times are reported, formal fairness metrics (e.g., Jain's fairness index, SLA violation rates per tenant) are absent.
  • No online reclassification of queued requests: Once a request is classified and queued, its class does not change even if the adaptive estimator learns new bias values. The author identifies this as future work.
  • Small model scale: At 1.8B parameters, the model may not exhibit the same KV-cache pressure, memory contention, or token-drift magnitude as larger production models.
  • No statistical significance testing: Reported averages across three runs are provided without confidence intervals or variance measures.
