> **一句话总结：** 本文提出了首个针对 AMD SEV 虚拟机可信执行环境的完整形式化验证框架，使用 Rosette 语言建立设计级与属性级抽象模型，并通过符号执行结合 Z3 SMT 求解器成功验证了 14 条机密性、7 条完整性和 2 条可用性属性，发现 SEV 能保证宿主机活性但无法在恶意监控程序存在时确保客户机侧活性。

> **TL;DR:** This paper introduces the first comprehensive formal verification framework for AMD SEV VM-based TEEs, building design-level and property-level abstractions in Rosette and using symbolic execution with the Z3 SMT solver to verify 14 confidentiality, 7 integrity, and 2 availability properties, revealing that SEV guarantees host-side liveness but not guest-side liveness under a malicious hypervisor.

## 核心贡献

- 提出了机密虚拟机（CVM）安全边界的首个综合形式化模型，明确考虑了拥有高级攻击向量的对抗者。
- 为基于虚拟机的可信执行环境形式化定义了机密性（Confidentiality）、完整性（Integrity）和可用性（Availability）三大安全属性。
- 使用 Rosette 语言开发了 AMD SEV 架构（含 SEV-ES 和 SEV-SNP 扩展）的详细形式化模型，涵盖客户机生命周期、内存加密和页状态转换。
- 系统性地对比分析了 AMD SEV 与 Intel TDX 在信任根、安全监控器、认证流程和密钥管理方面的架构差异。
- 通过机器可检查的证明正式验证了 AMD SEV 的 CIA 安全保证：21 条属性通过验证，1 条可用性属性（客户机侧活性）未能满足，与架构预期一致。

## Key Contributions

- Presents the first comprehensive formal model defining the security boundaries of confidential VMs, explicitly considering adversaries with access to advanced attack vectors.
- Formally defines confidentiality, integrity, and availability notions specifically for VM-based trusted execution environments.
- Develops a detailed formal model of the AMD SEV architecture (including SEV-ES and SEV-SNP extensions) using the Rosette language, covering guest lifecycle, memory encryption, and page-state transitions.
- Systematically compares AMD SEV with Intel TDX across root of trust, security monitor, attestation flow, and key management.
- Provides machine-checked proofs of AMD SEV's CIA security guarantees: 21 properties verified, with 1 availability property (guest-side liveness) failing as architecturally expected.

## Methodology

The authors employ a two-pronged abstraction strategy to manage the complexity of AMD SEV. **Design-based abstraction** captures all architecturally defined behaviors and state transitions from the public specification, forming a reusable operational semantics baseline. **Property-based abstraction** simplifies the system by focusing only on components relevant to specific security properties.

The formal model is implemented in **Rosette** (a solver-aided programming language extending Racket), which enables symbolic execution. Key symbolic data structures include the Guest Context Table (GCTX), ASID Table, Page Encryption Table, Guest Policy Table, Guest State Machine (GSTATE), Memory Encryption Key Table (MEKT), and for SEV-SNP, the Reverse Map Table (RMP). The model covers four categories of ABI functions: guest lifecycle (LAUNCH_START, LAUNCH_FINISH, etc.), execution (ACTIVATE, DEACTIVATE), migration (SEND_START, RECEIVE_START), and teardown (DECOMMISSION).

Security properties are derived from formal CIA definitions and checked using Rosette's symbolic execution engine backed by the **Z3 SMT solver**. The threat model assumes a privileged adversary (malicious hypervisor) that can pause, terminate, and initiate VMs, manipulate system state, and execute arbitrary instructions—but cannot directly modify the confidential VM state protected by hardware.

## Results & Highlights

- **Verification Coverage**: 23 total properties evaluated—14 confidentiality (CP1–CP14), 7 integrity (IP1–IP7), and 2 availability (AP1–AP2).
- **Success Rate**: All confidentiality and integrity properties passed. AP1 (host-side liveness: hypervisor must eventually regain control) passed. **AP2 (guest-side liveness) failed**, which is architecturally expected because an untrusted hypervisor controls CPU scheduling and can arbitrarily delay guest execution.
- **Performance**: Verification completed in under 37 seconds total (CP: 20.52s, IP: 12.45s, AP: 3.67s) on an Apple M2 with 16 GB RAM in a Docker container.
- **Model Scale**: Approximately 1,360 lines of Rosette code (655 for data structures, 450 for ABI/state transitions, 140 for test cases, 115 for utilities).
- **Key Findings**:
  - SEV-base guarantees memory encryption isolation per-ASID but leaves CPU registers exposed.
  - SEV-ES closes the register-state gap by encrypting CPU registers on VMEXIT.
  - SEV-SNP introduces hardware-enforced memory integrity via the Reverse Map Table (RMP) and PVALIDATE, preventing remapping, aliasing, and rollback attacks.
  - The decentralized hardware-enforcement model of SEV (no firmware monitor like TDX's TDX Module) shifts trust to the AMD Security Processor (ASP) and hardware datapath.

## Critical Assessment

- **Novelty**: High. While prior work verified Intel SGX (enclave-based), ARM TrustZone, and Intel TDX, this is the first comprehensive formal verification effort specifically targeting AMD SEV with extensions (SEV-ES, SEV-SNP). The explicit comparison with Intel TDX and the architecture-specific modeling are valuable contributions.
- **Rigor**: Strong. The paper provides formal mathematical definitions for CIA properties, implements them in a machine-checkable framework (Rosette + Z3), and reports concrete verification times and line counts. The abstraction methodology is clearly justified.
- **Reproducibility**: Good. The authors state that the full implementation is available on GitHub (`https://github.com/UFESL/TEE-Checking/tree/stable/sev`). The model is based on publicly available AMD specifications, making independent replication feasible. However, without permanent artifact evaluation badges or a DOI, long-term accessibility is uncertain.
- **Significance**: Moderate-to-High for the hardware security and cloud-confidential-computing communities. Formal verification of widely deployed TEEs is critical given repeated attacks on SEV implementations. The finding that guest-side liveness cannot be guaranteed is a useful clarification of architectural limitations.
- **Red Flags**:
  - The model abstracts low-level microarchitectural details (cache timing, speculative execution side channels). This is reasonable for a design-level verification but means implementation-level vulnerabilities (e.g., cache-based side channels) are out of scope.
  - Platform provisioning is assumed correct and excluded from verification, which is justified but narrows the threat model.
  - The paper does not report whether the model was validated against known CVEs or real-world attacks to demonstrate its bug-finding capability.
  - SEV-TIO (Trusted I/O) is explicitly excluded, which is a growing attack surface.

## Strengths

- First end-to-end formal verification of AMD SEV covering base, ES, and SNP extensions.
- Clear, pedagogical presentation of formal definitions with illustrative figures for CIA properties.
- Honest reporting of the availability limitation (AP2 failure) rather than overstating guarantees.
- Strong industrial collaboration ensuring specification fidelity.
- Efficient verification times (~37 seconds total) suggest the framework is practical for iterative design exploration.

## Limitations

- Does not model microarchitectural side channels (cache timing, speculative execution, power analysis).
- Platform provisioning and SEV-TIO are out of scope.
- No empirical validation against known vulnerabilities or bug injection to demonstrate that the model catches real bugs.
- The symbolic model is an abstraction; gaps between the model and actual silicon/firmware implementation remain.
- Guest-side availability cannot be guaranteed by any VM-based TEE with an untrusted hypervisor, which the paper correctly notes but does not propose mitigations for.

## Relevance

该工作为可信执行环境提供了严格的形式化安全保证，其设计抽象与属性检验的方法论对追求系统可靠性数学基础的读者具有参考价值；尽管本文聚焦于硬件虚拟化安全而非 ML 系统本身，但其形式化建模思路与对抗者建模框架可启发 ML 训练基础设施（如机密 GPU/TPU 计算）的可靠性分析。

## Questions for Further Exploration

1. Can this formal model be extended to cover AMD SEV-SNP's interaction with IOMMU and DMA devices, or the recently introduced SEV-TIO extension?
2. How would the verification framework scale if extended to model cache-coherence protocols and transient-execution side channels (e.g., speculative store bypass) that have historically affected SEV?
3. Could the Rosette model be used for differential verification—automatically detecting semantic differences between successive SEV firmware versions or between SEV and Intel TDX implementations?
4. What mitigations exist (or could be formally modeled) to provide guest-side liveness guarantees, such as hardware-enforced scheduling quotas or trusted timers inaccessible to the hypervisor?
5. Has the model been tested against documented CVEs affecting SEV (e.g., SEVerance, CipherLeaked) to confirm it would flag the corresponding vulnerability classes?
