> **一句话总结：** 本文提出 PersistentKV，一个面向 GQA 的原生块表（block-table）decode 注意力引擎与页感知调度方案；通过序列切分（sequence splitting）和紧凑工作队列（compact workqueue），在消费级 RTX 3060 上的低活跃长上下文服务场景下，相对 FlashInfer 取得 1.063–1.399× 的同步墙钟吞吐提升，并通过标定自适应策略在会退化的 B4 边界情形回退到 FlashInfer。

> **TL;DR:** PersistentKV is a native block-table GQA decode attention engine plus a page-aware serving scheduler; on a commodity RTX 3060 it beats FlashInfer by 1.063–1.399× synchronized wall throughput in low-active long-context regimes by using sequence splitting and a compact non-empty workqueue, and falls back to FlashInfer where its split-merge overhead would regress. The honest claim is a workshop-level systems niche, not a universally faster kernel.

## 核心贡献
- **原生块表 GQA decode 引擎**：按 KV-head group 分配 CTA，复用 K/V tile，直接在 paged cache 布局上执行 decode 注意力，避免将 K/V repack 成连续张量；保持精确 online-softmax（FP32 累加）。
- **页感知序列切分 + 行本地长度边界**：对低活跃长上下文将序列维度切分为 S 段，每段计算部分 softmax 状态后由 merge kernel 精确合并；split 边界、tile 数、prefetch 大小都按真实 per-row 长度推导，空 split CTA 写中性状态并提前返回。
- **紧凑工作队列调度**：仅对非空的 row–KV-head–split 区间发射任务，把 ragged 一步的 CUDA launch 数从 16 降到 2，merge MB/step 从 4.06 降到 2.54，从而在 bimodal B8 上取得 1.265× 墙钟提升。
- **标定自适应路由策略**：B1 走 PersistentKV bucket、B8 long-context 走 PersistentKV workqueue、B4 边界回退 FlashInfer；用 calibration seed (20260622) 选阈值、held-out seed (20260623) 报告，避免在测试行上选参数。
- **严谨的服务评测方法论**：同一页表分布、同一 GQA 形状、同一上下文长度、同一数值容差（max|e|<2e-3, mean|e|<3e-4 对 FlashInfer），同时报告 CUDA-event 时间与同步墙钟时间，并给出 CUDA graph replay、fused atomic merge、route bucketing 等多组消融。

## Key Contributions
- A native block-table GQA decode attention engine that maps CTAs by (request, KV-head, sequence-split), reuses K/V tiles across grouped query heads, and executes directly over paged cache layout without repacking.
- Row-local sequence-length scheduling and sequence splitting that expose sequence-parallel work for low-active long-context decode while computing exact merged softmax online.
- A compact non-empty workqueue scheduler that emits tasks only for non-empty row–KV-head–split intervals, cutting CUDA launches/step from 16 to 2 on ragged bimodal B8 and keeping one route bucket while preserving sequence parallelism.
- A calibrated adaptive routing policy (PersistentKV bucket for B1, workqueue for B8 long-context, FlashInfer for the B4 boundary) selected on a calibration seed and evaluated on a held-out seed.
- A serving-trace methodology with correctness checks against FlashInfer, dual timing views (CUDA-event and synchronized wall), and ablations over route bucketing, raggedness, workqueue scheduling, CUDA graph replay, and fused split merging.

## Methodology
PersistentKV targets grouped-query attention (GQA) with Llama-style shape Hq=32, Hkv=8, G=4, d=128, FP16, page size 16. The system has four technical pieces. (1) Native block-table execution: K,V pages are stored as `[physical page, Hkv, page_size, d]`; a page accessor translates token indices to physical page addresses; the attention loop processes 32-token tiles with async copies and maintains exact online-softmax state (m, ℓ, acc) in FP32. (2) Sequence splitting with row-local bounds: the sequence dimension is split into S disjoint ranges, each producing a partial softmax state combined exactly by a merge kernel; split bounds use true per-row effective length so empty splits write neutral states and early-exit. (3) Compact workqueue: for each active row, KV-head, and split, the scheduler emits a task only if the tile interval is non-empty, producing a 1-D CUDA grid; a second merge kernel performs segmented softmax-state reduction, keeping one route bucket for a ragged step without launch fan-out. (4) Calibrated adaptive policy: `route(B, Nmax)` selects PersistentKV length-bucket for B≤1, PersistentKV workqueue for B≥8 ∧ Nmax≥16K, and FlashInfer otherwise; thresholds and per-trace split counts (32/20/24/28) are fixed on seed 20260622 and reported on held-out seed 20260623. The serving harness simulates continuous decode steps with synthetic trace families (bucketed, homogeneous, bimodal, uniform, Zipf) that deliberately separate context length, active batch size, and raggedness. Page tables are generated by assigning seeded permuted physical page IDs; main traces use 0% hole fraction and isolated tests use 50%. Both CUDA-event timing (queued GPU work) and synchronized wall timing (incl. host planning, metadata, sync) are reported.

## Results & Highlights
- On isolated single-request native-paged attention (50% holes), FlashInfer 0.2.5 is the fastest single-call kernel (0.8889 ms @64K); PersistentKV (1.2175 ms) sits near vLLM and does not beat FlashInfer on the roofline — motivating the scheduling (not kernel-math) claim.
- Held-out calibrated-policy serving (seed 20260623): wall-clock gains of **1.399× on B1 bucketed**, **1.265× on bimodal B8**, **1.082× on uniform B8**, **1.063× on Zipf B8**; B4 bimodal routed to FlashInfer (1.000×) to avoid regression. All pass correctness (max error 6.104e-5, far below the 2e-3 threshold).
- Scheduler-derived counters on held-out bimodal B8: launches/step drop 16.00 → 2.00 (workqueue) and merge MB/step 4.06 → 2.54; merge launches 8.00 → 1.00.
- Negative-result ablations are reported honestly: fused atomic-completion merge is correct but slower on RTX 3060; CUDA graph replay does not fix the B4 two-kernel cost. These confirm the B4 loss is not merely host overhead.

## Authors & Credibility
Single author: **Muhammad Ahmed**. No institutional affiliation is listed in the paper header, and the author list is a single name. The work is explicitly self-positioned as a "workshop study / workshop prototype," with a dedicated Threats-to-Validity section. The methodology is unusually disciplined for a single-author systems note: identical page layout/shape/length/tolerance across engines, calibration-vs-held-out seed separation, dual timing views, and explicit refusal to claim hardware counters (Nsight performance counters were blocked by ERR NVGPUCTRPERM, so only structural scheduler counters are reported). The lack of institutional affiliation and the absence of any full-runtime integration keep the credibility ceiling modest, but the internal honesty is high.

## Critical Assessment
- **Novelty: Medium.** The pieces (paged attention via FlashAttention-style tiling, sequence splitting with online-softmax merge, GQA tile reuse, ragged workqueues) are individually well-established. The genuine novelty is the *adaptive page-aware serving policy* that treats work assignment — not attention math — as the decisive variable, plus the disciplined identification of which regimes favor each kernel.
- **Rigor: Good (within its stated scope).** Same page tensors/IDs/shape/lengths/tolerance for all engines; calibration/held-out seed protocol; correctness verified before timing; dual timing views; negative ablations reported. Weak spots: only one held-out seed (authors acknowledge ≥5 needed for statistical claims), and no hardware profiling counters available.
- **Reproducibility: Moderate-to-Good.** Exact versions (FlashInfer 0.2.5, vLLM 0.6.4.post1, TensorRT-LLM 0.8, CUDA 12.1, PyTorch 2.5.1), trace-generation procedure, and routing thresholds are specified. But no code/data release is referenced, traces are synthetic, and the RTX 3060 (28 SMs, ~302 GB/s) is an unusual target whose specific SM count matters for the split-count tuning.
- **Significance: Narrow but real.** The wins are confined to commodity-GPU, low-active long-context decode. The claim is explicitly workshop-level and does not generalize to datacenter GPUs (Hopper FA-3 features untested) or full serving stacks.
- **Red flags:** (a) Single author, no affiliation listed — worth noting for vetting. (b) "PersistentKV" name is potentially misleading: the paper clarifies it means persistent *KV-cache layout*, not persistent CUDA-kernel residency; readers may assume the latter. (c) Headline speedups come from synthetic traces on a single GPU with thresholds tuned on one seed. (d) Kernel-only decode loop — no MLP layers, sampling, networking, or admission control — so request-level latency claims are not supported. (e) GQA-reuse mechanism is fixed at G=4 and not ablated over G∈{1,4,8}, so the GQA-specific contribution is a design choice rather than a demonstrated separate speedup.

## Strengths
- Honest, scoped claims with a dedicated Threats-to-Validity section — rare and commendable.
- Clean controlled methodology: identical layout/shape/length/tolerance across engines removes a common confound in serving benchmarks.
- The adaptive-policy framing (route by request state, not one-kernel-fits-all) is a genuinely useful systems insight, especially the decision to *fall back* to FlashInfer at the B4 boundary.
- Strong methodological hygiene: separate calibration vs held-out seeds, dual timing views, correctness-first reporting.
- Useful structural counters (launches/step, merge traffic) that make the scheduling mechanism legible.

## Limitations
- Synthetic traces only; no real production traffic, admission control, or prefill/decode interference.
- Single hardware target (RTX 3060); Hopper/datacenter behavior untested, and split policy must be retuned per GPU.
- Single calibration seed and single held-out seed — insufficient for statistical claims.
- Kernel-only decode loop; no integration into vLLM/TensorRT-LLM, no MLP/sampling/network, no request-level latency.
- Hardware profiling counters blocked, so achieved occupancy, L2 hit rate, and DRAM bytes are not measured.
- GQA contribution not isolated (G fixed at 4); no sweep over GQA ratios.
- Fused-merge and CUDA-graph optimizations tried and abandoned — leaving the B4 regime genuinely unsolved.

## Relevance
高度相关：本文直接属于 LLM serving / 推理系统领域，聚焦长上下文 decode 阶段的页感知调度与 KV-cache 访问，正好命中你关注的"LLM 服务与推理可靠性"主题——它揭示了在低活跃长上下文场景下，工作分配策略（而非单纯注意力算子）会成为服务吞吐的决定性变量，对消费级 GPU 部署和调度策略设计有直接参考价值。

## Questions for Further Exploration
- How does the adaptive policy transfer to datacenter GPUs (A100/H100/H200) where FA-3 async/tensor-memory features and higher HBM bandwidth may erase the RTX-3060-specific gains?
- Would an online bandit- or learned-cost-model router (replacing the manual thresholds) generalize across traffic patterns without per-trace split tuning?
- What happens once PersistentKV is integrated into a full serving runtime (vLLM/SGLang) with real traces, admission control, and prefill/decode interference — does the 1.265× wall gain survive at request-level p50/p99 latency?
- Can a lower-overhead segmented merge (e.g., cooperative persistent kernel, warp-level reduction) close the B4 regression that atomic-completion and CUDA-graph replay failed to fix?
- How does GQA tile reuse actually scale as G sweeps over {1,4,8,16}, especially for newer wide-GQA models?
- Does page-table lookup hoisting or fused RoPE/KV-cache-write/dequantization change the routing boundaries for quantized (INT8/FP8) long-context serving?
