Compiler-Driven Approximation Tuning for Hyperdimensional Computing

arXiv: 2606.26547

Date: 2026-06-25

https://arxiv.org/abs/2606.26547

一句话总结： 本文提出 ApproxHDC，一个基于 HPVM-HDC 编译器基础设施的自动调优框架，通过 HDC 专用的搜索空间剪枝与统一软件/硬件近似（approximation）调优，将 HDC 工作负载可重定向地编译到 CPU、GPU 与模拟的 SpecPCM/ReRAM 内存计算加速器，在满足用户指定 QoS 约束下取得最高 17.25×(CPU)/15.02×(GPU)/4.69×(SpecPCM) 加速，并比 SOTA 的 MicroHD 框架快 3.49×。
TL;DR: ApproxHDC is a compiler-driven framework that automatically searches both software-level (reduction perforation, binarization, encoding dimensionality, element type, algorithmic choice) and hardware-level (ADC resolution, write-verify cycles, quantization scale, MLC bits) approximations for Hyperdimensional Computing. Across four HDC++ benchmarks (HD-Classification, HD-Clustering, RelHD, HD-Hashtable) on ISOLET and Cora datasets, it reaches up to 17.25× CPU and 15.02× GPU speedup; on the Micro-HD-Classification port of MicroHD on ISOLET it finds a 24.65× speedup (4.8% accuracy loss) versus MicroHD's 7.07× (2.4% loss) — a 3.49× advantage at the same 0.845 accuracy bar.

核心贡献

  • 提出 ApproxHDC 自动调优系统：首个面向 HDC 的、自动识别并应用领域特定近似（domain-specific approximations）的框架，把近似从手工标注提升为编译器内可被外部调优器驱动的自动决策，并在软件与硬件两个抽象层联合优化端到端性能、能耗与 QoS。
  • 扩展 HPVM-HDC 编译器基础设施：通过两个专用编译 pass（HDC 原语识别 + HDC 近似标注）实现可重定向编译，从一份 HDC++ 源程序统一映射到 CPU、GPU 与模拟的 ReRAM/SpecPCM 内存计算加速器。
  • 统一软件与硬件近似搜索空间：首次把内存计算加速器的硬件旋钮（ADC 分辨率、write-verify 深度、模拟量化尺度、MLC 位数）纳入编译器驱动的调优循环，与软件近似（reduction perforation、automatic binarization、encoding dimensionality、element type、algorithmic choice）联合搜索。
  • HDC 专用的搜索空间剪枝：结合自动领域感知剪枝与开发者 __hetero_hdc_no_approx 注解，把设计空间缩减多达 86 个数量级（如 HD-Classification 从 10^89 降到 10^3），在分钟级时间预算内发现高质量配置。
  • 跨四个基准的广泛评测：在 HD-Classification、HD-Clustering、RelHD、HD-Hashtable 上覆盖多数据集与多硬件目标，并给出对 MicroHD 的直接对比。

Key Contributions

  • ApproxHDC framework: An auto-tuning system external to the HPVM-HDC compiler that automates approximation selection (software + hardware) to jointly optimize end-to-end performance, energy, and quality of service across heterogeneous targets, subject to a user-specified QoS constraint.
  • Two new compiler passes on HPVM-HDC: a HDC Primitive Identification pass that statically extracts every HDC primitive invocation (with function/BB/execution-order context) into a JSON view, and a HDC Approximation Annotation pass that consumes the tuner's per-operation configuration and re-inserts source-level annotations into the HDC++ program.
  • Unified software + hardware approximation space: software knobs (Reduction Perforation with base/stride/extent, Automatic Binarization, Encoding Dimensionality, Element Type Representation, Algorithmic Choice such as cosine vs. Hamming, random projection vs. ID-based) plus hardware knobs exposed through SpecPCM/ReRAM backends (ADC resolution, write-verify cycles, analog quantization scale/floor/ceiling, MLC bits per cell).
  • HDC-specific search-space pruning: automatic domain-aware pruning plus a lightweight __hetero_hdc_no_approx source annotation, reducing the space by up to 86 orders of magnitude so OpenTuner-based search finds good configs within minutes.
  • Broad empirical evaluation across four HDC++ benchmarks, multiple datasets (ISOLET, Cora, genomic long-read search), CPU/GPU/SpecPCM targets, and a head-to-head port of the MicroHD baseline.

Methodology

Compiler substrate (HPVM-HDC / HDC++). ApproxHDC extends the HPVM-HDC compiler (itself built on the HPVM heterogeneous dataflow framework). HDC++ is a C++-embedded DSL exposing HDC primitives (e.g. __hetero_hdc_matmul, __hetero_hdc_hamming_distance, __hetero_hdc_cos_sim, __hetero_hdc_arg_min) as first-class operations over hypervector/hypermatrix types. Each primitive lowers to a unique HPVM-HDC IR operation. ApproxHDC is implemented in Python (not C++) and lives outside the compiler; it communicates via JSON views.

Three-stage flow. (1) HDC Primitive Identification — a static pass enumerates every HDC primitive invocation, disambiguating repeats in the same basic block by execution order, and emit a JSON listing each op with its intrinsic name, argument types, calling function and ID. (2) ApproxHDC Tuning System — given the op list it constructs the software + hardware approximation design space, and uses OpenTuner as the underlying ensembler/bandit auto-tuner. The user supplies an application driver that runs the compiled binary with the right dataset and reports a QoS metric (e.g. inference accuracy or NMI). For each candidate configuration ApproxHDC emits a JSON specifying per-op approximation parameters, compiles + runs via HPVM-HDC, and records QoS, runtime and energy. (3) HDC Approximation Annotation — a second pass re-inserts the chosen annotations (e.g. __hetero_hdc_red_perf(base, end, stride)) into the HDC++ source so HPVM-HDC can lower them. Tuning iterates until a time/iteration budget is hit.

Approximation transformations. Reduction Perforation skips iterations along the reduction axis at a configurable stride (with start/extent/stride knobs; dot products are rescaled to compensate). Automatic Binarization packs hypervectors into ±1 single-bit representations and rewrites ops to xor/popcount. Encoding Dimensionality shrinks D (cost grows ~quadratically in encoding, linearly in inference). Element Type Representation switches between int 8/16/32/64-bit, FP16/FP32 or 1-bit. Algorithmic Choice swaps cosine↔Hamming similarity or random-projection↔ID-based encoding. Hardware (SpecPCM/ReRAM) knobs are modelled by the simulator: ADC resolution, write-verify cycles, analog quantization scale (floor/ceiling/step), and MLC bits-per-cell, each with calibrated latency/energy/error behaviour (e.g. prior PCM data shows >10% error at 3 bits/cell, while 1–2 bits/cell stay stable; 2 bits/cell ≈ 2× throughput).

Pruning. Because the cross-product grows exponentially, ApproxHDC provides __hetero_hdc_no_approx to exclude sensitive stages (nearly all evaluated apps need it to protect the encoding stage), plus domain-aware reduction-perforation pruning (the paper studies four pruning series: Full RedPerf, End-only+hint, No RedPerf, Full RedPerf+hint).

Hardware & datasets. CPU: two AMD EPYC 7453 28-core processors (56 threads, 503 GB). GPU: NVIDIA GeForce RTX 2080 Ti (11 GB), host Intel Xeon Silver 4216 (16 cores / 32 threads, 192 GB). Applications from the HPVM-HDC suite: HD-Classification and HD-Clustering on ISOLET (spoken letters); RelHD on Cora (scientific-publication graph); HD-Hashtable on genomic long-read sequences. QoS target is baseline QoS − 0.05; tuning budget 3 h (2 h for the MicroHD comparison).

Results & Highlights

  • Peak speedups (within QoS = baseline − 0.05). "ApproxHDC achieves up to 17.25× speedup on CPUs, 15.02× on GPUs, and 4.69× on SpecPCM while staying within user-specified QoS constraints."
  • HD-Clustering on ISOLET (baseline NMI 0.72): 17.25× on CPU (NMI 0.6737) and 13.41× on GPU (NMI 0.6708), driven by switching to int datatype + Hamming similarity and cutting training iterations from 20 to 3, holding D=1024; CPU additionally sets reduction-perforation extent to 0.9.
  • RelHD on Cora (baseline float, D=2048): 11.4× on CPU and 15.02× on GPU, both at 0.71 accuracy, primarily from shrinking dimensionality from D=2048 to D=128 (16× reduction); reduction perforation contributed only ~0.3× extra on CPU.
  • HD-Classification on ISOLET (baseline D=2048, 10 train iters, float+cosine, 0.87 accuracy): 11.64× on CPU and 12.93× on GPU, both at 0.82 accuracy, by reducing D to 256 (8×) and training iterations to 3; the redperf-OFF series found the best configs.
  • HD-Hashtable (baseline D=8192, KMERS_PER_HV=400, 0.83 accuracy): 14.66× on GPU at 0.78 accuracy (D→512, KMERS_PER_HV=20, THRESHOLD_MULT=0.589, redperf extent 0.646); only 2.01× on CPU (Full RedPerf) because its search space (10^16) and long runtime starved the 3 h budget (<100 configs evaluated).
  • MicroHD head-to-head (ISOLET, shared D=1024/L=128/Q=16, cosine sim, accuracy cutoff 0.845). ApproxHDC finds a 24.65× speedup at 4.8% accuracy loss versus MicroHD's best 7.07× at 2.4% loss — a 3.49× advantage at the same accuracy bar. ApproxHDC crosses MicroHD's max at t≈282 s (~4.7 min), already at 13.52×. The structural reason: ApproxHDC picks D=512 with per-op reduction-perforation (≈44% of training cossim, ≈82% at inference), a knob MicroHD's PyTorch/TorchHD pipeline lacks.
  • Search-space pruning. HDC-specific pruning reduces the design space by up to 86 orders of magnitude (e.g. HD-Classification Full RedPerf 10^89 → with hint 10^3); no single pruning strategy is uniformly best.
  • SpecPCM hardware tuning (HD-Classification). Reference (D=1024, TRAINING_ITER=10, MLC=1, ADC_RES=8, accuracy 0.84); best config (TRAINING_ITER=2, ADC_FLOOR=−3, accuracy 0.8138) yields 4.69× speedup and a coincident 4.69× energy reduction; a config dropping ADC_RES 8→6 and ADC_STEP→9 trades an extra 0.55% accuracy for up to 7.12× energy reduction. Most of the speedup came from cutting training iterations 10→2; the hardware knobs mainly affected energy.

Critical Assessment

  • Headline numbers are tuning-budget contingent. Several reported maxima are explicitly called "conservative estimates" within the fixed 3 h budget — HD-Clustering's GPU/CPU runs wasted budget on configs below the (mis-set) NMI cutoff, and HD-Hashtable on CPU only managed 2.01× because fewer than 100 configs were evaluated. So the per-app maxima are lower bounds, not asymptotic optima.
  • Weak/absent baseline on hardware. The SpecPCM comparison has no real-device counterpart — all ReRAM/PCM results rest on the SpecPCM simulator's device models, and the paper itself notes >10% error at 3 bits/cell in prior PCM measurements, so MLC/ADC energy claims depend on model fidelity that the paper does not validate against silicon.
  • Software baselines are narrow. The only HDC-specific approximation baseline is MicroHD, ported to HDC++ without its sub-word quantization knob Q (because HDC++ does not support it), so MicroHD is compared at a structural disadvantage the authors acknowledge; no comparison to ApproxHPVM/ApproxTuner/EnerJ on the HDC workloads is given.
  • Strong reliance on the no_approx annotation. The paper states "nearly all the applications we evaluate ApproxHDC on require this annotation, specifically to prevent the application-specific encoding stages from being approximated," meaning the headline results depend on manual developer insight about which stages are sensitive — undercutting the "automated" framing.
  • No error/reliability bound. QoS is enforced purely empirically (accuracy threshold), with no formal accuracy-loss or numerical-error guarantee during search; "minimal accuracy loss" is operationalized as baseline − 0.05 only.

Strengths

  • Genuinely cross-layer: integrates software and hardware approximation in one compiler-driven tuning loop, including emerging in-memory (SpecPCM/ReRAM) knobs.
  • Built on a real, named compiler base (HPVM-HDC / HDC++) and a real auto-tuner (OpenTuner), with concrete per-op JSON interfaces.
  • Strong quantitative breadth: four HDC++ applications, three target classes, two datasets, and a MicroHD head-to-head with a 3.49× win.
  • Honest reporting of where pruning helps or hurts and where the budget starves search (HD-Hashtable CPU).
  • Pruning mechanism is developer-controllable and composable (annotation + domain-aware perforation pruning).

Limitations

  • All ReRAM/PCM results are simulation-only; no fabricated-device validation.
  • The "automated" pipeline still needs a developer-supplied QoS driver and, in practice, no_approx annotations on the encoding stage.
  • HDC accuracy itself is below modern DNNs on the same tasks (e.g. 0.87 on ISOLET classification), so speedups preserve "HDC-level" accuracy, not DNN-level accuracy.
  • Tuning cost: 2–3 h budgets per app per target; search does not obviously transfer across datasets/backends (the paper notes configs "rarely transfer").
  • Single GPU (RTX 2080 Ti) and single CPU socket configuration; no scaling study across GPU count or newer accelerator generations.

Relevance

直接映射到您的 「计算可靠性与精度误差（Computation Reliability & Precision Errors）」 与 「ML 可靠性与数学基础（ML Reliability & Mathematical Foundations）」 两个兴趣方向：ApproxHDC 的核心就是在软件（精度/ perforation）与硬件（ADC 位宽、write-verify、MLC）层引入有控的近似，并研究这些近似对 HDC 推理质量（QoS）的影响与剪枝方法，本质上是"可控精度损失 vs 性能/能耗"的可靠性工程问题。其内存计算（ReRAM/PCM）硬件旋钮部分也与 Silent Data Corruption (SDC) 中的设备级噪声/写入不可靠性话题相关，但本文不直接讨论 SDC 检测。

Questions for Further Exploration

  1. How does the chosen OpenTuner ensemble's performance scale with the number of HDC primitives, and would a Bayesian-optimization or RL-driven tuner find better configs within the same 3 h budget on the large-space apps (HD-Hashtable CPU)?
  2. The paper notes MLC at 3 bits/cell already shows >10% PCM error from prior work [8] — how do ApproxHDC's selected MLC/ADC configurations degrade if the SpecPCM simulator's noise model is replaced by measured device traces?
  3. Could the no_approx encoding protection be inferred automatically (e.g. via a sensitivity-analysis pass) instead of requiring developer annotations, and what is the accuracy risk if it is omitted?
  4. Is there a formal upper bound on accuracy loss given the joint software + hardware approximation configuration, or is the baseline − 0.05 QoS target the only guarantee?
  5. The paper cites ApproxHPVM [29] and ApproxTuner [30] as the closest DL-side analogues — would porting ApproxTuner's three-stage (dev/install/runtime) tuning onto HDC++ improve wall-clock tuning time relative to OpenTuner?
  6. How transferable are the discovered configurations across datasets (e.g. ISOLET vs. another speech dataset) and across GPU generations, given the authors' note that configs "rarely transfer"?
