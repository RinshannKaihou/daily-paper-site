> **一句话总结：** 本文针对 LLM 驱动的望远镜调度中数据引用不一致、推理错误与决策不可执行等问题，提出多层级验证 + 可追溯推理框架（原子推理单元 ARU 与依赖 DAG），并在决策执行前用仪器/天文/计算/原始数据/一致性五层校验过滤错误，结合失败反馈闭环；在 ZTF 瞬变源调度基准上把 Qwen3(30B) 的可执行率（EER）从 0.692 提到 1.000、参考策略可接受率（RPA）从 0.531 提到 0.944。

> **TL;DR:** The paper inserts an independent validation layer between an LLM-generated telescope scheduling proposal and execution. It decomposes model reasoning into Atomic Reasoning Units (ARUs) `a_i = ⟨e_i, c_i⟩` organized as a dependency DAG, and runs a 5-level pre-execution check (instrument state, astronomical knowledge, computational re-evaluation, raw-data authenticity, decision/data consistency). On a controlled 6-scenario / 360-case benchmark built from ZTF transient-alert candidates, the framework lifts Qwen3-30B's External Executability Rate (EER) from 0.692 [0.642,0.737] to 1.000 [0.989,1] and Reference Policy Acceptability (RPA) from 0.531 [0.479,0.582] to 0.944 [0.916,0.964]; in the hardest multi-alert scenario S5 RPA rises from 0.050 (pure Qwen3) to 0.933 (Qwen3+Full) versus a greedy baseline at 0.483 and ILP at 1.000.

## 核心贡献

- **提出多层级验证框架**：在 LLM 生成调度提案与执行之间插入独立验证层，不干预 AI 决策生成过程，仅校验其"正确性（可执行 + 一致）"而非"最优性"。
- **五层校验体系**：仪器状态约束（如云量 >90% 禁用）、天文知识约束（滤镜有效性、时间窗、安全规则）、计算约束（用传统方法重算角距离等数值，规避概率模型的数值不可靠）、原始数据真实性与可追溯校验（核对目标/仪器/约束字段是否能在数据库中解析）、决策表达与原始数据一致性校验（描述与所引用数据量是否匹配）。
- **原子推理单元（ARU）与依赖 DAG 的可追溯表示**：把每个推理步骤定义为最小不可分因果单元 `a_i = ⟨e_i, c_i⟩`（e_i 为支撑数据字段，c_i 为原子结论），用有向无环图 `G=(V,E)` 编码依赖关系；任何 ARU 失效即整条提案被阻断，支持按拓扑路径定位错误。
- **失败阻断 + 闭环反馈**：未通过的提案输出结构化错误信息（失败项 ID、失败类型、关联字段、依赖路径），可回灌模型输入触发再生成；历史记录沉淀为高质量训练数据，形成"AI ↔ 验证"闭环。
- **针对失败模式的分类法**：给出 AI 调度失败模式与对应校验机制的映射表（参考错误→原始数据一致性；约束不一致→多层约束校验；无效坐标→天文/仪器约束；推理跳跃→ARU/DAG 依赖检查；伪造干扰源→原始数据真实性校验）。

## Key Contributions

- **Independent validation layer** positioned between LLM proposal generation and observation execution, redesigning model I/O around a *claims / arguments / supporting evidence* structure (Figure 1) and judging executability + consistency rather than optimality.
- **5-level pre-execution validation**: (1) instrument-state constraints (e.g. block when cloud >90% or instrument unavailable), (2) astronomical-knowledge constraints (invalid filters, expired windows, unsafe parameters), (3) computational re-evaluation of model-asserted numbers (e.g. angular distances) with classical methods to counter probabilistic-model numeric unreliability, (4) raw-data authenticity/traceability (resolve referenced targets/instruments/fields against DB + state dictionary), (5) decision-expression vs raw-data consistency (e.g. "frequent interference" claim must have ≥1 supporting record).
- **Atomic Reasoning Unit (ARU) + dependency DAG** for traceability: each ARU is the smallest indivisible causal unit `a_i = ⟨e_i, c_i⟩`; reasoning consistency holds only when every ARU is individually validated, and the acyclic DAG enables local blocking, reusable verified sub-chains, and topological error localization.
- **Closed-loop failure blocking + feedback**: structured error payloads (failed-item ID, failure type, fields, dependency path) are fed back to the model to regenerate constraint-compliant proposals; history becomes a refinement dataset.
- **Failure-mode taxonomy** mapping each AI scheduling failure to its validator (Table 1), plus a *telescope scheduling state dictionary* (action_wait / action_adjust_exptime / action_change_target / action_keep_plan / target_match / interference_pollution, …) that constrains ARU expression to reduce variability.

## Methodology

**System architecture (§2).** The LLM receives a JSON prompt aligned to database field names containing current alerts (GCN transient alerts), instrument status, scientific objectives, and the telescope-state dictionary, and emits a scheduling proposal plus a set of ARUs. The proposal is decomposed into (1) a unique task id, (2) the decision/instruction, (3) the justification linking data→decision, (4) supporting data sources/conditions. Each ARU `a_i = ⟨e_i, c_i⟩` carries the data fields it depends on (with table+field identifiers) and an atomic conclusion (e.g. "execute action A on target T in window [t1,t2] at node k"). ARUs are wired into a DAG `G=(V,E)`; an edge `a_j → a_i` means `a_i`'s validity depends on `a_j`. The DAG is constrained acyclic so validation can run along topological paths and localize the offending ARU.

**Validation loop (§2.3).** Proposals pass through the 5 levels above; only all-passing proposals proceed to execution. Failures emit structured error info and, in the feedback configuration, are re-injected into the model. The mechanism is explicitly *step-level consistency checking* — it does not certify global optimality or ultimate truth of the decision, only that each building block and its connections are sound.

**Models.** Two LLMs are evaluated: **Microsoft Phi-4 (14B)** and **Qwen3 (30B)**. The paper notes Phi-4 hallucinates fewer spurious interference signals than Qwen3, but neither is hallucination-free.

**Scenarios (§3.2).** Six ToO (Target-of-Opportunity) scenarios covering interrupt-driven scheduling, priority conflicts and multi-target competition: S1 single high-priority transient; S2 single low-priority transient (priority conflict); S3 multiple simultaneous alerts (multi-priority competition); S4 low-priority ToO with long windows under irrelevant/outdated interference; S5 high-priority ToO in complex environments (dozens of irrelevant inputs + recent changes); S6 abnormal device / out-of-bound physical constraints (invalid RA/Dec, invalid filters). The main comparison benchmark is built from historical **ZTF** transient-alert candidates (basic attributes: sky position, brightness, detection history) with controlled injection of current plan, environment, interruptions — 60 cases × 6 scenarios = 360 cases per method.

**Metrics (§3.5).** Pass Rate PR = N_pass/N; Repair Success Rate RSR1 = N_repair,1/N_block; Failure Distribution; Reasoning Reusability. For the controlled benchmark three metrics with 95% CIs (Wilson for rates, bootstrap for ranks): External Executability Rate EER (action ∈ feasible set F_i), Reference Policy Acceptability RPA (action ∈ independently-specified reference set R_i ⊆ F_i, built by restricting to highest feasible priority tier then keeping Pareto-efficient actions under latest-start-time / filter-switch cost / motion overhead / environment risk), and Reference Rank Score RRS (ranking quality within the reference list).

**Baselines.** Science-first greedy scheduler [7], deadline-oriented rule scheduler, single-step integer-linear-programming (ILP) dispatch scheduler [9,18,19], plus four Qwen3 configurations: Qwen3 Only, Qwen3+Validation, Qwen3+Validation+Feedback, Qwen3+Framework (full).

## Results & Highlights

- **Reliability, no-feedback (Table 2, >2000 tests, 2433 total).** Phi-4 (14B) average pass rate 79.3% over 1231 cases; Qwen3 (30B) average 84.5% over 1202 cases. The worst case for both is S6 (abnormal/out-of-bound physical constraints): Phi-4 69/204 passed, Qwen3 40/202 passed — i.e. models largely fail to reject physically invalid coordinates/filters without validation.
- **Staged validation gains (cumulative, §3.6.1).** Adding output-level validation alone raises average pass rate by **+18.1%**; adding feedback-guided regeneration adds a further **+12.6%**; adding ARU/DAG reasoning verification adds a further **+16.8%**, with the reasoning gain concentrated in complex scenarios (**+31.35% in S4**, **+25% in S5**).
- **Controlled benchmark (Table 3, 360 cases/method, 95% CIs).** Pure Qwen3: EER **0.692 [0.642,0.737]**, RPA **0.531 [0.479,0.582]**, RRS **0.612 [0.565,0.657]**. Qwen3+Full: EER **1.000 [0.989,1]**, RPA **0.944 [0.916,0.964]**, RRS **0.983 [0.975,0.990]**. Deterministic baselines: Greedy RPA 0.769 [0.723,0.810] / RRS 0.885; Deadline RPA 0.711 / RRS 0.711; ILP RPA 1.000 / RRS 1.000.
- **Hardest scenarios (S3, S5).** In S5 (high-priority ToO, complex environment) RPA goes from **0.050** (pure Qwen3) to **0.933** (Qwen3+Full), versus greedy 0.483 and ILP 1.000; in S3 (multi-alert competition) from 0.050 to 0.733. By contrast in clean S1/S2/S6 scenarios most methods already hit RPA ≈ 1.000.
- **Ablation (Panel B).** Validation alone restores executability (EER 1.000) but leaves RPA 0.683 / RRS 0.765; feedback pushes RPA to **0.928** / RRS **0.978**; the full reasoning-audit layer gives the final RPA **0.944** / RRS **0.983** and adds explicit audit flags + traceability hooks.
- **Runtime cost.** Per-case latency rises from <0.001 s (Greedy/Deadline) and 0.024 s (ILP) to **174.7 s** (Qwen3 Only) and **184.3 s** (Qwen3+Full) — i.e. the validation+feedback loop adds ~10 s over pure inference.
- **Reasoning reusability (§3.6.2, Qwen3 30B, >2000 cases).** With ~300–320 frequent atomic causal units the average reuse rate is ~80% across scenarios (87.5% S1, 86.5% S2, 90% S3). Although >7000 distinct reasoning paths can be generated, only **104 atomic reasoning units** are activated from the expert knowledge base, bounding reasoning complexity.
- **Error-type findings.** The most common failures are: handling abnormal physical values/instrument states beyond reasonable ranges; omitting current-telescope-state constraints; confusing interference *presence* vs *intensity*; in rare cases fabricating interference signals absent from the input. Phi-4 (14B) hallucinates fewer spurious interference signals than Qwen3 (30B).

## Critical Assessment

- **The framework does not beat the strongest deterministic baseline on clean cases.** The authors themselves state the framework's goal is executability/correctness, not optimality: in clean scenarios S1/S2/S6 most methods already saturate at RPA 1.000, and ILP achieves RPA/RRS 1.000 across the board — so the LLM+framework combination's advantage is confined to *complex* multi-constraint scenarios, where it still trails ILP (e.g. S5: Qwen3+Full 0.933 vs ILP 1.000).
- **Reference set is partly framework-co-defined.** The paper acknowledges the reference decision set R_i is built from operational attributes and "is intended to reduce, rather than eliminate, the circularity associated with a framework-defined success criterion" — i.e. RPA is not measured against an entirely independent ground truth, which weakens the headline RPA gains.
- **The 16.8% reasoning-stage gain is only described as "cumulative and sequential"** with no per-scenario confidence intervals reported for that specific increment (CIs are given for EER/RPA/RRS in Table 3 but the staged +18.1/+12.6/+16.8% numbers in §3.6.1 are reported without intervals), making the relative contribution of the ARU/DAG layer vs feedback harder to scrutinize.
- **Cost is large.** At ~184 s per case, the full pipeline is 4 orders of magnitude slower than ILP (0.024 s); for an LSST-scale "millions of alerts per night" setting (the paper's own motivation) this latency is a real deployment obstacle that the paper does not address.
- **Two-model sample.** Only Phi-4 (14B) and Qwen3 (30B) are tested; no frontier closed model (GPT-/Claude-class) is evaluated, so generalization across model families is unverified.

## Strengths

- Clean separation of concerns: validates *executability + consistency* without trying to replace the LLM's optimization, leaving scheduling-quality improvement to the model.
- The ARU+DAG representation is genuinely auditable — it enables localizing the exact failing reasoning step and reusing verified sub-chains, which the paper backs with the 104-activated-units / 80% reuse data.
- Failure-mode taxonomy (Table 1) is concrete and maps each AI failure to a specific validator, useful as a design template beyond astronomy.
- Honest reporting: explicit acknowledgment that pure Qwen3 is not competitive with constraint-aware deterministic baselines, and that the framework's gains are concentrated in hard multi-constraint scenarios.
- Controlled benchmark built from real ZTF alert attributes (sky position, brightness, detection history), with 95% Wilson/bootstrap confidence intervals.

## Limitations

- No comparison against frontier proprietary LLMs; only two open models (Phi-4 14B, Qwen3 30B).
- Per-case runtime ~184 s makes the approach impractical for the LSST alert volumes the introduction motivates.
- Step-level consistency checking explicitly does *not* verify global correctness — a proposal can pass all ARU checks yet still be suboptimal or, in edge cases, globally wrong.
- Benchmark cases are 60/scenario (360 total) — relatively small per cell, and S6 is acknowledged to be more of a safety stress test than a discriminative ranking case.
- No real on-sky deployment: the evaluation is a controlled benchmark from historical candidates, "rather than a full replay of a historical observing night or a complete operational deployment study."
- The "Reference Policy" is constructed by the authors and inherits some circularity they only partially mitigate.

## Questions for Further Exploration

1. The framework judges executability/consistency but explicitly not global optimality — could the ARU/DAG representation be extended to carry quantitative utility scores so that "passes validation but is dominated" proposals can also be flagged?
2. The authors acknowledge residual circularity in the reference decision set R_i; how would RPA change if R_i were derived purely from an independent ILP/optimal solver rather than from operational attributes the framework itself uses?
3. Per-case latency is ~184 s; can the validator's computational-re-evaluation and DB-resolution stages be batched or pre-computed to bring the loop within the seconds-level latency that LSST-scale alert rates would demand?
4. Only Phi-4 (14B) and Qwen3 (30B) are tested — how does the failure-mode distribution (Table 1) and ARU reuse rate change on a frontier closed model, and do the same ~300–320 frequent ARUs still cover ~80% of reasoning?
5. The paper notes S3 strains Qwen3's ARU normalization and causal rigor under multi-target coupling; does adding an explicit multi-target coupling validator (beyond the current per-ARU check) recover S3 RPA closer to the 0.933 seen in S5?
6. Could the closed-loop feedback history (validated proposals) be used to fine-tune the base LLM, and would that reduce the +18.1/+12.6/+16.8% staged gains by making the raw model more constraint-aware — i.e. is the framework self-limiting?
