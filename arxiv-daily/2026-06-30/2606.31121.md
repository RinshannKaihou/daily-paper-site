> **一句话总结：** 本文提出 Janus，一个即插即用的记忆控制器，通过 Memory Momentum Trigger（基于更新方向 EMA 的余弦偏离检测）在"可疑"的轨迹转折点触发新旧记忆对比，并在覆盖/边界/新鲜任务组成的紧凑混合评测集上择优部署记忆；在六个数据集、两个骨干 LLM、两个更新器上，比基础更新器平均提升 +2.7 到 +4.6 个百分点，且不修改底层更新规则。

> **TL;DR:** Janus is a plug-in, method-agnostic memory controller for sequentially evolving LLM memory. It decides whether to accept or reject each candidate memory update using a Memory Momentum Trigger (a directional-deviation check on the update trajectory's EMA) and a compact hybrid evaluation set (coverage + boundary + fresh tasks). Across six datasets, two LLMs, and two updaters, it yields +2.7 to +4.6 average accuracy gains over base updaters, approaching "always-compare" performance at a fraction of the replay cost.

## 核心贡献

- **提出 Janus，一个即插即用、与方法无关（method-agnostic）的记忆控制器**：包裹现有记忆更新器（如 DC-RS、ExpeL），在每一步把"接受候选记忆 vs. 保留旧记忆"建模为部署决策 `M_t ∈ {M_{t-1}, cM_t}`，而不修改更新器本身的更新规则。
- **Memory Momentum Trigger（MMT）解决"何时对比"问题**：用文本编码器把记忆状态映射到向量空间，计算候选更新方向 `z_t = φ(cM_t) − φ(M_{t-1})`，并维护其指数移动平均 `m_t`；仅当 `cos(z_t, m_{t-1}) < τ`（即偏离近期更新方向）时才触发新旧对比，避免每次或定频触发的开销与漏检。
- **紧凑混合评测集解决"用什么对比"问题**：由三部分组成——coverage 集（聚类代表全局任务分布）、boundary 集（记录"翻转任务"即记忆选择改变正确性的样本）、fresh 集（自上次触发后新见到的任务），三者合计 `K=20 + K_F=5` 任务，构成对全量历史回放的低成本近似。三个子集会随任务流动态刷新。
- **跨方法、跨模型、跨数据集的稳健验证**：在六个数据集、两个骨干 LLM、两个更新器上，Janus 相比对应基础更新器平均提升 +2.7 到 +4.6 个百分点；并通过四类消融（触发策略、评测集组成、部署决策、超参敏感性）系统验证了各组件必要性。

## Key Contributions

- **A plug-in, method-agnostic memory controller (Janus):** Wraps existing updaters and reframes each memory update as a deployment decision—accept the candidate or retain the previous memory—without altering the updater's update rule.
- **Memory Momentum Trigger (MMT):** Decides *when* to compare by tracking the candidate update direction `z_t` against an EMA of recent directions `m_t`; it triggers comparison only when directional misalignment `cos(z_t, m_{t-1}) < τ` occurs, capturing meaningful memory-transition points rather than relying on fixed schedules.
- **Compact hybrid evaluation set:** Decides *what* to compare by combining a **coverage** set (cluster-based global representatives), a **boundary** set (tasks whose correctness flips between the two memories), and a **fresh** set (tasks seen since the last trigger). Default sizes: K=20 (12 coverage + 8 boundary) + K_F=5 fresh, all dynamically refreshed.
- **Strong, consistent empirical gains:** +2.7 to +4.6 average accuracy improvements over base updaters, validated across six datasets, two backbone LLMs (Qwen3-8B, DeepSeek-V4-Flash), and two updaters (DC-RS, ExpeL), plus four ablation studies.

## Methodology

**Problem setting.** Following the sequentially evolving memory formulation, an LLM `L` solves a task stream `D={(x_t,y_t)}` while maintaining external memory. At step `t`, it predicts using `M_{t-1}`, receives feedback `f_t`, and a base updater proposes a candidate `cM_t = Update(M_{t-1}, (x_t, ŷ_t, f_t), L)`. Existing methods deploy directly (`M_t = cM_t`); Janus instead chooses `M_t ∈ {M_{t-1}, cM_t}`.

**Core problem (motivation).** Memory-update trajectories exhibit *non-monotonic* test performance (shown on GPQA, Fig. 1): more updates do not always mean better future behavior, because updates optimized locally for the recent task can overwrite useful knowledge, inject over-specific rules, or bias memory toward recent examples. This creates a **memory validation problem** that requires answering two coupled questions: *when* and *what* to compare.

**Janus (two components):**

1. **Memory Momentum Trigger (when).** Map memory states via text encoder `φ`. Candidate update direction `z_t = φ(cM_t) − φ(M_{t-1})`; EMA of directions `m_t = β·m_{t-1} + (1−β)·z_t` (β=0.9). Trigger a comparison only when `cos(z_t, m_{t-1}) < τ` (τ=0.0). Intuition: when consecutive updates are aligned (incremental refinements), repeated comparison adds little; a sharp directional deviation signals the candidate may introduce useful new knowledge *or* distort memory with recent-task-specific content—exactly when validation is worth its cost. If not triggered, Janus accepts the candidate directly.

2. **Hybrid Trigger-Time Evaluation Set (what).** When triggered, compare `M_{t-1}` and `cM_t` on `E_t = S_cov_t ∪ S_bdry_t ∪ F_t`:
   - *Coverage set:* refreshed by clustering embeddings of all seen tasks, keeping the task nearest each centroid (global representativeness).
   - *Boundary set:* built from the **flip set** `{x : 1_corr(x; M_{t-1}) ≠ 1_corr(x; cM_t)}`—tasks where the memory choice actually changes correctness (memory-sensitive cases).
   - *Fresh set:* tasks sampled since the last trigger step `ℓ(t)`, preventing the comparison from becoming a closed loop over a stale replay buffer.
   
   Janus deploys the memory with higher performance on `E_t`; each subset is dynamically refreshed as the stream evolves. The whole design is method-agnostic.

**Baselines:** Memory-free; ExpRAG (lightweight top-k experience retrieval); DC-RS (Dynamic Cheatsheet, Suzgun et al. 2026); ExpeL (reflection-based insights, Zhao et al. 2024).

**Backbones:** Qwen3-8B (thinking mode) and DeepSeek-V4-Flash (non-thinking mode).

## Results & Highlights

**Main results (Table 1, accuracy %):** Janus-enhanced variants are the best method in every LLM block.
- *Qwen3-8B:* DC-RS+Janus lifts average 79.5→83.2 (+3.7); ExpeL+Janus lifts 78.3→81.5 (+3.2). E.g., GPQA DC-RS 73.4→81.5; MMLU-Pro(Phy.) ExpeL 90.4→92.8.
- *DeepSeek-V4-Flash:* DC-RS+Janus 76.7→81.3 (+4.6); ExpeL+Janus 79.6→82.3 (+2.7). E.g., MATH500 DC-RS 79.0→86.2; MMLU-Pro(Eng.) ExpeL 73.2→75.6.
- A striking finding: stronger memory abstraction (DC-RS/ExpeL) does *not* consistently beat lightweight ExpRAG—motivating the central thesis that *whether* to deploy matters as much as *how* to generate memory.

**MMT trigger ablation (Table 2):** Janus matches or beats "Always" (compare every step) at far fewer triggers, and clearly beats Random/Periodic at comparable trigger budgets. On GPQA with DC-RS, Janus (81.5, 72.4% trigger rate) even outperforms Always (80.0, 100%); on HumanEval it reaches Always-level accuracy at only 12–20% trigger rate. This shows trigger *timing* matters, not just trigger frequency.

**Support-set composition ablation (Fig. 3):** Removing coverage, boundary, or fresh all hurt; removing the **fresh** set causes the largest drop on both GPQA and HumanEval—confirming a stale fixed support set becomes self-reinforcing.

**Memory deployment ablation (Fig. 4):** Across 20/40/60/80/100% of the stream, base updaters often improve early then *stagnate or degrade*, while Janus maintains a steadier upward trend and better final accuracy—direct evidence that selectively accepting/rejecting (not just rescheduling) is what helps.

**Hyperparameter sensitivity (Table 3):** K=20 is optimal (larger K raises cost without gain); larger τ increases trigger rate/cost and can raise accuracy but with diminishing returns. Janus thus offers a controllable accuracy-vs-cost trade-off; defaults (K=20, τ=0.0) give a strong balance.

## Critical Assessment

- **Novelty:** Recasting LLM memory updates as an *accept/reject deployment decision* governed by a momentum-style trigger is a fresh, well-motivated angle for LLM agent memory. The EMA-cosine trigger and the coverage/boundary/fresh hybrid set are simple but non-obvious; the boundary (flip-set) idea in particular is elegant. It is not a fundamental algorithmic breakthrough, but a sharp systems-level reframing.
- **Rigor:** Solid empirical breadth (6 datasets, 2 LLMs, 2 updaters) and four well-designed ablations directly targeting the "when" and "what" questions. Per-dataset numbers are reported (not just averages). However, gains are modest (+2.7–4.6 avg), and a few cells show small or uneven improvements (e.g., HumanEval moves are tiny). No statistical significance/variance is reported.
- **Reproducibility:** Method is described in algorithmic detail (equations given, hyperparameters specified, appendices referenced). Method-agnostic design aids adoption. *Caveat:* the abstract/paper does not clearly state code release; the text encoder `φ` choice and clustering implementation details should be verified.
- **Significance:** Practically relevant for memory-augmented LLM agents where unchecked updates degrade long-term performance. The plug-in nature keeps adoption cost low. Conceptually, it argues for "selective scaling" of inference effort—spend validation tokens only when an update may change future behavior.
- **Red flags / open questions:** (1) The compact evaluation set still requires per-trigger LLM rollouts over ~25 tasks—absolute cost (tokens/time) is not reported, only relative trigger counts. (2) τ=0.0 as default yet sensitivity table shows τ=0.2 best on GPQA—sensitive to tuning. (3) Binary accept/reject ignores potentially valuable *partial* updates. (4) Scope is limited to prompt-based memory; training/RL-based memory (e.g., Memory-R1, SkillRL) is explicitly excluded. (5) "Memory-free" sometimes already strong (e.g., MMLU-Pro Physics 84.0 with Qwen3-8B), so absolute headroom varies a lot by task.

## Strengths

- Simple, method-agnostic plug-in design that complements rather than replaces existing updaters.
- Directly targets a well-motivated and empirically demonstrated failure mode (non-monotonic memory utility).
- Efficient decision-making: MMT + compact hybrid set avoids full-history replay, approaching "always-compare" quality at a fraction of the cost.
- Elegant "boundary set" = flip-set design focuses validation where memory choice actually changes outcomes.
- Broad, consistent empirical coverage with thorough ablations isolating each design choice.

## Limitations

- Average gains are modest; some dataset/LLC cells show near-flat or uneven improvements.
- Absolute compute/token cost of the hybrid evaluation set is not reported; only trigger counts and an approximate "cost" proxy are given.
- Trigger threshold τ is sensitive (default 0.0 vs. best 0.2 on GPQA)—likely needs per-task tuning.
- Binary accept/reject; no mechanism for partial or compositional memory updates.
- Scope restricted to prompt-based memory; does not cover training-/RL-based memory evolution.
- Evaluation coverage excludes long-horizon interactive, multi-agent, and non-stationary environments (acknowledged by authors).

## Relevance

该工作与 LLM Agent 记忆系统、test-time 适应及持续学习方向高度契合，提出了即插即用、与方法无关的记忆部署控制器，可直接用于构建更稳健的长程记忆型 Agent；对关注 agent memory、lifelong learning、self-evolving agents 的研究具有较高参考价值。

## Questions for Further Exploration

- What is the absolute token/latency cost of the hybrid evaluation set per trigger, and how does net efficiency compare to simply running a stronger base updater?
- How sensitive is MMT to the choice of text encoder `φ`—does a cheap embedder vs. the LLM's own representation change trigger behavior?
- Could the binary accept/reject be relaxed to partial updates (e.g., merge salient new entries while rejecting over-specific rules)?
- How does Janus compose with hierarchical/retrieval-augmented memory (e.g., G-Memory, A-Mem) or with RL-trained memory (Memory-R1, SkillRL)?
- What are the false-negative (and false-positive) rates of MMT—do harmful updates ever pass unchallenged, and how does the error rate scale with stream length?
- Does Janus still help when the task distribution is non-stationary or adversarially ordered (vs. the shuffled streams studied)?
- Can the boundary/fresh-set mechanism be reused for *memory forgetting/compaction* rather than only accept/reject?
