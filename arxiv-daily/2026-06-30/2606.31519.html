RaBitQCache: Rotated Binary Quantization for KVCache in Long Context LLM Inference

02 Why this matters

A four-in-one design — unbiased estimator + proven O(1/√D) error bound + adaptive Top-p + hardware-aware system — advances long-context LLM deployment, with both theoretical grounding and real engineering payoff.

03 Critical edge

theory under pressure

The error bound assumes uniform keys/queries on the unit hypersphere — but real distributions are clustered (Xing et al. 2026). Robustness under decode-phase drift is unresolved; the 2.16× is also normalized within-system, not head-to-head.

01

Methodology — from float dot-products to binary×INT4

Re-center & normalize

Remove query/key bias using prefill centroids c_q, c_k; precompute all query-only & key-only terms once.

Random rotation → sign bits

One fixed orthogonal matrix P. Keep only the sign of Pᵀk_c — a 1-bit codebook per Key.

c̄ = (1/√D)·sign(Pᵀk)

Store scalar α

One correction factor α[i] = ⟨c̄, Pᵀk_c⟩ per token keeps the estimator exactly unbiased.

Decode: INT4 query

Rotate & uniformly quantize each new query to 4-bit, replacing float inner products with binary×INT4.

Unbiased proxy score

⟨c̄,q′⟩/α estimates true cosine similarity — proven unbiased with O(1/√D) error (Thm A.1/A.2).

Adaptive Top-p

Retrieve the minimal token set whose cumulative softmax mass ≥ p — budget self-sizes to sparsity.

Hybrid attention

Merge retrieved KV with a local sliding window; full-precision write-back buffer doubles as the window.

Hardware-aware pipeline

Async prefill hides O(L) indexing behind O(L²) attention; lazy batched re-quantization on decode.

Kernel: Int4DotBinary GEMV 32 binary elements packed into one 32-bit word → 8× bandwidth cut · shared-memory tiling of the INT4 query · 128-bit vectorized bit-extraction with unrolling. A fused linear-time ternary-search Top-p operator adapts FlashInfer's nucleus kernel. Result: 2.27×–3.43× over naive CUDA.

03

Strengths & limitations

Strengths

  • Genuine unbiased estimator with a proven error bound — rare in KV-cache literature.
  • Adaptive Top-p is principled and tuning-free vs the dominant static Top-k paradigm.
  • Honest empirical work: 8+ tuned baselines, Oracle upper bound, per-task tables, multiple ablations.
  • Real systems engineering — async prefill, lazy updates, custom fused CUDA — not algorithmic hand-waving.
  • Production stack (vLLM + FlashInfer + LMCache) with public code; transparent about limits.

Limitations

  • Uniformity assumption is challenged by clustered real distributions; decode-drift robustness open.
  • Speedups rely on Hopper-specific kernels + vLLM v0.10.2; portability unverified.
  • Cross-method latency is normalized within-system, not a single head-to-head bake-off.
  • Per-token overhead (1-bit index + α + rotation P) not foregrounded in memory accounting.
  • Validated only to ~32K; 128K+ behavior is extrapolated, not benchmarked.
04

Open questions

  1. Does the O(1/√D) bound stay tight at 128K+ under empirically clustered (not uniform) key/query distributions?
  2. Can a distribution-aware rotation relax the uniformity assumption so the guarantee survives decode drift?
  3. What is the absolute head-to-head wall-clock on identical hardware, removing the within-system normalization?
  4. Where is the memory break-even — does per-token overhead (1-bit + α + P) pay off at extreme contexts?
  5. Can the binary×INT4 kernel fuse with FlashAttention or generalize to other bit-widths (e.g. FP8 keys)?
  6. Does composing Top-p with downstream KV compression (KIVI-style) compound gains or add instability?
  7. How robust is the single hyperparameter p under adversarial / out-of-distribution long contexts?
