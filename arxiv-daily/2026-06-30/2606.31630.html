Why this matters

Compile + unit-test is the default judge for LLM code. For probabilistic programs it fails systematically.

The paper reframes correctness as calibration and turns the Bayesian workflow — PPC, SBC, sampler geometry, held-out density — into a deployable verifier-and-repair loop for LLM-written statistical code.

→ reliable LLM scientific-code generation · agentic feedback design
Critical edge
The headline 0.97 / 88% is an upper bound — it needs the correct reference program.

The deployable reference-free figure is only 62–78%. Worse, in repair loops unit-test feedback is significantly worse than no feedback (Claude 25% vs 75%) — passing tests breed false confidence.

Methodology

01 / HOW

A task is a brief plus data from an unknown process p⋆. A program is statistically correct iff its posterior predictive is indistinguishable from p⋆ — two programs can be syntactically near-identical yet encode different densities (Poisson vs. NegativeBinomial differ by one token).

The calibration oracle aggregates four diagnostic families into one verdict:

  • Posterior predictive checks — tail probabilities over {mean, sd, max, #zeros}; near 0/1 flags aspects the model can't reproduce.
  • Simulation-based calibration — rank-uniformity of prior draws; flags inference/specification error.
  • Sampler geometry — split-R̂ <1.05, divergences ≤15, bulk-ESS; catches pathological posteriors (e.g. funnel).
  • Held-out predictive density (lppd) — a strictly proper scoring rule; misspecified models score strictly lower (Gibbs' inequality).

Bug taxonomy: code-visible (hard-coded noise, absurd prior) vs. code-invisible (Gaussian for heavy tails, Poisson for over-dispersed counts). The invisible class is exactly where tests are powerless (Prop. 1) and calibration retains power (Prop. 2).

Repair loop (Alg. 1): LLM drafts → NUTS fits → oracle queried → if not calibrated, a plain-language diagnostic ϕ naming the misfit statistic is fed back, up to K=3. Compared feedback maps: none / test / diag.

Results

02 / WHAT
84%real-LLM-program repair by calibration (n=98) — beats every baseline at p<0.05
  • Detection tiers: reference-free 62% → GLM baseline 68% → model-search 78% → reference oracle 88% (AUC 0.97). Unit tests: 0%.
  • Component ablation: lppd 59%, PPC 56%, sampler 6%, SBC alone just 5% — internally-coherent-but-wrong models pass SBC.
  • Strong-but-unsaturated gains: GPT-5.1 33→92% (+59), Claude 75→100% (+25).
  • Unit tests actively harmful: test ≤ none for capable models (Claude 25% vs 75%); McNemar p=3.4×10⁻³.
  • Real programs: 15–47% of runnable LLM-written code is statistically wrong; unit tests flag none.

Strengths

03 / +
  • Memorable, well-argued thesis backed by both theory (two propositions) and extensive empirics.
  • Honest three-tier framing of the oracle (reference-free → practical → upper bound), not just the best number.
  • Surprising, actionable result: unit-test feedback is worse than none — important for agentic-coding design.
  • Strong, escalating baselines on real LLM-written programs, not just injected bugs.
  • Non-monotone capability analysis yields a clean deployment rule: pair strong-but-unsaturated writers with calibration.
  • PPL-agnostic: R̂, divergences, ESS, PPC, SBC, lppd are first-class in Stan / PyMC / Pyro / NumPyro.

Limitations

04 / −
  • Reference-free detection (the deployable setting) tops out at 62–78%; the upper figure needs an automated model search.
  • SBC is impractical in high dimensions; limited power against untracked statistics (clamped/identity link: 26%).
  • Predictive checks test distributional fit, not causal/generative structure — same-posterior different-mechanism models are indistinguishable.
  • Poor NUTS mixing masquerades as misspecification → false alarms on hard-to-fit families.
  • Diagnosis without remedy: weak writers get a correct diagnostic but can't act (Qwen-7B stays 0%).
  • Scope limited to classical low-dimensional models; high-dimensional / spatial / deep is future work.

Open questions for further exploration

05 / ?
  1. How robust is the reference-free detector to families outside the 10 studied — hierarchical, non-conjugate, discrete-latent, spatial?
  2. Can SBC be scaled into the repair loop for high-dimensional models, adding value beyond PPC + held-out density?
  3. Does the "test feedback is harmful" effect generalize to causal-inference or simulation code — or is it specific to probabilistic programs?
  4. How stable are the strong-but-unsaturated gains (GPT-5.1 +59, Claude +25) across prompt paraphrases and newer versions?
  5. Can passing unit tests be recombined with calibration signals instead of replaced — recovering cheap signal without the false-confidence trap?
  6. What statistics beyond {mean, sd, max, #zeros} close the gap on the clamped-link and stochastic-volatility blind spots?
