> **一句话总结：** 本文提出一个由通用编码 LLM（Claude Opus，经 Claude Code 接口）驱动的多智能体自动形式化框架：单一 orchestrator 编排"陈述形式化"与"证明形式化"两条流水线，借助 type-first 分解与 Auxiliary Lemma（类单元测试）验证机制动态扩展 Mathlib 之外的类型定义；在 PutnamBench 上以 32/32 全对（全库 91.3% 下界，95% 置信）≈$5/题的成本超越专用 prover，并完整形式化 5 篇 STOC 论文主定理（其中 2 篇除 Lean 内核外零公理），还发现了一处 STOC 已发表论文中的证明缺口。

> **TL;DR:** An orchestrator-driven multi-agent framework (Claude Opus via Claude Code) that autoformalizes research-level math into Lean 4 via a "type-first" decomposition strategy and a unit-test-style Auxiliary Lemma technique to dynamically extend concepts beyond Mathlib; it solved 32/32 PutnamBench problems (≥91.3% lower bound at 95% confidence) at ~$5/problem and fully formalized the main theorems of 5 STOC papers — two with no axioms beyond Lean's kernel — while surfacing a verified gap in a published STOC proof.

## 核心贡献

- 提出**单一 orchestrator 编排的多智能体自动形式化系统**，通过 Claude Code 会话动态调度（可重排阶段、回溯、子任务委派），克服了单智能体上下文饱和与固定流水线无法回退的缺陷。
- 引入 **"type-first" 类型分解范式**：在形式化主定理前，先按依赖图顺序显式定义 Mathlib 之外所需领域概念，构建数学词汇表。
- 提出 **Auxiliary Lemma（辅助引理）技术**，类比软件工程中的单元测试——为新类型生成多条尽量通用的性质引理并尝试证明，若失败则说明定义有误，触发重试，从源头防止错误传播。
- 设计 **Faithfulness Judge（忠实度裁判）**：用盲测（back-translation 回译 + 多模型交叉判定）与直接比较双重验证 Lean 代码是否忠实反映原始非正式陈述，单一确认错误即判为无效。
- 构建递归式 **证明形式化流水线**：将证明视为树，每个节点先证明再递归子引理；Proof Critic 在自然语言层审计推理漏洞；Claim Check 确保证明不能偷偷修改待证命题。
- 实证验证：PutnamBench 32/32（≥91.3% 全库下界）、5 篇 STOC 论文主定理形式化（2 篇零额外公理），并发现 STOC 论文 Pham 的 Lemma 3.6 证明存在计算机验证的反例缺口。
- 大幅降低门槛：整个项目运行在 $200/月 Claude 订阅上，5 个月实际花费约 $1,000（API 等价约 $10,762），无需本地 GPU。

## Key Contributions

- A **single-orchestrator multi-agent framework** (Claude Code session as manager) that dynamically reorders stages, backtracks, and delegates sub-tasks — overcoming the context-saturation of single agents and the no-backtrack rigidity of fixed pipelines.
- A **"type-first" decomposition paradigm**: before the main theorem, the orchestrator explicitly defines missing domain-specific concepts (in dependency-graph order) to build the needed mathematical vocabulary beyond Mathlib.
- An **Auxiliary Lemma technique** analogous to software unit testing — generating multiple general-purpose property lemmas for each new type and attempting to prove them; failure flags an incorrect/poor definition and triggers retry, preventing foundational errors from propagating.
- A **Faithfulness Judge** that verifies whether Lean code faithfully captures the informal statement via two blind checks (Informalizer back-translation + cross-model adjudication) and two direct checks; one confirmed error invalidates the formalization.
- A recursive **proof-formalization pipeline** treating a proof as a tree where each node is proved before recursing into its children; a natural-language **Proof Critic** audits reasoning gaps; **Claim Check** guarantees a proof cannot silently weaken the statement it closes.
- Empirical validation: 32/32 on a random PutnamBench sample (≥91.3% lower bound over the full 672-problem benchmark at 95% confidence), main theorems of 5 STOC papers formalized and machine-checked (two with no axioms beyond Lean's kernel), and a computer-verified gap uncovered in a published STOC proof (Pham, Lemma 3.6).
- Drastically lowers the entry barrier: the entire project ran on a $200/month Claude subscription (~$1,000 over 5 months actual vs. ~$10,762 API-equivalent), with no local GPUs.

## Methodology

The system is invoked through the **Claude Code interface**, taking a paper's PDF + LaTeX plus a natural-language prompt, and returning Lean code. A single **Orchestrator** (the Claude Code session) manages two pipelines and can reorder stages, open feedback bridges, and roll back to previous states on failure — deliberately blending structured guidance with non-linear flexibility.

**Statement Formalization pipeline:**
1. **Extractor** — finds the main theorem plus all required definitions/assumptions from PDF/LaTeX.
2. **Type Planner** — determines which types beyond Lean/Mathlib are needed, producing a type plan in dependency order.
3. For each type, a **Lemma Planner** identifies well-known general property lemmas (kept as general as possible, not tailored to the problem). The orchestrator then spawns `k` parallel **Type Formalizer** agents, each translating the type + lemmas into Lean, attempting proofs, and passing them through the Faithfulness Judge.
4. An **Auctioneer** applies best-of-k selection over candidates using three criteria: proportion of auxiliary lemmas proved, semantic alignment from the Faithfulness Judge, and Lean code length. If no candidate passes, additional Type Formalizers are spawned until a correct definition is found.
5. The **Theorem Formalizer** then translates the primary theorem, iterating with the Faithfulness Judge; if it cannot express the statement faithfully, the orchestrator diagnoses and may add/revise upstream types.

**Auxiliary Lemma technique (core novelty):** analogous to unit testing — if a newly defined type is wrong or poorly formalized, its auxiliary lemmas become unprovable, signaling the orchestrator to retry. This guards foundational correctness before any theorem-level work.

**Faithfulness Judge:** runs four independent verifications (two blind, two direct), each pair through different models (e.g., Claude + OpenAI). Blind: an *Informalizer* back-translates the Lean into natural language for comparison; Direct: compares comment-stripped Lean against the informal statement. A judge agent reviews flagged discrepancies and marks each VALID or FALSE POSITIVE; a single VALID concern invalidates the formalization.

**Proof Formalization pipeline:** treats a proof as a recursive tree of nodes (main theorem = root), processed by the same chain at every node:
- **Natural Language Prover** proposes an informal proof (from the statement alone for benchmarks; from the paper's argument for research math), iterated with a **Proof Critic** that raises concrete questions (unstated hypotheses, undefined terms, hidden case splits, swapped quantifiers, silent weaker/stronger steps) until none remain.
- **Proof Detailer** expands into tactic-sized steps; **Lemma Breakdown** produces topologically ordered sub-lemmas with stable identifiers; **Lemma Leanifier** translates each into a Lean statement (body = `sorry`).
- The orchestrator scores each node (depth, statement length, quantifier nesting, source cues) to decide **leaf-or-recurse**; non-leaf nodes are proved *before* their children using children's statements as unproven premises, so a too-weak/wrong child statement is caught at the parent level first.
- **Claim Check** accepts a proof only if the node's statement is preserved byte-for-byte (rejecting import cycles against ancestors), preventing silent weakening.
- **Structured-failure feedback:** on failure the prover reports *missing hypothesis / too-weak-or-contradictory child / stuck goal*, and the orchestrator routes each case differently (re-breakdown, re-formalize child, promote leaf to non-leaf), giving dynamic backtracking. Retry/escalation budgets and a max recursion depth ensure termination.
- **Prior-work handling:** lemmas cited from published prior work are admitted as Lean axioms (verified by the Faithfulness Judge) — the only way external axioms enter. This path is **disabled for benchmark runs** so PutnamBench proofs stay self-contained (only Lean + Mathlib axioms).

## Results & Highlights

- **PutnamBench:** Random sample of 32 problems (seed = 0), all 32 solved end-to-end by the pipeline (statement-only input, no prepared/hand-corrected proofs, internet access disabled to prevent contamination). Yields a **≥91.3% lower-bound accuracy over the full 672-problem benchmark at 95% confidence** (Wilson bound).
- **Cost-efficiency advantage:** ~$5/problem actual cost on a $200 Claude subscription (~$150 for all 32 over three weeks; ~$29/problem at API list rates for comparison). Competitors: AlephProver 94% at ~$54/Q (GPT-5.2, ~1,834 tool calls) or 99.4% at ~$68/Q (Pass@3, $1,400 cap); Seed-Prover 1.5 86% requiring ~10 H20-GPU-days (~$168/Q). Specialized provers (DeepSeek-Prover-V2 7%, Goedel-Prover-V2 12.8%) are far cheaper but vastly weaker.
- **5 STOC papers** formalized (Pham; Mackenzie & Saffidine; Gravin & Jia; Rivkin et al. from STOC 2025; Kalai et al. from STOC 2010), spanning combinatorics, communication complexity, mechanism design, information-theoretic lower bounds, and learning theory — all human-expert-validated and Lean-machine-checked.
- **Axiom spectrum readout (faithfulness signal):** Mackenzie & Saffidine and Gravin & Jia are **fully axiom-free** beyond Lean's kernel (even reproving Chung's product theorem and the Kushilevitz–Nisan log-rank bound from scratch). Pham admits 1 axiom (its Lemma 3.9 bound); Rivkin et al. admits 2 (Prékopa–Leindler, Stirling-type estimate); Kalai et al. admits 3 (Hummel–Gidas, real-analytic implicit function theorem, genericity of analytic zeros).
- **Scientific finding:** While formalizing Pham (STOC 2025), the system **uncovered a computer-verified gap** — the paper's Lemma 3.6 is false as stated (explicit counterexample on a legitimate non-p-small family), so the surjection Ψᵤ used in the proof of the Lemma 3.9 bound is not surjective. The theorem may still be correct; only this proof step fails, and the bound is admitted as a single scoped axiom.
- **Total project cost:** ~$1,000 over 5 months on the $200/month plan (10.8M input / 45.3M output / 11.6B cache-read tokens; ~$10,762 at API list rates — nearly an order-of-magnitude saving from the flat-rate model).
- **Models used:** Claude Opus 4.7 for PutnamBench; Claude Opus 4.8 for STOC papers.

## Critical Assessment

- **Novelty:** High. The standout ideas are (a) treating definitions as first-class objects to be "unit-tested" via Auxiliary Lemmas before any theorem-level formalization, and (b) a genuinely non-linear orchestrator with dynamic backtracking versus the fixed loops of prior pipelines (MerLean, Seed-Prover, Hilbert, Aristotle, M2F). The combination specifically targets the "research math beyond Mathlib" gap that most prior autoformalization work sidesteps.
- **Rigor:** Strong for formalization artifacts (all outputs are machine-checked in Lean; Claim Check prevents statement-weakening; the Pham gap finding is backed by an explicit counterexample). However, several headline numbers rest on a **single n=32 random sample** — the 91.3% is a *lower bound*, not a measured solve rate, and the head-to-head cost comparison uses competitors' self-reported or author-estimated figures (flagged with ⋆) rather than re-run head-to-head. "Validated by human experts" is asserted without a documented protocol (number of reviewers, reviewer expertise, conflict resolution, inter-annotator agreement).
- **Reproducibility:** Moderate-to-good. Formalizations are public and verifiable, which is the gold standard for this subfield. But the backing model is **closed-source Claude** (the authors openly flag this as a limitation: it may change over time and exact results are hard to reproduce). Exact prompts are deferred to an appendix (Appendix D); seed=0 sampling is disclosed. Token logs are priced transparently. No code repo for the orchestrator pipeline itself is promised in the main text.
- **Significance:** Substantial. Demonstrating end-to-end formalization of *published STOC theorems* (with two axiom-free proofs) is a meaningful milestone for AI-assisted mathematics, and the cost story (≈$5/problem, no GPUs) materially changes who can participate. The Pham gap-finding is a concrete example of the pipeline producing scientific value beyond benchmark scores.
- **Red flags / concerns:**
  1. The PutnamBench headline is a 32-problem lower bound, not a full run — direct comparability with systems that ran all 672 problems (Aleph 668/672, Seed-Prover 1.5 581/672) is imperfect. The authors are careful to call it a lower bound, but casual reading could over-state it.
  2. The cost comparison conflates different pass budgets, backbones, and compute paradigms; the ~$5 figure depends entirely on a *flat-rate subscription* not generally available or rate-stable.
  3. "General coding LLMs outperform fine-tuned models" is asserted via leaderboard positioning rather than a controlled backbone swap.
  4. Selection of only the *main* (non-algorithmic) theorems of 5 papers is a curated, favorable target; the authors are transparent about this (Appendix A), but it bounds the generalization claim.
  5. The system depends on frontier closed models (Opus 4.7/4.8), so progress is partly a statement about model capability rather than a transferable method.

## Strengths

- Tackles genuinely hard **research-level** mathematics (STOC theorems), not just competition/textbook problems.
- The Auxiliary Lemma technique is an elegant, transferable idea — applying software-engineering unit-testing discipline to mathematical definitions.
- Orchestrator design with **dynamic backtracking and structured-failure feedback** is a principled advance over rigid fixed pipelines.
- **Publicly released, machine-checked formalizations** plus the Pham gap-finding give concrete, verifiable scientific value.
- Two fully axiom-free STOC formalizations signal strong formal quality and faithfulness discipline.
- Cross-domain evaluation (Putnam + 5 distinct theory areas) demonstrates breadth, and the cost/accessibility story is compelling.

## Limitations

- **Algorithmic results not formalized** (correctness + complexity of reconstruction/learning algorithms) — explicitly deferred to future work (Appendix A).
- **Closed-source backing model** (Claude) — no control over it, subject to change, hindering exact reproduction.
- **Small-scale benchmarking** (n=32) yields a lower bound, not a full PutnamBench solve rate; the ≥91.3% headline is a confidence-bound estimate.
- **Human-expert validation** is asserted without a documented review protocol (reviewers, expertise, adjudication).
- **Reliance on frontier closed models** limits accessibility and means results partly reflect proprietary model capability.
- Cost advantage hinges on a **flat-rate subscription** that may not persist or scale; the metered-API equivalent (~$10,762 for 5 papers) is far higher.
- Only **main theorems** of selected (curated) papers are targeted, not entire papers.

## Relevance

对于关注"大语言模型智能体 + 形式化验证/自动定理证明"方向的工作极具参考价值：本文展示了 orchestrator 编排 + type-first 分解 + Auxiliary Lemma 单元测试如何攻克"Mathlib 之外研究级数学"这一长期难题，并以极低成本（≈$5/题、无需 GPU）刷新了 PutnamBench 的可达性基线，且发现了一处真实 STOC 论文证明缺口——为可信 AI 与 AI-for-Math 提供了可直接借鉴的多智能体范式与公开基线。

## Questions for Further Exploration

- How does per-problem success rate vary across PutnamBench sub-domains (algebra, analysis, combinatorics, number theory), and what classes formalize most/least reliably?
- What is the true full-benchmark solve rate (all 672 problems), and does the 91.3% lower bound hold up?
- How sensitive are results to the backbone (Claude vs. GPT-5.x vs. open-weight), to temperature, and across repeated runs — do the same proofs reproduce?
- Does the Auxiliary Lemma technique scale as definitions grow more complex (e.g., measure-theoretic or sheaf-theoretic types), or does lemma-generation/proof cost explode?
- What are the dominant failure modes (timeouts, divergent proofs, wrong definitions), and what fraction of nodes ultimately require human intervention?
- Could the framework extend to **entire papers** (all theorems + algorithmic correctness/complexity) rather than just main non-algorithmic theorems?
- Can the cost advantage survive the move off a flat-rate subscription — what is the economically sustainable price point at metered rates?
- For the Pham gap: has the underlying theorem since been re-proved correctly, and does the pipeline's counterexample generalize to related fractional-cover arguments?
