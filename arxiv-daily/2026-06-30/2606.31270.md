> **一句话总结：** 本文提出一个失败驱动的自改进闭环——用 Claude 4.5 Sonnet 作为元控制器，分析计算机使用智能体（OpenCUA-72B）在 OSWorld 上的失败轨迹，自动诊断出定位误差、能力缺失、知识不足、冗余循环四类失败模式，并生成经人工轻量验证（97% 无需修改）的推理时代码补丁（视觉搜索、终端执行、知识检索、重复检测），在零训练成本下将成功率从 42.3% 提升至 48.9%（+6.6 pp），且该方法可跨模型规模与 4 个异构基准泛化。

> **TL;DR:** Instead of discarding failed trajectories, this work adds a "failure-case loop": an LLM (Claude 4.5 Sonnet) diagnoses failed OSWorld rollouts of OpenCUA-72B into four failure modes (grounding errors, competency gaps, knowledge deficiencies, redundant loops) and auto-generates lightly-verified inference-time code patches (visual search, terminal execution, knowledge support, repetition detection). It lifts OSWorld success from 42.3% to 48.9% (+6.6 pp, +15.6% relative) with zero training cost, ~8% more runtime, ~15% fewer steps, and generalizes across model scales and four heterogeneous benchmarks.

## 核心贡献

- **指出当前智能体改进流水线的根本低效**：成功驱动的自训练循环（agent → environment → 成功轨迹 → SFT）把失败轨迹全部丢弃，浪费了构建可验证环境的高昂工程成本。
- **提出互补的失败驱动自改进闭环（failure-case loop）**：以数据中心的方式，把失败轨迹转化为推理时的行为改进。闭环为 agent → environment → 失败轨迹 → LLM 分析与改进 → 新 agent，无需任何权重更新。
- **四类通用失败模式及其对应推理时策略**：定位误差→视觉搜索（zoom-in 自校验）；能力缺失→终端执行（Ctrl+Alt+T 热键）；知识不足→知识检索（GPT-5-mini 搜索引擎 + 软件手册）；冗余循环→重复检测（5 步滑窗、阈值 3 次的 thought/action/screen-state 重复检测）。
- **首个经验性证据**：在 SOTA 开源模型 OpenCUA-72B 上，OSWorld 成功率 42.3% → 48.9%（+6.6 pp，+15.6% 相对），零训练成本，运行时仅增加约 8%、交互步数减少约 15%。
- **强泛化性**：同一套从 OSWorld 挖掘的补丁，在 OpenCUA-32B、GUI-Owl-32B 上取得一致的 +10–12% 相对增益，并跨 OmniACT、AndroidControl、ScreenSpotPro、WebVoyager 四个异构基准均有提升。
- **近乎全自动**：LLM 生成的补丁超过 97% 无需人工修改即可接受，人工平均仅改动不足 3% 的代码行。

## Key Contributions

- **Identifies a fundamental inefficiency** in current agent-improvement pipelines: success-driven self-training loops discard all failed trajectories despite the high engineering cost of building verifiable environments.
- **Proposes a complementary failure-driven self-improvement loop (failure-case loop)**: a data-centric paradigm that turns failed trajectories into inference-time behavioral improvements without any weight updates. Loop: agent → environment → failed trajectories → LLM analysis & improvement → improved agent.
- **Four generalizable failure modes + matching inference-time strategies**: grounding errors → visual search (zoom-in post-action self-verification); competency gaps → terminal execution (Ctrl+Alt+T hotkey); knowledge deficiencies → knowledge support (GPT-5-mini search + software-manual retriever); redundant loops → repetition detection (5-step sliding window, threshold of 3, over thought/action/screen-state traces).
- **First empirical evidence** of a failure-driven loop meaningfully improving computer-use agents: OpenCUA-72B on OSWorld 42.3% → 48.9% (+6.6 pp, +15.6% relative), with zero training cost, ~8% runtime overhead, and ~15% fewer interaction steps.
- **Strong generalization**: the OSWorld-mined recipe transfers to OpenCUA-32B and GUI-Owl-32B (+10–12% relative gains) and to four heterogeneous benchmarks (OmniACT, AndroidControl, ScreenSpotPro, WebVoyager).
- **Largely automated**: over 97% of LLM-generated patches are accepted without modification; humans touch on average <3% of modified lines.

## Methodology

The paper adds a second, complementary loop to the standard success-driven self-training pipeline (which fine-tunes only on successful trajectories).

**Framework (Algorithm 1 / Figure 2).** An outer loop runs for K iterations (4 in practice). Each iteration: (1) the current agent rolls out on the task set, producing trajectories verified by the environment's reward function; failed trajectories are retained; (2) a diagnostic LLM (the "meta-controller", Claude 4.5 Sonnet) is fed each failed trajectory's instruction, action history, and thought process, and produces a structured diagnosis of the failure mode; (3) the LLM proposes an inference-time fix and generates a code patch; (4) a human performs light verification/selection; (5) patches accumulate into the agent's inference-time behavior and a new rollout round begins. The loop targets a *different dominant failure mode each round*, so patches compose constructively rather than pile up ad hoc.

**Meta-controller selection.** Claude 4.5 Sonnet was chosen over GPT-5.2, Gemini 3 Flash, and Qwen3-VL-32B-Instruct because it was best at jointly analyzing failures, proposing actionable improvements, and producing working code.

**The four derived strategies** (each addressing a failure category the LLM surfaced):

1. **Visual Search (grounding errors).** After any spatially-grounded action (click/moveto/dragto), a 400×400 region centered at the target is cropped, upscaled 2×, and overlaid with a red 7px-radius circle marking the intended point. The agent is re-prompted with this view + instruction + action history to verify/adjust coordinates. Closes the action→visual-verification loop.
2. **Terminal Execution (competency gaps).** Agents default to fragile low-level GUI manipulation. A Ctrl+Alt+T hotkey opens a terminal so the agent can translate operation-intensive tasks into concise shell commands. Rules force terminal-only access via the hotkey, syntax pre-verification via search, and explicit Enter to run commands.
3. **Knowledge Support (knowledge deficiencies).** Two mechanisms: (a) a search-engine interface to GPT-5-mini returning concise, action-oriented guidance (the agent may query multiple times per step); (b) a curated software-manual reference (e.g., LibreOffice Calc/Writer hotkey sheet) injected at test time.
4. **Repetition Warnings (redundant loops).** A sliding window (latest 5 steps) monitors three signals: thought repetition, action repetition (identical pyautogui commands), and screen-state repetition (a stable hash of the OSWorld accessibility-tree UI-node subtree). A warning is fired when any pattern recurs ≥3 times (screen-state: identical hash for 3 consecutive steps despite actions). The warning activates a "recovery mode" prompting the agent to switch strategy (search, different tool, or terminal).

**Evaluation.** OSWorld (real computer-use, 100 steps, success rate). Comparisons against the leaderboard. Main results are mean ± std over 3 independent runs (single-run configs marked *). Ablations use the OSWorld "small set" with a 30-step limit. Cross-model tests apply the same patch recipe to OpenCUA-32B and GUI-Owl-32B; cross-benchmark tests transfer OSWorld-mined patches to OmniACT, AndroidControl, ScreenSpotPro, WebVoyager using Qwen3-VL-32B-Instruct as backbone. Tool access, environment config, and action space are kept identical to the baseline for fairness.

## Results & Highlights

**Main result (OSWorld, 100 steps).** OpenCUA-72B: **42.3 ± 2.6 → 48.9 ± 1.2** (+6.6 pp, +15.6% relative), beating OpenAI CUA o3 (42.9%) and UI-TARS-1.5 (42.5%) on the leaderboard, at no training cost, ~8% more runtime, ~15% fewer interaction steps.

**Ablation (OSWorld small set, 30 steps, Table 1).** Baseline 41.67 →
- +Visual Search: 47.22 (single biggest individual contributor)
- +Terminal Execution: 47.19
- +Knowledge Support: 44.44
- +Repetition Detection: 44.40
- +Full Method (all four): **52.74** — combination clearly beats any single module.

**Cross-model (Table 2).** Uniform +10–12% relative gains regardless of origin/scale:
- GUI-Owl-32B: 19.0* → 21.3*
- OpenCUA-32B: 34.5* → 38.2*
- OpenCUA-72B: 42.3 → 48.9 (stronger models benefit most).

**Cross-benchmark (Table 3, OSWorld patches transferred).** All improve:
- OmniACT 4.77 → 6.90
- AndroidControl 28.37 → 36.23 (+7.86, largest — long-horizon planning + interactive commands)
- ScreenSpotPro 27.50 → 30.74
- WebVoyager 23.80 → 27.90 (+4.10)

**Failure-mode transition (Figure 6).** Before: action loops 43%, coordinate precision 24%, UI recognition 19%, multi-step planning 15%. After 4 rounds: remaining failures shift toward higher-level cognition — multi-step planning 28%, UI recognition 24%, action loops 24%, task misinterpretation 12%, CAPTCHA 12%. The framework cleanly removes low-level bottlenecks and pushes the residual frontier into semantic/reasoning capability.

**Automation level.** >97% of LLM-generated patches accepted without modification; human edits touch <3% of modified lines on average — the loop is "predominantly model-driven."

## Critical Assessment

- **Novelty: Moderate-to-high.** The high-level "learn from failures" idea is not new, but the concrete operationalization — an LLM meta-controller that auto-diagnoses failure modes and emits verified inference-time *code patches* that accumulate constructively over rounds for computer-use agents — is fresh and well-executed. The four failure categories are conventional (grounding, tools, knowledge, loops) rather than surprising.
- **Rigor: Moderate-to-high.** Main results report mean ± std over 3 runs; ablations isolate each module and show the combination wins; cross-model and cross-benchmark generalization strengthen the claim. However, several configs are single-run (*) due to compute constraints, and no statistical significance testing is reported.
- **Reproducibility: Moderate.** Code is released and the setup (named model + standard benchmark) is concrete. But the "light human verification" gate, the choice of meta-controller (Claude 4.5 Sonnet), and the curated software-manual reference are manual/prompt-engineering-dependent and harder to replicate exactly. Reliance on proprietary 2026-era frontier models (Claude 4.5 Sonnet, GPT-5-mini, GPT-5.2, Gemini 3 Flash) as moving targets is a practical concern.
- **Significance: Practical.** The "no training cost, inference-time patches" framing is genuinely attractive for a field where SFT/SFT-with-RL is expensive. The failure→patch reframing is a useful lens that other teams can adopt.
- **Red flags / concerns:**
  - The "self-improvement" branding is generous — this is largely *human- and LLM-guided inference-time engineering* (prompt scaffolding, tool hooks, repetition detectors). The agent's *weights* never change; it is the surrounding harness that improves. Whether this is "the agent improving itself" is debatable.
  - The cross-benchmark gains use a *different, weaker backbone* (Qwen3-VL-32B-Instruct) than the main result, making direct effect-size comparisons to OSWorld muddier.
  - Patches were mined from OSWorld; the cross-benchmark "generalization" partially conflates "general failure patterns" with "generic helpful scaffolding (zoom-in, terminal, search, loop-breaker)" that any agent benefits from regardless of failure analysis.
  - The Visual Search module adds an extra verification step per grounded action — the ~8% runtime overhead claim may not capture full cost on harder tasks.
  - No comparison to simply spending the equivalent compute on more successful-trajectory collection (the baseline the paper argues against), so the opportunity-cost argument is asserted, not measured.

## Strengths

- Exploits a genuinely underutilized signal (failed trajectories) that current pipelines discard.
- Zero training cost; purely inference-time, plug-and-play.
- Concrete, measurable, leaderboard-competitive gains on a recognized benchmark, with error bars.
- Honest ablations showing each module contributes and that they compose.
- Useful failure-mode taxonomy that other computer-use agent work can reuse.
- High automation (>97% patches accepted unmodified) addresses the obvious scalability worry.
- Code released.

## Limitations

- Human-in-the-loop verification, while minimal, is still a manual gate and a reproducibility hazard.
- No statistical significance testing; some configs single-run.
- The four strategies are fairly conventional engineering (zoom-in grounding, terminal access, web/manual search, loop-breaking) — the "failure-driven discovery" is partly post-hoc narrative for known agent weaknesses.
- Cross-benchmark evaluation uses a different backbone than the headline result.
- Reliance on proprietary frontier models (Claude 4.5 Sonnet, GPT-5-mini) whose availability/behavior may shift.
- The harness grows in complexity; long-term patch-conflict/saturation behavior across many rounds is not studied.
- "No training cost" omits the engineering and meta-controller (LLM API) cost of running the loop itself.

## Relevance

本文与计算机使用智能体（computer-use agents）、LLM 自改进、推理时能力增强（inference-time scaling）方向高度相关。对于关注"无需重新训练即可提升智能体成功率"以及"如何从失败数据中提炼可复用改进"的研究者与工程师，其失败→补丁闭环、四类失败模式分类法及跨模型/跨基准的泛化证据具有直接的参考价值。

## Questions for Further Exploration

- How does this compare, in equal-compute terms, to simply collecting more successful trajectories for SFT, or to one more round of RL fine-tuning?
- Do the four accumulated patches ever conflict or degrade performance as the loop runs beyond 4 rounds? Is there a saturation point?
- Can the human verification gate be fully removed with a learned reward/validity model, and at what cost to patch quality?
- Would the failure-mode taxonomy and patches survive a non-LibreOffice, non-Ubuntu OSWorld domain shift (e.g., Windows/macOS tasks)?
- How sensitive are the gains to the choice of meta-controller — would an open-source meta-controller (Qwen3-VL-32B-Instruct) recover most of the benefit?
- Does Visual Search's per-action overhead scale adversely on tasks with many grounded actions, and can it be triggered selectively?
- Is the cross-benchmark gain truly from "generalizable failure patterns" or mostly from generic helpful scaffolding (terminal/search/loop-breaker)?
