Why this matters

A plug-in, method-agnostic deployment controller that directly builds more robust long-horizon memory agents — highly relevant to agent memory, lifelong learning, and self-evolving agents.

Critical edge

Absolute token/latency cost of the ~25-task hybrid eval set per trigger is never reported — only trigger counts. Gains are modest and the threshold τ is sensitive (default 0.0, yet 0.2 is best on GPQA).

01

Methodology

two coupled questions · when & what to compare

Memory-update trajectories are non-monotonic: more updates do not always mean better future behaviour. Locally-optimal updates can overwrite useful knowledge, so Janus reframes every update as a binary deployment decision.

At step t the base updater proposes a candidate memory cM_t. Existing methods deploy it directly — Janus instead chooses M_t ∈ {M_{t−1}, cM_t}: accept, or retain the old memory.

① Memory Momentum Trigger — when. Encode each memory via φ; track the candidate update direction against an EMA of recent directions. Trigger comparison only on a sharp directional deviation.

z_t = φ(cM_t) − φ(M_{t−1})
m_t = β·m_{t−1} + (1−β)·z_t  β=0.9
trigger iff cos(z_t, m_{t−1}) < τ  τ=0.0

② Hybrid Evaluation Set — what. When triggered, compare the two memories on K=20 + K_F=5 tasks, each subset dynamically refreshed:

Coverage
cluster representatives of the global task distribution (global representativeness)
Boundary
the flip set — tasks whose correctness changes between the two memories
Fresh
tasks seen since the last trigger — prevents a stale, self-reinforcing replay loop

Janus matches or beats Always at far fewer triggers — on HumanEval it reaches Always-level accuracy at just 12–20% trigger rate. Trigger timing, not frequency, is what matters.
Stronger ≠ better
Higher-abstraction memory (DC-RS / ExpeL) does not consistently beat lightweight ExpRAG — motivating the thesis that whether to deploy matters as much as how to generate memory.
Selective deployment, not rescheduling
Across 20–100% of the stream, base updaters improve then stagnate or degrade; Janus keeps a steadier upward trend and better final accuracy. Removing the fresh set hurts most.
03

Strengths & Limitations

critical assessment

+ Strengths

  • Method-agnostic plug-in — complements rather than replaces existing updaters.
  • Targets a well-demonstrated failure mode: non-monotonic memory utility.
  • Approaches "always-compare" quality at a fraction of the replay cost.
  • Elegant boundary = flip-set design — validates where memory choice actually changes outcomes.
  • Broad coverage (6 datasets, 2 LLMs, 2 updaters) with thorough ablations isolating each choice.

− Limitations

  • Average gains are modest; some cells near-flat (e.g. HumanEval).
  • Absolute token/latency cost of the eval set is not reported — only trigger counts.
  • Threshold τ is sensitive (default 0.0 vs. best 0.2 on GPQA) — likely needs per-task tuning.
  • Binary accept/reject only; no partial / compositional updates.
  • Scope limited to prompt-based memory — excludes training-/RL-based (Memory-R1, SkillRL).
  • No long-horizon, multi-agent, or non-stationary evaluation (acknowledged).
04

Open Questions

for further exploration
What is the absolute token/latency cost per trigger, and how does net efficiency compare to a stronger base updater alone?
How sensitive is MMT to the choice of text encoder φ — a cheap embedder vs. the LLM's own representation?
Can binary accept/reject be relaxed to partial updates (merge salient entries, reject over-specific rules)?
How does Janus compose with hierarchical / retrieval memory (G-Memory, A-Mem) or RL-trained memory?
What are the false-negative / false-positive rates of MMT — do harmful updates pass unchallenged, and how does error scale with stream length?
Does it still help under non-stationary or adversarially ordered task streams (vs. the shuffled streams studied)?
