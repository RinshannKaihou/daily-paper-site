Directly relevant to computer-use agents, LLM self-improvement & inference-time scaling — lifting agent success without retraining, and mining reusable fixes from the failures every pipeline currently throws away.
Critical edge
The "self-improvement" branding is generous — the agent's weights never change. This is human- & LLM-guided inference-time engineering (prompt scaffolding, tool hooks, loop-breakers), not the agent improving itself.
Cross-benchmark gains use a weaker backbone (Qwen3-VL-32B); patches may be generic helpful scaffolding, not failure-specific insight. No significance tests; some configs single-run.
01 — Methodology

A second loop that feeds on failure, not success.

Standard self-training fine-tunes only on successful trajectories and discards the rest — wasting the high cost of building a verifiable environment. This work adds a complementary failure-case loop that turns failures into inference-time behavioural improvements, with no weight updates.

agent→ environment→ failed trajectories→ LLM analysis & patch→ improved agent
Outer loop, K=4 rounds. Each round: roll out → retain failures → a meta-controller (Claude 4.5 Sonnet, picked over GPT-5.2 / Gemini 3 Flash / Qwen3-VL-32B) diagnoses the dominant failure mode, proposes a fix, emits a code patch; light human verification; patches accumulate. Each round targets a different failure mode, so patches compose rather than pile up.
01
GROUNDING ERRORS

Visual Search

closes the action→verify loop

After any click/move/drag, crop a 400×400 region around the target, upscale 2×, overlay a red marker, and re-prompt the agent to verify or correct coordinates.

02
COMPETENCY GAPS

Terminal Execution

GUI fiddling → shell command

A Ctrl+Alt+T hotkey opens a terminal so operation-heavy tasks collapse into concise shell commands, with syntax pre-checks and explicit Enter.

03
KNOWLEDGE DEFICIENCIES

Knowledge Support

two retrieval paths

(a) A search interface to GPT-5-mini returning action-oriented guidance; (b) a curated software-manual reference (e.g. LibreOffice hotkeys) injected at test time.

04
REDUNDANT LOOPS

Repetition Warnings

5-step window · threshold 3

A sliding window watches thought / pyautogui-action / screen-state-hash repetition; on ≥3 recurrences it fires a recovery mode prompting a strategy switch.

03 — Strengths & limitations, side by side

A practical lens — with real caveats.

Strengths

  • Exploits a genuinely underutilized signal (failed trajectories) that pipelines discard.
  • Zero training cost; purely inference-time, plug-and-play.
  • Leaderboard-competitive gains on a recognized benchmark, with error bars.
  • Honest ablations: each module contributes and they compose.
  • Reusable failure-mode taxonomy for the computer-use community.
  • >97% patches auto-accepted — addresses the scalability worry.
  • Code released.

Limitations

  • Human verification gate, while minimal, is a reproducibility hazard.
  • No significance testing; some configs are single-run.
  • The four strategies are conventional engineering; "failure-driven discovery" is partly post-hoc narrative.
  • Cross-benchmark eval uses a different (weaker) backbone than the headline.
  • Relies on proprietary frontier models (Claude 4.5 Sonnet, GPT-5-mini) — moving targets.
  • Harness complexity grows; long-term patch-conflict / saturation beyond 4 rounds unstudied.
  • "No training cost" omits the meta-controller (LLM API) and engineering cost of running the loop.
04 — Open questions

Where the loop breaks down.

?

Equal-compute baseline? How does this compare to simply collecting more successful trajectories for SFT, or one more RL round? The opportunity-cost argument is asserted, not measured.

?

Patch saturation? Do the four accumulated patches ever conflict or degrade as the loop runs beyond 4 rounds?

?

Remove the human? Can the verification gate be fully replaced by a learned reward/validity model — and at what cost to patch quality?

?

Domain shift? Do the taxonomy and patches survive non-LibreOffice / non-Ubuntu OSWorld tasks (Windows, macOS)?

?

Meta-controller sensitivity? Would an open-source meta-controller (Qwen3-VL-32B) recover most of the benefit?

?

Generic scaffolding vs. failure insight? Is the cross-benchmark gain truly generalizable failure patterns — or mostly terminal/search/loop-breaker that helps any agent?
