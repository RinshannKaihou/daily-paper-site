> **一句话总结：** 本文提出 InterFLOPBench 基准（90 个 C 内核、1130 个测试样本，覆盖六类浮点错误），以 FPChecker（经硬化加固）+ Herbgrind 作为真值预言机，系统评测了 17 个大语言模型（2023.10–2026.02）对浮点错误的静态分类能力；最新模型（gpt-oss 120b、Qwen 3.5 27b）整体 F1 超过 0.90，但模型在显性错误（除零、NaN）上表现远好于隐性数值现象（抵消、下溢），且通过 LLM-与-预言机的分歧分析反过来发现并修复了 FPChecker 的三个盲区。

> **TL;DR:** The authors introduce InterFLOPBench (90 short C kernels, 1,130 samples, 6 floating-point error categories + no-error), using a hardened FPChecker plus Herbgrind as the ground-truth oracle, to benchmark 17 LLMs (Oct 2023–Feb 2026) on static floating-point error classification; the latest models (gpt-oss 120b, Qwen 3.5 27b) exceed 0.90 overall F1, dramatically outperforming on explicit errors (division by zero, NaN) versus subtle ones (cancellation, underflow), and the LLM-vs-oracle disagreement analysis itself surfaced three genuine FPChecker blind spots that were then patched.

## 核心贡献

- **构建 InterFLOPBench 多标签基准**：90 个手写 C 内核（10–30 行）+ 1130 个测试样本，覆盖七大标签（抵消 cancellation、比较 comparison、除零、上溢 overflow、下溢 underflow、NaN、no-error），共 1307 个错误实例（错误可共存，如除零常伴随上溢）。代码与数据集已开源：https://github.com/interflop/InterflopBench 。
- **将浮点错误检测建模为多标签分类**，以 F1-score（按类别与微平均）为主指标，使评估标准化、可复现。
- **跨 17 个模型的系统性横向评测**（2023.10–2026.02），覆盖不同规模、架构（Dense/MoE）与推理机制（CoT/RL 强化），揭示能力随时间显著提升，最新模型整体 F1 > 0.90。
- **提示工程的对照实验**：设计 3 档提示（类别清单 → 加定义 → 强制 detailed_explanation），证明精确错误定义与显式逐步推理可使 F1 提升最高达 +0.28（Gemma 3 27b：0.35→0.63），对无内部 CoT 的模型增益尤其显著。
- **LLM-与-预言机分歧分析反过来加固真值**：分析 gpt-oss-120b 与 FPChecker 的 86 处分歧，发现其中 28 例 LLM 正确而预言机漏报，由此定位并修复了 FPChecker 的三类盲区（FMA、`!=` 比较、math.h 库函数），证明 LLM 可作为传统动态分析工具的审计/互补层。

## Key Contributions

- **InterFLOPBench multi-label benchmark**: 90 hand-crafted C kernels (10–30 lines) with 1,130 samples spanning 7 labels (cancellation, comparison, division by zero, overflow, underflow, NaN, no-error; 1,307 non-exclusive error occurrences). Open-sourced at https://github.com/interflop/InterflopBench.
- **Multi-label classification formalization** with F1-score (per-category and micro-averaged) as the primary metric, yielding a standardized, reproducible evaluation.
- **Systematic 17-model evaluation** (Oct 2023–Feb 2026) spanning diverse sizes, architectures (Dense/MoE) and reasoning mechanisms (CoT/RL), documenting clear chronological improvement, with the latest models (gpt-oss 120b, Qwen 3.5 27b) clearing >0.90 overall F1.
- **Prompt-engineering ablation**: three prompt tiers (category list → +definitions → +`detailed_explanation`) yield up to +0.28 F1 (Gemma 3 27b: 0.35→0.63), with the largest gains for models lacking internal chain-of-thought.
- **LLM-vs-oracle disagreement analysis that hardens the ground truth itself**: of 86 disagreements between gpt-oss-120b and FPChecker, the LLM was correct in 28 cases where the oracle was silent, directly motivating three FPChecker fixes (FMA handling via `-ffp-contract=off`, rewriting `a != b` as `!(a == b)`, and forcing propagation through `math.h` via `+0.0f`). This establishes LLMs as auditors that surface blind spots in conventional dynamic instrumentation.

## Methodology

- **Task framing**: Static floating-point error detection is cast as a **multi-label classification** problem — given a C kernel plus anonymized JSON input datasets (the model sees inputs but not FPChecker's predicted errors), it must label each input with one or more of the six error categories (or `no_error`).
- **Benchmark construction**: 90 deliberately-crafted kernels that elicit specific numerical anomalies, each paired with a JSON metadata file of structured input datasets designed to trigger (or avoid) the target anomalies. Category distribution (occurrences): Comparison 249, Cancellation 208, Overflow 152, Underflow 93, Division by Zero 70, NaN 55, No-Error 480.
- **Ground-truth oracle**: A **hardened FPChecker** (instruction-level exception detection for NaNs, infinities, underflow, cancellation), cross-validated against **Herbgrind** (output-accuracy) with manual inspection of disagreements. Three FPChecker limitations were patched via code hardening (no change to the error taxonomy): FMA fusion masked cancellation (`-ffp-contract=off`), `!=` comparisons were overlooked (rewritten as `!(a==b)`), and `math.h` domain errors (e.g., `log(negative)`) weren't instrumented (forced via `+0.0f` propagation).
- **Models**: 17 LLMs spanning Oct 2023–Feb 2026 — Mistral 7B, GPT-4o, Llama 3.1 8B, DeepSeek R1 (8B/32B), Gemini 2.0 Flash, Gemma 3 (4B/12B/27B), DeepSeek R1T 685B, Phi-4 Reasoning 14B, Gemini 2.5 Flash, DeepSeek R1T2 671B, gpt-oss (20B/120B), Qwen 3 32B, Qwen 3.5 27B.
- **Prompts**: Three tiers tested on Qwen 3 32b (thinking-capable) and Gemma 3 27b (non-reasoning). Prompt 3 (full definitions + `detailed_explanation`) is retained for all evaluations.
- **Metric**: F1 = 2TP/(2TP+FP+FN), reported per-category and micro-averaged overall.

## Results & Highlights

- **Latest models clear >0.90 overall F1**, with gpt-oss 120b and Qwen 3.5 27b showing the highest scores and lowest per-category variability. Clear chronological improvement from Mistral 7B (Oct 2023) onward.
- **Large per-category variance**: explicit errors are much easier than subtle numerical phenomena — division by zero averages **F1 ≈ 0.83**, underflow **≈ 0.65**, cancellation **≈ 0.62**. Intermediate overflow (final result representable but intermediate path overflows) is another blind spot.
- **Prompt engineering matters, asymmetrically**: precise definitions help reasoning models (Qwen 3 32b: 0.67→0.86→0.89); the same definitions *hurt* Gemma 3 27b by biasing it toward over-predicting `comparison` errors, but the `detailed_explanation` field rescues it (0.35→0.44→0.63, +0.28 total). Eliciting explicit step-by-step reasoning is most beneficial for models without internal CoT.
- **LLM-as-auditor finding**: In 28/86 disagreements the LLM was correct and the (standard) FPChecker was silent — covering FMA-related cancellation, unsafe `!=` comparisons, and `math.h` domain errors. The reported F1 is measured against the *hardened* ground truth; against standard FPChecker many LLM "false positives" would actually be true positives.
- **LULESH real-world case study**: a guaranteed NaN `(a−a)/(a−a)` injected into the full 60k-token LULESH `.cu` file. Gemini 2.0/2.5/3.0 Flash (1M-token context) all caught the direct pattern; under progressively harder formulations, Gemini 2.0 Flash misclassified the hardest case (computed-variable numerator/denominator) as "Division by Zero", while 2.5/3.0 succeeded.
- **Training methodology > raw scale**: models with explicit reasoning reinforcement (GRPO/CoT-RL) outperform larger non-reasoning peers — Phi-4 Reasoning (14B) surpasses GPT-4o. Multi-step numerical reasoning, not data exposure alone, is the key differentiator.
- **Cost**: non-reasoning models emit 500–1,000 tokens/sample; reasoning models (DeepSeek R1, Qwen 3.5) emit 5,000–10,000 (5–10× overhead). The overhead is justified for the hardest categories (cancellation, underflow); non-reasoning models are more cost-effective for division-by-zero/NaN.

## Critical Assessment

- **Novelty**: High within its niche. Benchmarking LLMs *specifically* on IEEE-754 floating-point error *classification* (not generic bug detection, not input generation) is genuinely under-explored, and the multi-label framing with a runtime oracle as ground truth is clean. The most novel element is the bidirectional result: the LLM-vs-oracle disagreement loop *itself* improved the oracle, turning the benchmark into a debugging tool for FPChecker.
- **Rigor**: Strong. Ground truth is defended explicitly (hardening + residual-disagreement discussion); multi-label TP/FP/FN accounting is laid out in a table; per-category F1 is reported (not just headline numbers); cost is reported. The authors are refreshingly honest that the reported F1 *slightly underestimates* LLMs because of deliberately-unrelabelled silent-precision-loss cases.
- **Reproducibility**: High in principle — fixed benchmark, open repo (`github.com/interflop/InterflopBench`), a published per-benchmark disagreement-analysis page, named models and prompts. Caveats: exact prompt text and per-sample raw outputs should be verified in the repo; proprietary models (Gemini, GPT-4o, gpt-oss) are non-deterministic and version-drifting.
- **Significance**: Meaningful for HPC / scientific-computing reliability and for "LLM code intelligence." Practitioner value is real (cheap semantic pre-filter for FP debugging). Academic novelty is carried by the disagreement-analysis loop rather than by the headline F1.
- **Red flags / concerns**: (1) **Scale is modest** — 90 kernels / 1,130 samples; per-category counts are small (NaN 55, Division-by-Zero 70, Underflow 93), so category-level F1 differences carry real noise. (2) **Hand-crafted kernels** may resemble common textbook patterns the models memorized; the LULESH injection (`a-a)/(a-a)`) is syntactically trivial, weakening the "real-world" claim. (3) **Single language (C), single-turn, isolated kernels** — the authors own this limitation. (4) The headline figures rose between the abstract in the metadata service (14 models, >0.88) and the PDF (17 models, >0.90) — benign version drift, but worth noting.

## Strengths

- First dedicated, open benchmark for LLM-based floating-point error classification with a defensible runtime ground truth.
- Honest, bidirectional evaluation: the benchmark *audits the auditor*, uncovering three real FPChecker blind spots.
- Clean prompt-engineering ablation with a non-obvious asymmetric finding (definitions help reasoning models but can hurt non-reasoning ones).
- Per-category + cost reporting, and explicit acknowledgment of where the metric underestimates LLMs.
- Strong domain credibility (InterFLOP/Verificarlo authors, Intel co-author).

## Limitations

- **Scale**: 90 kernels / small per-category counts (NaN 55, DivZero 70) limit statistical confidence on the hardest categories.
- **Hand-crafted, C-only, isolated kernels**: risk of pattern memorization; no Fortran/C++/Python/NumPy; no multi-file or agentic pipelines.
- **Single-turn single-prompt**: no iterative refinement or tool-use that could change model ranking.
- **Trivial real-world case**: the LULESH injection is a syntactically obvious NaN; harder bugs are only tested across 3 scenarios on 3 Gemini models.
- **Proprietary model drift**: Gemini/GPT-4o/gpt-oss results are not bit-reproducible over time.
- **Oracle conflation**: exception-based (FPChecker) vs output-accuracy (Herbgrind) semantics are deliberately not merged, so some genuinely dangerous silent-precision-loss bugs are scored as LLM false positives.

## Relevance

该论文处于「代码大模型 + 数值可靠性」的交叉点，与代码静态分析、LLM 代码智能评估及 HPC/科学计算可靠性等方向高度相关；其「LLM 反向审计传统工具」的思路对构建混合（LLM + 动态/形式化）分析流水线尤具启发，可作为衡量大模型深度数值推理能力的参考基准。

## Questions for Further Exploration

- Would scaling the kernel set (e.g., 500+) and balancing per-category counts change the ranking, especially for cancellation/underflow?
- Can the disagreement-analysis loop be automated — i.e., systematically mine LLM "false positives" against a tool to discover new tool blind spots?
- How does performance change on real, non-injected bugs mined from HPC projects (versus hand-crafted textbook kernels)?
- Does extending to Fortran / Python+NumPy / Julia preserve the per-category ordering of model difficulty?
- Could agentic/multi-turn pipelines (LLM calls FPChecker/Herbgrind as tools) close the cancellation/underflow gap without prohibitive cost?
- Is the asymmetry of prompt-engineering effects (definitions help reasoning models, hurt non-reasoning ones) robust across more model pairs?
- How much of the latest models' edge comes from exposure to numerical-analysis corpora vs. reasoning ability — and is it testable via contamination checks?
