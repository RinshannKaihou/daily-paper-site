Key Finding

Memory aging in GPU LLM serving is universal — and completely invisible to client-side monitoring.

Characterizing Software Aging in GPU-Based LLM Serving Systems ↗ Domenico Cotroneo Bojan Cukic UNC Charlotte 2606.11916

Why This Matters

Standard latency and throughput dashboards report no anomaly even as host memory slowly bleeds. Production operators are flying blind — their current monitoring cannot detect this class of degradation before it compounds.

Critical Edge

Single run per configuration. Confidence intervals capture only within-run variance. Run-to-run variability is unquantified, making specific slope estimates unreliable until replicated — the authors acknowledge this honestly but it limits actionability.

Statistical Pipeline
  1. Hamed-Rao corrected Mann-Kendall — handles near-unit autocorrelation (ρ ≈ 0.99) by inflating variance via effective sample size from empirical ACF
  2. Theil-Sen slope estimator — robust median-of-pairwise-slopes with CI variance inflated by AR(1) factor (1+ρ)/(1−ρ)
  3. Benjamini-Hochberg FDR control — q = 0.10 across ~200 simultaneous tests; guards against multiple comparison inflation
  4. Trend declared significant only when MK rejects null and 95% Theil-Sen CI excludes zero — double gate

Counterintuitive ordering: most optimized engine leaks least; naive PyTorch sits in the middle; production-hardened Triton+V0 stack leaks most. Aging source is the runtime/orchestration layer, not the shared CUDA inference path.

GPU VRAM: stationary in all runs — paged KV-cache management successfully contains device-side fragmentation.
Load independence: E3b at 0.05 RPS leaks more (+103 KB/h vs +31 KB/h) than high-load E3 — aging is time-driven, not request-volume-driven.

Strengths & Limitations
Strengths
  • First SAR study targeting GPU-resident, framework-mediated LLM serving with factorial engine comparison
  • Autocorrelation correction + FDR control — above the SAR field norm
  • 2×2 factorial disentangles engine generation from hosting layer
  • Qualitatively novel step-wise growth pattern in Triton+V0 with practical relevance
  • Honest scoping; thorough threats-to-validity section; no overselling
Limitations
  • Single run per config — within-run CIs only; run-to-run variance unquantified
  • Version confound in factorial (V0 ≠ V1 engine generation is aliased with vLLM version)
  • Synthetic Poisson workload; no bursty arrivals, cache hits, or temporal locality
  • Single hardware platform (L40S) and single model (Qwen2.5-7B); no H100/B200 data
  • No mechanism identification — heap profiling is listed as future work
  • No rejuvenation strategies proposed or evaluated

Monitoring blind spot: Latency (p50/p95/p99), TTFT, throughput, and drop rate are stationary in virtually all 36-hour windows. The ~5 ms/h p50 drift in E1 is statistically significant but operationally negligible (~2% over 36 h).

Recommendation: Add USS/RSS process-private memory trending to LLM serving dashboards. Plan periodic rejuvenation (rolling restart) for Triton+V0 deployments, especially where step-wise jumps indicate unbounded anonymous mappings.

Open Questions
  • Q1
    What causes the step-wise growth in Triton+V0? Heap profiling needed to distinguish Python object accumulation, C++ runtime issue, or OS-level anonymous mappings never returned.
  • Q2
    Do leak rates generalize to larger models (70B, 405B) and newer GPUs (H100, B200)? KV-cache pressure and memory behavior may differ substantially.
  • Q3
    How do production traffic patterns — bursty arrivals, shared prefix caching, high prompt repetition — alter the aging signature? Synthetic Poisson loads may mask or amplify mechanisms.
  • Q4
    What is the leak rate after ≥3 independent replications per config? Run-to-run variance is the most urgent empirical gap before acting on specific slope estimates.
  • Q5
    Does periodic rejuvenation (restart or warm reload) effectively mitigate observed aging? At what interval is the restart cost justified given current leak rates?
  • Q6
    Can specific vLLM V1 redesigns (async task management, new resource release paths) be credited for the 7–27× reduction in leak rate vs V0?
