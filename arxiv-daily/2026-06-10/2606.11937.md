> **一句话总结：** 本文提出 Cholesky-Bench 基准框架，在 128 核 AMD Zen 2 双路服务器上系统量化了 OpenMP（GCC/LLVM）与 HPX 异步多任务运行时的瓦片 Cholesky 分解性能，发现 HPX 异步任务比 OpenMP 同类实现快约 26%、任务开销仅为 2μs（OpenMP 为 7.6μs），折叠 fork-join 变体可无代价地追平同步任务性能。

> **TL;DR:** Cholesky-Bench is an open-source benchmark framework that exhaustively compares four parallelization variants (classical fork-join, collapsed fork-join, synchronous tasking, asynchronous tasking) on two runtimes (OpenMP/GCC, OpenMP/LLVM, HPX) for tiled right-looking Cholesky decomposition on a 128-core AMD Zen 2 server. Key finding: HPX async tasks are up to 26% faster than OpenMP async tasks with ~3.8× lower per-task overhead (2 μs vs. 7.6 μs); the collapsed fork-join variant nearly matches synchronous tasking at no extra programming effort.

## 核心贡献

- 提出 **Cholesky-Bench**：基于统一软件栈的开源基准框架（MIT 许可证，GitHub: `constracktor/Cholesky-Bench`），用于对比多种并行化策略。
- 系统覆盖 **4 种并行变体 × 2 个运行时**（OpenMP/GCC、OpenMP/LLVM、HPX）= 8 种组合，对多种矩阵尺寸（$2^8$–$2^{16}$）与瓦片数（4–1024 tiles/dim）进行全面扫描。
- 首次量化 **任务开销**：HPX 约 2 μs/task，GCC OpenMP 约 7.6 μs/task，差距约 3.8×，与 HPX 面向异步任务的设计预期一致。
- 揭示 **折叠 fork-join 的高性价比**：将内外循环折叠（`collapse(2)`）后性能接近同步任务，是低成本的实用优化。
- 发现 **GCC vs. LLVM 的编译器差异**：LLVM 在折叠 fork-join 变体上比 GCC 慢约 44%，根源在于非矩形循环的 `collapse` 语义分歧和调度策略的不同实现。

## Key Contributions

- **Cholesky-Bench framework**: open-source (MIT), reproducible benchmark with `sbatch` scripts for HPC clusters; covers the right-looking tiled Cholesky with four parallelization strategies on a unified software stack.
- **8-way exhaustive comparison**: four variants (fork-join, fork-join collapsed, task synchronous, task asynchronous) × two runtimes (OpenMP via GCC 14.2 / LLVM 22.1, and HPX 1.11.0), sweeping tile sizes from 4 to 1024 tiles per dimension and problem sizes from $2^8$ to $2^{16}$.
- **Per-task overhead measurement**: isolated via no-op BLAS substitution. HPX: ~2 μs/task; GCC OpenMP: ~7.6 μs/task — a confirmed 3.8× advantage for HPX.
- **Compiler effect disclosure**: GCC is 44% faster than LLVM on the collapsed fork-join variant due to a non-standard extension divergence in how `schedule(dynamic)` interacts with non-rectangular `collapse(2)` loops.
- **PLASMA reference comparison**: Cholesky-Bench matches or exceeds PLASMA at optimal tile sizes, with the performance gap attributable to tile size choice rather than implementation quality.

## Methodology

**Algorithm.** The study focuses on the **right-looking tiled Cholesky decomposition**. A symmetric positive-definite matrix **A** is partitioned into $M \times M$ square tiles. Each outer iteration $J$ proceeds with four BLAS kernels:
- **POTRF**: in-place Cholesky on the diagonal tile $\mathbf{A}_{J,J}$.
- **TRSM**: triangular solve updating each column tile $\mathbf{A}_{I,J}$ for $I > J$.
- **SYRK**: symmetric rank-$k$ update on each diagonal trailing tile.
- **GEMM**: general matrix-matrix multiply updating each off-diagonal trailing tile.

The right-looking variant exposes an irregular (inhomogeneous) task graph, making it sensitive to synchronization overhead — ideal for probing AMT benefits.

**Four parallelization variants (implemented in both OpenMP and HPX):**

1. **Fork-join**: annotate outer loops with `#pragma omp parallel for`; only the outer submatrix-update loop is parallelized (static schedule for TRSM loop, dynamic for trailing update), leaving the inner GEMM loop unexposed. HPX uses `hpx::experimental::for_loop`.
2. **Fork-join collapsed**: fuse the SYRK and GEMM updates into a single loop nest and apply `collapse(2)`, exposing both dimensions of work simultaneously. Retains implicit barriers but saturates cores better at fine tile counts.
3. **Task synchronous**: launch each BLAS call as a task (`#pragma omp task` / HPX dataflow) but enforce global `taskwait` barriers matching the fork-join synchronization points. Isolates task-creation cost from scheduling benefit.
4. **Task asynchronous**: express data dependencies explicitly (`#pragma omp task depend` in OpenMP 4.0; future chaining via `hpx::dataflow` / `.then()` in HPX), removing all global barriers. The runtime schedules tasks as soon as their inputs are ready.

HPX uses `hpx::future<T>` (or `hpx::shared_future<T>` for tiles read by multiple tasks) wrapping `std::vector<T>` buffers that expose raw pointers to the cBLAS interface.

**Experimental setup:**
- Hardware: dual-socket node, 2× AMD EPYC 7742 (Zen 2), 64 cores / 128 threads per socket; SMT enabled but workers pinned one-per-physical-core; 2046 GiB DDR4 RAM.
- OS: Ubuntu 24.04.4 LTS, Linux kernel 6.8.0-106.
- Compilers: GCC 14.2.0 (OpenMP 4.5) and LLVM/Clang 22.1.2 (OpenMP 5.1).
- HPX: version 1.11.0 with jemalloc allocator.
- BLAS: OpenBLAS 0.3.28 via cBLAS; sequential build inside tasks (runtime is the sole parallelism source).
- PLASMA: version 24.8.7 as reference.
- Metrics: median over 20 independent runs; error bars = min/max; FP64 double precision throughout.
- Task overhead isolation: all BLAS calls replaced with no-ops; effective per-task overhead computed by dividing total no-op runtime by task count $n + n(n-1)/2 + n(n-1)/2 + n(n-1)(n-2)/6$.

## Results & Highlights

**Tile-size scaling (problem size $2^{16}$, 128 cores):**
- All variants exhibit a "sweet spot" tile count that balances parallelism saturation against task-management overhead.
- Fork-join (naive) performs significantly worse than the other three variants at up to 256 tiles per dimension because the inner loop is not exposed to the scheduler.
- Collapsed fork-join nearly matches synchronous tasking — capturing most of the load-balance benefit at minimal code change.
- Asynchronous OpenMP tasking adds a further **7%** over synchronous tasking at the optimal tile size.
- HPX asynchronous tasking adds a further **14%** over collapsed HPX fork-join.

**Cross-runtime comparison (at each runtime's respective optimal tile size):**
- HPX fork-join: **30% faster** than OpenMP fork-join.
- HPX fork-join collapsed: **15% faster** than OpenMP fork-join collapsed.
- HPX task synchronous: **21% faster** than OpenMP task synchronous.
- HPX task asynchronous: **26% faster** than OpenMP task asynchronous.

**Task overhead (no-op measurement):**
- GCC OpenMP: ~7.6 μs/task.
- HPX: ~2 μs/task.
- Ratio: ~3.8× lower overhead for HPX, consistent across tile counts.

**Problem-size scaling:**
- For OpenMP, classical fork-join can *outperform* asynchronous tasking on small problem sizes due to OpenMP's high per-task cost overwhelming the scheduling benefit.
- For HPX, asynchronous tasking dominates across *all* problem sizes at tile counts of 32, 64, and 128 tiles per dimension — no crossover point where fork-join wins — a direct consequence of HPX's lower task overhead.

**Compiler comparison (GCC vs. LLVM on OpenMP):**
- Tasking performance and naive fork-join are essentially identical between GCC and LLVM at optimal tile sizes.
- GCC is **44% faster** than LLVM on the collapsed fork-join variant. Root cause: the fused trailing-update loop is non-rectangular, and the OpenMP spec does not permit attaching a `schedule` clause to the surrounding `collapse(2)`. GCC enforces this strictly (compilation error); LLVM silently accepts it as a non-standard extension but defaults to static scheduling, which performs poorly. Under standard-conforming code (no `schedule` clause on collapsed non-rectangular loop), LLVM's gap narrows to the level of the naive fork-join.

**PLASMA reference:**
- Both Cholesky-Bench stacks (OpenMP and HPX) outperform PLASMA at their respective optimal tile sizes. PLASMA's gap is due to its default tile size ($2^8$) being suboptimal for the $2^{16}$ problem, not an algorithmic disadvantage.

## Critical Assessment

**Novelty:** Moderate. Tiled Cholesky is a thoroughly studied benchmark kernel, and HPX-vs-OpenMP comparisons exist (Task Bench [33], prior HPX Cholesky work [36]). The contribution is the *principled, reproducible, unified-stack* comparison that previous work lacked — individual implementations embedded in separate libraries make fair comparison infeasible. The OpenMP priority hint null-result (annotating critical-path tasks with `priority` clause yielded no measurable improvement) is a minor negative finding worth noting.

**Rigor:** High. The experimental design is sound: same algorithm, same BLAS, same compiler flags, same hardware, pinned threads. Per-task overhead isolation via no-op substitution is a clean methodology. Reporting both median and min/max across 20 runs is appropriate for noisy HPC measurements.

**Reproducibility:** Excellent. Full software stack disclosed (hardware, OS, compiler versions, HPX version, BLAS version, build flags). Code is MIT-licensed on GitHub (`constracktor/Cholesky-Bench`) with `sbatch` scripts. This is significantly better than typical HPC papers.

**Significance:** Moderate-to-high for HPC practitioners. The 3.8× task overhead advantage of HPX over OpenMP is a concrete, actionable datum. The finding that fork-join can be competitive with AMT on small problems (due to OpenMP's high task cost) is a nuanced, practically useful caveat often glossed over in AMT advocacy.

**Red flags:**
- Results are from a *single* hardware platform (dual-socket AMD Zen 2). The paper acknowledges this and flags Intel, ARM, RISC-V as future work.
- Only right-looking Cholesky is studied; left-looking and top-looking variants are deferred to future work. The choice of traversal variant affects task graph regularity.
- The 30%/15%/21%/26% cross-runtime speedups must be interpreted as "these two stacks as typically deployed," not purely intrinsic runtime overhead differences — scheduling policy divergence contributes alongside task overhead.
- AI tools (Grammarly, ChatGPT, Claude) were used for manuscript clarity; authors confirm all technical content is original.

## Strengths

- Systematic, reproducible, openly available benchmark with full configuration disclosure.
- Clean isolation of task overhead via no-op substitution — a rarely seen but important methodological step.
- Nuanced finding that collapsed fork-join is a cheap, practical win before committing to full AMT.
- Compiler comparison (GCC vs. LLVM) exposes a real-world portability pitfall in `collapse(2)` scheduling that practitioners need to know.
- Honest acknowledgment of the single-platform limitation and a clear future-work roadmap.

## Limitations

- Single hardware platform (dual-socket AMD Zen 2); generalization to Intel, ARM Neoverse, or many-core accelerators is unvalidated.
- Single algorithmic variant (right-looking); left-looking and top-looking Cholesky may behave differently under the same parallelization strategies.
- HPX carries non-trivial adoption cost (C++ futures API, jemalloc dependency, separate build system) not reflected in runtime numbers — the performance advantage must be weighed against engineering overhead.
- Only shared-memory parallelism is studied; HPX's distributed-memory capabilities (MPI, Legion, FleCSI backends) are out of scope and would require a separate evaluation.
- The comparison does not include other AMT runtimes (StarPU, TTG, PaRSEC) in the same framework; extending Cholesky-Bench to these is listed as future work.

## Relevance

本文聚焦于高性能计算运行时的细粒度性能对比，与 ML 系统中分布式训练、稀疏计算和高性能线性代数加速有一定交叉，但核心受众是 HPC 系统软件开发者；若关注 ML 推理/训练中的并行任务调度（如异构多核加速），其方法论和量化结论有参考价值。

## Questions for Further Exploration

1. Does the ~3.8× task overhead advantage of HPX over OpenMP hold on Intel Xeon (Sapphire Rapids) or ARM Neoverse V2 architectures, where NUMA topology and cache coherence differ substantially?
2. Would left-looking or top-looking Cholesky — which expose more regular task graphs — reduce the advantage of asynchronous tasking over collapsed fork-join?
3. Can OpenMP's task overhead be reduced to HPX-level (~2 μs) with implementation improvements (e.g., OpenMP 5.x detached tasks, or compiler-specific optimizations in LLVM), or is it a fundamental design consequence of fork-join-centric runtime architecture?
4. How does Cholesky-Bench scale in a distributed-memory setting with HPX's MPI or FleCSI backend — does the AMT advantage amplify or diminish as inter-node communication introduces new bottlenecks?
5. The paper mentions GPRat (Gaussian process regression) as a real application using HPX tiled Cholesky. What is the end-to-end GPRat speedup from switching to asynchronous HPX tasking, compared to this microbenchmark figure of 26%?
