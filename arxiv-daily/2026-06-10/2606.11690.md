> **一句话总结：** 现有所有公开LLM成本计算器将GPU利用率视为固定输入，忽略了通过Little's Law决定并发量的请求率λ；作者提出C_eff = f(H, M, Q, λ, L)参数化模型，通过42次H100基准测试证明同硬件下有效成本跨越$0.21–$15.25/MTok（低负载下高估自托管经济性达36.3倍），并发布开源工具vllm-cost-meter。

> **TL;DR:** All surveyed public LLM cost calculators assume fixed GPU utilization, causing them to understate self-hosted inference cost by exactly 1/U — a 2.5–36x error at enterprise loads. This paper formalizes a concurrency-aware cost model C_eff = f(H, M, Q, λ, L) where offered request rate λ drives utilization via Little's Law, validates it across 42 benchmarks on H100 NVL and 56 runs on A100 PCIe, and releases `vllm-cost-meter` — an open-source tool that measures real $/M-tokens against a live vLLM server's actual traffic rather than vendor benchmarks.

## 核心贡献

- **揭示成本计算器的系统性缺陷**：对Helicone、LiteLLM、YourGPT、llm-prices.com等15+公开计算器的调研显示，所有工具均将GPU利用率作为固定输入（100%或用户自定义），而非从实际请求率测量；学术分析（Pan et al.假设100%利用率，SLaM假设80%）同样如此。
- **建立并发感知成本模型**：形式化C_eff = f(H, M, Q, λ, L)，其中λ（请求到达率）通过Little's Law N≈λW决定实际并发量，是决定成本的核心运营商可控变量；自托管盈亏平衡点是(λ, L, H)空间中的一个曲面，而非单一数字。
- **量化低负载成本惩罚**：在H100上，λ=1 rps时有效成本$7.60–$15.25/MTok，比朴素估计高17.5–36.3倍；λ=10 rps时惩罚仍有2.5–3.8倍；λ≥50 rps后服务器进入队列饱和区，惩罚接近1×。
- **揭示FP8量化的架构条件效益**：FP8对MoE架构吞吐量提升（Mixtral +69%，Qwen3-30B-A3B +74%）显著大于密集模型（Llama 3.1 8B +31%），且在无原生FP8 Tensor Core的A100 PCIe上，密集模型的FP8优势反转。
- **发布开源实时成本工具**：vllm-cost-meter挂载于live vLLM服务器的Prometheus指标，按操作员真实流量报告$/M-tokens，MIT许可证开源。

## Key Contributions

1. **Empirical demonstration of cost spread on identical hardware**: Effective cost-per-million-tokens varies by up to 36.3x (17.5–36.3x across configurations) on the same H100 hardware as a function of offered request rate λ, using 42 systematic vLLM benchmark runs (6 server configurations × 7 arrival-rate levels).

2. **Concurrency-aware cost attribution formula**: Formalizes C_eff = f(H, M, Q, λ; L) where utilization U(λ, L | H, M, Q) = Θ_achieved(λ, L) / Θ_max(H, M, Q) is a *dependent variable* determined by (H, M, Q, λ, L) — not a user-supplied input. Effective cost is C_eff = (P_GPU × 10^6) / (3600 × Θ_achieved(λ, L)).

3. **Corrected crossover analysis**: Shows that existing break-even analyses produce misleading crossover points; the true self-host vs. API crossover is a surface in (λ, L, H) space, not a single token-volume threshold.

4. **Open-source real-time cost meter**: Releases `vllm-cost-meter` that attaches to a live vLLM server and reports real $/M-tokens against the operator's own traffic — the only open-source tool that does this.

## Methodology

**Cost model formulation.** The paper separates two distinct quantities: (1) the offered request rate λ (operator-controlled, requests/sec arriving at the server) and (2) in-flight concurrency N (an emergent quantity set by λ via Little's Law N ≈ λW). Prior calculators conflate or ignore this distinction. The utilization function is:

U(λ, L | H, M, Q) = Θ_achieved(λ, L) / Θ_max(H, M, Q)

And effective cost is:

C_eff = P_GPU × 10^6 / (3600 × Θ_achieved(λ, L))

The naive model's error is exactly the underutilization penalty: C_eff / C_naive = 1/U.

**Hardware.** Single Azure ML node with 2× NVIDIA H100 NVL 96GB GPUs (NVLink 900 GB/s, $13.96/hr node = $6.98/GPU). Single-GPU models (Llama, Qwen) costed at $6.98/hr; Mixtral at $13.96/hr (TP=2). Cross-hardware validation: 56 runs on NVIDIA A100 80GB PCIe.

**Model selection.** Three models chosen to span an architectural sparsity gradient:
- **Llama 3.1 8B Instruct**: Dense (8B/8B active), single H100, TP=1
- **Qwen3-30B-A3B Instruct**: Ultra-sparse MoE (3B/30B active, July 2025 release), single H100, TP=1
- **Mixtral 8x7B Instruct**: Sparse MoE (12.9B/46.7B active), two H100s, TP=2

**Sweep protocol.** λ ∈ {1, 5, 10, 25, 50, 100, 200} requests/sec. Each configuration: 100 warmup requests (discarded) + 500 measured requests. Synthetic random-token prompts, 512 input / 256 output tokens. Prefix caching explicitly disabled (`-no-enable-prefix-caching`) to isolate pure offered-load effects; speculative decoding disabled. Arrivals: Poisson (`-burstiness 1.0`). Reported figures are conservative upper bounds on cost (production optimizations only lower cost).

**Derived metrics.** Effective C_eff = P_GPU×10^6 / (TPS×3600) where TPS is aggregate *output*-token throughput. Underutilization penalty = C_eff / C_naive at each λ. Crossover rate λ* where C_eff(λ*) = C_API(model_tier).

**Variables held constant.** Inference engine (vLLM 0.19.0 with continuous batching + PagedAttention), pricing mode (Azure on-demand list), I/O ratio (uniform 512:256), parallelism strategy (TP=1 for dense, TP=2 for Mixtral), arrival distribution (Poisson). These make the reported costs conservative lower bounds on cost dispersion.

## Results & Highlights

**Offered load vs. effective cost (Section 5.1).** At λ=1 rps, C_eff ranges from $7.60/MTok (Llama FP16/FP8) to $15.25/MTok (Mixtral FP16). As λ increases, C_eff drops precipitously: by λ=25 rps all single-H100 models are within 20% of peak-throughput cost. At λ≥50, the server is queue-limited and throughput plateaus.

Naive model underestimates self-hosted costs by **17.5–36.3×** at λ=1. Even at λ=10 rps (not typically considered "low"), the naive model is wrong by 2.5× (Llama) to 3.8× (Qwen).

**Architecture-dependent scaling (Section 5.2).** At peak throughput: Llama 3.1 8B achieves 6,238 tok/s ($0.311/MTok), Qwen3-30B-A3B achieves 5,319 tok/s ($0.364/MTok), Mixtral 8x7B achieves 4,454 tok/s ($0.871/MTok). The three curves converge to within ~3× at saturation. The cost-structure differences between dense and ultra-sparse MoE manifest primarily as level shifts (higher idle cost, different saturation floor) rather than slope shifts — but large enough that a single cost formula calibrated on dense models systematically misestimates MoE costs.

**Quantization impact (Section 5.3).** FP8 peak throughput gains: +31% for Llama 3.1 8B (6,238→8,155 tok/s), +74% for Qwen3-30B-A3B (5,319→9,271 tok/s), +69% for Mixtral 8x7B (4,454→7,524 tok/s). The larger FP8 gain for MoE models is consistent with their memory-bandwidth-bound profile: sparse activation requires scattered parameter reads, and halving parameter precision directly doubles effective bandwidth. **Notable**: at peak offered load, Qwen3-30B-A3B FP8 ($0.209/MTok) is cheaper than Llama 3.1 8B FP8 ($0.238/MTok) despite 30B total parameters — active parameter count, not total size, dominates saturation economics.

At λ≥100 rps, TTFT_p50 exceeds 2.5 seconds — well beyond the example production SLO of TTFT_p99 ≤ 300ms — confirming that the λ=50 minimum-cost point is a *no-SLO floor*.

**A100 cross-hardware validation (Section 5.9).** The load-driven cost spread reproduces at 7.0–11.4× per configuration on A100 80GB PCIe (compressed by cheaper, slower silicon). Active-parameters-beat-total-parameters ordering survives at FP8. The dense-FP8 advantage *inverts* on A100 — hardware without native FP8 tensor cores — confirming that Q must be treated as a hardware-conditional input in C_eff.

## Critical Assessment

**Novelty.** The core insight — that utilization is a function of λ, not a fixed input — is textbook queuing theory (M/G/1 or M/M/c queue dynamics). WiNGPT (Zhuang et al., cited as [7]) demonstrated a 2.6× cost reduction from concurrency 8 to 48 on A800 GPUs using vLLM. What is new here is: (1) systematic quantification of the full load-range spread from near-idle to saturation across three architectural classes, (2) the crossover-as-surface framing, (3) the open-source live-server measurement tool, and (4) the hardware-conditional FP8 finding. The framing as "every public calculator ignores this" is pointed and practically useful even if the underlying principle is not new.

**Rigor.** Moderate-to-good for a single-author empirical systems paper. 42 H100 runs + 56 A100 runs provide reasonable coverage. Benchmark design choices (synthetic tokens, no prefix caching, Poisson arrivals, no speculative decoding) are explicitly conservative upper bounds and well-justified. The n=3 FP8/architecture comparison is explicitly acknowledged as insufficiently powered for strong generalization. The paper's own framing — "treat this as suggestive" for the active-parameter claim — models appropriate epistemic humility.

**Reproducibility.** High in principle. Hardware is commercially available (Azure Standard_NC80adis_H100_v5), costs are under $65 total, model checkpoints are public, and the vLLM version is pinned. The `workload_protocol` schema is designed for crowd-sourced curve contributions. One gap: the exact benchmark harness commit (`1c40823`) is cited but the repository URL is not visible in the reviewed pages.

**Significance.** High for practitioners. Organizations plan multi-figure monthly LLM infrastructure budgets using the calculators surveyed here. The finding that a Mixtral 8x7B deployment at 1 rps costs $15.25/MTok — more expensive than Claude Sonnet 4.6 at $0.87 (stated in the introduction) — could materially change build-vs-buy decisions for low-traffic workloads. The crossover-as-surface insight is particularly important: prior "self-hosting is cheaper above 50M tokens/month" advice collapses under this analysis.

**Red flags.**
- Single author, independent researcher, no peer review prior to arXiv posting.
- n=3 architectures is a thin base for the active-parameter-dominance claim, explicitly acknowledged.
- The 36.3× headline figure corresponds to λ=1 rps (footnote 1: Qwen3-30B-A3B FP8 measured against Mixtral FP16 at saturation), a somewhat cherry-picked extreme. The practically relevant range (1–10 rps enterprise) shows 2.5–24×, still large but less dramatic.
- Input/output ratio fixed at 512:256 (ShareGPT-derived median); actual enterprise workloads with long prompts or variable I/O ratios may show different penalty shapes.
- Pricing uses Azure on-demand list prices; spot instance multipliers (0.3–0.7×) reduce C_eff by a roughly constant scalar without altering the load-cost dynamics — but the absolute crossover points shift significantly.
- A100 results compressed by "cheaper, slower silicon" — the framing conflates two hardware differences (price and performance) in a way that makes direct architectural comparison harder.

## Strengths

- Addresses a real, widespread, and consequential blind spot: every named public calculator (Helicone, LiteLLM, YourGPT, llm-prices.com, Mem0.ai, DocsBot, LLM Inference TCO Calculator) treats utilization as a fixed input.
- Methodological transparency is exemplary: held-constant variables are enumerated, conservative choices are explicitly flagged, and results are framed as upper bounds on cost.
- The utilization-penalty formula C_eff/C_naive = 1/U is simple, memorable, and actionable.
- Dual-hardware validation (H100 + A100) specifically designed to test hardware-conditional behavior, yielding the practically important FP8 inversion finding.
- Open-source tool release with MIT license; design accommodates crowd-sourced curves via `workload_protocol` schema.
- Cost of replication is under $65, making independent verification unusually accessible for a systems paper.
- Explicit crossover-as-surface framing refutes a common practitioner heuristic ("self-hosting breaks even above X tokens/month") in a principled way.

## Limitations

- Single author, no institutional review; experimental design gaps may exist.
- n=3 architectures (dense, ultra-sparse MoE, sparse MoE) insufficient for strong architectural generalization. Findings should be treated as suggestive pending broader validation (Llama 70B, DeepSeek-MoE, Qwen-MoE variants, etc.).
- Fixed I/O ratio (512:256): variable input/output distributions common in production (long-context, summarization, RAG) may amplify or dampen the penalty; prefill-dominated workloads narrow the dense/MoE distinction (explicitly noted in §5.2).
- Poisson arrival assumption (CV=1): supplementary Gamma probe at CV=2 found negligible impact on one configuration (C4), but burstier production workloads may amplify the penalty — the paper treats current results as conservative lower bounds.
- vLLM-specific: findings tied to vLLM's continuous batching + PagedAttention engine. TensorRT-LLM is explicitly out of scope; SGLang shares the same apparatus but is secondary.
- Hardware scope: H100 NVL and A100 PCIe only. AMD MI300X, NVIDIA H200, AWS Inferentia, Google TPU v5 are uncharacterized.
- Pricing denominator choice (co-tenant vs. dedicated node) doubles absolute C_eff for single-GPU configurations under a dedicated-node deployment model — practitioners must apply the correct denominator for their situation.
- Latency SLO (L) is held implicit in the reported sweeps; incorporating L as an active sweep dimension (noted as future work in §6.9) is necessary for SLO-aware cost planning.

## Relevance

本文对任何考虑自托管LLM推理（尤其是低流量或中等流量场景）的AI基础设施团队具有直接参考价值；其方法论揭示了现有成本分析工具的系统性盲点，对研究LLM推理效率、MoE架构经济学以及量化感知硬件性能的人员同样重要。

## Questions for Further Exploration

1. How does the cost model extend to production workloads with Gamma or Weibull inter-arrival distributions (CV > 1), where bursty traffic could amplify the underutilization penalty beyond the conservative Poisson baseline reported here?
2. The crossover point λ* shifts with latency SLO L — can the paper's framework be extended to produce SLO-aware crossover surfaces that account for TTFT_p99 and TPOT_p99 constraints simultaneously?
3. The active-parameter-dominance finding (Qwen3-30B-A3B FP8 cheaper than Llama 3.1 8B FP8 at saturation) rests on n=3 with a ~12% margin. Does it hold for denser sparsity ratios (e.g., Llama 70B vs. a hypothetical 70B MoE with 10B active parameters)?
4. How should multi-replica or auto-scaling deployments be modeled? The current framework assumes a fixed hardware allocation; real production operators scale replicas in response to λ, changing the effective denominator.
5. Given that A100 PCIe shows a dense-model FP8 inversion, what is the precise hardware-capability threshold (e.g., tensor core FP8 native support) that determines whether FP8 benefits dense vs. MoE models more, and how should this inform GPU procurement decisions?
