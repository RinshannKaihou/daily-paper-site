> **一句话总结：** ModSleuth 是一个智能体系统，通过递归解析公开制品（论文、模型卡、代码库）自动重建 LLM 的依赖图，在四个主流开源模型上共恢复 1,060 条经验证的依赖关系，揭示了跨跳许可证风险、训练-评测耦合及文档不一致等普遍问题。

> **TL;DR:** ModSleuth is an agentic pipeline that recursively reconstructs source-verified LLM dependency graphs from public artifacts; applied to four open-model families (Olmo 3, Nemotron 3, DR Tulu, SmolLM3) it recovers 1,060 verified dependencies—3× more than the best single-prompt baseline—and surfaces license risks, train–eval coupling, and documentation gaps that are invisible from any single source.

## 核心贡献

- **形式化递归依赖追踪任务**：区分"直接依赖"（影响权重或训练数据）和"间接依赖"（影响开发决策但不进入训练），并通过以操作为中心的关系表示消除固定类型分类法的局限性。
- **身份格（identity lattice）设计**：解决跨文档的制品身份歧义问题——同一模型在不同来源中可能以家族名、版本号、仓库 slug 等不同形式出现。
- **三阶段智能体流水线**：来源收集 → 实体发现与解析 → 依赖构造与对账，以 Claude Code 为执行引擎，支持广度/深度/束搜索三种递归展开策略。
- **大规模实证发现**：在聚合图中恢复 2,526 个节点、9,112 条依赖边、36,187 条证据锚，揭示了多跳许可证义务、训练-评测结构性耦合、代码级来源与卡片摘要的出入等重大发现。
- **开源发布**：ModSleuth 系统与生成的依赖图均公开，支持 AI 生态透明度研究。

## Key Contributions

- **Task formalization**: Defines recursive LLM dependency tracing as constructing evidence-grounded graphs that distinguish direct dependencies (affecting weights or training data) from indirect ones (affecting development decisions without entering training).
- **Operation-centered representation**: Encodes dependencies as structured operations with free-form natural-language descriptions, coarse type labels, and source anchors, avoiding the rigidity of a fixed taxonomy.
- **Identity lattice**: A hierarchical structure for resolving artifact identity across heterogeneous sources, handling ambiguity between family names, checkpoint IDs, and repository slugs without forcing premature resolution.
- **Staged agentic pipeline**: Three phases—source gathering, entity discovery and resolution, and dependency construction—implemented with Claude Code as the extraction engine and supporting recursive BFS/DFS/beam expansion.
- **Empirical scale**: Applied to four fully open LLM releases (Olmo 3, Nemotron 3, DR Tulu, SmolLM3), recovering 1,060 source-verified dependencies under the unbounded scope (3× the best single-prompt baseline) and 1,654 under BFS reachability.

## Methodology

**Problem framing.** Modern LLM training pipelines rely on upstream models for data generation, filtering, rewriting, OCR, evaluation, and preference learning—not just for initialization. This creates recursive, multi-hop dependency chains scattered across technical reports, model cards, code repositories, and dataset cards. ModSleuth's goal is to reconstruct an evidence-grounded dependency graph for a given target LLM release using only public sources.

**Dependency semantics.** A *direct dependency* is any upstream artifact that materially affects the target's training or weights, including initialization checkpoints, synthetic-data generators, OCR systems, and filtering models. Direct dependencies are recursive: the training data of a direct dependency is itself a direct dependency. An *indirect dependency* influences development decisions (e.g., evaluation models, ablation baselines) without entering training. Artifacts that are only baseline comparisons or vague inspirations are excluded.

**Identity lattice.** Each artifact family has a root node for underspecified references, intermediate nodes for partially resolved identities, and canonical leaves anchored to concrete URLs or release identifiers. Identity is an open-vocabulary set of facets (e.g., `{family:Olmo 3, size:32B, stage:Think}`), preserving uncertainty without forcing premature resolution. Hugging Face metadata is retrieved deterministically to resolve dataset subsets, derived variants, and internal names.

**Three-phase pipeline:**
1. *Source gathering*: For the target release, all official public artifacts are collected (technical reports, model/dataset cards, code repositories, release blogs, linked upstream artifacts) and organized into topically coherent batches.
2. *Entity discovery and resolution*: The agent identifies dependent artifacts from each batch, recording mentions exactly as they appear, then maps them into the identity lattice. Collisions, nonexistent artifacts, and inconsistent resolutions are flagged before dependency construction.
3. *Dependency construction and reconciliation*: The agent re-reads each batch against the resolved lattice and emits structured operations. Edges record upstream/downstream artifacts, a free-form description, a coarse dependency type, and source anchors. Conflicting claims are flagged for human audit; verified claims are merged while preserving the most specific identity.

**Recursive expansion.** After recovering the local neighborhood, ModSleuth recursively applies the same pipeline to each discovered upstream artifact, using BFS for maximum coverage, DFS for targeted chain investigation, or beam search for structurally central ancestors. This converts one-hop evidence into a full transitive dependency graph.

**Evaluation.** Because ground-truth dependency graphs do not exist, the metric is the number of *verified* dependency relationships. Verification is performed post-hoc using the cited evidence: Claude Sonnet 4.6 with web search reads each cited URL, independently corroborates it, and returns a verdict of *verified*, *refuted*, or *unclear*. The BFS-scope aggregate uses Claude Opus 4.7 with extended thinking. Three reporting scopes are defined: depth-1 (subject is the target's canonical ID), unbounded (forward-reachable in the merged graph with provenance attribution), and BFS reachability (all forward-reachable edges regardless of which sub-investigation found them).

**Baselines.** GPT-5.5 Pro, GPT-5.4 Pro, Claude Code single-prompt (CC-single), and ChatGPT Deep Research, all given the same high-level task specification plus the same target model and recursive graph-construction instructions.

## Results & Highlights

**Quantitative.** Under the unbounded scope, ModSleuth recovers 1,060 verified relationships—more than 3× the strongest baseline (GPT-5.5 Pro: 314). Under BFS reachability, the merged graph contains 1,654 verified forward-reachable edges across the four targets. The aggregate graph has 2,526 artifact nodes, 9,112 dependency edges, and 36,187 evidence anchors. Gains are largest for Olmo 3 (481 vs. 91 at depth-1) and Nemotron 3 (1,023 vs. 156), which have complex multi-stage data pipelines.

**Direct vs. indirect split.** Direct dependencies account for 72% of verified edges (1,191 of 1,654 BFS edges). Among direct dependencies, upstream operations on training data (generation, filtering, transformation, embedding, decontamination) account for 350 verified edges, while weight-level model lineage (trained_from, merged_from, quantized_from) accounts for only 28—meaning upstream models shape downstream training far more through data operations than through checkpoint inheritance.

**Internal vs. external.** 75–82% of verified edges come from outside the target organization. Qwen and OpenAI/GPT are the most depended-on external families across all four targets.

**Key qualitative findings:**
- *Multi-hop upstream models*: DR Tulu depends on Claude Sonnet 3.7 via a ScholarQA trajectory-generation chain not mentioned in the main paper. Olmo 3 RL-Zero inherits Qwen2.5-Coder-32B-Instruct through an intermediate data construction chain not stated in downstream RL-Zero cards.
- *Training–evaluation coupling*: Olmo 3 training mixes reuse IFEval-sampled constraints while also reporting IFEval as an evaluation target; GSM8K appears both as training data (via synthetic expansion) and as an evaluation target across Olmo 2 and 3.
- *License-relevant paths*: SmolLM3's FineMath pretraining traces back to Llama-3-70B-Instruct annotation via finemath-classifier, potentially implicating the Llama 3 Community License even though Llama is not named in the SmolLM3 card.
- *Model-mediated selection*: Qwen3-32B judges Olmo 3 RL data; Nemotron-3-Super uses a Qwen-bootstrapped generative reward model. Among generators, filters, and judges, Qwen2.5-32B-Instruct (70 uses), Llama-3.3-70B-Instruct (36), Qwen3-32B (33), and GPT-4.1 (32) dominate.
- *Code-level provenance*: Nemotron training blends have placeholder rows requiring a separate `fill_placeholders.py` script. SmolLM3's SmolTalk pipeline collapses multiple upstream models (Qwen2.5-72B-Instruct for summarization, Llama-3.1-405B-Instruct + Llama-3.1-8B-Instruct for filtering) into one dataset name on the card.

## Critical Assessment

**Novelty.** The work addresses a real and underexplored problem: the recursive, fragmented nature of LLM dependency chains. The formalization of dependency types, the identity lattice, and the agentic multi-phase pipeline are genuinely novel contributions. The key insight—that the bottleneck is semantic/representational rather than pure information extraction—is well-argued and supported by evidence.

**Rigor.** The evaluation design is careful given the absence of ground truth: post-hoc source verification with a separate model, three distinct scoping levels, and explicit acknowledgment that the recovered graph is a lower bound. Quantitative results clearly exceed baselines. The authors are appropriately transparent about what ModSleuth cannot recover (undocumented, private, or proprietary dependencies).

**Reproducibility.** Code and the resulting dependency graphs are released. However, both ModSleuth and the verifier rely on Claude Code and Claude Sonnet 4.6/Opus 4.7, making exact reproduction dependent on specific API versions and model behavior. Shared modeling biases between extractor and verifier are a legitimate concern that the authors flag but cannot fully resolve.

**Significance.** The findings are practically important: hidden license obligations, train-eval coupling, and documentation gaps are not hypothetical risks—the paper provides concrete, source-verified examples. The ecosystem-level view (a few model families dominating generator/judge roles across many releases) is a novel empirical finding with implications for model collapse, bias amplification, and governance.

**Red flags.** (1) Evaluation quality is bounded by the verifier's own accuracy; the paper uses Claude to verify Claude-extracted claims with no independent cross-check. (2) The comparison is restricted to four well-documented, fully open releases; performance on partially closed or less-documented releases is unknown and likely much lower. (3) The paper does not quantify false negatives (missed dependencies), which could be large and systematic given the lower-bound nature of the approach. (4) Human-in-the-loop baselines are not evaluated, leaving the contribution of ModSleuth's task decomposition somewhat conflated with raw agentic capability improvements.

## Strengths

- Addresses a genuinely important and growing problem: the opacity of modern LLM supply chains.
- Principled formalization (direct/indirect distinction, operation-centered representation, identity lattice) that is more expressive than prior work.
- Large-scale empirical demonstration with concrete, specific, verifiable findings—not just aggregate statistics.
- Honest about limitations: explicit "lower bound" framing, declared-only scope, and acknowledgment of verifier bias.
- Code and data released for community use.
- The qualitative findings (train-eval coupling, hidden license paths, code-level discrepancies) are independently interesting and actionable.

## Limitations

- **Only declared dependencies**: ModSleuth cannot recover undocumented, proprietary, or private dependencies. The gap between declared and true dependencies is unknown and likely large.
- **No ground truth**: Absolute recall cannot be measured; the evaluation is necessarily recall-incomplete.
- **Self-referential verification**: Both extraction and verification rely on Claude, creating shared failure modes and potential circular validation.
- **Scope of evaluation**: Only four well-documented, fully open releases; generalization to less-documented or partially closed models is unvalidated.
- **No human-in-the-loop comparison**: The authors do not evaluate expert-guided interactive dependency tracing, leaving the task-decomposition contribution partially conflated with model capability.
- **Static snapshot**: The recovered graphs reflect public documentation at a single point in time; LLM pipelines and their documentation evolve rapidly.

## Relevance

本文直接关乎 LLM 训练数据来源、许可合规与生态透明度等核心治理问题。对于研究现代 LLM 数据工程、模型审计或 AI 供应链的读者，ModSleuth 提供了一套可复用的分析框架和目前最大规模的公开 LLM 依赖图数据集，具有较高的工具价值与参考意义。

## Questions for Further Exploration

1. How would ModSleuth perform on partially-closed releases (e.g., Gemini, GPT-4o) where most pipeline details are withheld? Can the approach meaningfully bound the true dependency structure even with sparse public evidence?
2. The paper shows that a small number of model families (Qwen, Llama, GPT-4) dominate generator/judge roles. What are the downstream consequences for model diversity and potential correlated failure modes across the ecosystem?
3. Can the identity lattice and operation-centered representation be standardized into a disclosure schema that model developers could adopt proactively, analogous to SBOMs (software bill of materials) in software supply chains?
4. The train–eval coupling findings suggest that standard decontamination practices may be structurally insufficient. How should benchmark developers respond to models that structurally reuse benchmark-derived data in multi-hop pipelines?
5. What does the dominance of a few upstream model families as judges and generators mean for preference leakage and self-reinforcing behavioral biases across the open-model ecosystem over successive generations?
