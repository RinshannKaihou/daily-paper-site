Selective retrieval beats global injection — but retrieval quality beats retrieval quantity: Top-K > 10 degrades accuracy

External Experience Serving in Production LLM Systems: A Deployment-Oriented Study of Quality-Cost Trade-offs ↗
Lin Sun Heming Zhang Xiangzheng Zhang Qiyuan Tech (360)

Why This Matters
In decode-heavy tasks, retrieval can actually reduce latency by shortening the generation path — flipping the expected cost trade-off. Retrieval quality degrades when weakly matched items dilute high-precision signals.
⚠ Critical Edge
Primary task uses proprietary data — core results cannot be reproduced externally. The 19.6% baseline is artificially low (risk-focused benchmark, not traffic-matched). Qwen3 model family only; generalization unverified. Retrieval system details withheld.
Methodology

Framework: external experience serving — injecting operational knowledge (policy rules, failure cases, edge-case traces) from a pre-built repository into LLM prompts at serving time. Framed as two break-even decisions:

  • 1. Serving-interface break-even: global injection vs. selective retrieval
  • 2. Task cost-structure break-even: whether quality gains justify online cost given output regime

Three tasks: content moderation (Qwen3-8B, short-output), tool-use (Qwen3-8B, short-output), GPQA (Qwen3-30B-A3B, decode-heavy). All run with paired Instruct and Thinking inference modes. Prefix caching enabled for fair comparison.

Experience items structured with [Trigger] / [Common Failure] / [Action] fields. 105K items for moderation from ~300K business-labeled examples.

Strengths / Limitations
  • Production grounded: real 12B tokens/day system, 105K-item repository — not a synthetic benchmark.
  • Decode-heavy insight: retrieval can reduce total latency in long-generation tasks — underappreciated in practice.
  • Random control included: isolates relevance from mere context augmentation.
  • Oracle bounds: 14.7–43.2% retrieval reduction possible with optimal selective serving.
  • Proprietary primary task: core results cannot be reproduced externally.
  • 19.6% baseline is artificially low: risk-focused benchmark, not traffic-distribution-matched.
  • Qwen3 only: generalization to other model families unverified.
  • No full prefill/decode breakdown: cost proxied by token counts and end-to-end latency only.
Open Questions
  1. How is the retrieval system implemented — what embedding model, index structure, and query formulation are used? How does retrieval latency factor into the full cost breakdown?
  2. Can the oracle selective-serving estimates (14.7–43.2% retrieval reduction) be approximated with a lightweight gating classifier?
  3. What is the quality-coverage trade-off in repository construction — how many items are needed before retrieval quality saturates for a given domain?
  4. Is there a principled way to decide when to enable Thinking vs. Instruct for a given serving budget in short-output settings?
  5. How does performance degrade over time as policy and category taxonomy evolve — what is the "freshness" requirement for the experience repository?
  6. Would findings transfer to other safety-critical applications (medical triage, legal compliance) where experience also accumulates operationally?
