01 /Methodology

Three-phase recursive pipeline, source-verified end-to-end.

ModSleuth treats dependency tracing as graph construction. A direct dependency is any upstream artifact that materially affects weights or training data (initialization, synthetic-data generators, OCR, filters); an indirect dependency shapes development decisions without entering training. An identity lattice resolves the same model across family names, checkpoint IDs, and repository slugs without forcing premature collapse.

PHASE 1Source GatheringCollect reports, cards, code, blogs into coherent batches.
PHASE 2Entity DiscoveryFind mentions, map to the lattice, flag collisions and ghosts.
PHASE 3Dependency BuildEmit structured ops with evidence anchors; reconcile conflicts.

Recursion re-applies the pipeline to each upstream artifact (BFS / DFS / beam), converting one-hop evidence into a full transitive graph. A separate Claude Sonnet 4.6 reads each cited URL and returns verified / refuted / unclear—the BFS-scope aggregate uses Opus 4.7 with extended thinking.

03 /Strengths

What works.

  • Formalization (direct/indirect + operation-centered) is more expressive than prior work.
  • Concrete, source-verified findings—not just aggregate statistics.
  • Honest "lower bound" framing; declared-only scope is acknowledged.
  • Code and the recovered dependency graphs are released for community use.
  • Qualitative findings (hidden license paths, train–eval coupling) are independently actionable.
04 /Limitations

Where it breaks.

  • Only declared, public dependencies—undocumented chains stay invisible.
  • No ground truth: absolute recall cannot be measured.
  • Self-referential verification (Claude extracting + Claude verifying).
  • Evaluated only on four fully open releases; closed models untested.
  • No human-in-the-loop baseline; decomposition conflated with agent capability.
  • Static snapshot—pipelines and documentation evolve rapidly.

Questions for Further Exploration

  1. How would ModSleuth perform on partially-closed releases (Gemini, GPT-4o) where most pipeline details are withheld?
  2. What are the downstream consequences of a handful of model families (Qwen, Llama, GPT-4) dominating generator and judge roles?
  3. Could the identity lattice become a standardized disclosure schema—an SBOM for LLM supply chains?
  4. Train–eval coupling suggests standard decontamination is structurally insufficient. How should benchmark developers respond?
  5. What does persistent judge/generator dominance mean for preference leakage and self-reinforcing biases across generations?
