> **一句话总结：** "Every Eval Ever"(EEE) 提出首个面向 LLM 评测结果的社区治理 JSON Schema 与 Hugging Face 众包仓库，覆盖 22,235 个模型 × 2,273 个基准 × 31 种评测格式，并附自动格式转换器、Pydantic 校验管道与四项实证用例（成本-准确率权衡、困惑度规范差异、HELM 复现性审计、IRT 元分析）。

> **TL;DR:** EEE is a community-built JSON schema plus a Hugging Face–hosted repository that unifies AI evaluation results from HELM/lm-eval-harness/Inspect AI logs, leaderboards, and papers. The schema has five blocks (source metadata, model info, generation config, results, optional instance-level data) with Pydantic-based validation and per-run UUIDs. Already aggregating ~22k models × ~2.3k benchmarks across 31 formats, the artifact enables four concrete studies — cost-accuracy tradeoffs in agentic evals, version-dependent perplexity discrepancies, HELM instance-level reproducibility audits, and cross-benchmark IRT — that would otherwise require costly re-evaluation. It's a credible first attempt at infrastructure-level standardization for an ecosystem that has long suffered from incomparable scores.

## 核心贡献

- 提出**首个社区治理的 AI 评测结果 JSON Schema**（`evaleval/every_eval_ever`），含五大元数据块：source metadata / model info / generation config / evaluation results / 可选 instance-level 数据。
- 实现**自动转换器**覆盖 HELM、lm-eval-harness、Inspect AI 三大主流评测框架，加上 community-contributed leaderboard 抓取脚本；每个转换器产出 schema-compliant `.json` + 可选 `_samples.jsonl`。
- 部署**两层验证管线**：本地 CLI（rich/JSON/GitHub annotation 输出）与 Pull Request 阶段的 EvalEvalBot，基于 Pydantic v2 模型强制 schema 合规。
- 发布**众包社区仓库** `evaleval/EEE_datastore`：截至 2026-05-04 收录 22,235 个模型、2,273 个 unique benchmark、31 种评测格式、229K 条聚合结果。
- 通过四个**实证用例**展示统一 schema 催化的研究：在 CocoaBench / CORE-Bench Hard 揭示 agent scaffold 与 backbone 的成本-准确率 trade-off；证明 WikiText perplexity 在 token/word 归一化下相差 3+ 点（OPT-6.7B、Llama-2-7B）；用 instance-level 复现审计发现 HELM Entity-Matching 数据漂移与 SyntheticReasoning-Natural 空响应等问题；对 GPQA Diamond / Wordle Arena / JudgeBench 拟合 unidimensional IRT 比较难度与能力分布。

## Key Contributions

- A **versioned, community-governed JSON schema** for AI evaluation results (`evaleval/every_eval_ever`), structured into five modular blocks plus an optional instance-level companion file.
- **Automatic converters** for HELM, lm-eval-harness, Inspect AI and leaderboard/community formats — including per-instance outputs where source logs provide them.
- A **Pydantic v2 validation pipeline** exposed both as a CLI and as a GitHub bot (EvalEvalBot), with explicit, first-class handling of missing fields rather than silent defaults.
- A **crowdsourced Hugging Face dataset** (`evaleval/EEE_datastore`) with 22,235 models, 2,273 benchmarks, 31 formats, 229K aggregate results, and 28 contributing organizations.
- A **governance model** with three roles (core maintainers, contributors, community reviewers), immutable records, and explicit correction/retraction mechanisms that preserve history.
- Four **empirical case studies** that exploit the unified corpus: cost-accuracy tradeoffs in agentic evals (§7.1), perplexity normalization effects (§7.2), HELM instance-level reproducibility audit (§7.3), and cross-benchmark IRT meta-analysis (§7.4).

## Methodology

**Schema design.** Inspired by Croissant, EEE's schema was iterated with ~40 structured and ~110 unstructured inputs from benchmark creators, harness developers, governance experts, leaderboard operators, and industry practitioners; field inclusion required (i) reported in ≥1 existing framework, (ii) deemed necessary for interpretability by the majority, and (iii) anticipated to be available to others. Design principles (Table 1): accept partial records via minimal required fields, capture wide typed metadata, preserve run identity with UUIDs rather than canonical fingerprints (to surface duplicates instead of merging them), version/extend without rewriting old records, separate aggregate and instance-level records via sidecar `_samples.jsonl`, and organize reusable information blocks.

**Five schema blocks.**
1. `source_metadata` — `source_type` (documentation vs. evaluation_run), `evaluator_relationship` (first_party / third_party / self).
2. `model_info` — standardized `developer/name`, `access mode` distinguishing `inference_platform` (e.g., openai, anthropic) vs. `inference_engine` (e.g., vLLM); the latter exposes hidden confounds from serving stacks.
3. `generation_config` — first-class fields for temperature, n_samples, stop sequences; unknowns are *omitted* rather than silently defaulted.
4. `evaluation_results` — one entry per scored metric with a `metric_config` carrying `lower_is_better`, type (continuous / binary / ordinal), range, and uncertainty (SE, CI).
5. `instance_level` (optional `_samples.jsonl`) — single_turn (output object), multi_turn (messages array), or agentic (messages + nested tool_calls with sandbox/tool configuration).

**Converters + validation.** Three first-party converters (HELM, lm-eval-harness, Inspect AI) plus community-contributed leaderboard scrapers. Each contribution is validated at submission time by Pydantic v2 models generated from `eval.schema.json`; CLI supports rich/JSON/GitHub annotations; EvalEvalBot flags duplicate records and missing files in PRs.

**Governance.** Three roles (core maintainers / contributors / community reviewers); records are immutable once accepted with explicit correction/retraction mechanisms (so the LLaMA 65B MMLU 63.7-vs-48.8 discrepancy stays visible as two distinct records with metadata, rather than being silently "resolved").

**Repository statistics.** As of 2026-05-04: 229K aggregated results, 22,235 models, 2,273 benchmarks, 135 data PRs, 127 code PRs, 31 formats, 28 source organizations.

## Results & Highlights

- **Schema footprint:** 31 formats from 28 orgs unified into a single schema; conversion + validation pipeline demonstrated at scale.
- **§7.1 (agentic cost-accuracy):** On CocoaBench, Codex and OpenClaw with GPT-5.4 reach the same accuracy but Codex is cheaper/faster — pure scaffold effects invisible from scalar accuracy. On CORE-Bench Hard (HAL), Claude Code is cheaper than CORE-Agent for both Opus 4.5 and 4.1, but accuracy moves in opposite directions across the two backbones — scaffold and backbone effects are entangled and direction-dependent.
- **§7.2 (perplexity normalization):** GPTQ-style token-perplexity vs. lm-eval byte/word-perplexity differ by 1.43 (OPT-6.7B) and 3.33 (Llama-2-7B) on WikiText — same summed cross-entropy, different denominator; EEE's metric_config prevents silent aggregation.
- **§7.3 (HELM reproducibility audit):** Across three models × 14 single-turn HELM benchmarks, agreement between official HELM and local re-runs after EEE conversion ranged from ~78% (BoolQ on Falcon-7B) to 100%; Entity-Matching was incomparably drifted due to data-processing row-order changes; SyntheticReasoning-Natural revealed a serving artifact (empty official completions vs. non-empty local ones); WikiFact held ~92% consistent with stochastic sampling.
- **§7.4 (IRT meta-analysis):** Unidimensional IRT fit on GPQA Diamond / JudgeBench / Wordle Arena shows Wordle Arena has wider item-difficulty spread and the highest difficulty estimates, while GPQA's difficulty distribution suggests impending saturation — only visible because per-item outputs are co-located.
- **Macro-average metadata fill rates across 31 formats (Table 2):** model name 100%, HF repo URL 96%, metric ID 77%, temperature/n_samples only 23%, model parameters 3%, model license 3% — empirically confirming the authors' central thesis that even "comparable" fields are sparsely reported.

## Critical Assessment

- **Novelty:** EEE is not the first benchmark-aggregation project (HELM, Open LLM Leaderboard, Chatbot Arena, OpenCompass, etc.) and explicitly positions itself as a *translation layer above* harnesses, not a new harness. The real novelty claim is "first **unifying schema** across all these sources simultaneously with instance-level support." Plausible, though concurrent work (Abbas et al. 2025; another Inspect-AI-specific repo) is acknowledged and folded in — honest about the ecosystem.
- **Rigor:**
  - **+** Schema inclusion criteria are explicit (three-gate rule), field coverage is documented in App. B, converters are listed in App. C.
  - **+** Empirical demonstrations go beyond "we built a thing" — they show concrete reproduction failures (HELM audit) and concrete score-meaning collisions (perplexity) that *justify* the schema's existence.
  - **−** The four case studies are mostly illustrative, not full benchmarks of "how good is the schema." There is no formal lossiness audit per converter (which fields are dropped in HELM→EEE?), no schema-minimality proof, no inter-annotator agreement on schema compliance.
  - **−** "31 formats" is impressive-sounding but the macro-average fill-rate table (Table 2) simultaneously reveals how shallow coverage of key fields is across those formats. Some "format support" may be near-empty records.
- **Reproducibility:** Strong on infrastructure side (open schema, public dataset, public converters, validation at PR time). Weaker on outcome side: case studies reuse prior reproductions (HELM official records) and the authors themselves note EEE doesn't run evaluations, so serving-only confounds remain unrecoverable from the schema alone.
- **Significance:** Potentially high. The field's irreproducibility crisis (LLaMA 65B's 63.7 vs. 48.8 on MMLU is a textbook example) is widely acknowledged but under-instrumented; EEE is the first serious attempt at schema-level remediation with cross-org buy-in. If adopted by ≥2 of HELM/lm-eval/Inspect AI, downstream meta-analysis becomes substantially cheaper.
- **Red flags / weaknesses to note honestly:**
  1. **Adoption is the whole game** — a schema no one uses is a README. The paper acknowledges this but cannot solve it.
  2. **"Comparable format" ≠ "comparable scores."** EEE co-locates results but does not normalize them; the perplexity case study (§7.2) explicitly shows the same numeric label means different things across harnesses. The schema can prevent silent aggregation but does not by itself produce a valid cross-harness leaderboard.
  3. **Coverage bias is acknowledged but not measured.** The 28-org concentration on US-based, English, text, single-model evaluations is itself a finding ("five companies account for 23 of the 24 most frequently evaluated systems") — but the dataset is still mostly Open LLM Leaderboard-shaped.
  4. **Governance is heavyweight but unproven.** Three-role model + immutable records + correction flow is sensible but only ~one PR cycle old at paper time.
  5. **No rigorous baseline comparison vs. existing aggregators** (OpenCompass meta-benchmarks, HELM's own comparison UI). "First of its kind" is asserted but the line between EEE and a HELM-wide-export pipeline is thin in places.
  6. **Schema versioning & migration policy** is mentioned but not deeply specified in the main text (App. E.2 referenced).

## Strengths

- **Genuine, well-targeted problem.** Cross-harness irreproducibility is documented and acknowledged in the field; EEE's LLaMA-65B-on-MMLU example lands.
- **Right architectural shape.** Modular five-block schema + sidecar instance-level data + UUID-per-run + Pydantic validation is a textbook defensible design for an evolving artifact.
- **Real cross-community coalition.** 47 authors from 28 orgs, explicit roles, public governance doc — not a single-lab overreach.
- **Concrete empirical value.** All four case studies use the unified corpus to produce findings (cost-accuracy tradeoffs, perplexity denominator effect, HELM reproducibility gaps, IRT difficulty distributions) that were genuinely hard before.
- **Honest positioning.** Explicitly *not* a new evaluation harness; explicit limitations section; explicit concurrent-work citation and merging.
- **Hugging Face hosting + Pydantic validators** lower the friction for both consumption and contribution.

## Limitations

- **Coverage ≠ comparability:** Co-location of results does not normalize them; cross-harness score comparisons remain unsafe without case-by-case justification.
- **Strong text/single-model bias:** Multimodal, human preference (Chatbot Arena Elo), and multi-agent evaluations are only "partially supported" (authors' words).
- **Adoption-dependent value:** If HELM/lm-eval/Inspect AI do not adopt EEE export paths natively, the schema is reduced to a downstream aggregator.
- **No formal lossiness/audit per converter:** Converters are listed but their drop-rate of source fields is not systematically reported.
- **Schema governance is unproven:** Three-role model is sensible but no track record of resolving contested schema-change proposals.
- **Participation-based bias:** Coverage is biased by who contributes; 28-org concentration on US/English/text benchmarks is a selection effect, not a true cross-section of AI evaluation.
- **Deduplication left to consumers:** UUID-per-run preserves information but shifts duplicate-detection burden to analysis layer.
- **Cannot capture serving-stack confounds:** EEE surfaces mismatches (e.g., empty HELM completions) but cannot always attribute them to specific runtime causes.

## Relevance

该工作直接命中"LLM 评测与基准"研究中的基础设施痛点 —— 不同 harness / leaderboard / paper 报告的分数名义上同名却不可比。对关注评估方法学、跨基准元分析、或希望系统性复现论文报告分数的研究者，本工作既是工具也是问题清单。

## Questions for Further Exploration

- How lossy is each converter in practice — what fraction of HELM/lm-eval/Inspect AI fields are dropped or transformed into `additional_details`, and how is this measured?
- Will HELM, lm-eval-harness, and Inspect AI ship first-party EEE export, or only downstream converters? (Long-term adoption hinges on this.)
- Does the schema capture **sufficient** metadata to distinguish, e.g., `vLLM 0.4.x` vs. `vLLM 0.5.x` serving the same weights? (Serving-stack confounds were a major source of HELM mismatches in §7.3.)
- How does EEE interact with existing meta-evaluation work (BIG-bench, OpenCompass, MLPerf) — complement, overlap, or compete?
- For multimodal and agentic evaluations, where is the schema thin, and what concrete field additions would close those gaps?
- Can EEE instance-level data be combined across datasets to produce **cross-benchmark difficulty priors** (IRT/Rasch as in §7.4) at scale, enabling something like a "global LLM ability calibration"?
- What is the empirical duplication rate of records once contributed (e.g., the same model evaluated twice under slightly different prompts) — and is there a canonical deduplication reference implementation, as the limitations hint?
- How does the immutability + correction/retraction design interact with EU AI Act audit-trail requirements? Is EEE's record format a viable substrate for regulatory compliance logging?
- Coverage bias: how representative are the 28 contributing orgs of the global AI evaluation ecosystem (industry, academia, non-English, government)?
- What is the cost trajectory? The paper conservatively estimates hundreds of thousands of dollars to reproduce current EEE records; how does this scale as more contributors add agentic / multimodal runs?
