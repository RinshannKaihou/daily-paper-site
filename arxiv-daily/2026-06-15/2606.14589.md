> **一句话总结：** 基于一套自 2026 年 3 月持续生产运行的个人助手 LLM 代理运行时 (openclaw-model-bridge) 8 周内 22 次事件 / 28 次沉默失败的根因复盘，提出五类机制导向的失败分类法，并以 Class D (chained hallucination & fabrication, 命名为 *fail-plausible*) 突出 LLM 系统的独有风险——LLM 把错误信号改写为流畅的假故事交给用户。

> **TL;DR:** A longitudinal case study of an in-production LLM agent runtime (openclaw-model-bridge, running since March 2026 on a single macOS host: ~40 scheduled jobs, 8 LLM providers, ~1,100-note KB, defended by 4,286 unit tests / 827 governance checks / 90 invariants / 23 meta-rules / 14 mechanized scanners) catalogues 22 incidents / 28 silent-failure manifestations over an 8-week window into a **mechanism-oriented five-class taxonomy**: (A) environment & platform quirks, (B) design-assumption mismatches, (C) error swallowing & dilution, (D) chained hallucination & fabrication, and (E) operational omission & forensic blind spots. **Class D ("fail-plausible") is the LLM-specific, most dangerous class**: a pollution chain makes the model convert an error log into confident, fluent fabrications delivered to the user (e.g. an HTTP 400 "Hugging Face platform crisis" industry digest, a fabricated macOS Full-Disk-Access remediation, a fabricated software release). Three quantified findings: **~70% of silent failures were caught by human user-view observation**, not by tests/audits; a 15-incident retrospective audit = **0% ex-ante prevention vs 87% ex-post regression blocking** (audits are regression engines, not prediction engines); **incident latency (13h–60d) tracks mechanism layer, not code complexity** — the longest silences lived in the *seams* between components, not in complex code. A defense framework (sabotage-validated guards, declared-state convergence, layered anti-fabrication, monitoring that monitors itself) and a "Sunset Law" (retire complexity before adding defense) are proposed. All postmortems and the governance engine (also released on PyPI as `openclaw-ontology-engine` v0.1.0) are public.

## 核心贡献

- **五类机制导向的沉默失败分类法** (A–E)，从 22 个生产事件 / 28 次失败中归纳；按机制 (mechanism) 而非按位置 (location) 分类，原因是同一机制会跨组件复发，对一个机制做防御可以"免疫一类"而非"免疫一个文件"。
- **fail-plausible 概念**：区别于传统 gray failure 的"观察者失明"，LLM 代理系统会主动把错误信号转写为上下文合适、流畅、虚假的输出给用户，定义为 gray failure 的"差分可观测性"在 LLM 时代的升级。
- **Class D 的四起生产事件**：D1 把一个 UTF-16 代理项编码错误转写成"Hugging Face 平台危机"行业分析；D2 把系统告警污染进聊天上下文后产出虚构的 macOS 修复指令；D3 在 LLM 调用失败后用机械行过滤输出"看似合理的周报"并写 `llm: true`；D4 从论文列表推断"项目一定已发布"并伪造内部版本号——四种 pollution chain 都满足"触发-放大-隐藏"三段式结构。
- **量化三个反直觉发现**：~70% 沉默失败由人眼发现、回归审计 0% 事前预防 vs 87% 回归拦截、事件潜伏期 13h–60d 跟机制所在层 (observational distance) 强相关、跟代码复杂度弱相关。
- **三段防御成熟路径**：点修复 (point fix) → 元规则 (meta-rule) → 机械化扫描器 (scanner)。论文量化：止于点修复的教训几天内在 8 个站点复发；走到 scanner 的元规则截至研究截止零复发。
- **"日落法则" (Sunset Law)**：每次新增防御前先尝试退役等效的一个；"一个逻辑实体必须只有一个物理表示"；防御本身也是事件面，优先减缝 (seam reduction) 而非堆盾 (defense accretion)。
- **可复现工件**：22 篇 postmortem、治理本体 (90 invariants / 827 checks / 23 meta-rules)、14 个扫描器、`openclaw-ontology-engine` 作为独立包发布到 PyPI。

## Key Contributions

- A **mechanism-oriented five-class taxonomy of silent failures** derived from 22 production incidents / 28 silent-failure manifestations in a real, continuously running LLM agent runtime. Classes A/C/E have direct ancestry in the gray-failure and fail-slow literature; **Class D (chained hallucination & fabrication) is LLM-specific**.
- Coining **fail-plausible**: the LLM-era escalation of gray failure's differential observability — the observer is not just blind, it is *convincingly lied to* by the failure itself. Four production incidents are documented in which LLMs converted polluted context (error logs, stale alerts, ungrounded enrichment) into confident fabrications pushed to the user.
- **Three-layer causal decomposition** (trigger / amplifier / concealer) that recurs in nearly every postmortem, with the prescriptive implication that trigger-only fixes are cosmetic.
- **Quantitative findings**: ~70% discovery via human user-view observation; 15-incident retrospective audit = 0% ex-ante prevention vs 87% ex-post regression blocking; 80% of audit misses are "blank categories" (dimensions the audit had never conceived); incident latency 13h–60d tracks mechanism layer, not code complexity.
- **Three-step defense maturation** (point fix → meta-rule → mechanized scanner) with empirical evidence that lessons stopping at step 1 recur within days and lessons reaching step 3 have zero recorded recurrences.
- **Sunset Law** and seam-reduction design principles, motivated by the empirical observation that the longest-lived failures lived in component seams (deployment topology, dev-vs-target-OS, declared-vs-runtime state, observer-vs-observed).
- A **defense framework** with audited scorecard: declarative governance (90 invariants / 827 checks), sabotage validation, declared-state convergence engine, context hygiene + six-level cumulative anti-fabrication prompt guards, and monitoring that monitors itself; adversarial audit of 16 destruction scenarios scored 16/16 after its own blind-spot round.
- All artifacts (postmortems, governance ontology, scanners, governance engine as PyPI package `openclaw-ontology-engine` v0.1.0) released publicly.

## Methodology

- **System under study**: `openclaw-model-bridge` — a two-layer middleware connecting self-hosted and commercial LLMs to the open-source OpenClaw personal-agent framework (bridging WhatsApp and Discord). In continuous production on a single macOS host since early March 2026.
  - **Three-plane architecture**: control plane (tool-governance proxy with tool cap, schema repair, alert-context stripping, custom tool interception; declarative governance engine; SLO + circuit breakers; **convergence engine** that machine-syncs declared state to runtime state); capability plane (8-provider adapter with self-hosted Qwen3-235B primary, Qwen2.5-VL multimodal route, plus commercial fallbacks and capability-scored fallback chains); memory plane (~1,100+ notes with local-embedding RAG, multimodal media index, conversation-harvest, and LLM-driven daily synthesis jobs — "dream", "evening digest", "deep dive").
  - **Scale at study cutoff (2026-06-11)**: ~40 registered scheduled jobs across two cron schedulers, 8 LLM providers, 3 supervised long-running services, 4,286 unit tests in 121 suites, 22 published postmortems. One human operator + one AI engineering collaborator (Claude) develop and operate the system; the runtime itself is powered by Qwen3-class models.
- **Observation window**: 2026-04-09 to 2026-06-02 (eight weeks).
- **Selection criteria for incidents**: (a) reached production, (b) had a silent phase (failing while all automated indicators stayed green), (c) closed with a full postmortem. **22 incidents qualify.**
- **Postmortem protocol ("exception-analysis constitution")** — required before any fix:
  1. Full causal-chain diagram (timeline × layer × logic × architecture, with code-branch truth values).
  2. Three-layer root cause: **trigger** (external spark) / **amplifier** (architectural flaw that spread it) / **concealer** (absence that hid it).
  3. Timeline reconstruction with per-minute precision where logs allow.
  4. Condition-combination analysis (why now and not before).
  5. Feeding the governance ontology (new invariants, meta-rule candidacy, catalog entry).
- **Taxonomy construction**: inductive, mechanism-first (location classes discarded as having no predictive/defensive value). Each new postmortem matched against existing classes' "true root cause"; two consecutive non-matches forced a split or new class. The five-class scheme has been stable since mid-May 2026 over the last 9 incidents.
- **MR-4 meta-pattern counter**: silent-failure-as-a-bug counter started early in the study; reached ≥ 28 across 22 incidents.
- **Retrospective audit (Q1/Q2/Q3)**: mid-study audit of 15 incidents with three questions: could the audit as it existed have caught it? why not? do the post-incident guards block the same class going forward?
- **Adversarial audit**: 16 destruction scenarios injected against the live repository (10 replaying known incidents, 6 probing suspected blind spots) — useful precisely because its first run did not score 16/16.
- **Two operators (human + AI collaborator) classified incidents without independent annotators** — no κ reported (explicitly acknowledged as a limitation).

## Results & Highlights

- **Class A — Environment & platform quirks (1 + 6 sub-events; hours–weeks silence)**: bash 3.2 not propagating ERR traps into functions without `set -E` silently disarmed a watchdog's self-alarm; BSD `awk` aborting on invalid UTF-8 killed a monitoring script for 7 days; absence of GNU `timeout` on stock macOS; CJK full-width punctuation adjacent to unbraced shell variables; messaging client folding long messages at an undocumented ~4,000-character threshold. **Signature: dev always green; only target OS reveals the bug.** Defense: cross-OS quirk scanner (6 patterns), meta-rule requiring framework-level fixes to be validated on the target environment.
- **Class B — Design-assumption mismatches (4 incidents; days silence)**: metadata-resolution function shipped with three path candidates, none covering the actual production location — fell back to unfiltered mode for 5 days; LLM-output parser indexed lines positionally (`lines[i+1]`, `lines[i+2]`, stride 3) — when the model omits one line, every subsequent field shifts. **Signature: tests mirror the assumption rather than the caller.** Meta-rule: parsers must be key-based, never positional.
- **Class C — Error swallowing and dilution (5 incidents; hours–days silence)**: three mechanism variants — (i) **swallowing** (a counter filtered on `status=="fail"` and missed `status=="error"`, governance audit printed "all invariants hold" over dead checks; a separate governance-executor YAML key mismatch caused **67 checks across 21 invariants to execute `exec("")` for months** — vacuous passes — discovered only when a new invariant's sabotage validation refused to fail); (ii) **dilution** (HTTP 502 wrapped, body never read, "client" reduced to `f"HTTP {code}: {reason}"` — three hops each stripping cause, alert that reached the human contained zero actionable bits); (iii) **amplified swallowing** (an automated batch tool injected `os.environ` reads into 8 job scripts without injecting the import — `NameError` in all 8, caught by fail-open guard, every test green).
- **Class D — Chained hallucination & fabrication / fail-plausible (4 incidents; hours–days silence)**:
  - **D1 — fabricated platform crisis**: Unicode surrogate in scraped content → `json.dump` raises mid-write → truncated body → HTTP 400 → 456-byte HTML error page → logging function writes diagnostics to **stdout** → caller captures stdout via command substitution as the signal payload → cache fills with `HTTP Error 400: Bad JSON` strings → reduce-step LLM, prompted to find cross-domain signals, composes a confident "Hugging Face platform crisis" analysis → pushed to user as routine insight digest. **Every component worked as designed.** Fix: a single `>&2` redirection; defense-in-depth added (surrogate sanitization, encoding error policy, anti-pollution prompt guards).
  - **D2 — fabricated remediation**: a system alert was persisted into chat history as an ordinary assistant message; 36 minutes later the user asked an unrelated architecture question; the model replied that it had "received the system alert follow-up task" and instructed the user to grant **Full Disk Access to a cron binary in macOS System Preferences** — off-topic and technically fabricated for the scenario. (Forensic twist: weeks later, an unrelated 60-day investigation established that the general remediation direction the model fabricated was coincidentally relevant to a real problem elsewhere — fabrication is not refuted by occasional accuracy.)
  - **D3 — fabricated success**: a weekly review job's LLM call failed → fell back to a mechanical line-filter that emitted leftover container headings as if they were review content → wrote `"llm": true` into its status file unconditionally → passed casual inspection for weeks. **Fabrication does not require a model**: a fallback that manufactures plausible-shaped output is a hallucination implemented in shell.
  - **D4 — fabricated release**: an evening digest, given a list of the day's high-alignment papers as context enrichment, inferred the user's project "must have shipped" and announced a community release of an internal version number that exists only in the project changelog.
- **Class E — Operational omission and forensic blind spots (8 incidents; days–60 days silence — the longest)**:
  - **Omission**: a new daily-analysis job was fully implemented, tested, registered in the job registry, and deployed — and **never ran**, because the final step (writing the crontab line) was a human memory item. Three independent small bugs hid it (preflight grepped for only one of two registry-drift warning strings; crontab helper did not check its own install exit code and compared counts with `<`; the job's absence produced no log). → catalyzed the **convergence engine**.
  - **HEARTBEAT.md reserved-file incident**: the agent wrote "task complete" notes into a file the runtime treats as a heartbeat control file; gateway then strips the heartbeat acknowledgment token from outbound messages → **13 hours of total silence** with every component functioning exactly as designed. Meta-rule: any file path carrying special runtime semantics must be unwritable by the LLM's generic file tools.
  - **60-day external-SSD backup EPERM**: six hypotheses (filesystem format, ownership UIDs, ACLs, daemon contention, snapshot locks, physical disconnection) each falsified by data; breakthrough came from `log show` revealing **macOS TCC sandbox denials** — cron-derived processes lack Full Disk Access by default. **Methodological finding**: during those weeks, the forensic collectors themselves (`lsof`, ACL listing, snapshot enumeration) were being silently denied by the same sandbox and returning empty output — diagnosis pipeline recorded "normal/empty". An instrument that cannot distinguish "nothing there" from "I was not allowed to look" manufactures false reassurance. All collectors now capture stderr separately and tag `[sandbox denied]` as a first-class observation.
  - Also: 9-hour total gateway outage (quiet-hours filter suppressed both channels including the emergency one; keepalive logged WARN without alerting; restart script reported success without verifying health) → meta-rule: alert path must not depend on the failing subject; watchdog that died of a Class A quirk and stayed dead for 7 days because nothing watched the watcher.
- **Cross-cutting findings**:
  - **Latency tracks mechanism layer, not code complexity** (Fig. 4 in paper). Code-level bugs die young (unit tests and the next run catch them). What survives for weeks lives where no test runs — deployment topology, OS policy, monitoring-of-monitoring, declared-vs-runtime state. **Latency is a measure of observational distance**: the further a mechanism sits from any existing observer, the longer it lives. For silent failures, **time-to-detect dominates time-to-repair by 1–2 orders of magnitude**.
  - **Discovery channel distribution** (Table 2): ~70% human user-view (a weekly 30-min observation ritual: alert noise, push latency, information density, response quality); high share via target-environment execution (all Class A); rising share via self-observation (governance auditing governance); **~0% via unit tests for this corpus** (tautological by selection, but the magnitude of the first row is not).
  - **Silent failure is a bug class, not a bug**: the MR-4 manifestation counter kept advancing at a roughly constant rate even as defenses accumulated — never twice in the same form (swallowing → dilution → fabrication → fix-introduced-worse-silence → correct-fix-to-wrong-bug → vacuous verification).
  - **Defense maturation (point fix → meta-rule → scanner)**: every meta-rule that reached step 3 has zero recorded recurrences as of study cutoff; every recurrence in the corpus involves a lesson that stopped at step 1. Import-omission bug fixed at one site recurred at **8 sites** via an automated injector two days later because the lesson existed only as a diff.
  - **Audits are regression engines, not prediction engines** (15-incident Q1/Q2/Q3 audit): 0% ex-ante prevention, 13% partial early warning, 87% ex-post regression blocking, 80% of misses were "blank categories" (dimensions no invariant had ever contemplated).
- **Defense framework scorecard**:
  - **Post-deployment measurement of anti-fabrication ladder**: targeted fabrication patterns (multi-hop causal chains, "therefore"-style necessity claims) dropped **53–92%** in the affected synthesis job; tagged-claim usage went from 0 to ~9 per day.
  - **Adversarial audit**: 16/16 destruction scenarios after the blind-spot batch drove its own round of guard additions — useful precisely because its first run did not score 16/16.
  - **Sabotage validation caught**: guards that matched their own assertion strings (self-referential greps), the 67 vacuous checks (when a sabotage refused to fail), tests whose fixtures mirrored the wrong assumption.
  - **Convergence engine** retired the largest Class E mechanism from "human memory item" to "machine-closed loop" — but its own history is a cautionary tale: an audit that applies synchronization is an observer mutating the observed, and one drift bug caused a thrice-recurring crontab duplicate until the audit path was forced to dry-run → standing rule that **audit observes and never mutates**.

## Critical Assessment

- **Novelty**: The taxonomy itself (A/C/E) overlaps with established reliability engineering (Heisenbugs, error masking, fail-slow, gray failure); the contribution is the *mechanism-first* reframe and the LLM-specific **Class D (fail-plausible)**. The *fail-plausible* coinage is genuinely new and clearly articulated. The "audits are regression engines, not prediction engines" framing is sharp and broadly applicable. The **trigger/amplifier/concealer** decomposition and the **fix→rule→scanner maturation path** are the paper's most transferable structural contributions. The **seam-not-component** reframing of complexity is the most operationally useful insight in the paper.
- **Rigor**: Concrete numbers throughout (22, 28, 70%, 87%, 0%, 80%, 53–92%, 16/16) — falsifiable and traceable to a public repo, which is unusual for this kind of work. **However**: (i) no formal definition distinguishing "incident" from "silent-failure manifestation" is given — the same incident contains multiple manifestations and the counter has "deliberate gaps"; (ii) no inter-rater reliability for taxonomy assignment (two-operator, no independent annotation); (iii) classification is **load-bearing** (each class drives a real scanner with objective findings), which partially mitigates the confirmation-bias risk; (iv) postmortem quality varies with log retention; (v) the AI collaborator co-wrote the postmortems that the AI collaborator's own work is graded against — the author offers the public repo and traceable numbers as the primary check on narrative bias, which is the right move but does not eliminate it.
- **Reproducibility**: The taxonomy, the governance ontology, the scanners, and the `openclaw-ontology-engine` PyPI package are reusable. The system itself is a personal-assistant runtime on a single macOS host with a single human operator — it cannot be replicated. The 22 postmortems and a data inventory mapping every number to a repository source are the primary reproducibility surface.
- **Significance**: High for the LLM-agent reliability and observability communities. *Fail-plausible* deserves to enter the standard vocabulary. The empirical claims (humans catch 70%, audits prevent 0%, latency 13h–60d) are large-magnitude effects that will likely generalize in direction if not in number. The "seam reduction over defense accretion" posture is broadly applicable beyond LLM agents. The **adversarial audit scoring 16/16 only after its blind-spot batch** is a refreshingly honest result.
- **Red flags**:
  - **Selection / survivorship bias**: corpus contains only failures that *stopped* being silent. Failures silent at study end are by construction absent. The 60-day tail may be a lower bound on the true tail. (Author acknowledges this; it argues for, not against, the thesis.)
  - **Single-system, single-operator-pair**: the 70% user-view share "may reflect this system's unusually attentive operator" (author's own caveat) — the magnitude is presented as an existence proof, not a constant. But Class D frequencies and the precise ratio of mechanisms may not transfer.
  - **Mechanism boundaries are blurry in practice** (e.g., when does C "error swallowing" become D "chained hallucination"?) — no formal decision procedure is given. The author notes "borderline cases were included when the actionable signal was absent" and that "reasonable observers could draw the line differently for 2–3 incidents."
  - **No baseline comparison** to non-LLM production runtimes of similar scale — the gray-failure and fail-slow lineages are cited but never quantitatively compared.
  - **The "70% human-caught" number has no counterfactual**: would more aggressive alerting change the mix between classes, or just shift which class a failure is filed under?
  - **The defense framework is sketched and partially evaluated, not end-to-end evaluated**. The 53–92% drop in fabrication patterns is reported for the affected synthesis job only; the system-wide incident rate is reported as "declined" in the five weeks since adopting the Sunset Law with "appropriate caution about confounds."
  - **The "audits as regression engines, not prediction engines" claim is presented as audit-as-it-existed**; the audit framework itself has been extended during the study, so the 0% ex-ante figure is a snapshot of an evolving system, not a steady-state property of "audits" in general.
  - **AI-collaborator entanglements**: the postmortem protocol's rigidity is *attributed in part* to managing AI-collaborator behavior (an AI will "pattern-match symptoms to plausible fixes" otherwise) — and the same AI co-wrote the postmortems. The author frames this as "AI collaboration shifts the binding constraint of reliability engineering from implementation bandwidth to judgment discipline," which is plausible but is also a self-serving description of the arrangement the author chose.
  - **"Every meta-rule that reached step 3 has zero recurrences"** is a strong claim and the supporting evidence base (count of rules, time window, recurrence-detection mechanism) is not given in the abstract / main text I sampled — it should be in §5.5.
  - **Some Class-D incidents involve genuinely impressive chains** (D1 in particular) and are the most quotable part of the paper, but the paper would be stronger with a class-D incident that *did not* require a model to fabricate (D3 is the closest, and the lesson "fabrication does not require a model" is well-drawn).

## Strengths

- Reframes a real, painful class of LLM-system failure with a memorable term (*fail-plausible*) and a clean mechanism-level taxonomy that is load-bearing in the actual defense framework.
- Reports concrete, falsifiable numbers rather than vague claims; ties every number to a public repository.
- **Honest scorecards**: 0% ex-ante prevention is reported as a *job description* of audits, not as a failure; 70% human-catch is reported as an existence proof, not a constant; the adversarial audit scored 16/16 *only after* its own blind-spot round.
- The "audits are regression engines, not prediction engines" insight is broadly applicable beyond LLM agents — it should be cited by anyone building post-incident infrastructure.
- The **trigger / amplifier / concealer** decomposition is the paper's most operationally useful contribution; the prescriptive implication (fix the concealer, not the trigger) is non-obvious and well-supported by the corpus.
- The **defense maturation path** (point fix → meta-rule → scanner) is the right abstraction and the empirical evidence for it is sharp.
- The **Sunset Law** and the seam-vs-component reframing are non-obvious and broadly applicable.
- Public artifacts (postmortems, governance engine as a standalone PyPI package) raise the floor for the field.
- The AI-involvement disclosure is unusually honest and the paper uses its own arrangement as a research finding rather than hiding it.

## Limitations

- Single-system, single-author, single-host-OS, single-operator-pair, eight-week case study; frequencies and shares are descriptive of one case, not population estimates.
- The system is unusually heavily defended (4,286 unit tests, 827 governance checks, 90 invariants, 14 scanners) — findings may not transfer to lightly tested agent stacks.
- No formal taxonomy validation: no inter-rater agreement, no held-out incidents, no formal decision procedure for class boundaries (author flags 2–3 borderline cases).
- Defense framework is partially evaluated (53–92% drop in targeted fabrication patterns in one synthesis job; "incident frequency declined" with explicit confounds caveat); no controlled comparison.
- Right-censored latency distribution: failures still silent at study end are absent by construction.
- AI-collaborator entanglements are acknowledged but not independent of the analysis they support.
- 70% user-view discovery share may not generalize; the author presents it as an existence proof, not a constant.

## Relevance

本文与 LLM 代理系统可靠性、可观测性、幻觉防御研究直接相关；其五类机制导向的分类法、*fail-plausible* 概念、trigger/amplifier/concealer 三段式归因、fix→rule→scanner 防御成熟路径以及"日落法则"对设计、评审、运维 LLM 代理运行时具有直接方法论价值。

## Questions for Further Exploration

- Does the five-class taxonomy transfer cleanly to agent runtimes built on different stacks (LangGraph, AutoGen, custom) and to multi-agent systems (where Cemri et al.'s MAST taxonomy covers a related but distinct failure space)?
- Can *fail-plausible* outputs be detected automatically (e.g., via contradiction checks against structured tool-call logs, or provenance-trust scoring of injected context) without imposing heavy latency cost on every step? What is the recall/precision tradeoff at the prompt-injection time credibility-tier step?
- What is the minimum viable observability surface for a Class-D pollution chain to be caught before reaching the user — and at what point does the defense cost dominate the synthesis job's value?
- The 70% human-catch rate assumes a vigilant human in the loop with a weekly observation ritual. What is the equivalent rate when the agent operates truly autonomously (no user-view checkpoint), and can the LLM-as-observer pattern recover the gap without inheriting the same failure classes?
- The 60-day TCC-sandbox silence was a single observation. How do sandbox / OS-policy seams behave in containerized or multi-OS agent deployments, and does the "instrument that cannot distinguish 'nothing there' from 'I was not allowed to look'" anti-pattern generalize?
- The "audits are regression engines, not prediction engines" position is presented as a job description, not a limit. What *would* an ex-ante prediction engine look like for novel mechanism classes — and is "adversarial review + target-environment exposure + user-view observation" the best we can do, or are there structured methods (e.g., counterfactual trace replay, mutation testing at the orchestration layer) that move the needle?
- "Audit observes and never mutates" is a strong standing rule. Are there classes of agent-runtime reliability work where a *read-only* audit is provably insufficient and a controlled mutation channel is the only way to close the loop? How would such a channel be sandboxed?
- The fix→rule→scanner maturation path claims zero recurrences for rules at step 3. What is the empirical observation window and recurrence-detection mechanism, and how does this hold up as the rule base grows from 23 to hundreds?
- Class D's pollution chains all involve text-based pollution entering an LLM context. As agents gain tool-use over more stateful surfaces (databases, file systems, calendars), do new pollution vectors (e.g., a poisoned calendar entry that becomes a Class-D trigger) appear, and does the taxonomy extend cleanly?
- The AI-collaborator finding (judgment discipline > implementation bandwidth) is presented as an observation, not a verdict. Under what collaboration patterns does it *not* hold — e.g., when the AI collaborator has its own incentives, when the human is not a domain expert, or when the system operates in adversarial settings?
