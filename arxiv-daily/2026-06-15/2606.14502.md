> **一句话总结：** 这是一篇由腾讯优图实验室主导的综述,把LLM的演进刻画为"认知核心(Chatbot → Thinking LLM)×工具执行(Agent → OpenClaw)"的二维演化,并把"Workspace + Skill"作为从瞬时对话转向持久数字同事工作的核心机制,辅以数据(指令对→CoT→State-Action-Observation轨迹)与评估(答案打分→过程验证→任务闭环)的同步迁移论述。

> **TL;DR:** A 70+ page survey/position paper (20 authors, primarily Tencent Youtu Lab + Tsinghua + UIUC) that frames the LLM → autonomous-AI transition as a 2D axis — (i) cognitive core: stateless "fast thinking" chatbots → inference-time deliberate "Thinking LLMs" (CoT, PRM, RLVR), and (ii) tool-augmented execution: episodic tool-calling agents → persistent "OpenClaw"-style Workstations that combine a durable Workspace, reusable Skills (e.g. SKILL.md packages), verification loops, and runtime governance. The central thesis is that "Workspace + Skill" is the mechanism that turns conversation into "task closure" — leaving a correct, auditable, repeatable final state. The paper also argues the data paradigm shifts from instruction-response pairs → long CoT + process reward data → State-Action-Observation trajectories, and the evaluation paradigm shifts from answer correctness → process verification → task-closure rate → workspace safety. Empirically anchors the narrative with a 50%-time-horizon chart showing agent task length growing from 2.4 s (GPT-2, 2019) to 718.8 min (Opus 4.6, 2026).

## 核心贡献
- 提出"Chatbot → Digital Colleague"二维分类框架,把LLM范式转变拆解为**认知核心**(Chatbot vs Thinking LLM)与**工具执行**(Agent vs OpenClaw)两个耦合轴
- 主张"Workspace + Skill"是实现持久化、可重用、可审计任务闭环的核心机制,给出了Agent Era到OpenClaw Era的明确边界条件(7个维度的对比表)
- 梳理数据范式迁移:从"指令-回复"对,到长CoT+过程奖励数据(PRM),再到State–Action–Observation轨迹
- 梳理评估范式迁移的四阶段演进:输出准确率 → LLM-as-Judge过程验证 → 任务闭环率 → 工作站级能力与安全
- 给出可量化的"代理能力时间地平线"演化图(2.4秒到718.8分钟)与代表性系统的发布时间线
- 面向"自演化AI生态"提出6个开放挑战(长程可靠性、安全治理、人机协作伦理、记忆/上下文、超越梯度的学习、可组合技能与多代理)与5个未来方向(从prompt工程到harness工程、AI-Native工作空间为数字具身、超越梯度的系统维护、可组合技能市场、治理下的自演化)

## Key Contributions
- A unifying **two-dimensional framework** ("Chatbot → Digital Colleague") that decomposes the LLM-agent evolution into the cognitive-core axis and the tool-augmented-execution axis.
- A clear **boundary definition** between the Agent Era and the OpenClaw Era along 7 dimensions (organizing abstraction, unit of action, state model, knowledge reuse, task objective, evaluation focus, failure recovery, safety boundary) — see Table 4 of the paper.
- A **taxonomy of representative systems** (OpenClaw, OpenHands, SWE-agent, SemaClaw, Sema Code, Voyager, Anthropic Agent Skills, OpenClaw Skills, etc.) organized into Workspace / Skill / Task-Closure / Evaluation / Reliability / Security / Governance categories (Table 5).
- A **data-construction shift narrative**: instruction-response SFT → long CoT + process reward models → State-Action-Observation trajectories for agents and workstation systems.
- A **four-stage evaluation shift**: (I) final-output accuracy, (II) LLM-as-Judge for long-chain reasoning, (III) task closure rate, (IV) workspace/OpenClaw capability + safety.
- A **socio-technical roadmap** for reliable autonomous AI: long-horizon reliability, memory and context, safety/governance, beyond-gradient learning, composable skills, multi-agent ecosystems, and self-evolving systems.
- An **empirical time-horizon chart** showing exponential growth in median task length for frontier agents (2.4 s for GPT-2 in 2019 → 718.8 min for Opus 4.6 in 2026), used to anchor the "Digital Colleague" claim.

## Methodology
- This is a **survey / position paper**, not an empirical study. Methodology is **conceptual synthesis**:
  - **Two-dimensional taxonomy** that maps the field's evolution along (i) cognitive core and (ii) tool-augmented execution.
  - **Annotated timeline** of representative model/system releases from 2019 to 2026, across text models, multimodal models, thinking models, agent cognition, agent-environment interaction, and workspace/skill stacks.
  - **Comparative tables** contrasting the Agent Era and the OpenClaw Era on 7–8 dimensions (Table 4) and cataloging ~30 representative works in the OpenClaw era (Table 5).
  - **Cross-cutting discussion of data and evaluation paradigms** through the same evolutionary lens.
  - **Qualitative discussion** of open challenges and future directions for self-evolving AI ecosystems.
- Empirical content is limited to a single chart of "50%-time horizon" (median task length), which is presented as evidence for the trend, not as a novel measurement contribution.

## Results & Highlights
- **50%-time-horizon growth** (Figure 2): the median length of coding tasks frontier agents can complete has grown roughly exponentially from 2.4 s (GPT-2, 2019) → 8.9 s (GPT-3) → 36.3 s (GPT-3.5) → 3.9 min (GPT-4) → 38.8 min (o1) → 119.7 min (o3) → 224.3 min (Gemini 3 Pro) → 303.0 min (GPT-5) → 718.8 min (Opus 4.6, early 2026). The paper interprets this as the empirical signature of the Chatbot → Digital Colleague shift.
- **Agent Era vs OpenClaw Era — seven-axis comparison** (Table 4): the OpenClaw era is characterized by (1) organizing abstraction = persistent workspace (vs environment-action-feedback loop), (2) unit of action = workspace operation (vs tool call), (3) state model = durable files / sessions / repositories / local memory (vs episodic observations), (4) knowledge reuse = reusable skill packages (vs ad-hoc prompts), (5) task objective = deliver a correct, inspectable, recoverable final state (vs produce useful intermediate actions), (6) evaluation focus = task closure + repeatability + auditability (vs action correctness), (7) failure recovery = structured verification, rollback, rerun, sandboxing, state repair (vs best-effort retry), (8) safety boundary = runtime permissions, provenance tracking, audit logs (vs prompt-level guardrails).
- **30-work catalog** of OpenClaw-era systems (Table 5) organized into 7 buckets: Workspace (OpenClaw, OpenHands, SWE-agent, SemaClaw, Sema Code), Skill (Voyager, Anthropic Agent Skills, OpenClaw Skills, SkillFortify), Task-Closure benchmarks (SWE-bench, Terminal-Bench), Evaluation (OSWorld, WebArena, WorkArena, TheAgentCompany), Reliability (Reliability of Computer-Use Agents, Science of Agent Reliability, Verifiers for Computer-Use Agents), Security (Your Agent Can Hurt You, Systematic OpenClaw Security, OS-Harm), Governance (Taming OpenClaw, Don't Let the Claw Grip Your Hand, OpenClaw PRISM, ClawGuard, Agentic Forensics).
- **Representative OpenClaw system**: Tencent Youtu Lab's "OpenClaw" is described as a local personal assistant wrapping a gateway, workspace, communication channels, prompt files, skills, and tool integrations; presented as the embodiment of the "Workspace + Skill" thesis.

## Critical Assessment

**Novelty.** The two-dimensional (cognitive core × tool execution) framing is **a useful, original synthesis** — most prior surveys cover either reasoning models (e.g. the System-1/System-2 surveys) or tool-use agents (e.g. the LLM-agent surveys) in isolation, and very few connect them tightly to the workstation / persistent-state paradigm. The "Workspace + Skill" name and the 7-axis Agent↔OpenClaw boundary table are genuinely clarifying.

**Rigor.** As a survey, the paper is **broad rather than deep**: it lists many systems and concepts but rarely compares them on a shared quantitative metric. The empirical centerpiece (Figure 2 — the 50%-time-horizon chart) is borrowed from `theaidigest.org/time-horizons` and is presented as a motivating trend, not as the paper's own measurement. The claim that "Workspace + Skill is the key leap" is asserted more than it is proven; no head-to-head experiments of, e.g., a tool-calling agent vs an OpenClaw-style agent on the same benchmark are provided.

**Reproducibility.** The paper is **conceptual**; no code or models are released. "OpenClaw" is referenced as a representative system, but it is unclear from the paper alone whether it is open-source, closed, or hypothetical. The project page is mentioned but the full-text discussion of OpenClaw is light on technical details (architecture, API, training procedure, evaluation protocol).

**Significance.** The framing is **useful for the field**: it gives the community a shared vocabulary for the Agent → persistent-workstation shift, paralleling the Anthropic / Claude Code / Devin / OpenHands direction in industry. The Section 6 future-directions list (harness engineering, AI-native workspaces, beyond-gradient learning, composable skills, self-evolving systems) is essentially a research agenda for the next 2–3 years of agent research.

**Red flags.**
- **Self-promotion risk**: "OpenClaw" is the paper's exemplar of the entire OpenClaw Era, and it is Tencent Youtu Lab's own product. The framing is naturally slanted to validate Tencent's positioning. A reader should treat the OpenClaw case study as marketing-flavored rather than as a neutral empirical example.
- **Date and citation issues**: the paper's release-date line reads "arXiv:2606.14502v1 [cs.AI] 12 Jun 2026", and the timeline includes several 2026 model releases (e.g. "MiniMax M2.7", "Hy3-Preview", "GLM-5.1", "DeepSeek-V4", "Kimi K2.6", "Opus 4.6", "GPT-5.5"). Some of these names look like forward projections or placeholder strings; the bibliography contains a very large number of 2025–2026 entries that should be spot-checked for verifiability.
- **Vague boundary criteria**: the Agent ↔ OpenClaw distinction is presented as a 7-axis qualitative table. There is no operational test (e.g., "if a system persists state across sessions, it is OpenClaw-era") — many real systems (e.g. Devin, Claude Code) sit ambiguously on the boundary.
- **Light on quantitative comparisons**: the paper would be much stronger with a single table comparing, e.g., SWE-bench Verified scores of tool-calling agents vs OpenClaw-style agents, or a benchmark showing the "skill reuse" effect on long-horizon task completion. None is provided.
- **Overclaim on "self-evolving ecosystems"**: the future-directions section (6.2.5) is speculative; the paper does not present any concrete self-evolution experiment.

## Strengths
- The **two-dimensional framework** is clean, memorable, and pedagogically useful — it would serve well as the organizing structure for a graduate seminar or a product strategy doc.
- The **boundary table** between Agent and OpenClaw eras is a real contribution: it gives practitioners and researchers a concrete checklist for what "persistent workstation AI" actually requires.
- The **time-horizon chart** is an effective visual anchor and is, to my knowledge, one of the most accessible single images of agent capability growth over the past 7 years.
- The **catalog of representative works** (Table 5) is a useful starting point for anyone surveying the OpenClaw-style ecosystem.
- The **data and evaluation paradigm shifts** (Part IV) are well-articulated and connect the model-level and system-level discussions in a way that most agent surveys do not.

## Limitations
- **No new empirical experiments** — the paper is conceptual; for a 70+ page survey, this is acceptable, but readers looking for ablation evidence or comparative numbers will not find them here.
- **Forward-dated / unverified citations** — a meaningful fraction of the 200+ references are 2025–2026 preprints or model releases; several of the model names in Figure 1 look like they may be projections or placeholders. A careful reader should independently verify the cited works.
- **OpenClaw-centric framing** — the choice of "OpenClaw" as the era's exemplar naturally promotes Tencent's own product; alternative framings (e.g. "Anthropic Agent Skills + Claude Code", "Devin + SWE-agent", "MCP-style workstations") might emphasize different design dimensions.
- **Thin on safety / governance mechanics** — Sections 3.2.2 and 6.1.2 discuss safety and governance, but mostly as a research agenda; there is little concrete protocol, threat model, or empirical evaluation.
- **Heavy on terminology, light on math** — the paper uses many new terms ("Workspace intelligence", "skill-based task closure", "harness engineering", "AI-native workspace", "beyond-gradient learning", "self-evolving ecosystem") but provides few formal definitions; this is appropriate for a position paper but may frustrate readers looking for a precise technical model.
- **No reproducibility artifacts** — no open-source release accompanying the survey, beyond the project page.

## Relevance
- 与LLM代理、自主智能体方向高度相关,提出了"Workspace + Skill"作为agent从episodic tool-use升级到persistent task-hosting的关键抽象,与Claude Code、Devin、OpenHands、Cursor等工业趋势一致;同时其关于"评估从输出正确性到任务闭环"的论述对长程agent benchmark设计(SWE-bench、OSWorld、Terminal-Bench)有直接参考价值。

## Questions for Further Exploration
- Is the "OpenClaw" system open-source? If so, where is the code, architecture documentation, and training data? If not, how should the community evaluate the central exemplar of the paper's thesis?
- How does the **50%-time-horizon** metric actually decompose? The paper presents the growth as exponential, but a log-scale plot and a per-domain breakdown (coding vs web vs OS) would help separate general capability gains from benchmark-specific optimizations.
- Does the paper (or its companion works) provide **empirical head-to-head comparisons** of tool-calling agents vs OpenClaw-style workstation agents on a shared benchmark such as SWE-bench Verified or OSWorld? Without such data, the "Workspace + Skill is the key leap" claim is unfalsified.
- How are the **safety/governance claims** (PRISM, ClawGuard, Agentic Forensics) operationalized? The paper gestures at "defense-in-depth over the agent lifecycle" but does not specify how the defenses compose or how they would be evaluated.
- Several **model names in the timeline** (e.g. "MiniMax M2.7", "Hy3-Preview", "DeepSeek-V4", "GPT-5.5", "Opus 4.6") appear forward-projected; a reader should confirm whether these are real releases, preprints, or aspirational placeholders.
- What is the relationship between the proposed "**skill packages**" and existing software-engineering artifacts (libraries, Docker images, CI pipelines, infrastructure-as-code)? Are skills simply a new name for well-established software abstractions, or do they add something specifically for LLM agents (e.g., natural-language triggers, learned preconditions, LLM-friendly I/O schemas)?
- The **"beyond-gradient learning"** idea (Section 6.2.3) is intriguing but underspecified; what concrete mechanisms prevent the executable environment from accumulating stale or overfitted skills, and what evaluation would distinguish a system that truly "self-evolves" from one that merely has a self-editing code base?
