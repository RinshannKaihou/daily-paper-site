> **一句话总结：** 提出 **EurekAgent**,主张自主科学发现的瓶颈已从"设计智能体工作流"转向"设计智能体运行环境",通过**权限 / 产物 / 预算 / 人在回路**四项环境工程(配合 prepare→propose→implement 三阶段循环 + Claude Code + GLM-5.1),在数学 3 题、TriMul kernel 工程、MLE-Bench Lite 7 题上同时刷新 SOTA,并以**全部 < $17 平均、单题 < $11** 的 API 成本在 26 圆 packing 上发现新记录(2.635999)。
>
> **TL;DR:** This position + system paper argues the **bottleneck for autonomous scientific discovery has shifted from agent workflow design to *agent environment engineering*** — the resources, constraints, and interfaces that shape agent behavior. The authors instantiate this in **EurekAgent**, a thin orchestration layer over an off-the-shelf CLI agent (**Claude Code**) running **GLM-5.1**, with no test-time training. The system runs a three-stage loop `PREPARE → PROPOSE_r → {IMPLEMENT_{r,p}}` (R rounds, P parallel sessions), and engineers the environment along four axes: (1) **Permissions** (Docker sandbox + hidden-evaluator service + same-round isolation + default-deny GPU locking), (2) **Artifacts** (filesystem + Git history as cross-session memory), (3) **Budget** (wall-clock + API-cost limits, with a time-checker helper API and deadline-warning injection), (4) **Human-in-the-loop** (terminal UI for live intervention + web monitor for score evolution and session logs). Empirically, EurekAgent sets **new SOTA on every evaluated task**: 26-circle packing **2.635999** (vs. AlphaEvolve 2.635986, prior best human ≈2.634), Erdős min overlap **0.380870** (vs. 0.380876), 1st-autocorr-ineq **1.502861** (vs. 1.502863), GPUMODE TriMul median **2005.03 µs** (≈4.3% better than the top regraded leaderboard solution, ≈10.8% better than TTT-Discover), and 85.71% any-medal on a 7-task MLE-Bench Lite subset (vs. AIBuildAI 71.43%, Famou-Agent 71.43%). All three math SOTAs are reached with **average API cost < $17** and the 26-circle packing at **< $11** — and notably, EurekAgent beats prior test-time training systems (gpt-oss-120b, R1-Distill-Qwen3-8B) without any model updates, using only environment engineering.

## 核心贡献

- **范式主张:从"工作流设计"转向"环境工程"**。基于"通用 CLI 智能体(Claude Code、Codex)已在 ResearchClawBench 40 题上击败所有专用研究型智能体"这一证据,论文论证自主科学发现的瓶颈已迁移到**运行环境**——Gibson affordances 的视角:好的环境应**放大**有益行为(开放探索、准确奖励、跨智能体协作、便捷的人类监督),**抑制**有害行为(评估器污染、产物篡改、reward hacking)。
- **提出"环境工程"四维框架**(EurekAgent 的核心机制):
  - **权限工程(Permissions)**:Docker 沙箱、隐藏评估器(智能体只能提交、不能看/改)、同轮隔离(parallel implement 互相不可见以避免过早坍缩到单一方向)、GPU 默认拒绝 + 锁所有权(每个物理 GPU 一次只被一个 session 持有)。
  - **产物工程(Artifacts)**:文件系统 + Git 历史作为**跨 session 长期记忆**;准备摘要、提案清单、解法代码、评估器反馈、已评分提交皆持久化;ranked solution history 显式维护;每次 commit 要求描述"独立方案 + 相对上版的变化"。
  - **预算工程(Budget)**:同时管控**墙钟时间 + API 成本**;为 propose / implement 设独立时长;暴露 `time-checker` 工具 API 让智能体主动查询剩余时间;临近截止且未交付时,系统被动注入"停止探索、补齐产物"警告;成本预算耗尽时保留 workspace snapshot;**支持中断恢复**(persist stage id、elapsed、effective budget)。
  - **人在回路(HITL)**:terminal UI(CLI 风格 live 输出 + 输入框直接与会话对话)+ web monitor(分数演化、每轮/全局最佳、完整 session transcript)。
- **系统:EurekAgent**。薄编排层,协调多 session off-the-shelf CLI agent;核心循环 `PREPARE → PROPOSE_r → {IMPLEMENT_{r,p}}`(R 轮,每轮 P 个并行实现),三阶段:prepare(读任务、探评估器、装环境,含糊则问人)→ propose(读历史 + 可选 web 搜索,产出 P 个独立可执行假设)→ implement(并行启动 P 个 session,各持独立 workspace,提交到隐藏评估器,自动 rank)。
- **三域 SOTA 实证**:
  - **数学(3/3)**:Circle packing 2.635999、Erdős 0.380870、Autocorr 1.502861,**全部击败 SOTA 且不使用 test-time training**。
  - **Kernel 工程**:GPUMODE TriMul(本地 A100 复评)best 2005.03 µs median,比 regrade 后的榜首快 ≈4.3%,比 TTT-Discover 快 ≈10.8%;**前 4 名都是 EurekAgent 方案**(4 个解法,中位数 <2031 µs,说明非孤例)。
  - **MLE 工程**:7 题 MLE-Bench Lite 子集 **85.71% any-medal / 71.43% gold / 100% above-median**,any-medal 排名第一,gold 率在非商用开源模型中最高。
- **成本/能力反差**。用开源 GLM-5.1 + Claude Code 编排,**全 3 道数学题平均 API 成本 < $17**,circle packing 单题 < $11;EurekAgent 即便不用最强商用模型,也能在多项 baseline(全部用 Claude-Opus-4.6 / Gemini-2.5-Pro / Gemini-3-Pro)之上拿到最佳 any-medal。

## Key Contributions

- **Position: the bottleneck has shifted to the environment.** Building on the observation that off-the-shelf CLI agents (Claude Code, Codex) outperform research-specific agent systems on ResearchClawBench (40 tasks, 10 domains), the paper argues the limiting factor is no longer the *workflow* but the *environment* (resources, constraints, interfaces). The framing is grounded in Gibson's affordance theory: a good environment amplifies productive behaviors and suppresses harmful ones.
- **Four-axis "environment engineering" framework** — concretely instantiated in EurekAgent as:
  - **Permissions engineering.** Docker sandbox for the run, **hidden evaluator exposed only through a secure grading service** (agents submit and receive official scores but cannot read/modify the evaluator), **same-round isolation** (parallel implement sessions cannot inspect each other, avoiding premature collapse), **default-deny GPU policy** with a GPU-helper API recording lock ownership (each physical GPU held by at most one session).
  - **Artifact engineering.** Filesystem + **Git history** as cross-session long-term memory. Preparation summaries, proposal manifests, hypothesis descriptions, solution code, evaluator feedback, scored submissions are all persisted. **Ranked solution history** is automatically maintained. Commit messages must describe both the standalone solution and the diff vs. the previous version.
  - **Budget engineering.** Joint control of **wall-clock time + API cost**. Separate per-stage time budgets. A `time-checker` helper API lets the agent inspect elapsed / remaining time. **Deadline-warning injection**: when the deadline is approaching and required deliverables are still missing, the system injects a warning asking the agent to stop exploring and emit the artifacts. On cost-budget exhaustion the workspace snapshot is preserved; runs **persist stage id / status / elapsed / effective budget** so an interrupted run resumes from the latest filesystem state under the remaining budget. Users may also grant extra resume time.
  - **Human-in-the-loop engineering.** Two complementary interfaces: a **terminal UI** (CLI-style live session outputs, per-approach progress, bottom input box to chat with active sessions) and a **web monitor** (score evolution chart, per-round and global best approaches, complete session transcripts).
- **System: EurekAgent.** Thin orchestrator over off-the-shelf CLI agent sessions. The loop is `PREPARE → PROPOSE_r → {IMPLEMENT_{r,p}}_{p=1..P_r, r=1..R}`. The environment only handles *outer-loop coordination* (workspace init, stage transitions, objective/deliverable specification, tool/resource interface exposure, scoring + ranking, state persistence, budget enforcement). Within these boundaries, sessions freely pick their own research strategy, experiment plan, and refinement process.
- **Empirical SOTAs across three domains** (single config: Claude Code + GLM-5.1, no test-time training):
  - **Mathematics (3/3 SOTA).** Circle Packing **2.635999** (prev AI 2.635986, prev human ≈2.634, AlphaEvolve-style evaluator with 1e-6 tolerance); Erdős min overlap **0.380870** (prev 0.380876, gpt-oss-120b); 1st autocorr ineq **1.502861** (prev 1.502863, gpt-oss-120b). **All three beat the prior best AI, which was set by test-time training systems, without any model updates.**
  - **Kernel engineering (TriMul, A100 local regrade).** Best median **2005.03 µs** (CUDA Graph), 4/4 top entries are EurekAgent solutions (2005.03 / 2006.99 / 2016.57 / 2030.68 µs), all under 2031 µs. ≈**4.3% better than the strongest regraded leaderboard submission (josusamartin, 2096.04 µs) and ≈10.8% better than TTT-Discover (2247.78 µs)**.
  - **ML engineering (MLE-Bench Lite, 7-task subset).** **85.71% any-medal / 71.43% gold / 100% above-median**, single run per task. Beats AIBuildAI (Claude-Opus-4.6, 71.43%), Famou-Agent (Gemini-2.5-Pro, 71.43%), LoongFlow (Gemini-3-Flash-Preview, 71.43%), and CAIR MARS+ (Gemini-3-Pro-Preview, 71.43% / 71.43% gold). Notably, EurekAgent does this with **GLM-5.1 (open)**, while all listed baselines use **closed commercial models**.
- **Cost-effectiveness as a first-class result.** All three math SOTAs achieved with **average API cost < $17**; the 26-circle packing SOTA at **< $11** total API cost. This is positioned as evidence that environment engineering converts a strong base agent into a cheap, reliable research system.
- **Open-source release** of code, configuration, and discovered results (https://github.com/THU-Team-Eureka/EurekAgent); positioning of "environment engineering" as a community research direction.

## Methodology

- **System loop.**
  - `PREPARE` (once). Reads problem description, evaluator-facing submission spec, optional initial code; **tests the hidden evaluation service**; installs / validates runtime dependencies. May pause for human clarification. Outputs a preparation summary and a completion artifact, both of which become shared context for downstream stages.
  - `PROPOSE_r` (each round). Reads task inputs + preparation summary + ranked best solutions from prior rounds; may inspect prior-round workspaces and use web search/browser. Outputs a manifest of up to **P** candidate hypotheses, each with an implementation-ready description.
  - `IMPLEMENT_{r,p}` (P parallel per round). Each session is assigned a separate workspace and its own hypothesis; iteratively refines, debugs, modifies the solution based on feedback from the **hidden evaluator**. Submissions flow through the secure grading service, which records all evaluated submissions and maintains both intermediate and best-valid results. After parallel sessions finish (or exhaust budgets), EurekAgent **automatically ranks valid submissions** and updates the ranked history.
- **Environment engineering (the four axes).**
  - *Permissions.* **Docker sandbox** for the run; **hidden evaluator outside the agent-visible workspace**, exposed only through a secure grading service that records submissions and returns official scores; controllers own and automatically update authoritative result files with **hooks blocking agent modification**; **same-round isolation** (parallel implement sessions may learn from prior rounds but not from peers); **default-deny GPU** with a GPU-helper API that records lock ownership and ensures each GPU is held by at most one session at a time.
  - *Artifacts.* Filesystem stores stage deliverables (prep summaries, proposal manifests, hypotheses, code, evaluator feedback, scored submissions). System-managed artifacts: web-search history (cache of explored internet info) and official scores (auto-recorded and ranked). All run artifacts persisted under the run directory for traceability, interruption recovery, and resumability. Within each session, **Git commits track solution evolution**; commit messages must describe both the standalone solution and the diff vs. the previous version.
  - *Budget.* Two axes: **wall-clock time + API cost**. Separate per-stage time budgets (proposal shorter, implementation longer). **Active** time-awareness: a `time-checker` helper API the agent can call. **Passive** time-awareness: when a stage's deadline is approaching and required deliverables are still missing, EurekAgent **injects a warning message** asking the agent to stop exploring and emit the artifacts. **API cost is not exposed to the agent**; only tracked by the system. On cost-budget exhaustion the run is aborted and the workspace is preserved as a final snapshot. **Interruption recovery**: stage id, status, elapsed time, effective budget are persisted, so a run resumes from the latest filesystem state under the remaining budget. Users may also revise time limits or grant explicit extra resume time.
  - *HITL.* **Terminal UI** (CLI-style live session view; users can preview all sessions or enter any session and chat via an input box). **Web monitor** (status logs, score evolution chart, per-round / global best, complete session transcripts).
- **Experimental setup.**
  - **CLI agent:** Claude Code. **Base LLM:** **GLM-5.1** (open model, from Zhipu / Z.ai). No test-time training; all gains come from environment engineering.
  - **Tools enabled:** Web Search Prime MCP (search engine) + Playwright MCP (browser).
  - **Hyperparameters (Appendix A).** Circle Packing R=5, P=3, t_propose=20 min, t_implement=120 min; Erdős R=8, P=3, 20/120 min; 1st Autocorr Ineq R=8, P=3, 20/120 min; TriMul R=13, P=3, 20/160 min, A100; MLE-Bench Lite R=12, P=3, 20/100 min, 1 GPU.
- **Tasks.**
  - **Math (3).** (i) **26-circle packing** in the unit square, maximize sum of radii, OpenEvolve-style evaluator with 1e-6 boundary/overlap tolerance; (ii) **Erdős minimum overlap**; (iii) **1st autocorrelation inequality** (nonnegative construction certifying the tightest known upper bound on an autoconvolution constant).
  - **Kernel eng.** **GPUMODE TriMul** (triangular matrix multiplication on A100). Official leaderboard is closed, so the authors **regrade top leaderboard solutions locally** under the same released TTT-Discover TriMul setting; 3 warmup rounds + 10 measured rounds with shuffled order; report median and mean geometric-mean runtimes.
  - **ML eng.** **MLE-Bench Lite** (Chan et al., 2025), 7-competition subset sampled from 22 Lite tasks to balance cost/diversity/difficulty: 2 Easy (histopathologic-cancer-detection 57.0%, plant-pathology-2020-fgvc7 49.7%), 2 Medium (aerial-cactus-identification 26.2%, the-icml-2013-whale-redux 23.5%), 3 Hard (jigsaw-toxic-comment 9.1%, dog-breed-identification 0.4%, tabular-playground-may-2022 0.4%). Single run per competition, 24h / 1 GPU per MLE-Bench's official setting.
- **Baselines.** Math: prior best AI (AlphaEvolve for circle packing, gpt-oss-120b test-time training for Erdős / autocorr). Kernel: top GPUMODE leaderboard solutions regraded locally, plus TTT-Discover. ML: MLE-Bench public leaderboard entries (AIBuildAI Claude-Opus-4.6, Famou-Agent Gemini-2.5-Pro, LoongFlow Gemini-3-Flash-Preview, CAIR MARS+ Gemini-3-Pro-Preview, etc.). For baselines with multiple runs, the authors report the **upper end of the reported range**.

## Results & Highlights

- **Headline SOTA table (Table 1 in the paper).**

  | Domain | Task | Prev. Best Human | Prev. Best AI | EurekAgent |
  |---|---|---|---|---|
  | Math | Circle Packing (↑) | ≈2.634 | 2.635986 (AlphaEvolve) | **2.635999** |
  | Math | Erdős Min Overlap (↓) | 0.380927 | 0.380876 (gpt-oss-120b) | **0.380870** |
  | Math | 1st Autocorr Ineq (↓) | 1.509730 | 1.502863 (gpt-oss-120b) | **1.502861** |
  | Kernel | TriMul (↓) | 2096.04 µs | 2247.78 µs (TTT-Discover) | **2005.03 µs** |
  | ML | MLE-Bench (↑) | N/A | 71.43% (AIBuildAI) | **85.71%** |

- **Circle packing trajectory (Fig. 1).** First valid baseline 1.439; R1 climbs to a 2.632 local basin; **R2 uses web search to discover and adopt the public AlphaEvolve solution (~2.635983), then iterates from it**; R3 crosses the previous SOTA; R4–R5 further improve with relaxed LP, adaptive tolerance, and joint SLSQP. This is itself a vivid demonstration of the artifact-engineering claim: prior public solutions are first-class inspectable context, and the agent can lift from them and continue.
- **Kernel — all 4 top entries are EurekAgent solutions** (Table 3): 2005.03 / 2006.99 / 2016.57 / 2030.68 µs median. Stable high-quality optimization, not a single lucky candidate. ~4.3% over best regraded leaderboard, ~10.8% over TTT-Discover.
- **MLE-Bench Lite (Table 4).** EurekAgent is **first on any-medal (85.71%) and gold (71.43%, tied for first with CAIR MARS+)** and **first on above-median (100%, tied)** — while running on **GLM-5.1 (open)** against baselines that all use closed commercial models.
- **Cost.** All three math SOTAs with **average API cost < $17**; the 26-circle packing SOTA at **< $11** total. The paper positions this as evidence that environment engineering unlocks cheap, reliable discovery with a strong open base model.

## Critical Assessment

- **Novelty.** The *position* — that environment/scaffold engineering dominates results — is a real and useful reframing, and the prior evidence the authors cite (ResearchClawBench, where off-the-shelf CLI agents beat research-specific systems; documented reward-hacking incidents; MLE-STAR/CORAL's task-specific safeguards) supports it. The four-axis taxonomy is actionable. What is *not* novel is the individual mechanism: Docker sandboxes, Git-tracked artifacts, budget-controlled execution, and HITL UIs are all widely used. The contribution is the **integration and the framing**, not the components.
- **Rigor.**
  - **Math baselines are honest and strong.** The comparison is against AlphaEvolve on circle packing and against test-time-training systems (gpt-oss-120b, R1-Distill-Qwen3-8B) on Erdős and autocorr — current SOTA — under the same evaluator tolerance. EurekAgent is explicitly *training-free*, so beating TTT systems on the same metrics is the right comparison.
  - **Kernel baseline is *partially* fair and clearly disclosed.** The official GPUMODE leaderboard is closed, so the authors regrade top leaderboard scripts **locally** under the same TTT-Discover TriMul setting, with 3 warmup + 10 measured rounds, shuffled order, and unchanged correctness tests, benchmark cases, scoring rule, and timing logic. This is the cleanest reproducible comparison one can do given the constraints, and the paper is explicit about it. Caveat: regrade on a single A100 is sensitive to driver / CUDA / clock state, and absolute µs numbers are not interchangeable with the official leaderboard. The ~4.3% margin over the top regraded solution is comfortable enough that clock noise is unlikely to flip the ordering, but a multi-GPU / multi-seed reproduction would be the gold standard.
  - **MLE-Bench baseline is the *upper end of the reported range*** for systems with multiple runs. This is generous to the baselines, and EurekAgent still wins on any-medal, so it does not change the conclusion — but a same-budget, same-seed, multiple-run EurekAgent comparison would be a much stronger test.
  - **No multi-seed / no statistical tests** on the headline numbers. The kernel results (4/4 top entries being EurekAgent) and the MLE-Bench any-medal (6/7 vs. 5/7) are reasonably robust, but the math SOTAs are improvements in the **4th–5th decimal place** (e.g. 2.635999 vs. 2.635986, a delta of 1.3e-5; 0.380870 vs. 0.380876, a delta of 6e-6). These are real but razor-thin, and one wonders how much they are subject to evaluator tolerance / numerical noise.
  - **No ablation of the four axes.** The paper does not present a "remove permissions engineering" / "remove artifact engineering" / etc. decomposition. Without ablations, the "environment engineering is all you need" claim is supported by *correlated* evidence (the system works) but not by *causal* evidence (each axis is necessary). This is the **single biggest weakness** of the paper.
  - **Single LLM backbone.** Every experiment uses Claude Code + GLM-5.1. The headline claim that "environment engineering, not model strength" is what matters is therefore not directly tested — the strongest closed baselines (Claude-Opus-4.6, Gemini-3-Pro-Preview) lose to EurekAgent-on-GLM-5.1 on MLE-Bench, but that is one data point per baseline and the task subset is small (n=7).
  - **Cherry-picking risk in MLE-Bench subset.** The 7-task subset is hand-curated by the authors from 22 Lite tasks "to balance cost, diversity, and difficulty." Selection bias is real; on a different 7-task subset, the rank order could change. The standard MLE-Bench protocol is to report results on a fixed split; using a custom subset should be flagged more loudly.
  - **"$11" headline is a single data point.** The cost number depends on the model, the per-token price, the trajectory, and the task. It is impressive, but it is a single run; without a cost-vs-SOTA frontier curve, "cheap" is not yet a property of the system so much as a property of *this* task + *this* model.
- **Reproducibility.** Promising (open-source code + artifacts), but in this class of paper, reproducibility hinges on (i) the exact environment manifests and Docker images being released, (ii) the discovered artifacts (e.g. the 26-circle configuration + independent verifier) being externally re-checked, and (iii) wall-clock / per-task cost / per-stage token usage being reported alongside the headline SOTA.
- **Significance.** If the claims hold, this is a meaningful **vocabulary contribution** to the AI-scientist literature — naming the env-design work that everyone does implicitly is genuinely useful. The fact that the system works with an open model is a real counter-narrative to "you need GPT-5 / Claude-Opus to do agentic science." However, "is the env all you need" is rhetoric; in practice the system still depends on a strong base LLM and on a non-trivial amount of human engineering of the four axes, which the paper under-counts.
- **Red flags to watch.**
  - No ablations on the four axes.
  - Math SOTAs are razor-thin 4th-5th-decimal improvements.
  - MLE-Bench is on a hand-curated 7-task subset, not the full Lite split.
  - Cost is a single number; no cost-vs-quality frontier.
  - All experiments with a single model backbone.
  - The same hidden evaluator that defines the task is the only signal — no held-out verification of the "new SOTA" 26-circle configuration is reported.

## Strengths

- **Clear, defensible position** grounded in cited evidence (ResearchClawBench, Anthropic system cards, documented reward-hacking incidents) — names the implicit work that "AI scientist" papers do without naming it.
- **Actionable four-axis taxonomy** that practitioners can copy: permissions, artifacts, budget, HITL.
- **Breadth of empirical claims** (3 math + 1 kernel + 7 ML tasks) — unusual in this subfield, and all in one paper with one config.
- **Concrete mechanisms for each axis** — e.g. "default-deny GPU + lock ownership", "Git commit messages must describe both the standalone solution and the diff", "deadline-warning injection when deliverables are missing", "ranked solution history as cross-session memory".
- **Honest disclosure of baseline caveats** — explicit that the kernel leaderboard is closed, so solutions are regraded locally; explicit that for multi-run MLE-Bench baselines they report the upper end of the range.
- **The circle-packing trajectory (Fig. 1)** is a memorable, falsifiable micro-narrative: R2 web-searches the public AlphaEvolve solution, adopts it as a stepping stone, and EurekAgent then *pushes past* it. This is exactly the "environment as substrate" story the paper tells.
- **Open-source release** of code and discovered results.
- **Cost evidence** that an open-source base model can compete with closed-model SOTA on agentic science, with single-digit-dollar API budgets.

## Limitations

- **No ablations of the four axes.** The paper does not quantify the marginal contribution of permissions / artifacts / budget / HITL. Without this, the "is all you need" framing is overstated.
- **Single LLM backbone (GLM-5.1).** The "environment > model" claim is not directly tested against a stronger closed backbone running through the *same* EurekAgent harness.
- **Math SOTAs are razor-thin** (4th–5th-decimal-place improvements) and lack statistical reporting. Are they robust to evaluator tolerance, to multiple seeds, to slight changes in initial code?
- **MLE-Bench on a hand-curated 7-task subset** rather than the full Lite split; selection bias is real.
- **No cost-vs-SOTA frontier.** The "<$11" headline is a single number; the right metric is "for a given budget, where does the system land on the cost-vs-SOTA curve?" — and that curve is not reported.
- **Kernel evaluation is a local regrade** on a single A100; µs numbers are not directly comparable to the (now-closed) official leaderboard. The ordering is likely robust, but absolute numbers are not.
- **"Environment engineering is all you need"** is rhetoric. The system still depends on a strong base LLM (Claude Code + GLM-5.1), on a non-trivial amount of human engineering of the four axes, and on a hidden evaluator that defines the task. The paper under-credits the model and the human work.
- **Failure modes are not quantified.** The paper gestures at "evaluation tampering, artifact manipulation, reward hacking" as things the environment suppresses but does not report how often each residual risk remains, nor how the system degrades under adversarial prompts.
- **Closed evaluator = closed ground truth.** The hidden evaluator is the sole feedback signal; an external held-out verification of the "new SOTA" 26-circle configuration (or the kernel solutions) is not reported.
- **The four-axis taxonomy is system-specific.** Permissions engineering matters most when there is a hidden evaluator and untrusted code; artifact engineering matters most when solutions are code; budget engineering matters most when API costs are non-trivial. The framework's portability to non-code / non-metric-driven research is asserted but not demonstrated.

## Relevance

直接命中你"**Self-Evolving Agents & Automated Research**"兴趣主线:本篇把"环境工程"作为自主科学发现范式的**核心轴**提出,并用一个完整系统(Clore Code + GLM-5.1,无 test-time training)在数学 / kernel / MLE 三域同时刷新 SOTA,代表 AI 研究员方向从"工作流设计"向"环境 + 基础设施设计"的范式转移。对关注 autonomous research loop、agentic scientific discovery、self-modifying architectures 的读者是关键文献,建议与 The AI Scientist、AlphaEvolve、AIDE、Karpathy autoresearch、ResearchClawBench 对照阅读,理解当前自主发现系统在"环境 vs 工作流 vs 模型"三角上的最新分工。

## Questions for Further Exploration

- **Ablations.** What is the marginal contribution of each of the four axes (permissions / artifacts / budget / HITL)? E.g. remove the same-round isolation, remove the ranked-solution-history, remove the deadline-warning injection, remove the GPU lock — which of these is doing the work?
- **Cost-vs-SOTA frontier.** For a fixed SOTA delta, what is the minimum API cost? For a fixed budget, what is the achievable SOTA? Without a curve, "$11" is a single point.
- **Model substitution.** Run EurekAgent with Claude-Opus-4.6, Gemini-3-Pro, and a smaller open model (e.g. Qwen3-8B) as the base. Does the env-engineering advantage **persist**, **shrink**, or **invert** as the base model changes? The "environment > model" claim is the load-bearing one for the paper and is not yet directly tested.
- **Statistical robustness of math SOTAs.** Re-run circle packing, Erdős, and autocorr with 3–5 seeds for both EurekAgent and the strongest baseline (AlphaEvolve / gpt-oss-120b-TTT). Are 4th-5th-decimal improvements robust to evaluator tolerance and to seed?
- **Independent verification of the 26-circle SOTA.** Submit the discovered configuration to an independent geometric/optimization verifier (not the same hidden evaluator the agent optimizes against). Does the SOTA hold, and is it consistent with the public AlphaEvolve solution (which the agent explicitly adopted as a starting point)?
- **Kernel reproduction.** Re-grade the top 4 EurekAgent kernel solutions and the strongest leaderboard solution on **multiple A100 instances and CUDA versions**. Are the ~4.3% and ~10.8% margins robust?
- **MLE-Bench generalization.** Report results on the **full 22-task MLE-Bench Lite split** (not the hand-curated 7), ideally with 3 seeds per competition for both EurekAgent and the strongest baseline (AIBuildAI). Does the rank ordering hold?
- **Failure-mode surface.** How often do reward hacking, evaluator leakage, score tampering, and same-round copying still occur in EurekAgent runs? What are the residual risks, and how would an adversarial user exploit them?
- **Closed-evaluator vs. open-evaluator generalization.** All reported tasks have a hidden / executable evaluator. How does EurekAgent degrade when the metric is **noisy, delayed, multi-faceted, or human-judged** (e.g. paper-writing quality, hypothesis novelty)?
- **Environment × self-improvement.** Does environment engineering compose with **agent self-modification** (e.g. an agent rewriting its own tool surface, or proposing a new affordance to the controller)? Is that part of the framework or out of scope?
- **Portability to non-code / non-metric research.** Can the four-axis framework be ported to **wet-lab protocols, embodied robotics, or qualitative analysis**, where the "evaluator" is not a script and the "artifacts" are not Git-trackable code? The current paper is silent on this.
