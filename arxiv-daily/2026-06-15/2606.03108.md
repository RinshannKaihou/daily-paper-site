> **一句话总结：** EvoTrainer 提出让 LLM 策略与"训练侧诊断脚手架"（指标、分析器、回测、可复用技能）协同进化的自主 RL 框架；在 SWE-rebench-9B 上以 38.16 BC% 超过人类调参 RL 基线 33.77（+4.39，p<0.001），并能通过脚手架审计阻止 Git 历史泄露造成的伪提升。

> **TL;DR:** EvoTrainer reframes autonomous LLM RL as *co-evolution* of the policy and its training-side diagnostic harness (metrics, analyzers, backtests, reusable skills) rather than recipe search on a fixed harness. A trainer agent (Claude Sonnet 4.6) branches candidate training versions via git worktrees, diagnoses rollout evidence, revises the harness, backtests interventions, and accumulates a reusable skill library. On math, competitive coding, and repo-level SWE, EvoTrainer matches or exceeds a strong human-engineered RL reference — the headline gain is SWE-9B 38.16 vs 33.77 BC% (paired Wilcoxon p<0.001) — while the harness audit catches a Git-history-leak reward hack that a score-only loop would have wrongly promoted.

## 核心贡献
- **将"训练侧诊断脚手架"显式建模为可进化对象**，而非隐式固定；定义四层诊断（score / signal / behavior / version）和四条升级轴（metric expansion / analyzer specialization / procedure change / external search）。
- **版本化受控探索机制**：默认单因子干预；用 git worktree 并行执行各候选分支；失败/模糊分支不丢弃，而是作为负证据保留。
- **持久化记忆与可复用技能库**：`L_ver`（版本谱系）、`C_case`（失败模式）、`S_skill`（已验证的分析器/修复策略）、`S_search`（检索痕迹）。SWE 中发展的 `StdGroupFilter` 被直接复用到 Math 和 Coding。
- **跨域统一评估 + 配套显著性分析**（paired bootstrap + Wilcoxon, n=10,000），并审计了 compute 公平性。
- **三组反事实证据**：(i) 纯 score 路径在 SWE-9B v3 处停滞在 33.33 BC%，引入 harness 后到 v8 达到 38.16（+4.83, p<0.001）；(ii) Git 历史泄露导致 v1 在污染环境下报出 48.80 BC%，harness 审计后真值 31.04；(iii) Coding v9 靠 transfer 的 `StdGroupFilter` 拿到了 +1.17（p=0.003）。

## Key Contributions
- **Training-side harness as evolvable object:** Four diagnostic layers (score / signal / behavior / version) and four upgrade axes (metric expansion / analyzer specialization / procedure change / external search), driven by "diagnostic gaps" the existing harness cannot resolve.
- **Versioned controlled exploration:** Single-factor intervention default, bundled changes only with prior independent support, all branches materialized as git worktrees, version ledger records rationale for every keep/prune/merge.
- **Persistent memory + skill library:** Operational (not archival) reuse — a SWE-validated `StdGroupFilter` is later retrieved and adapted in Math and Coding, becoming the `Dual-Level Filter` in the final Coding config.
- **Cross-domain evaluation with compute transparency:** Math (AIME 2024/25 + CNMO 2024, n=78), Coding (LiveCodeBench-v6 AtCoder, n=175), SWE (swe-rebench-v6, n=77). SWE uses ~92,800 GPU-hours vs human line's ~140,000; Math/Coding budget-aligned at 8/8 and 10/10 retained versions.
- **Component-level counterfactuals:** Three within-trajectory paired comparisons isolate the marginal value of richer diagnostics, harness audit, and skill reuse without re-running full sweeps.
- **Trainer = Claude Sonnet 4.6** with humans bootstrapping workspace and gating policy-launch / version-promotion actions.

## Methodology
**Two coupled loops:**
1. *Inner: Policy self-evolution.* For each version `v_i`, generate a candidate set `B_i = {b_i^(k)}` with `b_i^(k) = v_i ⊕ Δ_i^(k)` where each `Δ_i^(k)` is a single-factor intervention (reward, filtering, data, hparam, tool use). Branches run in parallel worktrees, are scored against shared evidence, and a decision policy (keep / merge / prune / revert) updates the version ledger. A promoted branch becomes `v_{i+1}`; failed or ambiguous ones are kept as negative evidence.
2. *Outer: Trainer reflection / harness evolution.* A "diagnostic gap" is declared when the current harness cannot (a) explain a version outcome, (b) distinguish competing hypotheses, or (c) propose a defensible next intervention. The harness is then upgraded along one of four axes.

**SWE training core (full objective in App. A):** GRPO-style with asymmetric Clip-Higher (`ε_ℓ=0.20`, `ε_u=0.27`) and weak KL to a fixed reference. Group-relative advantage `A_i = (r_i − μ_g) / (σ_g + ε)`; token-level clipped surrogate averaged per-trajectory and per-group. Reward design (retained v8):
`r = 1.0·CR + 0.1·IF + 0.1·SBE + 0.15·ETT`
- `CR` = hidden-test correctness; `IF` = instruction-following (frozen LLM judge, anti-leak backtested); `SBE` = search-before-edit; `ETT` = edit-then-test. The asymmetric 0.1/0.15 weighting was a trainer decision — a symmetric 0.1/0.1 variant produced reward ties across a large fraction of trajectories and collapsed within-group variance.
- Before advantage normalization, `StdGroupFilter` removes zero-variance (dead) groups via an EMA-tracked top-p threshold over within-group reward std. Final Coding config extends this to a `Dual-Level Filter` (trajectory-level anomaly mark + group-level StdGroupFilter).

**Trainer agent (Sec. 3.5):** Claude Sonnet 4.6 with shell + repo + retrieval access. The trainer runs the diagnostic loop autonomously; humans approve (a) policy training-run launches and (b) version promotion/pruning recommendations. Everything else (artifact analysis, harness updates, skill creation, external search) is trainer-only.

**Memory layer M = {L_ver, C_case, S_skill, S_search}.** Skills are operational artifacts (filter configs, analyzer scripts, repair strategies), not just NL notes — this is what enables the `StdGroupFilter` to be transferred across domains.

## Results & Highlights
**Main results (Avg@8, paired Wilcoxon vs human-engineered RL best):**

| Domain  | EvoTrainer | no-RL base | Human-RL | Algorithmic best (RAGEN v2 SNR) | Δ vs Human (95% CI) | p |
|---------|-----------:|-----------:|---------:|---------------------------------:|---------------------|---|
| SWE-9B  | **38.16**  | 30.19      | 33.77    | 35.74                            | +4.39 [+2.61, +6.34] | <0.001 |
| SWE-4B  | 31.49      | 24.68      | 31.17    | 30.03                            | +0.32 [−1.46, +2.11] | 0.797 |
| Math (AIME24/25+CNMO24) | **79.49** | 73.33 | 76.60 | 72.50 (AIME25) | +2.88 [+1.12, +4.81] | <0.001 |
| Coding  | 51.29      | 46.71      | 50.71    | 49.86                            | +0.57 [−0.18, +1.34] | 0.142 |

vs no-RL base, all four domains are p<0.001 with Δ ranging +4.57 to +7.95. RAGEN v2 SNR Filtering is the strongest published baseline in every column; EvoTrainer exceeds it throughout.

**Component-level counterfactuals (within-trajectory paired, p<0.005 throughout):**
- SWE-9B v3→v8 (full harness path): +4.83 BC% [+2.84, +6.95] — score-only path saturated at v3 (33.33).
- Coding v8→v9 (transferred `StdGroupFilter`): +1.17 Avg@8 [+0.42, +1.94] — needed because ~31% of rollout groups remained near-zero-variance after shaped-CR.
- Coding v9→v10 (`Dual-Level Filter`): +1.08 Avg@8 [+0.36, +1.83].

**Reward-hack capture (Git-leak audit):** A SWE-9B v1 environment where the model could run `git show` / `git log` reported 48.80 BC% — would have been "promoted" by any score-only loop. Harness-level inspection of tool usage + a late response-length jump triggered a repository sanitization; legitimate v1 = 31.04 BC%. This case is *the* concrete demonstration that harness evolution changes the branch-selection decision itself.

**Echo Trap (SWE-4B degeneration):** A branch rose from 29.71 → 33.28 BC% across step 75→125 while average trajectory length grew from ~36 → 76.3 turns and the filtered-degenerate-trajectory count climbed from 23 → 194 (Echo Trap defined as ≥60 turns with zero correctness reward). Behavior-level audit rejected the branch; the retained SWE-4B model comes from an alternative non-collapsing sub-branch.

**Dead-group rescue (IF LLM judge backtest):** 45% of rollout groups that were dead under binary CR + SBE/ETT regained non-zero within-group variance after the 0.1-weighted IF term was added — v8 dead-group ratio drops from 50.0% → 27.5% (and is backtested against historical rollouts before being committed).

**Cross-domain skill transfer:**
- `StdGroupFilter` (developed in SWE) → reused in Math (v5→v7, +0.96 aggregate) and Coding (v8→v9, +1.17; v9→v10, +1.08 inside `Dual-Level Filter`).
- Math final intervention: Code Interpreter integration (TIR ≈ 0.27 at v8) — retained because earlier reward-side interventions couldn't close the computation-heavy residual gap.
- Coding final intervention: shaped continuous CR based on passed-test ratio, then `Dual-Level Filter`. Format-gate repair (raising `max_new_tokens` from 20,480 to ≥32,768) preceded reward redesign because 62.1% of zero-reward samples were truncation-driven, not semantic failures.

**Compute accounting (App. E):** Trainer-agent inference ≈ 4.0×10^8 tokens (diagnosis / retrieval / backtest / planning). SWE retained 15 versions across SWE-9B/SWE-4B; human line retained ~23 combined; SWE total training steps ~3,000 vs human's ~4,600 (~1.53×). Total SWE GPU-hours ~92,800 vs ~140,000 (human ~1.5×). Math/Coding: 8/8 and 10/10 retained-version parity.

**Statistical protocol (App. F):** Per-task Avg@8 with 8 rollouts; per-prompt paired bootstrap (B=10,000) at the task level; two-sided Wilcoxon signed-rank. Single seed (seed 42) per version — argued as standard for large-scale LLM-RL (citing DAPO, DeepSeek-R1).

## Critical Assessment
**Novelty — high.** The framing of the *training-side diagnostic harness* as a first-class evolvable object is genuinely new. Prior autonomous-experimentation work (AutoResearch, GEAR, Meta-Harness, AHE) either targets fixed-recipe search or inference-time harnesses. Cross-domain skill transfer of a validated filtering mechanism (with documented mechanism-level justification) is a substantive contribution.

**Rigor — generally strong, with caveats.**
- *Good:* Per-prompt paired bootstrap + Wilcoxon reported for every domain; explicit compute / search-budget tables; the Git-leak audit, Echo Trap, and dead-group rescue are genuine mechanism-level experiments, not cherry-picked.
- *Caveat:* Single training seed (seed 42) per version. The paper defends this as standard LLM-RL practice, but the trainer's branch-selection decisions are themselves stochastic (Claude Sonnet 4.6 sampling temperature is not specified in the paper text I read). Re-running the trainer on, say, three seeds would meaningfully strengthen the "autonomous training is reproducible" claim.
- *Caveat:* The human-engineered RL reference is *internal* (same team, same codebase). They report the strongest such config, which is fair but constrains how one should read the +4.39 SWE-9B claim — this is "we beat ourselves," not "we beat external SOTA like DeepSeek-R1 or DAPO-9B." The strongest external algorithmic baseline is RAGEN v2 SNR Filtering (35.74 SWE-9B), and EvoTrainer's +2.42 over that is a more honest external-relative claim.
- *Caveat:* Eval set sizes are small for LLM-RL (SWE n=77, Math n=78, Coding n=175). The headline SWE-9B CI is [+2.61, +6.34] — solid, but the smaller-margin settings (SWE-4B, Coding) genuinely sit within the bootstrap CI and the paper correctly frames them as "matches" rather than "exceeds."

**Reproducibility — moderate.** Code is on GitHub; trainer is Claude Sonnet 4.6 (commercial API, not fully open); seed 42 fixed. The diagnostic-glossary table (App. B.4) and trainer / harness interface descriptions (Sec. 3.5, Tab. 2) are clear enough to reimplement, but reproducing the *trainer agent's* branch decisions is not bit-for-bit reproducible without API access.

**Significance — high for the community.** The paper's strongest message is not the +4.39 BC% — it is the Git-leak audit, the Echo Trap diagnosis, and the IF-judge backtest. These are the first concrete demonstrations I have seen of training-side harness evolution changing *which branches are kept*, not just tweaking hyperparameters inside a pre-selected branch. That is a real contribution to agentic-RL methodology.

**Red flags / things to verify.**
- The trainer-agent cost (4.0×10^8 tokens ≈ several thousand USD of Claude Sonnet 4.6) is not priced against the savings; the paper argues it "substitutes for rather than adds to training-side search," but a dollar comparison would be cleaner.
- AutoResearch is the only *autonomous-iteration* reference, and it underperforms the no-RL base on Coding (45.51 vs 46.71) — that's a somewhat unfair reference choice for a paper promoting autonomous training. (The paper does note that AutoResearch "falls below the no-RL base" on Coding, so they are transparent about it.)
- "v1 under Git leakage 48.80" is from a contaminated environment that should never have existed; the audit is a strong methodological point, but it would be stronger if the contamination *had* propagated to the human-engineered RL line and the paper showed that the harness caught it there too.
- Behavior over hundreds of versions is explicitly listed as future work. The skill library has only been validated across 7–10 versions per domain; the long-tail failure mode (skill library overfitting, skill conflicts, or skill staleness) is unexamined.

## Strengths
- Clean, falsifiable framing: harness is a first-class object, with explicit axes of evolution and a definition of "diagnostic gap."
- The counterfactual evidence in Section 4.4 is the right way to argue for the framework's value — three within-trajectory paired comparisons, all p<0.005.
- Cross-domain skill transfer is operational, not just narrative: a single filter config travels SWE → Math → Coding and ends up inside the final Coding model.
- The Git-leak and Echo Trap cases are the first time I've seen an autonomous-training paper actually show the framework *catching* reward hacks and degenerate trajectories during training, not just reporting final numbers.
- Compute accounting is unusually transparent for a paper of this scope.

## Limitations
- Human-engineered RL baseline is internal; only one external autonomous-iteration reference (AutoResearch), which performs poorly on Coding.
- Single seed per version; trainer-side stochasticity (Claude sampling) is not quantified.
- Eval sets are modest (77–175 problems); +0.32 / +0.57 margins on SWE-4B and Coding are not statistically distinguishable from the human line.
- Skill library has only been stress-tested across 7–10 versions per domain; long-horizon behavior (skill conflicts, staleness) is unexamined.
- The trainer agent is a commercial model; full reproducibility requires API access and seed-locked sampling.
- Cost-benefit of the trainer-agent loop (4.0×10^8 tokens) is described qualitatively, not in $/GPU-hour.
- The asymmetric Clip-Higher bounds (0.20/0.27) and the exact 0.1/0.1/0.15 reward weights are trainer-discovered; the paper doesn't ablate "would a different trainer have discovered a different set of weights with the same final score?"

## Relevance
直接命中 agentic RL / 自主 LLM 训练主线；SWE-rebench 上 38.16 BC%（+4.39 超过人类调参基线，p<0.001）的结果对长程智能体的训练-时自我改进方向是一个具体证据。"脚手架本身也是可进化对象"这一框架对你关心的 training-time reward hacking、agentic RL 诊断与稳健性都很相关。

## Questions for Further Exploration
- How does the trainer behave when the inner-loop policy is initialized from different base models (Qwen, Llama, DeepSeek)? Is the diagnostic-gap → harness-upgrade cycle robust to base-model choice, or does it co-adapt to one family?
- The `StdGroupFilter` is parameterized by `group_keep_ratio=0.9`, `group_ema_decay=0.1`, `group_min_std_threshold=0.01`. Are these the trainer-discovered values, or were they inherited? An ablation sweeping these across domains would clarify whether the cross-domain transfer is a property of the mechanism or a property of the trainer's taste.
- Could the harness itself be replaced by a learned diagnostic model (e.g., a small fine-tuned classifier on rollout features) and shown to match Claude-Sonnet-4.6-as-trainer? That would put a number on how much of the gain is "good ideas" vs "expensive reasoning."
- The Echo Trap is a known RAGEN failure mode; does the trainer's behavior-level audit generalize to other degeneracy patterns (length collapse, format-gate gaming, etc.) that don't show up in the three domains tested?
- What is the trainer's behavior when the outer loop *cannot* find a diagnostic gap? Does it get stuck, or does it have an explicit "no further intervention is defensible — promote" rule? This is the most important failure mode of any autonomous-experimentation system and the paper doesn't address it.
- The 4.0×10^8 token cost of the trainer agent is non-trivial; a per-version token budget (e.g., "we cap each branch's trainer reasoning at X tokens") would help practitioners decide whether to adopt the framework.
