> **一句话总结：** CARE 是一个面向高通量科学实验的可审计控制器，把 LLM 限定为"挑战者策略"的提议者（用 Python 排序函数表达），由一个仅依赖公开证据的"干预门"决定是否让其覆盖传统 BO 现任动作；它在 Minerva/Olympus（80.0→88.5）和 ChemLex（83.9→92.1）两个匹配 30 种子基准上 SOTA，且 LLM 后端可换为 Gemini-2.5-Flash 不影响结论。

> **TL;DR:** CARE is a controller for finite-pool high-throughput experimentation (HTE) replay that keeps a classical BO/heuristic *public incumbent* as the default action path and uses an LLM only to *propose* (or patch) executable Challenger ranking policies. Before any reveal, a **Public-Evidence Intervention Gate** computes a gain−risk authorization score κ_t from pre-reveal public evidence (expert ranks, novelty, categorical support, etc.) and either authorizes the challenger or falls back to the incumbent; every decision is written to a pre-reveal audit log. Across 30 matched seeds with 5 initial observations and 10 reveal rounds, CARE beats the public incumbent, GP-EI, GP-UCB, BoTorch, TPE, BayBE/BoFire-style mixed BO, Gryffin, EDBO-GP-EI, an LMABO-style LLM-BO, and both gated/ungated LLM baselines on final-best, AUC, and simple regret, with the gate authorizing only 52/300 (Minerva) and 36/300 (ChemLex) challenger actions. Ablations show the frontier proposal mode is the dominant contributor on Minerva/Olympus, the recovery proposal mode on ChemLex, and that simply enlarging the public incumbent to 6 experts does *not* close the gap. The takeaway for NLP/agent evaluation: treat LLM self-evolution as *proposal generation under control*, not as direct action selection.

## 核心贡献

- 将"LLM 控制科学实验"重新建模为**公开信息边界（public-information boundary）下的有限池序贯交互**——agent 只能看到已揭示结果、公开候选特征和自身日志，杜绝 oracle 泄漏。
- 设计 **CARE 控制器**：默认动作路径由"公开现任控制器"（一个非 LLM 的 BO/启发式专家融合）给出；LLM 通过 Policy Planner 决策 `reuse / patch / create` 三个动作，**只生成/修补可执行的 Python `rank_candidates` 排序策略**，不直接选实验。
- 提出 **Public-Evidence Intervention Gate**：在揭示结果前用 `κ_t(x) = αᵀ g_t(x) − βᵀ q_t(x)`（公开增益 − 公开风险）授权；授权门槛为 `κ_t ≥ 0.0`（干预余量）。所有决策写入**揭示前审计日志**。
- 两种**有界提议模式**：recovery（预算 2/轮，弱轨迹时启用）+ frontier（预算 1/轮，仅当 schema 暴露决策可见特征时启用），均仅生成提案，由门决定是否实际执行。
- 在 Minerva/Olympus（5670 候选，Suzuki 偶联）和 ChemLex（11088 候选，酸-胺湿实验）上**两个基准、30 匹配种子**都达到最终最佳值与 best-so-far AUC SOTA，并报告了完整消融（w/o Planner / w/o Gate / w/o recovery / w/o frontier / 6-expert）、后端可替换性（GPT-5.4 → Gemini-2.5-Flash）、揭示协议扫描与干预余量敏感度。

## Key Contributions

- Formalizes LLM-in-the-loop scientific experimentation as a **finite-pool sequential environment interaction** under a strict public-information boundary: only revealed outcomes, public candidate features, and logged optimizer state are admissible.
- Designs the **CARE controller** with three clear roles: (1) a non-LLM *public incumbent controller* (ensemble of public experts + public prior/novelty/categorical terms) produces the default action; (2) a *Policy Planner* decides whether the LLM should reuse/patch/create an executable Challenger Policy; (3) a *Public-Evidence Intervention Gate* grants or withholds action authority using pre-reveal evidence alone.
- Introduces two **bounded proposal modes** (recovery, budget 2/replay; frontier, budget 1/replay) that expand the proposal space without letting the LLM directly choose experiments.
- Provides a **per-round audit ledger** (public interface, reference action, challenger artifact, pre-reveal audit record, post-reveal outcome) that decouples proposal generation from action authority.
- Empirically validates the design on two real HTE replay suites under matched seeds: Minerva/Olympus final-best 80.0 → 88.5 (regret 9.2 → 0.7), ChemLex 83.9 → 92.1 (regret 16.1 → 7.9), with 95% bootstrap CIs and paired seed-level win/tie/loss counts against every control.
- Reports full ablations, backend swap to Gemini-2.5-Flash, replay-protocol robustness (4 settings varying init/budget), and intervention-margin sensitivity.

## Methodology

### Problem setting — finite-pool replay
A pool of N candidates `X = {x_i}` with hidden objectives `{y_i}`. At round t the controller sees the revealed index set `O_t` and history `H_t = {(x_i, y_i) : i ∈ O_t}` plus unrevealed pool `C_t = {x_i : i ∉ O_t}`, picks `a_t ∈ C_t`, then learns `y_{a_t}`. Given a reveal budget B, the policy is scored by **final-best** `f_B★ = max_{1≤t≤B} y_{a_t}` and **best-so-far AUC** `AUC_B = (1/B) Σ_{t=1}^B max_{1≤s≤t} y_{a_s}`.

### Public-information boundary
Every policy (LLM or not) acts only on revealed outcomes, public candidate features, and logged optimizer state. Unrevealed outcomes, oracle ranks, evaluator-private artifacts, and post hoc information are off-limits. A policy *written* by the LLM only gets deployed after passing the same boundary and audit checks applied to the non-LLM baselines (full prompt interface in Appendix F; validation in §4.2/§B.4).

### Public Incumbent Controller (§4.1)
An ensemble of public experts `E^pub_t` with rank-fusion weight `ρ(r) = r^(−1/2)` for finite ranks and `ρ(∞) = 0`:
`S^pub_t(x) = Σ_{e∈E^pub_t} w_{t,e} · ρ(r_{t,e}(x)) + λ_p P_t(x) + λ_n N_t(x) + λ_d D_t(x) + λ_q Q_t(x)`,
where each term (public prior, novelty/density, categorical evidence) is a public-state function. The reference action `a^ref_t = argmax_{x∈C_t} S^pub_t(x)` is computed before any LLM intervention; this is the **audit reference** for the gate.

### Policy Planner + LLM Policy Generator (§4.2)
The planner sees a public trajectory representation `z_t = Φ(H_t, s_t, ℓ_{t−1})` (recent improvement, stagnation, poor-reveal events, prior gate reports) and picks `u_t = Σ(z_t) ∈ {reuse, patch, create}`. This is a **planner action on the Challenger Policy artifact only** — it never selects the environment action. The Challenger Policy is an executable `rank_candidates(observed_df, candidate_df, memory, tool_state)` Python function producing a full-pool score `S^ch_t(x) = π_θ^ch(H_t, C_t, s_t) x` with deterministic `candidate_id` tie-breaks, row-order invariance, no private attrs, no network/file/eval/exec.

### Challenger Candidate with public rationale (§4.3)
`a^ch_t = Ψ_t(S^pub_t, S^ch_t, M_t; H_t, C_t)`, where `M_t` records public ranks, expert agreement, categorical evidence, novelty, and schema-conditioned public-feature evidence. Two bounded proposal modes sit on top:
- **Recovery mode** (budget 2/replay): under weak trajectory signals, scan the top challenger-ranked candidates that *also* carry public support.
- **Frontier mode** (budget 1/replay): only when the public schema exposes a decision-visible feature view; schema-conditioned alternative exploration.

Both modes output *proposals*; the gate still assigns action authority.

### Public-Evidence Intervention Gate (§4.4)
For each candidate x, the gate assembles a public gain vector `g_t(x)` (public margins, support, rank bonuses, novelty, categorical evidence, schema-conditioned evidence) and a public risk vector `q_t(x)` (missing public rank, low support, challenger-only support, deviation from reference, negative public margins). The authorization score is
`κ_t(x) = αᵀ g_t(x) − βᵀ q_t(x)`,
and the selected action is
`a_t = a^ch_t  if γ_t = 1 (gate authorizes, κ_t(a^ch_t) ≥ intervention margin),`
`a_t = a^ref_t if γ_t = 0 (incumbent fallback).`
The fixed instantiation uses **intervention margin = 0.0**; all gate weights, planner thresholds, challenger budgets, the prompt interface, and the LLM sampling temperature are frozen before matched replay and reused across both datasets.

### Audit ledger
Per-round entries are written *before* outcome reveal: (1) public interface, (2) public reference action, (3) Challenger Policy artifact with compile/validation status, (4) challenger candidate with public rationale, (5) pre-reveal audit record (public evidence, gain/risk, gate decision), (6) final action log. The post-reveal outcome is appended only after the audit record is sealed.

### LLM backend
OpenAI-compatible Responses API, `gpt-5.4`, reasoning effort medium, temperature 1.0, structured JSON outputs, verbosity low, max output tokens 6000 (Minerva/Olympus) / 9000 (ChemLex). Total cost: **497 API calls / 3.73M tokens** on Minerva/Olympus (16.6/seed) and **525 / 4.10M** on ChemLex (17.5/seed), over 30 seeds. A Gemini-2.5-Flash secondary backend is reported in Appendix B.2.

### Anytime trajectories (Figure 2)
CARE pushes normalized regret down early on Minerva/Olympus and maintains the lowest final regret on both datasets. The middle rounds on ChemLex are close, but CARE still ends lowest.

### Ablations (Tables 3, 16, 17)
Mean drop from full CARE (final-best / AUC), 30 seeds:

| Variant | Minerva/Olympus | ChemLex |
|---|---|---|
| w/o Policy Planner | 1.3 / 2.4 | 4.7 / 2.9 |
| w/o intervention gate | 1.2 / 1.7 | 4.6 / 2.3 |
| w/o recovery proposal mode | 0.9 / 1.6 | **6.9 / 3.1** |

| 6-expert public incumbent | 11.6 / 15.4 | 10.2 / 6.4 |

- The **frontier mode** is the dominant contributor on Minerva/Olympus; the **recovery mode** dominates on ChemLex. Both proposal modes matter; both dataset-specific.
- **Simply expanding the public incumbent to 6 experts (76.9/69.0 Minerva, 81.9/75.2 ChemLex) does *not* close the gap to CARE** — the gain is not from richer public expertise alone.

### Intervention diagnostics (Table 6, 300 planner calls per dataset)

| Metric | Minerva/Olympus | ChemLex |
|---|---|---|
| Eligible Challenger Policies | 197 | 225 |
| Authorized challenger actions | 52 | 36 |
| Immediate improvements | 21 | 9 |
| Mean active policies | 6.5 | 7.0 |
| Recovery-mode authorizations | 34 | 36 |
| Frontier-mode authorizations | 18 | 0 |

- The gate is **sparse**: ~17% / ~12% of decisions change the selected action. This is the intended "intervention funnel" — the LLM proposes often, the gate is conservative.
- Minerva/Olympus uses both modes; ChemLex uses recovery only, because its high-cardinality categorical schema does not expose a decision-visible feature view for frontier exploration.

### Backend robustness (Table 8, Gemini-2.5-Flash)
Swapping the proposal LLM to Gemini-2.5-Flash with all other settings held fixed:
- Minerva/Olympus: 88.2 / 83.3 (regret 1.0), 42/20 auth/improvements.
- ChemLex: 90.8 / 80.1 (regret 9.2), 35/7 auth/improvements.
The control interface is **backend-agnostic** — gate, prompt interface, and protocol stay the same.

### Replay-protocol robustness (Figure 3, Tables 10-11, 10 matched seeds per setting)
Four protocol variants (init, reveal budget): Main (5,10), Low-init (3,10), Long-budget (5,20), Warm-start (10,10). CARE keeps **positive mean margins** vs the public incumbent and the w/o-intervention-gate ablation in **every dataset–protocol cell**. Minerva/Olympus Long-budget CARE reaches 89.2 (vs incumbent 85.8); ChemLex Long-budget reaches 97.8 (vs 96.2).

### Intervention-margin sensitivity (Table 9, Minerva only, 10 seeds)
Margins −0.1, 0.0, +0.1, +0.2 give final-best 86.8 / 86.6 / 86.5 / 86.5 and AUC 83.5 / 83.6 / 83.8 / 80.3. The pre-specified margin of 0.0 is essentially the break-even point of gain−risk; nearby margins preserve qualitative behavior, +0.2 lowers anytime AUC without helping final-best.

### Policy evolution in practice (Table 4 + Appendix D)
The Challenger Policy is an *executable Python `rank_candidates`* function, validated then logged. Concrete patches in the appendix:
- **Minerva/Olympus** (seed 3): initial policy ranks by best-ligand proximity. After a successful challenger action (L3, residence time 120, T=110, loading 2.515 → post-hoc yield 89.23), the patch keeps best-ligand preference, adds a **diversity penalty** (`5.0 * nearest_same if tested_same_count > 0 else 1.5 * nearest_any`) and a small **exploration bonus** for unseen ligands.
- **ChemLex** (seed 11): initial policy blends positive acid–amine similarity, failed-pair avoidance, and component evidence. After low-reward reveals, the patched policy strengthens positive-similarity weight, softens negative penalties, and keeps novelty secondary — yielding two gate-authorized recovery actions (C2 → 93.42, C3 → 94.74).

## Critical Assessment

**Novelty.** The framing is genuinely new and useful. CARE is not a new acquisition function or a new BO surrogate; it is a **control architecture** that says "language models are best used as proposal-space expanders behind a fixed public-evidence gate." The pre-reveal audit ledger and the strict public-information boundary are a more disciplined take on the "LLM agent for science" trend than most contemporaneous work. The closest analogue is LMABO-style LLM-BO, but those works let the LLM touch the acquisition function directly; CARE refuses to.

**Rigor.** Strong, with caveats:
- *Matched-seed protocol* (same 5 initial observations, same remaining pool, same reveal budget, deterministic tie-breaking, 30 seeds) is the right design and the paper uses it throughout.
- *95% bootstrap CIs* on every reported number. *Paired W/T/L* on the primary controls. Good.
- *Comprehensive ablation suite*: the 6-expert public incumbent test is the most informative ablation — it falsifies the "just add more BO experts" alternative cleanly.
- *Backend swap* (GPT-5.4 → Gemini-2.5-Flash) is a useful transfer test and survives.
- *Replay-protocol robustness* across 4 settings shows the gain is not a 10-reveal artifact.
- *Caveat:* the LLM backend is `gpt-5.4` (reasoning_effort=medium, temperature=1.0). No ablation over backend temperatures or prompt-interface variants. The ChemLex top-10 hit is only 13% (vs 10% incumbent), much smaller than Minerva's 100% vs 47% — the paper is honest about this but the headline numbers on ChemLex are not as dramatic as they look in Table 2.

**Reproducibility.** High in principle:
- Full controller specification in §B.5: fixed term definitions, schedules, gain/risk terms, weights, intervention margin.
- Full prompt templates in Appendix F: planner prompt card, policy-synthesis prompt card, both with allowed actions, allowed skill families, and explicit forbidden-info clauses.
- Per-round algorithm (Algorithm 1) with explicit ordering.
- Audit ledger schema (Table 12) — the artifacts that determine the selected action are written before reveal.
- Reported token/call budget: 497/525 API calls, 3.73M/4.10M tokens.
- *Caveat:* the upstream datasets (Minerva/Olympus, ChemLex) are explicitly non-commercial (CC BY-NC 4.0); the paper says it does not redistribute raw archives, so a re-implementer needs the original Zenodo record (Zhong 2025) and the Olympus repo (Häse et al. 2021b).

**Significance.** High for the niche of LLM-driven scientific experimentation. CARE's lesson is generalizable: when a learned policy cannot be trusted with direct action authority, **separate proposal generation from action authority and gate the latter with a fixed public-evidence rule plus a pre-reveal audit log**. This is a reusable template for agent deployment under irreversibility, regulatory, or safety constraints, not just HTE.

**Red flags / things to double-check:**
- *Oracles are dataset-specific and reachable.* Minerva/Olympus oracle is 89.23; CARE hits 88.5 — essentially oracle. The "near-oracle" framing is a fair reading but worth flagging: the controller is being asked to find a near-best candidate in a pool of 5,670 with only 10 reveals (≈0.18% of the pool). Whether the same control architecture scales to pools that are 10⁴–10⁵× larger and budgets that are not so generous is unclear.
- *The ChemLex "more challenging" claim is partially about the BO baselines being closer*, not about the controller being weaker. The paired delta vs the strongest BO baseline (EDBO-style GP-EI) is +3.2 final-best and +1.1 AUC; modest but consistent.
- *The frontier mode is gated on the schema exposing a decision-visible feature view.* On ChemLex (high-cardinality categorical), it never fires. This means the "two bounded proposal modes" is in practice a *single-mode* controller on a large class of HTE problems.
- *Cost is reported but not normalized.* 16–17 LLM calls per seed is modest, but the paper does not report wall-clock time or dollar cost in the main text. Without that, the "is the gain cost-effective?" question is open.
- *The intervention margin (0.0) is the "break-even point of gain minus risk, not a value fitted to replay outcomes"*, but the margin-sensitivity table only varies Minerva. ChemLex is not sensitivity-checked.
- *No ablation on the LLM backend's temperature or prompt-card details.* The "sparse intervention" pattern is robust to backend swap, but a prompt-level ablation would strengthen the claim.

## Strengths

- Clean **separation of concerns**: classical optimizer = safety floor, LLM = exploration engine, gate = arbiter. The pre-reveal audit log makes the decision boundary reproducible.
- **Pre-outcome gating** is the right design choice for irreversible experiments — it forces the gate to use *only* causally available information.
- The **6-expert public incumbent ablation** is the most important control in the paper: it falsifies the "just add more BO experts" alternative and isolates the LLM-gate contribution.
- **Backend transfer test** (GPT-5.4 → Gemini-2.5-Flash) without touching the gate, prompt, or protocol demonstrates that the control interface is genuinely modular.
- **Reasonable statistical reporting** (95% bootstrap CIs, paired W/T/L, 30 seeds, matched protocol) — far above the agent-paper median.
- **Full prompt and controller specification** in the appendix — a re-implementer can in principle reproduce the run.
- The qualitative claim — "LLM self-evolution is more reliable when it expands the proposal space under an auditable controller" — is **supported by the trajectory plots, ablations, and intervention diagnostics**, not just the headline numbers.

## Limitations

- **Replay-only evaluation.** Live-lab constraints (robot scheduling, execution failures, measurement delays, batch-dependent availability, multi-objective constraints) are explicitly out of scope. The controller is validated offline, not in a closed loop.
- **Single-objective, finite-pool, batch=1.** Multi-objective, cost-aware, batched, and safety-constrained HTE are listed as future work.
- **Assumes informative public descriptors.** When the candidate schema is sparse or uninformative, fewer challenger actions survive the gate (this is the ChemLex "recovery-only" case). CARE does not propose a remedy beyond "fall back to the public incumbent."
- **ChemLex top-10 hit is 13%.** CARE is near-oracle on Minerva (100% top-10) but not on ChemLex, where the categorical structure is harder. The paper is honest about this; readers should not over-read the +8.2 final-best delta on ChemLex.
- **No wall-clock / dollar cost in the main text.** 16–17 LLM calls/seed is modest but not benchmarked against, say, doubling the public BO budget.
- **Margin sensitivity** is only checked on Minerva/Olympus.
- **No prompt-level ablation.** The planner-prompt and policy-synthesis prompt are reported in full but not perturbed to test sensitivity.
- **The frontier mode is silent on ChemLex.** In practice, the "two proposal modes" is one mode on a large class of HTE problems.

## Relevance

直接命中"用 LLM 辅助科学发现/高通量实验"和"高风险/不可逆实验中的可审计决策"两条主线。CARE 把"提议生成"与"行动授权"在控制器层面解耦，再用揭示前审计日志固化边界，这对材料发现、催化剂筛选、反应条件优化等真实 HTE 闭环有直接借鉴价值；其思想（"LLM 拓宽提议空间 + 固定公开证据门"）也可迁移到任何需要把 LLM 智能体放进不可逆决策场景的部署形态。

## Questions for Further Exploration

- How does CARE scale to **larger pools** (10⁵–10⁶ candidates) and **smaller reveal budgets** (e.g., 3–5 reveals)? The Minerva result is near-oracle at 5670 candidates / 10 reveals; the headroom for the LLM-gate may shrink.
- Can the **gate itself become learned**? The audit log is rich enough to train a neural gate that inherits interpretability constraints from the public-evidence rule. Would that close the remaining ~10 AUC gap on ChemLex without losing auditability?
- What happens to **cost-effectiveness** if we compare CARE not against 5-init/10-reveal BO but against *longer-budget* BO? The Long-budget CARE-on-Minerva reaches 89.2 (regret 0.0) — does a non-LLM BO with the same 20-reveal budget close the gap?
- Can **other non-LLM proposal families** (acquisition portfolios, GP-UCB variants, expert-hedging) be put behind the same gate? The paper explicitly raises this as a natural next step.
- How does CARE behave under **adversarial / noisy LLM** challengers? The intervention margin is a fixed 0.0; can a robust margin be derived from the public evidence alone?
- Does the **frontier mode** generalize to categorical-heavy HTE if we pre-compute decision-visible public-feature views (e.g., learned embeddings over the candidate schema)? The current "schema exposes it, or it doesn't" rule is the main practical limitation.
- Can CARE be deployed **on a live self-driving lab** under the listed live constraints (robot scheduling, batched execution, failure modes)? The replay-to-deployment gap is the obvious next validation.
