Why This Matters
Current LLMs cannot reliably re-plan under accumulating real-world constraints — a concrete ceiling blocking trustworthy agent deployment.
Critical Edge
Household tasks only; LLM judges introduce bias; "progressive disclosure" is self-induced by the agent's own plan choices — all limit generalizability beyond the benchmark.
Methodology

How AdaPlanBench Works

01
Dataset Base

307 household tasks from MacGyver, rewritten to strip explicit resource hints. Three difficulty tiers by constraint count: E_low (~21 total), E_mid (~42), E_high (~80).

02
Dual-Constraint Pipeline

Multi-agent chain (rewriter → samplers → extractor → merge → checker) generates paired world constraints (tool/environment limits) and user constraints (attribute-level preferences) for every query.

03
Runtime Protocol

Agent proposes plan → LLM judges flag violated constraints → user simulator feeds back natural-language violations → agent re-plans. Constraints are only disclosed upon violation; agents must explore to trigger disclosure.

04
Evaluation

7 metrics (Acc., VPR, Avg Turns, AWRV, AURV, ATWC, ATUC). 4-dimension Rubric (Tool-Use Feasibility, Physical Plausibility, Effectiveness, Safety) scored 1–5 by human-validated LLM judge.

Key Findings
Memory ≠ Success Explicit constraint tracking lifts VPR by 5–15%, but accuracy improves less than 3% — the bottleneck is plan construction, not memory.
User > World Difficulty User constraints are consistently harder than world constraints; preferences implicitly rule out large portions of the tool space.
Scale Doesn't Help Qwen3-8B, 14B, 32B all score ~14–18% Acc., suggesting model size alone cannot fix adaptive planning failures.
Explore to Succeed Proactive constraint elicitation (ATWC/ATUC) correlates with accuracy at r = 0.898/0.919 — diversity of plan proposals is key.
Strengths

What Works Well

✦

First benchmark to jointly test multi-turn adaptive re-planning under both world and user constraints with progressive disclosure.

✦

Negative results (memory and rubric feedback don't help) are clearly demonstrated, not buried — rare and valuable.

✦

Strong correlation finding (proactive exploration ↔ accuracy, r ≈ 0.9) gives a concrete direction for model improvement.

✦

Scalable automated construction pipeline; code and dataset are publicly released.

✦

Human-validated LLM judge adds credibility; confidence intervals in appendix.

Limitations

Where to Be Cautious

▲

Household domain only — findings may not generalize to web navigation, code generation, or scheduling.

▲

Text-only interaction — no perception, embodied execution, or real environment grounding.

▲

LLM-based judge introduces systematic bias; partially mitigated but not eliminated.

▲

"Progressive disclosure" is self-induced — difficulty is partially determined by the agent's own plan choices, a subtle confound.

▲

Early-stopping criterion (no new violations for 2 turns) may prematurely end trajectories that could eventually succeed.

▲

Preprint — not yet peer reviewed.

Open Questions

Directions for Further Exploration

Q1

Why does explicit constraint tracking fail to improve final accuracy — is it a planning-search problem, a goal-specification problem, or reasoning failure under constraint accumulation?

Q2

Can chain-of-thought or scratchpad-based plan revision significantly narrow the 67.75% GPT-5 accuracy ceiling?

Q3

How would multimodal inputs (visual grounding) reduce physical plausibility failures?

Q4

Can a dedicated user-preference modeling module close the user-constraint difficulty gap — the hardest constraint type in this benchmark?

Q5

Does MacGyver's household focus create systematic biases? Would travel, scheduling, or coding domains yield qualitatively different failure patterns?
