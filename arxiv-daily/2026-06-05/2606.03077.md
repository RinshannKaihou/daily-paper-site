> **一句话总结：** Libra 是一个专为 Agentic RL 后训练设计的资源管理系统，通过周期性跨阶段 GPU 重分配规划器和因果驱动多级反馈队列调度器，在 48 张 A800 上实现最高 3.0× 吞吐量提升和 2.5× 奖励收敛加速。

> **TL;DR:** Libra is a resource orchestration system for agentic RL post-training that jointly optimizes GPU allocation across rollout and training clusters via a periodic global planner with elastic execution, and routes rollout requests using a causality-driven multi-level feedback queue (C-MLFQ) that leverages tool-return outcomes rather than fragile length predictions — achieving up to 3.0× throughput and 2.5× faster reward convergence on 48 A800 GPUs across three agentic RL benchmarks.

---

## 核心贡献

1. **跨阶段联合优化框架**：将 Rollout 和 Training 的资源分配视为一个耦合优化问题，周期性地重新规划全局 GPU 分配，而非静态分配。这是现有系统（无论是 Colocated 还是 Disaggregated）从未解决的盲区。

2. **弹性混合池（Elastic Hybrid Pool）**：维护三类 GPU 池（Core Training Pool、Core Rollout Pool、Elastic Hybrid Pool），混合工作节点可在 Rollout 和 Training 模式间非阻塞切换，切换开销仅约 15 ms（训练→Rollout）和 4.4 s（Rollout→训练，异步后台完成）。

3. **因果驱动多级反馈队列 C-MLFQ**：不依赖脆弱的序列长度预测，而是利用工具调用返回结果（payload 大小、成功/失败状态）作为确定性因果信号，通过前缀树（Causality-Aware Prefix Tree）将请求路由到异质 Tensor Parallel 桶，达到 91.1% 的每次决策路由准确率（Oracle 上界 100%，预测方法仅 65.2%）。

4. **层次化自动并行规划器**：外层决策树枚举训练侧并行策略（TP/PP/DP），内层动态规划求解 Rollout 侧异质划分，配合多项式代价评估器（MAPE < 6.35%），使规划搜索时间在 memoization 下控制在 1.9 s 以内（128 GPU 规模）。

---

## Key Contributions

1. **Cross-stage joint optimization**: Treats rollout and training GPU allocation as a coupled problem, periodically re-planning the global configuration as the workload distribution drifts — a dimension ignored by all prior colocated and disaggregated RL systems.

2. **Elastic Hybrid Pool for non-blocking reallocation**: Three-pool architecture (Core Training, Core Rollout, Elastic Hybrid) allows lightweight GPU reallocation between stages. Worker transitions incur only ~15 ms teardown (training→rollout) and ~4.4 s background snapshot reload (rollout→training), invisible to the critical path.

3. **C-MLFQ: causality-aware scheduling without length prediction**: Builds a lightweight prefix-trie over historical tool-call return-state sequences to derive the remaining-length distribution at any mid-trajectory point. Routes requests to heterogeneous TP buckets on each tool return, achieving 91.1% per-decision accuracy (vs. Oracle 100%, prediction-based 65.2%, reactive MLFQ 44.8%, load balancing 31.2%).

4. **Hierarchical auto-parallel planner with cost evaluator**: Decision-tree enumeration of training parallelism strategies combined with DP-based rollout partitioning, underpinned by a polynomial cost evaluator with MAPE 3.1–5.9% for rollout and 2.4–5.5% for training time prediction.

## Methodology

### Problem Setup

Agentic RL post-training differs from standard LLM RL in three fundamental ways:
- **Long-tail rollout**: Tool invocations cause trajectory length to vary from <8K to >41K tokens (R2E-Gym), creating pronounced stragglers.
- **Cross-stage asymmetry**: Rollout is memory/bandwidth-bound and scales linearly with sequence length (30 s → 2,850 s as length grows 1K→32K); training is compute-bound and amortizes length through batching (135 s → 527 s over the same range). Static GPU splits therefore leave one stage perpetually starved.
- **Non-stationarity**: The policy evolves continuously, shifting the trajectory-length distribution (e.g., Search-R1 average length grows ~2,500→~11,500 tokens), so any fixed allocation becomes progressively suboptimal.

Existing systems either colocate (verl, OpenRLHF) or disaggregate (AReaL, StreamRL, OrchestrRL) but all fix the rollout/training GPU ratio statically or with hand-tuned heuristics. None address the cross-stage coupling.

### Periodic Global Resource Planner

The end-to-end iteration time is `T_iter = max(T_rollout, T_train)`. The planner minimizes `T_iter` by jointly searching over:
- **Training side**: a topology-aware decision tree that enumerates valid (TP, PP, DP) factorizations with branch pruning on memory, communication-to-compute ratio, and pipeline bubble ratio. Reduces the exponential search to dozens of candidates.
- **Rollout side**: dynamic programming over heterogeneous TP partitions. Requests are sorted by generation length; DP assigns contiguous sorted segments to TP instances, minimizing makespan. Rollout DP results are memoized per `n_rollout`, so outer loop reuse avoids redundant computation.

The planner is invoked every K training steps, reads the latest workload statistics, and triggers reallocation only if the projected throughput gain exceeds the reconfiguration cost.

### Elastic Hybrid Pool

To realize GPU reallocations without global communicator rebuilds, Libra separates roles:
- Core Training Pool: fixed collective group, immutable topology, performs synchronized weight updates.
- Core Rollout Pool: heterogeneous TP inference, handles trajectory generation.
- Elastic Hybrid Pool: workers hold complete, unsharded model weights and optimizer states. They join as external data-parallel replicas of the training pool (routing gradients through an asynchronous side channel, not the main All-Reduce), or switch to rollout mode without reshaping the training topology.

**Non-blocking joining protocol**: When a hybrid worker returns to training, active training workers asynchronously snapshot their state each step. The rejoining worker fetches this snapshot via P2P (CPU-memory, RDMA-enabled) and emits a zero-gradient placeholder until the snapshot is loaded — mathematically preserving gradient equivalence (proven formally in Appendix B).

### C-MLFQ Scheduler

The heterogeneous rollout cluster has buckets `B_1 < B_2 < ... < B_m` ordered by TP capacity. Routing proceeds in three phases:

1. **Phase 1 — Initial placement**: All requests start in B_1 (smallest TP, lowest overhead for short sequences).
2. **Phase 2 — Per-tool-return routing**: On each tool return, the scheduler walks the causality-aware prefix tree from the root following the observed return-state sequence (encoding tool type, payload size class, success/failure). The reached node stores offline statistics on remaining trajectory length (mean and P90). Migration occurs only when both statistics agree on the target bucket (conservative rule against outlier-driven thrashing).
3. **Phase 3 — Tree update**: Completed trajectories are inserted offline along their return-state sequences, updating node statistics incrementally. Full tree rebuilds every T=50 steps take <1 s.

**KV-cache migration**: When a request migrates across TP buckets, the KV cache is offloaded to CPU pinned memory (during tool execution, so zero critical-path cost), CPU-resharded to match the target TP's attention-head partition, then GPU-reloaded at tool return. Cross-node transfer stays below 733 ms even at 40K-token prefixes; the crossover where recomputation is cheaper falls around 4K tokens.

## Results & Highlights

### End-to-End Throughput (48 A800 GPUs, GRPO, Qwen3-14B / Qwen3-30B-A3B)

| Benchmark | Libra | vs. AReaL-Static-Optimal | vs. verl-Greedy-Heuristic | vs. verl-Colocated |
|-----------|-------|--------------------------|---------------------------|---------------------|
| Search-R1 | ~2,700 tok/s | +63% | +80% | +300% |
| DAPO-Math-17K | — | +60% | +40% | +200% |
| R2E-Gym | — | +49% | +83% | +209% |

### Reward Convergence (wall-clock time to equivalent reward)

- Search-R1: **17.9 h** (vs. AReaL-Static-Optimal 28.6 h, i.e., **1.6× faster**; vs. verl-Static-Uniform ~2.5× faster)
- DAPO-Math-17K: **26.7 h** (~1.6× faster than AReaL)
- R2E-Gym: **63.2 h** (~1.5× faster than AReaL, ~1.85× vs. verl-Greedy-Heuristic)

### Routing Accuracy (Search-R1, same TP bucket config)

| Method | Per-Decision Accuracy | Migration Ratio | Throughput |
|--------|-----------------------|-----------------|-----------|
| Load Balancing | 31.2% | 0% | 1,502 tok/s |
| MLFQ | 44.8% | 10.8% | 1,917 tok/s |
| Prediction-Based | 65.2% | 48.9% | 2,400 tok/s |
| **C-MLFQ** | **91.1%** | 8.2% | **2,848 tok/s** |
| Oracle | 100% | 0% | 3,102 tok/s |

### Ablation (R2E-Gym, Qwen3-14B, 48 GPUs)

Starting from verl-Static-Uniform baseline (423 tok/s):
- +Planner (homogeneous TP): 510 tok/s (+20%)
- +Planner (heterogeneous TP): 551 tok/s (+10%)
- +C-MLFQ: 666 tok/s (+21%)
- +Elastic execution: 763 tok/s (+15%) — full Libra, **78% above baseline**

### Cost Evaluator Fidelity

- Rollout step-time MAPE: 3.1–5.9% across sequence lengths 512–40K and TP=1–8
- Training iteration MAPE: 2.4–2.6% (Qwen3-14B), 5.0–5.5% (Qwen3-30B-A3B)
- End-to-end scatter: overall MAPE 6.35%, max error 9.30% across 100 estimations

## Critical Assessment

**Strengths**

- **Problem formulation is genuinely novel**: framing rollout/training GPU allocation as a jointly coupled optimization problem is well-motivated by the empirical characterization (Figures 3a/3b showing 95× vs. 3.9× latency scaling divergence and workload drift).
- **C-MLFQ's causal insight is elegant**: exploiting tool-return outcomes as deterministic causal signals for routing avoids the fundamental accuracy degradation of any embedding-based predictor under RL (since the LLM is continuously updated). The 91.1% accuracy approaching the Oracle's 100% validates this.
- **Non-blocking elasticity is carefully engineered**: the zero-gradient placeholder trick preserving mathematical equivalence during asynchronous re-joining is technically sound and empirically validated (Figure 13 showing identical final reward between Libra-Non-Blocking and Libra-Sync-Join, with Sync-Join incurring ~150 s spikes per transition).
- **Comprehensive evaluation**: three benchmarks, two model families (dense Qwen3-14B and MoE Qwen3-30B-A3B), four baselines, ablations for each component, and overhead microbenchmarks.
- **Honest overhead reporting**: transition costs are quantified (Table 3), planner search time with/without memoization shown (Figure 11b), KV-cache migration latency profiled (Figure 11a).

**Limitations**

- **Single-cluster evaluation at 48 GPUs**: all experiments use exactly one 6-node, 8-GPU-per-node cluster. Scalability to 256+ GPU setups (common in production RL training) is not demonstrated. The planner's DP complexity is O(n_rollout · L² · |T|), which may become non-trivial at very large scales, and the paper does not profile this beyond 128 GPUs.
- **Algorithm-agnostic claim is partially untested**: the paper states Libra is algorithm-agnostic (GRPO, PPO, etc.) but all experiments use GRPO only.
- **C-MLFQ requires an offline profile phase**: before training, Libra must run the full corpus through the policy to build the initial prefix tree and calibrate the cost evaluator (Table 4: 1.24–3 h depending on benchmark). This adds non-trivial pre-training overhead and must be repeated when the base policy changes significantly.
- **Heterogeneous rollout assumes TP as the primary parallelism axis**: Pipeline Parallelism and Expert Parallelism are handled as extensions (Appendix C) rather than first-class citizens, which may limit applicability to very large models that require PP for memory reasons.
- **Workload coverage**: all three benchmarks involve external tool calls; the system's benefits for standard (non-agentic) RL where trajectory lengths are more stable would be marginal and are not discussed.

## Questions for Further Exploration

1. **Scalability**: How does the planner's optimality degrade at 256–512 GPU scales where the search space explodes and communication topology constraints multiply? Does the memoized DP remain tractable?

2. **Policy-drift sensitivity of C-MLFQ**: The prefix tree is rebuilt every T=50 steps, but Table 5 shows the fallback ratio is still ~3.68% at steps 101–200. How does C-MLFQ perform during rapid policy transitions (e.g., reward-shaping changes mid-training)?

3. **Interaction with RLHF/PPO**: The paper claims algorithm-agnosticism, but PPO requires a critic and reference model that consume additional GPU memory. How does the planner's cost model account for the memory footprint of these additional models when determining TP/PP/DP splits?

4. **KV-cache migration vs. recompute tradeoff**: The crossover where recompute is cheaper than migration is ~4K tokens. As models scale to longer context windows (128K+), does this crossover shift, and does Libra's per-request decision logic adapt accordingly?

5. **Cold-start overhead amortization**: For short production runs or rapid experimentation, the 1.24–3 h offline profile phase may dominate total training time. Is there a warm-start mechanism from a prior run's profile, or a cheaper adaptive approximation for the first few hundred steps?
