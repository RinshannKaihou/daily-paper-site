> **一句话总结：** QuBLAST 通过块级灵敏度分析 + 精度锁定 + 激活缩放，实现跨多种注意力架构的混合精度 PTQ，在 5% 困惑度约束下压缩 LLM 内存 40–45%。

> **TL;DR:** QuBLAST is a post-training quantization framework that performs block-level sensitivity profiling, precision lockout, and activation scaling to find mixed-precision configurations for LLMs, achieving 40–45% memory reduction across four architecturally diverse models while keeping perplexity within 5% of the FP16 baseline.

---

## 核心贡献

1. **块级压缩框架（Block-Level Compression）**：将模型的注意力块抽象为"公共注意力块（CAB）"单元，对每个 CAB 独立进行交叉熵损失灵敏度分析，避免了逐层枚举的指数级搜索空间。

2. **精度锁定（Precision Lockout）**：基于用户定义的可接受损失阈值（如 2% 或 5%），自动排除对量化高度敏感的位宽配置，将候选搜索空间收缩至可管理范围，同时保护关键块免受过度压缩。

3. **激活缩放策略（Activation Scaling Strategy）**：引入可学习/可搜索的缩放映射 M，在量化前将每个 CAB 的输入/输出激活值控制在有界范围内，有效抑制激活异常值对量化表示能力的破坏，同时节省 42.4%–48.2% 内存。

4. **跨架构适配**：在标准 Transformer（Llama3-8B、Mistral v0.1-7B、Qwen3-8B）和混合 Mamba 架构（Falcon H1R-7B）上均验证有效，证明 CAB 抽象具备架构无关性。

## Key Contributions

1. **Block-Level Compression with CABs**: Abstracts repeated transformer blocks into "Common Attention Blocks" (CABs) as the basic quantization unit, enabling sensitivity analysis without enumerating every individual layer.

2. **Precision Lockout**: Uses cross-entropy loss increase (not perplexity, which over-inflates small differences) as the sensitivity metric. Bit-width configurations causing loss increase above a user-defined threshold T_loss are locked out per block, shrinking the search space while protecting sensitive blocks.

3. **Activation Scaling Strategy**: Introduces a per-block scaling map M that scales actual activation values A_i to a controlled range A_hat_i before quantization, then reverses the scaling on outputs. This directly suppresses outlier activations without the complex rotation operations used by methods like SmoothQuant or SpinQuant.

4. **Cross-Architecture Generality**: Validated on Qwen3-8B (FlashAttention), Llama3-8B (standard), Mistral v0.1-7B (QK rotation), and Falcon H1R-7B (hybrid SSM+self-attention), demonstrating that the CAB abstraction accommodates non-conventional attention structures.

## Methodology

QuBLAST operates as a three-stage PTQ pipeline, implemented in PyTorch on a single NVIDIA A100 40GB GPU, with no fine-tuning or retraining.

**Stage A — Network Model Analysis:**
- Parse the pre-trained LLM's architecture to identify the repeating attention blocks.
- Categorize layers into: embedding (input), attention, and output layers. Only attention blocks are quantized; embedding and output layers remain at FP16 due to higher sensitivity.
- Group repeated attention layers into CABs — the coarser-grained quantization unit that makes the analysis tractable for models with 250+ layers.

**Stage B — Block-Level Compression:**

*Weight Quantization Sensitivity (Algorithm 1):*
- Candidate bit-widths: {2, 4, 6, 8, 10, 12, 14, 16}.
- For each block b and each bit-width q, temporarily quantize block b to q bits (all others remain FP16), compute cross-entropy loss L[b][q] on N_eval calibration batches, and record the loss increase ΔL[b][q] relative to the FP16 baseline.
- This produces a per-block sensitivity profile revealing that early blocks (block-0 to block-2) and final blocks are most sensitive, while middle blocks tolerate aggressive quantization.

*Precision Lockout:*
- If ΔL[b][q] > T_loss (e.g., 2% or 5%), bit-width q is locked out for block b.
- Example: if block-0 fails at 2-bit and 4-bit, its minimum allowed bit-width becomes 6-bit, reducing configuration candidates.

*Activation Scaling (for activation quantization):*
- Introduce scaling map M per CAB: scaled input Â_i = A_i + M, scaled output Â_o = A_o * M.
- M constrains activations to a quantization-friendly range, mitigating outlier impact without expensive rotation matrices.

**Stage C — Quantization Setting Selection (Algorithm 2):**
- Evaluate remaining valid configurations (after precision lockout) by computing full perplexity on WikiText-2 and WikiText-103.
- A configuration is "valid" if its perplexity increase stays within T_perf% of the FP16 baseline (5% in reported experiments).
- Among all valid configurations, select the one with the minimum memory footprint.
- Memory footprint is estimated analytically: non-quantized parameters (embedding + output) at 16-bit, quantizable block parameters at their assigned bit-widths.

## Results & Highlights

**Memory Reduction (Table II, WikiText-2/103, T_loss = 2% or 5%):**

| Model | Memory Savings |
|---|---|
| Qwen3-8B | 40.0% |
| Llama3-8B | 40.7% |
| Mistral v0.1-7B | 45.2% |
| Falcon H1R-7B | 42.7% |

**Perplexity Increase (Table III, at the same thresholds):**

| Model | WikiText-2 PPL Increase | WikiText-103 PPL Increase |
|---|---|---|
| Qwen3-8B | 3.95% | 3.36% |
| Llama3-8B | 4.64% | 4.94% |
| Mistral v0.1-7B | 2.27% | 1.54% |
| Falcon H1R-7B | 1.06% | 2.65% |

**Activation Quantization (Table IV):** With activation scaling enabled, QuBLAST achieves 42.4%–48.2% memory savings (Qwen3-8B: 42.4%, Llama3-8B: 43.5%, Mistral: 48.2%, Falcon: 44.7%), all within 5% perplexity degradation.

**Block Sensitivity Findings (Figure 8):** Cross-entropy loss profiles consistently show:
- Blocks 0–2 are most sensitive across all models and bit-widths.
- Most middle blocks stabilize between 6-bit and 8-bit (further bit-width increase yields minimal loss change).
- This non-uniform pattern is the key motivation for mixed-precision over uniform assignment.

**Comparison vs. Baselines (Table I, Llama3-8B and Mistral v0.1-7B on WikiText-2):**
- Uniform W8 + AS: ~50% memory, but 1.85%/0.68% PPL increase — QuBLAST achieves similar PPL with only 40.7% memory (slightly less savings but architecture-agnostic).
- Uniform W4 + AS: ~75% memory, but >100% PPL increase — unusable. QuBLAST avoids this failure mode by locking out aggressive bit-widths for sensitive blocks.
- GPTQ (W4, ~75% memory): 18.0%/59.3% PPL increase — far worse quality.
- SmoothQuant (W8A8): ~50% memory — QuBLAST is competitive at less uniform configuration.
- SpinQuant-nohad (W4A8): 5.6% PPL on Mistral, QuBLAST achieves 2.27%.

**Threshold Robustness:** Using T_loss = 2% vs. 5% converges to the same final selected configuration across all four models, demonstrating stability of the block sensitivity structure.

## Critical Assessment

The paper addresses a real and practically important problem: PTQ methods that assume uniform quantization across all blocks leave significant memory savings on the table, and methods that handle outlier activations (SmoothQuant, SpinQuant) add computational overhead that complicates deployment. QuBLAST's three-stage pipeline is conceptually clean and the CAB abstraction is a sensible engineering choice.

However, several limitations temper enthusiasm:

- The evaluation is exclusively on **weight-only or weight+activation quantization at 8-bit weight** regimes. The paper does not compare at the frontier of 4-bit weight, 4-bit activation (W4A4), which is increasingly the deployment target for on-device inference.
- The **search cost** is not explicitly reported. Evaluating every (block, bit-width) pair on calibration data — even with block-level coarsening — can be non-trivial for 30+ block models.
- The **activation scaling map M** is described but its derivation / optimization procedure is underspecified in the paper. It is unclear whether M is calibrated, optimized, or heuristically set.
- Evaluation benchmarks are limited to **perplexity on WikiText-2 and WikiText-103**. Downstream task benchmarks (MMLU, ARC, HellaSwag) would better validate whether perplexity preservation translates to capability preservation.
- The 40–45% memory savings, while solid, fall below what aggressive uniform 4-bit methods claim (though those methods sacrifice quality, as shown in the comparison table).

---

## Strengths

- **Architecture-agnostic design**: The CAB abstraction cleanly handles both standard transformers and hybrid SSM architectures (Falcon H1R-7B), which most PTQ papers ignore.
- **No retraining required**: Pure PTQ pipeline makes it deployable on any pre-trained checkpoint.
- **Methodologically transparent**: Algorithm 1 and Algorithm 2 are provided in full pseudocode, and Figure 4 gives a clear system overview.
- **Consistent results across datasets**: WikiText-2 and WikiText-103 results align closely, suggesting the method generalizes beyond a single calibration domain.
- **Threshold robustness**: Convergence to the same configuration under 2% and 5% lockout thresholds is a reassuring indicator of stability.
- **Activation scaling is simple and effective**: Compared to rotation-based methods, the per-block scaling map approach has lower overhead and is easier to integrate.

---

## Limitations

- No downstream task evaluation (MMLU, BBH, etc.); perplexity is a necessary but not sufficient quality metric.
- Activation scaling procedure is underspecified — reproducibility may require careful reading of the implementation.
- No latency or throughput measurements reported; memory savings do not automatically translate to inference speedup without hardware-aware kernel support.
- Search cost (number of calibration forward passes) not quantified.
- Comparison with state-of-the-art methods like LLM.int8(), AQLM, or QuIP# is absent.
- Models tested are all in the 7–8B parameter range; scalability to 13B, 70B, or MoE models is untested.

---

## Relevance

**High relevance** for researchers working on:
- LLM deployment on edge/embedded hardware where memory is the primary constraint.
- Mixed-precision PTQ and block-level quantization strategies.
- Architecture-aware model compression for emerging LLM designs (SSM, hybrid models).
- Activation outlier mitigation without complex linear algebra operations.

Less directly relevant to researchers focused on training efficiency, scaling laws, or purely software-level inference optimization without hardware memory constraints.

## Questions for Further Exploration

1. How does QuBLAST perform on **W4A4 quantization**? The activation scaling approach seems principled for this regime, but results are absent.
2. What is the **calibration cost** (number of forward passes, time) relative to GPTQ or AWQ? Is the block-level approach actually faster in practice?
3. How is the **activation scaling map M derived** — is it a learned parameter, a percentile-based clipping value, or something else? The paper describes the mechanism but not the optimization.
4. Does the **CAB abstraction hold** for MoE architectures (e.g., Mixtral, Qwen-MoE) where different experts may have very different sensitivity profiles?
5. Can QuBLAST's sensitivity profiles be **reused across similar models** (e.g., Llama3-8B → Llama3-70B), or must the calibration be re-run from scratch for each checkpoint?
6. How do the selected mixed-precision configurations **generalize to downstream tasks** — does 2–5% perplexity increase correspond to an acceptable capability drop on MMLU or instruction-following benchmarks?
