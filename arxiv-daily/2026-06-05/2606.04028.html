Novel Aspects of IEEE SA P3109 Arithmetic Formats for Machine Learning

A parameterized family of narrow-bitwidth ML floating-point formats — unified, formally verified, replacing vendor-fragmented FP8/FP4 with a single interoperable standard.

Andrew Fitzgibbon (Graphcore) · Christoph M. Wintersteiger (Imandra) · Jeffrey Sarnoff (IEEE SA)

Key Metric
~500
Mechanically-checked theorems in Imandra IML, with independent Lean cross-validation — an unprecedented formal-verification scale for a numerical standards document.
Why This Matters
One standard to replace them all
Unifies OFP8, MXFP8, bfloat16 successors under a single parameterized framework — critical for cross-vendor ML hardware interoperability and reproducibility.
Critical Edge
The exponent bias design creates a ~2× value-range gap vs. already-deployed OFP8 on H100 / B200 GPUs. The authors claim block scaling compensates during training — but present zero empirical ML validation. Inference model conversion risk is real and unquantified.

All operations are exception-free: NaN, ±∞, and division-by-zero are communicated through return values, not hardware exceptions. Single NaN, single zero (no −0), NaN at code point 2K−1.
κ-Approximation
Scale-invariant accuracy metric counting representable format steps between approximate result r̃(x) and correctly-rounded result r̂(x). Analogous to ULP but precisely defined for all edge cases (subnormals, infinities) across all P3109 configurations.
Rounding modes: nearest-even, nearest-away, toward ±0/∞, inexact-to-odd, and three parameterized stochastic modes (A/B/C). Saturation modes: SatFinite, SatPropagate, SatNone.
Results & Highlights
  • Coherent format family: covers 4-bit through 15-bit, all signedness/domain combinations. Table I shows all six 4-bit signed formats with consistent special-value code-point placement.
  • ~500 theorems in IML including square-root convergence and operation equivalence proofs. A Lean reformulation (FLoPS, arXiv:2602.15965) adds independent validation of further properties (e.g., FastTwoSum).
  • External format interop: all P3109 operations instantiate with binary16, binary32, bfloat16 — enabling mixed-format stochastic rounding pipelines.
  • Fixes IEEE-754 anomaly: nextUp(+∞) = +∞ is inconsistent with IEEE-754's own textual spec; P3109 returns NaN for boundary cases in finite-domain formats.
  • Bias design: exponent bias = 2K−P−1 places 1.0 at the midpoint code point. Switching Extended ↔ Finite does not alter finite value mappings.
  • Block operations: BlockExp, BlockReduceAdd, BlockDotProduct defined uniformly for shared-scale sequences; result scale factor sr is implementation-determined.
Strengths & Limitations
Strengths
  • ▲Auto-generated standards text from machine-checked specs — novel for arithmetic standards authorship
  • ▲Principled design: single NaN, single zero, fixed code points, 1.0 at midpoint
  • ▲Full stack: encoding → block ops → approximate implementation description
  • ▲Formal spec + interim report both publicly available (github.com/P3109/Public)
  • ▲Extensible: 6-bit, 12-bit, and future formats fall in-framework without standard revision
Limitations
  • ▼No empirical ML validation — no training or inference accuracy benchmarks vs. OFP8/bfloat16
  • ▼~2× exponent range gap with deployed OFP8 on H100/B200: adoption barrier unanalyzed
  • ▼No hardware κ benchmarks — κ-approximation framework described but not applied to real accelerators
  • ▼Block result scale factor left to implementations: interoperability risk in practice
  • ▼K ≥ 3 minimum; 2-bit formats excluded (appendix in interim report)
Open Questions
Q1 — Adoption H100/B200 GPUs already ship OFP8 (E4M3/E5M2). Is the realistic path a P3109 overlay on existing silicon, or new hardware? What is NVIDIA's / AMD's stance?
Q2 — κ in practice What κ values do current FP8 accelerators achieve for fused-multiply-add or exp? Is κ ≤ 2 achievable at 8-bit for all common ML ops?
Q3 — Stochastic rounding What are the ML training convergence differences between stochastic modes A, B, and C? Which should be the default recommendation for transformers?
Q4 — Block scaling Is there de facto convergence toward max-abs-finite for result scale selection? Should a future revision standardize this to restore interoperability?
Q5 — Lean gaps What properties remain unproved in the FLoPS Lean reformulation? Are there known divergences between the IML and Lean versions of P3109?
