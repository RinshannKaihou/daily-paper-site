Key Finding — 2606.03458 · 2026-06-05

Token magnitude errors dominate KV-cache quantization failure; fixing them closes the gap to FP16 at 2-bit precision

arxiv.org/abs/2606.03458 ↗ Muller, Bich, Boretti, Chang, Zhuang, Cavigelli · Huawei CSL
Key Metric
60.0%
AIME24 accuracy (Qwen3-4B, 2.3 bits/element) — vs. 56.7% SOTA (QuaRot) and 61.1% FP16
Why This Matters
Reasoning tasks (chain-of-thought) decode autoregressively over thousands of tokens. Quantization errors accumulate across timesteps — a failure mode prior static evaluations completely miss.
Critical Edge
MLA architectures (DeepSeek family) explicitly excluded. Per-token-per-channel FP16 scale storage grows linearly with context — memory overhead at 100k+ tokens is uncharacterized. Tested only up to 14B.
Methodology

Root-Cause Diagnosis: Magnitude vs. Directional Error

For key vector K and dequantized approximation K_dq:
‖K−K_dq‖² = E_M + E_D
where E_M = magnitude error, E_D = directional error.
For the top 0.1% largest errors, >85% is E_M — incorrect token norms after quantization, not angular distortion.

KVarN Pipeline

X (raw cache) → [HX] Hadamard rotation (online, O(N log N)) → [VarN] Dual-axis variance normalization (per 128-token block, Sinkhorn) → [RTN] Round-to-nearest @ 2 bits → Q + s₁ (FP8 scale) + s₂ (FP16 per-token-per-channel scale) Total: 2.3 bits/element

Pseudo-Decode Evaluation (new protocol)

Prior work evaluates in static prefill (one-shot quantization). KVarN introduces pseudo-decode: divide prefill into blocks of b tokens; after each block, quantize and re-run subsequent states through the quantized cache. This directly measures error compounding — the regime that matters for reasoning.

VarN: Calibration-Free

Sinkhorn-style iterative normalization in log-domain. No offline calibration datasets — scales derived purely from the block being quantized. Overhead: 1.9ms per 128-token block on Qwen3-4B vs. 1050ms for generation (0.18% overhead).

Strengths / Limitations
Strengths
  • Clean E_M / E_D decomposition — first to identify token magnitude error as dominant failure mode
  • Pseudo-decode evaluation: principled protocol the community should adopt as standard
  • Calibration-free — no offline datasets, directly deployable
  • 0.18% quantization latency overhead — negligible in practice
  • 3 model scales, 4 task types, 6+ baselines — comprehensive
  • Open vLLM-integrated implementation
Limitations
  • MLA (DeepSeek family) and SSM/Mamba architectures explicitly excluded
  • Per-token-per-channel FP16 s₂ scale: memory grows with sequence length — uncharacterized at 100k+ tokens
  • Models up to 14B only; behavior at 70B+ untested
  • VarN Sinkhorn convergence properties not theoretically analyzed
  • Ablation (VarN vs. Hadamard alone) is visual, not tabular across all benchmarks
  • No end-to-end wall-clock throughput in production serving
Open Questions

  • Q2Does VarN benefit scale with model size? Phi-4-14B shows larger gains than Qwen3-4B — is this systematic?
  • Q3At what bit-budget does the pseudo-decode accumulation regime become negligible for 4-bit quantization?
  • Q4Can a VarN variant address token-magnitude error in MLA architectures (DeepSeek-V3/R1)?
  • Q5How sensitive are results to block size G? Smaller G = more frequent updates but more scale storage overhead.
