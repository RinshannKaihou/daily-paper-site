Methodology

The Key Design Decision: Mixed Layout

The SymbolicLight V1 model has two distinct path types in each block. Each path has different memory access patterns — getting one right hurts the other:

Dense projections (Q/K/V/O, FFN-up)
→ Row-major storage
Dot-product reads contiguous rows
Good L1/L2 locality
Sparse spike-conditioned (TCAM, FFN-down)
→ Column-major storage
Active spike columns are contiguous
Reduces cache miss on binary inputs

Naive full-column-major actually regressed decode speed because dense output accumulation overflowed L1 cache. The mixed layout is the key engineering insight.

INT8 Quantization

s_m = max_k |W_{m,k}| / 127 Q_{m,k} = clip(round(W_{m,k} / s_m), −127, 127) Sparse path: accumulate INT32, apply scale once at end → eliminates per-element scale lookups in hot loop

Why INT4 Failed

Per-group INT4 scales complicate the sparse inner loop — each active column needs a group-scale lookup. Result: INT4 decode is only 10.62 tok/s vs 23.31 for INT8. Bandwidth savings are outweighed by the scale lookup overhead in the sparse hot path.

Strengths
  • Honest reporting: PPL gap and INT4 failure are openly disclosed, not buried. The author explicitly distinguishes runtime contribution from language-quality claims.
  • Concrete systems insight: Mixed layout ablation has clear quantitative support — pure row-major vs. pure column-major vs. mixed all measured.
  • Informative negative: INT4 failure mechanism in sparse paths is a useful finding for other SNN quantization researchers.
  • Narrow but complete scope: Thread-scaling, prefill-scaling, PPL rerun, and correctness caveats all documented clearly.
  • Edge AI relevance: Direct application to robotics / embedded scenarios where no GPU is available.
Correctness note: C++ path aligns with PyTorch reference for step-30k export (top-5 token match, mean abs error 0.005). Newer SFT checkpoints show larger logit error (mean 0.80–1.23) — flagged as an export alignment issue, not a runtime bug.
Limitations
  • Language quality gap is large: PPL 24.80 vs. 8–15 for dense baselines. The source of the gap (training steps, 512-token context cap, architecture) is not isolated — no controlled dense Transformer ablation.
  • Single machine, x86 only: All experiments on AMD Ryzen 7 5800X. ARM/NEON, Apple Silicon, Intel AVX-512 are not covered.
  • Energy claims unverified: CPU power is an estimated utilization-based watt figure — no RAPL or external power meter measurement.
  • Code not open-sourced: Runtime, benchmark scripts, and model export tools not yet public; reproducibility depends on a future release.
  • Comparison is uncontrolled: llama.cpp baselines use different tokenizers, training data, and context lengths — deployment reference, not architecture comparison.
Quality vs. speed: Faster inference is valuable only if the model is good enough. At PPL 24.80, the SNN produces noticeably worse text than TinyLlama-1.1B (8.51). The runtime engineering is solid; the underlying model needs more training.
Questions for Exploration
What is the PPL bottleneck? Training steps (186k), 512-token context limit, or architecture? A controlled dense Transformer on the same data/steps would isolate these factors.
How sparse is the SNN in practice? The paper doesn't measure per-layer spike activation rates during inference — does sparsity vary meaningfully by token type or language?
Does mixed layout transfer to ARM/Apple Silicon? NEON SIMD width differs from AVX2; Apple Silicon's memory bandwidth profile may change the column-major benefit.
Can INT4 be salvaged with per-tensor quantization? Per-group scales cause the sparse hot-loop stall — per-tensor would eliminate the lookup at the cost of more quantization error.
When will real energy measurements be available? The RAPL / external power meter data is the missing piece for the core "edge AI efficiency" claim.
