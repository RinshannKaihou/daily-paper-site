The periodic global planner jointly minimizes T_iter = max(T_rollout, T_train) by searching TP/PP/DP factorizations for training and DP-assigning contiguous sorted request segments to rollout TP instances. Results are memoized per rollout count.

C-MLFQ routes requests using tool-call return outcomes (payload size, success/failure) as causal signals, building a prefix trie over historical tool-return state sequences to predict remaining trajectory length — no fragile length prediction from embeddings required.

Strengths
  • First system to treat rollout/training GPU allocation as jointly coupled — all prior work fixes splits statically
  • C-MLFQ's causal insight avoids embedding-based predictors that degrade as the LLM changes during training
  • Non-blocking hybrid worker re-joining proven to be mathematically equivalent to synchronous join (Appendix B)
  • Three diverse benchmarks: info retrieval (Search-R1), software engineering (R2E-Gym), math (DAPO)
  • Cost evaluator MAPE < 6.35% across 100 estimations — planner predictions are reliable
Limitations
  • Single 48-GPU cluster only — scalability to 256+ GPU production setups undemonstrated
  • All experiments use GRPO; algorithm-agnosticism claim untested with PPO (which adds critic + reference model memory)
  • C-MLFQ requires 1.24–3 h offline profiling before training — non-trivial for rapid experimentation
  • PP and Expert Parallelism are appendix extensions, not first-class citizens — limits applicability to very large models
  • Benefits only accrue for agentic (tool-use) RL; standard RL with stable lengths gains little
Open Questions
  • Does the planner's DP (O(n_rollout · L² · |T|)) remain tractable at 256–512 GPUs where the search space explodes?
  • How does C-MLFQ perform during rapid policy transitions (mid-training reward shaping changes) beyond the studied 50-step tree rebuild interval?
  • At 128K+ context windows, does the KV-cache migration crossover (currently ~4K tokens) shift enough to require redesign?
  • Is there a warm-start mechanism to reuse prior run profiles, reducing the 1.24–3 h cold-start overhead for iterative experiments?
Critical edge: Libra's gains are impressive but the 48-GPU ceiling is a real constraint — production agentic RL training often runs at 256+ GPUs. The offline profiling phase is also non-trivial: 1.24–3 h upfront cost dominates short experimental runs. And the entire system is untested on PPO, which adds a critic and reference model that may shift the optimal GPU allocation substantially.
