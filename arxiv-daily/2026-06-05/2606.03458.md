> **一句话总结：** KVarN 通过 Hadamard 旋转 + 双轴方差归一化，系统性地消除自回归解码中 KV-Cache 量化的 token 幅度误差累积，在 2-bit 精度下于 AIME24、MATH500、HumanEval 等推理基准上全面超越现有 SOTA。

> **TL;DR:** KVarN addresses a previously overlooked failure mode of KV-cache quantization: during autoregressive (test-time scaling) decoding, quantization errors accumulate across timesteps because standard methods fail to preserve per-token magnitudes. KVarN combines Hadamard rotation (incoherence processing) with a calibration-free dual-axis variance normalization (VarN) adapted from Sinkhorn-style weight quantization, achieving new state-of-the-art at 2.3 bits/element on AIME24 (60.0% vs 56.7% KIVI), MATH500 (79.2% vs 77.8%), HumanEval (88.4% vs 86.4%), and IFEval — with only 0.18% quantization latency overhead.

## 核心贡献

1. **问题诊断：token 幅度误差是量化误差的主因。** 论文首次系统性地将 KV-Cache 量化误差分解为"幅度误差"（$E_M$）与"方向误差"（$E_D$）两部分，实验表明：前 0.1% 的最大误差中，超过 85% 来自幅度偏移（token norm 在量化后被错误缩放），而非方向失真。

2. **伪解码评估框架（Pseudo-Decode Setting）。** 现有工作几乎都在 prefill（静态）场景下评估量化质量，忽视了自回归解码时误差随 token 数累积的现象。论文提出将 prefill 序列切成块，逐块量化并用量化后 KV-Cache 推断后续状态，从而真实模拟解码时的误差传播。

3. **KVarN 方法：Hadamard 旋转 + 双轴方差归一化。** 对 K/V 矩阵先做 Hadamard 旋转（消除通道空间异常值），再用 VarN（基于 Sinkhorn 迭代的对数域双尺度方差归一化）在 token 和通道两个维度同时均匀化方差，使量化前每个 token 的 norm 得到正确保留。

4. **校准免费（Calibration-Free）。** VarN 无需任何校准数据，缩放向量直接从待量化块的统计量迭代得出，适合在线部署。

5. **vLLM 开源实现。** 提供完整的 vLLM 集成，代码开源于 https://github.com/huawei-csl/KVarN。

## Key Contributions

1. **Root-cause diagnosis: token magnitude errors dominate quantization outliers.** The paper decomposes KV-cache quantization error into a magnitude component ($E_M = (\|K\| - \|K_{dq}\|)^2$) and a directional component ($E_D = 2\|K\|\|K_{dq}\|(1-\cos\theta)$). For the top 0.1% largest errors, over 85% of error energy comes from magnitude shifts — incorrect token norms after quantization — rather than angular distortion.

2. **Pseudo-decode evaluation protocol.** Prior literature evaluates in a static prefill setting (quantize a fixed context in parallel), which misses the autoregressive error accumulation regime. The authors introduce a "pseudo-decode" setting: divide the prefill into blocks of size $b$; after each block, quantize the new KV pairs and re-run subsequent token states through the quantized cache. This directly measures error compounding over decoding time.

3. **KVarN: Hadamard rotation + dual-axis variance normalization.** Every token is Hadamard-rotated online in the channel dimension (absorbed into weight matrices offline following QuaRot). Every block of $G=128$ tokens is then variance-normalized jointly in both token and channel dimensions via VarN — a Sinkhorn-style iterative algorithm in log-domain that produces two scale vectors ($S_c \in \mathbb{R}^{1 \times C}$, $S_r \in \mathbb{R}^{N \times 1}$). This prevents the rounding process from scaling outlier token norms incorrectly.

4. **Calibration-free design.** Unlike weight quantization schemes that use calibration datasets to approximate input statistics, KVarN's VarN derives scales purely from the block being quantized — making it suitable for online, streaming inference.

5. **Minimal runtime overhead.** The VarN normalization per 128-token block takes 1.9 ms on a Qwen3-4B layer vs. 1050 ms for token generation — 0.18% overhead. Dual-scale dequantization adds at most 1.4% latency vs. single-scale KIVI (fused into the Triton dequant kernel).

## Methodology

### Error Decomposition

For a key vector $K$ and its dequantized approximation $K_{dq}$, the squared $\ell_2$ error is:

$$\|K - K_{dq}\|^2 = \underbrace{(\|K\| - \|K_{dq}\|)^2}_{E_M,\text{ Magnitude Error}} + \underbrace{2\|K\|\|K_{dq}\|(1-\cos\theta)}_{E_D,\text{ Directional Error}}$$

The key empirical finding (Figure 1a): for the top-$k$% largest errors, the fraction $E_M / E_T$ exceeds 85% at $k=0.1\%$ and remains above 15% even at $k=50\%$. Standard KIVI quantizes per token (one scale per token) but fails to control the token norm after quantization, causing $E_M$ to dominate.

### Why Outliers Dominate End-to-End Quality (Figure 3)

The paper shows — perhaps surprisingly — that replacing the worst 5% of quantization errors with oracle values improves end-to-end KL-divergence more than fixing the remaining 95%. This is because outlier errors produce anomalously large attention logits that distort the softmax distribution disproportionately, even though their MSE contribution is small.

### KVarN Pipeline (Figure 2)

```
X (raw cache) → [HX] Hadamard rotation (online, O(N log N))
             → [VarN(HX)] Dual-axis variance normalization (per 128-token block)
             → [RTN] Round-to-nearest quantization
             → Q (2-bit quantized cache) + s1 (standard scale) + s2 (VarN scale)
```

- **Hadamard rotation:** Applied online after RoPE embedding; absorbed into $W_K, W_V, W_O$ offline. Complexity $O(N \log N)$ per layer, negligible vs. linear layers.
- **VarN (Algorithm 1):** Iterative alternating log-domain Sinkhorn normalization. Maintains log-scale vectors $L_c$ (channel-wise) and $L_r$ (token-wise), updating each by half the log-variance in that dimension. Tracks the tile with minimum imbalance (IMB) and returns those scales. Clamps prevent extreme scaling.
- **Storage:** K and V elements stored at 2 bits; one FP8 scale absorbs the Sinkhorn scale into RTN's scale ($s_1$, per group of $G=128$); one second FP16 scale $s_2$ (per token per channel) — totaling 2.3 bits/element including auxiliary storage.
- **Quantization layout:** $S$ sink tokens in FP16, $G=128$ token groups quantized, $R=128$ trailing tokens in FP16 (too recent to form a full group).

### Pseudo-Decode Evaluation

Tokens are divided into blocks of $b$ tokens. After each block, KV pairs are quantized. All subsequent transformer computations use the quantized cache — so errors from block $t$ propagate to blocks $t+1, t+2, \ldots$ This is the realistic regime for long chain-of-thought reasoning.

|---|---|---|---|
| FP16 (baseline) | 16.0 | 61.1% | 82.6% |
| KIVI | 2.3 | 55.5% | 77.8% |

| KVQuant-1% | 2.4 | 40.9% | 67.5% |

| TurboQuant | 4.5 | 49.9% | 77.0% |

On Phi-4-14B: KVarN achieves 61.7% (AIME24) and 84.8% (MATH500), vs. KIVI at 57.8% and 74.4% — a 4-point and 10-point improvement respectively.

KVarN matches FP16 performance on HumanEval (88.4% vs. 88.8% for Qwen3-4B; 88.2% vs. 88.9% for Phi-4-14B) — the only 2-bit method to do so.

### IFEval (Table 3)

KVarN achieves 80.4%/83.4% (Strict/Loose) on Qwen3-4B — best among all 2-bit methods and matching or exceeding Hadamard-only and KIVI baselines across all three models.

### Line Retrieval (Table 4)

At 600 lines (long-context), KVarN achieves 85% accuracy on Qwen3-4B vs. 74% for KIVI and 85% for Hadamard-only — demonstrating that error accumulation suppression matters most at longer contexts.

## Critical Assessment

**What the paper does well:** The error decomposition into magnitude vs. directional components is clean and mathematically well-grounded, and the pseudo-decode evaluation protocol is a genuine methodological contribution that exposes a blind spot in prior work. The empirical results are comprehensive (3 model families, 4 benchmarks, 6+ baselines) and the gains on AIME24 (+3.3–4 points) and MATH500 (+1.4–10 points) over same-bit-budget competitors are significant.

**What deserves scrutiny:**

- **Ablation depth:** The paper shows that VarN alone (VarN(K)) and Hadamard alone (HK) each partially help, with KVarN combining both. However, the ablation is primarily visual (Figure 1b) rather than a systematic table across all benchmarks. It is not entirely clear how much each component contributes to the AIME/MATH gains specifically.

- **Token-dimension VarN cost:** The paper notes that applying Hadamard in the token dimension during online dequantization would be too expensive (128² operations per 128 tokens per channel). VarN sidesteps this but requires storing a second per-token-per-channel scale. The actual memory overhead relative to claims of "2.3 bits/element" deserves careful parsing — the FP8 scale absorbs one factor, but the FP16 $s_2$ scale is an additional 16 bits per token per channel, which could be significant for very long sequences.

- **MLA architectures excluded:** The paper explicitly excludes multi-head latent attention models (e.g., DeepSeek-V2 family), which are increasingly common. This limits applicability.

- **SSM models excluded:** State-space models (Mamba etc.) have no KV-Cache, making KVarN inapplicable to that architecture class.

- **No public serving framework integration:** End-to-end evaluation on vLLM's production serving path is reported as blocked by lack of 2-bit KV-Cache support in current frameworks, requiring the authors' own vLLM fork.

---

## Strengths

- **Novelty of diagnosis:** The distinction between token magnitude error and directional error in KV-Cache quantization is well-motivated and empirically validated. Prior work (KIVI, QuaRot) did not identify this as the dominant failure mode.
- **Pseudo-decode evaluation:** A principled, reproducible protocol for measuring error accumulation that the community should adopt as standard.
- **Calibration-free:** No offline calibration datasets required — directly applicable in deployment.
- **Lightweight overhead:** 0.18% quantization latency, ~1.4% dequantization latency increase — negligible in practice.
- **Comprehensive experiments:** Three model scales (4B, 8B, 14B), four task types (math reasoning, coding, instruction following, retrieval), multiple context lengths.
- **Open implementation:** vLLM-integrated code released.

---

## Limitations

- **Scope limited to standard multi-head / grouped-query attention.** Multi-head latent attention (MLA, e.g., DeepSeek family) and state-space models are explicitly out of scope.
- **Memory overhead at very long sequences.** The per-token-per-channel $s_2$ scale in FP16 adds memory proportional to sequence length × channels, which may become non-trivial for 100k+ token contexts. The paper's "2.3 bits/element" figure is computed for the experimental setting (up to ~32k tokens) and may not hold asymptotically.
- **Approximation of calibration-free VarN:** The VarN iteration is heuristic (clamped log-domain Sinkhorn); convergence properties and sensitivity to the number of iterations $K$ are not thoroughly analyzed theoretically.
- **Evaluation models are relatively small** (up to 14B). Behavior at 70B+ scale, where KV-Cache pressure is most acute, is untested.
- **No wall-clock end-to-end throughput numbers** for production serving; the latency numbers are for isolated kernel benchmarks.

---

## Relevance

This paper is highly relevant to anyone working on:
- **LLM inference efficiency**, particularly for long chain-of-thought / test-time scaling workloads (o1-style, DeepSeek-R1-style)
- **KV-Cache compression** at 2-bit precision
- **Quantization for reasoning models**, where long decoding sequences make error accumulation especially damaging
- **vLLM / production inference** with memory-constrained deployments

The insight that "outlier token magnitude errors drive end-to-end degradation disproportionately" has implications beyond KVarN itself — it suggests that any KV-Cache compression method should be evaluated in the pseudo-decode (accumulated) setting, not just the static prefill setting, before claims about reasoning task performance are made.

## Questions for Further Exploration

1. **How does KVarN interact with speculative decoding?** Speculative decoding produces draft tokens that may be rejected; how does this affect the block-based VarN normalization that assumes a fixed window of 128 tokens?

2. **Does VarN's benefit scale with model size?** The gains on Phi-4-14B (reasoning-plus) are larger than on Qwen3-4B in some benchmarks. Is there a systematic relationship between model scale / architecture and the magnitude of token-scale errors?

3. **Can the pseudo-decode evaluation protocol reveal when 4-bit quantization starts accumulating errors?** The paper focuses on 2-bit compression; it would be valuable to know at what bit-budget the accumulation regime becomes negligible.

4. **What is the interaction with MLA (multi-head latent attention)?** DeepSeek-V3/R1 use MLA, which compresses the KV-Cache at training time. Does token-magnitude error accumulation still occur, and can a variant of VarN address it?

5. **How sensitive are the results to the block size $G=128$?** The paper uses $G=128$ throughout; smaller blocks would allow more frequent normalization updates but increase scale storage overhead, while larger blocks may allow more accumulation within a block.
