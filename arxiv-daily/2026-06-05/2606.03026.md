> **一句话总结：** 本文为稀疏脉冲语言模型（SymbolicLight V1）实现了一套专用的 C++ CPU 推理运行时，通过混合内存布局 + AVX2 INT8 量化将吞吐量从 9.5 提升至 22.63 tokens/s，但代价是语言质量（困惑度）明显差于同等规模的稠密基线。

> **TL;DR:** A single-author systems paper that builds a C++ CPU inference runtime specialized for sparse spiking language models. By treating binary spike states as a first-class execution primitive — using mixed row/column memory layouts and AVX2 INT8 kernels — the runtime achieves 22.63 tokens/s single-thread on an 874M-parameter SNN, outpacing TinyLlama-1.1B (16.31) and Falcon3-1B (11.26) under llama.cpp, but with substantially worse WikiText-2 perplexity (24.80 vs. 8–11 for the dense baselines). The result is framed honestly as a runtime systems contribution, not a language-quality claim.

## 核心贡献

1. **专用 C++ 推理运行时**：首次为脉冲门控语言模型实现了端到端的 C++ CPU 推理引擎，包含 manifest 驱动的权重加载器、PyTorch→二进制导出路径，以及稠密路径与脉冲条件路径的分离执行逻辑。

2. **混合内存布局策略**：稠密矩阵乘法保持行主序（Row-major）以保证 dot-product 局部性；稀疏脉冲条件投影改用列主序（Column-major），使活跃输入列成为连续读取，显著减少 cache miss。纯列主序会拖慢稠密路径，因此"混合"布局是关键设计。

3. **AVX2/FMA + INT8 量化**：采用逐输出通道对称 INT8 量化（per-channel symmetric），稀疏路径在整数域内累积（INT32 buffer），最后一次性应用缩放因子，避免每个活跃元素的重复 scale 查找。

4. **量化消融研究（INT4 负面结论）**：INT4 在稀疏路径上因频繁 per-group scale 查找导致热循环复杂化，纯 INT4 吞吐量仅 10.62 tokens/s（INT8 为 23.31），说明稀疏路径应优先保证累积简单性而非极限带宽压缩。

5. **可复现基准测试包**：包含单线程/多线程解码、长提示预填充、WikiText-2 困惑度重跑及稠密 C++ 基线对比，并明确列出所有正确性注意事项。

---

## Key Contributions

1. **Dedicated C++ inference runtime** for the SymbolicLight V1 spike-gated language model family: manifest-driven weight loader, PyTorch-to-binary export, and separate dense vs. spike-conditioned execution paths — no PyTorch dependency at inference time.

2. **Mixed memory layout** (Table 1): dense matvec stays row-major for dot-product locality; sparse spike-conditioned projections are stored column-major so active input columns become contiguous reads. A naive full-column-major design actually regressed decode speed because dense output accumulation overflowed L1 cache.

3. **AVX2/FMA kernels + per-channel INT8 quantization**: integer-domain accumulation for the sparse hot loop (INT32 buffer, one scale application at the end) eliminates repeated per-element scale lookups that would stall the sparse path.

4. **INT4 negative result**: Pure INT4 reduces weight footprint to ~0.8 GB but slashes decode speed to ~10–13 tokens/s vs. ~23 for INT8, because per-group scales complicate the sparse inner loop. A mixed INT4mix variant (dense in INT4, sparse TCAM/FF-down in INT8) partially recovers but still trails INT8.

5. **Honest benchmark package** with thread-scaling, prefill-scaling, WikiText-2 PPL rerun, and explicit correctness caveats, all on a single AMD Ryzen 7 5800X.

## Methodology

### 模型背景 / Model Background

目标模型是 **SymbolicLight V1**（arXiv:2605.21333），一个解码器-only脉冲门控语言模型，每个 SymbolicLight block 含：
- 稠密投影（Q/K/V/O、FFN-up）：带宽密集，适合低比特打包
- 脉冲条件稀疏投影（TCAM 投影、FFN-down）：输入为稀疏二值或稀疏激活向量

基准测试用的导出检查点：886 步-186k，874M 参数，INT8 量化后权重 1.09 GB（FP32 原始 3.49 GB）。

### 运行时设计 / Runtime Design

| 组件 | 设计选择 | 理由 |
|------|----------|------|
| 权重加载 | Manifest-driven strict parser | 避免权重键名子串歧义 |
| 稠密路径内存 | Row-major | dot-product 连续读 |
| 稀疏路径内存 | Column-major | 活跃脉冲列连续读 |
| Embedding/Norm | Native FP32 | 无法从 INT8 matvec 获益 |
| 量化粒度 | Per-output-channel symmetric INT8 | 稀疏热循环中 scale 只应用一次 |
| SIMD | AVX2 + FMA (x86) | 不依赖专用硬件 |

**INT8 量化公式：**
```
s_m = max_k |W_{m,k}| / 127
Q_{m,k} = clip(round(W_{m,k} / s_m), -127, 127)
```
稠密路径按行反量化；稀疏列主序路径对活跃列在 INT32 域内累积后统一转换。

### 正确性验证 / Correctness Checks

C++ 前向路径对 step-30k 导出与 PyTorch 参考对齐：
- Top-5 token 精确匹配
- 词表级平均绝对误差 0.00529，最大 0.0157

较新的 SFT checkpoint 出现更大 logit 误差（mean 0.80–1.23，max 4.12–4.67），作者将其归为导出对齐问题而非运行时 bug，并如实标注。

## Results & Highlights

### 性能演进（step-30k 导出）

| 运行时变体 | 权重大小 | 解码速度 | 相对 FP32 基线 |
|-----------|---------|---------|--------------|
| Scalar FP32 baseline | 3.49 GB | 9.5 tok/s | 1.00× |
| AVX2 mixed-layout FP32 | 3.28 GB | 14.7 tok/s | 1.55× |
| AVX2 INT8 | 1.06 GB | 19.9 tok/s | 2.09× |
| AVX2 INT4mix | 0.88 GB | 13.7 tok/s | 1.44× |

### 最终基准（186k-step 874M INT8 导出，单线程，AMD Ryzen 7 5800X）

| 模型 | WT2 PPL | 解码速度 | 权重大小 | RSS |
|------|---------|---------|---------|-----|
| **SNN-0.8B, C++ INT8** | **24.80** | **22.63 tok/s** | 1087 MB | 1176 MB |
| Qwen2.5-0.5B, Q8_0 | 15.44 | 32.76 tok/s | 507 MB | 565 MB |
| TinyLlama-1.1B, Q8_0 | 8.51 | 16.31 tok/s | 1116 MB | 1072 MB |
| Falcon3-1B, Q8_0 | 11.53 | 11.26 tok/s | 1696 MB | 1487 MB |
| Qwen2.5-1.5B, Q8_0 | 10.18 | 9.70 tok/s | 1570 MB | 1634 MB |

- SNN 在同等或更大参数量的模型（≥0.8B）中解码速度最快
- WikiText-2 困惑度 24.80 远差于所有稠密基线（8.51–15.44）

### 多线程扩展

| 线程数 | 1 | 2 | 4 | 8 | 16 |
|-------|---|---|---|---|----|
| SNN tok/s | 22.63 | 39.08 | **47.90** | 47.28 | 41.80 |

4 线程时吞吐最优（47.90 tok/s），之后因内存带宽饱和回落。

### 预填充扩展（512 tokens, 1→8 线程）

29.86 → 94.68 tok/s（3.17× 提升），预填充阶段比解码阶段具有更好的并行化收益。

### GPU 参考（RTX 2080 Ti，仅作对比）

GPU 上 SNN 解码 11.9 tok/s，WT2 PPL 24.66，功耗 140W；相比 Qwen2.5-0.5B（65W, 5.4 tok/s）功耗高但速度也更高。GPU 实验是参考性的，因为稀疏 LIF 状态更新天然不适合稠密 GPU kernel。

## Critical Assessment

本文是一篇**推理系统工程论文**，范围定义清晰，自我评价诚实。其核心贡献是一套可运行的 C++ 运行时和量化消融，而非新架构或新训练方法。

主要价值在于：
1. 验证了"混合内存布局"对脉冲模型的必要性（纯行主序和纯列主序均不如混合方案）
2. 给出了 INT4 在稀疏路径上失效的具体机制解释（per-group scale 开销 > 带宽节省）
3. 为 SNN 语言模型的 CPU 部署建立了一条基线参考路径

主要弱点在于：语言质量（困惑度）的差距太大，使得吞吐量优势的实际意义存疑——如果模型质量不足，更快的推理速度意义有限。

---

## Strengths

- **Honest reporting**: 作者没有掩盖困惑度劣势或 INT4 的负面结论，明确区分"运行时贡献"与"语言建模质量主张"。这在以自家模型为基础的小型独立研究中尤为难得。
- **Concrete systems insights**: 混合布局（dense=行主序, sparse=列主序）的设计决策有清晰的量化 ablation 支撑（Table 2, 3），不是空泛主张。
- **INT4 negative result is informative**: 稀疏架构中 INT4 因 per-group scale 开销反而更慢的结论，对其他做 SNN 量化的研究者有实际参考价值。
- **Narrow but complete scope**: 明确说明不做机器人任务评估、不做能效声明（CPU 能耗仅为利用率估算，非功率计数据），边界清晰。
- **Thread-scaling and prefill-scaling data**: 提供了详细的多线程/长提示扩展表，有助于评估在不同部署场景下的实用性。

---

## Limitations

- **单一模型，单一机器**：所有实验在一台 AMD Ryzen 7 5800X 上进行，缺乏跨硬件复现（ARM/NEON、Intel AVX-512、Apple Silicon 均未覆盖）。
- **语言质量差距显著**：WikiText-2 PPL 24.80 vs. 稠密基线 8–15，差距来源（训练数据量、上下文长度限制为 512、架构本身）未被隔离分析。没有受控的稠密 Transformer 对照实验。
- **能效声明不可信**：CPU 能耗是基于利用率估算的"估计瓦数"，而非 RAPL 或外部功率计测量，不能作为能效结论的依据。
- **代码未开源**：运行时源码、benchmark 脚本和模型导出工具尚未公开发布，可复现性依赖后续版本。
- **比较不受控**：llama.cpp 基线使用不同分词器、训练语料和上下文长度，是"部署参考"而非"架构公平比较"。
- **单一作者，独立机构**：缺乏外部同行评审，SymbolicLight V1 本身的架构主张（高激活稀疏性、训练证据）也尚未经过独立复现。

## Questions for Further Exploration

1. **模型质量瓶颈在哪里？** WT2 PPL 24.80 的原因是训练步数不足（186k steps）、512-token 上下文限制、还是架构本身？如果用同等数据和上下文训练一个稠密 Transformer，差距会缩小多少？

2. **稀疏度实际有多高？** 论文引用了 SymbolicLight V1 的激活稀疏性数据，但未在本文推理过程中直接测量各层实际脉冲激活比例。稀疏度随 token 类型或提示语言变化吗？

3. **混合布局对 ARM/Apple Silicon 是否同样有效？** AVX2 的 SIMD 宽度与 ARM NEON 不同，且 Apple Silicon 的内存带宽特性完全不同，列主序稀疏路径的收益是否可迁移？

4. **INT4 的失败能用分组更大的 per-tensor 量化缓解吗？** 现有 INT4 实验用的是 per-group，若改为 per-tensor 或 per-channel（接受更大精度损失），稀疏热循环的 scale 开销是否消失？

5. **能耗问题何时能有实测数据？** 论文指出 RAPL 或外部功率计是正确的测量方式，SymbolicLight Research 是否有计划提供 tokens/joule 的可信测量？对于"边缘部署能效"这一核心动机，这是必要证据。
