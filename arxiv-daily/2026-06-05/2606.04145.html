Key Finding — 2606.04145 · 2026-06-05 · ICML 2026

Downstream eval score — not training loss — enables precise RLHF overoptimization detection at the scheduler layer, saving compute without destroying healthy jobs

arxiv.org/abs/2606.04145 ↗ Zhang, Sun, Sarkani, Fossaceca · George Washington University
Key Metric
98.3% / 1.5%
Precision / FPR — 80% RLHF workload, 64 GPUs. Recall 99.3%. Wasted compute −21.8%.
Why This Matters
JCT-only metrics hide 64.5% false-positive rates. A system that looks best on throughput can silently destroy two-thirds of healthy RLHF jobs. Quality requires a detection axis evaluated independently of JCT.
Critical Edge
Simulator-only — never validated on real TRL or OpenRLHF training curves. Eval noise is modeled as i.i.d. Gaussian; real benchmark variance is heavy-tailed and auto-correlated. k selection in deployment is unspecified.
Methodology

Three signals during RLHF training — only one reveals overoptimization:

Training Loss Monotonically decreasing throughout — including during hacking. Indistinguishable between hacking and healthy runs. Useless
Reward Model Score Always rising — policy directly optimizes this. Cannot detect that what it's maximizing is wrong. Blind
Downstream Eval Score Rises to peak at ~55% training progress, then degrades as hacking takes over. The only signal that reveals overoptimization. Signal

EvalStop Algorithm

on eval_result(j, score): if score < prev_score[j]: d[j] += 1 // consecutive decline else: d[j] = 0 // reset on any improvement if d[j] >= k_type(j): // k=2 RLHF, k=3 DPO mark j for stop schedule(cluster, waiting, running): for j marked for stop: terminate(j); release_gpus(j); save_best_ckpt(j) return base_scheduler.schedule(updated_cluster)

Composable wrapper: EvalStop pre-processes early-stop decisions, then delegates updated cluster state to any base scheduler (FIFO, SJF-Est, SRTF-Est, LossAware) with no modification to the base policy.

Strengths / Limitations
Strengths
  • Detection-framing insight: JCT metrics alone are deceptive and mask 64.5% FPR — surfacing this is the paper's strongest contribution
  • Algorithm is ~15 lines; wraps any base scheduler without modifying it
  • E8 cleanly shows StopAt-0.65 precision = base rate exactly — not a detector
  • 5 seeds + Welch's t-test; information boundary prevents label leakage
  • Eval frequency finding is operationally actionable: invest in more frequent eval to recover more than the cost
  • Fairness verified (Jain's index within noise of SRTF-Est)
Limitations
  • Simulator only — never validated on real TRL/OpenRLHF training runs
  • Eval cost not fully modeled — 5%-interval savings may overstate net gains
  • i.i.d. Gaussian noise; real benchmark variance is heavy-tailed and autocorrelated
  • Homogeneous hardware — no elastic scaling, gradient accumulation, or heterogeneous GPUs
  • k selection procedure in deployment for novel workload types is unspecified
  • No code release — full reproduction requires reimplementation
