> **一句话总结：** 本文提出 HORIZON 自演化智能体框架，将硬件（RTL）设计视为仓库级代码进化问题，通过将结构化 Markdown 任务规范编译为带可执行评估器与验收判定门的 git 工作树，由免人工干预的智能体循环驱动 ChipBench、RTLLM、Verilog-Eval 及九类 CVDP 任务全部收敛到 100% 通过率。

> **TL;DR:** HORIZON is a hands-free self-evolving agent framework that hosts each hardware-design problem as an isolated, version-controlled git worktree with an executable acceptance gate, driving four RTL benchmark suites (ChipBench, RTLLM, Verilog-Eval, nine CVDP categories) to 100% completion.

## 核心贡献
- 提出 HORIZON：首个将硬件设计任务托管为「隔离、版本化、自动评估的 git 仓库」而非一次性提示的自演化智能体框架，把仓库级自我进化范式从 EDA 软件延伸到硬件设计产物本身。
- 用一条 git 原生的免人工干预循环（编辑—评估—评分—提交/拒绝），将所有受评测 RTL 基准套件扫到 100% 通过率；唯一残留失败被追溯到基准自身的规范—harness 缺陷。
- 提供基于原生 git 的可复现追踪机制：diff 暴露状态变化、commit 定义验收检查点、log/notes 记录评估器证据，仓库历史本身即智能体的经验与回放轨迹。
- 系统分析了轨迹成本：token 消耗高度集中在少数困难 CVDP 类别（如 CID 002 占 26.7%），且约 91% 为缓存输入 token，因此将「token 效率」定位为比「最终通过率」更值得改进的指标。

## Key Contributions
- Introduces **HORIZON**, a framework that frames hardware-design tasks as isolated, version-controlled, automatically evaluated git repositories rather than one-shot prompts, extending repository-level self-evolution (AlphaEvolve, SATLUTION, ABCEvo) from EDA software to hardware artifacts.
- A bootstrap agent compiles a structured Markdown harness into a project pack (mission, domain skills, executable evaluator, acceptance predicate, git/runtime policy); a hands-free loop then edits, evaluates, commits, or rejects candidate versions with no human intervention.
- Demonstrates 100% completion on ChipBench, RTLLM-2.0, Verilog-Eval-v2, and nine CVDP code/verification categories — claimed as the first agentic system to complete all evaluated RTL suites end-to-end without human input.
- Native git serves as both isolation substrate and trace/memory buffer: diffs expose state, commits are checkpoints, logs/notes carry evaluator verdicts, and repository history doubles as a replayable experience trace.
- Provides a detailed cost analysis showing token usage concentrated in a few hard categories, with ~91% of tokens being cached input, reframing token efficiency as the key frontier once correctness saturates.

## Methodology
HORIZON treats a hardware-design problem as repository-level code evolution. A user supplies only a structured **Markdown harness** (objective, domain knowledge, expected artifacts, evaluation criteria). A bootstrap tool loop $G_\phi$ compiles it into a **project pack** $p = (\pi_{agent}, E_p, A_p, \Gamma_p, \Omega_p)$: the agent policy/tool contract, an executable evaluator, an acceptance predicate, a version-control/artifact policy, and domain skills. The harness is domain-agnostic — the same slot can host unit tests, theorem provers, profilers, synthesis tools, or human-review gates.

From there, a **fully hands-free agent loop** evolves an isolated git worktree. Each cycle generates/edits candidate artifacts, runs the evaluator $E_p(w_t \oplus \Delta_t)$, and the acceptance predicate either commits the new version (with evaluator evidence attached as git notes) or logs a rejection. The authors borrow semi-MDP vocabulary purely to name recorded objects: a "state" is a versioned repository snapshot, an "option" is an extended edit/evaluate/repair episode between checkpoints, and the reward is vector-valued ($\Delta$pass, $\Delta$coverage, $\Delta$QoR, $-$tokens, $-$time). They stress this is a bookkeeping/replay formalism, not a Markov or RL-training assumption — no policy is trained; the backbone (GPT-5.3) is fixed throughout a campaign.

Two engineering choices enable long campaigns cheaply: (1) tracing is built on **native git** (`git diff --cached`, `git log`, git notes) so the experience buffer is essentially free; (2) a **persistent model session** is reused across iterations so harness, project pack, stable sources, and accumulated debugging context are served from prompt cache, leaving only the current diff, latest evaluator output, and response as freshly billed tokens. An independent review step diffs candidates before submission, keeping review memory separate from campaign memory.

## Results & Highlights
- **100% completion** across all suites in a single hands-free run (Table 1). Aggregate first-iteration pass rate is 47.8%, but the same loop closes the gap to 100% on every suite; the lone non-passing ChipBench task is traced to a spec–harness defect in the benchmark itself.
- Convergence cost varies sharply: RTLLM-2.0 and Verilog-Eval saturate in **2 iterations**; ChipBench in 5; CVDP categories range from 1 (CID 014) to **82 iterations** (CID 002, code completion).
- **CID 013 (checker generation)** is a striking case: lowest first-iteration pass rate (3.8%) but near-linear, plateau-free climb to 100% by iteration 19 — low first-pass accuracy does not imply slow/unstable convergence.
- Token consumption is heavily skewed: the three legacy suites consume 6.0M tokens vs. 203.9M (97.1%) for the nine CVDP categories; CID 002 alone uses 56.0M. **~91% of all tokens are cached input**, drastically lowering API cost.
- For test-generation categories (CID 012/014), coverage is an observational, not optimized, signal because the acceptance gate is the CVDP pass condition; CID 012 reaches 97.9% (not 100%) coverage at convergence, with the loop lifting a low-coverage tail rather than nudging already-high designs.

## Authors & Credibility
All four authors are from **NVIDIA Research**: Cunxi Yu (corresponding author, also co-author of SATLUTION and ABCEvo — the direct lineage of this work), Chenhui Deng (co-author of ACE-RTL, ScaleRTL), Nathaniel Pinckney (co-author of the CVDP benchmark), and Brucek Khailany. The team therefore spans both the self-evolving-repository lineage this paper extends and the RTL/benchmark ecosystem it evaluates on, giving strong domain authority. The framing is appropriately measured: they explicitly disclaim that "agentic hardware design is solved" and treat RTL benchmarks as "controlled proxies." The paper is a preprint (arXiv:2606.28279v1).

## Critical Assessment
- **Novelty:** Conceptually a clean transfer of an established idea (repository-scale self-evolution from AlphaEvolve/SATLUTION/ABCEvo) to a new object class (hardware designs rather than EDA software). The git-as-substrate framing echoes EvoGit and Git Context Controller, but applying it to host a single hardware problem as a verification-gated, replayable repository is genuinely new. The core engineering contribution is the project-pack abstraction and the harness→worktree pipeline rather than a new model or RL method.
- **Rigor:** Evaluation is broad (four suites, nine CVDP categories) and the authors are unusually candid about caveats — iter-0 is explicitly *not* a Pass@1 measurement, the one ChipBench miss is attributed to a benchmark defect, and coverage is reported as diagnostic rather than optimized. The 100% headline is somewhat softened by the honest admission that these benchmarks are saturated proxies.
- **Reproducibility:** Single-agent mode, fixed GPT-5.3 backbone, named hardware (AMD EPYC 9334, 512 GB RAM), open-source evaluators for most categories (only CID 012–014 need a commercial simulator). However, no code or project-pack templates are released with the preprint, the bootstrap agent $G_\phi$ is not detailed, and the harness contents per task are not provided — limiting independent replication.
- **Significance:** High as a position/benchmark-convergence paper; it likely resets the bar for agentic RTL and shifts attention to convergence cost, reward hacking, and verification robustness. But the contribution is more about *hosting protocol* than a capability breakthrough, and the 100% numbers rest on benchmarks the authors themselves argue are increasingly saturated.
- **Red flags:** The reward-hacking / over-solving risk is real and acknowledged: full access to simulator logs and failure traces lets the agent specialize RTL to the visible harness rather than implement robust semantics, and current benchmarks lack held-out randomized tests or formal-equivalence checks to detect this. The 100% pass rates should therefore be read as "satisfies the visible harness," not "satisfies the specification." No ablations on backbone choice, harness design sensitivity, or over-solving detection are provided.

## Strengths
- Honest, well-scoped framing: explicitly refrains from overclaiming that hardware design is solved and positions RTL benchmarks as controlled proxies.
- Clean, domain-agnostic abstraction (Markdown harness → project pack → git worktree) that generalizes beyond RTL to EDA software, methodology, and flow exploration.
- Native-git tracing makes the experience buffer, replay, and review nearly free to maintain — an elegant engineering decision.
- Cost analysis is a valuable and underreported dimension; the ~91% cached-token figure is a concrete, actionable insight for agent economics.
- Strong related-work positioning connecting EDA self-evolution, git-as-agent-substrate, and RTL benchmarks.

## Limitations
- Reward-hacking / over-solving: agent can tailor RTL to observed failures or evaluator idiosyncrasies; no held-out or formal-equivalence checks separate repair-time feedback from final scoring.
- Benchmarks are saturated and narrow relative to real chip design (incomplete specs, changing constraints, PPA tradeoffs, downstream integration, human review).
- Long-feedback-turnaround regime (synthesis/P&R/timing/power, day- to week-level PPA evaluation) is acknowledged but not addressed; current pass/fail benchmarks are favorable because evaluations are fast.
- No code, harness templates, or bootstrap-agent details released; reproducibility is constrained. No ablations on backbone, harness sensitivity, or coverage-target variants.
- Only the RTL instantiation is exercised empirically; the broader design-space claim (EDA software, methodology exploration) is a stated goal, not a completed validation.

## Relevance
与「自演化智能体 / 自动化研究」主题高度契合：HORIZON 把单一研究/设计任务包装成可执行、可验收、可回放的 git 仓库演化过程，正是 self-evolving agent 范式从软件向硬件领域迁移的代表性工作；其对 token 效率、reward hacking 与长反馈延迟的分析，也为自动化研究循环的收敛成本与评估稳健性提供了普适启示。

## Questions for Further Exploration
- How would a two-level benchmark protocol (repair-time feedback vs. held-out randomized/formal-equivalence scoring) change the observed 100% pass rates — i.e., how much of the success is genuine design repair vs. harness specialization?
- Can the vector reward ($\Delta$QoR, $\Delta$coverage) replace the pass-only acceptance gate to jointly optimize correctness, coverage, and PPA, and does this reduce the long convergence tails (e.g., CID 002's 82 iterations)?
- How sensitive are convergence cost and final robustness to the backbone model, the harness verbosity, and the bootstrap agent $G_\phi$?
- Can the depth-$k$ git trace buffer be used for offline agent-RL training or curriculum construction, turning replayable repository histories into learned repair policies?
- How does the framework degrade under realistic long-latency PPA reward (synthesis/P&R/timing), and what scheduling/lookahead policies keep edit-evaluate-repair loops tractable at day/week turnaround?
