01

Methodology

HORIZON reframes a hardware-design problem as repository-level code evolution. The user supplies only a structured Markdown harness — objective, domain knowledge, expected artifacts, and evaluation criteria. A bootstrap tool loop G_φ compiles it into a project pack.

A fully hands-free agent loop then evolves an isolated git worktree: each cycle edits candidate artifacts, runs the executable evaluator, and an acceptance predicate either commits the new version (with evaluator evidence attached as git notes) or logs a rejection. No policy is trained — the backbone (GPT-5.3) stays fixed throughout a campaign.

Two engineering choices make long campaigns cheap: tracing rides on native git (git diff --cached, git log, git notes) so the experience buffer is essentially free; and a persistent model session reuses harness, project pack, and accumulated debug context from prompt cache — leaving only the current diff and latest evaluator output as freshly billed tokens.

03

Strengths & Limitations

● Strengths

  • Honest framing — explicitly avoids claiming hardware design is solved; positions RTL benchmarks as controlled proxies.
  • Clean, domain-agnostic abstraction (Markdown harness → project pack → git worktree) that generalizes beyond RTL to EDA software and methodology.
  • Native-git tracing makes the experience buffer, replay, and review nearly free to maintain — an elegant engineering call.
  • Cost analysis as a first-class dimension; the ~91% cached-token figure is concrete and actionable for agent economics.
  • Strong related-work positioning connecting EDA self-evolution, git-as-substrate, and RTL benchmarks.

● Limitations

  • Reward-hacking / over-solving — agent can tailor RTL to observed failures or evaluator quirks; no held-out or formal-equivalence checks separate repair feedback from final scoring.
  • Benchmarks are saturated and narrow relative to real chip design (incomplete specs, PPA tradeoffs, downstream integration, human review).
  • Long-feedback regime (synthesis/P&R/timing, day-to-week PPA) is acknowledged but unaddressed; fast pass/fail benchmarks favor the system.
  • Reproducibility constrained — no code, harness templates, or bootstrap-agent G_φ details released; no ablations.
  • Only RTL instantiation validated; the broader design-space claim (EDA software, methodology) is a stated goal, not completed work.
04

Open Questions

Q1How much of the 100% is genuine design repair vs. harness specialization?›
A two-level protocol — repair-time feedback vs. held-out randomized / formal-equivalence scoring — would expose how much success is real repair versus over-fitting to the visible harness. Currently no signal separates the two.
Q2Can vector reward (ΔQoR, Δcoverage) replace the pass-only gate?›
Jointly optimizing correctness, coverage, and PPA could cut long convergence tails like CID 002's 82 iterations — and move the loop beyond binary accept/reject.
Q3How sensitive is convergence to backbone, harness verbosity, and G_φ?›
No ablations are provided. Backbone choice, harness verbosity, and the bootstrap agent's design could each swing both convergence cost and final robustness — open territory.
Q4Can the depth-k git trace feed offline agent-RL training?›
Replayable repository histories could become curricula or learned repair policies — turning the (currently bookkeeping-only) trace buffer into training data.
Q5How does the framework degrade under real day/week-latency PPA reward?›
Synthesis/P&R/timing rewards turn around over days or weeks. What scheduling or lookahead policies keep edit-evaluate-repair loops tractable at that latency remains unaddressed.
