Why this matters

First solver-agnostic, reproducible yardstick for differentiable-PDE gradients — pick and trust backends for any training, optimal-control, or inverse-problem pipeline.

Critical edge ⚠

Tasks are deliberately simpler than state-of-the-art, so the headline “gradients are good enough” may not transfer to harder, high-dimensional, or chaotic problems.

Methodology

three objects: setup · task · backend — protocol independent of solver internals

Mosaic frames evaluation around three objects: a physical setup (geometry + boundary conditions + regime), a benchmark task (a fixed optimization problem with a control variable, objective, and reference solution), and a solver backend (forward map + optional gradient).

Each backend ships as a Tesseract container exposing apply(inputs)→outputs and vector_jacobian_product(inputs, cotangents)→vjp; forward-only solvers (OpenFOAM, deal.II) get automatic finite-difference gradients as reference. Because solver APIs differ too widely for one generic wrapper, each task defines a task-specific interface schema many solvers implement — making the protocol independent of internals.

Gradient accuracy is checked against central finite differences along 6–20 random directions (cosine similarity, relative ℓ₂), with full Jacobian SVD spectra for Navier–Stokes. Performance is measured at 3–4 problem sizes on 4× V100 GPUs.

F2 drag task · solver attritionsetup compatibility gates the field

Headline insight: on the negative-control domains (H, S) all solvers converge to the same optimum; attrition appears only on the harder physics — so setup compatibility, not gradient quality, is what first constrains solver choice.

Strengths

what lands well
  • Genuinely uniform interface across 14 heterogeneous solvers spanning 4 languages and 5+ AD/adjoint strategies — swap backends via a single config string.
  • Clean experimental design with a negative-control domain (heat) that disentangles AD bugs from chaotic / ill-conditioned physics.
  • Highly practitioner-relevant findings: rollout length is the critical scaling parameter; conditioning predicts stability better than pointwise accuracy; reverse-mode AD offers the best effort-to-capability tradeoff.
  • Strong reproducibility posture: Apache 2.0, pinned lockfile, tagged reproduction tag (v0.1+paper-repro), CI-driven continuous re-evaluation.
  • Honest about scope — acknowledges solver-version pinning, V100 specificity, and the standalone-task proxy assumption.

Limitations

where it strains
  • Tasks are deliberately simpler than SOTA challenges — the “gradients are good enough” conclusion may not extrapolate to harder, ill-posed, or high-dimensional problems.
  • Relies on central finite differences as ground truth, itself sensitive to perturbation size (acknowledged but inherent); ε-tuning unstable on ill-conditioned problems.
  • Results are hardware-specific (4× V100) and pinned to specific solver versions — relative rankings may shift on CPU or newer GPUs.
  • No learned / hybrid components tested, so failure modes of solver-in-the-loop ML pipelines are not captured.
  • Only four domains; multiphysics, compressible flow, and Enzyme/LLVM-IR-level AD backends left to future work. The bespoke-LOC integration metric is coarse and excludes glue code.

Open Questions

where the work points next
01Do “gradients good enough” and “conditioning predicts stability” hold on harder, higher-dimensional, or chaotic (e.g. turbulence) problems beyond the tested setups?
02How do rankings shift on modern GPUs (H100) and CPU-heavy backends, given all timings were V100-only?
03What failure modes appear when these solver gradients are embedded in solver-in-the-loop ML training versus the isolated optimization tasks tested here?
04Can the finite-difference reference be replaced or augmented by an analytical or higher-precision reference for ill-conditioned problems where ε tuning is unstable?
05How does Enzyme / LLVM-IR-level AD compare to framework-level (JAX/PyTorch) and hand-derived adjoints when added as backends?
