The Speedup Paradox: Rethinking Inference Speed-Quality Trade-off in Embodied Tasks

Yujin Wang et al. — Tsinghua University · 6 authors · closed-loop inference analysis

Faster per-step inference can backfire: in embodied control, lossy speedups produce non-monotonic task-level effects — and the sweet spot drifts across hardware.

一句话总结　在具身任务的闭环执行中，有损推理加速（量化／剪枝／异步推理）并不单调改善任务级表现：静态任务的端到端时间与动态任务的成功率都会出现非单调"甜点"，且甜点位置会随部署硬件漂移。

Why this matters

Per-step latency is a systematically misleading proxy for embodied inference efficiency. Benchmarks must report rollout length, observation freshness, and the deployment hardware.

Critical edge

Real-world validation is limited to a single UR5e and two task families, and the dynamic-success model Φ(q,z) lacks empirical calibration — so some paradox cases may not generalize uniformly.

01

Methodology — TISED

TISED (Task-level Inference Speedup Effect Decomposition) places policy-intrinsic and execution-aware optimizations on one shared closed-loop time axis. Each cycle is one policy inference followed by n executed actions, repeated until rollout termination. It decomposes end-to-end effects into per-chunk speedup, rollout length N, and action delay Dact.

per-chunk time Tchunk = Tinf + n · Tact policy-intrinsic (quant / prune / sampling-step reduction), latency ratio α ∈ (0,1] Tplcycle = α · Tinf + n · Tact execution-aware async (overlap depth n′) Teacycle = Tinf + n · Tact − min{ Tinf, n′ · Tact }

End-to-end task time is Ttask = N · Tcycle, where the chunk count N is an outcome, not a constant — it grows monotonically with lightweighting strength as action quality degrades. For dynamic tasks, success is modeled as Psucc = Φ(q, z) with staleness exposure z = Dact · ωenv, trading action quality against observation freshness. A progressive-lightweighting sweep is then run across task types, model backbones, optimization methods, and compute platforms.

03

Strengths & Limitations

Strengths

  • Unifies heterogeneous techniques (quant / prune / sampling / async) under one principled closed-loop timing framework.
  • Cleanly explains why per-step metrics fail via rollout-length inflation, observation staleness, plus hardware dependence.
  • Cross-hardware study is rare and directly actionable — the same config can be optimal or harmful by platform.
  • Combines simulation breadth (5 benchmarks, 6 models) with real-world UR5e physical evidence.

Limitations

  • Abstracts away second-order effects: scheduling jitter, OS latency, queueing in production async schedulers.
  • Static chunk count N modeled from action quality only; motion smoothness observed, not formalized.
  • Dynamic surrogate Φ(q,z) not empirically validated as a specific functional form.
  • High-frequency dynamic tasks (e.g. table tennis) out of scope; real-world validation = one UR5e, two task families.
04

Open Questions

  1. How do the paradoxes behave under adaptive online lightweighting that adjusts α or n′ from detected dynamics within a single rollout?
  2. Can the hardware-dependent sweet spot be predicted a priori from ρ and ωenv — enabling auto-tuning at deployment without the full sweep?
  3. Does the framework extend to multi-agent / shared-compute settings where policies contend for one accelerator (queueing effects)?
  4. Would learned / stochastic Φ(q,z) surrogates fit from data shift the predicted sweet spots versus the monotonic-derivative assumption?
  5. Do the findings transfer to locomotion / whole-body control with continuous high-frequency dynamics, beyond manipulation?
  6. Could a single benchmark protocol — reporting per-task time/success across a hardware ladder — become a community standard for embodied inference papers?
