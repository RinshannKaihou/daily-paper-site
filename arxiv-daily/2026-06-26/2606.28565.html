Methodology

Roofline × Efficiency · Discrete-Event

KernelSight-LM fuses a kernel-level latency predictor with Vidur's discrete-event serving simulator. Each kernel's cost is modeled as t = max(t_roof·η, t₀), where the roofline bound is computed entirely from datasheet peaks (compute, bandwidth, SM count) plus a wave-quantization correction.

The decisive move is a clean factorization: the roofline absorbs all hardware scaling, so the only quantity that must transfer across GPU generations is a single learned, dimensionless efficiency factor η — trained on portable features (arithmetic intensity, occupancy, wave count, L2 residency) with low-capacity per-family heads selected from a small pool (regularized linear, GBDT, small MLP, and a bounded analytical form adapted from NeuSight).

Per-kernel estimates aggregate up the transformer hierarchy, then combine with an explicit host-overhead model capturing CPU/GPU overlap. The simulator faithfully replays vLLM v1's token-budget scheduling loop — continuous batching, chunked prefill, two-tier prefix cache — pricing mixed steps kernel-by-kernel. Tier B adds a density-routed measured efficiency grid (falling back to the learned model outside the convex hull) and calibrates host constants from a short serving sweep.

The two tiers share one roofline×efficiency formula: Tier A = datasheet-only extrapolation; Tier B = one-shot model-agnostic microbenchmark refinement.

Core Equations

t = max(troof·η, t₀)
tstep = max(tgpu + n·cdisp, ℓ) + max(0, covl − tgpu) + ctail
t_roof — datasheet roofline bound + wave-quantization u
η — learned dimensionless efficiency (the only transferred qty)
t₀ — portable launch floor
c_disp / c_ovl / c_tail — host overhead: dispatch, overlap, tail sync

Strengths  ⇄  Limitations

Critical Assessment

Strengths

  • 01
    Unified two-tier system spanning profiling-free extrapolation to high-fidelity in-distribution prediction within one roofline×efficiency formula.
  • 02
    Faithful vLLM v1 modeling — continuous batching, chunked prefill, prefix cache — plus a host-overhead channel prior simulators omit. Scheduler tracks real engine at Spearman +0.985.
  • 03
    Honest limitation analysis quantifying exactly which kernel families transfer across generations and which don't.
  • 04
    Production fused kernels (FlashAttention) instead of isolated synthetic operators — captures real engine dispatch.
  • 05
    Distributional error reporting (p50/p90/p99/max) across a wide model/hardware matrix on a realistic production trace.

Limitations

  • 01
    Attention η doesn't transfer. It's large (~3–3.5) and varies up to 3.8× across devices — closing the gap needs ≥1 serving-matched measurement per new architecture.
  • 02
    Per-kernel ≠ end-to-end. GEMM dominates runtime and is only a tie, so realized gains are smaller than per-kernel means suggest.
  • 03
    TTFT collapses near saturation: error rises from ~15% at ρ≲0.9 to 74% at ρ≈1.2 as queue buildup amplifies per-step error.
  • 04
    Single held-out generation. Only GB200 used as target — broader cross-gen validation (future architectures) is untested.
  • 05
    No code/data release mentioned; the evaluation trace is proprietary, limiting independent reproduction.

Open Questions

For Further Exploration
Q.01 · TRANSFERABILITY

Can attention efficiency η be made transferable via richer dimensionless features (achieved-bandwidth ceilings, cache-residency distributions) — or does it fundamentally require per-architecture calibration?

Q.02 · NEW ARCHITECTURES

How does the predictor degrade on architectures with qualitatively different memory hierarchies — GDDR7/LPDDR5X edge parts, 3D-stacked logic-in-memory?

Q.03 · SOURCE LADDER

What is the minimum source-GPU "ladder" size & diversity needed to keep cross-generation error below a target threshold?

Q.04 · SATURATION REGIME

Can the host-overhead model be extended to accurately cover the saturated regime (ρ≳1) where TTFT error explodes to 74%?

Q.05 · NEW ATTENTION

Does the kernel-level breakdown shift meaningfully under Mamba/SSM, grouped-query, sliding-window attention, or speculative decoding?

Q.06 · CLOSED-LOOP

Can the simulator couple with an optimizer for closed-loop auto-tuning of TP, batching & prefix-cache — vs. Bayesian-optimization config search?
