Does the Same Token Mean the Same State? MoE Routing as Signal for Reasoning Control

[Kang Chen, Minshen Yu, Junjie Nian, Yaoning Wang, Yixin Cao, Yugang Jiang]

arXiv: 2606.22798

Date: 2026-06-22

Tags: [mixture-of-experts, test-time-compute, self-consistency, routing-analysis, multi-rollout-reasoning]

https://arxiv.org/abs/2606.22798

Summary / 概述

This paper investigates whether Mixture-of-Experts (MoE) routing patterns can serve as an answer-string-free signal for selecting among multiple reasoning rollouts at test time. The authors discover that even when the same token id appears at repeated anchor positions across rollouts, the experts that produce it differ -- retaining information about problem identity, reasoning trajectory, and effort level. They operationalize this finding into Routing Agreement Decoding (RAD), a method that selects the rollout whose routing neighborhood is densest (most agreed-upon) without ever parsing, normalizing, or voting over answer strings. Across 10 sparse-MoE configurations (gpt-oss, Qwen3-MoE) and 6 datasets (math, GPQA, code), RAD matches textual Majority voting where string voting works well, and provides a meaningful selection signal on code generation and agentic SWE-bench tasks where exact-string voting degenerates.

本文研究了 Mixture-of-Experts (MoE) 路由模式能否作为一种无需解析答案字符串的信号，用于在测试时从多条推理 rollout 中做出选择。作者发现，即使在不同 rollout 的重复锚点位置出现相同的 token id，生成该 token 的专家组合却不同——仍然保留了问题身份、推理轨迹和推理力度等级的信息。基于此发现，他们提出了 Routing Agreement Decoding (RAD)，一种在不解析、不归一化、不对答案字符串投票的前提下，选择路由邻域最稠密（共识度最高）rollout 的方法。在 10 种稀疏 MoE 配置（gpt-oss、Qwen3-MoE）和 6 个数据集（数学、GPQA、代码）上，RAD 在字符串投票有效的场景中与文本多数投票持平，而在精确字符串投票退化的代码生成和智能体 SWE-bench 任务中提供了有意义的选择信号。

Key Findings / 主要发现

  1. Same token, different experts (Insight 1): At fixed anchor token ids (e.g., Qwen EOS <|im_end|>, OSS boundary triplet <|channel|>final<|message|>), the generating MoE routes still separate problem identity (< 5% cross-problem leakage) and reasoning-effort mode (< 1% cross-level contamination). Token identity is therefore not a sufficient statistic for router state.

相同 token，不同专家（洞察 1）： 在固定锚点 token id 处，生成该 token 的 MoE 路由仍然能区分问题身份（跨问题泄漏 < 5%）和推理力度模式（跨等级污染 < 1%）。因此，token 身份不是路由状态的充分统计量。

  1. Routing neighborhoods align with answer basins (Insight 2): Near boundary and delimiter anchors, rollouts sharing the same normalized final answer cluster together in routing space. The alignment is strongest at delimiter@16 (a 16-token window starting at the answer-opening delimiter such as \boxed{), where chance-corrected K-NN purity reaches 0.86 on math datasets.

路由邻域与答案盆地对齐（洞察 2）： 在边界和定界符锚点附近，共享相同归一化最终答案的 rollout 在路由空间中聚类。该对齐在 delimiter@16（从答案开始定界符如 \boxed{ 起的 16 个 token 窗口）处最强，数学数据集上机会校正后的 K-NN 纯度达到 0.86。

  1. RAD competitive with Majority on well-posed tasks: On the well-posed math+GPQA regime, RAD scores 73.9 vs. Majority 73.6 (pooled problem-weighted average across all 10 configurations). The paired delta is +0.28 pp (bootstrap 95% CI [-0.50, +1.07], McNemar p=0.52) -- small and not statistically significant, but positive.

RAD 在定义明确的任务上与多数投票持平： 在定义明确的数学+GPQA 场景下，RAD 得分 73.9 vs. 多数投票 73.6（所有 10 种配置的加权平均）。配对差异为 +0.28 pp（95% CI [-0.50, +1.07]，McNemar p=0.52）——小且不显著，但为正。

  1. RAD excels where string voting fails: On LiveCodeBench (code generation), where code outputs are near-unique and exact-string Majority/DeepConf-vote degenerate to singletons, RAD and RAD+DC still produce meaningful pass@1 selections (60.5 and 61.3 for gpt-oss-20B), improving over the Avg@64 random-rollout floor.

RAD 在字符串投票失效处表现突出： 在 LiveCodeBench（代码生成）上，代码输出近乎唯一使得精确字符串多数投票/DeepConf 投票退化为单例，而 RAD 和 RAD+DC 仍然能产生有意义的 pass@1 选择（gpt-oss-20B 上分别为 60.5 和 61.3），超过 Avg@64 随机基线。

  1. Agentic SWE-bench Verified: On best-of-16 patch selection (N=16 rollouts x 500 problems per model), RAD improves over random selection by +4.7/+4.8/+5.0 pp on the decidable subset across three models, and by +2.2 to +2.9 pp end-to-end. This uses only the boundary anchor (thinking-to-action transition), with no answer string to vote on.

智能体 SWE-bench Verified： 在 best-of-16 补丁选择（每模型 N=16 rollout x 500 问题）上，RAD 在三个模型的可判定子集上相比随机选择分别提升 +4.7/+4.8/+5.0 pp，端到端提升 +2.2 至 +2.9 pp。这仅使用边界锚点（思考到行动的转换），无需答案字符串投票。

Method / 方法

Anchor-Conditioned Routing: RAD aligns rollouts at fixed token positions called anchors. Three types are used:

  • Trajectory anchors (e.g., So, Now): recurring discourse markers in reasoning traces, read cheaply and near-universally.
  • Boundary anchors (e.g., , <|channel|>final<|message|>): final-response transition markers.
  • Delimiter anchors (e.g., \boxed{, code fences): fixed answer-opening delimiters.

锚点条件路由： RAD 在称为锚点的固定 token 位置对齐各 rollout。使用三种类型：

  • 轨迹锚点（如 So、Now）：推理轨迹中的反复出现的话语标记。
  • 边界锚点（如 、<|channel|>final<|message|>）：最终响应转换标记。
  • 定界符锚点（如 \boxed{、代码围栏）：固定的答案开始定界符。

Routing Representation: For each rollout, RAD reads the sparse MoE routing vector (selected expert ids + post-softmax weights) in a W=16 token window starting at the delimiter anchor. These per-token sparse vectors are averaged into a per-rollout route vector z_i. Pairwise Weighted-Jaccard similarity forms an N x N consensus matrix S.

路由表示： 对于每个 rollout，RAD 读取从定界符锚点开始的 W=16 token 窗口内的稀疏 MoE 路由向量（选中的专家 id + softmax 后权重），将逐 token 稀疏向量平均为每个 rollout 的路由向量 z_i。成对加权 Jaccard 相似度构成 N x N 共识矩阵 S。

Selection via K-NN Density: For each rollout i, RAD computes a local agreement density q_i = sum of similarities to its k_rad nearest neighbors, then selects i* = argmax q_i. The selected rollout's answer string is read only after this routing-only selection is made. RAD never parses, normalizes, or votes over answer strings during selection.

K-NN 密度选择： 对每个 rollout i，RAD 计算局部一致密度 q_i = 其 k_rad 个最近邻的相似度之和，然后选择 i* = argmax q_i。被选中 rollout 的答案字符串仅在此纯路由选择完成后才被读取。RAD 在选择过程中从不解析、归一化或对答案字符串投票。

RAD+DC (Confidence Fusion): Combines RAD with DeepConf's per-rollout scalar confidence score. Rollouts are first filtered to the highest-confidence half (by cBW), then RAD density is computed within the retained subset. This fuses two near-orthogonal signals (consensus + quality).

RAD+DC（置信度融合）： 将 RAD 与 DeepConf 的逐 rollout 标量置信度分数结合。先将 rollout 按置信度过滤至最高的一半（按 cBW），然后在保留子集内计算 RAD 密度。这融合了两个近乎正交的信号（共识 + 质量）。

Experimental Setup / 实验设置

  • Models: 10 sparse-MoE configurations spanning 4 families: gpt-oss-20B (32 experts, top-4), gpt-oss-120B (128 experts, top-4), Qwen3-30B-A3B (128 experts, top-8), Qwen3-Next-80B-A3B (512+1 shared experts, top-10).
  • Datasets: AIME24/25, BRUMO25, HMMT25 (math reasoning), GPQA (science QA), LiveCodeBench v5 (code generation).
  • Scale: 4,850 model-problem cells, 310,400 total rollouts, N=64 rollouts per problem.
  • Baselines: Random (Avg@64), textual self-consistency Majority (Cons@64), DeepConf confidence-weighted vote.
  • Routing capture: Top-K expert ids and post-softmax weights captured during vLLM generation, adding ~7% overhead.
  • 模型： 4 个家族的 10 种稀疏 MoE 配置：gpt-oss-20B、gpt-oss-120B、Qwen3-30B-A3B、Qwen3-Next-80B-A3B。
  • 数据集： AIME24/25、BRUMO25、HMMT25（数学推理），GPQA（科学问答），LiveCodeBench v5（代码生成）。
  • 规模： 4,850 个模型-问题单元，310,400 个总 rollout，每个问题 N=64 个 rollout。
  • 基线： 随机（Avg@64）、文本自洽多数投票（Cons@64）、DeepConf 置信度加权投票。
  • 路由捕获： 在 vLLM 生成时捕获 Top-K 专家 id 和 softmax 后权重，增加约 7% 开销。

Critical Assessment / 批判性评估

Strengths / 优势:

  • The core observation is genuinely novel: identical tokens at anchor positions are generated by different expert ensembles that retain upstream context, making routing a richer signal than token identity alone.
  • The answer-string-free interface is a real practical advantage. Many important domains (code, agentic workflows) have no well-defined answer string to vote on, making traditional self-consistency inapplicable. RAD provides a principled alternative.
  • Thorough experimental coverage across 10 model configurations and 6 datasets, with careful statistical reporting (bootstrap CIs, McNemar tests, exact random envelopes).
  • The strict delimiter@marker readout variant shows that most of the signal comes from routing at the marker tokens themselves (no answer content), recovering 71.1 out of the full 73.9 RAD score.
  • 核心观察确实新颖：锚点位置的相同 token 由保留上游上下文的不同专家集合生成，使路由成为比 token 身份更丰富的信号。
  • 无需答案字符串的接口是真正的实用优势。许多重要领域（代码、智能体工作流）没有定义明确的答案字符串可供投票，使传统自洽方法不适用。RAD 提供了一种有原则的替代方案。
  • 在 10 种模型配置和 6 个数据集上的全面实验覆盖，附有严谨的统计报告。
  • 严格的 delimiter@marker 读取变体表明，大部分信号来自标记 token 本身的路由（无答案内容）。

Weaknesses / 不足:

  • On the well-posed math+GPQA regime where string voting works, the advantage over textual Majority is statistically insignificant (+0.28 pp, p=0.52). The method's distinctive value is limited to the string-voting-degenerate regime.
  • RAD requires white-box MoE-router access. It does not apply to dense models or closed APIs -- a significant practical limitation given that most deployed frontier models are accessed via API.
  • RAD is not a verifier -- a dense wrong routing basin can still win, exactly as textual Majority can follow a dense wrong consensus. The S-curve analysis (Figure 4 right) shows RAD is below chance when correct answers are a minority.
  • The agentic SWE-bench result is small in scale (best-of-16 on one rollout pool per model) and still leaves a wide gap to the any-of-16 oracle (69.8/75.8% vs. deployed 51.4/57.8%).
  • The computational overhead, while modest (~7% at generation), requires storing full routing tensors for all rollouts (~1.2 TB for the full experimental pool), which may not be practical at scale.
  • 在字符串投票有效的定义明确的数学+GPQA 场景下，相对于文本多数投票的优势在统计上不显著（+0.28 pp，p=0.52）。该方法的独特价值局限于字符串投票退化的场景。
  • RAD 需要白盒 MoE 路由器访问权限，不适用于密集模型或封闭 API——鉴于大多数部署的前沿模型通过 API 访问，这是重大的实际限制。
  • RAD 不是验证器——稠密的错误路由盆地仍可能获胜。S 曲线分析显示，当正确答案是少数时，RAD 低于随机基线。
  • 智能体 SWE-bench 结果规模较小，与 any-of-16 oracle 之间仍有较大差距。
  • 计算开销虽适中（生成时约 7%），但需要存储所有 rollout 的完整路由张量（约 1.2 TB），在规模化时可能不实际。

Relevance / 相关性

This work is relevant to anyone working on test-time compute scaling, multi-rollout reasoning, or MoE model interpretability. The key insight that MoE routing carries rollout-level semantic structure beyond token identity opens a new angle for using internal model signals as a selection mechanism. The answer-string-free property makes RAD particularly interesting for code generation and agentic applications where traditional self-consistency cannot be applied directly. The slogan captures the conceptual contribution well: "Majority counts answer strings; RAD aggregates routing agreement."
