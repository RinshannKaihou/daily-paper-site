# Litmus: Zero-Label, Code-Driven Metric Specification for Evaluating AI Systems

## Summary / 摘要

**English**: Litmus is a zero-label system that designs evaluation and monitoring metrics for AI pipelines by eliciting evaluation intent from source code and targeted interrogation, rather than assuming the evaluation target is already known. It first identifies what must be measured and why, then converts those answers into constraints for constructing a justified, per-stage metric portfolio. Evaluated on three real, code-defined AI pipelines -- financial account grouping, scientific QA, and inherent risk assessment -- Litmus achieves the broadest concern coverage, near-zero redundancy, and ranks first in validity (agreement with per-row quality labels) on all three pipelines, despite using no labels at design time.

**中文**: Litmus 是一个零标签系统，通过从源代码中提取评估意图并进行针对性质询，为 AI 流水线设计评估和监控指标，而非假设评估目标已知。它首先识别需要测量什么以及为什么，然后将这些答案转化为约束条件，用于构建有理有据的、按阶段划分的指标组合。在三个真实的、代码定义的 AI 流水线（金融账户分类、科学问答、固有风险评估）上的评估表明，Litmus 在关注点覆盖率上最广，冗余度接近零，且在所有三个流水线上的有效性（与逐行质量标签的一致性）排名第一，尽管在设计阶段未使用任何标签。

## Key Contributions / 主要贡献

1. **Reframing metric design as goal elicitation by interrogation / 将指标设计重新定义为目标引导式质询**: Rather than selecting metrics post-hoc, Litmus connects metric validity concerns to clarification-question generation and inquiry-based uncertainty resolution. The question that precedes "which metric?" is "what is this component supposed to do, how can it fail silently, and what is therefore worth measuring?"
   - 不再事后选择指标，而是将指标有效性问题与澄清性提问和基于质询的不确定性消解相连接。在"选哪个指标"之前先问"这个组件应该做什么、可能如何静默失败、因此值得测量什么"。

2. **Litmus system with five-phase pipeline / 五阶段 Litmus 系统**: A zero-label system that derives a justified, per-stage evaluation and monitoring portfolio from source code, with practitioner answers and internal validity checks represented as explicit metric-design constraints. The five phases are: (1) Architecture Reconstruction, (2) Pattern Classification, (3) Interrogation, (4) Metric Design, (5) Traceability & Export.
   - 一个零标签系统，从源代码出发导出有理有据的、按阶段划分的评估与监控指标组合。从业者的回答和内部有效性检查被表示为显式的指标设计约束。五个阶段为：架构重建、模式分类、质询、指标设计、可追溯性与导出。

3. **New portfolio-level evaluation axes / 新的组合级评估维度**: Beyond standard validity (label agreement), the paper introduces coverage (fraction of domain-specific failure concerns addressed), grounding (how concretely metrics reference implementation artifacts), and redundancy (fraction of metric pairs measuring the same thing) as axes for characterizing metric-design system outputs.
   - 除了标准的有效性（标签一致性），论文引入了覆盖率（领域特定失败关注点的覆盖比例）、扎根度（指标对实现制品的具体引用程度）和冗余度（测量相同内容的指标对比例）作为评价指标设计系统输出的维度。

## Methodology / 方法论

### Architecture Reconstruction / 架构重建
A deterministic scanner parses the codebase, builds import and symbol/call graphs, computes centrality scores, and selects salient files for LLM analysis. An LLM synthesizes an 8-20 node component graph describing the system's major modules, their roles, AI capabilities, and associated risk surfaces. An adversarial critic then validates this graph against code evidence, pruning phantom components and flagging missed ones.

确定性扫描器解析代码库，构建导入和符号/调用图，计算中心性分数，并选择关键文件供 LLM 分析。LLM 合成一个 8-20 节点的组件图，描述系统的主要模块、角色、AI 能力和相关风险面。对抗性评审器随后根据代码证据验证该图，裁剪幻影组件并标记遗漏组件。

### Pattern Classification / 模式分类
Each component is mapped to an evaluation taxonomy of pipeline-stage patterns (data acquisition, knowledge base, AI core, assurance) and cross-cutting strategies (e.g., RAG, agentic orchestration). This narrows the design space by flagging which metric families are plausibly relevant per component.

每个组件被映射到流水线阶段模式（数据获取、知识库、AI 核心、保障）和跨切面策略（如 RAG、智能体编排）的评估分类体系。这通过标记每个组件可能相关的指标族来缩小设计空间。

### Interrogation / 质询
The core innovation: Litmus generates two channels of questions. **Channel A** produces practitioner-facing clarification questions targeting under-specified decisions (e.g., which processing tier carries production traffic, whether a fallback path fires often or rarely). **Channel B** produces internal adversarial validity questions that challenge each candidate metric on pipeline fit, data assumptions, measurement validity, and direction of goodness. Answers become **confirmed facts** -- hard constraints that admit, reject, or re-scope metrics.

核心创新：Litmus 生成两个通道的问题。通道 A 产生面向从业者的澄清性问题，针对未充分指定的决策（如哪个处理层承载生产流量、回退路径是否经常触发）。通道 B 产生内部对抗性有效性问题，质疑每个候选指标在流水线适配性、数据假设、测量有效性和改善方向上的合理性。答案成为"已确认事实"——用于接受、拒绝或重新界定指标的硬约束。

### Metric Design and Export / 指标设计与导出
Conditioned on the reconstructed architecture and confirmed facts, the designer emits a small number of high-signal metrics per component: deterministic checks, LLM-as-judge metrics (scored on a five-level rubric with evidence-backed rationale), and composite metrics. Each metric specifies its evaluated component, purpose, required data, computation, direction of goodness, and validity conditions. Litmus also maps each evaluation metric to a production monitoring counterpart.

基于重建的架构和已确认事实，设计器为每个组件生成少量高信号指标：确定性检查、LLM-as-judge 指标（使用有证据支持的五级评分标准）和复合指标。每个指标指定其评估的组件、目的、所需数据、计算方法、改善方向和有效性条件。Litmus 还将每个评估指标映射到生产监控对应物。

## Key Results / 关键结果

Evaluated on three real pipelines against AutoMetrics and three DynamicRubric baselines:

| Axis | Litmus Performance |
|------|-------------------|
| **Validity** (Spearman rho) | Ranks first on all three pipelines: rho = 0.51 (account grouping), 0.72 (scientific QA), 0.32 (risk assessment) vs. best baseline rho = 0.47, 0.27, 0.28 |
| **Coverage** | Top or tied on all pipelines (0.80, 1.00, 0.83) |
| **Grounding** | Top on P1 and P3 (0.45, 0.42); close second on P2 (0.32 vs. 0.35) |
| **Redundancy** | Near-zero throughout (0.02, 0.01, 0.05) vs. baselines up to 0.35 |
| **Degradation sensitivity** | Highest on P1 (0.77) and P3 (0.84); competitive on P2 (0.94 vs. 0.95) |

The single-vector validity statistic (stitched per-row score) preserves this ranking with Litmus first and individually significant throughout (p < 0.001), though on P1 and P3 its confidence interval overlaps the strongest baseline.

在三个真实流水线上对比 AutoMetrics 和三种 DynamicRubric 基线进行评估。Litmus 在所有三个流水线上有效性排名第一（Spearman rho 分别为 0.51、0.72、0.32），覆盖率最高或并列最高，冗余度接近零，且是唯一在所有三个流水线上同时在有效性和退化敏感性上表现强劲的系统。

## Technical Details / 技术细节

- **Pipeline scope**: Litmus emits per-stage portfolios (14 metrics for account grouping, 13 for scientific QA, 14 for inherent risk assessment) including monitoring metrics, scope-matched LLM-as-judge metrics, and code-evaluated checks -- unlike baselines that score only the final output.
- **Architecture verification**: For PaperQA2 (scientific QA), Litmus recovers a 30-node graph that is 65% edge-verified against the tree-sitter call graph, maps every recovered module to real code (coverage F1 1.00, zero phantoms), and earns an adversarial-critic confidence of 0.78.
- **Graph quality for account grouping**: Of 49 proposed edges, 32 are verified, 1 inferred, and 16 unsupported (0 discarded). Critic confidence: 0.78 (lowest node: 0.65). Zero phantom nodes pruned.
- **Scoring protocol**: All LLM-based scoring uses temperature 0 with up to three retries. The same judge-model family is used across all systems to keep scoring consistent, though this introduces possible judge-family alignment bias.

- **流水线范围**：Litmus 为每个阶段生成指标组合（账户分类 14 个、科学问答 13 个、风险评估 14 个），包括监控指标、范围匹配的 LLM-as-judge 指标和代码评估检查，而基线仅对最终输出打分。
- **架构验证**：对 PaperQA2 流水线，Litmus 恢复了一个 30 节点的图，65% 的边通过 tree-sitter 调用图验证，覆盖率 F1 为 1.00，无幻影节点。
- **评分协议**：所有基于 LLM 的评分使用温度 0，最多三次重试，所有系统使用同一判断模型族。

## Critical Assessment / 批判性评估

**Strengths / 优势**:
- The conceptual reframing of metric design as interrogation-driven goal elicitation is well-motivated and practically useful. It addresses a real gap: most automatic evaluation systems assume the evaluation target is already known, which is rarely true for production pipelines.
- The zero-label constraint is significant -- Litmus achieves stronger validity than label-fitted AutoMetrics without consuming any labels at design time.
- The per-stage metric portfolio (vs. single final-output scores) is genuinely more useful for production debugging, monitoring, and audit. This is a meaningful architectural difference from baselines.
- The paper is intellectually honest about limitations and avoids overclaiming.

**Weaknesses / 不足**:
- All evaluation uses a single judge-model family, risking judge-system alignment bias. The paper acknowledges this but has not yet conducted multi-judge-family replication.
- The three evaluation pipelines, while real, are all scored by one assessor family; generalization to other domains and judge models remains unvalidated.
- Coverage is monotone in portfolio size -- Litmus emits 14 metrics vs. 1-8 for baselines, and the reference concern lists were authored by the paper's authors, creating a potential circularity. The coverage gap is better read as an upper bound.
- The clarification-question channel (practitioner answers shaping metrics) is established at the specification level but its end-to-end benefit (whether answers measurably change the resulting metrics) has not been tested in a human study.
- On P1 and P3, Litmus's validity confidence intervals overlap the strongest baseline, so the advantage is a favorable ranking rather than a uniformly significant margin.

- **优势**：将指标设计重新定义为质询驱动的目标引导是切实有用的概念创新；零标签约束下仍优于使用标签的基线；按阶段的指标组合对生产环境更有实用价值；论文对局限性的讨论诚实充分。
- **不足**：仅使用单一判断模型族，存在评判偏差风险；覆盖率与组合规模单调相关且参考关注列表由作者自行编写；澄清性提问通道的端到端效果未经人类研究验证；部分流水线上有效性置信区间与最强基线重叠。

## Relevance to My Research / 与我研究的相关性

This paper is relevant to anyone building or evaluating production LLM pipelines. The key insight -- that evaluation should begin with goal elicitation rather than metric selection -- applies broadly to agentic systems, RAG pipelines, and multi-stage AI workflows. The interrogation-driven approach to surfacing implicit assumptions about what should be measured could complement existing evaluation frameworks. The portfolio-level axes (coverage, grounding, redundancy) provide a useful vocabulary for assessing metric-design quality beyond simple label agreement.

本文与构建或评估生产 LLM 流水线的研究者相关。核心洞察——评估应从目标引导而非指标选择开始——广泛适用于智能体系统、RAG 流水线和多阶段 AI 工作流。质询驱动的方法可用于揭示关于应测量什么的隐含假设。组合级评估维度为评价指标设计质量提供了有用的词汇。
