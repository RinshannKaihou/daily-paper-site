> **一句话总结：** 提出Darwin Mobile Agent开源基础设施，通过云端Android异步并行环境和基于GiGPO/VERL的多轮RL框架，对UI-TARS-7B进行策略优化，在SPA-Bench任务上实现稳定的自我进化学习，并提出从任务课程、结果验证、记忆管理三个维度逐步去除人工先验的三阶段路线图。

> **TL;DR:** Darwin Mobile Agent is an open-source infrastructure for training self-evolving mobile GUI agents via reinforcement learning. It uses cloud-hosted Android instances with an asynchronous rollout architecture to overcome the data-collection bottleneck, fine-tunes UI-TARS-7B with PPO (built on GiGPO/VERL), and demonstrates stable policy improvement on SPA-Bench tasks (success rate ~0.2 to ~0.8). The paper further proposes a three-stage roadmap for progressively removing human priors from task curricula, outcome verification, and agent memory -- though only stage 1 (human-designed) is implemented.

## 核心贡献
- 提出首个端到端开源系统，通过强化学习训练移动GUI智能体，使用云端Android集群实现异步数据收集，解决移动环境高延迟、高变异性导致的数据收集瓶颈
- 设计基于Gymnasium的异步移动环境接口，将环境Worker与策略推理解耦，支持8-32台云手机的弹性扩展，且在off-policy数据增加时保持学习稳定性
- 提出自我进化智能体的"功能解剖学"（functional anatomy），将任务课程、结果验证和智能体状态（记忆）定义为三大核心组件，并为每个组件规划了从人工设计到外部模型再到完全内化的三阶段演进路线
- 引入任务生命周期协议（Task Lifecycle Protocol），将交互分为Setup/Execution/Teardown三阶段，使智能体能学习长期复杂工作流而无需外部重置
- 发现critic初始化偏差和bootstrapping策略对多轮RL训练稳定性的关键影响：校准初始化（bias=0.1）可防止策略崩溃，将截断轨迹视为终止状态优于从learned critic进行bootstrapping

## Key Contributions
- First open-source end-to-end system for training mobile GUI agents via RL, using cloud Android clusters with asynchronous data collection to overcome the mobile environment's high-latency bottleneck
- Gymnasium-based asynchronous mobile environment that decouples environment workers from policy inference, supporting elastic scaling (8-32 cloud phones) with consistent learning despite increased off-policy data
- "Functional anatomy" of a self-evolving agent: task curriculum, outcome verification, and agent state (memory), each with a three-stage roadmap from human-designed to externally-automated to fully internalized
- Task Lifecycle Protocol decomposing interaction into Setup/Execution/Teardown phases, enabling agents to learn complex long-horizon workflows without artificial external resets
- Key finding on critic stability: calibrated initialization (bias=0.1, sigma=0) prevents policy collapse; treating truncated trajectories as terminal (V(s')=0) outperforms bootstrapping from a learned critic

## Methodology

**Environment.** The system uses cloud-hosted Android instances controlled via ADB, wrapped in a Gymnasium interface. Observations are raw pixel screenshots plus natural-language task instructions. The action space includes click, long_press, type, scroll, drag, open_app, press_home, press_back, wait, and finished. The environment avoids XML phone states for scalability and uses persistent state across episodes (minimal cleanup via home-screen return rather than full reflash).

**Asynchronous Architecture.** Each worker manages a dedicated cloud device in an isolated process. Workers push completed steps to a central inference queue asynchronously, eliminating synchronous vectorization bottlenecks. The learner consumes batches as soon as a minimum batch size is met, enabling concurrent GPU computation and environment execution. The number of workers is decoupled from inference batch size.

**Agent Learning Framework.** Built on GiGPO and VERL (HybridFlow), the framework uses two primary abstractions: *Agents* (handle context processing and action parsing) and *Workflows* (manage the agent-environment loop). The base model is UI-TARS-7B. Policy optimization uses PPO with token-level GAE advantage estimation, operating on atomic transitions rather than full trajectories. This allows credit assignment at token granularity across arbitrary task horizons.

**Rollout and Optimization Loop.** The system retrieves only fully completed episodes from the replay buffer to avoid value-estimation bias from truncated trajectories. During optimization, step data is reconstructed into temporal order for accurate GAE computation. A 30-epoch critic warm-up phase precedes policy updates.

**LLM Judge for Verification.** An LLM-based judge (Gemini-2.5-flash) provides binary success/failure reward signals. The optimal configuration uses full visual and action history with marked interaction coordinates, achieving F1=0.91 (90% accuracy) on English tasks and F1=0.85 (89% accuracy) on Chinese tasks against human-labeled ground truth.

## Results & Highlights

- **Policy Improvement:** UI-TARS-7B mean success rate improves from ~0.2 to ~0.8 over 200 training steps on 8 SPA-Bench tasks, demonstrating stable learning in the mobile GUI domain
- **Distributed Scalability:** Scaling from 8 to 32 phones shows consistent convergence despite increased off-policy data. Wall-clock throughput improves from 8 to 16 phones but saturates at 32 (bottleneck shifts to model inference). The system is resilient to hardware instability (network outages)
- **Task Scalability:** Scaling from 8 tasks (16 phones) to 16 tasks (32 phones) with proportional hardware yields similar convergence, supporting horizontal scaling for "Big World" training
- **Task Lifecycle Protocol:** The cycle protocol successfully manages automated Setup/Teardown transitions; agents learn auxiliary skills without degrading core task performance
- **Critic Robustness:** Random critic initialization without warm-up causes immediate policy collapse. Calibrated initialization (bias=0.1) alone prevents collapse and enables faster learning. Treating truncations as terminal (V(s')=0) outperforms bootstrapping from the learned critic
- **Training runs take 24-36 hours** on a multi-GPU server with 8-32 cloud phones

## Critical Assessment

**Novelty:** Moderate. The core technical novelty is the asynchronous cloud-phone RL infrastructure for mobile GUI agents -- a genuine engineering contribution but not a conceptual breakthrough. The three-stage self-evolution roadmap is conceptually interesting but only stage 1 is implemented, making the "roadmap" largely aspirational. The PPO + token-level GAE formulation is standard.

**Rigor:** Reasonable for an infrastructure paper. The experiments validate stability and scalability across multiple dimensions (phone count, task count, lifecycle protocol, critic robustness). However, the evaluation is limited to 8-16 tasks from SPA-Bench, a relatively narrow task distribution. No comparison with existing mobile agent training approaches (e.g., supervised fine-tuning baselines, or other RL frameworks). The ablation on critic bias/bootstrapping is the most rigorous experimental contribution.

**Reproducibility:** Good in principle -- the system is claimed open-source (github.com/ai-agents-2030/darwin-mobile-agent), built on public frameworks (VERL, Gymnasium), and uses a public benchmark (SPA-Bench). However, reproducing results requires access to cloud Android infrastructure, which is non-trivial.

**Significance:** The infrastructure itself fills a real gap -- there is no prior open-source end-to-end RL training system for mobile GUI agents. The roadmap articulates important research directions. However, the gap between "infrastructure works" and "self-evolving agent" is vast, and the paper only covers the first step.

**Red Flags:**
- The "self-evolution" framing in the title overpromises relative to what is demonstrated (stable RL training on curated tasks with an external LLM judge)
- No comparison with any baseline method or competing system
- The roadmap's stages 2-3 are entirely speculative with no preliminary results
- Limited task diversity (8-16 tasks from a single benchmark)
- The LLM judge (Gemini-2.5-flash) introduces an external dependency that contradicts the "remove human priors" vision

## Strengths
- Addresses a genuine infrastructure gap: no prior open-source system for RL-based mobile GUI agent training
- Thoughtful architectural design: asynchronous rollout, persistent state management, task lifecycle protocol
- Comprehensive scalability evaluation across multiple dimensions
- Practical insights on critic initialization and bootstrapping that generalize beyond this specific domain
- Clean separation of concerns (Agent/Workflow abstractions) enabling modular research
- Open-source commitment with public code repository

## Limitations
- Only stage 1 of the three-stage roadmap is implemented; stages 2-3 remain speculative
- No baseline comparisons (e.g., supervised fine-tuning, other RL methods, other mobile agent frameworks)
- Narrow task distribution (8-16 SPA-Bench tasks); unclear how the system handles truly open-ended mobile tasks
- Relies on an external LLM judge (Gemini-2.5-flash) for reward signals, which is itself a significant human prior
- Cloud Android infrastructure requirement limits accessibility for many researchers
- No evaluation of the learned policy on held-out tasks or transfer to new apps
- The paper does not report final absolute performance numbers on standard benchmarks for comparison with SoTA GUI agents

## Relevance
该论文提出了首个面向移动GUI智能体的开源端到端RL训练系统，与LLM agent自主学习、强化学习基础设施、GUI自动化等研究方向高度相关。其异步训练架构设计和critic稳定性发现对广义的多轮LLM RL训练（如reasoning model训练）也有参考价值。

## Questions for Further Exploration
1. How does RL-trained UI-TARS-7B compare with SoTA GUI agents (e.g., supervised fine-tuned models, GPT-4V-based agents) on standard benchmarks like SPA-Bench full set or AndroidWorld?
2. Can the task lifecycle protocol scale to truly complex, multi-app workflows that require coordination across applications?
3. What is the practical path from stage 1 (human-curated tasks + external LLM judge) to stage 3 (fully internalized curriculum and verification)? Are there intermediate results suggesting feasibility?
4. How sensitive is the system to the choice of LLM judge? Would a weaker/cheaper judge degrade learning, and could the agent eventually replace the judge?
5. What is the computational cost breakdown (cloud phones vs. GPU training vs. LLM judge API calls) and how does it compare to supervised approaches?
