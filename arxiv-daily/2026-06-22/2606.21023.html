Why This Matters
Enables reproducible LLM deployment in regulated domains — healthcare, law, finance — without the crippling cost of full FP32 inference.
Critical Edge
~89.5% TPOT overhead vs BF16 is non-trivial for latency-sensitive apps; no code released; only validated up to 32B parameters.

Methodology

HEAL is built on a precise diagnosis: SASS instruction-level profiling reveals that critical kernels (FlashAttention, GEMM) internally use FP32 accumulators. The instability comes exclusively from truncating FP32 results back to BF16/FP16 at kernel output boundaries.

Diagnosis
SASS profiling identifies boundary truncation as sole root cause. Arithmetic error is negligible.
→
INT16 Q/K/V
Uniform INT16 quantization achieves lower MSE than BF16/FP16 by exploiting narrow value distributions. Dual-FP16 unpacking yields 20 bits effective precision.
→
GEMM Compensation
All GEMMs promoted to FP16; dual-FP16 algebraic decomposition approximates FP32 precision on 16-bit Tensor Cores.
→
Result
FP32-level reproducibility. 16-bit memory footprint. Standard Tensor Core execution.

Strengths

  • Hardware instruction-level root cause analysis (SASS profiling) goes far beyond typical software-level reasoning
  • Counterintuitive insight that INT16 beats BF16/FP16 for Q/K/V tensors, well-supported mathematically
  • Maintains 16-bit memory footprint; runs on standard Tensor Cores across NVIDIA and AMD
  • Clean progressive evaluation (Base through Full) isolates each component's contribution
  • Divergence threshold framework is a reusable conceptual tool: you don't need bit-exact FP32 parity

Limitations

  • ~89.5% TPOT overhead vs BF16 is significant; paper frames it as a win vs FP32, but practitioners compare against BF16
  • No code release; custom CUDA/Triton kernels needed to reproduce
  • Only evaluated up to 32B parameters; 70B+ scaling behavior unknown
  • Dual-FP16 decomposition ~3x Attention FLOPs and ~2x GEMM FLOPs; roofline argument may not hold for all batch sizes
  • FP16 overflow risk handled by rescaling but without formal guarantees

Open Questions

1How does HEAL scale to 70B+ models where error accumulation across layers and KV cache memory pressure are both more severe?
2Can the divergence threshold be estimated cheaply at runtime, enabling adaptive per-token or per-layer precision switching?
3How does HEAL interact with existing KV cache compression (GQA, sliding window, token pruning) that already modify Q/K/V representation?
4What explains the residual 0.1–0.3% flip rate in HEAL-Full — remaining numerical noise, or inherent model sensitivity at decision boundaries?
5Could INT16 quantization for Q/K/V be applied during training to improve stability or enable lower-precision training regimes?
