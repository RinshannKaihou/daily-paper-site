> **一句话总结：** Sakana AI 提出 Fugu 系列编排器模型，通过轻量级选择头 + 监督微调 + 进化算法（Fugu）和基于 Conductor 框架的 GRPO 强化学习（Fugu-Ultra），将多个前沿 LLM 的互补专长组合为统一接口的集体智能系统，在 SWE-Bench Pro、Terminal Bench、GPQA-Diamond 等多个基准上超越所有单一前沿模型。

> **TL;DR:** Sakana Fugu is a family of learned LLM orchestrators that present a multi-model agent team as a single-model interface. Fugu (latency-optimized) uses a lightweight selection head with SFT + evolutionary search to route each query to the best worker model; Fugu-Ultra (quality-optimized) uses a Conductor framework trained with GRPO to compose multi-step agentic workflows across multiple frontier models. Both variants achieve SOTA on SWE-Bench Pro (73.7%), Terminal Bench 2.1 (82.1%), GPQA-Diamond (95.5%), and other benchmarks, outperforming Claude Opus 4.8, GPT-5.5, and Gemini 3.1-Pro individually.

## 核心贡献

- 提出"行为级模型组合"范式：将前沿 LLM 视为黑盒 agent，通过学习型编排器在行为层面组合其互补能力，无需参数访问或架构兼容
- 发布两个变体：Fugu（低延迟，轻量级选择头 + singular-value 微调 + sep-CMA-ES 进化优化）和 Fugu-Ultra（高质量，Conductor 框架 + GRPO 强化学习，支持多步 agentic 工作流）
- 在 SWE-Bench Pro、Terminal Bench 2.1、GPQA-Diamond、LiveCodeBench、Humanity's Last Exam 等多个基准上取得 SOTA，超越所有单一前沿模型（包括 Opus 4.8、GPT-5.5、Gemini 3.1-Pro）
- 提出工作流内 agent 隔离和跨工作流持久共享内存机制，解决多 agent 函数调用中的"编排坍缩"问题
- 在 AutoResearch、古典日文假名阅读顺序恢复、CAD 生成等端到端任务上展示编排优于单模型的实际价值

## Key Contributions

- Introduces "behavioral-level model composition": treats frontier LLMs as black-box agents and learns to orchestrate their complementary strengths without parameter access or architectural compatibility
- Releases two variants: Fugu (low-latency routing via lightweight selection head + singular-value fine-tuning + sep-CMA-ES) and Fugu-Ultra (quality-maximizing multi-agent workflows via Conductor framework + GRPO reinforcement learning)
- Achieves SOTA on SWE-Bench Pro (73.7%), Terminal Bench 2.1 (82.1%), GPQA-Diamond (95.5%), LiveCodeBench v6 (92.0%), Humanity's Last Exam (50.0%), CharXiv Reasoning (86.6%), and SciCode (58.7%), surpassing all individual frontier models
- Introduces intra-workflow agent isolation and inter-workflow persistent shared memory to prevent "orchestration collapse" in multi-agent function-calling scenarios
- Demonstrates orchestration value on end-to-end tasks: autonomous ML training optimization (AutoResearch), classical Japanese kana reading-order recovery, and CAD generation

## Methodology

**Fugu (latency-optimized):**
1. **Parametrization**: Attaches a lightweight prediction head after the final hidden layer of an orchestrator backbone LM. Given hidden state h, the head outputs L logits scoring which of L worker models to select. Uses the orchestrator's logits (not generated text) for routing decisions, making inference cheap. Also fine-tunes singular-value scales of selected parameter matrices in the backbone via SVD decomposition.
2. **Supervised Fine-tuning (SFT)**: Assembles a large collection of single-step verifiable tasks (coding, math, reasoning, language understanding). Runs every worker model on each task n times, computes average rewards, converts to a soft target distribution via softmax with temperature tau, and trains via KL divergence loss.
3. **Evolutionary Optimization**: After SFT, optimizes on end-to-end multi-turn agentic tasks using sep-CMA-ES. Collects real multi-turn trajectories from coding-assistant environments. At each turn, Fugu selects a worker based on the hidden state; terminal reward is binary task completion. The evolutionary search refines routing over these complex trajectories more stably than gradient-based fine-tuning.

**Fugu-Ultra (quality-optimized):**
1. **Conductor Framework**: An RL-trained LM that outputs full agentic workflows as natural language -- sequences of workflow steps, each specifying a subtask, assigned worker agent, and access list (which prior outputs to share). Supports arbitrary topologies: best-of-N, chains, trees, parallel branches.
2. **Function Calling & Memory**: Introduces intra-workflow agent isolation (each agent sees only its own tool-call history + access-list outputs) to prevent orchestration collapse. Permits inter-workflow shared memory so agents across successive workflows retain context from prior tool calls.
3. **Training**: Uses GRPO without KL penalty on a mixture of public and expert-designed end-to-end tasks. Worker pool includes Gemini-3.1-Pro, Claude-Opus-4.8, and GPT-5.5. Reward: 0 for unparseable workflows, 1 for correct final output, 0.5 otherwise.

## Results & Highlights

**Benchmark Performance (Table 1):**
| Benchmark | Fugu-Ultra | Fugu | Opus 4.8 | Gemini 3.1 | GPT-5.5 |
|---|---|---|---|---|---|
| SWE-Bench Pro | **73.7** | 59.0 | 69.2 | 54.2 | 58.6 |
| Terminal Bench 2.1 | **82.1** | 80.2 | 74.6 | 70.3 | 78.2 |
| LiveCodeBench v6 | **92.0** | 90.3 | 90.3 | 88.9 | 90.7 |
| GPQA-Diamond | **95.5** | **95.5** | 92.0 | 94.3 | 93.6 |
| Humanity's Last Exam | **50.0** | 47.2 | 49.8 | 44.4 | 41.4 |
| CharXiv Reasoning | **86.6** | 85.1 | 84.2 | 83.3 | 84.1 |
| SciCode | **58.7** | 60.1 | 53.5 | 58.9 | 56.1 |

- Fugu-Ultra's SWE-Bench Pro gain (73.7% vs. 69.2% for Opus 4.8) is described as "consistent with entire generational improvements" -- equivalent to a next-generation model release.
- Both Fugu variants surpass Mythos Preview and Fable 5 model class (not publicly accessible) on GPQA-Diamond, CharXiv, and Terminal Bench.
- Domain adaptivity: routing distributions match known model strengths -- GPT-5.5 dominates Terminal Bench, Gemini dominates GPQA-Diamond, balanced distribution on HLE.

**End-to-End Tasks:**
- **AutoResearch**: Fugu-Ultra achieves best mean validation BPB (0.9774 vs. 0.9781 for Model C, the next best) across 123 autonomous experiments, with lowest variance.
- **Classical Japanese kana reading order**: Fugu-Ultra achieves mean NED 0.776, far ahead of best frontier baseline (0.642).
- **CAD generation**: Fugu-Ultra produces the only mechanically correct iris mechanism across models.

## Authors & Credibility

Sakana AI, a Tokyo-based research lab co-founded by former Google Brain researchers (David Ha, Llion Jones). Known for evolutionary model merging (Nature Machine Intelligence, 2025) and nature-inspired AI research. Project lead: Yujin Tang. The team has relevant prior publications in evolutionary optimization and multi-agent systems.

## Critical Assessment

**Novelty**: Moderate-to-high. The core idea of learned LLM routing is not new (RouterDC, Trinity exist), but the production-scale execution, the Conductor-based multi-step workflow generation via RL, and the intra-workflow isolation/shared memory mechanisms are genuine contributions. Fugu-Ultra's ability to generate arbitrary coordination topologies per query is a meaningful advance over fixed routing.

**Rigor**: Mixed. Benchmark results are strong and cover a wide range of tasks. However, baseline comparisons have a notable asymmetry: Fugu orchestrates Opus 4.8, GPT-5.5, and Gemini 3.1-Pro, then compares against each individually. This is inherently favorable -- the ensemble has access to the union of all models' capabilities. The paper acknowledges this ("to assess Fugu's ability to harness and amplify the skills of its individual workers") but could be more explicit that the comparison is not compute-matched. The qualitative case studies (Section 4.3) use anonymized baselines, which is good for reducing bias but limits reproducibility.

**Reproducibility**: Low. The orchestrator models, training data composition, worker pool configuration, and infrastructure details are not fully disclosed. The system relies on proprietary API access to multiple frontier models. The paper reads more as a product announcement than a reproducible research contribution.

**Significance**: High for the industry, moderate for research. Demonstrates that orchestration is a viable scaling axis complementary to model size. The practical implication is clear: Sakana can offer frontier-competitive performance without training a frontier-scale model from scratch.

**Red flags**:
- The paper compares a system that uses 3+ frontier models against each model individually, which is not a fair compute or cost comparison. Latency and cost analysis is notably absent.
- No ablation isolating how much of the gain comes from orchestration intelligence vs. simply having access to multiple models (e.g., random routing baseline, majority-vote baseline).
- "Shoulder-to-shoulder with Fable 5 and Mythos Preview" -- these are non-public models, making the claim unverifiable.
- The AutoResearch BPB improvements (0.0007-0.0051) are very small in absolute terms; the paper frames them as meaningful but acknowledges they are "modest in absolute magnitude."

## Strengths

- Comprehensive benchmark coverage across coding (SWE-Bench Pro, Terminal Bench, LiveCodeBench), reasoning (GPQA-Diamond, HLE, CharXiv), and scientific tasks (SciCode)
- Two clearly differentiated variants serving different use cases (latency vs. quality)
- Elegant training pipeline: SFT for stable initialization, then evolutionary optimization for end-to-end tasks (Fugu); GRPO for free-form workflow generation (Fugu-Ultra)
- The intra-workflow isolation + inter-workflow shared memory design is a well-motivated solution to a real multi-agent problem
- Qualitative case studies (kana reading order, CAD generation, AutoResearch) go beyond standard benchmarks to show practical value
- The "orchestration as scaling axis" framing is compelling and well-argued

## Limitations

- No cost or latency analysis: Fugu-Ultra composes multi-agent workflows that invoke multiple frontier models, but the paper never reports end-to-end latency or dollar cost vs. single-model baselines
- No ablation against simpler baselines (random routing, majority vote, oracle routing) to quantify how much of the gain is from learned orchestration vs. model diversity
- Dependent on proprietary API access to all worker models; any provider's API change, deprecation, or rate-limiting could break the system
- Reproducibility is limited -- no open-source release of the orchestrator, training data, or evaluation harness
- The orchestrator backbone itself is not described (architecture, parameter count, which base model)
- The comparison is structurally asymmetric: an ensemble of 3 frontier models vs. each model alone

## Relevance

Sakana Fugu 的"学习型 LLM 编排"范式直接关联多 agent 系统、模型路由、集体智能等研究方向。其 Conductor 框架的 RL 训练方法和工作流生成机制对构建自适应 AI 系统有参考价值；同时，"编排作为新的 scaling axis"这一观点值得关注——它意味着前沿能力未必需要训练更大的模型，而可以通过智能组合现有模型获得。

## Questions for Further Exploration

1. What is the end-to-end latency and API cost of Fugu-Ultra workflows compared to single-model baselines? Is the performance gain worth the added cost?
2. How much of the performance improvement comes from learned orchestration vs. simply having access to multiple models? What would a random-routing or majority-vote baseline achieve?
3. What is the architecture and parameter count of the orchestrator backbone? How does orchestrator scale affect downstream performance?
4. How robust is the system to changes in the worker pool (e.g., adding a new model, removing one, model version updates)?
5. Can the Conductor framework generalize to open-source model pools, and would the orchestration benefits persist with weaker workers?
