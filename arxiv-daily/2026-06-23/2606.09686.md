> **一句话总结：** 本文提出一个包含 83 种数值格式（横跨 13 个家族）的开源目录 t27，配套发布 6 个 bit-exact 一致性测试包（GF16、MXFP4、BF16、FP8 E4M3、FP8 E5M2、E8M0 block scale），并通过 IEEE P3109 v3.2.0 交叉映射表，揭示了 FP8 E4M3 溢出策略与 MXFP4/NVFP4 块结构两类被规范允许的"interpretation gap"，为跨厂商 ML 加速器数值一致性提供一把"中立刻度尺"。

> **TL;DR:** This paper releases the **t27 catalog** — a single source of truth (SSOT) enumerating **83 numeric formats across 13 families** (GoldenFloat, IEEE-754 binary, Posit/Unum III, ML low-precision including BF16/FP8, OCP MX microscaling, IntegerFixed, LNS, HistoricalVendor, Compression/scaling, etc.). Alongside it come **six bit-exact conformance packs** (GF16, MXFP4 element, BF16, FP8 E4M3, FP8 E5M2, E8M0 block scale) with SHA-256 fingerprints, shared row schema, and an anchor vector encoding 3.0 via the identity φ² + 1/φ² = 3. Packs are cross-validated against **ml_dtypes 0.5.4** (Google/JAX oracle); every divergence is documented, not hidden. The work exposes two important **spec-permitted interpretation gaps**: (A) FP8 E4M3 overflow (tt-metal/AMD saturates to 448.0, JAX/TPU overflows to NaN), and (B) MXFP4 vs NVFP4 block structure (32-element + E8M0 scale vs 16-element + FP8 E4M3 scale). An **IEEE P3109 v3.2.0 cross-walk** maps each pack to its standards-track configured format (Binary8p3se ↔ FP8 E4M3 with SatMax vs OvfInf; Binary4p1sf ↔ MXFP4 element). All artifacts are MIT-style open licensed at `github.com/gHashTag/t27`, with cross-compiled output to Markdown/JSON/Python/Rust/C/TypeScript via Jinja2 templates.

## 核心贡献

- **83 格式 × 13 家族的统一目录 (t27)**：覆盖 GoldenFloat (22)、HistoricalVendor (10)、Posit/Unum III (8)、IntegerFixed (8)、MLLowPrecision (7)、IEEE754 binary (5)、LNS (4)、Theoretical (4)、Compression/scaling (4)、IEEE754 decimal (3)、Extended (3)、Microscaling OCP MX (3)、QuantTuned (2)，通过 15 条 CI 不变式（如 CI-01 总数 = 83、CI-11 SHA-256 锚定、CI-13 anchor 解码为 3.0）严格保证。
- **6 个 bit-exact 一致性测试包**：每个包均为自包含 JSON 文档，含共享行 schema（name / input f64 / `<fmt>` bits hex / decoded f64 / abs error / category）、SHA-256 自指纹、对应的 ml_dtypes 版本锚点、以及至少一个 `anchor*` 向量编码恒等式 φ² + 1/φ² = 3（值 3.0 在所有 6 个包中均为精确表示，构成跨包 sanity check）。
- **跨 oracle 验证**：以 Google/JAX 的 `ml_dtypes 0.5.4` 作为 ground-truth；BF16 21/21 全匹配，FP8 E5M2 17/17 全匹配，FP8 E4M3 15/16（仅 1000.0 溢出差异），MXFP4 与 GF16 因 ml_dtypes 无对应类型仅做 round-trip 自检。
- **IEEE P3109 v3.2.0 交叉映射**：将 6 个包对应到 P3109 配置格式（Binary8p3se ↔ FP8 E4M3，Binary4p1sf ↔ MXFP4 element），明确标注饱和策略与 NaN 编码差异；GF16/BF16/FP8 E5M2/E8M0 在 P3109 v3.2.0 中无直接对应。
- **揭示两类 interpretation gap**：FP8 E4M3 溢出策略分歧（饱和 vs NaN）以及 MXFP4/NVFP4 块结构差异（32 elem + E8M0 vs 16 elem + FP8 E4M3 scale），并强调"暴露分歧"本身就是一致性测试套件的价值。
- **15 条 CI 不变式 + 跨语言代码生成**：通过 Jinja2 模板从单一 JSON 源生成 Markdown / JSON / Python dataclasses / Rust serde structs / C headers / TypeScript enum literals，所有生成文件 commit 入仓并经 GitHub Actions 矩阵重建。

## Key Contributions

- **Vendor-neutral 83-format registry** spanning 13 clusters with a strict, invariant-enforced schema (bit layout, bias, inf/NaN policy, saturation policy, max-finite, min-normal, min-subnormal, cluster label, claim-status tag ∈ {Verified, Empirical fit, Open conjecture, Risk, Retracted}).
- **Six SHA-256-anchored bit-exact conformance packs** (GF16, MXFP4 element, BF16, FP8 E4M3, FP8 E5M2, E8M0 block scale) — each as a JSON document with header (format spec, saturation policy, max-finite, SHA-256 self-fingerprint, ml_dtypes version anchor, anchor identity reference) plus a vector array. GF16 + MXFP4 are live in `tt-lang-t27` PyPI package v0.3.1; BF16/FP8-E4M3/FP8-E5M2/E8M0 are new in v0.4.0-pre (PR #6).
- **Golden ratio anchor vector**: every pack includes a vector that encodes 3.0 via φ² + 1/φ² = 3 — a single-line cross-pack sanity check, since 3.0 is exactly representable in all six formats.
- **Honest abs-error reporting norm**: every vector where `decoded ≠ input` carries the actual absolute error magnitude (including overflow-to-Inf and underflow-to-zero cases); no values are suppressed to inflate match statistics.
- **IEEE P3109 v3.2.0 cross-walk**: maps each pack to its P3109 configured-format counterpart (direct match: Binary8p3se ↔ FP8 E4M3 SatMax vs OvfInf; Binary4p1sf ↔ MXFP4 element; no-match: GF16, BF16, FP8 E5M2 → hypothetical Binary8p2se absent from v3.2.0; orthogonal: E8M0 scale-only).
- **Two named interpretation gaps**: (Gap A) FP8 E4M3 overflow — OCP MX v1.0 permits both saturation-to-max (448.0 / 0x7E) and overflow-to-NaN (0x7F), and tt-metal/AMD pick SatMax while JAX/TPU pick OvfNaN; (Gap B) MXFP4 vs NVFP4 share S1E2M1 element layout but differ in block size (32 vs 16) and scale format (E8M0 vs FP8 E4M3), giving different effective bits/element (4.25 vs 4.50, ~5.9% overhead delta).
- **15 CI invariants** (CI-01 sum=83, CI-03 bit-width consistency, CI-04 bias range, CI-13 anchor decodes to 3.0, CI-14 codegen compiles, etc.) enforced on every commit.
- **Cross-language codegen** via Jinja2: Markdown table, JSON, Python dataclasses, Rust serde structs, C headers, TypeScript enums.

## Methodology

The work follows a **registry-filling** methodology (explicitly not a benchmark or competitive evaluation). The pipeline has five steps:

1. **Catalog schema design.** Each of the 83 formats is recorded with: `name`, `bits`, `exp`, `mant`, `bias`, `has_inf`, `has_nan`, `saturation_policy` ∈ {SatFinite, OvfInf, OvfNaN}, `max_finite`, `min_normal`, `min_subnormal`, `cluster`, `claim_status`. The schema is the **single source of truth (SSOT)** stored as JSON at `github.com/gHashTag/t27`.

2. **Conformance pack construction.** For each pack, a JSON file is hand-crafted with (i) a header object specifying the format's (E, M, bias, infNaN policy), saturation policy, max-finite value, SHA-256 self-fingerprint (over canonical JSON serialization), the ml_dtypes version anchor, and the GoldenFloat anchor identity reference; (ii) a vector array where each row contains `name`, `input_f64`, `input_f64_hex`, `<fmt>_bits_hex`, `<fmt>_bits_int`, `decoded_f64`, `decoded_f64_hex`, `abs_error`, and `category` ∈ {zero, normal, subnormal, inf, nan, overflow, underflow, rounding, anchor, transcendental}.

3. **Two-step verification per pack.** (a) **Round-trip self-check**: `decode(encode(input)) = decoded` for each vector, with the stored abs error consistent with the deviation. (b) **Cross-check against ml_dtypes 0.5.4** (Google/JAX) using `ml_dtypes.bfloat16`, `ml_dtypes.float8_e4m3fn`, `ml_dtypes.float8_e5m2`, `ml_dtypes.float8_e8m0fnu`. Every divergence is recorded in the pack header's `divergences` list and described in the paper text.

4. **Golden ratio anchor.** Every pack contains a vector named `anchor*` encoding 3.0 via the identity φ² + 1/φ² = 3 (where φ = (1+√5)/2). This identity, formalized in companion preprint arXiv:2606.05017 (GoldenFloat), is the L2 anchor; the value 3.0 is exactly representable in all six pack formats (normal range, zero mantissa error), giving a single-line cross-pack sanity check. The anchor's canonical SHA-256 fingerprint is `218403e344779c890f302ad2c70af21fb765060dd794d793c7eacc1ef8f80e6b`.

5. **CI-invariant enforcement and codegen.** A 15-invariant CI matrix (Table 2) runs on every commit; Jinja2 templates read the canonical JSON catalog and emit Markdown / JSON / Python / Rust / C / TypeScript outputs, all of which are committed to the repo and rebuilt via GitHub Actions.

### FP8 E4M3 overflow interpretation gap (Table 5)

| Input | Implementation | Bits | Decoded | Policy |
|---|---|---|---|---|
| 1000.0 | this pack (tt-metal/AMD convention) | 0x7E | **448.0** (max-finite) | saturate-to-max |
| 1000.0 | ml_dtypes 0.5.4 (JAX/TPU convention) | 0x7F | **NaN** | overflow-to-NaN |

Both choices are permitted by OCP MX v1.0. The pack adopts SatMax; the divergence is explicitly documented.

### MXFP4 vs NVFP4 structural parameters (Table 7)

| Parameter | OCP MX MXFP4 | NVIDIA NVFP4 |
|---|---|---|
| Element layout | S1E2M1 (4 bits) | S1E2M1 (4 bits) |
| Block size | 32 elements | 16 elements |
| Scale format | E8M0 (8-bit) | FP8 E4M3 (8-bit) |
| Scale exponent / mantissa bits | 8 / 0 | 4 / 3 |
| Scale dynamic range | 2⁻¹²⁷ to 2¹²⁷ | ≈ 2⁻⁹ to 448 |
| Scale granularity per decade | 1 (power-of-two only) | 8 (3-bit mantissa) |
| Bits/element including scale | 4 + 8/32 = **4.25** | 4 + 8/16 = **4.50** |

| Our format | P3109 name | Match | Key difference |

| GF16 | (none) | No match | P3109 does not address 16-bit phi-anchored formats |

### 15 CI invariants (selected from Table 2)

CI-01 (sum = 83), CI-02 (no name collisions), CI-03 (1 + exp + mant = bits for standard layout), CI-04 (bias ≤ 2^(exp-1) − 1), CI-07 (min-normal ≤ max-finite), CI-10 (bits ≥ 2), CI-11 (SHA-256 anchor present), CI-12 (anchor vector present), CI-13 (anchor decodes to 3.0), CI-14 (codegen targets compile Python + Rust), CI-15 (no duplicate SHA-256 across packs).

### Key cross-pack empirical findings

- **BF16** exhibits the **highest inter-vendor agreement** of the six formats: Google bfloat16, Intel BFLOAT16, ARM BFloat16, and NVIDIA TF32-paired BF16 share IEEE 754-style sub/inf/NaN semantics with RTE on the lower 16 bits of FP32. No divergences observed in 21 boundary cases.
- **FP8 E5M2** matches ml_dtypes exactly across all 17 boundary cases (zero, normals, subnormals, ±∞, NaN, overflow, underflow, RTE midpoints, anchor 3.0).
- **FP8 E4M3** matches 15/16; the single divergence is the overflow case 1000.0, attributed to a **spec-permitted policy choice** rather than a bug.
- **E8M0 block scale**: 0xFF encodes NaN (uninitialized/invalid scale); the anchor 3.0 encodes to closest representable power-of-two scale = 2¹ = 2, with a documented nonzero abs error.
- **MXFP4 element**: 12 vectors cover the 15 representable finite values + zero + saturation case; verified round-trip only (no ml_dtypes equivalent).

## Critical Assessment

This is a useful **community plumbing** paper, but it should be evaluated honestly rather than rubber-stamped.

**What works well:**
- The **interpretation-gap-as-feature** framing is genuinely valuable. The FP8 E4M3 overflow case (1000.0 → 448.0 vs NaN) is a real cross-vendor pain point that has bitten production deployments, and having it pinned down with a SHA-256-anchored test vector is a concrete service.
- The **honesty norm** on abs error (no zeroed-out or suppressed values) is the right ethical stance and contrasts favorably with suites that hide overflow/underflow cases.
- The **P3109 cross-walk** is genuinely useful for compiler/test-harness authors who need to map their internal format catalog to the standards-track configured formats.
- The **GoldenFloat anchor** is a clever way to get a single cross-pack sanity check without inventing a new format — the value 3.0 happens to be exactly representable in all six layouts.

**What is weaker:**
- **Single-oracle bias**: ml_dtypes 0.5.4 is the only ground truth for BF16/FP8/E8M0, and the only documented divergence (FP8 E4M3 overflow) is precisely the case where ml_dtypes picks OvfNaN — i.e. the suite cannot independently adjudicate between tt-metal and JAX. P3109 FLoPS (Lean 4 formalization, [4]) is cited but **not yet wired into the verification harness** (acknowledged as Track 2).
- **Asymmetric coverage**: the catalog advertises 83 formats but the Tier-1 packs cover only 6. Posit/Unum III (8 formats), LNS (4), NF4, BitNet, TF32, FP6 — none have conformance packs. The catalog is therefore a **registry first, a verification harness second**, which the author acknowledges but which is easy to over-read from the title.
- **Vendor-neutrality claim is partially self-undermined**: the GoldenFloat cluster accounts for 22/83 (26.5%) of the catalog, and GF16 is one of only two packs already "LIVE" in a PyPI package. The author's own format is over-represented relative to its production relevance, even if each entry follows the same schema.
- **NVFP4 gap is structural, not just missing**: the most important emerging 4-bit format (NVIDIA Blackwell/Rubin) is documented in Section 7.2 but **not packed** — the closing deliverable is punted to Track 2 (Q3 2026).
- **Citation echo**: the anchor identity φ² + 1/φ² = 3 comes from the author's own companion preprint (arXiv:2606.05017); the SHA-256 fingerprint `218403e3...` covers "the canonical UTF-8 encoding of the identity string," not a numerical computation — so it is really a citation fingerprint rather than a numerical anchor. This is a subtle conflation.
- **No novel research contribution** in the academic sense — this is engineering / infrastructure work, which is fine but should not be over-claimed.
- **Operation layer completely absent**: the suite covers representation-layer bit patterns only. For actual ML workloads (matmul, FMA, accumulation), the conformance story is far from complete.

**Red flags to note:**
- Single-author preprint with strong self-citation to companion work (GoldenFloat).
- The paper explicitly disclaims model-accuracy, FLOP, and performance comparisons — readers expecting benchmarks will be disappointed.
- "Honest abs error" framing implicitly criticizes other suites (without naming them) that suppress overflow/underflow — this is rhetorically effective but also positions the work competitively rather than purely registry-filling.

## Strengths

- **Concrete, deployable artifacts**: 6 JSON packs with full SHA-256, machine-verifiable, hosted on GitHub and a PyPI package. Anyone can fetch and replay.
- **The interpretation-gap framing is the right metaphor**: a conformance suite earns its keep when it exposes divergences that would otherwise be invisible — Section 7 makes this point precisely and persuasively with the FP8 E4M3 1000.0 case.
- **Schema discipline**: shared row schema across all six packs, 15 CI invariants enforced on every commit, single source of truth.
- **P3109 cross-walk** gives the work standards-track grounding without requiring the catalog itself to be a standard.
- **Cross-language codegen** via Jinja2 reduces downstream friction: ML practitioners get Python dataclasses, systems engineers get C headers / Rust structs, web/JS folks get TypeScript enums.
- **Honest limitations section (Section 10)**: explicitly states element-layer-not-operation-layer, single-oracle bias, asymmetric coverage, NVFP4-not-packed, no optimality claim, honest abs error norm — this is more transparent than typical preprints.
- **Author invites independent verification** (Section 9.4 "Open Invitations"): explicit asks to ml_dtypes, OCP MX WG, IEEE P3109 editors, NVIDIA NVFP4, Pychop, libtakum, IREE/vLLM/llama.cpp/onnxruntime to validate and report divergences — this is the correct collaborative posture for a registry.

## Limitations

- **Representation layer only**: encode/decode round-trip is verified; **operation-layer** semantics (add, multiply, FMA, accumulation) are explicitly out of scope and deferred to Track 2 (Q3 2026).
- **Single ground-truth oracle**: only ml_dtypes 0.5.4 is wired into the harness. P3109 FLoPS Lean 4 ([4]) is cited but not yet integrated; Pychop ([10]) and libtakum ([12]) are mentioned as future Track 2 oracles.
- **Asymmetric pack coverage**: 6/83 formats have packs; the remaining 77 (Posit/Unum III, LNS, NF4, BitNet, TF32, FP6, decimal, integer-fixed, etc.) are cataloged but unverified end-to-end.
- **NVFP4 not packed**: the most important emerging 4-bit format (Blackwell/Rubin) has a structural gap documented in Section 7.2 but no Tier-1 pack; cross-pack divergence vectors at scale-quantization boundaries are listed as Track 2.
- **GF16 anchor is a citation fingerprint, not a numerical one**: the SHA-256 `218403e3...` is over the UTF-8 identity string, not the result of `decode(encode(3.0))` — minor but worth noting.
- **Self-citation footprint**: GoldenFloat (the author's own format) accounts for 22/83 catalog entries and is one of two LIVE packs.
- **No benchmarks, no model accuracy, no FLOPs, no vendor comparison**: explicit scope limit; readers expecting competitive evaluation will find none.
- **Reproducibility envelope**: depends on continued availability of ml_dtypes, OCP MX v1.0 spec, and IEEE P3109 interim doc — no mirror layer is implemented.
- **Hash of E8M0 anchor vector is non-zero abs error**: 3.0 encodes to closest power-of-two scale = 2¹ = 2 (the anchor vector "decodes" with nonzero error against the input 3.0) — the cross-pack anchor is therefore "encode of 3.0" rather than "round-trip of 3.0" for this format.
- **GF16 SHA-256 not pinned in v0.4.0-pre manifest** (the paper notes "see repo" for the GF16 fingerprint, which is a minor provenance gap).

## Relevance

该工作与 ML 加速器生态系统中所有涉及跨厂商数值一致性的工作直接相关：编译器作者（IREE/vLLM/llama.cpp/onnxruntime）需要明确的 FP8 E4M3 溢出策略声明以保证跨硬件可复现性，硬件验证团队需要 P3109 对齐的位精确测试向量，而 MXFP4/NVFP4 块结构差异的精确文档则对 Blackwell/Rubin 部署的工程师尤为关键。与本人关注的低精度推理 + 量化研究兴趣高度契合。

## Relevance (English)

This work is directly relevant to anyone porting models across ML accelerators (Tenstorrent tt-metal, AMD, NVIDIA Blackwell/Rubin, JAX/TPU) where silent numerical divergences — like FP8 E4M3 overflow policy (448.0 vs NaN) or MXFP4/NVFP4 block-structure mismatch — can corrupt cross-vendor benchmarks. The P3109 cross-walk, SHA-256-anchored test vectors, and honest abs-error reporting make it a foundational reference for the low-precision inference and quantization research I follow.

## Questions for Further Exploration

- When P3109 FLoPS Lean 4 ([4]) is wired into the verification harness as a second oracle, will the FP8 E4M3 1000.0 case resolve to SatMax, OvfNaN, or be explicitly polymorphic at the proof level? And will ml_dtypes be updated to align?
- What is the empirical impact of the FP8 E4M3 overflow gap on real model accuracy — does saturation vs NaN at 1000.0 ever matter in practice, or is it always inside the saturated-weights regime of quantized models?
- How will the Track 2 NVFP4 pack handle scale-format divergence vectors — is there a clean way to encode "same element bits, different scale, different decoded value" as a single round-trip test, or does it require a tensor-level pack?
- Could a property-based fuzzer (Section 9.3) discover divergences that hand-crafted boundary vectors miss, particularly for E8M0 block scale at the 2¹²⁷ / 2⁻¹²⁷ extremes?
- Does the GoldenFloat cluster (22/83 formats) deserve a Tier-1 pack per format, or is it covered by the single GF16 pack + a codegen template? The paper is silent on per-GFN pack plans.
- Will the operation-layer (Track 2) conformance vectors — particularly FMA across BF16/FP8/MXFP4 — expose further spec-permitted interpretation gaps beyond the representation-layer overflow case?
- How does this catalog relate to Pychop ([10]) and libtakum ([12]) operationally — could Pychop serve as the second oracle for FP8 (representation layer), with libtakum as the Posit/Unum III oracle, to address the single-oracle bias acknowledged in Section 10?
- Is there a path to align the catalog schema with IEEE P3109's `StandardOperations.yaml` and the kappa-approximation accuracy measure ([19]), so that the conformance story extends from representation to operation in a single framework?
- The fingerprint `218403e3...` covers the **UTF-8 string** of the anchor identity, not the numerical round-trip result. Could this be sharpened to a hash of `(encode_F(3.0) for each pack F)` so the anchor fingerprint is genuinely bit-exact across packs?
