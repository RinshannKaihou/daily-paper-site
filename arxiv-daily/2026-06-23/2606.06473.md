# MLEvolve: A Self-Evolving Framework for Automated Machine Learning Algorithm Discovery

> **一句话总结：** MLEvolve 提出了一种自演化的多智能体框架，通过渐进式蒙特卡洛图搜索（Progressive MCGS）、回溯式记忆（Retrospective Memory）和分层自适应代码生成，把端到端机器学习工程（MLE）的长程搜索从孤立的树扩展到跨分支融合的图搜索，在 12 小时预算内于 MLE-Bench 上达到 65.3% 的平均奖牌率，并在 15 个数学优化任务上击败 AlphaEvolve。

> **TL;DR:** MLEvolve is an LLM-based self-evolving multi-agent framework for end-to-end ML algorithm discovery. It addresses three long-horizon limitations of existing MLE agents — branch isolation, memoryless search, and one-shot code generation — by (1) extending MCTS into **Progressive MCGS** with graph reference edges and an entropy-inspired soft switch from exploration to exploitation, (2) pairing a cold-start domain knowledge base with a dynamic global memory (lexical + FAISS + RRF retrieval) for experience reuse, and (3) decoupling the Planner from the Coder with three adaptive coding modes (full rewrite / stepwise / diff edit). On MLE-Bench (75 Kaggle tasks) it reaches 65.3% average medal rate and 100% valid submission rate in 12 hours (half the standard budget), and wins 11 of 15 AlphaEvolve mathematical optimization tasks, demonstrating strong cross-domain generalization.

## 核心贡献

- **Progressive MCGS**：把 MCTS 从树扩展为有向图 $G=(V, E_T \cup E_{\text{ref}})$，通过跨分支参考边实现跨分支知识流动；同时引入基于 Shannon 熵的渐进式探索调度，让搜索从广探索平滑过渡到精英引导的聚焦开发。
- **回溯式记忆（Retrospective Memory）**：把静态领域知识库（冷启动）与动态全局记忆（任务级经验）结合，使用词法+向量（FAISS）+Reciprocal Rank Fusion 混合检索，**无需额外 LLM 反射**即可自动积累并复用经验。
- **分层规划 + 自适应代码生成**：把"为什么改 / 改什么"（Planner）与"怎么实现"（Coder）解耦，并根据当前搜索状态在 Base / Stepwise / Diff 三种编码粒度之间自适应切换，提升长程迭代稳定性。
- **实证表现**：在 12 小时（标准预算一半）下达到 MLE-Bench SOTA（65.3% 奖牌率、100% 有效提交率、76.0% 超中位数率、34.7% 金牌率），并在 AlphaEvolve 数学任务上跨域泛化更优。

## Key Contributions

- We introduce **MLEvolve**, a self-evolving multi-agent framework for end-to-end MLE that unifies progressive graph search, retrospective memory, and hierarchical adaptive code generation to support long-horizon iterative optimization.
- We propose **Progressive MCGS**, which resolves inter-branch isolation via graph-based cross-branch information flow (primary edges $E_T$ + reference edges $E_{\text{ref}}$) and a progressive exploration schedule that mixes UCT-based exploration with Elite-Guided exploitation according to a time-decreasing soft switch $w(t)$.
- We propose **Retrospective Memory**, combining a curated cold-start knowledge base with a dynamic global memory that automatically accumulates and retrieves task-specific experience through hybrid lexical + FAISS retrieval fused by RRF, without requiring extra LLM reflection calls.
- We introduce **Hierarchical Planning with Adaptive Code Generation** that decouples strategic planning from code implementation and switches among full rewrite (Base), module-by-module (Stepwise), and patch (Diff) coding modes based on the search state.
- We report state-of-the-art results on **MLE-Bench** (65.3% medal rate under a 12 h budget, beating all open-source and proprietary baselines including MARS+, AIBuildAI, ML-Master 2.0) and **11/15 wins on AlphaEvolve mathematical optimization tasks**.

## Methodology

### Problem formulation
Each candidate solution $s \in S$ covers preprocessing, feature engineering, model training and inference. The objective is $s^* = \arg\max_{s \in S} h(T, s)$, where $h(T,s)$ is task-specific (accuracy / AUC / loss). The solution space is organized as a directed graph and explored iteratively.

### Progressive Monte Carlo Graph Search (Progressive MCGS)
- **Graph-based search space.** Each node $v \in V$ maps to a candidate solution; edges $E = E_T \cup E_{\text{ref}}$ distinguish parent–child generative edges (used for selection and backprop) from cross-branch reference edges (used for compositional reuse but **excluded from credit assignment**).
- **Selection with progressive schedule.** Tree backbone follows UCT with exploration constant $c(t)$ that decays from $c_0$ to $c_{\min}$. At each step the policy mixes UCT-based exploration and Elite-Guided exploitation through $P(S_t=\text{UCT}) = w(t)$, where $w(t)$ is chosen so that empirical branch-selection entropy $H(\pi_t)$ decreases monotonically over time. Elite selection draws from the top-$K$ global nodes weighted by $1/\text{rank}$.
- **Expansion (four types).** Unified as $v_{\text{new}} = g_o(v_t, R)$ with reference set $R$:
  1. **Primary expansion** ($R=\varnothing$) — Draft/Improve/Debug.
  2. **Intra-branch evolution** ($R = R_{\text{hist}}(v_t, k)$) — reflect over the last $k$ siblings on the same branch.
  3. **Cross-branch reference** ($R = R_{\text{cross}}(N)$) — borrow top-$N$ solutions from other branches.
  4. **Multi-branch aggregation** ($R = R_{\text{agg}}$) — synthesize a fresh branch root from top trajectories of multiple branches.
- **Reward & backprop.** Discrete reward $R(v) \in \{-1, 1, 2\}$ for fail / succeed-no-improve / branch-best-refresh; propagated along $E_T$ only.
- **Stagnation triggers.** Branch-level ($τ_{\text{branch}}$ consecutive non-improving expansions → intra-branch evolution, then cross-branch reference) and global-level ($τ_{\text{global}}$ no global improvement → multi-branch aggregation).

### Retrospective Memory
- **Domain knowledge base (cold-start).** Manually curated lists of suitable models per task type (image classification, NLP, tabular regression, …) with concise usage notes; retrieved by keyword match $R_{KB}(T)$.
- **Dynamic global memory.** After each valid node execution, stores structured records (plan, outcome, analysis, feedback). Retrieved via hybrid lexical + FAISS + RRF: $\text{score}(d) = \alpha \cdot \frac{1}{k + r_{\text{lex}}(d)} + (1-\alpha) \cdot \frac{1}{k + r_{\text{vec}}(d)}$.
- **Stage-aware queries.** Planning stage uses the current free-text plan as query (retrieve both successes and failures). Debugging stage uses the error message as query.

### Hierarchical Planning + Adaptive Code Generation
- **Planner–Coder decoupling.** Planner operates at module level (decides *what*/*why*), Coder implements at code level (decides *how*) while preserving existing structure.
- **Adaptive coding modes** (chosen by search state):
  - **Base** — full rewrite from scratch (cold start).
  - **Stepwise** — module-by-module generation for complex multi-stage pipelines.
  - **Diff** — patch-based local edits when a working solution exists.
- Specialized agent team: Draft / Improve / Debug / Evolution / Fusion / Aggregation / Code Review / Data Leakage / Result Parse agents.

- **76.0% above-median** — submissions beat the human Kaggle median in more than three quarters of tasks.
- Strongest relative gain on **high-complexity tasks** (46.7% medal rate, tying AIBuildAI but at half the time).

### Cross-domain generalization — AlphaEvolve (15 math tasks)

MLEvolve achieves the best result on **11 / 15** mathematical optimization problems, beating AlphaEvolve, AlphaEvolve-v2, SimpleTES, TTT-Discover, OpenEvolve on packing, additive combinatorics and autocorrelation inequalities. Notable wins:
- Sums differences problem 1: **1.1902** vs AlphaEvolve 1.1480.
- Third autocorrelation inequality variant: **1.4588** vs AlphaEvolve 1.4688.
- First autocorrelation inequality: **1.5028628749** (matches TTT-Discover best to reported precision).

### Ablation on MLE-Bench Lite (22 tasks)

| Configuration | Medal (%) | Gold (%) | Beat Ratio (%) |
|---|---|---|---|
| **MLEvolve (full)** | **81.82** | **54.55** | **88.39** |
| w/o Progressive MCGS | 68.18 | 40.91 | 79.91 |

All three components matter; removing Progressive MCGS is the largest single drop (-13.64 medal pts, -8.48 beat-ratio pts).

### Additional analyses
- **Entropy dynamics.** Effective active branches $\exp(H(\pi_t))$ decrease smoothly from 4.8 -> 2.8 over the search, validating the soft-switch schedule. Vanilla MCTS stays flat at ~4.3.
- **Backbone-agnostic.** Tested on Gemini-3.1-Pro-preview, GPT-5.5, DeepSeek-v4-Pro, Kimi-K2.6 across Image / NLP / Audio; all remain competitive. Best per domain: GPT-5.5 on NLP (96.2%), Kimi-K2.6 on Audio (99.2%).
- **Time efficiency.** On representative tasks MLEvolve reaches **98.2%** beat ratio at 12 h; vanilla MCTS plateaus at ~70% after the early stage.

## Critical Assessment

- **Novelty.** The individual ingredients — graph search (MCGS), memory + FAISS, planner/coder split, hierarchical multi-agent MLE — each have prior art. The genuine novelty is *integration*: showing that combining cross-branch reference edges, entropy-soft exploration-exploitation switching, and stage-aware memory retrieval lets a single agent family outperform every existing MLE agent at half the compute. The graph-with-reference-edges abstraction is a clean formalization worth borrowing in other search-agent work.
- **Rigor.** Mean ± SEM over 3 seeds, ablations on a held-out 22-task subset, comparison against 11 baselines (proprietary + open-source) at matched budgets, plus cross-domain validation on 15 math tasks. Reward shaping is explicitly defined and grounded in execution semantics. The exclusion of reference edges from backprop is principled and well-justified.
- **Reproducibility.** Code is released; hyper-parameters and hardware are documented. However, results depend on (a) Gemini-3.1-Pro-preview, (b) the cold-start KB quality, (c) FAISS index seeding. The KB itself is curated by the authors — its exact contents are not exhaustively enumerated in the paper, so a third party re-running may see different cold-start behaviour.
- **Significance.** Sets a new SOTA on MLE-Bench at 12 h — important for any practitioner trying to use agentic MLE under tight compute budgets. Demonstrates that *search architecture* (not just stronger LLMs or longer training) is the binding constraint for MLE agents.
- **Red flags / watch-outs.**
  - 100% valid submission rate on MLE-Bench full set is unusual — worth checking whether failure modes are silently downgraded or retried.
  - The cross-branch reference and aggregation operations make credit assignment harder; the paper's reward shaping sidesteps this by only backprop-ing on $E_T$, but the agent may still over-credit solutions built from references.
  - Some AlphaEvolve wins are by very small margins (e.g. Heilbronn triangles differ only in trailing digits), so 11/15 should be read as "competitive on essentially all 15."
  - The progressive schedule is hand-tuned ($w_0, w_{\min}, c_0, c_{\min}, K, \tau$); robustness across tasks is asserted but not deeply studied.

## Strengths

- **State-of-the-art across the board at half the budget.** 65.3% medal rate on the full 75-task MLE-Bench, with 100% valid submission and 76.0% above-median rate.
- **Backbone-agnostic design.** Verified on four different LLMs (Gemini, GPT, DeepSeek, Kimi) across three modalities.
- **Cross-domain transfer.** Wins on AlphaEvolve math tasks without modification, indicating the self-evolving pattern is not MLE-specific.
- **Clean abstraction.** Graph $G=(V, E_T \cup E_{\text{ref}})$ + soft switch $w(t)$ + reward $R(v) \in \{-1,1,2\}$ is a compact, reusable formulation.
- **Practical stability.** Planner–Coder decoupling + diff mode avoids the "every iteration rewrites everything" failure mode common in one-shot agents.
- **No additional reflection LLM** — Retrospective Memory accumulates experience without paying extra LLM inference cost per reflection, unlike MARS / ML-Master 2.0.

## Limitations

- **Single proprietary backbone dominates headline numbers.** Gemini-3.1-Pro-preview is not generally available; open-source replications will see a sizeable gap.
- **Curated cold-start KB.** Effective initialization depends on hand-built domain priors; building/maintaining the KB is itself engineering work that limits transfer to new domains.
- **Manual hyper-parameters for the progressive schedule.** $w(t), c(t), K, τ_{\text{branch}}, τ_{\text{global}}$ are fixed across 75 tasks; no learned or per-task adaptive variant.
- **Compute cost.** 21 vCPUs + 234 GB RAM + 1×H200 per task, even at 12 h — still far more than a typical Kaggle contestant would spend.
- **No theoretical convergence analysis.** The progressive MCGS lacks guarantees on convergence / sample efficiency; empirical dominance is shown, not proved.
- **Limited study of multi-branch aggregation failure modes.** When $R_{\text{agg}}$ produces poor starting points, the paper does not analyse recovery.
- **Scope of cross-domain eval.** The AlphaEvolve tasks are mathematical/programming only; nothing in physics, chemistry, or general science benchmarks.

## Relevance

与本项目关注的 **LLM 智能体 / 自动机器学习 / 长期规划搜索** 主题高度相关。MLEvolve 把"自演化"从口号落到具体机制：图结构的跨分支信息流、熵驱动的探索-开发切换、以及无需额外反射 LLM 的经验复用，对任何长程搜索型 agent（自动实验、AI4Science、代码生成）都有借鉴价值。

## Questions for Further Exploration

- 能否用强化学习 / 元学习自动调节 $w(t), c(t), K, τ_{\text{branch}}, τ_{\text{global}}$，让渐进式调度真正自适应任务难度，而不是依赖人工先验？
- 跨分支参考边是否引入隐式的"信用泄漏"风险？如果一个 child 由 parent 和一个高奖励参考点共同生成，仅沿 $E_T$ 回传奖励是否系统性地低估了参考点的贡献？
- 回溯式记忆用 RRF 融合词法+向量检索，但规划阶段和调试阶段的 query 形态差异极大，是否需要为不同阶段训练专用检索头？
- Planner–Coder 解耦在多大程度上依赖底座 LLM 的指令遵循能力？把同一个 agent 在开源 LLM（DeepSeek-V3 / Qwen3）上做 ablation，会复现同样的差距吗？
- 把 Progressive MCGS 推广到 **AI for Science**（材料设计 / 蛋白质工程 / 自动化实验）场景，是否需要修改节点表示和奖励定义？作者已宣告未来方向，但具体接口（候选 = 代码 vs 候选 = 实验配置）尚未明确。
- 100% valid submission rate 反过来说明 MLEvolve 的"硬失败"几乎被消解——是否意味着它实际上是在一个更窄的有效子空间里做更深的搜索？有效提交率与奖牌率的解耦值得进一步刻画。
- 图结构是否可被可视化或事后分析（例如聚类参考边的语义）以辅助人工 debug agent 行为？目前仅给出了 effective branch count 这一个全局指标。
