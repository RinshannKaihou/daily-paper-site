OpenSkill: Open-World Self-Evolution for LLM Agents

An agent that builds its own skills — and its own tests — from nothing but the open web.

Zhiling Yan, Dingjie Song, Hanrong Zhang, Wei Liang, Yuxuan Zhang, Yutong Dai, Lifang He, Philip S. Yu, Ran Xu, Xiang Li, Lichao Sun · Lehigh · UIC · UBC+Vector · Salesforce AI · MGH/Harvard

LLM agents self-evolution open-world retrieval self-verification SkillsBench cross-model transfer
Key Finding

An LLM agent can bootstrap both a portable skill library and a supervision-free verifier from public web and repository resources — and those skills beat the strongest closed-world baseline by +8.9 pp on SkillsBench while transferring unchanged to four weaker model families.

OpenSkill 让智能体只凭任务指令和公网资源，自行合成技能并搭建可监督自己的"虚拟测试"，零目标监督下在 SkillsBench 上以 +8.9 pp 拿下最佳自动通过率，且同一份技能文件可零适配迁移到 Haiku / Qwen / DeepSeek / Mistral。

Key Metric
+8.9pp
Best-automated pass rate on SkillsBench (11 domains) over the strongest closed-world baseline — within 1–3 points of the human upper bound (44.5%).
Why This Matters

A reproducible recipe for post-deployment agent self-improvement: skills become portable artifacts that travel across model families without retraining or re-tuning.

Critical Edge

The "no target-task supervision" claim rests on a name-based query filter — open-web retrieval could still leak via StackOverflow-style snippets that reveal evaluation criteria, even after benchmark names are stripped.

Methodology

Three stages, a leakage barrier, and a self-built verifier.

  • Stage 1 — Open-World Knowledge Acquisition. Gemini Deep Research fetches background knowledge ki (APIs, conventions, best practices) and independently verifiable anchors kvi (reference values, dataset row counts, cross-validation procedures).
  • Stage 2 — Leakage-Free Skill Evolution. An Opus 4.6 base agent inside a Docker container synthesises 1–4 skills, refined across up to J = 3 rounds against a separate Verifier LLM session that generates pytest assertions from kvi — no shared context with the creator.
  • Stage 3 — Zero-Shot Target Evaluation. The final skill file Ŝi* is copied to the target workspace and scored under neval = 5 independent runs.
  • Four-layer Leakage Barrier. Code (no solution/test paths), container (GT dirs never mounted), oracle (evolution loop overrides parent GT oracle), log-level audit (zero GT references in trajectories).
Results

Best automated across 3 benchmarks × 2 agents.

43.6% / 42.1%

SkillsBench overall pass rate on Opus 4.6 / GPT 5.2 — within 1–3 points of human upper bound; best-or-tied in 8 / 7 of 11 domains.

  • SocialMaze: 82.7% / 70.7% — best automated, +0.9 to +2.2 pp.
  • ScienceWorld: 90.0% / 85.3% — best automated.
  • Cross-model transfer: +5.5 to +14.8 pp on Haiku 4.5, Qwen 3 Coder, DeepSeek V3, Mistral Large 3.
  • Verifier agreement: 60.7% with GT outcomes (precision 56.9%, recall 80.5%); 88.9% coverage of human test intents.
Strengths

Clean formulation, honest reporting.

  • Auditable leakage barrier — the constraint is mechanically enforced, not rhetorical.
  • First cross-model transfer demonstration using identical skill files across 4 weaker LLM families.
  • Novel verifier anchoring: pytest assertions tied to dataset statistics and documented APIs, not target answers.
  • Honest numbers: 60.7% agreement, iteration overfitting at J > 3, Manufacturing collapse at 0% — all reported.
  • Open-source pipeline + public benchmarks + multi-model harness enable replication.
Limitations

Where the recipe still breaks.

  • Web noise: sources may be outdated, contradictory, or contain solutions to benchmark tasks — provenance tracking is shallow.
  • Verifier–task mismatch: 12 of 25 false positives were "high-accuracy near-misses" (0.81 mean accuracy) that GT scored as 0.
  • Pytest-centric: verifier fails entirely on 5 non-standard tasks (CVE patches, build systems, proof assistants).
  • Manufacturing collapse: 0% across every automated method — likely a class where open-world acquisition alone is insufficient.
  • Cost: ~$1,800 per SkillsBench run with Opus 4.6; skill creation is one-time but the budget is steep.
Questions for Further Exploration

Six open threads this paper raises.

  • Does the virtual verifier generalise beyond code — to web navigation, embodied control, dialogue — where pytest is inapplicable?
  • How robust is the name-based leakage filter against indirect leakage (task-specific datasets, repos containing benchmark tests)? An adversarial audit would help.
  • Skills transfer Opus → Haiku / DeepSeek / Mistral / Qwen — but what about the reverse direction? Is the gain bounded by the construction model?
  • The J = 3 peak and decay at J = 5 / 10 — is this overfitting to verifier artefacts or genuine diminishing returns?
  • Cross-benchmark transfer (SkillsBench → SocialMaze and vice versa) is unreported — how domain-specific are the synthesised skills?
  • Manufacturing fails for everyone — what is qualitatively missing? Open-world acquisition, verifier anchoring, or simulation environments?
