Key Metric
+38.0% average reward across ALFWorld, WebShop & ScienceWorld (4 backbones, vs ReAct+Skills baseline)
Why This Matters

Directly addresses how agents distill experience into reusable knowledge — core to evolving long-horizon agent capabilities and taming real-world skill explosion.

Critical Edge

All results are temperature-0 single deterministic runs with no reported variance — confidence in score-difference significance is therefore weakened.

01 — METHODOLOGY

How the pyramid gets built, used, and grown

Problem Formulation

An agent receives task x and rolls out a trajectory τ = (x, o₀, a₀, …, o_{T+1}, r). A skill is a structured natural-language program s = (n, d, c) — name, description, content. The goal is to organise a flat skill set S = {s₁, …, s_N} into a multi-level pyramid P = {L₀, L₁, …, L_K}, increasingly abstract from bottom to top.

Analyzer → Builder Pipeline

  1. Coarse grouping — the Relation Analyzer clusters skills by name/description alone.
  2. Fine-grained analysis — each cluster emits relation-construction tasks D_m = {(gₖ, τₖ, eₖ)}.
  3. Downward Atomic Extraction — the Builder pulls a reusable atomic skill aₖ = B_atom(gₖ) into a layer below its source, attaching an explicit reuse reference ρ(aₖ).
  4. Upward Abstract Induction — the Builder lifts a shared high-level schema uₖ = B_abs(gₖ) one layer above its source.
  5. The loop iterates until no new skills emerge or the round budget T is hit.

Skill Creator (§2.3)

Given a new task x, the Creator retrieves a candidate set R(x) ⊆ S and synthesises ŝ = C_create(x, R(x), P) in two passes: abstract skills scaffold the task frame, atomic skills fill in executable details. If an existing skill already covers x, it is reused verbatim — no noisy synthesis from scratch.

Self-Evolution (§2.4)

At time t the system maintains (S_t, P_t). The Analyzer treats the newly created ŝ as a query, the Builder produces a delta ΔP_t, and P_{t+1} = Update(P_t, ŝ, ΔP_t) merges the change incrementally — the whole pyramid is never rebuilt.

Implementation

Qwen3.5-35B-A3B powers the Analyzer/Builder; Qwen3-Embedding-4B handles embedding retrieval. Outer pipeline ≤ 3 rounds; intra-group fine analysis ≤ 5 rounds; inference temperature = 0 for reproducibility.

02 — RESULTS & HIGHLIGHTS

What the numbers say — main table, ablations, evolution curves

  • Main table (Table 1) · ALFWorld / WebShop / ScienceWorld × {DeepSeek-V3.2, GPT-4.1, Gemini 2.5 Pro, Qwen3-235B}. Mean reward 73.7 vs ReAct 53.4 / Reflexion 59.6 / ExpeL 61.3 / ReAct+Skills 65.8. Unseen-task reward 84.8 vs ReAct+Skills 73.9. Mean steps 14.6 vs 20.2 / 18.5 / 17.4 / 17.7. Since ReAct+Skills shares the same initial library, the gain is attributable to structure, not access.
  • Component ablation (Table 2, DeepSeek-V3.2) · Drop atomic skills: ALFWorld 81.4 → 78.6, ScienceWorld 76.3 → 75.3. Drop abstract skills: ALFWorld 81.4 → 80.0, ScienceWorld 76.3 → 71.1 — ScienceWorld suffers more. Freeze self-evolution: 76.4 / 73.7. Synthesise from scratch (no reuse): 64.3 / 70.1 — below the flat-skill ReAct baseline (75.7 on ALFWorld), showing that rootless synthesis injects noise.
  • Atomic vs abstract roles (Table 3) · Atomic-heavy (184 tasks, short dep chains) drops 6.5 pts without atomic skills but only 2.1 without abstract. Abstract-heavy (147 tasks, long chains) drops 8.0 pts without abstract and only 2.4 without atomic — a clean functional split.
  • Self-evolution curves (Figure 3) · Full SkillPyramid stays stably high on ALFWorld; static version degrades over time; append-only version oscillates as redundancy accumulates; from-scratch version never catches up.
  • Web-skill extension (Table 4, GAIA-Lite, 75 tasks) · 150K web candidates → 2,271 high-quality skills. Flat web skills lift overall accuracy 52.0 → 56.0; SkillPyramid pushes it to 62.7, steps 5.2 → 4.8 — hierarchical consolidation still pays off under realistic noise.
03 — STRENGTHS

What the paper gets right

  • Clean module boundaries. Analyzer–Builder + Skill Creator + Self-Evolution is decomposable, ablatable, and extensible.
  • Broad evidence chain. 3 long-horizon benchmarks × 4 backbones, plus main table, ablation, task-stratified analysis, evolution curves, and noisy-web extrapolation.
  • Honest attribution. ReAct+Skills control cleanly separates “access to skills” from “organisation of skills”; atomic-heavy vs abstract-heavy splits give interpretable blame.
  • Incremental update. P_{t+1} = Update(P_t, ŝ, ΔP_t) avoids rebuilding the whole pyramid as the library grows.
04 — LIMITATIONS

Where the cracks show

  • Initial pyramid construction still analyses the full library — no lightweight construction method for very large skill corpora.
  • Analyzer/Builder lean on a strong backbone; degradation under weaker models is not systematically studied.
  • Evaluation stays in text environments (ALFWorld, WebShop, ScienceWorld, GAIA-Lite) — no multimodal or embodied settings.
  • Every reported number is a temperature-0 single deterministic run; run-to-run variance is never reported.
  • The pyramid models only an “atomic ↔ abstract” axis; temporal, causal and cost dependencies are out of scope.
05 — QUESTIONS FOR FURTHER EXPLORATION

Where to push next

Q · 01

At 10⁴–10⁵ real skills, what approximation strategy is needed to keep construction tractable and merging clean?

Q · 02

When the backbone shrinks to 7B–13B, how badly does the Analyzer/Builder misjudge — and is a voting/verifier loop needed?

Q · 03

Can the pyramid grow into multi-axis structure (temporal, causal, cost dependencies) for richer multi-step planning and backtracking?

Q · 04

Over thousands of tasks, how do drift, forgetting and a capability ceiling manifest — and can a “skill value evaluator” govern merge / retire / retain?

Q · 05

Paired with parametric memory (Skill-Pro non-param PPO, AgentPro LoRA), which capabilities belong to the structured pyramid vs the implicit skills?

Q · 06

In multimodal or embodied settings, how should “atomic actions” and “abstract schemas” be redefined for continuous control or visual state spaces?
