> **一句话总结：** SPEAR 提出了一种"输入自适应 + 模块选择性"的 4-bit 量化误差补偿系统：先用一个带轻量门控的低秩 Error Compensator 按 token 动态调整补偿强度，再用 CKA 引导的熵感知策略把补偿预算集中到量化损伤最重的层；配套 phase-aware kernel 融合、epilogue P2P dual-write 和 SLO 约束的 chunked prefill 调度，在 < 1% 模型内存开销下把 W4-to-FP16 的 perplexity gap 缩小 56–75%，decode 延迟保持在 W4 MARLIN 量级。

> **TL;DR:** SPEAR is a deployment-aware post-quantization compensation stack for low-bit LLM serving. The authors observe that quantization error is strongly **input-dependent** (per-token cosine similarity spans ~0.4 between easy and hard tokens on Llama-3.2-1B) and **module-concentrated** (a small subset of layers carries most of the damage). SPEAR attacks both inefficiencies: a low-rank Error Compensator (EC) modulated by an input-dependent gate γ(Ax) = 1 + tanh(·) modulates compensation strength per token, and a **CKA-guided, entropy-aware module selector** concentrates the tight parameter budget on the layers with the highest damage/cost score. To avoid the 5× decode-latency blow-up that naive EC insertion causes, SPEAR fuses ECs into the MARLIN 4-bit GEMM epilogue for decode, uses a CUDA-Graph-overlapped semi-fused path for prefill, replaces NCCL AllReduce with **NVLink P2P dual-write** epilogue-integrated peer reduction under tensor parallelism, and adds an **SLO-constrained EC-aware chunk scheduler** that calibrates kernel-latency lookup tables to keep P99 ITL under target. Across RTN/GPTQ/AWQ/OmniQuant on Llama-3.2-1B/3B and Llama-2-7B/13B/70B, SPEAR uses only 31–66% of the memory of prior static compensation methods (LoftQ/LQER/QERA/EoRA/ASER), closes 56–75% of the W4→FP16 perplexity gap in challenging per-channel settings, and stays within ~25% of W4 MARLIN decode latency on a single GPU while remaining comparable to W4 MARLIN on TP=2/3/4.

## 核心贡献

- **首次系统刻画量化误差的输入依赖性与模块集中性**：在 9 条不同输入序列上，FP16 与 4-bit 隐藏态的 per-token 余弦相似度跨度高达 0.42；skip-one 探针显示，Llama-3.2-1B 和 Llama-2-7B 上仅前 5 个模块就承担了大部分 perplexity 增量。这意味着"对所有 token、所有层施加同一补偿"的静态做法本质上是低效的。
- **提出 Error Compensator (EC)**：在低秩分支 Ax → γ(Ax) → B 中插入由 bottleneck MLP 计算的门控 γ(z) = 1 + tanh(W₂ ReLU(W₁ z + b₁) + b₂)，把"1 +"残差化设计使得 EC 可以从静态低秩适配器稳定初始化；总额外参数仅 8r² + 6r，远小于 token-wise expert 或全宽 conditional adapter。
- **CKA-guided 熵感知模块选择**：用训练免费的 Centered Kernel Alignment 计算每个模块的量化"损伤分数"，再用归一化损伤熵 Hnorm 动态决定 top-K% 选择比例——损伤集中时减少模块数并提高每模块 rank，损伤分散时扩大覆盖；并以 sensitivity − λ · deployment cost 联合打分保护损伤最重的模块不被高代价模块挤掉。
- **Phase-aware 自适应 kernel 融合调度**：decode (M=1) 阶段把整条 EC 链折叠进 MARLIN 4-bit GEMM epilogue，单 kernel 消除 launch gap；prefill (M>1) 阶段切换到半融合 + CUDA Graph DAG 跨流 overlap，避免与 compute-bound GEMM 抢资源。
- **Epilogue-integrated peer reduction (P2P dual-write)**：利用 EC 的非线性门控不能对 TP partial 直接作用的特性，把跨卡 AllReduce 折叠进 GEMM epilogue——MARLIN 在写本地 W4 输出 Ybase 的同时通过 NVLink P2P 把 Ax 的 partial 写到对端 staging buffer，把剩下 EC 操作（gate、B、残差、RMSNorm）压成一条 fused post-EC tail，避免显式 NCCL launch/sync。
- **SLO 约束的 EC-aware chunk 调度**：离线性微基准得到 ℓ_EC(M)、ℓ_W4(M)、ℓ_attn(M) 三张查找表，运行时按当前 CKA 选择 S 二分搜索最大 prefill chunk 使得 T_S(d) + T_S(c) ≤ T_SLO，从而在 TTFT 与 P99 ITL 之间自动重平衡。
- **跨量化后端 × 跨模型规模 × 单/多卡一致收益**：在 RTN/GPTQ/AWQ/OmniQuant 上对 Llama-3.2-1B/3B 与 Llama-2-7B/13B/70B 做 W4 per-channel 与 group-128 评测，并在 GH200 (NVLink) 上验证 TP=2/3/4 部署延迟；< 1% 模型内存开销下相对 QERA 节省 34–59% 补偿内存。

## Key Contributions

- **Empirical characterization of two structural inefficiencies in 4-bit quantization**: (i) token-level error spread — per-token cosine similarity between FP16 and W4 hidden states varies by ~0.42 across 9 sequences on Llama-3.2-1B; (ii) module-level damage concentration — skip-one probes show a long-tail distribution where the top-5 modules account for most of the perplexity increase.
- **Input-adaptive Error Compensator (EC)**: a low-rank module y = Wcx + α·B(γ(Ax) ⊙ Ax) where γ = 1 + tanh(MLP(·)) is computed in the rank-r latent space (r ≪ d_out, d_in). The residual "1 +" form improves calibration stability by initializing EC ≈ a static low-rank adapter. Total extra parameters are 8r² + 6r.
- **CKA-guided entropy-aware module selection**: training-free per-module damage via Centered Kernel Alignment, adaptive top-K% based on the normalized entropy of the damage distribution (clipped τ_eff ∈ [0.15, 0.6]), and a hybrid score score_i = δ̃_i − λ·t_dep_i that protects the highest-damage modules while penalizing deployment-expensive layers.
- **Phase-aware adaptive kernel fusion dispatch**: full GEMM-epilogue fusion for memory-bound decode (M = 1); semi-fused + CUDA-Graph DAG overlap for compute-bound prefill (M > 1). Decision is made at runtime per iteration.
- **Epilogue-integrated peer reduction**: instead of launching NCCL AllReduce between GEMM and EC, the MARLIN epilogue performs a P2P dual-write that simultaneously commits Y_base locally and the low-rank Ax partial to peer-GPU staging memory. The remaining EC path (gate, B, residual, RMSNorm) executes as a fused post-EC tail — collapsing TP communication into a 2-kernel pipeline.
- **SLO-constrained EC-aware chunk scheduling**: offline microbenchmarks build latency lookup tables ℓ_EC(M), ℓ_W4(M), ℓ_attn(M); online binary search picks the largest prefill chunk satisfying T_S(d) + T_S(c) ≤ T_SLO under the current EC selection S, automatically rebalancing TTFT ↔ P99 ITL across varying EC densities.
- **Comprehensive evaluation** on Llama-3.2-1B/3B and Llama-2-7B/13B/70B, four quantization backends (RTN, GPTQ, AWQ, OmniQuant), per-channel and group-128 granularities, single-GPU and TP=2/3/4 on NVIDIA GH200. SPEAR uses < 1% extra model memory and closes 56–75% of the W4→FP16 PPL gap in challenging per-channel settings.

## Methodology

**Problem formulation.** Given a frozen W4 model W_c = Q(W), SPEAR attaches lightweight auxiliary modules θ to recover quality loss while respecting parameter budget b and serving-latency overhead ΔT. The framework treats compensation as a constrained co-design over three coupled components — compensation parameters θ, placement strategy S, and deployment plan Π (kernel fusion, TP communication, scheduling):

min_{S,θ,Π} L(Q(θ))  s.t.  |θ| ≤ b,  T(θ, Π) ≤ T(W_c) + ΔT.

Existing static methods primarily optimize θ while treating S and Π as fixed. SPEAR jointly optimizes all three.

**Algorithm 1 — Input-Adaptive Error Compensator.** The EC is a low-rank update modulated by an input-dependent gate γ:

y = W_c x + α · B(γ(Ax) ⊙ Ax),  with  γ(z) = 1 + tanh(W₂ ReLU(W₁ z + b₁) + b₂),

where A ∈ ℝ^{r×d_in}, B ∈ ℝ^{d_out×r}, and γ is a bottleneck MLP operating entirely in the rank-r latent space. Because γ operates on the low-rank coordinates rather than full hidden dimensions, the EC adds only 8r² + 6r parameters — negligible relative to A, B and far cheaper than token-wise experts or full-width conditional adapters. The "1 + tanh" residual form initializes EC ≈ a static low-rank adapter, which improves calibration stability. Calibration is **data-free**: sequences are self-sampled from the FP16 model and the compensated model is optimized with KL divergence L(θ) = E_x KL(P_fp(·|x) ∥ P_θ(·|x)) in two stages — first (A, B) with γ ≡ 1, then γ with (A, B) frozen.

**Algorithm 2 — CKA-Guided Entropy-Aware Module Selection.** Per-module damage is estimated without training via linear Centered Kernel Alignment between FP16 hidden states H_fp and the hidden states when only module (l, m) is quantized to W4: δ_l^{(m)} = 1 − CKA(H_fp, H_{l,m}). Damage distributions are heavy-tailed and configuration-dependent (Figure 4), so SPEAR (i) computes normalized entropy H_norm of the damage distribution to capture concentration vs. diffusion, (ii) selects the smallest top-K% whose cumulative noise-floor-adjusted damage exceeds a τ_eff(H_norm) ∈ [0.15, 0.6] threshold, and (iii) computes a hybrid placement score score_i = δ̃_i − λ · t_dep_i that protects the highest-damage modules under the raw CKA ranking and uses the hybrid score only for the remaining budget.

**System 1 — Phase-Aware Adaptive Kernel Fusion Dispatch.** A naive EC chain (low-rank projection → gate → reprojection → accumulation) is launched as 10+ separate kernels whose inter-launch gaps dominate compute time on the decode critical path. SPEAR fuses the entire EC chain into the MARLIN 4-bit GEMM epilogue for decode (M = 1), collapsing everything to a single kernel that reuses register/shared-memory resources released after the GEMM main loop. For prefill (M > 1), where the GEMM is compute-saturated, full fusion competes for tensor-core cycles, so SPEAR switches to a semi-fused overlapped path that runs the EC as a separate kernel on a different CUDA stream, captured in a statically replayed CUDA Graph DAG that synchronizes only at the final accumulation.

**System 2 — Epilogue-Integrated Peer Reduction (P2P Dual-Write).** Under tensor parallelism, each GPU computes a local Ax partial Ax_r; the gate is non-linear so gate(Σ_r Ax_r) ≠ Σ_r gate(Ax_r). Applying the gate independently on local TP partials is incorrect (Figure 7 left). A standard fix — launching NCCL AllReduce between GEMM and EC — adds 60–116 µs of standalone sync/query overhead even though only <5 µs of data movement is needed. SPEAR instead folds the AllReduce into the MARLIN epilogue via a P2P dual-write: the epilogue simultaneously writes (a) the local W4 output Y_base to local HBM and (b) the EC activation partial Ax_r to peer-GPU staging memory over NVLink. The remaining EC path (gate, B, residual, RMSNorm) then executes as a fused post-EC tail inside the same kernel, avoiding the standalone NCCL hop.

**System 3 — SLO-Constrained EC-Aware Chunk Scheduling.** Static chunk sizes tuned for plain W4 backbones no longer provide a stable latency-throughput tradeoff once selective EC introduces per-configuration cost variance. SPEAR builds offline microbenchmark latency lookup tables ℓ_EC(M), ℓ_W4(M), ℓ_attn(M); online, given CKA selection S, iteration latency T_S is estimated by aggregating layer entries (EC table for selected layers, backbone table otherwise), and binary search returns the largest prefill chunk c satisfying T_S(d) + T_S(c) ≤ T_SLO for the current decode count d. The lookup-and-aggregate step adds only a few hundred cached table accesses and µs-scale overhead.

**Evaluation setup.** Hardware: NVIDIA GH200 (120 GB HBM3, CUDA 12.6, NCCL 2.27.5); multi-GPU via NVLink. Models: Llama-3.2-1B/3B and Llama-2-7B/13B/70B with trained SPEAR checkpoints. Quantization backends: RTN, GPTQ, AWQ, OmniQuant under W4 per-channel and group-128; SPEAR uses INT8 EC parameters under a fixed memory budget. Quality: WikiText-2 / C4 perplexity and 7-task average accuracy (PIQA, ARC-e/c, HellaSwag, WinoGrande, BoolQ, LAMBADA, MMLU) via lm-evaluation-harness. Baselines: LoftQ, LQER, QERA, EoRA, ASER. Serving baselines: FP16 cuBLAS + FlashAttention, W4 MARLIN, naive W4+EC. Workloads: kernel microbenchmarks (500 timed iterations after 200 warmups), end-to-end decode latency at M=1 with prompt length 128 and 64 generated tokens, multi-GPU TP=2/3/4 over NVLink with NCCL collectives (vLLM stack), and serving-scale replay of 300 ShareGPT requests with Poisson arrivals (Sarathi-Serve).

## Results & Highlights

**Algorithmic quality recovery (Table 1).** Across RTN, GPTQ, AWQ, OmniQuant × per-channel/group-128 × five model scales, SPEAR consistently closes a large fraction of the W4→FP16 gap using < 1% extra model memory:
- Llama-3.2-1B RTN per-channel: PPL 20.46 → 12.40 (75% gap recovery); C4 PPL 31.34 → 18.04; 7-task ZS 0.477 → 0.536.
- Llama-2-7B RTN per-channel: Wiki PPL 6.56 → 5.92 (60% gap recovery); C4 PPL 8.65 → 7.72; ZS 0.634 → 0.653.
- Llama-3.2-3B RTN per-channel: PPL 10.54 → 8.98 (56% gap recovery).
- Llama-2-13B RTN per-channel: Wiki PPL 5.28 → 5.21.
- Gains are larger under per-channel settings where the baseline gap is wider, but group-128 also improves (e.g., RTN 7B: 5.82 → 5.67). SPEAR is **complementary** to strong quantizers like OmniQuant (7B PPL 5.79 → 5.72).

**Quality–memory tradeoff vs. static compensation (Table 2).** Under per-channel RTN, SPEAR achieves the best or tied-best C4 PPL across all five scales while using only **31–66% of baseline memory**:
- 1B: 9.8 MB / PPL 18.04 (vs. QERA 14.8 MB / 19.84; LoftQ/LQER/EoRA/ASER ≈ 22.5 MB / 20.53–25.98).
- 7B: 24.9 MB / 7.72 (vs. QERA 60.3 MB / 7.84; full-rank baselines ≈ 80 MB / 7.81–8.62).
- 70B: 148 MB / 5.84 (vs. QERA 301 MB / 5.88; full-rank baselines 414 MB / 5.88–6.12).
The advantage grows with scale, reflecting SPEAR's selective adaptive placement vs. dense static residual correction.

**Ablation: input-adaptive gate (Figure 10, Table 4).** Per-token loss-reduction analysis on Llama-2-7B shows the gate concentrates compensation on hard tokens (top quartile gains substantially more than the bottom quartile). Removing γ (γ ≡ 1) degrades C4 PPL by 0.18 / 0.41 / 1.03 on 7B / 3B / 1B respectively, confirming the value of input-adaptive modulation. CKA-guided placement consistently outperforms random placement at the same memory budget (Δ PPL 0.2–3.4) and comes within 0.1–0.15 PPL of full-module coverage while using only ~31% of the memory.

**Single-GPU decode latency (Figure 8).** Naive EC insertion inflates decode latency by ~5× over plain W4 MARLIN (1B: 10.1 ms vs. 2.0 ms; 3B: 16.9 ms vs. 3.4 ms; 7B: 19.1 ms vs. 4.7 ms). SPEAR brings latency back to within ~25% of W4 MARLIN: 2.5 / 4.3 / 5.9 ms at 1B / 3B / 7B. At larger scales, SPEAR even **outperforms FP16 decode**: 1.09× FP16 throughput at 3B and 1.25× FP16 throughput at 7B, while smaller models expose weaker bandwidth bottlenecks.

**Multi-GPU tensor-parallel decode (Figure 9, Table 5).** Naive EC deployment inflates TP decode to 31–35 ms at 13B and > 61 ms at 70B. SPEAR brings 13B to 6.0–6.3 ms/token across TP=2/3/4 (matching or slightly beating W4 MARLIN) and 70B to within +6% to +14% of W4 MARLIN while still achieving 1.24–1.44× FP16 throughput. Progressive ablation shows epilogue fusion cuts naive latency by > 2× across all scales, and adding kernel-fused P2P reduction closes the remaining gap (e.g., Llama-2-70B TP=2: 61.27 → 30.28 → 20.64 ms).

**SLO-constrained scheduling (Table 3).** On Llama-2-7B at 16 req/s, static chunked-prefill baselines exhibit a sharp TTFT↔ITL tradeoff — static-512 violates both 22 ms and 16 ms SLOs at all EC densities; static-64 satisfies SLOs but inflates TTFT to 1394.7–2682.8 ms under Dense selection. SPEAR automatically adapts: under 22 ms SLO, ITL stays at 20.8–21.2 ms with TTFT 44.5–60.9 ms; under 16 ms SLO, ITL stays at 15.3–15.8 ms with TTFT 50.4–88.4 ms — up to **2.7× lower TTFT than SLO-compliant static baselines** while maintaining SLO compliance across all densities.

## Critical Assessment

**Novelty.** The paper's strongest contribution is the **algorithmic insight** that quantization error is simultaneously input-dependent and module-concentrated — a structural observation that frames why prior static compensation is fundamentally inefficient. The token-adaptive gate γ(Ax) is a clean and parameter-efficient realization (8r² + 6r), and the entropy-aware CKA selection that maps damage concentration to top-K% is principled rather than heuristic. The systems co-design (phase-aware fusion, P2P dual-write epilogue, SLO-constrained scheduler) is genuinely deployment-grade, not a toy implementation. Compared to ASER's output-aware SVD or QERA's regression-based residual, SPEAR's adaptive gate is a more direct attack on the input-dependence problem, though the gate's additional compute means the systems work was unavoidable.

**Rigor.** Experiments are thorough: four quantizers × two granularities × five scales × two deployment regimes × systematic ablations. Kernel microbenchmarks follow the MARLIN protocol (500 timed after 200 warmup). Baselines are reproduced under identical pipelines. Per-token loss-reduction analysis (Figure 10) and progressive system ablation (Table 5) cleanly attribute gains to individual design choices.

**Reproducibility concerns.** (1) No code link in the excerpted text; the entropy-aware selection threshold τ_eff clipping range [0.15, 0.6] and the λ cost-weighting in score_i are stated but not fully reproducible without reading supplementary. (2) "Self-sampled calibration sequences" from the FP16 model is conceptually clean but the sampling temperature, sample count, and sequence length are not specified. (3) The 70B numbers are reported only for per-channel RTN (Table 2) — broader 70B coverage would strengthen the scale-claim. (4) P2P dual-write timing "+0.4 µs" is mentioned only once in Figure 7; a clearer breakdown of NCCL launch/query overhead reduction would help.

**Significance.** SPEAR pushes the Pareto frontier of low-bit serving: **better quality at lower memory than static compensation, at latency comparable to plain W4**. The 31–66% memory saving vs. QERA at matched-or-better PPL is the single most actionable number for practitioners. The TP P2P dual-write is a generalizable primitive for any system attaching input-dependent computation to quantized layers.

**Red flags.** None major. The "+25% over W4 MARLIN at 1B" is the weakest data point — small models are bandwidth-bound enough that 4-bit advantage is modest, and EC overhead is harder to hide. The work is squarely an engineering systems paper with a clear algorithmic punchline rather than a theory result, so expectations should be calibrated accordingly.

## Strengths

- **Algorithmic clarity.** The two-axis decomposition (input-adaptive × module-selective) is conceptually crisp and well-motivated by the per-token cosine similarity and per-module damage distributions.
- **Strong quality-memory Pareto.** SPEAR uses 31–66% of the compensation memory of the best prior methods at matched-or-better PPL across five model scales from 1B to 70B.
- **Deployment-grade systems engineering.** Phase-aware fusion dispatch, epilogue-integrated P2P dual-write, and SLO-constrained chunk scheduling collectively bring naive EC overhead from 5× to ~25% of W4 decode latency on a single GPU and to within +6–14% of W4 MARLIN on TP=2/3/4.
- **Plug-and-play with existing quantizers.** Works on top of RTN, GPTQ, AWQ, and OmniQuant without modification; gains stack with stronger backends.
- **Comprehensive ablation.** Both algorithmic (gate on/off, CKA vs. random vs. full) and system (fusion on/off, P2P on/off) ablations cleanly attribute contributions.
- **SLO compliance under continuous batching.** The latency-aware scheduler up to 2.7× improves TTFT over SLO-compliant static baselines while maintaining ITL targets across all EC densities.

## Limitations

- **Small-model single-GPU latency gap.** At 1B, SPEAR is +25% over W4 MARLIN (2.5 vs. 2.0 ms/token) and only 7% faster than FP16 — the EC overhead is harder to hide when the bandwidth bottleneck is weaker. The authors acknowledge this scales-related limitation.
- **70B evaluation coverage.** Quality numbers for 70B are reported mainly under per-channel RTN (Table 2); broader 70B × quantization-backend coverage would strengthen the claim that the approach generalizes at the largest scale.
- **Architecture-specific assumptions in deployment.** Phase-aware fusion is tuned for MARLIN-style W4 kernels; portability to W4A16 / W4A8 (ATOM) or LUT-based (T-MAC) serving paths is not evaluated. The P2P dual-write assumes NVLink topology; cross-node or PCIe-only deployments would need a different TP communication strategy.
- **Self-sampled calibration only.** While data-free is a plus, the calibration data characteristics (temperature, sample count, sequence length) and the sensitivity of the entropy-aware K% to those choices are not deeply explored.
- **Limited 3-bit and mixed-precision results.** 3-bit results are deferred to supplementary Appendix D; the headline numbers are W4-focused.
- **No discussion of MoE / multimodal extensions.** The authors explicitly flag this as future work — extending to MoE is non-trivial because expert-routing adds another input-dependent nonlinearity that interacts with the EC gate.
- **Reproducibility.** No public code link is visible in the excerpted text, and several hyperparameters (τ_eff clip range, λ cost weight, calibration sampling configuration) require supplementary material to reproduce exactly.

## Relevance

与本仓库关注的 **LLM 推理系统、量化与高效部署**主题高度相关。SPEAR 解决的"量化误差跨 token 异质、跨模块集中"问题对低比特 LLM 部署具有普遍意义，其 phase-aware kernel fusion、epilogue P2P dual-write 和 SLO-aware chunk scheduling 三项系统优化都是可直接迁移到其他低比特服务栈（vLLM、Sarathi-Serve、TGI）的工程范式。对于关注 4-bit 以下推理、INT4/INT8 补偿与混合精度服务的研究者与工程师，这是一篇算法-系统协同设计的范本。

## Questions for Further Exploration

- 在非 NVLink 拓扑（PCIe-only、跨节点 InfiniBand）下，epilogue-integrated P2P dual-write 的等价原语应该是什么？是否可以用 NVSHMEM 或 RDMA write 做"单次 epilogue 写 + 远端隐式 reduce"？
- CKA-based 模块选择对 MoE 架构是否仍然有效？expert-routing 本身是输入依赖的非线性，与 EC gate 串接后是否会破坏专家负载均衡？
- 熵感知 top-K% 的阈值 τ_eff 区间 [0.15, 0.6] 是经验选择，是否存在基于 Fisher 信息或 Hessian 迹的更原则化替代？
- 把 EC gate 与 speculative decoding 的 draft model 共享参数是否有收益？draft 阶段本身就是输入依赖的，可能进一步摊薄 EC 的 per-token 开销。
- 在 W4A16 / W4A8（ATOM）或 LUT-based（T-MAC）路径下，phase-aware fusion 的最优切分点是否会改变？特别是 activation quantization 引入的额外非线性是否需要新的 epilogue 集成方案？
- 自采样校准数据与真实部署 prompt distribution 的分布漂移如何影响 CKA 损伤估计的稳定性？是否需要在线 CKA 重估计机制？
- 与 QLoRA、LoRA-style 适配器的数学关系：EC 的低秩 (A, B) 是否可以与下游 LoRA 参数合并，从而在推理路径上完全消失？这种"训练期补偿、推理期透明"是否可行？
