01
Methodology · Simulate → Package → Diagnose → Evolve
Methodology

A four-stage agentic loop that edits code, not weights

A 2D incompressible Navier–Stokes solver (WaterLily.jl / BDIM, GPU-capable, Julia) is coupled to a deformable dogfish-like swimmer. The body has a fixed thickness profile w(s) swept on a two-joint bendable spine (joints at s = 1/3, 2/3). Actuation is restricted to joint angular accelerations (ϕ̈₁, ϕ̈₂), clipped under 45° bend, 140°/s rate, 900°/s² acceleration. Global motion (centre-of-mass rc, heading ψ) is an emergent response to hydrodynamic force F and moment Mz integrated over the body surface.

"The agent rewrites the source-code controller — the dual artifact (executable policy + guidance + experience notes) is the actual evolving object."
  • Simulate Candidate — CFD rolls out the current source-code policy.
  • Package Evidence — compact multimodal summaries: aggregate score, per-target diagnostics, sampled trajectories, vorticity keyframes.
  • Diagnose — agent identifies the active failure mode; consults a passive Knowledge Shelf only when local diagnosis demands architectural help.
  • Policy Evolve — LLM rewrites source code, updates paired optimizer guidelines.

Hard constraints on the evolved policy: no hidden state, no random numbers, no target-specific branching — a single readable feedback law for all four canonical targets. Builds on the authors' prior Evolutionary Ensemble of Agents framework (arXiv:2605.09018).

03
Strengths

What genuinely lands

  • Radically interpretable. Every component of the final policy is a named mechanism (signed tail curvature, cadence boost, …) — no black-box weights.
  • Sample-efficient by orders of magnitude. 6 iterations to 4/4; 20 to refinement — versus the thousands of DRL rollouts typical for opaque policy fitting.
  • Generalization demonstrated. 8 held-out static targets + a moving curved target reached by the same iter-20 policy.
  • Hard, physics-rich benchmark. Real 2D FSI with underactuation, hydrodynamic coupling, Re = 2000.
  • Auditable discovery trace. The full source-code evolution log + simulation evidence form a transparent scientific record.
  • Layered growth. Later iterations stack on earlier ones — clean engineering of evolutionary discovery.
04
Limitations

What's still open

  • Single highlighted run — no statistical replicates; seed-to-seed variance unknown.
  • No ablation on Knowledge Shelf content, LLM choice, or population size.
  • No external generalization — different Re, body shapes, 3D swimming, disturbances, currents all untested.
  • No strong baselines. Well-tuned DRL, MPC, or trajectory-optimization not compared head-to-head.
  • Reproducibility gap. Model identity, prompt templates, sampling temperature, code release all missing from the manuscript.
  • Score opacity. Aggregate score's sign convention and weighting only partially described.
  • Narrow validation domain. Targets stay "left-side" of the start; no 180° turn or wrong-pose recovery.
05
Open Questions

Lines worth chasing

  • Does removing the Knowledge Shelf collapse or preserve the emergent architecture?
  • Can the loop transfer to 3D FSI, higher Re, currents, multi-swimmer coordination?
  • Does it scale to other morphologies — eel spines, flapping foils, jellyfish pulsation?
  • What's the LLM's contribution vs. the diagnostic + scoring loop? Swap in CMA-ES over source code to isolate.
  • How does iter count & score vary across LLM seeds, sizes, prompts? Report ≥10 runs.
  • Is the emergent architecture stable, or does it co-evolve with prompt/context? Map the attractor landscape.
  • Could the shelf be replaced by an active retrieval-augmented agent over fresh literature?
  • Can the readable source code be formally verified against reachability / stability specs for physical robots?
