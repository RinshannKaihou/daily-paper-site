> **一句话总结：** AutoMedBench 是一个面向医疗 AI 自主研究的"流程感知"基准，把 24 个跨模态任务统一进 Plan→Setup→Validate→Inference→Submit 五阶段工作流，通过阶段打分 + 五类错误码诊断揭示当前 LLM 智能体"会跑但不会验"的工程瓶颈。

> **TL;DR:** AutoMedBench is a workflow-aware benchmark that evaluates LLM-based agents on 24 end-to-end medical-AI research tasks (segmentation, image enhancement, VQA, report generation, lesion detection) across CT/MRI/X-ray/pathology/microscopy/dental/video modalities. Each run is decomposed into a shared S1–S5 pipeline (Plan, Setup, Validate, Inference, Submit) and scored on both workflow completion (AGENTIC) and deterministic held-out task quality (TASK), with two difficulty tiers (LITE / STANDARD) and five cause-based error codes (E1–E5). Across thousands of runs over six frontier models, validation is the weakest stage on average, verification (37.7%) and submission (38.1%) errors dominate, and one fired error code already cuts overall score by ~48%. The benchmark ships an open-source containerized harness and a public leaderboard.

## 核心贡献

- **面向医疗 AI 自主研究的端到端基准**：首次把"完整科研工作流（理解任务 → 配置环境 → 验证管线 → 跑推理 → 提交产物）"作为评估对象，区别于已有的医疗问答、临床对话、EHR 交互类基准。
- **统一 S1–S5 阶段化评分协议**：每个任务在相同数据/指标下接受 LLM-as-judge（Plan/Setup/Validate）+ 确定性检查（Inference/Submit）打分，权重 25/15/35/15/10，Validate 占 35% 强调"先验证再放大"。
- **LITE 与 STANDARD 双难度层**：固定数据与提交 schema，仅改变任务简报里的脚手架（是否给定方法/依赖/验证提示），从而隔离"模型能力 vs. 脚手架效应"。
- **执行后五类错误码诊断（E1–E5）**：理解 / 数据-模型搭建 / 验证恢复 / 实现执行 / 产物提交，独立于阶段分，可定位"哪里崩"与"为什么崩"。
- **流程分 vs. 任务分双轴刻画**：暴露"工作流能跑完 ≠ 产物真的对"的张力，并指出当前最强的模型（Opus 4.6）成本也最高，便宜模型（GLM-5）可能更划算。
- **开源可复现 harness + Leaderboard**：容器化 agent 与隔离 evaluator 两层设计，禁止读私有 ground truth，违规即零分。

## Key Contributions

- A **workflow-aware benchmark for autonomous medical-AI research**, evaluating agents on end-to-end pipelines rather than isolated predictions or short-form QA.
- A **shared five-stage protocol (Plan → Setup → Validate → Inference → Submit)** with weighted stage scoring (S1=25%, S2=15%, S3=35%, S4=15%, S5=10%) and a complementary **deterministic held-out task score** averaged 50/50 into the OVERALL score.
- **24 tasks across 5 tracks** (segmentation, image enhancement, VQA, report generation, detection) and 7+ modalities (CT, MRI, X-ray, pathology, microscopy, dental, medical video), instantiated under **LITE and STANDARD tiers** (48 task-tier settings).
- **Post-run cause-based error codes (E1–E5)** plus **full interaction trace logging**, enabling diagnostic analysis beyond the final metric.
- **Empirical evidence on six frontier agents** (Opus 4.6, GLM-5, Gemini 3.1 Pro, GPT-5.4, MiniMax-M2.5, Qwen3.5) showing that (i) workflow scores exceed task scores for every model, (ii) Setup is the strongest / Validate the weakest stage, (iii) one fired error code costs ~48% of overall score, and (iv) stronger agents recover better after ≥2 fired errors rather than merely avoid them.
- **Open-source two-container harness** (agent container with GPU + public inputs; offline evaluator with private references), public leaderboard, and GitHub release.

## Methodology

### Benchmark construction
Each task is an instance `T = (D_pub, D_priv, b, A, S, m, τ)`. The agent sees only `D_pub` (mounted read-only) and the task brief `b`, plus a writable workspace; `D_priv` (ground truth) is only mounted in the offline evaluator. A run is one continuous long-horizon interaction (mean ≈ 33 agent turns) inside a sandboxed container with GPU + network access. Two-container isolation prevents data leakage; sandbox violations zero all stage scores.

### Five-stage workflow
- **S1 Plan (25%, knowledge, LLM-judged)**: write `plan.md` with chosen method, dependencies, and validation strategy.
- **S2 Setup (15%, engineering, LLM-judged)**: install deps, load models, verify paths/outputs.
- **S3 Validate (35%, engineering, LLM-judged)**: pilot case + intermediate inspection + self-correction — the highest-weight stage because it gates full inference.
- **S4 Inference (15%, engineering, deterministic)**: completion rate + OOM/timeout checks on full evaluation inputs.
- **S5 Submit (10%, engineering, deterministic)**: schema validation of submitted artifacts.

### Scoring
`OVERALL = 0.5 · AGENTIC + 0.5 · TASK`, with `AGENTIC = 0.25·S1 + 0.15·S2 + 0.35·S3 + 0.15·S4 + 0.10·S5`. TASK uses track-specific metrics: macro-Dice (segmentation), mean SSIM (enhancement), VQA accuracy, report-generation composite (BLEU/METEOR/ROUGE-L/F1RadGraph/micro-P/R/F1), mAP@0.5 (detection).

### Difficulty tiers
- **LITE**: brief specifies method, key dependencies, and per-stage hints.
- **STANDARD**: only bounded method families are named; agent must choose implementation.

### Error codes (E1–E5)
E1 understanding hallucination · E2 wrong dependency / failed model load · E3 skipped validation / ignored bad output · E4 runtime crash / shape mismatch · E5 missing files / wrong format. Labels are diagnostic only — they do **not** enter OVERALL.

### Evaluation protocol
6 agents × 24 tasks × 2 tiers = **288 evaluation cells**, default N=10 runs/cell (5 segmentation tasks use N=20 for variance control). Fixed time cap per task; cost normalized via Table 11 snapshot.

## Results & Highlights

### Overall leaderboard (mean OVERALL across 24 tasks × 2 tiers)
| Agent | Seg | Enh | VQA | Report | Det | **Overall** |
|---|---|---|---|---|---|---|
| **Opus 4.6** | 67.2 | 78.3 | 55.5 | 55.8 | 85.7 | **66.5** |
| GLM-5 | 58.0 | 68.8 | **64.0** | 48.6 | 83.3 | 61.6 |
| Gemini 3.1 Pro | 54.7 | 70.9 | 62.3 | 47.7 | 77.2 | 59.0 |
| GPT-5.4 | 59.4 | 75.1 | 36.5 | 40.8 | 73.8 | 55.3 |
| MiniMax-M2.5 | 46.5 | 74.5 | 55.8 | 28.9 | 80.0 | 51.6 |
| Qwen3.5 | 42.8 | 63.5 | 57.0 | 38.7 | 81.4 | 51.2 |

15.3-point spread; **no uniformly best profile** — the VQA leader (GLM-5) is not the segmentation leader (Opus 4.6).

### Stage-level pattern
- **Setup (S2) is the strongest stage on average; Validate (S3) is the weakest.** Agents are good at producing runnable pipelines but bad at verifying their reliability.
- Across all six agents, **AGENTIC > TASK**, i.e. workflow progress overstates artifact quality.

### Tier sensitivity
Scaffolding does **not** monotonically help. ∆(STANDARD→LITE) is mixed: GLM-5 +8.3%, Gemini +2.8%, Qwen +8.0%, MiniMax +1.7% — but **Opus −0.9% and GPT-5.4 −16.3%**. Extra hints can constrain or distract stronger agents.

### Cost vs. performance
Pearson r between per-track cost and score is weak and track-dependent; VQA shows almost no correlation. **Higher spending ≠ better results** — what matters is whether agents spend compute on validation and recovery. Opus 4.6 has the highest cost, GLM-5 is a cheaper 2nd-best overall.

### Error-code analysis (Fig. 7)
- E3 Verification: **37.7%** of all fired codes.
- E5 Submission: **38.1%** (combined with E3 ⇒ ~76% engineering-shaped failures).
- E4 Execution, E2 Setup: bulk of remainder.
- E1 Understanding: only **0.9%** — rare.
- **One fired code ⇒ ~48% lower overall score**; runs with ≥2 codes stay in the low-score regime.

### Recovery
Strong agents aren't those with the fewest errors; they're those that **recover after ≥2 fired codes** and still complete end-to-end submission. This decouples reliability from raw capability.

## Critical Assessment

**Strengths of the experimental design:**
- The dual-score (AGENTIC + TASK) + stage-level scoring is genuinely novel for medical agents and captures failure modes that final-only metrics erase.
- The E1–E5 error taxonomy is empirically motivated by trace inspection, and the ~48%-per-fired-code cliff is a useful diagnostic.
- Public leaderboard + open harness lowers the barrier to third-party verification.

**Honest caveats — do not rubberstamp:**
- **Only 6 agents evaluated, all frontier proprietary or near-frontier open-weight.** Smaller / specialized / open-source medical agents (e.g. MedSAM-driven stacks, MONAI-Agent, BioMedCLIP-augmented agents) are absent, so the "leaderboard" mostly tells us about the top of the capability distribution.
- **LLM-as-judge for S1/S2/S3 is a known confounder.** The paper does not report inter-judge agreement, human calibration, or whether judge model is shared with the evaluated agents (potential bias if the same model family rates itself).
- **Task score uses official metrics (Dice/SSIM/VQA-accuracy/mAP@0.5/report-composite)** — appropriate but somewhat coarse; clinical utility (e.g. downstream treatment-decision impact) is not measured. The 50/50 OVERALL weighting treats "wrote a good plan" and "made the mask correct" as equally important, which may understate the clinical cost of bad artifacts.
- **Tier sensitivity result is striking but under-explored.** GPT-5.4's −16.3% under LITE is reported without ablation: is it prompt format, hint specificity, or something deeper? Worth follow-up.
- **Cost normalization via Table 11 snapshot** ignores prompt-cache and negotiated enterprise rates, which can shift ordering for hosted models by 3–5×.
- **No statistical significance / confidence intervals** reported on the leaderboard spread (15.3 points); with N=10/20 per cell, run-to-run variance is non-trivial, especially for the LITE/STANDARD comparison.
- **Reproducibility hinges on data access**: MIMIC-CXR requires PhysioNet credentialing and fastMRI requires the NYU DUA, so an external lab can't fully rerun without legal onboarding.
- **Scope claim "medical auto-research" is somewhat aspirational** — all tasks are inference-time; no training/fine-tuning, no real experimental design beyond method selection, no scientific hypothesis generation. The benchmark tests "can an agent run a known pipeline on a known dataset?", not "can an agent pose and answer a new medical-AI question?".
- **No comparison to humans** (e.g. a graduate student running the same pipeline) — without that anchor, the "66.5 is good/bad" calibration is uncertain.

## Strengths

- **Diagnostic depth**: stage-level + error-code analysis surfaces real failure modes (skipped validation, wrong filenames) that are invisible to accuracy-only benchmarks.
- **Heterogeneous coverage**: 24 tasks × 5 tracks × 7+ modalities is the broadest medical-agent benchmark to date and the only one with a unified cross-task workflow.
- **Engineering rigor**: two-container isolation, fixed cost snapshot, N=10/20 replication, atomic ledger via `pipeline.py` semantics, no vendor-specific orchestration.
- **Practical leaderboard signal**: the recovery-after-≥2-errors finding reframes "agent reliability" from error-counting to error-tolerance.
- **Open infrastructure**: GitHub + leaderboard + containerized harness lowers entry cost for follow-up work.

## Limitations

- LLM-judge bias and lack of human calibration for S1/S2/S3 scores.
- Limited agent coverage (6 frontier models); no open-source / specialized medical agents.
- No training-time tasks; "auto-research" is restricted to inference-time pipeline execution.
- No comparison to human researcher performance; absolute calibration is unclear.
- Some datasets require credentialed access (MIMIC-CXR, fastMRI), limiting full external reproducibility.
- 50/50 weighting of AGENTIC and TASK may underweight clinically critical task accuracy.
- Cost analysis uses a static price snapshot without prompt-cache or negotiated discounts.
- No reported confidence intervals on leaderboard ordering; N=10–20 per cell is modest given observed variance.
- Some failure modes (e.g. silent numerical drift in SSIM outputs) may evade both stage scoring and error codes.

## Relevance

对当前关注 **医疗 AI agent / 自主科研 / Agent 评估** 的研究者非常相关：它首次让"流程化质量"成为医疗 agent 评测的一等公民，并揭示了 S3 Validate 是真正的瓶颈——这与软件工程 agent 基准（SWE-bench, MLE-bench）的发现一致，但被搬到了医疗影像与多模态场景。对于做 AutoResearch、agentic workflow、benchmark 设计的工作，本文的双层打分 + E1–E5 错误码 + LITE/STANDARD tier 模板可直接借鉴。

## Questions for Further Exploration

1. **如何把 S3 Validate 从 LLM-judge 升级为可执行的自动化检查？** 比如让 agent 必须在 Validate 阶段生成一组 unit tests / sanity checks（output shape、dtype、空文件检测、gold sample 对比），并据此打分而非依赖 judge。
2. **GPT-5.4 在 LITE 下 −16.3% 的反直觉下滑** 是 prompt 模板、过拟合 STANDARD 风格，还是 LITE 提示里的某条具体建议误导？需要 controlled ablation。
3. **Recovery-after-errors 能否成为新的可靠性指标**？是否应把"在 ≥2 个 fired error 后仍提交 scoreable artifact"加入 OVERALL？
4. **能否引入人类医学生 / 临床研究者 baseline**？哪怕只跑 3–5 个任务，也能给 OVERALL 提供绝对刻度。
5. **训练时扩展（fine-tune / continual pretrain）是否在 AutoMedBench 上有合法路径？** 当前 inference-only protocol 排除了"用 LoRA 微调 nnU-Net"这类真实医学 AI 工作流。
6. **LLM-judge 与被评 agent 同源时的偏差有多大？** 用一个 held-out judge family（e.g. Claude judge on GPT-5.4 产物）做交叉验证即可回答。
7. **错误码能否在线触发干预**？例如 E3 验证错误一旦出现，harness 自动注入一道"重新检查输出"的强制步骤，观察 score 恢复情况。
8. **多模态扩展**：超声、内窥镜、皮肤镜、眼底这些 AutoMedBench 未覆盖的模态，stage 评分协议是否仍合理？
