# SPARQLe: Sub-Precision Activation Representation for Quantized LLM Inference

> **一句话总结：** SPARQLe 是一个面向 W4A8 量化大模型推理的硬件–软件协同设计框架，它将每个 8-bit 激活张量分解为稠密的 LSB4 + 由精度位图压缩的稀疏 MSB4，并配套一种轻量级按列剪裁算法与一套能直接在该混合格式上运行的混合 PE 加速器，从而在保持 8-bit 激活精度的同时把整型算术路径降到 Int4×Int4，实现了 16–24.3% 的 prefill 延迟下降与 6.5–26.7% 的能耗下降。

> **TL;DR:** SPARQLe exploits the statistical fact that a large fraction of LLM activations are concentrated near zero, causing their most-significant 4 bits (MSB4) to be zero. It decomposes each 8-bit activation tensor into a dense LSB4 tensor and a sparse MSB4 tensor compressed via a precision bitmap (PBM), and boosts MSB4 sparsity through a column-importance–aware clipping algorithm with both global and per-layer learnable clipping constants. A hybrid accelerator operates directly on this representation using Int4×Int4 MACs, packing ~50% more PEs than an iso-area Int8×Int8 dense baseline. Evaluated on BitNet 3B (W2A8KV4), Llama2-7B (W4A8KV4), and Llama3-8B (W4A8KV4), SPARQLe cuts prefill latency by 16–24.3%, decode latency by 13.5–23.4%, prefill energy by 17–26.7%, and decode energy by 6.5–14.2%, with only minor accuracy loss — and crucially without altering the underlying quantization scheme.

## 核心贡献

- **亚精度激活表征 (Sub-precision activation representation)**：提出把每个 8-bit 激活张量结构化分解为稠密 LSB4 + 稀疏 MSB4 + 精度位图 PBM 三件套，并设计基于「权重列重要性」(L1 范数最小的 k 列) 的轻量级剪裁算法，提供「全局」与「逐层可学习」两种精度–效率权衡模式。
- **硬件–软件协同设计 (Hardware–software co-design)**：提出统一混合 PE，既能用同一套 Int4×Int4 MAC 处理稠密 LSB4 计算，又能在稀疏 MSB4 计算阶段通过 PBM 旁路掉被剪掉的激活，省去「解压–计算–重压」循环；相比等面积 Int8 基线多放置约 50% 的 PE，且面积/功耗开销仅 5.5% / 7%。
- **与量化方法正交互补**：不动基础量化方案，直接作用于已量化激活；和现有 W4A8 / W2A8 量化方案天然可叠加；PBM 阈值由列重要性决定，使得精度–效率可按列细粒度控制。

## Key Contributions

- **Structured 8-bit activation decomposition.** Each activation tensor is split into a dense LSB4 tensor, a sparse MSB4 tensor compressed by a Precision BitMap (PBM), and the PBM itself — yielding a formal compression-ratio expression $C\% = \frac{sp/2 - 1}{p} \cdot 100$ and ops-reduction $O\% = \frac{s \cdot p}{2}$ for natural sub-precision sparsity $s$.
- **Lightweight, importance-aware clipping algorithm.** A column-importance mask (built from the L1 norm of each weight row) restricts clipping to the $k$ least-important columns, and clipping constants $\ell, h$ are either calibrated globally or learned per layer as trainable parameters while base weights remain frozen. The penalty term in the loss function explicitly trades accuracy for MSB4 sparsity.
- **Hybrid accelerator operating directly on the decomposed format.** A unified PE array with shared Int4×Int4 MACs, sparsity logic (PBM-driven operand gating), bypass-able sparse-load datapath, and a drain unit with MSB4/LSB4 splitters — eliminating decompress–compute–recompress cycles and amortizing sequential dense–sparse execution whenever MSB4 sparsity exceeds ~30%.
- **Complementary to quantization.** Drop-in on top of W4A8/W2A8 quantized models; validated on BitNet-3B, Llama2-7B, and Llama3-8B with up to 24.3% prefill-latency reduction, 23.4% decode-latency reduction, and 26.7%/14.2% prefill/decode energy reduction.

## Methodology

**Representation.** For 8-bit two's-complement activations, values in $[0, 15]$ already have MSB4=0 (or sign-extension zeros). SPARQLe generalizes this by storing the LSB4 of every element densely, the MSB4 of only the elements that need it, and a 1-bit PBM indexing which elements have a non-zero MSB4. With $s$ = fraction of activations whose MSB4 is zero, the format saves roughly $\frac{s \cdot p}{2}\%$ of operand bandwidth and compute, while preserving the original Int8×Int4 / Int8×Int2 numerical result.

**Sub-precision sparsity enhancement.** Two clipping strategies widen the natural zero-MSB4 range:
- *Global:* a single $[\ell, h]$ is calibrated offline on WikiText by sweeping accuracy/sparsity tradeoffs.
- *Per-layer (fine-grained):* $\ell_L, h_L$ become trainable parameters per linear layer; only these (and the column mask) are trained while base weights are frozen. The loss $\mathcal{L} = \text{MSELoss}(M_{\text{clip}}(D), M_{\text{base}}(D)) - \alpha \sum_L \frac{1}{N_L} \sum_i \text{mask}_{L,i}$ jointly preserves fidelity and rewards more clipped columns.

The clipping mask is restricted to the $k$ least-important columns of each activation matrix — identified by the L1 norm of the corresponding weight rows — so that quantization error multiplies with low-magnitude weights.

**Hardware.** The accelerator uses an output-stationary dataflow. Each PE holds separate LSB4 / MSB4 / PBM / weight / output register files, a shared Int4×Int4 MAC, and a sparsity-acceleration block that gates operands via PBM during the sparse phase. The load unit shares one SRAM-to-PE datapath for both dense and sparse flows (with a bypass-able sparse-byte-select module for compressed MSB4+PBM). The drain unit uses per-column MSB4/LSB4 splitters to keep two parallel write-combine streams (8B/cycle LSB4 + 16B/cycle MSB4) flowing to separate SRAM banks.

**Pipelining.** As shown in Fig. 5, activations/weights are prefetched ($t_0$–$t_1$), dense LSB4 compute runs first ($t_2$–$t_3$), sparse MSB4 compute follows ($t_3$–$t_4$) using the same MACs, and tiles are drained only after both partial sums are accumulated ($t_7$–$t_8$). Sparse partial sums are left-shifted by 4 bits before accumulation to recover Int8×Int{4,2} numerical equivalence.

**Experimental setup.** Three models — BitNet-3B (W2A8KV4), Llama2-7B (W4A8KV4), Llama3-8B (W4A8KV4, SmoothQuant/QServe). Evaluation: WikiText perplexity + LM-Harness (BQ, OQ, PQ, ARC-E/C, WGe) at sequence length 2048. BitNet uses per-layer clipping over 23 calibration iterations; Llama models use global clipping. Hardware comparison is against an iso-MAC 256-PE Int4×Int4 dense accelerator (3-level memory, 1.5 MB SRAM, 1 GB DRAM), synthesized at 7 nm in Synopsys DC, and modeled with an analytical energy–latency cost model similar to FlexNPU.

## Results & Highlights

- **Latency.** Average prefill (TTFT) speed-ups of **24.3% / 17.2% / 16.0%** and decode (TPOT) improvements of **23.4% / 14.6% / 13.5%** for BitNet-3B / Llama2-7B / Llama3-8B respectively.
- **Energy.** Prefill energy savings of **26.7% / 18.4% / 17.0%** and decode energy savings of **14.2% / 7.1% / 6.5%** for the same models.
- **Compute vs. data-transfer.** Compute acceleration 16.9–27.1%; data-transfer acceleration 14.2–24.4%. Unlike most compression work that targets only one, SPARQLe hits both because it compresses the operand format *and* avoids recompression in compute.
- **MSB4 sparsity.** 61.8% / 47.0% / 44.4% average across BitNet-3B / Llama2-7B / Llama3-8B — comfortably above the ~30% breakeven needed to amortize sequential dense–sparse execution. BitNet wins because its 2-bit weights leave more activation headroom and it uses per-layer clipping.
- **Accuracy.** Perplexity rises modestly (Llama3-8B: 6.72 → 7.05; Llama2-7B: 5.67 → 6.00; BitNet-3B: 10.21 → 12.19). On ARC-E/C, SPARQLe *outperforms* the corresponding W4A4 baseline by 3.82% (Llama2-7B) and 5.47% (Llama3-8B) on ARC-E and by 3.23% (Llama3-8B) on ARC-C — i.e., it occupies the favorable operating region between W4A8 (more accurate, more expensive) and W4A4 (cheaper, much less accurate).
- **Hardware cost.** Only +5.5% area and +7% average power relative to the iso-MAC Int4 baseline; these overheads are dwarfed by runtime savings.
- **Sensitivity.** The $k$ sweep shows that even $k=0$ gives 35.6% natural MSB4 sparsity (boostable to 52% with clipping); benefits are largest on `o_proj` and `down_proj` (more Laplacian-like activations in attention and FFN intermediate projections).

## Critical Assessment

**Novelty.** Genuinely new framing. Most prior work on activation quantization tweaks the *quantization scheme* (rotation, scaling, mixed precision, etc.). SPARQLe's insight is that within an already-quantized 8-bit activation, the *sub-operand* MSB slice is often all-zero — a redundancy that sits between classical dense and sparse regimes. Neither existing dense nor sparse accelerators can exploit it directly. The hardware is essentially a purpose-built dual-mode datapath for that exact niche.

**Rigor.** The comparison is iso-MAC and iso-area aware (256 Int4×Int4 MACs vs. 256 Int8 baseline scaled by bit-width, with ~50% more PEs at Int4 — they make the area/Power delta explicit). The cost model is built on FlexNPU, an independently published accelerator. They report perplexity *and* downstream tasks, and they explicitly position themselves between W4A8 (accuracy ceiling) and W4A4 (efficiency ceiling) on the operating-point frontier.

**Reproducibility.** Model and dataset choices are standard; clipping-constant calibration is described algorithmically (Algorithm 1). Hardware RTL synthesis conditions (7 nm, Synopsys DC) and memory hierarchy are spelled out. No code release mentioned in the extracted text — a notable gap for an architectural paper.

**Significance.** For edge LLM deployment, where decode latency/energy dominates, getting ~14–23% decode speedup with <1 percentage-point perplexity hit on top of an already W4A8 model is a meaningful incremental win. The wins stack with quantization, which is rare.

**Red flags / things to watch.**
- Accuracy on ARC-C drops from 52.05 → 46.92 for Llama3-8B and ARC-E from 80.72 → 75.88 — non-trivial downstream regressions even though perplexity is fine.
- Global clipping is used for Llama models (per-layer only for BitNet), and the paper notes this "limits per-layer sparsity adaptation" — so the headline numbers are not the best SPARQLe could do.
- No wall-clock silicon numbers; the energy/latency figures come from an analytical cost model. Behavior on real DRAM (excluded from the model) and across diverse batching strategies is not shown.
- Only three models, all decoder-only transformers — claims about broader applicability (CNNs, MoE, multimodal) are forward-looking.

## Strengths

- **Sub-precision sparsity is a genuinely under-explored niche** between dense and sparse compute, and SPARQLe names it crisply.
- **Hardware–software co-design is coherent**: the format, the clipping algorithm, and the PE architecture reinforce each other; the ≥30% MSB4 breakeven analysis is a clean engineering argument.
- **Complementarity, not replacement**: SPARQLe stacks on top of existing W4A8/W2A8 quantization, lowering adoption friction.
- **End-to-end validation**: perplexity + LM-Harness + analytical area/power, all on real quantized LLMs.
- **Favorable operating-point positioning vs. W4A4** on accuracy benchmarks while still delivering W4A4-class efficiency — i.e., it gets more of both than the standard W4A8 vs W4A4 trade.
- **Practical knob (k)**: the column-importance mask + tunable $k$ gives users a clean way to trade accuracy for efficiency post-training.

## Limitations

- **Modest accuracy regression on harder benchmarks** (ARC-E/C drops of ~5 pp on Llama3-8B) — may matter for safety-critical or reasoning-heavy applications.
- **Only Int4×Int4 PE precision** in the presented accelerator; Int8×Int8 reuse of MSB4 + LSB4 reconstructed values is not the primary path, so workloads that need true Int8 throughput won't fully benefit.
- **Global vs per-layer clipping inconsistency** across models makes the "headline sparsity" hard to interpret as a single number — BitNet uses per-layer, Llama models use global.
- **No public code release** mentioned; no silicon measurements; DRAM is excluded from the cost model.
- **Sequential dense–sparse execution** is amortized only when MSB4 sparsity > ~30%; for models that don't naturally hit this, SPARQLe's gains shrink.
- **Limited model coverage**: BitNet, Llama2, Llama3 are all homogeneous decoder-only transformers; the paper does not address encoder-decoder or MoE architectures.

## Relevance

与本人研究方向高度相关：本文聚焦 W4A8 / W2A8 量化 LLM 在边缘推理场景下的「激活侧」瓶颈，提出亚精度（sub-precision）稀疏这条介于稠密与稀疏之间的第三条路径，与本人关注的量化协同设计、激活分布稀疏性、定制加速器微架构等议题高度契合，可作为量化感知的硬件协同设计的优秀参考案例。

## Questions for Further Exploration

- How does SPARQLe compose with KV-cache compression schemes (e.g., GEAR, KIVI, H2O)? Are the MSB4 sparsity gains preserved across long contexts, or does attention's softmax-bounded range hurt the Laplacian-like distribution?
- Can the per-layer learnable clipping constants be merged with QAT/LoRA-style adapters so that the same approach handles downstream fine-tuned models without retraining $k$, $\ell$, $h$ from scratch?
- The accelerator uses Int4×Int4 MACs — what would the area/energy picture look like if the PEs were Int4×Int2 (for W2A8 models like BitNet) or if Int8×Int8 dense fallback existed for sparsity-starved layers?
- The cost model excludes DRAM. With modern LPDDR5X bandwidth limits, how much of the decode-time speedup actually survives when memory energy is included end-to-end?
- How transferable is sub-precision sparsity to other activation-heavy layers beyond linear projections — e.g., convolutions in vision-language models, MoE expert routing decisions, or attention's QK^T/Softmax stages?
- Is there a way to make the column-importance mask *dynamic* (per token) without losing the offline pre-computation benefit, given that LLM activations are explicitly input-dependent?
- Does the format compose with weight-only sparsity methods (2:4 sparse, Wanda), or do the bit-packing assumptions conflict?
