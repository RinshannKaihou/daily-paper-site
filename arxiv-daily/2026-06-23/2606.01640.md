> **一句话总结：** MobEvolve 提出了首个面向人类出行生成的"智能体自演化启发式系统"，将可解释的活动-出行启发式分解为四个行为子模块，并由 LLM 智能体通过 rollout-attribution-evolution-memory 四阶段循环迭代优化内部逻辑，在 Singapore 与 Montreal 两个城市基准上同时达成 SOTA 的分布对齐、行为合理性，并保持毫秒级推理速度。

> **TL;DR:** MobEvolve is the first agentic self-evolving heuristic framework for human mobility generation. It initializes a behavior-inspired heuristic system (factorized into travel participation, mobility retrieval, individual adaptation, and behavioral refinement) and uses an LLM coding agent to iteratively evolve its internal logic via validation-driven attribution, targeted heuristic updates, and an accumulated evolution memory. On Singapore and Montreal benchmarks, MobEvolve beats state-of-the-art deep generative models and LLM-based baselines in trip fidelity, distributional alignment, and behavioral plausibility (e.g., −34.1% overall error on Singapore), while preserving interpretability and achieving 7.65 ms/sample inference — orders of magnitude faster than LLM-based methods.

## 核心贡献

- **首个智能体自演化启发式生成框架**：将可解释的出行启发式建模、行为合理的出行链合成、经验分布对齐和高效人口级生成首次统一在同一个系统中，打破了传统启发式、深层生成模型和 LLM 方法只能满足部分需求的现状。
- **数据驱动的演化机制**：MobEvolve 用训练集模板库初始化启发式，然后通过"验证集驱动的归因"——对比经验数据与生成结果，定位经验性偏差与失败案例——由 LLM 智能体（Codex + GPT-5.5）提出针对性更新，并通过演化记忆（attribution_record/ 与 trial_memory/）累积有效或失败的经验，实现累积式自我改进。
- **行为可分解的生成器**：将出行链生成因子化为四个行为决策——出行参与 (p(zi|xi))、移动检索 (p(ri|xi,zi))、个体适配 (p(ỹi|xi,zi,ri))、行为精修 (p(ŷi|ỹi,xi))——既保留行为可解释性，又为智能体提供有意义的搜索空间。
- **在 Singapore 与 Montreal 两个真实城市基准上同时取得 SOTA**：Singapore 上 Overall 分数 0.5176，相比最佳基线（AIRL BC 0.7854）相对下降 34.1%；Montreal 上 Overall 分数 1.6762，相对最佳基线（AIRL BC 1.8790）下降 10.8%；推理延迟仅 7.65 ms/样本，与轻量级 MNL（3.05 ms）同量级，远快于 LLMob（2200–3200 ms）。

## Key Contributions

- **First agentic self-evolving heuristic generator for human mobility.** MobEvolve unifies interpretable heuristic modeling, behaviorally plausible synthesis, empirical distributional alignment, and efficient population-scale generation in a single framework — properties that prior paradigms (deep generative models, LLM-based methods, traditional heuristics) could only satisfy partially.
- **Novel data-driven evolution mechanism.** The system initializes its heuristic state from training-set template libraries (routine templates, segment-level participation rates, slot marginals, transition statistics, location marginals, destination-choice evidence, region land-use features, spatial centroids), then evolves through a validation-driven attribution–proposal–verification–memory cycle, achieving cumulative self-improvement with a fully auditable trail.
- **Behaviorally grounded factorization of trip-chain generation.** The generator is decomposed into four interpretable decisions — travel participation, mobility retrieval, individual adaptation, and behavioral refinement — which provide the LLM agent with a meaningful, localized search space for heuristic revision.
- **State-of-the-art empirical results.** MobEvolve achieves best overall score on both benchmarks (Singapore 0.5176 vs. 0.7854 best baseline; Montreal 1.6762 vs. 1.8790 best baseline) with consistent gains across Individual, Distribution, and Behavior metrics, while running at 7.65 ms/sample.

## Methodology

**Problem formulation.** Given individual features xi ∈ X (age, gender, household, home location, mobility access, occupational status) and a reference mobility dataset D = {(xj, yj)} of typical-day 24-slot trip chains, the task is to learn a conditional generator G ∈ G that produces trip chains ŷi = {(ai,t, li,t)} aligned with target observations Dtar under an evaluator Emob of behavioral plausibility and distributional alignment.

**Interpretable Mobility Heuristic System (the evolvable basis).** Generation factorizes as:
  p(ŷi | xi) = p(zi | xi) · p(ri | xi, zi) · p(ỹi | xi, zi, ri) · p(ŷi | ỹi, xi)

- **Travel Participation (p(zi | xi))**: Decides whether person i conducts out-of-home activities given socio-demographics.
- **Mobility Retrieval (p(ri | xi, zi))**: Selects an empirical trip chain from behaviorally similar records in D as a routine pattern.
- **Individual Adaptation (p(ỹi | xi, zi, ri))**: Adjusts the retrieved pattern to reflect individual behavioral variation.
- **Behavioral Refinement (p(ŷi | ỹi, xi))**: Transforms the adapted chain into a coherent, behaviorally plausible output.

**Self-Evolution loop (4 stages per round).** The LLM agent autonomously evolves the four components via:

1. **Mobility Rollout.** Apply current generator G(k) to evolution population Xtar, producing synthetic trip chains Ŷ(k) under a fixed random seed.
2. **Distributional-Behavioral Attribution.** Compare Ŷ(k) with Dtar via the evaluator Emob to produce an attribution record ∆(k) that localizes failures by time slot, persona segment, and representative trip chains.
3. **Heuristic Evolution.** Given G(k), ∆(k), and evolution memory M(k), the agent proposes a candidate G̃(k+1) by updating one targeted heuristic module; the candidate is promoted only after empirical verification.
4. **Memory Evolution.** The completed trial τ(k) — failure mode, hypothesis, intervention, outcome, accept/reject decision — is appended to M(k+1), so that later rounds are cumulative rather than independent searches.

**Implementation.** Modular Python system under `hs/`, with frozen data, frozen evaluator, and an evolvable generator. The coding agent is Codex + GPT-5.5 with high reasoning effort; verification includes regression checks, validity tests, and the empirical acceptance rule. Train-derived index artifacts (templates, segment participation, slot marginals, transitions, location marginals, destination choice, region features) are part of the heuristic state, not learned parameters.

## Results & Highlights

**Held-out test scores (Table 1, lower is better).**
- **Singapore**: MobEvolve Overall = 0.5176 (Individual 0.3839, Distribution 0.0578, Behavior 0.0759) — beats the strongest baseline (AIRL BC 0.7854) by **34.1%** relative reduction; Behavior score is 3.8× better than AIRL BC's 0.2880.
- **Montreal**: MobEvolve Overall = 1.6762 (Individual 1.4530, Distribution 0.1060, Behavior 0.1172) — beats the strongest baseline (AIRL BC 1.8790) by **10.8%**; Behavior score is 2.2× better than AIRL BC's 0.2556.
- LLM-based baselines remain weak on Distribution (LLMob zero-shot 2.3958 / 4.7400); deep generative models often suffer high Behavior errors; only MobEvolve consistently improves all three dimensions.

**Generation efficiency (Table 2, Singapore test, ms/sample).**
- MobEvolve: **7.65 ms** — same order of magnitude as MNL (3.05 ms) and Markov (0.44 ms).
- LLMob few-shot: 3200 ms; LLMob zero-shot: 2200 ms; TVAE 180 ms; CTGAN 240 ms.
- MobEvolve effectively breaks the accuracy–speed trade-off: best quality + high throughput.

**Self-evolution trajectory (Figure 3, Singapore validation).**
- Validation score decreases from 0.8621 (initial heuristic) to 0.5118 (final evolved heuristic) — a **40.6% relative reduction** over hundreds of rounds.
- Improvements are not smooth; they correspond to accepted updates targeting concrete behavioral failures (e.g., "Keep senior workers at Work later on weekends", "Remove unrealistic extra Work hours", "Preserve reliable retrieved routines").
- Distinct phases: early updates make adaptation more conservative (Behavior 0.0983 → 0.0692); middle updates preserve sparse interior Travel anchors; later updates refine population-segment-specific schedules (homemaker vs. worker, senior weekend, etc.).

**Ablations (Table 3, Singapore).**
- Designed initial heuristics (with 4-component structure): Val 0.8621 → Test 0.8539.
- Unstructured initialization (monolithic, no 4-component priors): Val 7.8753, Test 7.0256 — evolution improves it to Val 0.8753, Test 1.0256, but it cannot recover the structural prior; final score 2× worse than structured init.
- Different random seeds (2026 vs 2012) produce comparable trajectories, showing the system is not driven by a single "lucky" path.

**Interpretability case studies.**
- 55-year-old Singapore worker: MobEvolve preserves the workday structure (Home → Travel → Work → Home with only 1-hour shift at the commute boundary), while DBC places Work at midnight and breaks daily structure.
- Montreal student: MobEvolve matches 22/24 joint activity-location slots and preserves the school-day loop; DBC matches only 3/24 and keeps the student in an incorrect school region.
- Singapore evolved heuristics: group-conditioned rhythms (Home anchors most of the day; Travel marks sparse transitions; Work appears only when supported by role and surrounding schedule).
- Montreal evolved heuristics: spatial allocation (Work/School destinations adjusted by home region, population group, weekday context, activity type).

## Critical Assessment

**Novelty.** The contribution is twofold and genuinely new. (1) The self-evolving heuristic pattern — where an LLM agent evolves the internal logic of an interpretable, behaviorally structured generator — has not been applied to human mobility generation before; prior self-evolving agents (FunSearch, Eureka, AlphaEvolve, Darwin Gödel Machine, CORAL, Meta-Harness) focus on math/reward/algorithm/harness discovery. (2) The specific behaviorally grounded factorization (participation → retrieval → adaptation → refinement) provides a meaningful, interpretable search space for the agent — without it, the ablation in Table 3 shows unstructured init converges to a solution 2× worse.

**Rigor.** The paper is empirically strong: two real-world datasets (Singapore HTS 2012–2013 and Montreal 2023), a frozen evaluator with detailed per-component JSD diagnostics, a comprehensive baseline set spanning 3 paradigms (traditional, deep generative, LLM-based), and a meaningful ablation isolating the contribution of structural prior vs. evolution. The system organization (frozen evaluator + frozen data + evolvable hs/) prevents leakage and makes evolution auditable. However, some rigor questions remain: (i) the agent is Codex + GPT-5.5 with "high reasoning" — closed-source, version-pinned but not reproducible in detail; (ii) the "audit trail" framing depends on the agent's attribution honesty, which is not independently verified; (iii) only two cities and only typical-day trip chains are tested — generalization is asserted but not demonstrated.

**Significance.** For urban planning, this matters: policy simulations require interpretable, fast, population-scalable mobility generation with empirical alignment. MobEvolve is the first method to deliver all four properties in a single system. The audit trail per round is genuinely useful for planners: each performance jump can be linked to a behavioral failure and a heuristic update.

**Red flags.** (1) "Self-evolving" risks overfitting to the validation evaluator — the paper acknowledges this in limitations but does not run evolution on a different split to bound the risk. (2) "Interpretability" is partly cosmetic: the four components are conceptual, and the actual code in `hs/` is still procedural Python that requires effort to read. (3) Heavy reliance on Codex + GPT-5.5 makes exact replication hard; paper provides "Codex notes" and "code diff" artifacts, but the prompt/chain is not fully open. (4) The agent can propose "overly specific" revisions — a concern the authors raise but do not quantify. (5) The terminology "agentic" is in vogue; the system is, in practice, a structured evolution loop with an LLM in the proposal step, not a fully autonomous agent in the sense of long-horizon planning.

## Strengths

- Unifies **all four** desired properties (interpretability, behavioral plausibility, distributional alignment, efficient inference) in one system — a genuine first.
- Strong empirical gains: **−34.1%** (Singapore) and **−10.8%** (Montreal) overall error vs. strongest baselines, with consistent improvement across Individual, Distribution, and Behavior metrics.
- **Audit trail per evolution round** — each performance jump is linked to a concrete behavioral failure and a targeted heuristic update, making the system genuinely inspectable.
- **Inference latency 7.65 ms/sample** — orders of magnitude faster than LLM-based methods (2200–3200 ms) and competitive with the lightest baselines, breaking the accuracy–speed trade-off.
- Comprehensive baseline set spanning 3 paradigms; rigorous ablation isolating the role of the structural prior; multi-seed robustness check.
- Detailed metric definitions in the appendix (Singapore: 17 behavior JSDs; Montreal: 10 behavior JSDs) make the evaluator reproducible.
- Strong, well-balanced author team with both domain and methodological depth.

## Limitations

- **Reference-data dependency**: the system depends on the quality and coverage of empirical observations for retrieval, evaluation, and evolution; under-represented population segments, rare patterns, or spatial contexts will limit what the agent can discover or repair.
- **Limited scope of evaluation**: only two cities (Singapore, Montreal) and only typical-day trip chains over 24 hourly slots; transfer to other cities, finer temporal resolutions (e.g., 15-min slots), multi-day horizons, or richer mobility constraints is not demonstrated.
- **Overfitting to the evaluator**: although the system uses a frozen evaluator + validation split, repeated evolution rounds can overfit; the paper notes this but does not quantify the gap.
- **Agent effectiveness is not guaranteed**: the LLM agent may still propose ineffective or overly specific heuristic revisions, particularly for long-tail failures; the system relies on fixed evaluators and validation procedures to mitigate this.
- **Interpretability is partial**: the four-component structure is conceptual; the actual implementation is modular Python whose readability is not formally evaluated (e.g., with a user study).
- **Closed-source agent stack**: dependence on Codex + GPT-5.5 with high reasoning makes exact replication difficult; only "notes" and "diffs" are released, not full prompts or evolution chains.
- **Single-trip-maker-per-record design**: each individual has a single trip chain; the system does not model within-person variability or day-to-day dynamics.
- **No privacy or fairness audit**: although the Ethical Considerations section raises these concerns, the paper does not run segment-level bias audits (e.g., across income, age, geography) to quantify them.

## Relevance

- 与研究兴趣高度相关：该工作把 LLM 智能体自我演化范式（self-evolving agents）落地到一个具体可解释的城市出行生成任务中，并提供完整的 rollout-attribution-evolution-memory 演化闭环设计，是 agentic systems + urban computing 的代表性交叉工作。

## Questions for Further Exploration

- How does MobEvolve transfer to cities with substantially different mobility cultures (e.g., car-dominated North American suburbs, transit-rich East Asian megacities, or low-income settings with sparse data)? Does the template library + retrieval heuristic generalize, or does it need a different factorization?
- Can the four-component factorization be replaced or extended (e.g., adding a destination-choice module, a mode-choice module, or a multi-day dependency module) and still benefit from agentic self-evolution?
- What is the right way to bound overfitting to the frozen evaluator? Would a "held-out evolution set" or cross-city transfer evaluation reveal how much of the gain is validation-specific?
- How sensitive is performance to the choice of LLM agent (e.g., GPT-5.5 vs. Claude Opus 4 vs. open-source models like DeepSeek-Coder or Qwen-Coder)? Is the evolution loop robust to weaker agents, or does it require frontier reasoning?
- Can the evolution memory be turned into a learned, retrieval-augmented module (e.g., vector store of accepted/rejected trials) to scale to thousands of rounds without context bloat?
- How does the system behave when the reference data is intentionally biased (e.g., under-represents low-income or minority groups)? Does the agent surface the bias in its attribution record, and can it be steered to repair it?
- Is the "audit trail" actually used by downstream urban planners? A user study with transport-authority analysts would be a strong follow-up.
- Can the framework be extended beyond trip chains to joint simulation of mobility + energy consumption, or mobility + land use, where the same behavioral factorization might apply?
- What is the cost–benefit of the modular Python implementation vs. a neuro-symbolic representation that is more compact and easier for the agent to edit without breaking constraints?
