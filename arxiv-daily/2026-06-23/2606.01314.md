> **一句话总结：** SkillSmith 提出了一种协同感知的能力–工具共同进化框架，通过原子化提案、生态动力学（Lotka–Volterra）和反模式记忆三类机制，让 LLM 智能体在技能库、工具库与失败记忆三者之间联合演化，从而突破现有技能进化系统在工具层固定、单点评估和失败经验丢失三方面的瓶颈。

> **TL;DR:** SkillSmith is a **synergy-aware skill–tool co-evolution framework** for LLM agents that jointly evolves three external, non-parametric state components — Skills $S_t$, Tools $T_t$, and an anti-pattern memory $F_t$ — via atomic bundle proposals. It introduces three innovations: (1) **bundle-based co-evolution** with five typed tool lifecycle operations (WRAP / EDIT / COMPOSE / SPLIT / RETIRE); (2) **Lotka–Volterra ecological dynamics** that estimate a skill interaction matrix $\beta_{ij}$ from execution traces to capture complementarity and conflict; and (3) an **anti-pattern memory** that records failure signatures and vetoes repeated mistakes. On OfficeQA, SealQA, and WildClawBench across five Qwen3.5 scales (9B–397B), SkillSmith achieves +18.3% on OfficeQA and +18.9% on SealQA at 397B, with gains that grow monotonically with model size and multi-skill co-activation, while sustaining improvement on WildClawBench for 6+ days where skill-only baselines plateau by Day 2–3.

## 核心贡献

- **提出 Skill–Tool 共同进化的统一原子化提案空间**：打破现有系统将工具层视作固定的假设，反思输出不再是单一技能的修改，而是一个"原子 bundle"，允许在同一提案里同时改写技能并对工具执行类型化的生命周期操作（WRAP / EDIT / COMPOSE / SPLIT / RETIRE），从而在保持工具库结构安全的前提下修复工具级失败。
- **生态效用模型（Lotka–Volterra 动力学）**：把技能库建模为共享生态，依据执行日志估计技能成对交互矩阵 $\beta_{ij}$（正值为互补，负值为冲突）和个体效用 $\hat{u}_i^{(t)}$，通过轻量级的 Lotka–Volterra 更新平滑得到动态效用 $u_i^{(t+1)}$，用于指导检索打分、变异优先级与退役决策。
- **反模式记忆（Anti-pattern Memory）**：将失败签名 $p$、归因 $a$ 与修复方法 $c$ 结构化记录，在反思阶段既加速诊断也作为"否决器"，阻断系统重复已经验证无效的技能–工具配置。
- **实验覆盖三大基准与五个 Qwen3.5 规模**：在 OfficeQA、SealQA、WildClawBench 上，9B–397B 五个规模均一致超越 Base Agent 与 EvoSkill 等强基线，优势随模型规模与多技能共激活率放大；WildClawBench 上能持续演化至第 6 天，而 SkillClaw 等基线在第 2–3 天即陷入平台。
- **消融与稳定性证据完备**：锁定工具层导致 WCB 下降 6.8%、无生态治理导致库膨胀至 21 技能 + 7 工具、移除反模式记忆使回归率从 2.1% 升至 7.6%；100 轮演化下面对两次环境扰动仍可恢复到 ~70%。

## Key Contributions

1. **Bundle-based Skill–Tool co-evolution.** The proposer $R$ emits an atomic bundle $L$ that jointly mutates skills and tools; tool evolution is restricted to five typed primitives — `WRAP`, `EDIT`, `COMPOSE`, `SPLIT`, `RETIRE` — preventing the joint search space from exploding. The formal argument: single-type mutation operators are not closed over the valid state set $\Omega$ satisfying skill–tool dependency constraints, so a sequence of individually valid edits can require intermediate states outside $\Omega$ that validation rejects. SkillSmith's bundle mutation $\mathcal{L}: \Sigma_t \mapsto \Sigma_{t+1}$ with $\Sigma_t, \Sigma_{t+1} \in \Omega$ expands the set of reachable improvements under validity constraints rather than merely increasing edit size.

2. **Synergy-aware ecological dynamics.** For each execution, raw score $P(x)$ is converted to a difficulty-normalized residual $z(x) = P(x) - b(x)$ (where $b(x)$ is the historical mean for the task's category) to mitigate task-difficulty confounding. Per-skill observation is the EMA-smoothed difference in mean residuals between executions with vs. without skill $s_i$ activated; per-pair synergy $\beta_{ij}^{(t)}$ is the gain of co-activation residuals over the better single-activation residual (only for pairs with co-occurrence count $\geq n_{\min}$). These feed a lightweight Lotka–Volterra update:
$$u_i^{(t+1)} = u_i^{(t)} + \epsilon \, \hat{u}_i^{(t)} u_i^{(t)} \left(1 - \frac{\sum_j w_{ij} u_j^{(t)}}{K}\right),$$
with $w_{ii}=1$, $w_{ij}=-\beta_{ij}^{(t)}$ for qualifying pairs (others 0), and clipping to $[u_{\min}, u_{\max}]$ for stability.

3. **Anti-pattern memory with veto.** Each entry $\phi = (p, a, c)$ stores a failure signature, a causal attribution, and a remedy. At reflection time, similar past failures are retrieved to inject attributions into the proposer context, and a veto check rejects proposals that resemble known-bad configurations — explicitly preventing the agent from accidentally regenerating retired skills or repeating the same broken skill–tool bundles.

4. **Retrieval scoring under interaction.** Skill selection at execution time uses $\text{score}(s_i, q, S_{\text{act}}) = \alpha \cdot \text{sim}(s_i, q) + \gamma u_i^{(t)} + \delta \sum_{s_j \in S_{\text{act}}} \beta_{ij}^{(t)} - \eta \cdot \text{cost}(s_i)$, balancing semantic relevance, dynamic utility, interaction compatibility, and execution cost.

5. **Synergy-aware merge + instance-level Pareto management.** Two non-dominated states $\Sigma_i, \Sigma_j \in G$ sharing an ancestor $\Sigma_a$ can be crossed over by taking the differing descendant version of each skill (or the higher-utility one if both diverged). New combinations whose pairwise $\hat{\beta}_{ij}^{(t)} < -\beta_{\text{thresh}}$ are flagged for extended validation. The Pareto front $G$ is maintained **at instance granularity** rather than aggregate score, and sampling probability for state $\Sigma_k$ is proportional to the number of training instances on which it is uniquely best — preserving strategic diversity.

6. **Progressive validation pipeline.** A candidate $\Sigma'$ must pass Tool unit testing → end-to-end integration testing → regression checking; merged candidates additionally condition regression checks on the task families where either parent lineage had demonstrated strength.

## Methodology

### State representation

At time $t$, the agent's external non-parametric state is a triple $\Sigma_t = (S_t, T_t, F_t)$:
- **Skill** $s = (m, w, r, u)$ — metadata $m$ (name, trigger, version), workflow body $w$, reference resources $r$ (templates, domain knowledge), scalar utility $u$.
- **Tool** $\tau = (d, f, \sigma)$ — interface description $d$, executable implementation $f$, type signature $\sigma$.
- **Anti-pattern** $\phi = (p, a, c)$ — failure signature, causal attribution, remedy.

### Per-iteration loop

Each iteration has four stages:

1. **Sample & execute.** Sample $\Sigma_k$ from the Pareto front $G$ (proportional to its unique-instance coverage), execute tasks on minibatch $M$, collect failure set $\{x \in M : P(x) < \theta\}$.
2. **Reflect & propose.** Proposer $R$ receives three auxiliary signals: matched anti-pattern records (accelerate diagnosis), ecological dynamics $\Delta u_i^{(t)}$ and $\hat{\beta}_{ij}^{(t)}$ (prioritize repair targets), and a structured feedback function $\mu_f$ (compiler errors, missing documents, constraint violations when available). $R$ analyzes failures along two axes — vertical (skill orchestration defect vs. insufficient tool capability) and horizontal (multi-skill interaction) — then emits an atomic bundle $L$.
3. **Apply Tool-Smith if needed.** When the bundle contains tool operations, the Tool-Smith $B_\tau$ is restricted to five typed lifecycle primitives (`WRAP` / `EDIT` / `COMPOSE` / `SPLIT` / `RETIRE`).
4. **Validate & update.** Progressive validation (tool unit → integration → regression), admit $\Sigma'$ to $G$, recompute $\hat{u}_i^{(t+1)}$ and $\hat{\beta}_{ij}^{(t+1)}$, run one Lotka–Volterra step, store new failure patterns in $F_{t+1}$. Skills below the retirement threshold for $T_{\text{ret}}$ consecutive rounds are removed and their info transferred to $F_t$ as "epitaphs."

### Optimization objective

$$\max_{\{L_t\}_{t=0}^{T-1}} \mathbb{E}_{x \sim D_{\text{val}}} P(x \mid \pi, \Sigma_T) \quad \text{s.t.} \quad \Sigma_{t+1} = U(\Sigma_t, L_t), \quad \sum_{t=0}^{T-1} b(L_t) \leq B$$

where $b(L_t)$ is the rollout cost consumed by proposing and validating $L_t$.

### Benchmarks

- **OfficeQA** (246 questions, Mosaic Research, Databricks 2025): cross-document table localization and multi-step numerical reasoning — highly sensitive to skill precision.
- **SealQA** (111 questions, Pham et al. 2025): factual QA when search results contain noise and conflicting information.
- **WildClawBench** (Ding et al. 2026, InternLM): 15–50 step multi-modal tasks with multiple tools and high interaction density.

### Baselines

- **Base Agent**: initial skill library $S_0$ + fixed tool set $T_0$, no evolution.
- **EvoSkill** (Alzubi et al. 2026): failure-driven skill-only evolution + Pareto-front selection.
- **SkillClaw** (Ma et al. 2026): multi-user trajectory-driven collective skill evolution.

### Backbones

Five Qwen3.5 scales: 9B, 27B, 35B, 122B, 397B.

## Results & Highlights

### RQ1 — Main results on OfficeQA and SealQA (Qwen3.5, accuracy %)

| Method | OfficeQA 9B | 27B | 35B | 122B | 397B | SealQA 9B | 35B | 122B | 397B |
|---|---|---|---|---|---|---|---|---|---|
| Base Agent | 11.8 | 22.0 | 28.9 | 43.9 | 61.8 | 4.5 | 10.8 | 17.1 | 30.6 |
| EvoSkill | 13.4 | 25.6 | 34.1 | 53.3 | 71.5 | 5.4 | 14.4 | 23.4 | 41.4 |
| **SkillSmith (Ours)** | **14.6** | **28.5** | **38.6** | **60.2** | **80.1** | **6.3** | **17.1** | **27.9** | **49.5** |
| Δ vs Base | +2.8 | +6.5 | +9.7 | +16.3 | **+18.3** | +1.8 | +6.3 | +10.8 | **+18.9** |

The scaling trend is striking: SkillSmith's gain over Base Agent grows monotonically from +2.8% to +18.3% on OfficeQA and from +1.8% to +18.9% on SealQA as model size increases from 9B → 397B, suggesting larger models exploit the joint skill–tool evolution more effectively.

### WildClawBench — long-horizon multi-modal agent

- SkillSmith sustains improvement to **Day 6** on WildClawBench where SkillClaw plateaus by Day 2–4 due to tool-layer bottlenecks.
- Largest advantages on **tool-intensive categories**.

### RQ2 — Ablation (Qwen3.5-122B)

| Variant | SealQA | WCB Avg. | Regression Rate ↓ | Final Size | Tool Error ↓ |
|---|---|---|---|---|---|
| SkillSmith (Full) | 27.9 | 41.4 | **2.1%** | 14+6 | **3.2%** |
| Skill-only (tool layer locked) | 24.3 | 34.6 | 3.4% | 16+9 | — |
| FreeTool (unconstrained edits) | 26.1 | 37.8 | 8.9% | 14+9 | 14.7% (4.6× spike) |
| −Eco (no ecological governance) | 26.6 | 39.1 | 2.8% | **21+7** (bloated) | 4.1% |
| −Anti (no anti-pattern memory) | 25.2 | 38.2 | **7.6%** | 15+6 | 3.5% |

- **Locking the tool layer** is the single largest drop (−6.8% on WCB); the system overcompensates by generating bloated skills (16 skills) with critical breakdowns.
- **FreeTool** patches some tools but causes the tool error rate to spike 4.6× and regression to 8.9%, validating the typed lifecycle operations as a safety mechanism.
- **Without ecological governance**, the library bloats to 21 skills + 7 tools, slightly hurting performance and degrading retrieval accuracy.
- **Without anti-pattern memory**, regression jumps from 2.1% → 7.6% — the system repeatedly proposes previously failed configurations.

### RQ3 — Scaling effects

- Tasks with more steps exhibit higher skill co-activation rates; SkillSmith's gains grow from ~5–10% on simpler tasks to **>20% on highly complex tasks**.
- Factorized gains correlate positively with Tool Complexity, Interaction Density, Context Size, and Skill Synergy.

### RQ4 — Evolutionary stability (100 rounds, two perturbations)

- Perturbations at iteration 30 (increased task complexity) and iteration 60 (damaged underlying components) caused all methods to drop below 25%.
- **EvoSkill** (fixed tools) cannot repair damaged interfaces after iteration 60 and plateaus around 30%.
- **SkillSmith** reconstructs damaged tools and updates dependent skills in lockstep, recovering and stabilizing near **70% by iteration 100**.
- **−Eco** variant grows beyond 80 components → retrieval noise and context overflow drop accuracy from 68% → <35%.
- **−Anti** variant oscillates between 40%–60% (repeatedly explores ineffective paths).
- **Full SkillSmith** keeps the library under 28 components throughout.

### Case study — OfficeQA tool bug cascade (Figure 7)

Task: compute the change in federal debt held by the public between Q1 and Q3 of FY2019.

- **Before co-evolution (wrong):** The `table_extractor` v1 used integer indexing `(row=5, col=3)`. A merged header caused an off-by-one error returning "16,228" from the wrong cell for Q1; "16,174" for Q3; output = **−54M** (ground truth: **+187M**).
- **After co-evolution (Iter. 4, correct):** Atomic bundle $L_4$ does `TOOL — WRAP` with a header-resolution layer + `EDIT` signature to `(row_header: str, col_header: str)`; `SKILL — replace integer refs with header names; add unit verification`; `CASCADE — two other skills sharing this tool auto-migrate, fixing a class of errors across 3 skills at once`.
- Output: 16415 − 16228 = **+187M** ✓.

The case study shows the **cascade effect**: a single tool-level fix automatically propagates to all dependent skills via the bundle mechanism — a class of errors resolved in one iteration.

## Critical Assessment

- **Novelty (medium-high):** The **bundle-based atomic co-evolution** of skills and tools, plus the **Lotka–Volterra ecological dynamics for skill utility**, are genuinely new framings. Existing works (EvoSkill, SkillClaw, D2Skill, GEPA, EvoSkills) treat the tool layer as fixed; SkillSmith is the first to formalize skill–tool co-evolution as a single search problem with typed lifecycle operations. The ecological / interaction-matrix idea draws inspiration from population dynamics and is non-trivial to apply to discrete non-parametric assets.
- **Rigor (high):** The paper formalizes the problem as an optimization over a discrete structured state space $\Omega$, gives explicit update equations, ablations, instance-level Pareto management, and 100-round stability tests with two controlled perturbations. Validation is progressive (unit → integration → regression).
- **Reproducibility concerns (medium):** The implementation detail of the proposer $R$, the typed tool lifecycle primitives (`WRAP`/`EDIT`/`COMPOSE`/`SPLIT`/`RETIRE`), and the threshold values ($\theta$, $n_{\min}$, $\beta_{\text{thresh}}$, $T_{\text{ret}}$, $\epsilon$, $K$, $u_{\min}$, $u_{\max}$, retrieval weights $\alpha,\gamma,\delta,\eta$) are deferred to the appendix (Appendix B and D referenced but not in the excerpted pages). Without code release confirmation, exact reproduction will require careful re-implementation.
- **Threats to validity:** (1) The 5-primitive restriction may under-fit domains where richer tool operations are needed. (2) The ecological model assumes a global carrying capacity $K$ and quadratic interaction — oversimplified for highly nonlinear interactions. (3) The anti-pattern veto depends on signature-similarity matching quality; near-duplicate but differently-worded failures may slip through. (4) Cold-start noise when co-occurrence data is sparse is acknowledged but not fully quantified.
- **Significance:** Strong candidate for follow-up work in self-evolving agents. The "atomic bundle + ecological utility + failure memory" template is broadly transferable to other external-policy assets (memory stores, persona files, prompt caches). The 397B-scale results + 100-round stability demonstration are unusually thorough for a 2026-vintage paper.
- **Red flags:** None critical. The "co-evolution" framing risks over-claiming — at the smallest scale (9B) the gain is modest (+2.8% OfficeQA, +1.8% SealQA), so the framework clearly needs strong backbone models to shine. The case study is somewhat cherry-picked (a merged-header bug is a perfect demo for tool-level co-evolution), though the ablation evidence is consistent.

## Strengths

1. **Unified treatment of skills, tools, and failure memory** as a single evolutionary state — addresses the three previously independent problems (skill mutation, tool extension, failure retention) in one framework.
2. **Formal validity argument**: atomic bundles guarantee both $\Sigma_t, \Sigma_{t+1} \in \Omega$, expanding the reachable valid-state space rather than just enlarging edits.
3. **Typed tool lifecycle primitives** prevent uncontrolled tool explosion while still enabling real tool-level repair — a clean safety/evolvability balance.
4. **Lotka–Volterra dynamics** provide a theoretically motivated, lightweight mechanism for handling skill interactions that is cheap to compute from existing execution logs (no extra ablation cost).
5. **Anti-pattern veto** is a simple but high-leverage addition that triples regression prevention (2.1% → 7.6% when removed).
6. **Comprehensive evaluation**: 3 benchmarks × 5 model scales × 3 baselines + ablation + 100-round stability + case study — above-average for a paper at this venue/era.
7. **Scaling behavior is the headline**: the gain-vs-baseline curves grow monotonically with both model size and task complexity, which is what a self-evolving-agent framework should do.
8. **Instance-level Pareto management** preserves niche strengths rather than collapsing to one globally-best state — important for diverse task distributions.

## Limitations

1. **Backbone dependence on reflection quality**: SkillSmith relies on the LLM's ability to propose useful bundles; if the proposer $R$ is weak (small or poorly instruction-tuned models), co-evolution can stall. The framework inherits all weaknesses of reflective prompting (cf. GEPA).
2. **Cold-start noise in the ecological model**: when co-occurrence data is sparse, $\beta_{ij}^{(t)}$ is unreliable; the paper acknowledges this but does not present a quantified mitigation.
3. **Scope limited to three benchmarks** — OfficeQA (structured docs), SealQA (noisy search QA), WildClawBench (multi-modal agent). **No evaluation** on embodied agents, multi-user / multi-agent settings, code-generation benchmarks (SWE-Bench, HumanEval), or tool-use arenas like τ-bench or BFCL — leaving the generalizability of the framework partly untested.
4. **Global carrying capacity $K$** is a coarse abstraction; nonlinear skill interactions may not be captured by a single scalar.
5. **Threshold sensitivity** — many hyperparameters ($\theta$, $n_{\min}$, $\beta_{\text{thresh}}$, $T_{\text{ret}}$, $\epsilon$, $K$, retrieval weights) are not explored in the excerpt; risk of brittle behavior on out-of-distribution tasks.
6. **No reported cost / efficiency analysis** — how many rollouts does the framework consume relative to skill-only baselines? If co-evolution is 10× more expensive, the headline gains need to be normalized.
7. **No discussion of safety / governance risks** when the agent can WRAP and EDIT its own tools (e.g., removing safety checks); presumably addressed in a longer appendix, but not surfaced in the main paper.
8. **Lotka–Volterra analogy is metaphorical**: skills do not literally reproduce or compete for resources; the mapping requires care and may not generalize beyond retrieval-budget-bounded settings.

## Relevance

直接命中"自我进化智能体"和"工具增强 LLM 智能体"两条研究主线，并把长期被忽视的**工具层可进化性**推到了与技能进化同等重要的位置，对研究 LLM 智能体外部可治理的非参数资产（技能 / 工具 / 失败记忆）的统一进化范式具有重要参考价值。对于构建长期自改进、可治理、可审计的智能体系统而言，这篇工作提出的"原子 bundle + 生态治理 + 反模式否决"三件套是一个非常实用的设计模板。

## Questions for Further Exploration

1. **How does SkillSmith behave on coding benchmarks** such as SWE-Bench Verified, HumanEval+, or τ-bench, where tool APIs are external and immutable — can the framework gracefully degrade when only skills (not tools) can be edited?
2. **What is the compute cost per iteration?** Co-evolution requires Tool unit testing, integration testing, and regression checks; a rollout-budget-normalized comparison vs. EvoSkill / GEPA is needed to know whether +18.9% at 397B justifies the extra rollouts.
3. **Can the typed tool lifecycle primitives be expanded** (e.g., `FORGET`, `VERSION`, `WRAP-INTO-MACRO`) without exploding the joint search space? Is there a principled way to learn the primitive set per domain?
4. **How does the framework handle adversarial / noisy tool environments** where $\mu_f$ (structured feedback) is unavailable or unreliable — purely trajectory-based reflection?
5. **Can the ecological dynamics be extended** beyond pairwise $\beta_{ij}$ to higher-order skill interactions (3-skill synergies, role-based complements), perhaps via a learned hypergraph?
6. **What happens when multiple agents share and co-evolve tools** simultaneously — does the framework support multi-user SkillClaw-style settings with safety guarantees?
7. **Safety audit of self-EDITed tools**: how to prevent an agent from removing its own safety checks or introducing insecure behaviors through the WRAP / EDIT operations?
8. **Cold-start regime**: what is the minimal seed data and number of iterations needed to escape the noisy ecological regime and converge to a stable $\beta_{ij}$ matrix?
9. **Interaction with other external assets** such as memory stores, persona files, or RAG indices — can the unified $\Sigma_t = (S_t, T_t, F_t)$ be generalized to $\Sigma_t = (S_t, T_t, F_t, M_t, \ldots)$?
10. **Theoretical analysis**: convergence guarantees, regret bounds, or PAC-style sample complexity for the Lotka–Volterra update under the constraint $\Sigma_t \in \Omega$ — currently only empirically demonstrated.
