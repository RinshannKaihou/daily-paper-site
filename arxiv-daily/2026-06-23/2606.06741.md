> **一句话总结：** OpenSkill 提出一种"开放世界自进化"框架，仅凭任务指令和公网资源，让 LLM 智能体自行从文档/仓库/网页抓取知识合成可迁移技能，再用自建的虚拟测试任务做无监督打磨，最终在 SkillsBench 等基准上以零目标监督取得最优自动通过率（+8.9/+8.8 pp），且技能可跨模型零适配迁移。

> **TL;DR:** OpenSkill defines a new "open-world self-evolution" setting where an LLM agent must bootstrap both its skill library and its own verifier from scratch using only a task prompt and public web/repository knowledge — no target-task supervision allowed. The pipeline has three stages: (1) Gemini Deep Research retrieves grounded domain knowledge + independent verification anchors (e.g., dataset statistics, reference values); (2) an in-container Opus 4.6 agent synthesizes 1–4 skills and iteratively refines them against self-built pytest assertions derived from those anchors; (3) the final skill is deployed zero-shot to a target agent (Opus 4.6 or GPT 5.2) and scored on the hidden ground-truth suite. On SkillsBench (11 domains), OpenSkill hits 43.6% (Opus) / 42.1% (GPT) — best automated, within 1–3 points of human upper bounds, beating the strongest closed-world baseline by +8.9/+8.8 pp. The same Opus-built skills transfer as-is to Haiku 4.5, Qwen 3 Coder, DeepSeek V3, and Mistral Large 3 (5.5–14.8 pp gains), and the self-built verifier covers 88.9% of human-authored test intents with 60.7% agreement on outcomes, despite never seeing ground-truth tests.

## 核心贡献

- **新问题定义：开放世界自进化（Open-World Self-Evolution）。** 智能体只拿到任务指令 Ii 和环境 Ei，必须从公网资源 K 中自行构建技能集和验证信号；目标任务的监督（参考解、hidden test、奖励信号）在学习阶段完全不可见，仅在最终评测时使用。
- **OpenSkill 三阶段流水线。** Stage 1 用 Gemini Deep Research 抓取背景知识 ki（API、约定、最佳实践）和独立可验证锚点 kiv（参考值、数据集不变量、跨验证方法）；Stage 2 由基于 Opus 4.6 的 base agent 在 Docker 沙箱内合成 1–4 个技能，并用独立的 Verifier LLM 会话基于 kiv 生成 pytest 断言进行无监督打磨；Stage 3 把冻结的技能文件零样本部署到目标 agent。
- **泄漏屏障（Leakage Barrier）。** 对所有检索查询做 sanitize_query，过滤基准名称及变体；GT 测试目录在容器外从未挂载；agent 函数签名排除 solution 和 test 路径；log-level 审计确认零 GT 引用。
- **经验性结论：** 在 SkillsBench（11 个领域）、SocialMaze、ScienceWorld 三个基准和两个目标模型上全面拿下"最佳自动方法"，且同一组 Opus 4.6 生成的技能无需适配即可迁移到四个更弱的模型。

## Key Contributions

- **Formalizes "open-world self-evolution"** as a setting where the agent observes only the task instruction Ii and environment Ei and must construct skill set Ŝi = f(Ii, Ei, K) without any dependence on the hidden ground-truth test T_iGT.
- **Three-stage OpenSkill pipeline** that bootstraps both skills and a supervision-free practice environment from open-world resources, with a "leakage barrier" that excludes benchmark names from all retrieval queries and isolates GT files at the container/prompt/log levels.
- **Virtual-Task Verifier** that produces deterministic pytest assertions anchored to independently verifiable facts (e.g., dataset row counts, library output formats, known reference values) rather than target-task answers, yielding a median of 3.4× more tests per task and 88.9% coverage of human-authored test intents.
- **Diagnostic-driven refinement loop** (J ≤ 3 iterations) that classifies failures as SELF-FIXABLE bugs vs. NEEDS-DR knowledge gaps and triggers targeted Gemini retrieval only when needed.
- **Empirical evidence** of best automated pass rate on 3 benchmarks × 2 target agents, with skills that transfer across 4 different model families (Haiku 4.5, Qwen 3 Coder, DeepSeek V3, Mistral Large 3) without retraining.

## Methodology

**Pipeline architecture.** OpenSkill instantiates Eq. 2 as a three-stage orchestration on a host, with skill creation and verification running inside a per-task Docker container. Stage 1 (Open-World Knowledge Acquisition) uses a Skill Planner LLM to synthesize a single sanitized research query from Ii, Ei, and an environment preview (text files truncated to 2K chars, PDFs to 5K chars; skills/ and doc/ directories excluded). The query is submitted to Gemini Deep Research (deep-research-pro-preview-12-2025, async polling every 10s up to 3600s timeout), producing background document ki. A second Skill Planner slices ki into 1–4 per-skill reference chunks to form plan pi. Concurrently, a separate Verification-Knowledge Retriever Dv (gemini-3.1-flash-lite, temperature 0.3, Google-Search grounded) fetches kiv with four anchor classes: hand-computable reference values, dataset-level statistical invariants (row counts, monotonicity), cross-validation procedures via alternative tools, and published reference I/O pairs.

**Stage 2 (Leakage-Free Skill Evolution).** A base agent π_θ (Opus 4.6) generates an initial skill set of size m ∈ [1, 4] from (pi, ki) in a long-lived session, then refines it across up to J = 3 rounds. Per round, an independent Verifier LLM session (no shared context with the creator) generates a deterministic pytest suite T̃i = g(Ii, Ei, kiv). The virtual pass rate r̃(j) = |T̃i|⁻¹ Σ_k t̃i,k(π_θ(Ii, Ŝi, Ei)) drives refinement via structured failure diagnostics F(j) (per-assertion results + root-cause + revision suggestions). A gemini-3.1-flash-lite classifier (temp 0.1, 200 max tokens) labels each failure as SELF-FIXABLE or NEEDS-DR; the latter triggers targeted retrieval (≤ 3 searches/task) injecting additional knowledge into the refinement context. Loop exits when r̃ = 1 or J is exhausted. Auxiliary guards: idle/stale episode limits (3/10), token budget (warn 70% / stop 90%), verifier/creator episode caps (20/60).

**Stage 3 (Zero-Shot Target Evaluation).** The last-refined skill set Ŝi∗ (edited in place, not best-of-N) is copied to the target workspace and evaluated under neval = 5 independent runs of the target agent π_θ′ (Claude Code for Opus 4.6, Codex for GPT 5.2, terminus-2 for other LLMs). PassRate = (1/n) Σ_i T_iGT(π_θ′(Ii, Ŝi∗, Ei)). Because the skill is an explicit artifact rather than weight deltas, the same Ŝi∗ transfers to π_θ′ ≠ π_θ.

**Information isolation audit (4 layers).** (1) Code-level: function signature excludes solution/test paths; verifier is a separate LLM session. (2) Container-level: GT directories live on host as siblings to environment/, never mounted into the container. (3) Oracle bypass: evolution loop overrides the parent class's GT oracle; loop exits on surrogate pass without invoking GT eval. (4) Log-level: zero GT file references in evolution_run_log.json agent trajectories. (5) Prompt-level: explicit "no access to solution or ground-truth tests" instructions.

## Results & Highlights

**SkillsBench (Table 1, 11 domains).** OpenSkill reaches 43.6% overall on Opus 4.6 and 42.1% on GPT 5.2, both within 1–3 points of the human upper bound (44.5% / 44.8%) and beating the strongest closed-world baseline by +8.9 / +8.8 pp (Skill-Creator / CoT). It is the only method strong on both agents: iterative methods collapse on GPT 5.2 (AutoSkill 24.7→11.2, Memento 30.1→15.6, both below the 25.0% no-skill floor), while single-pass variants (Self-Gen, CoT) hurt Opus 4.6. Best-or-tied-best in 8/11 (Opus) and 7/11 (GPT) domains. Largest gains in knowledge-intensive areas: Health 69.6% (Opus), Software 59.9% (Opus), Energy 80.0% (GPT), Cybersecurity 52.5% (GPT, above Human). Manufacturing collapses to 0.0% across all automated methods.

**Other benchmarks (Table 2).** SocialMaze: 82.7% (Opus) / 70.7% (GPT) — best automated, +0.9 to +2.2 pp over best baseline. ScienceWorld: 90.0% / 85.3% — best automated, larger gains on GPT 5.2.

**Cross-model transfer (Figure 3).** Opus-built skills transferred as-is to Haiku 4.5, Qwen 3 Coder, DeepSeek V3, Mistral Large 3 yield +5.5–14.8 pp over No-Skill floor. OpenSkill beats AutoSkill and Memento on every target; AutoSkill underperforms No-Skill on three of four (Haiku, DeepSeek, Mistral), suggesting its skills are model-coupled.

**Verifier quality (Section 4.2).** 60.7% overall agreement with GT outcomes (precision 56.9%, recall 80.5%, Fisher's OR = 2.97, p = 0.035, point-biserial r = 0.242, p = 0.027). On a 15-task sample, the verifier covers 88.9% of GT test intents (120/135) and emits a median 3.4× more tests per task (15.3 extra assertions/task). Uncovered 11.1%: anti-cheat meta-validation and deep semantic checks (taxonomy coherence, lemmatization).

**Ablations (Figure 4, SocialMaze/Opus).** (a) Iteration count: peaks at J=3 (82.7%), degrades at J=5 (79.9%) and J=10 (78.0%) — evidence of overfitting to virtual feedback. (b) Component contribution: parametric-only baseline 74.5%; +Virtual-Verifier alone → 80.8% (+6.3); +Open-World-Query alone → 80.6% (+6.1); both → 82.7% (+8.2 over baseline). Components are largely complementary with partial overlap.

**Cost (Table 7).** Per-task on SkillsBench/Opus 4.6: skill creation 749K tokens / 39 min, GT eval 400K tokens / 91.5 min; end-to-end ~$1,800 / 84 tasks at Opus 4.6 list pricing. Eval-only median per run: 368s (OpenSkill) vs 347s (No-Skill) — comparable.

## Critical Assessment

**Strengths of the work.**
- The problem formulation is precise and timely: the "leakage barrier" definition makes the constraint auditable rather than rhetorical.
- Strong empirical setup: 3 benchmarks × 2 target agents × 7 closed-world baselines, with a clear "best automated" framing that excludes human skills from comparison.
- Genuine cross-model transfer demonstration across 4 weaker models using identical skill files — this is the most compelling evidence that skills are first-class artifacts, not model-specific weights.
- Honest reporting: virtual-verifier agreement is 60.7% (not 95%), they document failure modes (false positives on near-miss tasks, verifier infrastructure failures on non-standard domains), and they note the "no human upper bound" comparison policy.
- Open-source code + website + public benchmarks enable replication.

**Rigor concerns.**
- The "no target-task supervision" claim rests on a name-based query filter. Open-world retrieval could still surface task-relevant snippets (e.g., StackOverflow threads about SkillsBench-style tasks) that indirectly reveal evaluation criteria. The 4-layer audit checks process, not retrieval content.
- Verifier coverage (88.9%) is measured by Opus 4.6 as LLM-judge against its own semantically matched taxonomy — this introduces circularity. A blind human audit would strengthen the claim.
- The Manufacturing domain collapse (0% across all automated methods) is reported but not investigated. Manufacturing seems to be a clear failure mode where open-world acquisition alone is insufficient.
- Self-Gen and CoT only marginally help (or hurt) on Opus 4.6, but the paper attributes this to "parametric skills being insufficient" without isolating whether the prompt format itself is suboptimal.
- Some baselines (AutoSkill, Memento) collapse on GPT 5.2, which the authors treat as evidence for OpenSkill's robustness but could indicate baseline mis-configuration or model-specific tuning needed.

**Reproducibility.**
- Code released; hyperparameters (Table 5) and four-layer audit (Appendix F) documented. Multi-model evaluation harness may be costly to replicate (~$1,800 for one SkillsBench run with Opus).
- Some model versions are short-name abbreviations; the appendix provides identifiers but version drift could affect reproduction.

**Significance.**
- This is a meaningful step toward practical "agent self-evolution after deployment" where curated skills and reliable feedback are unavailable. The transferable-skill artifact framing is a clean abstraction that separates knowledge acquisition, verification, and deployment.
- The 88.9% verifier coverage and 60.7% agreement are modest but honest; for self-built verifiers without GT access, this is a reasonable signal.

**Red flags.**
- No real ablation on how robust the leakage filter is against clever queries. If a benchmark has a distinctive task signature (e.g., unusual output format), retrieval could leak via content even after name stripping.
- "Best automated" framing conveniently sidesteps comparison with closed-weight frontier systems (e.g., Opus 4.6 with a longer reasoning budget, or human-in-the-loop), but the headline "+8.9 pp over strongest closed-world baseline" is robust on its own terms.
- Some component contribution ablations (Figure 4b) overlap partially — the +2.1 pp marginal gain from combining VV+OW over the better individual component is modest.

## Strengths

- Clean problem formulation with an auditable leakage barrier (not just rhetorical "no leakage").
- First demonstration of cross-model skill transfer without model-specific adaptation across 4 weaker LLMs.
- The virtual verifier concept is novel: anchoring pytest assertions to independently verifiable facts (dataset statistics, documented APIs) rather than target answers is a practical recipe.
- Honest reporting of verifier agreement (60.7%), iteration overfitting, and per-domain failure modes (Manufacturing collapse).
- Open-source pipeline + website + multi-model evaluation harness enable replication.
- Authors publish both the framework and several baselines (Memento, AutoSkill), indicating intimate familiarity with the comparison landscape.

## Limitations

- **Knowledge-source noise:** web and repository sources may be outdated, contradictory, or contain solutions to benchmark tasks; provenance tracking and source validation are not deeply integrated.
- **Verifier-task mismatch:** virtual tasks may overestimate skill quality (format-correct but semantically wrong); 12 of 25 false positives were "high-accuracy near-misses" where the verifier approved 0.81 mean accuracy solutions that GT scored as 0.
- **Pytest-centric verification:** the verifier fails entirely on 5 non-standard tasks (CVE patches, build systems, proof assistants) because pytest is unsuitable — limiting applicability beyond code-generation domains.
- **Manufacturing collapse:** 0% across all automated methods suggests a class of tasks where open-world acquisition alone is insufficient (likely requiring embodied/control knowledge).
- **Cost:** ~$1,800 per SkillsBench run with Opus 4.6; skill creation is one-time but the budget for routine use may still be prohibitive.
- **Hyperparameter sensitivity:** J = 3 was best among {1, 3, 5, 10}; the paper does not sweep other knobs (retrieval depth, skill count, verifier episode caps).
- **GPT 5.2 collapse of iterative baselines** is reported as evidence for OpenSkill's robustness, but could also indicate baseline tuning issues not fully explored.

## Relevance

与"用户关注 LLM 智能体自进化、可迁移技能库、开放世界学习循环"的研究兴趣高度契合——OpenSkill 直接给出了"零目标监督下从公网资源自举学习回路"的可复现范式，并把技能做成了可跨模型零适配迁移的显式 artifact，与 SkillsBench 的"skill 质量决定上限"观察一致。对于关注 agent self-improvement、agent skill curation、open-world retrieval 的研究方向具有直接借鉴价值。

## Questions for Further Exploration

- Does OpenSkill's virtual verifier generalize to non-code domains (web navigation, embodied control, dialogue) where pytest-based assertions are inapplicable? What alternative anchor types (visual invariants, API contract checks, reward-model probes) could substitute?
- The leakage barrier relies on query-string filtering; how robust is it against indirect leakage (e.g., retrieval of task-specific datasets, repositories that contain the benchmark's own test cases)? A formal adversarial audit would strengthen the claim.
- Skills transfer from Opus 4.6 to Haiku 4.5 / DeepSeek V3 / Mistral / Qwen — but what about transfer in the opposite direction (skills built on weaker models then deployed on stronger)? Is the gain bounded by the construction model?
- The iteration count ablation peaks at J = 3 and degrades with more. Is this overfitting to verifier artifacts, or genuine diminishing returns? An ablation with verifier noise injection could disentangle these.
- Cross-benchmark skill transfer (skills built for SkillsBench used on SocialMaze or vice versa) is not reported. How domain-specific are the synthesized skills?
- Manufacturing domain fails across all methods — what is qualitatively missing? Open-world acquisition, verifier anchoring, or something deeper (e.g., need for simulation environments)?
- The +2.1 pp marginal gain of combining open-world query and verifier (over either alone) is modest. Are there failure modes where one component systematically fails and the other compensates?
