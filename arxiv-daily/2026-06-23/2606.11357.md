> **一句话总结：** TileFuse 是面向 AMD XDNA2 NPU 的近金属 (close-to-metal) 混合精度内核库,通过融合解压 + 反量化 + GEMM/GEMV、内 interleaved pre-tiling 布局,以及针对 batch-1 生成阶段的 4×8 阵列级数据流,使 AWQ 风格的 W4A16/W8A16 量化 LLM 能在 Ryzen AI 笔记本上原生运行,端到端 prefilling 延迟最高降低 2.0×、能耗下降 64.6%。

> **TL;DR:** TileFuse is a close-to-metal mixed-precision kernel library for AMD XDNA2 NPUs that targets transformer linear layers in quantized LLM inference. It co-designs weight layout, metadata packing, fused GEMM/GEMV microkernels, and array-level dataflow to bring off-the-shelf AWQ-style W4A16 and W8A16 formats directly onto XDNA2 — without forcing users to reshape the model around an NPU-specific quantization scheme. Kernel-level gains reach 121.6% (GEMM) and 281% (GEMV) over BF16 baselines; end-to-end prefilling on Ryzen AI laptops improves by up to 2.0× with >64.6% lower energy versus a tuned llama.cpp iGPU baseline. The work also exposes a fundamental tension in NPU reconfigurability: the same flexibility that enables custom microkernels and full-array GEMV also incurs per-op dispatch overhead, which hurts latency-bound batch-1 token generation.

## 核心贡献

- **面向 AWQ 量化格式的可复用 NPU 内核库:** 首次在 AMD XDNA2 上原生支持 AWQ 风格 W4A16 (group-128 非对称) 与 W8A16 (per-channel 对称) 权重量化,而非要求用户重新量化模型以适配 NPU。
- **融合内核 (Fused Kernels):** 将 INT4/INT8 解包、反量化和 GEMM/GEMV 计算融合到单一内核流,权重在整个生命周期内保持压缩格式;通过本地内存缓存去量化结果,避免每次复用都重新反量化。
- **Interleaved Pre-Tiling 布局:** 离线将权重重排为 interleaved column-major 布局,使分配到同一 AIE 列的瓦片在内存中连续,从而绕过 DMA buffer descriptor 的 stride 限制,将支持的 GEMM 维度从 8K 提升到 32K。
- **GEMV 全阵列数据流重构:** 利用 memory core 作为中间分配层,让 1×8 广播激活向量与 4 个不同的 weight tile 在四行 compute core 上并行计算,使 batch-1 token-generation 阶段从仅用一行 compute core 提升到利用整个 4×8 阵列。
- **端到端实测与混合部署洞察:** 在 Ryzen AI 7 350 / HX 370 上跑通 Llama3-8B、Gemma-2B、Qwen2.5-3B 的完整推理,提出 "NPU 跑 prefilling + iGPU 跑 token generation" 的混合策略,利用统一内存避免一致性开销。

## Key Contributions

- **AWQ-friendly NPU kernel library:** First open-source fused mixed-precision GEMM/GEMV library for AMD XDNA2 that natively supports off-the-shelf AWQ-style W4A16 and W8A16, rather than NPU-proprietary quantization.
- **Single-kernel fusion of unpack + dequant + matmul:** On-the-fly dequantization in compute cores using local-memory caching of BF16 weights, avoiding redundant work and global-memory round trips.
- **Interleaved pre-tiling for large GEMMs:** Column-major tile reorganization that keeps same-AIE-column tiles contiguous in memory, lifting the supported GEMM dim from 8K to 32K.
- **4×8-array GEMV dataflow:** Memory cores used as an intermediate distribution stage so all four compute rows receive distinct weight tiles while sharing the broadcast activation, scaling GEMV from 1 to 32 output-tile groups.
- **Reconfigurability characterization:** Explicit analysis showing where XDNA2 reconfigurability helps (large GEMM, GEMV) vs. hurts (latency-bound short ops due to dispatch + fabric reconfiguration overhead).
- **End-to-end Ryzen AI validation:** Up to 2.0× prefill latency reduction and 64.6% energy reduction vs. llama.cpp iGPU baseline on real Llama3-8B, Gemma-2B, Qwen2.5-3B.

## Methodology

**Target hardware & stack.** XDNA2-class AMD NPUs in Ryzen AI SoCs (Strix Point HX 370, Krackan Point 7 350): a 4×8 array of 32 AIE compute cores, 8 memory cores, 8 shim cores with BD-based DMA. Programming via the open MLIR-AIE (IRON) framework, which exposes both microkernel design inside each compute tile and explicit data movement across the array.

**Offline weight pre-tiling.** At deployment time, quantized weights are reorganized:
- *Metadata packing:* For W4A16, each `k × n` tile stores packed INT4 weights contiguously, followed by BF16 scales and (duplicated) INT8 zero-points for each quantization group; W8A16 follows the same pattern with per-channel scales only.
- *Interleaved column-major layout:* Tiles assigned to the same AIE column are placed consecutively in memory. This eliminates the large stride across AIE columns that would otherwise exceed DMA BD stride limits and force kernel fragmentation for large FFN shapes.
- Overhead is negligible (~1.5% extra memory for zero-point duplication to satisfy DMA alignment).

**Mixed-precision GEMM microkernel.** Each compute tile decomposes the `m × k` activation tile and `k × n` weight tile into 8×8 sub-tiles. Per sub-tile:
1. Unpack: extract low/high 4-bit fields, widen to INT8.
2. Dequantize: convert to BF16, apply `w = s · (q − z)` (asymmetric) or `w = s · q` (symmetric), using the pre-packed scale/zero-point.
3. Cache the BF16 weight in local memory and reuse across subsequent activation rows (avoids redundant dequantization per inner-product iteration).
4. Issue AIE matrix-multiply primitives, accumulating in FP32.

The local-buffer traffic is hidden by VLIW overlap between memory ops and compute.

**Mixed-precision GEMV microkernel + dataflow.** The baseline GEMV broadcasts weights to a single compute row, leaving 3/4 of the array idle. TileFuse widens the inner microkernel to load a 64×8 weight block per iteration (vs. 16×8 baseline), reusing the 1×8 activation across 64 output rows to better fill vector registers. The optimized dataflow uses memory cores as an intermediate stage: each shim streams four tiles into its column's memory core, which then distributes them to the four compute rows. This exposes 32 output-tile groups at a time instead of 8.

**End-to-end runtime.** A custom C++/Python framework: Python frontend loads Hugging Face safetensors, runs offline pre-tiling/tokenization; a libtorch C++ backend dispatches transformer operators. Linear layers (attention projections, MLP projections) → NPU via precompiled TileFuse kernels; FlashAttention-2, softmax, residual add, normalization → iGPU via HIP/libtorch. NPU and iGPU execute sequentially (operator-level device selection, not concurrent) to isolate TileFuse's contribution to linear layers. Unified memory on Ryzen SoCs avoids coherence overhead.

**Baselines.** llama.cpp's fused quantized HIP kernels on the iGPU (Q4_K and Q8_0 formats) — chosen as the strongest publicly available quantized GEMM for RDNA iGPUs as of Jan 2026 after a recent 1.5× update. CPU Q4_K/Q8_0 also reported.

## Results & Highlights

**Kernel-level GEMM throughput** (Fig. 5):
- TileFuse W4A16 on Ryzen AI 7 350: peak **~12 TOPS**; W8A16: ~9 TOPS.
- Ryzen AI 9 HX 370: W4A16 ~9 TOPS, W8A16 lower.
- iGPU baseline (RDNA): 8-CU Krackan Point ~5 TOPS, 16-CU Strix Point ~7 TOPS.
- CPU: marginal ~200 GOPS.
- Smooth scaling across matrix dimensions on XDNA2 vs. visible zig-zag shape sensitivity on untuned iGPU GEMM.

**GEMV (1 × 4096 × 14336):**
- Krackan Point NPU: 206.21 GOPs (4-bit), 104.16 (8-bit).
- HX 370 NPU: 181.22 (4-bit), 96.95 (8-bit).
- CPU/GEMV often more competitive because memory-bound — NPU is compute-optimized and lower on pure GEMV.

**End-to-end prefill speedups (W4A16) on Ryzen AI 7 350 (the smaller iGPU makes NPU gains more visible):**
- Llama3-8B: 1.34× @ 256 tokens → **2.00× @ 2048** → 1.97× @ 4096 → 1.90× @ 8192.
- Gemma-2B: 1.70× @ 2048, 1.73× @ 4096, 1.56× @ 8192.
- Qwen2.5-3B: 1.57× @ 2048, 1.70× @ 4096, 1.64× @ 8192.
- HX 370: smaller speedups because its stronger 16-CU iGPU partially masks NPU gains; W8A16 on HX 370 is *slower* than iGPU across nearly all shapes.

**Token generation (W4A16, tok/s):** NPU underperforms iGPU across the board (e.g., Llama3-8B: NPU 7.42 / iGPU 15.05 on AI 7 350). GEMV executes in hundreds of µs but NPU dispatch + reconfiguration overhead is in the millisecond range — a fundamental mismatch.

**Energy (Llama3-8B prefill, 4096 tokens):**
- AI 7 350 NPU: 385.98 J vs. iGPU 1091.81 J → **64.6% energy reduction**.
- HX 370 NPU: 322.89 J vs. iGPU 684.59 J → 52.8% reduction.
- NPU path also lowers instantaneous current draw throughout execution.

**Ablations (Fig. 10/11):**
- Interleaved pre-tiling alone: up to 50% gain over BF16 baseline; required to scale beyond 8K GEMM dimensions.
- Mixed-precision + pre-tiling: up to 121.6% (GEMM) over BF16 baseline.
- GEMV: pre-tiling +20%; full microkernel + dataflow optimization yields up to 281% over BF16 baseline.
- Fixed-kernel baseline cannot run 8K×8K×16K or 32K shapes without splitting kernels — pre-tiling fixes this.

**Comparison with prior NPU works (Table 3, 1024-token prefill):**
- vs. FastFlowLM (Llama-3-8B): 501.9 vs. 345.9 tok/s → **1.45×**.
- vs. Scaling NPU (Qwen2.5-3B): 890.4 vs. 550.0 tok/s → **1.61×**.

## Critical Assessment

TileFuse is a strong, well-engineered systems paper that delivers a genuinely useful contribution: bringing AWQ — the de facto standard for edge LLM quantization — to AMD XDNA2 in a way that does not require users to re-quantize their models. The methodology is clean and the evaluation is unusually thorough (two platforms, three models, both 4- and 8-bit, kernel + end-to-end + energy + ablation + comparison against prior NPU systems). A few honest critiques:

- **NPU-vs-iGPU framing is slightly cherry-picked for headline numbers.** The 2.0× prefill speedup is reported on the Ryzen AI 7 350, whose smaller 8-CU iGPU is precisely the weaker baseline. On the HX 370 (a more representative flagship SKU with a 16-CU iGPU), the W4A16 speedup shrinks to 1.22× at 8192 tokens and W8A16 actually *loses* across all measured configurations. The abstract headline (2.0×) is true but does not generalize across the product stack.
- **W8A16 results are unflattering and the paper handles them honestly.** On the HX 370, W8A16 is 0.62–0.79× of iGPU — i.e., the NPU is *slower* despite using more memory bandwidth. The paper attributes this to less compact kernels and platform memory behavior but does not fully diagnose it; for users on HX 370 with W8A16 weights, this kernel library is actively a regression.
- **Token-generation story is a weakness, not a feature.** Section 5 reports NPU *loses* to iGPU in generation throughput (Fig. 8) because reconfigurability overhead dominates sub-millisecond GEMV operators. The proposed hybrid "NPU prefill + iGPU decode" is reasonable, but the current implementation does *not* actually execute them concurrently or even switch mid-pipeline — it just dispatches the remaining ops to iGPU because attention is on iGPU anyway. The hybrid claim is therefore forward-looking, not demonstrated.
- **Comparisons with prior work are qualified.** Table 3 selects baselines "aligned with our target model families" — i.e., it does not run an apples-to-apples benchmark of TileFuse vs. FastFlowLM on the *same* hardware with the *same* prompt. The 1.45×/1.61× speedups are credible but should be read as cross-paper numbers, not head-to-head.
- **Generalizability claims are mild.** The interleaved pre-tiling idea is described as portable to "other GPUs and NPUs", but the paper provides no evidence beyond XDNA2. The actual reusable artifact is the methodology; the kernel code itself is XDNA2-specific.
- **No discussion of accuracy / quality impact of the W4A16 + asymmetric zero-point path.** The fused dequantization is a numerical re-implementation of AWQ; the paper shows end-to-end LLM runs but does not report perplexity or token-level quality comparisons against fp16 or against the same model run on llama.cpp iGPU.
- **Tile-size rigidity hurts Qwen2.5-3B.** Section 5 explicitly notes Qwen2.5-3B MLP hidden dim is not a multiple of the minimum tile size, causing padding overhead — a practical deployment footgun that the paper surfaces but does not solve.

Overall: a solid contribution with strong engineering, honest about its limitations, and a clear step toward making XDNA2 usable for the AWQ-quantized LLM ecosystem. The headline numbers should be read alongside the per-platform breakdown.

## Strengths

- **Truly off-the-shelf AWQ support:** No model-side quantization reshuffle required; users can take an existing AWQ checkpoint and run it on XDNA2.
- **Kernel-level fusion is principled:** The local-memory dequant cache + VLIW-overlap design is a clean way to avoid the redundant dequantization pitfall of naive fusion.
- **Interleaved pre-tiling is a genuinely portable idea:** Even if the kernel is XDNA2-specific, the offline-layout reorganization is a pattern that any spatial-array NPU could benefit from.
- **GEMV dataflow co-design goes beyond microkernel:** Routing through memory cores to use all four compute rows is a non-obvious contribution that addresses the architectural underutilization at the array level.
- **End-to-end + kernel + energy + ablation + cross-paper comparison** is unusually comprehensive for a systems paper; the methodology is reproducible (open-source link provided).
- **Honest treatment of reconfigurability as a double-edged sword** — the paper characterizes when XDNA2's flexibility helps and when its overhead hurts, which is rare and valuable.

## Limitations

- **Token generation is worse than iGPU** on every measured model, and the hybrid NPU+iGPU strategy is described but not actually realized as a runtime switch.
- **W8A16 underperforms on the HX 370** across nearly all configurations — the library is not uniformly useful across the Ryzen AI product line.
- **No concurrent NPU+iGPU execution** in the current implementation; the operator-level dispatch design is sequential.
- **No quality/perplexity evaluation** of the fused dequantization path against reference implementations.
- **Tile-size rigidity** causes padding overhead for some models (Qwen2.5-3B MLP shapes).
- **Strix Point vs. Krackan Point discrepancy** is acknowledged but not fully diagnosed; AMD does not publish enough NPU memory-path detail to isolate causes.
- **Comparison with prior NPU works** is cross-paper rather than head-to-head on identical hardware.

## Relevance

TileFuse 直接对应本研究兴趣中的"高效 LLM 推理"与"低精度/量化部署"主线:它证明当 NPU 配套开源低层级编程框架 (MLIR-AIE/IRON) 时,主流 AWQ 风格量化模型可以无需重新量化即在端侧 NPU 上高效运行,并展示了端云协同 (NPU prefill + iGPU decode) 的可行性。对关注边缘 LLM 部署、AMD XDNA 生态、或量化感知内核设计的读者来说,这是一篇兼具工程深度与方法论启发性的工作。

## Questions for Further Exploration

- **Quantization-accuracy parity:** Does TileFuse's fused INT4 unpacking + per-group asymmetric dequantization produce numerically identical outputs to the reference AWQ dequantization? Could fused dequantization in a fixed-point path ever drift enough to affect downstream logits?
- **Real hybrid NPU-prefill + iGPU-decode runtime:** Can the current sequential operator dispatch be extended to a true pipeline where the next prefilling chunk overlaps with the current decode, exploiting unified memory?
- **Concurrent NPU+iGPU execution:** What is the actual synchronization cost under ROCm/HIP for split operators on Ryzen SoCs? Is the unified-memory claim robust to KV-cache traffic?
- **Tile-size flexibility:** Can a dynamic-tile or "ragged" GEMM kernel eliminate the padding overhead for non-multiple-of-tile hidden dimensions (Qwen2.5-3B)?
- **Activation quantization (W4A8 / W4A4):** Does the same interleaved pre-tiling + fused microkernel pattern extend to QServe-style W4A8KV4? This would substantially expand the deployable model set on XDNA2.
- **Beyond AWQ:** GPTQ and the newer K-quants in llama.cpp use different group structures and metadata layouts — how portable is the pre-tiling strategy?
- **Cross-architecture portability:** Can the interleaved pre-tiling pattern be applied to Qualcomm Hexagon NPU (Snapdragon) or Apple ANE without major surgery?
- **Power modeling:** The 64.6% energy reduction is measured end-to-end. How much of that comes from NPU vs. iGPU power characteristics vs. memory subsystem effects? A per-stage power breakdown would clarify the source.
- **Scaling to larger FFNs:** The 32K limit comes from DMA BD constraints. What happens at 64K+ shapes (GPT-OSS, Llama-4-class models)?
- **KV-cache placement:** Since token generation still runs on iGPU, where does the KV cache live during NPU prefill? A NPU-resident KV cache could avoid iGPU↔NPU traffic during long-context generation.
