> **一句话总结：** 本文以"严格闭环"为前提系统比较了四类自进化（SE）策略，在 Knights and Knaves (KK) 逻辑推理与 OpenThoughts3 数学推理任务上，发现闭环自监督虽能稳步提升基模型，但与有监督 oracle 之间始终存在 8–13% 的不可消除差距，只有 Gemma 12B 的多轮 RevisionSE 才几乎追平 oracle。

> **TL;DR:** This paper asks how close closed-loop self-evolution (SE) can get to oracle-supervised training when the learner only has an unlabeled prompt set and a single base model. Studying four canonical strategies (single-round verification, multi-turn revision, iterative training, and curriculum learning) under a unified offline preference-learning framework, the authors show on Knights-and-Knaves (KK) that SE consistently lifts the base model (Gemma 3 4B: 31.0% → 44.8%) but plateaus and still trails oracle supervision by 8–13%. The largest, multi-turn RevisionSE at 12B nearly matches the oracle (52.8% vs. 53.6%). On OpenThoughts3 / GSM8K / MATH500 the gains are much smaller, and the work concludes that internal verification mostly sharpens existing solution modes rather than unlocking new reasoning.

## 核心贡献

- 提出严格的 **closed-loop 设定**：除无标签 prompt 集 $D$ 和基模型 $M$ 外，所有推理轨迹、奖励、偏好标签均来自 $M$ 自身，从而能精确度量内部信号所能填上的差距。
- 把四种典型 SE 策略（SimpleSE 单轮验证、RevisionSE 多轮 critic–revision、IterativeSE 多轮迭代、CurriculumSE 由易到难课程）统一到同一个 **generator–verifier game** + DPO 离线偏好优化框架。
- 在 **Knights and Knaves (KK)** 这一具有确定解、可控难度的逻辑推理基准上进行系统实证，并扩展到 **OpenThoughts3** 等真实数学/推理任务。
- 给出关于 **计算-精度权衡** 的实用结论：扩 verifier 计算量比扩 generation 量更划算；阈值 $\tau \approx 0.7$ 是精度/召回的折中甜点。
- 提出两条核心 takeaway：**Capability Threshold**（SE 需要足够强的内部验证/修订能力）与 **Generalization Limit**（闭环 SE 主要 "sharpen" 而非解锁新能力）。

## Key Contributions

- A **strict closed-loop formulation**: only an unlabeled prompt set $D$ and base model $M$ are available; all traces, rewards, and preference labels come from $M$ itself. This isolates the gap that purely internal signals can close.
- A **unified generator–verifier framework** that subsumes four SE strategies — SimpleSE (single-round verification), RevisionSE (multi-turn critic + revision), IterativeSE (multi-round offline DPO), and CurriculumSE (easy → hard) — all trained with offline DPO preference learning.
- Systematic empirical analysis on **Knights-and-Knaves (KK)** with deterministic solutions and parameterizable difficulty (2–8 people), avoiding data contamination and enabling easy-to-hard generalization studies.
- Extension to real-world reasoning (OpenThoughts3 → GSM8K, MATH500, MATHHard, TabMWP, KK) with comparisons against INTUITOR and Absolute Zero (AZR).
- A **cost–performance study** quantifying the trade-off between candidate generations $n_1$ and verifier passes $n_2$, showing that scaling verifier compute is generally more impactful than scaling generation.
- Two crisp **take-aways**: (i) there is a capability threshold below which closed-loop SE backfires, and (ii) internal feedback mainly sharpens existing solution modes rather than unlocking new reasoning, leaving a persistent generalization gap.

## Methodology

### Closed-Loop Generator–Verifier Game

A single base model $M$ is instantiated twice via system prompts as a **generator $G$** and a **verifier $V$**. For prompt $q \in D$:

- Generator produces $k$ candidates $\hat{y}_i \sim G(\cdot \mid q)$.
- Verifier assigns $V(q, \hat{y}_i) \in \{\text{Correct}, \text{Incorrect}\}$.
- A **preference pair** $(y_w, y_l)$ is extracted iff one candidate is judged correct and another incorrect.

### Four SE Variants

1. **SimpleSE (single-turn verification, $\tau$-thresholded majority voting):** query the verifier $n$ times per candidate; empirical correctness rate $\hat{p}(q,\hat{y})$ is computed. Mark as **positive** if $\hat{p} \geq \tau$, **negative** if $1-\hat{p} \geq \tau$, else discard. Higher $\tau$ → higher precision but more data sparsity.
2. **RevisionSE (multi-turn critic + revision):** the generator iteratively refines $\hat{y}^{(t+1)} \sim G(\cdot \mid q, f(V(q,\hat{y}^{(t)})))$, and a preference pair is extracted between an incorrect solution and a later-correct revision.
3. **IterativeSE:** repeatedly apply the generator–verifier game on the *same* dataset $D$, each round producing a fresh offline preference set $P_t$ and fine-tuning the previous model $M_{t-1} \to M_t$.
4. **CurriculumSE:** partition $D$ into $D_{\text{easy}}, D_{\text{hard}}$ by KK difficulty (number of people), train on easy first, then hard.

### Preference Learning

All variants use **Direct Preference Optimization (DPO)** on triples $(x, y_w, y_l)$ with reference policy $\pi_{\text{ref}}$ held fixed:

$$
\mathcal{L}_{\text{DPO}}(\pi_\theta; \pi_{\text{ref}}) = -\mathbb{E}_{(x,y_w,y_l)\sim D_{\text{pref}}} \log \sigma\!\left(\beta \log \tfrac{\pi_\theta(y_w\mid x)}{\pi_{\text{ref}}(y_w\mid x)} - \beta \log \tfrac{\pi_\theta(y_l\mid x)}{\pi_{\text{ref}}(y_l\mid x)}\right).
$$

Crucially, all preference data is generated **offline**, so each iteration is a fully offline DPO refinement — no online RL rollouts.

### Oracle Supervision Baseline

The **oracle verifier** uses the ground-truth solutions and exact-match labeling. This serves as the upper bound that any closed-loop SE is benchmarked against.

### Evaluation Protocol

- **Models:** Gemma-3-IT (1B, 4B, 12B; 27B as an upper-bound roofline) and Qwen-2.5-Instruct (7B, 14B).
- **Primary testbed:** Knights-and-Knaves (KK), parameterizable by number of people (2–3, 4–5, 6–8).
- **Secondary benchmarks:** OpenThoughts3 prompts → GSM8K, MATH500, MATHHard, TabMWP.
- **Metric:** exact-match accuracy; one sample at temperature 0.7, averaged over 4 random seeds.

## Results & Highlights

### KK with Gemma-3-4B-IT (main results)

Training on KK with 2–3 people, evaluating on 2–8:

| Method | 2–3 ppl | 4–5 ppl | 6–8 ppl | All |
|---|---|---|---|---|
| Gemma-3-4B-IT base | 62.0 | 31.0 | 10.3 | **31.0** |
| SimpleSE ($\tau=0.6$) | 70.9 | 45.4 | 17.5 | **40.7** |
| RevisionSE | 75.8 | 46.4 | 17.1 | **42.2** |
| IterativeSE (3 rounds) | 75.2 | 49.6 | 19.7 | **44.1** |
| CurriculumSE (KK23→KK45) | 76.2 | 49.7 | 20.6 | **44.8** |
| Oracle verifier (single round) | 78.4 | 52.6 | 21.4 | **46.6** |
| Any SE rounds + **final oracle round** | 82–85 | 58–62 | 25–30 | **52.6–53.3** |

A clear pattern emerges: closed-loop SE monotonically improves with structure (SimpleSE → RevisionSE → IterativeSE → CurriculumSE), but a **8–13 pp gap to the oracle persists**. Only adding a final oracle round pushes accuracy to 52–53%.

### Scaling with Model Size

- Gemma-3-1B-IT: gains essentially zero (7.8% → 8.4%); RevisionSE can even hurt.
- Gemma-3-4B-IT: large relative gains (~31% relative).
- Gemma-3-12B-IT + **RevisionSE** = **52.8%** vs. oracle **53.6%** — nearly matches the oracle (98.5% of oracle performance).
- Gemma-3-27B-IT acts as a roofline at 51.1% (with KK-2–3 training).

### Iterative Training

Three unsupervised DPO rounds raise 4B from 31.0% → 44.1%, close to a single oracle round (46.6%), but adding a final oracle round yields the biggest jump (to 52–53%) — confirming the residual gap.

### Curriculum Learning

KK23 → KK45 with SimpleSE: 44.8% (avg), beating random mixing (41.2%). However, the same trend appears under the oracle, so curriculum does **not** specifically help closed-loop SE beyond what it already gives supervised training.

### Cost–Performance Trade-off

Sweeping $n_1$ (generations/query) and $n_2$ (verifier passes/candidate) across $\tau \in \{0.5, 0.6, 0.7, 0.8\}$:

- $\tau = 0.7$ with $n_1 = 8, n_2 = 8$ hits 41.7% avg accuracy, the best practical point.
- Very high $\tau$ (0.8) causes data sparsity, hurting recall.
- Verifier compute is more impact-per-token than generator compute (intuition: better verifier signal cleans the preference pairs more than adding more candidates helps).

### Real-World Reasoning (OpenThoughts3)

Training Gemma-3-4B-IT and Qwen-2.5-7B-Instruct on 20K OpenThoughts3 prompts and evaluating on five benchmarks:

| Model | GSM8K | MATH500 | MATHHard | TabMWP | KK |
|---|---|---|---|---|---|
| Gemma-3-4B-IT base | 89.2 | 75.8 | 53.7 | 84.5 | 31.0 |
| + SimpleSE | 89.0 | 77.4 | 55.1 | 87.4 | 33.2 |
| Qwen2.5-7B-Instruct base | 90.2 | 73.5 | 49.7 | 91.9 | 18.1 |
| + SimpleSE | 90.6 | 76.0 | 51.5 | 92.3 | 17.6 |
| Qwen2.5-7B + INTUITOR | 87.3 | 75.0 | – | – | – |
| Qwen2.5-7B + AZR | 84.0 | 74.4 | 32.8 | 68.8 | 5.1 |

Gains are **much smaller** (~+1.5% on MATH500, +2.9% on TabMWP for Gemma 4B) and can even degrade (KK Qwen). Scaling prompt count to 40K shows **diminishing returns** — the SE curve plateaus.

### Sharpening, Not Expansion

Pass@1 improves but Pass@32 is roughly unchanged under SE: closed-loop SE steers the model toward its existing solution modes (the **sharpening hypothesis**) rather than expanding the set of solvable problems.

## Critical Assessment

**Honest evaluation:**

- **Novelty:** The taxonomy and unified framework are useful, but most individual ingredients (self-verification, rejection sampling, iterative DPO, curriculum learning, multi-turn revision) already exist. The paper's main novelty is the *systematic empirical comparison under a strict closed-loop formulation*, not new algorithms.
- **Rigor:** The closed-loop formulation is well-defined. The choice of KK as the primary testbed is principled (deterministic, contamination-resistant, difficulty-tunable). However, restricting most conclusions to KK and OpenThoughts3 (mathematical reasoning) limits the generality — open-ended domains like dialogue, code generation, or science QA are not tested.
- **Reproducibility:** Models, datasets, and method are all public. Hyperparameters for thresholds and budgets are ablated. Some details (exact prompts for generator/verifier) are deferred to Section C of the appendix, which is reasonable.
- **Statistical rigor:** 4 random seeds with reported std. dev.; numbers are stable enough to support the main conclusions.
- **Conflict of interest:** Google authors evaluating Google (Gemma) models. While disclosed, this is a structural caveat — the result that "Gemma 12B RevisionSE nearly matches the oracle" should be re-verified by an independent team on a non-Google model. The Qwen2.5-7B/14B results only use SimpleSE (not RevisionSE), so the scaling conclusion leans heavily on one model family.
- **Scope:** "Closed-loop SE" is by design narrow — the authors explicitly exclude external tools, multiple models, or verifiable environments. This is a feature for clean analysis but also means the paper does not test setups like STaR, R-Zero, or RLVR with execution feedback.
- **Significance:** This is an *empirical limit paper* — it doesn't propose a new SOTA method. Its contribution is to **pin down where internal feedback works and where it doesn't**, which is valuable for the field's mental model of self-improvement.
- **Red flags:** None major. Minor: the conclusion that SE "sharpens" existing modes is supported by Pass@1 vs. Pass@32 numbers, but the Pass@32 evidence is mentioned as a Section A.3 appendix note rather than a featured figure, so the case is not hammered home as strongly as it could be.

## Strengths

- **Clean, controlled experimental design.** KK is an ideal testbed: deterministic, contamination-resistant, difficulty-tunable. The closed-loop formulation isolates exactly what is being measured.
- **Unified framework** subsuming four SE strategies, with consistent DPO training, the same model families, and the same evaluation protocol — making comparisons fair.
- **Multi-scale coverage** across 1B/4B/12B/27B Gemma-3 and 7B/14B Qwen-2.5 models, plus realistic downstream benchmarks (GSM8K, MATH500, MATHHard, TabMWP).
- **Practical engineering insights:** $\tau \approx 0.7$ sweet spot; verifier compute > generator compute; diminishing returns beyond ~20K–40K prompts.
- **Sharp, honest conclusions** about when closed-loop SE helps (capable model + verifiable task) and when it doesn't (open-ended tasks, OOD generalization).
- **Transparency:** conflict-of-interest disclosure, ablation tables with std. dev., explicit comparison against INTUITOR and AZR.

## Limitations

- **Closed-loop only.** Excludes tools, external verifiers, multi-model setups, and verifiable rewards — so conclusions do not directly inform production RLVR pipelines (e.g., DeepSeek-R1, OpenAI o-series).
- **Single primary testbed.** Most numerical claims are on Knights-and-Knaves; generalization to other logical or commonsense reasoning benchmarks is not deeply explored.
- **Limited model diversity for RevisionSE.** Only Gemma-3-IT is evaluated under RevisionSE; Qwen results only use SimpleSE. The "12B nearly matches oracle" claim is thus a single-model-family observation.
- **Online RL comparison is not fully controlled.** INTUITOR and AZR are reported in their *original* online-RL-from-base settings, while SimpleSE starts from *instruction-tuned* models. This makes the comparison illustrative rather than apples-to-apples.
- **Open-ended reasoning gains are tiny.** +1.5–2.9% on GSM8K/MATH500/TabMWP are arguably within noise; the paper could have done a tighter significance analysis or paired significance tests.
- **No theoretical analysis.** The paper is purely empirical. A formal account of *why* the gap persists (e.g., bounds on learning from synthetic distributions à la Amin et al. 2026) would strengthen the claims.
- **Gemma-only sharpening evidence.** Pass@1 vs. Pass@32 is reported only for Gemma; whether the same holds for Qwen is unclear.

## Relevance

对关注 **LLM 自我改进、可验证奖励 (RLVR) 替代方案、以及测试时计算/后训练研究** 的读者来说，本文是一份冷静的"经验边界"参考：它告诉我们在不引入外部信号的前提下，自进化能填上一部分缺口，但无法替代真实监督或可执行环境的奖励。论文与用户兴趣中"RL without ground truth"、"self-play / self-rewarding"、"iterative DPO"、"curriculum reasoning"高度相关，可在设计无标签数据增强、离线偏好优化或自博弈方案时作为基准参考。

## Questions for Further Exploration

1. **Hybrid strategy.** Can one use a small amount of human-labeled data to bootstrap internal verifier quality, then run closed-loop SE on the rest — and does this close more of the gap than either alone?
2. **RevisionSE beyond 12B.** Does the "nearly matches oracle" trend continue at 27B/70B scale, or does it saturate? An independent team running RevisionSE on non-Gemma models (e.g., Qwen, Llama) would test generalizability.
3. **Connection to sharpening theory.** Can the Pass@1 vs. Pass@32 pattern be formalized — e.g., does closed-loop SE provably reduce entropy over the policy without expanding its support?
4. **Better verifier signals.** What if the verifier is allowed to access chain-of-thought or be augmented with self-consistency — does the 8–13% gap shrink, or is it fundamental?
5. **Dynamic compute allocation.** The cost analysis shows static $n_1, n_2, \tau$ are suboptimal. Can a per-instance adaptive scheme (e.g., allocate more verifier passes only when generator entropy is high) close more of the gap at equal compute?
6. **Domain transfer.** Does the gap persist on truly open-ended domains (creative writing, dialogue, science QA) where even humans disagree on "correctness"?
7. **Theoretical limits.** Tighten the connection to recent work on learning from synthetic data (Amin et al. 2026, model-collapse literature) to give information-theoretic bounds on closed-loop SE.
8. **Combination with inference-time compute.** Does combining RevisionSE-trained models with self-consistency / best-of-$N$ at inference close the gap further, since the two attack different stages of the pipeline?
