Static auto-harness evolution breaks on open-ended streams — so this paper decomposes the regret into evolution + adaptation loss and patches both with a tree of harnesses routed at solve-time.

02WHY THIS MATTERS

One of the few works that attacks deployment regimes — not static benchmarks — for self-improving LLM agents, offering a falsifiable lens (Levo vs Ladapt) instead of ad-hoc leaderboard tricks.

Beats A-Evolve · GEPA · Meta-Harness · Continual Harness · SkillOS · OctoTools reference.

Methodology Analyze → Research → Build → Verify

Builderfile-level edits
Verifiertests + checks

Three complementary mechanisms:

  • Stateful multi-agent evolver (§3.3) — four phases each get their own full context budget; a temporal-reveal gate surfaces rewards only after resolution, killing label leakage; persistent cross-cycle workspace (task board, logs, README, tests) carries inter-cycle memory.
  • Harness tree + solve-time routing (§3.4) — a git-backed tree of regime-specific branches (sports, politics, crypto-classical, …); a router agent reads each branch's workspace via git show and checks out the best one per task (confidence ≥ 0.7 else main).
  • Structured HITL hooks (§3.5) — two third-axis triggers when history lacks the signal: task-board steering + research-phase assistance (credentials, novel sources).

Solver: Claude Sonnet 4.6 · Evolver: Claude Opus 4.6 · Router: Sonnet 4.6 · Seed harness intentionally minimal (24–114 LoC, zero skills / tools / memory entries).

  • All three variants jointly lead — Multi-agent, Adaptive, and Full are complementary, not substitutable; Multi-agent edges FutureX (49.5%) where source retrieval beats routing.
  • L_evo evidence (Fig. 4) — bottleneck tracks regime: calibration on PolyBench, source tier (offline → curated → open) on FutureX (34.0 → 47.6 → 57.1), payload size on CTF-Dojo (single-agent 81.8 → 30.4 with growing payloads).
  • L_adapt evidence (Fig. 5) — per-task adaptation lift over the median committed-harness baseline stays positive across all cycles & categories — one dense harness is insufficient on a heterogeneous stream.
  • Ablations (Fig. 6) — removing cross-cycle memory = broadest degradation; removing feedback hurts mainly PolyBench (asynchronous labels).
  • HITL slice (Fig. 9) — lift over no-HITL: 0 broad / +5 broad search-dependent / +20 on targeted finance&tech / +15 adjacent Western-specialty slices.

Strengths

  • Conceptual clarity — Levo / Ladapt maps to actionable design axes (better evolver vs. better routing) and is cleanly falsifiable.
  • Pragmatic system design — git-backed harness tree, phase-separated multi-agent evolver, structurally triggered HITL — minimal, motivated, auditable.
  • Comprehensive evaluation — five auto-harness baselines + human-designed OctoTools across three chronologically ordered streams; coverage of pass-rate and portfolio metrics; ablation + bottleneck + routing analyses.
  • Honest about limits — authors flag that diagnostic losses are not directly estimated, that Adapt captures only part of Oracle headroom, and that FutureX hits a source-retrieval ceiling.
  • Operational robustness — temporal-reveal gate + CTF-Dojo sandbox (constrained network policy, SHA-256 flag verification) suit infrastructure-grade reuse.

Limitations

  • Generalization — only three streams; authors call for testing on streams that "expand spatially and temporally."
  • Routing headroom gap — Adapt only partially closes the gap to Oracle branch selection (Fig. 7); system is better at constructing branches than routing to them.
  • Commercial-model dependence — Sonnet 4.6 / Opus 4.6 are provider-hosted and closed-weight; gains aren't obviously transferable to smaller / open models.
  • HITL dependency — fully autonomous deployments degrade to the no-HITL variant.
  • No formal estimator — Levo / Ladapt approximated via bottleneck / ablation, not measured against an oracle harness.
  • Cycle budget skew — PolyBench: 51 cycles / 5,075 tasks; CTF-Dojo: 14 cycles / 261 — smaller stream potentially under-evolved.

Questions for Further Exploration

  • Estimable decomposition? — Can Levo / Ladapt be turned into a directly estimated quantity (oracle-harness proxy or counterfactual replay) instead of diagnosed via bottleneck / ablation proxies?
  • Multi-modal & tool-rich streams? — How does the system behave on web-shopping, code migration, long-horizon robotics, where branch-relevant skills span modalities rather than topical regimes?
  • Federated / privacy-preserving deployment? — Does the harness-tree + git-routing pattern transfer where branches must be sandboxed or routing decisions auditable post-hoc?
  • Branch-opening trigger? — Is the EGL threshold (0.05 over 3 cycles) derivable from the Levo trajectory itself rather than empirical?
  • Model substitution sensitivity? — Sonnet 4.6 → open-weight 70B-class? Same model for solver + evolver to remove one variance source?
  • Human-agent co-evolution? — Can the HITL hooks generalize into a protocol where the human's role is value alignment under distribution shift, not just credential / source provision?
  • Adversarial drift? — How does the tree's branch routing survive a stream where the task distribution is intentionally perturbed to break it?
