TileFuse brings AWQ-quantized LLMs natively to AMD XDNA2 NPUs — 2.0× faster prefill, 64.6% lower energy.

TileFuse: A Fused Mixed-Precision Kernel Library for Efficient Quantized LLM Inference on AMD NPUs ↗

Wesley Pang, Gregory Hyegang Jun, Feiyang Liu, Deming Chen · UIUC · AMD Center of Excellence · Code & benchmarks open-source
AMD XDNA2 NPU AWQ Quantization Mixed-Precision Kernel Edge LLM Inference MLIR-AIE / IRON Ryzen AI
Key Metric
2.0×
End-to-end prefill speedup on Llama3-8B @ 2048 tokens vs. llama.cpp iGPU (Ryzen AI 7 350). NPU path also draws −64.6% energy on the same workload.
Why This Matters
AWQ-quantized LLMs now run natively on edge NPUs — no requantization, no model surgery — proving efficient edge LLM deployment is finally feasible on AMD Ryzen AI silicon, with NPU prefill replacing a tuned iGPU stack.
Critical Edge
The 2.0× headline holds only on the weaker 8-CU iGPU — on the 16-CU HX 370, W8A16 actually loses to iGPU across nearly all shapes, and token generation underperforms on every measured model because NPU dispatch overhead dwarfs sub-millisecond GEMV.

01Methodology方法

  • Hardware & stack. AMD XDNA2 NPU on Ryzen AI SoCs (Strix Point HX 370, Krackan Point 7 350): a 4×8 array of 32 AIE compute cores + 8 memory cores + 8 shim cores. Programmed via the open MLIR-AIE / IRON framework.
  • Offline pre-tiling. Per-tile metadata packing (INT4 weights + BF16 scales + duplicated INT8 zero-points) and an interleaved column-major layout that keeps same-AIE-column tiles contiguous — bypasses DMA BD stride limits and lifts supported GEMM dim from 8K → 32K at only ~1.5% extra memory.
  • Fused GEMM microkernel. Inside each 8×8 sub-tile: unpack INT4 → widen to INT8 → dequant to BF16 → cache BF16 in local memory for re-use across activation rows → accumulate in FP32. VLIW overlap hides local-buffer traffic.
  • 4×8 GEMV dataflow. Widens the inner kernel to a 64×8 weight block per iteration; routes through memory cores as an intermediate distribution stage so all 4 compute rows receive distinct weight tiles while sharing the 1×8 broadcast activation — scales output-tile groups from 8 to 32.
  • End-to-end runtime. Python frontend (HF safetensors + offline pre-tiling + tokenization) + libtorch C++ backend. Linear layers → NPU via precompiled TileFuse kernels; FlashAttention-2, softmax, residual add, norms → iGPU via HIP. Operator-level (not concurrent) dispatch to isolate TileFuse's contribution.
  • Baselines. llama.cpp fused quantized HIP kernels on the iGPU (Q4_K / Q8_0) — strongest publicly available quantized GEMM for RDNA iGPUs as of Jan 2026. CPU Q4_K / Q8_0 also reported.

03Strengths & Limitations优 / 缺

  • Truly off-the-shelf AWQ. Drop-in for existing AWQ checkpoints — no requantization to an NPU-proprietary scheme.
  • Principled kernel fusion. Local-memory dequant cache + VLIW overlap avoids the redundant-dequant pitfall of naive fusion.
  • Portable pre-tiling idea. Even though the kernel code is XDNA2-specific, the offline column-major tile reorganization is reusable on any spatial-array NPU.
  • Dataflow co-design. Routing through memory cores to use all 4 compute rows addresses array-level underutilization, not just microkernel throughput.
  • Comprehensive evaluation. Kernel + end-to-end + energy + ablation + cross-paper comparison — unusually thorough for a systems paper.
  • Honest reconfigurability analysis. Explicitly characterizes when XDNA2's flexibility helps and when its dispatch overhead hurts.
  • Token generation is a regression. NPU loses to iGPU on every measured model — millisecond-scale dispatch overhead dwarfs sub-millisecond GEMV.
  • W8A16 on HX 370 is a real loss. 0.62–0.79× of iGPU across all shapes — the library is not uniformly useful across the Ryzen AI product line.
  • Hybrid NPU+iGPU is forward-looking, not realized. Current implementation dispatches remaining ops to iGPU because attention is on iGPU anyway — no runtime switch, no concurrency.
  • No quality / perplexity check. Fused INT4 unpack + asymmetric dequant is numerically a re-implementation of AWQ, but never compared against a reference dequant.
  • Tile-size rigidity. Qwen2.5-3B MLP hidden dim is not a multiple of the minimum tile → padding overhead.
  • Cross-paper, not head-to-head. Table 3 baselines are aligned with model families, not run on identical hardware & prompt.

04Questions for Further Exploration待探索

  • Quantization-accuracy parity. Does TileFuse's fused INT4 unpacking + per-group asymmetric dequant produce numerically identical outputs to the reference AWQ dequant? Could a fixed-point path ever drift enough to shift logits?
  • Real hybrid NPU-prefill + iGPU-decode runtime. Can sequential operator dispatch become a true pipeline where the next prefilling chunk overlaps the current decode, exploiting unified memory?
  • Concurrent NPU+iGPU. What is the actual ROCm/HIP synchronization cost for split operators on Ryzen SoCs? Is the unified-memory claim robust to KV-cache traffic?
  • Tile-size flexibility. Can a dynamic-tile / "ragged" GEMM kernel eliminate the padding overhead for non-multiple-of-tile hidden dimensions (Qwen2.5-3B)?
  • Activation quantization. Does the same pre-tiling + fused microkernel pattern extend to QServe-style W4A8KV4, expanding the deployable model set on XDNA2?
  • Beyond AWQ. GPTQ and llama.cpp's K-quants use different group structures and metadata layouts — how portable is the pre-tiling strategy?
  • Cross-architecture portability. Can interleaved pre-tiling apply to Qualcomm Hexagon (Snapdragon) or Apple ANE without major surgery?
  • Power decomposition. Of the 64.6% energy saving, how much comes from NPU power vs. iGPU power vs. memory subsystem? A per-stage power breakdown would clarify the source.
  • Scaling to 64K+ FFNs. The 32K limit comes from DMA BD constraints. What happens at GPT-OSS / Llama-4-class shapes?
  • KV-cache placement. Token generation still runs on iGPU — where does the KV cache live during NPU prefill? A NPU-resident KV cache could avoid iGPU↔NPU traffic during long-context generation.
