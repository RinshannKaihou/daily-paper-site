> **一句话总结：** 作者指出当前自动 harness 演化（A-Evolve / GEPA / Meta-Harness）在无界、异构、非平稳的开放任务流上会随时间退化，进而提出 Adaptive Auto-Harness：一个由"分析—研究—构建—验证"四阶段多智能体演化器、基于 git 的 harness-tree 求解路由、以及历史信号不足时的结构化人在环挂钩组成的系统，在 Polymarket、CTF-Dojo、FutureX 三个长程流上分别取得 80.9% 准确率、50.2% Pass@1 与 47.3% Pass@1，显著优于五种自动 harness 基线。

> **TL;DR:** The paper formalizes why static auto-harness evolution (A-Evolve, GEPA, Meta-Harness, Continual Harness, SkillOS) breaks on **open-ended task streams** — unbounded, heterogeneous, non-stationary. It decomposes the gap to an oracle harness into **evolution loss L_evo** (capabilities the evolver class cannot construct) and **adaptation loss L_adapt** (commitment to one dense harness before seeing the task). Adaptive Auto-Harness then proposes three complementary mechanisms: (1) a **stateful four-phase multi-agent evolver** with cross-cycle workspace memory and a temporal-reveal feedback gate, (2) a **harness tree with solve-time agentic routing** that isolates regime-specific skills and lets a router agent pick the best branch per task, and (3) **structurally triggered HITL hooks** for cases where history lacks the needed source / access signal. On PolyBench (5,075 markets), CTF-Dojo (261 CTFs), and FutureX (503 questions), the full system reaches 80.9% / 50.2% / 47.3% respectively, beating all five auto-harness baselines and a human-designed OctoTools reference, while ablations attribute gains to the three mechanisms independently.

## 核心贡献

- **部署体制的形式化分析**：将开放任务流解构为三个维度——无界流（D1）、任务异构（D2）、分布非平稳（D3），并首次给出针对 harness 的"演化损失 + 适配损失"二元分解（Proposition 1），为后续系统设计提供可证伪的目标。
- **状态化多智能体演化器**：用"分析 → 研究 → 构建 → 验证"四阶段流水线替代单 agent 演化器，配合跨周期的 workspace（任务板 / 研究日志 / README / 验证测试）和"时序揭示"反馈门，避免单上下文窗口的瓶颈，并解决标签异步到达的泄露问题。
- **Harness 树 + 求解时路由**：把演化空间的组织形式（线性链 → 树 / 图）与适配算子（检索 → 分支级路由）解耦；树用 git 仓库实现版本化与分支隔离，路由 agent 通过 `git show` 读取每个分支的 workspace 并为每个任务选择最合适的分支。
- **结构化人在环通道**：在 L_evo 与 L_adapt 都无法解决时引入第三轴——人类只在"任务板转向"与"研究阶段凭证 / 来源协助"两个挂钩上介入，避免污染求解与标签，并定位到缺失的外部信号。
- **三流统一实证**：在 Polymarket、CTF-Dojo、FutureX 三个具有不同瓶颈（校准 / 来源获取 / 文件大小）上的实验显示，三个机制是互补的而不是可互换的，并提供了细粒度的瓶颈诊断、组件消融和 HITL 分片分析。

## Key Contributions

- **Deployment-regime analysis**: Formalizes why one-shot, dense, repeatedly-updated harnesses fail on open-ended streams by isolating three dimensions (unbounded, heterogeneous, non-stationary) and deriving a regret decomposition into evolution loss (L_evo) and adaptation loss (L_adapt).
- **Stateful multi-agent evolver (§3.3)**: A four-phase (Analyze → parallel Researchers → Builder → Verifier) pipeline with persistent cross-cycle workspace (task board, research logs, README, verification tests) and a temporal-reveal gate that delays outcome labels until task resolution, removing the single-context-window bottleneck.
- **Solve-time adaptation via harness-tree routing (§3.4)**: A git-backed tree of regime-specific branches (sports / politics / binary-reversing / crypto-classical, etc.) plus an agentic router that reads each branch's workspace via `git show` and checks out the best branch per incoming task.
- **Human-in-the-loop as a third axis (§3.5)**: Two structurally triggered hooks — task-board steering (priority / domain guidance / source access) and interactive research-phase assistance (e.g., API credentials, novel web sources) — that engage only when history lacks the needed signal.
- **Comprehensive empirical validation**: Beats five auto-harness baselines (A-Evolve, GEPA, Meta-Harness, Continual Harness, SkillOS) and a human-designed OctoTools reference on PolyBench / CTF-Dojo / FutureX, with three diagnostic axes (per-cycle bottleneck, per-category adaptation lift, HITL slice) and component ablations.

## Methodology

- **Problem setup**: Open-ended task stream `x_1, x_2, …, x_T` with `x_t ∼ P_t`. Agent observes history `H_t = {(x_i, r_i, τ_i)}_{i=1..t-1}` (optional rewards, full trajectories). Harness `C = φ(H_t)` with capacity budget `|C| ≤ K` wraps a fixed LLM solver via policy `π(a | x_t, C_t)`.
- **Analytical decomposition (Prop. 1)**: For any deployed evolver `φ ∈ Φ`,
  - `L_evo(Φ) = E[V(H_t, x_t) − V(C*_Φ(x_t), x_t)]` — the evolver-class capability gap (e.g., single-agent editors cannot produce multi-file infrastructure).
  - `L_adapt(φ) = E[V(C*_Φ(x_t), x_t) − V(φ(H_t), x_t)]` — the cost of committing to one harness before seeing `x_t`, non-zero whenever tasks are heterogeneous.
  - HITL sits outside both terms: a third axis when `H_t` does not contain the needed source / access signal.
- **Multi-agent evolver (§3.3)**:
  - **Multi-agent with distinct roles**: Analyst → parallel Researchers (3 agents, hypothesis-test loop) → Builder (file-level edits + infrastructure) → Verifier (tests + contradiction checks), each with its own full context budget — eliminating the single-window bottleneck.
  - **Temporal-reveal feedback**: A gate surfaces each task's reward only after its resolution date, preventing future-leakage into evolver context.
  - **Persistent cross-cycle state**: A dedicated workspace (task board, research logs, README, tests) preserves inter-cycle knowledge so the evolver refines its construction ability over time.
- **Harness tree + routing (§3.4)**: The solver's workspace is a git repository; the evolver constructs regime-specific branches (e.g., `branch/sports`, `branch/crypto-classical`) during evolution; at solve time, a router agent reads each branch's workspace via `git show`, selects one based on the incoming task's context, and the solver checks it out. Routing uses a confidence threshold (0.7) to fall back to the main branch.
- **HITL hooks (§3.5)**:
  - **Task-board steering**: After the Analyst updates the task board, humans can add entries / adjust priorities / supply domain guidance and source access.
  - **Research-phase assistance**: When a Researcher hits a wall (auth wall, API key, novel source), the system prompts the human in real time to unblock it.
- **Implementation**:
  - Solver: Claude Sonnet 4.6 at T=0; Evolver: Claude Opus 4.6 at T=0; Router: Claude Sonnet 4.6.
  - Batch sizes 100/20/20 (PolyBench/CTF-Dojo/FutureX); 51/14/26 evolution cycles respectively.
  - Expected-Gain-from-Learning (EGL) trigger with threshold 0.05 over a 3-cycle window gates whether a new cycle runs.
  - Seed harness is intentionally minimal (24–114 LoC prompts, zero skills / tools / memory entries); all non-seed components are constructed by the evolver.

## Results & Highlights

- **Headline numbers (Full System vs best baseline)**:
  - **PolyBench**: 80.9% Accuracy and +330% Return, vs Meta-Harness 50.8% / +320% (Return best on Multi-agent variant +352%).
  - **CTF-Dojo**: 50.2% Pass@1 vs A-Evolve 45.2% (best single-agent Sonnet baseline 37.2%).
  - **FutureX**: 47.3% Pass@1, second to A-Evolve 47.5%; Multi-agent variant leads FutureX at 49.5%.
  - Adaptive variant leads PolyBench Return (+352%); Multi-agent leads FutureX (49.5%) where source/tooling matters more than routing.
- **Three variants jointly lead all metrics** — the three mechanisms (multi-agent evolution, solve-time routing, HITL) are complementary rather than substitutable.
- **Evolution loss evidence (L_evo) — benchmark-specific bottlenecks (Fig. 4)**:
  - PolyBench: confidence calibration against market consensus — multi-agent variant tracks the diagonal, single-agent stays flat / over-confident.
  - FutureX: source access — pass rate rises 34.0% → 47.6% → 57.1% as the retrieval tier moves from offline → curated web → open web.
  - CTF-Dojo: payload handling — best single-agent drops 81.8% → 30.4% as files grow from no-payload to >1MB; multi-agent retains a ~9-point margin throughout.
- **Adaptation loss evidence (L_adapt) (Fig. 5)**: Per-task adaptation lift over the median committed-harness baseline stays positive and meaningful across all cycles and task categories — a single committed dense harness is insufficient on a heterogeneous stream.
- **Ablations (Fig. 6)**: Stateful multi-agent evolver improves over single-agent on all three benchmarks: PolyBench CWR 20.3 → 44.3; CTF-Dojo Pass 38% → 43%; FutureX 38% → 44%. Removing cross-cycle memory causes the broadest degradation; removing feedback mainly hurts PolyBench (where labels resolve asynchronously).
- **Routing analysis (Fig. 7)**: On an evolved tree with one branch per category, Oracle (best branch per task) leaves meaningful headroom over both Naive (main workspace only) and Worst; Adapt (category-based routing) captures part but not all of that headroom. On FutureX, main outperforms Adapt because web-source retrieval dominates over specialization.
- **HITL evidence (Fig. 9)**: Lift over no-HITL run on 100 FutureX tasks — 0 on broad polymarket, +5 on broad search-dependent, +20 on the directly targeted finance&tech slice, +15 on adjacent Western-specialty slices. The mechanism helps when it injects missing external signal, not generic advice.
- **Failure example from Figure 1**: A skill `news_from_future.md` (138 correct vs 16 wrong BUYs) helps on a sports task yet misfires on a politics task — concrete demonstration that long, dense evolution overfits earlier stream evidence.

## Critical Assessment

- **Originality**: High. The decomposition of harness regret into L_evo and L_adapt, and the addition of HITL as a third axis, is a clean conceptual contribution that goes beyond the usual "more samples / longer context" recipe. The harness tree + git-based branch versioning is a pragmatic and unusually well-grounded design choice.
- **Methodology**:
  - Strong: multi-agent with explicit phase boundaries, temporal-reveal gate to prevent label leakage, and persistent cross-cycle workspace. The git-backed harness tree is reproducible and auditable.
  - Concerns: The ablation logic (Fig. 6) shows broad degradation when memory is removed but only PolyBench-specific degradation when feedback is removed — a useful diagnostic, but the per-component gains are not always cleanly separable (the three variants — Multi-agent, Adaptive, Full — sometimes differ by <1pt, e.g., CTF-Dojo 47.9 vs 50.2).
  - L_evo and L_adapt are analytical, not estimated — the bottleneck evidence in Fig. 4 is suggestive rather than conclusive, since "binding capability" is asserted by ablation rather than by a measurable oracle baseline.
- **Rigor**:
  - Temperature 0 throughout for solver and evolver gives point estimates but no error bars; results are deterministic per run, so any variance comes from benchmark-side non-stationarity or from the LLM provider's nondeterministic serving.
  - Provider-hosted commercial models (Claude Sonnet 4.6 / Opus 4.6) make the headline numbers hard to reproduce without paid API access; closed parameter counts mean the paper reports "task counts and evolution cycles" instead of GPU-hours.
- **Reproducibility**: Public code link provided; seed harness and hyperparameters (Table 3 / 4) are documented; the temporal-retrieval contract and FutureX sandbox network are non-trivial to replicate.
- **Significance**: This is one of the first papers to systematically attack "long-horizon auto-harness" deployment rather than "static benchmark harness improvement." If the proposed decomposition holds, it gives the community a falsifiable lens (L_evo vs L_adapt) instead of ad-hoc comparisons — a meaningful step toward a theory of self-improving agents.
- **Red flags**:
  - "Coverage-scaled CWR = Return" is an unusual portfolio metric; readers should not equate it with raw accuracy.
  - FutureX hit by ~0.3pp from A-Evolve (47.3 vs 47.5) — the system does not dominate on every row, and the adaptive routing variant can underperform main on FutureX.
  - The headline Fig. 1 demonstration uses a hand-picked stopping-cycle comparison; the broader claim "longer evolution is worse" is shown only for A-Evolve on Polymarket, not for the proposed system.

## Strengths

- **Conceptual clarity**: The L_evo / L_adapt decomposition is a useful organizing principle that maps to actionable design axes (better evolver vs. better routing) and is cleanly falsifiable.
- **Pragmatic system design**: Git-backed harness tree, phase-separated multi-agent evolver, and structurally triggered HITL are each minimal, well-motivated, and avoid over-engineering.
- **Comprehensive evaluation**: Five auto-harness baselines plus a human-designed reference across three chronologically ordered streams; coverage of pass-rate and portfolio-style metrics; ablation + bottleneck + routing analyses that separate headroom from realized routing quality.
- **Honest about limits**: Authors explicitly note that diagnostic losses are not directly estimated, that routing captures only part of the headroom, and that FutureX hits a ceiling where source retrieval dominates over specialization.
- **Operational robustness**: The temporal-reveal gate and CTF-Dojo sandbox (constrained network policy, SHA-256 flag verification) are appropriately defensive for a paper that will plausibly be used as infrastructure.

## Limitations

- **Generalization beyond three streams**: PolyBench / CTF-Dojo / FutureX cover heterogeneity, but the authors flag that the framing should be tested on additional deployment streams that "expand spatially and temporally."
- **Routing headroom gap**: Adapt only partially closes the gap to Oracle branch selection (Fig. 7); the system is currently better at constructing branches than at routing to them.
- **Sensitivity to commercial model behavior**: Claude Sonnet 4.6 / Opus 4.6 are provider-hosted, temperature-0, and not open-weight; gains are not obviously transferable to smaller / open models.
- **HITL dependency**: The structured hooks help when history lacks the needed signal, but require a responsive human operator; in fully autonomous deployments the system degrades to the no-HITL variant.
- **Limited evidence for the "non-stationarity" axis**: Most of D3 evidence is indirect (delayed labels, accuracy-vs-time analyses on Fig. 1); explicit drift quantification per cycle would strengthen the claim.
- **No formal L_evo / L_adapt estimator**: The decomposition is analytical; the paper approximates the losses via bottleneck / ablation diagnostics rather than estimating them against an oracle harness.
- **Batch size × cycle count trade-off**: PolyBench uses 51 cycles over 5,075 tasks, but CTF-Dojo uses only 14 cycles over 261 — the smaller stream is potentially under-evolved, which may bias the cross-benchmark comparison.

## Relevance

- **与"自动 harness 演化"研究高度相关**：本文直接针对 A-Evolve / GEPA / Meta-Harness 等系统在长程开放任务流上退化的问题，提出可证伪的 L_evo / L_adapt 二元分解，以及多 agent 演化 + 路由 + HITL 的三轴互补机制，是当前自演化智能体文献中少数明确面向"部署体制"而非"静态基准"的工作。
- **对工程化 LLM 代理部署具有参考价值**：git 化的 harness tree 与任务板、临时揭示门、结构化 HITL 挂钩都是可直接复用的工程模式；对做长程预测市场、持续 CTF、跨语言事件预测等项目的团队尤其值得借鉴。

## Questions for Further Exploration

- Can the L_evo / L_adapt decomposition be turned into a directly estimated quantity (e.g., via an oracle harness proxy or counterfactual replay) rather than diagnosed only through bottleneck / ablation proxies?
- How does Adaptive Auto-Harness behave on **multi-modal** or **tool-rich** open streams (e.g., web-shopping, code migration, long-horizon robotics) where branch-relevant skills span modalities rather than topical regimes?
- Does the harness-tree + git routing pattern transfer to **federated** or **privacy-preserving** settings where each branch must be sandboxed, or where routing decisions must be auditable post-hoc?
- What is the right **trigger** for opening a new branch vs. extending an existing one? The current EGL threshold (0.05 over 3 cycles) is empirical; can it be derived from the L_evo trajectory itself?
- How sensitive is the system to **solver / evolver model substitution** — e.g., Sonnet 4.6 → open-weight 70B-class models, or to using the same model for both solver and evolver to remove one source of variance?
- Can the **HITL hooks** be generalized into a "human-agent co-evolution" protocol where the human's role is not just credential / source provision but also **value alignment** under distribution shift?
- How does the system handle **adversarial drift** — i.e., a stream where the task distribution is intentionally perturbed to break the harness tree's branch routing?
