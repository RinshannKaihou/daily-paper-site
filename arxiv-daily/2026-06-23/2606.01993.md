> **一句话总结：** 提出 MMG2Skill-Bench 与 MMG2Skill 框架，将网页上多模态的"人类教程"自动编译为可编辑的 SKILL.md 技能，并通过闭环轨迹诊断在不使用 benchmark 分数的前提下持续修订技能，使 VLM 智能体在 GUI、游戏与策略任务上实现 +12.8 至 +25.3 pp 的稳定提升。

> **TL;DR:** MMG2Skill formalizes **guide-to-skill learning** — converting in-the-wild multimodal web tutorials into editable SKILL.md procedures and refining them from agent-visible trajectory feedback, without ever observing benchmark scores. Paired with the new MMG2Skill-Bench (130 success-inferable tasks spanning OSWorld GUI, OpenHA Minecraft and RLCard Doudizhu/Mahjong), the closed-loop framework wins in all 18 model–domain cells across six VLM backbones, saving 25–53% of attempts via analyzer-based early stopping. Raw guide prompting can actually *hurt*, and the results show that what matters is structured construction plus trajectory-driven revision, not access to guides.

## 核心贡献

- **MMG2Skill-Bench**：首个面向"将多模态网络教程转化为可执行技能"的评测基准，包含 130 个可从轨迹推断成败的交互任务，覆盖 GUI（OSWorld）、开放世界游戏（OpenHA Minecraft）、策略卡牌（RLCard 斗地主/麻将）三大领域；每个任务都配有一份任务相关的公开网页教程（HTML + 图像），但不含答案痕迹。
- **MMG2Skill 闭环框架**：把教程统一成可编辑的 SKILL.md（包含可复用步骤 `u`、适用条件 `c`、期望状态线索 `v`、恢复知识 `q`），冻结 VLM 策略，仅对技能集 `S ∈ Ω(G)` 做优化；执行 → 分析器 → 修订器循环，使用轨迹级根因反馈更新技能，而不消费 benchmark 评分。
- **三大实证发现**：(1) 公开教程中含有的流程知识是 vanilla 智能体无法稳定恢复的；(2) 单纯喂入原始教程甚至会降低性能，必须先结构化为技能再修订；(3) 修订收益非单调，需配合基于分析器的早停策略以避免后期退化。
- **可部署的早停机制**：分析器对 `likely_success` 的判定在 GUI/Game 上召回率 >95%，早停相比跑满 N=5 在 Game 上多 18 pp，同时节省 25–53% 步数与 23–41% VLM API 调用。

## Key Contributions

- **MMG2Skill-Bench (130 tasks, 3 domains)**: First benchmark pairing success-inferable interactive tasks with in-the-wild multimodal guides; spans GUI control, open-ended game play, and strategic card play; explicitly excludes benchmarks where outcome depends on private information not visible to the agent (e.g., No-Limit Hold'em goes to a separate diagnostic).
- **MMG2Skill framework**: A closed-loop pipeline with three operators — `ConstructSkills(G, I)`, `Analyze(I, τ)`, and `Refine(G, I, S, ρ₁:k)` — operating over an editable SKILL.md representation while keeping the VLM policy π_θ fixed. The framework's update variable is the skill set, not the policy.
- **Score-free revision**: Skills are updated purely from agent-visible trajectory diagnoses (evidence `eₖ` + self-judged outcome `rₖ`); benchmark scores are reserved for offline evaluation only, making the loop deployable without privileged feedback.
- **Empirical wins across backbones**: Positive gain in all 18 model–domain cells, macro-average +12.8 to +25.3 pp over vanilla across Claude-Opus-4.6, GPT-5.5, Claude-Sonnet-4.6, Kimi-K2.6, Gemini-3.1-Pro-Preview, Qwen3.6-Plus; largest single-cell jump +33.33 pp (Gemini on Game).
- **Ablations & diagnostics**: Raw-guide prompting can hurt (−1.67 pp on Game); "w/o revision" captures most of GUI gain but only ~10% of Game/Strategy gain; analyzer calibration table (P/R, attempts and calls saved); residual-failure attribution broken down by agent-loop / analyzer / refiner.

## Methodology

**Problem formulation.** Given a task instruction `I` and a multimodal guide `G`, the agent rolls out a trajectory `τ = (o₁, a₁, …, o_L, a_L)` while conditioning on a bounded history `hₜ` of the most recent `W−1` observation-action turns. The evaluation target is `S* ∈ arg max_S E_{τ∼p_θ(τ|I,S)}[s(τ)]` over skill sets `S ∈ Ω(G)` induced from the guide; crucially, the construction / analysis / refinement operators never observe `s(τ)`, only a trajectory-grounded diagnosis `ρₖ`.

**Stage 1 — Skill construction.** A VLM skill extractor normalizes `G` into a set of SKILL.md tuples `zᵢ = (uᵢ, cᵢ, vᵢ, qᵢ)` representing reusable procedure, applicability conditions, expected-state cues, and recovery knowledge. The tuple is a conceptual organization rather than a rigid schema — in implementation, fields are realized as procedural text, state descriptions, and referenced guide images.

**Stage 2 — Skill-conditioned execution.** At attempt `k`, the current skill set `Sₖ` is injected into the agent context and remains alongside the rolling history `hₜ` throughout the rollout. Each action is sampled as `aₜ ∼ π_θ(aₜ | hₜ, I, Sₖ)`, leaving the domain's action interface unchanged so the same conditioning interface works across GUI / Game / Strategy.

**Stage 3 — Analyzer + Refiner (closed loop).** After each rollout, an analyzer produces `ρₖ = (eₖ, rₖ)` where `eₖ` is trajectory-grounded evidence (what worked, what failed, where behavior diverged) and `rₖ` is a self-judged `likely_success` signal. The refiner then applies `Sₖ₊₁ = Refine(G, I, Sₖ, ρ₁:k)`, with the diagnosis chain accumulated without truncation and the original guide retained as input so omitted procedural details can be recovered. The loop terminates on `likely_success` or at attempt budget `N=5`.

**Guide corpus.** For GUI, product documentation and how-to articles for desktop apps; for Game, wiki pages and walkthrough guides for Minecraft; for Strategy, public rule descriptions and beginner strategy material. Each page pairs HTML content with image resources, preserving step order, visual examples, and interface/game-state cues. The paper claims no leakage: guides do not contain benchmark solution traces, gold actions, hidden labels, or environment-specific trajectories.

**Evaluation protocol.** Six VLM backbones are run with attempt budget N=5 on the 130 success-inferable tasks; main tables report the analyzer-selected early-stop view (return the first attempt whose analyzer says `likely_success`, fall back to N if none triggers). Domains: GUI uses OSWorld (Xie et al., 2024); Game uses OpenHA tasks in MineStudio (Cai et al., 2024); Strategy uses RLCard's Doudizhu and Mahjong (Zha et al., 2021). Domain-native scoring produces per-task scores in [0,1].

## Results & Highlights

- **Table 1 (main results)**: Every one of the 18 model–domain cells shows a positive gain over vanilla. Highlights: Claude-Opus-4.6 56.85→69.64 avg; GPT-5.5 54.08→67.41; Kimi-K2.6 44.36→58.81; Qwen3.6-Plus 26.67→51.94. Largest single-cell gain +33.33 pp (Gemini on Game, from 33.33 to 66.67).
- **Table 2 (ablation)**: Raw-guide injection is essentially flat on GUI but degrades Game/Strategy, with the largest loss −1.67 pp on Game; "w/o revision" matches or exceeds vanilla on all three domains, confirming that structured extraction provides a safer procedural prior; the full closed loop is needed for Game and Strategy (revision contributes >90% of total gain there, +22.22 pp on Game).
- **Figure 4 (revision dynamics)**: GPT-5.5 starts strong on GUI/Game with most remaining gains on Strategy; Qwen3.6-Plus starts much lower and accrues most gain through revision on Game/Strategy. Revision gain curves are non-monotonic — motivating calibrated stopping.
- **Figure 5 + Table 3 (early-stop deployment)**: Early-stop equals full-run on GUI (55.7 vs 55.7) but is substantially higher on Game (66.1 vs 47.8) and Strategy (61.7 vs 59.3 at K=5). Analyzer precision 74.2 / 87.0 / 84.6% on GUI/Game/Strategy; recall 98.5 / 95.8 / 70.8%; mean stopping attempt k̄ = 2.35 / 2.54 / 3.73; attempts saved 52.9 / 49.2 / 25.4%; VLM API call savings 41.8 / 37.5 / 23.6%.
- **Figure 6 (residual failure attribution)**: Error-tag distribution shows GUI failures are spread across grounding/diagnosis/skill information (coupled); Game failures are execution-heavy (grounding errors + budget exhaustion dominate); Strategy failures hinge on converting revised skills into context-dependent decision rules (wrong approach, skill-not-followed, suboptimal strategy loops).
- **Reflexion-style comparison (Appendix E.4)**: Accumulating root causes as ephemeral in-context history still trails MMG2Skill at N=5 on Kimi-K2.6 and Qwen3.6-Plus for GUI/Game, supporting the claim that persistent SKILL.md edits matter more than in-context feedback.
- **Gains are not from longer rollouts**: GUI and Game show *fewer* displayed-attempt steps on average under MMG2Skill, suggesting reduced exploratory detours rather than lengthier context-driven rollouts (Appendix F.1).

## Critical Assessment

- **Novelty**: The framing of *guide-to-skill learning* as an explicit problem, and the closed-loop pipeline that edits skill files (not the policy) from trajectory-level root-cause feedback without consuming benchmark scores, is a genuine reframing — most prior skill-learning work initializes from model rollouts, expert demonstrations, or training data. The pairing with a benchmark where each task is *intentionally* given public multimodal guides distinguishes it from CL-bench (curated text) and SkillsBench (verifier-based, expert/model-generated skills).
- **Rigor**: Strong on this dimension. Six VLM backbones × three domains × paired baselines (vanilla + Raw Guide + "w/o revision" + full closed loop), with ablation tables, revision-dynamics curves, analyzer calibration metrics (P/R, attempts/calls saved), and a fine-grained residual-failure error taxonomy. The matched-budget comparison against a Reflexion-style alternative is a useful control.
- **Reproducibility**: Public code repository, fixed prompts, fixed model versions, controlled comparisons, and explicit budget protocols all support reproducibility. The honest disclosure of evaluation cost (multimodal API calls for execution + iterative revision + analyzer + refiner) and the use of sandboxed benchmarks (no real accounts / production services) further strengthen this.
- **Significance**: The +12.8 to +25.3 pp range is substantial and consistent, and the "raw guides can hurt" finding is a non-obvious, actionable contribution for the in-context-learning / agent community. The result that revision matters most on Game/Strategy (where guides leave inventory/strategy under-specified) is a useful design rule: static extraction is enough when guides map cleanly to command-like operations, but interactive domains need runtime-grounded revision.
- **Red flags / caveats**:
  1. **Analyzer self-judgment is the loop's linchpin.** On Strategy, analyzer recall is only 70.79% (vs. >95% on GUI/Game), and the GUI essay-submission case (Appendix G.2) shows that "surface completion cues" can trigger false-positive `likely_success` even when the oracle judge assigns score 0. For high-stakes deployment, this means the loop's stopping signal is not perfectly calibrated.
  2. **No-Limit Hold'em is carved out as a diagnostic** because payoff depends on opponent private cards; this is a clean handling, but it also implicitly limits the framework's deployability to tasks where success is trajectory-inferable.
  3. **Guide discovery / retrieval is out of scope.** The paper treats `G` as a static, pre-selected input; in deployment, an agent must search, filter outdated/misaligned guides, and decide when no guide is reliable — all left as a separate problem.
  4. **Strategy domain needs further validation.** Despite improvements, residual Strategy failures are dominated by "wrong approach" / "skill-not-followed" / "over-tightened rules" / "broke winning policy", suggesting that converting strategy guides into context-dependent decision rules remains hard even with revision.
  5. **Macro-average gains are dominated by low-baseline backbones** (Qwen3.6-Plus: 26.67→51.94 ≈ +25 pp; Gemini: 35.82→54.17 ≈ +18 pp). Stronger backbones like Claude-Opus-4.6 still gain but only +12.8 pp average, which is more modest.

## Strengths

- Clean problem formulation (Eq. 1) that separates the evaluation target from the available feedback signal.
- Algorithm 1 is small and legible — one can read off the loop structure in a minute.
- The "raw guide vs. constructed skill" ablation is the paper's most valuable empirical contribution; it directly challenges the assumption that "more context = better."
- Cross-domain generality (GUI / Game / Strategy) with a unified interface — the same skill-conditioning interface works across all three.
- Analyzer calibration is reported numerically (precision/recall, savings in attempts *and* VLM calls), not just as a deployment suggestion.
- Honest failure attribution (Figure 6) by pipeline component, with the explicit caveat that it should be read as a residual-failure diagnostic rather than a stage ranking.

## Limitations

- **Guide discovery is external.** The framework assumes a task-relevant guide is supplied; it does not study retrieval, source filtering, staleness detection, or the "no reliable guide" case.
- **Strategy analyzer recall is weaker (70.79%)**, and the GUI essay-submission example shows false positives driven by surface cues. The early-stop policy is well-calibrated only on success-inferable tasks.
- **Evaluation is costly.** Multimodal API calls for execution + iterative revision + analyzer + refiner per task per attempt make continuous re-evaluation of new SOTA models expensive; the paper commits to fixed model versions and budgets.
- **Interactive rollout throughput is slower than static text evaluation**, and is further constrained by commercial API rate limits, especially with multiple revision attempts.
- **Skill schema is conceptual, not rigid.** SKILL.md fields `(u, c, v, q)` are realized as procedural text, state descriptions, and referenced images — there is no formal type system enforcing them, so the framework's quality depends on the underlying VLM's ability to populate these fields coherently.
- **No comparison against learning-from-rollouts baselines** (e.g., ExpeL, Voyager-style skill libraries) on the same benchmark — the comparison set is intentionally narrow (vanilla + Raw Guide + "w/o revision" + Reflexion-style in-context), which makes it easier to claim gains but harder to position the work against the broader self-improving-agent literature.
- **Ethical surface only briefly addressed.** The paper acknowledges potential misuse for automating unintended actions in real software but does not propose concrete safeguards beyond "external outcome verification, permission controls, and human oversight."

## Relevance

该工作直接命中本项目的核心关注方向之一：**vision-language agent 的程序性知识获取与自我改进**。它把"网络教程"作为弱监督信号、通过可编辑技能文件而非参数更新来实现持续改进的范式，对任何构建长时程、可自我演化 agent 系统的研究都有参考价值。

## Questions for Further Exploration

- Can the analyzer–refiner loop be replaced (or bootstrapped) by a learned verifier, so that deployment no longer depends on a single VLM's self-judgment?
- How does MMG2Skill degrade when guides are stale, contradictory, or partially relevant? A controlled study varying guide quality would isolate the framework's robustness.
- Could SKILL.md be replaced by a typed/structured skill representation (e.g., a small DSL with preconditions, effects, and recovery operators) to give revision symbolic grounding instead of free-form text edits?
- Does the framework extend to tasks where the "skill" is *not* a fixed action recipe but a learned policy prior (e.g., negotiation, mixed-strategy games)? The residual-failure analysis hints this is exactly where the current approach struggles.
- How would an upstream retrieval / source-filtering module change the early-stop calibration and the Strategy recall gap in particular?
- Is there a compute-efficient regime where revision runs on a smaller model than the executor, exploiting the asymmetry between "edit a SKILL.md" and "roll out a trajectory"?
- Does the persistent-skill-edit advantage over in-context feedback (Reflexion-style) generalize beyond GUI/Game to domains with very long context windows where history truncation is not the binding constraint?
