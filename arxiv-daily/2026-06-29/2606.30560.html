01

Methodology

TraceLab models a coding-agent run as a three-level hierarchy: a session persists accumulated context (files read or modified, tool outputs, conversation) and holds one or more requests; each request is resolved by an agentic loop of steps, where a step = one LLM invocation plus the tool calls it emits. A pipeline extracts timestamped events from Claude Code and Codex logs, normalizes both providers into a unified step-level schema, and sanitizes identifiers — only character counts, token usage, timestamps, and pseudonymous user IDs survive.

Input is decomposed as I = P + A: prefix tokens P are served from the prefix cache, append tokens A are fresh input (user messages, tool results, cache misses). Cost is derived from published API pricing. Spanning 4,265 sessions, 357K steps, 433K tool calls from 43 developers over ~8 months, this schema yields the first provider-comparable coding-agent serving trace.

Five serving opportunities

01
Denser, fused tool calls
Low-overhead tool runtime & approval paths to cut tail latency.
02
Append-aware prefill routing
Route on append length with adaptive kernels & parallelism.
03
Semantic tool-latency prediction
Predict tool duration from request features for smarter KV eviction.
04
Sparse attention for long prefixes
Target the 100k+ cached history that dominates prefill.
05
Human-paced KV retention
Cheaper storage + eviction/prefetching around idle gaps.

Strengths

assets
  • First open, real, cross-provider, multi-model coding-agent trace built explicitly for serving analysis.
  • Quantified, not anecdotal — full p50/p90/p99 distributions across session, request, and step granularity.
  • Clean methodology — explicit token decomposition, defined compaction criteria, output-attribution invariants, pricing-grounded cost model.
  • Actionable mapping — every finding lands on a concrete serving optimization.
  • Full open release — data + pipeline + analysis code; top systems-group authorship.

Limitations

caveats
  • Self-sourced, single institution — generalizability to other orgs, workflows, or agent domains is uncertain.
  • Externally visible logs only — backend mechanisms (scheduling, cache, routing) are inferred, not measured; the cached-vs-resend hypothesis is unconfirmed by provider telemetry.
  • Optimizations unvalidated — none is implemented or evaluated in a serving system; gains are prospective.
  • Narrow coverage — only Claude Code and Codex; no Gemini CLI, Cursor, or local agents; reasoning text is encrypted.
  • Sanitized release drops raw text, limiting content-level re-analysis.
04

Questions for further exploration

  1. How much latency/cost do the five optimizations actually deliver once implemented in engines like vLLM, SGLang, or Mooncake — and which yields the largest payoff?
  2. How generalizable is this single-org trace to enterprise or open-source agent populations, and what reweighting would make it representative?
  3. Do cache-hit patterns, tool-call tails, and context-growth dynamics differ across Cursor, Gemini CLI, local models, or broader computer-use agents?
  4. Can tool-call latency be predicted accurately from semantic request features + recent history, beating tool-name-only heuristics for KV eviction?
  5. Given 3.8× redundant prefill from human-paced gaps, where exactly does the KV storage-vs-eviction knee sit in practice?
  6. Can a serving system adaptively choose output-cache vs output-resend policy per step, and what are the gains?
