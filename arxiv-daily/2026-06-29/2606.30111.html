01

Methodology, distilled

§Methodology

AAS splits each system into an Executor — a typed node-and-wire graph of perception / memory / planning / action modules running in a simulator — and an external Optimizer, an LLM coding agent that edits the executor across development iterations but never acts in the environment.

Because embodied agents have no task-agnostic primitive palette, every search is method-seeded: it starts from a published executor (MapGPT, SmartWay, ExploreEQA, VoxPoser) and explores graph-level edits around it.

Two substrates carry the work. AgentCanvas is the executor substrate — typed JSON graphs backed by Python modules, edited via structured patches type-checked before expensive rollouts, every rollout instrumented into episode-level logs. A shared coding-agent harness (one Claude Code session, Claude Opus 4.7 orchestrator) runs all three variants so only proposer logic + memory differ — performance gaps reflect search policy, not tooling.

THINK→ CRITIC→ EXPERIMENT→ DISTILL· REFLECT (on stall)

The new KDLoop tracks intervention-axis coverage to escape local basins; ADAS and AFlow are faithful ports of prior text-domain AAS.

Strengths & Limitations

§Critical Assessment

Strengths

  • Clean Executor↔Optimizer split; shared harness fairly isolates search-policy differences.
  • Method-seeded framing tests real published executors — matches how systems are actually engineered.
  • Episode-level instrumentation is a genuine capability beyond text-domain AAS.
  • Methodological honesty: 3-pass reruns, explicit leak rejection, controlled-fault probe, "directional not certified" caveats.
  • Interpretable surviving edits (heading-band gate, stop-after-3→5) — not opaque reshuffling.

Limitations

  • Scope: AgentCanvas covers workflow-shaped agents only — not systems built dynamically at deployment.
  • Noisy rollouts obscure graph-edit effects; best single-pass score ≠ architecture quality; local basins persist.
  • Credit assignment from logs only partially emerges — SR alone is insufficient.
  • Scale: 4 executors × 3 optimizers, single backbone — generalization breadth is limited.
  • Cost: frontier orchestrator (Claude Opus 4.7, 1M ctx) + 4 simulators raise access barriers.
  • KDLoop is not uniformly better — its coverage bias under-exploits productive basins (e.g. SmartWay).

Open questions for further exploration

§Questions
Q1 · TRANSFER

Do discovered edits transfer across backbones (e.g. open VLA) and across unseen scenes/objects outside the search simulators?

Q2 · COST

Can a learned surrogate replace multi-episode rollouts to cut evaluation cost without masking real signal?

Q3 · LEAKS

Is leak detection (ground-truth wiring) automatable as a first-class attribution check, not KDLoop's late log inspection?

Q4 · STOPPING

What search-space stopping predicate lets ADAS/AFlow exit local basins without KDLoop's under-exploitation trade-off?
