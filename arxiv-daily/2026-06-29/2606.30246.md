> **一句话总结：** Clarus 提出了一套面向网络级科学协作的多智能体基础设施，将科研从"以代码为中心的执行回路"重构为开放、可审计、可溯源、资源感知的多阶段协作流程；通过最小化的"项目-智能体-资源"对象模型与四层架构（研究应用、数字协作、物理基底、物理世界），并把规划、路由、谈判、审计、影响更新做成可插拔策略，在一次受控的论文生成案例中用 8 个智能体、30 个任务、589 个可追溯事件和最终署名顺序匹配的贡献结算，验证了可追溯、可审查、可累积的协作闭环。

> **TL;DR:** Clarus reframes autonomous science from code-centered execution loops into an open, auditable, attributable, and resource-aware multi-phase *collaboration process*. It builds a minimal project-agent-resource object model plus a four-layer architecture (Research Application, Digital Collaboration, Physical Substrate, Physical World), exposes orchestration/negotiation/audit/impact-update as pluggable strategies, and validates a closed collaboration loop through a controlled "MirrorEval" paper-generation case study (8 agents, 30 DAG tasks, 589 trace events, credit-aligned author ordering).

## 核心贡献

- **范式转变**：首次将"自主科研"重新定义为面向网络规模科学协作的*基础设施问题*，而非单智能体助手或封闭流水线，强调科研反馈稀疏、目标开放、认知与资源分布三大差异。
- **最小化对象模型**：用三类原语节点（项目 P、智能体 A、资源 R）和最小边集（项目-智能体、智能体-资源）刻画 web 规模科研网络；智能体可以是 AI、人、团队、实验室或组织级参与者，避免了把人类/组织/任务都设为独立原语。
- **四层架构**：研究应用层（项目生命周期/DAG）、数字协作层（身份、发现、协作、效用）、物理基底层（可信资源访问：资源卡/租约/证据包）、物理世界层（真实设备与外部环境），实现职责分离与"信任反馈自下而上、任务流自上而下"。
- **可插拔策略机制**：将规划（模板/LLM DAG）、路由（能力/LLM 匹配）、谈判（投票/轮询/仲裁）、审计（规则/LLM/交叉验证）、影响更新（累加/加权/AgentRank）、物理访问（租约+证据包）抽象为可替换策略，并对每类给出形式化定义。
- **可审计/可累积闭环**：提出"研究即证据流"，用痕迹（trace）、审计记录（α）、信用记录（κ）、证据包（ϵ）支撑跨项目的影响更新（声誉 ρ 与智力 ι 双维度），将署名顺序从静态声明变为过程级记录。
- **原型案例验证**：通过 MirrorEval 基准论文生成案例，展示端到端流程，并在实验阶段触发审计失败→自动检查点→DAG 重规划（v1→v2）的闭环。

## Key Contributions

- **Paradigm shift**: Re-formulates autonomous research as a *web-scale scientific collaboration infrastructure* problem (not an assistant task or closed workflow), driven by the observation that research differs from code execution in sparse/delayed feedback, open-ended objectives, and distributed cognition/resources.
- **Minimal object model**: Models the research network as a typed graph with three primitive node classes (Projects, Agents, Resources) and a deliberately minimal edge set, where agents may be AI systems, humans, teams, labs, or organization-backed participants; ownership/control are expressed via agent-resource edges rather than extra primitive types.
- **Four-layer architecture**: Research Application (goal → phase/DAG lifecycle), Digital Collaboration (Identity, Discovery, Collaboration, Utility), Physical Substrate (trusted access via resource cards / revocable leases / evidence packages, keeping owners in control), and Physical World (real instruments/data/services), with structured objects+evidence exchanged between layers.
- **Pluggable strategies**: Formalizes Orchestration (planning/routing/replanning), Negotiation (centralized/decentralized/hybrid credit settlement), Audit (rule-based / LLM semantic / cross-validation), Impact Update (linear accumulation / penalty-weighted / graph-based AgentRank-style propagation), and a Physical-Access lease-and-evidence protocol — each as swappable functions with equations.
- **Evidence-bearing collaboration process**: Introduces the research process as a data flow of traces (τ), audit records (α), credit records (κ), and evidence packages (ϵ), enabling provenance, reproducibility, credit attribution, and long-term reputation/intelligence impact updates across projects.
- **Prototype validation**: Demonstrates an end-to-end "Idea-to-Paper" loop in the MirrorEval case study, including an audit-triggered checkpoint with automatic DAG repair (v1→v2).

## Methodology

**Problem framing.** The paper argues existing autonomous research systems are stuck in "code-centered execution loops" (write code → run → diagnose → optimize), which suits software but not science because (1) research feedback is sparse, costly, delayed, ambiguous; (2) objectives are open-ended (the problem/hypothesis/eval criteria themselves change); (3) cognition and resources are distributed across private data, tacit judgment, labs, compute. This motivates four system-level gaps: assistant → network participant; workflow automation → auditable collaboration; fixed pipelines → open discovery/negotiation/coordination; digital-only → physically grounded research. Unaddressed, open research networks risk becoming a "dark forest" of opaque, unverifiable participants.

**Object model (Section 2).** The web-scale research network is a typed graph G=(V,E), V=P∪A∪R, with E=EPA∪EAR (project↔agent, agent↔resource). Primitives:
- Project p=(idp, gp, Sp): goal + evolving state (phases, subtasks, artifacts, traces, strategies Πp).
- Agent a=(ida, Ca, Pa, za): capability set, permissions/governance, and impact state za=(ρa reputation, ιa intelligence).
- Resource r=(idr, typer, Cr, sr, Pr): capability, availability, access policy — covers software, private data, instruments, robotic labs.

**Research process as workflow.** A six-phase "Idea-to-Paper" pipeline: Explore → Ground → Design → Realize → Experiment → Compose. Each phase decomposes into a DAG of subtasks; each subtask emits a trace τv linking executor, artifact, provenance, audit αv, credit κv, evidence ϵv. Trust/audit/credit are treated as first-class network requirements (trust in identity/capability/permission/process/artifact/credit), and physical resources require lease + evidence-package records.

**Architecture (Section 3).** Four layers with separated responsibility:
- *Research Application*: team formation (open discovery via Holos), project container, phase DAGs, per-subtask artifact+evidence submission, audit/credit checkpoints, impact feedback.
- *Digital Collaboration*: Identity (DID, verifiable credentials, agent cards), Discovery (capability matching), Collaboration (group chat, hash-chain notarization, project logs), Utility (tools, knowledge bases, academic search).
- *Physical Substrate*: trusted access layer wrapping real devices/data into schedulable/authorizable/auditable objects via resource cards, revocable time-bounded leases, pre-execution safety/policy checks, and post-execution evidence packages — explicitly *not* real hardware control yet.
- *Physical World*: real devices/services/environments under owner control.

**Pluggable strategies (Section 4).** Each variable decision is a swappable strategy (Table 1):
- Orchestration: πplan (template/LLM DAG), πroute (capability/LLM matching + load balancing), πreplan (audit/triggered DAG repair).
- Negotiation: πneg producing prior credit κprior then settling κfinal after audit; centralized (arbitrator), decentralized (P2P/round-robin), or hybrid.
- Audit: πaudit → (quality score q, issue list I, control signal η = pass/warning/critical/human/replan); rule-based, LLM semantic, or cross-validation.
- Impact update: πimp updating (ρ, ι); linear accumulation, penalty-weighted, or graph-based signed random walk (AgentRank-style).
- Physical access: πlease (revocable authorization) + πevid (post-run evidence package).

## Results & Highlights

Validation is a *controlled, qualitative prototype run* (the paper is explicit this is feasibility evidence, not a benchmark of research quality):

- **End-to-end paper generation (MirrorEval case)**: Goal = build the MirrorEval benchmark (Personality Contract Framework) comparing RawLLM, LangGraph, OpenAI Agents SDK, Hermes, OpenCLAW, assembled into an "ACML 2026" paper. Clarus decomposed it into **6 phases**, **30 DAG tasks, 29 DAG edges**, involving **8 agents**, registering **128 workspace artifacts** (notes, experiment files, claims, figures, self-reports).
- **Audit-triggered replanning**: In the Experiment phase, DAG v1's adapter-provenance audit failed (score 0) due to insufficient local evidence; with "Auto checkpoint" enabled, Clarus generated a checkpoint + replan diff and produced repaired DAG v2 — demonstrating the closed execution-monitoring/audit/DAG-repair loop.
- **Credit attribution**: **19 credit records** generated; final author-order in the generated paper matched the process-derived credit settlement (e.g., Zihan Guo 25%, Bo Huang 23.3%, Shuai Shao 14.2%, etc.), turning authorship into an evidence-backed, process-level record rather than a static declaration.
- **Provenance tracking**: **589 trace events** across the run (Credit 181/30.7%, Artifact 128/21.7%, Task 97/16.5%, Other 72/12.2%, Audit 52/8.8%, Phase 45/7.6%, Routing 14/2.4%); a notary chain aggregates attested events per phase for verification.
- **Artifact taxonomy**: four categories — Logic (claims/plans/manuscripts), Physical (code/configs), Trace (DAGs/provenance/decisions), Evidence (outputs/metrics/audit/hashes) — so the *process* of producing outputs is auditable.
- The Physical Substrate is shown only as an *early mock interface* (resource card/lease/evidence-package objects) — not a real resource-scheduling experiment.

## Authors & Credibility

Large multi-institutional team (18 authors) spanning Sun Yat-sen University, Shanghai Jiao Tong University (corresponding author Weinan Zhang, a well-known ML/recommendation/agent-systems researcher), Tongji University, Jilin University, Harbin Institute of Technology, Shanghai Innovation Institute, and Shanghai AI Laboratory. The work ties into the **Holos** agentic-web infrastructure (holosai.io / clarus.holosai.io), suggesting an active, resourced research program. The breadth and formality of the design are consistent with a serious systems/architecture contribution, though the empirical evidence is still prototype-level.

## Critical Assessment

- **Novelty — High (conceptual), Low (empirical).** The core contribution is a *systems/architecture* reframe: treating autonomous science as collaboration infrastructure with a typed project-agent-resource ontology and a four-layer digital↔physical stack. This positioning is genuinely distinct from the dominant "single closed agent workflow" literature (e.g., AI-Scientist-style systems). The individual mechanisms (DAG planning, LLM routing, peer voting, reputation propagation) are individually well-known; novelty is in their *composition* and the auditable/attribution framing.
- **Rigor — Moderate.** The object model, typed relations, and strategies are given clean formal definitions (Eqs. 1–7). However, there is **no quantitative benchmark, no baseline comparison, and no statistical evaluation**. The case study reports descriptive counts (30 tasks, 589 events, 19 credit records) but not accuracy/quality of the generated research or robustness of credit allocation against ground truth. The paper explicitly disclaims that it does *not* prove the generated research has scientific novelty or that Clarus outperforms humans/single-agent systems.
- **Reproducibility — Low-to-Moderate.** A prototype is described (project room UI, artifact taxonomy, notary chain) and the system is "available at clarus.holosai.io," but the paper provides no code release statement, no agent/LLM configuration details, no audit prompts, and no reproducible experimental harness. The Physical Substrate is explicitly a mock. Independent reproduction of the case study would be difficult from the paper alone.
- **Significance — High ambition, unproven impact.** If the vision materializes, it addresses real gaps (attribution, provenance, physical-resource access) that current agent frameworks ignore, and frames a credible "research network" research agenda. But the gap between the ambitious "web-scale" framing and a single illustrative run is large; the paper itself redefines "web-scale" as *stress conditions to design for*, not current deployment size — an honest but notable caveat.
- **Red flags.** (1) Over-claiming risk in the title ("web-scale scientific collaboration") vs. a single controlled, self-selected case study whose benchmark (MirrorEval) and venue (ACML 2026) appear constructed *by the system itself* for demonstration. (2) No external validation that the generated paper is scientifically sound. (3) Several cited references are future-dated (2025–2026) — standard for a 2026 preprint but worth noting that the positioning leans heavily on a fast-moving agent literature. (4) The Physical Substrate — arguably the most distinctive layer — is not yet implemented, so the digital↔physical claim is currently aspirational.

## Strengths

- Conceptually reframes autonomous science around *collaboration infrastructure* with a clean, minimal, well-formalized ontology.
- Treats trust, audit, credit attribution, and physical-resource grounding as first-class design requirements rather than afterthoughts — a genuinely under-served area.
- Clean separation of concerns via the four-layer architecture; pluggable strategies make the system a credible *testbed* for comparing orchestration/audit/credit/impact strategies.
- Honest framing of "web-scale" as design stress conditions and explicit acknowledgement that the prototype is feasibility evidence, not a quality benchmark.
- Strong provenance/attribution story: 589 trace events, notary chain, credit-aligned author ordering, artifact taxonomy.

## Limitations

- **No quantitative evaluation**: no baselines, no success-rate/quality metrics, no ground-truth credit validation; the single case study is illustrative.
- **Physical Substrate not implemented**: the layer distinguishing Clarus from digital-only systems is only a mock interface.
- **Self-demonstration**: the generated paper and benchmark are produced by the system itself, with no independent scientific validation.
- **"Web-scale" unproven**: no federation, cross-organization pilot, abuse/collusion/Sybil resistance, or large-scale scalability tests — the authors flag these as future work.
- **Reproducibility gaps**: minimal implementation/LLM/prompt details; code availability not clearly stated in the body.
- **Security/adversarial**: only partial handling of collusion, prompt injection, audit manipulation, reputation gaming — acknowledged as open.

## Relevance

对关注智能体协作、科研自动化与系统基础设施的研究者具有高参考价值：Clarus 把多智能体科研从"封闭流水线"提升为可审计、可溯源、资源感知的开放协作网络，其"项目-智能体-资源"对象模型、四层架构与可插拔审计/信用/影响更新机制，可作为构建与评估自主科研系统、Agentic Web 及可信科研网络的系统级设计蓝图。

## Questions for Further Exploration

- What objective metric could validate that process-derived credit attribution matches *true* contribution, and how would it compare against human-authored authorship?
- How does the choice of pluggable strategy (e.g., LLM-based vs rule-based audit; centralized vs hybrid negotiation) quantitatively affect output quality and credit stability?
- How would the proposed Physical Substrate protocol actually integrate wet-lab/robotic resources, and can leases+evidence packages preserve owner control while enabling reproducibility?
- What are the empirical failure modes under adversarial conditions (Sybil, collusion, audit manipulation, prompt injection), and what governance mechanisms mitigate them?
- Can Clarus serve as a standardized *benchmark testbed* for comparing multi-agent research frameworks on attribution accuracy, provenance completeness, and cross-project impact?
