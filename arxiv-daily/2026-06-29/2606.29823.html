01 Methodology

search over experience graphs = a database access pattern

Trellis reframes long-horizon agent search as a new database workload. A two-loop architecture does the split: the inner loop runs a stateless, skill-driven agent session — LLM generates a candidate, executes/evaluates it in an isolated sandbox, then persists a reward-bearing node and terminates, holding no state. The outer loop picks which node to expand next through a single task-agnostic query interface.

Inner loop · the agent

consume task + context → LLM generates artifact → sandbox execute/evaluate → persist one node → terminate. Stateless by construction.

Outer loop · the search

reads fitness scores, selects frontiers via one data interface. one-shot / greedy / MCTS / evolutionary all write the same exploration_nodes table.

The experience graph — artifacts, tool outputs, rewards, sibling comparisons, causal lineage — lives as governed relational state on a four-level hierarchy: Tasks → Sessions → Nodes → Prompt histories (plus embeddings and object-store references). Axiom/Velox plan SQL + Cypher + vector retrieval into one physical plan in a single round-trip. Nodes are insert-only; mutated fields (visit counts, UCB scores, island ids) are versioned by logical step number, enabling AS-OF reconstruction (time travel).

Crucially, training data becomes a query, not a collection: SFT = root-to-leaf paths read as-of each step; DPO = sibling fitness-pattern matches; GRPO = cohort advantages via SQL window functions, collapsing multi-turn rollout into single-step expansions. Crash recovery, horizontal scaling, and the SFT/DPO/GRPO flywheel fall out of being a database.

03 Strengths & Limitations

side-by-side · the tradeoffs, plainly

▲ Strengths

  • Unifies many agent pain points — memory, reuse, crash recovery, concurrent search, RL-data generation, governance — under one coherent, well-motivated abstraction.
  • Credible cross-disciplinary author team straddling DB systems (Abadi), ML (Fergus, Goyal, Wu), and production deployment.
  • Honestly reports the central tradeoff (reuse accelerates but anchors) rather than overclaiming universal benefit.
  • Backed by a real production system, not toy benchmarks — plus a second deployment (silicon validation) showing domain-agnosticism.
  • Concrete, actionable database research agenda fit for the data-management community.

▽ Limitations

  • Empirical grounding is essentially one production case study, N=3 sessions/config; generalization to code gen / science / security is asserted, not measured.
  • No head-to-head comparison against existing agent-memory or retrieval baselines (e.g. generic vector-RAG).
  • Storage and query-latency overhead of time-travel change logs and multi-modal planning is not quantified.
  • Tight coupling to Meta-internal components (Axiom/Velox) limits external reproducibility.
  • The "experience graph" concept is defined loosely; its boundary with episodic memory could be sharper.

04 Open Questions

what the paper leaves unanswered

Q1 · SCALE

How do storage footprint and query latency scale across thousands of sessions and tasks?

Q2 · CONSISTENCY

What isolation level fits concurrent MCTS backpropagation — does relaxed consistency measurably degrade search quality?

Q3 · FLYWHEEL

Does the closed-loop actually improve quality on held-out tasks, or does reuse-induced anchoring cap asymptotic performance?

Q4 · INJECTION

How should injection rate be set adaptively (decayed, selectivity/confidence-driven) to balance convergence vs diversity?

Q5 · PORTABILITY

Can the abstraction decouple from Axiom/Velox and be evaluated on open benchmarks vs generic vector-RAG systems?

Q6 · GOVERNANCE

How are retractions propagated through materialized training views when source nodes are invalidated?
