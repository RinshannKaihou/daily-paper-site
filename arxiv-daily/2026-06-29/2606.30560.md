> **一句话总结：** TraceLab 采集并发布了首个跨厂商的大规模真实编码代理（Claude Code、Codex）服务 trace——4,265 个会话、35.7 万次 LLM 步骤、43.3 万次工具调用、约 549 亿输入 token，来自 43 名开发者、跨 23 个模型版本、历时约 8 个月——系统刻画出"长自主循环、长上下文短输出、重尾工具调用、高但不完美的 95.7% 前缀缓存命中"四大特征，并指出前缀 token 占 59.5% 成本、人类思考占 92.3% 墙钟时间，由此提出五类 LLM serving 优化方向。

> **TL;DR:** TraceLab releases the first large-scale cross-provider trace of real coding-agent serving workloads (4,265 sessions, ~357K LLM steps, ~433K tool calls, ~54.9B input tokens from 43 developers over ~8 months across 23 model versions of Claude Code and Codex), characterizing them as long autonomous loops with long-context/short-output, heavy-tailed tools, and 95.7% prefix-cache hits, and shows prefix tokens are 59.5% of cost while human thinking is 92.3% of wall-clock — pinpointing five concrete serving optimization directions.

## 核心贡献

- **首个面向 serving 分析的真实跨厂商编码代理 trace**：4,265 个会话、357,161 次 LLM 步骤、432,510 次工具调用，约 549 亿输入 token（其中 526 亿为前缀 token），来自 43 名开发者、跨 23 个模型版本（Claude Code + Codex）、历时约 8 个月（2025-09 至 2026-06），估算等价 API 花费约 4.04 万美元。弥补了 Mooncake/LMSYS/BurstGPT 等传统 trace 与 SWE-bench/Terminal-Bench 等能力基准都无法刻画真实多步工具调用服务负载的空白。
- **统一的 trace 格式与采集/脱敏流水线**：将 Claude、Codex 异构日志标准化为"一步=一次 LLM 调用+其工具调用"的统一 schema，把输入 token 分解为前缀 token P（缓存命中）与追加 token A（新鲜输入），并对所有标识符脱敏、仅保留字符计数/token/时间戳以保护隐私；流水线与代码全部开源。
- **系统化、可量化的工作负载特征刻画**：从会话/请求/步骤三级粒度，量化了自主循环长度、上下文增长与压缩、成本分布、时延分布、工具调用计数/延迟/开销、前缀缓存命中与回收等多维特征，给出大量 p50/p90/p99 与分布图。
- **可操作的 serving 优化方向**：将观察转化为五类研究机会——更密集/融合的工具调用与低开销工具运行路径；append 长度感知的 prefill 路由与自适应 kernel/并行度；语义感知的工具延迟预测用于 KV 回收；面向长前缀的稀疏注意力；围绕人类节奏间隙的更廉价 KV 存储与回收/预取。
- **完整开源**：数据集、采集流水线、分析代码全部发布（GitHub `uw-syfi/TraceLab` + 项目网站 tracelab.cs.washington.edu）。

## Key Contributions

- **First real cross-provider coding-agent trace built for serving analysis**: 4,265 sessions, 357,161 LLM steps, 432,510 tool calls, ~54.9B input tokens (52.6B of them prefix/cached), from 43 developers over ~8 months across 23 model versions of Claude Code and Codex, with an estimated ~$40.4K API-price equivalent. Fills the gap left by traditional traces (Mooncake, LMSYS-Chat-1M, BurstGPT, Splitwise) and capability benchmarks (SWE-bench, Terminal-Bench), neither of which captures real persistent, multi-step, tool-using agent traffic.
- **Unified trace schema and a privacy-preserving collection pipeline**: Normalizes Claude/Codex heterogenous logs into a step-level schema (one LLM invocation + its tool calls), decomposing input tokens into prefix tokens P (cache hits) and append tokens A (fresh input); sanitizes identifiers, drops raw text, keeps only counts/tokens/timestamps. Pipeline and code fully open-sourced.
- **Quantified, multi-granularity workload characterization**: Analyzes the three-level hierarchy (session/request/step) across autonomous-loop length, context growth/compaction, cost, latency, tool counts/latency/overhead, and prefix-cache hit/eviction, with extensive p50/p90/p99 stats and CDF/scatter plots.
- **Five actionable serving-optimization directions**: Denser/fused tool calls and low-overhead tool runtime/approval paths; append-length-aware prefill routing with adaptive kernels/parallelism; semantic-aware tool-latency prediction for KV-cache eviction; sparse attention for long prefixes; cheaper KV storage and smarter eviction/prefetching around human-paced gaps.
- **Fully open release**: dataset, collection pipeline, and analysis code (GitHub `uw-syfi/TraceLab`; project website tracelab.cs.washington.edu).

## Methodology

**Trace model (3-level hierarchy).** A *session* is the top unit persisting accumulated context (conversation, files read/modified, tool outputs). Within a session a user issues one or more *requests* (a natural-language task); each request is resolved by an *agentic loop* of *steps*, where each step = one LLM invocation + any tool calls it emits. First step of a request is user-initiated; subsequent steps are tool-initiated. The loop shape is `User Input → LLM → Tool Calls → Tool Results → LLM → ... → Final Answer`.

**Collection.** Coding agents log conversations and tool calls by default; the authors build a pipeline that (1) extracts events — metadata, user messages, encrypted reasoning, output text, tool calls (type + args), tool results — each timestamped with per-LLM-invocation token usage; (2) normalizes both providers into a unified step-level schema; (3) anonymizes via stable pseudonyms, drops raw user/tool text, keeps only character counts + token usage + timestamps + pseudonymous user IDs.

**Token decomposition.** Total input `I = P + A`, where prefix tokens `P` are history retrieved from the prefix cache and append tokens `A` are freshly added (user messages, tool results, cache-miss tokens). Provider-specific accounting: `P_Claude = cache_read_input_tokens`, `A_Claude = input_tokens + cache_creation_input_tokens`; `P_Codex = cached_input_tokens`, `A_Codex = input_tokens_total − cached_input_tokens`. Output tokens `O` (incl. reasoning) taken directly; Codex also reports separated reasoning tokens. Timing reconstructed from per-event timestamps; Codex reports tool internal execution time directly, Claude's tool latency is derived from call/return timestamps.

**Cost accounting.** Uses providers' published API pricing (Claude 5-min prompt-cache tier; Codex GPT-5.5/5.4 rates) to convert tokens into USD by category (append/prefix/output).

## Results & Highlights

**Dataset scale (Table 1).** 4,265 sessions (Claude 2,676 / Codex 1,589), 357,161 LLM steps (140K/217K), 432,510 tool calls (142K/290K), 43 distinct users, 23 models (e.g., opus-4-7 63.1%, gpt-5.5 47.5%), ~$40.4K est. price. Tokens: 54.9B input (52.6B prefix, 2.34B append), 186.9M output; Codex reasoning tokens 36.8M.

**Autonomous loop (Table 2).** Per session avg 9.2 requests, 8.9 user-initiated steps, 73.6 tool-initiated steps, 101.4 tool calls (p99 up to 1,438 tool calls). Per request avg ~8 steps / 10.8 tool calls, mostly tool-initiated ⇒ the loop is largely self-driving.

**Context growth & compaction (Tables 3–4).** Context almost always grows; reductions are rare for Claude (mostly major), more frequent for Codex (mostly micro/ordinary at user steps). Genuine compaction (≥64k drop near peak, slow recovery) hits 9.7% of sessions (Codex 18.4% vs Claude 4.5% — consistent with Codex's shorter context), averaging 3.7 among those, and is overwhelmingly tool-initiated (86.5%).

**Cost (Table 5).** Session avg $9.70 (median $0.61, p99 $178); request avg $1.01; step avg $0.11. **Prefix tokens dominate spend at 59.5%** (append 29.2%, output only 11.2%) — re-reading accumulated context, not generation, is the cost driver.

**Timing (Table 7).** **Human thinking is 92.3% of session wall-clock** (LLM gen 3.3%, tool exec 4.8%); median session is a single request but a heavy tail (p99 ~206h) hoards idle time. Within a request (response time 4.3 min avg, median 38s, p90 6.4 min), tools = 59.8% vs generation = 41.0%.

**Input/output (Tables 8–9, Figs 1–2).** Median step: Claude 126k prefix / 857 append / 252 output; Codex 116k / 886 / 184. **Bimodal input**: ~two orders of magnitude more prefix than append; >90% of steps append <1k tokens yet >70% of total append tokens come from rare 10k+ steps (cache misses / initial prefills). Outputs are short because frequent tool calls fragment a full response into ~8 steps.

**Output attribution (Fig 3).** Claude and gpt-5.5 mostly *re-send* prior output as next-step input; gpt-5.4 mostly *caches* it as prefix — hypothesized to reflect KV-pool management choices in PD-disaggregated serving.

**LLM timing (Fig 6).** Normalized decode median 46.8 tok/s (Claude) / 33.9 (Codex), CV>50% at every context length; Codex pure-decode median ~57 tok/s with residual TTFT median ~2.3s (p90 6.4s) growing with context ⇒ TTFT/autoscaling/admission optimization matters.

**Tool calls (Figs 7–9, Tables 10).** 54 distinct tools (Claude) / 31 (Codex); top 3 tools = 80% (Claude: Bash/Read/Edit) / 95% (Codex: exec_command/write_stdin/apply_patch) of calls. **Heavy latency tail**: Claude calls >1 min are only 4.9% of calls but 92% of total tool time (Codex: >1 min = 3.1% calls / 61% time). Codex end-to-end tool latency exceeds runner-reported internal time by 77.8h / 18.6% residual (exec_command alone 64.1h), implying approval/runtime/shell-startup overhead.

**Prefix cache (Tables 11–12, Figs 10–11).** Global hit rate **95.7%** (Claude 95.8% / Codex 95.7%); tool-result steps 97.5% vs user-initiated 84.4%. After a 1-hour gap almost all steps miss. Fresh tokens are only **19.0% of append tokens** (Claude 12.3% / Codex 25.8%) ⇒ ~81% of prefill is in principle cacheable, yet cache misses cause **3.8× more tokens to be prefilled** than truly unique input. A storage-vs-eviction trade-off table shows retaining cache is favorable up to ~7.4× storage at 5-min eviction (avg saved/reduced step $0.192).

**Five systems opportunities** distilled across sections: denser/fused tool calling; append-aware prefill routing; semantic tool-latency prediction; sparse attention for long prefixes; cheaper KV storage + eviction/prefetching around human gaps.

## Critical Assessment

- **Novelty (High for the resource, Moderate-High for insights).** This is, by the authors' claim and corroborated by their related-work survey, the first large-scale *cross-provider* real coding-agent trace purpose-built for serving-system analysis. Individual observations (long context, prefix-cache dominance, tool-call tails) partially echo general agentic-LLM intuition, but the systematic, quantified, multi-granularity treatment across two providers and 23 models is genuinely new and more rigorous than prior anecdotal/benchmark-based claims.
- **Rigor (Strong).** The methodology is careful: explicit token decomposition, normalized step-level schema, clearly defined compaction criteria (3 conjunctive tests), output-cached vs output-resend invariants, and a cost model grounded in published pricing. Statistics are reported with p25/p50/p90/p99 and distribution plots throughout. The 4,265-session / 357K-step scale is substantial for a measurement study.
- **Reproducibility (Very High).** Dataset, collection pipeline, and analysis code are all open-sourced — the ideal for this paper type. Caveat: only a *sanitized* trace is released (raw text dropped), so some finer analyses cannot be independently extended, but the released artifacts support the paper's claims.
- **Significance (High).** Efficiently serving coding agents is a timely, economically large problem (the paper cites Claude Code's ~$1B annualized run-rate, Copilot 20M users). A shared, realistic trace can catalyze systems research the way Mooncake/LMSYS did for chat serving, and the five opportunities are concrete and actionable.
- **Red flags / caveats.** (1) **Self-sourced, single-institution** trace — generalizability to other orgs/workflows/agent domains is uncertain; the authors acknowledge this. (2) Only **externally visible** logs — mechanisms (scheduling, cache, routing) are *inferred*, not measured from provider internals; output-cached vs output-resend conclusions rest on token-bookkeeping inference, not backend telemetry. (3) The proposed optimizations are **observational**, not yet implemented/validated in a real serving system — no end-to-end system evaluation is provided, so real gains are prospective. (4) Coverage is limited to Claude Code and Codex; other agents (Gemini CLI, Cursor, open-source) and broader computer-use agents are out of scope. None of these are disqualifying for a measurement paper, but they bound the strength of the systems-design conclusions.

## Strengths

- First open, real-world, cross-provider, multi-model coding-agent trace designed for serving analysis — a valuable community resource.
- Rigorous, quantified characterization with full distributions (not just means) across session/request/step granularities.
- Clean, well-defined methodology (token decomposition, compaction criteria, output-attribution invariants, cost model).
- Directly maps findings to five concrete, actionable serving optimizations.
- Full open-source release (data + pipeline + analysis code) maximizes reproducibility and follow-on value.
- Strong, credible authorship from a top systems group.

## Limitations

- Trace is self-sourced from one institution; may not generalize to other organizations, workflows, or agent domains (authors explicitly acknowledge).
- Analysis is limited to externally visible logs; backend mechanisms (scheduling, cache, routing) are inferred rather than measured — e.g., the output-cached/output-resend KV-management hypothesis is not confirmed by provider telemetry.
- Proposes optimizations but does **not** implement or evaluate any of them in a serving system, so real-world gains are unvalidated.
- Covers only Claude Code and Codex; no Gemini CLI, Cursor, or open-source/local agents; reasoning text is encrypted for both providers.
- Sanitized release drops raw text, limiting independent re-analysis of content-level patterns.

## Relevance

对于关注 LLM serving 系统、编码代理（Claude Code、Codex）效率优化与 prefix cache / KV-cache 管理的研究者与工程师，本文提供了首个高质量、跨厂商的真实负载 trace 与系统化量化特征，是设计 append 感知 prefill、语义工具延迟预测、稀疏注意力与人类节奏感知 KV 回收等专用优化的关键数据基础与基准。

## Questions for Further Exploration

- How well do the five proposed optimizations translate into measured latency/cost gains when implemented in serving engines like vLLM, SGLang, or Mooncake? Which yields the largest payoff?
- How generalizable is this self-sourced, single-org trace to enterprise or open-source coding-agent populations? What reweighting/bias-correction would make it representative?
- How do prefix-cache hit patterns, tool-call tails, and context-growth dynamics differ across other agents (Gemini CLI, Cursor, local models) and broader computer-use/automation agents beyond coding?
- Can tool-call latency be predicted accurately from *semantic* request features + recent history, and how much does that improve KV-cache eviction/scheduling versus tool-name-only heuristics?
- What is the optimal KV-cache retention strategy given that human-paced gaps cause 3.8× redundant prefill — i.e., where exactly should the storage-vs-eviction trade-off knee sit in practice?
- The output-cached vs output-resend behavior differs by provider/model version — can a serving system adaptively choose the policy per step based on output length and cache pressure, and what are the gains?
