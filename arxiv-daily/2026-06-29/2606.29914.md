> **一句话总结：** 本文提出 MemDelta 受控评测协议，在 LongMemEval-S（500 题、3 个模型家族）上每次只改变记忆管线中的一个组件，揭示出"仅更换 embedding 模型一项就能让准确率波动 +6.2pp 并翻转 Mem0 vs RAG 的胜负结论"——许多被归功于记忆架构的增益其实来自同时被更换的 embedding、LLM 或检索管线。

> **TL;DR:** MemDelta is a controlled evaluation protocol that varies one pipeline component at a time (embedding model, LLM, retrieval/write-path) on LongMemEval-S across three model families; it shows a single embedding swap shifts accuracy by +6.2pp and flips the Mem0-vs-RAG verdict, so reported memory gains are frequently confounded by simultaneous changes to other components and cannot be cleanly attributed to architecture.

## 核心贡献

- 提出 **MemDelta 受控评测协议**：以"每次只改变一个组件"为核心原则，构造 7 种记忆策略（S0 无记忆 → S1 全上下文 → S4/S4b 不同 embedding 的逐字 RAG → S2 自记忆 → S3 Mem0），通过 6 组配对比较逐一隔离检索质量、embedding 选择、模型行为、写入路径成本四个隐藏变量。
- 用**具体案例揭穿"架构胜利"假象**：同一问题下 MiniLM-RAG 失败而 cloud-RAG 成功、Mem0 仅在 embedding 较弱时才以 +11pp 看似胜出，换用 cloud embedding 后反而 −1.2pp 落败——证明增益来自 embedding 而非架构。
- 给出**跨 3 个模型家族的稳健性检验**：同一 S1-vs-S4 比较在 GPT-4o-mini（持平）、Gemini（全上下文 +14pp）、Sonnet（RAG +31pp）上得出三种相反结论，Sonnet 在 115K 上下文下 63% 直接拒答。
- 进行**成本匹配审计**：在 88 个可匹配实例上 Mem0 (72.7%) 与 cloud-RAG (73.9%) 打平（p=1.0），但写入成本高 50×（1000+ 次 LLM 调用、~120 分钟、$0.50+/实例）。
- 提出可操作的**评测改进清单**：固定命名 embedding、加入随机检索基线、至少两个模型家族、披露 embedding 及敏感性、报告写入成本、对高成本系统做实例匹配比较。

## Key Contributions

- **MemDelta controlled-evaluation protocol** built on a single principle — vary one factor at a time while holding the model, judge, questions, and input history fixed — operationalized as 7 memory strategies (S0 no-memory → S1 full-context → S4/S4b verbatim RAG with MiniLM vs. cloud embeddings → S2 self-memory scratchpad → S3 Mem0) connected by 6 controlled pairwise comparisons isolating retrieval quality, embedding choice, model behavior, and write-path cost.
- **Concrete reversal case studies** showing an apparent "+11pp architectural win" for Mem0 over MiniLM-RAG actually disappears (−1.2pp) when the baseline uses cloud embeddings — the gain was an embedding confound, not architecture.
- **Cross-model robustness probe** demonstrating the identical S1-vs-S4 comparison yields three contradictory conclusions across GPT-4o-mini (tie), Gemini (+14pp full-context), and Sonnet (+31pp RAG, driven by 63% full-context refusals on 115K tokens).
- **Cost-matched audit**: on 88 matched instances Mem0 (72.7%) ties cloud verbatim RAG (73.9%, p=1.0) but at ~50× the write-path cost (1,000+ LLM calls, ~120 min, $0.50+ per instance), framing accuracy-only claims as misleading.
- **Actionable evaluation checklist** (6 items) plus open-sourced protocol, code, and results.

## Methodology

MemDelta is explicitly a **measurement / methodology paper, not a new memory architecture** (the author stresses "MemDelta is a measurement paper rather than a new memory architecture"). Its single design principle is **vary one factor at a time** via paired conditions that isolate one variable while holding everything else fixed.

**Dataset — LongMemEval-S** [Wu et al., 2024]: a 500-question subset with 6 question types (single-session-user 70, single-session-assistant 56, single-session-preference 30, temporal-reasoning 133, knowledge-update 78, multi-session 133); each instance has 39–66 sessions (mean 50.2, ~115K tokens).

**Seven strategies** (ordered by write-path complexity, Table 1):
- **S0** No memory (question only) — lower bound.
- **S_rand** Random chunks (~5K tok) — controls for "having text" vs. "relevant text."
- **S1** Full conversation history (~115K tok) — diagnostic ceiling (not a deployment strategy).
- **S4** Verbatim RAG, MiniLM 384-d embeddings, 512-tok chunks, top-10 — no LLM calls.
- **S4b** Byte-identical to S4 except OpenAI text-embedding-3-small (1536-d) — isolates embedding choice.
- **S2** Agent scratchpad, 4096-tok budget, topic-organized, stale-fact overwriting (~250 LLM calls).
- **S3** Mem0 v2.0 extraction pipeline (1,000+ LLM calls, ~120 min, $0.50+/instance).

**Models**: full matrix on GPT-4o-mini (n=500); selected comparisons on Claude Sonnet (n=300–500) and Gemini 2.5 Flash (n=100), constrained by cost/rate-limits; samples are the first n instances in the benchmark's fixed ordering (no imputation).

**Six controlled comparisons** (Table 2): retrieval quality (S0→S4→S4b), retrieval relevance (S_rand vs S4), base-model behavior (S1 vs S4 ×3 models), embedding model (S4 vs S4b), write-path complexity (S4b vs S3), storage control (S2 vs S4).

**Per-question procedure** (5 stages): Reset → Ingest (record time/LLM calls/$) → Build context → Answer (temp 0, fixed prompt) → Judge (GPT-4o-mini binary correct/incorrect vs. ground truth).

**Metrics**: accuracy with 95% bootstrap CIs (2,000 resamples); paired significance via **McNemar's test** on per-question correctness; write-path cost (wall-clock ingest time, LLM call count, dollar cost). The paired design means constant judge bias cancels in pairwise comparisons.

## Results & Highlights

**Main results (GPT-4o-mini, n=500; McNemar vs S4, Table 3):**
| Strategy | N | Acc | 95% CI | vs S4 |
|---|---|---|---|---|
| S0 No Memory | 500 | 2.2% | [1.0,3.6] | −45.0pp*** |
| S_rand Random RAG | 500 | 3.2% | [1.8,4.8] | −44.0pp*** |
| S2 Self-Memory | 100 | 42.0% | [32.0,52.0] | −5.2pp* |
| S4 Verbatim RAG (MiniLM) | 500 | 47.2% | [42.8,51.4] | — |
| S1 Full Context | 500 | 49.8% | [45.6,54.0] | +2.6pp (ns, p=0.34) |
| S4b Verbatim RAG (cloud) | 500 | 53.4% | [49.0,57.8] | +6.2pp** (p=0.004) |

**Finding 1 — Retrieval quality**: S0 (2.2%) and random retrieval (3.2%) confirm memory is needed. Verbatim RAG (47.2%) matches full-context (49.8%, p=0.34); with cloud embeddings RAG reaches 53.4%, numerically exceeding full context (p=0.18). Agent self-memory (42%, n=100) underperforms basic retrieval and collapses on multi-session questions (3.3%).

**Finding 2 — Model behavior (Table 4)**: identical S1-vs-S4 gives three opposite verdicts — GPT-4o-mini tie (+2.6pp), Gemini full-context wins (+14pp), Sonnet RAG wins (+31pp). Sonnet's length-driven failure: 63% (188/300) of full-context errors are explicit refusals ("I don't see this in our conversation history") despite the answer being present in 115K tokens; truncating to ~47K cuts refusals to 41% and lifts accuracy 14%→38%, but still trails RAG (61%). Refusal rates vary by type: SS-User 70%, multi-session 70%, but SS-Pref only 7% (instead hallucinates 83% of the time).

**Finding 3 — Embedding choice (Table 5)**: a single embedding swap within identical code yields +6.2pp overall (p=0.004); largest gains on temporal (+10.5pp), multi-session (+11.3pp), SS-User (+11.4pp), but reverses on SS-Asst (−5.4pp) and SS-Pref (−10.0pp). This rivals or exceeds gains typically attributed to architecture.

**Finding 4 — Write-path cost (Tables 6–7)**: on 88 matched instances Mem0 (72.7%) does not beat cloud verbatim RAG (73.9%, p=1.0; 90% CI on Δ [−10.8,+8.4]pp — wide CI reflects small n, balanced disagreements, not proven equivalence). Yet Mem0 costs ~120 min + 1,000+ LLM calls + $0.50+/instance vs. ~60s + 0 LLM calls + $0.01 for S4b — a ~50× write-path cost penalty.

**Per-type pattern (Table 8)**: retrieval excels on single-session questions (S4b 84–87%); full-context excels on knowledge-update (72%, requires reconciling conflicting recent facts — a reasoning task retrieval doesn't solve) and multi-session. Cloud embeddings close much of the full-context "advantage" on cross-session questions (multi-session 24%→35%, temporal 30%→41%), suggesting part of that advantage is a retrieval-quality gap. Pattern holds for Sonnet (structural, not model-specific).

## Authors & Credibility

- Single author **Kuan Wang** (preprint, arXiv 2026-06-29). Credibility is strengthened by: (a) a genuinely rigorous one-variable-at-a-time paired design; (b) McNemar tests, 95% bootstrap CIs, and explicit p-values rather than headline-grabbing deltas; (c) honest reporting of negative/null results (e.g., p=1.0 correctly framed as "balanced disagreements, not proven equivalence"); (d) candid disclosure of the S4b-vs-S3 imperfect embedding match (text-embedding-3-small vs. ada-002) and of Mem0's 12/100 API failures being excluded (which may slightly favor S3). Open-sourcing protocol, code, and results further supports reproducibility.

## Critical Assessment

- **Novelty — High (methodological).** The contribution is not a new memory system but a controlled-protocol lens that exposes confounds; this is timely and arguably more valuable than yet another architecture given the current flood of "memory beats RAG" claims. The framing as "controlled deltas, not a causal decomposition" is appropriately scoped.
- **Rigor — High.** Paired McNemar testing, bootstrap CIs, multi-model robustness checks, write-path cost audited as a first-class metric, and careful per-type stratification. The author is unusually careful about what significance does and does not mean.
- **Reproducibility — Good.** Concrete models, fixed benchmark subset, 5-stage procedure, open-sourced code. Caveats: reliance on a commercial API (Mem0) whose behavior may drift, and judge = GPT-4o-mini for all models (though effect sizes are large enough that judge bias is unlikely to explain them).
- **Significance — Potentially field-shaping.** The 6-point evaluation checklist is directly actionable for benchmark authors (LongMemEval, MemoryAgentBench, MemoryArena, BEAM, etc.) and could become a standard reporting expectation.
- **Red flags / honest caveats**:
  - **Small n for the headline cost-matched claim** (n=88, only 2 of 6 question types) — explicitly admitted; the temporal/knowledge-update types where extraction "might help most" were not run. So the "Mem0 ties RAG" conclusion is narrow, not general.
  - **Synthetic-dialogue bias**: LongMemEval-S uses synthetic dialogues with high lexical overlap that may favor verbatim retrieval over extraction — generalization to real user data is untested.
  - **Approximate embedding control**: S4b (text-embedding-3-small) vs S3 (ada-002) are different OpenAI embedders, so the write-path comparison only approximately controls for embedding quality.
  - **Single author** → limited independent cross-checking of the pipeline.
  - **Model-version dependence**: the dramatic Sonnet refusal behavior (63%) is provider/version-specific and could change with updates, weakening the cross-model generality of that specific finding.
  - Not an additive causal model — variables may interact (acknowledged).

## Strengths

- Clean, principled "one variable at a time" design that directly targets the attribution problem in memory evaluation.
- Transparent, statistically grounded reporting (CIs, McNemar, explicit p-values, cost accounting).
- Honest framing of null/weak results and explicit acknowledgment of limitations.
- Concrete, immediately adoptable evaluation checklist plus open-source release.
- Compelling worked examples (Spotify playlist, commute duration, $5 coupon) make abstract confounds tangible.

## Limitations

- Headline cost-matched comparison covers only 88 instances / 2 of 6 question types (temporal & knowledge-update — arguably where extraction helps most — untested due to cost).
- Synthetic LongMemEval-S dialogues likely favor verbatim retrieval; real-user generalization untested.
- Imperfect embedding control (3-small vs ada-002) for the write-path comparison.
- GPT-4o-mini judges all models (no human spot-checks).
- Sonnet refusal findings are model-version-sensitive.
- Single author; not a formal/additive causal decomposition (interactions possible).

## Relevance

对关注 LLM Agent 记忆与检索评测严谨性的研究者高度相关——它系统揭示现有基准（LongMemEval、MemoryArena 等）的混淆问题并提供可直接采纳的受控协议，有助于在归因于"架构增益"前先排除 embedding、LLM、成本等隐藏变量，特别契合评估方法学与 RAG/记忆系统交叉方向。

## Questions for Further Exploration

- How do the findings transfer to **real-user (non-synthetic) dialogues**, where lexical overlap is lower and extraction-based memory might plausibly help more?
- Would running S3 (Mem0) on the **temporal-reasoning and knowledge-update** subsets (currently untested) change the cost-matched verdict?
- Does applying a **multiple-comparison correction** across the six pairwise tests and per-type breakdowns preserve significance?
- How stable is the **Sonnet refusal phenomenon** across API/model versions and providers (e.g., new Claude releases)?
- Can MemDelta's protocol be extended to **multi-hop, agentic (tool-use), and graph-based memory** (Graphiti, A-MEM) where retrieval is structurally harder?
- What is the full **accuracy-vs-write-cost Pareto frontier** across more embedding/retriever/reranker combinations when properly controlled?
- Is there a principled way to **decompose additive vs. interactive** contributions of the four confounds, beyond the controlled-contrast framing?
