Stabilizing Extrapolation in Looped Transformers via Learned Stochastic Stopping · Hsun-Yu Kuo et al. · EPFL & MPI-Tübingen · 2026-06-29
looped-transformers length-generalization reinforcement-learning algorithmic-tasks OOD-generalization

Key finding

Looped Transformers extrapolate unreliably because training-time loop count spuriously tracks sequence length — and randomizing that depth at training time stabilizes out-of-distribution prediction.

Headline metric
97.5% OOD

RL-Halting on Dyck-1 (formal language), up from the 84.6 best baseline. Across all four tasks run-to-run Std dropped to ≤ 3.6.

Why this matters

“When to stop” is reframed as a core training-time design choice for looped models — an actionable, cautionary signal for anyone deploying adaptive computation for length generalization.

Critical edge

Stability ≠ quality. On Copy, RL-Halting locks onto a suboptimal computation: OOD accuracy collapses to 43.2 (vs 68.7 for K=n) while variance hits its lowest. A stable schedule can select a non-extrapolating computation.

Methodology, distilled

A Looped Transformer reuses a single shared 3-layer block across iterations (fixed parameters, variable depth). Each step re-injects the original input, so hidden states evolve as:

H(t) = Blockθ( H(t−1) + H(0) ),  t = 1…K

The authors treat the loop count as a random variable K ∼ πstop(K|x) and minimize a depth-averaged objective Lπ = 𝔼K[ℓ] — sampling one depth per example per step. Their diagnosis: deterministic schedules supervise only one depth, leaving the chosen computation brittle to it. Three schedule families are compared:

  • Deterministic — fixed K=20 (latent-reasoning style) and length-matched K=n.
  • Hand-designed stochastic — K = clip(n+Δ, 1, Tmax), Δ ∼ Unif{−w…+w} (windows ±1/±5/±10).
  • RL-Halting — a stopping head parameterizes πstop,ϕ(τ|x); per step one depth is sampled & supervised, the policy trained with REINFORCE (reward R=−ℓ, moving-average baseline, entropy regularizer).

The headline diagnostic is oracle-over-iterations accuracy — it uses ground truth to pick the best depth, exposing whether an extrapolating computation exists, separate from whether any stopping rule finds it.

RL-Halting wins accuracy on 3 of 4 tasks and cuts run-to-run variance everywhere — but on Copy it trades accuracy for stability, foregrounded honestly by the authors. Headline numbers use oracle-over-iterations (ground-truth depth selection), an upper bound; controlling initialisation / data-order does not remove the baseline instability.

Strengths

  • Crisp, falsifiable diagnosis — length–loop-count spurious correlation as a training-time selection problem, backed by a dedicated oracle diagnostic.
  • Drop-in fix + principled upgrade — simple depth-window randomization, plus a loss-driven RL-Halting rule.
  • Variance-first evaluation with parameter-matched baselines, controlled ablations, and an explicit PonderNet comparison.
  • Intellectual honesty — the Copy failure and the informal scope of the theory are foregrounded, not buried.

Limitations

  • Scope — only synthetic algorithmic tasks; no realistic NLP / code / vision, so external validity is unknown.
  • Stability can mask failure — RL-Halting stabilizes a bad computation (Copy), demanding per-task validation.
  • Diagnostic optimism — headline OOD uses oracle depth selection; practical selected-rule accuracy is lower.
  • Theory is informal — Section 5 gives intuition, not guarantees; formal toy results rest on bounded-residual assumptions.
  • Added cost / code release — stopping head + REINFORCE hyperparameters need tuning; no compute-cost analysis, code link not confirmed in main text.

Open questions

  • TRANSFERDoes RL-Halting move beyond synthetic tasks to real language, code, or math — where “when to stop” matters for chain-of-thought-style latent reasoning?
  • SCALEDoes variance reduction hold at 2×–10× beyond the training distribution, or does the stabilized computation eventually collapse?
  • MECHANISMIs the length–loop correlation the only OOD-variance source, or do positional encoding / attention variance interact — and could randomized positions compound the gains?
  • INTERPRETABILITYDoes the learned policy converge to task-structured step counts (e.g., proportional to algorithmic complexity)?
  • COSTIs there a cheap non-RL heuristic that captures most of RL-Halting’s variance reduction?
  • FAILURE MODEWhy does it lock a suboptimal computation on Copy specifically — a trivial-task artifact, or a general failure of loss-driven stopping on easy tasks?
