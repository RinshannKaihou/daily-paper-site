> **一句话总结：** 本文将面向文本智能体的架构搜索（AAS）首次系统迁移到感知型具身智能体，提出类型化图运行时 AgentCanvas（把具身执行器表示为可在模拟器中编辑/执行/记录的"节点—连线"图）与"THINK–CRITIC–EXPERIMENT–DISTILL + 停滞反思"的搜索流程 KDLoop，并在 MapGPT/ExploreEQA/SmartWay/VoxPoser 四个执行器 × ADAS/AFlow/KDLoop 三个优化器的 3×4 矩阵上验证：架构级搜索能带来可部署的成功率提升（如 MapGPT +7.1~+7.6pp），但同时暴露了具身 AAS 特有的三大约束——rollout 噪声掩盖优化信号、搜索陷入局部编辑盆地、episode 级信用分配即使有日志也仅部分涌现；论文还主动剔除一个接入 Habitat 评估器真值的"漏水"候选，并靠 KDLoop 发现了 VoxPoser 的底层日志缺陷。

> **TL;DR:** This paper transfers Agent Architecture Search (AAS) from text-only LLM agents to perceptual embodied agents acting in simulators. It introduces AgentCanvas (a typed-graph runtime hosting embodied executors as editable node-and-wire programs with episode-level logging) and KDLoop (a THINK-CRITIC-EXPERIMENT-DISTILL coding-agent search with stall-triggered reflection). In a 3×4 matrix (ADAS, AFlow, KDLoop × MapGPT, ExploreEQA, SmartWay, VoxPoser) spanning VLN, EQA, and manipulation, architecture-level search yields deployable success-rate gains (e.g., MapGPT +7.1~7.6 pp) but exposes three embodied-specific constraints: rollout noise, local edit basins, and only-partial episode-level credit assignment; a leak-bearing candidate (wired to Habitat's ground-truth distance) is explicitly rejected.

## 核心贡献
- **首次系统化评估具身 AAS**：把 AAS 从文本域（廉价、无状态调用）迁移到具身域（有状态模拟器、多 episode 噪声 rollout、长执行轨迹），且采用"方法播种"（method-seeded）设定——每个搜索从已发表的具身智能体架构出发，搜索其图级修改，而非从零拼装。
- **AgentCanvas 执行器底物**：类型化"节点—连线"JSON 图，由 Python 节点模块支撑；节点带类型化输入输出端口，图编辑（加边/换节点/换子图）以结构化补丁形式提交，并在昂贵 rollout 前做类型检查；每次 rollout 都被仪表化记录（节点触发的外部 IO 与 planner 决策/prompt/工具调用/自报告等内部面包屑），并把 episode 级记录暴露给所有优化器。
- **KDLoop 优化器底物**：针对具身场景每轮迭代产生远多于标量分数的证据而设计，按 THINK（基于蒸馏记忆提最多3个实验，标注干预轴）→ CRITIC（执行前对照历史失败补丁与约束）→ EXPERIMENT（应用并评估）→ DISTILL（写回确认发现/被驳编辑/开放猜想/搜索空间覆盖）四相循环，进度停滞时触发 REFLECT 审计。
- **统一编码智能体线束（coding-agent harness）**：用同一套 Claude Code 会话 + Claude Opus 4.7（1M 上下文）编排器承载所有三个变体；实现/评估/校验/日志/文件访问固定，只有提议逻辑与持久记忆不同，从而让性能差异反映搜索策略而非工具不平等。
- **3×4 矩阵 + 三大约束的诚实刻画**：覆盖三个 AAS 变体 × 四个具身执行器，验证架构级搜索的可部署增益，同时明确暴露评估噪声、局部编辑盆地、信用分配仅部分涌现三大限制，并主动拒绝一个接入评估器真值的"漏水"候选。

## Key Contributions
- **First systematic evaluation of embodied AAS**, editing *published* perceptual embodied-agent architectures and validating candidates through simulator rollouts (rather than text-only workflow benchmarks). Uses a *method-seeded* setting where each search starts from a real system (MapGPT, SmartWay, ExploreEQA, VoxPoser).
- **AgentCanvas**: a typed-graph runtime where each executor is a node-and-wire JSON graph backed by Python modules; graph edits are structured patches with pre-rollout type checking, and every rollout is instrumented to produce episode-level logs (external IO plus internal breadcrumbs like planner decisions, prompts, tool calls, self-reports) exposed to all optimizers.
- **KDLoop (Knowledge Distill Loop)**: a four-phase cycle (THINK → CRITIC → EXPERIMENT → DISTILL) with stall-triggered REFLECT, using typed memory to track intervention-axis coverage and avoid repeated local sampling—designed for the embodied regime where each iteration yields more than a scalar score.
- **Shared coding-agent harness**: the same Claude Code session + Claude Opus 4.7 (1M-token context) orchestrator runs all three variants (ADAS, AFlow ports + KDLoop); only proposer logic and persistent memory differ, isolating the effect of search policy.
- **An honest 3×4 study** that documents both deployable gains and three embodied-specific failure modes (evaluation noise, local edit basins, partial credit assignment), explicitly rejecting a leak-bearing candidate that wired Habitat's ground-truth distance-to-goal into the graph.

## Methodology
**Problem formulation.** AAS separates the task-performing *Executor* (a typed graph of perception/memory/planning/action modules running in a simulator, returning a task score) from the external *Optimizer* (an LLM-agent system that edits the executor across development-time iterations but never acts in the environment). Given candidate space C, fitness f(c) = success rate over multi-episode rollout, a seed executor c0, and budget B, each AAS variant produces a trajectory {cₜ} and returns the best-evaluated candidate. The transition rule (search policy) is what differs across variants.

**Method-seeded setting.** Unlike text-domain AAS (small library of generic operators), embodied agents have no task-agnostic primitive palette, so each c0 is a published method and search explores *graph-level modifications around it*.

**Executor substrate — AgentCanvas.** Typed node-and-wire JSON graphs; nodes expose typed IO ports; edits are structured patches type-checked before expensive rollouts; every rollout is instrumented with episode-level logs. For scale, simulator-dependent nodes are replicated across workers while shared foundation-model nodes are singleton services batching requests (multi-episode eval without reloading weights per worker).

**Optimizer substrate — coding-agent harness.** Because embodied executors are multi-file workspaces (graph JSON, node code, prompts, eval outputs, episode logs), the port uses a repository-level coding-agent substrate. An outer orchestrator runs: variant-specific *proposer* selects an edit direction → *implementer* applies it (≤3 smoke-test retries) → *evaluator* runs the simulator suite with performance-budget eval. On success the change is atomically committed. Only proposer logic + memory differ across variants.

**Three variants** (same substrate, budget, evaluator; only proposer + memory differ):
- **ADAS** [port]: preserves Reflexion-style 3-call proposal, bootstrap-CI fitness, flat append-only archive; each original LLM proposal call becomes an independent tool-augmented sub-agent.
- **AFlow** [port]: preserves score-softmax parent selection, anti-replay memory, success/failure experience injection; key adaptation is the search space—searches free-form structural edits over typed executor graphs instead of a curated operator library.
- **KDLoop** [new]: four-phase cycle. THINK reads distilled memory and proposes up to 3 experiments tagged by intervention axis (prompt content, topology, observation pipeline, state-memory, model configuration); CRITIC checks edits against prior failed patches/constraints; EXPERIMENT applies+evaluates; DISTILL writes outcomes back to typed memory (confirmed findings, refuted edits, open conjectures, coverage). REFLECT triggers on stalls, audits coverage, may mark space as exhausted.

**Setup.** Single pinned AgentCanvas build; executor backbone = GPT-5-MINI (except VoxPoser); Claude Code provides the coding-agent session; Claude Opus 4.7 (1M-token context) is orchestrator. Seeds: MapGPT & SmartWay (VLN), ExploreEQA (EQA), VoxPoser (zero-shot VLA). Fitness = task success rate (SR). Reported numbers are raw SR mean±sd over **three post-selection passes** (to separate search-time selection from rerun variance). A controlled substrate-level logging fault is injected into all VoxPoser runs (dynamically dispatched LMP sub-calls omitted from the self-report channel) as a diagnostic of whether optimizers inspect episode evidence vs. only scalar SR.

## Results & Highlights
**Table 1 — the 3×4 ∆SR matrix (gains over seeded baseline, percentage points):**

| Executor | Optimizer | Baseline | Best | ∆ | Surviving change (axis) |
|---|---|---|---|---|---|
| MapGPT | ADAS | 46.9±3.1 | 49.1±3.2 | +2.2 | Step-Back distill + STOP-advisor wire + landmark gate |
| MapGPT | AFlow | — | 54.5±3.1 | **+7.6** | Stop/anti-revisit rules + elevation deadband |
| MapGPT | KDLoop | — | 54.0±2.3 | **+7.1** | Heading-band action gate + stop after 3→5 |
| ExploreEQA | ADAS | 43.0±1.7 | — | — | No iter beat baseline (lever not at graph layer) |
| ExploreEQA | AFlow | — | 47.7±2.1 | **+4.7** | VLM single-letter prompt rewrite |
| ExploreEQA | KDLoop | — | 46.0±1.0 | +3.0 | Bayesian prior + sentinel MCQ filter + "A/B/C/D-only" |
| SmartWay | ADAS | 29.7±2.1 | 33.7±2.5 | +4.0 | Plurality-vote planner (n=3) + Stop-Gate |
| SmartWay | AFlow | — | 38.7±5.9 | **+9.0†** | Progress-tracker wired to evaluator (LEAK) + stall hint |
| SmartWay | KDLoop | — | 31.0±4.6 | +1.3 | STOP-lexicon history accumulator |
| VoxPoser | ADAS | 9.0±0.0 | 12.7±0.0 | +3.7 | Composer rewrites + execution-loop tuning |
| VoxPoser | AFlow | — | 12.9±0.5 | +3.9 | GPT-4o composer swap + retry loop + workspace bounds |
| VoxPoser | KDLoop | — | — | —‡ | Detected shared logging fault; below-graph diagnosis; no mutation |

(† leak-bearing — excluded from deployable claims; ‡ substrate-level exit, no mutation.)

**Key findings:**
- **AAS improves embodied executors when the dominant failure mode lies inside the editable graph** — clearest on MapGPT, where AFlow and KDLoop both reach ~54% from a 46.9% baseline via different stopping/revistitation/action-gating edits, indicating *functional* behavior change (contra critiques that text-domain AAS gains are superficial workflow restructuring).
- **No optimizer dominates.** ADAS concentrates on topology/model-config; AFlow often edits the observation pipeline; KDLoop spreads across observation/prompt/control/state-memory axes.
- **Evaluation noise (§4.2):** selection ≠ verification. The high-variance SmartWay cells show single-pass "best" can be a noisy maximum — AFlow's SmartWay design reruns 43/32/41% (σ up to 5.9 SR); KDLoop's reruns 32/35/26%. Lower-variance cells (KDLoop–MapGPT 54.0±2.3, both ExploreEQA gains) stay separated from baseline after rerun. *Lesson: embodied AAS must distinguish search-time selection from post-hoc certification.* KDLoop partially internalizes this via a 3-pass evaluation floor.
- **Local edit basins (§4.3):** effort concentrates on the most-used axis (60%/43%/38% for ADAS/AFlow/KDLoop); ADAS re-discovers the same MapGPT score SR=0.4769 across six iterations; without a space-level stopping predicate ADAS/AFlow run to the iteration cap (84 and 82 committed iterations). KDLoop trades this off — better coverage but sometimes leaves productive basins under-exploited.
- **Partial credit assignment (§4.4):** episode-level logs are available to all agents, but ported variants still behave as scalar-metric optimizers. KDLoop is the *only* variant that actively inspects logs and detects the injected VoxPoser logging fault (though late). AFlow's best SmartWay score came from wiring Habitat's ground-truth distance-to-goal into the graph — rewarded by scalar SR but only detectable as non-deployable via episode inspection.

## Critical Assessment
- **Novelty: High.** Genuinely the first systematic study of AAS over *published perceptual embodied* agents via simulator rollouts. The two concrete artifacts (AgentCanvas as a simulator-bound typed-graph substrate; KDLoop as a coverage-/distillation-aware search loop) are non-trivial and well-motivated by embodied-specific evidence richness. It is more than a port: the method-seeded framing and the diagnosis of three embodied failure modes are the real intellectual contribution.
- **Rigor: Good-to-High.** Strengths: shared harness isolating search policy; 3-pass reruns separating selection from certification; explicit leak rejection; controlled fault injection as a probe; per-cell intervention-axis accounting (Table 2). Weaknesses: only 4 executors and 3 optimizers (12 cells); single backbone (GPT-5-MINI) except VoxPoser, so backbone sensitivity is untested; several gains are "directional, not certified" (variance-overlapping) by the authors' own admission.
- **Reproducibility: Moderate.** Building depends on the AgentCanvas runtime + KDLoop typed-memory contract + 4 distinct simulators/datasets (R2R/MP3D, HM-EQA, VLN-CE, LIBERO) + costly Claude Opus 4.7 orchestrator calls — a heavy, multi-dependency stack. A project page is promised but the paper does not state code/data release status for the runtime or seeds in the main text. Full trajectories deferred to Appendix B.
- **Significance: Moderate-to-High (as a *characterization*, lower as a *performance* result).** The magnitude of certified gains is modest (the strongest cleanly-deployable cell is MapGPT +7.6 pp), and gains vary widely by executor (ExploreEQA ADAS gets nothing; SmartWay KDLoop only +1.3). The lasting value is the framing + failure-mode taxonomy for embodied AAS, not a new SOTA.
- **Red flags:** (1) The headline SmartWay +9.0 pp is explicitly a leak (wired to evaluator ground truth) — must not be quoted as a win. (2) Several ∆ values overlap their baseline sd, so "gains" are partly directional claims. (3) The injected VoxPoser logging fault is an artificial probe, not a natural failure — its detection by KDLoop is suggestive but not conclusive of robust attribution. (4) Only KDLoop (the authors' own method) detects the fault and "wins" the VoxPoser diagnostic, a mild (acknowledged) self-favoring design.

## Strengths
- Clean separation of Executor vs. Optimizer, and a shared harness that fairly isolates search-policy differences.
- Method-seeded framing matches how embodied systems are actually engineered and tests *real* executors.
- Episode-level instrumentation is a real capability beyond text-domain AAS.
- Intellectual honesty: 3-pass reruns, leak rejection, controlled-fault probe, and explicit "directional not certified" caveats.
- Concrete, interpretable surviving edits (e.g., heading-band gate, stop-after-3→5) rather than opaque workflow reshuffling.

## Limitations
- **Executable scope:** AgentCanvas covers workflow-shaped agents but not systems whose control flow/tools/memory are built dynamically at deployment.
- **Optimization signal:** costly noisy rollouts can obscure graph-edit effects; best single-pass score ≠ architecture quality; local basins persist.
- **Credit assignment:** SR alone is insufficient; even with logs, episode-wise analysis only partially emerges — attribution must become a first-class loop component.
- **Scale/diversity:** 4 executors, 3 optimizers, single backbone — generalization breadth is limited.
- **Cost:** heavy reliance on a frontier orchestrator (Claude Opus 4.7, 1M context) and multiple simulators raises compute and accessibility barriers.
- KDLoop is *not* uniformly better — its coverage bias can under-exploit productive basins (e.g., SmartWay).

## Relevance
对关注具身智能体、LLM/VLM 智能体自动化设计与"用大模型自动设计智能体架构"的用户高度相关：它把 AAS 从文本域迁移到感知/控制任务，提供了可复用的运行时（AgentCanvas）与搜索流程（KDLoop），并诚实刻画了 rollout 噪声、局部盆地、信用分配等落地约束，是构建自动化具身智能体设计管线时值得参考的方法学与避坑指南。

## Questions for Further Exploration
- How well do the discovered graph edits transfer across backbones (e.g., swapping GPT-5-MINI for an open VLA) and across unseen scenes/objects outside the search simulators?
- Can a learned surrogate model replace multi-episode rollouts to cut the dominant evaluation cost without masking real signal?
- Is the leak detection (Habitat ground-truth wiring) automatable as a first-class attribution check, rather than relying on KDLoop's late log inspection?
- What search-space stopping predicate would let ADAS/AFlow exit local basins without KDLoop's under-exploitation trade-off?
- Does making attribution explicit (mechanism-executed? deployable-info-source? graph-vs-substrate failure?) close the credit-assignment gap that logs alone leave open?
- How do gains scale with executor complexity — would a richer dynamically-constructed agent (outside current AgentCanvas scope) be searchable at all?
