> **一句话总结：** 本文将长时序智能体搜索过程抽象为"经验图"（experience graph：可执行产物、工具输出、奖励、兄弟比较、因果谱系），并提出数据底座 Trellis，将其作为一等公民的数据库状态加以管理——把恢复、跨会话复用、训练数据提取、历史回溯统一映射为查询/向量种子图检索/物化视图/时间旅行查询，使智能体退化为无状态计算，崩溃恢复、水平扩展与闭环训练飞轮成为架构副产品；在 Meta 的 KernelEvolve 加速器内核优化器上的受控实验显示，跨会话复用使达到 1.2× 加速快约 10 倍、每个有效节点的 token 成本降低 52%，并揭示了复用与多样性之间的"探索—锚定"权衡。

> **TL;DR:** Long-horizon agentic tasks (code generation, scientific discovery, hardware design, security research) generate rich "experience graphs" (artifacts, tool outputs, rewards, sibling comparisons, causal lineage), but current frameworks discard them as JSON checkpoints / session logs that cannot survive crashes, be queried across users, or be materialized into training data. Trellis treats the experience graph as first-class, governed, queryable database state, reframing agent operations as database access patterns: frontier selection = query, cross-session reuse = vector-seeded graph retrieval, training-data extraction = materialized view, "what did the agent know at step t" = time-travel query. Agents become stateless serverless compute; crash recovery, horizontal scaling, and a closed-loop SFT/DPO/GRPO training flywheel emerge as byproducts. Grounded in KernelEvolve (Meta), a controlled study shows cross-session memory reaches the 1.2× speedup target ~10× faster (5 vs 51 steps) at 52% lower token cost per valid node, while exposing an exploration–anchoring tradeoff.

## 核心贡献

- **定义"自我改进智能体系统"为双环架构**：内环（技能驱动的智能体会话）+ 外环（在持久数据基底上进行的 RSI 树搜索），并扩展到多智能体"科学社会"。将搜索策略（one-shot、linear、greedy、MCTS、evolutionary）统一为对同一基底的不同控制平面策略。
- **指出三层记忆（声明式/程序式/情景式）是不完整特例**：现有生产级智能体记忆只记录"知道什么"，缺少承载奖励的"经验图"。Trellis 通过融合图遍历、向量相似度与结构化过滤的查询层，将四层记忆统一到同一基底。
- **将智能体操作重构为数据库访问模式**：识别出 Resume/Reuse/Repair/Train/Replay/Observe/Audit/Govern 八种操作均为查询；并刻画了独特的访问模式（append + 局部路径更新混合写；有序扫描、多跳遍历、向量检索、全表扫描四模态读），现有 OLTP/OLAP/图库/向量库均不适用。
- **生产环境受控实证（KernelEvolve）**：跨会话记忆使达到 1.2× 加速快约 10×（5 步 vs 51 步），buggy 节点率从 55% 降至 34%(p=0.1)/21%(p=0.5)，每个有效节点 token 成本降低 52%；并揭示"探索—锚定"权衡。
- **提出数据库研究新议程**：多模态查询规划、并发树搜索一致性、物理设计、受治理的物化视图维护、bi-temporal 记忆、对经验图的学习式控制等。

## Key Contributions

- **Defines the self-improving agentic system as a two-loop architecture**: an inner loop of skill-driven agent sessions and an outer loop of recursive-self-improvement (RSI) tree search over a persistent data substrate that renders agents stateless and serverless; extends to multi-agent "scientific societies." Search strategies (one-shot, linear, greedy, MCTS, evolutionary) are unified as different policies of one control plane over the same substrate.
- **Argues the three memory tiers (declarative/procedural/episodic) are an incomplete special case**: production agent memory captures *what the agent knows*, missing the reward-bearing *experience graph* of what its search tried. Trellis unifies all four tiers through a query layer composing graph traversal, vector similarity, and structured filters.
- **Reframes agent operations as database access patterns**: identifies eight operations (Resume/Reuse/Repair/Train/Replay/Observe/Audit/Govern) all expressible as queries; and characterizes a novel access pattern (mixed append + localized multi-hop path writes; ordered scans, multi-hop traversals, vector similarity, and full scans for reads) that no existing OLTP/OLAP/graph/vector store is designed for.
- **Controlled production empirical grounding (KernelEvolve)**: cross-session memory reaches the 1.2× speedup target ~10× faster (≈5 vs 51 steps), cuts buggy-node rate from 55% → 34% (p=0.1) / 21% (p=0.5), and lowers token cost per valid node by 52%, while exposing an exploration–anchoring tradeoff.
- **Articulates a new database research agenda**: multi-modal query planning, consistency for concurrent tree search, physical design, governed view maintenance, bi-temporal memory, and learned control over experience graphs.

## Methodology

**Conceptual framing.** The paper positions long-horizon agentic tasks as a new database workload (in the tradition of "one size does not fit all" / C-Store). The core thesis: *search over experience graphs is a database access pattern*.

**Two-loop architecture (Fig. 1).**
- *Inner loop*: a single agent session (Claude Code, Codex, or any tool-using LLM) driven by a **skill** (a reusable task spec: what to do, how to evaluate, what counts as success). The pattern is fixed: consume task + context → generate candidate artifact via LLM → execute/evaluate in an isolated sandbox → persist structured output (logs, metrics, fitness) as a node. The agent then terminates; it holds no state.
- *Outer loop*: decides which node to expand next per a search strategy (one-shot/linear/greedy/MCTS/evolutionary, Fig. 2). It is task-agnostic—it reads fitness scores and selects frontiers through one data interface. Crucially, all strategies write the same `exploration_nodes` table; only the selection query and mutated fields differ.

**Data foundation (Trellis).** Relational schema with a four-level hierarchy: **Tasks → Sessions → Nodes → Prompt histories** (plus task-description embeddings for vector search; large artifacts live in object storage by reference). The schema is algorithm-, task-, and skill-agnostic.
- *Disaggregated storage/compute*: a stateless query/execution engine over durable storage. Current implementation uses **Axiom** (a cost-based optimizer) over **Velox** (Meta's vectorized engine), planning **SQL + Cypher + vector retrieval** into one physical plan, routing fragments to an operational store (sub-50 ms frontier queries), a vector index, and a columnar warehouse.
- *Context as managed state*: the `prompt_history` table lets a child node inherit a parent session (cheap resume) or reconstruct context via an ancestor query, avoiding context discontinuity.
- *Vector-seeded graph retrieval*: ANN over task embeddings → relational joins to sessions/nodes → variable-length Cypher traversal (`[:HAS_CHILD*1..k]`) to expand full trajectories, fused into one planned statement (one round-trip instead of four RPCs).
- *Multi-version state / time travel*: nodes are insert-once, never deleted; mutated fields (visit counts, UCB scores, island ids) are captured in a change log keyed by *logical step number* (evaluation order, not wall-clock) enabling AS-OF reconstruction.

**From memory to training (Section 5).** Training data is *queried, not collected*:
- **SFT trajectories** = root-to-leaf paths through non-buggy high-fitness nodes, each node read as-of its own step (to avoid leaking future-mutated statistics).
- **DPO preference pairs** = sibling graph-pattern matches (`a.fitness_score > b.fitness_score + m`).
- **GRPO** = sample a state from the persistent buffer, generate N children, compute group-normalized advantages via SQL window functions over cohorts, append one canonical node—collapsing multi-turn rollout cost to single-step expansions.
- **Value models** over graph features (ancestor rewards, sibling diversity, failure signatures, diffs, visit counts, budget) for learned control of the outer loop.

**Closed loop / flywheel:** production exploration populates memory → improves retrieval/frontier selection → traces train better models (SFT/DPO/GRPO) → better exploration → higher-quality states. Every serving run is simultaneously a training-data generator.

## Results & Highlights

**Evaluation setting (Section 6):** KernelEvolve (built on Trellis), optimizing accelerator kernels across NVIDIA, AMD, MTIA, and CPU; the foundation has accumulated a substantial corpus of experience graphs. Controlled study holds model, 100-step budget, worker count, and greedy search fixed; **3 independent sessions per configuration** to control for LLM-sampling variance; compared no-memory vs injection rate p=0.1 and p=0.5.

- **Crash recovery / statelessness**: database-native state means a crashed worker resumes from the last committed node—recovery is a frontier query with no checkpoint deserialization or warm-up; described as inherited "for free," not engineered.
- **Cross-session reuse (Figs. 3–4)**:
  - Reaches 1.2× speedup in ≈5 steps vs **51 steps** cold start → **~10× faster convergence**.
  - Buggy-node rate cut from **55% → 34% (p=0.1) / 21% (p=0.5)**; valid nodes meeting baseline speedup rise from 79.5% → 90.8% / 100%.
  - **52% lower token cost per valid node** (driven by avoided debug-retry loops on buggy nodes).
- **Exploration–anchoring tradeoff**: higher injection lowers variance/bugs but *anchors* search—p=0.5 collapses onto 8 strategy combinations vs 20 for cold start, and the single best solution (1.49×) comes from *no memory* (vs 1.36×). Balanced p=0.1 retains most of the convergence win while producing the most high-quality (≥1.3×) solutions.
- **Training data without re-rollout**: SFT/DPO/GRPO data extracted as queries; multi-turn agentic-RL rollout (hours-to-days per tree) collapses to trajectory reuse + single-step expansions.
- **Generality (Section 7)**: retargeted from kernel optimization to **MTIA silicon validation** (finding hardware bugs vs the golden reference) by changing *only* the fitness function and skills—the experience graph, query layer, orchestration, training views, and observability carried over unchanged.

## Critical Assessment

- **Novelty**: High *conceptually*, modest *technically*. The reframing—agent search history as governed database state; frontier selection/reuse/training/replay as queries—is a genuinely cross-disciplinary lens rather than a new algorithm or benchmark. Most building blocks (MCTS, evolutionary search, vector retrieval, materialized views, time travel, Velox/Axiom) pre-exist; the contribution is their unification and the workload characterization.
- **Rigor**: Moderate. The headline numbers (10×, 52%) come from a **single controlled study within one production system (KernelEvolve)**, averaged over only **3 sessions per config**—appropriate as "early measurements" but underpowered for strong claims given acknowledged high LLM-sampling variance. The exploration–anchoring tradeoff is honestly reported. No comparison against alternative memory/reuse systems (e.g., a generic vector-RAG baseline) is provided.
- **Reproducibility**: Low. Tightly coupled to Meta-internal stack (Axiom, Velox, KernelEvolve, production hardware). No public dataset/benchmark/code is indicated. The paper is explicitly a vision/systems-architecture piece (CIDR style), not a benchmark paper, so this is partly inherent to the venue.
- **Significance**: Potentially high *if* the thesis is adopted—could shape agent infrastructure (memory, reuse, crash recovery, RL data pipelines) and open a concrete DB research agenda. The flywheel framing aligns with industry momentum (AlphaEvolve, AI Scientist, Omnigent, etc.).
- **Red flags**: Vision-paper rhetoric ("durable institutional asset," "logs made databases reliable; experience graphs may make agents cumulative"); the empirical section is explicitly "preliminary" with "fuller evaluation left to future work"; generalization beyond kernels/silicon is *asserted* via retargeting rather than quantitatively shown across diverse domains.

## Strengths

- Unifies many agent-infrastructure pain points (memory, reuse, crash recovery, concurrent search, RL-data generation, governance/audit) under one coherent, well-motivated abstraction.
- Strong, credible author team straddling DB systems, ML, and production deployment.
- Honest about the central tradeoff (reuse accelerates but anchors), rather than overclaiming memory as universally beneficial.
- Backed by a real production system, not only toy benchmarks; plus a second deployment (silicon validation) demonstrating domain-agnosticism.
- Concrete, actionable research agenda genuinely suited to the data-management community.

## Limitations

- Empirical grounding is essentially a single production case study with small N (3 sessions/config); generalization to code generation / scientific discovery / security research is asserted, not measured.
- Lacks head-to-head comparison against existing agent-memory or retrieval baselines.
- Storage/query-latency overhead of maintaining time-travel change logs and multi-modal query planning is not quantified.
- Tight coupling to Meta-internal components (Axiom/Velox) limits external reproducibility.
- The "experience graph" concept, while compelling, is defined somewhat loosely; boundary between it and episodic memory could be sharper.

## Relevance

与智能体基础设施、长期记忆、自我改进（RSI）及测试时搜索/强化学习方向高度相关：本文为如何将智能体探索过程沉淀为可查询、可复用、可训练、可治理的数据资产提供了系统化方案，并提出了与之匹配的数据库研究议程，对构建可累积进化的智能体系统具有直接参考价值。

## Questions for Further Exploration

- How do storage footprint and query latency scale as experience graphs accumulate across thousands of sessions and tasks?
- What is the right isolation/consistency level for concurrent MCTS backpropagation, and does relaxed consistency measurably degrade search quality beyond a bounded factor?
- Does the closed-loop training flywheel actually improve agent quality on held-out tasks, or does reuse-induced anchoring cap asymptotic performance?
- How should injection rate be set adaptively (per-query, decayed, or selectivity/confidence-driven) to balance convergence vs diversity?
- Can the abstraction be decoupled from Axiom/Velox and evaluated on open, reproducible benchmarks against generic vector-RAG agent-memory systems?
- How are governance/access-policy retractions propagated through materialized training views when source nodes are invalidated?
