§ 01 Methodology

The AI Training Manager is an LLM (GPT-5.4-mini) supervisor that never touches parameters, code, or architecture — it only nudges the surrounding training recipe (LR, dropout, weight decay, loss weights, action std, entropy, safety penalty). Its candidate decisions are not applied directly: a deterministic verifier parses the JSON, enforces a declared action surface, and maps any malformed or out-of-bounds response to a safe no-op.

The manager sees a three-part prompt — task-family guidance, a problem description that deliberately hides the injected failure mode, and a bounded action surface — plus a runtime context of live telemetry, the current recipe, and a compact history of prior decisions and their observed outcomes. Crucially it proposes only bounded relative updates (increase / decrease / hold within limits), never arbitrary absolutes.

Two protocols keep LLM latency off the gradient critical path: an asynchronous non-blocking loop for supervised training (stale decisions silently discarded at the next safe boundary), and a boundary-synchronous loop for RL, where mid-update changes proved unstable. Every request emits a full, auditable decision trace.

§ 03 Strengths & Limitations

▲ Strengths

  • Principled bounded design: verifier-enforced schema, no-op fallbacks, auditable traces — directly defuses LLM-agent safety concerns.
  • Asynchronous, non-blocking integration keeps LLM latency off the gradient critical path.
  • Multi-axis control (LR, regularization, loss weights, exploration) beyond single-axis schedulers, across supervised + RL.
  • Cleanest evidence: one prompt → opposite-direction intervention on scared vs reckless runs.
  • Honest limitations section that names its own key confounds.

▼ Limitations

  • Tiny scale, one environment per domain; large-pretraining / realistic-RL transfer unproven.
  • Apparent single-seed; no variance, no statistical significance.
  • Structured-memory confound: LLM reasoning vs memory unseparated (authors flag this).
  • Auxiliary dialogue task saturates early → multi-loss claim only weakly tested.
  • Async did not transfer to RL; needed boundary-synchronous fallback.
  • No formal optimality / safety guarantees; injected failure modes ≠ real long-tail pathologies.
§ 04 Open Questions
01 LLM reasoning vs structured memory — which actually drives the gains? An ablation isolating the two is the single most important missing control.
02 Does recovery (0.852→0.770, 0.0→0.94) survive multiple seeds, or does LLM non-determinism inject variance?
03 How does it scale beyond 6-layer GPT / 2k stories to real pretraining, and what is the inference-cost-to-training-cost ratio?
04 Can bounded interfaces reach formal safety guarantees (Lyapunov / shielded RL) instead of empirical bounding?
05 Would harder, non-saturating auxiliary objectives (entity / factual / style consistency) expose failures the dialogue-head test hides?
06 How brittle is the manager to prompt / schema design, telemetry features, and the underlying LLM version? Can async be stabilized for RL via decision-versioning over rollout buffers?
