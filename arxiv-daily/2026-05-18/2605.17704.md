> **一句话总结：** 本文在组合DNF可解释玩具模型中证明，中奖彩票（winning tickets）不是简单的权重空间稀疏子网络，而是特征空间中初始化时已接近最终特征编码的"前驱脚手架"（scaffold），稠密SGD通过超位置竞争选择兼容的编码位置族，且基于特征空间距离和运动的轻量探针在早期编码预测上优于传统权重剪枝方法。

> **TL;DR:** This paper reframes the lottery ticket hypothesis in feature-space terms using a combinatorial DNF toy model where internal representations are directly interpretable. Winning tickets correspond to precursor scaffolds in feature space — families of compatible code locations already near final feature-channel codes at initialization. Dense SGD resolves these through structured selection under superposition competition, and mechanism-inspired feature-space probes outperform weight-based methods (SNIP, GraSP, SynFlow, magnitude pruning) for early ticket discovery.

## 核心贡献

- 在组合DNF（析取范式）玩具模型中建立了完全可解释的特征空间表示，定义了基于汉明距离的编码距离度量（dτ），可以精确追踪特征编码的演化过程
- 证明中奖彩票在特征空间中对应初始化时已接近最终特征通道编码（4P/3N1P模式）的前驱位置，而非仅仅是权重空间中的大权重子集
- 揭示稠密SGD的超位置竞争机制：候选编码位置在拥挤行（high-load rows）上被优先拒绝，说明SGD在求解全局的特征放置问题而非局部优化
- 证明剪枝改变了整个特征空间表示（C₁ = W₁C₀），而非仅删除部分权重——掩码创建了全新的特征空间初始化状态
- 通过固定参数预算实验证明OBS彩票支持恢复约53%的稠密编码身份（vs 随机稀疏扩展仅20%），Jaccard重叠仅约10%说明两者选择了完全不同的编码位置
- 设计了基于特征空间距离和运动的机制探针，在广泛的参数扫描中（宽度16-32、子句8-32、稀疏度50%-90%）在准确率和精确编码恢复上均优于权重空间基线（SNIP、GraSP、SynFlow、magnitude pruning）
- 发现彩票保持的对象是族级别（family-level）的编码结构，而非微观行身份——稀疏重训练常在不同行上重新表达相同的子句/模板族

## Key Contributions

- Establishes a fully interpretable feature-space representation in a combinatorial DNF (Disjunctive Normal Form) toy model with Hamming-distance-based code distance metrics (dτ), enabling precise tracking of feature-code evolution during training
- Demonstrates that winning tickets correspond to precursor locations in feature space already near final feature-channel codes (4P on positive-output rows, 3N1P on negative-output rows) at initialization — not merely large-weight subsets in parameter space
- Reveals dense SGD's superposition competition mechanism: candidate code locations are preferentially rejected on crowded (high-load) rows, showing SGD solves a global feature placement problem rather than merely pushing every near-code toward exactness
- Proves that pruning changes the entire feature-space representation (C₁ = W₁C₀), not just deletes weights — the mask creates a new projected feature-space initialization state
- Shows via fixed-budget experiments that OBS lottery supports recover ~53% of dense code identities (vs ~20% for random sparse expansion), with Jaccard overlap ~10% indicating fundamentally different code complexes
- Introduces mechanism-inspired probes (static distance, dynamic motion, combined) that outperform weight-space baselines (SNIP, GraSP, SynFlow, checkpoint magnitude) across broad sweeps (width 16–32, clauses 8–32, sparsity 50%–90%)
- Identifies that the preserved object in a lottery ticket is family-level code structure rather than microscopic row identity — sparse retraining often re-expresses the same clause/template family on different rows

## Methodology

The authors study lottery tickets in a clause-structured toy DNF model (introduced by Adler et al., 2025) where the internal feature space is directly readable rather than opaque.

**Model architecture:** A two-layer ReLU network with a learnable linear projection C₀ simulating prior layers. The network computes DNF (Disjunctive Normal Form) tasks with clause size four. The key analytical object is C₁ = W₁C₀ ∈ ℝ^(h×d_in), the feature-space clause-local representation that disentangles functional structure from sparsity in W₁.

**Feature-space framework:** The authors define task-aligned code families — 4P (all positive) patterns on positive-output rows and 3N1P (one positive, three negative) patterns on negative-output rows — as the canonical feature-channel codes. They introduce combinatorial distance dτ(u, T) counting sign defects from exact codes and margin m(u, T) measuring stability.

**Experimental design (six main components):**

1. **Ticket cycle visualization** (Fig. 1): Traces one representative run through dense training → masked final → dense initialization → sparse rewind → sparse retraining, showing that applying the mask to initialization unexpectedly creates more codes by removing smaller weights.

2. **Dense SGD analysis** (Sec. 5): Categorizes locations into oracle-supported stable codes, oracle-supported lost locations, recruited final codes, and close-but-lost locations. Shows rejection concentrates on denser rows.

3. **Fixed-budget sparse expansion** (Sec. 6): Compares 16×16 dense baseline (256 params), random 32×16 50% sparse expansion (256 params), and OBS lottery ticket (256 params) — isolating the contribution of support structure from parameter count.

4. **Code identity overlap analysis**: Uses strict same-site recall and Jaccard overlap to show OBS tickets and random expansion produce fundamentally different code complexes.

5. **Initial precursor diagnostics**: Conditions near-code fractions on eventual final code sites, demonstrating OBS masks place precursors at the right locations (69–79% near-codes at own final sites vs ~55% for random).

6. **Broad diagnostic sweep** (Sec. 8): Sweeps hidden width {16, 32}, clause counts {8, 16, 32}, sparsities {50%, 75%, 90%}, and probe epochs {0–20}, comparing feature-space rules against SNIP, GraSP, SynFlow, Early-Bird magnitude, and checkpoint magnitude.

## Results & Highlights

- **Fixed-budget hierarchy:** Random sparse expansion (acc 0.736) < OBS ticket from init (0.767) ≈ OBS ticket rewind (0.769) ≈ full dense 32×16 reference (0.771), all at the same 256-parameter W₁ budget
- **Code preservation:** OBS tickets recover 52.8–54.3% of dense code identities vs 20.0% for random expansion; Jaccard overlap between random and OBS tickets is only ~10%
- **Precursor placement:** At initialization, 69.3% (OBS-ticket-init) and 78.9% (OBS-ticket-rewind) of eventual final code sites are already near-codes, vs 54.6% for random expansion
- **Feature-space probes win early:** Combined static+dynamic feature-space rules lead on accuracy; static distance and dynamic motion lead on exact-code recovery — all outperforming weight-space baselines for early code prediction
- **Weight magnitude is a "shadow":** Checkpoint magnitude catches up in accuracy at later epochs precisely because dense training has projected feature-channel organization into W₁ magnitude ordering
- **Scaling:** Appendix B experiments with width up to 256 show the qualitative story transfers to larger settings within this model class
- **Mask creates codes:** Applying the final mask to random initialization unexpectedly increases code count and accuracy (e.g., 0.856 from 0.978→0.725 masked final), providing a feature-space explanation of the Supermask phenomenon (Zhou et al., 2019)

## Critical Assessment

- **Novelty:** The reframing of lottery tickets from weight-space masks to feature-space scaffolds is a genuine conceptual advance. The combinatorial distance framework and the superposition-competition mechanism for SGD are novel contributions. However, the core insight is tested exclusively in one specific toy model family.
- **Rigor:** The experimental design is strong — fixed-budget comparisons isolate support structure from parameter count, code-identity overlap analysis distinguishes structural preservation from functional equivalence, and the broad sweep tests robustness across hyperparameters. The separation between "probe as diagnostic" and "probe as pruning algorithm" is methodologically sound. Weakness: no error bars or statistical tests reported for several key comparisons in the main text (though Tables 1–4 report SEM).
- **Reproducibility:** The toy DNF setting with explicit code definitions should be straightforward to reproduce. The probe rules are clearly defined. The paper provides an appendix roadmap with implementation details. Lack of official code release is a minor concern.
- **Significance:** If the feature-space scaffold insight transfers to realistic architectures, it could shift how the community understands lottery tickets — from weight-space pruning to feature-space structure preservation. The superposition-competition mechanism is a concrete, testable hypothesis for general networks. However, the gap between 4P/3N1P codes in Boolean DNF detectors and features in transformers is enormous. The authors acknowledge this honestly.
- **Red flags:** (1) The "toy setting" limitation is fundamental, not cosmetic — the entire feature-space framework relies on knowing task-aligned code families in advance, which is precisely what's unavailable in general networks. (2) The feature-space probes' advantage over magnitude pruning narrows substantially at later epochs, suggesting the practical benefit may be limited to very early training. (3) The translation problem (feature-space locations → sparse W₁ coordinates) is acknowledged but unresolved, which is a significant gap. (4) No comparison to SAE-based feature discovery as a potential bridge to realistic models.

## Strengths

- Exceptionally clear and honest about limitations — the Folkman analogy sets the right expectations
- Clean experimental methodology with fixed-budget comparisons that isolate causal factors
- The feature-space scaffold concept is precise and testable, not just metaphorical
- Visual explanations (Fig. 1 walkthrough) make the complex setup accessible
- Broad diagnostic sweep provides strong empirical support across configurations
- Family-level preservation insight adds important nuance beyond standard LTH

## Limitations

- All results are in a Boolean DNF toy model — no demonstration of transfer to any standard architecture (MLP on MNIST, ResNet on CIFAR, transformer on language)
- The combinatorial code families (4P, 3N1P) are task-specific and require knowing the ground-truth clause structure in advance
- The translation problem from feature-space locations to weight-space masks remains only partially solved
- Feature-space probes' advantage diminishes at later training epochs when magnitude-based methods catch up
- No connection to SAE-based or other post-hoc feature discovery methods that could bridge to realistic models
- The paper does not establish whether feature-space geometry governs ticket structure in any setting beyond this specific model family

## Questions for Further Exploration

- Can SAE-derived features in transformers serve as approximate code families for a feature-space ticket analysis, even without knowing ground-truth task structure?
- Does the superposition-competition mechanism (rejection on high-load rows) manifest in transformer attention heads or MLP layers during training?
- How does the feature-space scaffold picture interact with grokking dynamics — does the scaffold form during the memorization phase and get cleaned up during generalization?
- Could the family-level preservation insight explain why tickets found via magnitude pruning transfer across tasks in multitask settings?
- What is the minimal feature-space information needed to discover tickets in a standard architecture — could probing at initialization predict trainability without any dense training?
