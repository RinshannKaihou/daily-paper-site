01 Methodology · three pillars

GKVC

Global KV Cache

A pre-allocated 3D static buffer in L3. Each decode step writes fresh K/V by an address-jump stride — no Concat, no CPU sync, no realloc. Keeps the whole pipeline on the accelerator, unblocking downstream fusion.

Step = h·(Nₚ+Nᵣ)·d + Nₚ·d
TPSFLAT

Fused L1 Attention

FLAT-derived dataflow specialized for the token phase (B=N=1). Two-level H-tiling; one shared RoPE call; RoPE + QKᵀ + Softmax + PV fused continuously inside L1. Only the KV the cache needs ever touches L3.

QK ∈ R^{2×B×N×1×d} · OI ×3
FUSION MLP

Interlaced Gate/Up

Gate & up weights split along K and interleaved into one Wgu ∈ R^{D×2K} Matmul; SiLU + elew-Mul fused into a single siLu·Mul. L1↔L3 round-trips cut 7 → 2; a pipeline schedule overlaps weight load with compute.

I/O: 7 → 2 · ~1.3× data volume

03 Assessment · strengths vs. limitations

Strengths

  • Clean three-pronged design — each piece (GKVC / TPSFLAT / Fusion MLP) addresses a distinct bottleneck and ties to a measurable roofline or IOI metric.
  • Genuinely strong accuracy — <0.22% max BF16 error that improves with sequence length; ~10× edge over FlashAttention.
  • Auditable — Algorithms 1–4 plus a transparent cache-usage table (Table 4) make the dataflow checkable even with no released code.
  • Real silicon — deployed evaluation on AMD NPU hardware with cross-vendor MLPerf numbers in the appendix, not a simulator.
  • Composable — explicit GQA compatibility (≤7.2% overhead) and a quantization integration plan.

Limitations

  • Weak baselines — HuggingFace eager / unfused kernels are not competitive NPU-optimized stacks; no head-to-head vs. FlashAttention-3, PagedAttention, or vLLM-class serving.
  • CPU cost omitted — measured while excluding CPU overhead, which biases the GKVC comparison (the Concat-removal win).
  • Decode-only, B=1 — no prefill, no batching, no speculative decoding: the regime where real serving systems live.
  • Hardware scope — evaluated only on AMD XDNA NPU; GPU/TPU/ASIC portability is asserted, not demonstrated.
  • Missing content — Table 8 referenced but absent; no perplexity/downstream quality, only max-element error. No code release.
  • Vendor framing — all authors are AMD employees; cross-vendor tables are selective despite AMD losing several absolute-TPS comparisons.

04 Open questions · for further exploration

How does FastTPS behave under continuous batching / B>1 and the prefill phase, where the N=1 assumption and per-token scheduling break down?
What is the end-to-end latency including CPU overhead the paper excludes — does GKVC's win survive once CPU scheduling is counted for the unfused baseline?
How does TPSFLAT's speed (not just accuracy) compare vs. FlashAttention-3 / FlashInfer / PagedAttention on the same NPU via an open compiler stack?
Can the interlaced gate-up layout be retrofitted to existing weights without retraining (offline repack cost?), or does it need a one-time transform?
Do the ≤0.22% error guarantees hold at long contexts (32k+, where FlashAttention degrades) and for the INT8/INT4 variants promised as future work?
Is the cross-vendor gap (AMD trailing Nano / Lunar Lake on Llama2/Llama3 absolute TPS) closable with FastTPS alone, or is it a fundamental XDMA bandwidth limit?
