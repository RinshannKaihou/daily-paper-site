> **一句话总结：** StructAgent 提出"统一状态 + 结构化工作流"的因果框架，让规划/执行/验证三方共享同一份可审计的任务进度状态，仅当验证器给出证据支撑的判定时才提交进度，在 OSWorld-Verified 上将 Qwen3.5-9B 从 27.0% 提到 46.9%、MiniMax-M3 达到 78.9% 的开源新 SOTA。

> **TL;DR:** StructAgent is a state-centered agent framework that maintains a compact, typed, verifier-maintained task state `(s_req, s_val, s_ver)` and a planner→actor→verifier loop where only verifier-backed decisions commit, preserve, or invalidate progress. On OSWorld-Verified it lifts Qwen3.5-9B from 27.0%→46.9% (+19.9), Qwen3.5-27B from 31.6%→62.2% (+30.6), and MiniMax-M3 from 75.2%→78.9% to a new open-source SOTA, beating reproduced Agent S3 / OS-Symphony / VLAA-GUI frameworks under matched runner settings, and the same design transfers to Mind2Web (52.3% on 27B) and Minecraft (76% vs Optimus-1's 59%).

## 核心贡献

- **统一状态（Unified State）：** 用 `s_t = (s_t^req, s_t^val, s_t^ver)` 三元组分别记录"当前子目标需求 / 已发现的有用值（文件路径、URL、选中实体等）/ 已验证的证据"，作为规划、执行、验证三个模块共享的唯一读写接口，只保留最小充分信息。
- **结构化工作流（Structured Workflow）：** 固定的 Planner→Actor→Verifier→State-update 循环；每个需求带 `PENDING/VERIFIED/INVALIDATED` 状态，只有验证器判定的证据才能改写状态，从而形成透明的因果链。
- **以验证为核心的因果转移：** 规划器和执行器的"DONE"声明只是提案而非进度更新，杜绝虚假进度提交；当状态无法推进时基于 `s^ver` 做定向恢复（重规划/重试/修复环境/人工接管）而非泛化重试。
- **广泛实证验证：** 在 OSWorld-Verified 上达到开源新 SOTA（MiniMax-M3 78.9%），并证明同一套"状态-验证"范式可跨域迁移到 Web（Mind2Web）与游戏（Minecraft，基于物品栏验证）。
- **系统化的失效分析：** 对 83 条零分轨迹按"主责模块 + 失效类型"归因（Actor 33%、Planner 30%、Verifier 30%、环境 7%），并对照评估结构化验证相对纯视觉验证的增益（召回 0.67→0.81）。

## Key Contributions

- **Unified agent state.** A compact, typed, and auditable state `(requirements, useful values, verified evidences)` that gives planning, acting, and verification a shared, minimum-sufficient representation of task progress instead of forcing each module to re-derive it from raw multimodal history.
- **Structured workflow with verifier-committed transitions.** A planning-acting-verification loop in which planner/actor completion claims are *proposals*; only verifier-backed decisions can commit, preserve, or invalidate state, making every progress update causally traceable to evidence.
- **Controllable long-horizon capabilities that fall out of the design.** Progress checkpointing/resuming (state history is a reusable snapshot), targeted failure recovery from `s^ver`, and tool/skill-supported execution whose outcomes are still only committed through the verifier.
- **New open-model SOTA and cross-domain generalization.** 78.9% on OSWorld-Verified with MiniMax-M3 (open-source SOTA), plus instantiations on Mind2Web (web) and Minecraft (inventory-verified embodied), showing the framework is not tied to desktop screenshots.
- **Diagnostic evidence on *why* it works.** Isolated verifier study (recall 0.67→0.81, F1 0.70→0.76 on 354 DONE-gate decisions) and a per-module failure audit of 83 score-zero trajectories, identifying state-verification errors as the single largest remaining failure type.

## Methodology

StructAgent replaces continuous generation over a growing raw history with a state-centered control loop built on two complementary designs.

**1. Unified agent state.** The state is a typed triple `s_t = (s_t^req, s_t^val, s_t^ver)`:
- `s_t^req` — current requirements of the active subgoal (what must be achieved and what the verifier should check); each requirement carries a status `σ ∈ {PENDING, VERIFIED, INVALIDATED}`.
- `s_t^val` — durable useful values discovered during execution (file paths, URLs, selected entities, extracted fields) needed by downstream steps.
- `s_t^ver` — verified evidences collected by the verifier, so later turns can audit and reuse them.

A *causality-oriented state transition* rule guarantees that each requirement status changes only in response to an explicit verifier decision `d_t` (verify, invalidate, or leave unchanged). If the verifier also surfaces useful values, they are written into `s_{t+1}^val` together with their supporting evidence in `s_{t+1}^ver`.

**2. Structured workflow.** A fixed loop `s_t —Planner→ g_t —Actor→ τ_t —Verifier→ d_t —Update→ s_{t+1}`:
- *Planner* reads the state (not raw history) and picks the next subgoal `g_t`.
- *Actor* receives `g_t` plus the current state, predicts an action trace `τ_t` (GUI clicks/typing on a 0–1000 grid, browser DevTools navigation, host/VM `cli_run`/`edit_json`, or structured Office actions backed by UNO), and executes it.
- *Verifier* probes task-relevant evidence and emits the state-update decision. It can either judge from observation or request one of seven typed `VerifySpec` kinds — `file_grep`, `url_match`, `a11y_match`, `calc_verify`, `writer_verify`, `impress_verify`, `shell_command` (read-only guarded) — each gated by a per-domain trust map.
- *Memory* is explicitly separated from state: it stores reusable experience (plans that worked, pitfalls, verifier recipes) mined offline from CUA-Gym rollouts, OpenCUA desktop trajectories, and Multimodal-Mind2Web web trajectories; it can *suggest* but never commit progress.

**3. Controllable long-horizon capabilities.** Because progress lives in the state, StructAgent gets (a) checkpointing/resuming without replaying history, (b) targeted recovery diagnosed from `s^ver` (insufficient evidence / conflicting requirement / repeated failure / environment block → replan, retry, repair, or human takeover inserted as a new subgoal), and (c) tool-supported execution where recurring patterns are summarized into typed procedures whose outcomes still require verifier commitment.

## Results & Highlights

- **OSWorld-Verified overall success rate (controlled, matched runner, 100-step budget):** StructAgent improves **Qwen3.5-9B from 27.0% to 46.9% (+19.9)**, **Qwen3.5-27B from 31.6% to 62.2% (+30.6)**, and **MiniMax-M3 from 75.2% to 78.9% (+3.7)**; the 78.9% figure is the reported new open-source SOTA, ahead of reproduced Agent S3 (27B: 60.2%), OS-Symphony (27B: 52.8%), and VLAA-GUI (27B: 48.2%) under the same backbone, and ahead of reported framework references VLAA-GUI/Opus 4.5 (76.3%) and Agent S3/Opus-GPT bBoN (72.6%).
- **Per-domain Office breakdown (Qwen3.5-9B, StructAgent vs single-model baseline):** Calc jumps **8.5% → 59.6%** and GIMP **26.9% → 61.5%**, the clearest evidence that durable intermediate state (files, cells, app checks) is where verifier-maintained progress pays off most.
- **Verification quality (isolated on 354 Qwen3.5-27B OSWorld-Verified trajectories, OSWorld checker as ground truth):** structured verification raises **recall 0.67 → 0.81** and **F1 0.70 → 0.76** while keeping precision near the visual-only verifier (≈0.72–0.73); in a 100-case disagreement review, structured verification is correct in **63%** vs **31%** for visual-only (6% unclear), with gains coming mainly from latent evidence (files, command outputs, URLs) not visible in the final screenshot.
- **Web generalization (Mind2Web, WebJudge-style protocol):** StructAgent reaches **47.3% (9B)** and **52.3% (27B)** overall, both leading their controlled backbone group (strongest reproduced baseline Agent S3 at 46.0% / 51.7%); gains concentrate in information-seeking tasks.
- **Cross-domain (Minecraft, GPT-4o backbone, inventory-verified):** StructAgent achieves a **76%** task-weighted average success rate vs **59%** for published Optimus-1, with large gains on long-dependency tiers — Iron **65% vs 47%**, Golden **86% vs 9%**, Redstone **57% vs 25%** — though Optimus-1 still wins Stone (92 vs 70).
- **Failure audit (83 score-zero 27B trajectories, ≤20/domain):** remaining failures are distributed across modules — Actor 33%, Planner 30%, Verifier 30%, Environment 7% — with *state verification errors* (accepting weak evidence, missing a completed condition, or failing to detect invalidated earlier progress) the single largest failure type, arguing against the view that reliability is only an actor-grounding problem.
- **Test-time scaling (Behavior Best-of-N, optional, not a core component):** going from N=1 to N=5 rollouts with structured candidate-level evidence lifts overall from 57.0% to 67.6% on the studied subset, with most gain early (N=3 already 63.7%).

## Critical Assessment

- **Novelty:** The individual ingredients (planner/actor/verifier roles, environment-state verification, skill memory) all exist in prior work — Agent S3, OS-Symphony, VLAA-GUI, OpenComputer, GUI-Shepherd, Voyager. The genuine contribution is the *fusion*: making verification the sole committer of a *shared typed state*, so that memory, tools, planner, and actor all read/write through one verifier-maintained causal surface. The paper's own Table 4 honestly shows this is a combination column (StructAgent is the only row with all six design flags `Plan/Ver/Rec/T/S/SS/VC` checked), which is a moderate-but-real architectural novelty rather than a new component.
- **Rigor:** Strong on the OSWorld-Verified main result (controlled matched-runner protocol, full per-domain breakdown in Table 5). Weaker on ablation: the paper studies BBON scaling and a Minecraft memory-retrieval ablation (F.5: sequential vs combined retrieval, 0/17 → 11/17 Iron passes), but there is **no clean component ablation of the unified state itself** on OSWorld — e.g., turning off `s^val`, or allowing actor-committed updates — so the reader cannot fully attribute the 27B +30.6 gain to "verifier-committed state" vs "better verifier" vs "better memory." The Q2 verifier isolation (recall 0.67→0.81) partially addresses this but only at the DONE gate.
- **Reproducibility:** Good. Fixtures are explicit (1920×1080, 100 steps, single rollout unless BBON, aliases for planner/actor/verifier/judge), runner artifacts are listed (Table 13), verifier shell probes are read-only guarded, and the Minecraft failure modes (Table 12) are named with deterministic signatures. However, memory-bank construction depends on non-public internal CUA-Gym rollouts and normalized OpenCUA/Mind2Web trajectories whose exact contents are not released, and the project page / code link is referenced but the artifact URL is not embedded in the PDF text.
- **Significance:** The 78.9% open-source SOTA on OSWorld-Verified is meaningful, but note it is built on MiniMax-M3 — a frontier open model whose own single-model baseline is already 75.2%. On the same backbone the StructAgent delta is only **+3.7**, and on the hardest Multi-Apps group with the weaker Qwen3.5-9B, StructAgent (27.5%) still trails OS-Symphony (32.6%), so the framework's leverage shrinks as the backbone gets stronger and as the task becomes more about cross-application dependency reasoning than single-app verification.
- **Red flags / quirks:** (1) Mind2Web is judged by **Qwen3.5-27B itself** as the WebJudge model — the same family used as a backbone — and Table 6 shows an 86% human-agreement / 4.7pp SR gap, so web numbers should be read as relative-only. (2) Mind2Web gains are marginal in several categories (Entertainment 9B: StructAgent 24.1% vs OS-Symphony 31.0%; Shopping 9B: StructAgent 36.4% vs VLAA-GUI 52.6%), undercutting the "generalizes to web" claim for dynamic-site tasks. (3) The Minecraft instantiation reuses the Optimus-1 codebase and its GUI-blind JARVIS-1 craft helper, whose "multi-cell craft retry storm" accounts for most Iron+ failures — explicitly orthogonal to the framework contribution but it does mean the 76% number is capped by a borrowed actor rather than a clean test. (4) Several cited "2026" arXiv references (VLAA-GUI 2604, OS-Symphony 2601, OpenComputer 2605) are future-dated relative to a Jul 2026 submission, which is normal for the venue timeline but worth noting for citation tracing.

## Strengths

- Principled, cleanly stated architecture: the `(s^req, s^val, s^ver)` triple plus verifier-only-committed transitions is easy to reason about and genuinely fixes the "raw-history soup" problem for long-horizon tasks.
- Large, consistent gains on weak/mid open backbones (9B +19.9, 27B +30.6), exactly the regime where structured state should help most.
- Honest evaluation hygiene: controlled matched-runner comparisons, reported-vs-self-run separation, isolated verifier study with precision/recall/F1, and a human-style failure audit instead of aggregate-only numbers.
- Cross-domain instantiation (Minecraft inventory verification) demonstrates the *abstraction* — verifier-committed progress is not bound to screenshots — which is the more durable claim beyond the leaderboard number.
- Rich engineering disclosure: the seven `VerifySpec` kinds, per-domain trust maps, read-only shell guards, and typed `VerificationRecord`/`Outcome` objects make the system concrete rather than notional.

## Limitations

- No clean on-OSWorld ablation isolating the unified state from the improved verifier or the memory module; attribution of the headline gains is therefore underdetermined.
- Gains compress sharply at the strong-backbone / cross-application frontier (MiniMax-M3 +3.7 overall; 9B Multi-Apps still behind OS-Symphony), indicating the framework does not solve inter-app dependency reasoning.
- Web (Mind2Web) results are modest and partly judge-dependent (same-family judge, 86% human agreement), with regressions on Entertainment/Shopping where success hinges on dynamic site flows.
- Minecraft results are bottlenecked by a borrowed GUI-blind craft helper (JARVIS-1) and the STEVE-1 actor's ~6s spatial-memory limit, capping the demonstration; the authors acknowledge these are orthogonal to the framework claim.
- Cost/latency and token overhead of running a separate planner + actor + agentic verifier per subgoal are not reported, so the price of the extra reliability is opaque.
- Memory bank depends on internal CUA-Gym rollouts whose contents are not publicly released, weakening exact reproducibility of the memory component.

## Relevance

高度相关于长程数字智能体（computer-use / GUI agent）方向：其"统一可验证状态 + 验证器独占提交"的范式直接回应了长 horizon 任务中进度难追溯、易误判完成的痛点，可作为评估和改进现有 CUA 框架（如 OS-Symphony、VLAA-GUI、Agent S）状态管理的对照基线与设计参考。对关注智能体可靠性、过程验证与多步规划的研究者，OSWorld-Verified 上的开源新 SOTA 与详细的失效归因具有直接的参考价值。

## Questions for Further Exploration

- What is the per-component ablation on OSWorld-Verified — how much of the 27B +30.6 comes from verifier-committed state vs. the structured verifier probes vs. the memory module vs. recovery?
- How does StructAgent's token/latency cost compare to single-model and to Agent S3/OS-Symphony at equal success rate, and does the verifier overhead amortize on longer tasks?
- Can the unified state schema learn adaptive `VerifySpec` selection (currently a fixed per-domain trust map) instead of hand-authored probe kinds?
- Why does the framework *regress* on 9B Mind2Web Entertainment/Shopping, and would a browser-domain-specific verifier (DOM/XHR probes) close the dynamic-site gap?
- Does the state-commit discipline compose with RL-trained actor/verifier backbones, or does it mostly help prompted VLMs — i.e., does a trained MiniMax-M3 still gain +3.7 if it is also trained to emit evidence?
- Could the verifier-invalidates-progress path (`INVALIDATED`) be exposed as a training signal to learn better long-horizon credit assignment, rather than only used at inference?
