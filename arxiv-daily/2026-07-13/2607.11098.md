> **一句话总结：** AgentCheck 是一个开源 Web 工作台，把 MCP 服务器变成可干预的故障注入面——它在同一缓存下记录干净运行、注入单一工具响应故障后重放、再在有缓解措施下重放同一故障，从而形成"复现—干预—确认"闭环；在 120 个场景、12 种故障类型上，最强 agent DeepSeek 通过 105/120，最弱 Llama 仅 77/120，重试缓解把 Llama 的超时故障成功率从 3/10 提升到 10/10，但陈旧数据类故障在所有缓解下仍停滞在 3–4/10。

> **TL;DR:** AgentCheck is an open-source browser workbench that turns any MCP server into a controlled fault-injection surface, running clean, faulted, and mitigated passes over a shared response cache to form a reproduce-intervene-confirm loop. On a suite of 120 scenarios across 12 tool-fault types, the best agent (DeepSeek, ReAct) passes 105/120 while the weakest (Llama-3.3-70B) passes only 77/120 — a 28-point spread — and a retry wrapper lifts Llama's timeout-fault success from 3/10 to 10/10 while stale-data faults stay stuck at 3–4/10 no matter the mitigation, exposing silent data-quality failures as the dominant, unmitigable weakness that happy-path benchmarks (GAIA, SWE-bench, ToolBench) cannot see.

## 核心贡献

- **可控故障注入工作台：** 首次把 MCP 服务器作为干预面，在共享的工具响应缓存上做"干净运行 → 故障重放 → 缓解重放"三向对照，使开发者能在自己的 agent 上复现一个具体故障、打补丁、再用同一故障验证修复是否生效（`fix_confirmed`）。
- **三族十二类故障分类法：** 把工具故障组织为 A 工具执行（超时/API 错误/权限拒绝/Schema 漂移）、B 数据质量（陈旧/矛盾/答非所问/空结果）、C 安全（提示注入/描述投毒/虚假声明/数据外泄），每类配确定性的 pass 准则。
- **120 场景库（含来源标注）：** 36 个改编自 ToolMisuseBench、20 个改编自 MCPTox、64 个新撰写，覆盖金融/健康/地理政治/代码技术/常识五个领域，符合发布 schema 即自动发现。
- **双层评分与人类对齐验证：** 主判定用确定性规则，辅以 LLM 裁判（Claude 4.5 Haiku）做诊断标签，并在 224 个标注实例上与两位人类标注者对齐（scorer-vs-human Cohen's κ 0.66–0.87）。
- **可比较的 agent 画像与缓解实验：** 在固定相同响应下评测五个配置，证明数据质量类故障是所有 agent 的共同软肋；并在最弱的 Llama 上系统比较 Retry/Schema/Injection-filter/全栈缓解，量化每种故障的可修复边界。

## Key Contributions

- **An MCP-native intervention surface.** AgentCheck sits between agent and tools, caching every response and replaying identical responses while changing exactly one at a chosen injection point, so any behavioral divergence is attributable to the injected fault rather than environment noise.
- **A reproduce-intervene-confirm loop.** After a faulted run fails deterministic checks, a developer wraps the agent with a mitigation and re-runs against the identical `(p, g)` fault; the system returns a deterministic `fix_confirmed` verdict (Algorithm 1), closing the loop that offline benchmarks and trace monitors leave open.
- **A 3-category / 12-fault taxonomy with executable pass criteria** (Table 2), distinguishing infrastructure-level faults (A), silent data-quality faults (B), and adversarial security faults (C) — with the key insight that Category B carries no error code and therefore slips past happy-path benchmarks.
- **A validated, open 120-scenario suite** (Table 3) with documented provenance (ToolMisuseBench, MCPTox, newly authored), auto-discovered via a published schema.
- **A two-tier scoring design** (deterministic primary checks + LLM-judge diagnostic labels) with human-alignment evidence, plus a comparative demonstration that retry mitigates execution faults but data-quality faults resist output-layer fixes.

## Methodology

AgentCheck's core is **controlled comparison over a shared cache**. Given an agent `A`, task `q`, tools `T`, fault `(p, g)` (injection point + injector), and optional mitigation `m`:

1. **Clean run** `τ_clean = RUN(A, q, T)` records every tool response into a cache `K: (tool, args, i) → r`.
2. **Faulted run** replays via `REPLAY(tool, args, i)`, returning `g(r)` only when `(tool, i) = p` and the cached `r` otherwise — so every tool call except the targeted one is byte-identical across runs.
3. **Mitigated run** wraps the agent as `m(A)` and re-applies the same fault, then checks whether previously-failed checks now pass.
4. The harness records the first divergence point between clean and faulted trajectories and applies fault-specific `CHECKS(p, g)`.

The system has four components: (i) an **Agent Harness Layer** supporting both iterative ReAct-style reason-act loops (Yao et al., 2022) and native tool-calling loops, both routed through AgentCheck so it can record and perturb responses; (ii) a **Fault Injection Engine** implementing the 12 fault types (full specs in Appendix A — e.g., A1 delays past timeout threshold, A2 returns 5xx, B1 swaps in a stale value, C4 instructs exfiltration to an attacker URL); (iii) **Primary Pass/Fail Checks + Diagnostic Labels**, where the primary checks are deterministic and pattern-bound, and the LLM judge (Claude 4.5 Haiku) produces three interpretive labels — Failure Detection, Recovery Action (recovered/safe_abort/propagated/crashed), Uncertainty Communication — used only as a secondary diagnostic; (iv) an **Interactive Dashboard** that places clean/faulted/mitigated runs side-by-side and lets the developer toggle mitigations.

Five agent configurations were evaluated (Appendix D, Table 6), all at temperature 0 with a 10-step cap on the bundled suite so every agent sees identical fixed responses: Gemini-2.5-Flash under both zero-shot and ReAct harnesses (to isolate harness effect), DeepSeek-V4-Pro (ReAct), Llama-3.3-70B-Instruct (ReAct), and GPT-4.1-mini (ReAct). Mitigation wrappers tested on the weakest agent (Llama) were Retry-with-backoff, Schema validation, Injection filter (scan), and a combined "All" stack.

## Results & Highlights

- **Cross-agent spread is large and invisible to task-success benchmarks.** On the 120-scenario suite, DeepSeek (ReAct) passes **105/120**, GPT-4.1-mini **89/120**, both Gemini variants **96/120**, and Llama-3.3-70B only **77/120** — a **28-point range** (DeepSeek − Llama = 105 − 77 = 28) that would not appear in GAIA/SWE-bench/ToolBench where tools always behave.
- **Silent data-quality faults (Category B) are the universal weakness.** Mean per-fault pass counts out of 10 are **B1 Stale Data = 5.0, B2 Contradiction = 5.2, B3 Wrong Answer = 4.6, B4 Silent Empty = 4.2**, versus ≥8.6 for all Category-A execution faults and ≥7.8 for Category-C security faults. Even the strongest agent DeepSeek scores only **29/40 on Category B**, and Llama **13/40**.
- **Retry mitigation closes execution faults but not data-quality faults on Llama** (Table 5): A1 Timeout rises **3/10 → 10/10**, A2 API Error **5/10 → 10/10**, A3 Permission Denied **3/10 → 10/10**, whereas B1 Stale Data stays **4/10 → 3/10** under retry and **3/10** under the full stack; B3 Wrong Answer stays **3/10 → 4/10** under the full stack.
- **Schema validation specifically rescues B4** (Silent Empty) from **1/10 → 4/10**, reaching **7/10** in the combined stack — the one data-quality fault an output-layer wrapper can address.
- **Security faults are largely handled already.** Per-fault means: **C1 Prompt Injection = 10.0, C2 Description Poisoning = 9.2, C3 False Claim = 9.8**, with only **C4 Data Exfiltration lower at 7.8** and unchanged (7/10) by every mitigation tested, because none of the mitigations targets exfiltration.
- **Scorer reliability is strong and repeatable.** Fault injection succeeded in all 120 scenarios; re-running a 36-scenario subset under identical cache gave identical pass/fail outcomes; re-scoring the same traces three times agreed perfectly on every diagnostic dimension. Scorer-vs-human Cohen's κ on 224 labeled instances: **Failure Detection 0.75, Recovery Action 0.85, Uncertainty Communication 0.66** (inter-annotator κ 0.78 / 0.87 / 0.69).

## Authors & Credibility

- **Aritra Mazumder** (University of Utah) and **Nusrat Jahan Lia** (University of Dhaka) — a small two-author team. The paper ships a public MIT-licensed repository (`github.com/aritra741/AgentCheck`), a YouTube demo, the full 120-scenario library with provenance, the LLM-judge prompts verbatim (Appendix B), agent configuration table (Appendix D), and scored traces. The artifact commitment and the reproduction protocol (shared cache, temperature 0, 10-step cap) are concrete and reproducible. The work is positioned carefully relative to prior benchmarks (GAIA, SWE-bench, ToolBench, τ-bench, ToolMaze, PlanBench-XL, ToolMisuseBench, MCPTox, MCPSecBench, MCPEval) and trace tools (AgentDiagnose, LangSmith, Langfuse, AGDebugger), with a clear Table 1 comparison on four axes (Inj / Own / Loop / HV) where AgentCheck is the only system satisfying all four.

## Critical Assessment

- **Novelty:** Genuine and well-scoped. The distinguishable idea is not "another fault benchmark" but the **reproduce-intervene-confirm loop over a shared cache on the practitioner's own agent/MCP server** — Table 1 shows no prior system combines fault injection + own-agent support + closed-loop fix confirmation + human-validated fault-handling scoring. The caching-based controlled comparison (change exactly one response) is a clean experimental-design contribution.
- **Rigor:** Mixed. The controlled-comparison methodology is sound and the human-alignment study is a real strength (κ reported on both legs). However, the **LLM judge is acknowledged non-deterministic and possibly over-firing**, so the reported propagation rates (Fig. 3, e.g. GPT-4.1-mini 58%) are explicitly labeled an *upper bound* rather than a measurement — a caveat the authors handle honestly but which weakens the diagnostic-label claims. The deterministic checks are the load-bearing verdict and they are *pattern-bound*, so a novel fault expression could evade them.
- **Reproducibility:** High. MIT license, public repo, fixed configs at temperature 0, shared cache making runs deterministic, verbatim judge prompts. The main reproducibility gap is dependency on commercial APIs (DeepSeek-V4-Pro, GPT-4.1-mini, Gemini-2.5-Flash, Claude 4.5 Haiku) with no cost/latency table.
- **Significance:** Practically meaningful — it directly answers a deployment-readiness question ("did my fix close this specific failure?") that no benchmark or monitor currently answers. The finding that data-quality faults are both dominant and unmitigable by output-layer wrappers is actionable.
- **Red flags / specific weaknesses:**
  - The **weakest baseline is Llama-3.3-70B-Instruct**, already a non-frontier model at the 2026-07 publication date; the 28-point spread and the mitigation deltas on Llama may overstate the practical gap, since frontier models would likely cluster near the top. The paper does not test a frontier open model.
  - **Category-B scenarios are newly authored (64/120) and the authors themselves state "its Category-B scenarios are authored, so the findings should be read as a hypothesis"** — the headline "silent data-quality faults are the dominant weakness" rests partly on researcher-authored scenarios rather than deployment-mined failures, a potential selection bias.
  - **Single-fault-at-a-time injection**: compound and cascading failures (a timeout that triggers a retry that hits a poisoned description) are explicitly out of scope and untested — a real deployment gap.
  - **C4 Data Exfiltration left at 7/10 with no mitigation tested against it** (the tested mitigations target execution and C1-style injection), so the security story is incomplete for the one fault class with real attack precedent (MCPTox's 72.8% attack-success rate).
  - The mitigation ablation (Table 5) reports only a single "All" combined column with no analysis of mitigation interactions or regressions beyond green/red coloring.

## Strengths

- Clean experimental design: holding all responses constant except one isolates causality in a way offline benchmarks cannot.
- The reproduce-intervene-confirm loop directly addresses a real engineering question ("does my fix work?") with a deterministic verdict.
- Honest, unusually detailed limitations and ethics section; the authors explicitly warn that `fix_confirmed` is *not* a robustness certificate and that the judge is fallible.
- Strong artifact packaging: open source, demo video, verbatim prompts, full config table, scored traces, documented scenario provenance.
- The taxonomy's separation of "no-error-code" data-quality faults (Category B) from crash-style faults (Category A) is conceptually useful and explains why happy-path benchmarks miss the dominant failure family.

## Limitations

- One fault injected at a time; no compound or cascading failure modes evaluated.
- The 12-fault library does not cover all long-horizon or policy-specific failure modes; Category-B scenarios are authored, so conclusions about them are hypothesis-level.
- LLM judge is non-deterministic and may over-fire; propagation rate is an upper bound, not a precise estimate.
- Deterministic primary checks are pattern-bound and could miss novel fault expressions.
- Five-agent panel skews toward one non-frontier open model (Llama-3.3-70B); no truly frontier open-weights model tested.
- Experiments rely on commercial model APIs; no compute-cost or latency reporting; live comparisons recommended only for read-only/sandboxed servers.
- C4 (data exfiltration) has no targeted mitigation tested, leaving the security story partial.

## Relevance

对研究 LLM agent 工具使用鲁棒性、可信评测与 MCP 生态安全的工作高度相关：AgentCheck 提供了可在自己 agent 上复现—干预—验证故障的开放工作台与 120 场景库，直接填补"工具失效时 agent 行为如何"这一评测空白，可作为后续 agent 可靠性、对抗鲁棒性与自动缓解研究的基础设施和对照基线。

## Questions for Further Exploration

- How do failure rates and mitigation effectiveness change under **compound/cascading faults** (e.g., a timeout that forces a retry which then hits a poisoned description)?
- Would **frontier open-weights models** (e.g., a 2026-era 400B+ model) close the Category-B gap, or is silent data-quality failure a fundamental model-capability limit rather than an alignment one?
- Can a **freshness-aware retrieval / cross-call consistency** harness component (sketched in Table 7) actually move B1/B2/B3 off the 3–4/10 floor that output-layer wrappers cannot?
- How stable is the LLM-judge propagation rate across judge models and seeds, and can a **lightweight non-text detector** (Gurram, 2026; Advani, 2026) replace it as the primary signal?
- Does `fix_confirmed` on one fault generalize — i.e., does a mitigation that closes A1 on Llama also close A1 on GPT-4.1-mini and DeepSeek, or is mitigation efficacy agent-specific?
- Can the controlled-comparison cache protocol be extended to **non-MCP tool environments** (e.g., raw function-calling APIs, browser-automation agents) without losing determinism?
