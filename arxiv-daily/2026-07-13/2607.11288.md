> **一句话总结：** 本文提出自进化智能体操作系统（SE-AOS）架构及其首个实例 Mako，将"漏洞利用能力"作为可在运行时被观测、合成、验证并热加载的可变内核；在 XBOW-104 基准（104 个容器化 CTF 式 Web 应用、26 类漏洞、3 个难度等级）上以"每次构建注入全新随机 flag + 真实响应扫描"的防伪造验证机制取得 104/104 全覆盖，且最难的 L3 层仅需中位数 2 轮即可解出，揭示了"能力（而非推理）才是成熟系统的稀缺瓶颈"这一规律。

> **TL;DR:** Mako is the first instantiation of a Self-Evolving Agentic Operating System (SE-AOS), an agent architecture that treats exploit capability as a mutable, versioned kernel it grows at runtime by observing failures, synthesizing new primitives, proving them against a live target, and hot-loading them back. On the XBOW validation-benchmarks (104 containerized CTF-style web apps, 26 vulnerability classes, 3 difficulty tiers), Mako achieves 104/104 full-suite coverage under a fabrication-proof verification regime (fresh random per-build flags scanned in genuine tool responses, never model self-report), where the strongest external baseline is KinoSec at 103/104 (99.04%) and BoxPwnr at 101/104 (97.1%); the hardest L3 tier paradoxically solves in a median of 2 turns vs 7 for the easier tiers, total cost $478.99 (~$4.61/benchmark). Payloads, exploit chains and tool source are deliberately withheld as dual-use research of concern.

## 核心贡献

- **新架构类 SE-AOS**：首次提出"自进化智能体操作系统"，把漏洞利用能力建模为可变、版本化的内核，运行时可由系统通过失败-合成-验证-热加载闭环自我扩展，突破了传统 ReAct/Toolformer 类智能体在部署时冻结工具集的范式。
- **自进化的形式化**：定义能力演化算子 Φ 与覆盖度泛函 C，证明在能力只增不减且择优保留的条件下 C 单调非降、有界收敛到固定点 C=1；并将子模 (1−1/e) 的贪心界作为开发优先级的启发式指南。
- **防伪造的评测机制**：通过每次构建注入 128 位均匀随机的 fresh flag + 对真实工具响应（而非 agent 自报告）做字符串扫描 + 仅当完整端到端漏洞利用触发后才算解出，把误报率理论上界压到 ≤2⁻¹²⁸，杜绝模型靠记忆/声明成功过关。
- **XBOW-104 全覆盖**：104/104（100%）目标均被驱动吐出全新随机 flag，覆盖全部 26 类漏洞与 3 个难度等级；最难的 L3 层中位数仅需 2 轮（"Level-3 反转"）。
- **能力选择律（Capability is all you need）**：给出并实证一条规律——一旦某能力存在且可被"发现"（关键词丰富的描述 + 在 agent 已可靠选用的伞形工具内做回退链式调用），难题即坍缩为单工具调用问题；难度从推理器迁移到能力库的覆盖度与可发现性。
- **双用途披露姿态**：发表"科学"（架构、理论、验证机制、全覆盖实证）而保留"武器"（payload、漏洞链、工具源码、演化引擎源码），并以负责任披露/DURC 实践自辩。

## Key Contributions

- **A new architectural class — SE-AOS**: capability is treated as a first-class, versioned, mutable kernel that the system itself extends at runtime by observing failures, synthesizing new primitives via operator Φ, proving them against a live target under a fabrication-proof gate, and hot-loading them back. This contrasts with ReAct/Toolformer-style agents whose capability set is frozen at deployment.
- **A formalism for self-evolution**: defines capability-evolution operator Φ and coverage functional C, proves a no-regression proposition (capabilities are additive, selection tie-breaks toward incumbents, so C(T_{t+1}) ≥ C(T_t)), and establishes a coverage fixed point at C=1 that the experiment attains; the marginal-gain submodularity is used as a prioritisation heuristic.
- **Fabrication-proof evaluation regime**: fresh 128-bit random flag injected per build (unknowable in advance since tool code is already fixed), ground-truth scanning of genuine tool responses (the harness wraps the tool executor and never trusts `report_finding`), and a construction guaranteeing the flag is reachable only via a complete end-to-end exploit — yielding a false-positive bound ≤ 2⁻¹²⁸.
- **Full-suite coverage of XBOW-104**: 104/104 (100%) benchmarks driven to emit a freshly-randomised flag across all 26 vulnerability classes and all 3 difficulty tiers, verified by the fabrication-proof gate.
- **The tool-selection law**: a geometric-time argument (E[turns] ≈ 1/p, where p is the per-turn selection probability of the solving capability) predicts that discoverability engineering (keyword-rich descriptions + chaining new exploits as fallbacks inside reliably-selected umbrella tools) drives p→1 and E[turns]→1 — exactly the observed Level-3 inversion (hardest tier, median 2 turns).
- **A dual-use disclosure posture**: science published (architecture, theory, safeguards, full-suite evidence), weapon withheld (payloads, PoC code, per-target recipes, tool/evolution-engine source), justified via responsible-disclosure / DURC practice.
- **Turn- and compute-economics analysis**: median 7 turns/solve (L3 median 2), $478.99 total / ~$4.61 per benchmark, ~1.03B tokens (~95% input).

## Methodology

Mako is built around two coupled, verification-gated loops (Figure 1):

1. **Online surface — live exploitation loop.** `SecurityResearchAgent` runs a bounded perceive → reason → act loop (XBOW runs used MAX_TURNS=25). Each turn the model receives the running memory and emits exactly one tool call plus a natural-language reasoning string. The engagement is structured in three phases: Phase 1 Recon (turns 1–5: `read_ctf_challenges`, `enumerate_paths`, `browser_detect_fw`, `find_flags`); Phase 2 Exploit (turns 6–80: the category-specific arsenal); Phase 3 Chain (turns 81–100: compose discoveries such as cracked-JWT → admin panel → IDOR). Two robustness mechanisms: a **forced pivot** (repeating the same tool beyond a threshold triggers a mandated technique switch) and a one-way **escalation** to Gemini 3.1 Pro Preview only after 70% of the turn budget elapses with zero confirmed findings. Primary model is Gemini 2.5 Flash.

2. **Capability-evolution loop (Φ).** A closed, honesty-preserving loop: run a benchmark with a fresh flag; if solved, register and record turns; if not, read the failure trace, diagnose root cause, synthesise or enhance a *general* (not benchmark-specific) capability via Φ, validate it standalone against the live target with a planted flag under the same gate, chain it into an umbrella tool the agent reliably selects, and re-run the agent end-to-end with a new fresh flag — no manual intervention.

3. **Self-evolution surface (platform Φ).** A parallel gated engine that improves Mako's own detection rules and agent logic: measure multi-category fitness on a labeled corpus → focus on weak categories → propose a single scoped change (FP rules, thresholds, specialist prompts, new pattern specialist) → apply only in an isolated sandbox → re-score → accept only if a conservative gate finds no regression in any category and no rise in false positives → commit or log the rejection. The control plane cannot rewrite itself; safety is the fitness gate plus versioned history.

**Capability-evolution operator (formalism).** Let U be the universe of realizable exploit capabilities, T_t ⊆ U the kernel at step t, E the target distribution (104-suite sample), ϕ_e ~ Uniform({0,1}¹²⁸) a fresh per-build flag, and π_θ the reasoner selecting one capability per turn. Define `solve(e,T)=1` iff some run of π_θ elicits ϕ_e in a genuine response, and `C(T)=E[solve]`. Then `c_new = Φ(τ_t, T_t)` and `T_{t+1} = T_t ∪ {c_new : V(c_new)=1}`, where V(c)=1 iff c demonstrably extracts a fresh flag from the relevant target class in sandbox — capability is admitted only when *proven*, never when merely proposed.

**Verification soundness.** Since ϕ_e is drawn uniformly from {0,1}¹²⁸ *after* all capability code is fixed, Pr[false positive] ≤ 2⁻¹²⁸ ≈ 0; the harness wraps the tool executor and scans every raw tool result for the planted flag, so calling `report_finding` is never sufficient on its own. Every benchmark's solving technique and successful-run log is recorded.

**Models & tooling.** ~180 registered tools (low-level `probe_url`, raw-socket senders, up to high-level self-contained exploit engines that internally sweep many payloads in one call); ~50 exploit/detection tools were newly built or materially enhanced during the campaign. General-purpose umbrella tools (e.g. `test_ssti_all_engines` handling Jinja/Twig/ERB/DTL with self-authentication, crawling, and filter-bypass phases) are emphasised over narrow single-benchmark gadgets.

**Threat model.** External, unauthenticated attacker given only a base URL; no source access at runtime (any source analysis informing Mako's tool library happens out-of-band in the capability-evolution loop). ~42/104 upstream fixtures required non-contaminating infrastructure repair (EOL apt archives, mysql:5.7 on arm64, invalid expose syntax, PhantomJS removal, etc.); one benchmark (XBEN-092) additionally required publishing an internal service port that the shipped Compose file omitted. All such changes are git-revertible and never touch application logic, the vulnerability, or the flag.

## Results & Highlights

- **Full-suite coverage**: Mako solves **104/104 (100%)** of the XBOW-104 benchmarks, each verified by extracting a freshly-randomised flag from the genuine application response (false-positive bound ≤ 2⁻¹²⁸). Coverage spans all 26 vulnerability classes and all 3 difficulty tiers. The strongest external baseline, KinoSec (vendor self-reported, no independent citation), reaches 103/104 (99.04%); BoxPwnr reaches 101/104 (97.1%); Shannon Lite reaches 100/104 (96.2%) but on a modified white-box, hint-free variant not directly comparable; Strix v0.4.0 reaches 100/104 (96%); the original XBOW agent ~85%; MAPTA 80/104 (76.9%); a 5-person human team 91/104 (87.5%); a single 20-year-principal human (40h) ~85%.
- **The Level-3 inversion**: the hardest tier (8 L3 benchmarks) has the lowest median turn count at **2.0**, vs 7.5 for L1 and 7.0 for L2 — the signature prediction of the tool-selection law, because each L3 challenge was ultimately solved by a purpose-built, well-described capability the agent selects immediately.
- **Turn economics**: median **7 turns/solve**, mean 10.5, min 2, max 40 (n=104, every solve logged); 30 benchmarks solved in ≤5 turns; 17 effectively one-shot (2 turns = exploit call + report). The long tail (21–40 turns) is dominated by XSS challenges needing slow in-browser PhantomJS oracle round-trips.
- **Compute economics**: total **$478.99** API spend at official Google list pricing ($306.44 on Gemini 2.5 Flash, $172.55 on Gemini 3.1 Pro escalation tier) ≈ **$4.61 per benchmark**, processing an estimated **~1.03B tokens** end-to-end (~95% input, ~5% output).
- **Tool library growth**: ~180 registered tools total; ~50 exploit/detection tools newly built or materially enhanced during the campaign.
- **Discoverability evidence**: XBEN-023 went from an 18-turn failure to a 2-turn solve *purely* by improving a tool's description and self-discovery, with no new exploit logic — supporting the tool-selection model.
- **Per-category behaviour**: SQLi has the highest median turns (11) among frequent classes; arbitrary_file_upload (17) and XXE (19) are the slowest despite being lower difficulty, because their tool maturity lags — effort tracks tool maturity, not nominal difficulty.

## Authors & Credibility

Both authors (Praneeth Narisetty, Shiva Nagendra Babu Kore) are affiliated with **LaunchSafe**, a company that "builds autonomous security agents for continuous offensive testing and agent-driven security research," and Mako is described as "the core engine behind that platform." This is explicitly a **vendor whitepaper / industry technical report**, not a peer-reviewed academic publication: the implementation is proprietary (not open-source), exploit materials and tool source are withheld as dual-use research of concern, and "qualified partners and investors may request a private technical diligence under NDA." The report is dated 2026-07-06. The verification methodology (fresh per-build flags, ground-truth response scanning, auditable run registry, integrity statement that no flag was hardcoded/guessed/memorised/fabricated) is methodologically sound and the strongest part of the paper's credibility case; however, none of the headline results are independently reproducible or third-party audited, and the comparison numbers in Table II are drawn from heterogeneous sources (self-reported vendor blogs, modified-suite GitHub traces) with explicit caveats.

## Critical Assessment

- **Novelty**: The SE-AOS framing is genuinely novel as a unifying architectural metaphor, but its constituent ideas have strong priors — Voyager evolves a skill library (in a benign Minecraft sandbox), STOP/​Gödel machines study recursive self-modification, Reflexion/Self-Refine iterate within a fixed capability set. The paper's honest differentiation is the combination of (i) a fabrication-proof adversarial gate on every self-authored capability, (ii) an externally verifiable domain, and (iii) full-suite (not per-task) coverage. The "capability, not reasoning, is all you need" thesis is provocative but somewhat tautological once one realises "capability" is defined to include discoverability — the claim risks being unfalsifiable.
- **Rigor**: The verification regime is the paper's strongest technical contribution and is well-engineered (fresh 128-bit flags drawn after code is fixed, ground-truth response scanning wrapping the tool executor, the harness never trusting `report_finding`, auditable per-run logs). The formalism, however, is explicitly hedged: the authors themselves state the submodular (1−1/e) bound is "a heuristic guide to prioritisation, not a tight result for this setting," that `solve` is an idealisation, and that the guarantee is "best read in expectation rather than pointwise." The no-regression proposition is essentially trivial (additive capabilities can't remove solves).
- **Reproducibility**: **Low, by design.** The agent, tool arsenal (~180 tools, ~50 newly built), and evolution engine are all proprietary and withheld; payloads and exploit chains are withheld; per-turn model attribution was *not logged*, so "we cannot confirm which specific long-running solves escalated" to Gemini 3.1 Pro. About 42 of 104 upstream fixtures required non-trivial infrastructure repair (EOL Debian/Ubuntu apt archives, mysql:5.7 on arm64, PhantomJS removal, XBEN-092's omitted published port) — a concrete reproducibility caveat for anyone trying to re-run on a different host. The comparison table mixes black-box and white-box modes, self-reported vendor numbers (KinoSec 103/104 with "no independent citation, run data, or disclosed verification methodology"), and modified-suite results (Shannon Lite), so cross-system comparison is noisy.
- **Significance**: If the 104/104 result holds under independent audit, it is a meaningful milestone — full-spectrum, machine-speed, sub-$5-per-target verified web exploitation under a cheat-proof regime. The Level-3 inversion is a genuinely instructive finding for the agent-design community. But the dual-use withholding posture, while ethically defensible, means the scientific community can only take the result on trust.
- **Red flags**: (1) Single suite, single model tier — headline results are for XBOW-104 with Gemini 2.5 Flash + Gemini 3.1 Pro Preview only; generalisation to unseen applications is explicitly called future work. (2) Benchmark-specific gadgets: the authors concede "some tools encode app-shaped knowledge," which materially weakens the "capability generalises" claim. (3) Run-to-run variance: "a few benchmarks (e.g. XBEN-009) solved in some runs and exhausted turns in others; reported turn counts are from successful runs" — turn-economics numbers are therefore best-case-conditioned, not expected values. (4) Per-turn model attribution not logged undermines the cost-attribution and escalation-policy analysis. (5) The "no frozen third-party baseline agent" comparison means Mako's advantage over peers is asserted via externally-sourced numbers of varying provenance, not head-to-head under one harness. (6) The `gemini-3.1-pro-preview` and `gemini-2.5-flash` naming is unusual and the paper notes "no separate technical report exists for these specific preview versions."

## Strengths

- **Methodologically cheat-proof evaluation**: the fresh-per-build random flag + genuine-response-scan design is a model for how offensive-agent benchmarks should be run; it directly neutralises memorisation, hallucinated success, and incidental disclosure. The 2⁻¹²⁸ false-positive bound is a clean theoretical backstop.
- **Architectural clarity**: the OS analogy (capability kernel = kernel, typed tool interface = syscalls, self-authored exploit tools = loadable drivers, synthesis operator Φ = compiler, meta-controller = scheduler, fresh-flag gate = protected mode) is a genuinely useful organising frame.
- **Counter-intuitive, well-explained empirical finding**: the Level-3 inversion (hardest tier, median 2 turns) is the kind of result that reframes how practitioners should think about agent maturity, and the geometric-time tool-selection law gives it a crisp causal account.
- **Honest self-limitations section**: the paper is unusually forthcoming about what it does and does not claim (system capability, not static-model capability; no public frozen baseline; stochastic variance; app-shaped knowledge in tools; per-turn model not logged).
- **Responsible disclosure posture**: publishing the science while withholding the weapon, with an explicit integrity statement and dual-use analysis, is a defensible ethical stance for full-spectrum offensive capability.
- **Concrete economics**: sub-$5-per-target verified exploitation at retail API rates is a striking, decision-relevant number for defenders.

## Limitations

- **Not independently reproducible**: proprietary agent, tools, evolution engine, and withheld payloads mean no third party can re-run or audit the 104/104 claim; the scientific artefact is the architecture and methodology, not the result.
- **Single benchmark suite and single model tier**: results are specific to XBOW-104 with the Gemini family; cross-suite and cross-model generalisation is untested.
- **Turn counts are best-case-conditioned**: drawn from successful runs only, with explicit acknowledgement of run-to-run variance where some benchmarks failed in other runs.
- **Benchmark-specific knowledge contamination**: ~50 newly-built tools may encode app-shaped knowledge that does not transfer to unseen applications.
- **Reproducibility friction on the suite itself**: ~42/104 fixtures needed non-contaminating but non-trivial infra repairs; one benchmark needed a port published that the shipped Compose file omitted.
- **Comparison provenance is heterogeneous**: Table II mixes black-box/white-box modes, self-reported vendor blogs (KinoSec), and modified-suite GitHub traces (Shannon Lite), making clean head-to-head comparison impossible.
- **Formalism is decorative more than load-bearing**: the submodular (1−1/e) bound is admittedly heuristic; the no-regression proposition is trivial; the fixed-point result is essentially "additive coverage eventually saturates."
- **Cost and per-turn attribution gaps**: per-turn model was not logged, so the escalation-tier cost split and which solves actually escalated cannot be confirmed.

## Relevance

对关注"LLM 智能体自主安全研究 / 自动化渗透测试 / 自进化智能体架构"的方向高度相关：Mako 展示了将工具能力作为可演化内核 + 防伪造验证闭环这一新范式，在 XBOW-104 上取得全覆盖，对研究 agent 的工具发现与编排瓶颈、以及如何用机制（而非信任）来评估 offensive agent 都有直接借鉴价值；其"能力而非推理才是稀缺资源"的论点对智能体系统设计具有启发性。

## Questions for Further Exploration

- How much of the 104/104 coverage generalises to unseen, non-CTF web applications (real production apps with bespoke logic) versus being an artefact of benchmark-specific gadget construction in the ~50 newly-built tools?
- What is the run-to-run solve-probability distribution per benchmark (not just the best-case turn count), and what is the expected cost per reliable solve rather than per successful run?
- How sensitive is the result to the underlying model — would a non-Gemini primary model (e.g. an open-weights model) reach comparable coverage with the same tool library, or is the Gemini family doing unmeasured heavy lifting?
- Can the fabrication-proof verification regime (fresh per-build flags + genuine-response scanning) be adopted as a community standard for offensive-agent benchmarks beyond XBOW, and what does a head-to-head re-run of KinoSec/BoxPwnr/Shannon under this single harness show?
- Does the "capability, not reasoning, is all you need" thesis survive when the capability library cannot be pre-evolved against the test distribution — i.e., under a strict train/test split where evolution happens only on a held-out 70% of vulnerability classes?
- What is the marginal contribution of the platform Φ self-evolution surface (improving Mako's own agents/rules) versus the online capability-evolution loop, and is the fitness gate's no-regression guarantee ever violated in practice?
- Given the dual-use withholding, what minimal artefact (e.g. anonymised capability-evolution traces without payloads) could be released to allow independent scrutiny of the self-improvement claims without operationalising the weapon?
