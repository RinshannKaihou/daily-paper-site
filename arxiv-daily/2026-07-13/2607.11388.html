Unverified "done" commits are the dominant false-progress failure in long-horizon agents. A shared, verifier-committed state is a reusable design baseline — not just a leaderboard bump.

The mechanism

state · workflow · control

Unified, typed agent state

st = ( streq , stval , stver )
  • req  — current subgoal requirements; each carries σ ∈ {PENDING, VERIFIED, INVALIDATED}
  • val  — durable useful values: file paths, URLs, selected entities, extracted fields
  • ver  — verified evidences, auditable & reusable by later turns

Planner, actor, and verifier all read/write through one verifier-maintained surface. Memory is kept separate and can only suggest, never commit.

Verifier-committed loop

s_t ─Planner→ g_t ─Actor→ τ_t ─Verifier→ d_t ─Update→ s_t+1

Planner reads state (not raw history), picks the next subgoal.

Actor emits action traces — GUI grid clicks, DevTools nav, cli_run/edit_json, UNO-backed Office actions.

Verifier probes evidence via 7 typed VerifySpec kinds (file_grep, url_match, a11y_match, calc_verify, writer_verify, impress_verify, shell_command), each gated by a per-domain trust map.

Rule: planner/actor "DONE" is a proposal. Only d_t commits, preserves, or invalidates.

Strengths & limitations

side-by-side

STRENGTHS

  • Principled, cleanly stated architecture: the (sreq, sval, sver) triple + verifier-only transitions genuinely fixes the "raw-history soup" of long-horizon tasks.
  • Large, consistent gains exactly where structured state should help most — weak/mid open backbones (9B +19.9, 27B +30.6).
  • Honest evaluation hygiene: matched-runner protocol, reported-vs-self-run separation, isolated verifier study with P/R/F1, human-style failure audit.
  • Cross-domain instantiation (Minecraft inventory verification) proves the abstraction is not bound to screenshots — the more durable claim.
  • Concrete engineering: 7 VerifySpec kinds, per-domain trust maps, read-only shell guards, typed VerificationRecord.

LIMITATIONS

  • No clean on-OSWorld ablation isolating the unified state from the better verifier or memory — headline gains are underdetermined.
  • Leverage compresses at the strong-backbone / cross-app frontier (MiniMax-M3 only +3.7; 9B Multi-Apps still trails OS-Symphony 27.5 vs 32.6).
  • Mind2Web numbers are modest and partly judge-dependent (same-family judge, 86% human agreement); regressions on Entertainment/Shopping.
  • Minecraft capped by a borrowed GUI-blind craft helper (JARVIS-1) and STEVE-1's ~6s spatial-memory limit — orthogonal, but it caps the demo.
  • No cost/latency/token overhead reported for running planner + actor + agentic verifier per subgoal.
  • Memory bank depends on non-public internal CUA-Gym rollouts — exact reproducibility of memory is weakened.
