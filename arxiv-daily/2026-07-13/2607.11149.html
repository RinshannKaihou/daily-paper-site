Why this matters

Establishes the fourth evaluation axis beyond task-success, reliability, and inference cost — with a reproducible, serialization-aware meter directly useful for agent selection, retention governance, and compliance audits.

Critical edge

The 15.7× spread compares documented defaults that bundle different recovery, audit, and crash-resume capabilities — it is not capability-normalized, so readers could over-read it as “LangGraph is wasteful.”

01

Methodology — a serialization-aware meter

Footprint = bytes left inside a per-run sandbox boundary after a run, attributable to that run. Workspace artifacts plus every framework-side store (session/checkpoint DBs, logs, snapshots, traces, hidden home-directory state). Bytes are classified four ways: decisions, observations, derivations, duplicates.

The methodological crux: raw-file chunking masks duplication. SQLite fragments large values across 4 KB pages; JSON escaping rewrites embedded bytes — so copies no longer fingerprint-match. LangGraph's checkpoint store reports D=1.01 under naive chunking but D=12.1 on logical streams (and zstd still compresses it 142×). The meter extracts logical streams (per SQLite cell, per JSONL line) before content-defined chunking (FastCDC + SHA-256).

Harness: 8 frameworks × 1,061 sandboxed runs (509 main + 552 replication/variant), identical backend (DeepSeek-V4-Flash, T=0), identical minimal tools, three repetitions, container-validated isolation. A fixed-trace control replays one identical trajectory through seven adapters to isolate persistence-layer amplification (1.03×–6.86×) from agent behavior.

Serialization masking

LangGraph checkpoint store: D=1.01 naive → D=12.1 logical, while zstd still hits 142×. The duplication was always there — SQLite paging and JSON escaping hid it by up to 12×.

Residue dominates artifacts

A 39-byte summary file leaves 128 KB–1 MB of residue — up to 24,033× the delivered artifact. In production: 131 MB for one supply-chain task, ~56 GB over 103 rounds.

SWE-bench in the wild (108 systems)

Exported trajectory volume spans 1,617× (6.6 KB–10.6 MB/instance). Resolve rate shows no correlation with bytes: Kendall τb=0.080, p=0.222.

Compactor (existence proof)

A ~300-line content-addressed compactor cuts retention 4.8–32.7× (AutoGen 32.7×, Agno 20.5×) across 70 runs — every R score reproduced, all 310 restore checks pass.

03

Strengths & limitations

Strengths

  • Defines a genuinely unmeasured evaluation axis, with clean separation of logical volume (behavior) vs amplification (persistence layer) — and a fixed-trace control that isolates the second.
  • The D=1.01 vs 12.1 serialization-masking insight transfers to anyone measuring deduplication on serialized stores, not just agent researchers.
  • Exceptional hygiene: per-run sandboxes, container validation, 3 reps, 20k-resample bootstrap CIs, two-model backend replications (ρ=0.96 / 0.89), per-commit automated verification.
  • Intellectually honest: six explicit boundary caveats; CAS store flagged as an existence proof with real obligations (hash salting, cross-tenant deletion, existence probes).
  • Practical: the 24,033× residue ratio and the 131 MB production example translate directly to compliance and retention-policy decisions.

Limitations

  • Controlled suites are retrieval/retention-focused and small; heterogeneous families use single runs with no repetition variance — generalization to complex reasoning is untested.
  • The 15.7× spread is not capability-normalized: defaults bundle different crash-resume / audit / debug-provenance capabilities.
  • The CAS compactor is a ~300-line reference, not a production system; unsalted hashes leak content-equality, shared blobs complicate deletion.
  • SWE-bench data is exported trajectory volume (a lower bound subject to per-team export policy), not runtime residue — the no-correlation result partly conflates footprint with export-policy heterogeneity.
  • α is a descriptive fit over only four horizons; local slopes diverge within one framework, so ±0.01 precision overstates regularity.
04

Open questions

  1. Does the 15.7× dispersion collapse or persist under capability-normalized configurations (equal crash-resume, workflow-state, debug-provenance)?
  2. Can α be predicted from a framework's documented archival/context policy, enabling a static storage-risk classifier without running the stress task?
  3. Does the τb=0.080 no-correlation result hold on other leaderboards (τ-bench, GAIA), or is it SWE-bench-specific?
  4. What is the latency cost of the content-addressed compactor relative to its storage savings — does rehydration-on-demand degrade step latency?
  5. How do rankings behave under multi-agent / shared-thread deployments (fresh sandboxes understate shared threads: 95.2 vs 5.8 MB)?
  6. Can R be extended to a richer capability score (crash resume, workflow state) so “removable redundancy” is tested against capability-normalized reconstructability?
