AgentCheck: A Reproduce–Intervene–Mitigate Workbench for LLM Agents over MCP

AgentCheck turns any MCP server into a fault-injection surface, exposing silent data-quality failures that happy-path benchmarks miss.

01

Methodology — controlled comparison over a shared cache

change exactly one response
STEP 1 · CLEAN

Record

τ_clean = RUN(A, q, T) caches every tool response into K: (tool, args, i) → r.

STEP 2 · FAULTED

Replay

Returns g(r) only at injection point p; every other call is byte-identical.

STEP 3 · MITIGATED

Wrap & re-run

Agent wrapped as m(A), same fault re-applied — do the failed checks now pass?

STEP 4 · VERDICT

Confirm

First divergence logged; deterministic CHECKS(p,g) return a fix_confirmed verdict.

The loop offline benchmarks leave open: inject a fault → it fails → wrap a mitigation → re-run against the identical (p, g) fault → get a deterministic fix_confirmed. Caching makes every run reproducible.

Agent panel (all temp 0, 10-step cap, identical fixed responses): DeepSeek-V4-Pro (ReAct) · Gemini-2.5-Flash (ReAct + zero-shot) · GPT-4.1-mini (ReAct) · Llama-3.3-70B-Instruct (ReAct). Mitigations tested on weakest agent (Llama): Retry · Schema validation · Injection filter · combined "All" stack.

3 categories · 12 fault types. A Execution (timeout / API error / permission / schema drift)  ·  B Data quality (stale / contradiction / wrong answer / silent empty) — no error code  ·  C Security (prompt injection / description poisoning / false claim / exfiltration).  Suite = 120 scenarios (36 ToolMisuseBench · 20 MCPTox · 64 newly authored).

03

Strengths & Limitations

side-by-side

✓ Strengths

  • Clean experimental design. Holding every response constant except one isolates causality in a way offline benchmarks cannot.
  • Reproduce-intervene-confirm loop answers a real engineering question — "does my fix work?" — with a deterministic verdict.
  • Honest limitations. Authors explicitly warn fix_confirmed is not a robustness certificate and the judge is fallible.
  • Strong artifact packaging: MIT repo, demo video, verbatim prompts, full config table, scored traces, documented provenance.
  • Useful taxonomy: separating "no-error-code" data-quality faults (B) from crash-style faults (A) explains why happy-path benchmarks miss the dominant family.

▲ Limitations

  • Single-fault injection only; compound / cascading failures are explicitly out of scope and untested.
  • Category-B scenarios are authored (64/120) — conclusions about them are hypothesis-level, a potential selection bias.
  • LLM judge is non-deterministic and may over-fire; propagation rate is an upper bound, not a measurement.
  • Deterministic checks are pattern-bound and could miss novel fault expressions.
  • Panel skews to one non-frontier open model (Llama-3.3-70B); no frontier open-weights model tested.
  • Commercial APIs only; no compute-cost or latency reporting. C4 exfiltration has no targeted mitigation — security story partial.

Open Questions

for further exploration
Q1 · COMPOUND
How do failure rates shift under cascading faults — e.g. a timeout forcing a retry that then hits a poisoned description?
Q2 · FRONTIER
Would a frontier open-weights model close the Category-B gap, or is silent data-quality failure a capability ceiling?
Q3 · B-FLOOR
Can a freshness-aware / cross-call consistency harness lift B1/B2/B3 off the 3–4/10 floor output wrappers cannot?
Q4 · JUDGE
Is the judge propagation rate stable across models — and can a lightweight non-text detector replace it as primary signal?
Q5 · GENERALIZE
Does fix_confirmed transfer across agents — does an A1 fix on Llama also close A1 on GPT-4.1-mini / DeepSeek?
Q6 · PORT
Can the controlled-comparison cache extend to non-MCP tool environments (raw function-calling, browser agents) without losing determinism?
