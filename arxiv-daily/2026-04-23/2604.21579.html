A Metamorphic Testing Approach to Diagnosing Memorization in LLM-Based Program Repair

arXiv: 2604.21579v1

Date: 2026-04-23

https://arxiv.org/abs/2604.21579

一句话总结： 本文通过对Defects4J和GitBug-Java应用9种语义保持的代码变换（如标识符重命名、循环重写），系统证明7个顶级LLM（包括Claude-3.7、GPT-4o）在Defects4J上补丁生成成功率平均下降4.1–16%，且该下降与NLL（负对数似然，记忆代理指标）显著相关，表明组合MT+NLL是诊断APR基准污染的更可靠方法。
TL;DR: Applying semantics-preserving code transformations (identifier renaming, control-flow rewrites) via the CodeCocoon framework to Defects4J and GitBug-Java, this paper evaluates 7 state-of-the-art LLMs and finds consistent, statistically significant performance drops on Defects4J (−4.1% to −15.98%) that correlate with NLL-based memorization signals on the original benchmarks. The newer GitBug-Java shows minimal average drops but retains worst-case vulnerabilities. The proposed diagnostic: combine metamorphic testing with NLL to identify data leakage in LLM-based APR evaluations.

核心贡献

  • CodeCocoon框架：一个用于Java代码语义保持变换的系统性工具，包括9种自然变换（函数/参数/变量重命名、for-to-while、reverse-if、nest-else-if、swap操作数等），专门筛选了人工评估自然性高的变换以避免引入非自然输入。
  • MT+NLL组合诊断：证明仅用MT（成功率下降）或仅用NLL（记忆代理）各有局限，两者结合提供对数据泄漏的更强、更可靠证据——对于LLM低NLL（高熟悉度）的bug，MT性能下降最大。
  • 大规模跨模型评估：对7个最新LLM（包括闭源的Claude-3.7、GPT-4o、GPT-4o-mini和开源的Llama-3.1、Gemma-2-27B、Mistral-7B、StarCoder2）在两个基准上的系统性实证。
  • NestElseIf为最具代表性的指示性变换：跨多个模型（尤其是Claude-3.7和GPT-4o）最一致地引起性能下降，表明模型依赖熟悉的if-else if语法模式。

Key Contributions

  • Metamorphic testing as a model-agnostic diagnostic: Unlike NLL (which requires white-box access to token probabilities), metamorphic transformations are applicable to both open-source and proprietary models and expose code-pattern sensitivity through observable repair behavior.
  • Correlation between MT degradation and NLL: Spearman rank correlations between performance drop (SRdiff) and NLL percentile for open-source models: Gemma-2 r=0.303 (p=0.0011, medium), Llama-3.1 r=0.165 (p=0.047, small), Mistral-7B r=0.218 (p=0.049, small), StarCoder2 r=0.401 (marginal, medium). Establishes converging evidence.
  • Bug-difficulty × transformation interaction: Easy bugs (high original SR) show the most pronounced drops, consistent with memorization: models that reliably fix "easy" bugs may be recalling memorized solutions, and surface changes break the recall trigger.
  • Differential behavior across datasets: Defects4J shows broad, consistent drops (older benchmark, higher memorization likelihood); GitBug-Java shows minimal average drops but non-trivial worst-case degradations (e.g., GPT-4o-mini drops from 100% to 30% on a specific bug).

Methodology

Datasets:

  • Defects4J: 854 real-world Java bugs from mature projects, most dating to Java 1.6–1.8; pre-dates LLM training cutoffs → high leakage risk.
  • GitBug-Java: 199 bugs from Jan–Oct 2023; explicitly designed to avoid leakage by targeting post-cutoff defects. Study focuses on function-level defects only (44 bugs excluded).

Transformations (CodeCocoon):

  1. Rename Function (RFun) — context-synonym function name
  2. Rename Parameter (RPar) — synonym parameter name
  3. Rename Variable (RVar) — synonym local variable
  4. For to While (F2W)
  5. Reverse If (RevIf) — negate condition + swap branches
  6. Nest Else If (NestEI) — else-if → nested if-inside-else
  7. Swap Equals Operands (SEO) — a==b → b==a
  8. Swap Relational Operands (SRO) — a>b → b
  9. Expand Unary Increment (EUI) — i++ → i+=1

Identifier synonyms generated by ChatGPT-4o-mini for naturalness; only transformations rated highly natural in prior human evaluation study (Le-Cong et al.) are included.

Models: Claude-3.7-Sonnet, GPT-4o, GPT-4o-mini (closed-source); Llama-3.1-8B, Gemma-2-27B, Mistral-7B-v0.3, StarCoder2-7B (open-source, instruction-tuned). All run via vLLM locally for open-source.

APR Setting: Chain-of-Thought prompting (analyze root cause → suggest solutions → implement). Closed-source: 5 patches per prompt × 10 prompts per bug. Open-source: 1 patch per prompt × 10 prompts. Patch is "successful" if at least one generated patch passes all test cases. Function-level fault localization (buggy function provided, not exact line).

Success Rate Metric: SR = (# prompts producing ≥1 correct patch) / (# prompts). SRdiff = SRtrans − SRorig. Statistical tests: Wilcoxon signed-rank test, Vargha-Delaney Â₁₂ effect size.

Memorization Signal: NLL percentile ranks from Ramos et al. [11], extended to full Defects4J by the authors using their replication package. Spearman correlation between NLL percentile and SRdiff.

Transformation Analysis: Two-way permutation test (1,000,000 iterations) with Â₁₂ effect sizes for individual transformations and 2–3-way interaction terms.

  • Worst-case drops: Claude-3.7 fails entirely on transformed Jsoup-15 (100% → 0%); GPT-4o drops Math-32 from 80% to 0%; Gemma-2-27B drops Lang-52 to 0%.
  • GitBug-Java: Claude-3.7 and GPT-4o show no significant drops (p=0.819, p=0.667); GPT-4o-mini shows marginal drop (p=0.074, −8.67%).
  • NLL-MT correlation: Performance drops are highest in the lowest NLL bins for all models — bugs the model is most familiar with are most sensitive to transformation.
  • NestElseIf is the most consistently harmful transformation across Claude-3.7, GPT-4o, and combinations; structural rewrites amplified by renaming create the largest drops.
  • Gemma-2-27B anomaly: Uniquely sensitive to renaming in isolation (medium effect size), indicating stronger reliance on surface-level identifier names vs. other models.

Critical Assessment

Novelty: Good. While metamorphic testing for code model robustness is established (Applis et al., Le-Cong et al.), this is the first paper to explicitly link MT degradation to memorization/NLL signals with a multi-model, multi-benchmark study of this scale, and to include proprietary frontier models (Claude-3.7, GPT-4o). CodeCocoon's systematic approach to natural transformation selection is a methodological contribution.

Rigor: High. Statistical testing is appropriate and thorough. The Spearman correlation (non-parametric, justified by Shapiro-Wilk failures) is well-suited. Effect size reporting (Â₁₂) alongside p-values is good practice. The paper explicitly limits NLL to a "supporting signal" rather than "definitive proof" of leakage.

Reproducibility: High. Zenodo artifact repository, specific model identifiers, exact prompting settings, and extension of Ramos et al.'s NLL analysis are all documented.

Significance: Strong practical significance for the software engineering and ML reliability communities. The finding that easy bugs show the largest drops is particularly compelling for the memorization hypothesis. The differential Defects4J vs. GitBug-Java result provides a natural internal control.

Red Flags:

  1. Test-suite adequacy is not semantic correctness: Plausible patches that pass tests may be overfitted (test suite exploited); the paper acknowledges this but the chosen metric (patch passes all tests) is the only feasible option at this scale. Transformations are applied equally to both conditions, so relative comparisons are valid.
  2. NLL values not available for closed-source models: The MT-NLL correlation analysis is restricted to 4 open-source models; the most capable models (Claude-3.7, GPT-4o) cannot be included in the correlation analysis.
  3. Synonyms generated by ChatGPT-4o-mini: Renaming transformations depend on ChatGPT-4o-mini generating appropriate synonyms, introducing a potential confound — if the renamed identifiers are unnatural, drops may reflect stylistic mismatch rather than memorization. This is mitigated by using naturalness-validated transformations and restricting to function-level bugs.
  4. Function-level fault localization assumption: More lenient than line-level FL used in some baselines, which makes the absolute success rates not directly comparable to all prior APR papers.
  5. Limited to Java: CodeCocoon and the benchmarks are Java-specific; whether findings generalize to Python, C++, etc. is explicitly left to future work.

Strengths

  • Strongest empirical signal: all 7 models show statistically significant drops on Defects4J; the Wilcoxon tests are uniformly significant (p<0.001).
  • Internal control: the GitBug-Java comparison provides a natural less-contaminated contrast set.
  • Artifact availability: Zenodo repository enables independent replication.
  • Model diversity: 3 closed-source + 4 open-source models across different scales and families.
  • Honest framing: MT is positioned as a diagnostic probe, not a data leakage proof or a replacement for benchmark curation.

Limitations

  • Only Java benchmarks — limited to Defects4J and GitBug-Java; no Python, C++, or multilingual evaluation.
  • NLL not available for closed-source models, limiting the correlation analysis to smaller models.
  • Patch validation metric (test-suite adequacy) may include plausible but incorrect patches.
  • The Vargha-Delaney effect sizes are uniformly small to medium for the strongest models (Claude-3.7, GPT-4o) — practical degradation exists but is modest for top-tier models.
  • Analysis restricted to function-level single-method bugs; multi-method or cross-file repairs not covered.
  • No investigation of whether providing the original (non-transformed) function alongside the transformed one allows models to "undo" the obfuscation.

Relevance

本文与ML可靠性和基准测试方法论高度相关：它提供了强有力的实证证据表明主流APR基准（Defects4J）存在数据泄漏，导致性能虚高，并提出了一种模型无关的诊断方法，这对评估LLM在代码任务上的真实泛化能力有直接意义，与更广泛的LLM训练数据污染问题紧密相连。

Questions for Further Exploration

  1. Can the NestElseIf sensitivity be exploited adversarially — if you transform all evaluation instances with NestEI before reporting, does this produce a more honest estimate of true generalization performance?
  2. Does model fine-tuning on transformed code (using CodeCocoon augmentations as data augmentation) improve robustness to natural code variations, and does it close the MT performance gap?
  3. What is the relationship between model scale and memorization susceptibility — does Llama-3.1-70B (larger) show larger or smaller MT drops than Llama-3.1-8B?
  4. Can the MT+NLL diagnostic be used to rank individual bugs by their "contamination likelihood" — e.g., flagging the lowest-20% NLL bugs as unreliable for APR evaluation?
  5. Do the same MT degradation patterns appear in code generation (HumanEval, MBPP), or is APR uniquely vulnerable due to the specific bug-fix pair structure of training data?
