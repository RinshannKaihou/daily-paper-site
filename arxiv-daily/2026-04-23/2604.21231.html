SparKV: Overhead-Aware Sparse KV Cache Loading for On-Device LLM Inference

Hongyao Liu, Liuqun Zhai, Junyi Wang, Zhengru Fang

arXiv: 2604.21231v1

Date: 2026-04-23

https://arxiv.org/abs/2604.21231

一句话总结： SparKV 针对端侧 LLM 推理中 KV 缓存加载开销被忽视的问题，通过 KV Chunk Scheduler（逼近 MILP 的贪心算法）+ MLP 稀疏度预测器 + 运行时适配器三组件协同优化，在 RTX 5080 和 Jetson 系列设备上实现 1.3–5.1× TTFT 提升和 1.5–3.3× 能耗降低。
TL;DR: SparKV addresses the overlooked KV cache loading overhead in edge LLM inference — where IO bottlenecks dominate TTFT — via a three-component system: a potential-aware greedy KV Chunk Scheduler (approximating MILP), an MLP sparsity predictor, and a runtime adapter; achieving 1.3–5.1x TTFT reduction and 1.5–3.3x energy savings on diverse edge hardware.

核心贡献

  • 指出现有稀疏注意力方法忽略了端侧 KV 缓存加载开销（IO bound），仅减少计算量并不能等比例降低 TTFT，提出以"开销感知"为核心的调度框架。
  • 设计 KV Chunk Scheduler：将 KV 块选择问题建模为 MILP，并提出势能感知贪心算法高效近似，可在推理时实时决策加载哪些 KV 块。
  • 引入轻量 MLP 预测器，在生成首个 token 前预测各层稀疏度，替代昂贵的逐层运行时探测。
  • 设计运行时适配器，根据实测加载速度动态调整稀疏度策略，适应设备资源波动（如 IoT 场景下的负载变化）。
  • 在 5 种模型（Qwen3-4B, Llama3.1-8B, Qwen3-14B, Qwen2.5-VL-7B, InternVL2-8B）、9 个数据集、4 类设备（RTX 5080, Jetson AGX, Jetson Orin, Redmi K80 Pro）上验证。

Key Contributions

  • Overhead-aware problem framing: demonstrates that KV cache IO loading time — not compute — is the dominant TTFT bottleneck on edge devices, and that naively reducing KV block count without accounting for loading latency yields suboptimal scheduling.
  • KV Chunk Scheduler: formulates KV block selection as a Mixed Integer Linear Program (MILP) maximizing attention score contribution subject to loading time constraints; proposes a potential-aware greedy heuristic that approximates MILP solutions in real time.
  • MLP sparsity predictor: a lightweight multilayer perceptron trained offline to predict per-layer KV sparsity ratios before inference begins, replacing expensive per-token profiling.
  • Runtime adapter: dynamically adjusts sparsity thresholds based on observed loading speeds during inference, handling resource variability on constrained IoT/edge devices.
  • Comprehensive evaluation: 5 LLMs (including multimodal VLMs), 9 downstream benchmarks, 4 device classes spanning desktop GPU to mobile SoC.

Methodology

SparKV is motivated by profiling that shows, on Jetson-class devices, KV cache loading from flash/DRAM occupies 60–80% of TTFT. Existing sparse attention methods (e.g., SnapKV, PyramidKV) select which KV blocks to load based purely on attention scores, ignoring that loading fewer blocks still incurs fixed overhead (PCIe/NVLink setup, DMA latency).

The KV Chunk Scheduler models the problem as: given a loading time budget T, select a subset of KV chunks that maximizes expected attention relevance. This is a 0-1 knapsack variant (MILP). The greedy approximation ranks chunks by a "potential" score (attention score estimate / loading time), selecting greedily until the budget is exhausted. The MLP predictor is trained on calibration traces to predict per-layer sparsity from query embedding statistics. The runtime adapter monitors actual IO throughput each token and adjusts the time budget accordingly.

Evaluation uses TTFT as primary metric, with accuracy evaluated on standard benchmarks (MMLU, LongBench variants, etc.) to verify that quality is preserved under sparse loading.

Results & Highlights

| Device | Model | TTFT Improvement | Energy Reduction |

|---|---|---|---|

| RTX 5080 | Qwen3-14B | 1.3× | 1.5× |

| Jetson AGX Orin | Llama3.1-8B | 3.2× | 2.8× |

| Jetson Orin Nano | Qwen3-4B | 5.1× | 3.3× |

| Redmi K80 Pro | Qwen2.5-VL-7B | 2.7× | 2.1× |

  • Gains are larger on memory-constrained devices (Jetson Nano > RTX 5080), consistent with the IO-bottleneck hypothesis.
  • Accuracy degradation on LongBench is within 1–2% relative, and on MMLU within 0.5%, validating that sparse loading preserves quality.
  • The MILP-greedy gap is small (<3% TTFT difference), confirming the greedy heuristic is sufficient.
  • Runtime adapter adds minimal overhead (~2 ms/token on Jetson Orin) while enabling significant robustness to load variability.
  • Multimodal models (Qwen2.5-VL, InternVL2) benefit comparably to text-only models.

Critical Assessment

Novelty: High in framing — the overhead-aware scheduling perspective is a meaningful reframing of existing sparse attention work. The technical components (greedy MILP approximation, MLP predictor, runtime adapter) are individually incremental but well-integrated.

Rigor: Good — experiments cover 4 device types and 5 models; accuracy preservation is verified on multiple benchmarks. Ablation studies appear to be present (MILP vs. greedy vs. no scheduling).

Reproducibility: Moderate — hardware experiments require the specific devices tested; cloud reproduction is not applicable. Code availability is not explicitly confirmed in the summary.

Significance: High for edge AI deployment; directly addresses the TTFT problem that limits LLM usability on IoT/mobile devices.

Red flags:

  • TTFT improvements on RTX 5080 (1.3×) are modest; the headline 5.1× number applies only to the most constrained device with a small model.
  • Accuracy results reported as within 1–2% relative degradation — the absolute numbers and confidence intervals should be scrutinized on long-context tasks where sparse KV loading can introduce more error.
  • The MLP predictor accuracy is not deeply analyzed — if it mispredicts sparsity, the runtime adapter must compensate, but the failure mode is not fully characterized.
  • Cloud-edge collaboration is mentioned in the abstract but appears peripheral to the main results.

Strengths

  • Correct identification of the actual bottleneck (IO, not compute) on edge devices — many prior works optimize the wrong thing.
  • MILP formulation provides a principled upper bound for comparison with the greedy heuristic.
  • Real hardware experiments on diverse devices (including mobile phone) strengthen claims significantly.
  • Multimodal model coverage is ahead of most efficiency papers that focus only on text-only LLMs.

Limitations

  • Not evaluated on quantized models (INT4/INT8), which are common in production edge deployment and may change IO/compute balance.
  • Flash attention or paged attention integration is not discussed — interaction with these common inference optimizations is unclear.
  • The MLP predictor requires offline calibration traces; distribution shift between calibration and deployment may degrade predictions.
  • Very long contexts (>128K tokens) are not evaluated; sparse loading benefits may differ at extreme context lengths.

Questions for Further Exploration

  1. How does SparKV interact with quantized KV caches (INT4/INT8 keys and values) — does quantization change the IO/compute balance and shift the optimal sparsity ratio?
  2. Can the MLP sparsity predictor be updated online during deployment to adapt to distribution shift, and what is the cost of such continual calibration?
  3. How does the greedy scheduler perform when loading time variance is high (e.g., due to thermal throttling on mobile devices) — does the runtime adapter fully compensate?
  4. Is there a theoretical bound on the quality of the greedy approximation relative to the MILP optimum in terms of attention score loss?
  5. How does SparKV compare to KV cache eviction strategies (e.g., H2O, Scissorhands) that discard KV entries entirely rather than selectively loading them?
