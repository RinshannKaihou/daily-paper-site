TingIS: Real-time Risk Event Discovery from Noisy Customer Incidents at Enterprise Scale

一句话总结： TingIS是蚂蚁集团部署的企业级实时风险事件发现系统，将LLM语义蒸馏、LSH高效聚类、级联路由和多维去噪四类模块串联，从每分钟2000条嘈杂用户投诉中以3.5分钟P90延迟发现95%的高优先级风险事件，并将噪声告警数从512条压缩至29条（降噪94.3%）。
TL;DR: TingIS is a production-deployed, end-to-end system that transforms noisy customer incident streams (2000/min) into structured risk events by chaining LLM-based semantic distillation, LSH+LLM clustering, cascaded domain routing, time-decay historical linking, and multi-dimensional denoising — achieving 95% recall at 3.5-minute P90 latency on a large fintech platform.

核心贡献

  • 提出"客户投诉驱动的风险事件发现"系统框架，明确了客户事件（原子噪声反馈）与风险事件（持久化结构化故障表示）的概念区分。
  • 设计了五模块级联架构（M1语义蒸馏→M2级联路由→M3事件链接→M4状态管理→M5多维去噪），各模块独立可插拔。
  • M3中提出"时间衰减加权"历史关联机制，将语义相似度与时间近邻性结合，防止"历史惯性"导致无关事件的错误合并。
  • M5多维去噪融合了误报知识库匹配、动态基线统计过滤（μ+2σ）和行为约束（含坡度穿透机制），在不牺牲高优先级召回的前提下大幅降低噪声。
  • 在蚂蚁集团金融科技平台完成生产部署，并提供工业NLP实践的详细经验总结。

Key Contributions

  • Conceptual framework distinguishing customer incidents (noisy, colloquial, atomic) from risk events (structured, persistent, expert-verified), with a defined mapping task between them.
  • A five-module architecture designed around three core principles: semantic convergence and identity persistence, synergy of hybrid intelligence (LLM + efficient classical methods), and multi-constraint SNR balance.
  • Time-decay weighting for historical event association: similarity score × e^(-k·Δt) prevents stale events from absorbing new, unrelated incidents.
  • A three-layer denoising pipeline: source suppression (false-positive KB matching), statistical filtering (dynamic μ+2σ baseline), and behavioral constraints (alert silencing with slope-based penetration).
  • Production benchmark suite: Alarm Replay Set (~50K incidents), Benchmark Events (12 annotated events), Event Identity Set (~1,400 for clustering quality), Routing Set (~3,200) — all derived from real production data.

Methodology

M1 (Semantic Distillation): Each incoming incident is normalized by an LLM (Qwen3-8B) into a structured "subject + problem" summary, stripping PII, filler, and subjective language, then embedded via M3-embedding (BGE-M3).

M2 (Cascaded Routing): A two-stage system routes each summary to a business domain (biz_code): (1) keyword matching against an "entity-priority" KB for high-precision cases; (2) multi-path vector retrieval + BGE-ReRanker-V2-M3 for recall in long-tail cases. The cascaded architecture prevents noisy keyword results from polluting the reranker's candidate pool.

M3 (Event Linking): In-batch: LSH for preliminary grouping, Kimi-K2 LLM for cluster purity checks and title generation, producing refined cluster summaries. Cross-batch: each cluster title is embedded and searched against a historical risk event knowledge base; time-decay weighted scores determine merge vs. create decisions, with LLM final adjudication.

M4 (State Management): Three-layer data model: mutable State Layer (current volume, timestamps), immutable Audit Layer (full evidence chain), and periodic Snapshot Layer (volume timeline for dynamic baselines).

M5 (Denoising): Sequential application of false-positive KB suppression, dynamic baseline thresholding, and alert silencing with slope-based penetration to handle non-linear volume surges.

Results & Highlights

  • Online production (1 month): 95% high-priority incident discovery rate, P90 alert latency 3.5 minutes.
  • Offline SNR: TingIS full (29 alerts) vs. TingIS w/o Denoising (512 alerts) — 94.3% noise reduction with no drop in detection rate. Event-to-Alert ratio of 1.23 (near-ideal 1.0) vs. 2.18 for w/o Denoising.
  • Event identity quality (B³-F1): TingIS 0.826 vs. DBSCAN 0.673, Keyword Grouping 0.745, Vector Matching 0.744. Mismerge rate reduced from 64.3% (DBSCAN) to 21.5%.
  • Routing accuracy: cascaded architecture (Acc@1: 0.669, Coverage: 0.881) outperforms parallel fusion (0.460, 0.680).
  • Ablation confirms: business partitioning is the most critical component (−15.6% B³-F1 without it), followed by LLM summary in M1 (−7.0%) and two-stage M3 LLM application (~5% combined).
  • Latency: average 12.4 seconds per batch end-to-end; LLM components account for 69.7% of latency.

Critical Assessment

Novelty: The individual components (LSH clustering, LLM-based NLU, cascaded retrieval, time-decay scoring, dynamic baselines) are all established techniques. The novelty is in the system design, specifically the principled combination for the enterprise incident discovery problem, and the time-decay weighted historical association mechanism.

Rigor: Online validation is compelling (1-month production run). Offline benchmarks are constructed from production data with SRE expert annotation. Baseline comparisons include relevant alternatives. Ablation studies are thorough.

Reproducibility: The system is deployed on proprietary Ant Group infrastructure; full reproduction is not possible. The algorithmic descriptions are detailed enough to reimplement, but without the production data and knowledge bases, reproduction is limited.

Significance: The system addresses a high-stakes, high-volume real-world problem that is poorly served by existing academic event detection literature. The 94.3% noise reduction without recall loss is a strong practical result.

Red flags: (1) The "12 benchmark events" annotation is a very small evaluation set for a system of this scale; statistical significance of detection rate claims is uncertain. (2) The biz_code taxonomy is proprietary and not released; generalizability to other enterprise settings is unclear. (3) The false-positive KB requires manual curation and may become stale under domain drift. (4) LLM component costs (8.0M tokens/day) represent a non-trivial ongoing expense not discussed in terms of cost-benefit. (5) The system's performance under adversarial or unusual failure modes (e.g., a novel failure type not resembling any historical event) is not evaluated.

Strengths

  • Production-validated at enterprise scale (300K incidents/day, 2000/min peak) with quantitative latency and recall numbers.
  • Principled three-insight design framework that balances accuracy, efficiency, and maintainability.
  • The time-decay historical association mechanism is a practical and elegant solution to event identity drift.
  • Multi-dimensional denoising achieves SNR improvement without sacrificing recall for critical events.
  • Appendix provides practitioner-oriented lessons on data skew, routing design, and responsible LLM integration.

Limitations

  • Proprietary data and knowledge bases prevent external reproduction and generalizability assessment.
  • Benchmark event set (12 events) is small for statistical confidence in recall claims.
  • LLM inference costs are substantial and not addressed in terms of operational economics.
  • The system is designed for structured financial service domains; adaptation to other verticals (healthcare, logistics) would require significant knowledge base re-engineering.
  • Alert silencing with 2-hour windows creates a fixed blind spot that may miss rapidly evolving incidents within that window (mitigated by slope penetration but not fully eliminated).

Questions for Further Exploration

  1. How does TingIS handle distribution shift when a new business line or product is launched that has no historical events in the knowledge base?
  2. Could the time-decay weighting parameter k be learned adaptively based on domain volatility rather than fixed?
  3. What is the false-positive rate for newly introduced failure types that don't appear in the false-positive KB — is there a cold-start recall penalty?
  4. How would the system perform if the LLM distillation step were replaced with a smaller, faster, locally-deployed model — what is the accuracy-latency tradeoff?
  5. Can the audit trail (M4 Audit Layer) be used to automatically detect degradation in the knowledge bases over time, enabling automated KB maintenance?
