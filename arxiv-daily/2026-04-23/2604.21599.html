Verifying Machine Learning Interpretability Requirements through Provenance

["Lynn Vonderhaar", "Juan Couder", "Daryela Cisneros", "Omar Ochoa"]

arXiv: 2604.21599v1

Date: 2026-04-23

https://arxiv.org/abs/2604.21599

一句话总结： 本文提出用 ML 溯源（Provenance）将不可量化的可解释性非功能需求（NFR）分解为可验证的功能需求（FR），通过 yProv4ML（基于 W3C PROV-O 本体的 Python 库）在线性回归和反事实解释两个实验中演示如何保存并验证溯源数据，从而隐式验证可解释性 NFR。
TL;DR: This paper addresses the challenge that ML interpretability is a Non-Functional Requirement (NFR) that is currently immeasurable and thus unverifiable. The proposed approach decomposes interpretability into provenance-based Functional Requirements (FRs) specifying what data to save during ML development, then verifies those FRs. Verification of the provenance FRs in turn verifies the interpretability NFR. The approach is demonstrated using the yProv4ML Python library (PROV-O compliant) on linear regression and counterfactual explanation experiments.

核心贡献

  • 明确识别可解释性作为 ML 非功能需求（NFR）的核心可验证性缺口：现有文献虽定义了可解释性，但未提供度量和验证方法。
  • 提出将可解释性 NFR 分解为溯源数据需求（FR），以"需要保存哪些溯源数据"作为可量化、可核查的功能需求。
  • 提供具体的需求规范模板，涵盖从线性回归（固有可解释模型）到反事实解释（DiCE，代理方法）两类典型可解释性方法的溯源需求。
  • 实现并演示了基于 yProv4ML（兼容 W3C PROV-O）的实际工具链，可输出 JSON 格式或图形格式的溯源图，支持由第三方或工具自动验证。

Key Contributions

  • Identifies that while ML interpretability is widely defined as an NFR, existing literature provides no measurement or verification method, making it an "immeasurable requirement" that cannot be definitively confirmed.
  • Proposes decomposing the interpretability NFR into concrete, verifiable provenance Functional Requirements (FRs): specifying exactly what provenance data must be saved at each stage of model development.
  • Demonstrates the approach through two concrete interpretability methods from different categories: (1) linear regression (inherently interpretable model—provenance allows behavior simulation), (2) counterfactual explanations via DiCE (feature attribution/surrogate—provenance maps input space to understand behavior indirectly).
  • Provides formal requirement specifications for both experiments (Req. 1–7 for linear regression, Req. 1–8 for counterfactuals), linking each FR to its provenance data type.
  • Implements the approach using yProv4ML, a Python library compliant with W3C's PROV-O (PROVenance Ontology), using PROV-JSON and PROV-O graph output formats.
  • Code and provenance data available at https://github.com/lynndalou/ProvInterpretExplain.

Methodology

Core insight: Interpretability is the ability for humans to draw connections among model data, training, and behavior. Storing model provenance throughout development allows humans to trace behavior from training data to model deployment. The storage of necessary provenance data forms verifiable FRs whose satisfaction implies the interpretability NFR is met.

Provenance categories (from Table II):

  • Data provenance: datasets, data type, source, fidelity, metadata, collection method, preprocessing steps, labeling, train/val/test split, class proportions.
  • Model provenance: pretraining info, model size/speed, parameters/weights/biases, hyperparameters, architecture comparisons, optimization algorithms, pipelines, execution logs, environment configurations, source code.

Tool — yProv4ML: A Python library compliant with W3C PROV-O ontology (OWL, PROV-DM). Uses three core PROV-O classes: Agent (Person, Organization, SoftwareAgent), Entity (Collection, Bundle, Plan), Activity. Provenance is serialized as PROV-JSON and can be visualized as a directed graph. Key API: yprov4ml.start_run(), yprov4ml.log_dataset(), yprov4ml.end_run(create_graph=True).

Linear Regression Experiment:

  • Model: inherently interpretable; provenance allows full behavior simulation.
  • Source: Kaggle simple linear regression example.
  • FRs (Req. 4–7): save training dataset, testing dataset, preprocessing steps, linear regression function to provenance.
  • Verification: Microsoft Excel used to independently reproduce the linear regression from saved provenance data, confirming model behavior matches Kaggle code. Full code + provenance JSON + graph at GitHub.

Counterfactual Explanation Experiment (DiCE):

  • Model: black-box binary classifier; behavior cannot be directly simulated.
  • Approach: save counterfactual explanations (input perturbations that change output) to provenance, mapping input space structure.
  • FRs (Req. 4–8): save training/testing datasets, preprocessing steps, selected features, and counterfactual explanations to provenance.
  • Verification: provenance of CE maps demonstrates understanding of input space boundaries, implicitly verifying interpretability NFR through data traceability.

Interpretability method categorization (Table I): Inherently interpretable (linear/logistic regression, decision trees/rules), surrogate models (LIME), feature attribution (SHAP, counterfactual explanations, Anchors).

Results & Highlights

  • Successfully saved and visualized provenance graphs for both linear regression (Figures 3–4: model parameters, execution metadata, training/test data) and counterfactual explanations.
  • Linear regression provenance verification: following saved provenance steps in Excel reproduces the Kaggle model's regression function (same line of best fit), thereby simulating model behavior and verifying interpretability.
  • Counterfactual explanation provenance verification: saved CEs map the input space to show which inputs lead to different outputs, providing indirect behavioral understanding without direct simulation.
  • Demonstrates that different interpretability methods require different provenance—the specific FRs are not universal but must be tailored per interpretability method and use case.
  • The approach uses existing MLOps tooling (MLflow, DVC, W&B, Neptune AI, Comet are noted as insufficient for full PROV-O provenance) and extends it with yProv4ML for PROV-O compliance.

Critical Assessment

Novelty: Moderate. The core idea—using provenance to ground interpretability requirements—is practical and fills a genuine gap in ML engineering practice. However, the technical novelty is limited: yProv4ML is a prior tool, PROV-O is a standard ontology, and the experiments are deliberately simple (linear regression on a Kaggle dataset, DiCE on a binary classifier). The paper's strength is conceptual framing rather than algorithmic contribution.

Rigor: Limited. The two experiments are proof-of-concept demonstrations with simple models. No quantitative evaluation of the approach's effectiveness, scalability, or completeness is provided. The question of whether the provenance FRs are sufficient to verify interpretability (not just necessary) is not formally addressed—there could be interpretable models whose provenance FRs are all met yet the model is still opaque in practice.

Reproducibility: Good given the GitHub release. The requirement specifications are explicit (Req. 1–7/8 listed), the tool (yProv4ML) is described, and the code is publicly available. However, the Excel verification step for linear regression is manual, not automated.

Significance: Moderate for the ML engineering community, particularly for practitioners working on safety-critical ML systems where audit trails and interpretability certifications are required. The requirements engineering framing is valuable for teams building ML systems under regulatory constraints.

Red flags: (1) The approach is only demonstrated on toy examples (simple linear regression, binary classifier). Whether the provenance FRs scale to large neural networks, transformers, or multi-modal models is not addressed. (2) The completeness question: do the specified FRs fully verify interpretability, or only provide evidence of it? The paper conflates provenance adequacy with interpretability verification without formally justifying the equivalence. (3) There is no evaluation of overhead: how much storage does provenance tracking require? What is the runtime impact? (4) The "verification" using Excel is manual and subjective—there is no automated oracle for interpretability.

Strengths

  • Fills a genuine gap: translates an immeasurable software quality attribute (interpretability) into measurable, auditable FRs.
  • Requirements engineering framing provides structure for ML compliance teams and safety auditors.
  • Tool (yProv4ML) is publicly available and PROV-O compliant, enabling standard ontology tooling.
  • Concrete requirement specifications (numbered Req.) are directly actionable.
  • Code and provenance artifacts are publicly released.

Limitations

  • Only demonstrated on two simple, small-scale examples.
  • Completeness of provenance FRs as a proxy for interpretability is asserted rather than formally justified.
  • No scalability analysis for complex models (deep neural networks, LLMs).
  • Manual verification step (Excel) is not automatable as specified.
  • The approach requires domain-specific tailoring of FRs for each interpretability method and use case—there is no universal provenance specification.
  • Does not address dynamic interpretability requirements (e.g., model updates, concept drift, retraining).

Questions for Further Exploration

  1. Can the provenance FRs be formalized as a complete and sound axiom set—i.e., is there a proof that satisfying the specified FRs is both necessary and sufficient for the interpretability NFR to hold?
  2. How does the approach scale to large language models where the "model provenance" (billions of parameters, pretraining data) is partially proprietary and incomplete?
  3. Can yProv4ML be extended to automatically verify the provenance FRs against a specification, replacing the manual Excel step with an automated compliance check?
  4. How should the provenance FRs differ for post-hoc interpretability methods (SHAP, LIME) where the explanation depends on the test instance and may not be globally reproducible?
  5. What is the relationship between this provenance-based approach and existing model cards / data cards standards—can provenance graphs automatically generate structured documentation?
