> **一句话总结：** 本文提出 TRACE，一个将用户在会话中给出的自然语言纠正实时挖掘为原子规则、并编译成编码智能体完成任务前必须通过的运行时检查的 skill 层，从而把“记住偏好”升级为“强制遵守偏好”。

> **TL;DR:** This paper introduces TRACE, a skill layer that mines natural-language user corrections, converts them into atomic rules, and compiles them into runtime checks that a coding agent must pass before completing future tasks—showing that remembering a preference is not enough; it must be enforced.

## 核心贡献

- 识别并量化了交互式编码智能体中“偏好可访问但未被遵守”的缺口：在基于真实用户摩擦案例构造的 19 个诊断任务（29 条偏好检查）上，Mem0 记忆仅能让遵守率达到 42.5%，而直接把规则写入上下文的基线也仅达 55.0%。
- 提出 Test-time Rule Acquisition and Compiled Enforcement（TRACE），一个可即插即用到 Claude Code / Codex CLI 等编码智能体运行时的 skill 层，将用户纠正转换为持续增长的个人规则库。
- 在 ClawArena 和 MemoryArena 派生任务上验证：TRACE 显著降低偏好违反率，同时保持或提升任务通过率，且在分布外编码任务上把违反率从 100% 降至 2%。

## Key Contributions

- Identifies and quantifies the access–compliance gap in interactive coding agents: on 19 held-out diagnostic tasks with 29 preference checks built from real-user friction cases, Mem0 achieves only 42.5% compliance, and even prompting with all rules reaches only 55.0%.
- Presents Test-time Rule Acquisition and Compiled Enforcement (TRACE), a drop-in skill layer for coding-agent runtimes (Claude Code and Codex CLI) that converts user corrections into a persistent, per-user library of atomic, runtime-enforceable rules.
- Validates TRACE on ClawArena and MemoryArena-derived tasks, showing large reductions in repeated preference violations while preserving or improving task pass rates, including a drop from 100% to 2% violation on out-of-distribution coding tasks.

## Methodology

- **Problem formulation.** The paper frames the failure mode as *preference access ≠ preference compliance*. Memory and retrieval make a correction visible, but visibility does not force the agent to satisfy it. TRACE therefore changes the representation of a correction from advisory natural language into an executable constraint.
- **Pipeline overview.** TRACE is a skill layer inserted into the agent runtime. It scans the live user–agent interaction stream; ordinary messages pass through, while correction signals are routed to a compiler.
- **Correction-signal detection.** A lightweight LLM (Gemma 4 31B served via DeepInfra, configurable via `COMPILE_MODEL`) evaluates every user utterance for durable-preference signals such as repeated errors, workflow friction, or explicit instructions. Non-signals are logged but ignored.
- **Atomic-rule extraction.** For each detected correction, TRACE builds a correction record linking the feedback to the agent behavior and workspace state, then extracts an atomic rule paired with an applicability condition (e.g., “clean up temporary files only when the agent has created artifacts”).
- **Rule lifecycle resolver.** Because corrections arrive incrementally, each candidate rule is reconciled against the current per-user library via one of five actions: **Noop** (add evidence), **Update** (compatible refinement), **Supersede** (conflict, archive old rule), **Split** (multiple distinct preferences), or **New**.
- **Compiled enforcement.** Each rule is compiled into three functional components: an applicability check, a behavior instruction injected into the agent context, and a verifier registered to runtime hooks (prompt, tool-use, file-write, termination). Three enforcement tiers are supported: deterministic (regex / file-state / command checks), semantic (model-based evaluation of text/files), and intent-level (runtime reminders). The deployed 47-entry library uses deterministic verifiers with a verify-retry marker that blocks the hook event on failure.
- **Execution.** At task start, applicable rules are matched to the task and loaded. During execution, hooks intercept matching events; a failing verifier reports the violation and forces revision. The task is allowed to terminate only after all active verifiers pass. Retries are capped (3 in the diagnostic; bounded by a two-turn simulator budget in the main experiments).
- **Implementation safeguards.** Compilation uses a four-stage self-verify loop: JSON parse, schema check, unit test on canonical inputs, and sandbox replay of the originating correction-conflict pair. The deployed library contains 47 entries: 37 atomic, 7 refinement, and 3 composite.

## Results & Highlights

- **Access-compliance diagnostic (Section 3).** Built from 32 long-context coding-agent transcripts (~1M tokens each) yielding 142 correction-conflict records, filtered to 19 held-out tasks with 29 manually annotated preference checks.
  - No Rules: 31.6% compliance.
  - All Rules (full rule set in context): 55.0%.
  - Mem0 memory backend: 42.5%.
  - Relevant Rules (only applicable rules in context): 54.0%.
  - **Compiled Rules: 70.1%**, demonstrating that operational, enforceable representations outperform retrieval or prompt-based access.
- **ClawArena in-distribution (Section 5.1).**
  - TRACE violation rate: **37.6%** vs. >50% for all memory baselines (three of four >94%).
  - Task pass remained within the same narrow band as No Memory and ReMe-Light, showing enforcement does not degrade coding success.
- **ClawArena out-of-distribution (Section 5.2).**
  - TRACE violation rate: **2.0%**, an order of magnitude below the next baseline, with zero observed violations on three of four models.
  - Task pass again comparable to no-memory, indicating clean transfer of compiled rules to unseen scenario families.
- **MemoryArena-derived tasks in-distribution.** Because task pass here includes satisfying hidden user/project constraints, enforcement and pass are coupled. TRACE achieved the highest task pass (**17.3%**) and the lowest violation rate (**60.5%**).
- **MemoryArena-derived tasks out-of-distribution.** Task-pass scores clustered within one point, but TRACE was the only method below 99% violation (**97.0%**) and below 90% mean corrections (**86.5%**), suggesting marginal benefit even when many constraints are uncovered.
- **Test-time efficiency (ClawArena, Section 5.3).**
  - Mean user turns per round dropped from 2.00 (No Memory) to **1.37 ID** and **1.02 OOD** for TRACE; Mem0 closed part of the gap (1.51 ID, 1.31 OOD).
  - Average wall-clock time: **42.5s ID** and **41.8s OOD** for TRACE, within seconds of No Memory and below all memory baselines except comparable ones (ReMe-Light was an outlier at 228s ID / 174s OOD due to retrieval overhead).
- **Simulated-user fidelity (Appendix A).** Validated on 790 held-out judgments (357 correction, 433 non-correction cases): Precision 0.864, Recall 0.953, F1 0.906, Specificity 0.940; rule recall on positive examples was 0.668.

## Critical Assessment

- **Novelty and conceptual contribution.** TRACE’s central insight—that personalization should couple memory with runtime enforcement—is well motivated and distinct from prior work that treats memory and guardrails as separate systems. The paper explicitly positions itself against both retrieval-based personalization and pre-authored guardrails, carving out a useful middle ground.
- **Empirical strength.** The OOD results on ClawArena are striking: a 2% violation rate with preserved task pass is a strong signal that compiled rules transfer across unseen coding scenarios. The efficiency analysis also makes a practical case that enforcement does not introduce prohibitive latency.
- **Rigor and reproducibility.** The authors detail the diagnostic construction, simulator validation, rule-library structure, and resolver protocol, and provide code/skill links. However, the detector is not evaluated on a held-out conversational corpus (acknowledged in Appendix C.1), and the diagnostic corpus is single-user, so generalization across users with conflicting preferences is not tested.
- **Limitations and red flags.**
  - The MemoryArena numbers (even ID task pass of 17.3%) are low in absolute terms, suggesting the augmented tasks remain hard and that enforcement is not a panacea.
  - OOD violation on MemoryArena is still 97%, indicating that compiled rules only help when the relevant rule exists.
  - The semantic and intent enforcement tiers are described but not required in the 47-rule snapshot; the paper’s empirical claims are therefore really about deterministic checks.
  - Rule conflict resolution relies on an LLM-based resolver and manual audit/rollback; at scale, mis-classified Supersede decisions could silently change agent behavior.
  - There is no user study on acceptability of blocking task completion, nor on how users manage a growing rule library over months.

## Strengths

- Targets a concrete, daily-use failure mode—users repeating the same correction across sessions—with a clear engineering solution.
- Strong, reproducible empirical design: diagnostic + two benchmarks × ID/OOD × efficiency.
- Excellent out-of-distribution compliance on coding tasks (2% violation) without task-pass degradation.
- Low runtime overhead and reduced user-turn cost, making enforcement practical for real coding agents.
- Modular, drop-in skill architecture for popular runtimes (Claude Code, Codex CLI) with open-source artifacts.

## Limitations

- Diagnostic derived from a single user’s transcripts; multi-user preference conflicts are not studied.
- No held-out evaluation of the correction-signal detector itself; detector errors are a clear failure mode.
- Semantic/intent enforcement tiers are not exercised in the reported experiments, limiting claims about complex, non-deterministic rules.
- MemoryArena OOD results show high residual violations, so coverage of the rule library remains a bottleneck.
- Conflict resolution and rollback currently require manual audit; auto-rollback is not implemented.
- No human-in-the-loop user study on transparency, control, or frustration when tasks are blocked.

## Relevance

- 本文对从事编码智能体、AgentOps、LLM 安全与人机协作的研究者和工程师极具参考价值。它把“个性化”从提示工程/记忆检索推进到运行时执行约束，为构建长期可用、越用越顺手的智能体协作系统提供了可直接落地的范式。

## Questions for Further Exploration

- How does TRACE scale to thousands of rules across many users, and what indexing or conflict-resolution mechanisms are needed?
- Can the correction-signal detector be benchmarked on a held-out, multi-user conversational corpus, and what is its precision/recall trade-off?
- How well do semantic/intent-level enforcement tiers perform on rules that cannot be expressed as deterministic checks?
- What is the user experience of living with a growing enforced rule library—do users accept blocked completions, and how do they edit or revoke rules?
- Can the same compile-and-enforce paradigm be extended to non-coding agents (e.g., research assistants, customer-service agents) with different action spaces?
- How can TRACE handle contradictory corrections from multiple stakeholders on shared projects?
