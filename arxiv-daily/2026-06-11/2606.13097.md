> **一句话总结：** FCGraft 通过维护函数级已验证代码骨架的双层 KV 缓存，并用“拼接（stitching）+ 局部修补（patching）”合成新策略，在开放域具身任务上同时实现了比 RAGCache 高 18.31% 的成功率与 2.3 倍的延迟降低。

> **TL;DR:** FCGraft maintains a two-tier function-level KV cache of validated code skeletons and synthesizes new embodied-agent policies by cache-stitching and cache-patching, achieving 18.31% higher success rate and 2.3× lower latency than RAGCache on open-domain benchmarks.

## 核心贡献

- 提出 **FCGraft（Functional Cache Grafting）**，将 CodeLLM 的 KV 缓存从 prompt 级推进到**函数级**，支持开放域具身任务的快速、鲁棒代码策略合成。
- 设计**双层代码缓存** `H = (I, C)`：Function-Interface 层 `I` 存储函数签名与 KV cache，Function-Code 层 `C` 存储完整实现与 KV cache。
- 提出 **cache-stitching（缓存拼接）** 与 **cache-patching（缓存修补）** 两个相互依赖的阶段：stitching 通过复用已验证函数消除结构性错误并避免重复 prefill；patching 在执行异常时仅对错误片段做最小化解码修复。
- 在 ALFRED、TEACh、RLBench 以及真实 Franka 机器人操作任务上验证，统一在成功率（SR）与策略合成延迟（PSL）上取得最佳权衡。

## Key Contributions

- Introduces **FCGraft**, a function-level KV caching framework that moves beyond prompt-level caching for CodeLLM-based embodied control.
- Designs a **two-tier code cache** `H = (I, C)`: the Function-Interface tier `I` stores callable signatures and KV states; the Function-Code tier `C` stores validated implementations and KV states.
- Proposes **cache-stitching** and **cache-patching** as interdependent stages: stitching composes validated function segments to eliminate internal errors and redundant prefill; patching localizes and regenerates only erroneous spans when runtime exceptions occur.
- Demonstrates strong empirical results across ALFRED, TEACh, RLBench, and real-world Franka manipulation, consistently improving success rate while reducing policy synthesis latency.

## Methodology

**Problem formulation.** The open-domain embodied task is formalized as a tuple `(D, S, A, F)`, where `D` is the set of domains, `S` the state space, `A` the action space, and `F(d) = {Ω_d, G_d, P_d}` maps a domain to its observation function, goal states, and transition function. A CodeLLM `π_θ` generates an executable code policy `π_code` from observation `o_t` and instruction `τ_d`. The idealized objective trades off task success rate (SR), policy synthesis latency (PSL), and code similarity (CSIM):

```
π_θ* = arg max E_{d~D} E_{π_code~π_θ(·|τ_d,o_t)} [ SR(Exec(π_code), g_d) − η PSL(π_θ) + μ CSIM(π_θ) ]
```

**Two-tier code cache.** After each successful execution, FCGraft decomposes the generated policy into reusable functions and stores them in:

```
H = (I, C),
I = {(f_k, i_k^text, i_k^KV)},
C = {(f_k, c_k^text, c_k^KV)}
```

`I` provides lightweight modular interfaces that can be reinjected into the model without recomputing cross-attention, mitigating attention-sink issues when multiple cached functions are composed. `C` retains full implementations, linked at execution time and used as prefix context during patching.

**Cache management.** Each function receives a locality score combining recency, invocation frequency, conditional co-occurrence with other functions, and semantic relevance (via perplexity):

```
ℓ(f_k) = (1 − ℓ_curr(f_k)) · (α·ℓ_freq(f_k) + β·Σ_j ℓ_asso(f_j|f_k) + γ·ℓ_sema(f_k)) + ℓ_curr(f_k)
```

Low-scored entries (especially heavy `c^KV` in `C`) are offloaded to DRAM, while high-scored entries stay on GPU for fast reuse.

**Cache-stitching.** Given `o_t`, `τ`, and the interface KV states `I^KV`, the CodeLLM generates a high-level policy `π_code` that calls cached functions. The final executable program `π_exec` is assembled by linking the corresponding implementations from `C`:

```
π_code ~ π_θ(· | o_t, τ, I^KV)
π_exec = π_code || { c_k^text | f_k ∈ Call(π_code) }
```

Because internal implementations are reused, errors in stitched policies are mainly surface-level (parameter mismatches, variable values), which can be repaired efficiently.

**Cache-patching.** When execution raises an exception `E`, `π_exec` is split into prefix `x_pre^text`, erroneous span `x_err^text`, and suffix `x_suf^text`. FCGraft reuses the prefix KV state and generates only a corrected middle span `x_mid^text` guided by a chain-of-thought root-cause analysis, then reassembles:

```
π_exec' = x_pre^text || x_mid^text || x_suf^text
x_mid^text ~ π_θ(x_pre^KV, CoT(o_t, τ, c_k^KV, E))
```

This bypasses prefill for the preserved prefix and focuses decoding on the localized error span.

## Results & Highlights

**Simulation benchmarks.** FCGraft is evaluated on ALFRED, TEACh, and RLBench under three open-domain scenarios of increasing difficulty: **Open-Composition** (curriculum-style composition), **Open-Perturbation** (unexpected object-state changes), and **Open-Evolution** (varying observations and goals). Across all benchmarks:

- **vs. RAGCache:** +18.31% average success rate (SR) and **2.3× reduction** in policy synthesis latency (PSL).
- **ALFRED:** +4.04% SR and **2.67× lower PSL** vs. LRLL, the second-best baseline by rank.
- **RLBench:** +16.21% SR and **2.25× lower PSL** vs. CAG.
- **TEACh:** similar improvements to ALFRED (reported in Appendix 14).

**Real-world robot experiments.** Using a 7-DoF Franka Emika Research 3 on office-desk rearrangement and cooking-workstation preparation tasks:

- **+37.04% SR** vs. CaP.
- **2.88× lower PSL** vs. RAGCache.
- Reliable performance across 9 trials, with qualitative examples showing cache-stitching composing reusable functions and cache-patching correcting runtime errors (e.g., replacing an inconsistent object-region parameter `drawer_high` with `drawer_top`).

**Behavioral consistency.** Snapshot evaluations over a 27-task continual stream show:

- Positive **forward transfer (FWT)** — newly cached functions help future tasks.
- Non-negative **backward transfer (BWT)** — patching’s localized corrections do not disrupt prior code.
- High **code similarity (CSIM)** — stitching preserves validated structural templates.

**Ablations (Table 5).**

- Removing `I`, `C`, or the entire cache `H` causes substantial SR drops, confirming the two-tier design is essential.
- Removing the semantic term `ℓ_sema` reduces cache hit rate and SR.
- Replacing semantic scoring with pure retrieval degrades hit rate and raises latency.
- Replacing cache-patching with full regeneration substantially increases PSL and reduces SR.

**Model choice.** FCGraft works with Qwen2.5-Coder-14B, Gemma, Llama 2, and DeepSeek. CodeLLM variants consistently outperform general-purpose LLM variants (+4.67% SR, 1.08× lower PSL, +8.31% hit rate on average) and CaP.

## Critical Assessment

**Novelty and significance.** The core idea — reusing function-level KV caches rather than whole-prompt caches — is well motivated and technically distinct from prior KV-cache methods such as RAGCache, PromptCache, and EPIC. The decomposition into stitching (structural composition) and patching (localized repair) is a clean architectural choice that addresses both latency and robustness simultaneously. The paper also connects neatly to the skill-learning and continual-learning literature.

**Rigor.** The empirical design is comparatively thorough: three simulation benchmarks, three open-domain scenarios each, real-robot validation, ablations, and a behavioral-consistency analysis. Standard deviations over three seeds are reported. However, several caveats remain:

- **Small real-world sample:** only 9 trials per scenario are reported, so the impressive real-world gains have limited statistical power.
- **Benchmark-specific prompts and oracle prior information:** All methods, including FCGraft and the KV-cache baselines, are initialized with the same “external oracle policies for basic functions” supplied in each method’s native format. This helps fairness but still means performance depends heavily on the curated starting library.
- **Reproduced/adapted baselines:** Some baselines (notably EPIC) lack public implementations and were reproduced from the paper, which can introduce implementation bias.
- **No formal significance testing:** The paper reports means and standard deviations but does not state whether differences are statistically significant.
- **Metric coupling:** The Rank metric combines SR and inverse-normalized PSL with equal weights; this is a reasonable composite but not the only possible trade-off.
- **Generality beyond compositional tasks:** Cache-stitching is most natural when tasks decompose into reusable functions. Tasks requiring genuinely novel control structures not present in the cache may not benefit as much.

**Reproducibility.** Without released code, exact reproduction is difficult, though the appendix provides substantial implementation detail (prompt construction, cache decomposition, server architecture, baseline recipes).

## Strengths

- Tackles the two main practical barriers to Code-as-Policies deployment — latency and generative brittleness — with a unified function-level cache-grafting approach.
- Strong and consistent empirical gains across three simulators and a real robot, balancing success rate and latency.
- Clean separation between stitching (structural reuse) and patching (localized adaptation), enabling both efficiency and continual improvement without catastrophic forgetting.
- Thoughtful cache-management score that combines recency, frequency, association, and semantic diversity.
- Detailed appendix covering baselines, metrics, and system configuration.

## Limitations

- **No public code or artifact release** for FCGraft, hindering independent verification.
- **Real-world evaluation is small-scale** (9 trials), limiting confidence in physical-robot generalization.
- **Reliance on a curated initial function library** derived from oracle policies; performance may degrade if the initial library is incomplete or poorly matched to the deployment domain.
- **Isolated per-agent cache:** the current design does not share caches across agents, which the authors identify as a future direction.
- **Static model and greedy decoding:** all experiments use a single model size and sampling strategy; robustness to model scale, stochastic sampling, and multi-GPU serving is not explored.
- **Task decomposition assumption:** benefits are largest for tasks that can be expressed as compositions of cached functions; highly novel tasks may still require near-full generation.

## Relevance

对于关注具身智能体、Code-as-Policies、LLM 代码生成与推理加速的研究者来说，FCGraft 把 KV cache 复用从“prompt 级”下沉到“函数级”，并提供了一套可解释、可累积的策略合成机制，是在开放域机器人任务中同时提升成功率与响应速度的有力候选方案。

## Questions for Further Exploration

- How sensitive is FCGraft to the quality and coverage of the initial function library? Can it bootstrap from weak or human-annotated examples?
- What is the transfer curve as the cache grows — at what cache size do hit rates and success rates saturate?
- How does FCGraft perform under stochastic decoding (temperature > 0) or larger models where prefill latency differs?
- Can the two-tier cache be compressed or quantized without losing stitching/patching fidelity?
- How would cross-agent cache sharing affect performance in multi-robot or collaborative settings?
- Can the locality score be learned online rather than hand-weighted, and would it improve cache hit rate further?
