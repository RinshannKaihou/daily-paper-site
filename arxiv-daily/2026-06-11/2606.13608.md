> **一句话总结：** AgentBeats 提出“Agentified Agent Assessment (AAA)”框架，通过 Judge Agent 与 A2A/MCP 标准协议把 agent 评估从固定 harness 改造成开放、可互操作、可复现的体系，并用 5 个月竞赛（298 个 judge agent、467 个 subject agent）和 coding case study 验证了大规模可行性。

> **TL;DR:** AgentBeats proposes Agentified Agent Assessment (AAA): judge agents evaluate subject agents via standardized A2A task management and MCP tool protocols, replacing rigid LLM-centric harnesses with an open, interoperable, and reproducible evaluation framework validated by a 5-month competition (298 judge agents, 467 subject agents) and a controlled coding-agent case study.

## 核心贡献

- 提出 **Agentified Agent Assessment (AAA)** 范式：把 benchmark 本身视为 agent，由 **judge agent**（评估 agent）对 **subject agent**（被测 agent）进行任务分发、环境准备、交互监控与结果打分；delegator 只负责发起评估请求。
- 通过生产级协议实现统一接口：**A2A** 负责任务管理、进度追踪与双向通信；**MCP** 负责工具访问。这样把传统 N×M 的 agent-benchmark 集成工作量降到 N+M 的协议级集成。
- 引入 **AgentBeats** 作为 AAA 的具体实现，提出五种运行模式：Local、Remote、Hosted、Proxy、CI，分别适配本地开发、远程私有 agent、公开蓝图、开发调试与可审计 CI 评估等场景。
- 论证 AAA 支持两种评估逻辑“内部化”方式：**programmatic internalization**（精确复刻原 benchmark 行为）与 **semantic internalization**（用自然语言 prompt 表达评估逻辑，支持 LLM-as-judge）。
- 通过 5 个月开放竞赛（AgentX-AgentBeats）与 coding agent case study，验证 AAA 在**覆盖度（coverage）、实用性（practicality）、保真度（fidelity）**三个维度上的大规模可行性。

## Key Contributions

- Proposes **Agentified Agent Assessment (AAA)**: benchmarks themselves become agents; a **judge agent** orchestrates task distribution, environment setup, interaction monitoring, and scoring for **subject agents**, while a delegator only initiates the request.
- Achieves a unified interface through production protocols: **A2A** for task management, progress tracking, and bidirectional communication; **MCP** for tool access. This reduces integration cost from N×M agent-benchmark pairs to N+M protocol-level integrations.
- Introduces **AgentBeats**, a concrete realization of AAA, with five operation modes: Local, Remote, Hosted, Proxy, and CI, covering local development, remote private agents, public blueprints, development debugging, and auditable CI-based assessment.
- Shows that AAA supports two forms of evaluation-logic internalization: **programmatic internalization** (hard-coding original benchmark logic) and **semantic internalization** (using natural-language prompts, enabling LLM-as-a-judge).
- Validates AAA at scale on three dimensions—**coverage, practicality, and fidelity**—through a five-month open competition and a focused coding-agent case study.

## Methodology

### AAA 角色与交互流程

论文把一次评估抽象为三个角色：

1. **Delegator**：选择 judge agent 与 subject agent，发起 A2A kick-off 请求。
2. **Judge agent**：实现 agentified benchmark，负责数据采样、环境准备（MCP server）、任务分发、交互监控、性能打分与结果报告。
3. **Subject agent**：被测 agent，只需支持 A2A 任务接收与 MCP 工具调用。

五步交互流程：

1. **Evaluation request delegation**：delegator 发送 kick-off A2A 消息。
2. **Task instruction distribution & access authorization**：judge agent 准备环境（启动 MCP server）并分发任务。
3. **Task fulfillment**：subject agent 在时限内完成任务，可与 judge agent 多次交互、调用 MCP 工具。
4. **Performance scoring**：judge agent 回收权限，检查最终状态与响应，按指标打分。
5. **Result reporting**：judge agent 向 delegator 返回结果。

### AgentBeats 五种运行模式

| 模式 | 场景 | 特点 |
|------|------|------|
| **Local** | 本地开发 / 私有 on-prem | 所有服务跑在同一主机，轻量脚本作 delegator |
| **Remote** | 公开评估私有 agent | agent 部署在远程服务器，向平台注册 URL |
| **Hosted** | 公开评估公开蓝图 | 提交 Git/Docker，平台负责实例化与健康检查 |
| **Proxy** | 本地开发对接远程 agent | 本地临时 A2A 服务通过平台反向代理与其他 agent 通信 |
| **CI** | 可审计公开评估 | 利用 GitHub Actions 等 CI 基础设施自动执行 |

### Coding Agent Case Study 设计

- 评估四个模型-harness 组合：Claude Opus 4.7 × Claude Code、GPT-5.4 × Codex CLI、Gemini 3.1 Pro × OpenCode、Qwen3.5-397B-A17B × mini-SWE-agent (MSA)。
- 在三个 benchmark 上进行 agentified 评估：DevEval（函数级代码生成）、SWE-Bench Pro（仓库级 issue 修复）、Terminal-Bench 2.0（终端工作流）。
- 设计模式：judge agent 准备隔离容器与代码库；subject agent 可选择“注入容器运行”或“远程 shell 驱动”。
- 额外做 **harness-swapping experiment**：把 GPT-5.4 与 Claude Code 配对、Opus 4.7 与 Codex 配对，测量模型-harness 共适配（co-adaptation）效应。

## Results & Highlights

### 大规模田野研究（Field Study）

- 竞赛历时 5 个月，共吸引 **298 个 judge agents** 与 **467 个 subject agents**，覆盖 **12 个类别**（coding、web、browsing、healthcare、finance、multi-agent games、cybersecurity 等）。
- 多个知名 benchmark 被成功 agentified：Tau²-Bench、MedAgentBench、FinanceAgent、OfficeQA、PersonaGym、OSWorld 等。
- 提交语言以 Python 为主（judge 99%、subject 98%），但也出现 TypeScript、Rust 实现，说明协议级约束不限制实现语言。
- **78% 的 judge agents** 包含至少一个 prompt，提示 semantic internalization 被社区广泛采用；prompt 平均 241 行代码 / 11.6k 字符，约占代码字符数的 5%。
- Tau²-Bench 收到 347 次评估、42 个不同 subject agent，16 名开发者迭代提交 ≥10 个版本。

### Coding Agent Case Study 结果

| Benchmark | 最佳组合 | 关键结果 |
|-----------|---------|---------|
| **DevEval** | GPT-5.4 × Codex CLI | 94.8% solve rate，$0.20/inst |
| **SWE-Bench Pro** | Claude Opus 4.7 × Claude Code | 69.1% solve rate，$2.28/inst |
| **Terminal-Bench 2.0** | Claude Opus 4.7 × Claude Code | 68.5% solve rate |

- **没有单一系统在所有 benchmark 上领先**：GPT-5.4 在 DevEval 领先，Opus 4.7 在 SWE-Bench Pro 与 Terminal-Bench 2.0 领先。
- **成本差异在 2.4× 以内**，SWE-Bench Pro 与 Terminal-Bench 2.0 显著贵于 DevEval。
- **Gemini 3.1 Pro × OpenCode 在 DevEval 上出现成本异常**：$0.52/inst 最高，但 solve rate 仅 88.8%（最低）；轨迹分析显示仓库探索低效且 tool-call 错误率高达 2%（其他 agent 的 10 倍以上）。
- **与原公开记录整体一致**，差异主要源于：memorization screen、SWE-Bench Pro 修正版本、thinking effort 设置不同。

### Harness-Swapping 实验

- 在 6 组对比中，**原生（native）model-harness 配对在 5 组中表现最佳**，平均领先 swapped 配对 **5.3 个百分点**。
- Terminal-Bench 2.0 是唯一例外：Codex 在默认时限内完成更多任务；但给足时间后，Opus 4.7 × Claude Code 达到 79.8%，Opus 4.7 × Codex 为 73.6%，原生优势恢复。
- **GPT-5.4 在原生 Codex harness 下输入 token 消耗显著高于 swapped 到 Claude Code**，可能与 Codex 的大块文件读取策略有关。

## Critical Assessment

- **协议依赖风险**：AAA 的核心价值建立在 A2A/MCP 成为事实标准的前提下。若社区出现多个互不兼容的协议扩展，N+M 的集成优势会被削弱。
- **Judge agent 质量决定评估质量**：judge agent 也是 LLM 驱动，其 prompt、budget、tool 设计会系统性影响评分；论文未报告不同 judge agent 配置下的评估方差或 inter-rater 一致性。
- **竞赛数据的解释深度有限**：虽然 298/467 的规模令人印象深刻，但论文未详细披露有效提交比例、失败原因、重复提交去重策略，以及 leaderboard 排名的稳定性分析。
- **公开记录的差异归因较合理**：论文对与原公开记录不一致之处给出了具体解释（memorization screen、数据集版本、thinking effort），体现了 intellectual honesty。
- **Harness-swapping 实验规模受控**：为了控制成本，DevEval 与 SWE-Bench Pro 使用 20% 子样本，结论的代表性需要更大样本验证。
- **缺少对恶意行为的系统讨论**：开放竞赛中可能存在针对评分规则的 reward hacking、数据污染或 collusion，论文仅在 extensions 中提及 governance，未深入。
- **成本与可扩展性 trade-off 被承认**：judge agent 越智能、越通用，评估的 orchestration 成本越高；论文建议 case-by-case 分析，但未给出通用预算公式。

## Strengths

- 直面当前 agent 评估碎片化的根本问题，提出了从协议层统一评估接口的系统思路。
- 通过真实社区竞赛验证可行性，是少数把“开放生态”落到大规模实验的 agent 评估工作。
- 五种运行模式设计务实，兼顾 local 开发、远程私有服务、公开蓝图、CI 审计等多场景。
- Coding case study 提供了有价值的 head-to-head 对比与“模型-harness 共适配”这一具体研究洞察。
- 与 CUBE 等并行标准的关系讨论清晰，展示了良好的生态兼容意识。

## Limitations

- 摘要和正文对具体评估指标、置信区间、统计显著性检验披露较少，主要依赖 solve rate 与平均成本。
- 竞赛覆盖虽广，但 12 个类别的 judge agent 质量参差不齐，难以保证所有 agentified benchmark 的可靠性。
- 对于需要复杂 UI 交互、长视频/多模态输入的 benchmark，A2A/MCP 的协议开销与实际可行性仍需更多案例验证。
- 未提供针对非 A2A/MCP native agent 的迁移成本量化数据。
- 对 closed-source / proprietary judge agent 的可审计性与结果可复现性讨论有限。

## Relevance

- 如果你关注 LLM agent 的评估标准化、开放 benchmark 生态、多 agent 系统互操作性，或者正在设计需要跨模型/跨 harness 公平对比的评估流程，AgentBeats 提供了一个从协议出发、经过大规模社区验证的实用框架。

## Questions for Further Exploration

- 不同 judge agent（模型能力、推理 budget、tool access）对评估一致性与排名稳定性的影响有多大？能否建立 judge agent 的元评估协议？
- 在 AgentBeats 框架下，如何设计抗 reward hacking、collusion 与数据污染的机制？
- 相比传统固定 harness，AAA 在延迟、token 成本、部署复杂度上的具体 trade-off 能否被系统量化？
- 哪些任务类别最能从 A2A/MCP 标准化评估中受益，哪些仍需要保留 domain-specific harness？
- 能否将 AAA 与 SWE-bench、GAIA、OSWorld 等主流 benchmark 进行标准化对接，并发布可复现的 baseline？
- Harness-swapping 中观察到的 co-adaptation 效应是否也存在于非 coding 领域（如 browser、computer-use agents）？
