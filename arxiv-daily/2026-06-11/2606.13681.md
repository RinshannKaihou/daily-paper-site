> **一句话总结：** 本文提出 EvoArena 多领域动态环境基准与 EvoMem 补丁式记忆机制，通过显式记录记忆覆盖、更新理由与证据，使大语言模型智能体在演化终端工作流、软件仓库与用户偏好场景中显著提升鲁棒性，并泛化到 GAIA、LoCoMo 等标准基准。

> **TL;DR:** This paper introduces EvoArena, a multi-domain benchmark for persistent environment evolution, and EvoMem, a patch-based memory paradigm that records overwritten memory states, update rationales, and supporting evidence, substantially improving LLM-agent robustness on evolving terminal workflows, software codebases, and user preferences while also generalizing to standard benchmarks such as GAIA and LoCoMo.

## 核心贡献

- 提出 **EvoArena** 基准套件，将静态基准扩展为持续演化的版本链，覆盖终端工作流（Terminal-Bench-Evo）、软件工程（SWE-Chain-Evo）和个性化对话记忆（PersonaMem-Evo）三大领域。
- 提出 **EvoMem**，一种轻量级、类似 Git 的补丁式记忆范式：在基础记忆系统之上维护追加式补丁历史，记录非累加性记忆更新（更新前状态、更新后状态、更新理由、语义摘要、触发证据）。
- 在 EvoArena 上验证当前智能体在持续演化环境中表现有限（平均 step 准确率 39.6%），而 EvoMem 在 Terminal-Bench-Evo、SWE-Chain-Evo、PersonaMem-Evo 上分别平均提升 2.4%、0.5%、1.8% step 准确率，并在 chain 级别平均提升 3.7%。
- 在标准基准上验证 EvoMem 的泛化能力：GAIA 平均提升 6.1%，LoCoMo 平均提升 4.8%。
- 通过机制分析揭示 EvoMem 的价值：当智能体真正将检索到的补丁过渡信息操作化时增益最大；在 SWE-Chain-Evo 上减少 PASS_TO_PASS 回归；在 PersonaMem-Evo 上提升时间轨迹与多模式综合问题的证据保留。

## Key Contributions

- Introduces **EvoArena**, a benchmark suite that extends static benchmarks into persistently evolving version chains across terminal workflows (Terminal-Bench-Evo), software engineering (SWE-Chain-Evo), and personalized conversational memory (PersonaMem-Evo).
- Proposes **EvoMem**, a lightweight git-like patch-based memory paradigm that augments a base memory system with an append-only patch history recording non-additive memory updates: pre-update state, post-update state, update rationale, semantic summary, and triggering evidence.
- Shows that current agents struggle under persistent evolution on EvoArena (39.6% average step accuracy), while EvoMem improves average step accuracy by 2.4%, 0.5%, and 1.8% on Terminal-Bench-Evo, SWE-Chain-Evo, and PersonaMem-Evo respectively, and improves chain-level accuracy by 3.7% overall.
- Demonstrates generalization to standard benchmarks: GAIA improves by 6.1% on average and LoCoMo by 4.8% on average.
- Provides mechanistic evidence for EvoMem's value: gains are largest when agents operationalize retrieved transition patches; it reduces PASS_TO_PASS regressions on SWE-Chain-Evo and improves preservation of temporal and dispersed preference evidence on PersonaMem-Evo.

## Methodology

**EvoArena Construction.**

The authors identify three complementary evolution regimes and convert existing static benchmarks into versioned evolution chains:

1. **Terminal-Bench-Evo** (executable workflow evolution). Starting from 89 original Terminal-Bench tasks, the authors extract a structured workflow state (objective, files, dependencies, interfaces, validation rules), design an ordered evolution plan over mutable components (I/O/protocol, CLI/API, dependency/toolchain, workspace/module, policy/semantic), realize each version by inheriting the previous version's environment, and apply quality control plus oracle validation. This yields 352 evolved versioned tasks across chains of mean length 4.96. The largest change categories are I/O/protocol (49.1%) and workspace/module changes (13.4%).

2. **SWE-Chain-Evo** (software evolution). The authors select 12 active GitHub repositories, extract contiguous commit windows, group related commits into coherent milestones, synthesize SWE-bench-style task descriptions, build Docker evaluation environments with Fail-to-Pass and Pass-to-Pass tests, and assemble milestones into chronological chains using oracle-state progression. This yields 50 evolution chains with 493 chain-step instances (145 unique milestones), mean chain length 9.86, across Go (81.9%) and Python (18.1%) repositories. Notably, 29.8% of non-initial milestones modify files touched by earlier milestones, creating realistic cross-step dependency.

3. **PersonaMem-Evo** (preference evolution). Built on PersonaMem-v2, the authors sample personas, generate categorized preferences, construct implicit multi-turn conversation histories, introduce ordered preference trajectories over five change families, generate four question types (single-pattern transfer, multi-pattern synthesis, conflict resolution, temporal trajectory prediction), and apply dual-blind filtering to remove shortcuts. Evaluation uses step accuracy per multiple-choice question and chain accuracy requiring all questions in a preference-evolution chain to be correct.

**EvoMem Design.**

EvoMem augments a base memory system rather than replacing it. Given base memory updater Mt = U(Mt-1, xt), EvoMem computes ∆t = Diff(Mt-1, Mt) and records a patch pt = (τt, Ct-, Ct+, rt, zt, et) only for non-additive updates (revisions, overwrites, reinterpretations), where τt is temporal metadata, Ct-/Ct+ are pre/post memory content, rt is rationale, zt is semantic summary, and et is supporting evidence. At inference, the agent retrieves from the latest memory cmem = Rmem(q, MT) and from patch history Pq = Rpatch(q, P1:T), concatenating both into the final context c(q). The authors instantiate EvoMem across four agents: Terminus2 (terminal), OpenHands (software engineering), A-Mem (conversational memory), and Memento-Skill (reusable skill memory).

## Results & Highlights

- **Baseline struggle on EvoArena.** Average step accuracies are 43.6% (Terminal-Bench-Evo), 29.2% (SWE-Chain-Evo), and 46.5% (PersonaMem-Evo); chain-level accuracies drop to 21.5%, 10.6%, and 39.1% respectively, showing that isolated step success does not imply reliability across persistent evolution.

- **EvoMem improves EvoArena.** Average step gains are +2.4% Terminal, +0.5% SWE, +1.8% Persona; chain gains are larger: +6.1% Terminal, +2.9% SWE, +3.0% Persona. Overall across all settings, EvoMem yields +2.6% step-level and +3.7% chain-level improvement.

- **Notable per-model gains on Terminal-Bench-Evo step accuracy:** GPT-5.5 +2.3% (65.1%), Gemini-3.1-Pro +2.7% (56.5%), DeepSeek-V4-Pro +3.1% (40.4%), GLM-5.1 +3.5% (55.3%), Qwen3.6-27B +3.3% (40.9%). Chain gains are especially strong for GPT-5.5 (+13.7%), Kimi-K2.6 (+7.8%), and DeepSeek-V4-Pro (+8.9%).

- **Standard benchmark generalization.** On GAIA, EvoMem improves GPT-5.5 by 0.0%, Gemini-3.1-Pro by +8.0%, DeepSeek-V4-Pro by +10.0%, GLM-5.1 by +7.0%, Qwen3.6-27B by +5.0%, and Gemma4-31B by +9.0% (average +6.1%). On LoCoMo, average improvement is +4.8%, with Gemini-3.1-Pro gaining +7.5%.

- **Mechanistic findings.** On Terminal-Bench-Evo, when retrieved patch terms are actually reused in the agent's reasoning or commands, gains rise to +8.3% versus only +2.6% when no patch uptake occurs. On SWE-Chain-Evo, PASS_TO_PASS regression rate drops from 9.09% to 6.32% on average (Kimi-K2.6: 7.14% → 3.33%). On PersonaMem-Evo, temporal trajectory and multi-pattern synthesis questions each improve by +5.2%, and row-level evidence capture improves from 72.5% to 74.9% (with temporal trajectory row capture up +4.4%).

- **Efficiency–accuracy trade-off.** Higher token usage does not guarantee higher accuracy. In Terminal-Bench-Evo, GPT-5.5 reaches 62.8% accuracy but consumes 505.0M tokens, while Gemini-3.1-Pro and GLM-5.1 achieve strong accuracy with ~80M tokens.

## Authors & Credibility

The 14-author team spans National University of Singapore, Singapore Management University, University of Washington, University College London, University of Pennsylvania, Nanyang Technological University, Recursive, and MIT, with senior contributors including Caiming Xiong (formerly Salesforce AI Research), Hae Won Park, and Bryan Hooi. The work releases code, data, and a project page, which supports reproducibility. The empirical design combines benchmark construction, multi-agent instantiation, and cross-benchmark validation, making the claims well-supported. However, as with any newly introduced benchmark, long-term adoption and independent replication will further establish its reliability.

## Critical Assessment

**Novelty & Significance:** The paper addresses a genuinely underexplored problem—persistent environment evolution—by contributing both a benchmark suite and a conceptual memory mechanism. The patch-based framing of memory as an evolution trace is natural and well-motivated, and the cross-domain coverage strengthens the claim that the insight generalizes.

**Rigor:** The construction of EvoArena is described in substantial detail (with extensive appendices), and the evaluation includes multiple backbone models, step/chain metrics, and standard-benchmark transfer. The reported gains are generally modest at the step level (often 1–3%), but they are consistent and become more meaningful at the chain level, which is arguably the more important metric for deployment reliability. Some numbers in the abstract (e.g., 39.6% average accuracy) are rounded aggregate summaries of the per-domain values reported in the main paper. Statistical significance testing is not reported, so it is difficult to judge whether small gains (e.g., +0.5% on SWE-Chain-Evo) are robust.

**Reproducibility:** The authors provide project page, code, and dataset links, which is commendable. The detailed appendices describe construction and instantiation procedures. Independent replication will still require substantial compute and careful reproduction of the agent-specific integrations.

**Limitations & Open Questions:**
- The gains on SWE-Chain-Evo step accuracy are small (+0.5% average) and some models show slight degradation (Gemini-3.1-Pro -2.4%, Kimi-K2.6 -2.6%), suggesting the software-evolution setting remains very hard or that patch retrieval does not always help.
- The social-preference evaluation relies on synthetically generated personas and conversations; while dual-blind filtering reduces shortcuts, the ecological validity of generated preferences versus real user data is unclear.
- The paper does not deeply analyze the cost of storing and retrieving patch histories at scale, nor does it study very long chains (e.g., hundreds of versions) where patch retrieval noise could compound.
- The mechanism analysis is observational rather than causal; stronger ablations that control for retrieval strategy or patch granularity would strengthen claims.

## Strengths

- Tackles a realistic and important deployment gap: static benchmarks do not capture real-world environment evolution.
- Provides both evaluation infrastructure (EvoArena) and an algorithmic/memory abstraction (EvoMem) that is agent-agnostic.
- Demonstrates cross-domain and cross-benchmark improvements, including standard long-horizon and tool-use benchmarks.
- Includes detailed construction methodology and releases artifacts (code, data, project page).
- Goes beyond aggregate accuracy with chain-level metrics, regression analysis, evidence-capture metrics, and efficiency analysis.

## Limitations

- Step-level absolute gains on EvoArena are modest, especially on SWE-Chain-Evo, and not uniformly positive across all models.
- The benchmark instances are constructed semi-automatically; quality control and oracle validation are described but the process depends on LLM-based generation and human review, which may introduce subtle biases.
- Real-world deployment involves even richer forms of evolution (adversarial changes, partial observability, multi-user settings) not fully captured here.
- The cost and scalability of maintaining and retrieving large patch histories are discussed briefly but not thoroughly evaluated.
- Some reported abstract numbers differ slightly from the main table values (e.g., 39.6% aggregate step accuracy versus per-domain 43.6/29.2/46.5); readers should consult the tables for precise figures.

## Relevance

对于从事大语言模型智能体长期部署、动态环境适应、记忆系统设计的研究者而言，EvoArena 提供了一个及时且多维度的评估平台，而 EvoMem 将“记忆更新历史”显式化的思路对构建更可靠的智能体具有直接启发意义。其跨基准泛化结果也表明，记录记忆演化不仅是动态场景的需求，也可能提升常规任务表现。

## Questions for Further Exploration

1. How does EvoMem scale to much longer evolution chains (hundreds of versions) and how does retrieval noise affect performance?
2. Can the patch-generation and retrieval components be learned or optimized end-to-end rather than relying on heuristic diff and retrieval?
3. How robust is EvoMem to adversarial or contradictory environment updates, and can it detect when a patch should be reverted?
4. Does EvoMem help in multi-agent or multi-user settings where different agents observe different subsets of environmental changes?
5. What is the computational and storage overhead of maintaining full patch histories in production deployments?
6. Can the insights from EvoMem be integrated with continual-learning or self-evolving agent systems that update their own skills over time?
