ICML 2026 2606.13097 Embodied AI Code-as-Policies KV-Cache

Functional Cache Grafting:
Robust & Rapid Code-Policy Synthesis for Embodied Agents

Saehun Chun · Wonje Choi · Sera Choi · Sanghyun Ahn · Honguk Woo | Sungkyunkwan University | 2026-06-11
FCGraft 通过维护函数级已验证代码骨架的双层 KV 缓存，并用"拼接（stitching）+ 局部修补（patching）"合成新策略，在开放域具身任务上同时实现了比 RAGCache 高 18.31% 的成功率与 2.3 倍的延迟降低。
Key Finding
Function-level KV-cache reuse via stitch + patch
Moves beyond prompt-level caching. Composes validated code skeletons, then locally repairs errors — eliminating structural mistakes and redundant prefill.
Key Metric
+18.31% SR · 2.3× faster
vs RAGCache on simulation benchmarks. Real robot: +37.04% SR, 2.88× lower latency.
Why This Matters
Unblocks real-robot deployment of code-as-policies
Solves the two blocking barriers — latency and generative brittleness — simultaneously, with a unified cache-grafting approach.
Critical Edge
No public code · 9 real-robot trials
Relies on curated oracle function library. Reproducibility limited. Real-world generalization uncertain at this sample size.

Method Architecture

/* Two-tier code cache H = (I, C) */ Interface tier I = { (f_k, signature_text, KV_state) } ← lightweight, reusable prefixes Code tier C = { (f_k, implementation_text, KV_state) } ← full validated bodies /* Stage 1: Cache-stitching */ π_code ~ LLM( observation, instruction, I^KV ) ← calls cached interfaces π_exec = π_code ∥ { c_k^text | f_k ∈ Call(π_code) } ← link implementations from C /* Stage 2: Cache-patching (on exception E) */ split π_exec → prefix | err_span | suffix x_mid ~ LLM( prefix^KV, CoT(obs, instr, c_k^KV, E) ) ← regenerate only error span π_exec' = prefix ∥ x_mid ∥ suffix ← reassemble & retry /* Locality score for cache eviction */ ℓ(f_k) = (1−ℓ_curr)·(α·ℓ_freq + β·Σℓ_asso + γ·ℓ_sema) + ℓ_curr

Assessment

Strengths ✓

  • Unified approach to latency and brittleness — the two practical blockers for code-as-policies
  • Consistent gains across 3 simulators + real Franka robot
  • Clean stitching/patching separation enables continual improvement without catastrophic forgetting
  • Thoughtful cache management: recency × frequency × association × semantic diversity
  • Detailed appendix with baseline recipes and system configuration

Limitations ✗

  • No public code or artifact release — blocks independent verification
  • Real-world evaluation: only 9 trials per scenario
  • Relies on curated oracle initial function library
  • Isolated per-agent cache — no cross-agent sharing (acknowledged)
  • Single model size + greedy decoding only; scale/sampling robustness unknown
  • Benefits largest for compositional tasks; novel control structures may still need full generation

Open Questions

How sensitive is FCGraft to initial function library quality? Can it bootstrap from weak or human-annotated examples?
What is the transfer curve as the cache grows — at what size do hit rates and success rates saturate?
Can the two-tier cache be compressed or quantized without losing stitching/patching fidelity?
How would cross-agent cache sharing affect performance in multi-robot or collaborative settings?
Can the locality score be learned online rather than hand-weighted?
