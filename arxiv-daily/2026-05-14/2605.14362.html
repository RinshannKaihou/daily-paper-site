Correctness-Aware Repository Filtering Under Maximum Effective Context Window Constraints

arXiv: 2605.14362

Date: 2026-05-14

https://arxiv.org/abs/2605.14362

一句话总结： 本文提出了一种基于文件大小的预执行启发式过滤框架（SizeFilter/HybridFilter），利用OS级stat()元数据在token化之前移除大型非代码产物，在10个开源仓库上实现79.6%–89.3%的token削减，且任务准确率从25%提升至72%、幻觉率从61%降至17%。
TL;DR: A pre-execution, size-based heuristic filtering framework that removes large non-code artifacts from software repositories before tokenization using only OS-level stat() metadata, achieving 79.6–89.3% token reduction at sub-millisecond overhead while improving downstream task accuracy from 25% to 72%.

Key Contributions

  • Pre-execution size-based heuristic filtering framework: stack-agnostic, non-blocking, zero-index, deployable without manual configuration, with empirical evidence of correctness preservation
  • Formal cost model (Eqs. 1–4) linking file size to token consumption under the MECW constraint, validated with Pearson r=0.997 across 2,688 files
  • Taxonomy and evaluation of eight composable filters across 10 real open-source repositories (22,046 files, five languages: JS, Python, Go, Ruby, TS/C++)
  • Threshold sensitivity analysis across θ ∈ {50 KB, 100 KB, 500 KB, 1 MB, 5 MB} with statistical grounding; θ=1 MB identified as optimal tradeoff
  • Tail-at-scale distribution analysis explaining why size-based filtering is structurally effective: 0.5–2% of files account for 80–94% of bytes in data-heavy repositories
  • Zero Disk I/O testing methodology: 45-test, sub-50 ms deterministic CI validation via in-memory virtual filesystem
  • Limited-scope empirical task evaluation (18 tasks, 2 repos, CodeLlama-7B-Instruct) showing accuracy improves from 25% to 72% and hallucination rate drops from 61% to 17%

Methodology

The framework intercepts repository scans at a pre-tokenization stage (Stage S2 in a six-stage pipeline). A single recursive directory traversal (depth-limited at 20) prunes system directories (node_modules, .git, __pycache__) and applies filters in one pass. The core decision uses only OS-level stat() metadata—no file content is read.

SizeFilter: A single comparison f.size > θ flags files exceeding the threshold (recommended θ=1 MB). Token count is approximated as tokens(f) ≈ k · s where k=0.250 tokens/byte (validated empirically, r=0.997).

HybridFilter: Chains four gates in ascending computational cost: (1) BinaryFilter (magic-byte, 8 bytes read, <0.01 ms), (2) ExtensionFilter (blocklist, no read), (3) SizeFilter (stat(), no read), (4) SemanticFilter (keyword density, ~4 KB read). Files exit on first trigger.

Validation approach: Token-density study across 2,688 files, statistical tests (Wilcoxon signed-rank, Wilson CIs), FPR analysis via manual inspection, and a limited-scope task evaluation with CodeLlama-7B-Instruct (4-bit GGUF via Ollama) using fixed 4,096-token truncation to isolate filtering effects.

Results & Highlights

  • SizeFilter (θ=1 MB): 79.6% (±13.2%) mean token reduction at 0.30 ms overhead; 55% lower variance than ExtensionFilter
  • HybridFilter: 89.3% (±9.0%) reduction—the lowest variance of any filter evaluated; overall corpus reduction from 154.0M to 4.6M tokens (94.1%)
  • Heuristic validation: Pearson r=0.997, R²=0.995, MAE <0.1%, k=0.2500 tokens/byte across 2,688 files
  • Statistical significance: Wilcoxon signed-rank W=68, p=0.047 for SizeFilter vs. ExtensionFilter; Wilson 95% CI [68.4%, 87.6%]
  • Task evaluation: File accuracy Top-1: 25%→72.2% (+47.2 pp); Function accuracy: 12.5%→56.3% (+43.8 pp); Hallucination rate: 61.1%→16.7% (−44.4 pp)
  • FPR ≈ 0%: Manual inspection across fastapi, django, tensorflow revealed no incorrectly excluded source files at θ=1 MB
  • Comparison with semantic methods: Operates at <0.01 ms per file vs. 50–300 ms for RepoCoder/GraphRAG/dense embeddings, requires no indexing, and handles binary files

Critical Assessment

  • Novelty: Moderate. The idea that large files should be filtered before tokenization is intuitive, but the paper provides rigorous formalization and extensive empirical validation. The hybrid gate architecture and zero-disk-I/O testing are nice engineering contributions. The MECW framing elevates this from cost optimization to correctness preservation, which is a meaningful reframing.
  • Rigor: Strong in the token-reduction evaluation (10 repos, 22K files, statistical tests, sensitivity analysis). However, the task evaluation is explicitly labeled "limited-scope" — only 18 tasks across 2 repositories with a single 7B quantized model. The authors acknowledge this openly and reserve SWE-bench evaluation for future work.
  • Reproducibility: Excellent. All code and data are released. The Zero Disk I/O testing methodology (in-memory virtual filesystem, 45 deterministic tests) is a strong reproducibility contribution. Token counting uses standard tiktoken cl100k_base encoding.
  • Significance: Practically significant for LLM-based developer tools. The sub-millisecond overhead and zero-index requirements make deployment trivial. The finding that 0.5–2% of files cause 80–94% of token bloat is actionable.
  • Red flags: (1) Single-author independent researcher — unusual for this scope but the work is thorough. (2) The task evaluation (18 tasks, CodeLlama-7B) is too small to be conclusive; the accuracy gains could be partly artifact of the specific repos chosen. (3) The BinaryFilter only covers 11 magic-byte signatures, limiting coverage for ML-heavy repos with TFRecord/Parquet/Arrow files. (4) No comparison against a random baseline or a naive "filter by extension" oracle that includes data file extensions.

Strengths

  • Extremely low overhead (0.30 ms for SizeFilter) with no indexing requirement — trivially deployable
  • Comprehensive evaluation across 10 diverse repositories (22,046 files, 5 languages) with statistical rigor
  • Strong heuristic validation: near-perfect linear relationship (r=0.997) between file size and token count
  • Honest about limitations — the "limited-scope" task evaluation is clearly labeled and caveated
  • Complementary to semantic retrieval systems rather than competing with them
  • Zero Disk I/O testing methodology is a practical contribution for CI pipelines

Limitations

  • Task evaluation scale: Only 18 tasks, 2 repositories, 1 model (CodeLlama-7B-Instruct, 4-bit quantized) — far from benchmark-quality. SWE-bench evaluation is deferred to future work.
  • Large legitimate files: Auto-generated protocol buffer bindings or machine-generated parser tables exceeding 1 MB would be incorrectly flagged. The override mechanism mitigates but does not eliminate this.
  • Binary detection coverage: Only 11 magic-byte signatures; ML-heavy repos with TFRecord, Parquet, or Arrow files are poorly covered.
  • Semantic filter is keyword-based: Relies on fixed English keyword list, limiting language-agnostic applicability. Integration with lightweight embeddings is suggested but not implemented.
  • No evaluation on enterprise monorepos: The 10 open-source repos may not represent the complexity and scale of enterprise codebases.
  • Static threshold: θ is fixed per configuration; adaptive thresholding based on residual MECW capacity is described as future work (S5→S2 feedback arc in Fig. 1).

Relevance

该工作直接解决了LLM在软件工程场景下的上下文窗口效率问题，对构建更可靠的代码生成和仓库理解工具具有实际价值。其形式化的MECW约束模型和低成本过滤方法论对研究LLM上下文管理和长上下文退化具有参考意义。

Questions for Further Exploration

  1. How does SizeFilter/HybridFilter perform on frontier models (GPT-4, Claude, Gemini) with much larger context windows — does the MECW constraint still apply at 128K+ tokens?
  2. Can adaptive thresholding based on per-repository file-size distributions (e.g., P95) reduce FPR without sacrificing token reduction?
  3. What is the interaction effect when SizeFilter is used as a pre-pass gate before semantic retrieval systems like RepoCoder or GraphRAG?
  4. How would the framework perform on enterprise monorepos with millions of files and cross-language dependencies?
  5. Does the k=0.250 tokens/byte constant generalize to non-ASCII-heavy codebases (e.g., CJK comments, Unicode identifiers)?
