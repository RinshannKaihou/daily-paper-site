# Dual-Pool Token-Budget Routing for Cost-Efficient and Reliable LLM Serving

> **一句话总结：** 提出双池 token-budget 路由方法，将 vLLM 同质舰队拆为短/长上下文双池并按在线校准的 token 预算分发请求，在 Azure 与 LMSYS 真实轨迹上把 GPU-hours 降低 31–42%（年省 $2.86M）、抢占率下降 5.4×，且 P99 TTFT 改善 6%。

> **TL;DR:** On two real-world traces (Azure LLM Inference Dataset, LMSYS-Chat-1M, 100K requests each) serving Llama-3-70B on A100-80GB GPUs, dual-pool token-budget routing versus a homogeneous single-pool baseline reduces GPU instances by 41.9% (Azure) and 31.3% (LMSYS) at 1,000 req/s, lowers preemption 5.4× (47.3‰ → 8.7‰), and improves P99 TTFT by 6% (1.82s → 1.71s); a Qwen3-235B-A22B on AMD MI300X case study projects $15.4M/yr savings at 10,000 req/s.

## 核心贡献

- 提出 token-budget 池路由（fleet 级别分发算法）：将同质 vLLM 舰队拆为短上下文高并发池（Cmax=8K，128 slots）与长上下文全覆盖池（Cmax=65K，16 slots），按估计的总 token 预算 O(1) 分发，与 PagedAttention、continuous batching、prefill-decode disaggregation 正交可叠加。
- 自校准 token 估计：用每类别 bytes-per-token 的指数移动平均（EMA，β=0.95）从 `usage.prompt_tokens` 反馈在线学习，配合保守偏置（γ=1.0 减去标准差项）与总预算路由（Lin + Lout），消除对 tokenizer 的依赖，跨 CJK/code/prose 类别收敛于约 50 个样本。
- 给出闭式成本模型 `savings = α·(1−1/ρ)`，仅用流量 CDF 与实测吞吐比即可在部署前估算上界 GPU 节省，便于团队审计机会。
- 在两个真实轨迹（Azure、LMSYS）与前沿硬件（Qwen3-235B-A22B on MI300X）上系统评估，展示 31–42% GPU 削减、5.4× 更少抢占、$15.4M/yr 节省。

## Key Contributions

- Token-budget pool routing: a fleet-level O(1) dispatch algorithm splitting a homogeneous vLLM fleet into a high-concurrency short-context pool (Cmax=8K, 128 seq slots) and a high-capacity long-context pool (Cmax=65K, 16 seq slots), composable with PagedAttention, chunked prefill, and prefill-decode disaggregation.
- Self-calibrating token estimation: a per-category EMA (β=0.95) that learns bytes-per-token from `usage.prompt_tokens` feedback with an asymmetric-error-aware conservative bias (γ=1.0 subtracting the EMA std), routing on total budget Lin + Lout, and eliminating the need for a model-specific tokenizer at the router.
- A closed-form cost model `savings = α·(1−1/ρ)` (α = short-traffic fraction, ρ = short/long throughput ratio) that predicts fleet-level GPU savings from two pre-deployment observable quantities (traffic CDF and profiled throughput), acting as a cheap upper bound.
- Comprehensive evaluation on two real-world traces (Azure, LMSYS) plus a frontier-hardware case study (Qwen3-235B-A22B on AMD MI300X), reporting 31–42% GPU reduction, 5.4× fewer preemptions, and $15.4M/yr projected savings at 10,000 req/s.

## Methodology

The method splits a homogeneous vLLM fleet into two specialized pools: a short pool Ps (Cmax=8K, 128 concurrent sequences, µ=11.2 req/s/inst) and a long pool Pl (Cmax=65K, 16 sequences, µ=2.8 req/s/inst). For each request r of traffic category k (code, prose, CJK, mixed), the router estimates total token budget `Ltotal = ⌈|r|/ĉk⌉ + r.max_output_tokens` where ĉk is the calibrated bytes-per-token ratio; the estimate is O(1) (one division, no tokenizer). Cold start defaults c0=4.0 (English-prose average). Closed-loop calibration updates ĉk via EMA `ĉk ← β·ĉk + (1−β)·cobs` with β=0.95, where `cobs = |r| / usage.prompt_tokens`; because routing errors are asymmetric (long→short causes preemption, short→long only wastes throughput), the router uses a conservative estimate `ĉ_route = ĉk − γ·σ̂k` with γ=1.0. Dispatch (Algorithm 1) enforces a hard feasibility check (`Ltotal > Cmax(Ps)` → Pl), then budget-based routing against threshold Bshort (default 8192), then load-aware spillover when the preferred pool is saturated, and a final safety check. Routing on Ltotal=Lin+Lout (not Lin alone) is critical: early prototypes on Lin alone caused preemption storms from short-prompt/long-generation requests (e.g., Lin=200, Lout=8192).

The closed-form cost model gives fractional GPU savings `ΔG/Ghomo = α·(1 − 1/ρ)` where α=F(Bshort) is the short-traffic fraction and ρ=µ(CS)/µ(CH) is the throughput gain ratio; production traces give α∈[0.80, 0.95] and vLLM profiling ρ∈[4, 8] for 8K vs 64K. The evaluation simulates Llama-3-70B-Instruct (BF16, 80 layers, 8 KV heads, dh=128) on NVIDIA A100-80GB (TP=2) using a discrete-event simulator calibrated against Vidur; two 100K-request Poisson-arrival traces are used — Azure-Derived (80% below 2K tokens, tail to 64K) and LMSYS-Derived (mean Lin=69.5, Lout=214.5). SLO targets: P99 TTFT ≤ 2s, P99 TPOT ≤ 80ms. A frontier case study projects Qwen3-235B-A22B (235B params, 22B active, 94 layers, 4 KV heads GQA 16:1, dh=128, FP8) on AMD MI300X (192GB HBM3, TP=8, 23.5 KB/token/GPU, 133.4 GB for KV after weights/activations/safety margin).

## Results & Highlights

- GPU cost: on the Azure trace at 1,000 req/s with Bshort=8192, token-budget routing needs 208 GPUs vs 358 homogeneous = 41.9% savings (Table 2); on LMSYS, 246 vs 358 = 31.3% savings. At $2.21/GPU-hr (AWS p4d.24xlarge) the Azure saving is $238K/month = $2.86M/year.
- Reliability at 1,000 req/s, 90% utilization (Azure, Table 3): preemption drops 5.4× from 47.3‰ (homogeneous) to 8.7‰ (overall dual-pool); OOM events/hr drop 5.3× from 2.1 to 0.4; rejection rate from 0.31% to 0.05%; success rate rises 99.69% → 99.95%. The short pool is effectively failure-free (0.0 OOM/hr, 0.00% rejection) because every routed request fits by construction.
- Latency (Azure, Table 4): P50 TTFT improves 33% (0.42s → 0.28s) and P99 TTFT 6% (1.82s → 1.71s); P50 TPOT 11% (28ms → 25ms) and P99 TPOT 7% (67ms → 62ms). Cost and latency improve as co-benefits, not a trade-off.
- Calibration convergence (Table 5): per-category EMA converges within 50 observations to ≤3.5% relative error (English prose 1.6%, code 1.4%, CJK 3.5%, mixed 1.8%); mis-route rate falls from 4.1% (global static c=4) to under 1% for all categories — CJK benefits most because static c=4 overestimates its bytes-per-token by 2×.
- Threshold robustness (Figure 6): any Bshort in 4K–16K delivers >80% of peak savings; the product α·(1−1/ρ) is flat near the optimum (α≈0.80, ρ≈4.0 at 8K).
- Frontier projection (Table 6): Qwen3-235B-A22B on MI300X at 10,000 req/s needs 137 nodes (1,096 GPUs) vs 197 nodes (1,576 GPUs) homogeneous = 30.5% reduction = $15.4M/yr at $3.67/GPU-hr cloud ($6.3M/yr on-premise); the 8K pool supports 4× more concurrent sequences than a 32K pool (676 vs 169).

## Critical Assessment

The headline savings rest on a discrete-event simulation calibrated against Vidur rather than on a physical fleet deployment — the Azure/LMSYS 31–42% GPU reductions and the reliability/latency tables are simulator outputs, and the closed-form model overpredicts by 18.1pp on Azure (predicted 60.0% vs simulated 41.9%) and 19.7pp on LMSYS (51.0% vs 31.3%), because the formula assumes perfect packing while the simulator includes queuing delays, load imbalance, and integer-GPU ceiling effects; the authors honestly position the formula as a cheap upper bound, but readers must not conflate it with realized savings. The Qwen3-235B-A22B/MI300X numbers are an analytical projection (applying Equation 1 to vendor data sheets and a 10% safety margin), not a measured deployment. The two evaluation traces, although real, are each only 100K Poisson-arrivals and are re-derived/simulated rather than replayed against live serving software, so end-to-end correctness of the dispatch path (Algorithm 1, including the spillover transition that produces the residual 1.2‰ short-pool preemption) is validated only through the simulator's queueing model, not against actual vLLM preemption behavior. Multi-model correctness of the tokenizer-free estimator is argued via per-category EMA on four coarse categories, but the paper does not report performance under adversarial or distribution-shift traffic where the cold-start default c0=4.0 could misroute a large fraction before convergence.

## Strengths

- Directly attacks a real, well-characterized production failure mode (configuration-traffic mismatch) and ties cost waste (4–8× throughput) to specific reliability failures (OOM, preemption storms, rejections, head-of-line blocking) with a clear root-cause analysis.
- The tokenizer-free, O(1), self-calibrating estimator is genuinely practical for multi-model fleets where the router sits upstream of heterogeneous backends, and the asymmetric-error-aware conservative bias is well-motivated by the cost asymmetry of routing errors.
- The closed-form cost model `α·(1−1/ρ)` gives practitioners a no-infrastructure-change audit tool, and the threshold-robustness result (4K–16K plateau) lowers deployment risk.
- Composability is argued carefully: each pool can internally use chunked prefill/PagedAttention/disaggregation, and the paper explains why chunked prefill is necessary-but-insufficient (it solves compute scheduling but not the KV-cache provisioning problem).

## Limitations

- All headline results are simulator-derived (Vidur-calibrated), not from a physical vLLM deployment; the 5.4× preemption reduction and 6% P99 TTFT gain are simulated metrics.
- The cost model systematically overpredicts savings (by 18–20pp on the two traces) and is positioned as an upper bound, so realized production gains may be materially smaller.
- The Qwen3-235B-A22B/MI300X $15.4M/yr figure is a projection from data-sheet arithmetic, not a measured result.
- Evaluation is limited to two 100K-request traces and four coarse content categories; behavior under distribution shift, adversarial traffic, or pre-convergence cold-start is not reported.
- No code release URL is given in the PDF despite a referenced reproducibility script, limiting independent verification.

## Questions for Further Exploration

- If the threshold Bshort were made adaptive using runtime preemption/OOM/rejection signals (as the paper's future-work section suggests), how would the savings curve in Figure 6 change under non-stationary traffic, and would the 4K–16K plateau still hold?
- How does token-budget routing interact with speculative decoding and KV-cache compression (e.g., LaCache), which also change the effective µ(Cmax) relationship the cost model depends on?
- Can the per-category EMA estimator be validated against a real multi-model vLLM fleet to confirm the ≤1% mis-route rate and the cold-start convergence within 50 requests, rather than only in simulation?
- Does the 5.4× preemption reduction survive replay against the actual vLLM scheduler (not Vidur), where preemption heuristics and PagedAttention page allocation may differ from the simulator's queueing model?
- For workloads where α < 0.70 (e.g., long-context-heavy coding agents), at what α does dual-pool routing stop being worth the operational complexity of two pools?
