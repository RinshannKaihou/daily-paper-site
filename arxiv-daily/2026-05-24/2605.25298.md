> **一句话总结：** 本文提出Prism，一种基于eBPF的应用无关性能退化诊断方法，通过在六个内核子系统中收集16个细粒度线程-资源交互指标来扩展线程状态分析（TSA），并利用选择性线程追踪算法从入口点线程追踪性能问题传播路径，在MySQL、Kafka、Redis等六种应用上成功诊断了CPU、磁盘、锁和外部服务竞争，开销仅0.06–0.37ms。

> **TL;DR:** This paper presents Prism, an application-agnostic performance degradation diagnosis method that extends Thread State Analysis (TSA) by implementing 16 eBPF-based metrics across six kernel subsystems (scheduling, VFS, networking, futex, multiplexing IO, block IO) at thread-to-resource granularity. A selective thread tracking algorithm traces degradation propagation from entry-point threads through inter-thread dependency chains. Validated on six OLDI applications (MySQL, Kafka, Cassandra, Redis, TeaStore, ML-inference), it successfully diagnoses CPU, disk, lock, and external service contention with 0.06–0.37ms overhead.

## 核心贡献

- 提出一种应用无关的性能退化诊断方法，通过在六个内核子系统（调度、VFS、网络、futex、多路复用IO、块IO）中收集线程到资源粒度的16个eBPF指标，扩展了传统的线程状态分析（TSA）
- 设计了选择性线程追踪算法（Algorithm 1），从入口点线程出发，沿着线程间依赖链追踪性能问题的传播路径，时间复杂度为O(k·n)
- 实现了名为Prism的工具（Rust + libbpf），提供OLAP数据库存储、UI分析界面和模板化查询，并已开源
- 在MySQL、Kafka、Cassandra、Redis、TeaStore和ML推理服务六种OLDI应用上验证了方法的有效性
- 通过分布对比（baseline vs. comparison）结合统计检验（Mann-Whitney U、KS检验）来减轻指标噪声，提高诊断准确性

## Key Contributions

- An application-agnostic method extending TSA by collecting 16 eBPF-based metrics at thread→resource granularity across six kernel subsystems, capturing not just where time is spent but how threads interact through shared resources
- A selective thread tracking algorithm (O(k·n)) that starts from entry-point threads and iteratively follows inter-thread dependency chains (via futex, pipe, socket BRIs) to trace degradation propagation paths
- Prism: a complete toolchain (Rust + libbpf) with OLAP storage (DuckDB), a UI with templated queries, and automated process dependency discovery via transitive IPC tracing
- Distribution-based comparison (baseline vs. degraded) with statistical tests (Mann-Whitney U, Kolmogorov-Smirnov, Wasserstein distance) to mitigate metric noise
- Comprehensive evaluation on six diverse OLDI applications covering thread pools, async multiplexing IO, JVM, Python VM, and process pools — successfully diagnosing CPU, disk, lock, and external service contention
- Open-source release with full reproducibility artifacts (GitHub: EC-labs/prism)

## Methodology

The method operates in three stages:

**1. Monitoring (automated):** A user specifies a bootstrapping process. Prism deploys eBPF probes and automatically discovers all directly and indirectly communicating processes via IPC (sockets, pipes), constructing a complete process dependency graph. The collector reads eBPF statistical summaries at 1-second intervals, storing metrics in an OLAP database (DuckDB).

**2. Metric Collection (eBPF instrumentation):** 16 metrics are collected across six kernel subsystems:
- *Scheduling:* runtime (on-CPU time), rq time (runqueue wait), block time, iowait time, sleep time — at thread granularity
- *Pipes/Sockets:* pipe wait time/count, socket wait time/count — at thread→pipe/socket granularity via Backing Resource Identifiers (BRI = inode i_ino + superblock s_dev)
- *Futex:* futex wait time/count, futex wake count — at thread→futex granularity, tracking both lock contention and work scheduling patterns
- *Multiplexing IO:* epoll wait time/count, epoll file wait — supporting select, poll, and epoll APIs
- *Block IO:* sector count — at thread→device granularity

**3. Analysis (Selective Thread Tracking):** Given a target application and KPI time series:
- Identify entry-point threads via socket wait activity
- For each tracked thread, check all TSA metrics for distribution shifts between baseline and degraded periods
- For flagged IPC metrics (futex, pipe, socket), resolve the BRI to find all counterpart threads interacting with the same resource
- Iteratively expand the tracking set until no new threads are added
- Returns the set of inter-dependent threads and their flagged metrics, revealing the degradation propagation path

The tool provides a web UI with KPI visualization, process dependency graphs (Ripple page), and a Debug page for custom/templated SQL queries against the metrics database.

## Results & Highlights

**Disk Constrained:**
- *MySQL:* Identified two degradation paths — disk contention (TPCC throughput drop linked to worker thread iowait increase) and lock contention (YCSB read/write workload mutual futex wake activity between threads t3/t4)
- *Kafka:* Traced throughput degradation (55–77% below target at 450K events/s) to worker thread block time saturation stalling internal pipe writes, which cascaded to the network handler thread via epoll

**CPU Constrained:**
- *Cassandra:* Discovered that expected lock contention was actually CPU contention (increased rq time); and disk contention manifested as memory contention (increased runtime) due to Cassandra's asynchronous fsync (commitlog_sync=periodic, 10s interval)
- *ML-inference:* 100x response time increase (75ms→7500ms) traced to CPU saturation in futex-synchronized handler threads within gunicorn workers
- *Redis:* Single-threaded t1 runtime spike + background thread t2 competing for same CPU core, raising rq time for both threads

**Externally Constrained:**
- *TeaStore:* webui component's degradation traced to external persistence service via socket wait time histogram showing shifts from 0–0.1s to 0.9–1.0s at peak load — correctly identifying the issue as external to the monitored service

**Overhead:** Latency overhead measured at 0.061ms (Kafka) to 0.372ms (Redis), with 10% of a single CPU core and 115KB/s disk writes under saturation.

## Critical Assessment

- **Novelty:** The key novelty is the combination of TSA with thread→resource granularity metrics and the selective thread tracking algorithm. While individual eBPF observability tools exist, the systematic integration capturing inter-thread dependency propagation at this granularity is original. The tool fills a gap between coarse-grained service-level tools (Coroot, Pixie) and fine-grained but context-limited thread-state tools (xCapture).

- **Rigor:** Evaluation covers six diverse applications with different threading models (thread pools, async IO, process pools), runtimes (JVM, Python VM, native), and IPC mechanisms. The induced degradation scenarios (CPU, disk, lock, external) are realistic. However, all experiments run on a single machine (8-core i7-11850H), and scalability to multi-node or containerized environments is not evaluated.

- **Reproducibility:** Excellent. Open-source code, full reproducibility artifact with Nix-pinned dependencies, Docker-based setup, and an artifact description/evaluation appendix. Estimated reproduction time: ~30 minutes.

- **Significance:** The work addresses a practical gap in production systems observability. The ability to diagnose degradation propagation paths (not just identify the constrained subsystem) is genuinely useful for SREs and systems engineers. The low overhead and application-agnostic design make it deployable in production.

- **Weaknesses:** The selective tracking algorithm assumes degradation propagates toward entry-point threads (optimistic assumption acknowledged by authors). The current method is limited to single-host scenarios — no distributed tracing integration. System-level information integration is left for future work. The method also cannot diagnose application-level logic issues (e.g., inefficient algorithms) that don't manifest as kernel-visible resource contention.

## Strengths

- Application-agnostic: no recompilation, modification, or prior instrumentation required — works with any Linux application at runtime
- Low overhead (0.06–0.37ms) suitable for production deployment
- Comprehensive metric coverage across six kernel subsystems enables diagnosis of diverse contention types
- The selective thread tracking algorithm efficiently narrows the diagnostic search space with O(k·n) complexity
- Excellent reproducibility with open-source tool, Nix-managed dependencies, and detailed artifact appendix
- Distribution-based comparison with statistical tests provides robust noise mitigation
- Automated process dependency discovery via transitive IPC tracing reduces manual configuration
- Surprising diagnostic insights (e.g., Cassandra's expected lock contention was actually CPU contention) demonstrate the method's ability to challenge assumptions

## Limitations

- Single-host only; no support for distributed or multi-node diagnosis — the method cannot trace degradation across service boundaries beyond socket-level external dependency detection
- The selective tracking algorithm optimistically assumes degradation propagates toward entry-point threads; if the root cause doesn't propagate this way, a full (expensive) metric search is needed as fallback
- No integration with system-level metrics (e.g., /proc, hardware counters) — acknowledged as future work
- Cannot diagnose application-level performance issues (inefficient algorithms, memory leaks) that don't manifest as kernel-visible resource contention
- Evaluation limited to a single 8-core machine; scalability to high-core-count servers or containerized/Kubernetes environments is untested
- eBPF kernel version dependency (requires Linux 6.8+ with BPF/BTF enabled)
- The method requires manual selection of baseline and comparison time windows for degradation analysis

## Questions for Further Exploration

- How does Prism scale to machines with 64–128 cores and thousands of threads, where the process dependency graph and BRI tracking could become significantly larger?
- Can the selective thread tracking algorithm be automated end-to-end (automatic baseline/comparison selection, automatic root cause report) without human analyst intervention?
- How could Prism integrate with distributed tracing systems (e.g., OpenTelemetry, Jaeger) to extend diagnosis across service boundaries?
- What is the detection latency — how quickly after a degradation event begins can Prism's 1-second sampling interval identify the root cause?
- Could the metric set be extended to cover memory subsystem interactions (e.g., page faults, TLB misses, cache contention) to diagnose memory-related degradation patterns?
- How robust is the distribution-based comparison approach to gradual performance degradation (as opposed to sudden onset)?
