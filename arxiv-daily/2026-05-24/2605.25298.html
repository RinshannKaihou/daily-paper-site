Beyond Thread States: Diagnosing Performance Degradation with eBPF and Thread Dynamics

Diogo Landau · Jorge G. Barbosa · Nishant Saurabh — Utrecht University · Universidade do Porto

Prism extends Thread State Analysis with 16 eBPF metrics across 6 kernel subsystems and a selective thread-tracking algorithm that traces degradation propagation paths through inter-thread dependency chains — application-agnostic, zero-modification, production-ready.

Key Finding

Prism captures thread→resource interactions at kernel granularity (scheduling, VFS, networking, futex, epoll, block IO) and traces degradation propagation from entry-point threads through IPC dependency chains — revealing root causes that coarse service-level tools miss.

Key Metric
0.06–0.37
ms latency overhead
Why This Matters

Traces degradation propagation paths through dependency chains, not just identifies constrained subsystems — production-ready with near-zero overhead.

Critical Edge

Single-host only — no distributed tracing. Tracking algorithm assumes degradation propagates toward entry-point threads (optimistic). Requires Linux 6.8+ and cannot diagnose application-level logic issues that don't surface as kernel resource contention. Evaluated on a single 8-core machine only.

Methodology

Monitoring

User specifies a bootstrapping process. Prism deploys eBPF probes and discovers all communicating processes via IPC (sockets, pipes), building a complete process dependency graph. Metrics sampled at 1-second intervals into DuckDB.

Metric Collection

16 eBPF metrics across 6 kernel subsystems:

  • Scheduling — runtime, rq wait, block, iowait, sleep
  • Pipes/Sockets — wait time/count via BRI
  • Futex — wait/wake by futex address
  • Multiplexed IO — epoll wait/count
  • Block IO — sector count by device

Analysis

Selective Thread Tracking (O(k·n)): start from entry-point threads → check metrics for distribution shifts (Mann-Whitney U, KS test) → resolve BRI to find counterpart threads → iterate until no new threads added. Returns the degradation propagation path.

Results

Disk Constrained

MySQL: Two degradation paths — disk contention (TPCC throughput drop → worker iowait) and lock contention (YCSB futex wake between t3/t4)
Kafka: 55–77% throughput drop at 450K events/s traced to worker block-time saturation → pipe stall → epoll cascade

CPU Constrained

Cassandra: Expected lock contention was actually CPU contention (↑rq time); disk issue manifested as memory contention (↑runtime) due to async fsync
ML-inference: 100× latency spike (75ms→7500ms) from CPU saturation in futex-synchronized gunicorn handlers
Redis: Single-thread t1 runtime spike + background t2 competing for same core

Externally Constrained

TeaStore: webui degradation traced to external persistence service via socket wait shift from 0–0.1s → 0.9–1.0s — correctly identified as outside the monitored service

Assessment

Strengths

  • No recompilation or modification — works with any Linux application at runtime
  • 0.06–0.37ms overhead suitable for production
  • Covers 6 kernel subsystems for diverse contention diagnosis
  • O(k·n) tracking algorithm narrows search efficiently
  • Excellent reproducibility: open-source, Nix-pinned, Docker-based
  • Statistical tests (Mann-Whitney U, KS, Wasserstein) for noise mitigation
  • Automated process dependency discovery via transitive IPC tracing
  • Reveals surprising insights (Cassandra lock contention was actually CPU contention)

Limitations

  • Single-host only; cannot trace across service boundaries
  • Optimistic assumption: degradation must propagate toward entry-point threads
  • No system-level metrics integration (/proc, hardware counters)
  • Cannot diagnose application-level issues (bad algorithms, memory leaks)
  • Evaluated on single 8-core machine; multi-node untested
  • Requires Linux 6.8+ with BPF/BTF enabled
  • Manual selection of baseline vs. comparison time windows

Open Questions

  1. How does Prism scale to 64–128 cores with thousands of threads?
  2. Can baseline/comparison selection and root-cause reporting be fully automated?
  3. Could integration with OpenTelemetry or Jaeger extend diagnosis across services?
  4. What is the detection latency from degradation onset to root-cause identification?
  5. Could memory subsystem metrics (page faults, TLB misses, cache contention) be added?
  6. How robust is the distribution comparison to gradual (vs. sudden) degradation?
