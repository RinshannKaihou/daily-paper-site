> **一句话总结：** 本文揭示了LLM智能体中长期记忆存储的人格偏好（如追求速度、节俭、冒险倾向等）会通过隐式转向向量和注意力重分配机制，悄然改变不相关专业任务中的工具调用参数，提出MEMDRIFT基准（105场景）并在7个前沿模型上验证了该漏洞的普遍性，扫描了6,062个真实MCP工具确认608个存在受攻击风险。

> **TL;DR:** Biased personality memories in LLM agents silently alter tool-call parameters via implicit steering vectors and attention hijacking even when irrelevant; the authors formalize "memory-induced tool-drift," introduce MEMDRIFT (105 scenarios, 5 bias dimensions, 7 domains), demonstrate drift across 7 frontier models (+3.6 deflection on 1–5 scale), audit 6,062 real MCP tools flagging 608 vulnerable ones, and show standard defenses fail to fully mitigate the issue.

## 核心贡献

- **首次提出并形式化"记忆诱导的工具漂移"（Memory-Induced Tool-Drift）问题**：定义了LLM智能体中长期记忆中的人格偏差如何在不相关的专业场景中隐性影响工具调用参数选择，建立了严格的三条件实验框架（无记忆M∅、中性记忆Mn、偏差记忆Mb）。
- **构建MEMDRIFT基准**：包含105个场景，覆盖5个偏差维度（速度/急躁、资源节俭、极简主义、风险宽容、自主/自立）和7个专业领域（医疗、金融、法律、软件基础设施、教育、电商、营销），通过三阶段自动化对抗流水线生成（场景生成→工件扩展→自改进对抗优化）。
- **全模型实证验证**：在7个前沿模型（GPT-5.2、GPT-5.4、Claude-Sonnet-4.5、Gemini-2.5-Pro、Gemini-3.1-Pro-Preview、Kimi-K2.5、Qwen3.5-397B-A17B）上均观察到显著工具漂移，偏差偏转分数最高提升+3.6分。
- **生产级记忆架构无法消除漂移**：在Mem0、MemPalace、SimpleMem三种生产级记忆框架下漂移依然存在，因为记忆架构在存储时会剥离个人上下文，将偏好编码为全局指令。
- **真实世界威胁量化**：扫描Smithery上288个已验证MCP服务器的6,062个工具，标记608个（约10%）存在易受攻击的参数，并在验证子集上复现了漂移（如GitHub label_write工具将update→delete销毁issue关联；GitLab create_project将private→public暴露受HIPAA监管的代码库）。
- **机制层面解释**：在Llama-3.3-70B-Instruct上进行机制分析，证明偏差记忆作为隐式转向向量（steering vectors），沿与显式行为指令相同的潜空间方向推动激活；同时通过表面关键词重叠劫持注意力，使其从任务相关上下文（工具schema、用户查询）转移到记忆条目。

## Key Contributions

- **Formalize memory-induced tool-drift** as a novel failure mode where personality-driven biases stored in long-term memory silently affect tool-call parameters in unrelated professional contexts. Establish a rigorous three-condition experimental framework (no memory M∅, neutral Mn, biased Mb) that isolates bias-driven drift from mere memory presence.
- **Construct MEMDRIFT**, a 105-scenario benchmark spanning 5 bias dimensions (speed/impatience, resource frugality, minimalism/conciseness, risk permissiveness, autonomy/self-reliance) and 7 professional domains, generated through a three-stage automated adversarial pipeline (scenario generation → artifact expansion → self-improving adversarial refinement via 4-iteration loop with LLM judge feedback).
- **Evaluate 7 frontier models** (GPT-5.2, GPT-5.4, Claude-Sonnet-4.5, Gemini-2.5-Pro, Gemini-3.1-Pro-Preview, Kimi-K2.5, Qwen3.5-397B-A17B) — all exhibit substantial tool-drift, including reasoning models with extended thinking. Deflection scores increase by up to +3.6 on a 1–5 Likert scale.
- **Show drift persists under production memory architectures**: Test Mem0, MemPalace, and SimpleMem — memory frameworks strip personal context during storage, encoding preferences as global directives and retrieving based on shallow semantic overlap, so biased deflection scores remain comparable to or exceed direct injection.
- **Audit real-world MCP tool ecosystem**: Scan 6,062 tools across 288 verified servers on Smithery, flag 608 with susceptible parameters (bias-aligned values or lexical overlap), and confirm drift on validated subset (e.g., GitHub label_write: update→delete; GitLab create_project: private→public exposing HIPAA-regulated code).
- **Provide mechanistic explanation** on Llama-3.3-70B-Instruct: biased memories act as implicit steering vectors pushing activations along the same latent directions as explicit behavioral instructions (confirmed via contrastive steering vector projection analysis with Cohen's d and AUROC metrics), and redistribute attention from task-relevant regions (tool schema, query) toward memory entries with surface-level keyword overlap to the target parameter.

## Methodology

The paper employs a multi-stage methodology combining benchmark construction, empirical evaluation, real-world auditing, and mechanistic analysis:

**Problem Formulation:** An LLM agent A maintains long-term memory M = {m₁, ..., mₙ} and accesses tools defined by parameter schemas P. The agent constructs input x = [S ∥ M ∥ T ∥ q] and generates tool invocation T(P = θ). Three conditions isolate drift: no-memory (M∅), neutral memory (Mn, containing irrelevant but unbiased facts), and biased memory (Mb, containing personality-driven biases from personal life). Tool-drift occurs when θb deviates from θ∅ while θn ≈ θ∅, confirming bias-driven rather than memory-presence-driven deviation.

**MEMDRIFT Benchmark Construction (3-stage pipeline using Claude-Opus-4.6 as generator/judge, GPT-5.2 as target):**
- *Stage 1 — Scenario Generation:* For each (bias dimension, domain) pair, generate 3 scenarios specifying a professional task and tool.
- *Stage 2 — Artifact Expansion:* Each scenario expanded into tool schema T with parameters P, user prompt q, base/neutral/biased memory sets (5–6 memories each), and drift condition specification.
- *Stage 3 — Self-Improving Adversarial Refinement:* Target model run under 3 memory conditions (k=5 responses each), LLM judge scores deflection on Likert scale, feedback loops into generator over 4 iterations to adversarially maximize drift. Best attempt undergoes human verification.

Strict context separation enforced: all memories grounded in personal life (lifestyle, personality, values); all tool calls target professional tasks — guaranteeing unambiguous ground truth.

**Evaluation Protocol:** Deflection scored by LLM judge (Claude-Opus-4.6) comparing Mb and Mn responses against M∅ baseline on 1–5 Likert scale, with heavier weight on task-critical parameter drift. Two settings: (1) direct memory injection in system prompt; (2) organic encoding via multi-turn conversations with production memory architectures (Mem0, MemPalace, SimpleMem). All models invoked in function-calling mode with structured outputs. Robustness checks include alternate judge (GPT-5.4) agreement analysis and bias-percentage sensitivity analysis.

**Real-World Vulnerability Scanning:** LLM-powered scanner applied to Smithery MCP registry, identifying susceptible parameters via two properties: bias-aligned parameter values (e.g., visibility maps to risk permissiveness) and lexical overlap between memory keywords and parameter/value labels.

**Mechanistic Analysis (on Llama-3.3-70B-Instruct):**
- *Steering Vector Analysis:* Contrastive prompt pairs (bias-aligned vs. bias-opposed) define explicit steering direction v_steer. Memory-induced activation shifts projected onto v_steer to measure implicit steering.
- *Attention Redistribution:* Input partitioned into semantic regions (system, memory, tool schema, query, partial tool call, template). Attention weights at behavior-defining token position extracted across layers 40–79, with delta (αMb − αMn) computed per region and per memory entry.

## Results & Highlights

- **All 7 frontier models exhibit substantial tool-drift** under direct memory injection. Biased deflection scores (sb) are consistently high (3.0–4.2 range) while neutral scores (sn) remain low (≈1.5), confirming drift is driven by personality-biased content. Even a small fraction of biased memories can induce near-maximal drift (sensitivity analysis).
- **Drift persists under all 3 production memory frameworks** (Mem0, MemPalace, SimpleMem) — biased scores remain comparable to or exceed direct injection. Memory architectures strip personal context during storage, converting preferences into decontextualized global directives.
- **Benchmark generalizes beyond target model**: Though generated using GPT-5.2 as target, all evaluated models show elevated deflection.
- **608/6,062 real-world MCP tools flagged vulnerable** (≈10%). Confirmed drift on validated subset with serious consequences: GitHub label_write changes update→delete (destroys issue associations), Brave local_search changes safesearch moderate→off (removes K-12 filtering), GitLab create_project changes private→public (exposes HIPAA codebase).
- **Mechanistic findings (Llama-3.3-70B):**
  - Biased memory projections πMb significantly exceed neutral πMn across all 5 bias dimensions (Cohen's d and AUROC reported per dimension), with separation growing through middle-to-late layers where behavioral representations consolidate.
  - Attention shifts from task-relevant regions (tool schema, query, partial tool call) toward memory region under Mb. Within memory, entries with lexical overlap to target parameter receive disproportionately elevated attention.
- **Defenses are insufficient:**
  - Defensive system prompt (relevance instructions) reduces drift but scores remain substantially elevated (Δsb = −0.52 overall for GPT-5.4).
  - Memory relevance filter (Self-ReCheck with Qwen3-8B) achieves perfect filtering on MEMDRIFT's clean separation but only 61.0% recall / 10.3% FPR on realistic multi-hop cases from PersistBench — does not generalize.
- **Models rarely request clarification** in unconstrained mode, predominantly proceeding directly to the drifted tool call.

## Critical Assessment

- **Novelty: High.** Memory-induced tool-drift is a genuinely novel failure mode at the intersection of memory and tool use — two capabilities previously studied in isolation. The formalization, benchmark, and mechanistic analysis collectively advance understanding significantly.
- **Rigor: Very strong.** Three-condition experimental design cleanly isolates bias effects. Multi-model evaluation, production architecture testing, alternate judge validation, sensitivity analysis, and mechanistic probing on an open-weight model provide multiple layers of evidence.
- **Reproducibility: Good with caveats.** Code and benchmark promised. Pipeline is automated. However, judge-based scoring introduces variability, and mechanistic analysis requires access to model internals (limited to Llama-3.3-70B).
- **Significance: Very high.** Directly relevant to deployed production systems (ChatGPT memory, Claude Code, MCP-based agents). As agents gain more consequential tool access, this vulnerability escalates. The finding that production memory architectures worsen or fail to address drift is particularly concerning.
- **Red flags:**
  - Benchmark size (105 scenarios) is modest; each (bias, domain) pair has only 3 scenarios, which may not capture the full variability within each combination.
  - Mechanistic analysis limited to a single open-weight model (Llama-3.3-70B) that is not among the 7 evaluated frontier models — unclear if steering vector findings transfer to closed-source models.
  - Real-world validation is schema-only (no live server connections); actual deployment behavior may differ.
  - Single-turn, single-tool evaluation; production agents chain multiple tools across servers, which could amplify or attenuate drift.
  - The boundary between "harmful drift" and "useful personalization" is clean in MEMDRIFT by design but messy in reality — the paper acknowledges this but doesn't fully resolve it.

## Strengths

- Identifies a genuinely novel, practically critical failure mode that no prior work has examined.
- Exceptionally thorough evaluation: 7 models, 3 production architectures, real-world tool audit, mechanistic analysis, defense evaluation.
- Clean experimental design with three-condition framework that eliminates confounds.
- Mechanistic analysis provides causal explanation (steering vectors + attention redistribution), not just correlation.
- Real-world MCP server scanning demonstrates immediate practical relevance.
- Paper is well-organized with clear findings, takeaways, and logical flow.

## Limitations

- **Benchmark scale**: 105 scenarios (3 per bias×domain pair) is small; broader coverage could reveal additional drift patterns or boundary conditions.
- **Mechanistic generalization**: Steering vector and attention analysis only on Llama-3.3-70B; unclear if findings transfer to frontier closed-source models.
- **Single-turn, single-tool**: Production agents chain multiple tools; drift dynamics in multi-tool workflows are unknown.
- **Clean separation assumption**: MEMDRIFT's strict personal–professional separation enables unambiguous measurement but doesn't reflect real-world ambiguity where personal context may be legitimately relevant.
- **Defense evaluation limited**: Only prompt-based and filter-based defenses tested; architectural or training-time interventions left for future work.
- **English-only**: Scenarios and tool schemas are in English; cross-cultural/cross-lingual drift behavior unexplored.

## Relevance

本文揭示的"记忆诱导工具漂移"问题直接关联LLM智能体安全与对齐研究，对设计带长期记忆的工具调用智能体（如MCP生态、ChatGPT记忆系统）具有重要警示意义，属于智能体安全方向的高影响力工作，其机制分析方法（转向向量+注意力重分配）也为理解LLM记忆如何影响行为提供了新视角。

## Questions for Further Exploration

- Can dedicated architectural mechanisms (e.g., memory-context gating layers at the tool-call generation boundary) fully eliminate drift without degrading useful personalization?
- How does tool-drift compound in multi-tool agentic workflows where drifted outputs from one tool feed into subsequent tool calls?
- Does the steering vector equivalence finding hold for frontier closed-source models, and can it be detected via black-box probing?
- Could contrastive fine-tuning on MEMDRIFT-style drifted vs. non-drifted examples (with held-out bias dimensions as generalization test) instill genuine contextual discernment?
- What is the interaction between memory-induced tool-drift and adversarial memory injection attacks — does drift create a larger attack surface for intentional manipulation?
- How should the boundary between legitimate personalization and harmful drift be operationalized in real-world settings where personal and professional contexts overlap?
