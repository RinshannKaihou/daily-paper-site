> **一句话总结：** 本文提出了"随机-确定边界"(SDB)作为生产级LLM Agent运行时的核心架构原语，围绕协调/状态/控制三大关注点组织了六种运行时模式，并通过五步选型方法论和故障签名目录将分布式系统的成熟模式系统地适配到随机工作者的场景中。

> **TL;DR:** This paper formalizes the Stochastic-Deterministic Boundary (SDB) — a four-part contract (proposer, verifier, commit step, reject signal) governing how stochastic LLM outputs become deterministic system actions — as the foundational primitive for production LLM agent runtimes. It organizes runtime design into three concerns (Coordination, State, Control), presents six composable patterns (hierarchical delegation, scatter-gather+saga, event-driven sequencing, shared state machine, supervisor+gate, human-in-the-loop) with distributed-systems lineage, and provides a five-step selection methodology with decision predicates, a failure-signature diagnostic catalog, and a novel "replay divergence" failure mode. Applied to five workloads with a runnable LangGraph/Google ADK reference implementation for a 90-day contract-renewal agent. The reliability decomposition y(t) = μt + σ·ξ(t) argues that architectural momentum (μ) dominates per-call model variance (σ) as models improve.

## 核心贡献

- 命名并形式化了**随机-确定边界（SDB）**：由proposer（LLM提出输出）、verifier（确定性验证）、commit step（写入系统状态）、reject signal（拒绝并回退）四部分组成的契约，将LLM的非确定性输出转化为确定性系统动作
- 将Agent运行时设计组织为**协调（Coordination）、状态（State）、控制（Control）**三大关注点，三者的交集构成生产运行时
- 提供了**六种运行时模式**的目录（P1-P6），每种模式追溯到分布式系统概念（saga、两阶段提交、事件溯源、Erlang监督树等），并分析当worker是随机模型时需要哪些变化
- 提出了**五步选型方法论**：分类运行时 → 选择状态骨架 → 包装协调模式 → 绑定控制模式 → 确定构建顺序，每步都有明确的决策谓词和方法门槛（method gate）
- 建立了**故障签名目录**：为每种模式定义了功能性故障、性能故障和漂移故障的症状-原因-修正三元组
- 发现了**重放分歧（replay divergence）**这一新型故障模式：同一输入事件在新模型版本下回放产生不同的下游事件，这是从P3迁移到P5的触发信号
- 通过**可靠性分解** y(t) = μt + σ·ξ(t) 将架构动量μ与每次调用的模型方差σ分离，论证随模型改进σ压缩，μ成为长期可靠性的主导杠杆
- 审计了五个开源Agent框架（LangGraph、Google ADK、CrewAI、OpenAI Agents JS、AutoGen）的SDB实现，分类了21篇公开故障事后报告

## Key Contributions

- **SDB formalization**: Defines a four-part contract (proposer, verifier, commit step, reject signal) that explicitly marks where stochastic outputs cross into deterministic system actions — audited against five open-source frameworks (LangGraph, Google ADK, CrewAI, OpenAI Agents JS, AutoGen)
- **Three-concern decomposition**: Organizes runtime design into Coordination (how work splits/merges), State (what the system remembers between failures), and Control (who decides what runs and when to stop), with State as the "spine"
- **Six-pattern catalog** with explicit distributed-systems lineage: P1 (hierarchical delegation → supervisor trees), P2 (scatter-gather+saga → Garcia-Molina sagas), P3 (event-driven sequencing → Kreps' log), P4 (supervisor+gate → Erlang OTP), P5 (shared state machine → CAS + Raft), P6 (human in the loop → four control planes: kill switch, escalation, approval, throttling)
- **Five-step selection methodology** with decision predicates and method gates: classify runtime (Conversational/Autonomous/Long-Horizon) → choose spine (P3 vs P5 based on three predicates) → wrap with coordination (P1 vs P2) → bound with control (P4 ± P6) → sequence the build (dashboard before agent)
- **Failure-signature catalog** (Section 5.2): Per-pattern symptom/cause/correction triples covering functional, performance, and drift failures
- **Replay divergence**: A discrete cross-version failure mode where LLM-based consumers of a deterministic event log produce different downstream outputs under model-version or prompt changes — distinct from continuous architectural momentum
- **Reliability decomposition** y(t) = μt + σ·ξ(t): As calendar time grows, the architecturally-shaped trajectory μt dominates bounded per-call noise σ·ξ(t)
- **Five worked applications** spanning all three runtime classes, with a runnable reference implementation in LangGraph and Google ADK for a 90-day telco contract-renewal agent
- **Six-line architecture decision record**: The concrete output artifact — runtime class, spine, coordination, control, build sequence, date/model version — that auditors can read instead of guessing

## Methodology

The paper is conceptual and methodological rather than empirical. Its approach:

1. **SDB audit of existing frameworks**: Examines five open-source agent frameworks to identify which SDB components they implement and which they omit (e.g., OpenAI Agents JS had a bug where rejected tool calls were reported as `status: 'completed'`, causing hallucinated success). Classifies 21 published failure post-mortems.

2. **Pattern catalog construction**: Each pattern is defined by answering three questions: which concern it addresses, what failure mode it prevents that no existing pattern prevents, and its typed-contract specification (input type, output type, deadline, retry budget, partial-result policy).

3. **Five-step selection methodology**:
   - **Step 1 — Classify runtime**: Duration of work unit + whether the world changes midflight → Conversational, Autonomous, or Long-Horizon. Determines which concern dominates.
   - **Step 2 — Choose spine**: Three predicates determine P5 (shared state machine) vs P3 (event-driven) vs none: (1) pauses > 1 hour, (2) state not reconstructible from input, (3) world changes during pause. Requires senior engineer sign-off.
   - **Step 3 — Wrap with coordination**: P1 if single owner + independent sub-tasks + deterministic merge feasible. P2 if peers hit external systems with side-effects or partial failures are costly. Both can compose.
   - **Step 4 — Bound with control**: P4 (supervisor+gate) always if any external side-effects. P6 (human-in-the-loop) if legal/financial consequence, cases outside policy envelope, or auditors will ask.
   - **Step 5 — Sequence the build**: Dashboard before agent. State schema + observability → gate + audit log → orchestrator + first sub-agent → remaining sub-agents → P6 control planes in order: kill switch, escalation, approval, throttling.

4. **Diagnostic procedure**: When reliability degrades: (1) pin failing model version, (2) replay on prior version, (3) if persistent → functional failure → apply signature catalog, (4) if resolved on prior version → replay divergence → migrate P3→P5, (5) if neither reproduces → variance → increase pass@k.

5. **Three observability lenses**: Operational (health: P95 latency, queue depth), Business (outcomes: what happened), Compliance (proof: decision lineage, policy/model version per decision). Unified by a single request identifier.

## Results &amp; Highlights

- **Five workloads demonstrate methodology responsiveness**: Two Long-Horizon workloads (Number Port-in vs Lead Warming) select different spines (P5 vs P3) because predicate (2) fires differently — the methodology gives different answers when predicates differ and same answers when they match.
- **Contract Renewal reference implementation**: Full LangGraph + Google ADK implementation composing all six patterns, running against IBM Telco Customer Churn dataset (7,043 records, ~26.5% churn rate) projected into 100 renewal scenarios. Available at `github.com/vasundras/agent-runtime-patterns`.
- **Billing & Payment Assist (Conversational)**: No durable spine needed — P1 + P4 sufficient. Adding P5 would over-engineer; adding P6 on top of an expert-facing tool creates circular review.
- **Order Management Fall-out Scanner (Autonomous)**: P3 spine (upstream already emits events) + P2 coordination (heterogeneous sources fail independently) + light P6 (throttling only).
- **Number Port-in (Long-Horizon)**: Full P5 + P1+P2 + P4+P6 — cross-carrier sagas handle the painful case where new-carrier provisions then old-carrier rejects.
- **Replay divergence evidence**: Promptfoo documented a 23-point drop (94%→71%) in prompt-injection resistance when upgrading GPT-4o→GPT-4.1, illustrating cross-version drift. OpenAI Agents JS issue #1104 documents a reject-signal bug causing hallucinated tool-call success.
- **The methodology produces a six-line architecture decision record** (Table 2) that is the concrete handoff artifact for reviewers and auditors.

## Critical Assessment

- **Novelty**: Moderate-to-high. Individual patterns are not new (sagas date to 1987, Erlang supervision to 2003), but the systematic composition for stochastic workers is original. The SDB abstraction is essentially a two-phase commit with a stochastic proposer — useful naming but the concept is recognizable. **Replay divergence is the most genuinely novel contribution**: it names a failure mode that practitioners experience but hasn't been formally characterized. The reliability framing y(t) = μt + σ·ξ(t) is explicitly presented as a metaphor, not a derived model.

- **Rigor**: The paper is honest about its limitations. Section 8.2 explicitly lists threats to validity: construction bias in worked examples (not independently audited), recent timeframe (tested against Claude Sonnet 4.6 only), arbitrary predicate thresholds (1 hour, 3 retries), replay availability dependency, and the audit/classification being the author's own. The failure post-mortem classification (21 incidents) is reproducible from cited sources but subjective. The reliability decomposition is "stylized" — acknowledged as metaphorical, not mathematically derived.

- **Reproducibility**: Strong. Companion repo with MIT-licensed code, runnable against a public dataset (IBM Telco Churn). The six-line architecture decision record format is concrete and auditable. The five-step methodology is prescriptive enough to replicate.

- **Significance**: High for practitioners, moderate for researchers. This fills a real gap: production teams building LLM agents lack a systematic architecture selection procedure. The failure-signature catalog is immediately actionable. However, the paper does not demonstrate that teams using this methodology produce more reliable systems than teams that don't — there is no controlled comparison.

- **Red flags**:
  - Single authorship is ambitious for this scope. The five worked applications are all constructed by the author — no independent validation.
  - The telecommunications-domain bias may limit generalizability to other sectors (healthcare, finance, etc.).
  - No production deployment data: the paper argues for production reliability but validates only against synthetic scenarios.
  - The open pattern catalog (P7-P9 are speculative) means the methodology is incomplete.
  - The claim that μ dominates σ over time is structurally intuitive (linear growth vs bounded noise) but lacks empirical calibration — how quickly does this crossover happen in practice?

## Strengths

- Addresses a genuinely unsolved practical problem with a systematic, repeatable procedure
- The SDB abstraction provides crisp vocabulary for a boundary that every production agent system has but few name explicitly
- Excellent grounding in distributed-systems literature — each pattern's lineage is clearly traced
- The failure-signature catalog (Section 5.2) is immediately useful for on-call engineers debugging production agent failures
- Replay divergence is a novel, well-named, practically important concept
- The six-line architecture decision record is a concrete, auditable output artifact
- Honest about threats to validity and what the methodology does not solve (Section 8.3)
- The "dashboard before agent" sequencing advice (Step 5) is counterintuitive and valuable
- The two Long-Horizon contrasts (Port-in vs Lead Warming) effectively demonstrate that the methodology responds to workload predicates, not just runtime class labels

## Limitations

- **No empirical validation of the methodology itself**: No A/B comparison of teams using vs. not using the methodology, no measured reliability improvements from adopting specific patterns
- **Single author, single domain**: All worked applications are telecommunications-adjacent; no validation in healthcare, finance, or other regulated domains
- **Synthetic validation only**: The reference implementation runs against a projected dataset, not a real production system with real customers
- **Predicate thresholds are conventions**: The 1-hour pause threshold, 3-retry CAS threshold, and other numeric cutoffs are defaults, not empirically calibrated
- **Pattern catalog is incomplete**: P7 (Shared Memory Store), P8 (Tenant Isolation), P9 (Cross-Runtime Handoff) are named but not developed
- **Model-dependent timeframe**: Tested against Claude Sonnet 4.6; subsequent model improvements could collapse some patterns (e.g., eliminating replay divergence would weaken the P3→P5 migration argument)
- **No guidance on upstream concerns**: Token budgets, model selection, RAG index sizing, prompt management are explicitly out of scope
- **Organizational prerequisites**: P6 (human-in-the-loop) requires approval SLAs and specialist queues that the methodology makes visible but does not help negotiate

## Relevance

本文提出的SDB概念和六种运行时模式对构建生产级LLM Agent系统具有直接参考价值。五步选型方法论和故障签名目录可立即应用于Agent系统的架构设计与生产故障排查，replay divergence这一故障模式对Agent可靠性工程有重要启发意义。

## Questions for Further Exploration

- Can architectural momentum μ be measured empirically from production telemetry, and how quickly does it dominate per-call variance σ in real deployments?
- How should the SDB contract be adapted for multi-agent systems where multiple stochastic proposers interact without a single deterministic verifier?
- What are the failure signatures for the anticipated patterns P7 (Shared Memory Store), P8 (Tenant Isolation), and P9 (Cross-Runtime Handoff)?
- How does replay divergence interact with RAG systems where the retrieval index changes across versions in addition to the model?
- Can the five-step methodology be partially automated — e.g., can a static analyzer infer the runtime class and spine predicates from an existing codebase?
- How do the patterns compose when an agent must operate across runtime classes (e.g., a Long-Horizon workflow that hands off to a Conversational sub-agent)?
- What is the quantitative cost of migrating from P3 to P5 mid-production, and when is it cheaper to accept replay divergence than to migrate?
