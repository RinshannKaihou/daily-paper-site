GEM: GPU-Variability-Aware Expert to GPU Mapping for MoE Systems

Sourish Wawdhane · Avinash Kumar · Poulami Das UT Austin

First framework that accounts for GPU hardware performance variability when mapping MoE experts to GPUs. Balances per-GPU completion latencies rather than token loads, and separates correlated temporal experts across GPUs.

Key Finding

Even within a homogeneous GPU cluster, hardware performance variability is a first-order source of stragglers in MoE expert-parallel inference. Across 128 NVIDIA L40 GPUs, the fastest GPU completes MoE-layer work 27.7% sooner than the slowest. GEM exploits this by assigning more tokens to faster GPUs so all GPUs finish synchronously — a latency-balancing paradigm that inverts the conventional load-balancing assumption.

Why This Matters

  • GPU variability worsens with scale: gap grows from 11.9% at 4 GPUs to 23.4% at 64 GPUs
  • MoE deployments are exploding (Mixtral, Llama-4, Qwen3, DeepSeek) — all face this bottleneck
  • Orthogonal to existing MoE optimizations — composable for additive gains
  • Near-zero deployment cost: minutes of profiling + seconds of mapping

Methodology Pipeline

1
Expert Utilization Trace — Collect per-expert token counts over 16 generation steps during online inference with vLLM's default linear mapping.
2
Tile-Boundary Profiling — Micro-benchmark per-GPU latency at tile boundaries (multiples of 32/64 tokens) only. Cuts profiling from 12+ hours to 0.5–3.6 minutes.
3
Iterative Mapping Search — Greedy initial placement (heaviest experts first), then pairwise swaps minimizing straggler score S(M). K=30 random restarts with 20% noise.
4
Static Deployment — Optimized mapping replaces linear mapping at model load time. Overhead comparable to vLLM's EPLB redistribution.

Two Expert Patterns

CONSISTENT
Always active at high frequency. Placed by minimizing the maximum per-GPU load.
TEMPORAL
Infrequently but simultaneously active. Detected via Pearson correlation and separated across GPUs.

Llama-4-Scout gains 3.5% from temporal placement alone, even with uniform GPUs.

Strengths

  • Identifies a real, previously overlooked problem with empirical evidence across 128 GPUs
  • Clean two-insight formulation: balance latencies not loads + separate correlated temporal experts
  • Extremely low deployment overhead — minutes of profiling, seconds of search
  • Comprehensive evaluation: 5 diverse MoE architectures, 2 datasets, 3 variability levels
  • Orthogonal to existing MoE optimization techniques (expert replication, communication reduction)

Critical Edge

  • Single-node only: All experiments on 4× H200 with emulated variability. No real multi-node validation.
  • Static mapping: Cannot adapt to workload drift once deployed. 16-step trace assumed representative.
  • No optimality guarantee: Heuristic search provides no bounds, though convergence is fast.
  • Equal experts assumed: Does not handle heterogeneous expert sizes (e.g., DeepSeek-V3 shared+routed).
  • Memory overhead: Expert swapping cost is not quantified.

Limitations

  • Mapping is fixed post-deployment — no online adaptation
  • Requires hardware access for variability emulation via power capping
  • Limited to expert-parallel within a single MoE layer
  • Assumes equal number of experts per GPU — breaks for heterogeneous memory

Open Questions

  1. How does GEM interact with dynamic expert replication (e.g., vLLM EPLB) in a combined system?
  2. Can traces be continuously updated to detect workload drift and trigger remapping?
  3. What is the impact on models with heterogeneous expert sizes (DeepSeek-V3)?
  4. How does GEM perform in multi-node settings with NVLink topology variability?
  5. Could a learned mapping policy (RL) outperform the heuristic for 128+ experts?
