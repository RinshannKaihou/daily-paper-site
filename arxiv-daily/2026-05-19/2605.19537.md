> **一句话总结：** 本文系统研究了不同推理后端（vLLM、SGLang、llama.cpp等）对LLM基准测试结果的显著影响，发现仅切换后端即可导致高达16.6个百分点的性能差异，且在35000篇ML论文中推理栈几乎未被报告。

> **TL;DR:** This paper shows that the choice of inference backend (vLLM, SGLang, llama.cpp, LMDeploy, Ollama) can shift LLM benchmark scores by up to 16.6 percentage points even with identical model weights and greedy decoding, driven by hidden engine defaults and numerical drift from system-level optimizations. A survey of 35,000 ML papers reveals the inference stack is almost never reported, undermining reproducibility.

## 核心贡献

- 调研了200个推理引擎的生态系统，建立了自托管库、托管平台、聚合路由器三级分类体系
- 在控制变量条件下系统评估了5种主流推理后端对LLM基准测试的影响，量化了性能差异
- 分析了35000篇顶级ML/NLP会议论文，发现推理后端的报告率极低，构成严重的可复现性隐患
- 通过消融实验追溯到两大根源：可纠正的引擎默认参数和优化引入的数值漂移

## Key Contributions

- Surveyed 200 inference engines, establishing a taxonomy (self-hosted libraries, managed platforms, aggregators/routers)
- Conducted controlled experiments across 5 backends (vLLM, SGLang, llama.cpp, LMDeploy, Ollama), 5 models, and 4 benchmarks showing up to 16.6 pp score variance
- Analyzed 35,000 papers from top ML venues (NeurIPS, ICML, ICLR, ACL, EMNLP) finding inference stacks are rarely documented
- Root cause analysis identifying hidden engine defaults (BOS token injection, repetition penalties) and optimization-induced numerical drift (prefix caching, CUDA graphs, custom kernels) as primary sources of divergence

## Methodology

The authors design a rigorously controlled experimental framework isolating the inference backend as the sole variable:

- **Backends:** HuggingFace transformers (reference), vLLM, SGLang, llama.cpp, LMDeploy, Ollama — all with fixed versions
- **Models:** Llama 3.1 8B, Qwen3 4B, Qwen3 30B (standard); DeepSeek R1 Distill Qwen 7B, Qwen3 Thinking 30B (reasoning)
- **Benchmarks:** GSM8K (math), GPQA Diamond (science), SimpleQA Verified (factuality), LiveCodeBench v6 (code)
- **Controls:** Greedy decoding (T=0), FP16 precision, externally applied chat templates, batch size 1, 12 seeds per configuration, single NVIDIA H100 GPU, unified Docker container
- **Metrics:** Benchmark accuracy, disagreement rate vs. reference, length error, divergence index (first mismatched token position), logprob RMSE, top-5 token Jaccard similarity
- **Literature survey:** Keyword filtering + LLM-as-judge (Qwen3-235B) on 35,000 papers from NeurIPS/ICML/ICLR/ACL/EMNLP (2023–2025), classified into 4 reproducibility tiers

## Results & Highlights

- **Up to 16.6 pp accuracy spread:** DeepSeek R1 Distill Qwen 7B on GSM8K ranged from 61.87% (Ollama) to 78.47% (transformers)
- **No backend matches transformers perfectly** across all benchmarks
- **Reasoning models are most affected:** Early token-level divergence reshapes chain-of-thought; Ollama produces DeepSeek R1 outputs ~9,000 tokens shorter than reference on GPQA
- **Disagreement rates exceed accuracy gaps:** e.g., 27.37% for DeepSeek R1 on Ollama/GSM8K, meaning backends flip the model's decision boundary on hundreds of questions
- **Safety implications:** JailbreakBench shows DeepSeek R1's adversarial vulnerability fluctuates by 8.9 pp depending on backend
- **Root causes — systematic defaults:** Ollama's hidden repetition penalty (1.1) and forced BOS token injection account for the most extreme outliers; correcting these recovers ~12 pp for DeepSeek R1
- **Root causes — numerical drift:** Prefix caching, CUDA graphs, multi-threaded Top-K tie-breaking, and FP32 accumulation in llama.cpp/Ollama introduce compounding drift across layers
- **Paper survey:** Of 9,018 relevant papers, only 36% without code report their backend; 37% of GitHub repos lack dependency files; inference backend is effectively an undocumented variable

## Critical Assessment

- **Novelty: High.** The inference backend as a reproducibility variable has been largely ignored. The systematic quantification and root cause analysis are novel contributions.
- **Rigor: Strong.** Controlled experiments with fixed hardware, Docker containers, fixed software versions, 12 seeds, externally applied chat templates. The ablation studies isolating individual optimizations are well-designed.
- **Reproducibility: Good.** Fixed versions reported, Docker container specified, standardized pipeline. However, code is not yet released ("upon publication").
- **Significance: High.** The finding that backend choice alone can shift scores by 16+ pp — enough to change leaderboard rankings — is consequential for the entire field.
- **Red flags:**
  - Greedy decoding is suboptimal for reasoning models (the authors acknowledge this); the absolute numbers for DeepSeek R1 may not reflect typical usage, though relative differences remain valid
  - The paper survey relies on an LLM judge (Qwen3-235B) which introduces its own classification noise; manual verification is limited to random subsets
  - Not all 200 engines are evaluated experimentally (only 5 + reference), limiting generalizability of specific numerical claims
  - The study represents a lower bound — real production variance may be higher, which the authors acknowledge

## Strengths

- Comprehensive and systematic: combines landscape survey, controlled experiments, literature meta-analysis, and root cause analysis
- Practical recommendations for researchers, leaderboard maintainers, and engine developers
- Honest about limitations (greedy decoding, lower-bound nature of findings)
- Safety angle (jailbreak vulnerability fluctuation) adds real-world urgency beyond academic benchmarking
- Clear taxonomy of 200 inference engines provides lasting reference value

## Limitations

- Only 5 of 200+ engines tested experimentally
- Greedy decoding may not represent typical deployment conditions for reasoning models
- LLM-as-judge for paper classification introduces measurement uncertainty
- Code and artifacts not yet publicly available
- Single GPU architecture (H100) as primary testbed; L40 ablation shows slightly different magnitudes but limited hardware diversity
- Ablating every custom kernel is acknowledged as infeasible

## Relevance

该论文直接揭示了LLM评测中推理后端这一被忽视的变量，对任何依赖benchmark进行模型选型或对比研究的场景都有重要参考价值，与LLM评估方法论和可复现性研究高度相关。

## Questions for Further Exploration

- How does backend-induced variance interact with quantization methods (INT8, INT4) — does quantization amplify or mask backend differences?
- What is the minimum set of reporting requirements needed to make LLM benchmark results reproducible?
- Could a standardized "backend benchmarking protocol" be adopted by major leaderboards (Open LLM Leaderboard, Chatbot Arena)?
- Do closed-source API providers (OpenAI, Anthropic, Google) exhibit similar hidden variance across API versions?
- How does backend variance scale with model size beyond 30B parameters?
