A Methodology for Selecting and Composing Runtime Architecture Patterns for Production LLM Agents

Key Finding

The Stochastic-Deterministic Boundary is the foundational architectural primitive for building reliable production LLM agent runtimes.

Six runtime patterns and a failure-signature catalog give production teams a systematic architecture selection procedure for LLM agents — where none existed before.

No controlled comparison. The methodology is unvalidated — there is no evidence that teams using it build more reliable systems than teams that don't.

Methodology

The SDB Contract

Four parts govern how stochastic LLM outputs become deterministic system actions:

Proposer
LLM generates output
Verifier
Deterministic check
Commit
Write to system state
Reject
Roll back & retry

Five-Step Selection

  1. Classify runtime — Duration + world-change → Conversational, Autonomous, or Long-Horizon
  2. Choose spine — Three predicates determine P5 (shared state machine) vs P3 (event-driven)
  3. Wrap coordination — P1 (hierarchical) if single owner; P2 (scatter-gather + saga) if peers with side-effects
  4. Bound with control — P4 (supervisor+gate) always if external side-effects; P6 if legal/financial consequence
  5. Sequence the build — Dashboard before agent. Observability → gate → orchestrator → sub-agents → control planes
y(t) = μt + σ·ξ(t)

Reliability decomposition: architectural momentum μ dominates per-call variance σ as models improve

Six Runtime Patterns
P1 Hierarchical Delegation → Erlang supervisor trees
P2 Scatter-Gather + Saga → Garcia-Molina sagas (1987)
P3 Event-Driven Sequencing → Kreps' log (Kafka)
P4 Supervisor + Gate → Erlang OTP
P5 Shared State Machine → CAS + Raft consensus
P6 Human in the Loop → Kill switch / escalation / approval / throttling
Results & Highlights
⚠
Replay divergence demonstrated: Promptfoo recorded a 23-point drop (94%→71%) in prompt-injection resistance on GPT-4o → GPT-4.1 upgrade — a cross-version failure mode this paper formally names for the first time.
  • Five workloads across three runtime classes demonstrate methodology responsiveness
  • Two Long-Horizon workloads (Port-in vs Lead Warming) select different spines when predicates differ
  • Reference implementation: LangGraph + Google ADK, IBM Telco Churn dataset (7,043 records, ~26.5% churn rate)
  • OpenAI Agents JS issue #1104: rejected tool calls reported as completed — hallucinated success
  • Output artifact: a six-line architecture decision record auditors can read instead of guessing
Strengths & Limitations

Strengths

  • Addresses an unsolved practical problem with a repeatable procedure
  • SDB provides crisp vocabulary for a boundary every system has but few name
  • Excellent distributed-systems grounding — each pattern traced to lineage
  • Failure-signature catalog is immediately actionable for on-call engineers
  • "Dashboard before agent" sequencing advice is counterintuitive and valuable

Limitations

  • No A/B comparison — no measured reliability improvements
  • Single author, single domain (telecom); no healthcare/finance validation
  • Synthetic validation only — projected dataset, not real production
  • Predicate thresholds (1h pause, 3 retries) are conventions, not calibrated
  • Pattern catalog incomplete: P7–P9 are named but not developed
