Switching inference backends alone shifts LLM benchmark scores by up to 16.6 percentage points—even with identical model weights and greedy decoding.
The Silent Hyperparameter: Quantifying the Impact of Inference Backends on LLM Reproducibility · David Pape, Jonathan Evertz, Lea Schönherr · CISPA Helmholtz Center · 2026-05-19
llm-evaluation reproducibility inference-backends benchmarking system-optimization
Key Metric
16.6 pp
Accuracy spread on GSM8K for DeepSeek R1 Distill Qwen 7B—61.87% (Ollama) to 78.47% (transformers)—from backend alone.
Why This Matters
Inference backends are an undocumented variable in LLM evaluation—switching them alone reshapes model rankings, undermining every benchmark comparison you rely on.
Critical Edge
Only 5 of 200+ engines were tested experimentally, limiting generalizability of specific numerical claims—real production variance may be even higher.

Methodology

Rigorously controlled experiments isolating the inference backend as the sole variable:

  • 6 backends: HuggingFace transformers (reference), vLLM, SGLang, llama.cpp, LMDeploy, Ollama — all fixed versions
  • 5 models: Llama 3.1 8B, Qwen3 4B, Qwen3 30B (standard); DeepSeek R1 Distill Qwen 7B, Qwen3 Thinking 30B (reasoning)
  • 4 benchmarks: GSM8K (math), GPQA Diamond (science), SimpleQA Verified (factuality), LiveCodeBench v6 (code)
  • Controls: Greedy decoding (T=0), FP16, externally applied chat templates, batch size 1, 12 seeds, single NVIDIA H100, unified Docker
Identical weights → Fixed hardware → Vary backend only → Measure divergence

Additional metrics: disagreement rate vs. reference, divergence index (first mismatched token), logprob RMSE, top-5 token Jaccard similarity.

Literature survey: LLM-as-judge (Qwen3-235B) classified 35,000 papers from NeurIPS/ICML/ICLR/ACL/EMNLP (2023–2025) into 4 reproducibility tiers.

Results

FindingDetail
Max accuracy spread 16.6 pp (DeepSeek R1, GSM8K)
Decision flips 27.37% disagreement rate on Ollama/GSM8K
Reasoning model sensitivity Ollama outputs ~9,000 fewer tokens on GPQA
Safety variance 8.9 pp jailbreak vulnerability shift (JailbreakBench)
Defaults fix Removing Ollama's hidden rep-penalty recovers ~12 pp

Root causes identified:

  • Hidden defaults: Ollama injects BOS tokens and applies repetition penalty (1.1) by default
  • Numerical drift: Prefix caching, CUDA graphs, multi-threaded Top-K tie-breaking, FP32 accumulation compound across layers

Of 9,018 relevant ML papers surveyed: only 36% without code report their backend; 37% of GitHub repos lack dependency files. The inference stack is effectively an undocumented variable.

Strengths

  • Combines landscape survey, controlled experiments, meta-analysis, and root cause analysis in one study
  • Practical recommendations for researchers, leaderboard maintainers, and engine developers
  • Safety angle (jailbreak vulnerability fluctuation) adds real-world urgency beyond benchmarking
  • Clear taxonomy of 200 inference engines provides lasting reference value
  • Transparent about limitations and lower-bound nature of findings

Limitations

  • Only 5 of 200+ engines tested experimentally
  • Greedy decoding may not reflect typical deployment for reasoning models
  • LLM-as-judge classification introduces measurement uncertainty
  • Code and artifacts not yet publicly available
  • Single GPU architecture (H100) as primary testbed; limited hardware diversity
  • Ablating every custom kernel is acknowledged as infeasible

Questions for Further Exploration

  • How does backend-induced variance interact with quantization (INT8, INT4)—does quantization amplify or mask backend differences?
  • What minimum reporting requirements would make LLM benchmark results reproducible?
  • Could a standardized “backend benchmarking protocol” be adopted by major leaderboards (Open LLM Leaderboard, Chatbot Arena)?
  • Do closed-source API providers (OpenAI, Anthropic, Google) exhibit similar hidden variance across API versions?
  • How does backend variance scale with model size beyond 30B parameters?
