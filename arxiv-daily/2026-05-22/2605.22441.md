# A Constant-Time Implementation Methodology for Activation Functions on Microcontrollers

## 一句话总结

本文提出了一种面向嵌入式微控制器上神经网络激活函数的恒定时间实现方法，通过无分支掩码选择、固定开销Padé有理近似和周期对齐技术，在ARM Cortex-M4上实现ReLU、sigmoid、tanh、GELU和Swish五种激活函数的完全输入无关执行时间（88/108周期），有效消除计时侧信道泄漏。

## TL;DR

Embedded neural-network inference on microcontrollers can leak information through timing side channels in activation functions. This paper proposes a constant-time implementation methodology combining branchless masking-based selection, fixed-cost Padé-type rational approximation, dummy arithmetic where needed, and cycle alignment. Validated on ARM Cortex-M4 for ReLU, sigmoid, tanh, GELU, and Swish, the protected implementations achieve identical cycle counts for all inputs (88 cycles for the three-function setting over [-8,8], 108 cycles for the five-function setting over [-500,500]) while maintaining high numerical accuracy. The authors also demonstrate that a desynchronization-based countermeasure remains vulnerable to template-based timing attacks, directly motivating the constant-time approach.

## 核心贡献

1. **恒定时间激活函数方法论**: 提出了一套完整的恒定时间激活函数实现方法论，涵盖基于掩码的无分支选择、固定开销Padé有理近似、冗余运算和周期对齐，有效消除基于激活函数的计时侧信道泄漏。
2. **统一的有理近似核心**: 所有非线性激活函数（sigmoid、tanh、GELU、Swish）共享同一个基于Padé近似的 R_tanh(x) 计算核心，通过数学恒等式（如 sigmoid(x) = (1+tanh(x/2))/2）派生各函数，降低了实现复杂度。
3. **反制措施安全性分析**: 评估了一种基于去同步化的对策（随机延迟插入），证明其对基于模板的计时攻击仍然脆弱——攻击者在约2000次测量后即可区分激活函数类型。
4. **精度与安全性的平衡**: 在保证恒定执行时间的同时，近似函数保持高精度（如sigmoid的最大绝对误差仅 7.51e-6），且对于sigmoid和tanh，受保护实现反而比未保护版本更快。
5. **开源实现**: 代码公开于 https://github.com/andrewyvt/activation_timing。

## Key Contributions

1. **Constant-time activation function methodology**: A complete methodology combining branchless masking-based selection, fixed-cost Padé-type rational approximation, dummy arithmetic, and cycle alignment to eliminate timing side-channel leakage from activation function evaluation.
2. **Unified rational approximation core**: All nonlinear activation functions (sigmoid, tanh, GELU, Swish) share a single R_tanh(x) Padé rational approximation core, derived through mathematical identities (e.g., sigmoid(x) = (1+tanh(x/2))/2), reducing implementation complexity.
3. **Countermeasure vulnerability analysis**: Demonstrated that a desynchronization-based countermeasure (random delay insertion) remains vulnerable to template-based timing attacks -- an adversary can distinguish activation function types after approximately 2,000 measurements.
4. **Accuracy-security trade-off**: Achieved identical cycle counts across all inputs while maintaining high numerical accuracy (e.g., sigmoid max absolute error only 7.51e-6). Notably, protected sigmoid and tanh implementations are faster than their unprotected counterparts.
5. **Open-source implementation**: Code publicly available at https://github.com/andrewyvt/activation_timing.

## Methodology

The proposed methodology is structured around four tightly integrated techniques:

### 1. Branchless Masking-Based Selection (Section IV-B)
Instead of conditional branches (`if-else`), the implementation uses bit-mask-based selection on IEEE-754 floating-point representations. A comparison creates a 32-bit all-zero or all-one mask, which is applied to the bit patterns of two candidate values via bitwise AND/OR operations. This ensures the instruction sequence is identical regardless of input:

```c
float ct_select_f32(float a, float b, uint32_t mask) {
    uint32_t ua = std::bit_cast<uint32_t>(a);
    uint32_t ub = std::bit_cast<uint32_t>(b);
    uint32_t ur = (ua & ~mask) | (ub & mask);
    return std::bit_cast<float>(ur);
}
```

### 2. Fixed-Cost Padé Rational Approximation (Section IV-C)
Nonlinear functions are approximated using a shared Padé rational approximation for tanh:

R_tanh(x) = x * P(x^2) / Q(x^2)

where P and Q are low-degree polynomials. This requires only a fixed sequence of additions, multiplications, and one division -- no data-dependent loops or input-dependent interval selection. All other nonlinear functions are derived from R_tanh through mathematical identities:
- **Sigmoid**: S(x) ≈ 1/2 + (1/2) * R_tanh(x/2)
- **Swish**: Swish(x) ≈ x * (1/2 + (1/2) * R_tanh(x/2))
- **GELU**: GELU(x) ≈ (x/2) * (1 + R_tanh(sqrt(2/pi) * (x + 0.044715*x^3)))

### 3. Masking-Based Saturation (Section IV-D)
For large-magnitude inputs outside the approximation interval, the implementation applies saturation (e.g., tanh → ±1, sigmoid → 0/1) using the same masking-based selection rather than branching. The input to the rational approximation is also clamped using masking, avoiding evaluation outside the valid interval.

### 4. Cycle Alignment
All functions within a given setting (3-function or 5-function) are padded to the same cycle count. For ReLU, which is inherently simple (~12 cycles unprotected), this means adding dummy operations to reach the target cycle count.

### Experimental Platform
- **Hardware**: STM32F411E-DISCO board with ARM Cortex-M4 at 84 MHz
- **Software**: C++ (GNU++14), STM32 HAL, compiled with STM32CubeIDE (arm-none-eabi-g++), hardware FPU enabled (-mfloat-abi=hard)
- **Timing measurement**: DWT cycle counter with synchronization barriers (__DSB, __ISB), interrupts disabled during measurement
- **Input ranges**: [-8, 8] with step 0.01 (3-function setting), [-500, 500] with step 1.0 (5-function setting)

## Results & Highlights

### Timing Regularity
- **Three-function setting** (ReLU, sigmoid, tanh over [-8,8]): All three protected implementations execute in exactly **88 cycles** (1.048 us), regardless of input.
- **Five-function setting** (ReLU, sigmoid, tanh, GELU, Swish over [-500,500]): All five protected implementations execute in exactly **108 cycles** (1.286 us), regardless of input.

### Execution-Time Cost (3-function setting)
| Function | Unprotected | Protected | Change |
|----------|------------|-----------|--------|
| ReLU | ~12 cycles (0.143 us) | 88 cycles (1.048 us) | 7.3x slower |
| Sigmoid | ~221 cycles (2.63 us) | 88 cycles (1.048 us) | 2.5x faster |
| tanh | ~403 cycles (4.80 us) | 88 cycles (1.048 us) | 4.6x faster |

The protected sigmoid and tanh are faster because the Padé rational approximation replaces expensive standard-library exponential computations.

### Numerical Accuracy (3-function setting, [-8,8])
| Function | MSE | RMSE | Max abs error |
|----------|-----|------|---------------|
| Sigmoid | 2.93e-12 | 1.71e-6 | 7.51e-6 |
| tanh | 6.14e-10 | 2.48e-5 | 9.59e-5 |

### Numerical Accuracy (5-function setting, [-500,500])
| Function | MSE | RMSE | Max abs error |
|----------|-----|------|---------------|
| Sigmoid | 5.78e-12 | 2.40e-6 | 4.54e-5 |
| tanh | 1.72e-11 | 4.15e-6 | 9.08e-5 |
| GELU | 4.45e-10 | 2.11e-5 | 4.17e-4 |
| Swish | 2.96e-9 | 5.44e-5 | 1.11e-3 |

Swish has the largest error because it multiplies the sigmoid component by x, amplifying small approximation errors near saturation thresholds.

### Desynchronization Countermeasure Attack
The authors demonstrate that a desynchronization-based countermeasure (random delay insertion) can be broken using template-based timing analysis. After approximately 2,000 measurements, the accumulated class scores for the correct activation function clearly separate from competing hypotheses across all three tested functions (ReLU, sigmoid, tanh).

## Critical Assessment

### Strengths

- **Practical and timely**: As neural-network inference moves to edge devices (medical IoT, wearables), timing side-channel protection for activation functions is an under-explored but critical problem.
- **Unified core design**: Sharing a single R_tanh Padé approximation across all nonlinear functions is elegant and reduces implementation/maintenance burden.
- **Faster for nonlinear functions**: The protected sigmoid (88 cycles) and tanh (88 cycles) are substantially faster than their unprotected counterparts (221 and 403 cycles), meaning security comes with a performance benefit rather than a penalty for these functions.
- **Honest security evaluation**: Demonstrating the weakness of an existing countermeasure (desynchronization) with a concrete attack (template-based, ~2000 measurements) strengthens the paper's motivation.
- **Concrete, verifiable metrics**: Exact cycle counts, error tables, and open-source code make results reproducible.
- **High-level language implementation**: The entire methodology is implemented in C++ without inline assembly, increasing portability and auditability.

### Limitations

- **Single architecture validation**: Only ARM Cortex-M4 (STM32F411) is tested. Compiler behavior for branchless selection was verified only for -O1, -O2, -O3; debug builds (-O0) are acknowledged as unreliable, and link-time optimization may break the intended structure.
- **ReLU overhead is significant**: ReLU goes from ~12 cycles to 88 cycles (7.3x slower) -- for networks heavily using ReLU, this could be a meaningful throughput regression.
- **Limited to timing side channels**: Power analysis, electromagnetic leakage, and cache-timing attacks are not addressed. The authors acknowledge this is complementary to, not a replacement for, other countermeasures.
- **No formal verification**: Constant-time property is demonstrated empirically (cycle counting) rather than formally verified. Compiler optimizations or microarchitectural changes could potentially reintroduce timing variation.
- **Special IEEE-754 inputs excluded**: NaN and infinity inputs are outside the considered model.
- **No end-to-end evaluation**: The impact on full neural-network inference (accuracy degradation, total latency) is not measured.

## Relevance

This work is relevant to researchers and engineers working on:
- **Embedded/edge AI security**: Protecting neural-network inference on microcontrollers from side-channel attacks, especially for sensitive applications like medical IoT.
- **Side-channel countermeasures**: Providing a systematic methodology for constant-time implementations that goes beyond cryptographic algorithms into ML inference.
- **Privacy-preserving ML**: Ensuring that ML inference does not leak information about inputs or model parameters through timing channels.
- **Efficient approximation**: The Padé-based rational approximation technique for activation functions is independently useful for resource-constrained deployment, even apart from security concerns.

## Questions for Further Exploration

1. How does the methodology perform on end-to-end neural-network inference tasks (e.g., image classification on CIFAR-10 or keyword spotting)? Does the approximation error accumulate across layers?
2. Can formal verification tools (e.g., ct-verif, FaCT) be applied to prove the constant-time property of the generated implementations, rather than relying on empirical cycle counting?
3. How does the approach generalize to other architectures (RISC-V, ARM Cortex-M0/M33, ESP32) where the compiler may generate different instruction sequences for the same C++ source?
4. What is the interaction between constant-time activation functions and other side-channel countermeasures (masking, shuffling) -- can they be composed effectively?
5. Could the Padé approximation degree be made configurable to trade off between numerical accuracy and instruction count for different application requirements?
