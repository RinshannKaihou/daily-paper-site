CompPow: A Case for Component-level GPU Power Management

Relevance
- 对 ML 系统研究者: 直接相关。随着 LLM 训练和推理规模的持续增长，GPU 功耗已成为数据中心运营的核心瓶颈。CompPow 提供的组件级功耗优化思路可为未来的 ML 框架调度和编译器优化提供新维度。 - 对 GPU 架构设计者: 高度相关。论文的软硬件协同设计建议（组件亲和性提示、操作关键性提示）为下一代 GPU 的功耗管理接口设计提供了明确方向。 - 对数据中心运营商: 中等相关。GPU 内部功耗优化的累积效应可在大规模部署中产生显著的能源和成本节约。
Methodology
- 平台: AMD Instinct MI300X 8-GPU 节点，全连接拓扑。MI300X 采用 chiplet 设计，包含 XCD（加速计算复合芯片，含计算单元和 L2 缓存）、IOD（I/O 芯片，含 LLC/Infinity Cache 和 HBM 接口）、HBM（高带宽内存堆栈）三大组件。 - 功耗测量: 使用 FinGraV 工具获取细粒度组件级功耗分布，提供 XCD/IOD/HBM 三个组件的功耗分解视图。仅使用相对功耗数据（非绝对值）。 - 评估负载: GEMM（来自 LLaMA-405B、LLaMA-70B 等模型的实际矩阵尺寸）和 all-gather 通信集合（不同数据量）。 - 执行场景: 独立执行（standalone）和并发执行（concurrent，GEMM 与 all-gather 同时运行）。 - 优化手段: GPU 级功耗限制（component-agnostic）、XCD 频率限制（component-aware）、CU 分配约束（用于模拟功耗重新分配）。
Results & Highlights
| 场景 | 优化手段 | 能效提升 | 性能影响 | |------|----------|----------|----------| | All-gather（独立） | GPU 级功耗限制 | 3.47% | -0.06% | | All-gather（独立） | XCD 频率限制 | 10.13% | -1.36% | | GEMM/All-gather（并发） | 模拟 IOD->XCD 功耗重分配 | -- | +5% | - 核心发现 1: GEMM 会触发 GPU 功耗上限限制（power-constrained），GPU 时钟被迫降低；all-gather 不会触发功耗限制。 - 核心发现 2: GEMM 的算术强度越低，IOD 功耗占比越高，XCD 功耗相应减少——存在组件间功耗竞争。 - 核心发现 3: ML 训练中高达 47% 的执行时间涉及并发内核执行，GEMM/通信并发是最主要的并发模式。 - 核心发现 4: 并发执行时 IOD 承受更大压力（GEMM 和通信都涉及数据移动），导致 XCD 功耗被动降低。

Critical Assessment

**视角独特**
在集群级功耗优化的大趋势下，转向 GPU 内部组件级功耗分析，填补了重要的研究空白。
**实用性导向**
基于真实硬件平台（MI300X）和真实 ML 负载（LLaMA 系列模型的 GEMM 尺寸），实验结论具有直接的生产环境参考价值。
**分析全面**
覆盖独立执行和并发执行两种场景，并分别给出了不同的优化策略，分析的广度值得肯定。
**软硬件协同设计建议具体可行**
提出的 component affinity hints 和 operation criticality hints 概念清晰，落地路径明确（可通过 profiling、算术强度推断等方式获取）。
**优化手段受限**
由于商业 GPU 未暴露所有组件级功耗控制旋钮（如 IOD/HBMC 频率限制），部分实验依赖模拟（通过限制 CU 分配来间接模拟功耗重分配），而非直接操作。实际效果可能与模拟结果存在差距。
**仅评估单一平台**
实验全部基于 AMD MI300X，未在 NVIDIA 或 Intel GPU 上验证。不同架构的组件划分和功耗管理机制差异较大，结论的通用性有待考察。
