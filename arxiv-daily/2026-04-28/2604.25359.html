The Structured Output Benchmark: A Multi-Source Benchmark for Evaluating Structured Output Quality in Large Language Models

Abhinav Kumar Singh, Harsha Vardhan Khurdula, Yoeven D Khemlani, Vineet Agarwal (JigsawStack, Inc.)

arXiv: 2604.25359v1

Date: 2026-04

https://arxiv.org/abs/2604.25359

一句话总结： 提出了 SOB，一个跨文本、图像、音频三模态的结构化输出评测基准，发现模型 schema 合规率接近完美但 Value Accuracy 最高仅 83.0%（文本）、67.2%（图像）、23.7%（音频），揭示了“产生合法 JSON”与“产生正确 JSON”之间的巨大鸿沟。
TL;DR: SOB is a multi-source benchmark evaluating structured JSON output from LLMs across text, images, and audio. Across 21 models, near-perfect schema compliance masks a large value-correctness gap: best Value Accuracy reaches only 83.0%/67.2%/23.7% on text/images/audio, demonstrating that producing valid JSON is far easier than producing correct JSON.

核心贡献

  • 构建了首个跨三模态（文本、图像、音频）的结构化输出基准 SOB，包含 5,000 条文本记录、209 条图像记录和 115 条音频记录。
  • 提出了 7 项细粒度评估指标（Value Accuracy, Faithfulness, JSON Pass Rate, Path Recall, Structure Coverage, Type Safety, Perfect Response Rate），将 schema 合规性与值正确性解耦评估。
  • 设计了可复现的 benchmark 构建流程：人工撰写 schema 与 ground truth，结合 LLM 交叉校验（Gemini 2.5 Flash）和 Pydantic/JSON Schema 验证。
  • 评估了 21 个前沿及开源模型，发现 schema 合规率普遍 >95%，但 Value Accuracy 显著偏低，模型排名在不同模态间会发生变化。
  • 揭示了结构化输出中的五类典型错误：解析失败、schema 违规、值错误、缺失字段、类型不匹配，其中值错误是最主要的问题。

Key Contributions

  • Introduced SOB, the first multi-source benchmark for structured output spanning text (HotpotQA), images (7 OCR-processed PDF types), and audio (AMI meeting corpus), all using text-normalized inputs for fair cross-model comparison.
  • Defined 7 fine-grained metrics that separately measure schema compliance (JSON Pass Rate, Type Safety, Structure Coverage) and value correctness (Value Accuracy, Faithfulness, Path Recall, Perfect Response Rate).
  • Established a reproducible benchmark construction pipeline combining human authoring, LLM cross-check (Gemini 2.5 Flash), Pydantic validation, and JSON Schema validation.
  • Evaluated 21 frontier and open-weight models (ranging from 8B to 358B parameters), finding that structural metrics mask severe value errors: best models achieve only 83.0% (text), 67.2% (images), and 23.7% (audio) Value Accuracy.
  • Demonstrated that model rankings shift across modalities (e.g., GLM-4.7 leads text, Gemma-4-31B leads images, Gemini-2.5-Flash leads audio), proving that single-source benchmarks are insufficient.

Methodology

Benchmark Construction: Three source-specific data loaders (HotpotQA for text, olmOCR-bench for images, AMI Meeting Corpus for audio) feed into a unified pipeline. Each record contains a context, a natural-language question, a JSON schema the model must follow, and a ground-truth structured output. For text: 25,091 full records, 5,000 held-out test split, requiring multi-hop reasoning across Wikipedia passages. For images: 209 records from OCR-processed PDFs across 7 categories (multi-column layouts, dense tables, headers/footers, scanned historical documents, small-print text, mathematical typesetting, scanned math). For audio: 115 records from AMI meeting transcripts formatted with timestamps and speaker labels (average ~7,300 tokens per record, 3-5 speakers).

Input Normalization: All models receive text-only representations regardless of source modality. Image records are supplied as OCR-rendered markdown transcripts; audio records as speaker-labelled conversation transcripts. This isolates structured-output capability from raw vision/speech processing quality.

Evaluation Pipeline: Each model response is first checked for parse validity and schema compliance (hardening gate). Passing responses are path-flattened to leaf nodes with concrete array indices and compared field-by-field against ground truth to compute 7 metrics. Metrics are aggregated with schema-complexity weighting (easy=1.0, medium=2.0, hard=3.0). An overall coverage factor adjusts for partial model runs.

Models: 21 models evaluated (20 on audio due to Phi-4's 16K context limit), including GPT-5.4, GPT-5, GPT-4.1, GPT-5-Mini, Claude-Sonnet-4.6, Gemini-2.5-Flash, Gemini-3-Flash-Preview, GLM-4.7, Qwen3-235B, Qwen3.5-35B, Qwen3-30B, Gemma-4-31B, Gemma-3-27B, Phi-4, Nemotron-3-Nano-30B, Schematron-8B, IBM-Granite-4.0, DS-R1-Distill-32B, GPT-OSS-20B, Ministral-3-14B, Interfaze-Beta. All evaluated at temperature 0.0 with reasoning disabled where possible.

Results & Highlights

  • Overall Leaderboard: GPT-5.4 ranks first with an adjusted overall score of 0.870, followed by GLM-4.7 (0.861) and Qwen3.5-35B (0.861). The gap between JSON Pass Rate and Value Accuracy is 15-25 percentage points across all models.
  • Text Performance: GLM-4.7 leads at 0.830 Value Accuracy. Structural metrics cluster above 0.95 for most models, but Value Accuracy spans a 13.7-point range (0.693 to 0.830).
  • Image Performance: Gemma-4-31B leads at 0.672 Value Accuracy. The image leader reaches only ~81% of the text ceiling.
  • Audio Performance: Gemini-2.5-Flash leads at 0.237 Value Accuracy, less than one-third of the text ceiling. Perfect Response Rate is near zero (<1%) for all models on audio.
  • Model Size Irrelevance: Phi-4 (14B) scores 0.798 Value Accuracy on text, above GPT-5 (0.795) and GPT-5-Mini (0.779). Schematron-8B (0.754) outperforms GPT-OSS 20B (0.693).
  • Cross-Modality Ranking Shifts: No single model dominates all three modalities. GPT-5.4 ranks 3rd on text but 9th on images. Gemma-4-31B ranks 11th on text but 1st on images.
  • Schema-Constrained Decoding Ablation: Adding explicit schema enforcement changed Value Accuracy by only -0.007 to +0.033, confirming it is not a confound.
  • Error Taxonomy: Value errors (correct structure, wrong content) are the dominant failure mode. Even models with >97% schema compliance have 17-31% incorrect leaf values.

Authors & Credibility

All four authors are affiliated with JigsawStack, Inc., a company that provides structured extraction APIs (Interfaze-Beta is their own model, which performs competitively at overall 0.855). This is an industry research paper with a clear product motivation. The authors are transparent about potential conflicts: they include their own model in the evaluation and report its results alongside competitors without artificial boosting. The benchmark is released publicly with code and data, and the construction methodology is documented in detail. The grounding in real-world extraction tasks (invoices, medical records, meeting transcripts) adds practical credibility. However, the team is small and all from one company, and the paper does not appear to have undergone formal peer review. Ground truth relies on human authoring with LLM cross-check using Gemini models, introducing potential reviewer-model bias.

Critical Assessment

Novelty: Moderately high. The key novelty is combining multi-source evaluation (text, image, audio) with per-field value correctness assessment under a unified framework. Individual components (schema compliance benchmarks, value accuracy evaluation, multi-hop QA) exist separately, but no prior benchmark integrates all three dimensions. The text-normalized input design is a smart methodological choice that isolates structured-output capability from perception.

Rigor: Generally solid. The evaluation covers 21 models with consistent inference settings (temperature 0.0, greedy decoding). The 7-metric framework is well-defined with clear mathematical formulas. The hardening gate prevents inflated semantic scores from unparseable outputs. The schema-complexity weighting is well-justified. However, the image (209) and audio (115) sample sizes are small compared to text (5,000), which limits statistical power for per-modality comparisons. The ~3% estimated residual error rate in ground truth (from 5x100 record sampling) is acknowledged but only measured for text.

Reproducibility: High. The authors release the full benchmark, evaluation pipeline, and model outputs. The construction pipeline is documented with clear stages (load, author, validate, cross-check). All models use well-known APIs or vLLM serving. A worked scoring example (Appendix F) makes metric computation transparent.

Significance: High for practitioners deploying LLMs for structured extraction. The central finding — that schema compliance masks value errors — has immediate implications for production systems that rely on LLM-generated structured data. The cross-modality ranking shifts demonstrate that single-domain benchmarks are insufficient for model selection. For researchers, the benchmark provides a standardized testbed for improving structured output quality beyond syntactic validity.

Red Flags: (1) All authors from the same company that builds a competing product, and their own model (Interfaze-Beta) is included in evaluations — though they position it transparently and it does not artificially top the leaderboard. (2) Ground truth verification uses Gemini 2.5 Flash, which could introduce systematic bias favoring Gemini-family models (Gemini-2.5-Flash leads audio by a wide margin). (3) The audio source uses gold human transcripts, bypassing real ASR errors — the reported audio Value Accuracy of 23.7% is therefore an upper bound and real-world performance would be worse. (4) No statistical significance testing (confidence intervals, bootstrap) is reported for model comparisons, making it unclear whether small differences are meaningful.

Strengths

  • The text-normalized input design is a thoughtful methodological choice that cleanly isolates structured-output capability from perception quality.
  • The 7-metric framework captures a comprehensive picture: structural compliance, value correctness, soft partial-credit, and strict exact match.
  • The hardening gate and coverage factor provide robust safeguards against inflated scores from unparseable outputs or partial evaluation runs.
  • Multi-source coverage (text, image, audio) reveals capability gaps invisible to single-domain benchmarks, with cross-modality ranking shifts.
  • Practical relevance: the benchmark directly tests what production structured-extraction systems need, making results immediately actionable.
  • Extensive appendices with worked examples, error taxonomy, full metric definitions, and construction pipeline details.
  • Transparent about limitations including ground truth quality, sample size imbalance, and the text-only input design choice.

Limitations

  • Image (n=209) and audio (n=115) sample sizes are small, limiting statistical power and generalizability of per-modality rankings.
  • Ground truth creation uses Gemini 2.5 Flash as an automated reviewer, potentially introducing systematic bias favoring Gemini-family models.
  • The text-normalized input design, while methodologically clean, does not reflect end-to-end deployment where models must handle raw images and audio directly.
  • Audio source uses gold human transcripts (no ASR errors) — real-world audio extraction performance would be considerably worse than the reported 23.7% ceiling.
  • No confidence intervals or statistical significance testing for model comparisons.
  • Schemas reflect the authors' design conventions; real-world schemas may differ in complexity and style.
  • All authors are from the same company, and the company's own model is evaluated — though this is disclosed transparently.

Relevance

该工作直接关心中文 LLM 生态中的结构化输出能力评估（GLM-4.7 在多项指标中表现突出），对于关注 LLM 实际部署、JSON 生成可靠性、以及多模态信息提取的研究者和工程师具有重要参考价值。其揭示的 “schema 合规但值错误”问题在生产环境中具有普遍性，值得在中文场景下进行类似的系统性评测。

Questions for Further Exploration

  1. How well do these findings generalize to non-English languages and non-Latin scripts? The benchmark is English-only; structured extraction from Chinese documents, for example, may present different challenges.
  2. Would end-to-end evaluation with native image/audio inputs (bypassing OCR/ASR preprocessing) change the model rankings, and by how much?
  3. How do reasoning-enabled models perform when allowed to use chain-of-thought for structured extraction, given that the paper deliberately disables reasoning?
  4. Can fine-tuning specifically for value accuracy (rather than schema compliance) close the 15-25 percentage point gap, and what training data strategies work best?
  5. How does the benchmark's schema complexity distribution compare to real-world production schemas, and are there schema patterns (e.g., conditional fields, oneOf/anyOf) that SOB does not yet cover?
  6. What is the relationship between context length and Value Accuracy degradation? The audio source's ~7,300 tokens average context suggests long-context handling is a key bottleneck, but a controlled study would be valuable.
