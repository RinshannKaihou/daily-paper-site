> **一句话总结：** 本文提出一种由专家角色分工的多智能体闭环自动研究框架，通过在三个真实训练配方基准（Parameter Golf、NanoChat-D12、CIFAR-10 Airbench96）上共1,197次无人干预实验，分别取得 0.81%、38.7% 和 4.59% 的验证指标改进，核心机制是将每次实验的失败信号（崩溃、超时、精度未达标）通过共享谱系记忆传递给后续智能体，从而将边界失败转化为程序级代码改动。

> **TL;DR:** This paper introduces a closed-loop multi-agent system where specialist LLM agents (powered by Claude Opus 4.7) iteratively edit ML training recipes, submit them to an external evaluator, and feed measured outcomes—including crashes, budget overruns, and accuracy-gate failures—forward through a shared "lineage" log. Over 1,197 human-free headline trials across three compute-budgeted benchmarks, the loop improves Parameter Golf validation loss by 0.81%, raises NanoChat-D12 CORE score by 38.7%, and cuts CIFAR-10 training wallclock by 4.59%. The key finding is that lineage feedback (not just role partitioning) is the essential mechanism: ablating it collapses valid improvements from 16 to 3 in a matched 200-trial window. The system does not invent new architectures; it combines and refines known techniques guided by empirical failure signals.

## 核心贡献

- **闭环自动研究范式**：将机器学习研究定义为"提案→可执行代码编辑→外部评估→反馈→下一次提案"的可审计轨迹，而非生成报告或单一检查点，使研究过程本身成为可复现的工件。
- **专家角色分工与共享谱系记忆**：不同专家智能体（如架构、优化器、数据、系统等角色）负责不同的配方搜索面，通过共享谱系日志（包含历史分数、代码差异、崩溃摘要）跨角色传递测量证据，有效防止重复探索和坏方向堆积。
- **失败信号作为研究信号**：系统将崩溃、预算超支、规模失败和精度门未达标等非成功结果纳入谱系，允许后续智能体将边界失败转化为有效的程序级编辑（例如将一次精度近失直接导向最终 CIFAR-10 warmup 修复方案）。
- **三环境验证**：在 Parameter Golf（大小与预算约束）、NanoChat-D12（固定预算预训练）和 CIFAR-10 Airbench96（精度门控速度优化）三个互补约束环境中验证了同一闭环框架的有效性。
- **可复现公开轨迹**：全部试验日志、代码差异和评估记录不依赖私有模型内部状态，可公开发布用于审计和后续分析（GitHub: https://github.com/cxcscmu/Auto-Research-Recipes）。

## Key Contributions

- **Closed-loop auto research formulation**: Reframes ML research as a measured empirical loop whose unit is a submitted trial (hypothesis + code edit + evaluator score + feedback label), producing an inspectable trajectory rather than a paper or checkpoint.
- **Role-partitioned specialist swarm with shared lineage**: Task-specific specialist agents (5–10 per environment, covering architecture, optimization, data, schedule, systems, regularization, etc.) each own a distinct recipe surface. A shared lineage log carries measured scores, diff summaries, crash details, and status labels across role boundaries, enabling cross-role idea transfer confirmed by proposal entropy analysis.
- **Failure-to-edit feedback translation**: Crashes, size-blocks, budget overruns, and accuracy-gate near-misses are retained in lineage as actionable signals rather than discarded, allowing the loop to turn boundary violations into valid follow-up edits.
- **Multi-environment empirical validation**: Three benchmarks with complementary constraint regimes demonstrate that the same loop design generalizes across size-constrained LM training, fixed-budget pretraining, and accuracy-gated speed optimization.
- **Auditable, releasable trace**: All trial records are free of private model internals, enabling public release for reproducibility and human-versus-agent comparisons.

## Methodology

### System Architecture

The system is a closed empirical loop instantiated on top of the Anthropic Claude SDK. Each iteration consists of four stages: (1) a specialist agent reads the current lineage state from a shared blackboard, proposes a hypothesis, and implements it as an executable code edit; (2) the edited recipe is submitted to an external evaluator (not under agent control) that runs the full training pipeline; (3) the evaluator returns a score, status label, timing data, and failure details; (4) these results are appended to the lineage log that seeds the next agent session.

### Specialist Roles and Lineage Memory

Specialist roles partition the editable recipe surface. Parameter Golf uses 10 specialists (architecture, optimization, quantization, regularization, curriculum, loss, evaluation, tokenizer, test-time training, meta search). NanoChat-D12 uses 5 (architecture, optimization, data, schedule, systems). CIFAR-10 uses 5 (architecture, optimization, augmentation, loss, regularization). Each specialist receives a system prompt assembled from: (a) static knowledge files, (b) a global rules protocol enforcing hard limits and the per-session workflow, and (c) a domain preamble defining scope, non-scope, and the edit radius historically associated with valid improvements.

The per-iteration user message renders a compact lineage slice: the current best row, the specialist's recent rows, adjacent-specialist rows, and a delta-saturation signal when recent own-role improvements have been negligible. This keeps the frontier visible and failed directions apparent without replaying the full transcript.

### Environments and Feedback Signals

- **Parameter Golf**: Minimizes FineWeb validation bits-per-byte under a 16 MB artifact cap and 10-minute GPU budget. Feedback includes score, status, byte count, and per-phase timing.
- **NanoChat-D12**: Maximizes CORE score from a fixed d12 nanochat pretraining run. Main feedback is wallclock headroom under the fixed budget (faster code recovers time for more tokens).
- **CIFAR-10 Airbench96**: Minimizes shell-measured wallclock training time, but only when mean CIFAR-10 accuracy meets a strict 0.96 gate. The accuracy-gate near-miss returns timing plus accuracy deficit, making a failing trial usable as a directional signal.

### Anti-Anchoring and Crash Feedback

A per-role banlist of patterns that previous sessions tried and failed is rendered alongside the lineage slice to reduce re-exploration of known dead ends. Crash handling surfaces the deepest exception line and the deepest training-script frame, giving the next specialist an explicit failure mode rather than a generic crash status.

### Infrastructure

All specialists run on Claude Opus 4.7 with 8,000-token extended thinking budget and a 200-turn cap per session. Bash is sandboxed (read-anywhere, write-only-to-cwd). `Write` is deliberately not in the allowed-tools list to prevent sidecar file inflation toward the size cap. Parameter Golf and NanoChat-D12 trials run on fresh eight-H100 workers; CIFAR-10 runs on a long-lived GPU worker to preserve pre-warmed CUDA context. Total estimated active accelerator time is approximately 4,000 H100-hours for 1,500 Parameter Golf trials and 2,400 H100-hours for 200 NanoChat-D12 trials.

### Throughput and Parallel Submission

Parallel throughput is modeled as τ_e = τ_run + τ_eval + τ_queue + τ_log. The ten-specialist role swarm achieves 18.15 trials/hour versus 2.26 for a single generalist, with a parallel efficiency η ≈ 0.80 relative to the ideal 10× speedup (sharing GPU cluster, blackboard filelock, and queue overhead accounts for the gap).

## Results & Highlights

### Headline Results (Table 1)

| Environment | Metric | Start | Best | Relative Improvement | Valid Trials |
|---|---|---|---|---|---|
| Parameter Golf | val_bpb (lower better) | 1.0810 | 1.0722 | **-0.81%** | 36 of 900 |
| NanoChat-D12 | CORE (higher better) | 0.1618 | 0.2244 | **+38.7%** | 5 of 200 |
| CIFAR-10 Airbench96 | train_s (lower better) | 26.356 s | 25.146 s | **-4.59%** | 4 of 97 |

### Ablation: Role Organization and Lineage (Figure 3, Table 3)

In a matched 200-trial Parameter Golf window:
- **Role swarm + lineage**: 16 valid drops, reaches 1.073142 bpb.
- **No-lineage ablation**: 3 valid drops, reaches 1.077413 bpb; then runs 125 consecutive trials without a new improvement. The no-lineage tree collapses to 3 active parent heads versus 15 with lineage.
- **Generic 10-agent control** (shared lineage, no role partitioning): 10 valid drops, 41.1 effective proposal clusters (vs. 134.8 for role swarm); concentrates on fewer ideas.
- **Single generalist**: 14 valid drops but stream concentrated (largest cluster consumes 35 of 200 submissions around polar-coefficient edits).

Role partitioning increases proposal entropy (134.8 effective clusters vs. 41.1 for generic multi-agent), while lineage is the mechanism that carries measured evidence across sessions and prevents repeated dead ends.

### Representative Non-Trivial Edits (Table 2)

- **Parameter Golf**: Recurrent residual scaling; separate RoPE/NoPE query gains; per-head data-dependent attention-output gate.
- **NanoChat-D12**: SSSL-to-L attention path rewrite; all 12 masked SDPA layers moved to Flash SDPA (systems-level diagnosis from a single runtime trace, amplified by later proposals spending recovered wallclock on more tokens).
- **CIFAR-10**: Residual-preserved ConvGroup depth reductions; wider-shallower blocks passed under the accuracy gate.

### Scope of Non-Trivial Edits

13.1% of 1,197 headline trials (157) were classified as architecture-domain or systems-domain submissions—a conservative lower bound on non-scalar edits, since loss and schedule specialists also rewrote executable structures.

## Critical Assessment

**Novelty**: The framing of auto research as a closed empirical trajectory (rather than a one-shot generated output) is a meaningful conceptual contribution. The specific combination of role-partitioned specialists + shared lineage with failure signal propagation is not identical to prior work (AlphaEvolve, AI Scientist, MLGym). However, the individual components—evaluator-driven search, multi-agent role decomposition, shared memory logs—are each present in the literature. The novelty is primarily in the integration and the empirical demonstration at scale on real compute-budgeted training benchmarks.

**Rigor**: The paper is unusually rigorous for an agentic AI paper. Ablations are clean and directly address the paper's central claims (lineage vs. no lineage; role swarm vs. generic multi-agent vs. single generalist). The evaluator is external and cannot be tampered with by agents; anti-reward-hacking is enforced at the harness level (CIFAR timing cannot be self-reported; Parameter Golf byte counts are checked). The trace records all 1,197 trials including failures, not just successes. Starting-point calibration is handled carefully to avoid stale denominators.

**Reproducibility**: High within stated conditions. The code and trace artifacts are promised for public release. The frozen run protocol (harness, prompt templates, specialist taxonomy fixed before launch; no human intervention during search) is well-specified. The main reproducibility caveat is significant compute cost (~thousands of H100-hours) and dependency on Claude Opus 4.7 API, which may drift.

**Significance**: The NanoChat-D12 38.7% CORE improvement is impressive but partly reflects that the starting recipe had a clear systems bottleneck (attention-path inefficiency) that a systems-aware agent could diagnose from a runtime trace—a situation that may not generalize to more mature recipes. The Parameter Golf 0.81% and CIFAR 4.59% improvements are modest in absolute terms, though they are measured against competitive public starting points.

**Red Flags**:
- The loop explicitly does not produce paradigm-level architectural inventions (the authors acknowledge this as a scope boundary). All improvements are compositional—combining and refining known techniques.
- Valid improvement rate is low (13.1% of trials produce a score improvement; the "keep" rate is much lower: 36 of 900 for Parameter Golf). The loop is expensive per useful signal.
- The 38.7% NanoChat-D12 gain is large, but only 5 trials out of 200 were valid improvements. The lineage mechanism is critical, but so is having the right systems specialist diagnose the correct bottleneck early (trial 007 for the attention-path rewrite). Sensitivity to early lucky discoveries is not fully characterized.
- All reported runs use Claude Opus 4.7 exclusively. Cross-model generalization is not tested.

---

## Strengths

- Strong ablation design that directly isolates the contribution of lineage feedback versus role partitioning versus both.
- Empirical object is a full measured trajectory (proposals, code diffs, scores, failures, timing) rather than a generated narrative—significantly more auditable than most auto-research papers.
- Failure feedback is treated as a first-class research signal, not discarded, which is both methodologically correct and practically valuable.
- Anti-reward-hacking measures are well-designed and enforced externally.
- Compute accounting is transparent and conservative (upper bounds, not claimed actuals).
- The proposal entropy analysis (TF-IDF clustering of hypothesis text, effective cluster count via Shannon entropy) provides a principled way to measure idea diversity beyond just score outcomes.

---

## Limitations

- The system combines and refines existing techniques; it does not synthesize fundamentally new ideas (e.g., no novel attention mechanism was discovered, only an existing kernel path was swapped).
- Compute cost is substantial and scales poorly: ~4,000 H100-hours for 1,500 Parameter Golf trials. This is feasible only for well-resourced groups.
- Valid improvement rates are low (often under 5%). Much of the compute is spent on failures that contribute to lineage but do not advance the recipe.
- The benchmarks (Parameter Golf, NanoChat-D12, CIFAR-10 Airbench96) are compute-efficient by design, which is what enables the loop—but this limits generalizability to full-scale training runs (GPT-4 scale, for instance).
- Role taxonomy and specialist preambles are manually designed before each run, requiring human expertise to construct effective partitions. Automation of this meta-level design is left for future work.
- The study covers only language model and vision classifier training recipes; extension to reinforcement learning, diffusion, or multimodal recipes is untested.
- All runs use Claude Opus 4.7; it is unclear how much the results depend on this specific model's coding and reasoning capabilities.

## Questions for Further Exploration

1. **Generalization beyond compute-budgeted recipes**: Can the same loop apply to full-scale pretraining runs (hundreds of GPU-hours per trial)? What changes in the feedback and lineage design are needed when per-trial cost is much higher?
2. **Paradigm-level invention**: The paper explicitly notes the loop does not produce novel architectures. What would be needed—in terms of agents, tools, or evaluation environments—for the loop to propose and validate structurally novel designs rather than compositional refinements?
3. **Role taxonomy automation**: The specialist roles and preambles are hand-designed. Could the loop itself propose and refine its own role decomposition? What would a meta-loop over role design look like?
4. **Cross-model robustness**: All runs use Claude Opus 4.7. How much of the improvement is attributable to the loop design versus the specific reasoning and coding capabilities of this model? Would the same framework yield comparable results with weaker models?
5. **Transfer across environments**: The lineage is task-specific. Could a cross-task lineage—where solutions from NanoChat-D12 inform Parameter Golf proposals—accelerate search? What representation would enable this?
6. **Diminishing returns and horizon length**: The loop was run for up to 900 trials. Does performance plateau, and if so, what triggers plateau—exhaustion of the compositional search space, or lack of paradigm-level moves?
7. **Human-versus-agent comparison on identical traces**: The paper mentions releasing traces to enable this. What does the comparison reveal about the kinds of edits humans would prioritize that agents miss, and vice versa?
