Systems · KV Cache Reuse

Adaptive KV Cache Reuse for Fast Long-Context LLM Serving

Fei Li · Song Liu · Yan Liu · Jinhua Cui · Shiqiang Nie · Jinyu Wang · Weiguo Wu
3.72–4.86× TTFT speedup with 94.8% quality retention

Frequency-domain analysis identifies semantically critical KV pairs for selective recomputation, combining sparse async transfer with hardware-aware adaptive scheduling to accelerate long-context LLM serving across GPU, SSD, and HDD cache tiers.

TL;DR — CacheTune applies rFFT along the sequence dimension of KV tensors, uses low-frequency energy scores to select the most semantically important tokens for recomputation, and reuses the rest from external cache. A roofline model derives the optimal recomputation ratio per hardware, refined by golden-section search. Sparse transfer + dual-stream async overlap + deferred RoPE recovery eliminate serial overhead. Evaluated on Mistral-7B, Llama3-8B, and Qwen2.5-32B across five tasks on A100 and RTX 4090.

01
Frequency-Domain Selective Recomputation
rFFT along sequence dim → retain lowest α=0.5 frequency fraction → inverse rFFT → per-token score si = ½(|K̃i|² + |Ṽi|²) → TopK(r·N) selects recomputation candidates. Low-frequency KV pairs encode global cross-chunk dependencies.
02
Sparse Offloading Pipeline
Transfer Stream prefetches non-selected KV for next layer. Recompute Stream regenerates selected tokens with correct positions. Default Stream runs current-layer attention+FFN. Layers classified as collection / check / fusion.
03
Deferred RoPE Recovery
Stores raw pre-RoPE Keys in cache pool. Applies RoPE with true global positions during online inference — eliminates positional encoding misalignment in non-prefix reuse. Zero serial cost (folded into Recompute Stream).

Strengths
  • Novel frequency-domain perspective on KV importance — beyond attention-weight heuristics
  • Principled roofline model with closed-form optimal ratio
  • Complete end-to-end system: offline analysis → online recomputation → adaptive scheduling
  • Strong results across 3 models, 5 tasks, 7 baselines, 2 hardware platforms
  • Graceful degradation on slow HDD via adaptive tuning
  • Deferred RoPE elegantly solves positional misalignment at zero serial cost
Limitations
  • rmin=15% quality bound is empirical — may not generalize to all architectures
  • No evaluation beyond ~32K tokens; 128K+ scalability untested
  • Roofline model assumes steady-state pipelining; startup bubbles only partially captured
  • Per-deployment calibration adds ~3–4 min operational overhead
  • Frequency indices tied to chunk composition — RAG document changes may require regeneration
  • No comparison with LMCache or Mooncake in non-prefix reuse setting
Open Questions
  1. Does low-frequency dominance hold at 128K+ tokens, or do mid-frequency components become important?
  2. Can offline frequency analysis be incremental — updating indices when cached chunks are re-composed without full recomputation?
  3. How do low-frequency KV pairs interact with quantization, pruning, or distillation-based cache compression?
  4. Does the 94.8% quality figure hold for code generation, math reasoning, or agentic workflows where token-level precision matters more?
  5. Could the roofline model incorporate variable chunk sizes for heterogeneous document workloads?
