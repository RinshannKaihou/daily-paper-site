> **一句话总结：** AutoSG 用 Claude Opus 4.6 通过RAG检索OpenAlex/arXiv文献→两阶段代码生成→单步结构保持自精炼→无实例Elo LLM-as-Judge评估的端到端流水线，从自然语言任务描述自动生成昂贵优化求解器，在 BBOB(20D)、CEC2013(905–1000D) 和 Bayesmark HPT(2–8D) 三类基准上取得 24/0/0、15/0/0 等压倒性胜率，平均排名均列前三。

> **TL;DR:** AutoSG is a fully automated LLM-driven pipeline (backbone: Claude Opus 4.6) that turns a natural-language task prompt into an executable expensive-optimization solver via three modules — RAG-grounded two-stage code generation from OpenAlex/arXiv literature, a one-step structure-preserving self-refinement operator, and an instance-free Elo-based LLM-as-a-Judge evaluator. Across the BBOB (24 problems, 20D, 300 FEs), CEC2013 (15 problems, 905–1000D, 11×D FEs) and Bayesmark HPT (8 datasets × 5 ML models, 2–8D, 30 FEs) suites over 25 independent runs each, the three independently generated AutoSG solvers take the top-3 average ranks, beating CMA-ES, TuRBO, HEBO, LLaMEA-BO and LSEO-S3 with Wilcoxon win/tie/loss counts such as 24/0/0 (vs CMA-ES on BBOB) and 15/0/0 (vs SADE-AMSS on CEC2013) at p<0.05. The LLM-as-a-Judge mechanism identifies the top solver using only 63.64% of exhaustive pairwise matches.

## 核心贡献

- 提出 AutoSG，首个完全从自然语言任务描述 `Ptask=⟨Tdesc, d, B, X⟩` 端到端生成可执行昂贵优化求解器的工作流，输出已按维度 d 与预算 B 实例化。
- 设计 RAG 求解器生成引擎：LLM 改写 K=8 个查询 → 同时检索 OpenAlex(M_OA=30) 与 arXiv(M_ax=10) → 去重排序留 N_OA=25、N_ax=15 → 合并 N=40 候选 → LLM 重排选 Top-1 论文 → 解析全文 PDF 作为 D_pdf → 两阶段（先忠实复现 S_pre，再按伪代码/公式/超参打磨得 S_init），消除幻觉。
- 设计"结构安全"的单步自精炼算子：先让 LLM 逆向工程出 S_init 的语义描述 D_init，再用"learn-and-improve"提示 Prefine 生成 C=10 个候选，严格最小化对 Pimp 标注的关键组件的修改，避免破坏局部最优结构。
- 提出无实例 Elo LLM-as-a-Judge 评估机制：动态 K 值 `K_i = 32×(1 + 50/(m_i+10))`、两阶段智能配对（T_init=3、T_total=6）、置信区间剪枝（|ΔR|>Δmax=400 且 95%CI 不重叠则跳过），仅需 63.64% 的穷举配对即可定位最优。

## Key Contributions

- AutoSG, the first end-to-end LLM-driven workflow mapping a natural-language task prompt `Ptask = ⟨Tdesc, d, B, X⟩` directly into an executable, dimension- and budget-instantiated solver for expensive optimization.
- A RAG-based solver generation engine that rewrites the task into K=8 search queries, retrieves from OpenAlex (M_OA=30/query) and arXiv (M_ax=10/query), deduplicates to N_OA=25 and N_ax=15, merges to N=40, LLM-reranks to a Top-1 paper, parses its full-text PDF as D_pdf, and runs a two-stage generation (faithful reproduction S_pre → polishing against pseudo-code/formulas/hyper-parameters to S_init) — explicitly to eliminate factual hallucinations.
- A structurally safe one-step self-refinement operator that reverse-engineers S_init into a semantic description D_init, then generates C=10 candidate refinements under a "learn-and-improve" prompt that strictly minimises modifications to Pimp-annotated critical components, avoiding the catastrophic destruction of locally optimal heuristics seen in multi-step/crossover-mutation variants.
- An instance-free Elo-based LLM-as-a-Judge evaluation mechanism with dynamic K-scaling, a two-phase smart-pairing tournament (T_init=3, T_total=6, Δmax=400, λ=10), and 95%-CI-based pruning, which identifies the top solver using only 63.64% of exhaustive pairwise matches.

## Methodology

AutoSG takes `P_task = ⟨T_desc, d, B, X⟩` (task description, dimensionality, FE budget, search-space bounds) and applies three modules (backbone LLM throughout: **Claude Opus 4.6**).

**Module 1 — RAG-based solver generation (§3.2).** Two paradigms are contrasted: (i) *ungrounded* `S̃ ← LLM(P_task ⊕ Q_meta)` — shown to suffer Source Fabrication and Implementation Degradation; (ii) the adopted *grounded* pipeline:
- *Retrieval:* LLM rewrites the task into K=8 domain queries; OpenAlex (M_OA=30/query) and arXiv (M_ax=10/query) APIs return up to K×(M_OA+M_ax) papers; pools are deduplicated and sorted by year (latest N_OA=25, N_ax=15); the N=40 merged candidates' metadata (titles/citations/venues/abstracts) are LLM-reranked to a single Top-1 paper whose full-text PDF is parsed as `D_pdf`.
- *Two-stage generation:* Stage 1 — LLM acts as a "faithful reproducer" using system prompt P_G1, an extensible template T_code (instantiates both Bayesian-optimization and surrogate-assisted solvers), and an instruction P_imp that forces explicit annotation of critical solver components → `S_pre`. Stage 2 — a polishing prompt P_G2 cross-references S_pre with the paper's pseudo-code, formulas, and hyper-parameters → SOTA-level `S_init`.

**Module 2 — One-step self-refinement (§3.3).** Because S_init is already near a local optimum, multi-step evolution is argued to be redundant and risks catastrophic forgetting. The operator: (i) reverse-engineers S_init with P_reverse into a semantic description D_init; (ii) under P_refine (a "learn-and-improve" prompt that strictly minimises changes to Pimp-annotated critical components) adapts the implementation to P_task, generating C=10 candidate solvers `{S_L1,…,S_LC}` for evaluation.

**Module 3 — Elo-based LLM-as-a-Judge evaluation (§3.4).** Candidate pool `S_candidate = {S_init, S_L1,…,S_LC}` (|S_candidate|=11). Each solver starts at R_i^(0)=1500 with deviation r_d^(0)=350 (approximate 95% CI [R_i ± 1.96·r_di]); r_d decays as max(30, 0.95·r_d) per match. Winner/loser updates: ΔR_w=K_w(1−E_{w,l}), ΔR_l=−K_l·E_{l,w}, with E_{w,l}=1/(1+10^{(R_l−R_w)/400}) and dynamic K_i=32×(1+50/(m_i+10)). Two-phase smart pairing: Phase 1 ensures ≥T_init=3 matches per solver prioritising fewest combined matches; Phase 2 extends to T_total=6 with priority `prio_ij = 1000/(|R_i−R_j|+1) + λ·(T_total − (m_i+m_j)/2)`, λ=10. Redundant matchups are pruned when |R_i−R_j|>Δmax=400 and 95% CIs are disjoint. The candidate with the highest final Elo rating is selected as S*.

**Evaluation suites (§4.1).** (1) BBOB (24 continuous problems, D=20, 300 FEs, 25 runs) vs CMA-ES, TuRBO, HEBO, LLaMEA-BO, TREvol; (2) CEC2013 large-scale (15 problems, D=905–1000, 11×D FEs, 25 runs) vs L2SMEA, SAEA-RFS, SADE-AMSS, LSEO-S3, LLaMEA-BO; (3) Bayesmark HPT (8 datasets × 5 ML models, D=2–8, 30 FEs, 5 runs) vs logEI, CMA-ES, HEBO, TuRBO, LLaMEA-BO. Statistical testing: Wilcoxon rank-sum test at α=0.05, reported as +/≈/− win/tie/loss. AutoSG is independently run 3 times per suite to verify stability.

## Results & Highlights

- **BBOB (Table 1):** the three AutoSG solvers S*_BBOB,1/2/3 take average ranks **1.71 / 2.71 / 3.50** — the top three positions — against CMA-ES (6.88), TuRBO (5.50), HEBO (3.67), LLaMEA-BO/TREvol (4.04). Wilcoxon vs CMA-ES: **24/0/0** (win on all 24 functions); vs TuRBO: 24/0/0; vs HEBO: 21/1/2.
- **CEC2013 large-scale (Table 2):** S*_CEC,1/2/3 take average ranks **2.40 / 2.73 / 2.73** (top three) vs L2SMEA (5.27), SAEA-RFS (5.80), SADE-AMSS (5.33), LSEO-S3 (3.73). Wilcoxon vs SADE-AMSS: **15/0/0**; vs SAEA-RFS: 13/0/2; vs L2SMEA: 12/0/3.
- **Bayesmark HPT (§4.2.2, Fig. 2):** on the *private (unseen) datasets*, the three AutoSG solvers S*_HPT,1/2/3 secure the top three positions in average regret; S*_HPT,1 achieves the best overall performance. On public datasets most solvers are comparable but S*_HPT,1 still leads. (Regret curves in Fig. 2, with private-dataset Avg Regret values plotted in the ~0.30–0.55 range.)
- **LLM-as-a-Judge efficiency (Fig. 3):** the Elo mechanism reliably identifies the top-performing S* using only **63.64% of exhaustive pairwise matches** (ground truths averaged over 5 executions).
- **Ablation — one-step refinement (Table 4, vs S*_CEC,2):** AutoSG beats `AutoSG w/o SelfRefine` (9/2/4), `AutoSG w/ C-M` (crossover-mutation from LLaMEA-BO, 13/2/0 — i.e. C-M is worse), `AutoSG w/ 2-step` (1/13/1), and `AutoSG w/ 10-step` (2/13/0) — confirming that extra iterations on RAG-grounded SOTA solvers are computationally redundant and can degrade performance.
- **Implementation faithfulness (§4.2.3):** cross-verification of S_init against the BBOB public GitHub repo (and against D_pdf for the other two tasks) using **Gemini 3 Pro with chain-of-thought** yields a **95% logical consistency rate** across all three tasks.

## Critical Assessment

- **The "instance-free" evaluation is only indirectly validated:** the Elo LLM-as-a-Judge picks S* without running candidates on training instances, but Figure 3 reports the *true* ranking of the selected S* within S_candidate averaged over only 5 ground-truth executions "due to computational costs" — a small sample for judging judge-quality, and the paper does not report a correlation coefficient (e.g., Kendall τ) between Elo ranking and true ranking across the full candidate pool.
- **Self-selection bias in the judge:** Claude Opus 4.6 is used both to *generate* solvers (via RAG) and to *judge* them via LLM-as-a-Judge — a known confound (the paper cites Boubdir et al. on LLM position/randomness bias but does not test whether a different family judging a different family's outputs changes the ranking).
- **The headline BBOB win rests on RAG retrieving the BBOB SOTA:** since only "the retrieved literature for the BBOB suite provides a public repository," S_init is essentially a faithful reproduction of an existing strong BBOB solver — so the 24/0/0 vs CMA-ES partly measures "did the retriever find a better-than-CMA-ES paper" rather than "did AutoSG invent a better solver." The 95% consistency-vs-GitHub check confirms reproduction fidelity, not novel design.
- **No comparison on truly novel / literature-sparse tasks:** all three suites (BBOB, CEC2013, Bayesmark) are heavily represented in the literature, so the RAG engine is operating in its comfort zone; the paper does not report a task where retrieval fails.
- **Cost accounting missing:** the paper motivates the work by the 75-hour single-evaluation antenna-design example, but does not report AutoSG's own LLM API cost (K=8 queries × reranking × PDF parsing × 2-stage generation × 10 refinements × Elo tournament on Claude Opus 4.6), which is the relevant comparison.

## Strengths

- Clean, well-motivated three-module design that directly maps each of the three stated failure modes (hallucination, structural destruction, expensive evaluation) to a specific mechanism (RAG, one-step refinement, instance-free Elo).
- Strong empirical breadth: three suites spanning 20D (BBOB), 905–1000D (CEC2013), and 2–8D real-world HPT, with 25 runs and Wilcoxon tests.
- Stability check via three independent regenerations per suite is a good practice rarely seen in LLM-for-code papers.
- The Elo pruning design (95%-CI disjoint + Δmax) gives a concrete 63.64% match-count saving that is directly actionable.

## Limitations

- Generation and judging share the same LLM family (Claude Opus 4.6), creating a self-judge bias risk that is not controlled for.
- All three benchmarks are well-covered in the literature; no evaluation on retrieval-sparse or genuinely novel tasks.
- No LLM API cost / latency reporting, despite the motivation being expensive evaluation.
- Average-rank and win/tie/loss reporting obscures per-function magnitudes (e.g., by how much on F8 BBOB).
- No public code repository URL in the paper body; reproducibility depends on Appendix C artifacts.

## Questions for Further Exploration

- What is the Kendall τ / Spearman correlation between the Elo LLM-as-a-Judge ranking and the true performance ranking across the *full* candidate pool (not just the selected S*), and how does it degrade when the judge LLM is from a different family than the generator?
- How does AutoSG perform on a retrieval-sparse or genuinely novel expensive-optimization task where OpenAlex/arXiv return no high-quality paper?
- What is the total LLM API cost (USD and wall-clock) of one AutoSG run, and how does it compare to the 75-hour antenna-design evaluation it aims to replace?
- Does the one-step refinement result generalise, or is it an artefact of S_init already being a near-SOTA reproduction (i.e., would multi-step help more when retrieval is weak)?
- Can the instance-free Elo mechanism be transferred to other automated-research loops (e.g., auto-generated ML training pipelines) where ground-truth evaluation is similarly expensive?
