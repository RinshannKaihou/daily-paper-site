> **一句话总结：** Fuzzy PyTorch 是一个集成到 PyTorch 中的浮点数值变异性评估框架，通过 PRISM 库实现随机舍入和上下舍入两种模式，相比 Verrou 实现 5×–60× 的加速，可扩展到 3.41 亿参数的深度学习模型。

> **TL;DR:** Fuzzy PyTorch integrates stochastic arithmetic into PyTorch via the PRISM library, supporting Stochastic Rounding (SR) and a novel Up-Down Rounding (UD) mode, achieving 5–60× speedups over the state-of-the-art Verrou tool while maintaining comparable numerical variability assessment accuracy across models from 1M to 341M parameters.

## 核心贡献

- 提出 **PRISM 库**：一个 C++ 库，实现了高效的随机舍入（SR）和全新的上下舍入（UD）模式，利用 Highway 库实现向量化指令加速，支持多线程独立随机数生成器
- 将随机算术**无缝集成到 PyTorch** 中：仅需修改 PyTorch 代码库中的一行代码，即可对深度学习模型的浮点运算进行全面的数值变异性分析
- 与 Verrou 进行**全面对比评估**：在 MNIST 数字分类、FastSurfer 脑部分割（CNN）和 WavLM 帕金森病语音分类（Transformer）三个任务上验证了框架的有效性
- 提出**上下舍入（UD）模式**：对已舍入结果以等概率向上或向下扰动，虽然不保留精确浮点运算，但在大规模代码上产生与 SR 类似的结果且通常更快

## Key Contributions

- **PRISM library**: A C++ library implementing fast probabilistic rounding (SR and UD modes) leveraging the Highway library for vectorized SIMD instructions, with per-thread random generators for lock-free multithreaded execution
- **Seamless PyTorch instrumentation**: Fuzzy PyTorch requires modifying only one line of PyTorch source code (removing `noexcept` from a move constructor), compiling BLAS/LAPACK and ONNX runtime with Verificarlo to achieve complete floating-point operation coverage
- **Up-Down Rounding (UD)**: A novel rounding mode that perturbs already-rounded results by ±1 ULP with equal probability — faster than SR for CNN workloads while producing comparable variability patterns
- **Comprehensive evaluation**: Validated across harmonic series, NAS Parallel Benchmarks, and three DL use cases (1M–341M parameters), demonstrating 5–60× speedups over Verrou with comparable variability assessment
- **Published in TMLR** (January 2026), with open-source code and Docker build at [github.com/big-data-lab-team/fuzzy-pytorch](https://github.com/big-data-lab-team/fuzzy-pytorch)

## Methodology

**PRISM library design:**
- Implements SR and UD rounding for {+, −, ÷, ×, √, FMA} operations using the Highway library for architecture-adaptive SIMD (AVX2, AVX-512)
- SR operators (except FMA) use rounding-mode-free algorithms from Fasi & Mikaitis (2021); FMA extends these with an algorithm inspired by Boldo & Muller (2010)'s ErrFmaNearest
- Supports both static dispatch (compile-time vector types) and dynamic dispatch (runtime pointer-to-scalar for portability)

**PyTorch instrumentation:**
- PyTorch 2.2.1 compiled with Verificarlo 2.0.0 using PRISM as backend (LLVM 7.0.0)
- Open-source BLAS/LAPACK compiled with Verificarlo replaces Intel MKL
- ONNX runtime also instrumented for comprehensive coverage
- Exponential, logarithmic, and argmax functions excluded due to instability under probabilistic rounding

**Evaluation protocol:**
- **Harmonic series** (n=10² to 10⁷, binary32): validates correctness of rounding modes vs. CADNA, MCA RR, Verrou, FM SR
- **NAS Parallel Benchmarks** (dataset classes S and A): measures runtime overhead relative to IEEE-754 baseline
- **DL tasks**: MNIST (small CNN), FastSurfer (CNN brain segmentation, 341M params), WavLM (Transformer-based Parkinson's classification from speech)
- Variability measured via significant digits metric and minimum Sørensen-Dice scores (for segmentation)

## Results & Highlights

- **Runtime speedup**: 5–60× faster than Verrou for UD mode; 7–49× for SR mode across DL workloads — achieved despite using less optimized Netlib BLAS vs. Verrou's Intel MKL
- **NAS benchmarks**: PRISM UD consistently yields the lowest overhead among all evaluated tools; PRISM SR comparable to or better than Verrou SR and CADNA
- **Numerical variability**: PRISM SR exhibits variability comparable to Verrou SR, MCA RR, and FM SR (Levene's test, p=0.29); UD shows higher variability as expected
- **MNIST**: ~6.17 significant digits across metrics; variability mainly in loss function; predictions remain stable
- **FastSurfer**: Minimum Sørensen-Dice scores ≥0.9985 across all brain regions and modes; UD shows slightly higher but acceptable variability
- **WavLM**: Output metrics show ~6.17 significant digits (comparable to IEEE), but pre-max class probabilities drop to ~4 significant digits — revealing hidden instability masked by the argmax operation
- **Embeddings**: Internal feature maps show consistent, task-dependent variability patterns; FastSurfer background instability traced to max-pooling under stochastic arithmetic (motivated subsequent Conservative & Aggressive NaNs work)

## Critical Assessment

**Novelty (moderate):** The core idea of integrating stochastic arithmetic into PyTorch is straightforward from a systems perspective. The UD rounding mode is a simple perturbation scheme (±1 ULP on already-rounded values). The primary technical contribution is the engineering effort of PRISM + Verificarlo + PyTorch integration rather than a fundamentally new algorithm or insight.

**Rigor (strong):** Evaluation is thorough — harmonic series validation, NAS benchmarks, three diverse DL tasks with different architectures (CNN, U-Net, Transformer). Statistical testing (Levene's test) supports claims. Comparison against multiple baselines (Verrou, CADNA, FM SR, MCA RR) is comprehensive.

**Reproducibility (strong):** Open-source code, Docker build, explicit software versions, documented infrastructure. The single-line PyTorch modification is clearly described.

**Significance (moderate-high):** The 5–60× speedup is meaningful for practitioners who need numerical variability analysis. The framework makes such analysis feasible for large models (up to 341M params) where existing tools like Verrou are prohibitively slow. However, the CPU-only limitation restricts immediate applicability to GPU-dominated DL workflows.

**Red flags:**
- CPU-only — no GPU support yet, which limits practical utility for most modern DL training
- Some critical functions (exp, log, argmax) excluded from instrumentation due to instability — these are operationally important
- Uses LLVM 7.0.0 (2018) for experiments despite newer LLVM versions being available; this may raise compatibility concerns
- Verrou runs single-threaded in experiments (Valgrind limitation), which somewhat inflates the speedup numbers — a fairer comparison would be on equal threading footing

## Strengths

- Substantial practical speedup (5–60×) over the best existing tool, making numerical variability analysis feasible at scale
- Minimal code modification required (one line in PyTorch)
- Two complementary rounding modes (SR for accuracy, UD for speed) give users flexibility
- Comprehensive vectorized implementation via Highway supports modern SIMD (AVX2, AVX-512)
- Thorough evaluation across diverse model architectures and established baselines
- Open-source with reproducible Docker build
- Discovered practical insights (e.g., hidden instability in WavLM, max-pooling instability in FastSurfer) that motivated follow-up optimization work

## Limitations

- **CPU-only**: No GPU support; GPU implementation faces fragmented tooling (NVBit for NVIDIA, IREE for broader support) and is listed as future work
- **Excluded operations**: Exponential, logarithmic (SLEEF), and argmax functions cannot be instrumented, leaving gaps in coverage for models relying heavily on these
- **UD rounding inaccuracy**: UD does not preserve exact floating-point operations and converges toward binary32 results rather than binary64 reference values — it is a proxy for variability, not a precise measurement
- **Threading comparison fairness**: Verrou serializes multithreaded execution due to Valgrind, so single-threaded comparisons may overstate Fuzzy PyTorch's advantage in real multi-threaded scenarios
- **Limited DL task diversity**: Only classification and segmentation tasks evaluated; no training-time variability analysis, no generative models, no reinforcement learning
- **Stale toolchain**: Experiments use LLVM 7.0.0 and PyTorch 2.2.1; compatibility with current versions (PyTorch 2.6+ mentioned but not benchmarked) is uncertain

## Relevance

本文与深度学习系统的可复现性和数值稳定性直接相关。Fuzzy PyTorch 提供了一种实用的工具来量化模型推理中的浮点不确定性，对于关注模型鲁棒性、硬件可移植性以及低精度计算（如量化、混合精度训练）的研究者具有重要参考价值。框架目前仅支持 CPU 是其主要局限。

## Questions for Further Exploration

1. How would Fuzzy PyTorch's variability analysis change under mixed-precision (FP16/BF16) settings commonly used in modern GPU training?
2. Can the UD rounding mode's higher variability be leveraged as a form of stochastic regularization during training rather than just analysis?
3. What is the numerical variability profile of large language models (LLMs) and diffusion models under Fuzzy PyTorch instrumentation?
4. How do the excluded operations (exp, log, argmax) affect the completeness of variability assessment for attention-based architectures?
5. Could PRISM's vectorized SR/UD be adapted to GPU tensor cores via compiler-based approaches like IREE or Triton?
