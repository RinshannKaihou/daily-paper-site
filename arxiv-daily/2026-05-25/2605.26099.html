Language Models Need Sleep

Sangyun Lee, Sean McLeish, Tom Goldstein, Giulia Fanti · CMU & UMD

25 May 2026 · Preprint

+52%

Accuracy improvement on GSM-Infinite 2-op problems (59.6% → 90.5%) via sleep-time consolidation — offline recurrent passes that refine fast weights before evicting KV cache.

TL;DR

Before evicting the KV cache at context-window boundaries, the model performs N offline recurrent passes to iteratively refine SSM fast weights — a "sleep" phase that decouples consolidation compute from inference latency. More sleep passes monotonically improve performance, with the largest gains on deeper reasoning tasks.

Methodology

Architecture: SSM-attention hybrid (e.g., Gated Delta Net blocks) with fixed context window L. At each eviction boundary:

  1. Sleep phase — N recurrent forward passes over the current context window. SSM fast weights updated via gated Hebbian rule: St = αt·St-1 + βt·vt·ktT
  2. Cache eviction — KV cache fully cleared (hard) or mostly cleared retaining L−1 tokens (sliding-window)
  3. Prediction — Single forward pass using refined fast weights only; no CoT or extra loops

Training is end-to-end: gradients backprop through all N sleep passes. Compatible with fine-tuning existing hybrids (Jet-Nemotron 2B) or inserting SSM layers into depth-recurrent models (Ouro 1.4B).

Key Contributions

  • Sleep-like consolidation: N recurrent passes refine fast weights before cache eviction
  • Identifies that organizing stored information — not storage capacity — is the bottleneck in hybrid models
  • Decouples compute from inference latency: sleep adds compute during consolidation; prediction stays single-pass
  • Validated on three tasks: Rule 110, Depo multi-hop retrieval, GSM-Infinite math reasoning
  • Works with both hard-eviction and sliding-window strategies

Strengths

  • Elegant biological motivation with clean engineering realization
  • Isolates reasoning depth from memory load via synthetic tasks
  • Consistent scaling: more sleep → better performance
  • Works with both from-scratch training and fine-tuning
  • Minimal architectural changes (<10% parameter increase)
  • Compatible with hard-eviction and sliding-window strategies

Limitations

  • Training sequential across context windows; cost ∝ N
  • Backprop through many recurrent passes can be unstable
  • Modest absolute gains on harder tasks (8-op: +3.7pp)
  • Requires SSM blocks — not a drop-in for pure transformers
  • Evaluated only on synthetic/procedural benchmarks
  • Effectiveness depends on SSM fast-weight quality, varies by architecture

Questions for Further Exploration

  1. How does sleep consolidation compare to streaming LLM, memory-augmented, and RAG approaches under controlled compute budgets?
  2. What is the optimal sleep schedule — fixed N per eviction, or adaptive based on context complexity?
  3. Does the learned consolidation rule generalize across tasks, or overfit to training distribution?
  4. How does the method scale to 7B+ models and diverse real-world long-context tasks?
  5. Can sleep-time compute be amortized across multiple queries sharing the same context prefix?
  6. What happens when fast-weight capacity is insufficient — graceful degradation or catastrophic forgetting?
