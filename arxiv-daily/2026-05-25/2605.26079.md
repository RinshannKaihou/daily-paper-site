> **一句话总结：** 本文提出 Auto Benchmark Audit (ABA)，一个基于 Claude Code CLI (Opus 4.7) 的智能体框架，对 168 个前沿 LLM 基准共 34,285 个任务进行系统审计，发现 25.7% 的任务存在重大缺陷（指令歧义/环境冲突/评测错误），并在 Terminal-Bench 2 上对维护者独立提交的修复 PR 取得 66.7% 严格召回率，过滤这些问题任务后 SWE-bench Verified 与 Terminal-Bench 2 的平均性能分别上升 9.9% 与 9.6%。

> **TL;DR:** ABA is an agentic auditing framework (run via Claude Code CLI with Opus 4.7, v2.1.96) that audits benchmark tasks in static or trajectory mode along three axes (Instruction, Environment, Evaluation) with a 0/1/2 severity scale. Across 168 benchmarks and 34,285 tasks spanning 9 domains, 25.7% of tasks carry at least one major (severity-2) issue (highest: Safety/Alignment 42.2%, Medical 41.5%, Professional 38.4%; lowest: Math 13.2%, Coding 14.1%). Validated against Terminal-Bench 2 maintainer fix PR #53, ABA recovers 14/21 issues under strict matching (66.7% strict / 81.0% partial recall) and independently flags the two SWE-bench Verified tasks (pylint-dev__pylint-4551, sympy__sympy-18199) that OpenAI cited when retiring the benchmark. Filtering flagged tasks shifts leaderboard rankings, raising mean performance on SWE-bench Verified by 9.9% and Terminal-Bench 2 by 9.6%.

## 核心贡献

- 提出 ABA：一个 benchmark-agnostic 的智能体审计框架，结合静态审计（仅读任务定义）与轨迹审计（重放记录的 agent 运行）两种模式，输出统一 schema 的结构化发现。
- 在 9 个领域、168 个基准、34,285 个任务上进行大规模审计，并公开发布全部审计记录、per-task 证据与评分标注（https://github.com/IsThatYou/autobenchaudit, https://www.autobenchaudit.com/）。
- 通过三层证据验证 ABA 的发现：(1) "fixes in the wild"——实践者独立提交的修复 PR 或基准退役决定；(2) BenchGuard 的 gold-issue 集；(3) 内部人工评分研究。
- 量化证明有问题任务对模型排行榜的扭曲：过滤 trajectory 审计标记的任务后，5 个基准的平均排名提升 10.5%，其中 SWE-bench Verified +9.9%、Terminal-Bench 2 +9.6%、Dabstep 在 major+minor 过滤下达 +12.8 个百分点。

## Key Contributions

- An agentic auditing framework (ABA) that combines static and trajectory auditing modes with a benchmark-agnostic evidence collector resolving heterogeneous artifacts (repo dirs, JSON task_patches, simple instruction/gold pairs) into a uniform manifest, onboarding a new benchmark with just a name + URL.
- A large-scale audit of **168 benchmarks and 34,285 tasks across 9 domains** (Science, Multimodal, Professional, Agentic/Tool Use, Coding, Medical, Math, Retrieval/RAG, Safety/Alignment), released as a public artifact with per-task evidence.
- A three-axis issue taxonomy (Instruction / Environment / Evaluation) with a 0/1/2 severity scale (0 clean, 1 minor discoverable gap, 2 major unsolvable), each finding carrying category + severity + evidence file paths + claim-and-why-it-matters + suggested fix.
- Empirical validation via three tiers: maintainer fix PR #53 on Terminal-Bench 2 (independent practitioner action), OpenAI's retirement of SWE-bench Verified, the BenchGuard gold-issue sets on BixBench/ScienceAgentBench, and an internal human-rater study.
- Demonstration that audit-flagged tasks distort leaderboards: trajectory-mode filtering raises mean performance by 10.5% across 5 benchmarks (9.9% SWE-bench Verified, 9.6% Terminal-Bench 2, ~12.8 pp on Dabstep with major+minor filter).

## Methodology

**Pipeline (Figure 2).** Given a benchmark name + URL, ABA runs:
1. **Evidence Collector Agent** — downloads GitHub repos / HuggingFace datasets, and maps each benchmark's heterogeneous representation (Terminal-Bench 2 directory of files, SWE-bench JSON task_patch, or HLE/GPQA-style (instruction, gold_answer) pairs) into a uniform manifest of file-path references: `task_config × N` = {instruction_text, solution_ref, environment_ref, tests_ref, trajectory_path*, test_output_path*} (* trajectory mode only). Lightweight data is inlined; heavyweight assets and execution traces are isolated via filesystem references. No per-benchmark prompt/rubric/schema changes are needed.
2. **Auditor Agent** — initialised with the task config and file paths; uses shell tools (file read/write, bash, search) to inspect artifacts and audit under a standardised rubric. Two modes:
   - *Static mode:* reads only the task definition (instruction, tests, eval config), no execution.
   - *Trajectory mode:* additionally reads recorded agent traces and test outputs, catching runtime issues static inspection cannot.
3. **AuditRecord** — each finding has 5 fields: category (Instruction/Environment/Evaluation, per Figure 3 definitions), severity (0/1/2), evidence (filesystem paths), claim & why-it-matters (NL description + downstream failure mode), suggested fix (concrete change to prompt/environment/tests).
4. **External Validation** — cross-referenced against upstream PRs, published papers/blogs, and expert manual review.

**Agent configuration.** ABA is run through the **Claude Code CLI with default settings: Opus 4.7, default tool permissions (file read/write, bash, search), default system prompt and agent loop, version fixed at v2.1.96.** For the BenchGuard comparison on BixBench/ScienceAgentBench, ABA is run with Claude Opus 4.6 to match the BenchGuard single-model baseline.

**Benchmark sources.** (1) Frontier Benchmarks drawn from five recent frontier LLM releases (Anthropic Opus 4.7, OpenAI GPT-5.4, Zhipu GLM-5.1, Moonshot Kimi K2.6, MiniMax M2.7); (2) NeurIPS 2025 Datasets & Benchmarks Track accepted papers. Inclusion criteria: general-purpose LLMs/agents (excludes audio/embodied/remote-sensing) and verifiable ground truth (excludes subjective or meta-evaluation benchmarks). Total: 168 benchmarks, 34,285 tasks.

**Alignment judge.** Strict match = aligned with gold issue; partial match = aligned + partial. Recall = gold issues recovered; Precision = findings that match gold; Redundancy = average aligned findings per unique gold issue caught.

## Results & Highlights

- **Overall issue rate (Table 1, static mode):** across 168 benchmarks / 34,285 tasks, **25.7% of tasks carry ≥1 major (severity-2) issue** and 15.1% carry only minor issues — i.e., **fewer than 60% of tasks are clean**. There are 14,024 major issues across 8,819 unique affected tasks.
- **Per-domain major-issue rate (Table 1):** Safety/Alignment **42.2%**, Medical **41.5%**, Professional **38.4%**, Science 25.7%, Multimodal 24.7%, Agentic/Tool Use 20.6%, Retrieval/RAG 17.1%, Coding **14.1%**, Math **13.2%**. The top-3 (Safety, Medical, Professional) account for 42.6% of major-issue tasks despite only 26.5% of all tasks.
- **Issue-type patterns:** domains with deterministic graders are instruction-issue-dominant — Math has 65% of major issues from instructions (some benchmarks >90%, e.g., IMOAnswerBench), Science 69%; both have >25% from evaluation-setups (incorrect gold answers).
- **Terminal-Bench 2 maintainer PR #53 (Table 2, fixes in the wild):** ABA's trajectory audit independently recovered **14/21 strict (66.7%) and 17/21 partial (81.0%)** of the issues the maintainers fixed without sight of ABA's findings. Precision vs PR #53 alone is 58.3%/70.8% (a lower bound since the PR is not exhaustive); on 7 hand-reviewed Major tasks outside the PR, ABA achieves 72.7% strict / 81.8% aligned-or-partial precision.
- **SWE-bench Verified retirement:** the two tasks OpenAI's blog cited when retiring SWE-bench Verified (pylint-dev__pylint-4551, sympy__sympy-18199) both appear as major issues in ABA's audit, flagged independently without reading the blog or querying git history.
- **BenchGuard comparison (Table 2):** on ScienceAgentBench, ABA recovers **10/12 strict, 11/12 partial** (matching Opus 4.6 baseline). On BixBench, ABA improves strict recall over BenchGuard (62.5% vs 54.2%) at matched partial recall (79.2%) with higher precision (50.0%/71.4% vs 38.7%/67.7%) and lower redundancy (1.05× vs 1.10×).
- **Leaderboard distortion:** trajectory-mode filtering of major+minor issues raises mean per-model performance by **10.5% across 5 benchmarks**, with **+9.9% on SWE-bench Verified, +9.6% on Terminal-Bench 2**, and **~12.8 percentage points on Dabstep**; several-point increases remain under the conservative major-only filter.
- **Trajectory vs static:** trajectory mode flags more issues (e.g., 591 vs 323 Major tasks on a representative subset) but the substantial overlap supports the robustness of static mode.

## Critical Assessment

- **Precision is only a lower bound on the most important validation:** the 58.3%/70.8% precision vs Terminal-Bench 2 PR #53 is explicitly acknowledged as a lower bound because the PR "is not exhaustive" — but this means ABA's *false-positive rate against the true (unknown) issue set* is not measured; the 72.7%/81.8% figure comes from only 7 hand-reviewed tasks outside the PR, a small sample.
- **Single-auditor-LLM dependency:** all main results use Claude Opus 4.7 (or 4.6 for the BenchGuard comparison), so it is unclear how much of the recall/precision is attributable to the auditor model vs the ABA protocol; no cross-model robustness check (e.g., GPT-5.4 or GLM-5.1 as auditor) is reported, despite those models being in scope of the frontier-benchmark source set.
- **Trajectory mode needs recorded agent runs** that not all benchmarks provide, so the higher-recall trajectory mode is available only where recorded trajectories exist — the static-mode headline (25.7% major) may under-count on benchmarks without trajectories, and the paper does not report what fraction of the 34,285 tasks had trajectory data.
- **The "fixes in the wild" validation, while strong, is only n=2 instances** (Terminal-Bench 2 PR, SWE-bench Verified retirement); the broader precision claim rests on BenchGuard's curated gold sets on just two scientific benchmarks, which are not representative of the Safety/Alignment and Medical domains where the highest major-issue rates are reported.
- **Self-referential risk:** ABA audits benchmarks that include tasks used to evaluate the same model family that powers ABA (Anthropic Opus 4.7 appears both as the auditor and as one of the five frontier models whose benchmarks are audited) — a potential conflict of interest the paper does not address.

## Strengths

- Benchmark-agnostic design: onboarding a new benchmark requires only name + URL, and the framework spans static-QA, patch-based, and multi-step agentic benchmarks under one rubric.
- Validation by "fixes in the wild" (PRs and benchmark retirements by practitioners acting independently of ABA) is a much stronger epistemic standard than constructed gold sets alone.
- Large scale (168 benchmarks, 34,285 tasks, 9 domains) with full public release of findings, per-task evidence, and tooling.
- Concrete demonstration that audit-flagged tasks shift leaderboards by ~10% quantifies the practical stakes.
- The three-axis (Instruction/Environment/Evaluation) × three-severity taxonomy is simple, reproducible, and maps directly to suggested fixes.

## Limitations

- Precision is bounded below by an incomplete PR; true false-positive rate against the real issue set is unknown.
- Single auditor model (Opus 4.7/4.6); no cross-model robustness check.
- Trajectory mode (higher recall) requires recorded agent runs not available for all benchmarks.
- "Fixes in the wild" validation is n=2; broader precision rests on two BenchGuard scientific benchmarks unrepresentative of the worst-affected (Safety/Medical) domains.
- Potential conflict: the auditor model family is also among the audited frontier models.

## Relevance

直接映射到「LLM Inference Evaluation」（自动化基准审计、排行榜可信度、评测错误对模型能力评估的扭曲）与「Self-Evolving Agents & Automated Research」（用 agent 自动发现评测缺陷并建议修复的自动化研究闭环）两个方向；为任何依赖公开基准做模型选择或对比的工作提供了量化可信度证据与可复用的审计工具。

## Questions for Further Exploration

- How sensitive are ABA's recall/precision to the auditor model — do GPT-5.4, GLM-5.1, or Kimi K2.6 as auditors recover the same issue set, and does the 25.7% major-issue rate change materially?
- What is the true false-positive rate against a constructed exhaustive gold-issue set on a Safety/Alignment or Medical benchmark (where the highest major-issue rates are reported), rather than only on the two BenchGuard scientific benchmarks?
- What fraction of the 34,285 tasks have recorded trajectories available, and how much does the static-mode 25.7% under-count relative to trajectory mode on the subset where both are possible?
- Can ABA's suggested fixes be auto-applied and re-audited in a closed loop to harden benchmarks without human intervention?
- Does the leaderboard-distortion result (10.5% mean lift) hold when issues are filtered by an independent auditor model rather than the same Opus 4.7 used to produce the audit?
