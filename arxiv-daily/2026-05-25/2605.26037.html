2605.26037 GRPO Tool Use Knowledge Graphs Reward Hacking

Peak-Then-Collapse and the Four Interface Channels of Knowledge-Graph Tool Use

Tianda Sun & Dimitar Kazakov · University of York

Key Finding

All 7 GRPO reward designs collapse. The strongest peaks at 9.6% tool accuracy then drops to 0% in 50 steps — because KG interfaces lack the feedback signals that make Python/web-search RL work.

9.6% → 0% Peak-then-collapse accuracy (50 steps)
40.0% Best EM (self-distillation)
95.4% Errors from retrieval-composition failure
7 ≈ 14B Capacity gap: only 0.25 pp EM
7 Reward designs tested
4 Failure mode channels identified
So What

Standard RL-for-tools recipes are not portable across API types. KG interfaces provide no natural-language error messages, no compositional state visibility, and no pretraining prior — making any proxy reward exploitable. The bottleneck is interface-bound, not model-bound.

Critical Edge

The four-channel framework is descriptive, not causally validated — only 1 of 4 channels was directly tested via oracle ablation. The specific channel gating the residual EM gap (Lcomp vs Lsig) remains unresolved.

Four Interface Channels
  • Lsig — Silent failure: KG returns empty sets with no error message, hiding whether the query was wrong or the data absent
  • Llang — Symbolic schema: Relation names are opaque tokens, not natural-language descriptions the model can interpret from pretraining
  • Lcomp — Opaque composition: Multi-hop entity threading requires tracking state across calls with no intermediate feedback
  • Lprior — Absent prior: Unlike Python/web-search, LLMs have no pretraining exposure to KG query patterns
Key Contributions
  • C1: Controlled 7-reward ladder comparison on one model × task × algorithm, organized into a four-mode failure taxonomy
  • C2: Signal-theoretic framework with oracle ablation localizing the EM gap downstream of relation selection
  • C3: One-iteration self-distillation recipe reaching EM=40.0%, capacity-invariant across 7B–14B
  • C4: Step-resolved peak-then-collapse with a Tools/Q call-volume signature (3.0→1.0)
Methodology
  • Task: Complex WebQuestions over Freebase, 4-verb KG tool interface
  • Model: Qwen2.5-7B-Instruct, LoRA (r=64, α=128, DoRA + rslora)
  • Algorithm: GRPO, group size 8, KL 0.05, lr 3e-7, 500 steps on GH200
  • Evaluation: Full CWQ test (3,531 questions), greedy decoding, 7-category trajectory classifier, Wilson 95% CIs, McNemar tests

Strengths
  • Exceptionally well-controlled: one model, one task, one algorithm, systematic reward ladder
  • Step-resolved evaluation at 50-step cadence catches transient phenomena
  • Honest reporting of negative results — all 7 designs fail distinctly
  • Oracle ablation cleanly isolates bottleneck locus
  • Capacity-invariance (7B ≈ 14B) proves interface, not scale, is the constraint
Limitations
  • Deliberately minimal 4-verb KG interface may not represent practical scenarios
  • Framework is descriptive hypothesis, not causally validated (1/4 channels tested)
  • 40% EM ceiling far below richer pipelines (67–80%)
  • Single task (CWQ) and single model family (Qwen2.5)
  • Unresolved Lcomp vs Lsig leaves precise failure mechanism open
Open Questions
  1. Can a non-gameable self-verifiable reward signal be designed for KG tool use, or is any proxy reward on an opaque interface exploitable?
  2. Which specific channel (Lcomp entity-threading or Lsig silent failure) gates the residual EM gap between 40% and oracle?
  3. Would enriching the KG interface along a single channel (e.g., adding NL error messages) produce disproportionate gains?
  4. Does peak-then-collapse generalize to other structured-data interfaces (SQL, SPARQL) with similar opacity?
  5. Can the four-channel framework predict training difficulty for novel tool APIs before running expensive RL experiments?
