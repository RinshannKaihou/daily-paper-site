> **一句话总结：** 本文为Jamba等混合Mamba-Transformer模型提出非对称虚拟内存分页（AVMP）分配器，将KV缓存与SSM状态分离到两个物理异构池并在分配失败时按需迁移容量，在RTX 3060 12GB的180单元合成+60单元ShareGPT回放实验中，相对最优静态双池基线将OOM降低7.6%（510 vs 552）、吞吐在uniform_short上提升13.3×、在ShareGPT上提升2.36×。

> **TL;DR:** Hybrid Mamba-Transformer models (e.g., Jamba) need two asymmetric caches — linearly-growing KV pages and fixed per-layer SSM blocks — but vLLM-style unified pools pad SSM states to attention page sizes (7.3× overestimation on Qwen3.5-4B-AWQ) and SGLang-style static dual pools (mamba_full_memory_ratio=0.9) trigger 1221 vs 552 OOMs at ratio 0.5. AVMP introduces a 32-bit VirtualHandle over two physically distinct stores with CapacityError-triggered migration, validated on an RTX 3060 12GB across 180 synthetic cells + 60 ShareGPT cells. Dynamic AVMP (b=128) reduces OOMs 7.6% (510 vs fixed_dual_mr05's 552, 95% CI [−5.83, −1.39]/cell) and lifts goodput 13.30× on uniform_short (434.24 vs 32.65 req/s), 2.39× on mixed_long, 1.83× on agentic_burst, 2.36× on ShareGPT — all bootstrap CIs exclude the null.

## 核心贡献

- 提出"非对称虚拟页表"抽象：用统一的32位VirtualHandle覆盖两个物理异构后端存储（KVPagesStore与SSMBlocksStore），将GPU虚拟内存思想从单一KV缓存扩展到两种非对称池类型。
- 实现"容量按需迁移"机制：在allocate()的CapacityError异常处理器中触发池间容量迁移（而非预测式采样钩子），用逻辑操作计数器（min_rebalance_interval_ops=1000）而非挂钟时间保证确定性与逐单元字节级可复现。
- 在RTX 3060 12GB上对5种分配器变体×3种合成负载×2种模型规格×2种池容量×3随机种子共180单元进行系统扫描，并用配对bootstrap 95%置信区间给出显著性。
- 开源参考实现与合成负载框架：https://github.com/codepawl/cachepawl 。

## Key Contributions

- **Asymmetric Virtual Page Table abstraction (§3):** a unified virtual handle space over two physically heterogeneous backing stores (KV pages with native attention_page_tokens=16 sizing; SSM blocks matching exact state_dim×bytes_per_element), extending PagedAttention / GPU virtual memory to two asymmetric pool types with byte-identical OOM counts (552) vs the static dual baseline.
- **CapacityError-triggered dynamic rebalancing (§3.2):** a 3-state machine (BALANCED / KV_PRESSURED / SSM_PRESSURED / REBALANCING) that migrates capacity between pools only when allocation would otherwise fail, gated by threshold_high=0.30 free fraction and a logical operation counter (not wall-clock) for byte-identical reproducibility.
- **Empirical validation on hybrid architectures (§4):** a 180-cell sweep (5 allocators × 3 synthetic workloads × 2 models × 2 pool sizes × 3 seeds) plus a 60-cell ShareGPT replay on an RTX 3060 12GB, with paired-bootstrap 95% CIs (B=10000, RNG seed 20260520).
- **Open-source prototype:** reference Python implementation and synthetic workload harness with committed sweep artifacts at https://github.com/codepawl/cachepawl.

## Methodology

**Problem framing.** Hybrid models like Jamba mix attention (KV cache = O(n) per token) with Mamba SSM (fixed O(1) state per layer) at a 1:7 attention-to-SSM ratio, supporting 256K context with ~12B active params in MoE. Two failure modes are identified and quantified:
- *Unified pools* (vLLM HybridKVCacheCoordinator, issue #37121): the capacity estimator incorrectly multiplies the O(1) SSM state by the sequence token count, padding SSM to attention page size — **7.3× KV cache overestimation on Qwen3.5-4B-AWQ, leaving only 13.7% effective VRAM utilisation** at peak.
- *Static dual pools* (SGLang HybridReqToTokenPool + HybridLinearKVPool): a fixed `mamba_full_memory_ratio` (default 0.9) cannot rebalance without a process restart — on identical workloads, ratio 0.9 triggers **1221.3 OOMs vs 552 at ratio 0.5**.

**AVMP design.** Allocations return an opaque 32-bit `VirtualHandle` tagged with `pool_id` and `page_id`; the `VirtualPageTable` resolves it to a physical offset. KV pages use `attention_page_tokens × per_token_bytes` (default 16 tokens/page); SSM blocks use exact `state_dim × bytes_per_element`. Alignment: 128-byte slab global, 16-byte page within slabs. The rebalancing state machine fires inside the `CapacityError` handler of `allocate()`: if one pool fails and the other's free fraction > threshold_high (0.30), the donor is shrunk at the high end of its address space and the recipient grows; backing tensors are oversized at init to span both pools' maxima, so the current Python prototype does no physical data copy (a future cuMemMap implementation would do on-demand remapping). Migration is throttled to ≥1000 logical operations apart.

**Evaluation setup.** Single NVIDIA RTX 3060 12GB, CUDA 13.0, PyTorch 2.12.0+cu130, Python 3.10.19, WSL2 Ubuntu. The 180-cell main sweep covers 5 allocator variants (`padded_unified`, `fixed_dual_mr05`, `fixed_dual_mr09`, `avmp_static_mr05`, `avmp_dynamic_b128`) × 3 synthetic workloads (`uniform_short`, `mixed_long`, `agentic_burst`) × 2 model specs (`jamba_1_5_mini`, `mamba2_1b3`) × 2 pool budgets (1 GiB, 4 GiB) × 3 seeds — wall time 16:14. Supplementary sweeps add wall-clock decomposition (180 cells, 18:46) and ShareGPT trace replay (60 cells, 1:56).

**Baselines.** Allocator-level models of vLLM (padded unified) and SGLang (fixed dual at mr=0.5 and mr=0.9), *not* full engine reproductions — kernel execution and request scheduling are explicitly out of scope.

**Statistics.** Paired bootstrap 95% CIs (B=10000, RNG seed 20260520) over matched (workload, model, pool, seed) tuples; pre-registered justification threshold: goodput must exceed 1.10× the best baseline on at least one workload. Determinism is enforced via a logical operation counter (not wall-clock) so event-deterministic fields reproduce byte-identically.

## Results & Highlights

- **OOM reduction:** `avmp_dynamic_b128` records **510 cross-workload OOMs vs 552 for the best static baseline `fixed_dual_mr05`** — a 7.6% reduction, paired bootstrap 95% CI [−5.83, −1.39] OOMs/cell (cross-workload), excluding zero; per-cell this is a 3.0–12.7% cross-workload reduction.
- **Goodput (req/s):** `avmp_dynamic_b128` vs `fixed_dual_mr05` — **13.30× on uniform_short (434.24 vs 32.65 req/s, 95% CI [11.18, 16.00])**, 2.39× on mixed_long (65.07 vs 27.25, CI [1.70, 3.04]), 1.83× on agentic_burst (46.91 vs 25.69, CI [1.42, 2.60]); all CIs exclude both unit ratio and the 1.10× pre-registered threshold. ShareGPT trace replay gives **2.36× goodput improvement**.
- **Per-workload OOM wins:** AVMP beats static on mixed_long (364.3 vs 387.3, CI [−10.3, −1.33]) and agentic_burst (136.7 vs 157.3, CI [−9.3, −1.3]); on uniform_short the delta is +0.4 (CI [0.0, +1.25]) — *statistically inconclusive* on the KV-only workload, consistent with a within-~1.7-OOM tie.
- **Failure of existing designs:** `padded_unified` triggers **1567.7 OOMs** (worst; CI [+46.2, +128] vs fixed_dual_mr05); `fixed_dual_mr09` (SGLang default ratio) triggers **1221.3 OOMs** (CI [+29.2, +86.9]).
- **Equivalence control:** `avmp_static_mr05` ties `fixed_dual_mr05` at exactly 552 OOMs (CI [0, 0]), validating the virtual-handle abstraction introduces no measurable overhead.
- **Dominant config axis:** parameter sweep identifies `migration_batch_size=128` as the dominant axis; the Stage-2 threshold sweep returns a strict null result.
- **Trade-off:** AVMP maintains a **2× VRAM footprint, peaking at 9216 MiB vs 5120 MiB** for static baselines (backing tensors oversized to enable migration without reallocation); cuMemMap integration would remove this.

## Critical Assessment

- **No real inference engine integration:** the baselines (`padded_unified`, `fixed_dual_mr05`, `fixed_dual_mr09`) are *allocator-level models* of vLLM issue #37121 and SGLang, not reproductions against vLLM-main or SGLang-main; the paper explicitly states "A full reproduction against vLLM main or SGLang main would require integrating AVMP into those engines, which we leave to future work." So the 13.3× goodput is measured against a modelled baseline, not the production system — and the prototype "does not implement kernel execution and request scheduling."
- **Single low-end GPU:** evaluation is limited to one RTX 3060 12GB; behaviour on A100/H100 80GB (where OOM pressure is far lower) or multi-GPU is unknown. The 2× VRAM footprint trade-off (9 GiB peak on a 4 GiB pool) would consume a larger fraction of a small GPU's budget and may distort the headline ratios.
- **The 13.3× headline is on uniform_short, the KV-heavy workload where AVMP's OOM delta is itself statistically inconclusive** (CI [0.0, +1.25]); the paper attributes the goodput win to "faster allocation calls" rather than to OOM reduction, which is a weaker and more implementation-dependent mechanism (pure-Python allocator overhead).
- **Pure-Python prototype, Triton/CUDA integration deferred:** wall-clock goodput numbers include Python overhead that a kernel-level implementation would not have, so the absolute req/s values are not directly comparable to production engines.
- **effective_batch_size_p50 is identical (129/132/284) across all 5 variants per workload**, meaning AVMP does not increase sustained concurrency — the win is purely in OOM-recovery latency, a narrower claim than "higher throughput" suggests.

## Strengths

- Precisely identifies and quantifies (with the 7.3× overestimation and 1221-vs-552 OOM numbers) a real, documented failure mode in production engines (vLLM #37121, SGLang's default ratio).
- Clean abstraction: the 32-bit tagged VirtualHandle over two physically heterogeneous stores is a simple, reusable idea, and the static-equivalence control (552=552 OOMs) proves it adds no overhead.
- Lazy, CapacityError-triggered migration keeps behaviour deterministic (logical op counter, not wall-clock) and byte-identical across reruns.
- Strong statistical hygiene: pre-registered RNG seed, pre-registered 1.10× threshold, paired-bootstrap CIs on every headline claim, open-source code + artifacts.

## Limitations

- Allocator-level baselines only; no integration with vLLM-main or SGLang-main, so gains are vs a model of those systems, not the systems themselves.
- Single RTX 3060 12GB; no large-memory or multi-GPU evaluation.
- Pure-Python prototype with 2× VRAM footprint overhead; Triton/cuMemMap integration is future work.
- The headline 13.3× goodput comes from the workload (uniform_short) where the OOM reduction is statistically inconclusive; the mechanism shifts to "faster allocation calls," which is Python-overhead-dependent.
- Only Jamba (1:7) and Mamba2 model specs tested; MoE routing memory is explicitly out of scope.

## Questions for Further Exploration

- Do the OOM and goodput gains survive when AVMP is integrated into vLLM-main or SGLang-main with real attention/SSM kernels, rather than an allocator-level model?
- How does the 2× VRAM footprint trade-off behave on A100/H100 80GB, and does cuMemMap-backed on-demand remapping eliminate it?
- Is the 13.3× uniform_short goodput gain an artefact of pure-Python allocator overhead on the KV-heavy path, or does it persist in a Triton implementation?
- How does AVMP interact with KV-cache compression (quantization, eviction, recomputation) and with request preemption policies used in production engines?
- Can the CapacityError-triggered migration policy extend to multi-GPU / disaggregated prefill-decode settings where pool pressure crosses device boundaries?
