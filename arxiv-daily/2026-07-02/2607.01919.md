> **一句话总结：** ElephantAgent 提出一种基于可验证上下文状态连续性的协议，通过两个组件（主机侧 Context Guard 与基于 Nimble 账本+Intel TDX 可信硬件的远程 State Continuity Module）在每次查询前重算并校验上下文状态摘要，以确定性地防御 MCP 工具投毒与记忆投毒等带外篡改，并辅以 Historical Traceability 支持带内语义滥用的事后审计与回滚；实测在三项 Agent 任务上端到端开销仅 1.02×–1.04×。

> **TL;DR:** ElephantAgent is a verification-based security protocol that defends agentic systems (instantiated over MCP) against tool-descriptor and memory *poisoning* by enforcing **Contextual State Continuity**: before each query, a host-side Context Guard (CG) recomputes the digest of the bounded, security-critical contextual state (tool descriptors + memory) and verifies it against the latest authorized digest held in a linearizable, append-only ledger maintained by a remote, replicated, TDX-backed State Continuity Module (SCM, built on Nimble). Out-of-band tampering is *deterministically* rejected (not statistically detected), while in-band semantic abuse is handled by a complementary **Historical Traceability** property enabling post-hoc audit and rollback. The authors give formal security proofs and report only **1.02×–1.04× end-to-end overhead** across three agent tasks (email triage, code bug fix, calendar scheduling), with verification ≈4.3 ms and update ≈5.3 ms per query.

## 核心贡献

- **统一威胁建模**：将 MCP 工具投毒（含"rug pull"攻击）与记忆投毒归因为同一根源——智能体上下文状态缺乏可验证的连续性，并形式化定义了"上下文状态" C_t = ⟨T_t, M_t⟩（工具描述 + 由长期指令、对话记录、受管产物三部分组成的记忆）。
- **Contextual State Continuity（上下文状态连续性）形式化与协议**：定义主安全属性，使真实系统在每一步诱导的状态对 (id, h) 与可信参考系统账本一致；设计 CG + SCM 双组件协议，采用 commit-before-finalize 工作流，带外篡改被 VerifyState 确定性拒绝。
- **Historical Traceability（历史可追溯性）形式化**：作为补充属性，针对 Contextual State Continuity 无法阻止的带内语义滥用（如 prompt injection 通过合法工作流提交的恶意状态转换），提供条件性事后审计与回滚到已知良好状态。
- **基于可信硬件的可实用实例化**：SCM 用 Nimble 的线性化账本在 Intel TDX 保护的 VM 中以复制服务实现，CG 内嵌于 TEE 中的 MCP Host；并附 TDX 远程证明与信道绑定认证。
- **完整的形式化证明**：在 6 条假设 (A1–A6) 下给出 Lemma 1（单步完整性与新鲜性）、Corollary 1（真实与理想状态序列一致）、Lemma 2（授权/记录/有序的有效转换）与 Corollary 2（条件可审计与可恢复），证明放入附录 C。
- **低开销实测验证**：在三项真实 Agent 任务上端到端开销 1.02×–1.04×，初始化一次性开销 8.22×（远程证明 + SCM 建立）；并给出"证明式 MCP Server"消融实验。

## Key Contributions

- **Unifying threat model.** Reframes tool poisoning (including MCP "rug pull") and memory poisoning as different symptoms of one root cause — the absence of *verifiable continuity* in the agent's contextual state — and formally scopes the contextual state as C_t = ⟨T_t, M_t⟩ (tool descriptors + memory split into long-term instructions M_I, query-level transcripts M_R, and managed artifacts M_A, following Codex/Claude Code abstractions).
- **Formalization of Contextual State Continuity (Definition 1).** The primary property: for an approved initial state C_0, any query sequence Q, and any resolution Ω of benign nondeterminism, the real system induces the same sequence of (id, h) pairs as an ideal reference system's ledger L*. This targets out-of-band tampering (Scenarios I & II).
- **Formalization of Historical Traceability (Definition 2).** A complementary property for in-band semantic abuse (Scenario III): after an undesired committed transition or crash, the state evolution can be audited and the system recovered to a known-good prior state, *provided* local ledger + snapshots are available.
- **The ElephantAgent protocol.** Two add-on components: a host-side **Context Guard (CG)** inside a TEE that recomputes/verifies digests and enforces a **commit-before-finalize** workflow, and a remote **State Continuity Module (SCM)** — a replicated TEE-backed service maintaining a linearizable, append-only ledger of authorized digests, issuing signed receipts/proofs with fresh read-challenge nonces. Protocol-agnostic: also instantiated for CLI-based agents (Appendix A).
- **Practical SCM instantiation.** SCM is built on the **Nimble** linearizable ledger running inside **Intel TDX** VMs (chosen over SGX to avoid enclave-specific refactoring), with TDX-adapted remote attestation binding the SCM identity to the TLS channel.
- **Formal security analysis.** Under six assumptions (A1–A6: SCM ledger correctness, signature unforgeability, hash collision-resistance, trusted/non-bypassable CG, attested SCM access, local artifact availability), the authors prove Lemma 1 (single-step integrity + freshness), Corollary 1 (sequential agreement with the ideal reference), Lemma 2 (authorized/logged/ordered effective transitions), and Corollary 2 (conditional auditability + recovery).
- **Low-overhead empirical evaluation.** End-to-end overhead of only **1.02×–1.04×** across three agent tasks; per-query verification ≈4.2–4.3 ms and update ≈5.3–5.4 ms; plus an ablation on attested-MCP-server mode.

## Methodology

**Architecture (two augmentation components):**
- **Context Guard (CG)** — runs inside a TEE on the MCP Host, mediates the planning/execution path. Before each query it recomputes the contextual-state digest from local tool state + memory and verifies it against SCM's latest authorized digest. On updates it uses **commit-before-finalize**: commit next digest to SCM → seal pending record → finalize local side effects only after a valid SCM receipt → append to local ledger atomically (crash-consistent).
- **State Continuity Module (SCM)** — a remote, replicated TEE-backed service exposing a linearizable abstraction with two primitives, `Commit(ℓ, id, h)` and `GetLatestState(ℓ, n)`. It enforces strictly monotonic, gap-free identifiers, rejects rollback/fork attempts, and signs receipts/proofs bound to a fresh nonce. SCM is instantiated with **Nimble** running inside Intel TDX VMs.

**Contextual state scoping.** C_t = ⟨T_t, M_t⟩ where T_t = registered tool descriptors (names, descriptions, I/O schemas, permissions) and M_t = ⟨M_I (system prompt/skills), M_R (query/response transcript), M_A (managed artifacts)⟩. Scope is justified by two criteria: (i) components are stable input to future planning cycles, (ii) deterministically encodable under the hash. Components governed by other layers (e.g., TLS transport state) are explicitly excluded.

**Workflow (Algorithm 5), three phases, fail-closed:**
1. *State initialization* — TOFU-based `ApprovePolicy`, compute h_0, anchor in SCM, snapshot C_0 into a sealed store. (Also serves as the recovery primitive.)
2. *State verification* (per query) — fresh nonce read-challenge to SCM, signature verification, digest/ledger/id equality check; any mismatch aborts.
3. *State update* — side-effect-free `PreviewApply` derives h_next, commit to SCM, seal pending, finalize, optionally snapshot, append local ledger entry.

**Security analysis.** Formal proofs (deferred to Appendix C) establish single-step integrity+freshness (Lemma 1), sequential agreement with an ideal reference system (Corollary 1), authorized/logged/ordered transitions (Lemma 2), and conditional auditability+recovery (Corollary 2). An explicit attack-vector catalog (Appendix B) covers 14 vectors: V1–V7 in-scope (deterministically rejected: rug-pull, memory tampering, memory rollback, stale-proof replay, forged proof/receipt, SCM impersonation, interrupted update); V8–V9 conditionally in-scope (in-band abuse, local artifact tampering); V10–V14 out-of-scope (TEE compromise, LLM-channel compromise, availability/deletion, SCM quorum loss, compromised initial trust root).

**Evaluation setup.** Intel Xeon 6982P-C (TDX-capable) Alibaba Cloud `ecs.g9i.xlarge` VMs; SCM replicated across TDX2/TDX3 (5 nodes); MCP Host on TDX1, MCP Server on NT1. Agent LLM = qwen-flash (chosen so ms-scale CG/SCM cost is observable); uses server-side `req-cost-time` to isolate from network latency. 19 hand-built MCP tools across three tasks. Compared ElephantAgent (confidential mode) vs. NoGuardAgent (baseline, no CG/SCM). 50 valid runs per task.

## Results & Highlights

- **End-to-end overhead: 1.02×–1.04×** across Email Triage (2249→2324 ms), Code Bug Fix (2359→2412 ms), Calendar Scheduling (2831→2950 ms).
- **Per-query overhead is small and stable:** verification 4.23–4.33 ms, update 5.25–5.39 ms, extra data-loading time <1 ms across all tasks.
- **Initialization cost: 8.22×** (219 ms → 1800 ms) due to remote attestation + SCM setup — but it is a one-time, amortized cost.
- **Ablation (attested MCP Server):** running the MCP Server inside a TDX VM with attestation adds 1.51× initialization overhead and 1.12×–1.22× tool-invocation overhead, growing with baseline invocation time — modest enough for security-critical MCP servers.
- **Defense is deterministic, not statistical.** The authors explicitly argue (§6.1) that, under A1–A5, in-scope out-of-band tampering is *always* rejected by VerifyState, so reporting detection rates (as learned detectors do) is inappropriate; they instead measure the *cost* of enforcement, following prior state-continuity work (Memoir, ROTE, Nimble).
- **Comparison table (Table 1)** positions ElephantAgent as the only surveyed defense covering *both* tool and memory against out-of-band tampering, plus audit + recovery for in-band abuse — vs. MindGuard (tool only, no recovery), SecureMCP (tool only, blocking), A-MemGuard/SuperLocalMemory (detection/isolation, no continuity verification).

## Authors & Credibility

All seven authors (Jiankai Jin, Xiangzheng Zhang, Zhao Liu, Wenzhuo Xu, Dongdong Yang, Deyue Zhang, Quanchen Zou) are from **360 AI Security Lab** (Beijing, China). Notably, the lead author **Jiankai Jin** is a co-author of "Elephants do not forget: Differential privacy with state continuity" (CCS 2024, ref [19], "ElephantDP") and the system borrows from **Nimble** (OSDI 2023) and ROTE/Memoir state-continuity lineage — giving the team strong, directly relevant prior art in TEE state continuity. The framing (cs.AI + cs.CR), formal proofs, and honest discussion of out-of-scope vectors reflect a security-systems paper written by researchers fluent in the TEE/state-continuity literature.

## Critical Assessment

- **Novelty — moderate, well-scoped.** The core idea is a *transfer* of established state-continuity techniques (Nimble/ROTE/Memoir/ElephantDP) from OS/execution-state and DP-budget domains to the *contextual state* of LLM agents. The conceptual reframing of tool/memory poisoning as a state-continuity violation is genuinely useful and timely, and the two-tier design (deterministic prevention + conditional recovery) is sensible. But the mechanism itself (digest + replicated TEE ledger + commit-before-finalize) is a well-trodden pattern; the contribution is the *application* and the threat-model unification, not new cryptographic primitives. The authors are commendably explicit about this lineage rather than overclaiming originality.

- **Rigor — high.** Genuine formal proofs with clearly stated assumptions (A1–A6), an inductive argument for sequential agreement, a 14-vector attack catalog with explicit in-scope/conditional/out-of-scope boundaries, and deferred proofs in the appendix. The fail-closed commit-before-finalize design and nonce-bound proofs are cryptographically sound. This is more rigorous than typical agentic-defense papers, which lean on statistical detectors.

- **Reproducibility — medium.** Implementation details are reasonably specific (Nimble in Rust, Python for the rest, TDX VMs, qwen-flash, Alibaba Cloud instance types, 50 runs). However, the prototype depends on specific TDX-capable hardware and a Nimble deployment that most labs lack; SCM reconfiguration and KMS integration are left to "production deployment." No code release is mentioned.

- **Significance — promising but deployment-gated.** The 1.02×–1.04× overhead is genuinely attractive *if* one already operates TDX infrastructure. For teams without TEE hardware, the protocol is largely inaccessible. The deterministic-rejection framing is a meaningful upgrade over brittle ML-based detectors.

- **Red flags / weak points to scrutinize:**
  1. **Small, synthetic tasks.** Email triage, a single QuixBugs case, and calendar scheduling are toy-scale; the authors themselves note LLM time dominates real workloads, which would shrink relative overhead further — but also means the *interesting* large-context/large-memory regime (where digest recomputation and snapshot storage actually scale) is untested. No measurement of digest cost vs. context size, or snapshot storage growth.
  2. **No adversarial end-to-end evaluation.** Despite being a defense paper, there is no experiment *demonstrating* an actual poisoning/rollback attack being caught — the authors argue this is unnecessary because rejection is deterministic. That argument is logically sound but leaves the empirical story feeling incomplete; a single worked attack example (e.g., rug-pull detected and blocked) would strengthen it substantially.
  3. **Weak initial trust root.** The prototype uses naive TOFU (`ApprovePolicy`), which the authors themselves flag (V14) as a risk "advisable to strengthen for production." A compromised initial state defeats the entire chain.
  4. **Availability is explicitly out of scope.** An adversary controlling the host can delete the local ledger/snapshots and brick the agent (fail-closed) or constrain recovery to stale states. This is a real liveness concern for hostile deployment environments.
  5. **SCM as a trusted, centralized service.** The protocol trades a host trust problem for a (replicated) SCM trust problem; quorum loss loses liveness. Operational burden is non-trivial.
  6. **LLM-channel blind spot.** Prompt/response tampering on the path to the external LLM API is out of scope (V11); the suggested mitigation (local confidential GPU model) is a heavy lift.

Overall: a solid, honestly-written security-systems paper with a clean formalization and encouraging overhead numbers, whose practical impact hinges on TEE availability and on future work addressing the initial-trust-root and large-scale-regime gaps.

## Strengths

- Clean, unifying threat model tying tool poisoning + memory poisoning + rollback to one root cause (state-continuity violation), with a precise, justified scoping of the "contextual state."
- Deterministic, proof-based defense rather than a brittle learned detector — a meaningful categorical improvement over MindGuard/SecureMCP/A-MemGuard.
- Formal security analysis with explicit assumptions, lemmas, corollaries, and a 14-vector attack catalog; intellectually honest about in-scope vs. out-of-scope.
- Low, well-measured per-query overhead (1.02×–1.04×), with sensible design choices (commit-before-finalize, nonce-bound proofs, crash-consistent local ledger).
- Protocol-agnostic design demonstrated for both MCP and CLI deployments.
- Authors credit prior art (Nimble, ElephantDP) rather than overclaiming.

## Limitations

- Evaluated only on small, synthetic tasks; the large-context/large-memory regime where digest/snapshot costs actually matter is untested, and no scaling analysis is provided.
- No end-to-end demonstration of an attack being detected/blocked — relies on the (valid but unsatisfying) "deterministic by construction" argument.
- Initial trust root is naive TOFU in the prototype; vulnerable at anchor time.
- Strong hardware dependency: requires Intel TDX-capable CPUs and a replicated Nimble SCM deployment, limiting accessibility.
- Availability/liveness is sacrificed for safety under adversarial interference (fail-closed); host adversary can brick the agent or force recovery to stale states.
- Out of scope: TEE side-channel attacks, LLM-service channel compromise, and in-band semantic abuse prevention (only post-hoc recovery).
- No code/artifacts released (as of the paper).

## Relevance

本文聚焦于 LLM Agent（尤其是基于 MCP 协议的 Agent）系统的安全防护，针对工具投毒、记忆投毒与回滚攻击这三类当前 Agent 部署中最具现实威胁的攻击面，提出了基于可信硬件的可验证状态连续性方案。对于关注 Agent 安全、可信执行环境（TEE）以及 MCP 工具链生态安全性的研究者与工程团队具有直接且较高的参考价值。

## Questions for Further Exploration

- How does per-query digest recomputation and snapshot storage cost scale with context size, memory depth, and number of tools — i.e., what happens in the realistic large-context regime the paper avoids?
- Can the initial TOFU trust root be replaced by something stronger (e.g., signed manifests, a transparency log of approved tool descriptors) without breaking usability, and how would that change the threat model?
- What is the operational cost and failure behavior of the replicated Nimble SCM under realistic churn, network partitions, and partial TEE compromises?
- Could Historical Traceability's recovery handle *irreversible* external side effects (e.g., a financial transaction already executed via a tool), or is recovery limited to the in-agent contextual state only?
- How does ElephantAgent compose with complementary defenses — e.g., LLM-channel encryption/local confidential models, runtime prompt-injection detection — to close the remaining out-of-scope vectors (V11, in-band abuse prevention)?
- Is there a path to a non-TEE instantiation (e.g., a blockchain/transparency-log-backed SCM) that broadens deployability while preserving the linearizability guarantees?
