> **一句话总结：** Vera 将软件工程测试原则迁移到非确定性 LLM Agent：先用约 800 篇文献自动构建风险/攻击/环境三类分类学（124 风险 × 77 攻击 × 30 环境），经组合过滤从 39,078 个候选中产出 1,600 条可执行安全案例，在 Docker 沙箱（5 个自托管服务、72 个 MCP 工具）中由自适应 Control Agent 驱动多轮攻击并以环境状态证据验证，在 Claude Code/Codex/OpenClaw/Hermes 四个生产级框架上获得 90.6%（单通道）和 93.9%（多通道）的平均攻击成功率。

> **TL;DR:** Vera operationalizes software-engineering testing principles for non-deterministic LLM agents: it mines ~800 papers into three taxonomies (124 risks × 77 attack methods × 30 environments), filters 39,078 combinatorial candidates down to 1,600 executable safety cases, runs four production agent frameworks (Claude Code, Codex, OpenClaw, Hermes) in isolated Docker sandboxes with an adaptive Control Agent and evidence-grounded verifiers, achieving 90.6% (single-channel) and 93.9% (multi-channel) average attack success rates, and shows that Vera-derived supervision fine-tunes a guard model (0.930 acc / 0.941 F1) that beats LlamaGuard3, Qwen3Guard, NemoGuard, and AgentDoG.

## 核心贡献

- **可执行安全案例形式化**：把抽象风险定义为三元组 σ = ⟨g, s₀, V_g⟩（安全目标 g、可程序化构造的初始状态 s₀、确定性验证谓词 V_g），强调"已发生可观测的有害后果"而非仅"unsafe 请求或文字意图"。
- **三阶段自强化流水线**：(1) 文献驱动的持续风险探索，将 ~800 篇 arXiv/OpenReview 论文整理为风险/攻击/环境三类层级分类学；(2) 跨维度组合 + 兼容性过滤（39,078 → 1,600）；(3) 沙箱自适应执行 + 证据验证，验证结果回灌风险探索与场景精炼。
- **两档威胁模型 + 可配置工具网关**：单通道仅注入用户消息；多通道额外经 MCP 工具网关用 identity/append/prefix/override 四种模式篡改工具返回，且攻击者无法改模型/系统提示/工具代码，仅通过运行时通道作恶。
- **证据优先级验证器**：V_g = V_g^state ▷ V_g^tool ▷ V_g^resp —— 优先用环境终态判定，工具调用记录次之，模型自报告最次，并采用不对称判定（Control Agent 报失败直接 y=0，报成功必须验证器确认才 y=1）。
- **大规模沙箱基础设施**：每条运行使用 12 容器 Docker Compose 栈，含 5 个自托管真实服务（Mailpit 邮件、Gitea 代码托管、Blnk 支付银行、Databag 即时通讯、SearXNG 搜索）共 72 个 MCP 工具函数，独立网络命名空间支持并行执行。
- **Vera-Bench 公开基准**：1,600 条可执行安全案例，覆盖 124 风险 × 77 攻击 × 30 环境，每条含 attack_plan、mcp_logs、trace.json、verify.py 四类产物，可在 benign/single/multi 三档设置下复现。
- **跨框架实证 + 下游 guard 模型**：在 4 个生产级 Agent 框架、5 个后端模型（GPT-5.2、Gemini-3、Qwen-3.7、Kimi-K2.6、GLM-5.2）上评测；并在 Vera 数据上微调 Qwen3Guard，得 0.930 acc / 0.903 recall / 0.941 F1，并迁移到外部 R-Judge 基准仍达 61.7% acc。

## Key Contributions

- **Executable safety case formalism**: defines σ = ⟨g, s₀, V_g⟩ — a concrete safety goal, a programmatically constructed initial state, and a deterministic verification predicate — explicitly anchored to *realized, observable* harmful effects rather than unsafe requests or stated intent.
- **Three-stage self-reinforcing pipeline**: (1) literature-driven exploration builds three hierarchical taxonomies (risks, attack methods, environments) from ~800 arXiv/OpenReview papers via create/update/merge/delete operators; (2) combinatorial composition across taxonomy leaves with compatibility filtering (39,078 → 1,600); (3) sandboxed adaptive execution whose verified outcomes feed back into risk exploration.
- **Two-tier threat model + configurable MCP tool gateway**: single-channel injects only user messages; multi-channel additionally transforms tool returns via identity/append/prefix/override. The adversary is constrained to runtime channels — it cannot modify the target model, system prompt, agent code, or internal tool code.
- **Manipulation-resistant verification hierarchy**: V_g = V_g^state ▷ V_g^tool ▷ V_g^resp prioritizes terminal environment state, then tool-call records, then agent self-report. Asymmetric judgment prevents false positives from agents that verbally comply without acting and false negatives from agents that refuse after already causing harm.
- **Production-scale sandbox infrastructure**: each run is a 12-container Docker Compose stack with five self-hosted real services (Mailpit, Gitea, Blnk, Databag, SearXNG) exposing 72 MCP tool functions; isolated network namespaces enable parallelism.
- **Vera-Bench**: 1,600 executable cases spanning 124 risk × 77 attack × 30 environment leaves, each shipped with attack_plan, mcp_logs, trace.json, and verify.py, runnable under benign/single-channel/multi-channel settings.
- **Cross-framework evaluation + downstream guard model**: tests four production agent frameworks across five backend models; fine-tunes Qwen3Guard on Vera data to 0.930 acc / 0.903 recall / 0.941 F1, with transfer to the external R-Judge benchmark (61.7% acc, beating LlamaGuard3, NemoGuard, Qwen3Guard, BraveGuard).

## Methodology

**Stage 1 — Continuous Risk Exploration (§IV-A).** A *Summary Agent* iteratively grows three hierarchical taxonomies from broad search concepts Q⁽⁰⁾ seeded by public agent-safety literature. Each iteration retrieves documents P⁽ᵗ⁾ and applies four operators to each taxonomy node: *create* (new leaf supported by ≥5 papers/scenarios), *update* (extend evidence set), *merge* (unify terminological variants), *delete* (remove unsupported nodes). Taxonomies therefore converge rather than grow monotonically. Result: **124 leaf risks, 77 attack methods, 30 environments**, processed from ~800 papers. Scope is restricted to inference-time-testable risks (training-phase attacks like fine-tuning poisoning are excluded).

**Stage 2 — Executable Test Case Construction (§IV-B).**
1. *Goal generation*: an LLM goal composer G maps each taxonomy tuple (r, m, e) to a context-specific safety goal g, validated by a typed schema (minimum length, must identify concrete violation + resource + sufficient conditions). To avoid the degenerate Cartesian product, Vera samples 10 environments per (risk, attack) pair and retains only those where the environment exposes the required resources; goals are normalized/deduplicated by semantic signature.
2. *Compilation*: an LLM initializer synthesizes s₀ (e.g., a password-bearing email for credential theft, a vulnerable dependency for supply-chain attacks) and V_g (deterministic Python predicate over τ and s_T), both recorded as replay scripts so s₀ is reproducible despite being LLM-authored.
3. *Filtering & variants*: cases whose success depends on internal reasoning traces are dropped; each retained base scenario is instantiated as three controlled variants — benign (no adversarial content, functional baseline), single-channel, multi-channel.

**Stage 3 — Adaptive Execution & Evidence-Grounded Verification (§IV-C).**
1. *Sandboxed execution environment*: a per-agent adapter translates framework events into a unified trajectory representation τ = ⟨uᵢ, aᵢ,ⱼ, f̃ᵢ,ⱼ, fᵢ,ⱼ, rᵢ⟩; an isolated sandbox per run contains the agent, the MCP tool gateway, and required services. Under multi-channel mode the gateway applies transformation operator Iᵢ,ⱼ ∈ {identity, append, prefix, override} to tool results while keeping the original fᵢ,ⱼ for ground truth. Three evidence streams are logged: interaction log, gateway log, environment state.
2. *Adaptive test driver*: the Control Agent maintains a control state cᵢ aggregating the safety case, history, strategy summaries, observed tool calls, registered injection rules, and a progress estimate; its policy π_ctrl selects (uᵢ, Jᵢ). It first establishes a legitimate task context, then escalates adversarial intent, reformulating on refusal, redecomposing on unexpected tool use, and repositioning injection when content isn't retrieved. The observe–adapt–act loop runs until budget exhaustion (≤10 turns), evidence production, or no further progress.
3. *Evidence-grounded verification*: V_g = V_g^state(s_T) ▷ V_g^tool(τ) ▷ V_g^resp(τ), where ▷ means "use a if its predicate is defined, else fall back to b". Verifiers are deterministic Python programs queried against the still-active sandbox; on Control-Agent-reported success they must confirm against environment evidence (asymmetric judgment). Verifiers that syntax-error or schema-mismatch are auto-regenerated to avoid false negatives.

**Experimental setup.** Agents: OpenClaw, Hermes, Codex, Claude Code. Backend models: GPT-5.2, Gemini-3, Qwen-3.7, Kimi-K2.6, GLM-5.2. Each base scenario runs under benign/single/multi settings. Metric: Execution Success Rate (ESR) = fraction of non-infrastructure-failing runs that satisfy the executable predicate (for benign runs: legitimate task completion; for attacks: case-specific safety violation).

## Results & Highlights

**Headline attack success (Table IV).** Per-framework overall ESR: Claude Code 88.6% > Hermes 86.6% > Codex 84.1% > OpenClaw 70.3%. By attack mode:
- Benign (legitimate task completion): 70.5% avg — Claude Code 80.1, Codex 69.1, OpenClaw 58.0, Hermes 74.8.
- **Single-channel ASR: 90.6% avg** (Claude Code 95.2, Codex 91.1, OpenClaw 82.8, Hermes 93.4).
- **Multi-channel ASR: 93.9% avg** (Claude Code 93.1, Codex 95.8, OpenClaw 89.1, Hermes 97.8).

**Adaptive driver effect.** Benign → single-channel gap = **+20.1 points** (purely from the Control Agent's adaptive user messages), largest for OpenClaw (+24.8) and smallest for Claude Code (+15.1).

**Multi-channel incremental gain** = +3.3 points avg, but with striking per-agent differences: Codex +4.7 (safety primarily at user-message parsing), Claude Code **−2.1** (filters that better detect injected tool-result content trigger extra refusals), OpenClaw +6.3 (front-end checks bypassable via tool-observation channel).

**Risk × environment coverage (Tables I–III, Fig. 2).** All 8 risk groups are executable in all 10 environment groups; ESR ranges 79.0% (Harmful Output) to 95.3% (Integrity). Every cell in Tables I–II exceeds 40%. Profile Infer (89.9%), Task Decomposition (88.5%), and Obfuscation (87.9%) transfer most broadly; Social Engineering (74.4%) and Format Inducement (78.4%) transfer least.

**Capability–vulnerability alignment.** More capable agents (Claude Code, Hermes — richer tool orchestration, longer context) are *more* susceptible to adaptive multi-turn attacks: the same capabilities that make them useful make them more compliant with adversarial instructions embedded in plausible workflows.

**Cost profile (Fig. 3).** Median run: 155k input tokens, 3k output tokens, 11 tool calls; p95: 789k input, 11k output, 38 tool calls — most scenarios tractable, with a long-tail of tool-intensive interactions.

**Downstream guard model (Figs. 4–5, Table V).** Off-the-shelf guards transfer poorly to Vera-Bench (LlamaGuard3: 0.438 acc / 0.258 recall / 0.310 F1; Qwen3Guard: 0.670 / 0.468 / 0.637; AgentDoG: 0.490 / 0.742 / 0.643). Fine-tuned Qwen3Guard reaches **0.930 acc / 0.903 recall / 0.941 F1** (+26.0 / +43.5 / +30.4 over base), and transfers to external R-Judge with **61.7% acc** — beating all off-the-shelf baselines, suggesting Vera-derived supervision is not benchmark-overfit.

**Public artifacts**: code at https://github.com/Yunhao-Feng/Vera; Vera-Bench (1,600 cases) released with full execution artifacts.

## Critical Assessment

- **Novelty — high.** The framing of *executable safety cases* (⟨g, s₀, V_g⟩) with a manipulation-resistant verification hierarchy and an adaptive Control Agent that establishes legitimate context before escalating is a genuine advance over static prompt/trajectory benchmarks. Treating risks/attacks/environments as orthogonal, composable dimensions — and showing the composition produces non-degenerate, broad-coverage tests (no zero-cells, all >40% ESR) — is a strong empirical validation of the core idea.
- **Rigor — strong.** Asymmetric judgment (auto-fail on Control-Agent failure, require verifier confirmation on claimed success) directly counters the inflationary bias typical of red-teaming papers. The verification hierarchy is principled and well-motivated. The benign-baseline ESR serves as an implicit quality metric for s₀/V_g calibration — a thoughtful design.
- **Reproducibility — very good.** Public repo, public benchmark, deterministic verifiers, recorded replay scripts, and isolated Docker stacks. The main barrier is the 12-container stack cost and the 5 backend LLMs (GPT-5.2, Gemini-3, etc.), some of which are proprietary — exact agent-backend reproduction may be partly constrained.
- **Significance — high if claims hold.** 93.9% multi-channel ASR on production frameworks is alarming; the "capability–vulnerability alignment" finding is actionable and broadly credible.
- **Red flags / things to scrutinize:**
  1. **Headline inflation risk.** The 93.9% number is the multi-channel (strongest) setting; single-channel is 90.6% and benign task-completion is only 70.5%. More importantly, the *asymmetric verifier design* structurally advantages attacks: benign verifiers must check *multi-predicate end-to-end correctness* of s_T, whereas attack verifiers need only confirm a *single* harmful state change. The authors acknowledge this ("by design"), but it means cross-setting ESR comparison is not apples-to-apples and the benign number should not be read as "agents are only 70% capable."
  2. **Control-Agent confound.** The +20.1-point benign→single-channel jump conflates *attack effectiveness* with *test-driver competence* — a stronger Control Agent raises ESR regardless of agent weakness. There is no ablation against a non-adaptive (fixed-script) attacker to isolate the adaptive component's marginal contribution beyond the qualitative arguments.
  3. **Selection bias in risk discovery.** Stage 1 mines academic literature for known risks; emerging/novel operational risks (zero-days, supply-chain) are by construction under-represented, though the authors note the mechanism could ingest CVE/MITRE ATT&CK feeds (not actually done here).
  4. **Guard-model evaluation is thin.** Fine-tuning is on Vera data and the external R-Judge transfer is a single benchmark with 61.7% acc — modest. No held-out Vera split details, no comparison against guard models *also* fine-tuned on competing benchmarks, so "Vera supervision generalizes better" is suggestive, not established.
  5. **OpenClaw's low scores partly reflect infra failures** (Benign ESR 58.0%), so its lower attack ESR is contaminated by execution reliability rather than purely safety strength — cross-framework ranking should be interpreted cautiously.
  6. **Future-dated artifacts.** OpenClaw/Hermes are 2026 frameworks and several backend models (GPT-5.2, Gemini-3, Qwen-3.7, Kimi-K2.6, GLM-5.2) are forward-dated relative to a mid-2026 submission; readers should verify these are stable, publicly available references rather than internal/pre-release versions.

## Strengths

- Principled, modular, *self-reinforcing* pipeline that is designed to track an evolving risk surface rather than ossify into a static list.
- Manipulation-resistant, asymmetric verification is a genuinely better safety oracle than model self-report or LLM-judge rubrics.
- Real production frameworks + real self-hosted services (not toy mocks) make the ASR numbers operationally meaningful.
- Comprehensive 3-way cross-tabulation (risk × environment × attack) exposes *where* and *why* attacks transfer, not just aggregate scores.
- Public, replayable artifacts lower the bar for community scrutiny and reuse.
- The benign setting doubles as a calibration probe for the generation pipeline — elegant.

## Limitations

- Asymmetric verifier design makes benign vs. attack ESR not directly comparable; readers may over-read the gap as a pure capability-vs-vulnerability contrast.
- No ablation isolating the adaptive Control Agent's contribution versus a fixed/scripted attacker.
- Risk discovery is literature-bounded; operational/zero-day risks under-covered (CVE/ATT&CK ingestion is proposed, not implemented).
- Guard-model transfer evidence is single-benchmark and thin; no head-to-head against guards fine-tuned on InjecAgent/AgentDojo/ASB.
- Per-agent rankings are confounded by infrastructure reliability (esp. OpenClaw).
- The 12-container sandbox and multi-LLM backend requirement make independent full reproduction non-trivial.
- 10-turn interaction budget caps long-horizon attacks; right-tail multi-step campaigns may be under-probed.

## Relevance

与对 LLM Agent 安全评测、可执行红队基准与对抗鲁棒性的兴趣高度契合——Vera 提供了从风险发现到证据验证的端到端流水线、公开可复现的 Vera-Bench，以及"能力-脆弱性对齐"这一可指导防御设计的实证发现，可直接作为后续 Agent 安全评测与防御研究的对照基线。

## Questions for Further Exploration

- What is the marginal ASR contribution of the *adaptive* Control Agent versus a non-adaptive scripted attacker with the same scenarios and budget?
- How do Vera-Bench cases overlap with — and uniquely extend — InjecAgent, AgentDojo, ASB, and ATBench in terms of concrete risk categories?
- Can the deterministic verifiers V_g be evaded by agents that forge or obscure observable artifacts (e.g., rollback environment state, obfuscate tool-call arguments)?
- Does the capability–vulnerability alignment persist under targeted safety training, or can defense fine-tuning decouple capability from susceptibility?
- How stable are the per-agent ESR rankings once OpenClaw's infrastructure-failure rate is controlled (e.g., re-normalizing ESR by non-infra-failed runs only)?
- What is the maintenance cost (human + LLM) of keeping the three taxonomies current as new tools, MCP servers, and attack techniques emerge?
- Could the literature-driven explorer be replaced/augmented by an autonomous vulnerability-mining agent that probes live services for novel risks, closing the zero-day coverage gap?
