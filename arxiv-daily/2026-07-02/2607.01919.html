01 / METHOD

Two add-on components, commit-before-finalize

Protocol-agnostic augmentation; instantiated for MCP hosts and CLI-based agents.

Context Guard CG · inside TEE

Runs inside a TEE on the MCP Host, mediating the planning/execution path. Before each query it recomputes the contextual-state digest and verifies it against SCM's latest authorized digest.

  • commit-before-finalize: commit → seal pending → finalize side effects only after a valid SCM receipt → append local ledger atomically (crash-consistent).
  • Fail-closed: any digest / id / signature mismatch aborts the query.
  • Trusted and non-bypassable (assumption A4).

State Continuity Module SCM · Nimble + Intel TDX

A remote, replicated TEE-backed service exposing a linearizable abstraction with two primitives — enforcing strictly monotonic, gap-free identifiers.

  • Commit(ℓ, id, h)  ·  GetLatestState(ℓ, n)
  • Append-only ledger; rejects rollback / fork attempts.
  • Signs receipts bound to a fresh read-challenge nonce; TDX remote attestation binds SCM identity to the TLS channel.
  • Built on Nimble (chosen over SGX to avoid enclave refactoring).

Algorithm 5 · Three-phase workflow (fail-closed)

1
State initialization

TOFU ApprovePolicy, compute h0, anchor in SCM, snapshot C0 into sealed store. Also serves as the recovery primitive.

2
State verification / query

Fresh nonce read-challenge to SCM, signature + digest + ledger-id equality check; any mismatch aborts.

3
State update

Side-effect-free PreviewApply derives hnext; commit to SCM → seal → finalize → optional snapshot → append local ledger.

05 / ASSESSMENT

Strengths vs. Limitations

Solid, honestly-written; impact hinges on TEE availability.

▲ Strengths

  • Unifying threat model — ties tool poisoning, memory poisoning, and rollback to one root cause, with a precise scoping of "contextual state."
  • Deterministic, proof-based defense instead of a brittle learned detector — a categorical improvement over MindGuard / SecureMCP / A-MemGuard.
  • Formal analysis with explicit assumptions (A1–A6), lemmas, corollaries, and the 14-vector catalog.
  • Low, well-measured overhead (1.02×–1.04×) with sound design: commit-before-finalize, nonce-bound proofs, crash-consistent ledger.
  • Protocol-agnostic — demonstrated for both MCP and CLI deployments.
  • Authors credit prior art (Nimble, ElephantDP) rather than overclaiming.

▼ Limitations

  • Toy-scale tasks only — the large-context / large-memory regime where digest & snapshot costs actually matter is untested; no scaling analysis.
  • No end-to-end attack demo — relies on the valid but unsatisfying "deterministic by construction" argument.
  • Naive TOFU trust root in the prototype; vulnerable at anchor time.
  • Strong hardware dependency — requires Intel TDX-capable CPUs and a replicated Nimble SCM.
  • Liveness sacrificed for safety (fail-closed); a host adversary can brick the agent or force stale recovery.
  • Out of scope: TEE side-channels, LLM-channel compromise, in-band prevention (only post-hoc recovery). No code released.
06 / OPEN

Questions for further exploration

Where the formal story meets real deployment.
Q1

How do per-query digest recomputation and snapshot storage cost scale with context size, memory depth, and tool count — the realistic large-context regime the paper avoids?

Q2

Can the TOFU trust root be replaced by signed manifests or a transparency log of approved tool descriptors without breaking usability?

Q3

What is the operational cost and failure behavior of the replicated Nimble SCM under realistic churn, partitions, and partial TEE compromise?

Q4

Can Historical Traceability's recovery handle irreversible external side effects (e.g., an already-executed financial transaction), or only in-agent state?

Q5

How does ElephantAgent compose with LLM-channel encryption and runtime prompt-injection detection to close the remaining out-of-scope vectors?

Q6

Is there a path to a non-TEE instantiation (blockchain / transparency-log-backed SCM) that broadens deployability while preserving linearizability?
