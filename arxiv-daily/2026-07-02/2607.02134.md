> **一句话总结：** 作者提出 Paper-replication 工作流（实现为 Codex/Claude Code 的 coding-agent skill），通过"持久化工作区 + 外部验证检查"（harness engineering）将每条论文计算性声明设为带证据包的复制目标，在 4 篇 SciML 论文 × 3 次独立运行 = 12 次运行中全部 158 个目标通过完成门控；同时用层次贝叶斯模型量化了目标划分、数值保真度、耗时、纠错工作与判定规则上的运行间差异。

> **TL;DR:** The authors reframe paper replication as a *target-level evidence contract* enforced by harness engineering rather than free-form prompting: each computational claim becomes a "target" requiring an evidence bundle (generated output + run record + provenance + comparison + report coverage), and completion is a *workspace state* gated by external validation checks—**not** the agent's final message. Implemented as a reusable coding-agent skill (SKILL.md + `scripts/paper_replication.py`) and run with Codex/GPT-5.4, all 12 independent runs across 4 SciML papers (PIFT, PINN-I, PINN-II, SINDy) pass the completion gate with all 158 targets matched. Honest hierarchical-Bayesian analysis then shows substantial residual variance: target-decomposition ratios up to 3.1×, PINN-II discrepancies varying by ~14× across reruns, elapsed times from 1.2–13 h, and acceptance-rule agreement ranging from 0.46 (PINN-II) to 0.95 (SINDy). Worth reading if you care about agent reliability, reproducibility, or harness design—skip if you want a new ML model.

## 核心贡献

- **把论文复制重新定义为"目标级证据契约"**：每个计算性声明（图、表、报告标量、习得场、分布、轨迹、稀疏支撑等）都成为一个"目标"(target)，必须用完整证据包 `E_j = (ŷ_j, R_j, P_j, C_j, G_j)`（候选输出、运行记录、来源、对比、报告覆盖）来满足。
- **提出"持久化工作区 + 外部验证检查"的 harness engineering 机制**：将进度状态存于工作区文件（manifest、reproduction matrix、task ledger、specification files、run/provenance 记录、报告）而非聊天记录；验证检查在 agent 之外运行，使"完成"成为可检验的工作区状态。
- **形式化的完成门控公式**：`V_complete = V_spec ∧ V_progress ∧ V_report ∧ (∀j s_j=MATCHED) ∧ [a=∅] ∧ [report PDF exists]`，从根本上杜绝"agent 自述完成"。
- **实现为可移植的 coding-agent skill**：包含 SKILL.md 指令层（适配 Codex 与 Claude Code）与 `scripts/paper_replication.py` 工作区工具层，且区分"论文提供资产"与"生成输出"（哈希校验防止把论文原图/表当作复制结果）。
- **设计并应用"重复运行案例研究"分析框架**：用层次贝叶斯模型（Gamma-Poisson 目标计数、Normal headroom、log-normal 耗时）量化完成性、目标覆盖、论文锚定数值保真度、努力量与判定差异。
- **诚实披露运行间差异**：即使所有工作区都"完成"，仍报告目标划分比、数值保真度波动、耗时分布、被替换执行次数与接受规则一致性。

## Key Contributions

- **Reframes replication as a target-level evidence contract.** Each computational claim becomes a *target* whose satisfaction requires linking a generated output to the paper claim, the agent's reconstruction of the method, a successful run, provenance, comparison evidence, and report coverage. An output artifact alone does **not** count as evidence.
- **Harness engineering via persistent workspace + external validation checks.** Rather than relying on prompt instructions, the workflow changes the *environment*: state is stored in workspace files, and validation checks run *outside* the agent before any target can be marked MATCHED. This directly targets known failure modes where LLMs cannot reliably self-verify or where prompts degrade across long horizons.
- **A formal completion gate** (`V_complete`, Eq. 3) that makes completion a workspace state—every target matched, no active target, report PDF exists, and three check families (specification, progress, report-coverage) pass. Completion is *not* a statement in the agent's final message.
- **Reusable coding-agent skill implementation.** Two layers: instruction layer (`SKILL.md` with separate Codex/Claude Code prompt files) and workspace utilities (`scripts/paper_replication.py`). Separates paper-provided assets from generated outputs and uses hash checks to detect copied figures/tables/TeX source.
- **Claim-type-aware acceptance rules.** Numeric targets use a discrepancy metric + tolerance; distributional targets compare distributional properties (mean, variance, quantiles, coverage, modes); structural targets check patterns (sparsity, ordering, stability, attractor geometry); visual targets require visual comparison only when the claim demands it.
- **A repeated-run case-study analysis** with explicit hierarchical Bayesian models quantifying target coverage, paper-anchored numeric fidelity (headroom), elapsed time, correction work (superseded executions), and judgment variation (acceptance-rule agreement).
- **Released artifacts**: skill for both Codex and Claude Code, prompts, all 12 agent-generated workspaces, and analysis scripts (GitHub: PredictiveScienceLab/paper-replication-paper).

## Methodology

### Core workflow (Paper-replication)
The agent receives a prompt specifying the paper's LaTeX source, datasets, compute environment, and policy choices (author code **not** allowed in this work). The workflow then enforces:

1. **Source inspection** — Build `spec/paper_inventory.json` from the main TeX file, recording included files, bibliography, appendices, figure assets, data references. Paper figures are indexed under `artifacts/paper_figures/` with hashes; generated outputs go separately to `artifacts/figures/`, `artifacts/tables/`.
2. **Target extraction** — Record each computational claim in a *reproduction matrix* with: paper location, supporting source/assets, data & method component, required output, judgment rule, status, and report location.
3. **Method reconstruction** — Specification files record the agent's reconstruction, plus *recorded hypotheses* for every under-specified detail (seed, tolerance, preprocessing, plotting).
4. **Single-active-target discipline** — The task ledger names exactly one active target; a validation check rejects multiple active targets or no active target while work remains. Forces closing one evidence contract before moving on.
5. **Run recording** — Every substantive execution passes through a run recorder storing command, working directory, start/finish times, success, messages/errors, expected outputs, output hashes (under `artifacts/runs/`). Failed/replaced runs are *kept*, preserving the trial-and-correction path.
6. **Provenance** — For each candidate output, `artifacts/provenance/` links it to implementation file, config, seed, method components, and the paper passages/equations justifying the implementation. Hashes prevent silent post-hoc edits.
7. **Comparison + report coverage** — Comparison `C_j` judges `ŷ_j` vs `y*_j` under the target's recorded acceptance rule; `G_j` records where the result appears in `report/main.tex`.

### Evidence bundle and completion gate
For target `t_j`, candidate `ŷ_j = F_j(D_j; θ_j, ω_j)`, evidence bundle `E_j = (ŷ_j, R_j, P_j, C_j, G_j)`. Completion gate:

`V_complete = V_spec ∧ V_progress ∧ V_report ∧ (∧_j [s_j = MATCHED]) ∧ [a = ∅] ∧ [report PDF exists]`

### Validation checks (external to the agent)
- **Paper-asset checks**: path/hash separation rejects copied paper material.
- **Active-target check**: ledger and matrix must agree.
- **Specification check**: manifest, ledger, inventory, hashes, spec files, matrix all present and complete.
- **Progress/provenance check**: matched targets must have run-generated provenance and the comparison evidence required by their acceptance rule; method provenance must trace implementation to the paper's method (not just output similarity).
- **Report-coverage check**: matched figure/table targets appear in `report/main.tex`; report PDF must exist.

### Skill implementation
- Instruction layer: `SKILL.md` + agent-specific prompt files (Codex, Claude Code).
- Workspace utilities: `scripts/paper_replication.py` (init/sync commands, source inspection, run recorder, provenance writer, validation checks).

### Evaluation (case-study analysis)
- **Papers**: PIFT (physics-informed information field theory), PINN-I (data-driven solutions), PINN-II (data-driven discovery), SINDy (sparse identification of nonlinear dynamics). Chosen to require diverse target evidence (learned fields, errors, coefficients, posterior structure, sparse support, trajectories, figures).
- **Design**: 3 independent runs per paper (N=4, R=3 → 12 runs), same materials/prompts/tools/compute/no-author-code policy, workspaces share no code/state.
- **Agent**: Codex with GPT-5.4, Extra High reasoning. Hardware: MacBook Pro M4 Max (128 GB) + Purdue Gautschi cluster (CPU 192-core/384 GB nodes; GPU nodes 8×H100, 112 CPU cores, ~1 TB). PIFT/SINDy run locally; some PINN targets use cluster.
- **Continuation**: fixed queue of 10 follow-up prompts (resume workspace, run status check, continue) because long agents may stop early.
- **Statistical models** (fit in NumPyro, NUTS, 4 chains × 2000 draws):
  - Target coverage: Gamma-Poisson (Eq. 4).
  - Headroom `h_pcr = log10(τ_pc / ed_pcr)` (Eq. 5): Normal model with paper-, claim-, and run-level terms (Eq. 6); positive headroom = inside fixed threshold.
  - Effort `log10 H_pr`: Normal with paper-level term (Eq. 7).
- **Paper-anchored thresholds** (fixed across runs, independent of agent-recorded tolerance): PINN-I relative L2 ≤ 10⁻²; PINN-II coefficient % error ≤ 10%; SINDy Lorenz coefficient relative error ≤ 10⁻³. PIFT has no scalar anchor → distributional/structural only.

## Results & Highlights

- **Completion**: 12/12 workspaces pass the gate; **158/158 recorded targets MATCHED with report coverage.** Completion is stable across runs; decomposition, effort, and execution work are not.
- **Target coverage (decomposition variation)**:
  - PINN-I: 8/8/8; SINDy: 20/20/20 (no variation, ratio 1.0).
  - PIFT: 8/8/25 (ratio **3.1**); PINN-II: 9/9/15 (ratio **1.7**).
  - Extra PIFT targets = separate targets for panels of a composite figure; extra PINN-II targets = appendix inference claims. Differences are *decomposition choices*, not incomplete work.
- **Paper-anchored numeric fidelity** (13 scalar anchors; PIFT excluded—no scalar anchor):
  - PINN-II has the largest run-residual scale `σε,p`: median **1.16**, 95% CI [0.86, 1.66] → ~**14× discrepancy variation** across reruns.
  - PINN-I: median 0.41 [0.26, 0.76]; SINDy: median 0.31 [0.11, 1.60] (wide—only one SINDy anchor).
  - For non-scalar targets: PINN tables preserve accuracy *patterns* across data size/collocation/noise even when cell digits differ; SINDy recovers exact sparse support in every run; cylinder-wake target supports quadratic structure/geometry but not digit-level coefficients (paper omits simulation trajectories/preprocessing).
- **Effort / elapsed replication time** (posterior medians): PINN-II **6.9 h** [3.0, 13.4]; PINN-I **5.0 h** [2.5, 9.9]; PIFT **2.2 h** [1.1, 4.4]; SINDy **1.9 h** [1.0, 4.3]. Observed range 1.2–13.0 h. P(PINN paper longer than non-PINN) = 0.947–0.972.
- **Correction work**: 25 superseded tracked executions total — PINN-I 11, PINN-II 10, PIFT 3, SINDy 1. PINN-I run 1 alone had 71 tracked executions (11 superseded) vs runs 2&3 with 12 each (0 superseded). Most churn is in PINN optimization/training.
- **Judgment variation** (same acceptance-rule-type fraction): SINDy **0.95** (19/20), PIFT 0.73 (8/11), PINN-I 0.50 (4/8), PINN-II **0.46** (5/11). PINN papers vary most—same claim sometimes classified numeric, sometimes structural.

## Critical Assessment

**Novelty (genuine):** The conceptual move—making completion a *workspace state gated by external validation* rather than an agent self-report—is simple, well-motivated, and addresses a real and repeatedly documented failure mode (LLMs cannot reliably self-verify; long-horizon agents drift). It is a *protocol/harness* contribution, not a new model, but it is the right kind of contribution for the reproducibility problem. The target-evidence-bundle formalism and the claim-type-aware acceptance rules are non-trivial and well-engineered. The honest hierarchical-Bayesian decomposition of residual variance is a strength rarely seen in agent-benchmark papers.

**Rigor (strong, with caveats):**
- The authors are admirably careful to *separate* "workspace MATCHED" (under the agent-recorded acceptance rule) from "paper-anchored numeric fidelity" (fixed external threshold). This is the intellectually honest move and they make it explicitly. The 100% pass rate is explicitly qualified as "relative to the recorded target set and validation checks"—they do not overclaim that every aspect of each paper was replicated.
- Statistical models are properly Bayesian with priors justified; NUTS, 4 chains, 2000 draws is a reasonable setup.
- *However*, the headline "158/158 targets matched" should not be read as "all numbers reproduced within tolerance." The paper-anchored scalar analysis (13 anchors) is the stricter test, and it shows PINN-II discrepancies swing by ~14× across reruns—i.e., a target can be MATCHED while the reproduced number is far from the paper's. The MATCHED label is a *process/coverage* metric, not a fidelity metric, and the authors say so—but a casual reader of the abstract will miss this.

**Reproducibility (excellent):** Skill, prompts, all 12 workspaces, and analysis scripts are released for *both* Codex and Claude Code. This is the gold standard for an agent-tooling paper. The workspaces themselves are inspectable artifacts.

**Significance (moderate-to-high in niche):** The "completion by evidence, not by message" principle and the target-evidence-bundle pattern generalize well beyond SciML to any agentic task where verifiability matters. The harness-engineering framing is timely.

**Red flags / things to scrutinize:**
- **No ablation against an unstructured prompt.** The authors explicitly state this: "It does not include an ablation without the skill, so it characterizes what Paper-replication produces rather than estimating the workflow's effect on an unstructured prompt." This is the biggest methodological gap—you cannot tell from this paper *how much* the skill improves over naive prompting. The motivation rests on citing prior work about prompt-only failures rather than demonstrating the contrast.
- **Self-assessment circularity.** The *same agent* (a) records acceptance rules, (b) generates evidence, and (c) judges comparisons. External validation checks enforce *structure and provenance* but do **not** independently re-derive the numbers. The authors acknowledge checks "do not prove that the agent's reconstruction is the only implementation consistent with the paper." Acceptance-rule agreement as low as 0.46 (PINN-II) means the agent is effectively choosing its own grading rubric run-to-run.
- **Narrow corpus.** 4 SciML papers, all classical (PINN/SINDy/PIFT era ~2016–2023), single agent/model/setting (Codex + GPT-5.4 Extra High). No generalization to LLM-training papers, systems papers, or modern large-scale experiments. Between-paper variance is explicitly under-identified.
- **Threshold sensitivity.** Only 13 scalar anchors; PIFT has none. The 10% PINN-II threshold is an "analysis convention" tied to the paper's reported robustness scale, and the authors note a different defensible threshold could change how many anchors fall inside.
- **Single judge per distributional/structural target** — inter-rater reliability of the structural/distributional comparisons is future work.
- **Compute asymmetry** baked into results: PIFT/SINDy local, PINN cluster—confounds cross-paper effort comparison (acknowledged: "not that the PINN papers are intrinsically harder").

## Strengths

- Clear, well-motivated problem with a concrete, non-trivial mechanism (target-evidence bundle + external validation gate).
- Shifts completion from natural-language self-report to inspectable workspace artifacts—a genuinely useful primitive for agentic reliability.
- Claim-type-aware acceptance rules (numeric / distributional / structural / visual) show real engineering thought, not a one-size-fits-all metric.
- Rigorous, honest statistics: hierarchical Bayesian models with properly separated completion vs. fidelity analyses.
- Transparent reporting of residual variance (decomposition ratios, ~14× discrepancy swings, 0.46–0.95 rule agreement) rather than only the rosy 100% headline.
- Exemplary artifact release (dual-agent skill, prompts, 12 full workspaces, analysis code).
- Honest limitations section that names the missing ablation and the under-identified between-paper variation.

## Limitations

- **No ablation** against direct prompting → cannot quantify the skill's marginal benefit.
- **Small, classical corpus** (4 SciML papers); generalization to other ML subfields untested; between-paper variance under-identified.
- **Self-assessment circularity**: the agent sets acceptance rules, produces evidence, and judges it; external checks enforce structure/provenance, not numerical truth.
- **MATCHED ≠ reproduced-within-tolerance.** A target can pass while the reproduced number diverges substantially (PINN-II ~14× swing).
- **Few scalar anchors (13)**; PIFT has none; threshold choices are conventions.
- **Single agent/model/setting** (Codex + GPT-5.4 Extra High); no sensitivity to model or reasoning level.
- **Single judge** for distributional/structural targets; inter-rater reliability unmeasured.
- Compute asymmetry across papers confounds effort comparisons.
- Niche scope: only papers with explicit computational claims; qualitative/empirical-data-collection claims out of scope.

## Relevance

与"agentic workflows / coding-agents 自动化科研"和"科学机器学习可复现性"两条主线都高度相关：它把"完成判定"从 agent 自述转为可验证的工作区状态，并提供跨多次运行的实证与统计刻画，正是评估 coding-agent 在科研自动化中可信度的核心议题；其 target-evidence-bundle 与外部验证门控的设计模式可直接迁移到其他需要可验证性的 agentic 任务。

## Questions for Further Exploration

- How large is the gap between Paper-replication and a matched unstructured-prompt baseline (the missing ablation)? Does the skill primarily improve completion rate, fidelity, or just inspectability?
- Can the workflow be extended to *detect genuine non-reproducibility* (i.e., legitimately fail targets), or does the gate bias toward "completed"? What is the false-MATCH rate?
- How does the ~14× PINN-II discrepancy swing decompose into prompt sensitivity, training stochasticity, and underspecified implementation choices?
- Does replacing the self-judged acceptance rule with an independent verifier (or a second agent) materially change which targets pass?
- How does the approach scale to non-SciML domains—LLM training runs, systems papers, or papers where the central claims are qualitative?
- Could the target-decomposition variance (ratios up to 3.1×) be reduced by a canonical target-schema, and would that improve cross-run fidelity comparability?
- What is the cost/quality trade-off across agent models and reasoning settings (only GPT-5.4 Extra High tested)?
- Is the skill composable with human-in-the-loop checkpoints to reduce self-assessment circularity without losing automation?
