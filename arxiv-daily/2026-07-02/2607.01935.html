Methodology — a state-alignment overlay, not a new substrate

A-TMA wraps an existing host (Graphiti/Zep, A-Mem, InsideOut, SeCom…) along three decoupled layers, preserving history for auditability rather than overwriting it.

BANK layer 1

Sentry + Judge maintenance

A lightweight dual-head Sentry (nomic-embed + topic/logic heads) proposes state-sharing candidates; a Judge (Qwen2.5-3B, QLoRA-SFT then GRPO) commits structured verdicts — marking records active→superseded and adding typed links, while coexisting facts stay active.

RETR layer 2

State-aligned retrieval

A query profiler assigns a state view; the host seed set is expanded via relation + semantic hops into pool C_q. A bounded controller (≤3 of ≤10 candidates) may reorder but is constrained to C_q — it cannot retrieve missing or invent facts.

QA layer 3

QA state conditioning

A deterministic serializer tags each evidence line with a role λ ∈ {cur, hist, tran, link, raw}; a rule set constrains the generator (qwen2.5:3b, k=5) to follow the requested view — current memory is active state unless the question asks about the past.

Assessment — strengths vs. limitations

+
Strengths
  • Clean problem framing. Separates a genuinely under-measured failure mode from generic RAG / retrieval quality.
  • Non-destructive design. Preserves history for auditability rather than overwriting — valuable for trustworthy assistants.
  • Rigorous benchmark construction. Documented leakage prevention, structural validators, human verification (IAA up to 100%).
  • Methodological discipline. Decoupled bank/retrieval/QA metrics, a bounded controller that cannot invent facts, explicit ablation.
  • Intellectual honesty. Results repeatedly framed as host- and metric-dependent with a "bounded conclusion," not overclaimed.
−
Limitations
  • Absolute capability stays low. LoCoMo temporal F1 only 0.1705 even after gains — the problem is far from solved.
  • Narrow ablation. 3 profiles on a single host (A-Mem); component-level contribution across hosts unverified.
  • Benchmark co-design risk. LTP is LoCoMo-derived with only 10 profiles; validators reduce but don't eliminate the concern.
  • Single 70B judge with no reported judge-reliability analysis or human-vs-judge calibration.
  • Small QA model (qwen2.5:3b) — some gains may reflect rescuing a weak reader, not fundamental memory improvement.
  • Added system complexity (Sentry + Judge + controller + traces) with unclear latency / cost at deployment scale.

Open Questions for further exploration

01

Do component contributions (Sentry/Judge/controller/QA-label) shift across hosts — does the dominant failure layer differ on Graphiti vs. InsideOut?

02

How sensitive are QA Acc and conflict accuracy to the choice of offline judge? Would a judge swap or calibration change rankings?

03

Do gains hold with a stronger answerer (e.g. 70B), or does it erase A-TMA's benefit by inferring state from raw text?

04

Is LTP robust to benchmark/method co-design? Could an independent, non-LoCoMo-derived benchmark replicate the ordering?

05

How does A-TMA handle ambiguous or partial transitions (contradictory updates, conditional/scope changes) vs. clean supersessions?

06

What is the latency / compute overhead of the Sentry+Judge pipeline and per-query evidence-packet construction in deployment?

07

Can A-TMA support transition-targeted QA — "what changed and when" — which is currently interface-only, not evaluated?

08

Does the bounded controller's constraint (i_m ∈ C_q) become a bottleneck when the host retriever's seed set is weak?
