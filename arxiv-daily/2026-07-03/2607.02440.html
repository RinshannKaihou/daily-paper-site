EvoPolicyGym: Evaluating Autonomous Policy Evolution in Interactive Environments

arXiv: 2607.02440

Date: 2026-07-02

https://arxiv.org/abs/2607.02440

一句话总结

EvoPolicyGym 提出了一个受控的"自主策略演化"评测设定：让 harness-model agent 在固定交互预算下反复编辑可执行的策略系统，从而区分 agent 能否真正把反馈迭代地转化为策略改进，而非只看最终分数或混入开放式软件工程进度。

TL;DR

The paper introduces Autonomous Policy Evolution, a controlled evaluation setting where a harness-model agent iteratively edits an executable policy system under a fixed interaction budget, and instantiates it as EvoPolicyGym, a benchmark of compact interactive RL environments. Beyond a leaderboard (where GPT-5.5 ranks highest), it offers trajectory-level diagnostics of how agents allocate budget and convert feedback into parametric tuning, showing that strong policy evolution depends on discovering task-appropriate mechanisms under bounded feedback rather than isolated task wins.

核心贡献

  1. 提出新的评测设定：定义"自主策略演化（Autonomous Policy Evolution）"，把"通过反馈改进可执行策略"这一过程单独抽离出来评估，避免被最终分数或开放式 SWE 进度所掩盖。
  2. 构建 EvoPolicyGym 基准：由紧凑的交互式 RL 环境组成的套件，专门评估 agent 在受限预算下迭代改进已探索策略的能力。
  3. 轨迹级诊断：不只给排行榜分数，还提供诊断指标，区分 agent 如何分配预算、如何把反馈转化为参数化调优。
  4. 核心洞见：优秀的自主策略演化不只依赖单次任务获胜，更取决于能否发现"任务适配的机制"并在有限反馈下精炼策略。

Key Contributions

  • A new controlled evaluation setting, Autonomous Policy Evolution, in which a harness-model agent repeatedly edits an executable policy system under a fixed interaction budget — isolating feedback-driven improvement from confounds like open-ended SWE progress or a single final score.
  • EvoPolicyGym, a benchmark suite of compact interactive RL environments designed to evaluate iterative policy improvement.
  • Trajectory-level diagnostics that go beyond leaderboard scores to characterize how agents allocate their interaction budget and turn feedback into parametric tuning.
  • An empirical insight: strong autonomous policy evolution depends not on isolated task wins but on discovering task-appropriate mechanisms and refining policies under bounded feedback.

Methodology

  • Autonomous Policy Evolution (the evaluated object). Unlike open-ended SWE benchmarks, EvoPolicyGym makes policy evolution itself the evaluated object: the agent repeatedly edits an executable policy system, submits it under a fixed interaction budget, and receives server-generated feedback from sandboxed rollouts. Train submissions return aggregate + trajectory-level feedback; validation and held-out cases remain server-side, so the agent must generalize, not memorize. The bounded budget is part of the capability being measured.
  • Core16 suite. The validated subset is Core16 — 16 environments across four families: Gym/Box2D, MuJoCo, MiniGrid, and Robotics/Driving. (A broader adapter surface — Classic Control, Toy Text, Atari/ALE, MiniWorld, HighwayEnv, Gymnasium-Robotics, MO-Gymnasium, BrowserGym, MiniWoB++, MetaWorld — is supported at the wrapper/interface-test level but not yet fully validated.)
  • Per-environment protocol. Each agent gets a 128-episode training budget, with 16 validation cases and 32 held-out cases reserved server-side. Final reported number is the validation-selected policy's mean return on held-out cases. Each leaderboard cell = one 128-episode optimization run for one model on one environment.
  • Harness–model agents (4 evaluated). The harness is part of the evaluation dimension: GPT-5.5 via the Codex harness; Claude Opus 4.7, MiniMax-M3, and DeepSeek-V4-Pro via the Claude Code harness. All agents face the same Core16 suite, case splits, submission interface, and scoring protocol.
  • Trajectory-level diagnostics. EvoPolicyGym records the full execution–feedback–revise trajectory, enabling comparisons of (i) how agents allocate budget across episodes, (ii) how they convert feedback into parametric tuning, (iii) how they diagnose failures, and (iv) audited-trace revisions — diagnostics that relate policy improvement to budget-conditioned policy evolution, not just final held-out score.

Results & Highlights

  • GPT-5.5 (Codex harness) achieves the highest Core16 aggregate rank score and top-two performance on all 16 environments — the only model with consistently strong cross-environment coverage.
  • Claude Opus 4.7 (Claude Code harness) leads the MiniGrid family, indicating environment-family-specific strengths rather than uniform dominance.
  • MiniMax-M3 and DeepSeek-V4-Pro achieve isolated task wins but substantially lower aggregate rank — i.e. they win specific environments but lack cross-environment generalization.
  • Empirical insight (the paper's central diagnostic finding): strong autonomous policy evolution depends not on isolated task wins but on discovering task-appropriate mechanisms and refining policies under bounded feedback — agents that win one environment but cannot transfer the mechanism score lower in aggregate rank.
  • The benchmark doubles as a diagnostic substrate for studying self-evolving agents under bounded budgets, exposing failure modes (retry loops, verification gaps, budget misallocation) that aggregate success rates hide.

Authors & Credibility

  • A 16-author team spanning multiple institutions (affiliations to be confirmed from the PDF). Names include researchers associated with LLM/agent evaluation work (e.g., Yafu Li, Yu Cheng, Derek F. Wong, Yang Yang).
  • The framing directly engages with a known weakness in current agent evaluations: collapsing iterative improvement into a single end score, or confounding it with open-ended SWE benchmarks.

Critical Assessment

  • The framing is well-motivated: existing benchmarks (SWE-style, final-score-only) genuinely fail to isolate the iterative, feedback-driven policy refinement capability that matters for autonomous agents.
  • The emphasis on trajectory-level diagnostics is a meaningful contribution over pure leaderboard benchmarks.
  • Risks of circularity: "policy evolution" could collapse into "agent that edits code well," so the strength of the setting depends heavily on how tightly the environments bind feedback to policy parameters (details pending from PDF).
  • Generalizability across domains (RL control vs. tool use vs. code) is unclear from the abstract alone.

Strengths

  • Cleanly isolates a capability (iterative policy improvement) that other benchmarks conflate or ignore.
  • Provides both a leaderboard and diagnostic axes for budget allocation and feedback conversion.
  • Compact RL environments make the setting reproducible and cheaper than full SWE-scale benchmarks.
  • Reports results across multiple frontier models, enabling relative comparison.

Limitations

  • Without the full paper, the precise definition of "policy system" and how feedback is generated/bounded is not verifiable.
  • Generalization beyond the 16 compact RL environments is uncertain.
  • The setting may still reward general code-editing competence over genuine policy-evolution skill.
  • Reliance on a "harness-model agent" introduces harness-design sensitivity that the abstract does not address.
  • Leaderboard dominated by GPT-5.5 may reflect model strength rather than benchmark discriminative power for the mid-tier.

Relevance (zh)

这项工作直击当前 LLM agent 评测的痛点：很多基准要么只看最终结果，要么与开放式软件工程任务混在一起，难以判断 agent 是否真正具备"基于反馈迭代改进策略"的能力。对于从事 agent 评测、RL-from-feedback、self-improving agents 以及自动化策略/代码优化的研究者，EvoPolicyGym 提供了一个可控、可诊断、预算受限的实验场。其"轨迹级诊断"思路对设计 reward shaping、process supervision 与 agent self-refinement 方法也具有参考价值。

Questions for Further Exploration

  1. "Policy system" 在 EvoPolicyGym 中具体指什么？是显式的可执行代码、参数化的 RL 策略，还是两者混合？反馈信号如何生成？
  2. 固定交互预算的设置是否对所有模型公平？更强的模型是否因更快收敛而获得不公平优势？
  3. 轨迹级诊断指标是如何定义和计算的？它们与最终分数的相关性如何？
  4. 在 16 个 RL 环境上观察到的结论，能否迁移到工具使用、网页导航或真实代码库改进等更开放的场景？
  5. GPT-5.5 在所有环境上的优势主要来自哪种能力——更快的反馈吸收、更稳健的探索，还是更强的初始策略？
  6. 如何防止 agent 通过"记环境"或过拟合单个 benchmark 来作弊？
