> **一句话总结：** 本文揭示了小语言模型在严格JSON输出格式下"答案正确但格式不可用"的系统性失败（NAIVE提示output accuracy为0%），并提出AloLab——一个基于Claude Sonnet 4.5的迭代系统提示优化器，仅需黑盒API访问即可将GSM8K上的output accuracy提升至84-87%（GPT-4o达95.2%），且推理延迟接近无格式约束的基线，远优于约束解码方案。

> **TL;DR:** Small language models (7-9B) can solve math problems correctly but systematically fail to produce valid JSON output — NAIVE prompting yields 0% output accuracy despite 77-85% task accuracy on GSM8K. Constrained decoding fixes syntax but adds 3.6-8.2x latency overhead and sometimes degrades task performance. The authors propose AloLab, an iterative black-box prompt optimizer using Claude Sonnet 4.5 as a meta-agent, which achieves 84-87% output accuracy on GSM8K and 34-40% on MATH across three models at near-baseline latency. The approach also generalizes to GPT-4o (95.2% output accuracy). Key finding: the bottleneck is structured communication, not reasoning capability.

## 核心贡献

- 揭示了小语言模型在严格JSON格式约束下的**结构性输出可靠性差距**：即使数学推理正确（task accuracy达85%），模型仍100%无法产出可解析的JSON，导致output accuracy为0%
- 提出**AloLab**——一个仅需黑盒API访问的迭代系统提示优化框架，通过分析模型输出行为自动发现并修复格式失败模式（如Gemma的markdown包裹、Llama的LaTeX转义问题）
- 系统性比较了五种策略（NAIVE、REFERENCE、CONSTRAINED、REF+CONSTRAINED、ALOLAB），证明AloLab在output accuracy和推理延迟上均显著优于约束解码方法
- 通过消融实验证明**元代理能力**是优化质量的关键驱动因素（Haiku替代Sonnet 4.5后均值从84%降至61%，标准差从<1pp升至21.8pp），且发现了一种"推理正确但答案字段写错"的格式-执行不一致现象（占剩余失败的1.5-1.8%）

## Key Contributions

- Identifies a systematic **structured-output reliability gap** in small language models: NAIVE prompting achieves 0% output accuracy across all models and datasets despite 77-85% task accuracy on GSM8K, demonstrating that the bottleneck is format compliance, not reasoning
- Proposes **AloLab**, an iterative black-box system-prompt optimizer using Claude Sonnet 4.5 as a meta-agent, requiring only API access to the target model — no weight access, gradients, or fine-tuning
- Provides a comprehensive comparison of five prompting strategies across three 7-9B models on GSM8K and MATH, showing AloLab achieves 84-87% output accuracy on GSM8K and 34-40% on MATH at near-NAIVE inference latency
- Demonstrates generalization to GPT-4o (95.2% output accuracy on GSM8K), proving the format failure is model-specific rather than size-related
- Conducts rigorous ablations: meta-agent quality ablation (Sonnet 4.5 vs Haiku) confirms optimizer capability as a primary driver; history-only ablation shows explicit model knowledge adds only 0.7pp, meaning observed execution patterns carry the optimization signal
- Identifies a **format-execution inconsistency** phenomenon (1.5-1.8% of AloLab failures) where models reason correctly but write the wrong value into the answer field, verified by an independent LLM judge

## Methodology

The paper evaluates three open-weight models (Llama 3.1-8B Instruct, Gemma 2-9B IT, Qwen 2.5-7B Instruct) and GPT-4o on two math benchmarks (GSM8K and MATH) under a strict JSON output contract requiring `reasoning` and `answer` fields.

**Metrics:** Task accuracy (math correctness regardless of format), `json_valid` (parseable JSON with required fields), and **output accuracy** (the joint event — primary metric).

**Five strategies tested:**
1. **NAIVE** — no system prompt at all
2. **REFERENCE** — minimal hand-written JSON format instructions
3. **CONSTRAINED** — grammar-based constrained decoding (GSM8K only)
4. **REF+CONSTRAINED** — REFERENCE prompt with constrained decoding (GSM8K only)
5. **ALOLAB** — iterative prompt optimizer with 5 independent runs per model/dataset

**AloLab pipeline** (4 epochs per run): Loader → Solver (queries target model via black-box API) → Evaluator (deterministic metric computation) → Analyzer (identifies failure/success patterns using Sonnet 4.5) → Optimizer (rewrites system prompt using Sonnet 4.5). The best validation checkpoint is selected for test evaluation. Five independent runs provide bootstrap confidence intervals.

**Statistical rigor:** Paired McNemar tests (29/30 significant at p < 0.05), 95% bootstrap CIs from 10,000 resamples.

## Results & Highlights

- **Format gap is near-universal:** NAIVE prompting yields 0% output accuracy for all models on both datasets, including GPT-4o
- **REFERENCE barely helps:** 0% output accuracy for Gemma and GPT-4o (markdown-fence wrapping); 44% for Llama; 74% for Qwen on GSM8K
- **Constrained decoding trades quality for latency:** 3.6-8.2x latency overhead; Gemma produces 52.4% duplicate outputs under constraint; output accuracy still below AloLab in all cases
- **AloLab achieves best output accuracy at lowest latency:** 84-87% on GSM8K, 34-40% on MATH; inference latency is 0.63-1.06x NAIVE (sometimes faster due to more directive prompts eliciting shorter completions)
- **GPT-4o generalization:** AloLab lifts GPT-4o from 0% to 95.2% output accuracy on GSM8K — same markdown-fence failure as Gemma, resolved identically
- **Meta-agent quality matters enormously:** Replacing Sonnet 4.5 with Haiku drops mean output accuracy from 84.15% to 61.0% and increases std from <1pp to 21.8pp (bimodal distribution: 2/5 runs succeed, 3/5 fail)
- **Difficulty stratification:** Output accuracy declines monotonically with problem complexity (91.2% at 1-2 steps → 66.7% at 7+ steps on GSM8K; 67.1% at Level 1 → 17.9% at Level 5 on MATH)
- **Format-execution inconsistency:** 1.5-1.8% of AloLab failures show correct reasoning in the `reasoning` field but wrong value in `answer` — a persistent decoupling phenomenon

## Authors & Credibility

The authors are from **Alomana** (a small company based in Grottaglie, Italy). The paper is methodologically sound with extensive experimental design: multiple models, multiple datasets, five independent runs with bootstrap CIs, paired McNemar tests, and well-designed ablations. The statistical reporting is thorough (29/30 comparisons significant). The authors are transparent about limitations (optimizer dependence, benchmark scope, Qwen MATH variance). The work addresses a genuinely practical problem in LLM deployment. No prior major publications from this group are evident, but the work stands on its rigor.

## Critical Assessment

- **Novelty (Moderate):** The iterative prompt optimization paradigm (OPRO, DSPy) is not new. The contribution is the systematic characterization of the structured-output reliability gap and the application of black-box optimization specifically to format compliance. The format-execution inconsistency finding is novel and interesting. The generalization to GPT-4o adds value.
- **Rigor (High):** Excellent experimental design with proper statistical testing (paired McNemar, bootstrap CIs), multiple models and datasets, ablation studies, and transparent reporting of failures (Qwen MATH variance, the single non-significant comparison). The decision to use math benchmarks as a controlled testbed is well-motivated.
- **Reproducibility (Moderate-High):** Models and datasets are public. The pipeline is described in detail. However, AloLab depends on Claude Sonnet 4.5 as the meta-agent, which is a closed-source model — exact reproducibility of the optimization trajectories is not guaranteed. The code does not appear to be publicly released.
- **Significance (Moderate-High):** The practical problem is real and important. The finding that constrained decoding can degrade task performance while adding latency is a valuable empirical contribution. The result that prompt optimization can match or exceed constrained decoding at lower cost is actionable for practitioners.
- **Red flags:** (1) The paper is from the authors' own company (Alomana → "AloLab"), though this is disclosed. (2) AloLab's heavy dependence on Claude Sonnet 4.5 as the meta-agent raises questions about cost and accessibility — the optimization requires ~30 Sonnet 4.5 calls plus thousands of target-model calls. (3) The benchmark scope is limited to math; generalization to other structured output tasks (API calls, information extraction, etc.) is claimed but not demonstrated. (4) The Qwen MATH results are quite unstable (CI spans 20.8% to 47.7%), which somewhat undermines the consistency narrative.

## Strengths

- Clean, well-motivated problem formulation: output accuracy as the joint event is the right metric for deployment
- Comprehensive experimental comparison across multiple strategies, models, and datasets
- Strong statistical rigor with paired tests and bootstrap confidence intervals
- The GPT-4o generalization probe strengthens the claim that this is a universal problem, not a small-model artifact
- Insightful ablations (meta-agent quality, history-only) that isolate what drives the optimization
- The format-execution inconsistency finding (reasoning correct but answer field wrong) identifies a concrete direction for future work
- Practical cost-effectiveness analysis (Figure 2 Pareto frontier) is immediately useful for practitioners

## Limitations

- **Benchmark scope:** Only mathematical tasks are tested. Whether AloLab generalizes to other structured output scenarios (tool-use APIs, schema extraction, code generation) remains unproven
- **Optimizer dependence:** Requires a capable (and expensive) meta-agent. The Haiku ablation shows catastrophic variance, and no middle-ground is explored
- **Qwen MATH instability:** 2/5 runs collapse due to double-escaped LaTeX, yielding a CI of [20.8%, 47.7%] — the approach is not universally reliable
- **No code release:** Reproducibility would be strengthened by open-source code
- **One-time optimization cost:** The optimization phase requires 5,000-10,000 target-model calls plus ~30 Sonnet 4.5 calls, which is non-trivial for rapid deployment scenarios
- **Format-task tension acknowledged:** On Qwen MATH, enforcing JSON format reduces recovered task accuracy from 47.5% to 34.3% output accuracy — the format constraint itself can hurt performance
- **Context window limitation:** Fixed 8,192 tokens for MATH may artificially cap performance for some models

## Questions for Further Exploration

1. How does AloLab perform on non-mathematical structured output tasks such as tool-use API generation, schema-based information extraction, or code generation with type annotations?
2. What is the cost-benefit trade-off when the target model itself is used as the optimizer (self-optimization) rather than relying on a separate, more capable meta-agent?
3. Can the format-execution inconsistency (correct reasoning, wrong answer field) be mitigated by architectural changes such as separate generation heads for structured fields?
4. How stable are the optimized prompts under distribution shift — e.g., if the test distribution differs from the optimization split in problem type or difficulty?
5. What happens when AloLab is applied to models that already undergo instruction tuning for JSON output (e.g., models fine-tuned with structured output data)?
