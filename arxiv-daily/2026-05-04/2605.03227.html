Evaluating Prompting and Execution-Based Methods for Deterministic Computation in LLMs

arXiv: 2605.03227v1

Date: 2026-05-04

https://arxiv.org/abs/2605.03227

一句话总结： 本文系统比较了CoT、Least-to-Most、PoT和Self-Consistency等提示策略在二进制计数、最长子串和算术求值等确定性计算任务上的表现，发现LLM在序列级精确计算上不可靠（最低仅40%准确率），而Program-of-Thought和轻量级CodeT5模型通过生成可执行代码达到了100%完美准确率，表明LLM更擅长模拟推理模式而非真正执行精确符号计算。
TL;DR: This paper systematically evaluates prompting strategies (CoT, Least-to-Most, PoT, Self-Consistency) on deterministic computation tasks—binary counting, longest substring detection, and arithmetic evaluation. LLMs using standard prompting struggle on sequence-based tasks (46–58% accuracy), with CoT providing negligible improvement and Least-to-Most degrading due to error accumulation. Program-of-Thought achieves perfect 100% accuracy across all tasks by delegating to an external code interpreter. A small CodeT5-small model fine-tuned for ~3 minutes on a single GPU also achieves perfect accuracy on held-out test data, suggesting specialized program-generating models are a more reliable and efficient path for exact computation than scaling up LLM prompting.

  • 证明Program-of-Thought通过生成可执行代码并交给外部解释器执行，可在所有任务上达到100%准确率，实现了推理与计算的解耦。
  • 展示了轻量级CodeT5-small模型（单GPU训练约3分钟）在生成可执行程序方面可达到与大型LLM+PoT相同的完美准确率，提供了一种高效且可扩展的替代方案。

Key Contributions

  • Constructed a synthetic dataset of 3,000 examples across three deterministic computation task types (binary counting, longest substring, arithmetic), each with diverse natural language instruction variants to test robustness beyond memorized templates.
  • Provided the first systematic comparison of five prompting strategies under exact-match evaluation for deterministic computation, revealing that CoT yields negligible gains and Least-to-Most suffers from error accumulation on sequence tasks.
  • Demonstrated that Program-of-Thought (PoT) achieves perfect accuracy on all tasks by generating executable Python code and delegating computation to an interpreter, cleanly separating reasoning from execution.
  • Showed that a small CodeT5-small model, fine-tuned in ~3 minutes on a single GPU, matches PoT's perfect accuracy on held-out test data across all tasks, offering a lightweight and scalable alternative to large LLM prompting.

Methodology

The paper introduces a controlled evaluation framework with three deterministic computation tasks:

  1. Binary Counting: Count 0s and 1s separately in binary sequences of length 80–120, including edge cases (all-0 or all-1 sequences). Output format: 0: | 1:.
  2. Longest Substring: Find the length of the longest consecutive block of a target character in sequences drawn from lowercase letters, digits, and symbols. Tests sequential state tracking.
  3. Arithmetic Computation: Evaluate expressions with 3–6 numbers and operators (+, -, *, /), constrained to produce integer results.

Each task uses multiple natural language instruction templates to prevent template memorization. The dataset contains 1,000 examples per task (3,000 total) in JSONL format with ground-truth answers and target executable code.

Five prompting strategies are evaluated on a gpt-oss-120b model under zero-shot settings: Plain Prompt, Chain-of-Thought (CoT), Least-to-Most decomposition, Program-of-Thought (PoT), and Self-Consistency (SC, 5 samples at T=0.7 with majority voting). PoT generates Python code executed in a sandboxed environment blocking unsafe operations.

Additionally, CodeT5-small (Salesforce/codet5-small) is fine-tuned to map task descriptions to executable Python programs, trained with lr=5e-5, batch size 8, 6 epochs, completing in ~3 minutes on a single GPU.

Evaluation uses exact-match accuracy. Single-task evaluation uses 100 samples per method (50 for SC); mixed-task evaluation uses 150 samples (50 per task; 90 for SC).

  • CoT provides no meaningful improvement over plain prompting on sequence tasks.
  • Least-to-Most degrades performance (40% on longest substring) due to error accumulation across decomposition steps.
  • Self-Consistency improves binary counting to 0.70 but barely helps longest substring (0.47), and requires 5× the compute.
  • PoT and CodeT5 both achieve perfect accuracy by offloading computation to code execution.
  • Aggregate mixed-task accuracy is misleading (dominated by trivial arithmetic); per-task analysis reveals the true capability gap.

Authors & Credibility

Single-author work by Hongkun Yu from Virginia Tech (ORCID provided). The paper is well-structured with clear problem formulation, controlled experiments, and honest reporting of results. References cover the relevant prompting literature (CoT, Self-Consistency, Least-to-Most, PoT, Toolformer, CodeT5). Dataset and code are claimed to be publicly released. The work appears to be a focused empirical study rather than a theoretical contribution. The use of gpt-oss-120b (a presumably open-source model) rather than GPT-4/Claude adds reproducibility value but may limit the generalizability of conclusions about frontier models.

Critical Assessment

Novelty: Moderate. The finding that LLMs struggle with exact computation is not surprising and has been informally observed. The systematic comparison across prompting strategies on specifically deterministic tasks is useful, though the contribution is primarily empirical rather than conceptual.

Rigor: Reasonable but with notable gaps. (1) Only one LLM is evaluated (gpt-oss-120b), making it unclear whether findings generalize to frontier models (GPT-4, Claude, Gemini). (2) The sample sizes are relatively small (50–100 per method per task). (3) No statistical significance tests or confidence intervals are reported. (4) The arithmetic task is trivially easy and inflates aggregate metrics, which the authors acknowledge but could have addressed by excluding it from the main analysis.

Reproducibility: Good. Synthetic dataset with automatic generation, public code release, detailed hyperparameters, and exact prompt templates in the appendix. The CodeT5 training setup is fully specified.

Significance: The paper provides a clean empirical demonstration of a well-known limitation, but the tasks are arguably toy problems. Binary counting of 80–120 character sequences is a narrow test. Whether these findings transfer to more complex real-world deterministic tasks (e.g., database query generation, formal verification) remains an open question that the authors flag as future work.

Red flags: (1) Single author with no peer review yet. (2) The model identity gpt-oss-120b is vague—it is unclear which specific model this is, which affects reproducibility. (3) The paper could benefit from testing at least one frontier proprietary model to validate that the findings are not model-specific. (4) The conclusion that "LLMs may simulate reasoning patterns rather than reliably perform exact symbolic computation" is strong but supported only by synthetic tasks with short sequences.

Strengths

  • Clean, well-controlled experimental design with a unified evaluation framework across methods.
  • Honest per-task reporting that avoids the trap of aggregate metrics masking poor performance.
  • Demonstrates a practical alternative (CodeT5 fine-tuning) that is both more accurate and more efficient than large LLM prompting.
  • Diverse instruction templates per task add ecological validity beyond single-prompt evaluation.
  • Public dataset and code release support reproducibility.

Limitations

  • Only one LLM (gpt-oss-120b) is evaluated; findings may not generalize to frontier models or different architectures.
  • Tasks are synthetic and relatively simple; real-world deterministic computation may involve more complex structures and longer sequences.
  • No error analysis beyond accuracy—understanding why LLMs fail (e.g., tokenization issues, attention limitations) would strengthen the contribution.
  • Self-Consistency uses only 5 samples; more samples or calibration of temperature could yield different conclusions.
  • The paper does not explore hybrid approaches (e.g., LLM deciding when to invoke tools vs. answering directly).

Relevance

本文与LLM推理能力评估高度相关，系统揭示了提示策略在确定性计算任务上的局限性，对设计LLM+工具的混合系统（如tool-augmented reasoning）具有直接参考价值。Program-of-Thought作为推理与计算解耦的范式，与当前agent框架中LLM调用代码解释器的趋势一致。

Questions for Further Exploration

  • How do frontier models (GPT-4, Claude, Gemini) perform on these same tasks? Does scaling model size improve exact computation, or is the fundamental limitation architectural?
  • Does tokenization of binary sequences (e.g., BPE merging of common bit patterns) contribute to counting errors, and would character-level tokenization help?
  • Can the CodeT5 approach generalize to more complex deterministic tasks (e.g., sorting, graph algorithms, constraint satisfaction) beyond the three synthetic tasks tested here?
  • How does the performance gap between prompting and program-based methods change as sequence length scales beyond 120 characters?
  • Could a hybrid approach—where the LLM detects whether a query requires exact computation and selectively delegates to code generation—outperform either pure prompting or pure program generation?
