# CodeSpecBench: Benchmarking LLMs for Executable Behavioral Specification Generation

> **一句话总结：** 本文提出了 CODESPECBENCH，首个覆盖函数级与仓库级任务、基于执行的可执行行为规范生成基准，发现即使最强模型在仓库级任务上的通过率也仅 20.2%，且生成规范显著难于生成代码，揭示了当前 LLM 对程序语义理解的深层局限。

> **TL;DR:** This paper introduces CODESPECBENCH, the first execution-based benchmark for executable behavioral specification generation spanning both function-level and repository-level tasks. Evaluating 15 state-of-the-art LLMs reveals that even the best model achieves only a 20.2% pass rate on repository-level tasks, and that generating executable specifications is substantially harder than generating code—indicating that strong coding performance does not necessarily reflect deep understanding of intended program semantics.

## 核心贡献

- 提出了 CODESPECBENCH，一个基于执行评估（execution-based evaluation）的可执行行为规范生成基准，涵盖函数级（CODESPECBENCH-Func，2,494 题）和仓库级（CODESPECBENCH-Repo，500 个真实 issue）两个粒度。
- 将规范表示为可执行的 Python 前置条件（preconditions）与后置条件（postconditions），同时评估正确性（correctness，接受所有合法行为）和完备性（completeness，拒绝所有非法行为）。
- 对 15 个当前最先进的 LLM 进行了全面评测，发现函数级任务上最佳模型 GPT-5-mini 的通过率仅为 47.0%，而仓库级任务上 Claude-4.5-Sonnet 仅达到 20.2%，开源模型在该设置下几乎失效（最高 6.8%）。
- 首次系统量化了“规范生成比代码生成更难”：在相同任务上，模型的规范生成通过率远低于其代码生成通过率（例如 Claude-4.5-Sonnet 77.2% 代码 vs 21.0% 规范）。
- 展示了生成的规范可用于语义重排序（specification-guided re-ranking），在 CODESPECBENCH-Func 上提升代码生成 pass@1 达 2.8%。

## Key Contributions

- Introduces CODESPECBENCH, an execution-based benchmark for executable behavioral specification generation covering both function-level (2,494 problems) and repository-level (500 real-world issues) tasks.
- Represents specifications as executable Python preconditions and postconditions, evaluating both correctness (accepting all valid behaviors) and completeness (rejecting all invalid behaviors).
- Conducts a comprehensive evaluation of 15 state-of-the-art LLMs, showing that the best model achieves only 47.0% pass rate on function-level tasks (GPT-5-mini) and merely 20.2% on repository-level tasks (Claude-4.5-Sonnet), with open-weight models struggling significantly (best open-weight: 6.8%).
- Systematically demonstrates that executable behavioral specification generation is substantially harder than code generation: specification pass rates lag far behind solution-generation pass rates on the same tasks.
- Shows that generated specifications can improve code generation via semantic re-ranking, improving pass@1 by up to 2.8% on CODESPECBENCH-Func.

## Methodology

CODESPECBENCH 采用基于执行的评估协议，要求模型将自然语言描述翻译为成对的 Python 函数 `preconditions(...)` 与 `postconditions(...)`，并通过断言（assert）来约束行为。函数级任务（Func）基于 LeetCode-Dataset，利用多模型（GPT-4o、DeepSeek-V3、Qwen-Max）生成测试输入，经官方在线评测系统（OJ）验证后获得 50 组正确输入、50 组错误输入以及 50 组正确输入-输出对；再通过让 LLM 生成有缺陷的解法来构造平均 67.8 组错误输出。仓库级任务（Repo）基于 SWE-bench Verified，并引入 UTBoost 增强的测试用例；模型需要阅读 issue 描述和相关代码文件，生成 `conftest.py` 文件，通过 pytest hook 在运行时动态注入前置/后置条件检查。评估指标包括：Correctness（接受所有合法测试）、Completeness（拒绝所有非法测试）以及同时满足两者的 Pass Rate。

## Results & Highlights

- **函数级性能有限，仓库级急剧下滑：** 在 CODESPECBENCH-Func 上，最强模型 GPT-5-mini 仅达 47.0% pass，Gemini-2.5-Pro 46.2%，开源的 GPT-OSS-120B 42.5%；在 CODESPECBENCH-Repo 上，Claude-4.5-Sonnet 仅 20.2%，Gemini-2.5-Pro 18.2%，多数开源模型接近 0%。
- **Func 上的正确性远高于完备性：** 大多数模型呈现出 correctness >> completeness 的模式（如 QWQ-32B 83.1% vs 37.3%），说明模型倾向于生成过于宽松的规范，能放过合法行为但无法拦住非法行为。
- **Repo 上的完备性高于正确性，但通过率仍低：** 在仓库级任务中，模型往往生成过于严格或有缺陷的规范（如 Gemini-2.5-Pro 53.0% complete vs 30.8% correct），导致大量合法行为被拒绝，通过率受 correctness 主导而受限。
- **错误模式分析：** 函数级失败主要由类型约束和范围约束错误主导（Type + Range Violations 占 48–54%）；仓库级失败则由依赖解析（Dependency Resolution，33.4% for Claude）和过度严格规范（Over-restrictive Specifications，18.7%）主导。
- **规范生成可用于提升代码生成：** 以生成的规范对 10 个候选解进行重排序，Qwen3-8B 的 pass@1 从 59.4% 提升到 62.2%，Qwen3-32B 从 71.2% 提升到 73.9%。
- **跨算法领域差异显著：** 排序类任务（Radix Sort、Merge Sort）的后置条件通过率最高（约 66%），而依赖全局结构或递归不变量的领域（如 Binary Search Tree 22.2%、Strongly Connected Component 28.4%）表现最差。

## Critical Assessment

这是一项非常重要且及时的基准建设工作。CODESPECBENCH 填补了 LLM 形式化语义理解评估的空白，其执行驱动、双粒度（Func/Repo）、双维度（Correctness/Completeness）的设计具有很强的现实意义。论文的核心发现——“代码生成强不代表语义理解深”——对当前 LLM 评估体系有重要警示作用。然而，CODESPECBENCH-Func 严重依赖外部在线评测平台（LeetCode）进行输入验证和获取标准输出，这使得基准构建流程并非完全自包含，长期可复现性存在风险。此外，规范形式被限定为 Python 的 assert 前置/后置条件，虽然实用但表达能力不及完整的形式化验证语言（如 Lean、Dafny），也无法提供数学上的正确性保证。最后，Repo 级别的低通过率多大程度上源于“规范生成本身的困难”，多大程度上源于“在复杂代码库中生成可执行 Python 代码的语法/环境困难”，仍值得进一步拆分（论文 Appendix F 已初步讨论 syntactic executability 的瓶颈）。

## Strengths

- 首个同时覆盖函数级和仓库级、基于执行的大规模可执行规范生成基准，评估维度全面（正确性 + 完备性）。
- 测试构造质量高：Func 级平均每题 217.8 个测试用例，语句/分支覆盖率高达 96.3%/93.6%；Repo 级引入 UTBoost 增强测试。
- 评测模型覆盖面广，涵盖开源（Qwen3、DeepSeek-V3.2、GPT-OSS）与闭源前沿模型（Claude-4.5-Sonnet、GPT-5、Gemini-2.5-Pro）。
- 发现具有挑战性：Repo 级最佳仅 20.2%，且规范生成显著难于代码生成，直接挑战了“代码生成能力 = 语义理解能力”的默认假设。
- 提供了细粒度的错误分类和跨算法领域的性能分析，为后续研究指明了改进方向。

## Limitations

- 执行验证是近似而非形式化证明：通过所有测试并不保证规范在所有可能输入上都正确或完备。
- CODESPECBENCH-Func 依赖外部 OJ 平台，基准构建流程不完全自包含，长期可复现性受第三方服务可用性影响。
- 规范形式仅限于 Python 的 assert 前置/后置条件，跨语言泛化能力受限，也无法表达某些需要复杂逻辑或数学归纳的约束。
- 仓库级任务中，低性能部分源于生成可执行代码的语法/环境困难（如依赖解析、签名不匹配），而非纯粹的语义理解失败。
- 未深入探讨不同 prompt 策略、多轮交互或 agent 工作流对规范生成性能的影响。

## Relevance

本文与 LLM 代码生成评估、软件工程形式化方法以及模型语义理解能力研究高度相关。CODESPECBENCH 为社区提供了一个新视角：除了测模型能不能“写出正确代码”，还要测模型能不能“准确描述正确代码应该满足什么约束”。在智能体协作、自动代码审查、合约驱动开发等场景中，可执行行为规范的自动生成能力至关重要。论文揭示的“代码生成强但规范生成弱”现象，意味着当前 LLM 在深层程序语义理解和精确约束表达上仍有巨大提升空间，对未来模型训练和评估具有重要指导意义。

## Questions for Further Exploration

- 如果为模型提供多轮反馈或自动调试循环（如 agentic workflow），仓库级规范生成的通过率能提升多少？
- 将规范形式从 Python assert 扩展到更丰富的表达（如 type hints、contract decorators、property-based tests）会如何影响模型表现和评估严格性？
- 能否将执行验证与符号验证或定理证明相结合，在保持可扩展性的同时提供更强的正确性保证？
- 对于 Repo 级任务，如何更好地区分“语义理解失败”与“代码注入/环境配置失败”，以得到更纯净的规范生成能力评估？
- 生成的 executable specifications 能否直接用于下游任务，如自动测试生成、代码审查、或强化学习中的 reward shaping？
