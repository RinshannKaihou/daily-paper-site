> **一句话总结：** Charon 是字节跳动提出的统一 LLM 训练与推理性能模拟器，采用编译器风格的图变换架构和混合多引擎后端（profiling + ML 预测 + 解析模型），在大规模 GPU 集群训练中预测误差低于 3.74%，并成功发现优于工程调优基线的推理部署配置。

> **TL;DR:** Charon is a production-grade LLM simulator from ByteDance that natively ingests PyTorch/HuggingFace/vLLM models and uses a compiler-style graph-IR frontend with a hybrid multi-engine backend (profiling, ML-prediction, analytical roofline) to simulate both training and inference at operator-level granularity. It achieves under 3.74% error on large-scale training (nearly 10K GPUs), under 5.35% overall, and discovered throughput-improving inference configurations in production deployment. Accepted at MLSys 2026.

## 核心贡献

- 提出 Charon，首个统一覆盖 LLM 训练与推理的细粒度模拟器，直接接受原生 PyTorch/HuggingFace/vLLM 模型输入，无需手动构建模型描述
- 采用编译器风格的 pass-based 图变换架构，将并行策略（TP/PP/DP/SP/EP）、算子融合、量化优化等抽象为可插拔的 graph pass，支持灵活组合
- 混合多引擎后端设计：profiling 引擎（最精确）、prediction 引擎（基于随机森林的 ML 模型，MAE 低于 2.22%）、analytical 引擎（roofline 模型），通过 fused engine 自适应选择
- 精确的算子级 overlap 建模：支持通信-计算和通信-通信重叠的 ratio-based slowdown 模型及带宽感知拥塞模型
- 内置设计空间探索（DSE）工具，支持规则剪枝加速搜索，在实际推理部署中发现优于工程调优基线的配置
- 在大规模 GPU 集群（近万卡）训练场景下端到端预测误差低于 3.74%，内存预测误差低于 0.4%

## Key Contributions

- Charon is the first unified fine-grained simulator covering both LLM training and inference, accepting native PyTorch/HuggingFace/vLLM models without manual model construction or preprocessing
- Compiler-style pass-based graph transformation architecture: parallelism strategies (TP, PP, DP, SP, EP), operator fusion, quantization, and optimizations are abstracted as composable graph passes
- Hybrid multi-engine backend: profiling engine (highest accuracy via cached profiling DB), prediction engine (random-forest-based ML with MAE under 2.22%), analytical engine (roofline model + hierarchical link-centric communication model), adaptively fused for speed-accuracy tradeoff
- Fine-grained operator-level overlap modeling: ratio-based slowdown for compute-communication overlap and bandwidth-aware congestion model for communication-communication overlap
- Built-in design space exploration with rule-based pruning, enabling rapid identification of optimal configurations
- Achieves under 3.74% end-to-end error on large-scale training (nearly 10K GPUs), under 5.35% overall, and memory prediction errors below 0.4%

## Methodology

Charon's architecture consists of three major components:

**1. Graph-based Frontend:**
- Uses `torch.fx.symbolic_trace` and `torch.compile` to automatically convert native PyTorch models into intermediate computation graphs
- For training, automatically generates backward graphs via `torch._function.aot_autograd`, with joint graph partitioning and dead code elimination
- Single transformer block optimization: extracts and simulates only one block for symmetric architectures, while tracing distinct layers for PP or asymmetric models
- Supports disaggregated serving by tracing prefill and decode into independent computation graphs
- Compiler-style passes for: operator rewrite/fusion, quantization (as a graph pass or via tracing quantized models), and all major parallelism strategies
- Parallelism insertion: shard-based (TP, SP, EP) insert all_reduce/all_gather/reduce_scatter; PP uses schedule pattern generator with 1F1B and DualPipe support; DP supports DDP, FSDP, ZeRO
- Liveness-based memory analysis: traverses the backward graph to precisely track tensor allocation/deallocation timing, enabling accurate peak memory estimation that layer-level simulators cannot achieve
- Multi-granularity analysis: from system-level (MFU, memory, power) to operator-level (PyTorch Profiler-style traces with 3D multi-GPU timelines)

**2. Multi-engine Backend:**
- Profiling engine: dispatches operators to GPU cluster, caches results in a profiling database for reuse
- Prediction engine: random-forest-based ML models trained on profiling data, generalizes to unseen tensor shapes (MAE: 1.44% Linear, 1.12% RMSNorm, 2.22% FlashAttention-3)
- Analytical engine: roofline model for computation; hierarchical link-centric model for communication with calibrated per-hop latency and effective bandwidth, supporting Ring and Tree collectives across diverse topologies
- Fused engine: prioritized fallback mechanism dynamically selects the best available backend per operator
- Supports FP32, BF16, FP16, FP8, INT8 precision formats

**3. Design Space Exploration:**
- Native search tool with user-defined pruning rules to skip known inefficient configurations
- Enables rapid "what-if" analysis for parallelism sizes, GPU counts, batch sizes, and scheduling strategies

## Results & Highlights

- **End-to-end accuracy (Figure 7):** Best among all compared simulators (ASTRA-SIM 2.0, SimAI, Vidur) across Qwen3-8B, LLaMA3-8B, and Qwen3-30B-A3B (MoE) for both training and inference
- **Large-scale training:** Under 3.74% error on nearly 10,000 GPU configurations with all parallelism types (DP, PP, EP, SP, TP)
- **Cross-GPU accuracy:** Under 4.98% error across Hopper (H800), Ampere (A100), Ada Lovelace (L20), and H20 GPUs
- **Memory prediction:** Errors of +0.39% (max allocated), +0.29% (max reserved), -0.03% (total footprint) for Qwen3-30B-A3B MoE training
- **Backend ablation:** Prediction engine achieves 1.44%-2.22% MAE vs. analytical engine's 31.84% MAE for FlashAttention-3
- **Case Study 1 -- Dynamic Sequence Parallelism:** Used Charon to design dynamic SP planning for LLaMA-3 70B, achieving 15% reduction in attention block latency vs. static zigzag attention
- **Case Study 2 -- Inference optimization:** Discovered deployment configurations for LLaMA-3 70B that outperformed manually-tuned baselines under strict 100ms SLO; revealed up to 7x throughput tradeoff along Pareto frontier; simulation completes within 2 minutes
- **Cost reduction:** More than 30x cost reduction compared to cluster profiling for large-scale experiments

## Critical Assessment

- **Novelty: HIGH.** The compiler-style pass-based architecture for LLM simulation is genuinely novel. Unlike prior simulators that treat parallelism and optimizations as hardcoded features, Charon's graph-pass abstraction makes it naturally extensible. The hybrid multi-engine backend with ML-based prediction is also a meaningful contribution over purely analytical or purely profiling-based approaches.
- **Rigor: HIGH.** The evaluation is thorough: compared against three competitive baselines (ASTRA-SIM 2.0, SimAI, Vidur) that were bug-fixed and re-calibrated for fairness. Evaluated on both dense and MoE models, across multiple GPU architectures (Hopper, Ampere, Ada Lovelace), and at scales up to ~10K GPUs. The ablation study (analytical vs. prediction engine) cleanly demonstrates the value of the ML backend. Operator-level breakdown tables and trace visualizations provide fine-grained validation.
- **Reproducibility: LOW-MODERATE.** No mention of open-sourcing. The system depends on an in-house GPU profiling cluster and internal inference framework integration. The profiling database is proprietary. While the methodology is well-described, full reproducibility would require releasing the code, profiling data, and prediction models. The HTML version is available on arxiv, which helps.
- **Significance: HIGH.** This is a production-validated simulator used at ByteDance for real deployment decisions. The 30x cost reduction over cluster profiling and the ability to find configurations that outperform engineering baselines demonstrate genuine practical impact. The work addresses a critical pain point in the LLM systems community.
- **Red flags:** (1) The abstract contains "Hooker Figure hypotheses" which appears to be a rendering artifact, not a substantive issue. (2) The prediction engine's training data comes from the same profiling database used for evaluation, raising potential concerns about generalization to truly unseen hardware -- though the cross-GPU evaluation partially addresses this. (3) The communication overlap model uses engineered slowdown factors from profiling data, which may not transfer well across different cluster configurations. (4) No comparison with analytical cost models like those in Megatron-LM's autosearch or Alpa, which are widely used in practice.

## Strengths

- Truly unified: handles training and inference in a single framework, eliminating the need for separate incompatible tools
- Native model input: directly ingests HuggingFace/vLLM/PyTorch models without manual model construction
- Compiler-style pass design is elegant and extensible -- new parallelism strategies or optimizations are just new graph passes
- The liveness-based memory analysis is a genuine improvement over layer-level memory estimation in prior simulators
- Strong empirical validation at production scale (~10K GPUs) with multiple GPU architectures
- Practical impact demonstrated through real deployment case studies, not just synthetic benchmarks
- Clear presentation with helpful architectural diagrams and trace visualizations

## Limitations

- Not open-sourced as of publication, limiting community adoption and independent verification
- Prediction engine relies on profiling data from the same hardware types used in evaluation; generalization to fundamentally different hardware (e.g., AMD GPUs, TPUs) is unclear
- The slowdown factors for operator overlap are empirically calibrated and may not transfer across cluster configurations
- Does not model stochastic runtime variability (network jitter, dynamic congestion, data-dependent kernel execution), which the authors acknowledge as a source of residual error
- No evaluation on models larger than 70B parameters, though the architecture should theoretically scale
- The dynamic SP case study shows 15% improvement on attention blocks but does not report end-to-end serving latency improvement
- The design space explorer uses simple rule-based pruning rather than more sophisticated Bayesian optimization or bandit methods

## Questions for Further Exploration

- How does Charon's prediction engine compare to learned cost models (e.g., neural network-based predictors) beyond random forests?
- Can Charon be extended to simulate heterogeneous clusters (mix of GPU generations or GPU+NPU)?
- How sensitive is the simulation accuracy to the quality and coverage of the profiling database?
- What is the wall-clock simulation time for the largest configurations (10K GPUs) compared to actual training run time?
- How would Charon handle emerging architectures like mixture-of-depths, sparse attention, or non-Transformer architectures (Mamba, RWKV)?
- Could the graph-pass framework be extended to simulate compiler-level optimizations like torch.compile's Inductor backend passes?
- Is there a path to open-sourcing at least the frontend and analytical engine components?
