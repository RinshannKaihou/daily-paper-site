Key Metric
< 3.74%
End-to-end prediction error on large-scale training across nearly 10,000 GPUs with all parallelism types (DP, PP, EP, SP, TP).
Why This Matters
Replaces expensive cluster profiling with fast simulation for LLM training and inference configuration, cutting costs 30x.
Critical Edge
Not open-sourced as of publication, limiting community adoption and independent verification of its claims.

Methodology

Charon has three major components:

  • Graph-based Frontend — Uses torch.fx.symbolic_trace and torch.compile to convert native PyTorch/HuggingFace/vLLM models into intermediate computation graphs. Backward graphs are auto-generated. A compiler-style pass system treats parallelism strategies (TP, PP, DP, SP, EP), operator fusion, and quantization as composable graph passes. Liveness-based memory analysis tracks tensor allocation/deallocation timing for accurate peak memory estimation.
  • Hybrid Multi-engine Backend — Three engines fused with prioritized fallback: a profiling engine (dispatches to GPU, caches results), a prediction engine (random-forest ML, 1.44–2.22% MAE across operator types), and an analytical engine (roofline compute model + hierarchical link-centric communication model). Supports FP32 through INT8.
  • Design Space Exploration — Native search tool with user-defined rule-based pruning for rapid what-if analysis across parallelism sizes, GPU counts, batch sizes, and scheduling strategies. Simulation completes within 2 minutes for large configurations.

Strengths

  • Truly unified: handles both training and inference in a single framework
  • Native model input: ingests PyTorch/HuggingFace/vLLM directly, no manual model description needed
  • Compiler-style pass architecture is elegant and extensible — new strategies are just new graph passes
  • Liveness-based memory analysis is a genuine improvement over layer-level estimation in prior simulators
  • Strong empirical validation at production scale (~10K GPUs) across multiple GPU architectures
  • Practical impact demonstrated through real deployment case studies, not just synthetic benchmarks

Limitations

  • Not open-sourced, limiting community adoption and independent verification
  • Prediction engine trained on same hardware types used in evaluation; generalization to AMD GPUs or TPUs is unclear
  • Overlap slowdown factors are empirically calibrated and may not transfer across cluster configurations
  • Does not model stochastic runtime variability (network jitter, dynamic congestion)
  • No evaluation on models larger than 70B parameters
  • DSE uses simple rule-based pruning rather than Bayesian optimization or bandit methods

Questions for Further Exploration

  1. How does Charon's prediction engine compare to neural-network-based learned cost models beyond random forests?
  2. Can Charon be extended to simulate heterogeneous clusters (mixed GPU generations, or GPU+NPU)?
  3. How sensitive is simulation accuracy to the quality and coverage of the profiling database?
  4. What is the wall-clock simulation time for 10K-GPU configurations compared to actual training run time?
  5. How would Charon handle non-Transformer architectures (Mamba, RWKV) or mixture-of-depths?
  6. Could the graph-pass framework simulate compiler-level optimizations like torch.compile's Inductor passes?
  7. Is there a path to open-sourcing at least the frontend and analytical engine?
