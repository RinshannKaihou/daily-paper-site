Tool-Making and Self-Evolving LLM Agents in Low-Latency Systems

arXiv: 2607.08010

Date: 2026-07-10

https://arxiv.org/abs/2607.08010

一句话总结： 针对生产环境 LLM 编码 Agent 每次请求都重新生成相同代码导致的延迟与可靠性问题，本文提出离线工具构造流水线，将标准操作流程（SOP）节点编译为环境落地、带标签验证、版本化的确定性工具；在亚马逊履约中心出库码头告警分诊系统中部署后，相对 CodeAct 子 Agent 基线 p50 延迟降低 42%，直接调用架构在受控消融中再降 62%，1500 条历史告警端到端错误率最高下降 53%，并借版本化工具暴露出三处上游数据不一致。
TL;DR: To stop production CodeAct-style LLM agents from regenerating the same procedural code on every request, the authors compile repeated SOP steps offline into environment-grounded, test-repaired, versioned tools via a data-collector sub-agent + tool-maker + reflector loop; deployed in an Amazon Fulfillment Center outbound-dock alarm-triage agent over a 44-node SOP, tool calls cut per-alarm p50 latency by 42% versus the sub-agent baseline (Qwen3 32B), a direct-call architecture ablation (GLM-4.5-Air, 3 seeds, 1,500 alarms) cuts p50 a further 62% (p99 -30%), and end-to-end error rate drops from 1.7% to 0.8% on GLM-4.5-Air (-53%) and from 2.8% to 1.8% on Qwen3 32B (-36%) on 1,500 historical alarms, with per-node pass@1 reaching 94.5% (GLM-4.7) and rising to 99.9% once underspecified SOP nodes are clarified.

核心贡献

  • 提出离线工具构造流水线，把每个 SOP 决策节点编译成一个签名统一 （warehouse, timestamp, context) -> verdict 的确定性工具，由三个模型组件组成：作为回退的 data-collector 子 Agent（跑生产 MCP 采执行轨迹）、生成候选工具的 tool-maker LLM、针对标注用例做 generate-test-repair 的 reflector 循环（3 轮预算）。
  • 通过双落地机制解决"SOP 文本缺执行细节"问题：把真实 schema/查询代码/标注判定的执行轨迹喂给 tool-maker，再加上测试-修复循环；在 GLM-4.7 上将逐节点 pass@1 从 SOP-only 的 75.7% 拉到完整配置的 94.5%，去掉修复循环掉 9.6pp，去掉轨迹掉 5.2pp。
  • 把该流水线集成进真实部署的告警分诊 Agent（蒸馏 GLM-4.7 教师 + LoRA 微调 Qwen3 32B / GLM-4.5-Air 学生，混合带工具/不带工具轨迹训练以保留回退能力），并给出工具替代编码后让主 Agent 直接调用、删除子 Agent 层的架构简化方案。
  • 给出生产可观测性闭环：版本化工具每次调用记录版本/输入/输出，监控 Agent 批量审日志，借此实际发现上游三处既有不一致（字段类型差异、累计 vs 当前事件计数、返回格式 75 vs 0.75），并支持配置级回滚、无需重训。
  • 实证定位真正瓶颈是 SOP 规格质量而非 Agent：4 个残留节点中，加数据/澄清 SOP 后 pass@1 从 94.5% 升至 99.9%；并给出少标签（N=5 即达 91.8%）与无标签（judge 过滤伪标签 + 多候选投票，91.6%）两种实用变体，均距有标签天花板 3pp 以内。

Key Contributions

  • An offline tool-making pipeline that compiles each of 44 SOP decision nodes into a model-agnostic tool with a uniform signature (warehouse, timestamp, context) -> verdict + supporting detail, built from three components: a data-collector sub-agent (the baseline CodeAct agent run against production MCP to gather real execution traces), a tool-maker LLM conditioned on SOP text + node position + traces, and a reflector-tool-maker test-repair loop run for a budget of three rounds against labeled cases.
  • Two grounding mechanisms that close the gap between sparse SOP text and executable code: the data-collection trace (schema, MCP values, working code, graded verdict) and the test-repair loop. On GLM-4.7 these jointly raise per-node pass@1 from 75.7% (SOP only) to 94.5% (SOP+D+R); the repair loop alone contributes +9.6pp and the trace +5.2pp, and only the full configuration exceeds 90%.
  • Integration into a deployed Fulfillment-Center alarm-triage agent, including distillation of a GLM-4.7 teacher into Qwen3 32B and GLM-4.5-Air students via LoRA with mixed-trajectory training (75% of trajectories have tools with 25% per-trajectory dropout, 25% have no tools) so the agent keeps a code-generation fallback; the GLM-4.7-generated library serves both student agents without regeneration.
  • An architectural simplification enabled by compact tool outputs: because tools return a few tokens, the main agent can call them inline, removing the sub-agent layer entirely; a controlled ablation (GLM-4.5-Air, 3 seeds, 1,500 alarms) shows direct calls alone cut p50 by 62% and p99 by 30% relative to the sub-agent architecture holding the model fixed.
  • Production-grade observability: every tool call logs version, inputs, and outputs, and a batch monitoring agent reviews logs to surface drift; this mechanism caught three pre-existing upstream inconsistencies (a type mismatch between offline and production MCP, cumulative-vs-current event counting, and a silent 75-to-0.75 percentage format change), and versioned tools make fixes a configuration change requiring no retraining.

Methodology

The system has two modes. Offline tool-making turns each SOP node into a validated tool. A data-collector sub-agent (the same CodeAct-style agent used as the runtime fallback, querying heterogeneous metric backends via Model Context Protocol) is run on three sampled training cases balanced across verdict labels; its full trace (query code, MCP responses, observed schema with field names/datatypes/value ranges, and the graded verdict) is recorded. A tool-maker LLM then writes a candidate tool conditioned on the SOP node text, the node's position in the decision tree (parent/children), and those traces. The candidate is executed against the full per-node labeled training set, and any disagreements enter a reflector-tool-maker test-repair loop: the reflector LLM writes a short diagnosis of failing (input, expected, predicted) triples, the tool-maker rewrites the candidate conditioned on that diagnosis, and this runs for up to three rounds with early stopping. The highest-pass-rate candidate is deployed. Each tool is a function with uniform signature mapping (warehouse, timestamp, context) to a verdict of true/false/no-data plus the observed value, threshold, and a textual explanation; exceptions trigger fallback to the CodeAct sub-agent.

Online runtime runs the deployed alarm-triage agent, which uses small models (Qwen3 32B and GLM-4.5-Air) on vLLM (8xH100) to meet latency targets. Tools are invoked in two configurations: the main agent delegates a node to a sub-agent that calls the node's tool, or the main agent calls the tool directly. The student agents are fine-tuned by distilling GLM-4.7 teacher trajectories that are rejection-sampled on full match of observations, root causes, and actions (acceptance ~92%), with mixed-trajectory training so the agent falls back to code generation when a tool is missing or fails. The full 44-node tool library is generated once at a cost of ~800K tokens and is parameterized by runtime thresholds so one library serves multiple physical sites.

The setting is an Amazon FC outbound dock over a 44 decision-node / 19 action-node SOP (three node kinds: observations, root causes, constraint gates). Evaluation uses 8,011 training and 8,022 held-out evaluation cases (per-node temporal split, 100-200 cases per node), plus 1,500 historical alarms with ground-truth observations, root causes, and actions for end-to-end correctness, and normalized production p50/p99 latency. Confidence intervals use 30 seeds (10 for some columns) with paired t-tests; end-to-end significance uses bootstrap-by-task (nboot=5,000).

Results & Highlights

  • Per-node tool quality (Table 1): The deployed SOP+D+R configuration reaches 94.5% pass@1 on GLM-4.7 and 93.6% on GLM-5, versus SOP-only baselines of 75.7% and 76.8% — the data-collection trace and repair loop are both necessary (removing repair costs -9.6pp, removing trace costs -5.2pp on GLM-4.7), and only the full configuration exceeds the 90% bar; weaker tool-makers (Qwen3 235B at 81.1%, GLM-4.7 Flash at 65.8%) do not improve monotonically.
  • Production latency (Table 3): Replacing the sub-agent's CodeAct loop with tool calls cuts per-alarm p50 by 42% on Qwen3 32B under the original two-tier architecture, with per-alarm output tokens dropping 58% (14,998 to 6,225), sub-agent turns dropping 45% (33.7 to 18.7), and sub-agent code-generation tokens falling 80%.
  • Direct-call architecture ablation: Holding the model (GLM-4.5-Air) and inference framework fixed (3 seeds, 1,500 alarms), direct main-agent tool calls reduce p50 by 62% and p99 by 30% relative to the sub-agent architecture, because compact structured verdicts remove the rationale for the sub-agent layer.
  • End-to-end accuracy on 1,500 alarms (Table 4): Tools reduce the error rate from 2.8% to 1.8% on Qwen3 32B (-36%) and from 1.7% to 0.8% on GLM-4.5-Air (-53%); the direct-call GLM-4.5-Air configuration reaches 99.3% overall accuracy versus the 98.2% no-tools sub-agent baseline, and improvements are significant at p < 0.001 by bootstrap-by-task (Tables 12-14).
  • Specification quality is the bottleneck (Table 2): Four residual nodes account for almost all remaining errors; augmenting edge-case data raises pass@1 from 94.5% to 96.6%, and clarifying one ambiguous averaging window SOP raises it to 99.9% on those nodes.
  • Data and label efficiency: With grounding, pass@1 reaches 91.8% at only N=5 training cases per node (within 3pp of the full-set ceiling), and a label-free variant using judge-filtered pseudo-labels and multi-candidate voting reaches 91.6% (vs. the 94.5% labeled ceiling).
  • Fallback preservation: The Qwen3 32B agent fine-tuned with tools, when evaluated with tools disabled, retains 96.9% overall accuracy — statistically indistinguishable from the 97.2% no-tools baseline — confirming mixed-trajectory fine-tuning preserves the code-generation fallback.
  • Production scale: Since deployment the generated tools have been invoked across over 5,000 production alarms, and the versioning mechanism surfaced three pre-existing upstream data inconsistencies.

Authors & Credibility

All seven authors (Kalle Kujanpää, Ning Liu, Shahnawaz Alam, Yeshwanth Reddy Sura, Tianyu Yang, Kristina Klinkner, Shervin Malmasi) are from Amazon Fulfillment Technologies & Robotics. The work builds directly on the Eluna agentic warehouse system (Liu et al., 2026b, with overlapping authorship including Ning Liu, Tianyu Yang, Shahnawaz Alam, Kristina Klinkner, Shervin Malmasi), so the deployment context and the multi-agent baseline are the authors' own production system rather than a borrowed benchmark. The contribution is a real production deployment (5,000+ alarms), not a benchmark-only study, and the closest peer work they cite (Pei et al., 2025 on SOP-to-code) is noted as benchmark-only. Statistical reporting is thorough: 30-seed pass@1 with 95% CIs, paired t-tests (Table 15), and bootstrap-by-task significance (nboot=5,000) for end-to-end accuracy.

Critical Assessment

  • Novelty: Moderate. The recipe — data-collect a trace, generate a tool from the trace, test-repair against labeled cases — is a straightforward industrialization of known tool-making ideas (Voyager-style build-time construction, LATM/Trove-style tool synthesis, self-debug execution feedback). The honest novelty is the production integration and the observation that specification quality, not agent capacity, is the binding constraint once tools are grounded.
  • Rigor: Generally strong (30 seeds, CIs, paired t-tests, bootstrap-by-task), with one clean design choice worth noting: Table 3 row 3 confounds the architecture change (direct calls) with the model change (Qwen3 32B to GLM-4.5-Air), but the authors are explicit about it and run a separate controlled ablation (GLM-4.5-Air held fixed, 3 seeds) to isolate the architecture effect — a degree of honesty uncommon in industry papers.
  • Weak baselines / comparability: The headline 94.5% pass@1 is only against the authors' own ablation columns; the closest SOP-to-code peer (Pei et al., 2025) is discussed but not benchmarked head-to-head, so it is hard to tell how much of the gain comes from the tool-making recipe versus the production-incident dataset. There is no comparison to a simpler retrieval-over-cached-code baseline that would test whether the full tool-maker is needed.
  • Reproducibility: Low. This is a proprietary Amazon FC system; absolute production latencies are withheld for competitive reasons (Table 3 is normalized to baseline), the 1,500-alarm evaluation set is internal, and no code or dataset release is mentioned. The method is described in enough detail to reimplement on a new SOP, but cannot be exactly reproduced.
  • Generalization is genuinely narrow: The authors themselves state the loop has been deployed on one application (outbound dock alarm triage) and that generalization to free-form runbooks is not established. This is the single biggest caveat on significance.
  • A telling negative result: On Qwen3 235B the data-collection trace is not significantly helpful (SOP+D+R vs SOP+R p=0.096), and manual inspection shows the model mis-infers schema details from the trace (wrong column types, missing columns), so the method's payoff is model-dependent — not every capable LLM profits from grounding.
  • Headline vs. ceiling gap: The jump from 94.5% to 99.9% under SOP clarification is presented as a finding, but it also means ~5pp of the "error" the tool-maker is credited with fixing is arguably unfixable specification debt, not an agent-capability gain.

Strengths

  • Real production deployment with measured latency and correctness, plus shadow-mode promotion and human review before going live — a credible industrial evaluation rather than a benchmark-only result.
  • Clean ablations that isolate both pipeline components (trace vs. repair loop) and the architecture effect (direct calls vs. sub-agent), with appropriate significance testing and honest treatment of confounds.
  • The specification-quality finding (4 nodes hold nearly all error; clarifying the SOP reaches 99.9%) is a transferable insight that reframes the bottleneck for any SOP-driven agent.
  • Practically useful observability story: versioned tools plus a monitoring agent surfaced three real upstream data inconsistencies that free-form code generation silently adapted to — a concrete, falsifiable operational benefit.
  • Label-efficiency data (N=5 within 3pp of ceiling; label-free within 3pp) materially lowers the adoption cost for new SOP nodes.

Limitations

  • Single-domain deployment (outbound dock alarm triage); generalization to free-form runbooks or non-SOP workflows is not established by the authors.
  • No public code, data, or model release; absolute latencies are withheld, so the 42%/62% reductions cannot be mapped to wall-clock numbers.
  • No head-to-head comparison against peer SOP-to-code systems (e.g., Pei et al., 2025) or against a simpler cached-code / retrieval baseline.
  • The method's benefit is model-dependent: Qwen3 235B does not significantly profit from the data-collection trace, and weaker tool-makers (GLM-4.7 Flash) plateau far below the ceiling.
  • The tool-making improvement loop still requires human oversight for production safety; full automation of drift detection and candidate promotion is left as open work.
  • The 99.9% number comes from a small, hand-selected set of four residual nodes after targeted SOP clarification, and should not be read as a system-wide accuracy figure.

Relevance

对研究"LLM Agent 自演化与工具构造"以及"工业级低延迟 Agent 系统"的方向高度相关：它给出了把 CodeAct 编码 Agent 升级为"离线编译工具 + 在线直接调用 + 版本化可审计"的具体落地范式，并实证证明瓶颈从 Agent 能力转移到 SOP 规格质量，这对任何 SOP 驱动的 Agent（运维分诊、流程自动化）都有方法论与工程参考价值。

Questions for Further Exploration

  • How does the tool-making pipeline transfer to free-form runbooks or non-decision-tree SOPs where there is no clean per-node labeled set to drive the test-repair loop?
  • Can the data-collection trace be replaced or augmented by an automatic schema-discovery pass, and does that close the Qwen3 235B gap where the trace currently does not help?
  • What is the wall-clock latency and cost breakdown (not just normalized ratios), and how do the gains hold under concurrent-alarm load at peak?
  • Does a simpler retrieve-and-cache-code baseline (store the sub-agent's first correct code per node, reuse it) capture most of the 42% latency win without the full tool-maker + reflector machinery?
  • How stable are the generated tools under upstream API drift over months, and what is the cadence of re-running the generate-test-repair loop needed to keep the monitoring agent's false-positive rate low?
