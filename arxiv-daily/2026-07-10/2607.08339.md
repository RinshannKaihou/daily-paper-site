> **一句话总结：** 作者用逐层线性探针在 SantaCoder-1.1B 与 CodeLlama-7B 的残差流上探测类型信息，构造 Java / 无标注 Python / 带标注 Python 三套 90K 对齐样本与对抗重命名集，发现跨语言可线性解码的类型表征（即使从无类型 Python 训练也能零样本迁移到 Java），但对词典级对抗扰动仅具部分鲁棒性。

> **TL;DR:** Using layer-wise linear probes on the residual streams of SantaCoder-1.1B and CodeLlama-7B over a parallel 90K-example Java/Python dataset, the authors show that linearly decodable, cross-lingual type representations emerge — even from untyped Python (pyUnt-trained probes transfer to Java with Task 1 selectivity up to 73 for SantaCoder and 47 for CodeLlama, against an 8-class baseline) — and that this structure is only partly robust to adversarial identifier renaming, with selectivity drops (Δ) as large as 41 points.

## 核心贡献

- 首次系统性地以**形式类型语义**（而非仅语法/标识符）为靶目标，对预训练代码模型的隐藏状态进行诊断探针分析，填补了代码模型可解释性研究在类型层面的空白。
- 设计了**跨语言对齐数据集**：Java、无标注 Python（pyUnt）、带标注 Python（pyTag）三套 90K 样本实现同一底层程序，标识符与字面量按分区独立随机化，从源头杜绝跨语言词汇捷径。
- 提出**三个分离的探针任务**（Task 0 函数预测、Task 1 参数类型 8 分类、Task 2 应用结果类型 8 分类），可隔离失败模式（例如结果类型预测正确但函数选择错误）。
- 通过**pyUnt → Java 的零样本迁移**证明类型相关方向可仅从无类型代码中恢复，暗示存在一个跨语言"类型流形"。
- 引入**对抗重命名分区**（标识符系统性违背真实类型），量化类型语义对词典线索的部分依赖与部分鲁棒性。

## Key Contributions

- First interpretability study to target **formal type semantics** (argument and result types, typed function application) directly, rather than syntax or identifiers, in pretrained code models.
- A **parallel cross-lingual probing dataset** of three 90K partitions (Java, pyUnt, pyTag) implementing the same programs, with identifiers and literals independently randomized per partition to eliminate lexical shortcuts.
- **Three disentangled probing tasks** (Task 0 binary function prediction, Task 1 8-class argument type, Task 2 8-class application result) that isolate failure modes such as correct result-type prediction despite incorrect function selection.
- Evidence of a **cross-lingual type manifold**: probes trained on untyped Python (pyUnt) transfer zero-shot to Java, indicating type-relevant directions are recoverable without explicit annotations.
- An **adversarial renaming protocol** in which identifiers systematically contradict true types, used to quantify how much type decodability leans on lexical cues versus formal type structure.

## Methodology

The authors probe the residual stream activations at the fill-in-the-middle (FIM) token position of two decoder-only code models: **SantaCoder-1.1B** (24 layers, 16 heads, hidden 2048, MQA) and **CodeLlama-7B** (32 layers, 32 heads, hidden 4096, MHA). Each example is a masked call site `b : τb = f_⟨FIM⟩(a)` where the correct candidate function is disambiguated purely by type compatibility among two in-scope functions with unique input types.

They construct three 90K-example partitions — Java, pyUnt (Python without annotations), pyTag (Python with annotations) — all implementing the same underlying programs, and a type mapping between operationally equivalent base and list types across Python and Java (str↔String, bool↔boolean, int↔int, float↔float, plus list containers). An adversarial partition rewrites every variable identifier to `var_{τ′}` and every function name to `f_{τr′}`, where τ′/τr′ are uniformly sampled from the *incorrect* types.

Three tasks are defined at the masked site: Task 0 (binary function prediction), Task 1 (8-class argument type τa), and Task 2 (8-class result type τb, probing a simple compositional step of typed function application). For every model, layer, task, training partition, and evaluation setting, a separate **linear probe** is trained exhaustively across all layers with cross-entropy loss, AdamW (lr 1e-4, weight decay 0.01), 20 epochs, 4-fold cross-validation (10% held-out per fold for in-distribution validation), without feature normalization; batch size 512 (CodeLlama) / 256 (SantaCoder), on a single NVIDIA H100 80GB via PyTorch and TransformerLens.

The principal metric is **selectivity** S = acc_task − acc_control following Hewitt & Liang (2019), but using a *per-token* random-label control rather than per-type, justified by per-example randomization of identifiers. Robustness is measured as the absolute drop Δ in selectivity between standard and adversarial settings, and cross-language generalization is assessed by zero-shot transfer (train on one partition, test on the two OOD partitions).

## Results & Highlights

- **Strong in-distribution type decodability.** On Task 1 (Argument Type, 8-class), CodeLlama-7B trained and evaluated on Java reaches peak selectivity S=87 at layer 16, and SantaCoder-1.1B reaches S=87 at layer 13 — against a control-adjusted ceiling of 87.5. Task 2 (Application Result) in-domain peaks at S=91 (L14) for CodeLlama Java→Java and S=83 (L12) for SantaCoder Java→Java.
- **Cross-lingual transfer from untyped code.** Probes trained on pyUnt generalize to Java: SantaCoder achieves S=73 (Task 1, pyUnt→Java), and even Task 0 function prediction transfers non-trivially (SantaCoder Java-trained → pyUnt S=01 fails, but pyUnt-trained probes recover type-relevant directions that align with the explicitly-typed signal). This is the central evidence that type structure is recoverable from code without any annotations.
- **Selective but non-uniform transfer.** In Task 2, CodeLlama's transfer is asymmetric: pyTag→Java reaches S=84 and pyUnt→Java S=45, but Java→pyUnt collapses to S=04; CodeLlama also over-predicts `int` under cross-lingual transfer, unlike SantaCoder whose confusion matrices stay more diagonal.
- **Adversarial renaming causes large but partial degradation.** Selectivity drops (Δ) between standard and adversarial settings reach 41 points for CodeLlama Task 1 Java→pyTag (77 → 36) and 33 points for Java→pyUnt (73 → 39); SantaCoder's pyUnt→pyTag Task 1 is far more stable (85 → 76, Δ=09). Adversarial renaming shifts SantaCoder toward int→float confusion and CodeLlama toward float→int confusion, while `str` stays comparatively stable.
- **Layer-wise localization.** Selectivity peaks in mid-to-late layers (SantaCoder L11–L16, CodeLlama L14–L18). pyUnt-trained probes exhibit a late-layer decodability drop that amplifies under adversarial evaluation, and this drop appears across all models and partitions when facing adversarial renaming.
- **Container vs. element recovery.** Confusion matrices for pyUnt indicate probes recover container structure (list vs. scalar) more reliably than the element type, whereas in-domain Java/pyTag confusion matrices are nearly diagonal.
- A notable reproducibility footnote: the authors discovered (prompted by reviewer comments) that their initial CodeLlama setup was splitting the FIM marker into four separate tokens instead of processing it as a single special token; correcting this preprocessing bug improved CodeLlama's probing accuracy, and all reported numbers reflect the corrected setup.

## Authors & Credibility

The authors are Giuliano Gorgone and Fausto Carcassi of the **Institute for Logic, Language and Computation (ILLC), University of Amsterdam** — an institution with strong credentials in logic and formal semantics, which fits the paper's emphasis on *formal* type systems rather than loose semantic proxies. The work acknowledges computing access on SURF's Snellius cluster, cites the standard probing-literature canon (Hewitt & Liang 2019, Belinkov 2022, Alain & Bengio 2017), TransformerLens (Nanda & Bloom), and the recent activation-steering work on type prediction by Lucchetti & Guha (BlackboxNLP 2025) that directly motivates this study. Code and datasets are released at `https://github.com/anticleiades/TypeProbe`. The paper transparently discloses a corrected preprocessing bug in response to reviewer feedback, which is a positive signal of rigor. The venue framing (ESSLLI registration waiver, ILLC travel support) suggests a short conference/workshop paper rather than a full archival venue, consistent with its concise scope.

## Critical Assessment

- **Novelty:** Genuine but incremental. The probing methodology is standard (Hewitt & Liang control tasks, linear probes); the contribution is applying it to *formal type semantics* and *cross-lingual* (Java↔Python) type transfer, which prior code-model probing work (Troshin & Chirkova 2022) had not directly targeted. The framing as a "cross-lingual type manifold" is novel in the code-LM setting but closely mirrors established multilingual-NLP manifold results (Kudugunta et al. 2019, Ammar et al. 2016), which the authors themselves acknowledge.

- **Rigor:** Methodologically careful in the right places — exhaustive layer-wise probing, control-task selectivity to avoid probe-capacity inflation, per-example identifier randomization to kill shortcuts, and a disentangled three-task design. However, the authors depart from Hewitt & Liang's per-type control and substitute a **per-token random-label control**, which they call "a reasonable approximation." This is a real weakening of the selectivity guarantee: per-token controls are noisier and could systematically under- or over-estimate selectivity relative to the canonical per-type baseline, and no sensitivity analysis is provided. Additionally, **linear probes confirm decodability but not causality** — the authors correctly flag this themselves, noting linear methods cannot recover features in superposition (Elhage et al. 2022) and cannot prove the extracted representations *drive* code generation.

- **Reproducibility:** Strong. Fixed 90K partitions, fixed type mapping, released code/data on GitHub, single-H100 compute budget, and a public, corrected CodeLlama preprocessing note. The fact that a tokenization bug slipped in initially is slightly concerning for out-of-the-box reproduction but is now disclosed and fixed.

- **Significance:** Modest. The scope is deliberately narrow: only **base types and list containers**, only **two languages** (Java, Python), only **two models** (1.1B and 7B), and only **two candidate functions** per call site (Task 0 is binary). Generalization to generics, subtyping, Rust ownership, or C++ templates is left entirely to future work. The practical upshot for type-constrained decoding (the motivating application, citing Mündler et al. PLDI 2025's finding that ~94% of TypeScript generation errors are type-check failures) is suggestive but not yet demonstrated — no downstream generation experiment connects the probeable representation to improved type accuracy.

- **Red flags:** (1) The reported selectivity numbers rest on a non-canonical control-task construction whose validity is asserted rather than validated. (2) The dataset is *programmatically generated* with controlled complexity ranging "from simple one-line functions to more involved conditionals and loops" — i.e., synthetic and far from real-world code, which limits how much the "type manifold" finding reflects naturalistic code distributions. (3) Several cross-lingual cells are near-chance (e.g., CodeLlama Task 2 Java→pyUnt = 04, Task 0 Java→pyUnt = 04 for both models), undercutting the universality of the cross-lingual claim; the authors are honest about these failures but the headline "cross-lingual type representations emerge" is somewhat stronger than the full transfer matrix supports.

## Strengths

- Clean, well-motivated experimental design with shortcut-killing identifier randomization across partitions.
- Disentangled three-task framework genuinely isolates distinct competencies (function selection vs. argument typing vs. compositional result typing).
- The pyUnt→Java transfer result is the most interesting and counterintuitive finding: type-relevant geometry is recoverable from code with no type annotations at all.
- Honest reporting of failure cells (near-chance transfers, asymmetric CodeLlama behavior, int over-prediction) rather than cherry-picking.
- Transparent disclosure and correction of the CodeLlama FIM tokenization bug.
- Public code and dataset release.

## Limitations

- Synthetic, programmatically generated dataset limited to base types and lists — no generics, subtyping, user-defined types, or real-world code.
- Only two languages (Java, Python) and two models (1.1B, 7B); no scaling analysis.
- Task 0 reduced to a binary choice between two candidate functions, a simplification of realistic function resolution.
- Non-canonical per-token control task substitutes for Hewitt & Liang's per-type control without sensitivity analysis.
- Linear probes cannot recover superposed features or establish causal roles; the authors recommend SAEs, dictionary learning, SVCCA, and activation patching as next steps.
- No downstream evaluation connecting the probeable type manifold to improved type-constrained decoding, so the practical payoff for code generation is unproven.
- Concise (short-paper) format limits depth of ablation and analysis.

## Relevance

对关注代码大模型可解释性与类型推理的研究者，本文提供了一套可控、可复现的探针框架与跨语言类型表征的初步实证，直接呼应 type-constrained decoding 与激活调控（activation steering）方向，可作为后续因果分析（SAE / activation patching）与更复杂类型系统研究的基线。

## Questions for Further Exploration

- Does the cross-lingual type manifold survive on real-world (non-synthetic) code corpora such as The Stack or permissively-licensed GitHub repositories, where identifier distributions are not randomized?
- Would a per-type control task (the canonical Hewitt & Liang construction) materially change the reported selectivity values, or is the per-token approximation truly innocuous?
- Can activation patching or SAEs establish a *causal* role for the identified type directions, e.g., by editing them to flip predicted return types in a controlled way?
- How does the type manifold scale with model size (beyond 7B) and does it extend to richer type systems — generics, subtyping, Rust ownership, C++ templates?
- Can the probeable type representation be exploited at inference time to reduce type-check failures in generation, closing the loop with type-constrained decoding (Mündler et al. 2025)?
- Why is Java→pyUnt transfer so much weaker than pyUnt→Java transfer for both models — does Java's explicit annotations create a representation that is *over-specialized* to typed surfaces?
