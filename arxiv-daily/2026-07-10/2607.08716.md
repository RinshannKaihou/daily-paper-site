> **一句话总结：** 提出一种"主动干预式"记忆代理（而非被动检索），在长时序任务中沿未修改的动作代理旁运行，通过两阶段流程（维护结构化记忆库 + 选择性注入提醒或保持沉默）缓解"行为状态衰减"问题，在 Terminal-Bench 2.0 上为 Sonnet 4.5 提升 +8.3 pp，在 τ²-Bench 上提升 +6.8 pp。

> **TL;DR:** The authors identify a failure mode they call **behavioral state decay** — decision-relevant state (requirements, failed attempts, diagnoses, open subgoals) stops influencing an agent's next action during long-horizon execution, even when still in context. They introduce a plug-and-play **memory agent** running alongside an unmodified action agent that performs a two-phase cycle (maintain a structured memory bank, then selectively inject a memory-grounded reminder or stay silent). On **Terminal-Bench 2.0**, memory intervention lifts Claude Sonnet 4.5 from 37.6% to 45.9% (**+8.3 pp**) and on **τ²-Bench** from 55.0% to 61.8% (**+6.8 pp**), beating passive bank exposure, always-on injection, advisor-only guidance, and Mem0 retrieval in ablations. An early open-weight study trains Qwen3.5-27B via SFT + GRPO, with partial transfer to held-out Terminal-Bench (+3.5 pp).

## 核心贡献

- **提出并命名"行为状态衰减（behavioral state decay）"这一失败模式**：在长时序执行中，本应约束未来动作的信息（任务要求、环境事实、过往尝试、错误诊断、未完成子目标）即便仍在上下文窗口内，也会停止对下一步决策产生可靠影响。这把记忆问题的焦点从"能否存取"转向"是否、何时、如何干预"。
- **设计了两阶段记忆干预架构（解耦维护与干预）**：Phase 1 通过显式工具调用（`memory_update_status` / `memory_save_knowledge` / `memory_save_procedural` / `memory_delete`）维护由"私有状态 + 知识记忆 + 程序性记忆"组成的结构化记忆库；Phase 2 在更新后的记忆库上决定向动作代理注入简短提醒还是保持沉默（null intervention 是显式动作）。
- **将"沉默"作为一等公民的干预策略**：明确鼓励记忆代理只在被记忆项可能影响下一步决策时才介入，避免泛泛的战略建议或重复可见信息；触发采用固定间隔以隔离干预策略本身的效应。
- **跨基准、跨模型强度的一致提升**：在 Terminal-Bench 2.0 与 τ²-Bench（airline/retail/telecom）上，弱（Sonnet 4.5）与强（Opus 4.6）动作代理均获益，且消融表明"选择性干预 + 维护式记忆"的组合最均衡。
- **初步验证干预策略可被开源模型学习**：在 SETA 上用 SFT 蒸馏 + GRPO 校准训练 Qwen3.5-27B（动作代理 Qwen3.5-122B-A10B 冻结），验证奖励从基线 0.709 提升至 GRPO 的 0.734，并在留出的 Terminal-Bench 上取得 +3.5 pp 的部分迁移。

## Key Contributions

- **Names and formalizes "behavioral state decay"** as a central long-horizon failure mode: execution state that should constrain future behavior stops exerting reliable control over the next action, even when still present in the context window — reframing memory from a storage/retrieval problem to an *intervention-control* problem.
- **A two-phase, decoupled memory-intervention architecture** that leaves the action agent unmodified: Phase 1 maintains a structured bank `(status, knowledge, procedural)` via explicit tool calls; Phase 2 decides whether to inject a concise, memory-grounded reminder or emit an explicit null intervention.
- **Elevates "silence" to a first-class policy action.** The memory agent is explicitly discouraged from generic advice, restating visible observations, or taking over planning; a fixed-interval trigger isolates the effect of the intervention policy itself rather than retrieval mechanics.
- **Consistent, ablation-backed gains across two benchmarks and two action-agent strengths.** Selective intervention beats passive full-bank exposure, always-on injection, advisor-only guidance (no bank), and Mem0 vector+BM25 retrieval on the macro-average across heterogeneous τ²-Bench domains.
- **Early evidence that the intervention policy is learnable by an open-weight model.** SFT distills prompted-memory trajectories (teaching the interface and discipline); GRPO further calibrates *when* remembered state should re-enter the control loop, with positive transfer to held-out Terminal-Bench 2.0.

## Methodology

The setting is a long-horizon action agent producing actions `a_t ~ π_A(· | x, τ_<t)` over a trajectory of observations. The scaffold's exposed context may be truncated/filtered, so not all trajectory information remains available. The authors augment — not modify — the action agent with a separate **memory agent π_M** invoked at the first step and then at a fixed interval.

At each memory step π_M observes the task description, a recent trajectory window `w_t = W_k(τ_<t, o_t)` (k=8 messages in experiments), and the current memory bank `B_{t-1}`, then executes two phases:

- **Phase 1 — Memory management (bank edits).** π_M returns a list of predefined tool calls; the system executes them. The four calls map to the bank's three components plus deletion: `memory_update_status` (a private progress/risk view never shown to the action agent), `memory_save_knowledge` (stable facts: requirements, environment properties, paths, configs, verified facts), `memory_save_procedural` (attempts/outcomes: failed commands, successful fixes, ruled-out hypotheses, diagnostic signals), and `memory_delete` (by identifier). Output is a sequence of structured bank edits, *not* a free-form summary.
- **Phase 2 — Intervention selection (transient injection).** Conditioned on the updated bank and recent trajectory, π_M either emits a targeted reminder `r_t` or a null intervention ∅. A non-null reminder is injected as a separate transient memory context into the next action-agent call; the action agent's base instructions, tools, and decoding are otherwise unchanged. Null is an explicit action — the policy must *decide* when remembered state is useful enough to enter the control loop.

The trigger function `g(t)` is a fixed interval (first step, then every step in the main runs) chosen to isolate the intervention policy's effect; more selective triggers (after tool errors, repeated commands, large context shifts) are noted as future work.

**Learning the policy (preliminary).** The architecture does not require training, but the authors explore an open-weight memory agent: SFT distills prompted-memory trajectories (supervising both Phase 1 operations and Phase 2 decisions), then GRPO optimizes intervention calibration. Because task-level verifier rewards are sparse over many memory calls, RL updates are focused on **pivot turns** that offline rollouts flag as likely to affect downstream success.

## Results & Highlights

- **Terminal-Bench 2.0 (full set, 85 paired tasks; memory agent = Claude Opus 4.6):** Sonnet 4.5 action agent improves **37.6% → 45.9% pass@1 (+8.3 pp)**; the stronger Opus 4.6 action agent still gains **43.5% → 45.9% (+2.4 pp)**, showing the benefit is not merely compensating for limited capacity.
- **τ²-Bench (278 tasks: 50 airline + 114 retail + 114 telecom; Sonnet 4.5 action agent):** task-weighted average improves **55.0% → 61.8% (+6.8 pp)**. Per-domain Sonnet lifts: airline **68.0% → 78.0% (+10.0 pp)**, retail **49.1% → 58.8% (+9.6 pp)**, telecom **55.3% → 57.9% (+2.6 pp)**.
- **τ²-Bench with Opus 4.6 action agent:** task-weighted average **66.2% → 68.7% (+2.5 pp)**; airline is exactly flat (76.0% → 76.0%, +0.0 pp, likely a ceiling effect), retail +4.4 pp, telecom +1.8 pp.
- **Ablations (Table 2, τ²-Bench, Sonnet 4.5, macro/micro):** The full two-phase agent leads the **macro-average at 64.3** (micro 61.2) vs. baseline macro 57.5. Removing Phase 2 (full-bank context) drops to macro 61.5 (−2.8); always-inject gives macro 63.5 but only ties micro (61.5) within run variance and loses on airline; injection-only/advisor (no bank) gives macro 61.0 and *hurts airline to 62.0* (below the 68.0 baseline); **Mem0 (vector+BM25 top-10) gives macro 62.1, micro 60.8** and does not beat baseline on airline, confirming retrieval ≠ deciding when to intervene.
- **Open-weight training (Table 4, Qwen3.5-122B-A10B action frozen, Qwen3.5-27B memory):** on SETA validation, action-only verifier reward = **0.709**; an *untrained* 27B memory agent *hurts* to 0.693 (−0.016); **SFT recovers to 0.720 (+0.011)**; **GRPO reaches 0.734 (+0.025)**. On held-out Terminal-Bench 2.0, the SETA-trained memory agent transfers: **37.6% → 41.1% pass@1 (+3.5 pp)**.
- **Qualitative mechanisms (Table 3):** memory helps by reactivating state right before it would be ignored — requirement/policy reactivation (airline compensation, retail modification rules), environment grounding (Git server setup, ARS file-write failure), failure-loop avoidance (adaptive rejection sampling retries), diagnostic carryover (regex edge cases, SQLite gcov config), and entity/progress tracking (telecom line lookup, retail auth state).

## Authors & Credibility

All eight authors (Yifan Wu, Lizhu Zhang, Yuhang Zhou, Mingyi Wang, Bo Peng, Serena Li, Xiangjun Fan, Zhuokai Zhao) are affiliated with **Meta AI**, consistent with a well-resourced industry lab with access to frontier Claude models (via APIs) and the compute needed for SFT/GRPO on 27B models. Code is publicly released at `https://github.com/yifannnwu/proactive-memory-agent`, and the framing is positioned carefully (and honestly) against a thorough related-work survey spanning MemGPT, Mem0, Memory-as-Action, Context-Folding, Mem-α, Reflexion, Self-Refine, Tree of Thoughts, and Anthropic's Advisor tool. The empirical study is internally consistent with its own claims (gains larger for weaker agents, ablations support the two-phase design, training is explicitly labeled "preliminary").

## Critical Assessment

- **Novelty:** The conceptual reframing — memory as a *selective intervention policy* where "silence" is an explicit action — is genuinely the core idea, and is more than a relabeling of advisor models or RAG. The "behavioral state decay" framing is a useful, citable diagnostic. However, the two-phase "maintain-then-maybe-inject" loop is a fairly natural composition of existing ingredients (structured memory banks + an LLM decider), so the novelty is more in the *formulation and ablation story* than in any single mechanism.
- **Rigor:** The ablation suite is the strongest part — it cleanly isolates Phase 1 vs. Phase 2, always-on vs. selective, bank vs. no-bank (advisor), and includes a Mem0 retrieval baseline. But **no confidence intervals or error bars are reported anywhere**, despite every τ²-Bench episode being a *single sampled conversation* (a notoriously high-variance setting). Several claimed deltas are small (Opus +2.4 / +2.5 pp; always-inject vs. selective micro gap of 0.3 pp) and sit well within plausible sampling noise. The paper itself concedes "within expected run variance" for the always-inject comparison, which undermines the strength of the ablation conclusions.
- **Reproducibility:** Code is released and the harness (Terminus-2-style, τ²-Bench) is standard, but **Table 1 and Table 2 report different numbers for nominally the same configuration**: e.g., τ²-Bench retail Sonnet 4.5 with the full memory agent is 58.8% in Table 1 but 57.0% in Table 2, and the task-weighted micro average is 61.8% (Table 1) vs. 61.2% (Table 2). These discrepancies are consistent with re-running single-sample episodes but are never acknowledged, which complicates exact reproduction. The training study uses **SETA**, an external task collection, with no detail on hyperparameters, pivot-turn labeling, or GRPO reward design beyond a one-paragraph description.
- **Significance:** Solid, practitioner-relevant gains on two credible long-horizon benchmarks, plus a real (if early) open-weight result. The architectural claim — that *when* to surface memory matters as much as *what* — is well-supported and likely to influence follow-on work on context/memory management for agents.
- **Red flags:** (1) **Table 1 vs. Table 2 number mismatches** for identical configs, unexplained. (2) **Terminal-Bench excludes 4 of 89 tasks** (85 reported) due to docker failures — reasonable, but with n=85 and single-sample pass@1, the +8.3 pp Sonnet headline rests on ~7 additional solved tasks. (3) The "stronger agent" gains concentrate in retail/telecom while **airline on Opus is exactly +0.0 pp**, hinting at ceiling effects that are not discussed. (4) No token-cost or latency overhead quantification, even though the memory agent (an Opus call at every step) roughly doubles model cost per step — a material deployment concern given the paper's own motivation about avoiding "unnecessary token or latency overhead." (5) The single memory-system baseline is Mem0; no comparison to Memory-as-Action, MemGPT, or Reflexion-style episodic reflection, though all are discussed.

## Strengths

- Clean, well-motivated conceptual contribution (behavioral state decay; memory-as-intervention) that reframes the problem usefully for the community.
- The action agent is left genuinely unmodified — the module is plug-and-play with frontier agents and existing harnesses, lowering adoption friction.
- Ablations are thoughtfully designed to test each design axis (bank vs. none, always-on vs. selective, advisor vs. grounded, retrieval vs. intervention) rather than only "with vs. without memory."
- Macro-averaging across heterogeneous τ²-Bench domains exposes that the *selective* policy is the most *balanced* even when always-inject ties on micro — a fair, non-cherry-picked presentation.
- Honest scoping: the training study is explicitly "preliminary," and the paper does not overclaim the open-weight result beyond "partial transfer."
- Code released; benchmarks are public and standard.

## Limitations

- **No uncertainty estimates** despite single-sample, high-variance evaluation; several key deltas (Opus gains, always-inject micro tie) are within plausible noise.
- **Unacknowledged number discrepancies between Table 1 and Table 2** for the same configurations complicate reproducibility.
- **Ceiling effects on the strongest agent/domain** (Opus airline +0.0 pp) are not analyzed.
- **Cost not quantified:** running a frontier-model (Opus 4.6) memory call every step roughly doubles per-step inference cost, yet the paper motivates itself partly by avoiding overhead. The fixed "every step" trigger maximizes cost; the authors defer more selective triggers to future work.
- **Narrow memory-system baselines:** only Mem0 is compared empirically; Memory-as-Action, MemGPT, Reflexion, and Context-Folding are discussed but not run.
- **Training study is small and task-specific:** SETA-only, one model size (27B), sparse-reward RL with an underscribed "pivot-turn" credit-assignment heuristic, and transfer tested on a single held-out benchmark.
- **Closed action/memory models in the main results** (Claude Sonnet/Opus) limit what the open-source community can directly reproduce from the headline numbers.

## Relevance

与"自演化智能体与自动化研究（Self-Evolving Agents & Automated Research）"方向高度相关：长时序智能体的状态维护与"是否干预"的决策，正是自改进/自主研究回路中持续运行多步工具调用的核心瓶颈；同时其"行为状态衰减"的诊断与选择性干预策略，对智能体推理质量评估（LLM Inference Evaluation）中的一致性与可靠性分析也有借鉴价值。

## Questions for Further Exploration

- How large is the **token-cost / latency overhead** of an every-step Opus memory call, and does a learned selective *trigger* (invoke only after tool errors, repeated commands, or large context shifts) recover most of the gain at a fraction of the cost?
- Would the Table 1 vs. Table 2 discrepancies survive **multi-sample pass@k with confidence intervals**, and do the small Opus deltas (+2.4/+2.5 pp) remain statistically significant?
- Can the memory agent be **co-trained with the action agent** (rather than frozen action + trained memory), and does that close the gap to or exceed a single frontier agent with native long-context?
- How does selective intervention compare empirically against **Memory-as-Action, MemGPT, and Reflexion**, not just Mem0?
- Is there a principled way to detect **ceiling effects** (e.g., Opus on airline at 76%→76%) and auto-deactivate memory to avoid wasted calls?
- Can the "pivot-turn" credit-assignment heuristic used for GRPO be **learned or verified**, rather than labeled from offline rollouts, to make the open-weight policy more generalizable beyond SETA/Terminal-Bench?
