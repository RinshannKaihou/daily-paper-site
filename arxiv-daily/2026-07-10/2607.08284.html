Understanding Axes of Difficulty For Long Context Tasks Via PredicateLongBench

arXiv: 2607.08284

Date: 2026-07-10

https://arxiv.org/abs/2607.08284

一句话总结： PredicateLongBench 提出一族"在长词表中找出满足谓词约束（前缀/后缀/包含/字典序）的最长连续子序列"的算法上简单却可沿难度轴系统缩放的任务，并定义了五条难度轴（计算 arity、量词复杂度、对抗 decoy、搜索空间大小、上下文结构）；在 128K–365K token 上下文上评测 GPT-5.4、Gemini 3.1 Pro、Opus 4.6 及多个开源模型后发现，闭源强模型在 baseline 上能达 87%–97%，但只要插入 8 个分散的 near-sorted decoy，所有模型全部崩溃到 ≤2%（合成数据）/ ≤36%（LongBench 真实文本），Opus 4.6（high reasoning）以 9/12 变体最佳但 decoy 任务上仍只有 0%–1%。
TL;DR: PredicateLongBench is a synthetic + real-text long-context benchmark built around a conceptually simple task — find the longest contiguous word subsequence satisfying a predicate (prefix/suffix/containment or lexicographic order) — whose difficulty can be scaled along five named axes (predicate arity, quantifier complexity, adversarial decoys, search-space size, context structure). On 128K-token synthetic lists and up-to-365K-token LongBench v2 documents, Opus 4.6 (high) hits 97% on the synthetic lexicographic baseline and GPT-5.4 (high) 95% on the unary baseline, but inserting just 8 scattered near-sorted decoys collapses every frontier model to ≤2% on synthetic data (Opus 97%→1%, GPT-5.4 62%→0%), and the universal-quantifier / "longest sequence" variants drop Opus from 97% to 23% and GPT-5.4 from 62% to 12%; scaling GPT-5.4 output budget from 16K to 128K tokens recovers the baseline (→92%) but not the scattered-decoy variants, while open-source models (Qwen 3.5 397B, GLM 5.1, MiniMax 2.7) sit near 0% on nearly all lexicographic variants.

核心贡献

  • 提出 PredicateLongBench 任务族： 把长上下文评测简化为"在一长串单词中找出满足给定谓词约束的最长连续子序列"，任务在算法上简单（无需 LLM 生成器或 LLM 裁判、答案可程序化验证），却能让前沿模型集体翻车。
  • 系统定义并探索五条难度轴： 计算复杂度（谓词 arity 1→2，即 unary prefix/suffix/containment → binary 字典序）、量词复杂度（∃any → ∀all，以及"找最长序列"=隐含 max）、对抗 decoy（near-miss near-sorted decoy 与 multi-list 跨边界 decoy）、搜索空间大小（token 数固定但单词数从 26K→60K）、上下文结构硬度（decoy 分散 vs. 聚类）。这是对现有基准"只沿 context length 单轴扩难度"的关键补强。
  • 两条互补的数据生成管线： 全合成管线（base-26 等长随机字符串，长度 8/12，按 Qwen tokenizer 约 128K token，每变体 100 例，含"唯一解"修复循环）；真实世界管线（从 LongBench v2 的 120 篇文档中过滤得到 100 篇，token 数最高 365K，对开源自回归模型取 <170K 的 93 篇），保留自然词频分布。
  • 给出量化的难度梯度与失败模式诊断： 用 9 个模型（GPT-5.4、Gemini 3.1 Pro preview、Opus 4.6、MiniMax 2.7、GLM 5.1、Qwen 3.5 397B，覆盖 high/low/none reasoning effort）在 12 个任务变体上的准确率表，证明性能随难度轴"均匀崩塌"；并诊断失败根因为 token 预算耗尽、模型退化到逐字复述输入、以及首词定位准确率为 0%（即模型不是"找到但复述不了"，而是根本没定位到）。
  • 揭示"搜索空间 > token 数"与"结构 > 分散"两条被低估的洞察： 固定 128K token 但把单词数从 26K 翻到 60K，GPT-5.4 high 准确率从 92% 掉到 10%；把分散 decoy 改成聚类在目标附近，Opus 从 1% 反弹到 98%（合成）/93%（LongBench），证明 decoy 的杀伤力本质上是"长上下文"属性而非 decoy 本身。

Key Contributions

  • Introduces PredicateLongBench, a family of long-context tasks that are algorithmically trivial to specify and verify (find the longest contiguous word subsequence satisfying a predicate) yet expose sharp failure modes in frontier LLMs, deliberately avoiding LLM-based generation or LLM-based judging.
  • Formalizes five axes of difficulty and instantiates task variants along each: (1) computation via predicate arity (unary prefix/suffix/containment vs. binary lexicographic ordering); (2) quantifier complexity (existential "any sequence" → universal "all sequences", and "find the maximum-length sequence"); (3) adversarial decoys (near-sorted near-miss decoys and multi-list cross-boundary decoys); (4) search-space size (fixed token budget but more, shorter words); (5) context-structure hardness (decoys scattered vs. clustered near the target).
  • Provides two complementary generation pipelines — fully synthetic (base-26 length-8/length-12 random strings, ~128K tokens by Qwen's tokenizer, with a fixup loop guaranteeing a unique length-k solution) and real-world (filtered LongBench v2 documents, up to 365K tokens, preserving natural lexical distribution; 100 examples per variant, 93 for open-source models on long docs).
  • Delivers a 9-model × 12-variant evaluation grid (GPT-5.4, Gemini 3.1 Pro preview, Opus 4.6, MiniMax 2.7, GLM 5.1, Qwen 3.5 397B across high/low/none reasoning), plus ablations on reasoning effort (Fig. 9), output-token budget (Fig. 10), first-word-locating accuracy, and qualitative failure-trace analysis of open-source models.
  • Surfaces two counterintuitive, underemphasized findings: search-space size hurts independently of raw token count (26K→60K words at fixed 128K tokens drops GPT-5.4 from 92% to 10%), and the lethality of near-sorted decoys is a long-context effect — clustering the same decoys next to the target restores Opus from 1% to 98% on synthetic data.

Methodology

PredicateLongBench is built around a single abstract task: given a long context that is a space-separated list of words and a predicate class H ⊆ { (a_n) → {0,1} }, identify the largest k such that a contiguous subsequence w = (w_1,…,w_k) satisfies a list of predicates f_1,…,f_j ∈ H (each applied at a fixed position or pairwise).

Predicates. Unary predicates operate on a single word: prefix pre_{s,i}, suffix suf_{s,i}, and containment con_{s,i} (string s is a prefix/suffix/substring of word w_i). Binary predicates model pairwise interactions; the paper uses lexicographic ordering, decomposed as f(w_1,…,w_k)=1 iff w_i ≼ w_{i+1} for all i, i.e. adjacent pairs must be non-decreasing.

Axes of difficulty (§3.3).

  • Computation: increasing predicate arity from 1 to 2 raises the per-step check from independent word tests to pairwise comparisons.
  • Adversarial decoys: (a) near-miss decoys — for unary tasks, 5% of words per position are edited to satisfy one predicate at random while keeping a unique length-5 solution; for lex tasks, 8 length-5 blocks that are sorted except for one swapped interior pair, with flanks preventing run extension. (b) Multi-list decoys — the list is split into 20 sublists and a valid length-k sorted run is planted across every adjacent boundary; the model must return a run contained in a single sublist, so each cross-boundary run is a decoy.
  • Search-space size: keep the token budget fixed (128K) but use shorter words so the prompt holds ~60K words instead of ~26K, increasing the search space an exact algorithm would need.
  • Quantifier complexity: switch the existential "any sequence" query to a universal "all sequences" query (still a unique answer), or ask for the maximum-length lexicographic run, forcing the model to verify the entire context.
  • Context structure hardness: cluster the near-sorted decoys together with the target versus scattering them, holding decoy count fixed.

Generation (§3.4). Synthetic words are sampled without replacement from {0,…,26^L−1} and base-26-encoded to length-L lowercase strings (L=8 for lex, L=12 for unary); the target plant block is built with sorted interior k positions and two flanks that prevent extension, then a fixup loop swaps words to destroy any unintended length-≥k run, guaranteeing exactly one valid solution. Real-world tasks draw 120 documents from LongBench v2 across 12 task categories (literary, legal, governmental, academic, financial, etc.), filter to 100 via Algorithm 7 (discard HTML, longest natural run > 10, runs dominated by a single character/word, duplicate words, apostrophes), and use the document's own word stream as the base context (k = the longest natural lexicographic run, length 6–10). Decoys for real text are built from the document's own vocabulary so they cannot be spotted as out-of-distribution.

Tasks (Table 1). Unary family: ∃-query baseline, ∀-query, ∀-query + 5% distractors (synthetic only). Lexicographic family: Locate-k (synth + LongBench), + near-sorted scattered decoys (synth + LongBench), + near-sorted clustered (synth + LongBench), + multi-list 20-sublist (synth), Longest (synth + LongBench), larger search-space (synth). k=5 for synthetic; k=longest natural run for LongBench.

Evaluation. 100 examples per variant (93 for open-source on LongBench), max output 16K tokens (some runs up to 128K), exact-match accuracy against the unique planted solution. Closed models called via API with reasoning effort high/low/none; open-source (MiniMax-2.7, Qwen 3.5 397B, GLM-5.1) run on A100/B200/H100 at temperature 1.0. The authors also probe why models fail: first-word accuracy of wrong outputs (0% everywhere), output-token-budget sweeps, and qualitative reads of open-source reasoning traces.

Results & Highlights

  • Unary ∃-baseline is nearly saturated by closed models, lex baseline is not. On the synthetic unary "any sequence" baseline, GPT-5.4 none hits 96% and Gemini high 93%, while Opus 4.6 high reaches 87% (its misses are all refusals, not errors). On the synthetic lexicographic Locate-k baseline, only Opus 4.6 high (97%) is strong; GPT-5.4 high is just 62%, Gemini high 43%, and the open-source trio (MiniMax 2.7, GLM 5.1, Qwen 3.5 397B) score 8% / 8% / 0%. On the LongBench lexicographic baseline Opus drops to 61% and GPT-5.4 to 36%; every open-source model is ≤3%.
  • Scattered near-sorted decoys collapse every model to ≤2% on synthetic data. Adding just 8 near-sorted decoys to the synthetic lex-locate task drops Opus 4.6 high from 97% → 1%, GPT-5.4 high from 62% → 0%, Gemini high from 43% → 2%; open-source models stay at 0%. On the unary ∃-task, adding 5% per-position distractors drops Opus 4.6 high from 87% → 7% and GPT-5.4 high from 95% → 5%.
  • Real text is a weaker adversarial setting than synthetic strings. The same scattered near-sorted decoys hurt far less on LongBench (Opus 61% → 49%, GPT-5.4 36% → 36%, Gemini 6% → 11%) — the authors attribute this to natural-text words carrying distributional distinguishing signal that uniform random strings lack.
  • Clustering the decoys restores or exceeds baseline — the hardness is a long-context property. Moving the identical near-sorted decoys next to the target lifts Opus 4.6 high from 1% → 98% (synthetic) and 49% → 93% (LongBench), GPT-5.4 high from 0% → 70% / 36% → 68%, and Gemini high from 2% → 64% / 11% → 56%.
  • Quantifier complexity (∃ → ∀ / max) is itself a difficulty axis even when the answer is unchanged. Switching unary "any sequence" to "all sequences" (same unique solution) drops most closed models (GPT-5.4 high 95% → 93%, Gemini high 93% → 59%), and the "longest lexicographic sequence" variant drops Opus high 97% → 23% and GPT-5.4 high 62% → 12%.
  • Search-space size matters independently of token count. Holding the prompt at 128K tokens but increasing the word list from 26K to 60K (shorter words) drops GPT-5.4 high on lex-locate from 92% (at 128K output budget) to 10%.
  • Reasoning effort and output budget help the baseline but not scattered decoys. GPT-5.4 with no reasoning collapses to ≤5% on every variant except the unary baseline; scaling GPT-5.4 high's output cap from 16K → 128K lifts lex-locate toward ~90% and clustered-decoy variants into the 80–90s, but scattered decoys and multi-list "barely move" with more budget.
  • Models fail at locating, not at reproducing. On all incorrect outputs the first-word accuracy of the model's proposed sequence is 0% across all models — i.e., the failure is search, not transcription.
  • Opus 4.6 dominates, open-source models mostly fail. Opus 4.6 high is the best model on 9/12 variants; open-source models (Qwen 3.5 397B, GLM 5.1, MiniMax 2.7) are at 0% on every lexicographic variant including the baseline.

Authors & Credibility

Both authors (Siddhartha Jain, Ameya Velingker) are at NVIDIA. The work is framed as a NeurIPS-format preprint (includes the NeurIPS Paper Checklist, marked for OpenReview/NeurIPS submission conventions) and the authors commit to releasing both the full prompt dataset and the code to generate, validate, and evaluate every task variant (checklist items 4 and 5 answered [Yes]). The benchmark is built on top of LongBench v2 (properly cited, Apache 2.0), and the experimental protocol — exact prompts in Appendix B, per-model inference setup in Appendix C, full construction algorithms (Algorithms 1–8) in Appendix A — is described in enough detail to reproduce the data without the released code. The model roster is current and broad (GPT-5.4, Gemini 3.1 Pro preview, Opus 4.6, MiniMax 2.7, GLM 5.1, Qwen 3.5 397B) and spans reasoning-effort levels. The main statistical weakness (acknowledged in the checklist, item 7 = [No]) is the absence of error bars or confidence intervals; the authors justify this by noting the effects are very large relative to any plausible noise.

Critical Assessment

  • Novelty: The core move — recasting long-context evaluation as "find a longest predicate-satisfying subsequence" and then parametrizing difficulty along named axes — is genuinely useful and not obviously covered by RULER, HELMET, NoLiMa, or the graph-based benchmarks the authors survey. The most original findings are the search-space-size-at-fixed-token-count axis (26K→60K words, 92%→10%) and the clustered-vs-scattered decoy ablation (Opus 1%→98%), neither of which is a lever that prior benchmarks isolate cleanly.
  • Rigor: The generation pipeline is carefully engineered to guarantee a unique solution per instance (plant block with non-extendable flanks + a fixup loop that destroys accidental length-≥k runs), which is what makes exact-match scoring legitimate. The "first-word accuracy 0% on wrong answers" probe is a clean way to rule out the "located-but-mistranscribed" confound. However, the paper reports single-run accuracies with no error bars, no seed variation, and no multiple seeds (checklist item 7 = [No]); for the variants hovering near 0–10% the per-instance count is only 100, so several of the open-source 0% numbers and the small-clustered deltas are within plausible sampling noise even if the headline effects are not.
  • Reproducibility: Strong on data and prompts (full algorithms + verbatim prompt templates + per-model hyperparameters), and the authors promise code + data release. Reproducibility is weaker on the closed-model side: GPT-5.4, Gemini 3.1 Pro preview, and Opus 4.6 are moving proprietary endpoints whose behavior the authors themselves note changed during the study (e.g., the Opus refusal artifact on the unary baseline that inflates the apparent ∀-gain), so the closed-model numbers are not bit-reproducible and the absolute rankings could shift between this preprint and any camera-ready.
  • Significance: The benchmark fills a real gap — most long-context suites scale only context length or distractor density, and many (NIAH variants) are saturated — and the failure modes it exposes (catastrophic sensitivity to scattered decoys; near-total open-source failure on binary predicates) are actionable. But the tasks are deliberately artificial, so the degree to which "PredicateLongBench accuracy" predicts real downstream long-context competence (document QA, codebase reasoning, RAG) is asserted rather than demonstrated; the LongBench-v2-sourced variant is the closest link and is only used for the lexicographic family.
  • Red flags / weak spots:
  • The headline "Opus dominates" conclusion is partly an artifact of Opus's refusal behavior on the unary baseline — the paper itself flags that Opus's baseline misses are all refusals, not errors, which makes the reported 87% understate its true capability and the apparent ∀-query "gain" (87%→99%) illusory. This is disclosed honestly but still skews the comparative table.
  • The quantifier-complexity axis is only stress-tested with a unique-solution dataset — a limitation the authors admit, but it undercuts the strongest possible version of the ∀ claim (which would require multiple satisfying sequences). The "Longest" variant is more convincing but conflates max-tracking with global scanning.
  • k=5 throughout synthetic tasks is small and fixed; the paper never shows how performance scales with k itself, so "longest subsequence" difficulty is only sampled at one resolution on synthetic data (and at a narrow 6–10 on LongBench).
  • The real-world pipeline's filtering (Algorithm 7) is aggressive — it discards documents with natural lex runs > 10 or runs dominated by a single repeated character/word — which biases the LongBench subset toward documents where a length-5–10 run is genuinely rare, potentially inflating the apparent gap between synthetic and real-text decoy sensitivity.

Strengths

  • Conceptually simple, fully programmable task family with exact-match grading and no LLM-as-judge — eliminating the contamination and judge-bias risks that plague LLM-generated/LLM-judged benchmarks.
  • Genuinely multi-axis difficulty design that lets each axis be ablated in isolation (the clustered-vs-scattered ablation is a model experiment in miniature).
  • Two data regimes (uniform synthetic strings vs. natural-text LongBench documents) that together separate "is this hard because of long context?" from "is this hard because of word distribution?".
  • Broad, current model coverage with reasoning-effort and output-budget sweeps, plus a refreshingly honest failure analysis (token-budget exhaustion, degenerate repetition, first-word accuracy).
  • Honest, paper-specific caveats (the Opus refusal artifact; the open "multiple satisfying sequences" gap) rather than generic limitations boilerplate.

Limitations

  • No error bars, single seed, n=100 (n=93 for open-source on LongBench). Several reported numbers in the 0–10% band are within plausible sampling noise; the very-large effects survive, but the finer rankings do not.
  • Tasks are artificial and only weakly tied to downstream long-context utility; the link to real-world QA/summarization/codebase tasks is argued, not measured.
  • Quantifier complexity is under-probed: the ∀-query is evaluated only on data with a single satisfying sequence, which is the easiest case for a universal quantifier.
  • k is fixed at 5 for all synthetic variants; no sweep over target length, so the "longest" axis is sampled at one resolution.
  • Closed-model numbers are non-reproducible at the level of absolute percentages because the underlying APIs (GPT-5.4, Gemini 3.1 Pro preview, Opus 4.6) are moving targets and the Opus baseline already shows a refusal artifact that distorts cross-row comparisons.
  • LongBench document filtering (Alg. 7) biases the real-text subset toward documents without long natural sorted runs, which may exaggerate the synthetic-vs-real decoy-sensitivity gap.
  • No k-ary predicates (k>2) — explicitly called out as future work, so the "computation" axis is only sampled at arities 1 and 2.

Relevance

This paper is highly relevant to the "LLM Inference Evaluation" and "LLM Hidden States & Interpretability" interest areas: it is a rigorous, multi-axis long-context benchmark that exposes systematic, reproducible inference failure modes (scattered-decoy collapse, open-source near-zero on binary predicates) rather than measuring average-case saturation. It also touches "ML Reliability & Mathematical Foundations" by reframing long-context retrieval as a predicate-satisfaction / search problem with controllable difficulty — a useful formal lens for failure-mode analysis.

Questions for Further Exploration

  • How do the scattered-decoy failure rates change when the target run length k is swept (k=3,5,8,12,16) rather than fixed at 5 — is the collapse a function of k, of decoy count, or of their ratio?
  • Would the clustered-vs-scattered gap (Opus 1%→98%) survive a control where decoys are clustered at a random location far from the target, to disentangle "locality helps" from "any clustering helps"?
  • Can the "search-space size at fixed token count" result (92%→10% at 60K words) be reproduced with a held-out model family, and does it correlate with any measurable attention-sparsity or retrieval-probe statistic — i.e., is it a tokenization artifact or a genuine retrieval limit?
  • On the universal-quantifier axis, how do accuracies degrade when the number of satisfying sequences grows from 1 to many, which is the regime the current unique-solution dataset cannot probe?
  • Do the open-source models' qualitative failure modes (locating the region then degenerating into verbatim repetition) persist under chain-of-thought scaffolding that explicitly forbids quoting the input list, and does that recover any accuracy on scattered decoys?
  • Is there a correlation between PredicateLongBench variant scores and scores on established real-world long-context suites (HELMET, LongBench v2, RULER) across the same model set, which would calibrate how predictive this synthetic probe is for downstream competence?
