> **一句话总结：** TTHE 把 LLM agent 的可执行控制程序（harness）当作测试时适应的状态：在评测过程中，仅用无标签执行轨迹驱动一个由"提议者-裁判"组成的种群搜索来不断改写 harness 代码，不更新权重、不需 gold 标签，在 BIRD 文本到 SQL 困难切片上把基线从 12.0% 拉到 50.0%，并在 LiveCodeBench、SWE-bench Verified、DS-1000 和 claw-eval 上一致取得提升。

> **TL;DR:** TTHE reformulates test-time adaptation for LLM agents as evolution over their executable harness program: during evaluation it maintains a population of candidate harnesses, has agentic proposers rewrite their code from unlabeled execution traces, and a label-free judge commits one to persist — with the frozen deepseek-v4-flash backbone lifting a hard BIRD slice from 12.0% to 50.0% (+38 pts over the ReAct baseline), SWE-bench Verified from 20.0% to 35.0%, and claw-eval from 48.9% to 69.8%, without any weight updates or gold labels.

## 核心贡献

- **提出"测试时 harness 进化"新范式：** 将 LLM agent 的可执行控制程序（harness，一个 Python 类）作为测试时适应的状态，把 harness 优化从开发期前搜索搬进评测过程本身；适应对象是任意程序代码而非 prompt 模板或有限超参开关，搜索空间严格包含 prompt-only 与固定工作流脚手架。
- **完全无标签、无权重更新、无外部教师模型：** 求解器、提议者、裁判只是同一冻结 backbone（主实验为 deepseek-v4-flash，通过 Claude Code 作为编码接口路由）下的不同角色与 harness；gold 标签、参考输出、隐藏测试仅在提交后用于测量，绝不进入适应回路。
- **种群式生成-裁判搜索：** 每个无标签 batch 上演化 G 条分支、迭代 R 轮；定义 OBSERVE / PROPOSE / JUDGE 三个算子，靠 lineage（父代沿分支累积）与 diversity（保守修复/探索/对抗三种角色）将逐次反思升级为持久、可检查的可执行策略。
- **跨五个执行落地任务一致有效：** 文本到 SQL（BIRD）、竞赛编程（LiveCodeBench）、真实软件工程（SWE-bench Verified）、数据科学代码（DS-1000）以及智能体工具使用（claw-eval），均超过固定 ReAct 基线，并产出可读的 grounding/验证/修复策略源码。
- **详尽的失败模式诊断：** 给出搜索预算、batch 大小、跨模型、跨 batch 累积等消融，并用 gold-only oracle 把误差分解为 selection regret（候选已有但裁判没选，14 点）与 coverage gap（任何候选都未生成，~15 题），把 proxy 可靠性与裁判校准定位为核心瓶颈。

## Key Contributions

- Formulates **test-time harness evolution** as a new adaptation paradigm: the executable harness (a Python class wrapping the frozen solver) is the adaptation state, and harness optimization is moved inside evaluation itself rather than being pre-searched on a labeled dev set and then frozen. The search space is arbitrary executable agent programs, which strictly contains prompt-only and fixed-workflow scaffolds.
- Realizes the paradigm as a **population-based generate-and-judge loop** with three label-free operators — OBSERVE (run candidates, collect traces + proxy signals), PROPOSE (G parallel coding agents that edit their own parent branch while reading sibling traces), JUDGE (agentic selector that commits one final-branch harness using only execution-derived evidence). Solver, proposers, and judge are different roles/harnesses around the same frozen LLM.
- Achieves **gains without weights, labels, or a stronger teacher**: every model call (including the Claude Code scaffold) is routed to the same frozen backbone (deepseek-v4-flash in main runs), so all adaptation is a code edit to the program around the frozen solver.
- Validates **breadth across five execution-grounded domains** — BIRD, LiveCodeBench, SWE-bench Verified, DS-1000, and the agentic tool-use benchmark claw-eval — and exposes the resulting harness source, traces, and per-batch decisions for inspection.
- Provides **diagnostics that isolate the bottleneck**: search-budget (G×R), batch-size, cross-model, and cross-batch-accumulation ablations, plus post-hoc gold-only oracle analyses that decompose the residual error into selection regret and a coverage gap, identifying proxy reliability and judge calibration as the central challenges.

## Methodology

TTHE optimizes the agent's harness *during evaluation*, using only the evidence that execution itself produces. The harness `H` is an executable control program (a Python class) that wraps a frozen model `M` and orchestrates context construction, tool calls, validation, and recovery. The measurement objective `J*(H) = E[ I[o_H(x) ≡ y] ]` requires gold and is unavailable during adaptation; TTHE instead drives the loop with execution-derived evidence and consults gold only *after* a harness is committed, for reporting.

Two questions organize the method:
- **(Q1) What is the optimization signal without gold/reference outputs/hidden tests?** Detailed execution traces (prompts, completions, tool calls, stdout/stderr, artifacts, runtime states, errors, probes) plus three lightweight, task-dependent **proxy signals**: execution health `s ∈ {0,1}` (runs without error, well-formed result), round-trip consistency `b ∈ [0,1]` (output inverted to a description and matched against the input), and public-test pass rate `p ∈ [0,1]` when public examples or repo tests exist. These are exposed *together with raw traces* to the proposer and judge rather than collapsed into a single scalar, because cross-sample agreement is not correctness and any single proxy is gameable.
- **(Q2) How is the harness actually improved given only that imperfect signal?** A batch-level population search. For each unlabeled batch `X_t`, TTHE instantiates G branches from the committed `H_t` and evolves them over R rounds. Each round, each branch's proposer (a coding agent) edits its *own* parent harness into a child while free to read the other branches' code and traces for context; invalid children fall back to the parent. Branches are steered toward different edit directions (conservative-repair, exploratory, adversarial). After R rounds, a JUDGE inspects the G final branches (code, traces, signals, optional re-execution) and commits one as `H_{t+1}`, which persists to the next batch. Two properties make this population search rather than single-candidate reflection: **lineage** (a branch carries its parent forward, so useful routines compound) and **diversity** (G branches explore several directions and can borrow from one another).

Three label-free operators formalize the loop:
- `OBSERVE(H, X_t)` runs every harness in H on every input, returns traces, cached outputs, and proxy signals.
- `PROPOSE({H_g}, T)` runs G proposers in parallel; proposer g edits its own parent branch `H_g` into one child.
- `JUDGE(F, T, X_t)` is an agentic selector over the final branch set F; it never sees gold.

The solver, proposers, and judge all use the same frozen backbone (deepseek-v4-flash in main runs); proposers and judge run inside a Claude Code agent scaffold used purely as a coding interface, with every model call routed to the same frozen backbone. The baseline `H_0` is a standard ReAct scaffold. Scoring is **transductive**: each batch is scored with the harness committed for it.

## Results & Highlights

- **Text-to-SQL (hard BIRD slice, 50 questions):** TTHE lifts the deepseek-v4-flash ReAct baseline from **12.0% to 50.0%** (+38.0 points). The slice is built from questions the baseline systematically fails, so the large gain measures recovery on those queries rather than an intrinsically easier task.
- **Competitive programming (LiveCodeBench hard slice, 60 problems):** **30.0% → 38.3%** (+8.3 points) over the ReAct baseline.
- **Software engineering (SWE-bench Verified hard slice, 40 instances):** **20.0% → 35.0%** (+15.0 points); the baseline here is the off-the-shelf mini-swe-agent whose scaffold TTHE then evolves.
- **Data-science code (DS-1000 hard slice, 50 problems):** **38.0% → 44.0%** (+6.0 points).
- **Agentic tool use (claw-eval, 30-task slice):** mean graded task score rises from **48.9% to 69.8%** (+20.9 points), improving on 28 of 30 tasks — evidence the same loop generalizes beyond code/query generation to multi-service office workflows scored by a rubric rather than pass@1.
- **Cross-model (hard BIRD slice, same B10/G3/R3):** MiMo V2.5 improves **32.0% → 52.0%** (+20.0), Kimi K2.5 improves **28.0% → 48.0%** (+20.0), showing the loop is not tied to one backbone; all three improve over their own model-matched baselines.
- **Search-budget ablation (hard BIRD slice):** best setting is **G=3, R=3 = 50.0%**; effect of more rounds is non-monotonic and interacts with G — at G=1 accuracy *falls* from 46.0% (R=1) to 40.0% (R=3), whereas at G=3 it rises from 44.0% (R=1) to 50.0% (R=3). Parallel lineages prevent a good early candidate from being lost with no choice left for the judge.
- **Batch-size ablation:** accuracy is non-monotonic and peaks at **B=10 (50.0%)**; B=5 gives 44.0%, B=25 gives 38.0%, B=50 gives 44.0% — though all remain far above the 12.0% baseline.
- **Cross-batch accumulation vs reset:** carrying the committed harness across batches reaches **50.0%** versus **44.0%** for resetting to baseline each batch, with the +6-point edge concentrated in the middle batches.
- **Selection regret vs coverage gap (gold-only oracle diagnostics):** an oracle restricted to the pool the judge actually saw reaches **64.0%** versus the judge's 50.0% (selection regret of 14 points); an oracle over all candidates ever generated still reaches only **70.0%**, so ~15 tasks are never produced by any candidate (coverage gap). The judge is measurably miscalibrated — committing a 3/10 branch over an available 4/10 in one batch and a 6/10 over a 7/10 in another while claiming near-perfect correctness for far-worse candidates.

## Critical Assessment

- **Novelty:** The reframing is genuinely fresh relative to pre-evaluation harness/workflow search (DSPy, AFlow, ADAS, Promptbreeder) and to per-response reflection (Reflexion, Self-Refine): TTHE keeps the harness as the object of optimization but moves it into evaluation, hides gold from both proposer and judge, and commits arbitrary code edits under incomplete evidence. The closest concurrent system, Meta-Harness (Lee et al., 2026), uses a similar agentic-proposer-reads-filesystem loop; TTHE's distinct contribution is the label-free, transductive evaluation-time setting and the explicit study of proxy reliability.
- **Rigor:** Mixed. Diagnostics are unusually candid — the selection-regret / coverage-gap decomposition (pool oracle 64.0% vs judge 50.0%; all-candidate oracle 70.0%) and the documented judge misranking (committing 3/10 over 4/10) are exactly the kind of honest failure-mode analysis often omitted. However, every ablation entry is "a single completed run" — **no multi-seed variance, no confidence intervals**, so the non-monotonic search-budget and batch-size curves could be partly noise. There are also **no compute-matched baselines** (best-of-N execution selection, dev-time optimizers like AFlow/DSPy run with the same wall-clock budget), so it is unclear how much of the gain comes from harness evolution vs. simply spending more LLM calls per batch.
- **Reproducibility:** Good on artifacts (code released, harness source and per-batch trajectories exposed, hard-slice construction described and fixed before running). Weaker on the selection criterion: the hard slices are conditioned on baseline failures (BIRD's 12% start is by construction), so absolute numbers and cross-model comparisons are explicitly not apples-to-apples — the within-model delta is the only clean claim.
- **Significance:** If the prequential generalization claim holds up, this is a meaningful new axis for label-free agent improvement; the transductive-only evidence is the main thing holding significance back.
- **Red flags / specific gaps:**
  - The setting is **transductive** (a batch is both adapted to and scored on itself); the authors explicitly state forward generalization is *not* established and that prequential scoring and shuffled-batch reruns are left to future work. Batch-local overfitting is not ruled out.
  - The claw-eval slice is difficulty-ordered, making the accumulation-vs-reset comparison (Table 3) potentially confounded; the authors flag this but do not fix it.
  - **Missing information ablation** — exposing proposer/judge to only scores, scores+summaries, or full traces — is proposed but not run, so the central claim that "raw-trace access enables the discovered policies" is not causally tested.
  - Shared frozen backbone for proposer + judge means additional search budget "tends to generate correlated variants that repeat the same errors," which is consistent with the non-monotonic curves and caps the coverage gap.
  - Runs use a restricted sandboxed executor; open-world/safety-critical extension is acknowledged as needing guardrails against the programs the proposer may write.

## Strengths

- A clean, well-motivated new problem formulation (evaluation-time, label-free, code-as-state) with a principled population-search instantiation and formally defined OBSERVE/PROPOSE/JUDGE operators.
- Honest, unusually detailed failure-mode diagnostics: the selection-regret vs coverage-gap split and the concrete judge misrankings are a model for how to report unsupervised agent improvement.
- Breadth across five diverse execution-grounded domains including a non-code agentic tool-use benchmark, with consistent positive deltas over ReAct baselines and across three backbones.
- Inspectability: evolved harnesses are readable source code, and the recovered policies (e.g., reproduce-first-locate-fix-reverify workflow on SWE-bench; execute-and-self-debug with database-grounded casing on SQL) are concrete and transferable intuitions.
- No weight updates, no gold labels, no stronger teacher — the adaptation boundary is tightly and credibly drawn.

## Limitations

- **Transductive setting only:** within-batch adaptation is measured; forward (prequential) generalization and the accumulation-vs-batch-local-specialization split are not established. This is the most important caveat for interpreting the headline gains.
- **Single-seed, no variance, no compute-matched baselines:** the non-monotonic ablation curves could partly reflect run-to-run noise; best-of-N and dev-time optimizers under equal compute are not compared.
- **Judge calibration is the bottleneck:** 14 points of selection regret are already in the candidate pool but not committed, and the agentic judge is shown to be measurably miscalibrated on subtle output-shape mismatches.
- **Coverage gap is intrinsic to the current search:** ~15/50 BIRD tasks are never produced by any candidate; shared-backbone correlated variants prevent more search budget from closing it.
- **Information ablation absent:** the claim that raw-trace access (vs. scalar scores) is what enables the discovered policies is not causally tested.
- **Restricted sandboxed execution:** open-world or safety-critical deployment would need additional guardrails on proposer-written code.

## Relevance

本文与"测试时适应（TTT/TTA）+ LLM agent 自演化"的研究兴趣高度相关：它把无监督测试时适应从权重/归一化统计扩展到 agent 的可执行控制程序，并系统刻画了 proxy 可靠性这一核心瓶颈，对关注 agent 自主改进、test-time scaling 与执行反馈学习的工作有直接参考价值。

## Questions for Further Exploration

- Does TTHE's gain survive **prequential scoring** — i.e., does the harness committed on batch t actually generalize to batch t+1 before it adapts, or is a large share of the gain batch-local overfitting to execution artifacts?
- How much of the improvement is **harness evolution vs. simply more compute**? A compute-matched comparison against best-of-N execution selection and dev-time optimizers (DSPy, AFlow) under the same per-batch LLM-call budget is the key missing control.
- Would an **information ablation** (scores-only vs. scores+summaries vs. full traces) confirm that raw-trace access is causally responsible for the discovered grounding/verification policies?
- Can the **selection regret** (14 points recoverable from the existing pool) be closed with a better-calibrated judge — e.g., a verifier that penalizes output-shape mismatches against the reference schema — without re-introducing gold into the loop?
- Is the **non-monotonic** search-budget / batch-size behavior stable across seeds, or does single-run noise account for the irregular curves?
- Does carrying the committed harness **across tasks** (not just across batches of one task) transfer useful structure, or does it overfit to one workload's execution shape?
- How does TTHE behave when proposer and judge use a **stronger backbone than the solver** (a teacher) — does that break the "no stronger teacher" claim, or is it a worthwhile tradeoff?
