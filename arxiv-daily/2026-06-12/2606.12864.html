Beyond Problem Solving: UOJ-Bench for Evaluating Code Generation, Hacking, and Repair in Competitive Programming

Tingqiang Xu, Hangrui Zhou, Tianle Cai, Alex Gu, Kaifeng Lyu  ·  Tsinghua · ByteDance Seed · MIT  ·  2606.12864
即使最前沿的模型也无法在单次推理中识别超过 50% 的已知错误提交 — Even frontier models miss bugs in >50% of incorrect submissions.
Key Finding
Best model (Gemini-3-pro) solves only 38.8% code-gen and 44.2% covert-error hacking at Pass@1.

Tasks & Dataset

Code Generation 672 problems from UOJ contests + IOI/NOI/JOISC
Code Hacking 479 Easy (overt errors) + 1,046 Hard (covert/community-hacked)
Code Repair 500 Easy + 216 Hard pairs (≥0.95 edit similarity)
Zero-Day Hacking 5,060 full-score submissions; LLM finds unknown bugs

Test-time scaling with GPT-OSS-120B achieves >90% hacking success at k=20,000 — but costs ~$1/problem. GPT-OSS-120B is the most cost-effective under scaling, beating Gemini at equal cost.

Key Results

  • Performance drops sharply from Easy → Hard (overt → covert errors). Gemini: 61% → 44% hacking.
  • Ultrahard code-gen: best model 18.6%; Qwen3-Coder scores 0%.
  • Agentic (ReAct) mirrors direct sampling for hacking but significantly outperforms for repair — iterative refinement works.
  • GPT-OSS-120B finds covert errors in >10% of zero-day submissions across ~30 problems.
  • Contamination: code-gen degrades on newer problems (memorization), but hacking/repair stay stable (genuine reasoning).

Strengths & Limitations

Strengths

  • Multi-task evaluation (gen + hack + repair) is genuinely novel
  • UOJ Hack mechanism provides real-world covert errors unavailable elsewhere
  • Native judging infrastructure eliminates surrogate-judge gap
  • Zero-Day Hacking is an innovative evaluation paradigm
  • Honest cost analysis — no hype about deployment readiness

Limitations

  • Heavily C++-biased (1,018/1,046 Hard hacking submissions)
  • Zero-Day sampling biased toward historically-hacked problems
  • Test-time scaling on only 100 hacking / 50 repair samples
  • Benchmark tied to UOJ infrastructure; API access needed for replication
  • Problem statements LLM-translated from Chinese; artifacts not analyzed

Relevance

UOJ-Bench 从代码生成、找错和修复三个维度全面评估 LLM 在竞技编程场景中的能力，与代码生成和 LLM 评估的研究兴趣高度相关。其 Zero-Day Hacking 发现——即 LLM 能在满分代码中发现未知隐性错误——对理解 LLM 的深层代码推理能力具有重要意义。
