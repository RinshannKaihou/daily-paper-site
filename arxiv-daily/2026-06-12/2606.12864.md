> **一句话总结：** 本文提出 UOJ-Bench，一个基于中国竞技编程平台 UOJ（Universal Online Judge）真实提交数据构建的基准，从代码生成、代码黑客攻击（找反例使人类代码出错）和代码修复三个维度评估 LLM 能力，涵盖显性错误和隐性错误两个难度级别，并引入 Zero-Day Hacking 评估 LLM 发现未知漏洞的能力；结果显示即使最强模型在单次推理下也无法识别超过 50% 的已知错误提交，测试时扩展虽可将成功率提升至 90% 以上但成本极高，不过 GPT-OSS-120B 在零日攻击中成功发现了超过 10% 提交中的隐性错误。

> **TL;DR:** UOJ-Bench is a comprehensive benchmark built from real submissions on the Universal Online Judge (UOJ) that evaluates LLMs on three competitive programming tasks: code generation, code hacking (generating counter-example test cases to break human-written code), and code repair (patching buggy code). It distinguishes overt errors (catchable by standard test cases) from covert errors (only discoverable via community hacking) and introduces a Zero-Day Hacking setting where LLMs attempt to find previously unknown bugs in full-score submissions. Even frontier models (Gemini-3-pro-preview, GPT-5, etc.) miss bugs in >50% of incorrect submissions under single-shot evaluation. Test-time scaling with GPT-OSS-120B achieves >90% success on known bugs but costs ~$1/problem at k=20,000; GPT-OSS-120B also discovers covert errors in >10% of "zero-day" submissions. The benchmark is publicly available at https://github.com/hehezhou/UOJ-Bench.

## 核心贡献

- **提出 UOJ-Bench 基准**：基于真实的竞技编程平台 UOJ 的提交数据构建，涵盖代码生成（672 题）、代码黑客攻击（479 Easy + 1,046 Hard 提交）和代码修复（500 Easy + 216 Hard 提交）三个任务。
- **区分显性错误与隐性错误**：Easy 级别评估能被标准测试用例捕获的显性错误，Hard 级别评估需要社区 Hack 机制才能发现的隐性错误。
- **引入 Zero-Day Hacking 设置**：在 5,060 个满分提交中评估 LLM 发现未知漏洞的能力，GPT-OSS-120B 成功发现超过 10% 提交中的隐性错误。
- **利用 UOJ 原生评测基础设施**：通过 UOJ 内部 API 进行评测，支持自定义 Checker 和严格时限检查，消除了代理评测管道的差距。
- **全面的成本-性能分析**：揭示了测试时扩展虽然能大幅提升成功率，但每年处理 UOJ 约 10 万次提交的成本约为 10 万美元，远超传统 CPU 评测的 500 美元/年。

## Key Contributions

- Introduce UOJ-Bench, a benchmark constructed from real user submissions on UOJ covering three tasks: code generation (672 problems), code hacking (479 Easy + 1,046 Hard submissions), and code repair (500 Easy + 216 Hard submissions), with problem statements translated from Chinese to English.
- Distinguish overt errors (detectable by standard test cases) from covert errors (only discoverable through community hacking), providing two difficulty levels for hacking and repair tasks.
- Introduce the Zero-Day Hacking evaluation setting, where LLMs attempt to discover previously unknown bugs in full-score submissions — analogous to zero-day vulnerability discovery in security.
- Leverage UOJ's native judging infrastructure via its internal API, supporting custom checkers and strict time-limit checking, eliminating the gap between surrogate and real-world evaluation environments.
- Provide comprehensive cost-performance analysis across 7 frontier models (Gemini-3-pro-preview, GPT-5, GPT-5-High, Kimi-k2-thinking, Deepseek-v3.2-thinking, GPT-OSS-120B, Claude-Opus-4.5, Qwen3-Coder), revealing the economic bottleneck of test-time scaling.

## Methodology

**Platform and Data Source.** UOJ (Universal Online Judge) is a community-driven online judge platform widely used in China, hosting problems from IOI, JOISC, NOI, NOIP, and original contests. Its unique Hack mechanism allows users to submit counter-example test cases targeting full-score submissions — successful hacks are automatically added to the Extra Tests and trigger re-evaluation of all submissions. All user submissions and hacks are open-access.

**Dataset Construction (cutoff: September 15, 2025):**
- **Code Generation (Task 1):** Union of all problems from hacking/repair tasks plus all UOJ original contest problems (672 problems total).
- **Code Hacking (Task 2):** Easy — 479 buggy submissions that failed standard tests; Hard — 1,046 submissions from successful community hacks (max 10 per problem, covering WA/TLE/RE/MLE failure modes).
- **Code Repair (Task 3):** Easy — 500 (buggy, fixed) submission pairs from partial-score submissions later corrected by the same author; Hard — 216 pairs from hacked submissions with subsequent fixes, enforcing ≥0.95 edit similarity to focus on precise debugging.
- **Zero-Day-Hacking-5K:** 5,060 full-score submissions sampled proportionally to historical hack frequency, re-verified against current UOJ test suite.

**Evaluation Protocol:**
- **Direct Evaluation:** Single-turn, zero-shot with ≥64k token output limits. Pass@1 metric. For repair, patches must modify ≤10% of original code.
- **Test-Time Scaling:** Pass@k evaluation on subsets (100 hacking, 50 repair samples) with k up to 20,000.
- **Agentic Evaluation:** ReAct framework with multi-turn feedback (compilation errors, verdicts, failed test case details) for up to 10 turns.
- **Contamination Analysis:** Temporal analysis showing code generation performance declines on newer problems (memorization effect) while hacking/repair remain stable (genuine reasoning).

**Error Taxonomy:** Corner Case Error (boundary conditions, overflow, precision), Careless Error (implementation oversights like wrong types or initialization), Wrong-but-Works Algorithm (fundamentally flawed logic that coincidentally passes most tests).

## Results & Highlights

- **Code Generation:** Gemini-3-pro-preview leads with 38.84% overall Pass@1, but drops to 18.62% on Ultrahard problems. Qwen3-Coder scores 0% on Ultrahard.
- **Hacking (Hard):** Even the best model (Gemini-3-pro-preview) achieves only 44.17% Pass@1 on covert errors; Qwen3-Coder scores 5.64%.
- **Repair (Hard):** Gemini-3-pro-preview achieves 48.15% Pass@1; most models score below 30%.
- **Easy vs Hard gap:** Models consistently perform better on overt errors than covert errors (e.g., Gemini: 61.38% → 44.17% for hacking).
- **Test-time scaling:** Gemini-3-pro-preview detects 93/100 hacking samples and repairs 41/50 repair samples under scaling. Cost ~$1/problem at k=20,000.
- **Cost-effectiveness:** GPT-OSS-120B is the most cost-effective model under test-time scaling, beating Gemini by a significant margin at equal cost.
- **Zero-Day Hacking:** GPT-OSS-120B uncovers errors in >10% of all submissions and >5% of full-score submissions across ~30 problems, demonstrating real-world complementary value beyond standard judging.
- **Agentic vs Direct:** Agentic (ReAct) approach closely mirrors direct sampling for hacking (feedback offers little advantage over enumeration), but significantly outperforms for repair (iterative refinement is highly effective).
- **Cost analysis:** LLM-based judging would cost ~$100,000/year for UOJ's ~100K submissions vs. ~$500/year for traditional CPU judging — a 200x cost disparity.

## Critical Assessment

- **Novelty:** High. The multi-task evaluation (generation + hacking + repair) with an explicit overt/covert error distinction is genuinely novel. The Zero-Day Hacking setting is a creative contribution that goes beyond any existing benchmark. The exploitation of UOJ's unique Hack mechanism for benchmark construction is elegant.
- **Rigor:** Strong. The data curation pipeline is thorough — filtering out randomization-dependent hacks, self-hacks, hard-coded fixes, with manual audit of ambiguous categories (100% LLM accuracy on the hardest categories). The use of native judging infrastructure avoids the surrogate-judge gap that plagues other benchmarks. The contamination analysis is a nice touch.
- **Reproducibility:** High. Benchmark is publicly available and UOJ's internal API access is offered to researchers. Problem statements are translated to English.
- **Significance:** Very high practical relevance. The finding that LLMs can find bugs in full-score submissions is actionable for OJ platforms and competitive programming education. The cost analysis ($100K vs $500/year) is sobering and honest about deployment feasibility.
- **Red flags / concerns:**
  - The benchmark is heavily C++-biased (1,018 of 1,046 Hard hacking submissions are C++), reflecting CP culture but limiting language diversity.
  - The 5% zero-day success rate across ~30 problems is a relatively small absolute number — it's impressive but the variance across problems is enormous (Figure 5), suggesting results may not generalize uniformly.
  - The test-time scaling evaluation uses only 100 (hacking) and 50 (repair) samples — somewhat limited for drawing strong conclusions.
  - The paper does not deeply analyze what types of covert errors are most amenable to LLM discovery, which would be valuable for guiding practical deployment.
  - The edit-distance constraint (≤10% modification, ≥0.95 similarity) for repair is sensible but may exclude meaningful fix types that require more substantial restructuring.

## Strengths

- Novel and well-motivated benchmark design that goes far beyond code generation to assess real-world code reasoning.
- Exploitation of UOJ's unique Hack mechanism provides a natural source of hard, real-world covert errors unavailable elsewhere.
- Native judging infrastructure evaluation eliminates a key weakness of existing benchmarks.
- Comprehensive evaluation of 7+ frontier models with detailed cost-performance tradeoff analysis.
- Zero-Day Hacking is a genuinely innovative evaluation paradigm.
- Honest about limitations — the cost analysis makes clear that large-scale deployment is currently economically infeasible.
- Thorough contamination analysis showing hacking/repair are more contamination-resistant than generation.

## Limitations

- Heavily C++-biased dataset (minimal Python/Pascal), limiting cross-language generalization.
- Zero-Day Hacking results are based on biased sampling (proportional to historical hack frequency) rather than uniform sampling, making it hard to estimate the true base rate of discoverable covert errors.
- Test-time scaling evaluation is on relatively small subsets (100/50 samples).
- The economic analysis, while honest, paints a somewhat pessimistic picture for near-term practical deployment.
- Problem statements translated from Chinese to English via LLM — potential translation artifacts not analyzed.
- The benchmark is tied to UOJ's infrastructure; researchers without API access cannot fully replicate the evaluation.

## Relevance

本文提出的 UOJ-Bench 从代码生成、找错（黑客攻击）和修复三个维度全面评估 LLM 在竞技编程场景中的能力，与代码生成、大语言模型评估和基准构建的研究兴趣高度相关。其 Zero-Day Hacking 发现——即 LLM 能在人类提交的满分代码中发现未知的隐性错误——对理解 LLM 的深层代码推理能力和在教育系统中的实际应用潜力具有重要意义。

## Questions for Further Exploration

- What specific types of covert errors (corner case, careless, wrong-but-works) are LLMs most effective at discovering, and which remain hardest?
- How would the results change with a more language-diverse benchmark including Python, Java, and Rust submissions?
- Can the high cost of test-time scaling be reduced through targeted techniques like guided search, curriculum-based sampling, or fine-tuning on hack patterns?
- How do the Zero-Day Hacking success rates correlate with problem difficulty, algorithm type, or code complexity metrics?
- Could a hybrid approach (CPU-based judging + selective LLM hacking on high-suspicion submissions) achieve a better cost-effectiveness tradeoff?
- How does model performance on UOJ-Bench hacking/repair correlate with performance on broader software engineering tasks like SWE-bench?
- Can the discovered zero-day hacks be used to improve the test suites of other OJ platforms (Codeforces, AtCoder)?
