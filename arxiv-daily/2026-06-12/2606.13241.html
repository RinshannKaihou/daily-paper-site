Brick: Spatial Capability Routing for the Mixture-of-Models (MoM) Paradigm

Francesco Massa · Marco Cristofanilli 2026-06-12 arxiv.org/abs/2606.13241 ↗

Brick 提出基于六维能力空间几何距离的多模型路由框架，通过 ModernBERT 分类器估计查询能力需求、独立复杂度分类器估计查询难度，并结合代价惩罚的几何调度规则，以最高质量配置达到 76.98% 准确率（超越最强单模型 +1.96 pp），中性配置下以 4.71× 成本节约实现 74.11% 准确率。

Key Finding

Spatial capability routing beats the best single model by +1.96 pp accuracy while being 28% cheaper per call — models have complementary error sets that a geometric router exploits.

Key Metric
76.98%

Why This Matters

Production LLM pools face 10-100× cost gaps. Brick turns quality-vs-cost into a deploy-time HTTP header — no code change needed.

Critical Edge

Only 3 models tested. In-domain calibration. Captures just 24% of oracle headroom. Router adds 494 ms latency (vs 15 ms for cascades).

Routing Pipeline — 6 Steps

Text normalize & truncate to 512 runes
Keyword prior injection (code fences, etc.)
ModernBERT capability classification → 6-dim probability vector
Qwen3.5-0.8B LoRA complexity estimator → blended difficulty τ
Asymmetric geometric scoring Jm = Dm + β·cm
argmin model selection with deterministic tie-break

Methodology

  • Three-model pool: qwen3.5-9b ($0.0014/call), deepseek-v4-flash ($0.0029), kimi2.6 ($0.031)
  • Shared 6-dim capability space: coding, creative synthesis, instruction following, math reasoning, planning/agentic, world knowledge
  • Skill matrix S ∈ (0,1)K×D calibrated via stratified 5-fold out-of-fold on Dataset A
  • Continuous preference knob r ∈ [-1, 1] modulates 4 scalars via asymmetric power laws — deploy-time quality/cost trade-off
  • Bayesian hyperparameter search (~25k configs) on 70/30 dev/test split

Strengths

  • Addresses a genuinely important 10-100× cost problem with clear economic motivation
  • Elegant geometric formulation — queries and models share the same space, dispatch is argmin
  • Unusually transparent: worked example with all numerical intermediates, full sensitivity analysis
  • Continuous preference knob bridges research prototype → production deployment
  • Timely cascade-vs-pure-routing argument for agentic N-step trajectories (43-47% cheaper)
  • All code, data, and models open-source (regolo-ai/brick-SR1)

Limitations

  • Three-model pool only — unclear how skill matrix scales to K=10 or K=20
  • In-domain calibration — strict OOD generalization untested
  • Router decision latency 494ms (production) vs 15ms for cascade routing
  • Static skill matrix — no online adaptation to model updates or distribution shifts
  • Six fixed dimensions — not justified beyond Dataset A composition
  • Captures only 24% of oracle headroom; τbase and β are fragile (±5.2 pp)

Relevance · 相关性

该论文提出的 MoM 路由框架与高效 LLM 部署直接相关，其能力空间几何路由思想和连续偏好旋钮设计对多模型服务架构具有重要参考价值。对于关注推理成本优化、多模型协作以及 agentic 系统设计的 AI 工程实践尤为相关。

Open Questions

  • How does routing degrade as the pool expands to K=10 or K=20 — does the skill matrix become too sparse?
  • Can the skill matrix be updated online without full recalibration?
  • Cross-domain transferability of the ModernBERT capability classifier on medical, legal, or multilingual queries?
  • Would a domain-conditioned complexity estimator close more of the 76% oracle gap?
  • How does Brick compare against simple ensemble (majority vote) in accuracy-per-dollar?
  • Can the thermodynamic analogy be formalized into a principled knob-free tie-breaker?
