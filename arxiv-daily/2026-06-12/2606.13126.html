MiniPIC: Flexible Position-Independent Caching in <100 LOC

Nathan Ordonez Thomas Parnell IBM Research Zürich

A minimal vLLM extension (78 LOC core) that enables position-independent KV caching for RAG and agentic workloads by storing unrotated keys and applying RoPE inside the attention kernel. Three user-facing special-token primitives let a single running instance express Block-Attention, EPIC, and Prompt Cache.

MiniPIC 通过在 KV cache 中存储未旋转的 key 向量并在 attention kernel 内动态施加 RoPE，配合三个用户控制的原语，仅用 78 行核心代码就在 vLLM 中实现了灵活的位置无关缓存，在 2WikiMultihopQA 上将 prefill 吞吐量提升 49%。

Key Finding
Position-free KV cache + 3 user primitives replaces entire engine-managed PIC subsystems
Key Metric
49% prefill speedup

Why This Matters
Three tokens replace engine complexity — user-controlled cache reuse is sufficient for RAG & agents.
Critical Edge
Silent false cache hits from bad marker placement; no accuracy eval; MEPIC not benchmarked.
Core Mechanism
Standard Apply RoPE → store rotated K → cache is position-locked
MiniPIC Store unrotated K̃ → apply RoPE on-the-fly in attention kernel using per-request logical positions
Benefit Moves positional encoding from shared GPU memory to temporary SRAM/registers — one physical block serves all positions
Overhead Only 5.7% worst-case when PIC disabled (30.37 vs 32.21 samples/s)

Three User-Controlled Primitives
PAD Block-aligned padding — ensures spans align to KV block boundaries so subsequent markers reliably appear at block edges.
SS_EP Span Separator — resets hash chain to global seed, making a span independently cacheable regardless of prefix. Enables Block-Attention and Prompt Cache.
PD_EP Prompt Depend — forces hash to depend on full preceding prompt, preventing false cache hits for suffix blocks (e.g., task instructions after documents).
These three tokens are sufficient to express Block-Attention, EPIC/LegoLink, and Prompt Cache within the same vLLM instance — no new scheduler request types, memory subsystems, or external caching services.

Strengths

  • 78 LOC core changes (61 new) — orders of magnitude simpler than SPNL or MEPIC
  • 49% throughput gain over baseline, 33% over SPNL
  • Only 5.7% overhead when PIC disabled
  • Clean separation: engine handles position-free caching; users handle policy
  • Native CPU offload compatibility — no special position tracking
  • Three primitives express multiple PIC methods in one instance

Limitations

  • RoPE-only — no path for ALiBi or NoPE architectures
  • No accuracy evaluation — span-based PIC approximates causal attention
  • Single model (8B), single GPU, single dataset — no generalization evidence
  • No eviction policy — cached spans may be evicted under high concurrency
  • Incorrect marker placement → silent false cache hits, no engine-side validation
  • MEPIC (most comparable system) not evaluated
Open Questions
  1. How does position-free caching interact with tensor parallelism across multi-GPU setups (70B+ models)?
  2. What is the quantitative accuracy impact of span-based PIC on downstream tasks?
  3. Can the three-primitive interface extend to attention-sink patterns (StreamingLLM) or sparse attention?
  4. How does MiniPIC perform under realistic online serving with mixed prefill/decode workloads?
  5. Could a compiler/verifier automatically check marker placement correctness?
