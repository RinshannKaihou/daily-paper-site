> **一句话总结：** Brick 提出了一种基于六维能力空间几何距离的多模型路由框架（MoM），通过 ModernBERT 分类器估计查询能力需求、独立的复杂度分类器估计查询难度，并结合代价惩罚的几何调度规则，在 5,504 条查询基准上以最高质量配置达到 76.98% 准确率（超越最强单模型 kimi2.6 的 75.02%），中性配置下以 4.71 倍成本节约实现 74.11% 准确率，中位延迟从 51.2 秒降至 22.8 秒。

> **TL;DR:** Brick is a calibrated Mixture-of-Models (MoM) router that represents both queries and models as vectors in a six-dimensional capability space (coding, creative synthesis, instruction following, math reasoning, planning/agentic, world knowledge), estimates per-query difficulty via a separate classifier, and dispatches via a cost-penalized geometric rule with a tunable preference knob. On a 5,504-query benchmark spanning three models (qwen3.5-9b, deepseek-v4-flash, kimi2.6), Brick at max-quality reaches 76.98% accuracy — beating the best single model (+1.96 pp) and all external routers (RouteLLM, FrugalGPT, Cascade Routing) — while being 28% cheaper per call. At a neutral cost-quality profile it achieves 74.11% at 4.71× lower cost with only 0.91 pp accuracy loss. The key insight is that capability-aware spatial routing captures within-domain difficulty variance that surface-feature routers miss, and that models have complementary error sets (e.g., kimi2.6 refuses 51.7% on world_knowledge where cheaper models succeed).

## 核心贡献

- **提出 MoM（Mixture-of-Models）范式**：将整个预训练模型作为路由单元（而非 MoE 中的层/专家），在推理时通过外部路由器为每个查询选择一个最合适的模型。
- **六维能力空间路由**：用 ModernBERT 分类器将查询映射到六个能力维度上的概率分布，每个模型有离线校准的技能矩阵，通过几何距离（非对称欠/超容量残差）选择最优模型。
- **代价惩罚的几何调度规则**：路由目标函数 `Jm = Dm + β·cm`，其中 `Dm` 是 L2 能力距离（区分欠容量和超容量惩罚），`β·cm` 是成本项，argmin 选择综合最优模型。
- **连续偏好旋钮 `r ∈ [-1, 1]`**：通过非对称幂律映射调制四个路由标量（µ, b, β, λ），运维人员可在部署时平滑切换最大质量与最大节约策略，无需代码修改。
- **显著实证结果**：在 5,504 条查询上超越所有基线路由器，中性配置下成本降低 4.71 倍仅损失 0.91 pp 准确率；揭示模型间存在互补错误集（如 kimi2.6 在世界知识上 51.7% 拒答率）。
- **论证纯路由优于级联路由**：在 agentic 部署场景下，级联路由在每一步都累积拒绝模型的成本，N=10 步时账单比纯路由高 43-47%。

## Key Contributions

- **Mixture-of-Models (MoM) paradigm**: Frames whole-model routing across a heterogeneous LLM pool as a first-class deployment pattern, distinct from Mixture-of-Experts (which operates at the layer/token level).
- **Six-dimensional capability-space routing**: Uses a fine-tuned ModernBERT classifier to map queries to a probability distribution over six capabilities (coding, creative_synthesis, instruction_following, math_reasoning, planning_agentic, world_knowledge). Models are represented by an offline-calibrated skill matrix `S ∈ (0,1)^{K×D}`.
- **Cost-penalized geometric dispatch rule**: The routing objective `Jm = Dm + β·cm` decomposes the quality gap into asymmetric under-capacity (model too weak) and over-capacity (overkill) residuals, combined via weighted L2 distance with an additive cost penalty. The argmin selector picks the cheapest model that geometrically "covers" the query.
- **Continuous preference knob `r ∈ [-1, 1]`**: Smoothly modulates four routing scalars (µ, b, β, λ) via calibrated asymmetric power laws, turning the quality-vs-cost trade-off into a deploy-time HTTP header parameter rather than a code change.
- **State-of-the-art routing results**: On 5,504 queries, Brick max-quality (76.98%) beats always-kimi (75.02%), RouteLLM (75.02%), FrugalGPT (69.42%), and Cascade Routing (73.40%). Neutral profile achieves 74.11% at 4.71× lower cost. Median latency drops from 51.2s to 22.8s.
- **Structural argument for pure routing over cascades**: In agentic deployments (N-step trajectories), cascade routers pay rejected-stage costs N times, resulting in 43-47% higher bills at N=10 compared to pure single-step routing.
- **Complementary error set analysis**: Identifies that kimi2.6 has a 51.7% refusal rate on world_knowledge queries where deepseek-v4-flash achieves 49% accuracy (vs kimi's 34%), which the capability-aware router exploits automatically.

## Methodology

**Model Pool**: Three models spanning cost/quality regimes — qwen3.5-9b (9B, $0.04/$0.15 per 1M input/output tokens), deepseek-v4-flash (under-500B, $0.20/$0.40), and kimi2.6 (frontier, $0.73/$3.49). The per-call costs are a$qwen=$0.001386, a$ds4=$0.002895, a$kimi=$0.030703.

**Pipeline (6 steps)**:
1. **Text normalization**: Trim whitespace, truncate to 512 runes for classification (full query forwarded to backend).
2. **Keyword prior injection**: High-precision keyword rules (e.g., code fences) bias the capability distribution.
3. **Capability classification**: Fine-tuned ModernBERT-base maps the query to a probability vector `p(x) ∈ Δ^{D-1}` over six capabilities. Trained on a 50k-query labeled corpus (Brick2 Dataset B), selected by macro-Pearson correlation (0.983).
4. **Complexity estimation**: A separate Qwen3.5-0.8B LoRA classifier (3-class: easy/medium/hard) produces a blended difficulty `τq` via confidence-weighted interpolation between the predicted label anchor and the medium anchor. Intentionally domain-agnostic.
5. **Routing math**: Maps difficulty to logit space `zq = b + µ·logit(τq)`, computes per-capability requirements `rq,c = pc·zq` and model capacities `vm,c = pc·logit(sm,c)`, decomposes into asymmetric residuals (under/over-capacity), and scores `Jm = Dm + β·cm` where `Dm = Σ(um,c² + λ·om,c²)`.
6. **Model selection**: `m⋆ = argmin Jm` with deterministic tie-breaking via expected success.

**Skill Matrix Calibration**: Each `sm,c` is a probability-weighted empirical success rate on Dataset A's results subset, clipped to [0.02, 0.98]. Evaluated via stratified 5-fold out-of-fold calibration to avoid per-example leakage.

**Preference Knob**: The four routing scalars are modulated by `r ∈ [-1, 1]` via one-sided power laws with independently calibrated asymmetric multipliers. For example, β deflates 6,559× at r=+1 (cost becomes free) and inflates 8.8× at r=−1 (cost dominates). The exponent α=2.92 concentrates action at the extremes.

**Hyperparameter Selection**: Class-(i) parameters (τbase, µ, b, β, λ) jointly tuned via Bayesian optimization (~25,000 configurations) on a 70/30 dev/test split. Class-(ii) constants (difficulty anchors, clip range, cost normalization) are design-time choices. Sensitivity analysis shows τbase and β are fragile (±5.2 pp and ±3.6 pp worst-case), while λ is robust.

## Critical Assessment

- **Novelty**: The spatial capability routing formulation is a genuine contribution. The key insight — representing both queries and models in a shared capability space and using geometric distance for dispatch — is elegant and well-motivated. However, the individual components (BERT classifier, complexity estimator, cost-penalized scoring) are fairly standard ML building blocks. The novelty lies in the composition and the calibration protocol, not in any single technique.
- **Rigor**: The evaluation is thorough along several axes (accuracy, cost, latency, route-exact efficiency, agent scaling). The 5-fold out-of-fold skill-vector calibration avoids leakage. The worked example (§7.2) is unusually transparent. The hyperparameter sensitivity analysis is commendable. **However**, the evaluation is **in-domain calibrated** — the skill matrix is fit on folds of the same dataset used for evaluation. The authors honestly acknowledge this limitation but it means we do not know how well Brick generalizes to unseen workload distributions.
- **Reproducibility**: Strong. All code, datasets, and models are publicly available. The hyperparameter search space, selection criterion, and sensitivity analysis are documented in detail. The 17-page appendix-level transparency is rare.
- **Significance**: The practical significance is high — production deployments genuinely face the 10-100× cost gap between frontier and open-weight models. The continuous preference knob is a production-friendly design. The cascade-vs-pure-routing analysis for agents is timely and important. The theoretical significance is moderate — the thermodynamic analogy (§10) is speculative and not formalized.
- **Potential red flags**:
  1. **Three-model pool only**: All results are on a pool of three models. It is unclear how the skill matrix estimation and routing math scale to K=10 or K=20 models (sparse cells, curse of dimensionality in the capability space).
  2. **In-domain evaluation**: Skill vectors are calibrated on Dataset A folds and evaluated on Dataset A. External generalization is untested.
  3. **Router overhead**: Brick's decision latency (p50=2998ms in benchmark) is 200× slower than Cascade Routing (15.1ms) due to two remote classifier calls. The authors note production latency is 494ms with co-located classifiers, but this is still non-trivial overhead.
  4. **The 6.27 pp oracle gap**: Brick captures only 24% of the available headroom. The remaining 76% represents information the router cannot extract, and the paper does not provide a clear path to closing most of it.
  5. **Sensitivity**: τbase and β are fragile parameters (±5.2 pp and ±3.6 pp worst-case degradation), which could be problematic in production if the workload distribution shifts.

## Strengths

- Addresses a genuinely important practical problem with clear economic motivation (10-100× cost gap at production scale).
- The geometric routing formulation is elegant: queries and models live in the same space, dispatch is argmin of a well-defined objective.
- Unusually transparent methodology: worked example with all numerical intermediates, full hyperparameter protocol, sensitivity analysis, honest limitations discussion.
- The preference knob is a pragmatic design that bridges the gap between research prototypes and production systems.
- The cascade-vs-pure-routing analysis for agentic deployments is a timely and important contribution — as agents become more common, this argument will only grow stronger.
- All code, data, and models are open-source with permissive licenses.
- The complementary error set analysis (refusal patterns, per-capability strengths) provides actionable insights beyond the routing system itself.

## Limitations

- **Three-model pool**: Results may not generalize to larger pools. The skill matrix estimation with probability-weighted averaging may become noisy with more models, and the Bayesian-smoothed variant is proposed but not implemented.
- **In-domain calibration**: The skill matrix and knob hyperparameters are calibrated on Dataset A. Strict out-of-distribution generalization is untested and acknowledged as a limitation.
- **Router decision latency**: 2.998s median in benchmark (494ms in production) is significant for latency-sensitive applications, especially compared to Cascade Routing's 15ms.
- **Static skill matrix**: The skill matrix is locked after calibration. It does not adapt online to changing model capabilities (e.g., model updates, RLHF iterations) or shifting query distributions.
- **Six fixed capability dimensions**: The choice of dimensions is not justified beyond the Dataset A source composition. Other domains (e.g., legal, medical, multilingual) may require different dimensional bases.
- **Domain-agnostic complexity head**: The complexity classifier deliberately ignores the capability distribution, leaving performance on the table. The authors acknowledge this but the current system pays the cost.
- **No vision/multimodal routing**: Despite the vision stated in the conclusion, the current system is text-only.

## Relevance

该论文提出的 MoM 路由框架与高效 LLM 部署直接相关，其能力空间几何路由思想和连续偏好旋钮设计对多模型服务架构具有重要参考价值。对于关注推理成本优化、多模型协作以及 agentic 系统设计的 AI 工程实践尤为相关。

## Questions for Further Exploration

- How does the routing performance degrade as the model pool expands to K=10 or K=20 models? Does the skill matrix become too sparse?
- Can the skill matrix be updated online as new query-response data arrives, without full recalibration?
- What is the cross-domain transferability of the ModernBERT capability classifier — does it work on medical, legal, or multilingual queries without retraining?
- How would a domain-conditioned complexity estimator (taking p(x) as input) affect the oracle gap? The authors identify this as the most promising direction.
- Is the 24% headroom recovery a fundamental limit of the six-dimensional spatial representation, or can richer feature extraction (e.g., attention-based query embeddings) close more of the gap?
- How does Brick compare against a simple ensemble (majority vote across all three models) in terms of accuracy-per-dollar?
- Can the thermodynamic analogy (query entropy, partition function) be formalized into a principled knob-free tie-breaker, or is it just a metaphor?
