> **一句话总结：** 本文提出 DUAL-BLADE 双路径 KV 缓存卸载框架，根据运行时内存可用性将 KV 张量动态分配至页缓存路径或绕过内核栈的 NVMe-direct 路径，配合自适应流水线并行，在 OPT-6.7B 边缘推理场景下将 prefill 延迟降低 33.1%、decode 延迟降低 42.4%，SSD 利用率提升 2.2 倍。

> **TL;DR:** DUAL-BLADE addresses three bottlenecks in NVMe-based KV-cache offloading for edge LLM inference (single-GPU, 8-32 GB DRAM): (1) page-cache thrashing that collapses hit ratios during decode, (2) kernel storage stack overhead that underutilizes NVMe bandwidth, and (3) LBA interleaving from blk-mq that breaks sequential locality. The solution combines a dual-path KV residency policy (page-cache path for tensors fitting in DRAM, NVMe-direct via io_uring_cmd for the rest), sequential-LBA placement to preserve device-level sequentiality, and adaptive pipeline parallelism to overlap storage I/O with GPU DMA. On OPT-6.7B with two different NVMe SSDs (PCIe Gen5 and Gen4), DUAL-BLADE reduces prefill latency by up to 33.1% and decode latency by up to 42.4%, while improving SSD utilization by up to 2.2x compared to the FlexLLMGen baseline across diverse memory budgets (2-11 GB).

## 核心贡献

- 首次系统性地实验识别并分析了 LLM 推理中 KV 缓存卸载场景下的页缓存抖动（page-cache thrashing）问题：当可用页缓存小于 KV 工作集时，LRU 策略导致级联驱逐，命中率从 42% 骤降至 <1%。
- 提出双路径 KV 驻留（Dual-Path KV Residency）机制：运行时动态估算可用页缓存预算，将 KV 张量分为 Group 1（页缓存路径）和 Group 2（NVMe-direct 路径），从结构上避免抖动。
- 设计 NVMe-direct 路径：通过 io_uring_cmd 绕过 VFS/文件系统/块层等内核存储栈，将 KV 张量映射到连续逻辑块地址（LBA）区域，消除内核开销并保持设备级顺序访问。
- 引入自适应流水线并行（Overlap-Intra / Overlap-Cross）：在多拷贝线程环境下动态选择流水线策略，重叠存储 I/O 与 GPU DMA，进一步隐藏延迟。
- 采用 GPU-NVMe Co-DMA 方案：GPU 和 NVMe 共享同一 pinned buffer，将数据搬运从 3-hop 缩减为 2-hop。

## Key Contributions

- Provides the first experimental identification of page-cache thrashing as a critical bottleneck in KV-cache offloading for LLM inference, demonstrating a sharp hit-ratio cliff from 42% to under 1% when available page-cache falls below the KV working set size.
- Proposes a dual-path KV residency framework that dynamically partitions KV tensors between a page-cache path (for tensors that fit within the estimated DRAM budget) and an NVMe-direct path (bypassing the kernel storage stack entirely via io_uring_cmd), structurally eliminating thrashing.
- Designs sequential-LBA placement for the NVMe-direct path, mapping all Group 2 KV tensors to one large contiguous NVMe namespace extent, preserving end-to-end sequentiality from tensor-level access down to device-level LBAs and reducing FTL/controller scheduling overhead.
- Introduces adaptive pipeline parallelism with runtime strategy selection between Overlap-Intra (parallel storage reads) and Overlap-Cross (staggered start overlapping storage I/O with GPU DMA), converging in 3-4 decode iterations.
- Implements GPU-NVMe Co-DMA where GPU and NVMe share the same 4-KiB-aligned pinned memory buffer, eliminating the memcpy hop present in the conventional 3-hop data path.

## Methodology

DUAL-BLADE is built on top of FlexLLMGen and consists of three complementary design components:

1. **Dual-Path KV Residency**: At initialization, each KV pair is abstracted as a KV Placement Unit (KPU). A budgeter estimates the available page-cache budget Bpc from cgroup memory stats, subtracting pinned memory reserved for GPU DMA. A knob X partitions KPUs into Group 1 (page-cache path, using mmap) and Group 2 (NVMe-direct path). The design is pluggable -- a ranker can drive X to prioritize high-reuse tensors for the page-cache path.

2. **NVMe-direct with Sequential-LBA Placement**: Group 2 KPUs are mapped onto a contiguous NVMe namespace extent via a hash map (tensor ID to start LBA and block count). Three invariants are enforced: alignment (tensor I/O unit is a multiple of LBA size), disjointness (no overlapping extents), and contiguity (each tensor begins where the previous ends). I/O is executed via io_uring_cmd, bypassing VFS, filesystem, and block layer. Each tensor transfer is chunked according to the NVMe controller's MDTS, and native NVMe commands are submitted asynchronously to the submission queue. On context teardown, TRIM commands reclaim LBA space.

3. **Adaptive Pipeline Parallelism**: In a two-copy-thread setting, the system profiles Overlap-Intra (parallel storage reads) and Overlap-Cross (staggered start, overlapping storage I/O with GPU DMA) during the first few decode iterations, then selects the higher-throughput strategy for all subsequent iterations.

**Experimental setup**: Single-GPU system with Intel Core Ultra 7 265K (20 cores), 16 GB host DRAM, NVIDIA RTX 5060 Ti (16 GB), Ubuntu 24.04.2, Linux 6.8.0, PyTorch 2.8.0 with CUDA 12.8. Two SSDs: SSD A (Samsung PM9D3a, PCIe Gen5, 4 KiB LBA, 256 KiB MDTS) and SSD B (Samsung 990 PRO, PCIe Gen4, 512 B LBA, 2 MiB MDTS). Model: OPT-6.7B, 512-token prompt, 32-token generation, batch size 32.

## Results & Highlights

- **End-to-end latency**: On SSD A, DUAL-BLADE reduces prefill latency by up to 33.1% and decode latency by 8.2-42.4% across 2-11 GB memory limits. On SSD B, prefill reduction up to 25.4% and decode reduction 11.7-57.8%.
- **Page-cache thrashing elimination**: DUAL-BLADE achieves linearly increasing page-cache hit ratios with increasing memory, completely eliminating the thrashing zone (2-7 GB) seen in the baseline where hit ratios collapse below 1%.
- **NVMe utilization improvement**: On SSD A, DUAL-BLADE achieves 2.2x higher NVMe busy ratio for prefill writes and near-100% utilization for decode reads/writes vs. 44-55% in baseline. The 256 KB decode write latency drops from 4.91 ms to 0.06 ms (98%+ reduction), demonstrating that baseline latency is dominated by software overhead, not device processing.
- **Disk throughput**: On SSD A, NVMe-direct outperforms baseline by 43% in prefill-write throughput and 120% (2.2x) in decode-read throughput. On SSD B, gains of 43% and 88% respectively.
- **Pipeline parallelism**: Enables further 7-9% decode latency reduction on SSD A and 3-5% on SSD B across varying DRAM-SSD tiering ratios.
- **Practical edge scenarios**: Evaluated on four data wrangling tasks (Entity Matching, Data Imputation, Error Detection) with OPT-6.7B under 4 GB memory limit; DUAL-BLADE outperforms baseline in most cases with latency reductions up to 0.85x.

## Critical Assessment

- **Novelty**: The dual-path residency idea is a natural but well-executed extension of existing tiering concepts. The page-cache thrashing identification for KV offloading is a genuine and valuable empirical contribution. The NVMe-direct path using io_uring_cmd with sequential-LBA placement is technically solid, though kernel-bypass I/O is well-established in the storage systems literature. The adaptive pipeline parallelism adds moderate novelty. Overall, the paper's novelty lies more in the systems integration and the thorough motivational analysis than in any single component.
- **Rigor**: Strong experimental methodology with two different SSDs, ablation study across three design variants, bpftrace-based NVMe instrumentation, sweep across memory limits from 2-11 GB, and evaluation on practical edge workloads. However, the evaluation is limited to OPT-6.7B only; testing on more modern architectures (e.g., LLaMA, Mistral) would strengthen generalizability claims. The batch size is fixed at 32 and generation length at 32 tokens; longer generation scenarios are not explored.
- **Reproducibility**: Good. The framework (FlexLLMGen with specific commit), hardware specifications, SSD models, and software versions are all specified. The io_uring_cmd-based NVMe-direct path requires Linux kernel support and char device access, which may limit portability. The page-cache budget estimation relies on cgroup stats, which is standard but environment-dependent.
- **Significance**: For the edge AI deployment niche (single GPU, tight memory, NVMe offloading), this is a significant systems contribution. The 42.4% decode latency reduction and 2.2x SSD utilization improvement are meaningful. However, the relevance is narrower for cloud-scale inference where GPU-SSD P2P DMA (GDS, BaM) and CSDs (computational storage) are available, as the authors acknowledge.
- **Red flags**: The paper claims the first identification of page-cache thrashing in KV offloading, but the phenomenon is a well-known consequence of LRU caching with scan-like workloads in OS literature. The evaluation uses only OPT models, which are outdated; modern decoder-only transformers (LLaMA family) with grouped-query attention would have different KV-cache size characteristics. The decode latency improvements are most dramatic under severe memory pressure (2 GB), which may not represent typical edge deployment configurations. The 2 GB scenario essentially demonstrates that the system avoids catastrophic failure rather than achieving high absolute performance.

## Strengths

- Thorough motivational analysis with clear root-cause identification of three distinct bottlenecks (page-cache thrashing, kernel stack overhead, LBA interleaving).
- Clean architectural design with three complementary components, each addressing a specific bottleneck, validated through ablation.
- Evaluation across two NVMe SSD generations (PCIe Gen5 and Gen4) demonstrates hardware-agnostic benefits.
- Practical relevance to edge AI deployment scenarios with realistic memory constraints.
- Well-structured paper with clear problem-solution mapping and per-component quantification of improvements.

## Limitations

- Only evaluated on OPT-6.7B; no validation on modern LLM architectures (LLaMA, Mistral, Gemma) which use grouped-query attention and have different KV-cache characteristics.
- Generation length fixed at 32 tokens; longer generation scenarios (hundreds to thousands of tokens) where decode I/O dominates even more are not explored.
- The NVMe-direct path requires io_uring_cmd and NVMe char device access, which may not be available on all platforms and requires privileged access.
- Single-batch evaluation; multi-request concurrent serving scenarios with shared KV-cache management are not addressed.
- The approach does not leverage GPU-SSD P2P DMA (GDS/BaM), which could further reduce latency on supported platforms.
- No power or energy consumption measurements; the paper focuses exclusively on latency and throughput.

## Relevance

本文对边缘端 LLM 推理系统的存储 I/O 优化具有直接参考价值，尤其是对研究 KV-cache 卸载、NVMe SSD 加速和内核旁路技术在推理场景中应用的工作。其双路径设计和页缓存抖动分析对构建高效的单 GPU 推理服务系统具有实用指导意义。

## Questions for Further Exploration

- How does DUAL-BLADE perform with modern LLM architectures using grouped-query attention (GQA) or multi-query attention (MQA), where the KV-cache per-token size is significantly smaller?
- What is the performance impact with much longer generation lengths (512-4096 tokens) typical of agentic AI workflows, where the KV working set grows substantially during decode?
- Can the dual-path residency policy incorporate importance-based KV cache eviction (e.g., from attention sinks or H2O) to further optimize the page-cache path utilization?
- How does DUAL-BLADE interact with emerging computational storage devices (CSDs) that can perform attention computation in-situ, as referenced in the related work (InstInfer, INF2)?
- What are the implications for multi-request serving scenarios with shared prefix caching, where multiple requests reuse the same KV-cache blocks?
