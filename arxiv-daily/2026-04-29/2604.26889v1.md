> **一句话总结：** 通过修改NVIDIA开源内核驱动并拦截GPU门铃寄存器写入，完整捕获闭源用户态驱动发出的硬件命令流，揭示了CUDA数据传输的双DMA模式机制（内联DMA启动延迟仅24ns，直接DMA带宽达22 GiB/s）以及CUDA Graph在新版本中通过缩减命令足迹降低启动开销的原理。

> **TL;DR:** This paper recovers the complete hardware command streams emitted by NVIDIA's closed-source userspace driver by instrumenting the open-source kernel driver and placing hardware watchpoints on the GPU doorbell register. Two case studies demonstrate the value: (1) cudaMemcpy uses two DMA modes -- inline DMA via the compute engine (24 ns startup, saturates at 17.5 GiB/s) for transfers <= 24 KiB, and direct DMA via the copy engine (500 ns startup, saturates at 22 GiB/s) for larger transfers; (2) CUDA 13.0 reduces graph launch overhead from 209 us to 5.9 us (at graph length 2000) compared to CUDA 11.8 by shrinking the command footprint from ~45 KB to ~2.2 KB and reducing doorbell writes. The methodology enables CUDA-bypassing direct DMA command issuance for isolating raw hardware performance from driver overhead.

## 核心贡献

- **命令流捕获与重建**：通过修改NVIDIA开源内核驱动的内存映射路径，在GPU门铃寄存器（doorbell）的用户空间映射上设置硬件观察点（watchpoint），在命令提交瞬间冻结用户态驱动，完整捕获并解析pushbuffer命令流，实现闭源驱动行为的高保真可视化。
- **CUDA数据传输机制深度分析**：揭示了cudaMemcpy内部的双DMA模式切换机制——小数据传输（<=24 KiB）使用计算引擎的内联DMA（启动延迟24ns），大数据传输使用专用拷贝引擎的直接DMA（启动延迟500ns、峰值带宽22 GiB/s），并通过绕过CUDA直接编程DMA引擎，首次分离出纯硬件DMA性能。
- **CUDA Graph开销优化解析**：通过对比CUDA 11.8与CUDA 13.0的命令流，发现新版本将graph长度2000时的命令足迹从~45KB缩减至~2.2KB、doorbell写入次数大幅减少，从而将启动时间从209us降至5.9us，揭示了命令足迹与启动开销的强耦合关系。
- **通用方法论贡献**：提出的方法不仅适用于数据传输和CUDA Graph，还可推广到其他CUDA API的分析与优化，为GPU中间件行为理解和软硬件协同设计提供实用基础。

## Key Contributions

- **Command-stream capture and reconstruction**: Modifies the NVIDIA open-source kernel driver's memory-mapping path and installs a hardware watchpoint on the userspace mapping of the GPU doorbell register. When the userspace driver rings the doorbell to submit work, the watchdog handler traps into the kernel, pauses the driver, and captures a complete, intact snapshot of the newly submitted pushbuffer and GPFIFO entries. A shadow doorbell page mechanism ensures transparent forwarding of the intercepted write to the real hardware doorbell after observation.
- **Dual-DMA-mode analysis in cudaMemcpy**: Identifies that the NVIDIA closed-source driver selects between inline DMA (compute engine, <=24 KiB threshold) and direct DMA (dedicated copy engine, >=24 KiB) for host-to-device transfers. Characterizes raw hardware performance: inline DMA achieves 24 ns startup latency and 17.5 GiB/s peak bandwidth; direct DMA achieves 500 ns startup latency and 22 GiB/s peak bandwidth.
- **CUDA-bypassing direct command issuance**: Leverages UVM address unification and knowledge of the submission architecture to inject custom pushbuffer commands directly to the GPU, bypassing the CUDA runtime entirely. This enables device-side timestamped measurement of DMA engine performance without driver overhead contamination.
- **CUDA Graph optimization analysis**: Shows that CUDA 13.0 achieves near-constant-time graph launch (1.9-5.9 us for graph lengths 1-2000) compared to CUDA 11.8's linear scaling (1.8-209 us), attributable to a dramatically smaller command footprint (~2.2 KB vs. ~45 KB) and fewer doorbell writes, establishing command size as a useful precursor for launch overhead.

## Methodology

The paper's methodology centers on a modified NVIDIA OpenGPU kernel driver that instruments the GPU command submission path at three levels:

1. **Shadow doorbell interception**: The original kernel driver maps the PCIe BAR0 doorbell register directly into userspace. The modified driver instead allocates a shadow RAM page and maps that to userspace. When the closed-source userspace driver writes to what it believes is the doorbell, the write lands in the shadow page and triggers a CPU debug register watchpoint. The watchdog handler then reads the channel ID, reconstructs the execution state, and forwards the write to the real doorbell.

2. **Execution state reconstruction**: Using the intercepted channel ID, the system locates the corresponding KernelChannel object, maps USERD and RAMIN into CPU virtual address space, walks the GPU MMU page tables to resolve physical addresses, and reads GPFIFO entries and pushbuffer contents. The pushbuffer commands are parsed using method listings from the open-sourced driver headers.

3. **Custom command issuance**: Exploiting the UVM address unification (Finding 1: GPU virtual addresses in pushbuffer commands are shared with the process's userspace virtual address space), the authors inject custom pushbuffer segments with DMA commands and semaphore-based progress trackers. All transfer commands and timestamps are coalesced into a single pushbuffer submission, enabling uninterrupted device-side execution and pure hardware timing via GPU-written timestamps.

The evaluation platform is an NVIDIA A40 (Ampere) GPU on an Intel Xeon Gold 6338 system with PCIe Gen4 x16, running Rocky Linux 9.5 with a patched Linux 5.14 kernel. Two CUDA stacks are compared: CUDA 11.8 (driver 520.61.07) and CUDA 13.0 (driver 580.105.08).

## Results & Highlights

- **Inline DMA vs. Direct DMA**: For host-to-device transfers, the driver uses inline DMA (data embedded in pushbuffer payload, dispatched to compute engine) for transfers <= 24 KiB, and direct DMA (source/destination addresses specified, dispatched to dedicated copy engine) for transfers >= 24 KiB. The compute engine achieves much lower startup latency (24 ns vs. 500 ns) but saturates quickly at 17.5 GiB/s, while the copy engine scales to 22 GiB/s at ~1 MiB.
- **Profiler vs. raw hardware disparity**: Nsight "CUDA HW" durations include significant non-hardware overhead. The fraction of profiler-reported latency attributable to raw hardware execution declines as transfer size grows (hardware time increasingly dominates), but for small transfers, the overhead fraction is substantial.
- **CUDA Graph launch time**: CUDA 11.8 shows linear growth in launch time with graph length (1.8 us to 209 us for lengths 1 to 2000), while CUDA 13.0 shows near-constant behavior (1.9 us to 5.9 us). Command size tracks launch time closely: CUDA 11.8 grows from 328 B to 45,476 B; CUDA 13.0 grows from 340 B to only 2,216 B.
- **Staircase pattern**: Under CUDA 11.8, both command size and launch time exhibit a discrete staircase pattern at fine granularity (0-200 kernel chain length), with step transitions closely aligned, further confirming the tight coupling between command footprint and launch overhead.
- **Submission pattern**: CUDA 13.0 also reduces the number of doorbell writes (distinct submission cycles) compared to CUDA 11.8, indicating a more efficient batching strategy.

The paper demonstrates deep familiarity with NVIDIA GPU internals, the OpenGPU kernel driver codebase, and low-level PCIe/MMIO mechanics.

## Critical Assessment

- **Novelty**: High. While prior work has modified the kernel driver for scheduling or security analysis, this is the first to reconstruct the complete end-to-end command trajectory from CUDA API calls through the closed-source userspace driver to hardware-level pushbuffer commands. The shadow doorbell technique and the CUDA-bypassing direct command injection are genuinely novel.
- **Rigor**: Strong. The paper provides detailed descriptions of the interception mechanism, state reconstruction procedure, and experimental setup. Two CUDA versions are compared. However, the evaluation is limited to a single GPU (A40) and a single CPU platform. The paper acknowledges this but argues the submission architecture is generation-invariant, which is reasonable but not empirically verified.
- **Reproducibility**: Moderate. The methodology requires modifying the kernel driver, patching the Linux kernel (exporting `sched_task_fork`), and deep knowledge of GPU MMU internals. While the open-source driver provides the base, the barrier to reproduction is high. No open-source code release is mentioned.
- **Significance**: High for the GPU systems community. The ability to separate hardware performance from driver overhead addresses a fundamental limitation in current GPU profiling. The findings about dual DMA modes and CUDA Graph optimization internals have practical implications for performance engineering.
- **Red flags**: The paper uses "xxx, 2026, xxx" as the venue header, suggesting it is a preprint under submission. The CUDA 13.0 version referenced appears to be a future/internal version not yet publicly available (as of April 2026, CUDA 12.x is the latest public release), which raises questions about general accessibility of the findings. The 24 KiB DMA mode switch threshold is stated for "our environment" without exploring whether it varies across hardware generations or driver versions.

## Strengths

- Provides unprecedented visibility into the closed-source NVIDIA userspace driver's command translation process.
- The shadow doorbell interception technique is elegant and minimally invasive -- it does not modify the userspace driver at all.
- The dual case study structure (hardware-centric data movement + software-centric CUDA Graph) effectively demonstrates the methodology's versatility.
- The discovery of UVM address unification being leveraged internally by the driver (Finding 1) is a useful insight for anyone working on GPU command streams.
- Clear and detailed technical writing with informative figures illustrating the command submission hierarchy.

## Limitations

- Single GPU platform (NVIDIA A40 only); results may differ on other architectures (Hopper, Ada Lovelace, etc.).
- No open-source code or tool release mentioned, limiting immediate community adoption.
- The CUDA 13.0 stack used may not be publicly available, which could hinder independent verification.
- The methodology requires a modified kernel driver and patched Linux kernel, imposing significant infrastructure requirements.
- The analysis of CUDA Graph internals is limited to macroscopic submission-level indicators (doorbell count, command volume) because many commands use NVIDIA-internal terms with undocumented semantics.
- The performance overhead of the watchpoint interception mechanism itself is not rigorously quantified (though the paper uses separate unmodified-driver runs for performance measurements).

## Relevance

本文对GPU系统性能分析和CUDA底层机制的研究具有重要参考价值，特别是其"绕过CUDA直接编程DMA引擎"的方法论可用于精确测量GPU互连的原始硬件性能，对研究GPU disaggregated memory和异构计算系统中软件开销占比问题特别有意义。

## Questions for Further Exploration

- How does the DMA mode switch threshold (observed at 24 KiB on A40) vary across GPU generations (e.g., H100, B200) and driver versions?
- Can the shadow doorbell interception technique be extended to capture multi-GPU communication primitives (NVLink, GPUDirect RDMA) with similar fidelity?
- What is the runtime overhead of the watchpoint interception itself, and could it be used in production profiling scenarios rather than offline analysis?
- Do the CUDA Graph command footprint reductions in CUDA 13.0 generalize to more complex graph topologies (branching, multi-stream)?
- Could the CUDA-bypassing direct command issuance methodology be used to construct lightweight GPU communication runtimes that avoid closed-source driver overhead entirely?
