LibEvoBench: Probing Temporal Knowledge Stratification in Code Generation Models

arXiv: 2606.25402

Date: 2026-06-24

https://arxiv.org/abs/2606.25402

一句话总结： 提出 LibEvoBench——一个覆盖 PyTorch/NumPy/SciPy 多版本、超 12.5 万实例的多任务基准，配合新指标 SEUS 与基于版本索引兼容性矩阵的确定性幻觉分类法，系统揭示当前主流 LLM 基本是"版本无感 (version-oblivious)"的：在演化 API（尤其新版本）上准确率持续下滑，而显式指定目标版本几乎无效，提供相关文档却能显著提升准确率，说明这是训练范式的结构性缺陷而非模型能力或规模上限。
TL;DR: LibEvoBench is a multi-task, multi-version benchmark (PyTorch/NumPy/SciPy, 29 library versions, 125k+ instances, 16k+ APIs) that evaluates whether code-generation LLMs can distinguish among coexisting API versions. Its new metric SEUS and a deterministic, matrix-driven hallucination taxonomy (separating anachronistic errors from fabrication) show that frontier models are fundamentally version-oblivious: accuracy degrades on evolving APIs (especially recent versions) while staying flat on stable ones; an explicit version constraint yields no benefit, whereas relevant documentation boosts accuracy 10–20 points; degradation is scale-invariant within a model family and driven by temporal confusion rather than hallucination.

核心贡献

  • LibEvoBench 基准：首个大规模、版本索引化的 Python 库 API 评测基准，覆盖 PyTorch、NumPy、SciPy 共 29 个版本，提供超 12.5 万评测实例、跨 3 个任务的 1.6 万+ API，据作者所称为目前最大的版本索引化 Python API 评测集。
  • 三大互补任务：API Calling (API-C，真实代码补全中的上下文式调用)、API Identification (API-I，根据模块路径与脱敏描述识别 API 名)、Signature Recall (SR，仅给全限定名与版本、凭记忆复现完整签名)，分别探测调用式、声明式与细粒度参数式知识。
  • SEUS 新指标 (Software Evolution Understanding Score)：以稳定 API 分数与演化 API 分数的调和平均为核心，并叠加版本间稳定性惩罚 (λ=0.5) 与时代错乱 (anachronism) 惩罚 (γ=0.5)，强制模型必须在稳定与演化 API 上同时、且跨版本一致地表现良好才能得高分。
  • 确定性幻觉分类法：基于版本索引的 API 兼容性矩阵 A 与签名兼容性矩阵 S，通过自动查表将每条预测确定性地判为 Correct / Invalid / Unknown / Anachronistic 四类，无需人工标注，且首次将"时代错乱"（同一 API 不同版本的混淆）与纯粹的"捏造"区分开来。
  • 稳定 vs 演化 API 二分法：将每个 API 划分为跨版本签名不变的稳定 API 与被引入/移除/修改的演化 API（占比 33–71%），用以定位 API 演化对 LLM 行为的精确影响。
  • 前沿模型评测：评测 OpenAI GPT-4.1/5/5.1/5.4/5.5、Google Gemini 2.0/2.5/3 Flash、Anthropic Claude Sonnet 4/4.6 以及 Qwen3.5 (35B/122B/397B)，覆盖 4 个模型族、13 个模型。
  • 结构性结论：模型在演化 API（尤其新版本）上持续退化、稳定 API 保持平稳；版本显式约束无效、文档显著有效；退化在单族内不随规模改变、跨代缩小但持续存在；错误多为"混淆"而非"捏造"。这指向训练范式在时间知识分层上的系统性缺陷。

Key Contributions

  • LibEvoBench: a large-scale, version-indexed benchmark for Python library APIs spanning PyTorch, NumPy, and SciPy across 29 versions, with 125k+ evaluation instances and 16k+ APIs across three tasks — claimed to be the largest version-indexed Python API evaluation to date.
  • Three complementary tasks: API Calling (API-C, contextual API use from real version-pinned code); API Identification (API-I, isolating declarative API knowledge from a redacted description); Signature Recall (SR, probing fine-grained parametric signature knowledge from memory).
  • SEUS (Software Evolution Understanding Score): a composite metric that harmonic-means stable- and evolving-API scores, then applies an explicit stability penalty (λ=0.5) and an anachronism penalty (γ=0.5), so that strong stable-only performance cannot mask weak or inconsistent version-specific performance.
  • Deterministic hallucination taxonomy: classifies every prediction by automated matrix lookup into Correct / Invalid / Unknown / Anachronistic at both API and parameter granularity, requiring no manual coding, and uniquely separates temporal confusion (anachronism) from outright fabrication (Unknown).
  • Stable-vs-evolving API partition: every API is split into stable APIs (identical signature across versions) and evolving APIs (added/removed/modified, 33–71% of tracked symbols), enabling precise localization of evolution effects.
  • Frontier-model evaluation: 13 models across 4 families — OpenAI GPT-4.1/5/5.1/5.4/5.5, Google Gemini 2.0/2.5/3 Flash, Anthropic Claude Sonnet 4/4.6, and Qwen3.5 at 35B/122B/397B — with the Qwen sweep isolating the effect of scale within one family.
  • Structural findings: models are version-oblivious (degrade on evolving APIs, flat on stable); explicit version constraints give no benefit; documentation gives a 10–20 point boost; degradation is scale-invariant within a family and narrows but persists across generations; errors are dominated by confusion rather than fabrication.

Methodology

LibEvoBench is built on an automated pipeline that fuses real-world GitHub usage with two independent reconstructions of each library's public API surface.

  • Library/version selection: PyTorch, NumPy, SciPy — chosen for rapid evolution and widespread adoption; 29 versions total (10, 10, 9 respectively).
  • Usage collection: mines public GitHub repos whose requirements.txt pins a specific version; keeps deduplicated, permissively-licensed repos with ≥10 stars; clones at the manifest commit and uses static analysis (import tracking, alias propagation, type/module resolution against the pinned version) to map each call site to a fully qualified API.
  • API surface reconstruction (two sources must agree):
  • Documented surface — official Sphinx objects.inv inventory files.
  • Runtime surface — install each version, walk the module tree, introspect every function/method signature.
  • An API counts as present only when both agree; a previously confirmed API later dropped from docs is treated as deprecated.
  • Compatibility matrices: A (API-level, exists per version) and S (signature-level, per (API, signature) pair); NA cells (irreconcilable disagreement) are only 3.4% of cells. These matrices induce the stable/evolving split and power the deterministic taxonomy.
  • Tasks & scoring:
  • API-C uses 4 noise-reduction levels — L0 (code only) → L1 (code + redacted docstring) → L2 (adds # requires:lib==ver) → L3 (adds API name, focuses on parameter prediction). Scored by exact match (L0–L2) and precision/F1 (L3).
  • API-I scored by exact match on the API name.
  • SR scored by parameter F1 against the ground-truth signature.
  • SEUS computation: B_v = 2·S_v·E_v / (S_v + E_v) (harmonic mean of stable S_v and evolving E_v, each averaged over 4 task-level metrics); final SEUS = mean over library–task pairs of E_v[B_v] − λ·Std_v(E_v) − γ·A_{ℓ,t}, with λ=γ=0.5. The stability penalty targets E_v (not B_v) precisely because only evolving-API variation reveals true version learning; rankings are verified robust to λ/γ choice (Appendix C).
  • Evaluation protocol: all models queried at zero temperature with no thinking budget, deliberately isolating parametric knowledge (the regime underlying implicit API selection in agentic coding).

Results & Highlights

  • Stable flat, evolving degrades: across all four model families, all three libraries, and all three tasks, evolving-API performance erodes progressively toward the most recent versions while stable-API performance stays nearly flat. Even the strongest models (GPT-5.5, Sonnet 4.6) lose 5–10 points on evolving APIs.
  • Scale shifts level, not curve: within Qwen3.5 (35B/122B/397B), larger size lifts quality uniformly but leaves the decay curve unchanged (near-parallel downward steps between versions), confirming degradation is a structural property, not a capability/scale ceiling. Newer GPT generations narrow but do not eliminate the gap.
  • Version constraint is useless; documentation is powerful: in the API-C noise-reduction staircase, adding a redacted docstring (L0→L1) yields a consistent 10–20 point gain across all models (evidence of genuine retrieval), whereas adding an explicit version tag (L1→L2) yields no measurable improvement — the version cue triggers a response with no version-differentiated knowledge behind it.
  • Errors are confusion, not fabrication: the taxonomy shows API-level errors cluster as Invalid (wrong-but-real symbol at the target version) and Unknown; anachronistic predictions at API-selection level are negligible because models fall back on high-frequency symbols valid across most versions. At the parameter level (SR), anachronistic errors are a substantial, consistent 6–14% of the error budget and intensify with version recency; SR amplifies both error types because it forces the model to commit to version-specific parameters, surfacing latent errors that API-C@L3 hides through implicit hedging.
  • SEUS leaderboard (top-tier excerpt): GPT-5.4 leads with SEUS 86.0 (stable 92.0, evolving 86.4, stability penalty 2.0 pp, anachronism penalty 1.0 pp); GPT-5.5 (85.1) has a slightly higher stable score (92.3) but ranks lower due to higher inconsistency (2.7 pp penalty) — SEUS rewards consistency over raw peak. Sonnet 4.6 (81.0) rounds out the top tier (>80); a middle cluster sits at 70–78; lower tier below 70; Qwen3.5-397B reaches 67.9.
  • Retention diagnostic: top-tier models preserve >93% of stable-API performance on evolving APIs; mid-tier drops below 86%; anachronism penalty grows from 0.9 pp (top) to 1.8 pp (bottom) — weaker models are both less accurate on evolving APIs and more prone to cross-version confusion.

Authors & Credibility

  • Daniele Cipollone, Sergey Titov, Maliheh Izadi, Egor Bogomolov (JetBrains Research, Amsterdam) and Arie van Deursen (Faculty of EEMCS, Delft University of Technology). The JetBrains Research + TU Delft software-engineering pedigree gives strong credibility on version-aware tooling and static-analysis methodology.
  • The benchmark construction is fully automated and grounded in real version-pinned GitHub code plus dual API-surface reconstruction (Sphinx inventories + runtime introspection), which is methodologically sound and reproducible in spirit.
  • The paper honestly positions itself against prior work (GitChameleon, CodeUpdateArena, VersiCode, LibEvolutionEval) and clearly articulates what each lacks (manual-curation limits, heuristic string matching, coarse F1 conflating version errors with general hallucination).
  • The framing is measured: claims are backed by deterministic matrix lookup rather than manual labeling, and the authors explicitly discuss the contamination threat and why their three indirect signals argue against memorization as the dominant driver.

Critical Assessment

  • Novelty — Strong on taxonomy, incremental on benchmark. The genuinely new contribution is the deterministic, version-indexed hallucination taxonomy that separates anachronistic errors (real symbol, wrong version) from Unknown (fabrication) at both API and parameter granularity — a diagnostic no prior scheme provides per prediction. The multi-version benchmark itself is a natural extension of an active line (VersiCode, LibEvolutionEval) and the real-code-calling + declarative-task design is well-executed but not conceptually unprecedented.
  • Rigor — High. The taxonomy classification by automated matrix lookup avoids subjective manual coding; the SEUS metric's robustness to λ/γ is checked; the stable/evolving split is principled; the zero-thinking-budget choice is deliberate and well-justified. NA cells are transparently reported (3.4%).
  • Generalizability — Limited. Only three Python libraries (albeit popular and fast-evolving) and at most 10 versions each. No non-Python languages, no web/JS or Rust ecosystems, and evolving-API coverage depends on how much a library actually churned. Findings about "structural version-obliviousness" are plausibly general but empirically demonstrated only for this trio.
  • The headline "version constraint yields no benefit" is the most striking claim — and also the one most in need of scrutiny. The L1→L2 null result hinges on prompt formatting (# requires:lib==ver as a comment). It is conceivable that a more salient or chat-formatted version instruction (e.g., a system message, or exposing the version inside a docstring header) could elicit different behavior; the authors do not ablate instruction phrasing.
  • Zero thinking budget is a defensible choice for isolating parametric knowledge but means the benchmark may understate what reasoning-capable models could do; the authors concede extended CoT might improve SR but cannot recover collapsed version-specific facts.
  • Contamination is acknowledged but not fully controlled — the three indirect signals are suggestive, not conclusive, and the authors honestly invite tighter protocols.

Strengths

  • Deterministic, fully-automated hallucination taxonomy via compatibility matrices — no manual labeling, high precision, uniquely isolates anachronism from fabrication.
  • Clean, interpretable SEUS metric that explicitly rewards cross-version consistency and penalizes temporal confusion, with sensitivity analysis on its hyperparameters.
  • Three complementary tasks probing distinct knowledge modalities (contextual calling, declarative identification, parametric recall), plus a 4-level noise-reduction staircase that disentangles retrieval from recall.
  • Scale-controlled analysis within the Qwen3.5 family cleanly attributes degradation to training structure rather than model capacity.
  • Strong practical signal for practitioners: RAG-with-documentation helps a lot; telling the model the version does not.

Limitations

  • Only three Python libraries and ≤10 versions each; no language/ecosystem diversity; coverage of evolving APIs is bounded by how much each library actually evolved.
  • The "version constraint useless" result is tied to a specific (comment-style) prompt format and is not ablated across instruction phrasings.
  • Zero thinking budget may understate reasoning-capable models, particularly for Signature Recall.
  • Training-data contamination cannot be ruled out conclusively; only indirect signals are provided.
  • Retention diagnostic is uninformative for weak models (e.g., Qwen3.5-122B shows 87.8% retention yet ranks 12th, because both stable and evolving scores are low).
  • No mitigation experiments — the paper diagnoses the problem but proposes no temporally-grounded training or retrieval fix to validate.

Relevance

对关注"代码大模型版本感知 / 持续学习 / API 演化 / 代码幻觉评测"方向的研究者与工程团队有直接价值：它给出了一个可复用的版本索引化基准、一个把"时代错乱"与"捏造"分离的确定性分类法，以及一条清晰的实证结论——当前 LLM 在演化 API 上是结构性版本无感的，加文档有效而加版本号无效。若你从事 RAG/文档检索、代码补全产品评估、或时间/版本知识注入研究，本文提供了扎实的评测底座与问题定义；若侧重训练方法，本文指明了"时间知识分层"这一待解的结构性瓶颈。

Questions for Further Exploration

  • Would a more salient version instruction (system message, structured prompt, or version embedded in the docstring) change the null L1→L2 result, or is version-obliviousness robust to instruction phrasing?
  • Can a temporally-grounded training or retrieval method (e.g., version-conditioned adapters, version-tagged training corpora) measurably close the evolving-API gap on LibEvoBench?
  • Does extended chain-of-thought reasoning materially improve Signature Recall, and does it introduce new error modes (e.g., confidently anachronistic parametrization)?
  • How do findings generalize beyond Python — to JS/TS, Rust, or Go ecosystems with different release cadences and deprecation cultures?
  • Is the degradation scale-invariant across much larger scales or only within the tested 35B–397B range; does it persist at frontier closed-model scales once newer training data cuts are included?
  • Can the deterministic anachronism taxonomy be extended to attribute errors to which specific prior version the model confused the target with, enabling more granular continual-learning diagnostics?
  • How sensitive are the SEUS leaderboard rankings to the evolving-API composition (the S/E ratios vary widely from 0.26 for SciPy-SR to 5.82 for PyTorch-API-C)?
