Constraint Tax in Open-Weight LLMs: An Empirical Study of Tool Calling Suppression Under Structured Output Constraints

arXiv: 2606.25605

Date: 2026-06-24

https://arxiv.org/abs/2606.25605

一句话总结： 本文报告了一个在真实生产 Agent 系统中可复现的故障模式——当 Tool Calling 与 JSON Schema 结构化输出约束同时开启时，多个开源权重模型（Qwen 系列、GPT-OSS-20B、Nemotron 3 Super 等）会完全停止调用工具（TIR 从 100% 跌到 0%），但各自单独评估时工具调用与 schema 合规都正常；作者追踪推理栈定位到根因是 JSON Schema 被编译成基于文法的 token 掩码，把以 < 开头的 XML 式工具调用 token 永久设为 −∞ 不可达，并提出无需重训的推理期缓解方案 "Transparent Two-Pass Execution"，可将工具调用率与端到端成功率恢复到 100%。
TL;DR: A reproducible production Agent failure: when Tool Calling and JSON Schema structured-output constraints are jointly enabled, multiple open-weight models (Qwen3.5/3.6 family, GPT-OSS-20B, Nemotron 3 Super) completely stop invoking tools (TIR drops 100% → 0%) even though both capabilities work fine in isolation. Inference-stack tracing localizes the root cause to grammar-based token masking that sets XML-style tokens to −∞ (unreachable). The authors propose the Constraint Priority Inversion (CPI) behavioral hypothesis and an inference-time fix, Transparent Two-Pass Execution (tools-first pass, schema-constrained second pass), which restores 100% TIR and 100% end-to-end success without retraining.

核心贡献

  • 识别并刻画"工具抑制 (Tool Suppression)"现象：在真实生产 Agent（外贸询盘分析）中，当 Tool Calling 与 JSON Schema 约束同时启用时，多个开源权重模型在 T2 条件下 TIR 从 100% 跌至 0%，而 T1（仅工具）和 T3（仅 schema）条件下能力均正常；首次把这种现象从传统的"答案质量下降型 Constraint Tax"升级为"动作执行层失效"的失败模式。
  • 给出实现层面的根因定位：通过追踪 SGLang 0.5.9 的完整调用链（response_format → xgrammar.compile_grammar() → GrammarMatcher.fill_next_token_bitmask() → apply_token_bitmask_inplace_triton()），证明 JSON Schema 被编译成有限状态机，在每个解码步把不在合法 JSON 状态中的 token logits 设为 −∞；对 Qwen 家族 XML 式 格式而言，< (U+003C) 在所有 FSM 状态下都被掩码，故工具调用 token 永远不可达。这是 paper 最硬的一条证据。
  • 系统排除大量混淆变量：在 7 个模型（20B–397B）、两种推理框架（SGLang/vLLM）、三种 schema 复杂度、四种工具强制策略（含 tool_choice="required"）、多种 SFT/GRPO 微调变体、本地/云端部署下，工具抑制都稳定复现，T2 TIR 始终为 0%——说明它不是模型规模、框架解析器、schema 复杂度或 prompt 弱化能解释的。
  • 提出 Constraint Priority Inversion (CPI) 行为假设：作为对"模型在 TS-C 模式下仍写出 need_search: true 却不调用工具"这一行为的解释假设，作者明确把 CPI 定位为"与观察证据一致的行为假设"而非"已验证的内部机制"——这是一种相当克制的因果归因态度。
  • 提出推理期缓解方案 Transparent Two-Pass Execution 并验证：Pass 1（Tools=ON, Schema=OFF）自由调用工具，Pass 2（Tools=OFF, Schema=ON）注入工具结果后生成 schema 合规响应；代价是多一轮推理 + 工具结果回传的 token 开销，但 TIR 0%→100%、ESR 0%→100%、JCR 保持 100%，无需改模型权重/训练/推理内核。
  • 给出工具抑制行为分类法 TS-A~TS-E：Empty Compliance、Simulated Retrieval、Intent Without Action、Tool-Free Hallucination、Frozen Required Tool，为后续跨模型、跨部署的抑制分析提供共同词汇，并标注了欺骗性风险等级。

Key Contributions

  • Identification and characterization of Tool Suppression. A reproducible production failure in which open-weight models stop invoking tools (TIR 100%→0%) exactly when Tool Calling and JSON Schema constraints are both on, while each capability works perfectly in isolation (T1, T3). Reframes Constraint Tax from answer-quality degradation to a behavior-level / action-execution failure.
  • Implementation-level root-cause localization to grammar-based token masking. Full inference-stack trace in SGLang 0.5.9 shows the JSON Schema is compiled into an FSM-based vocabulary mask; for Qwen-family XML syntax the < (U+003C) token is masked in every FSM state, so tool-call tokens are set to −∞ and become unreachable. This is the strongest single piece of evidence in the paper.
  • Systematic elimination of alternative explanations. The pattern holds across 7 models (20B–397B), two frameworks (SGLang, vLLM), three schema-complexity tiers, four tool-enforcement settings (including tool_choice="required"), multiple SFT/GRPO fine-tuned variants, and local/cloud deployment — T2 TIR is 0% throughout, so suppression cannot be attributed to scale, parser, schema complexity, or weak prompting.
  • Constraint Priority Inversion (CPI) behavioral hypothesis. Interprets TS-C (models emit need_search: true but no tool calls) as schema satisfaction dominating action selection. CPI is explicitly framed as a behavioral hypothesis consistent with evidence, not a verified internal mechanism — a notably cautious causal stance.
  • Transparent Two-Pass Execution, an inference-time mitigation with empirical validation. Pass 1 runs tools without schema; Pass 2 generates schema-compliant output from collected tool results. Restores TIR 0%→100%, ESR 0%→100%, JCR stays 100%, at the cost of one extra inference round; no retraining needed.
  • Tool Suppression behavioral taxonomy (TS-A…TS-E). Provides shared vocabulary (Empty Compliance, Simulated Retrieval, Intent Without Action, Tool-Free Hallucination, Frozen Required Tool) with deception-risk levels for cross-model analysis.

Methodology

The paper is structured as phenomenon discovery → root-cause tracing → behavioral interpretation → mitigation:

  1. Motivating production observation. A production foreign-trade inquiry-analysis Agent with an open-weight model + tools + JSON Schema enabled repeatedly failed to call tools; disabling only the schema constraint restored tool use. This triggered the investigation.
  1. Definitions and metrics (Sec. 3).
  • Tool Suppression = task requires tools AND model has tool capability AND model still omits tool execution under joint constraints. Distinguished from Tool Incapability.
  • Tool Invocation Rate (TIR) = N_tools_invoked / N_total.
  • Suppression Rate (SR) = 1 − TIR_constrained / TIR_baseline (normalizes for inherent calling tendency).
  • Also reported: JSON Compliance Rate (JCR), Avg Tool Calls/Session (ATC), End-to-End Success Rate (ESR).
  • A five-category behavioral taxonomy (TS-A…TS-E) on three axes (Tool Awareness / Tool Execution / Content Strategy).
  1. Controlled three-condition design (Sec. 4).
  • T1 (Baseline): Tools ON, Schema OFF → measures tool capability.
  • T2 (Joint Constraint): Tools ON, Schema ON → the production setting where suppression is detected.
  • T3 (Schema Control): Tools OFF, Schema ON → verifies schema compliance in isolation.
  • Identical system/user prompts, tool defs, parsing across conditions; only tools and response_format API params vary. 5 rounds, temperature 0.5, streaming, max 4096 tokens.
  1. Task set & models (Sec. 4). Five business-oriented task categories (Buyer Background, Company Verification, Market Search, Product Knowledge, Compliance Check), all genuinely requiring external info. Seven model instances from 20B–397B covering MoE, hybrid MoE-Mamba, VL, local/cloud, plus GPT-5.4-mini as a closed-source reference baseline.
  1. Parser-level validation & confound controls (Sec. 4.6–4.7). Raw streaming deltas inspected (no tool_calls delta in any suppression case → not a parser artifact). Confounds tested: framework (SGLang/vLLM), schema complexity (Simple/Medium/Production), tool choice (Optional / Required), scale (20B→397B), deployment (Local/Cloud), fine-tuning variants.
  1. Root-cause tracing (Sec. 4.8). Traced the API→tokenizer chain in SGLang 0.5.9 and directly inspected FSM states, showing <, , , websearch are all masked (✗) across the three representative JSON FSM states, whereas {, " are reachable only in their respective states.
  1. Mechanism analysis (Sec. 6). Two-layer interpretation: (a) decoding layer — grammar mask makes tool tokens unreachable, overriding any model preference regardless of weights (which is why SFT/GRPO cannot fix it — no gradient path back through the mask); (b) model layer — CPI as a behavioral hypothesis for TS-C-like patterns, kept as hypothesis.
  1. Mitigation: Transparent Two-Pass Execution (Sec. 7). Pass 1 (Tools ON / Schema OFF) freely invokes tools and collects outputs; Pass 2 (Tools OFF / Schema ON) receives tool results as context and emits the final schema-compliant response. Inference-time only; no weight/training/kernel changes. Cost: ~1 extra inference round + token overhead from re-transmitting context.

Results & Highlights

  • RQ1 — Reproducibility (Table 7). Under T1, TIR = 100% for all 8 models. Under T2, TIR = 0% for all 7 open-weight models (Qwen3.6-35B-A3B on both SGLang and vLLM, Qwen3.5-122B-A10B, GPT-OSS-20B, Nemotron 3 Super, Qwen3.5-397B-A17B, Qwen3-VL-235B-Thinking) → SR = 100% (complete suppression). The closed-source GPT-5.4-mini stayed at TIR 100% under T2 (SR 0%). T3 schema compliance was 80–100% across the board, proving each capability works alone.
  • RQ2 — Joint-constraint interaction. Suppression appears only in T2, never in T1 or T3, with identical prompts/tools/schemas → attributable to the interaction, not to either capability individually.
  • Schema complexity ablation (Table 8). T2 TIR = 0% for Simple (1–3 fields), Medium (5–10), and Production (20+) schemas alike — no threshold effect; even lightweight schemas trigger suppression.
  • Tool enforcement ablation (Table 9). T2 TIR = 0% under Optional, User-Prompt-Required, System-Prompt-Required, and even API-level tool_choice="required". Suppression persists past every prompt/API-level enforcement layer.
  • Fine-tuning ablation (Table 10). Base / Tool-Mandatory (200) / Schema-Injection (200) / GRPO (200) / Large SFT (6000) variants all kept T2 TIR = 0%. Conventional post-training does not remove suppression — consistent with the mask acting after logits, with no gradient path back to weights.
  • RQ3 — Framework independence (Sec. 5.6). SGLang and vLLM showed identical tool-execution behavior (T1 success, T2 full suppression); only minor JCR differences → not framework-specific.
  • Token-level exclusion evidence (Table 6). FSM state inspection confirms <, , , websearch are masked in all three sampled JSON FSM states; {/" reachable only in their proper states. This is the concrete implementation-level proof that tool tokens are unreachable by design.
  • Mitigation effectiveness (Table 11). Transparent Two-Pass Execution: TIR 0%→100%, JCR 100%→100% (unchanged), ESR 0%→100%, Avg Tool Calls/Session 0→5–8. Simulated-retrieval and tool-free hallucination patterns substantially reduced because real tool data now flows in.
  • Cost trade-off (Table 12). Two-Pass costs 2 LLM rounds + 5–8 tool calls vs. Single-Pass T2's 1 round + 0 calls; latency and token usage rise, but the baseline under suppression literally cannot complete tasks, so the trade-off is favorable for enterprise Agents.
  • Extended robustness (Appendix A.3.2). For Qwen3.6-35B-A3B, 200+ queries across 10 company profiles × 8 compliance markets (EU/US/Middle East/SEA/Australia/South America/Africa/Generic) all gave T2 TIR = 0%.

Critical Assessment

  • Novelty: High and timely. The original "Constraint Tax" concept (Ray, 2026, [9]) focused on answer-quality degradation under schema; this paper extends it to a fundamentally new and more severe failure mode — action-execution suppression in Agents — and, crucially, traces it to a concrete decoding-layer mechanism (FSM token masking) rather than stopping at a behavioral observation. The inference-stack trace through SGLang is the genuinely novel, load-bearing contribution.
  • Rigor: Strong on the empirical side (controlled T1/T2/T3 design, multiple confound ablations, parser-level validation, raw-stream inspection, FSM-state inspection). The mechanism claim (token unreachability) is a verifiable implementation fact, not a soft behavioral interpretation — that is the paper's strongest claim and it is properly evidenced. However, CPI as a "model-layer" explanation is under-evidenced: the authors never probe model internals (logits before masking, activations) to show CPI is a real phenomenon distinct from the mask; TS-C could simply be the model expressing intent in schema-allowed text before the mask blocks execution. The two-layer framing is reasonable but the second layer is essentially unfalsifiable as presented.
  • Reproducibility: Very good. Unified Python test harness, exact API params, model matrix, schema JSON, tool definitions, and an appendix listing test scripts by filename. The main reproducibility gap is that several "future" model names (Qwen3.5/3.6, GPT-OSS-20B, Nemotron 3 Super, GPT-5.4-mini) are 2026-vintage and may not be widely accessible; weights/quantization for local models are not pinned precisely.
  • Significance: High for the Agent-infrastructure and LLM-serving communities. The takeaway — "joint Tool Calling + Structured Output is a distinct evaluation regime; grammar-constrained decoding silently breaks XML-style tool calls; the fix is a two-pass orchestration" — is immediately actionable by any team deploying vLLM/SGLang-based Agents with response_format + tools. It also has clear implications for benchmark design (independent capability evals miss this).
  • Red flags / honest caveats:
  • The closed-source baseline comparison is under-interpreted and potentially misleading. Only one closed model (GPT-5.4-mini) is tested and it does not suppress; the authors speculate this is because it uses instruction-level rather than grammar-level enforcement, but provide no evidence. With n=1, "open-weight vs closed-source" framing is not justified — it could be an artifact of GPT's specific tool-call format (function-call JSON, not XML < tags) rather than open-vs-closed. The title's "Open-Weight LLMs" generalization is stronger than the evidence supports.
  • The mechanism is somewhat tautological for the Qwen family. If a model uses XML tags and the schema enforces strict JSON, then < being masked is almost definitional — the paper's contribution is naming and tracing this, not discovering an unexpected interaction. The surprising part would be whether models still try to call tools in their pre-mask logits, which the paper explicitly says it cannot determine.
  • Tool-call format is the obvious confound not fully isolated. XML-tag tool calls (Qwen) vs. function-call JSON tool calls (OpenAI-style) have very different compatibility with a JSON FSM. The paper does not test an open-weight model configured to emit function-call-style tool calls under the same schema, which would cleanly separate "format incompatibility" from "behavioral suppression."
  • The mitigation is a known engineering pattern, presented as novel. Two-pass / plan-then-format / decouple-tooling-from-formatting is standard in production Agent orchestration; the paper's contribution is showing why it is necessary here, not inventing it. The latency/token cost analysis is brief.
  • Scale of the main result. The headline T1=100%/T2=0% is striking but binary; there is no analysis of partial suppression regimes, of tasks where tools are only sometimes needed, or of how suppression interacts with multi-turn / multi-tool / parallel-tool-call settings.

Strengths

  • Traces the failure to a concrete, inspectable, implementation-level mechanism (FSM token bitmask with line-numbered SGLang source) rather than leaving it as a behavioral mystery — rare and valuable in empirical LLM papers.
  • Clean controlled T1/T2/T3 design with identical prompts/tools/schemas isolates the joint-constraint interaction convincingly.
  • Exhaustive confound ablation (framework, schema complexity, tool_choice, scale, deployment, fine-tuning including GRPO and 6k-sample SFT) makes the negative result robust.
  • Tool Suppression taxonomy (TS-A…TS-E) is a genuinely useful descriptive contribution with deception-risk annotations — TS-B (Simulated Retrieval) and TS-D (Tool-Free Hallucination) flag serious deployment-safety concerns.
  • Immediately deployable mitigation (Transparent Two-Pass) with before/after metrics on the same production pipeline; no retraining required.
  • Honest epistemic stance: CPI explicitly labeled a hypothesis, scope limits repeatedly acknowledged, no overclaiming about open-vs-closed universality.

Limitations

  • Single closed-source baseline (n=1). The "open-weight LLMs" framing in the title leans on GPT-5.4-mini being the only non-suppressing model; no second closed model, and no isolation of tool-call format (XML vs function-call JSON) as the true driver.
  • Tool-call format not isolated as a variable. The most plausible root cause — XML < tags are inherently incompatible with a strict-JSON FSM — is demonstrated but not experimentally separated from "open-weight behavioral" explanations by testing format variants on the same model.
  • CPI is under-evidenced. No internal-state probing (pre-mask logits, activations) to support the model-layer interpretation; TS-C is equally consistent with "model expresses intent then mask blocks execution."
  • Binary headline result. T1=100% / T2=0% leaves no room to study partial suppression, conditional tool need, or graded reliability — all of which matter in real deployments.
  • Mitigation cost under-analyzed. Only a small before/after table; no wall-clock or dollar-cost numbers, no comparison against alternative mitigations (e.g., grammar that explicitly permits tool tags, schema-relaxation, or hybrid function-call format).
  • Scope of mitigation validation. Tested mainly on single-turn tool-calling workflows; generalization to MCP ecosystems, multi-agent protocols, workflow agents, and computer-use agents is explicitly left open.
  • Limited academic benchmark coverage. Authors acknowledge the task set is smaller than large-scale Agent benchmarks; results are production-scenario-specific (foreign-trade inquiry analysis).

Relevance

对从事"开源 LLM Agent 生产部署 / vLLM、SGLang 推理框架 / Tool Calling + 结构化输出联合使用"的工程师与研究者高度相关且几乎是必读：它揭示了一个在真实生产环境中已被观测、可复现、且根因可定位到推理框架 token 掩码层的严重可靠性缺陷，并给出无需重训的工程缓解方案。对 Agent 评测基准设计者也有直接启发——独立评估工具调用与结构化输出会系统性漏掉这类联合约束失效；对关注 Agent 安全/可信赖性、constrained decoding、以及开源 vs 闭源模型行为差异的研究方向同样具有参考价值。

Questions for Further Exploration

  • Is the decisive variable tool-call format (XML vs function-call JSON) rather than open-vs-closed? Can an open-weight model configured to emit OpenAI-style function-call JSON tool calls still be suppressed under the same JSON Schema, and conversely can a closed model be made to suppress if forced into XML tags?
  • What do the model's pre-mask logits look like under T2 — does the model still assign high probability to tool-call tokens before the FSM mask zeroes them? This would distinguish "model wants to call tools but is blocked" (pure decoding-layer cause) from "model has genuinely deprioritized tool calling" (CPI as a real internal mechanism).
  • Can the grammar/FSM be extended to explicitly permit interleaved tool-call tokens while preserving JSON validity, and does that eliminate suppression without the latency cost of two-pass execution?
  • How does Tool Suppression behave in multi-turn, multi-tool, and parallel tool calling (e.g., the W&D parallel-tool scaling cited as [4]) settings — is it all-or-nothing or graded?
  • Does the phenomenon generalize beyond tool calling to other non-JSON token sequences agents must emit — workflow control tokens, multi-agent communication markers, MCP custom action tags, computer-use action sequences — as the paper speculates in Sec. 9.4?
  • How does Transparent Two-Pass Execution compare in cost/quality against alternative mitigations: schema relaxation, hybrid grammars, tool-call-format switching, or training-time alignment that accounts for the mask?
  • Can a standardized joint-constraint benchmark (Tool Calling + Structured Output as a first-class evaluation track, across XML/function-call/MCP formats and multiple schema complexities) be built to quantify suppression prevalence across the broader open-weight model landscape?
