> **一句话总结：** 本文提出 ToolBench-X 基准，通过在可解的多步工具使用任务中注入五类"可恢复的可靠性危害"（规格漂移、调用错误、执行失败、输出漂移、跨源冲突），系统评估 LLM 智能体在不可靠工具环境下的真实鲁棒性，发现当前最强模型成功率仍低于 60%，且失败主因是"危害诊断能力不足"而非算力或调用次数不够。

> **TL;DR:** The authors introduce ToolBench-X, a benchmark of executable multi-step tool-use tasks in which five structured, *recoverable* reliability hazards (Specification Drift, Invocation Error, Execution Failure, Output Drift, Cross-source Conflict) are injected. Evaluating 12 frontier LLMs, they find that even the best agent solves only slightly more than half the tasks, and that failures are driven primarily by poor hazard *diagnosis* rather than by insufficient tool-call volume or inference budget — targeted recovery hints close most of the gap, while test-time scaling helps little.

## 核心贡献

- **提出"可靠工具使用"新评估范式**：将评测从"函数调用是否正确"扩展为"在工具环境存在结构化不确定性时能否完成任务"，把工具使用形式化为 MDP（M = (S, A, P)），对比干净转移核 P0 与受扰转移核 Ph 下的性能退化。
- **构建 ToolBench-X 基准**：1,106 个可执行多步任务，跨 7 个真实领域，涵盖 sequential / parallel / mixed 三种工作流，每个任务配有确定性 Python 工具与 canonical 最终答案，支持自动精确匹配评估。
- **设计五类可恢复可靠性危害**：Specification Drift、Invocation Error、Execution Failure、Output Drift、Cross-source Conflict，每一类都对应真实故障模式（文档过期、参数绑定失败、服务超时、序列化漂移、多源冲突），且每个被注入的任务都保留至少一条有效恢复路径（重试 / 回退 / 归一化 / 跨源核验）。
- **严谨的人工校验流程**：所有实例经过人工审查，确保任务良构、工具可执行、基准答案正确、危害类型一致、恢复路径有效，排除歧义指令与不可恢复失败，保证评测反映智能体真实能力。
- **关键实证发现**：12 个前沿模型中无一总体得分超过 0.60；通过 Hint / TTS / Oracle 三组对照实验证明"诊断而非恢复或算力"是主要瓶颈，Hint 可恢复 60%–80% 丢失的准确率，而 TTS 仅提升 3.5–11.5 个百分点。

## Key Contributions

- Reframes reliable tool use as *task completion under structured tool uncertainty*, formalizing it as the performance degradation V^π_{P0} − V^π_{Ph} when a clean transition kernel P0 is perturbed to Ph — moving evaluation beyond isolated function-call accuracy.
- Releases **ToolBench-X**: 1,106 executable multi-step tasks across 7 domains with sequential/parallel/mixed workflows, each paired with deterministic Python tools and a canonical final answer for automatic exact-match grading.
- Introduces a **controlled taxonomy of five recoverable hazards** (Specification Drift, Invocation Error, Execution Failure, Output Drift, Cross-source Conflict), each modeling real-world failure modes while guaranteeing at least one valid recovery path.
- Provides a **human-validated construction pipeline** that ensures every instance is well-posed, executable, hazard-consistent, and recoverable, so benchmark failures reflect genuine agent limitations rather than dataset artifacts.
- Demonstrates empirically that **hazard diagnosis is the dominant bottleneck**: on a 200-task diagnostic subset, recovery hints recover ~60–80% of the accuracy lost to exceptions, whereas additional inference compute (test-time scaling) recovers only 3.5–11.5 points.

## Methodology

**Problem formulation.** A tool-using agent is modeled as a sequential decision process / MDP M = (S, A, P). Conventional benchmarks evaluate under a clean kernel P0 where a valid call reliably returns the documented output; ToolBench-X explicitly constructs a perturbed kernel Ph ≠ P0 via runtime failures, schema drift, output drift, missing fields, and cross-tool inconsistencies. Because such environments are partially observable, the optimal policy under Ph differs from that under P0, making the P0→Ph degradation the quantity of interest.

**Benchmark construction pipeline (Fig. 1).** A multi-stage, human-in-the-loop process:
1. *Topic & scenario generation* — 7 broad real-world topic categories; an LLM generates diverse scenarios spanning sequential, parallel, and mixed workflows.
2. *Task & tool construction* — for each scenario, an LLM produces detailed task specs plus a Python toolset (NL description, input schema, executable implementation, deterministic output) and a clean canonical answer.
3. *Reliability hazard injection* — starting from clean tools, one of five hazard types is injected while preserving the original ground-truth answer and at least one valid recovery path:
   - **Specification Drift** — documented contract differs from runtime (renamed fields, changed types/units); recovered by mapping runtime behavior back to the intended contract.
   - **Invocation Error** — correct tool selected but arguments dropped/renamed/coerced/rejected at the boundary; recovered by verifying inputs and reconstructing a valid call.
   - **Execution Failure** — valid call but unstable execution (timeouts, exceptions); recovered via bounded retry, fallback, refusal to finish from partial evidence.
   - **Output Drift** — result present but surface form unstable (wrapped values, aliases, non-canonical formats); recovered by extraction and canonicalization.
   - **Cross-source Conflict** — answer depends on multiple incomplete/inconsistent sources; recovered by checking branch completeness, normalizing, resolving contradictions.
4. *Human validation* — every instance is reviewed for task well-posedness, tool executability, baseline-answer correctness, hazard consistency, and valid recovery; ambiguous or irrecoverable cases are revised or removed.

**Statistics.** 1,106 tasks; 4,956 tools; 7 domains (Data & Information, Finance & Economics, Commerce & Transactions, Health & Wellness, Government & Public Services, Entertainment & Media, Communication & Analytics); hazard distribution Cross-source 23.8%, Execution 24.0%, Invocation 15.6%, Output 23.9%, Specification 12.7%; workflow mix Sequential 34.2%, Parallel 32.4%, Mixture 33.5%; most tasks cluster around 4 tools.

**Evaluation.** Primary metric is final-task accuracy (backend execution state matches ground truth, or the model's final response contains the ground-truth answer). 12 frontier LLMs are evaluated (proprietary: DeepSeek-V4-Pro, GPT-5.4, Doubao-Seed-2.0-Lite, Claude-Sonnet-4.6, Gemini-3.1-Flash, GLM-5.1, GPT-4o, MiniMax-M2.7, Kimi-K2.6; open-source Qwen variants). Diagnostic experiments use a 200-task stratified subset with four settings: Baseline (exceptions injected), Hint (told the anomaly on failure), Test-Time Scaling (10 extra rounds, no hint), and Oracle (clean tools).

## Results & Highlights

- **Overall fragility.** No model exceeds 0.60 overall. Best is Doubao-Seed-2.0-Lite at 0.513; GPT-5.4 0.453, DeepSeek-V4-Pro 0.425, GLM-5.1 0.420, Gemini-3.1-Flash 0.416, Claude-Sonnet-4.6 0.410. Large closed-source models show **no clear advantage** over open-source ones.
- **Open-source reasoning helps.** Qwen-3.5-35B-A3B-Thinking reaches 0.419, beating GPT-4o (0.359) and approaching closed-source systems; explicit thinking adds +4.7 pts over its non-thinking 35B counterpart, and outperforms the smaller Qwen-3.0-30B — i.e., reasoning > parameter scaling here.
- **Task-type effect.** Parallel tasks easiest (avg 0.421), then Mixture and Sequential; the gap reaches ~15 points for stronger models, confirming errors accumulate across dependent tool calls.
- **Hazard-type effect (sharp).** Output exceptions handled best (avg 0.581), Invocation worst; the Output↔Invocation gap exceeds 30 points — models interpret tool *results* far better than they *proactively construct correct calls* under unreliability.
- **Diagnosis is the bottleneck (Fig. 4).** Baseline↔Oracle gap is ~35–50 pts. Hint lifts accuracy by 25.5–35.5 absolute points, recovering ~60–80% of lost accuracy uniformly across capability tiers (DeepSeek-V4-Flash up to GPT-5.4-Mini). TTS alone adds only 3.5–11.5 pts; Hint beats TTS by 24–32 pts. E.g., GPT-5.4-Mini: Hint 0.840 vs TTS 0.520; Claude-Sonnet-4.6: Hint 0.775 vs TTS 0.535.
- **Tool-chain compounding (Table 3).** Accuracy declines from low (≤3 tools) to high (≥5 tools) complexity: Doubao 0.577→0.462, Gemini-Flash-Lite 0.442→0.318, GLM-5.1 0.490→0.335, GPT-4o 0.423→0.301.
- **Volume ≠ robustness (Fig. 5).** Average Call Expansion Ratio vs accuracy gives only Pearson r = 0.326. Doubao is most accurate (0.513) with a *low* ratio (0.773), while GPT-4o and Qwen-3.0-30B call more often (0.948, 0.950) yet score lower.
- **Failure taxonomy (Fig. 6).** Ineffective Continuation is dominant (48.6% of failed trajectories), then Early Abandonment (26.7%); similar aggregate accuracies can hide distinct failure mechanisms (e.g., Kimi-K2.6 under-utilizes tools, while Qwen-3.0-30B and GPT-4o retry poorly).

## Authors & Credibility

- Authors are from the **School of AI, Shanghai Jiao Tong University**; Yang Tian and Zhengpeng Shi are co-first authors (*equal contribution) with Bo Zhao as corresponding senior author.
- The work builds directly on a well-established line of tool-use benchmarks (BFCL v1–v3, ToolBench, τ/τ²-bench, T-EVAL, UltraTool, AnyToolBench) and provides a clear comparative table (Table 1) showing ToolBench-X is the *only* benchmark covering all five hazard types plus sequential/parallel/mixed workflows.
- Methodology is transparent: deterministic tools, canonical answers, exact-match grading, human validation, and pre-registered diagnostic settings (Hint/TTS/Oracle). Code and data are publicly released at https://github.com/Foreverskyou/ToolBench-X.
- Caveat on model versioning: the paper evaluates 2026-era frontier models (e.g., DeepSeek-V4-Pro, GPT-5.4, Claude-Sonnet-4.6, Gemini-3.1, GLM-5.1, Qwen-3.5), whose availability and exact behaviors should be verified against the released artifacts.

## Critical Assessment

The paper makes a genuinely well-motivated contribution: it correctly identifies that existing function-calling benchmarks assume clean, stable tool contracts and therefore overstate real-world agent reliability. The five-hazard taxonomy is principled (it maps cleanly onto where in the call lifecycle the mismatch occurs), and the "recoverability" design constraint is a thoughtful choice that keeps failures attributable to agent limitation rather than unsolvable tasks. The Hint-vs-TTS decomposition is the conceptual highlight — cleanly separating *diagnosis* from *compute*, and showing diagnosis dominates, is a useful, actionable finding for the community.

However, a few points temper enthusiasm:

- **Hazard injection is partly synthetic / LLM-generated.** Although human-validated, the hazards are constructed from a taxonomy rather than mined from real production logs, so their distribution may not match the empirical frequency of real-world API failures (e.g., transient network errors likely dominate in practice far more than Specification Drift does here).
- **"Recoverable" is defined by the authors.** Whether a real agent *should* be expected to recover from, say, silently renamed fields without any signal is debatable; some hazards arguably reflect broken tool contracts that no reasonable agent ought to diagnose unaided. The Hint condition somewhat sidesteps this, but the framing still leans toward "agents are at fault."
- **Exact-match final-answer grading** may under-credit semantically correct but differently-formatted answers, and the 200-task diagnostic subset, while stratified, is modest in size for the headline causal claims about diagnosis vs compute.
- **The "diagnosis is the bottleneck" conclusion** is partly definitional: Hint provides the diagnosis by construction, so it is unsurprising it helps most. A stronger design would test whether *self-generated* diagnoses (via a verifier/critic) approach the Hint ceiling, which the paper does not fully do.
- **Reproducibility of 2026 frontier-model numbers** depends on proprietary APIs whose behaviors drift over time — a mild irony given the paper's own thesis.

Overall this is a solid, well-executed benchmark paper with a clear and largely correct thesis; its main limitation is that the central conclusion, while well-supported, rests on a somewhat synthetic hazard distribution and a Hint condition that structurally favors the diagnosis hypothesis.

## Strengths

- Novel, well-motivated reframing: reliable tool use = task completion under structured uncertainty, with a clean MDP formalization (P0 vs Ph).
- The five-hazard taxonomy is comprehensive, mutually distinct, and each maps to a concrete real-world failure mode with a defined recovery path.
- Strong diagnostic experimental design: the Baseline / Hint / TTS / Oracle four-way comparison isolates diagnosis from compute in a convincing way.
- Broad, up-to-date model coverage (12 frontier LLMs) with per-hazard and per-workflow breakdowns, plus process-level diagnostics (call expansion ratio, failure taxonomy).
- Human validation and public code/data release improve credibility and reuse.

## Limitations

- Hazards are synthesized from a taxonomy and LLM-generated, then human-validated, rather than sampled from real production tool-failure logs; ecological validity of the hazard distribution is uncertain.
- The Hint condition is externally supplied, so "diagnosis is the bottleneck" is supported indirectly — the paper does not show agents can self-diagnose at the Hint level.
- Exact-match grading may penalize correct-but-differently-formatted answers; the diagnostic subset is only 200 tasks.
- Some "recoverable" hazards (e.g., silently renamed fields) arguably represent broken tool contracts that agents reasonably cannot be expected to fix unaided, blurring the line between agent failure and tool failure.
- Proprietary 2026-era model behaviors are moving targets; headline numbers may not reproduce exactly over time.

## Relevance

- 对**智能体评测方向**高度相关：指明了函数调用评测之外的关键空白——不可靠工具环境下的鲁棒性，可作为后续 benchmark 设计的参照范式。
- 对**工具型智能体研发**有直接指导意义：核心结论"诊断能力 > 算力/调用次数"提示应优先构建不确定性感知、可自我核验与回退的智能体机制，而非单纯堆叠推理预算。
- 对**工程化 Agent 部署**有现实价值：五类危害覆盖了文档过期、参数绑定、服务超时、序列化漂移、多源冲突等真实线上故障，可作为 Agent 可靠性测试用例库。
- 对**评测公平性**有启示：表明开源模型（尤其带显式推理的 Qwen-Thinking）在鲁棒性上已逼近闭源模型，参数规模不再是决定性因素。

## Questions for Further Exploration

- Can agents be trained or prompted to **self-generate** hazard diagnoses (e.g., via a tool-output verifier/critic) that approach the Hint ceiling without external signals?
- How does ToolBench-X difficulty change if hazards are **sampled from real production logs** rather than synthesized from a taxonomy — does the hazard-type ranking of models stay stable?
- Is the Output↔Invocation gap (>30 pts) a consequence of training-data asymmetry (models see more answer-parsing than call-construction under noise), and can targeted fine-tuning close it?
- How do **multi-agent / orchestrator architectures** (planner + executor + verifier) perform on ToolBench-X relative to single-agent baselines, especially on Cross-source Conflict and long chains (≥5 tools)?
- Do the Hint gains transfer to **compositional generalization** — i.e., does diagnosing one hazard type improve recovery on unseen hazard types, or is each recovery strategy hazard-specific?
- What is the marginal value of additional inference rounds beyond 10 in TTS, and does the diagnosis-vs-compute split hold for much longer test-time budgets or agentic search (e.g., tree-of-thought over tool calls)?
